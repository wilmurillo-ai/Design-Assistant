# 唯品会扫码状态轮询 Skill

本技能主要用于 AI 助手（Agent）在进行非阻塞登录交互时，负责在后台/前台轮询检查用户的扫码登录状态。唯品会扫码的基础发起与二维码生成已经集成到底层 Native Node.js 工具库中，此文档只介绍与 AI 交互相关的轮询控制。

## 准备工作
在开始执行前，必须确保系统已安装最新版本的 `vipshop-cli` 。
- 如果未安装或需要更新，请执行全局安装命令：`npm install -g vipshop-cli@latest`
安装或更新成功后，可通过 `vipshop --version` 确认版本，并直接在命令行调用 `vipshop` 命令。

## 使用场景

1. 先执行原生的 NPM 命令：`vipshop login` 启动登录流。
2. 控制台的输出会包含 `qrToken`，以及**二维码图片的本地保存路径**和**二维码的线上链接**。此时，如果是在聊天环境中，**必须将二维码的【链接】以及对应的本地【图片文件】在回复中全部返回给用户**，以便用户直接能在聊天窗口扫码。
3. 随后，AI 需要阻塞自身，主动发起状态轮询命令，直到该 `qrToken` 被用户拿手机扫码认证通过。

## 轮询指令 (Agent 职责)

为了保证交互的流畅性，`vipshop-cli` 内部不再包含死循环轮询。轮询逻辑由 Agent (AI) 负责：

1. **发起登录**: 执行 `vipshop login` 获取 `qrToken`。
2. **发送物料**: 立即向用户展示二维码链接和图片。
3. **主动轮询**: Agent 必须使用 `use_skill` 或命令工具，**每隔 3 秒**调用一次状态查询命令：
   ```bash
   vipshop login --poll <qrToken>
   ```
4. **轮询策略**: 
   - 每次调用后检查输出 JSON 中的 `status`。
   - **最多执行 10 次**轮询。
   - 如果 10 次之后状态仍非 `CONFIRMED`，Agent 应停止自动轮询，并告知用户：“已达到自动重试上限，请您扫描上方二维码并确认登录后，回复‘已扫码’通知我继续查询。”
5. **用户响应后的二次验证**：即使当用户主动回复“已扫码”或“已完成登录”，**Agent 绝不能仅凭用户口头表达就假设成功**。Agent 必须立即执行一次 `vipshop login --poll <qrToken>` 命令来查询真实的后端同步状态，只有当该命令返回 `CONFIRMED`（且 `tokenSaved: true`）时，才算真正登录成功，随后方可执行后续业务请求。

**轮询结果反馈解析：**
- **NOT_SCANNED**: 等待扫码，Agent 应在 3s 后再次尝试。
- **SCANNED**: 已扫码但未点确认，Agent 应在 3s 后再次尝试。
- **CONFIRMED**: 登录成功！Agent 停止轮询并继续之前的业务逻辑。
- **INVALID**: 二维码失效，提示用户重新发起登录。

## 状态与登出指令

### 1. 查看状态
Agent 可以随时通过以下命令检查用户当前是否已登录或 Token 是否过期：
```bash
vipshop status
```
**AI 行为建议**：在执行任何需要权限的搜索操作前，可以先静默执行此命令确认状态。

### 2. 登出
如果用户明确要求“退出登录”或“更换账号”，执行：
```bash
vipshop logout
```
执行后本地 token 将被清除。

## 技术细节
- 底层实现语言：基于 Native Node.js CLI。
- 零外部轮询依赖：完全依赖 `vipshop-cli` 自己的 HTTP 长连接或周期心跳握手，AI 无需编写任何 Fetch 与 Cookie 管理代码。
