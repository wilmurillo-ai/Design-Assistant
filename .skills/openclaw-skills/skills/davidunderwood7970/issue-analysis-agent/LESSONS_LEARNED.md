## 📋 一、错误清单

### 1. 字段名不匹配错误

| 项目 | 详情 |
|------|------|
| **问题** | JSON 中是"未解问题人 TOP5"，HTML 中读取的是 `unresolved_reporters` |
| **影响** | 数据显示错误，图表为空或显示错误数据 |
| **修复方案** | 统一字段名，确保 JSON 输出和 HTML 读取使用相同的字段名 |
| **如何避免** | 建立字段名对照表，前后端开发前确认字段命名规范 |

### 2. 数据硬编码错误

| 项目 | 详情 |
|------|------|
| **问题** | HTML 中数据是硬编码的，不是从 JSON 动态读取 |
| **影响** | 数据更新后 HTML 不更新，报告失去实时性 |
| **修复方案** | 修改 HTML/JS 从 JSON 接口动态读取数据 |
| **如何避免** | Code Review 时检查是否有硬编码数据，建立数据流检查清单 |

### 3. 标题打错字错误

| 项目 | 详情 |
|------|------|
| **问题** | "未解问题人"应该是"未解决问题人 TOP5" |
| **影响** | 用户体验差，显得不专业 |
| **修复方案** | 修正标题文字 |
| **如何避免** | 建立文案校对流程，发布前必须进行文字校对 |

### 4. 红色警告样式问题

| 项目 | 详情 |
|------|------|
| **问题** | 未解问题人图表有红色边框和红色标题 |
| **影响** | 用户困惑，误以为是错误或警告 |
| **修复方案** | 移除红色样式，使用统一的图表样式 |
| **如何避免** | 建立 UI 样式规范文档，明确各场景的颜色使用规则 |

### 5. COS 响应头设置错误 ⚠️ (多次修复)

| 项目 | 详情 |
|------|------|
| **问题** | Content-Disposition 设置不正确，导致浏览器下载而不是打开 |
| **影响** | 用户体验差，需要下载后打开，增加操作步骤 |
| **修复方案** | 设置 `Content-Disposition: inline` |
| **修复次数** | **3 次以上** (重复犯错！) |
| **如何避免** | 将 COS 配置写入标准化文档，添加到发布检查清单，设置自动化测试 |

### 6. 未解决问题模块 TOP10 显示为空

| 项目 | 详情 |
|------|------|
| **问题** | 数据字段名不匹配 |
| **影响** | 图表无法显示，关键信息缺失 |
| **修复方案** | 统一字段名 |
| **如何避免** | 建立 API 接口文档，前后端联调时验证所有字段 |

## 📖 二、最佳实践 (标准化流程)

### 2.1 开发流程

```
需求分析 → 字段设计 → 接口定义 → 开发实现 → 联调测试 → 发布检查 → 上线验证
```

### 2.2 字段命名规范

1. **统一使用英文 snake_case** 作为 JSON 字段名
2. **建立字段对照表**，记录 JSON 字段名与中文展示的映射关系
3. **前后端共同评审**字段设计，签字确认后方可开发

### 2.3 数据流规范

1. **禁止硬编码**：所有数据必须从接口动态读取
2. **数据验证**：前端收到数据后验证字段完整性和类型
3. **降级处理**：数据缺失时显示友好提示，而不是空白或错误

### 2.4 UI/UX 规范

1. **颜色使用规则**：
   - 🔴 红色：仅用于错误、警告、危险操作
   - 🔵 蓝色：信息、链接
   - 🟢 绿色：成功、完成
   - ⚫ 黑色/灰色：常规文本和边框

2. **文案规范**：
   - 标题必须完整、准确
   - 发布前进行文字校对
   - 使用统一的术语表

### 2.5 COS 配置规范

```javascript
// ✅ 正确配置 - 浏览器直接打开
Content-Disposition: inline
Content-Type: text/html; charset=utf-8

// ❌ 错误配置 - 触发下载
Content-Disposition: attachment
```

**将此配置写入模板代码，禁止手动修改！**

## 🤝 四、团队协作规范

### 4.1 角色职责

#### 🐛 找茬 (质量检查专家)

**职责**:
- 审查代码和输出，发现潜在问题
- 维护错误总结文档，记录重复犯错的问题
- 制定和更新检查清单
- 发布前进行最终质量验收
- 对重复犯错的问题提出警告和改进建议

**权力**:
- 有权阻止不符合质量的发布
- 有权要求返工和整改
- 有权更新检查清单和流程规范

#### 🎨 画师 (前端开发)

**职责**:
- 按照接口文档开发前端页面
- 确保数据从接口动态读取
- 遵循 UI/UX 规范
- 发布前完成自检清单
- 及时修复找茬提出的问题

**权力**:
- 有权要求清晰的接口文档
- 有权拒绝不合理的 UI 需求
- 有权提出技术改进建议

### 4.2 协作流程

```
1. 画师完成开发 → 提交自检报告
2. 找茬进行审查 → 提出问题清单
3. 画师修复问题 → 重新提交
4. 找茬再次审查 → 确认通过
5. 双方签字 → 允许发布
```

### 4.3 重复犯错处理

对于**重复犯错 3 次以上**的问题（如 COS 响应头设置）：

1. **升级处理**：报告给项目负责人
2. **流程改进**：将检查点自动化（如添加自动化测试）
3. **文档强化**：在显眼位置标注注意事项
4. **培训复盘**：组织团队复盘会议，分析根本原因

### 4.4 沟通原则

- **对事不对人**：指出问题，不指责个人
- **及时沟通**：发现问题立即反馈，不积累
- **记录在案**：所有问题和修复都要记录
- **持续改进**：每次错误都是改进流程的机会
