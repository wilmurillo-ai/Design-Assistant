# Promotion Targets

Use this guide when a learning is clearly durable enough to move beyond chronology or a lightweight log.

## Target selection

### Workflow guidance file
Promote here when the lesson changes:
- workflow sequencing
- delegation behavior
- task routing
- recurring execution discipline

Examples: `AGENTS.md`, process guide, operator handbook

### Tooling notes file
Promote here when the lesson is a durable tool or environment gotcha:
- auth quirks
- install/runtime caveats
- tool-specific warnings
- local environment notes

Examples: `TOOLS.md`, tooling notes, setup guide

### Principles or behavior file
Promote here only when the lesson affects stable behavior or communication principles.
Do this cautiously. Avoid changing persona or principles guidance based on one-off interactions.

Examples: `SOUL.md`, principles.md, behavior guide

### Decision or policy docs
Promote here when the lesson changes policy, authority, precedence, indexing rules, or system governance.

### Runbooks
Promote here when the lesson creates a reusable operational sequence or maintenance procedure.

### Backlog / feature request log
Promote here when the user requests a capability that is durable enough to revisit later.

## Promotion checklist

Promote only if most of these are true:
- recurring or likely to recur
- relevant to more than one task
- worth retrieving later
- not already covered by an existing durable doc
- concise enough to state as a reusable rule

## Anti-patterns

Do not promote:
- transient frustration
- one-off wording preferences unless explicitly durable
- low-value command output noise
- speculative rules that have not stabilized
