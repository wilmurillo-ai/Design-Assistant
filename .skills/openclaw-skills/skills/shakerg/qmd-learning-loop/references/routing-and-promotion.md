# Routing and Promotion Guide

## Routing rules

### Correction or misunderstanding
- Log chronology in a daily log or lightweight memory file.
- Promote only if it changes a stable rule or recurs.

### Command, tool, or workflow failure
- Log chronology in a daily log.
- If it is operationally useful or recurring, add an entry to an indexed incident or error log.
- If the fix becomes repeatable, update or create a runbook.

### User asks for a missing capability
- Add a durable request to an indexed backlog, strategy file, or feature-request log.
- If the feature is accepted and scoped, create a decision memo or project doc later.

### Better approach or recurring pattern
- If it is a workflow/tooling lesson, update a runbook or workspace guidance file.
- If it is a durable system rule, create or update a decision memo.
- If it is a behavior or communication principle, update the principles file cautiously.

## Promotion threshold

Promote when the learning is recurring, cross-cutting, policy-changing, or likely to save meaningful repeated effort.

## Anti-clutter rule

Prefer updating an existing durable doc over creating a new permanent file when the topic already has a home.

## QMD rule

If a durable rule may already exist, query QMD or inspect the relevant indexed docs before creating a new rule. Avoid duplicate policy documents.
