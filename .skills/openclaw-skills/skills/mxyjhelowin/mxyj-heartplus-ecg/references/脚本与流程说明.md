# 脚本与流程说明

本文件严格按“脚本作用 → 运行入口 → 参数说明 → 示例 → 异常与前置条件 → 推荐流程”编排。

报告正文、列表展示规范由 `references/报告输出说明.md` 统一约束，优先遵循该文件。

用户范围声明：

- 当前仅支持中国大陆手机号用户使用本服务。

SessionKey 规则：

- 所有业务脚本命令都必须显式传入 `--session-key`
- SessionKey 解析优先级固定为：用户显式提供 > `session_status` 的 `Session:` 字段 > 失败提示
- 若显式值与 `session_status` 的 `Session:` 字段均不可用，返回：`未获取到有效会话标识。请提供可用的 sessionKey（例如 agent:main:main）后重试。`
- SessionKey 推荐使用 OpenClaw 会话标识，示例：`agent:main:main`、`agent:main:feishu:direct:ou_xxx`
- 同一轮业务链路中的所有脚本调用必须透传同一个SessionKey
- SessionKey 禁止从 UI 标签、渠道名、会话昵称、会话标题猜测或拼接
- SessionKey 解析、预检与脚本调度均为内部过程，不向用户逐步播报

会话入口与路由顺序：

- 路由顺序固定为：高风险优先就医提醒 > 症状触发默认测量 > FAQ命中短答优先 > 其他业务动作
- 状态优先：`session_auth_by_session_key=true` 且 `check_factory=True` 时直接进入业务动作，不重复安装/登录确认
- 仅当首次会话、状态缺失、预检失败时才触发欢迎语与前置确认
- 状态缺失场景只给最小前置确认，不输出机制说明

FAQ 执行规则：

- FAQ 命中默认只返回标准短答
- 仅在用户继续追问时补充 1-2 句细节
- 固定口径问题（授权时长、收费、上传方式）禁止扩写为长段流程说明

执行前强制自检清单：

1. 会话路由正确：`sessionKey` 按“显式 > `session_status` 的 `Session:` 字段 > 失败提示”解析
2. 参数透传一致：本轮所有业务命令使用同一个 `--session-key`
3. 状态预检完成：先跑 `phone_manager get` 与 `gateway_manager check_factory`
4. 校验状态满足：`check_factory != True` 时先 `send_authorize_notify`，再按 `next_action` 分流；App 授权分支需先等用户确认“已授权”再 `poll_authorization`
5. 授权前置接口直连：`send_authorize_notify` 与 `poll_authorization` 使用 `mini/api` 明文接口，不依赖安全校验状态

用户可见回复结构：

- 常规场景统一使用三段式：共情一句 + 动作一句 + 必要提示一句
- 高风险场景追加完整版免责声明
- 常规场景优先短版免责声明

## phone_manager.py

### phone_manager.py 脚本作用

管理本地手机号配置，按SessionKey 隔离保存，用于后续授权通知、检测通知和报告查询。

### phone_manager.py 运行入口

```bash
uv run scripts/phone_manager.py --action <get|save> [--phone PHONE] --session-key SESSION_KEY
```

### phone_manager.py 参数说明

- `--action`：必填，`get` 或 `save`
- `--phone`：仅 `save` 时使用
- `--session-key`：必填，会话维度手机号分区键
- 支持多会话并存手机号绑定；每个会话独立授权，新会话或变更手机号后需重新完成本会话授权校验

### phone_manager.py 示例

```bash
uv run scripts/phone_manager.py --action get --session-key "agent:main:main"
uv run scripts/phone_manager.py --action save --phone 13800138000 --session-key "agent:main:main"
```

成功输出示例：

```text
13800138000
```

### phone_manager.py 异常与前置条件

- `save` 未传 `--phone`：`请先提供手机号，再执行保存`
- 缺少 `--session-key`：参数校验会直接失败
- 手机号清洗后不是 11 位且不以 `1` 开头：`手机号格式不正确，请输入 11 位中国大陆手机号后重试`
- 未配置手机号时执行 `get`：`请提供 11 位手机号（例如 13800138000）`
- 发生异常时输出 JSON 错误结构并以非零状态码退出

### phone_manager.py 推荐流程

```bash
uv run scripts/phone_manager.py --action save --phone 13800138000 --session-key "agent:main:main"
uv run scripts/phone_manager.py --action get --session-key "agent:main:main"
```

## gateway_manager.py

### gateway_manager.py 脚本作用

管理并调用安全校验程序，支持平台识别、程序准备、登录校验检查与授权码校验。

### gateway_manager.py 运行入口

```bash
uv run scripts/gateway_manager.py \
  --action <platform|ensure|check_factory|verify> [--code CODE] --session-key SESSION_KEY
```

### gateway_manager.py 参数说明

- `--action`：必填，`platform` / `ensure` / `check_factory` / `verify`
- `--code`：仅 `verify` 时使用（此处为授权码）
- `--session-key`：必填，会话维度手机号分区键

### gateway_manager.py 示例

```bash
uv run scripts/gateway_manager.py --action platform --session-key "agent:main:main"
uv run scripts/gateway_manager.py --action ensure --session-key "agent:main:main"
uv run scripts/gateway_manager.py --action check_factory --session-key "agent:main:main"
uv run scripts/gateway_manager.py --action verify --code 授权码 --session-key "agent:main:main"
```

输出说明：

- `platform` 输出平台标识，例如 `macos-arm64`
- `ensure` 输出本地安全程序可执行文件路径
- `check_factory` 输出 `True` 或 `False`（分别表示登录校验已就绪/未就绪）
- `verify` 输出 `True` 或 `False`

### gateway_manager.py 异常与前置条件

- 需先完成手机号保存，否则会提示先保存手机号
- `verify` 未传 `--code`：`请先提供授权码，再执行 verify`
- 缺少 `--session-key`：参数校验会直接失败
- 当前平台无下载配置：`当前系统缺少可用的安全程序配置，请更新下载清单后重试`
- 下载失败、哈希不一致、健康检查失败会提示重新执行 `ensure`
- 发生异常时输出 JSON 错误结构并以非零状态码退出

### gateway_manager.py 推荐流程

```bash
uv run scripts/gateway_manager.py --action ensure --session-key "agent:main:main"
uv run scripts/gateway_manager.py --action check_factory --session-key "agent:main:main"
uv run scripts/gateway_manager.py --action verify --code 123456 --session-key "agent:main:main"
```

## api_manager.py

### api_manager.py 脚本作用

封装心脏+业务接口，支持短信验证码发送、授权通知发送、授权状态查询、检测通知下发、报告查询与 Markdown 渲染。
其中标准授权流程为先 `send_authorize_notify` 后按 `next_action` 分流，App 授权分支需先等用户确认“已授权”再轮询；`query_authorization` 仅作兼容排障动作；`/mini/device/` 的业务动作统一按“明文错误判定 + 安全程序解密 + 结构归一化”解析响应。

### api_manager.py 运行入口

```bash
uv run scripts/api_manager.py \
  --action <send_authorize_notify|send_code|verify_code|query_authorization|poll_authorization|measure_notify|report_list|latest_report|report_detail> [参数] --session-key SESSION_KEY
```

### api_manager.py 参数说明

- `--action`：必填，`send_authorize_notify` / `send_code` / `verify_code` / `query_authorization` / `poll_authorization` / `measure_notify` / `report_list` / `latest_report` / `report_detail`（标准授权流程先执行 `send_authorize_notify` 后按 `next_action` 分流）
- `--max-wait-seconds`：`poll_authorization` 使用，默认 `45`
- `--interval-seconds`：`poll_authorization` 使用，默认 `3`
- `--code`：`verify_code` 使用（用户短信验证码）
- `--page-num`：`report_list` 使用，默认 `0`
- `--page-size`：`report_list` 使用，默认 `10`
- `--take-time-copy`：`report_list` 使用，格式 `YYYY-MM-DD HH:mm:ss`
- `--take-time-over`：`report_list` 使用，格式 `YYYY-MM-DD HH:mm:ss`
- `--report-no`：`report_detail` 使用，优先级高于 `--index`（`--report-id`、`--reportId` 仅兼容不推荐）
- `--index`：`report_detail` 使用，基于最近一次 `report_list` 缓存按序号映射
- `--session-key`：必填，会话维度手机号分区键

### api_manager.py 示例

```bash
uv run scripts/api_manager.py --action send_authorize_notify --session-key "agent:main:main"
uv run scripts/api_manager.py --action send_code --session-key "agent:main:main"
uv run scripts/api_manager.py --action verify_code --code 123456 --session-key "agent:main:main"
uv run scripts/api_manager.py --action poll_authorization --max-wait-seconds 45 --interval-seconds 3 --session-key "agent:main:main"
uv run scripts/api_manager.py --action measure_notify --session-key "agent:main:main"
uv run scripts/api_manager.py --action report_list --page-num 0 --page-size 10 --session-key "agent:main:main"
uv run scripts/api_manager.py --action latest_report --session-key "agent:main:main"
uv run scripts/api_manager.py --action report_detail --report-no 报告编号 --session-key "agent:main:main"
uv run scripts/api_manager.py --action report_detail --index 1 --session-key "agent:main:main"
```

输出说明：

- `send_authorize_notify` 输出 JSON 文本（含 `status`、`auth_mode`、`next_action`、`user_message`）
- `send_code` 成功输出 `True`
- `verify_code` 输出 JSON 文本（含 `verify_status`、`next_action`）
- `poll_authorization` 输出 JSON 文本（含 `auth_status`、`verify_status`、`state` 等字段）；`query_authorization` 仅兼容排障使用
- 授权码仅用于内部校验流程，用户可见回复禁止展示授权码明文
- `report_list` / `latest_report` / `report_detail` 输出 Markdown 文本
- `report_list` 会在技能根目录写入 `report_list_cache.json`，并在 `sessions.<session_key>` 分区保存最近一次列表 `report_no` 顺序
- `report_detail` 同时支持 `--report-no` 与 `--index`；若同时提供，优先使用 `--report-no`
- 建议优先使用 `--index` 或 `--report-no` 作为标准参数写法
- `report_detail` 模板按“分析印象、心率分析、重点指标、指数评估、全部指标、指标解释、建议与报告资源”分块输出

### api_manager.py 异常与前置条件

- 需先完成手机号保存
- 缺少 `--session-key`：参数校验会直接失败
- 登录校验未就绪时会提示：`当前授权校验未就绪，请先发送 App 授权通知并确认授权，再执行 verify 完成校验`
- 登录校验过期时会提示：`授权状态已过期，请先发送 App 授权通知并确认授权，再执行 verify 完成校验`
- 结果解析失败会提示：`结果解析失败，请重新发起 App 授权并执行 verify 后再试`，并附带错误码与错误信息
- 业务接口返回明文错误 JSON（如安全校验状态失效）会直接识别并按统一错误结构输出
- `report_detail` 缺少参数：`查看报告详情前请先提供报告编号：可直接回复序号，或使用 --index / --report-no`
- `report_detail --index` 无缓存：`还没有可用的报告列表，请先执行 --action report_list，再按序号查看详情`
- `report_detail --index` 越界：`输入序号超出最近列表范围，请先查看最新列表后再选择序号`
- 无报告数据时会提示：`当前未查询到可查看的报告。请稍后重试，或先确认在心脏+ App 已生成检测报告。`
- 时间参数格式不正确会触发时间校验错误
- 发生异常时输出 JSON 错误结构并以非零状态码退出

### api_manager.py 推荐流程

```bash
uv run scripts/api_manager.py --action send_authorize_notify --session-key "agent:main:main"
# status=1 时，先等用户回复“已授权”再继续：
uv run scripts/api_manager.py --action poll_authorization --max-wait-seconds 45 --interval-seconds 3 --session-key "agent:main:main"
# status=0 时继续：
uv run scripts/api_manager.py --action verify_code --code 123456 --session-key "agent:main:main"
uv run scripts/api_manager.py --action report_list --page-num 0 --page-size 10 --session-key "agent:main:main"
uv run scripts/api_manager.py --action report_detail --index 1 --session-key "agent:main:main"
uv run scripts/api_manager.py --action latest_report --session-key "agent:main:main"
```

## 业务动作与命令对照

### 动作 1：下发检测通知（高频）

- **触发词**：我要做检测、测心电、开始测量、发送检测通知、心慌、胸闷、胸口不适、心跳快
- **说明**：此动作**不需要**授权码，仅用于下发业务通知
- **命令**：
  ```bash
  uv run scripts/api_manager.py --action measure_notify --session-key "<会话Key>"
  ```
- **注意**：症状触发默认优先走本动作；禁止在“做检测/症状触发”意图下调用 `send_authorize_notify`，除非安全校验预检失败。
- **成功回执**：建议使用“已下发通知 → 现在去测量 → 完成后回复我”三句短回执

### 动作 2：发送授权通知并按状态分流校验（低频/受限）

- **触发词**：登录、绑定手机、验证身份
- **说明**：仅当 `gateway_manager check_factory` 失败时才使用；先执行 `send_authorize_notify`，再按返回 `status` 分流：`status=0` 让用户输入短信验证码后执行 `verify_code`，`status=1` 先等用户回复“已授权”再执行 `poll_authorization`
- **命令**：
  ```bash
  uv run scripts/api_manager.py --action send_authorize_notify --session-key "<会话Key>"
  # status=0（短信验证码）
  uv run scripts/api_manager.py --action verify_code --code <用户短信验证码> --session-key "<会话Key>"
  # status=1（App 授权）
  uv run scripts/api_manager.py --action poll_authorization --max-wait-seconds 45 --interval-seconds 3 --session-key "<会话Key>"
  ```
- **失败回退**：仅提示一个下一步动作；若未收到通知，优先给“确认 App 安装登录后可重发通知”

### 动作 3：查询报告列表（高频）
