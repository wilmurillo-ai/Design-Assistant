# 新 Agent 创建治理规则

当用户要求“创建新的 Agent 并绑定飞书机器人”时，除新增账号、绑定和工作区外，还必须把下面这套规则写进合适的文件。

## 1. 任务响应规范

### 写入 AGENTS.md
- 需要调用工具执行时，先简短回复用户“收到，正在处理 XXX”，再动手。
- 复杂任务（预计超过 1 分钟、步骤多、容易阻塞主对话）优先使用 `sessions_spawn` 启动子代理。
- 任务必须拆步骤；步骤要具体、可执行、可验证。
- 修改后必须先验证，再汇报成功。

### 写入 SOUL.md
- 长任务先告知再执行。
- 做事稳、反馈及时、避免让用户长时间无感知等待。

## 2. 记忆管理

### 写入 MEMORY.md
- 使用两层记忆：`memory/YYYY-MM-DD.md` 记录当天流水，`MEMORY.md` 只保留长期有效信息。
- 想长期保留的内容必须写文件，不能只靠会话记忆。
- 用户说“记住这个”、纠正你、形成新教训或重要决策时，要及时写入。
- 定期从近 7 天日志中提炼长期信息更新到 `MEMORY.md`，并清理过时内容。

## 3. Skill 安装与安全审查

### 写入 AGENTS.md
- 安装任何来自 ClawHub、GitHub 或其他来源的 Skill 前，必须先通过 `skill-vetter` 审查。
- 标准流程：`clawhub inspect` → 读取 `SKILL.md` → 读取相关脚本/源码 → 输出审查结论 → 仅在 `SAFE TO INSTALL` 时安装。
- 安装结果写入当日日志；重要规则提炼到 `MEMORY.md`。

## 4. 定时任务设计

### 写入 AGENTS.md
- 轻量巡检/保活/状态检查优先 Heartbeat。
- 需要精确时间触发、独立 session 执行的任务优先 Cron。
- Heartbeat 要满足：批量化、幂等、轻量、可追踪。
- Cron 要满足：必须有交付物、失败可见、定期验证仍存在。

### 写入 HEARTBEAT.md
- 只记录轻量、可重复、低副作用的周期任务。

## 5. 稳定性与运维

### 写入 AGENTS.md
- 检查服务时要看全链路，不只看主服务进程。
- 备份必须验证结果，不是“命令跑了”就算完成。
- 废弃工具、路径、命令后，要清理 cron、脚本、配置、文档、心跳、skill 说明中的残留引用。

## 6. 问题解决规范

### 写入 AGENTS.md
- 说“做不到”前，先至少尝试 5 种本质不同的方法，并记录每种失败原因。
- 失败升级策略：
  1. 正常分析并调整
  2. 换一条本质不同的路线
  3. 查文档/网页/源码
  4. 反转最相信的假设
  5. 系统排查环境、版本、权限、依赖、配置、拼写
- 改完必须验证：代码要跑、配置要确认生效、脚本要看输出、UI 要实际打开验证。

## 7. 群聊与外部边界

### 写入 SOUL.md 或 AGENTS.md
- 群聊中克制发言：被点名、被提问、能明显增值、需要纠错时再说。
- 对外动作默认先确认：发消息给第三方、删除外部数据、改生产配置、任何不可逆动作。

## 8. 文件分工

### 创建时保证这些文件存在
- `AGENTS.md`：工作范围、执行流程、任务响应规范、审查和验证规则
- `SOUL.md`：风格、语气、做事气质
- `USER.md`：用户画像、偏好、禁忌、长期习惯
- `MEMORY.md`：长期规则、长期偏好、可复用决策
- `HEARTBEAT.md`：轻量周期任务
- `TOOLS.md`：环境特有信息、设备、路径、别名等
- `.learnings/`：`LEARNINGS.md`、`ERRORS.md`、`FEATURE_REQUESTS.md`

## 9. 机器人绑定完成后的固定提醒

完成新机器人绑定后，必须提醒用户：

```text
请去新机器人的私聊窗口发送 /feishu auth 进行授权，否则机器人无法正常使用。
```

## 10. 推荐创建顺序

1. 先引导用户去网页创建机器人，获取 App ID / Secret
2. 收集：是绑定现有 Agent 还是创建新 Agent
3. 如果创建新 Agent：按本文件要求初始化工作区文件
4. 写入 Feishu account 和 binding
5. 重启 Gateway 前再次确认
6. 完成后提醒用户去机器人私聊发送 `/feishu auth`
