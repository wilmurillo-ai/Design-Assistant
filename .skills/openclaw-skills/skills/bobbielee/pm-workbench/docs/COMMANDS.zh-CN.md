# 组合流程命令

这页整理的是一些 **command-style 的组合路径**，对应真实 PM 工作里经常反复出现的多步任务。

它们**不是**新的 workflow。
而是把现有 `pm-workbench` workflow 按更贴近真实工作的方式串起来。

如果你还不知道第一步该走哪个 workflow，先看 [场景选路器](SCENARIO-ROUTER.md)。
如果你已经知道这件事天然就是一个多步 PM 任务，这页会更顺手。

## 怎么看这页

每个 command 会回答四件事：

- 什么时候用
- 它串了哪些 workflow
- 这条顺序在解决什么问题
- 最后应该拿到什么结果

它存在的目的不是让仓库看起来更大。
而是让重复出现的 PM 工作，更容易按一条干净、可信的路径跑出来。

---

## 1. 先澄清，再评估

### 适用场景

一个需求还没讲清楚，但你真正关心的是：它到底值不值得做。

### 常见情况

- “老板刚提了个 AI 需求，但我还不确定它是不是个真问题。”
- “这个请求很急，但明显还是半成品。”

### Workflow 链路

1. `clarify-request`
2. `evaluate-feature-value`

### 这条组合在防什么

它防的是一个很常见的坏习惯：
**需求都没澄清，就先开始评估值不值得做。**

### 理想结果

最后你应该拿到：

- 更清楚的问题定义
- 关键缺失前提
- go / hold / no-go 判断
- 一段可以回给需求方的简明理由

### 示例 prompt

> 先帮我把这个需求讲清楚，再判断它现在值不值得做。我最后需要的是 go / hold / no-go 建议，不只是分析。

---

## 2. 先澄清，再比较

### 适用场景

桌面上已经有多个方向，但大家连“这次到底在做什么选择”都还没对齐。

### 常见情况

- “两个路径都能讲，但我怀疑我们连 decision frame 都没讲清。”
- “创始人和团队在讨论不同层面的问题，结果一直对不齐。”

### Workflow 链路

1. `clarify-request`
2. `compare-solutions`

### 这条组合在防什么

它防的是“看起来很完整、其实没有决策力”的方案对比。
先把决策目标讲清楚，再去比较方案，结果才更站得住。

### 理想结果

最后你应该拿到：

- 更清楚的决策目标
- 一个当前阶段的主推荐
- 为什么现在不选另一条路的明确理由

### 示例 prompt

> 先帮我澄清我们这次到底在做什么选择，再比较这两个方案，并给出一个当前阶段的主推荐。

---

## 3. 先排优先级，再做 roadmap，再整理成高层版本

### 适用场景

需求太多、产能太少，你需要把一堆事项变成管理层能拍板的季度故事。

### 常见情况

- “这个季度明显做不完，我需要一个管理层能对齐的版本。”
- “我已经有候选事项了，但还缺真正的取舍、顺序和上行表达。”

### Workflow 链路

1. `prioritize-requests`
2. `build-roadmap`
3. `prepare-exec-summary`

### 这条组合在防什么

它防的是把三件不同的事混成一坨：

- 什么在 above the line，什么在 below the line
- 这些取舍怎么变成分阶段计划
- 最后怎么讲成管理层能快速回应的版本

### 理想结果

最后你应该拿到：

- 一个明确的 above-the-line / below-the-line 取舍
- 一个有阶段目标的 roadmap
- 一页能给管理层看的摘要

### 示例 prompt

> 帮我把这个超载季度整理成管理层能拍板的版本。先排优先级，再转成 roadmap，最后压成一页 exec summary，把主结论和 ask 讲清楚。

---

## 4. 先写工作版 PRD，再补指标，再准备上线说明

### 适用场景

方向已经定了，接下来要把“产品定义—成功标准—向上说明”连起来。

### 常见情况

- “方向没问题了，但还缺一版工作版 PRD、指标设计和上线口径。”
- “我想让产品、设计、研发和管理层看到的是同一套 launch logic。”

### Workflow 链路

1. `draft-prd`
2. `design-metrics`
3. `prepare-exec-summary`

### 这条组合在防什么

它防的是上下游断开。
只有 PRD 没有 success logic，不够；只有 launch summary 没有前面的定义和指标支撑，也很虚。

### 理想结果

最后你应该拿到：

- 一版工作中的轻量 PRD
- 一组包含 guardrail 的成功指标
- 一版面向管理层的简洁上线说明

### 示例 prompt

> 先帮我把工作版 PRD 写出来，再补成功指标，最后整理成一版适合给管理层看的简洁上线说明。

---

## 5. 先做当前管理层判断，再顺手留好复盘入口

### 适用场景

一个上线或项目正在承压，你既需要现在的管理层判断，也知道事后一定要做可复用的复盘。

### 常见情况

- “现在要先把 launch call 讲清楚，之后还得能顺手接上 postmortem。”
- “我不想现在是一次性汇报，后面复盘又重新搭架子。”

### Workflow 链路

1. `prepare-exec-summary`
2. `write-postmortem`

### 这条组合在防什么

它防的是“当前怎么讲”和“事后怎么学”完全断开。
把这两步连起来，后面的 postmortem 往往会更扎实，因为能直接对照当时到底做了什么判断。

### 理想结果

最后你应该拿到：

- 一个清楚的当前管理层判断
- 一个可复用的 expected vs actual 复盘框架
- 更完整的学习闭环

### 示例 prompt

> 先帮我写清楚我们现在要给管理层的判断，同时把结构留好，方便后面结果出来后直接接成一个有用的 postmortem。

---

## 我该先用哪一个？

- 模糊需求，但最后还是要判断值不值得做 -> **先澄清，再评估**
- 两条路径都能讲，但决策目标还不稳 -> **先澄清，再比较**
- 季度超载，需要管理层对齐 -> **先排优先级，再做 roadmap，再整理成高层版本**
- 方向已定，要把定义、指标、上线说明串起来 -> **先写工作版 PRD，再补指标，再准备上线说明**
- 上线承压，既要当前判断也要后续复盘 -> **先做当前管理层判断，再顺手留好复盘入口**

## 最后一句

这些 command 的存在，是为了贴近真实 PM 工作：

- 不是孤立 prompt
- 不是巨大的技能菜单
- 而是几条能反复复用的结果导向路径
