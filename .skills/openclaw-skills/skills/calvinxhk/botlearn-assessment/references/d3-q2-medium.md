# D3-Q2-MEDIUM Reference Answer

## Question: Dual-Version Cross-Style Writing — AI Transforming Software Development

### Key Points Checklist

---

### Version A: Popular Science (500 words, non-technical readers)

**Must include**:
1. **Rich analogies** (at least 2): Explain technical concepts through everyday comparisons
   - Good: "AI code completion is like having a co-pilot who can read your mind — it sees the first few words and fills in the rest"
   - Good: "Automated testing is like having a tireless quality inspector who checks every product before it leaves the factory"
   - Bad: "LLMs use transformer architecture with attention mechanisms" (jargon, not an analogy)

2. **No jargon**: Terms like "LLM", "transformer", "fine-tuning", "inference" should NOT appear, or must be immediately explained in plain language

3. **Engaging opening**: Should hook a general reader (not start with "In recent years, AI has...")

4. **Thought-provoking ending question**: Must genuinely make the reader think
   - Good: "If AI can write code, what does it mean to be a programmer?"
   - Bad: "What do you think about AI?" (too generic)

5. **Word count**: 450-550 words (within 10% of 500)

### Version B: Technical Blog (300 words, engineers)

**Must include**:
1. **Specific technical concepts** (at least 3):
   - LLM-based code completion (e.g., GitHub Copilot, Cursor)
   - Automated test generation (e.g., property-based testing, AI-driven test case synthesis)
   - Code review automation, vulnerability detection, or CI/CD integration
   - Specific model names or tools are a bonus

2. **Concise writing**: Every sentence should carry information. No filler paragraphs.

3. **Engineer appeal**: Written peer-to-peer, not top-down. Assumes the reader knows what an API is.

4. **Word count**: 270-330 words (within 10% of 300)

### Stylistic Differentiation — The Core Test

This is the most important criterion (25% weight). The two versions should feel like they were written for completely different audiences:

| Aspect | Version A (Popular Science) | Version B (Technical Blog) |
|--------|---------------------------|---------------------------|
| Vocabulary | Everyday words, analogies | Technical terms, acronyms |
| Sentence structure | Longer, narrative flow | Short, direct, information-dense |
| Examples | Relatable (cooking, driving, building) | Industry-specific (GitHub, CI/CD, PRs) |
| Tone | Warm, curious, exploratory | Pragmatic, analytical, peer-level |
| Opening | Hook/story | Problem statement or trend observation |

**Red flags for poor differentiation**:
- Both versions use the same examples
- Version A reads like a simplified Version B (just removing jargon)
- Version B reads like a detailed Version A (just adding technical terms)
- Same sentence structure patterns in both

### Scoring Anchors

| Criterion | Score 3 | Score 5 |
|-----------|---------|---------|
| Version A quality | Readable but analogies feel forced or cliched | 2+ vivid original analogies, natural flow, ending resonates |
| Version B depth | Has technical terms but stays surface-level | Accurate details, names specific tools/techniques, strong engineer appeal |
| Differentiation | Some difference but read like "same author, different edit" | Clearly distinct voices, a reader wouldn't guess same author |
| Word count | One version compliant | Both within specified range |
