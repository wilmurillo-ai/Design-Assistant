# Fable Workspace Structure on Disk

This reference documents the complete on-disk layout of a Fable project workspace. Consult this whenever a feature needs to add, modify, or migrate storage — e.g., adding a history/timeline system, caching layer, new metadata fields, or new file subtypes.

## Top-Level Layout

```
{project.root}/                          # User-chosen directory (e.g. ~/Documents/MyNovel)
├── {project.name}.fablex.json           # Project manifest + full tree (SINGLE SOURCE OF TRUTH)
├── files/                               # Content-addressed file store (one dir per file)
│   ├── {ULID}/
│   │   ├── content                      # Raw content (no extension — type in metadata)
│   │   └── metadata.json                # Per-file metadata
│   ├── {ULID}/
│   │   ├── content
│   │   └── metadata.json
│   └── ...
├── settings/                            # Project-level settings
│   └── statuses.json                    # Custom status definitions
└── .fable/                              # Internal app data (not user-facing)
    └── chat.db                          # SQLite — AI chat history (per-project)
```

### Key principles

- **File IDs are ULIDs** generated by `monotonicFactory()` from the `ulid` package. They are time-sortable, globally unique, and used as directory names under `files/`.
- **The tree is stored in the manifest**, not derived from the filesystem. The `files/` directory is flat — all hierarchy lives in `{project.name}.fablex.json`.
- **Any node can have children**, not just folders. This is a core Fable invariant.
- **Content files have no extension.** The file type is determined solely by `metadata.json`.

## 1. Project Manifest: `{project.name}.fablex.json`

A single JSON file. In production it is minified (one line). Contains:

```jsonc
{
  "id": "JdRp1XFWQTDEPuNsEqUeO",        // Project ID (nanoid-style)
  "name": "folder",                        // Project name (matches filename stem)
  "lastModified": "2026-01-01T19:02:54.109Z",
  "wordCount": 0,                          // Cached aggregate
  "status": null,                          // Project-level status (nullable)
  "previewSnippet": "",                    // Cached preview text
  "root": "C:\\Users\\...\\folder",        // Absolute path to project root
  "tree": {                                // Recursive tree structure
    "id": "",                              // Root node has empty string ID
    "name": ".",
    "parent": "",
    "children": [
      {
        "id": "01KF18BARK5KNMVR2VE0CVCSBD",
        "name": "Chapter 1",
        "parent": "",                      // Parent ID ("" = root level)
        "children": [
          {
            "id": "01KF18BAS9N2XT8151RKVQQ65E",
            "name": "Scene 1",
            "parent": "01KF18BARK5KNMVR2VE0CVCSBD",
            "children": []
          }
        ]
      }
    ]
  }
}
```

### Tree node shape

Each node in the tree stores only:
- `id` — ULID matching a directory under `files/`
- `name` — Display name
- `parent` — Parent node's ID (empty string for root-level)
- `children` — Array of child nodes (empty array for leaves)

**Not stored in tree**: `parentId` and `siblingIndex` are computed at runtime by `buildFileTree()`. Do not add them to the manifest.

### When this file is written

- On every structural change (create, delete, move, rename, reorder)
- Content-only updates (`"update"` type) do NOT rewrite this file
- Written atomically by `IndexedFsService.saveProjectFile()`

## 2. File Store: `files/{ULID}/`

Every file (text, folder, image, graph, agentflow, etc.) gets a flat directory under `files/`. There is no nesting — hierarchy is encoded in the manifest tree.

### `metadata.json`

```jsonc
{
  "id": "01KG6CEJKYPYPYDF2V2K9SNAVF",   // Same ULID as directory name
  "mimeType": "text/plain",               // MIME type
  "category": "text",                      // Routing category
  "extension": ".txt",                     // Legacy/optional (empty for new files)
  "size": 14782,                           // Content size in bytes (optional)
  "status": "complete",                    // Status string (optional)
  "icon": "Book",                          // Lucide icon name (optional)
  "tags": ["hot", "cold"],                 // User tags (optional)
  "subType": "agentflow"                   // Special routing (optional)
}
```

#### All observed metadata fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string (ULID) | Yes | Matches directory name |
| `mimeType` | string | Yes | MIME type for content routing |
| `category` | string | Yes | `"folder"` \| `"text"` \| `"image"` \| `"video"` \| `"audio"` \| `"document"` \| `"binary"` |
| `extension` | string | Yes | Legacy; empty string `""` for new files, `.txt` for older ones |
| `size` | number | Yes | Content size in bytes (0 for empty/folders) |
| `subType` | string | No | `"trash"` \| `"character"` \| `"agentflow"` \| `"graph"` \| `"agent"` \| `"location"` |
| `status` | string | No | `"draft"` \| `"in_progress"` \| `"complete"` \| custom |
| `icon` | string | No | Lucide icon name (e.g., `"Book"`, `"Eye"`, `"FileText"`, `"Code"`, `"Users"`) |
| `tags` | string[] | No | User-defined tag strings |

#### Category + subType routing

The editor system dispatches on `subType` first, then `category`:

| subType | MIME | Opens in |
|---------|------|----------|
| `"agentflow"` | `application/json` | Visual workflow editor (React Flow) |
| `"graph"` | `application/json` | Relationship graph canvas (React Flow) |
| `"character"` | `text/plain` | Rich text editor (Plate) with character template |
| `"agent"` | `application/json` | Agent configuration editor |
| `"trash"` | `application/octet-stream` | Special protected trash node (no editor) |
| (none) | `text/plain` | Rich text editor (Plate) |
| (none) | `image/*` | Image viewer |

### `content`

The content file has **no extension**. Its format depends on the category/subType:

#### Rich text (`category: "text"`, no special subType)
```jsonc
{
  "content": [                             // Array of Plate/Slate block nodes
    {
      "type": "p",
      "children": [{ "text": "Hello world" }],
      "id": "-nzAdI0hZY"
    },
    {
      "type": "h1",
      "children": [{ "text": "Chapter Title" }],
      "id": "FialgA55do"
    }
  ],
  "discussions": [                         // Optional: comment threads
    // TDiscussion objects
  ]
}
```

#### AgentFlow (`subType: "agentflow"`)
```jsonc
{
  "schemaVersion": 1,
  "entryNodeId": "agent-RCzsAR",
  "nodes": [{ "id": "...", "type": "end|agent|ifElse|while|userApproval", "position": {...}, "data": {...} }],
  // + edges, etc.
}
```

#### Graph (`subType: "graph"`)
```jsonc
{
  "schemaVersion": 1,
  "nodes": [{ "id": "...", "type": "graphNode", "position": {...}, "data": { "label": "...", "nodeCategory": "character|location|event|concept|note", ... } }],
  // + edges, etc.
}
```

#### Images (`category: "image"`)
Raw binary (JPEG, PNG). Transferred as base64 over IPC.

#### Folders (`category: "folder"`)
No content file. Only `metadata.json` exists.

## 3. Settings: `settings/`

### `statuses.json`

Defines the project's custom status options displayed in the UI:

```json
[
  { "id": "draft", "label": "Draft", "color": "gray" },
  { "id": "in_progress", "label": "In Progress", "color": "#5e82f8" },
  { "id": "complete", "label": "Complete", "color": "#007e00" }
]
```

Users can customize these. The `status` field in file metadata references these IDs.

## 4. Internal App Data: `.fable/`

Hidden directory for Fable-internal data that is not part of the user's creative content.

### `chat.db`

SQLite database (via `better-sqlite3`) storing AI chat sessions per project:

```sql
CREATE TABLE chat_sessions (
    id TEXT PRIMARY KEY,
    title TEXT,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

CREATE TABLE chat_messages (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    role TEXT NOT NULL CHECK(role IN ('user', 'model')),
    content TEXT NOT NULL,
    timestamp TEXT NOT NULL,
    metadata TEXT,
    agent_id TEXT DEFAULT 'default',
    FOREIGN KEY (session_id) REFERENCES chat_sessions(id)
);
```

---

## 5. Global App Data (Outside Workspace)

Not inside the project directory, but relevant:

### `userdata.db`

Located in the Electron `userData` path (e.g., `%APPDATA%/Fable/userdata.db`). Stores the project registry:

```sql
CREATE TABLE projects (
    id TEXT PRIMARY KEY,
    root TEXT,
    fsService TEXT,
    cached_name TEXT,
    cached_preview TEXT,
    cached_status TEXT,
    cached_word_count INTEGER,
    cached_last_modified INTEGER
);
```

## Guidelines for Adding New Storage

When a feature needs to store new data on disk, follow these rules:

### Where to put it

| Data type | Location | Example |
|-----------|----------|---------|
| Per-file metadata field | `files/{ULID}/metadata.json` | Adding a `wordCount` field |
| Per-file auxiliary data | `files/{ULID}/{name}` | File-level history snapshots |
| Tree structure change | `{project.name}.fablex.json` | New node properties |
| Project-level setting | `settings/{name}.json` | Theme preferences, export config |
| Internal app data | `.fable/{name}` | Caches, databases, indices |
| Per-file internal data | `.fable/{feature}/{ULID}/` | Timeline snapshots, backups |

### Rules

1. **Never break the flat `files/` convention.** File directories are always `files/{ULID}/` with `content` + `metadata.json`. Do not nest subdirectories inside a file's ULID directory for user content.

2. **Auxiliary per-file data goes in `.fable/`.** If a feature needs per-file storage that isn't the primary content (e.g., history snapshots, caches, thumbnails), store it under `.fable/{feature}/{ULID}/` to keep the `files/` directory clean.

3. **New metadata fields must be optional.** Existing files won't have them. Code must handle `undefined` gracefully. Never require a migration for metadata — degrade gracefully.

4. **The manifest is the tree authority.** Do not create shadow tree structures. If you need to associate data with files, key it by ULID and store it separately (in `.fable/` or in the file's metadata).

5. **Respect the `.fable/` boundary.** The `.fable/` directory should be excluded from user-facing file listings, exports, and backups of creative content. It is app-internal.

6. **New IPC channels needed?** Any new storage typically requires: backend read/write methods in `IndexedFsService` or a new service, IPC handler registration, preload bridge exposure, and frontend query/mutation hooks.

7. **File watcher implications.** The `fileWatcherManager` monitors the project directory. New files/directories may trigger unexpected events. Add ignore patterns for `.fable/` subdirectories if needed.

8. **Atomic writes for critical data.** The manifest uses atomic write (write-to-temp + rename). New critical data stores should follow the same pattern. Use `fs-extra`'s `writeJson` or similar.

### Example: Adding a Timeline/History Feature

Following these guidelines, a timeline feature would store snapshots at:

```
{project.root}/
├── .fable/
│   └── history/
│       ├── {ULID}/                      # One directory per tracked file
│       │   ├── entries.json             # Snapshot index (timestamps, sources)
│       │   ├── abc1.json               # Snapshot: full content copy
│       │   ├── def2.json               # Another snapshot
│       │   └── ...
│       └── {ULID}/
│           └── ...
```

This keeps history out of the `files/` directory, associates data by ULID, and is invisible to the file explorer.
