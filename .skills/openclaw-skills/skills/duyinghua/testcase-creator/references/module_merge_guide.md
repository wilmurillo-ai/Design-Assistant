# 功能模块归并指南

## 核心原则

**🚨 按完整功能流程组织测试，而非按需求文档章节划分**

测试用例是给测试人员用的，应按测试流程组织，而非按需求文档章节组织。需求文档章节可能按产品设计思路、开发实现思路、文档阅读便利性等组织，但测试用例结构应按测试执行思路、用户使用流程组织。

## 步骤1：识别章节关系

需求文档的章节通常有以下几种组织方式：
- **按功能模块划分**：每个章节对应一个独立功能
- **按功能维度划分**：同一功能的不同维度分散在不同章节
- **按实现层次划分**：前端展示、后端逻辑、数据模型等分开描述

## 步骤2：判断章节是否需要归并

使用以下判断标准：

1. **功能一致性检查**：
   - 这些章节是否描述同一个功能？
   - 用户是否会在同一个使用场景中使用？
   - 测试时是否需要一起验证？

2. **引用关系检查**：
   - 章节中是否有"详见XX章节"、"见XX部分"、"XX逻辑见XX"等引用？
   - 引用的内容是否是同一功能的实现细节？

3. **维度分析**：
   - 章节A描述的是功能的"是什么"（界面、交互）
   - 章节B描述的是功能的"怎么实现"（触发逻辑、业务规则）
   - 这种情况应归并为一个测试模块

## 步骤3：设计测试用例结构

对于每个识别出的功能模块，按以下流程组织测试：

```
功能模块
└── 子功能/入口/页面
    ├── 触发条件验证（什么情况下出现）
    ├── 业务规则验证（如何判断/计算）
    ├── 数据准备验证（需要什么数据）
    ├── 展示内容验证（展示什么内容）
    ├── 交互行为验证（用户如何操作）
    └── 结果处理验证（操作后的结果）
```

## 步骤4：通用示例说明

**示例1：章节描述同一功能的不同维度**
```
需求文档结构：
第一章 功能界面设计
  1.1 页面布局
  1.2 交互流程
  1.3 展示规则

第三章 业务逻辑实现
  3.1 触发条件
  3.2 计算规则
  3.3 异常处理

✅ 正确的测试用例结构：
一、XX功能
  1.1 页面/入口A
    - 触发条件验证（来自第三章）
    - 业务规则验证（来自第三章）
    - 页面展示验证（来自第一章）
    - 交互流程验证（来自第一章）
  1.2 页面/入口B
    - ...
```

**示例2：章节描述同一功能的不同层次**
```
需求文档结构：
第二章 用户界面
  2.1 登录页面
  2.2 注册页面

第四章 接口设计
  4.1 登录接口
  4.2 注册接口

第六章 数据校验规则
  6.1 用户名校验
  6.2 密码校验

✅ 正确的测试用例结构：
一、用户认证功能
  1.1 登录功能
    - 登录页面展示验证（来自第二章）
    - 登录接口调用验证（来自第四章）
    - 用户名校验规则验证（来自第六章）
    - 密码校验规则验证（来自第六章）
    - 登录成功流程验证
    - 登录失败场景验证
  1.2 注册功能
    - ...
```

## 步骤5：归并原则总结

| 需求文档章节关系 | 是否归并 | 归并方式 |
|-----------------|---------|---------|
| 不同功能的描述 | ❌ 不归并 | 各自独立成模块 |
| 同一功能的不同维度 | ✅ 归并 | 融入同一模块的测试流程 |
| 同一功能的不同层次 | ✅ 归并 | 融入同一模块的测试流程 |
| 功能描述+实现细节 | ✅ 归并 | 实现细节融入功能测试 |
| 功能描述+策略规则 | ✅ 归并 | 策略规则融入功能测试 |

## 步骤6：质量检查

归并完成后，检查以下内容：
- [ ] 测试人员能否在一个模块下完成该功能的完整测试流程？
- [ ] 是否需要在多个章节/模块间来回跳转？
- [ ] 每个功能的测试是否包含：触发→处理→展示→交互→结果？
- [ ] 是否有遗漏的需求细节？
