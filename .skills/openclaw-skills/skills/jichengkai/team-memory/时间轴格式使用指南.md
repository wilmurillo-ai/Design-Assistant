# 时间轴格式使用指南

## 📖 为什么用时间轴格式？

相比之前的事件列表，时间轴格式更符合**人类记忆习惯**：

1. **按时间顺序**: 像日记一样自然，容易追溯
2. **上下文完整**: 每个事件包含当时的情况、你的观察、后续行动
3. **可读性强**: 直接打开 Markdown 文件就能阅读，不需要 AI 解析
4. **开源可迁移**: 纯 Markdown，任何工具都能打开，终身可用

## 📁 文件结构

```
data/members/
├── 张三-时间轴.md      # 个人时间轴（按人组织）
├── 李四-时间轴.md
├── 王五-时间轴.md
└── 模板-成员时间轴.md  # 创建新成员时复制这个
```

**对比之前的方案**:

| 方案 | 优点 | 缺点 |
|------|------|------|
| **时间轴**（新） | 直观、可读性强、手工编辑友好 | 文件稍大 |
| **事件列表**（旧） | 结构化、AI 处理方便 | 需要 AI 解析才能阅读 |

**建议**: 时间轴为主，AI 辅助整理

## ✍️ 如何记录

### 日常记录（推荐）

**方式1: 直接编辑文件**（快速、直接）

打开 `张三-时间轴.md`，在顶部添加：

```markdown
### 2024-01-26（周五）
#### 15:30 - 代码Review
**事件**: 张三Review了我的PR，提出了3个很好的优化建议  
**类别**: 协作沟通  
**评价**: ⭐⭐⭐⭐ 良好  
**标签**: #代码质量 #技术深度

**观察笔记**:  
- 对代码质量要求高
- 能看出潜在的性能问题
- 表达方式友好，建议有建设性

**后续行动**:  
- [x] 已按建议修改
- [ ] 学习他提到的设计模式

---
```

**方式2: 通过 OpenCode**（方便、自然语言）

对 OpenCode 说：
```
记录：张三今天Review了我的代码，提出了很好的优化建议，表达方式也很友好
```

OpenCode 会帮你：
1. 提取关键信息（时间、人物、事件）
2. 自动分类和评级
3. 追加到对应的时间轴文件顶部

### 定期整理（每周/每月）

**每周五**（5分钟）：
1. 看一眼每个人的时间轴顶部
2. 更新"阶段总结"部分
3. 调整"发展计划"中的待办

**每月底**（10分钟）：
1. 让 OpenCode 帮你生成月度总结
2. 更新"数据汇总"表格
3. 调整"管理笔记"

## 🎯 核心字段说明

### 每条记录必须包含

```markdown
### {日期}（星期X）
#### {时间} - {一句话标题}
**事件**: {发生了什么，要具体}  
**类别**: {技术能力/协作沟通/项目交付/团队影响/成长潜力}  
**评价**: {星级或标签}  
**标签**: #{关键词}

**观察笔记**:  
- {你的观察和分析}

**后续行动**:  
- [ ] {待办}
```

### 字段说明

| 字段 | 说明 | 示例 |
|------|------|------|
| **日期** | 发生日期 | 2024-01-26（周五） |
| **时间** | 大致时间 | 15:30、上午、深夜 |
| **标题** | 一句话概括 | 代码Review、紧急响应、技术分享 |
| **事件** | 具体发生了什么 | 张三Review了我的PR，提出了3个优化建议... |
| **类别** | 5大维度之一 | 技术能力、协作沟通、项目交付、团队影响、成长潜力 |
| **评价** | 你的判断 | ⭐⭐⭐⭐⭐ 优秀 / ⭐⭐⭐⭐ 良好 / ⭐⭐⭐ 一般 / ⚠️ 需关注 |
| **标签** | 关键词 | #技术突破 #owner意识 #沟通需改进 |
| **观察笔记** | 你的分析 | 对代码质量要求高，能看出潜在问题... |
| **后续行动** | 待办事项 | - [x] 已修改 / - [ ] 下次1:1聊聊 |

## 📝 使用示例

### 示例1: 正面记录

```markdown
### 2024-01-26（周五）
#### 09:00 - 主动优化
**事件**: 张三发现生产环境接口响应慢，主动排查并优化，从500ms降到50ms  
**类别**: 技术能力 + 项目交付  
**评价**: ⭐⭐⭐⭐⭐ 优秀  
**标签**: #技术突破 #owner意识 #性能优化

**观察笔记**:  
- 主动发现问题，不等客户投诉
- 优化效果显著，10倍提升
- 还写了复盘文档分享给团队
- 这种owner意识值得表扬

**后续行动**:  
- [x] 已在早会表扬
- [ ] 纳入季度评估加分项
- [ ] 邀请在技术分享会分享

---
```

### 示例2: 需关注的记录

```markdown
### 2024-01-26（周五）
#### 14:00 - 会议冲突
**事件**: 下午的需求评审会上，张三和技术经理就架构方案产生激烈争论，声音较大，会议气氛尴尬  
**类别**: 协作沟通  
**评价**: ⚠️ 需关注  
**标签**: #沟通问题 #需关注 #情绪管理

**观察笔记**:  
- 技术观点是对的，但表达方式过于直接
- 没有照顾到技术经理的面子
- 可能最近项目压力大，情绪控制不佳
- 需要温和提醒，不是大问题

**后续行动**:  
- [ ] 今天下班前私下聊聊
- [ ] 提醒注意沟通方式：坚持技术观点是对的，但要注意场合和方式
- [ ] 1-2周后观察是否有改善

---
```

## 🔄 与 OpenCode 的配合

### OpenCode 能做什么

1. **自动追加记录**
   ```
   你：记录张三今天优化了接口，性能提升10倍
   OpenCode：已追加到 张三-时间轴.md 顶部
   ```

2. **生成总结报告**
   ```
   你：帮我总结张三最近的表现
   OpenCode：[读取时间轴文件] 张三最近技术突破明显，但需注意沟通方式...
   ```

3. **准备1:1材料**
   ```
   你：准备明天和张三的1:1
   OpenCode：[提取最近2周记录] 近期亮点：... 需关注：...
   ```

4. **年终总结**
   ```
   你：生成张三的年终总结
   OpenCode：[分析全年时间轴] 生成评估表格...
   ```

### 你需要做什么

- **每天**：花2-3分钟记录1-2条（口述给 OpenCode 或直接写）
- **每周五**：更新"阶段总结"部分（手动或让 OpenCode 帮你）
- **月底/季度**：让 OpenCode 帮你生成正式报告

## 💾 备份与迁移

### 备份

**方案1: Git 版本控制**（推荐）
```bash
cd data/members
git init
git add .
git commit -m "Daily update"
```

**方案2: 网盘同步**
- Dropbox/坚果云同步整个 `data/` 文件夹
- 或者每周手动复制到网盘

### 迁移

**场景1: 换电脑**
- 复制整个 `opencode-team-memory/` 文件夹
- 或者从 GitHub/Git 仓库 clone

**场景2: 换工具**（比如不用 OpenCode 了）
- 所有数据是 Markdown 格式
- 任何文本编辑器、Notion、Obsidian 都能打开
- 终身可用，不依赖特定软件

## 🎁 额外建议

### 手机端记录

如果在手机上想到什么：
1. 写在备忘录/微信文件传输助手
2. 晚上统一整理到时间轴文件
3. 或者用手机的 Markdown 编辑器直接编辑

### 语音输入

Mac/Win 都有语音输入功能：
- 按 `Win + H` (Windows) 或 `Fn` 两次 (Mac)
- 直接口述记录
- 稍微整理后粘贴到时间轴

### 定期回顾

**每月底问自己**:
- 这个月谁进步最大？
- 谁需要我更多关注？
- 谁准备好接受更大挑战？
- 团队整体氛围如何？
