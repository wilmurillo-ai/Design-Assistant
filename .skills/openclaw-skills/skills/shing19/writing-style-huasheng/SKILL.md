# 花生写作风格

## 核心风格要点

### 1. 叙事驱动

每篇文章是一个故事，不是一份报告。遵循"起因 → 转折 → 高潮 → 结尾"的叙事弧线。

- 正确：用时间线串联事件，在关键节点制造悬念
- 错误：罗列要点、堆砌信息、写成百科词条

### 2. 数据即可信度

精确数字替代模糊形容。数据不是装饰，是论据的骨架。

- 正确："训练成本从 1000 万美元降到 557 万美元，减少了 44%"
- 错误："大幅降低了训练成本"

**一手数据优先**：当能获取一手数据（API 查询、自有工具产出、独家采访）时，优先使用一手数据做论据，优先于转述他人推文或报道。一手数据的说服力远高于二手引用。

### 3. 三层递进

解释复杂概念时使用三层结构：概念 → 公式/细节 → 直觉类比。

- 第一层：给出概念名称和一句话定义
- 第二层：展开技术细节、公式、架构
- 第三层：用日常类比让非专业读者也能理解

**术语导读三步法（新增）**：面向教程/术语文章时，优先用“**一句定义 → 直觉类比 → 在 OpenClaw 里的用途**”解释术语，保证读者看完就能用。

**解释资产保留（新增）**：如果原稿里某段术语解释已经清晰易懂（定义准确、类比到位、读者反馈好），改稿时默认保留并微调，不要为统一结构整段重写掉。

**跨领域个人经历做类比**：在论证核心观点时，可穿插作者非技术领域的个人背景和经历（如工业设计、建筑等）做跨领域类比，增加文章辨识度和说服力。

### 4. 口语化但不随便

使用第一人称，短句为主（15-25 字），可以表达个人反应（惊讶、怀疑、兴奋），但不掉价。

- 正确："我看到这个数据的时候愣了一下"
- 错误："哇塞这也太牛了吧!!!!"

**语气克制**：避免"你一定""绝对""毫无疑问"等过强断言词，用"大概""往往""多数时候"替代。松弛、平等的语气比居高临下更好。

**领域精确**：在 AI/编程话题中使用圈内精确术语（token、上下文窗口、对话框），不用泛化比喻（"口头汇报"→"在对话中回复"，"最贵的"→"最费 token 的"）。

**主语不省**：允许短句和口语节奏，但每句话的主语必须可辨。禁止"名字叫 X""功能是 Y"这种省略主语的句式——读者不该猜"谁"叫 X。

**开头用问句推进**：开头段落倾向用反问句或短问句制造节奏（"这合理吗？还有救吗？"），而非叙述式铺垫。

### 5. 首尾呼应

开头优先用**对比式**（"虚假的X vs 真正的X"）或**结论前置式**，第一眼就给核心观点。避免"你一定经历过这种…"的共情铺垫开头——信息密度太低。

结尾回扣开头。模板 A/B 结尾偏好**情绪收束**——用排比、比喻把读者变成当事人（"如果你是 X…如果你是 Y…"），而非编号清单做理性总结。模板 C（教程方法论）用**暖调实用**收尾（"舒舒服服享受X帮你干活儿"），不用哲理式感慨（"然后去做真正想做的事"）。中间是论据和叙事。

**结尾不加解释性尾巴**：结尾段落用短句直接点题即可，不要在已经展示完的内容后面追加"其实这意味着…""本质上是…""X 的精髓不过是…"等解释性总结。展示即说服，不需要再翻译一遍。

### 6. 共鸣式标题

标题的目标不是概括内容，而是**说出读者心里那句没说出口的话**。

**核心原则**：找到目标读者普遍存在但很少被说清楚的困惑/痛点/感受，用一句话替他们问出来。读者看到标题的反应应该是"对！我也是这样！"→ 然后点进来。

**方法**：
1. 先问自己：目标读者在这个领域最常遇到的困惑是什么？
2. 用他们自己会说的话来表达（口语化，不端着）
3. 困惑本身就是 hook，不需要额外的标题技巧

**好标题的特征**：
- 说出了一个很多人有但没人讲透的困惑
- 读者看到会觉得"这不就是我吗"
- 暗示文章里有答案，但标题本身不给答案

**示例**：
- "为什么 AI 写的代码总是差那么一点" — 比 "AI 编程的 5 个技巧" 好
- "所有人都觉得没必要改的东西，他们改了" — 比 "DeepSeek mHC 论文解读" 好
- "Claude Code是最好的自动化写作Agent" — 用反直觉结论制造好奇

**避免**：
- 总结型标题（"XXX 的完整指南"）
- 数字堆砌型（"5 个技巧/10 个方法"）
- 抽象概念型（"论 AI 辅助编程的最佳实践"）

## 禁止清单

以下表达方式**绝对禁止**，发现一个删一个：

- **废话开场白**："在当今数字化时代…""随着 AI 技术的飞速发展…""众所周知…"
- **总结套话**："综上所述""总而言之""总的来说"
- **AI 味句式**："不是…而是…""值得注意的是""需要指出的是""本文将介绍"
- **模糊形容**："大幅提升""显著改善""非常优秀""极其强大"
- **商业黑话**："赋能""闭环""抓手""深耕""底层逻辑""打法""颗粒度"
- **强调拐棍**："说实话""不得不说""有一说一""坦白讲"
- **学术腔**："本文旨在探讨""笔者认为""研究表明"
- **首先/其次/最后** 三段式结构 → 改为自然过渡
- **过强断言开头**："你一定经历过""你绝对遇到过" → 改为"大概都有过""往往会遇到"
- **泛化比喻代替精确术语**："口头汇报"（AI语境）→ "在对话中回复"；"最贵的"→ 用具体度量如"最费 token 的"
- **预告/回顾式过渡**："下面是我做的X件事""做完之后效果如何""区别只是…" → 直接进入内容，不做章节预告或效果回顾
- **叙述排查/操作过程**："我帮他查了配置""花了三分钟找到原因" → 直接给结论（"原因很简单"）
- **代码示例暴露隐私**：禁止在代码示例中使用作者真实 agent 名、项目名、workspace 路径名 → 用通用编号（"1"/"2"/"3"）
- **为结构牺牲清晰解释**：禁止为了“看起来更系统”而删除已清晰、已被用户认可的术语解释段落

## 格式规则

- 每个章节至少 1 处**加粗**（核心数据或金句），增强扫读体验
- 全文加粗总数不超过 8 处，避免滥用
- 引用他人原话时加粗，让引用更醒目
- **Blockquote 扩展用法**：blockquote 可用于 (1) 引用原话 (2) 核心结论 / punchline (3) 独立操作建议 / tips，不仅限于引用他人原话
- **代码示例通用化**：代码示例中的 ID、路径、变量名使用通用编号（"1"/"2"/"3"），不使用语义命名或作者个人真实命名
- **非通用知名实体首次出场加定语**：首次提及非大众知名的媒体、工具、公司、人物时，用一个短修饰语定位身份（如"硅谷顶级科技媒体 VentureBeat"、"Treehouse 创始人 Ryan Carson"）。GitHub、Claude 等圈内广泛知名的不需要。
- **英文引用双语模式**：引用英文原文时，保留原文加粗 + 紧接 blockquote 中文翻译。格式：`**"English quote"**` → 换行 → `> "中文翻译"`

## 三种子模板

### 模板 A：技术解读型

**结构**：事件背景 → 技术核心（三层递进）→ 影响分析 → 哲理收尾

**适用**：论文解读、技术突破、产品发布、行业事件解读

**要点**：
- 如果素材是新闻事件，开头用**新闻式 lead**（事件事实 → 关键数据 → 反应/引用），把最有冲击力的素材放第一屏，概念分析放第二屏展开
- 非事件类素材，开头用一个事件或数据抛出悬念
- 中段按技术逻辑展开，每个关键概念用三层递进
- 穿插行业对比和历史脉络
- 结尾从技术回到人、回到趋势

**参考**：`experiments/1-writers/huasheng/article1.md`（技术叙事解读）

### 模板 B：人物叙事型

**结构**：人物登场 → 背景铺垫 → 关键转折 → 成就展开 → 人物弧光

**适用**：创业故事、人物传记、行业人物

**要点**：
- 开头用一个戏剧性时刻拉人进来
- 按时间线展开，在关键节点放慢节奏
- 交替使用数据（融资额、用户量）和细节（对话、场景）
- 结尾升华到人物代表的更大命题

**参考**：`experiments/1-writers/huasheng/article2.md`（人物传记叙事）

### 模板 C：教程方法论型

**结构**：痛点引入 → 方法概述 → 分步详解 → 效果验证 → 行动号召

**适用**：实操教程、工具评测、方法分享

**要点**：
- 开头用"对比式"（虚假X vs 真正X）或"结论前置式"直接切入
- 方法步骤清晰编号，每步有"做什么""为什么""怎么做"
- 穿插个人实践经验和踩坑记录
- 推荐工具/方案时**必须包含"不适用场景"或"前提条件"**（如硬件要求、版本限制），不能只说好处
- 结尾给出"你现在就能做的第一步"，用暖调实用语气收尾

**参考**：`experiments/1-writers/huasheng/article3.md`（教程方法论）

## 使用指南

1. 确认输入材料属于哪种子模板（技术解读/人物叙事/教程方法论）
2. 加载对应模板结构
3. 按照五条核心风格要点生成
4. 生成后自检禁止清单
5. 确保每个数字都有出处，每个概念都有类比
