# 三层记忆架构详解

> 本文档详细说明 OpenClaw Memory System 的架构设计原理和数据流转机制。

---

## 架构概览

```
用户对话（显式标记）
    ↓
物理日志 (.jsonl)
    ↓ [每小时:00, Prompt 驱动]
L1 提炼（LLM 读取 → 分析 → 写入）
    ↓
SESSION-STATE.md (L1)
    ↓ [23:05, Prompt 驱动]
L1→L2 归档（LLM 读取 → 按日期写入）
    ↓
memory/YYYY-MM-DD.md (L2)
    ↓ [周日 23:30, Prompt 驱动]
L2→L3 萃取（LLM 读取 → CAR 提炼）
    ↓
MEMORY.md (L3)
    ↓ [每天 23:10, Prompt 驱动]
Session GC（清理临时文件）
    ↓
心跳检查（每 30 分钟，健康监控）
```

---

## 三层记忆架构

### L1：当前任务状态（热记忆）

| 属性 | 说明 |
|------|------|
| **文件** | `SESSION-STATE.md` |
| **用途** | 实时工作记忆，当前任务状态 |
| **更新频率** | 每小时（9:00-23:00） |
| **内容** | 活跃任务、关键决策、洞察发现、完成项 |
| **游标** | `[Last_Extracted_Time]` 防重复提炼 |

**设计原理**：
- L1 是"工作台"，存放当前需要关注的信息
- 每小时提炼一次，避免信息过载
- 游标机制防止重复处理同一段对话

---

### L2：每日归档（温记忆）

| 属性 | 说明 |
|------|------|
| **文件** | `memory/YYYY-MM-DD.md` |
| **用途** | 每日原始工作记录 |
| **更新频率** | 每天 23:05 |
| **内容** | 当日 L1 内容完整归档 |
| **命名** | 动态日期获取，禁止硬编码 |

**设计原理**：
- L2 是"日志本"，记录每天发生了什么
- 按日期分组，方便按时间回溯
- 保留原始内容，不做提炼（提炼是 L3 的工作）

---

### L3：长期法则库（冷记忆）

| 属性 | 说明 |
|------|------|
| **文件** | `MEMORY.md` |
| **用途** | 结构化经验法则 |
| **更新频率** | 每周日 23:30 |
| **内容** | CAR 格式法则（Context-Action-Result） |
| **格式** | 结构化，便于检索和复用 |

**设计原理**：
- L3 是"经验手册"，存放可复用的法则
- 每周萃取一次，从原始记录中提炼精华
- CAR 格式确保每条经验都有场景、行动、规则

## 数据流转机制

### 提取规则

| 层级 | 读取范围 | 处理逻辑 | 写入目标 |
|------|---------|---------|---------|
| L1 | 最后 300 行日志 | 识别显式标记，提取关键信息 | SESSION-STATE.md |
| L2 | SESSION-STATE.md 全文 | 按日期归档 | memory/YYYY-MM-DD.md |
| L3 | 本周 memory 文件 | CAR 格式提炼 | MEMORY.md |

### 游标机制

**格式**：`[Last_Extracted_Time: YYYY-MM-DD HH:MM:SS]`

**工作原理**：
1. 每次提炼后，在 SESSION-STATE.md 顶部写入游标
2. 下次提炼时，读取游标，只处理游标之后的内容
3. 避免重复提炼同一段对话

## 设计决策

### 为什么用 Prompt-Driven 而非 Script-Driven？

| 维度 | Script-Driven | Prompt-Driven |
|------|--------------|---------------|
| 跨平台 | ❌ 依赖 Python/Bash | ✅ 纯文本，任何系统可用 |
| 可审计 | ❌ 脚本不透明 | ✅ Prompt 完全可见 |
| 可调试 | ❌ 需要改代码 | ✅ 改 Prompt 即可 |
| 用户门槛 | 高（需要编程） | 低（只需 OpenClaw） |

### 为什么用物理文件而非向量数据库？

| 维度 | 向量数据库 | 物理文件 |
|------|-----------|---------|
| 可审计 | ❌ 黑盒存储 | ✅ Markdown 可读 |
| 可备份 | ❌ 需要专门工具 | ✅ 直接复制文件 |
| 可迁移 | ❌ 依赖特定服务 | ✅ 通用格式 |
| 复杂度 | 高 | 低 |

### 为什么用 CAR 格式？

- **Context**：确保经验有场景，避免脱离上下文的"万能法则"
- **Action**：记录失效和成功两种动作，形成对比
- **Result**：提炼可复用的规则，而非临时方案

## Cron 任务说明

### 核心任务（4 个）

| 任务 | 触发时间 | sessionTarget | 说明 |
|------|---------|---------------|------|
| L1 每小时提炼 | `0 9-23 * * *` | isolated | 每天 9-23 点，每小时整点 |
| L2 夜间归档 | `5 23 * * *` | isolated | 每天 23:05，归档当日 L1 |
| L3 周度萃取 | `30 23 * * 0` | isolated | 每周日 23:30，萃取 CAR 法则 |
| Session GC | `10 23 * * *` | isolated | 每天 23:10，清理临时文件 |

### 可选任务（1 个）

| 任务 | 触发时间 | 说明 |
|------|---------|------|
| 心跳检查 | `*/30 * * * *` | 每 30 分钟，检查系统健康状态 |

### 可选任务（1 个）

| 任务 | 触发时间 | 说明 |
|------|---------|------|
| 外部待办同步 | `50 22 * * *` | 每天 22:50，同步外部待办到 L1 |
