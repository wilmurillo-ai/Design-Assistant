# 久坐提醒落地指南

这份指南用于把 sedentary-reminder skill 从“会写提醒”落到“真的能稳定提醒”。

重点不是每隔多久发一次消息，而是：
- 先定期检查
- 再判断现在该不该提醒
- 能延后就延后
- 该提醒时再提醒

## 一、推荐落地思路

最稳的方式不是“每 45 分钟强行发一次”，而是：

- **调度频率**：每 10 到 15 分钟检查一次
- **真正触发提醒**：根据久坐时长 + 当前状态 + 冷却时间来决定

也就是：

- 调度器负责“看表”
- 规则负责“判断现在发不发”

这样不会因为简单 cron 把提醒做成闹钟轰炸器。

## 二、推荐判断顺序

每次检查时，按这个顺序判断：

1. 用户现在是否在电脑前
2. 当前是不是会议 / 通话 / 演示
3. 当前是不是深度专注
4. 距离上次真正休息有多久
5. 距离上次提醒有多久
6. 当前连续久坐大概有多久
7. 有没有自然断点可提醒

## 三、建议状态模型

### 1. presence
- `present`：人在电脑前，最近有操作
- `uncertain`：不确定是否还在
- `away`：明显离开，锁屏，休眠，长时间 idle

### 2. workState
- `normal`
- `focus`
- `meeting`
- `study`
- `casual`

### 3. interruptionCost
- `low`
- `medium`
- `high`

如果拿不到完整状态，就保守一点，不要硬提醒。

## 四、推荐默认阈值

### Balanced 默认档
- 首次提醒：连续久坐 50 分钟
- 二次提醒：25 分钟后仍持续久坐
- 冷却时间：至少 15 到 20 分钟
- 开会时：延后
- 人离开时：不提醒
- 深度专注时：尽量等自然断点

### Focus 友好档
- 首次提醒：70 分钟
- 二次提醒：30 分钟后
- 文案：尽量极简
- 优先：自然断点 > 硬时间点

### Study 档
- 首次提醒：45 分钟
- 二次提醒：20 到 25 分钟后
- 100 分钟后可稍微加强提醒

### Casual 档
- 首次提醒：35 到 40 分钟
- 二次提醒：20 分钟后
- 文案可以轻松一点

## 五、什么时候适合提醒

优先选这些时机：

- 一段任务刚结束
- 用户刚发完消息或邮件
- build / 导出 / 上传刚完成
- 会议刚结束
- focus timer 刚结束
- 切窗口、切任务时
- 用户回来后又连续坐了一小段时间

## 六、什么时候不适合提醒

这些时机尽量别打断：

- 正在会议或通话
- 正在演示、共享屏幕
- 明显在深度专注
- 刚提醒过不久
- 人已经离开电脑
- 连续高压沟通或紧急处理问题中

原则很简单：
**晚一点提醒，通常比在错误时机提醒更好。**

## 七、推荐执行规则

### 触发条件
只有同时满足下面条件，才建议发提醒：

- `presence != away`
- `workState != meeting`，或者会议刚结束
- 距离上次提醒 >= 15 分钟
- 当前连续久坐达到该状态的阈值
- 当前不是明显高打断成本，或者已经连续久坐过长

### 延后条件
满足这些情况就先不发：

- 正在 meeting
- focus 中且没有自然断点
- presence = uncertain
- 上次提醒刚发没多久

### 加强条件
如果连续久坐已经 90 到 120 分钟，且用户仍在电脑前：

- 可以提醒得更明确一点
- 但仍然不要凶，不要责备

## 八、建议数据字段

如果你要配合 heartbeat、脚本或自动化，建议至少记录：

```json
{
  "lastReminderAt": 0,
  "lastBreakAt": 0,
  "lastPresence": "present",
  "lastWorkState": "normal",
  "estimatedSittingStartAt": 0,
  "lastEscalationLevel": "soft"
}
```

可选补充：

```json
{
  "lastNaturalBoundaryAt": 0,
  "lastUserActivityAt": 0,
  "lastMeetingEndAt": 0,
  "lastFocusBlockEndAt": 0
}
```

## 九、Heartbeat 落地建议

如果用 heartbeat，不建议让 heartbeat 直接无脑发提醒。

更好的方式是：

1. heartbeat 只负责周期检查
2. 每次检查读取状态
3. 判断是否满足提醒条件
4. 满足时再生成一条合适文案
5. 不满足就安静结束

### 推荐 heartbeat 提示词方向
让执行逻辑优先判断：
- 是否达到连续久坐阈值
- 当前是否可打断
- 是否处于冷却期
- 应该用轻提醒、正常提醒，还是久坐加强提醒

## 十、Cron 落地建议

如果用 cron，推荐：

- **检查周期**：10 到 15 分钟
- **不要**直接把 cron 写成“每 45 分钟发一次提醒”

推荐思路：
- cron 运行一个检查任务
- 任务根据状态判断要不要发
- 发什么文案由 skill 决定

## 十一、推荐文案分层

建议把中文文案至少分成三层：

### 1. 轻提醒
适合：刚到阈值、focus、低干扰场景

### 2. 正常提醒
适合：普通工作、学习、日常久坐

### 3. 久坐加强提醒
适合：连续久坐 90 到 120 分钟以上

这样提醒强度能随着时长变化，不会一上来就太重。

## 十二、实用默认策略

如果暂时没有更多状态输入，可以直接用这个：

- 每 15 分钟检查一次
- 连续活跃坐满 50 分钟触发首条提醒
- 15 分钟内不重复提醒
- 若检测到会议，延后
- 若检测到 idle / lock / away，取消本轮
- 若连续久坐超过 100 分钟，用更明确但依然友好的文案

## 十三、最重要的一句

久坐提醒最重要的不是“固定多久发一次”，而是：

**在用户大概率能接受、也最可能真的站起来的那个时机提醒。**
