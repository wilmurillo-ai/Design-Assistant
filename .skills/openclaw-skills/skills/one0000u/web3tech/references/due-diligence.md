# Due Diligence

## Use This Path For

- Full project research
- Technical risk review
- Whitepaper validation
- Developer or contributor quality checks
- Project comparison

## Default Workflow

1. Resolve the project with `web3tech_search_coins` if the name is ambiguous.
2. Run `web3tech_coin_analysis`.
3. Run `web3tech_project_timeline` if recent execution matters.
4. Run `web3tech_project_developers` if team quality matters.
5. Run `web3tech_developer_profile` only for the most relevant contributors.
6. Run `web3tech_code_similarity` if originality or fork risk matters.
7. Run `web3tech_whitepaper_analysis` if the user asks about claims, tokenomics, roadmap, or feasibility.
8. Run `web3tech_knowledge_search` only when you need technical background to interpret findings.

## Comparison Workflow

1. Resolve every project.
2. Run `web3tech_coin_analysis` for each.
3. Run `web3tech_project_timeline` for each if execution quality is part of the decision.
4. Run `web3tech_code_similarity` only if originality or code reuse matters.
5. Compare like with like; avoid fake aggregate scoring.

## Whitepaper Validation Workflow

1. Run `web3tech_whitepaper_analysis`.
2. Cross-check with `web3tech_coin_analysis`.
3. Use `web3tech_project_timeline` if you need to see whether execution supports the whitepaper story.
4. Use `web3tech_knowledge_search` only to interpret technical claims.

## Developer Review Workflow

1. Run `web3tech_project_developers`.
2. Identify the most central or most suspicious contributors.
3. Run `web3tech_developer_profile` only for those contributors.
4. Separate team size from team quality.

## Red Flags

- Weak or thin contributor base
- Low sustained activity
- High claim intensity with weak execution evidence
- Suspiciously generic whitepaper language
- Obvious code reuse without clear differentiation
