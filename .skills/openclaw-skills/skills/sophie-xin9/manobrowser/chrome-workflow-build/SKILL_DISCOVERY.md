# Skill智能发现机制

## 执行触发

本模块在以下情况触发：

1. **模式1（LOG.md）**: 执行完成、用户确认结果后立即执行（步骤3.2）
   - **触发时机**: 阶段1执行并记录完成 → 用户确认执行结果正确 → 立即触发Skill发现
   - **输入数据**: 当前上下文中的执行信息（user_query、执行链路、最终步骤）
   - **原因**: 执行刚完成，模型已完全理解用户意图，无需保存日志再读取

2. **模式2（DATASAVER.md）**: 读取DATASAVER录制工作流.json并提取description后立即执行（步骤2）
   - **触发时机**: 读取DATASAVER录制工作流.json文件 → 提取顶级description后立即触发
   - **输入数据**: `DATASAVER录制工作流.json`
   - **原因**: DATASAVER录制工作流.json已包含完整的业务需求描述

## 核心目标

避免重复创建功能相同的Skills，提升资源复用效率。

**核心价值**:
- **避免重复创建**: 减少生成重复功能的skills
- **提升效率**: 直接复用已有skill，无需重新执行和生成
- **用户友好**: 智能推荐，让用户知道已有资源

## 发现流程

### 1. 提取用户意图

根据调用来源不同，采用不同的提取策略。

**模式1（LOG.md）**：
- 从当前上下文中的user_query、整个执行链路和最终步骤综合分析业务需求
- 识别平台（小红书、抖音、B站等）
- 识别操作类型（搜索、发布、收集等）
- 识别目标对象（帖子、视频、用户等）

**模式2（DATASAVER.md）**：
- 从DATASAVER录制工作流.json整个文件分析业务需求（重点是description字段）
- 识别平台（小红书、抖音、B站等）
- 识别操作类型（搜索、发布、收集等）
- 识别目标对象（帖子、视频、用户等）

### 2. 扫描已有Skills

扫描当前环境可以使用的全部Skills，获取每个Skill的name和description信息。

**目标**：
- 获取所有已生成Skills的name（Skill名称）
- 获取所有已生成Skills的description（功能描述，核心匹配依据）
- 如果没有已有Skills，跳过发现流程，直接继续原流程

### 3. 智能匹配分析

对比用户意图和已有Skills的description，进行高度匹配分析。

**匹配要求**（评分0-100分）：
- **平台必须一致**：小红书、抖音、B站等平台必须完全匹配
- **业务需求必须一致**：操作类型（搜索、发布、收集等）和目标对象（帖子、视频、用户等）必须高度一致
- **推荐阈值**：>=90分才推荐给用户，<90分静默跳过

### 4. 用户交互决策

根据匹配度评分，决定是否推荐给用户。

**决策规则**：
- **匹配度 >= 90分**：向用户展示匹配的Skill信息，强制等待用户确认
- **匹配度 < 90分**：静默跳过，直接继续原流程

**用户确认"满足需求"**：
- 停止创建流程（不再执行后续阶段）
- 以文字形式指导用户如何使用已有Skill
- 结束chrome-workflow-build流程

**用户选择"不满足，继续创建"**：
- 继续执行原流程
- 最终生成新的Skill
