# API 逆向探索检查清单

## 阶段0: 鉴权探测

- [ ] 提取 document.cookie（chrome_execute_script, world: MAIN）
- [ ] 通过 Performance API 找到至少一个**数据** API endpoint
- [ ] ⚠️ 用 **curl**（Bash 工具）测试鉴权（禁止用 MCP 工具测试）
- [ ] 测试数据接口（不只是配置接口——同一平台不同接口鉴权可能不同）
- [ ] 判定架构路线：Type A（脚本自治）/ Type B（MCP 代调）

## 阶段1: 侦察

- [ ] 页面截图，确认数据展示方式
- [ ] 确认是 DOM 渲染还是 Canvas 渲染
- [ ] 识别页面上的筛选条件（类目/日期/维度等）
- [ ] 识别页面区块划分（一组筛选条件对应几个数据区块/API）
- [ ] ⭐ `chrome_network_debugger_start` 抓包（首选，能拿完整请求体+响应体）
- [ ] 若 debugger 报 "already attached" → 新开标签页，关闭旧标签后重试
- [ ] 启动 debugger 后通过切换筛选/点击触发数据请求（不要刷新页面）
- [ ] Performance API 获取 API 请求列表（补充，只有 URL）
- [ ] 筛选出可能的数据 API（排除埋点/静态资源）
- [ ] 若以上不够 → XHR 拦截器注入 + 切换筛选触发（备选）
- [ ] JS bundle / Webpack Chunk 搜索（最后手段，微前端架构下效率极低）
- [ ] 确认目标 API endpoint（URL + 方法）

## 阶段2: 逆向

- [ ] 调用配置/常量接口获取字段定义
- [ ] JS 源码搜索 API 调用处的参数构造
- [ ] 确认每个参数：名称、类型、必填/可选、来源
- [ ] ⚠️ 测试参数名大小写变体（同一 API 可能混用大小写）
- [ ] 发送最小参数集试探请求
- [ ] 逐步加参数验证返回变化
- [ ] 确认是否需要多次请求覆盖全部数据
- [ ] 识别动态参数（需从页面组件状态提取的参数）
- [ ] 区分用户参数 vs 固定参数

## 阶段3: 验证

- [ ] API 返回数据与页面截图逐项对比
- [ ] ⚠️ 发现并记录数据转换规则（缩放、过滤、格式化）
- [ ] 验证分页行为（实际返回条数 vs pagesize 参数）
- [ ] 不同参数组合测试正常
- [ ] 记录完整响应 JSON 结构及字段含义

## 阶段4: 生成

- [ ] 按架构路线选择模板（Type A / Type B）
- [ ] 生成 api_mapping.json
- [ ] 生成 Python 脚本（Type A: 全链路 / Type B: 仅格式化）
- [ ] 生成 SKILL.md（含参数确认路径分支、Cookie 刷新流程）
- [ ] 包含 ≥ 3 个测试问答对
- [ ] 记录所有发现的"坑"（大小写、分页、转换规则等）
- [ ] 展示摘要等待用户确认
- [ ] ⚠️ 稳定性自检（SKILL.md 行数、脚本独立性、数据处理归属）
