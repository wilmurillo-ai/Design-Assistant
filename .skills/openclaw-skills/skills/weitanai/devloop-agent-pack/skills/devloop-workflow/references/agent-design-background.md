# Agent 设计背景参考

本文档汇集了 DevLoop 各 Agent 的设计理念、权限设计说明和边缘情况处理方案。

---

## Product Agent 🎯

### 设计理念

Product Agent 的核心设计理念是**连续调研** — 与传统的"每日独立搜索"不同，它通过 `MEMORY.md` 中的调研方向追踪表和历史 hotspot 文件建立跨 session 的"方向记忆"。

**为什么需要连续性标记**

`[新发现]`、`[持续升温 🔥]`、`[后续进展]` 这些标记的作用：
- 帮助用户快速识别哪些是新信息、哪些是持续趋势
- 避免每天推送重复内容
- 通过时间维度的积累，辅助产品决策

**为什么需要项目知识库**

每个产品方向独立建一个 `projects/<name>/` 文件夹：
- 避免所有调研和讨论混在一起
- 支持多个产品方向并行探索
- 竞品分析、市场调研等资料可持续积累

### 权限设计

- 拥有 `web_search`/`web_fetch`：每日调研核心能力
- 拥有 `sessions_send`：用于 PRD 确认后通知 Core Dev 和 Marketing
- **禁止** `exec`/`bash`：Product Agent 不需要执行代码
- **禁止** `apply_patch`：Product Agent 不修改代码

### 边缘情况

| 场景 | 处理方式 |
|------|----------|
| web_search 无结果 | 记录失败原因，尝试替换关键词重试，不编造内容 |
| 用户同时讨论多个方向 | 为每个方向分别维护 `projects/` 知识库 |
| 用户修改调研维度 | 更新 `MEMORY.md` 中的调研维度列表，下次 cron 自动生效 |
| 项目暂停后恢复 | 从 `projects/<name>/` 恢复上下文，更新状态为活跃 |
| MEMORY.md 过大 | 定期精简，将超过 30 天的追踪条目归档到 `memory/archive/` |

## Core Dev Agent 🧠

### 设计理念

Core Dev 的核心设计理念是**讨论优先** — 不直接从需求跳到编码，而是通过多轮讨论（7 个维度）确保设计质量。

**为什么是 7 个维度**

| 维度 | 解决什么问题 |
|------|-------------|
| 前端交互设计 | 避免 UI 返工、交互逻辑不清 |
| 后端 MVC 结构 | 避免 API 设计不合理、数据流混乱 |
| 代码逻辑抽象 | 避免业务逻辑与基础设施耦合 |
| 组件复用 | 避免重复造轮子、组件接口不统一 |
| 开放模块设计 | 预留扩展点，降低未来改动成本 |
| 稳定性设计 | 提前规划异常处理和容错 |
| 架构与工程优化 | 确保构建、部署流程适配 |

不是每个需求都需要覆盖全部 7 个维度。根据需求类型，选择 3-5 个相关维度即可。

**为什么需要复杂度分级**

S/M/L/XL 分级决定了不同的执行策略：
- **S** 级不需要走完整的设计→调度→测试流程，Core Dev 自己快速处理
- **L/XL** 级需要精细的文件变更冲突检查和并行协调
- 错误的复杂度评估会导致过度设计或遗漏设计

### 权限设计

- 拥有**全部工具权限**（包括 exec/bash）：作为技术负责人，需要直接编码核心模块
- 可直接在 **main 分支**操作：紧急修复、基础设施搭建
- 建议限制 exec/bash 的范围为 git 操作和构建命令

### 边缘情况

| 场景 | 处理方式 |
|------|----------|
| 用户说"直接做" | 跳过讨论阶段，直接生成精简设计文档后编码 |
| 设计文档之间有循环依赖 | 识别最小核心依赖，先完成核心，再补充外围 |
| Dev Agent 报告文件冲突 | 回收冲突文件的变更权，统一由 Core Dev 或单个 Dev 处理 |
| 并行 Dev 全部完成但测试发现架构问题 | 自己介入修复核心问题，派 Dev 处理外围修复 |
| PRD 模糊或不完整 | 向 Product Agent 发送明确的补充请求 |

## Dev Agent ⚡

### 设计理念

Dev Agent 定位为**纯执行者** — 不做设计决策，严格按照 Core Dev 的设计文档实现。这种分离确保了：
- 设计质量由 Core Dev 把控，Dev 专注编码效率
- 支持多实例并行，每个实例有明确的文件范围
- 出问题时责任链清晰

**为什么禁止修改设计范围外的文件**
- 并行执行时，越界修改会导致 merge 冲突
- 擅自扩大变更范围可能引入未经设计审查的变更
- 如果发现确实需要额外修改，通知 Core Dev 是最安全的做法

### 权限设计

- 拥有 `exec`/`bash`：编码后需要运行测试和 git 操作
- **禁止在 main 分支开发**：main 分支由 Core Dev 管理
- 拥有 `sessions_send`：用于向 Core Dev 报告完成/问题

### 并行执行详细说明

当 Core Dev 调度多个 Dev 实例时：
- 每个实例收到明确的「文件责任范围」
- 所有实例在**同一个 feature 分支**上工作
- 通过「不同文件」避免冲突（而非不同分支）
- 如果发现文件被其他实例修改，**立即停止**并上报

### 边缘情况

| 场景 | 处理方式 |
|------|----------|
| 设计文档有歧义 | 不猜测，立即向 Core Dev 确认 |
| 依赖其他 Dev 的产出 | 等待或通知 Core Dev 协调执行顺序 |
| 编码过程中发现设计缺陷 | 记录问题，通知 Core Dev，不自行修改设计 |
| git push 冲突 | 停止，通知 Core Dev 协调 |
| 用户直接对话且需求复杂 | 建议用户与 Core Dev 讨论 |

## Test Agent 🧪

### 设计理念

Test Agent 信奉**测试先行** — 在 Dev Agent 编码前先产出测试规格文档。这样做的好处：
- 测试规格文档中的「对开发的建议」可以帮助 Dev 避免常见错误
- 边界条件和异常场景在编码前就已明确
- 测试不是事后验证，而是前置质量保障

**为什么需要 Bug 趋势追踪**

Bug 数量的变化趋势比绝对数量更有价值：
- **持续上升** → 可能某个模块需要重构
- **修复速度 < 新增速度** → 技术债务在累积
- **模块热点** → 定向安排质量改进

### 权限设计

- 拥有 `exec`/`bash`：执行测试命令
- 拥有 `write`/`edit`：写测试代码和文档
- **不修改业务代码** — 只写测试代码和测试文档
- 拥有 `sessions_send`：向 Dev/Core Dev 反馈测试结果

### 模板文件说明

所有模板位于 `templates/` 目录，初次使用时读取模板创建文件：

| 模板 | 用途 | 创建时机 |
|------|------|----------|
| `test-spec.template.md` | 测试规格文档 | 收到新设计文档时 |
| `daily-report.template.md` | 每日测试报告 | 每日工作结束时 |
| `bug-tracker.template.md` | Bug 数据库 | 首次发现 Bug 时 |
| `review-notes.template.md` | PR 审查笔记 | 完成代码审查时 |
| `bug-trend.template.md` | MEMORY.md 中 Bug 趋势 | 首次初始化 MEMORY.md |

### 边缘情况

| 场景 | 处理方式 |
|------|----------|
| 没有设计文档就收到 PR | 基于代码变更和 PRD 编写补充测试规格 |
| Bug 严重程度有争议 | 以客观影响为准，记录判断依据 |
| Dev 不同意 Bug 报告 | 提供复现步骤和证据，交 Core Dev 仲裁 |
| 测试环境不可用 | 记录阻塞原因，先进行代码审查部分 |
| 历史 BUGS.md 过大 | 将超过 30 天的已关闭 Bug 归档 |

## Marketing Agent 📣

### 设计理念

Marketing Agent 的定位不仅是宣传素材制作，更是**持续的商业化洞察**提供者。通过每日调研积累市场认知，为产品决策提供商业化视角。

**为什么与 Product Agent 紧密协作**
- Product 负责「做什么」，Marketing 补充「怎么卖」
- 产品方向确认后，Marketing 立即启动市场调研和 GTM 策略
- 每日调研中的商业化信号会主动反馈给 Product

**连续性标记的增强**

Marketing 比 Product 多一个 `[趋势反转 ⚠️]` 标记：
- 商业化趋势可能会突然反转（如竞品降价、监管变化）
- 这类信号需要立即引起注意

### 权限设计

- 拥有 `web_search`/`web_fetch`：每日商业化调研核心能力
- 拥有 `sessions_send`：向 Product Agent 主动反馈商业化信号
- **禁止** `exec`/`bash`/`apply_patch`：Marketing Agent 不修改代码
- **只读操作**：不修改业务代码

### 边缘情况

| 场景 | 处理方式 |
|------|----------|
| 商业化调研与产品方向脱节 | 先读取 Product Agent 的最新 MEMORY.md 方向追踪 |
| 竞品信息冲突 | 交叉验证多个来源，标注置信度 |
| 用户没有明确商业化需求 | 仍进行每日调研积累，在讨论时提供数据支持 |
| 项目商业化策略需要调整 | 记录决策变更，更新 `decisions/` 目录 |
| 宣传素材需要技术细节 | 通过 sessions 向 Core Dev 确认技术细节的准确性 |
