
### 2. 数据库表结构详解

#### 授权码表 (auth_codes)

授权码表用于存储所有用户的授权信息，包括权限类型、过期时间、使用次数等。

**核心字段：**
- id: 自增主键
- code: 授权码（AES加密后存储）
- permission: 权限类型（all/generate/execute）
- expire_time: 过期时间
- use_count/max_count: 使用次数控制
- is_active: 启用状态

#### 其他核心表

- test_cases: 测试用例表
- auto_scripts: 自动化脚本表  
- execute_records: 执行记录表
- test_reports: 测试报告表
- task_progress: 任务进度表

## 授权安全加固

### 安全风险说明

**重要提示**：当前授权机制是应用层中间件拦截，存在被绕过的风险！

如果攻击者能够修改应用源代码（例如删除 `app.add_middleware(AuthInterceptorMiddleware)`），就可以完全绕过授权验证继续使用平台。

### 风险等级

| 攻击场景 | 风险等级 | 可行性 |
|---------|---------|--------|
| 源代码被修改 | 🔴 高 | 如果攻击者获得服务器访问权限 |
| 反编译修改字节码 | 🟡 中 | Python可被反编译修改 |
| 数据库直接访问 | 🟡 中 | 需要数据库访问权限 |

### 推荐安全加固方案

#### 方案1：数据库层强制授权（强烈推荐 ⭐⭐⭐⭐⭐）

**核心思想**：将授权验证下沉到数据库层，即使应用代码被修改也无法绕过。

**实施步骤**：

1. **创建授权验证存储过程**

```sql
DELIMITER $$

CREATE PROCEDURE verify_and_use_auth_code(
    IN p_auth_code VARCHAR(100),
    IN p_required_permission VARCHAR(20),
    OUT p_is_valid BOOLEAN,
    OUT p_message VARCHAR(255),
    OUT p_remaining_count INT
)
BEGIN
    DECLARE v_permission VARCHAR(20);
    DECLARE v_expire_time DATETIME;
    DECLARE v_use_count INT;
    DECLARE v_max_count INT;
    DECLARE v_is_active TINYINT;
    
    -- 查询授权码信息
    SELECT 
        permission, 
        expire_time, 
        use_count, 
        max_count, 
        is_active
    INTO 
        v_permission,
        v_expire_time,
        v_use_count,
        v_max_count,
        v_is_active
    FROM auth_codes
    WHERE code = p_auth_code;
    
    -- 验证授权码
    IF v_is_active IS NULL THEN
        SET p_is_valid = FALSE;
        SET p_message = '授权码不存在';
        SET p_remaining_count = 0;
    ELSEIF v_is_active = 0 THEN
        SET p_is_valid = FALSE;
        SET p_message = '授权码已被禁用';
        SET p_remaining_count = 0;
    ELSEIF v_expire_time < NOW() THEN
        SET p_is_valid = FALSE;
        SET p_message = '授权码已过期';
        SET p_remaining_count = 0;
    ELSEIF v_use_count >= v_max_count THEN
        SET p_is_valid = FALSE;
        SET p_message = '授权码使用次数已达上限';
        SET p_remaining_count = 0;
    ELSEIF p_required_permission = 'generate' AND v_permission = 'execute' THEN
        SET p_is_valid = FALSE;
        SET p_message = '权限不足：需要生成权限';
        SET p_remaining_count = v_max_count - v_use_count;
    ELSEIF p_required_permission = 'execute' AND v_permission = 'generate' THEN
        SET p_is_valid = FALSE;
        SET p_message = '权限不足：需要执行权限';
        SET p_remaining_count = v_max_count - v_use_count;
    ELSE
        -- 验证通过，增加使用次数
        UPDATE auth_codes 
        SET use_count = use_count + 1 
        WHERE code = p_auth_code;
        
        SET p_is_valid = TRUE;
        SET p_message = '授权验证成功';
        SET p_remaining_count = v_max_count - v_use_count - 1;
    END IF;
END$$

DELIMITER ;
```

2. **在应用代码中调用存储过程**

```python
# app/services/auth.py
from sqlalchemy import text
from fastapi import HTTPException

class AuthService:
    def verify_auth_in_database(
        self, 
        db: Session, 
        auth_code: str, 
        required_permission: str
    ):
        """数据库层授权验证"""
        
        # 调用存储过程
        db.execute(
            text("CALL verify_and_use_auth_code(:code, :perm, @valid, @msg, @remaining)"),
            {"code": auth_code, "perm": required_permission}
        )
        
        # 获取输出参数
        result = db.execute(
            text("SELECT @valid as valid, @msg as message, @remaining as remaining")
        ).first()
        
        if not result.valid:
            raise HTTPException(
                status_code=403,
                detail={
                    "message": result.message,
                    "remaining_count": result.remaining
                }
            )
        
        return result
```

3. **在API端点中使用**

```python
# app/api/execute.py
@router.post("/run")
async def run_test(
    request: Request,
    db: Session = Depends(get_db),
    auth_service: AuthService = Depends()
):
    auth_code = request.headers.get("Authorization")
    
    # 数据库层强制验证
    auth_service.verify_auth_in_database(
        db, 
        auth_code, 
        required_permission="execute"
    )
    
    # 继续执行测试逻辑
    ...
```

**优势**：
- ✅ 即使应用代码被修改，仍需调用存储过程
- ✅ 数据库层强制授权，无法绕过
- ✅ 双重保护：应用中间件 + 数据库存储过程

#### 方案3：容器只读文件系统

**Docker部署时禁止修改代码**：

```yaml
# docker-compose.yml
services:
  app:
    build: .
    security_context:
      readOnlyRootFilesystem: true  # 只读文件系统

      - ./data:/app/data            # 只有数据目录可写
```

```dockerfile

```

**优势**：
- ✅ 运行时无法修改代码
- ✅ 即使获得容器访问权限也无法删除授权逻辑


### 常见安全问题

**Q1: 存储过程验证会影响性能吗？**

A: 影响很小。存储过程在数据库层执行，只需一次数据库调用，性能影响在毫秒级别。

**Q2: 如果忘记了数据库密码，会不会无法访问？**

A: 需要保留一个管理员授权码，并确保数据库管理员（DBA）可以重置授权。

**Q3: 已部署的系统如何升级？**

A: 
1. 先在数据库创建存储过程
2. 逐步更新API端点调用存储过程
3. 最后配置只读文件系统
4. 无需停机，可滚动升级

### 总结

**核心原则**：不要依赖单一防护层，建立纵深防御体系。

**最小安全配置**：
1. 数据库存储过程验证（必须）
2. 容器只读文件系统（必须）
3. 审计日志（必须）

完成这三项配置后，即使攻击者获得服务器访问权限并修改应用代码，也无法绕过数据库层的强制授权验证。
