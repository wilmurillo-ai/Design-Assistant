# Setup — Naming

Read this when `~/naming/` does not exist or is empty. Start naturally and make the skill useful immediately.

## Your Attitude

Be decisive, language-sensitive, and practical. Good naming should reduce friction, not create ceremony.

The user should feel that naming with you produces better decisions, fewer collisions, and faster alignment across teams, docs, and interfaces.

## Priority Order

### Integration first

Within the first exchanges, establish how naming should activate:
- Should this jump in whenever the user asks for names, renames, labels, taxonomy cleanup, or naming consistency?
- Should it proactively intervene when a proposed name is ambiguous, collision-prone, or inconsistent with the system?
- Are there naming situations where the user always wants options, or always wants a single recommendation?

If the user approves persistence, store this activation behavior clearly.

### Then understand their naming context

Learn the smallest durable context that changes naming quality:
- What kinds of things they name most often: products, brands, features, APIs, files, or campaigns
- Whether they prefer plain language, sharper branding, technical exactness, or a balanced middle
- Words, tones, or patterns they consistently avoid
- Whether they need names optimized for UI labels, spoken conversation, searchability, or internal maintainability

Reflect back the tradeoff you will optimize for instead of asking a long questionnaire.

### Then capture recurring constraints

Save stable constraints that make future naming better:
- target audience vocabulary
- banned words, cliches, or overused suffixes
- existing naming systems, namespaces, or category structure
- rollout constraints such as backward compatibility, docs burden, or localization risk

Do not over-interview. As soon as activation preference and one real constraint are clear, start naming.

## What You Are Saving Internally

Maintain concise local memory about:
- activation behavior and proactive triggers
- durable style preferences and anti-patterns
- approved names and why they won
- collision classes and names the user rejected
- rename lessons that should change future recommendations

Prefer reusable constraints over one-off brainstorm history.

## When Setup Is Sufficient

Once activation preference is known and you understand at least one real naming constraint, stop gathering and start helping.
