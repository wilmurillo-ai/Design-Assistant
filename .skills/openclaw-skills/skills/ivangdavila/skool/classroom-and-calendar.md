# Classroom and Calendar - Skool

## Build the Classroom Around the Next Member Move

- Organize courses by transformation, not by dumping all content at once.
- Each course should answer one clear promise and one clear completion state.
- Every lesson should point to the next lesson, task, template, or event.

## Access Logic

- Check whether the course is open, paid, level-gated, time-unlocked, or invite-driven before editing sequence or automation.
- Test unlock order with the real member path in mind, not just the admin view.
- If the business model changed, re-check old members, new members, and trial members separately.

## Event Design

- Calendar exists to pull members back into action, not to fill space.
- Name events by outcome, audience, and time expectation.
- Pair major classroom pushes with calendar moments so members know when to show up live.

## Operational Defaults

- Prefer one repeatable cadence over many ad hoc events.
- Capture which events are mandatory, optional, or replay-safe.
- If a course depends on attendance, make that dependency explicit in the welcome flow.

## Failure Patterns

- deep classroom, weak calendar -> members stop returning
- active calendar, weak classroom -> members show up but do not progress
- unclear unlock rules -> support load rises and trust drops
- copied courses without re-checking permissions -> the wrong people get the wrong access
