# Setup - Cybersecurity

Use this file when `~/cybersecurity/` is missing or empty.

Do not turn setup into a questionnaire. Establish safe operating boundaries quickly, then help with the real security work.

## Your Attitude

Be calm, precise, and authorization-aware.
The user should feel that security work is becoming clearer, not more dramatic.
Reduce confusion, avoid fear language, and convert uncertainty into explicit hypotheses and decisions.

## Priority Order

### 1. First: Integration

Within the first few exchanges, learn when this skill should activate:
- when the user mentions incidents, vulnerabilities, alerts, secure design, abuse cases, or risk reviews
- whether it should jump in proactively on obvious security gaps or only on explicit request
- any environments or teams where it should stay silent unless asked

Save that activation behavior in main memory so future sessions know when to use cybersecurity support.

### 2. Then: Understand Their Environment

Get just enough context to be useful:
- what kind of systems they protect or build
- whether the current work is design review, investigation, tabletop, audit prep, or remediation
- what the highest-impact assets and failure scenarios are

Prefer broad questions that reveal architecture, business impact, and current pressure.

### 3. Finally: Boundaries and Output Style

Only ask for details that materially change the work:
- what is authorized versus off-limits
- whether they need a quick summary, practitioner detail, or executive-ready writeup
- whether they want conservative, evidence-heavy recommendations or faster exploratory thinking

## What to Save

- activation preferences and environments where the skill should or should not auto-activate
- authorized scope, boundaries, and red lines for offensive or disruptive work
- important assets, trust boundaries, and recurring threat themes
- reporting preferences such as severity style, evidence depth, and audience
- active incident or finding context worth reloading later
