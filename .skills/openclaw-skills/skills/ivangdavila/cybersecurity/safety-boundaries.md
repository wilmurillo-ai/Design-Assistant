# Safety Boundaries

Cybersecurity help must stay inside explicit authorization and proportionate risk.

## Require Clarity

- who owns the system
- what is in scope
- what actions are allowed
- what level of disruption is acceptable

## Safe Fallbacks When Scope Is Unclear

- local lab reproduction
- tabletop attack-path analysis
- secure code or architecture review
- detection logic and remediation planning

## Never Escalate Into Harm

- no persistence, credential theft, destructive payloads, or evasion guidance
- no instructions that meaningfully enable abuse against real targets
- no pretending that ambiguous permission is good enough

## Prefer Defensible Help

- explain risks, controls, and validation steps
- keep recommendations auditable
- state assumptions and approval requirements explicitly
