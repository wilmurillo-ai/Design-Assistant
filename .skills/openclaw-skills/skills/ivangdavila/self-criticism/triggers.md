# Trigger Families

Use these families to decide when breakpoint critique should fire.

## Before branching

Pause when a rough direction is about to become a task tree, implementation plan, or delegated work packet.
Typical signs:
- turning a sketch into subtasks
- locking the architecture for several moving parts
- splitting work across files, agents, or teams

Core question:
- What flaw in the current frame would multiply across all later tasks?

## Before commitment

Pause before moves that are costly to unwind:
- irreversible edits
- external actions
- public recommendations
- decisions that constrain later options

Core question:
- What assumption is still unproven but is about to become expensive?

## After surprising evidence

Pause when tool output, user context, or reality contradicts the current model.

Core question:
- Which part of the working frame no longer matches the evidence?

## After user friction

Pause after correction, rejection, repeated revisions, or signs that the user expected a different shape of work.

Core question:
- Was the content wrong, or did critique happen too late in the process?

## At risky handoffs

Pause before handing work to execution, another agent, or the final answer when the next step depends on a clean frame.

Core question:
- What ambiguity would create downstream rework for the receiver?
