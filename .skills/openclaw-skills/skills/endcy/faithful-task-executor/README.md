## 📖 简介

**Faithful Task Executor** 是一个专注于"忠实执行初始规则"的任务处理 skill。它确保：

- ✅ 用户提出的工作要求被完整准确地执行
- ✅ 子代理/子任务始终遵循原始规则
- ✅ 不添加规则中不存在的要求
- ✅ 不忽略规则中的禁止事项
- ✅ 减少 AI 幻觉和执行偏差

---

## ✨ 核心特性

### 📜 规则持久化

- 初始要求写入规则文件，贯穿任务全生命周期
- 每个执行阶段强制读取并校验规则
- 禁止子代理添加不存在的要求或规则

### 🔄 并发任务编排

- 多子代理并行执行时，规则文件自动传递给每个子代理
- 子任务启动前必须确认已读取并理解规则
- 支持任务依赖关系和并行度控制

### ✅ 执行一致性校验

- 每个阶段输出前进行规则符合性检查
- 发现偏差立即纠正并记录
- 最终交付物必须通过规则校验

### 🛡️ 防幻觉机制

- 规则明确禁止的行为列表
- 模糊要求澄清机制
- 输出内容溯源验证

## 📋 完整工作流程

### Step 0: 规则提取与固化

从用户输入中提取明确规则，写入规则文件：

```markdown
# 任务规则说明书

## 任务概述
- **任务名称**: {任务名称}
- **任务类型**: {编码/文档/分析/设计/其他}
- **优先级**: {高/中/低}

## 明确要求（必须遵守）
1. {要求 1}
2. {要求 2}

## 禁止事项
- ❌ {禁止行为 1}
- ❌ {禁止行为 2}

## 规则版本
- v1.0: 初始版本
- 任何规则变更必须经用户确认
```

### Step 1: 任务分解与规划

将复杂任务分解为可执行的子任务：

```markdown
# 任务执行计划

## 阶段划分
- 阶段 1: {阶段名称} - {目标}
- 阶段 2: {阶段名称} - {目标}

## 子任务
- 子任务 1.1: {描述} - {依赖} - {预计耗时}
- 子任务 1.2: {描述} - {依赖} - {预计耗时}

## 并发策略
- 可并行：[子任务列表]
- 必须串行：[子任务列表]
- 最大并行度：{数字}
```

### Step 2: 子代理执行（带规则校验）

每个子代理启动前必须确认：

```
子代理 {label} 规则确认:
- [x] 已读取规则文件：.task-rules/rules.md
- [x] 理解所有明确要求
- [x] 理解所有禁止事项
- [x] 承诺不添加额外规则
- [x] 发现模糊将先询问
```

### Step 3: 规则符合性校验

每个阶段输出前进行校验：

```markdown
# 规则符合性校验清单

## 基本要求校验
- [ ] 所有明确要求已满足
- [ ] 所有禁止事项已避免
- [ ] 输出格式符合要求

## 防幻觉校验
- [ ] 所有陈述可溯源到输入或规则
- [ ] 未添加不存在的要求
- [ ] 未假设未提供的信息

## 校验结果
- **校验通过**: ✅ / ❌
- **发现问题**: {列表}
- **纠正措施**: {措施}
```

### Step 4: 任务汇总与交付

```markdown
# 任务汇总报告

## 子任务完成情况
| 子任务 | 状态 | 耗时 | 规则符合性 |
|--------|------|------|------------|
| 1.1 | ✅ | 8 分钟 | ✅ |
| 1.2 | ✅ | 12 分钟 | ✅ |

## 规则符合性总结
- ✅ 所有要求已满足
- ✅ 所有禁止事项已避免
- ✅ 无 AI 幻觉内容

## 交付物清单
1. {交付物 1}
2. {交付物 2}
```

## 🛡️ 防幻觉机制

### 1. 规则明确化

❌ 模糊："代码要高效"
✅ 澄清："高效"具体指什么？时间复杂度？空间复杂度？

### 2. 禁止事项清单

```
任务执行期间禁止:
- ❌ 添加规则中未提及的功能
- ❌ 假设未提供的数据
- ❌ 忽略明确的禁止事项
- ❌ 使用规则禁止的技术栈
```

### 3. 输出溯源

```
✅ "根据规则第 3 条要求，..."
✅ "基于用户提供的数据，..."
⚠️ "假设 X（未验证），建议确认"
❌ "我认为应该..."（无依据）
```

### 4. 模糊澄清

```
模糊要求 → 提出澄清问题 → 等待确认 → 更新规则 → 继续执行
```
