## 产品速查表

## 选型决策流程

### 前置判断：是否需要合单支付？


| 场景特征                                               | 结论                                             |
| -------------------------------------------------- | ---------------------------------------------- |
| 一笔支付中需要合并**多个子商户**（服务商模式下不同 sub_mchid）的订单，让用户一次性支付 | → **需要合单支付**（跳转下方「合单支付说明」确定具体支付方式后，使用合单下单 API） |
| 仅单一商户、无需合并多子商户订单的普通支付场景                            | → **不需要合单**，继续下方第一步选择支付方式                      |


> 合单支付不是独立的支付方式，而是在基础支付（JSAPI/APP/H5/Native/小程序）之上的一种**合并下单模式**。确定需要合单后，仍需按下方流程选择具体的支付方式（付款码支付不支持合单）。

### 第一步：判断是线上还是线下场景


| 场景特征                                    | 结论                         |
| --------------------------------------- | -------------------------- |
| **线下收银台**（超市/便利店/餐饮等），由商户用扫码枪扫用户手机上的付款码 | → **付款码支付**（跳转下方「付款码支付说明」） |
| 线上场景，或用户主动扫商户二维码                        | → 继续第二步                    |


### 第二步：根据用户发起支付的终端环境选择


| 用户所在终端环境                                            | 推荐支付方式       | 前置要求                                     | 关键注意                                            |
| --------------------------------------------------- | ------------ | ---------------------------------------- | ----------------------------------------------- |
| 微信客户端内（公众号 / 扫码打开的H5页面）                             | **JSAPI支付**  | 已认证的服务号/政府或媒体类公众号；需要用户openid、配置JSAPI授权目录 | —                                               |
| 微信小程序                                               | **小程序支付**    | 已认证的小程序；需要用户openid                       | 和JSAPI共用同一个下单API                                |
| 商户原生APP（iOS/Android）                                | **APP支付**    | 已认证的开放平台账号 + 移动应用APPID                   | 不要在APP内使用H5支付                                   |
| 手机浏览器（非微信客户端）                                       | **H5支付**     | 企业/事业单位/政府机关/社会组织（不支持个体工商户）；需配置H5支付域名    | 不支持在APP内使用；必须做跨域安全校验                            |
| PC浏览器访问**商户**网页/收银台（商户提供PC端收款页；**付款**靠用户手机微信扫页面上的码） | **Native支付** | 无特殊                                      | 展示 `code_url` 二维码供扫；微信「扫一扫」完成支付；不支持相册识别或长按识别二维码 |


### 付款码支付说明（与其他5种支付方式有本质区别）

**核心差异**：付款码是唯一的"被扫"模式 —— 由商户侧扫码设备扫描用户付款码，一步完成扣款。其他5种支付方式都是用户主动发起支付。

**交互流程**：

1. 用户打开微信 →"我"→"支付"→"收付款"，屏幕展示付款码（条码 + 二维码）
2. 收银员在商户系统生成支付订单，用户确认金额
3. 收银员用扫码设备扫描用户的付款码，商户系统调用付款码支付API直接提交扣款
4. 微信后台根据风控规则判断是否需要验证支付密码：
  - 免密交易：直接扣款成功
  - 需验密交易：用户微信端弹出密码输入框，确认后完成支付

**付款码编码规则**：18位纯数字，前缀 10~14 = 中国大陆用户，前缀 15 = 中国香港用户

**开发注意**：

- 付款码支付**没有 V3 版本**，必须使用 V2 接口（XML + MD5/HMAC-SHA256）
- 必须实现 USERPAYING（用户支付中）状态的**轮询查单** + **超时撤销**逻辑，否则会出现用户已付款但商户未感知的问题

### 合单支付说明（不是独立支付方式，而是一种合并下单模式）

**什么是合单支付**：合单支付允许商户将多个商户的订单整合为一个订单进行收款，用户只需支付一次。支持 JSAPI/APP/H5/Native/小程序 五种支付方式，付款码支付不支持合单。

**子订单数量与金额限制**：普通商户模式支持 **2~10笔** 子订单，服务商模式支持 **1~50笔** 子订单，每笔子单金额上限 **1000万**。单笔收款请使用对应的基础支付，不需要走合单。

**支付结果**：所有子单要么全部支付成功，要么全部失败，**不存在部分成功部分失败**的情况。

**与基础支付的关系**：

- 合单支付**不改变支付方式**——仍然通过 JSAPI/APP/H5/Native/小程序 调起支付，用户体验一致
- 区别在于**下单 API 不同**：基础支付用 `/v3/pay/transactions/{type}`，合单支付用 `/v3/combine-transactions/{type}`
- **查单和关单也使用合单专用接口**（按 `combine_out_trade_no` 操作），不能用基础支付的查单/关单接口
- **关单**：关闭**整个合单订单**，不支持只关闭部分子单；关单时子单信息必须与下单时完全一致
- **退款**：无法通过 `combine_out_trade_no` 退款，只能按**单个子单**维度发起退款
- **对账**：以子单为维度，各子单商户自行下载账单
- **回调通知**：回调报文结构与基础支付不同，解密后返回 `sub_orders[]` 数组（含各子单的 `trade_state`、`transaction_id` 等）

**各合单支付方式注意事项**：

合单支付的运行环境、调起方式与基础支付一致（参见上方产品速查表），用户在微信支付确认界面可点击"订单详情"查看各子单的收款方和金额。各方式特殊注意：


| 合单方式     | 关键注意                                                           |
| -------- | -------------------------------------------------------------- |
| JSAPI合单  | JSAPI 支付授权目录只需合单发起方配置                                          |
| APP合单    | Android 需实现 WXPayEntryActivity；Android13 需去除 intent-filter     |
| H5合单     | **不支持在APP内使用**；H5 支付域名只需合单发起方配置；需做跨域安全校验，传真实 `payer_client_ip` |
| Native合单 | **不支持相册识别或长按识别二维码**                                            |
| 小程序合单    | 交易类小程序结算周期受公众平台管控，需确认收货后结算                                     |


**准入条件**：

1. 合单发起方商户与所有子单参与方商户都需要**先开通对应的基础支付**
2. 合单发起方需申请**「发起合单支付」权限**，子单参与方需申请**「接收合单支付」权限**
3. 所有商户号（发起方 + 所有参与方）必须绑定**同一个** `combine_appid`

各支付方式的应用类型要求：


| 合单支付方式   | 前提                  | 应用类型要求                       |
| -------- | ------------------- | ---------------------------- |
| JSAPI合单  | 所有参与方均已开通 JSAPI 支付  | 已认证的服务号、已认证的政府或媒体类型的公众号      |
| APP合单    | 所有参与方均已开通 APP 支付    | 已认证的移动应用                     |
| H5合单     | 所有参与方均已开通 H5 支付     | 已认证的服务号/政府或媒体类型的公众号/小程序/移动应用 |
| Native合单 | 所有参与方均已开通 Native 支付 | 已认证的服务号/政府或媒体类型的公众号/小程序/移动应用 |
| 小程序合单    | 所有参与方均已开通小程序支付      | 已认证的小程序                      |


**合单下单核心字段**：


| 字段                     | 说明                                                     |
| ---------------------- | ------------------------------------------------------ |
| `combine_appid`        | 合单发起方的 APPID                                           |
| `combine_mchid`        | 合单发起方的商户号                                              |
| `combine_out_trade_no` | 合单总订单号，全局唯一                                            |
| `sub_orders[]`         | 子订单数组（2~10个），每个子订单包含 `mchid`、`out_trade_no`、`amount` 等 |


**调起支付**：合单下单返回的产物（`prepay_id` / `h5_url` / `code_url`）与基础支付相同，调起方式也完全一致。

**适用场景举例**：

- 电商平台：用户下单包含 A 店铺和 B 店铺的商品，平台合单后用户一次付款，资金分别结算给 A 和 B
- 合单发起方和子单参与方**可以是同一个商户号**；也支持**境内和境外商户混合**

**限制说明**：

- 合单支付**不支持电子发票**
- **小微类型主体商户号**不支持合单支付
- 不传 `time_expire` 时，默认订单支付有效期为 **7天**（超过后自动关单）

## 常见混淆点

### JSAPI vs 小程序支付

- 共用同一个下单接口（JSAPI/小程序下单）
- 区别在于**调起支付的前端环境**不同：JSAPI 在微信浏览器内用 WeixinJSBridge，小程序用 wx.requestPayment
- 两者都需要 openid，但 openid 的获取方式不同（公众号网页授权 vs 小程序 wx.login）

### H5支付 vs JSAPI支付

- H5 是在微信**外部**的手机浏览器中支付，会跳转到微信完成支付再返回
- JSAPI 是在微信**内部**浏览器中直接调起支付
- 判断标准：用户访问你网页时所处的环境

### Native支付 vs JSAPI扫码场景

- Native 是**商户提供PC网页收银**，页面上展示收款二维码；用户在**手机**微信里「扫一扫」完成支付（PC 浏览、手机扫码付款）
- JSAPI 的扫码场景是用户扫码后在微信内打开H5页面再支付（手机→手机）

### 合单支付 vs 基础支付

- 判断标准：一笔支付里是否涉及**多个子商户**需合并收款；仅单一商户普通收款时，用基础支付即可
- 详见上方「合单支付说明」

## 调起支付差异对比


| 支付方式    | 下单产物      | 调起方式                  | 产物有效期    |
| ------- | --------- | --------------------- | -------- |
| JSAPI   | prepay_id | WeixinJSBridge.invoke | 2小时      |
| 小程序     | prepay_id | wx.requestPayment     | 2小时      |
| APP     | prepay_id | OpenSDK sendReq       | 2小时      |
| H5      | h5_url    | 浏览器跳转链接               | **5分钟**  |
| Native  | code_url  | 转为二维码展示               | 2小时      |
| 付款码(V2) | 无（一步完成）   | 商户扫用户付款码直接扣款          | 付款码每分钟刷新 |


> ⚠️ **V2 与 V3 的关系**：6 种支付方式都有 V2 接口，但 JSAPI/APP/H5/Native/小程序 已推出 V3 接口（推荐使用）。**只有付款码支付没有 V3 版本**，必须使用 V2 接口（XML + MD5/HMAC-SHA256）。V3 使用 JSON 格式 + SHA256-RSA2048 签名，与 V2 的请求格式和签名方式完全不同。
