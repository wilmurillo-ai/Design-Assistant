# 签名与验签规则

## 一、签名基础

### 核心概念

微信支付 APIv3 使用 **SHA256-RSA** 数字签名算法（基于非对称密钥），区别于 V2 的 MD5/HMAC-SHA256。

| 概念 | 说明 |
|------|------|
| 商户API证书 | 商户在商户平台申请的证书，包含公钥和私钥 |
| 商户API私钥 | 用于生成请求签名和调起支付签名，**仅商户持有，不可泄露** |
| 商户API证书序列号 | 标识使用的是哪个证书，放在 HTTP 头中 |
| 微信支付公钥 | 用于验证微信支付响应和回调通知的签名 |
| APIv3密钥 | 用于回调通知中敏感信息的 AES-256-GCM 解密 |

### 接口请求签名（所有 API 调用通用）

所有 V3 API 调用（下单、查单、退款等）都使用同一种签名格式。签名串共 **5 行**，每一行为一个参数，结尾以 `\n`（换行符，ASCII 编码值为 0x0A）结束，**包括最后一行**。如果参数本身以 `\n` 结束，也需要附加一个 `\n`。签名结果放在 HTTP `Authorization` 头中：

```
HTTP请求方法\n
URL（不含域名）\n
请求时间戳\n
请求随机串\n
请求报文主体（GET请求为空）\n
```

Authorization 头格式：

```
Authorization: WECHATPAY2-SHA256-RSA2048 mchid="商户号",nonce_str="随机串",timestamp="时间戳",serial_no="证书序列号",signature="签名值"
```

### 不同参数类型对签名串的影响

签名串5行格式固定，但第2行（URL）和第5行（Body）因接口参数类型不同而有差异：

| 参数类型 | URL 构造 | Body 构造 | 典型接口 |
|---------|---------|----------|---------|
| Body 参数 | 接口路径原样 | JSON body 参与签名 | JSAPI下单 `POST /v3/pay/transactions/jsapi` |
| Path 参数 | 占位符替换为实际值 | 空（GET 请求） | 查询退款 `GET /v3/refund/domestic/refunds/{out_refund_no}` |
| Query 参数 | URL encode 后拼接到路径末尾 | 空（GET 请求） | 查询账单 `GET /v3/bill/tradebill?bill_date=xxx` |
| Path + Query | 先替换占位符，再拼 Query | 空（GET 请求） | 商户订单号查单 `GET /v3/pay/.../out-trade-no/{out_trade_no}?mchid=xxx` |

> **关键**：签名计算时 body 是什么样，发请求时 body 就必须是什么样。一行计算签名，发请求也必须一行。

## 二、各支付方式签名全流程

支付流程涉及两次签名：先调用下单 API（接口请求签名），再在前端调起收银台（调起支付签名）。两次签名格式完全不同，但**使用的商户API证书私钥必须保持一致**。

### JSAPI 支付

**第一步：下单（接口请求签名）**

- 接口：`POST /v3/pay/transactions/jsapi`
- 签名方式：接口请求签名（5行格式），Body 参数类型
- 返回：`prepay_id`

**第二步：调起支付（调起支付签名）**

签名串4行：

```
appId\n
时间戳\n
随机字符串\n
prepay_id=wx201410272009395522657a690389285100\n
```

- 第四行参与签名的是 **`package` 字段的值**，即 `prepay_id=xxx`
- `prepay_id=` 是 `package` 字段值的固有组成部分，不是可选前缀

前端传参（WeixinJSBridge）：

| 参数 | 值 | 是否参与签名 |
|------|------|---------|
| appId | 公众号AppID | 是（第1行） |
| timeStamp | 秒级时间戳 | 是（第2行） |
| nonceStr | 随机字符串 | 是（第3行） |
| package | `prepay_id=xxx` | 是（第4行） |
| signType | `RSA` | 否 |
| paySign | 签名结果 | — |

### 小程序支付

**第一步：下单（接口请求签名）**

- 接口：和 JSAPI 共用 `POST /v3/pay/transactions/jsapi`
- 签名方式：与 JSAPI 完全相同

**第二步：调起支付（调起支付签名）**

- 签名串与 JSAPI **完全相同**（4行格式，第四行也是 `package` 值）

前端传参（wx.requestPayment）：

| 参数 | 值 | 是否参与签名 |
|------|------|---------|
| timeStamp | 秒级时间戳 | 是（第2行） |
| nonceStr | 随机字符串 | 是（第3行） |
| package | `prepay_id=xxx` | 是（第4行） |
| signType | `RSA` | 否 |
| paySign | 签名结果 | — |

> **与 JSAPI 的区别**：小程序不传 `appId`（小程序自动获取），签名串中的 appId 使用小程序 AppID 而非公众号 AppID。

### APP 支付

**第一步：下单（接口请求签名）**

- 接口：`POST /v3/pay/transactions/app`
- 签名方式：接口请求签名（5行格式），Body 参数类型
- 返回：`prepay_id`

**第二步：调起支付（调起支付签名）**

签名串4行：

```
appId\n
时间戳\n
随机字符串\n
wx201410272009395522657a690389285100\n
```

- 第四行参与签名的是 **`prepayId` 字段的值**，即纯 prepay_id
- **不带 `prepay_id=` 前缀**

前端传参（OpenSDK sendReq）：

| 参数 | 值 | 是否参与签名 |
|------|------|---------|
| appId | 移动应用AppID | 是（第1行） |
| partnerId | 商户号mchid | 否 |
| prepayId | 纯 prepay_id 值 | 是（第4行） |
| packageValue | `Sign=WXPay`（固定值） | 否 |
| timeStamp | 秒级时间戳 | 是（第2行） |
| nonceStr | 随机字符串 | 是（第3行） |
| sign | 签名结果 | — |

### H5 支付

**第一步：下单（接口请求签名）**

- 接口：`POST /v3/pay/transactions/h5`
- 签名方式：接口请求签名（5行格式），Body 参数类型
- 返回：`h5_url`

**第二步：调起支付**

- **无需调起支付签名**，直接跳转 `h5_url` 即可
- 可在 `h5_url` 后拼接 `redirect_url` 指定支付完成后的回跳页面（需 URL encode）

### Native 支付

**第一步：下单（接口请求签名）**

- 接口：`POST /v3/pay/transactions/native`
- 签名方式：接口请求签名（5行格式），Body 参数类型
- 返回：`code_url`

**第二步：调起支付**

- **无需调起支付签名**，将 `code_url` 转为二维码展示给用户扫码即可

### JSAPI/小程序 vs APP 调起支付签名核心差异

| 差异点 | JSAPI / 小程序 | APP |
|-------|--------------|-----|
| 签名串第四行签的字段 | `package` | `prepayId` |
| 签名串第四行的值 | `prepay_id=xxx`（带前缀） | 纯 prepay_id（无前缀） |
| 签名结果字段名 | `paySign` | `sign` |
| 是否传 appId | JSAPI 传 / 小程序不传 | 传 |
| 额外字段 | signType=RSA | partnerId + packageValue=Sign=WXPay |

> **这是最容易出错的地方**：两者参与签名的字段本身就不同。JSAPI/小程序签的是 `package` 字段的值，APP 签的是 `prepayId` 字段的值。混用会直接导致签名验证失败。

## 四、验签与回调解密

### 验签（验证微信支付响应/回调）

1. 从 HTTP 头获取 `Wechatpay-Timestamp`、`Wechatpay-Nonce`、`Wechatpay-Signature`、`Wechatpay-Serial`
2. 构造验签串：`时间戳\n随机串\n响应体\n`
3. 使用微信支付公钥验证签名
4. 拒绝时间戳超过5分钟的过期请求

### 回调通知解密

- 回调通知的 `resource` 字段使用 **AES-256-GCM** 加密
- 解密需要 **APIv3密钥**（在商户平台设置）
- 解密参数：`nonce`（随机串）、`associated_data`（附加数据）、`ciphertext`（密文）
- **必须先验签再解密**，严禁不验签就处理回调

## 五、安全规范

### 私钥保护
- 私钥不要硬编码在代码中
- 不要提交到 git 仓库
- 通过环境变量或安全配置中心加载

### H5 支付额外安全要求
- 必须校验跨域请求的 Origin 头，限制白名单
- 下单前必须验证用户登录态
- 未遵循安全规范可能导致支付接口被关停

## 六、常见签名错误速查

### 接口请求签名（下单/查单/退款等 API 调用）

| 错误现象 | 可能原因 |
|---------|---------|
| 签名失败 | GET 请求的 body 为空但末尾缺少 `\n` |
| 签名失败 | URL 中的 Path 占位符未替换为实际值（如 `{out_trade_no}` 未替换） |
| 签名失败 | URL 中遗漏了 Query 参数（签名时的 URL 必须包含完整的 query string） |
| 签名失败 | Query 参数中的 JSON 值未做 URL encode |
| 签名失败 | 签名时的 body 和实际发送的 body 格式不一致（如签名用多行但发送用一行） |
| 签名失败 | 签名串换行符用了 `\r\n` 而非 `\n` |
| 签名失败 | 时间戳使用了毫秒级而非秒级 |

### 调起支付签名（前端/客户端调起收银台）

| 错误现象 | 可能原因 |
|---------|---------|
| JSAPI/小程序签名失败 | 第四行用了纯 prepay_id 值（应该用 `package` 值即 `prepay_id=xxx`） |
| APP 签名失败 | 第四行用了 `prepay_id=xxx`（应该用纯 prepay_id 值） |
| 签名失败 | 调起支付和下单使用了不同的商户API证书私钥 |
| APP errCode: -1 | AppID、包名/签名/Bundle ID 与开放平台配置不一致 |
| 小程序调起无反应 | 使用了公众号 AppID 而非小程序 AppID |
