# 基础支付排障手册

> 本文档覆盖商户模式和服务商模式的排障内容。各支付方式独有的前端报错见 `基础支付常见问题.md`，合单支付见 `合单支付常见问题.md`。
> 服务商模式特有问题集中在本文第八节。
>
> **错误码提取**：Request-Id 的格式为 `{请求标识}-{错误码}`，**截取最后一个 `-` 后面的数字即为错误码**。例如 `08E68E97CC0610DF0618AC86BCA80120082880CF05-268578704` → 错误码 `268578704`。
>
> **排障流程**：
> 1. **优先让用户提供 Request-Id**，提取错误码后在下方「错误码 TOP 20 速查表」中匹配，命中后直接使用对应的解决方案回复用户。
> 2. 若用户无法提供 Request-Id，或错误码不在 TOP 20 中，再按「快速定位流程」及后续各章节兜底排查。
> 3. 分账相关问题查阅 `分账常见问题.md`。
> 4. 服务商模式特有问题查阅第八节。
>
> ‼️ **语气要求**：排障回复应亲和、专业，像一个有经验的技术支持在帮忙。避免冷冰冰地罗列信息或堆砌文档目录。用自然的对话语气解释问题原因和修复方案。

## 错误码详细排查

### 268512771 — 订单不存在

**常见原因**：
- 商户订单号 `out_trade_no` 或微信订单号 `transaction_id` 传错了（大小写、多余空格）
- 查单/退款时用的商户号 `mchid` 与下单时不一致，导致跨商户号查不到
- 下单接口本身就没成功（没返回 prepay_id），订单根本没创建
- 退款时订单还没支付成功，未支付的订单无法退款
- **可能漏开发了查单接口**：商户没有实现查单逻辑，支付成功后无法确认订单状态，后续操作（退款/关单）因为状态不明而传了错误的订单号

**🔧 脚本确认**：向用户确认商户订单号和商户号后，提议「我帮你查一下这笔订单是否存在，需要你先在服务器上对查单请求做一次签名」，然后执行对应模式的查询订单脚本（商户：`scripts/商户/查询订单.py`；服务商：`scripts/服务商/查询订单.py`）。根据结果：
- 查到订单 → 告知用户订单状态，根据状态推荐下一步操作的代码示例（退款/关单）
- 仍查不到 → 说明订单号或商户号确实有误，帮用户排查

**💡 推荐集成**：刚才脚本调用的就是**商户订单号查单接口**，你的系统中如果还没有实现这个接口，建议补上。代码示例见对应接入模式的接口索引

---

### 268447840 — 请求频率超过限制

**场景**：高并发调用下单/查单等接口时触发。

**排查步骤**：
1. 降低请求频率，使用指数退避策略重试
2. 如业务量确实需要更高频率，联系微信支付运营申请提频

---

### 268435461 — 参数非法

**场景**：请求参数值格式或内容不合法。

**排查步骤**：
1. 检查金额字段单位是否为**分**（非元）
2. 检查时间格式是否为 RFC 3339（正确：`2025-08-01T00:00:00+08:00`）
3. 检查是否有必填参数缺失
4. 检查枚举值拼写是否正确
5. 检查是否有不可见字符（如全角空格、BOM 头）

---

### 268449538 / 268498689 — 订单已支付

**常见原因**：
- 用同一个 `out_trade_no` 重复下单，但该订单已经支付成功了
- 对已支付的订单调用了关单接口（已支付订单无法关单）
- **可能漏开发了查单接口**：商户在下单前没有先查询订单状态，不知道订单已经支付成功，导致重复下单或误调关单
- **可能漏开发了退款接口**：已支付订单需要取消时应走退款流程，而不是关单

**🔧 脚本确认**：提议「我帮你查一下这笔订单的当前状态，确认是否真的已支付」，执行对应模式的查询订单脚本。根据结果：
- 确认已支付（SUCCESS）且用户想取消 → 已支付订单只能退款不能关单，推荐退款代码示例
- 确认已支付且用户想重新下单 → 提醒须用新的 `out_trade_no`，推荐对应支付方式的下单代码示例

**💡 推荐集成**：刚才脚本调用的就是**商户订单号查单接口**，你的系统中如果在下单前先查一下订单状态，就能避免重复下单导致的报错。代码示例见对应接入模式的接口索引

---

### 270924358 — 参数中含有非法的额外参数

**场景**：请求体中包含了该 API 不支持的字段。

**排查步骤**：
1. 对照官方 API 文档，移除请求体中所有文档未定义的字段
2. 注意 V2 和 V3 的参数结构不同，不要混用

---

### 268443715 — 商户状态为不可用或者已经注销

**场景**：商户号被冻结、注销或状态异常。

**排查步骤**：
1. 登录微信商户平台确认商户号状态
2. 如已注销需重新申请，如被冻结需联系微信支付运营处理

---

### 268447843 — 商户号和订单不匹配

**常见原因**：
- 查单或退款时用的 `mchid` 与下单时不一致（常见于有多个商户号的商户，搞混了）
- `out_trade_no` 不属于当前商户号（可能是另一个商户号下的订单）
- **可能漏开发了查单接口**：商户没有在支付成功后通过查单确认订单归属，后续退款/关单时传错了商户号

**🔧 脚本确认**：提议「我帮你用正确的商户号查一下这笔订单，确认它属于哪个商户号」，执行对应模式的查询订单脚本。根据结果：
- 查到订单 → 告知用户订单归属的商户号，提醒后续操作需使用同一个 `mchid`
- 查不到 → 建议用户换另一个商户号再查

**💡 推荐集成**：刚才脚本调用的就是**商户订单号查单接口**，建议在业务系统中记录每笔订单对应的商户号，并在退款/关单前先查单确认。代码示例见对应接入模式的接口索引

---

### 268447914 — 该订单已失效

**常见原因**：
- 订单已被关单接口关闭，或超过 7 天自动关闭
- `time_expire` 设置过短，用户还没来得及支付订单就过期了
- `prepay_id` 超过 2 小时有效期后仍尝试调起支付（prepay_id 过期需重新下单获取新的）
- **可能漏开发了关单接口**：商户没有实现超时关单逻辑，未支付订单长期占用订单号，用户再次下单时用了同一个 `out_trade_no` 但订单已被微信侧自动关闭

**🔧 脚本确认**：提议「我帮你查一下这笔订单的状态，确认是否已经关闭」，执行对应模式的查询订单脚本。根据结果：
- 订单状态为 CLOSED → 告知用户需要用新的 `out_trade_no` 重新下单，推荐对应支付方式的下单代码示例

**💡 推荐集成**：刚才脚本调用的就是**商户订单号查单接口**，建议在业务流程中补上两个环节：① 下单后定期查单确认状态；② 未支付订单超时后主动关单。代码示例见对应接入模式的接口索引

---

### 268448010 — 此商家的收款功能已被限制

**场景**：商户或服务商存在违规被处罚。

**排查步骤**：
1. 登录「微信支付商家助手」小程序 → 风险处理，查看具体原因
2. 或登录 pay.weixin.qq.com → 账户中心 → 消息 → 违约记录
3. 按指引申诉（移动端违约查询：https://kf.qq.com/faq/240129aQZjiM240129QRBFRV.html ；PC端申诉：https://kf.qq.com/faq/240129iuEZBJ240129meAR3u.html ）

---

### 268449641 — 当前交易不允许进行普通关单（只能撤销）

**场景**：对付款码支付等场景的订单调用了关单接口。

**排查步骤**：
1. 付款码支付场景不支持关单，需改用**撤销接口**（reverse）
2. 支付失败/超时一律用撤销接口；只有支付成功后才用退款接口

---

### 268443699 — 签名错误

**场景**：请求签名验证不通过（HTTP 401）。

**排查步骤**：
1. 检查 Authorization 头格式是否符合规范
2. 确认 `serial_no` 与使用的私钥文件匹配
3. 检查签名串拼接：换行符、URL 格式、body 为空时末尾是否有 `\n`
4. 确认服务器时间偏差不超过 5 分钟
5. 建议使用官方 SDK 避免手动签名出错

---

### 268501013 — JSAPI支付必须传openid

**场景**：JSAPI 或小程序下单时未传 `payer.openid`。

**排查步骤**：
1. JSAPI 支付：通过公众号网页授权获取 openid
2. 小程序支付：通过 `wx.login` 获取 code，后端调用 `auth.code2Session` 换取 openid
3. 确保获取 openid 使用的 appid 与下单的 appid 一致

---

### 268498688 — 请求重入时，参数与首次请求时不一致

**场景**：同一 `out_trade_no` 二次下单但参数与首次不同。

**排查步骤**：
1. 同一 `out_trade_no` 必须用完全相同的参数重新请求
2. 如需修改参数（如金额），先调用关单接口关闭原订单，再用新的 `out_trade_no` 下单

---

### 268445527 — appid和mch_id不匹配

**场景**：下单时传的 appid 未与商户号建立绑定关系。

**排查步骤**：
1. 登录商户平台确认 appid 是否已绑定（绑定指引：https://pay.weixin.qq.com/doc/v3/merchant/4013287010 ）
2. 服务商场景下 `sp_appid` 也需与 `sp_mchid` 绑定
3. 绑定后需等待几分钟生效

---

### 268435464 — 参数超出取值范围

**场景**：参数值超出 API 允许的范围。

**排查步骤**：
1. 检查金额是否超出限额（单位为分）
2. 检查字符串字段是否超出最大长度
3. 对照 API 文档中各字段的取值范围说明

## 快速定位流程

```
用户报错
  ├─ 下单失败？
  │   ├─ HTTP 401 → 签名问题 → 查本文「一」
  │   ├─ HTTP 400 → 参数问题 → 查本文「一」
  │   ├─ HTTP 403 → 权限问题 → 查本文「一」
  │   └─ 业务错误码 → 绑定/权限/参数 → 查本文「二」
  ├─ 调起支付失败？ → 查各支付方式专项文件
  ├─ 回调收不到？ → 查本文「三」
  ├─ 退款异常？ → 查本文「五」
  └─ 账单异常？ → 次日10点前未生成 / 超3个月需商户平台下载 / 金额单位"元"vs API"分"
```

## 一、下单 HTTP 错误

| HTTP 状态码 | 含义 | 常见原因 | 排查要点 |
|------------|------|---------|---------|
| 401 | 签名验证失败 | 私钥与证书不匹配；serial_no 填错；签名串拼接有误（换行符/URL/body为空时缺末尾换行）；时间戳偏差过大 | 检查 Authorization 头格式；确认私钥文件正确加载；建议使用官方 SDK |
| 400 | 请求参数错误 | 必填参数缺失；金额单位是分不是元；时间格式不符 RFC 3339；JSON 层级错误 | 对照 API 文档逐项检查；`amount.total` 单位是**分**；`time_expire` 格式：`yyyy-MM-ddTHH:mm:ss+08:00` |
| 403 | 权限不足 | 未开通对应支付产品；IP 不在白名单；商户号状态异常 | 商户平台 → 产品中心确认开通状态 |

## 二、下单业务报错（TOP 20 未覆盖部分）

> 以下仅列出错误码 TOP 20 未覆盖的下单报错。TOP 20 已覆盖的报错请查上方错误码详细排查。

| # | 报错信息 | 原因 | 解法 |
|---|---------|------|------|
| 1 | "appid与openid不匹配" / "无效的openid" | openid 不属于该 appid（常见：下单和调起用了不同 appid） | 确保获取 openid 使用的 appid 与下单 appid 一致 |
| 2 | "商户传入的appid参数不正确" | 下单 appid 与调起 appid 不一致 | 两者必须用同一个 appid |
| 3 | "mchid不匹配" | Authorization 头中的 mchid 与请求体中的不一致 | 统一两处 mchid（服务商模式 Authorization 头中应填 sp_mchid） |
| 4 | "下单账号与支付账号不一致" | 下单时的 openid 与实际支付用户不是同一人 | 确保同一用户下单和支付 |
| 5 | "商户号该产品权限未开通" | 未开通对应支付产品 | 商户平台 → 产品中心开通 |
| 6 | "商户未申请过证书" | 未申请 API 证书 | 参考证书申请指引：https://pay.weixin.qq.com/doc/v3/merchant/4013053053 |
| 7 | "商户号或子商户号与订单号不匹配" | 商户订单号在同一商户号下（含所有子商户号）必须唯一 | 更换唯一的商户订单号 |
| 8 | SIGN_ERROR "商户证书序列号有误" | serial_no 与签名私钥不匹配 | 参考签名计算：https://pay.weixin.qq.com/doc/v3/merchant/4012365342 |

## 三、回调问题

> 回调问题在商户工单中占比近 1/3，是最常见的支付接入问题。
> ⚠️ 付款码支付没有异步回调，不适用本节。

### 收不到回调的排查清单（按优先级）

| 序号 | 排查项 | 说明 |
|------|--------|------|
| 1 | 地址不可达 | URL 配置错误、域名解析失败、localhost/内网 IP、服务未启动 |
| 2 | URL 格式问题 | URL 前有多余空格导致 DNS 解析失败 |
| 3 | 防火墙拦截 | 未对微信支付回调 IP 段开白名单（见下方 IP 列表） |
| 4 | 登录态拦截 | notify_url 不携带 session/token，须从鉴权中间件中排除 |
| 5 | 响应不正确 | 返回非 200（如 FAIL、404），微信重试后最终放弃 |
| 6 | 处理超时 | 未在 5 秒内返回应答 |
| 7 | 域名未备案 | 回调域名需完成 ICP 备案 |
| 8 | 商户号用错 | 实际已收到回调，但用另一个商户号查单导致"订单不存在" |

### 回调行为

| # | 问题 | 答案 |
|---|------|------|
| 1 | 怎么确认微信有没有发回调？ | 微信不提供回调日志查询，需检查自身服务器访问日志 + 查单接口确认 |
| 2 | 回调会重复收到吗？ | 会。商户未正确响应时微信会重试，业务必须做幂等 |
| 3 | 回调延迟正常吗？ | 正常，数秒到数十秒均属预期。建议回调 + 主动查单双保险 |
| 4 | 能直接将回调当最终结果吗？ | 不能。回调不保证送达，需结合查单接口确认 |
| 5 | 商户平台能查回调状态吗？ | 不支持，需调用查单接口 |
| 6 | 回调怎么测试？ | 无测试接口，需生产环境真实支付验证 |

### 回调解密与验签

| # | 报错信息 | 原因 | 解法 |
|---|---------|------|------|
| 1 | "cipher: message authentication failed" / `AEADBadTagException` | APIv3 密钥错误（最常见：密钥重置后代码未同步）或密文被截断 | 检查代码中的 APIv3 密钥是否与商户平台一致 |
| 2 | "证书序列号不一致" | 用了商户证书做验签（应用平台证书）或平台证书过期 | 改用平台证书并确保未过期 |
| 3 | "Last unit does not have enough valid bits" | 签名探测流量 | 检查 `Wechatpay-Signature` 是否以 `WECHATPAY/SIGNTEST/` 开头，是则返回非 2xx |
| 4 | 签名参数顺序错误 | 参数个数/顺序/大小写不对或末尾缺 `\n` | 严格按文档顺序拼接，末尾必须有 `\n` |

## 四、签名与证书

| # | 问题 | 答案 |
|---|------|------|
| 1 | 证书序列号怎么获取？ | 商户平台 → 账户中心 → API安全 → 商户API证书 → 管理证书 |
| 2 | 一个商户号能设多个 API 证书吗？ | 可以 |
| 3 | 平台证书过期怎么更换？ | 参考更换指引：https://pay.weixin.qq.com/doc/v3/merchant/4012068829 ，建议代码中实现证书自动轮换 |
| 4 | 更换证书编号（serial_no）后报签名错误？ | 证书编号与证书私钥一一对应，更新 serial_no 时必须同步更换私钥文件 |
| 5 | V2 签名失败怎么排查？ | 逐项对比签名原串：① 字段按 ASCII 字典序；② 大小写一致；③ 无多余空格或遗漏字段 |
| 6 | V2 签名方式？ | MD5 或 HMAC-SHA256（不是 V3 的 SHA256-RSA2048） |
| 7 | V2 支持公钥模式吗？ | 不支持，V2 使用 API 密钥 |
| 8 | API 只能通过域名访问吗？ | 是，不支持 IP 直连 |
| 9 | APIv2 密钥修改后验签失败？ | 密钥重置后代码中的密钥必须同步更新 |

## 六、订单与支付规则

> 订单相关报错（订单不存在、订单已支付、订单已失效、不允许关单等）已在上方 TOP 20 中详细排查，以下为补充的业务规则 Q&A。

| # | 问题 | 答案 |
|---|------|------|
| 1 | prepay_id 有效期？ | 固定 **2 小时**，与 `time_expire` 无关，过期需重新下单 |
| 2 | time_expire 设超过 2 小时有用吗？ | 订单有效期可超 2 小时（默认 7 天），但 prepay_id 仍需 2 小时内重新获取 |
| 3 | out_trade_no 能复用吗？ | 同一单号只能用完全相同的参数重新请求，不允许更换参数 |
| 4 | 同一订单能被不同用户支付吗？ | 不能。但同一微信号通过不同 appid（不同 openid）支付同一笔订单允许 |
| 5 | transaction_id vs prepay_id？ | prepay_id 下单后返回用于调起支付；transaction_id 支付成功后生成 |
| 6 | 关单接口可重复调用吗？ | 可以，关单接口可重入 |
| 7 | 下单金额传 0？ | 报错 PARAM_ERROR，最小值为 1（分） |
| 8 | 支付结果看哪个字段？ | 看 `trade_state`，不是 `result_code` |
| 9 | 支付失败应该用撤销还是退款？ | 支付失败/超时用**撤销接口**；只有支付成功后才用退款接口（仅付款码涉及撤销） |

## 八、服务商模式特有问题

> 以下问题仅在服务商模式下出现。

### 权限与受理关系

| 报错信息 | 原因 | 解决 |
|---------|------|------|
| "受理关系不存在" / "受理关系不可用" | 服务商与子商户之间没有对应支付产品的授权关系 | 服务商平台 → 产品中心 → 特约商户授权产品 → 发起邀请；子商户商户平台 → 产品中心 → 我的授权产品 → 授权 |
| "该产品授权已被解除" | 子商户解除了服务商授权 | 子商户商户平台 → 特约商户授权产品 → 重新开通 |
| "特约子商户商户号未授权服务商的产品权限" | 子商户未授权服务商对应支付权限 | 子商户商户平台 → 产品中心 → 特约商户授权产品 → 服务商授权；授权后等待约 10 分钟生效 |
| "此商家的收款功能已被限制，暂无法支付" | 子商户号被微信支付风控限制 | 子商户超级管理员通过"微信支付商家助手"公众号/小程序查看原因并申诉 |

### APPID 与商户号绑定

| 报错信息 | 原因 | 解决 |
|---------|------|------|
| "appid和mch_id不匹配" | `sp_appid` 与 `sp_mchid` 未绑定 | 服务商平台 → 产品中心 → APPID授权管理 中确认已关联 |
| "sub_mch_id与sub_appid不匹配" | `sub_appid` 未与 `sub_mchid` 绑定 | 服务商平台 → 合作伙伴功能 → 开发参数配置 → 特约商户APPID配置 |
| "子商户mchid不能与自身mchid相同" | `sub_mchid` 传了服务商自己的商户号 | `sub_mchid` 应传子商户的商户号 |
| "受理机构必须传入sub_mch_id" | 服务商模式下漏传 `sub_mchid` | 服务商模式所有下单/查单/关单接口都必须传 `sub_mchid` |

### openid 相关

| 报错信息 | 原因 | 解决 |
|---------|------|------|
| "appid与openid不匹配" | `sp_openid` 不是在 `sp_appid` 下获取的 | `sp_openid` 必须通过 `sp_appid` 对应的公众号/小程序获取 |
| "sub_appid和sub_openid不匹配" | `sub_openid` 不是在 `sub_appid` 下获取的 | `sub_openid` 必须通过 `sub_appid` 对应的公众号/小程序获取 |
| "无效的openid" | 传入的 `sp_openid` 或 `sub_openid` 不存在 | 重新获取正确的 openid |
| "下单账号与支付账号不一致" | 下单 openid 与实际支付用户不一致 | 确保下单传的 sp_openid/sub_openid 与实际支付用户一致 |

### 子商户号与订单

| 报错信息 | 原因 | 解决 |
|---------|------|------|
| 268447845 "子商户号和订单信息不匹配" | 查单/退款时传的 `sub_mchid` 与下单时不一致 | 确保 `sub_mchid` 与下单时保持一致，检查是否误用了其他子商户号 |
| "http header中的mchid与post payload中的mchid不匹配" | 请求头 mchid 与请求体不一致 | 服务商模式下 Authorization 头中的 mchid 应为 `sp_mchid`，而非 `sub_mchid` |

### 其他服务商常见问答

| 问题 | 答案 |
|------|------|
| 子商户配了 APPID 但没配 JSAPI 授权目录，却能成功调起支付？ | 服务商自身配置了匹配的授权目录。只要服务商或子商户其中一个配了就行 |
