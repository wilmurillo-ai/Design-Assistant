# 安装决策规则

> 无论用户如何触发安装（巡查推荐 / 直接说"安装第X个"），都必须先判断影响程度，再决定流程。

## 判断流程

```
收到安装请求
    ↓
评估该 skill 是否会影响：人设 / SOUL.md / MEMORY.md / 行为方式 / cron 任务 / 用户可感知交互
    ↓
  无影响（工具型）              有影响（增强型 / 可能改变规则或行为）
    ↓                               ↓
直接安装                     ① 描述 skill 功能
装完通知用户                 ② 列出所有影响点（⚠️ 有感知 / 🚨 不可逆）
新增了什么能力               ③ 等待用户明确确认
                             ④ 确认后才执行安装
```

**❌ 禁止**：有影响的 skill 直接装，装完才说有影响
**✅ 正确**：先判断 → 无影响直接装+通知 → 有影响先报告再确认

## 影响程度判断标准

| 类型 | 判断依据 | 处理方式 |
|---|---|---|
| 🔧 **工具型**（无影响） | 纯新增能力，与已有 skill/规则/行为无冲突 | 直接安装，装完通知 |
| 🧩 **增强型**（有影响） | 与已有 skill 或规则有重叠，或会改变行为方式 | 描述+报告影响→等确认→再装 |
| ⏭️ **重复型** | 已有 skill/规则完全覆盖该能力 | 跳过，说明已有什么覆盖 |

## 影响程度细分

### 有感知 = 必须确认（⚠️）
- 改变回复风格/语气/人格
- 改变与用户的交互方式（主动性、频率、回复长度等）
- 改变用户可见的任务执行步骤
- 修改 SOUL.md / USER.md / MEMORY.md 核心约定
- 新增或删除 cron 任务

### 无感知 = 可直接应用（静默）
- 内部工具调用顺序优化
- 错误处理逻辑改进
- 不影响输出结果的技术细节

### 不可逆 = 必须二次确认（🚨）
- 删除规则、覆盖 SOUL.md 内容、修改核心配置等

## 增强型内容分类标记

| 标记 | 判断条件 | 处理方式 |
|---|---|---|
| ✅ 可直接应用 | 与现有规则无冲突 | 列出内容，可静默应用 |
| 🔄 内部优化 | 与现有规则冲突，但用户无感知 | 列出内容，可静默应用 |
| ⚠️ 需用户确认 | 与现有规则冲突，且用户有感知 | 列出具体改动，等用户逐项确认 |
| 🚨 不可逆风险 | 删除规则、覆盖 SOUL.md 等 | 明确警告，需二次确认 |

**判断口诀：**
- 这是"一套操作流程" → 装 skill，让系统注入
- 这是"一个教训/习惯" → 写进 MEMORY.md 铁律
