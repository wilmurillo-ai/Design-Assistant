# Agent Skills 设计哲学详解

本文档是 Skill 验证器的完整参考手册，包含五大原则的详细验证标准、反模式、改进优先级定义和报告模板。

---

## 一、渐进式披露 (Progressive Disclosure)

### 三层加载模型

```
┌─────────────────────────────────────────────────────────┐
│ Level 1: 元数据层 (Always in context)                   │
│ - name + description                                    │
│ - 约 100 词                                             │
│ - 用于技能发现和触发决策                                │
├─────────────────────────────────────────────────────────┤
│ Level 2: 核心指令层 (In context when triggered)         │
│ - SKILL.md 主体内容                                     │
│ - 理想 < 500 行                                         │
│ - 包含核心工作流程和关键指令                            │
├─────────────────────────────────────────────────────────┤
│ Level 3: 资源层 (Loaded on demand)                      │
│ - references/, scripts/, assets/                        │
│ - 无限制                                                │
│ - 按需加载，不占用初始上下文                            │
└─────────────────────────────────────────────────────────┘
```

### 验证标准明细

| 检查项 | 标准 | 严重程度 |
|--------|------|----------|
| SKILL.md 行数 | < 200 行优秀，< 500 行理想，< 800 行可接受 | ⚠️ > 800 需拆分 |
| 元数据完整性 | name + description 必填 | ❌ 缺失则无效 |
| 资源目录使用 | 大型内容应移至 references/ | ⚠️ 建议改进 |
| 资源引用清晰度 | SKILL.md 中有明确的资源指针，含"何时读取"条件 | ⚠️ 建议改进 |
| 文件系统利用 | 整个文件夹被视为上下文工程手段，而非仅靠 SKILL.md 单文件 | ⚠️ 建议改进 |

### 常见反模式

1. **知识倾倒**：将所有内容塞进 SKILL.md，不使用 references/
2. **嵌套过深**：在 SKILL.md 中包含大量详细示例
3. **缺少指针**：有 references/ 但 SKILL.md 中未明确引用
4. **单文件思维**：未将文件夹视为上下文工程手段，所有内容集中在 SKILL.md 而非分布在 scripts/、assets/、references/ 中

## 二、知识差原则 (Knowledge Delta Principle)

知识差 = 专业领域知识 − 模型通用知识。Skill 只包含差值部分，且应聚焦于将模型推出常规思维模式的内容。

### 应包含的内容

| 类型 | 示例 |
|------|------|
| 领域专业知识 | 特定框架的高级用法、行业规范 |
| 组织特定知识 | 公司编码规范、内部 API |
| 工作流程知识 | 特定的多步骤流程、最佳实践 |
| 上下文特定知识 | 项目结构、命名约定 |

### 不应包含的内容

| 类型 | 示例 |
|------|------|
| 通用编程知识 | "Python 使用 def 定义函数" |
| 基础语法说明 | "JSON 是一种数据格式" |
| 模型已知内容 | 常见算法、标准库用法 |
| 可推导的知识 | 可从其他信息推断的内容 |

### 验证标准明细

| 检查项 | 标准 | 严重程度 |
|--------|------|----------|
| 冗余知识检测 | 无明显的模型已知内容 | ⚠️ 删除以节省 token |
| 专业知识密度 | 高价值、不可替代知识占比高 | ✅ 质量指标 |
| 具体 vs 抽象 | 具体指导优于抽象原则 | ⚠️ 太抽象则价值低 |
| 常见失败点聚焦 | 内容聚焦于模型的常见错误，而非重复通用最佳实践 | ✅ 高信号值 |

### 常见反模式

1. **教程化**：将 Skill 写成入门教程
2. **基础知识重复**：解释模型已熟知的通用概念
3. **过度抽象**：只有原则没有具体操作指导
4. **通用最佳实践堆砌**：罗列模型已经知道的常规做法，而非指出模型容易犯错的特定场景

## 三、提示词扩展架构 (Prompt Expansion Architecture)

Skills 本质上是**提示词模板**，通过注入指令来修改对话上下文，而非执行代码。

### Skills vs 传统工具

| 方面 | 传统工具 (Tools) | Skills |
|------|-----------------|--------|
| 执行模型 | 同步、直接执行 | 提示词扩展 |
| 本质 | 执行代码返回结果 | 注入指令修改上下文 |
| 目的 | 执行特定操作 | 指导复杂工作流 |
| 返回值 | 操作结果 | 上下文变化 |

### 验证标准明细

| 检查项 | 标准 | 严重程度 |
|--------|------|----------|
| 指令性质 | 内容是指令/指导，非可执行代码 | ❌ 违反则需重写 |
| 工作流导向 | 引导模型完成任务的方式 | ✅ 符合设计 |
| 工具使用指导 | 告诉模型何时用什么工具 | ✅ 符合设计 |

### 常见反模式

1. **代码即 Skill**：将 Python 脚本内容直接写入 SKILL.md
2. **函数定义式**：用函数签名的方式描述 Skill 功能
3. **执行逻辑混入**：包含条件分支、循环等程序逻辑

## 四、Gotchas 章节质量 (Gotchas Section Quality)

Gotchas 章节是 Skill 中最高信号值的内容。它记录了模型在使用该 Skill 时反复遇到的常见失败点，应随时间积累和更新。

### 验证标准明细

| 检查项 | 标准 | 严重程度 |
|--------|------|----------|
| 章节存在性 | 存在 Gotchas / 常见陷阱 / 注意事项 章节 | ⚠️ 缺失则信号值低 |
| 条目数量 | ≥ 3 条为基线，5-9 条为理想 | ⚠️ 过少则覆盖不足 |
| 条目具体性 | 每条描述具体场景和具体错误 | ⚠️ 泛泛的不如没有 |
| 可操作性 | 每条包含如何避免或修复的指导 | ⚠️ 只说问题不说方案 |
| 来源真实性 | 来自实际失败经验而非臆测 | ✅ 高信号值 |

### 好的 Gotchas 示例

- ✅ "模型在处理 PDF 表格时倾向于用 tabula，但 tabula 对合并单元格处理很差。应优先使用 camelot 的 lattice 模式。"
- ✅ "使用 openpyxl 写入公式时，模型经常忘记设置 `data_only=False`，导致公式被替换为 None。"

### 差的 Gotchas 示例

- ❌ "注意性能问题" — 太泛泛
- ❌ "确保代码质量" — 没有具体场景
- ❌ "处理边缘情况" — 没有说明哪些边缘情况

### 常见反模式

1. **无 Gotchas 章节**：完全缺失，Skill 的信号值大打折扣
2. **泛泛而谈**：条目过于抽象，如"注意错误处理"
3. **过时不更新**：Gotchas 应随使用经验持续积累
4. **臆测而非经验**：编造可能的问题，而非记录实际遇到的问题

## 五、Description 触发质量 (Description Trigger Quality)

Description 是 Skill 的"门面"，决定了模型能否准确匹配任务并触发 Skill。模糊的 description 会导致 Skill 很少被正确触发。

### 验证标准明细

| 检查项 | 标准 | 严重程度 |
|--------|------|----------|
| 具体性 | 具体描述 Skill 做什么，而非模糊的"有用的工具" | ❌ 模糊则无法触发 |
| Action-oriented | 以动作为导向描述触发条件 | ⚠️ 被动描述匹配率低 |
| 触发短语覆盖 | 包含用户可能使用的关键词和短语 | ⚠️ 缺失则遗漏触发 |
| 长度适当性 | 50-200 词的有效区间 | ⚠️ 过短匹配不准，过长噪音多 |
| 负向边界 | 明确说明何时不应触发 | ✅ 减少误触发 |

### 好的 Description 示例

- ✅ "Use this skill whenever the user wants to do anything with PDF files. This includes reading, merging, splitting, rotating pages, adding watermarks... If the user mentions a .pdf file or asks to produce one, use this skill."
- ✅ "MUST trigger proactively after ANY SKILL.md file is created, written, edited, or modified. Also trigger when user asks to 'validate skill', 'check skill design'..."

### 差的 Description 示例

- ❌ "project utilities" — 无法匹配任何具体任务
- ❌ "helpful patterns" — 模型不知道何时触发
- ❌ "A tool for doing stuff with files" — 过于模糊

### 常见反模式

1. **模糊泛化**：description 如"有用的工具"、"项目辅助"
2. **缺少触发短语**：未列出用户可能使用的关键词
3. **过短**：只有几个词，无法提供足够的匹配信号
4. **过长且散漫**：超过 200 词但未增加匹配精度
5. **缺少负向边界**：未说明何时不应触发，导致误触发

## 六、改进优先级定义

| 级别 | 含义 | 示例 |
|------|------|------|
| **P0 - 阻塞** | Skill 无法正常触发或工作 | description 缺失、name 不规范、结构严重错误 |
| **P1 - 重要** | 显著影响 Skill 效果 | SKILL.md 过长(>1000行)、缺少触发短语、大量冗余知识 |
| **P2 - 优化** | 可改进但不影响基本功能 | 渐进式披露可优化、写作风格可改进、资源组织可更好 |
| **P3 - 锦上添花** | 边际改进 | 添加更多示例、完善边缘情况、增加参考链接 |
