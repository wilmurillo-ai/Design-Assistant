## 概述

对各类技术方案或业务执行方案进行系统性的“压力测试”与沙盘推演。不仅检查文档结构的完整性，更核心的是通过逻辑演绎，提前暴露方案在极端场景、高并发或异常输入下的潜在崩溃点，确保方案具备健壮性、可扩展性以及完善的兜底能力。

# 审查提示词：方案推演（Solution Deduction & Walkthrough）

**适用文件夹名**：`方案推演` 或 `架构设计`
**核心价值**：逻辑自洽、防患未然、底线思维。

## Agent 执行说明

执行 Phase B 时，将本节 **审查维度表** 作为该类文档的 **唯一** 内容审查 checklist：要求 Agent 模拟“黑客攻击者”与“极端测试专家”的视角，对每一行维度逐条对照原文进行推演，给出结论并写入审查报告「3. 内容审查结论」；发现的逻辑漏洞和隐患写入「4. 分级问题清单」并按主 Skill 的严重度规则 @ 作者。**规则追溯**：本 Skill 的推演逻辑需严格基于上下文提供的技术栈与业务场景，避免脱离实际的空对空推演。

## 审查维度（Agent 自检），按六大维度推演

### 维度一：需求对齐与目标闭环（基石检查）

**检查项：**
- [ ] **目标锚定**：方案设计的最终输出是否与前置的业务需求（PRD/背景动机）强对应，是否存在“为了用新技术而用新技术”的过度设计？
- [ ] **逻辑闭环**：输入（Input）到输出（Output）的全链路是否完整，是否存在“数据流断层”或“关键模块缺失”？
- [ ] **衡量指标（Metrics）**：方案是否定义了明确的成功/验收标准（如 QPS、延迟、准确率提升百分比）？

---

### 维度二：技术逻辑与架构合理性（可行性审查）

**检查项：**
- [ ] **依赖耦合度**：上下游系统/模块的依赖关系是否清晰？是否存在严重的强耦合或单点故障（SPOF）风险？
- [ ] **技术选型**：所选用的组件/算法模型是否适合当前场景的数据规模与实时性要求？
- [ ] **状态与并发处理**：对于分布式系统或高并发场景，是否考虑了状态一致性（如锁机制、幂等性设计、分布式事务）？

---

### 维度三：边界条件与异常推演（Corner Cases，权重最高）

**检查项：**
- [ ] **极端输入测试**：如果输入为空、格式错误、超出最大限制或包含恶意注入，方案是否说明了拦截与处理机制？
- [ ] **链路超时与熔断**：当下游依赖服务宕机或响应严重超时，系统是否会发生雪崩？是否设计了降级（Degradation）和熔断（Circuit Breaking）机制？
- [ ] **数据冷启动/长尾问题**：对于算法方案，是否考虑了零样本（Zero-shot）、冷启动或者数据极度不平衡时的长尾失效表现？

---

### 维度四：资源开销与性能评估（ROI 检查）

**检查项：**
- [ ] **性能瓶颈（Bottleneck）预判**：随着数据量或流量的指数级增长，方案中最先扛不住的环节是哪里（DB、网络带宽、GPU显存）？
- [ ] **成本估算**：是否粗略评估了方案实施所需的硬件成本、API 调用成本或人力维护成本？ROI 是否合理？
- [ ] **扩展性（Scalability）**：方案是否支持水平扩容（Scale-out）？数据结构设计是否为未来的业务扩展留出了余地？

---

### 维度五：风险管控与兜底预案（底线思维）

**检查项：**
- [ ] **回滚预案（Rollback）**：如果上线后发生重大故障，是否有能在几分钟内恢复到旧版本的紧急切流/回滚方案？
- [ ] **灰度发布策略**：方案是否包含平滑上线的计划（如 AB 测试、按地域/白名单灰度）？
- [ ] **监控与告警**：方案上线后，如何第一时间知道它出问题了？是否设计了核心监控看板与告警阈值？

---

### 维度六：安全合规与代码规范

**检查项：**
- [ ] **数据隐私与合规**：方案是否涉及敏感用户数据（PII）的未脱敏落盘？是否符合相关数据保护法规（如 GDPR, 数据出境规定）？
- [ ] **权限与越权漏洞**：操作接口是否进行了严格的身份校验和权限鉴权（水平越权/垂直越权防范）？
- [ ] **日志合规**：业务日志是否可能打印了不该打印的敏感信息（如密码、密钥、Token）？
