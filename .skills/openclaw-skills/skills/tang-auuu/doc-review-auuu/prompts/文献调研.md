## 概述

对科研文献调研文档进行系统性评估。不仅检查文字质量，更核心的是评估调研是否达到了“支撑创新点”的目的，识别调研中的逻辑断层、关键文献遗漏以及技术对比的表面化问题，助力产出高水平学术成果（如 IEEE/ACM 顶刊级别）。

# 审查提示词：文献调研（Literature Review）

**适用文件夹名**：`文献调研`  
**核心价值**：寻根溯源、发现痛点与边界。

## Agent 执行说明

执行 Phase B 时，将本节 **审查维度表** 作为该类文档的 **唯一** 内容审查 checklist：对每一行维度逐条对照原文给出结论，写入审查报告「3. 内容审查结论」；问题写入「4. 分级问题清单」并按主 Skill 的严重度规则 @ 作者。**规则追溯**：本 Skill 的 AI 链接匹配逻辑需严格执行主 SOP 附录中的 URL 关键词清单。

## 审查维度（Agent 自检），按六大维度审查

### 维度一：学术诚信与 AI 痕迹（权重最高）

**检查项：**
- [ ] 是否存在未经引用的长段落描述（潜在剽窃风险）
- [ ] AI 对话链接检测：检查文中是否遗留 AI 分享链接（参考附录规则）
- [ ] 幻觉检查：若 Agent 具备联网检索工具，请随机抽样 2-3 篇文献校验真实性；若不具备，请仅对格式异常或逻辑不通的文献标注“疑似虚假，建议人工核实”。
- [ ] **AI 生成感排查**：是否存在大量“In conclusion”、“Moreover”、“It is worth noting”等空洞的 AI 常用连接词且无实质技术含量
- [ ] 敏感数据：是否错误地将未公开的内部实验数据作为文献证据引用

---

### 维度二：文献覆盖度与时效性（SOTA 检查）

**检查项：**
- [ ] **近三年文献占比**：是否包含 2024-2026 年的最新成果（针对快速发展的 LLM/VLM 领域）
- [ ] **顶刊/顶会覆盖**：是否包含领域内公认的关键文献（如 IEEE TII, TNNLS, Mechanical Systems and Signal Processing 等）
- [ ] 经典文献遗漏：该领域的奠基性工作是否被提及
- [ ] 地域/机构多样性：引用是否过于集中在单一课题组（评价客观性风险）

---

### 维度三：技术分类逻辑（Taxonomy）

**检查项：**
- [ ] **分类体系合理性**：是按时间罗列（流水账），还是按技术架构（如：数据驱动 vs 物理启发）分类？
- [ ] 维度一致性：同级分类的划分标准是否统一
- [ ] 覆盖完整性：分类是否涵盖了当前主流的所有技术路径

---

### 维度四：技术深度与对比分析

**检查项：**
- [ ] **优缺点评述**：是否深入分析了每类方法的限制（例如：计算开销、对标签数据的依赖、开放集下的表现）
- [ ] 对比维度：是否提供了多维度的对比（准确率、泛化性、推理速度、物理可解释性）
- [ ] **技术演进逻辑**：是否阐述了技术是如何从 A 演进到 B 的，中间解决了什么关键矛盾

---

### 维度五：研究缺口（Gap）识别

**检查项：**
- [ ] **痛点指向性**：调研得出的“现有技术不足”是否能自然推导出本文的“研究动机”
- [ ] 论证力度：对现有缺陷的描述是否有文献支撑，而非主观臆断
- [ ] 差异化分析：本文方案与最相似的 SOTA 工作之间的本质区别是否清晰

---

### 维度六：引用格式与学术规范

**检查项：**
- [ ] 引用一致性：正文引用编号（如 [1]）与末尾参考文献列表是否一一对应
- [ ] 格式统一性：作者名、期刊名缩写、DOI、年份格式是否统一（建议遵循 IEEE 格式）
- [ ] **绝对化用语排查**：严查“The best”、“Perfectly solved”、“Never fail”等非学术词汇

### 生成报告

按照下方【报告模板】输出 Markdown 格式报告，**创建为飞书云文档**。

**飞书操作**：
1. 使用 `createDocument` API 创建新文档，标题为 `review_<文档短标题>_<YYYYMMDD>`
2. 将报告内容（Markdown 格式）写入文档
3. 将文档移动到飞书知识库的 `审查报告/文献调研/` 文件夹
4. 将新创建的飞书文档链接记录到汇总表中
