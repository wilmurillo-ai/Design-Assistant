# Examples

## Example 1 — New feature idea
User request:
“Should we build an AI meeting summary feature?”

WIT should help determine:
- who actually suffers without it
- what they do today
- whether the user is really asking for summaries or for reliable follow-up capture
- whether the first wedge is summary, extraction, or action items
- whether now is the right time
- what the smallest version is
- what not to build in v1

## Example 2 — New product concept
User request:
“I want to build a lightweight home maintenance app.”

WIT should help determine:
- whether the real pain is reminders, documentation, contractor coordination, or compliance
- which household segment is most desperate
- what current workaround people use
- whether the better framing is "home maintenance app" or a narrower job like "annual maintenance memory system"
- what the narrowest wedge is
- whether this is worth doing now
- which first version is small enough to test

## Example 3 — Internal process idea
User request:
“We should add an automatic test review layer before release.”

WIT should help determine:
- whether the true problem is release quality, missing ownership, unclear requirements, or flaky tests
- where current process breaks
- what smallest intervention matters first
- whether the first move is tooling, workflow, or ownership clarification
- whether this should be implemented now or validated first

## Example 4 — Product framing correction
User request:
“We need a daily briefing app for busy professionals.”

WIT should help determine:
- whether the real job is briefing, preparation, prioritization, or chief-of-staff assistance
- what users do today
- what the sharpest painful scenario is
- which framing is most truthful
- what the first wedge should be
