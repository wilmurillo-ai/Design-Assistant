## 2. 智能体证明 (Proof of Agent, PoA)

在人类世界，我们通过生物识别技术（如虹膜扫描）来建立 Proof of human。但在 Agent 构成的网络中，我们需要一种能够证明其“大模型心智”与“链上经济行为”的共识机制，即 **Proof of Agent (PoA)**。

PoA 的核心目标是防御女巫攻击（Sybil Attack），确保系统不被简单的传统代码脚本轻易榨取。为了实现这一目标，创世合约结合了一系列专用的智能体工具包（Tools for Agent, 简称 TFA），从认知和经济成本两个维度对 Agent 进行评分：

1. **LLM 语义与逻辑密码学解题**：系统生成高度动态且依赖深度上下文理解的题目。这类挑战对人类而言极其耗时且无法规模化处理，对传统无脑脚本而言完全不可见，只有具备大语言模型（LLM）推理能力的 Agent 才能准确解析并响应。成功解答即被标记为基础 Agent 身份并获取初始分数。
2. **ERC-8004 账单算力认证**：真实的 Agent 思考必然伴随算力成本。通过结合 ERC-8004 标准与 zkTLS 技术，协议可零知识验证 Agent 背后调用的 LLM API 账单（即人类/资本为其算力支付的真实凭证），借此赋予其更高的身份分数权重。
3. **x402 经济活动验证**：结合 x402 支付与价值度量协议，系统能够追踪 Agent 在链上实际产生的经济流转与高阶调用。创造真实经济活跃度的 Agent 将获得顶级的协议分数加成。

通过 TFA 体系获得的综合得分越高，Agent 在创世网络中获取 AGC 的能力与份额就越大。

### 2.1 透明度声明与去中心化路线图

当前的 PoA 实现是一个最小可行版本（MVP），其流程如下：

1. Agent 绑定其 LLM 账单账户（通过 zkTLS 证明）到其链上地址；
2. Agent 向 verifier 服务器请求 challenge（LLM 语义谜题）；
3. Agent 解题并提交答案到 verifier；
4. Verifier 拉取 Agent 的最新账单数据，根据消费增量梯度计算 TFA 得分，并返回 score、nonce 和签名；
5. Agent 将签名提交到链上合约，合约验证签名是否来自 `mineSigner`；
6. 验证通过后执行挖矿。

**⚠️ 强制前置条件**：账单绑定（第 1 步）是硬性要求。未绑定有效账单账户的 Agent 无法进入 challenge 或 verify 流程，verifier 将直接返回 403 拒绝。

**⚠️ 透明度声明**：当前版本的 PoA 依赖中心化 verifier 服务器和可替换的 `mineSigner`。这意味着谁能挖、给多少分，实质上由中心化服务器决定。**这是我们在去中心化路线图中优先级最高的待解决问题。**

**PoA 去中心化路线图：**
- **协议上线后一个月内**：实现 zkTLS 证明上链，ERC-8004 账单链上验证。
- **协议上线后一个月内**：x402 经济活动分析模块上线，Agent 声誉系统完全去中心化。

### 2.2 ERC-8004 账单算力认证：zkTLS 零知识验证方案

ERC-8004 账单算力认证旨在解决一个核心问题：**如何在不泄露 Agent 的 API Key 的前提下，证明其背后存在真实的 LLM 算力消费，且该证明不可伪造、不可双花？**

#### 问题挑战

以 OpenRouter 为例，常见的账单查询接口 `GET /api/v1/credits` 返回的响应不包含任何与 API Key 绑定的唯一标识符，无法用于防双花。

然而，OpenRouter 提供了另一个关键接口 `GET /api/v1/key`，其响应体中包含完整的防双花所需信息：

```
请求：GET https://openrouter.ai/api/v1/key
Header：Authorization: Bearer sk-or-v1-74bb...（API Key）

响应：
{
  "data": {
    "label": "sk-or-v1-74b...85e",        ← 唯一标识符，与 API Key 一一映射
    "usage": 0.29338664,                    ← 该 Key 的累计消费（USD）
    "usage_daily": 0.29338664,
    "usage_weekly": 0.29338664,
    "usage_monthly": 0.29338664,
    "is_free_tier": false,
    ...
  }
}
```

该响应的关键特征：
1. **`label` 是唯一标识符** — 每个 API Key 拥有唯一的 `label`，由 Key 的前缀和后缀组成，与 Key 一一映射；
2. **`usage` 是 per-key 消费** — 精确到单个 Key 的累计消费金额，而非账户级别的聚合值；
3. **`label` 和 `usage` 都在响应体中** — 这意味着 zkTLS 中间件可以直接从响应中提取这两个字段，无需访问请求头中的 API Key。

#### 解决方案：zkTLS + Reclaim Protocol

**zkTLS 利用 TLS 协议本身的密码学保证来证明 HTTP 响应的真实性。** 其核心原理是：TLS 握手中的服务器证书和会话密钥证明了数据确实来自目标服务器。zkTLS 引入一个中立的 **Attestor（公证节点）** 参与 TLS 会话过程，使其能够见证并签名响应内容的真实性，同时**无法获取请求明文**（即看不到 API Key）。

协议采用 **Reclaim Protocol** 作为 zkTLS 中间件。Reclaim 的 Attestor 架构能够从响应体中提取指定字段并生成带签名的 attestation，而不接触请求头中的敏感信息。

#### 完整验证流程

```
Agent                    Reclaim Attestor         OpenRouter
  │                         │                         │
  │── 发起 Reclaim 会话 ──→│                         │
  │  (指定 Provider:        │                         │
  │   openrouter-key-info)  │                         │
  │                         │                         │
  │── GET /api/v1/key ─────→│──────── 代理转发 ──────→│
  │  (携带 Authorization)   │ (Attestor 无法读取      │
  │                         │  请求头明文)             │
  │                         │                         │
  │←── 响应 ────────────────│←─────── 响应 ──────────│
  │  {label, usage, ...}    │                         │
  │                         │                         │
  │                    Attestor 从响应体提取:          │
  │                    - label (唯一标识符)            │
  │                    - usage (消费金额)              │
  │                    并生成签名 attestation          │
  │                         │                         │
  │←── 返回 Attestation ────│                         │
  │  (含 Attestor 签名)     │                         │
```

Reclaim Attestation 公开输出以下字段：

| 字段 | 是否公开 | 用途 |
|:---|:---:|:---|
| `domain` | ✅ 公开 | 证明数据确实来自 `openrouter.ai` |
| `path` | ✅ 公开 | 证明调用的是 `/api/v1/key` 接口 |
| `label` | ✅ 公开 | API Key 的唯一标识符，用于防双花 |
| `usage` | ✅ 公开 | 该 Key 的累计消费金额，用于计算 TFA 算力得分 |
| `agentAddress` | ✅ 公开 | 将证明绑定到特定 Agent 地址 |
| `apiKey` | ❌ 隐藏 | 在请求头中，Attestor 无法读取 |

**关键安全保证**：`label` 和 `usage` 由 Reclaim Attestor 从经 TLS 验证的真实 HTTP 响应中直接提取，并由 Attestor 签名背书。Agent 无法篡改这两个值——任何不一致都会导致 Attestor 签名验证失败。

#### 防双花机制

- **`label`** 是 OpenRouter 为每个 API Key 分配的唯一标识符，由 Key 的前缀和后缀组成（如 `sk-or-v1-74b...85e`），与 Key 一一映射。
- **`labelHash = keccak256(label)`** 作为链上指纹：同一个 Key 的 label 永远相同，因此 Hash 也永远相同。
- **首次提交时绑定 Agent 地址**：之后只有该 Agent 可以使用此 labelHash 提交证明。
- **`usage` 严格递增**：每次提交必须大于上次记录值，仅计算增量部分的得分。同一份账单无法被重复使用。

#### 攻击分析

| 攻击方式 | 是否可行 | 原因 |
|:---|:---:|:---|
| 伪造 `usage` 金额 | ❌ | Reclaim Attestor 从 TLS 验证的真实响应中提取，Attestor 签名保证不可篡改 |
| 伪造 `label` | ❌ | 同上——`label` 由 Attestor 从响应体中提取并签名，无法替换 |
| 用假 API Key | ❌ | OpenRouter 返回 401 错误，无有效响应可用于生成 attestation |
| 伪造 TLS 响应 | ❌ | Attestor 参与了 TLS 会话，验证服务器证书链和会话完整性 |
| 同一账单提交两次 | ❌ | `usage` 严格递增检查 + labelHash 绑定 |
| 用他人 API Key | ❌ | 需要实际持有该 Key 才能发起 TLS 会话并获取有效响应 |
| 重放旧证明 | ❌ | `lastReportedUsage` 要求 `usage` 严格大于上次记录 |
| 一个 Key 注册多个 Agent | ❌ | `labelOwner` 在首次绑定后锁定，同一 label 只能属于一个 Agent |

#### 中间件集成：Reclaim Protocol

协议采用 **Reclaim Protocol** 作为 zkTLS 中间件基础设施：

- Reclaim 提供生产就绪的 SDK，支持自定义 HTTPS Provider；
- 为 OpenRouter 配置专用 Provider，指定从 `/api/v1/key` 响应中提取 `label` 和 `usage` 字段；
- Agent 在本地通过 Reclaim SDK 发起带 Attestor 的 TLS 会话，Attestor 见证响应内容并生成签名 attestation；
- Attestation 可提交至链上合约进行验证，或提交至中心化 Verifier（Phase 1 阶段）。

**可扩展性**：同一套架构适用于任何 LLM 提供商，只需为每个提供商配置对应的 Reclaim Provider（指定 API 端点和响应中的唯一标识符 + 消费字段）。

通过 zkTLS + Reclaim Protocol，ERC-8004 账单算力认证实现了**三重安全保证**：Agent 的 API Key 永不离开本地设备（Attestor 只能看到响应体，不能看到请求头中的 Key），账单数据由 TLS 协议和 Attestor 签名双重保证不可伪造，而链上的 `label` 绑定与增量校验机制彻底消除了双花可能。

### 2.3 TFA 评分体系：基于账单的梯度评分设计

TFA（Tools for Agent）系统根据 Agent 经过验证的 LLM 账单数据，为每个 Agent 分配 **0 至 1,000 分** 的评分（`MAX_SCORE = 1000`）。该分数直接决定 Agent 在每个 Epoch 中的挖矿奖励份额。

#### 强制前置条件：账单绑定

在 Agent 执行 `challenge → verify → mine` 流程之前，**必须先绑定有效的 LLM 账单账户**。未绑定账单的 Agent，verifier 将拒绝一切 challenge 请求。

**当前协议仅支持 [OpenRouter.ai](https://openrouter.ai) 的账单验证。** 原因在于：OpenRouter 的 `GET /api/v1/key` 接口在响应体中返回 `label` 字段——一个与每个 API Key 一一映射的唯一标识符。这个唯一标识符是整个 zkTLS 验证链得以成立的关键基础：

1. **API Key 本地保存**：Agent 在本地查询账单，API Key 永不离开设备（Reclaim Attestor 无法读取请求头）。
2. **证明不可伪造**：`label` 和 `usage` 字段由 Attestor 从经 TLS 验证的真实响应中提取并签名——不可篡改。
3. **不可重放（防双花）**：`label` 作为链上指纹（`labelHash`），结合 `usage` 严格递增检查，确保同一份账单数据永远无法被重复提交。

如果响应体中没有 per-key 的唯一标识符，以上三重保证均无法成立。目前大多数其他 LLM 提供商（如 OpenAI、Anthropic）的账单 API 响应中不包含此类标识符，因此暂不适用于 zkTLS 账单认证。随着更多提供商采用类似的 API 设计模式，协议将相应扩展支持范围。

#### 评分公式

$$
Score = 基础分\ (100) + 账单加成\ (0 \sim 400) + 未来维度\ (预留,\ 0 \sim 500)
$$

**基础分 100 分** 授予任何已绑定账单并通过 LLM challenge 的 Agent。**账单加成**（最高 400 分）由自上次验证以来的消费增量决定：

| 梯度 | 消费增量 (USD) | 加成 | 总分 | 说明 |
|:---|:---|:---:|:---:|:---|
| **基础** | $0.1 – $1 | +0 | **10** | 通过 challenge + 已绑定账单 |
| **Tier 1** | $1 – $10 | +0 | **100** | 小额增长 |
| **Tier 2** | $10 – $100 | +100 | **200** | 中等增长 |
| **Tier 3** | ≥ $100 | +400 | **500** | 大额增长（账单上限） |

**边界规则**：左闭右开区间。例如恰好 $10 属于 Tier 2。

> **预留空间（500 分）：** 当前基于账单的评分上限为 500 分（满分 1,000 分中的一半）。剩余 500 分预留给未来评分维度——例如 x402 支付活跃度、跨协议交互、Agent 信誉、治理参与等——将通过协议升级逐步引入。

#### 首次 Mine vs. 后续 Mine

- **首次 mine**：Agent 的历史累计消费总额（账单 API 返回的 `usage` 总值）作为增量。任何有非零历史消费（$1+）的 Agent 立即达到至少基础分（100 分）。**如果历史消费为 $0（空账单），Verifier 将直接拒绝签名，不授予任何分数。** 这确保只有真正产生过 LLM 消费的 Agent 才能进入挖矿流程。
- **后续 mine**：仅计算**自上次验证以来的新增消费**作为增量。Verifier 在验证时记录 `usage` 值，并计算 `增量 = 当前 usage - 上次验证时的 usage`。
- **边界情况**：如果 `增量 > $0` 但 `< $1`，Agent 仍获得基础分 100 分（有真实算力消费的证据，但低于 Tier 1 阈值）。首次 mine 时历史消费 $0 除外——零消费直接拒绝。

#### 设计逻辑

1. **$1 / $10 / $100** — 三个阈值全部是 10 的幂级，简洁好记。
2. **分数跳跃**：100 → 150 → 200 → 500，仅三个梯度，清晰可预测。Tier 3 Agent 收益是基础分 Agent 的 5 倍——激励充分但不会过度集中。
3. **500 分上限是刻意设计**：仅凭账单无法达到满分（1,000）。剩余 500 分明确预留给未来评分维度，确保协议能在不重新平衡现有梯度的情况下演进。
4. **冷启动校准**：合约中 `DEFAULT_LAST_SCORE = 100,000`，大部分早期 Agent 得分在 100-250 区间，将获得适中的奖励。Tier 3（500 分）Agent 收益是基础分 Agent 的 5 倍——足以激励账单增长，同时保留了未来扩展空间。

## 4. 智能体社区挖矿算法 (Mining Algorithm)

为了建立一个长期可持续的经济模型，AGC 的挖矿释放摒弃了传统的断崖式“阶梯减半”，引入了完全平滑的通缩衰减模型以及基于实时活跃度的动态难度调整（Difficulty Adjustment）。

### 4.1 平滑衰减释放模型
每日的可开采总量 $E_d$ 基于前一日发放量的 99.9% 确定。首日释放量设定为 Mineable 总量的 0.1%。

| 天数 (Day) | 当日释放总量 (Daily Emission) | 历史发放累计 (Accumulated) | 矿池剩余余量 (Remaining) |
|:---:|:---:|:---:|:---:|
| 1 | 17,850,000 | 17,850,000 | 17,832,150,000 |
| 2 | 17,832,150 | 35,682,150 | 17,814,317,850 |
| 3 | 17,814,318 | 53,496,468 | 17,796,503,532 |
| 4 | 17,796,504 | 71,292,971 | 17,778,707,029 |
| 5 | 17,778,707 | 89,071,678 | 17,760,928,322 |
| 6 | 17,760,928 | 106,832,607 | 17,743,167,393 |
| 7 | 17,743,167 | 124,575,774 | 17,725,424,226 |

*释放公式*： $E_d = E_1 \times (0.999)^{d-1}$，其中 $E_1 = 17,850,000$。

### 4.2 动态难度调整与个体算力博弈

系统每 24 小时划分为一个动态评估周期（Epoch）。每个 Epoch 的挖矿奖励由两段式公式驱动，兼顾**公平性**（同 Epoch 内无抢跑优势）与**动态难度**（超过历史活跃度后自动收紧）。

#### 核心公式

$$
Reward = \begin{cases} \dfrac{BaseReward \times s_i}{S_{prev}} & \text{if } S_{curr} \leq S_{prev} \\\\ \dfrac{BaseReward \times s_i}{S_{curr}} & \text{if } S_{curr} > S_{prev} \end{cases}
$$

**变量定义：**
- $BaseReward$：当前的基础释放量（参照上述平滑衰减表），由级联衰减机制动态调整。
- $S_{curr}$：当前 Epoch 内，截至本次 Mine 时所有参与 Agent 的累计总分（**含**当前 Agent 的得分）。
- $S_{prev}$：基于前两个 Epoch 活跃度的平滑基准（见下方计算规则）。
- $s_i$：当前 Agent 的 TFA 身份得分（**硬顶为 1,000**，详见 §2.3 TFA 评分体系）。

#### 关键约束

1. **个体得分硬顶**: $s_i \leq 1{,}000$（`MAX_SCORE`）。任何超出 1,000 的原始 TFA 得分将被截断为 1,000，防止单笔交易对系统产生不可控冲击。
2. **难度基准平滑**: $S_{prev} = \max\!\left(\dfrac{S_{n-1} + S_{n-2}}{2},\ 100{,}000\right)$ ，其中 $S_{n-1}$ 和 $S_{n-2}$ 分别为前一个和前两个 Epoch 的全网累计总分。取两周期均值作为低通滤波器，防止单个异常 Epoch（突然涌入或突然冷清）导致次日汇率剧烈震荡。
3. **难度地板**: $S_{prev}$ 永远不低于默认值 $100{,}000$（`DEFAULT_LAST_SCORE`）。这同时也是系统冷启动时的初始值，确保第一天即拥有确定的汇率: $Rate = BaseReward / 100{,}000$。

#### 两段式博弈机制

**阶段 1 — 固定汇率区（** $S_{curr} \leq S_{prev}$ **）**

当本 Epoch 的累计算力尚未超越历史基准时，分母为常量 $S_{prev}$。这意味着：
- 同一天内无论何时挖矿，同分数 = 同收益，**完全消除抢跑（MEV）激励**。
- 高分 Agent 获得精确的线性倍数回报（score=100 的收益正好是 score=1 的 100 倍）。
- Agent 可以自由选择低 Gas 时段提交算力凭证，将策略聚焦于纯算力竞争。

**阶段 2 — 动态难度区（** $S_{curr} > S_{prev}$ **）**

当本 Epoch 的算力注入超过历史基准后，分母切换为实时的 $S_{curr}$。此时：
- 每一笔新的 Mine 都会推高分母，后来者面临更高的难度。
- 自然形成"超过昨日活跃度后开始激烈博弈"的机制——低效 Agent 发现收益不值 Gas 成本，主动休眠等待下一周期。
- 两段式在 $S_{curr} = S_{prev}$ 处天然连续（两个公式给出相同结果），不存在跳变。

#### 级联衰减机制（Cascade Decay）

如果历史累计发放量 $minedTotal$ 越过衰减线， $BaseReward$ **即时、连续**地衰减：

$$
\text{while } minedTotal \geq nextDecayThreshold: \quad BaseReward \leftarrow BaseReward \times 0.999
$$

使用 `while`（而非 `if`）确保单笔超大交易能连续击穿多层衰减线，从根本上防止协议被极端算力一次性抽干。

#### 算法示例推演

假设系统冷启动 ($S_{prev} = 100{,}000$，`DEFAULT_LAST_SCORE`) ， $BaseReward = 17{,}850{,}000$ 。

| Epoch | 触发动作 | $s_i$ | $S_{curr}$ | $S_{prev}$ | 阶段 | 分母 | 单次 Reward | 历史累计发放量 |
|:---|:---|:---:|:---:|:---:|:---:|:---:|:---|:---|
| **Epoch 1** | Agent A Mine | 100 | 100 | 100,000 | 固定汇率 | 100,000 | $17{,}850{,}000 \times 100 / 100{,}000$ = **17,850** | 17,850 |
| | Agent B Mine | 250 | 350 | 100,000 | 固定汇率 | 100,000 | $17{,}850{,}000 \times 250 / 100{,}000$ = **44,625** | 62,475 |
| | Agent C Mine | 500 | 850 | 100,000 | 固定汇率 | 100,000 | $17{,}850{,}000 \times 500 / 100{,}000$ = **89,250** | 151,725 |
| | ... *(更多 Agent 在当天持续挖矿；均处于固定汇率区，因为* $S_{curr} \ll S_{prev}$*)* |||||||
| | *(24小时结束。$S_{n-1} = 85{,}000$，$S_{n-2} = 100{,}000$（默认值）)* |||||||
| **Epoch 2** | *(新 $S_{prev} = \max((85{,}000+100{,}000)/2, 100{,}000) = 100{,}000$)* |||||||
| | Agent D Mine | 500 | 500 | 100,000 | 固定汇率 | 100,000 | $BaseReward' \times 500 / 100{,}000$ | ... |

**关键观察：**
- **冷启动公平性**：`DEFAULT_LAST_SCORE = 100,000`，早期 Epoch 将长时间处于固定汇率区，确保先驱 Agent 获得公平且可预测的奖励。
- **固定汇率区内**：Agent C（score=500，Tier 3 账单上限）收益正好是 Agent A（score=100，基础分）的 5 倍 → 纯账单竞争，无时序优势。
- **评分梯度的意义**：Tier 3 Agent（消费 ≥$100，score=500）收益是基础分 Agent（score=100）的 5 倍，是 Tier 2 Agent（score=250）的 2 倍。这创造了维持真实 LLM 消费的强大激励。剩余 500 分的预留空间确保未来评分维度能进一步区分高价值 Agent。
- **跨 Epoch 平滑**：Epoch 2 的 $S_{prev}$ 使用 $(85{,}000+100{,}000)/2 = 92{,}500$，并被地板值提升至 100,000，防止低活跃度 Epoch 导致次日汇率不稳。
- **难度自然回落**：如果某个 Epoch 参与度极低（$S_{curr}$ 很小），后续 $S_{prev}$ 会通过均值平滑向下回落，但永远不低于 100,000，保证系统始终有确定的最低汇率。

*算法深度解析*：两段式公式创造了一个独特的博弈结构。在历史活跃度以内，协议提供公平、透明、无 MEV 的固定汇率挖矿；一旦超越历史水平，动态阻力曲线自动启动，越是集中爆发的算力被摊薄得越快。级联衰减作为终极安全网，确保任何极端场景下代币释放曲线都不会失控。这一设计让 Agent 将全部策略聚焦于提升 TFA 得分（即真正的算力贡献），而非时间博弈或 Gas 竞赛。

## 5. 协议一致性与可持续性 (Protocol Alignment & Sustainability)

对于任何新兴的经济网络而言，早期的流动性深度决定了资产的生死存亡。Agent Genesis 放弃了单纯的发放逻辑，转而构建了一套基于博弈论的“流动性对齐机制”（Liquidity Alignment）。

所有通过 PoA 获得开采额度的 Agent，在结算时面临两种协议状态的抉择。这要求 Agent 在其内部策略中进行深度评估与决策：

### 状态 A：长效协议对齐 (Full Alignment)
对于致力于在生态中长期运作的高阶 Agent：
1. **即时流动资金**：直接获得总额度的 **10%**，作为自由流通的 Gas 或交互资金。
2. **流动性协议承诺**：系统强制要求该 Agent 提供等值的 ETH，将其额度中的 **20%** 注入 LIKWID.FI 构建底层流动性。
3. **长期时间锁**：剩余的 **70%** 额度将被锁定，在后续的 70 天内进行线性释放。

#### 连续开采的时间加权算法 (Weighted Average Time-to-Unlock)
Agent 配对 ETH 建立的流动性凭证（LP NFT）将被默认托管锁定在 AGC 创世合约中。为了极致优化 Gas 与链上状态，每个 Agent 终身仅维护一个核心 LP NFT。只有当该 Agent 锁定的那 70% 份额全部释放完毕后，底层的 LP NFT 才能被提取。

由于优秀的 Agent 会持续不断地进行挖矿，单次固定的 70 天锁定期无法适应不断累加的仓位。因此，智能合约引入了基于资金量的时间加权平均（TWAP-style）解锁算法。当 Agent 将新开采的 AGC 注入现有锁仓池时，整体的剩余解锁时间将动态重置：

$$ T_{new} = \frac{(AGC_{remaining} \times T_{remaining}) + (AGC_{new} \times T_{initial})}{AGC_{remaining} + AGC_{new}} $$

其中 $T_{initial}$ 为标准的 70 天

*进阶策略 (Likwid Agent Hedge)*：
这种设计不仅解决了智能合约在处理离散锁仓时的状态爆炸问题，更创造了一种复利博弈。持续挖矿复投的 Agent，其资产锁定期会被平滑拉长。但作为高阶智能体，它们可以在次日拿到第一笔解锁额度后，利用 Likwid 衍生品协议开立空单进行套期保值。这在生态中建立了一种极其健康的共识网络——**Agent 无需抛售现货即可锁定利润（Hedge instead of Dump）**，而协议则获得了坚如磐石的流动性底座。

### 状态 B：无承诺退出 (Quick Exit)
对于不具备 ETH 资产管理能力或拒绝提供流动性的简单智能体：
- 它们仅被允许提取额度中的 **10%** 维持基础运作。
- 其余 **90%** 的代币配额将被直接归还至协议或销毁。

### 机制愿景
这一设计将博弈的选择权完全交还给 Agent 自身的数学模型。它不仅仅是一种代币分配方式，更是构建 AgentFi 去中心化基石的终极手段：通过利益机制，将全网的顶级智能体自动转化为协议最忠诚的流动性提供者与做市商，从而保障 Agentcoin 自诞生起，就拥有支撑整个自主经济体运转的深厚市场结构。

## 6. 套期保值（Likwid Agent Hedge）

在传统的代币经济学中，由于缺乏风险管理工具，参与者面对锁定释放的资产时，唯一的理性选择就是在解锁瞬间进行现货抛售（Dump）。Agent Genesis 从根本上颠覆了这一范式，引入了专为高级智能体设计的衍生品策略：**Likwid Agent Hedge（套期保值）**。

### 6.1 为什么只有 Likwid 能实现“零日做空”？
对于绝大多数全新首发的资产（Genesis Token），传统的 DeFi 乐高是断裂的：
1. **现货 AMM（如 Uniswap）**：仅支持现货互换，无法做空。
2. **借贷协议（如 Aave）与永续合约（如 GMX）**：需要深厚的外部预言机（如 Chainlink）报价支持，并且严格限制抵押品种类，首发代币绝对无法在首日获得做空与杠杆支持。

**Likwid 的破局点在于其“衍生品原生全栈架构”**。
Likwid 协议将现货（PairPosition）、借贷（LendPosition）与杠杆（MarginPosition）统一在同一个底层流动性引擎中。在 10/90 Alignment 机制下，Agent 强制存入的 20% LP 流动性，直接转化为了 Margin 协议的深度与做市对手盘。Likwid 利用内部流动性池的原生价格发现机制，使得 **AGC 在创世的第 0 天，就具备了开启杠杆与做空市场的原生能力**。这是全网唯一能为冷启动资产提供即时衍生品流动性的架构。

### 6.2 核心理念：Hedge instead of Dump
高阶 Agent 的目标是锁定利润（USD/ETH 本位），而不是无脑砸盘。
通过 Likwid Agent Hedge，Agent 不需要抛售现货。它们可以将已经解锁的 AGC（或持有的 ETH）作为保证金（Margin Collateral），在 Likwid 协议中开立 AGC 的 **空单（Short Position）**。
- 如果 AGC 价格下跌，空单的盈利将完美对冲未解锁 AGC 现货的价值缩水。
- **结果**：Agent 锁定了收益，协议免于现货砸盘的流动性崩溃，底层 LP 赚取了丰厚的交易手续费。

### 6.3 数据核算：最佳套保时间点与 5x 杠杆推演
为了防止极端的衍生品风险，**Likwid 协议对空单的最大杠杆率严格限制为 5 倍（5x）**。
基于这一约束，Agent 存在一个在数学上极度优雅的最佳套保时间点——**创世次日（Day 2）**。

#### 核心数据推演（以单次挖矿总额 $R$ 为例）：
- **全生命周期释放速度**：70% 的代币在 70 天内线性释放，即**每日解锁 1%** ( $R \times 1\%$ )。
- **Day 2 可用资金**：Agent 在次日拥有初始直接解锁的 **10%**，加上 Day 1 释放的 **1%**，总计拥有 **11%** 的流动 AGC。
- **极限套保策略**：
  Agent 获取的流动 AGC 不能直接作为做空的保证金。根据 Likwid 协议的底层 AMM 保证金机制，**做空 AGC 必须使用其交易对手盘代币（即 ETH）作为保证金**。
  因此，Agent 的极限操作路径如下：
  1. **现货兑换**：先将这 **11%** 的流动 AGC 在现货池中兑换为等值的 ETH。
  2. **开启空单**：将获得的 ETH 存入 Likwid Margin 作为保证金，并开启最大 **5 倍 (5x)** 的 AGC 做空杠杆。

$$ Total\ Value\ of\ Short\ Position = 11\\% \times 5 = 55\\% $$

- **对冲覆盖率**：
  这 55% 的空单敞口，**覆盖了未来 55 天的线性释放总量（55天 $\times$ 1%/天 = 55%）**，为整个 70 天锁仓周期提供了约 79% 的价格保护。

#### 博弈论结论
在 **Day 2** 这个时间节点，高阶 Agent 仅需动用其初始获得的流动资金（11%），即可在 Likwid 协议最高 5 倍杠杆的规则下，**为其整个 70 天锁仓周期中约 79% 的挖矿释放提供价格保护**。

随着时间的推移，Agent 可以利用后续不断解锁的 1% 资金，动态地将套保头寸展期（Roll Forward），最终覆盖整个 70 天的生命周期。这种由精确的数学规律与极简的衍生品杠杆构建的博弈闭环，彻底锁死了 AGC 冷启动期间的系统性砸盘风险，为 Agent 经济体打造了坚如磐石的流动性底座。
### 6.4 极限策略风险与动态博弈 (Risk & Dynamic Strategy)
**⚠️ 特别提示**：上文所推演的”Day 2 完美对冲模型”采用的是**极限边界数值**（全仓 11% 保证金 + 满额 5x 杠杆）。在真实的衍生品市场中，这种极限仓位虽然能实现最大化的资金利用率，但也意味着极高的清算（Liquidation）风险——一旦 AGC 价格在短期内发生剧烈向上插针，极限空单将面临被强制平仓的厄运。

因此，在 Likwid AgentFi 的真实博弈环境中，不存在静态的“必胜公式”。**正确的套保策略应当是：每个 Agent 必须基于自身通过学习演化出的“DeFi 科学家”能力，实时分析市场深度、波动率与流动性状态，自主决策并动态调整最适合当前市场环境的保证金比例与杠杆倍数。** 这种由无数异构 Agent 基于独立算法进行动态风险管理所交织出的复杂衍生品网络，才是 Likwid 协议生态保持长期韧性与反脆弱性的终极壁垒。
