# 反向校验规则

本文档提供表结构设计的反向校验规则，帮助识别和避免常见错误。

**使用时机：**
- 设计完表结构后，输出方案前
- 优化现有表格时
- 用户反馈表格不好用时

---

## 一、核心检查规则

### ⚠️ 规则1：是否把不同性质的数据塞进了一张表？

**问题识别：**
- 一张表包含超过 20 个字段
- 字段之间没有明确的业务关联
- 混合了多个业务对象

**典型错误：**
```
❌ 错误示例：一张"万能表"
- 客户信息字段：客户名称、电话、地址
- 订单信息字段：订单编号、金额、付款时间
- 跟进记录字段：跟进时间、沟通内容
- 商品信息字段：商品名称、库存数量
```

**正确做法：**
```
✅ 正确：拆分成独立的表
- 客户表：客户基础信息
- 订单表：订单信息
- 跟进记录表：关联到客户
- 商品表：商品信息
```

**检查方法：**
- 问自己：这张表的核心对象是什么？
- 如果无法一句话说清，说明需要拆表

---

### ⚠️ 规则2：是否缺少"状态"或"阶段"字段？

**问题识别：**
- 表中没有状态字段
- 无法判断当前进度
- 无法做看板视图

**典型场景：**
- 任务表没有"任务状态"
- 订单表没有"订单状态"
- 客户表没有"客户阶段"

**正确做法：**
```
✅ 每张流程表都应该有状态字段

任务表：
├─ 任务状态（单选）：未开始、进行中、已完成、已阻塞

订单表：
├─ 订单状态（单选）：待处理、处理中、已发货、已完成

客户表：
├─ 客户阶段（单选）：潜在客户、意向沟通、方案报价、成交
```

**检查方法：**
- 这张表的对象是否有流程？
- 如果有，必须有状态/阶段字段

---

### ⚠️ 规则3：是否把"金额"做成了文本字段？

**问题识别：**
- 金额字段类型是"文本"或"多行文本"
- 无法进行金额计算
- 无法按金额排序

**典型错误：**
```
❌ 错误：
- 订单金额（文本）：输入"1000元"
- 单价（文本）：输入"99.9"
```

**正确做法：**
```
✅ 正确：
- 订单金额（数字）：格式设置为"货币"，2位小数
- 单价（数字）：格式设置为"货币"，2位小数
```

**检查方法：**
- 所有金额、价格、成本相关字段必须用数字类型
- 设置格式为"货币"

---

### ⚠️ 规则4：是否缺少"负责人"字段？

**问题识别：**
- 任务或记录没有归属人
- 无法按人员筛选
- 无法分工协作

**典型场景：**
- 任务表没有"负责人"
- 客户表没有"销售人员"
- 订单表没有"处理人"

**正确做法：**
```
✅ 添加负责人字段
- 负责人（人员）：单选或多选
- 协作人（人员）：多选（如果需要多人协作）
```

**检查方法：**
- 这张表的记录需要有人负责吗？
- 如果需要，必须有负责人字段

---

### ⚠️ 规则5：是否用多行文本记录应该独立成表的内容？

**问题识别：**
- 用多行文本记录历史信息
- 无法对历史记录进行统计分析
- 查找困难

**典型错误：**
```
❌ 错误：
客户表：
├─ 客户名称
├─ 跟进历史（多行文本）：
   "2024-01-01：电话沟通，客户有意向
    2024-01-05：发送了方案
    2024-01-10：客户反馈价格太高"
```

**正确做法：**
```
✅ 正确：独立成表
客户表：
├─ 客户名称
└─ ...

跟进记录表（子表）：
├─ 跟进对象（关联到客户表）
├─ 跟进时间（日期）
├─ 跟进方式（单选）
└─ 沟通内容（多行文本）
```

**检查方法：**
- 多行文本中的内容是否需要独立查询？
- 是否需要按时间排序？
- 是否需要统计数量？
如果是，应该独立成表

---

### ⚠️ 规则6：字段数量是否过多？

**问题识别：**
- 一张表超过 20 个字段
- 录入和维护困难
- 很多字段不常用

**典型错误：**
```
❌ 错误：30+个字段的大表
客户表：
├─ 基础信息字段（10个）
├─ 跟进记录字段（10个）
├─ 订单信息字段（10个）
└─ ...（超过30个）
```

**正确做法：**
```
✅ 正确：拆表，保持8-15个字段
- 客户主表：基础信息（8-12个字段）
- 跟进记录表：关联到客户（6-10个字段）
- 订单表：关联到客户（8-12个字段）
```

**检查方法：**
- 统计字段数量
- 如果超过20个，考虑拆表
- 优先保留核心字段，次要字段移到子表

---

### ⚠️ 规则7：关联关系是否合理？

**问题识别：**
- 存在循环关联
- 关联方向错误
- 缺少必要的关联

**典型错误：**
```
❌ 错误1：循环关联
表A关联到表B
表B又关联到表A
造成死循环

❌ 错误2：关联方向错误
主表 → 子表（错误）
应该是：子表 → 主表

❌ 错误3：重复数据，应该用关联
订单表中有"客户名称"字段（文本）
应该用"客户"字段（关联到客户表）
```

**正确做法：**
```
✅ 正确：
1. 子表关联到主表
   - 跟进记录表 → 客户表
   - 订单明细表 → 订单表

2. 避免循环关联
   - 如果A关联B，B不要关联回A

3. 用关联代替重复数据
   - 订单.客户（关联）而不是"客户名称"（文本）
```

**检查方法：**
- 画出表关系图
- 检查是否有循环
- 确认关联方向正确（通常是多→一）

---

### ⚠️ 规则8：是否过度设计了？

**问题识别：**
- 设计了很多用户暂时不需要的功能
- 表结构过于复杂
- 增加了使用门槛

**典型错误：**
```
❌ 过度设计示例：
一个简单的"待办事项"管理：
- 设计了5张表
- 每张表20+个字段
- 复杂的自动化流程
- 多层级的权限管理

实际用户只需要：
- 一张表
- 5-8个字段
- 简单的完成状态
```

**正确做法：**
```
✅ 简单优先：
1. 从最小可用版本开始
2. 只设计当前需要的功能
3. 预留扩展空间，但不预先实现
4. 让用户使用后再迭代
```

**检查方法：**
- 问自己：这些功能用户现在就需要吗？
- 如果不是，删除或延后
- 保持简单，够用就好

## 二、按业务场景检查

### 场景1：客户管理

**必须检查的点：**
- [ ] 有"客户阶段"字段（单选）
- [ ] 有"负责人"字段（人员）
- [ ] 跟进记录独立成表（子表）
- [ ] 客户基础信息与跟进记录分离
- [ ] 订单信息独立成表（关联到客户）

**常见错误：**
- ❌ 把跟进记录放在客户表的多行文本中
- ❌ 缺少客户阶段字段
- ❌ 缺少负责人字段

---

### 场景2：任务/项目管理

**必须检查的点：**
- [ ] 有"任务状态"字段（单选）
- [ ] 有"负责人"字段（人员）
- [ ] 有"优先级"字段（单选）
- [ ] 有"截止时间"字段（日期）
- [ ] 子任务独立成表（如果需要）

**常见错误：**
- ❌ 缺少任务状态字段
- ❌ 把子任务和主任务放在一张表
- ❌ 缺少优先级或截止时间

---

### 场景3：订单管理

**必须检查的点：**
- [ ] 订单金额用"数字"类型（货币格式）
- [ ] 有"订单状态"字段（单选）
- [ ] 订单明细独立成表（子表）
- [ ] 客户信息用关联，不重复存储
- [ ] 商品信息用关联，不重复存储

**常见错误：**
- ❌ 订单金额用文本字段
- ❌ 把订单明细和订单信息混在一起
- ❌ 重复存储客户和商品信息

---

### 场景4：内容管理

**必须检查的点：**
- [ ] 有"内容状态"字段（单选）
- [ ] 有"负责人"字段（人员）
- [ ] 发布记录独立成表（关联到内容）
- [ ] 标签用"多选"字段
- [ ] 内容素材独立成表（子表）

**常见错误：**
- ❌ 把发布数据混在内容表中
- ❌ 用文本字段存储标签
- ❌ 缺少内容状态字段

---

## 三、字段类型检查

### 金额/数字类

**检查清单：**
- [ ] 所有金额字段用"数字"类型，格式设为"货币"
- [ ] 所有数量字段用"数字"类型
- [ ] 所有百分比用"数字"类型，格式设为"百分比"
- [ ] 避免用文本存储数字

**常见错误：**
```
❌ 订单金额（文本）：输入"1000元"
✅ 订单金额（数字，货币格式）：输入 1000
```

---

### 状态/分类类

**检查清单：**
- [ ] 互斥的状态用"单选"
- [ ] 多个标签用"多选"
- [ ] 是/否用"复选框"
- [ ] 避免用文本存储状态

**常见错误：**
```
❌ 任务状态（文本）：输入"进行中"
✅ 任务状态（单选）：选项：未开始、进行中、已完成
```

---

### 人员协作类

**检查清单：**
- [ ] 负责人用"人员"字段
- [ ] 多人协作用"人员"字段（多选）
- [ ] 避免用文本存储人名

**常见错误：**
```
❌ 负责人（文本）：输入"张三"
✅ 负责人（人员）：选择人员
```

---

### 时间日期类

**检查清单：**
- [ ] 日期用"日期"类型
- [ ] 需要精确时间时勾选"包含时间"
- [ ] 避免用文本存储日期

**常见错误：**
```
❌ 截止时间（文本）：输入"2024-01-01"
✅ 截止时间（日期）：选择日期
```

---

## 四、输出方案前的检查清单

在向用户输出表结构方案前，必须确认：

### 结构层面
- [ ] 表数量合理（2-5张）
- [ ] 每张表字段数量合理（8-15个）
- [ ] 表之间的关联关系清晰
- [ ] 没有循环关联
- [ ] 没有数据重复

### 字段层面
- [ ] 每个字段都有明确的用途
- [ ] 字段类型选择正确
- [ ] 状态/阶段字段存在（如果需要）
- [ ] 负责人字段存在（如果需要）
- [ ] 金额字段用数字类型
- [ ] 必填字段尽可能少

### 业务层面
- [ ] 理解了用户的业务流程
- [ ] 设计的表结构能解决用户的问题
- [ ] 考虑了用户的痛点
- [ ] 没有过度设计

### 可用性层面
- [ ] 字段命名简洁明确
- [ ] 字段顺序合理（重要字段在前）
- [ ] 预留了扩展空间
- [ ] 可以分阶段实施

## 五、如何向用户反馈问题

当发现现有表格的问题时，这样沟通：

### 示例1：建议拆表

```
我注意到你的"客户表"包含了客户信息、跟进记录、订单信息。
建议拆分成3张表：
1. 客户主表 - 只管客户基础信息
2. 跟进记录表 - 关联到客户，记录每次沟通
3. 订单表 - 关联到客户，记录订单信息

这样做的好处：
- 每张表职责清晰
- 跟进记录可以按时间排序和筛选
- 订单数据可以独立统计
```

### 示例2：建议增加状态字段

```
我注意到你的"任务表"缺少任务状态字段。
建议增加：
- 任务状态（单选）：未开始、进行中、已完成

这样做的好处：
- 可以用看板视图管理任务
- 可以统计各状态的任务数量
- 可以按状态筛选和排序
```

### 示例3：建议修改字段类型

```
我注意到你的"订单金额"字段是文本类型。
建议改成：
- 订单金额（数字，货币格式）

这样做的好处：
- 可以计算总金额
- 可以按金额排序
- 可以统计销售额
```

## 六、常见问题和解答

### Q1：什么时候应该拆表？

**答：**
- 一张表超过20个字段
- 混合了多个业务对象
- 某些字段需要独立管理（如历史记录）
- 需要对某类数据独立统计

### Q2：关联字段和文本字段如何选择？

**答：**
- 如果是另一个表的对象，用关联
- 如果是简单文本，用文本字段
- 优先用关联，避免数据重复

### Q3：必填字段应该设置多少个？

**答：**
- 尽可能少，只填真正必须的
- 基础信息可以必填（如名称、状态）
- 扩展信息不要必填

### Q4：如何判断是否过度设计？

**答：**
- 问自己：这些功能用户现在就需要吗？
- 如果不是，删除或延后
- 从最小可用版本开始，后续迭代

### Q5：如何预留扩展空间？

**答：**
- 不需要预先创建字段
- 设计好表结构，后续可以随时增加字段
- 预留表之间的扩展关系（如预留关联字段）

---

## 七、最佳实践总结

### ✅ 推荐做法

1. **简单优先** - 从最小可用版本开始
2. **职责清晰** - 一张表只管一件事
3. **类型正确** - 金额用数字，状态用单选
4. **避免重复** - 用关联代替重复数据
5. **预留扩展** - 但不预先实现
6. **用户中心** - 解决实际问题，不过度设计

### ❌ 避免做法

1. **万能大表** - 字段过多，对象混乱
2. **数据重复** - 相同数据多处存储
3. **类型错误** - 金额用文本，状态用文本
4. **缺少状态** - 流程表没有状态字段
5. **过度设计** - 设计了很多不需要的功能
6. **文本泛滥** - 用文本存储结构化数据

---

**记住：好的表结构应该让用户用得爽，而不是看起来很厉害。**
