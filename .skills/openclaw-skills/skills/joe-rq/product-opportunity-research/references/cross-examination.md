# Phase 2 交叉质询协议

## 目标

暴露盲点、剔除伪需求、校准优先级。Phase 2 的核心价值在于**制造高质量冲突**——如果 6 个 agent 完全一致，说明分析不够深入。

## 每个 Agent 的挑战义务

每个 agent 至少必须提出：

| 类型 | 数量 | 说明 |
|------|------|------|
| 对其他 agent 的反对意见 | ≥ 3 条 | 必须附带理由和替代观点 |
| 被高估的机会 | ≥ 2 个 | 指出哪些机会被其他 agent 过度乐观评价 |
| 被低估的机会 | ≥ 2 个 | 指出哪些机会被忽视或低估 |

## 四类核心冲突（必须重点暴露）

### 冲突 1：需求 vs 技术
- **表现：** 用户说"必须要"，但技术说"做不到/做不好"
- **质询焦点：** 是真的做不到，还是成本太高？有无降级方案？用户能否接受 80% 的解决方案？

### 冲突 2：商业 vs 用户
- **表现：** 商业说"能卖高价"，但用户说"现有方案够用"
- **质询焦点：** 用户的"够用"是真满意还是忍受？切换成本有多高？增量价值是否值得付费迁移？

### 冲突 3：技术 vs 合规
- **表现：** 技术说"容易做"，但合规说"风险等级会升高"
- **质询焦点：** 合规风险是硬约束（法规禁止）还是软约束（行业惯例）？有无合规友好的替代技术方案？

### 冲突 4：伪通用
- **表现：** 声称"跨场景通用"，但实际只在个别场景成立
- **质询焦点：** 逐场景检验——在场景 A 成立的逻辑，在场景 B/C 是否同样成立？通用化的技术成本与收益是否对等？

## 质询规则

1. **必须基于事实** — 每条反对意见必须引用 Phase 1 的具体数据或逻辑，不接受无依据的"我觉得"
2. **必须提供替代方案** — 反对不是目的，建设性替代才是
3. **允许升级** — 如果某个冲突无法在 agent 间解决，升级至总控裁决，标注为"待裁决"
4. **记录共识与分歧** — 明确标注哪些是达成共识的、哪些是悬而未决的

## 输出格式

每个 agent 的交叉质询输出：

```
### [Agent 名称] 的交叉质询

**反对意见：**
1. 对 [Agent X]：[具体反对内容 + 理由 + 替代观点]
2. ...
3. ...

**被高估的机会：**
1. [功能名称]：[为什么被高估 + 证据]
2. [功能名称]：[为什么被高估 + 证据]

**被低估的机会：**
1. [功能名称]：[为什么被低估 + 证据]
2. [功能名称]：[为什么被低估 + 证据]
```
