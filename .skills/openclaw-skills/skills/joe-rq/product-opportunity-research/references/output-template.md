# 10 节输出格式规范

全部使用中文输出。风格要求：**务实、尖锐、可落地**。

---

## 1) 执行摘要（Executive Summary）

- **一句话结论：** 本研究主题的主战场在哪里
- **最值得做的 3 类功能**（结论先行，每类一句话理由）
- **最容易踩坑的 3 类伪机会**（每类一句话警告）
- **首发场景推荐与理由**
- **研究过程中的关键假设声明**（基于研究主题合理推断并明确声明）
- **不该做清单**（≥5 项，合并各 Agent 的"不建议做"结论）：每项标注淘汰理由的主要维度（技术不可行 / 合规风险 / 伪需求 / 传统方案已够用 / ROI 不成立），避免散落在各节中被忽视

---

## 2) 真实场景拆解

按第一梯队场景（至少 3-4 个）分别拆解，每个场景包含：

- 核心流程与关键步骤
- 高认知负荷/高风险/高频节点
- 当前痛点与现有替代方案
- 新功能的增量价值
- 技术约束与风险
- 与其他场景的共性/差异

---

## 3) 6 个 Agent 的独立结论

每个 agent 分别输出：
- Top 10 机会点（标注：跨场景通用 / 场景专用）
- 核心论据
- 风险提醒
- 不建议做的点

---

## 4) 交叉质询记录

- 各 agent 的反对意见（逐条）
- 被高估/被低估的机会
- 总控的裁决与理由（保留/观察/淘汰）

---

## 5) 候选功能总表（核心表格）

表格列：

| 列 | 说明 |
|----|------|
| 功能名称 | 简明标识 |
| 功能类型 | 跨场景通用 / 场景专用 |
| 目标用户角色 | 核心/协作/管理/支撑/间接 |
| 覆盖场景 | 涉及的场景列表 |
| 对应流程环节 | 使用前/使用中/使用后/协同 |
| 痛点描述 | 无此功能时的具体困难 |
| Must-Have 理由 | 为什么不是 nice-to-have |
| 技术方案简述 | 一句话方案 |
| 关键技术难点 | 最难解决的一个点 |
| 合规/安全风险 | 风险等级与关键风险 |
| 商业模式 | 变现方式 |
| 付费对象 | 谁付钱 |
| 11 项评分 | D1-D11 各项分数 |
| 加权总分 | 按权重计算的总分 |
| 结论 | 保留 / 观察 / 淘汰 |
| 建议首发场景 | 从哪个场景切入 |

---

## 6) 三圈交集优先级清单

按四类分组：

**A. Must-Have × Solvable × Profit Center** — 最高优先，立即启动
- 功能列表 + 策略建议 + 首发场景

**B. Must-Have × Solvable** — 必须做但控制成本（标配/护城河）
- 功能列表 + 成本控制策略

**C. Must-Have × Profit Center** — 先做数据积累与可行性验证
- 功能列表 + 验证路径

**D. Solvable × Profit Center** — 谨慎投入，关注市场教育成本
- 功能列表 + 市场教育策略

每类须包含：首发场景推荐 + 扩展顺序建议。

---

## 7) 产品路线图

> 注意：此处"路线图 Phase"指产品交付时间阶段，与 skill 工作流的 Phase 0-4 是不同概念。

### 路线图 Phase 1（0-12月）
- MVP 功能清单 + 首发场景
- 最小技术方案
- 核心验证目标
- 关键里程碑

### 路线图 Phase 2（12-24月）
- 扩展功能 + 场景扩展
- 数据闭环建设
- 核心技术攻关
- 关键里程碑

### 路线图 Phase 3（24-36月）
- 高级功能 + 全场景覆盖
- 平台化/生态化
- 关键里程碑

每个 Phase 标注：**核心交付物、关键技术攻关点、所需资源、场景扩展计划**。

---

## 8) 验证方案

每个高优先功能需提供：
- **验证路径**（从最低成本到正式验证，阶梯式）
- **首发场景选择理由**
- **关键验证指标（KPI）**
- **Go / No-Go 判定标准**
- **验证所需时间与资源估算**

---

## 9) 商业打包策略

- **基础包 / 进阶包 / 旗舰包**内容划分
- 跨场景 vs 单场景定价策略
- 培训/咨询专项包
- 服务合同捆绑策略
- 对标竞品的定价与差异化叙事
- Build vs Buy vs Partner 决策建议（关键组件/技术）

---

## 10) 管理层可读摘要（1 页）

适合产品/市场/技术/管理层共识讨论：

- **核心结论**（3 句话）
- **优先级矩阵**（用文字描述 2×2：用户价值 vs 技术可行性）
- **首发场景与扩展路线**
- **投资回报预期**
- **关键风险与对策**
- **最重要的 3 个待验证假设**

---

## 质量检查清单

输出完成后，总控必须逐项核查：

- [ ] 每条结论都对应了"场景 + 环节 + 角色 + 具体价值 + 风险"
- [ ] 覆盖了协作角色/管理者/支撑团队/间接受益者（不仅仅是核心用户）
- [ ] 每个功能都回答了"谁付钱、怎么付"
- [ ] 每个功能都给出了 MVP 路径
- [ ] 指出了至少 5 个"看起来很酷但现在不该做"的功能
- [ ] 每个功能都有合规/安全分级
- [ ] 对"跨场景通用"功能逐场景验证了适用性
- [ ] 正面评估了最难的技术挑战
- [ ] 对每个高优功能说明了"为什么现在是窗口期"
- [ ] 对不确定结论写出了假设与验证方法
