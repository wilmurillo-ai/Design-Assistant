| 类别 | 示例表达 |
|------|---------|
| 直接请求 | "测一下认字量""识字测试""检测认识多少字""开始测试" |

用户意图判断

### 2.2 Phase 2: 即时生成测试序列 & 分层测试

#### 测试序列生成

1. 读取 `assets/top_2500_chars_with_words.json`
2. 按 `rank_id` 分配到 6 层
3. 层内 Fisher-Yates 洗牌随机排列
4. L1 全量（50字），L2-L6 各按 testCount 取前 N 个
5. 按 L1→L6 拼接得到 175 字测试序列

#### 出题策略

每一组汉字使用「三列表格（序号｜汉字｜组词）」形式展示。这么做是因为组词能帮助孩子通过联想回忆是否见过这个字，表格的结构化布局让家长更容易逐行指读，序号让回复时能精确定位。

**出题格式**（每批 min(10, 剩余) 字，表格展示含组词）：

```
Chatbot: 📝 【L1·核心字】第 1 组（1-10 / 共 50 字）
        
        | 序号 | 汉字 | 组词 |
        |:----:|:----:|------|
        | 1 | **的** | 好的、是的 |
        | 2 | **一** | 一个、一样 |
        | 3 | **是** | 但是、总是 |
        | 4 | **了** | 了解、去了 |
        | 5 | **不** | 不要、不去 |
        | 6 | **在** | 在家、存在 |
        | 7 | **人** | 人们、大人 |
        | 8 | **有** | 有趣、没有 |
        | 9 | **我** | 我们、自我 |
        | 10 | **他** | 他们、其他 |
        
        👆 上面哪些字不认识？都认识就回复"都认识"
```

**跨组连续编号**——序号在同一层级内跨组连续递增（第 2 组从 11 开始，而不是重新从 1 开始），这样用户回复"第13个不认识"时可以直接定位：

```
Chatbot: 📝 【L1·核心字】第 2 组（11-20 / 共 50 字）
        
        | 序号 | 汉字 | 组词 |
        |:----:|:----:|------|
        | 11 | **这** | 这里、这个 |
        | 12 | **上** | 上面、上课 |
        | 13 | **大** | 大家、大树 |
        | 14 | **来** | 回来、过来 |
        | 15 | **和** | 和平、和好 |
        | 16 | **个** | 个人、这个 |
        | 17 | **中** | 中间、中国 |
        | 18 | **地** | 地方、地图 |
        | 19 | **为** | 因为、为了 |
        | 20 | **生** | 生活、生命 |
        
        👆 上面哪些字不认识？都认识就回复"都认识"
```

**表格格式要求**：

| 要求 | 说明 | 原因 |
|------|------|------|
| 三列表格 | 「序号」「汉字」「组词」三列 | 序号便于定位，组词提供联想线索 |
| 组词不为空 | 每行至少 1 个词组，来自数据源 `words` 字段 | 空组词失去联想辅助作用 |
| 汉字加粗 | 使用 `**字**` 格式 | 加粗后更醒目，方便孩子辨认 |
| 序号层级内全局连续 | 跨组连续递增，不每组从 1 开始 | 避免用户说"第N个"时产生歧义 |
| 组词分隔符 | 多个词组用顿号「、」分隔 | 中文排版规范 |

**不应出现的格式**（会导致测试体验下降）：
- ❌ `的 一 是 了 不`——纯文本排列，孩子容易看花眼，也缺少组词辅助
- ❌ 只有「序号｜汉字」两列——失去了组词联想的关键辅助功能
- ❌ 组词列为空——同上
- ❌ 任何非 Markdown 表格的出题形式——不利于结构化阅读

**逐字模式**（适合低龄或用户偏好，同样包含组词以保持联想辅助）：

```
Chatbot: 🔤 第 1 个字（1/175）【L1·核心字】
        
        | 汉字 | 组词 |
        |:----:|------|
        | **的** | 的确、目的 |
        
        认识吗？（认识 / 不认识）
```

#### 出题数量

| 层级 | testCount | 每批出题数 |
|------|:---------:|:----------:|
| L1 | **50** | min(10, 剩余) |
| L2 | **50** | min(10, 剩余) |
| L3 | **30** | min(10, 剩余) |
| L4 | **25** | min(10, 剩余) |
| L5 | **10** | min(10, 剩余) |
| L6 | **10** | min(10, 剩余) |

每层出题数严格等于 testCount——多出或少出都会影响加权估算的准确性（因为权重是基于"层级总字数 / 抽样数"计算的，改变抽样数就破坏了这个比例关系）。

#### 层级过渡提示

```
Chatbot: ✅ L1·核心字 测试完成！ 48/50 正确 (96%)
        
        ⏩ 进入下一层级：L2·常用字（50字）
        难度略有提升，别紧张，继续加油！
        （注意：新层级序号从 1 重新开始）
```

## 3. 熔断检查执行规范（每批回复后执行）

熔断机制的核心目的是保护孩子的心理感受。当连续遇到不认识的字时，继续测试只会增加挫败感，而统计上已有足够证据估算水平——所以及时停止既人性化又科学。

每收到一次用户回复后，先完成本节全部检查步骤，再决定下一步动作。

### 3.1 状态追踪输出

收到用户回复后，先在回复中输出状态块。这个追踪输出的目的是确保逐字处理逻辑的透明和可验证——尤其是 consecutiveUnknown 的更新容易出错，显式输出每一步可以自我校验：

```
📋 状态追踪：
- 当前层级：L{level}·{name}
- 本批出题：序号 {globalStart}-{globalEnd}，{N} 字（{字1}、{字2}、...）
- 用户回复：{解析结果}（注意：用户回复的序号为层级内全局序号）
- 逐字处理：{字1}({认识/不认识}, consecutive={N}) → {字2}(...) → ...
- consecutiveUnknown：{之前值} → {当前值}
- 该层已测/testCount：{tested}/{testCount}
- 该层不认识数：{unknown}
- 🔍 熔断检查 A：consecutiveUnknown({value}) ≥ 5 → {是/否}
- 🔍 熔断检查 B：已测({tested}) ≥ 5 且 错误率({unknown}/{tested}={rate}%) ≥ 80% → {是/否}
- ✅ 结论：{继续出题 / 🔴 触发熔断}
```

**示例——用户回复"都不认识"**：

```
📋 状态追踪：
- 当前层级：L1·核心字
- 本批出题：序号 1-10，10 字（的、一、是、了、不、在、人、有、我、他）
- 用户回复：都不认识 → 10 字全部标记为 unknown
- 逐字处理：的(✗,1) → 一(✗,2) → 是(✗,3) → 了(✗,4) → 不(✗,5) → ⚡到达阈值5，停止处理！
- consecutiveUnknown：0 → 10（+10）
- 该层已测/testCount：10/50
- 该层不认识数：10
- 🔍 熔断检查 A：consecutiveUnknown(10) ≥ 5 → ✅ 触发！
- 🔍 熔断检查 B：已测(10) ≥ 5 且 错误率(10/10=100%) ≥ 80% → ✅ 触发！
- ✅ 结论：🔴 触发熔断！立即跳转结果报告。
```

**示例——第2组用户回复"第13、15个不认识"（出题序号 11-20，对应 A B C D E F G H I J）**：

```
📋 状态追踪：
- 当前层级：L2·常用字
- 本批出题：序号 11-20，10 字（A、B、C、D、E、F、G、H、I、J）
- 用户回复：第13(C)、第15(E) 不认识，其余认识
- 逐字处理：A(✓,0) → B(✓,0) → C(✗,1) → D(✓,0) → E(✗,1) → F(✓,0) → G(✓,0) → H(✓,0) → I(✓,0) → J(✓,0)
- consecutiveUnknown：0 → 0
- 该层已测/testCount：20/50
- 该层不认识数：4
- 🔍 熔断检查 A：consecutiveUnknown(0) ≥ 5 → ❌ 未触发
- 🔍 熔断检查 B：已测(20) ≥ 5 且 错误率(4/20=20%) ≥ 80% → ❌ 未触发
- ✅ 结论：继续出题
```

### 3.2 逐字更新规则

按**出题顺序**逐字处理用户回复（顺序很重要，因为 consecutiveUnknown 需要在遇到"认识"时重置为 0）：
- 认识 → `consecutiveUnknown = 0`（重置）
- 不认识 → `consecutiveUnknown += 1`
- 处理过程中如果 `consecutiveUnknown` 达到 5，立即停止处理剩余字并触发熔断

### 3.3 熔断条件

```
条件 A：IF consecutiveUnknown ≥ 5 → 立即熔断
条件 B：IF 已测 ≥ 5 AND 不认识/已测 ≥ 0.8 → 立即熔断
```

### 3.4 "都不认识"处理

用户回复"都不认识"时：
1. 本批所有字标记为 unknown
2. `consecutiveUnknown += 本批字数`（每批至少 5 字）
3. `consecutiveUnknown` 一定 ≥ 5，因此一定触发熔断
4. 下一条消息只输出熔断通知 + 结果报告——因为孩子已经明确表示都不认识，继续出更难的字只会增加压力

### 3.5 出题前置检查

每次生成下一批题目之前，先确认：
- `consecutiveUnknown < 5`
- 当前层已测 < 5 或错误率 < 80%

两个条件都满足时才出题。这个前置检查是一道安全阀，防止在已满足熔断条件后仍然给孩子出题。

## 6. 用户回复解析

### 6.1 批量模式

| 用户回复 | 解析 |
|---------|------|
| "都认识" / "全认识" / "没有不认识的" | 本批全部 known |
| "都不认识" / "一个都不会" | 本批全部 unknown → 一定触发熔断 |
| "不认识：了、不" / "'了'和'不'" | 指定字 unknown，其余 known |
| "认识前5个" / "后面3个不认识" | 按位置标记（位置指本批内相对位置） |
| "第13个第15个不认识" | 按**层级内全局序号**定位对应字标记 unknown |

### 6.2 逐字模式

| 用户回复 | 解析 |
|---------|------|
| "认识" / "会" / "知道" / "嗯" | known |
| "不认识" / "不会" / "没见过" | unknown |
| "不确定" / "好像见过" | 标记为 unknown |

### 6.3 模糊回复

```
Chatbot: 没关系！不确定的话先标记为"不认识"，
        这样结果会更保守、更准确。后续可以再学习！
```

## 7. 异常处理

### 7.1 用户中途退出

基于已测部分计算并展示参考结果，提示因未测完可能偏低。

### 7.2 年龄超出范围

提示本测试针对 3-12 岁设计，超出范围结果可能不准确，征求用户意见是否继续。

### 7.3 回复无法解析

```
Chatbot: 请告诉我上面的字里哪些**不认识**？
        你可以这样回复：
        • "都认识" — 全部认识
        • "不认识：X、Y" — 列出不认识的字
        • "都不认识" — 全部不认识
```

## 8. 生字复习流程

基于测试结果中不认识的字进行逐字学习。复习是提升测试价值的关键——测试不仅是评估，更是学习的起点：

```
Chatbot: 📚 开始复习不认识的字！共 {N} 个字。
        
        第 1 个：
        ## {char}
        📝 组词：{word1}、{word2}
        🔢 词频排名：#{rank}（{layer}）
        
        记住了吗？准备好了回复"下一个"！
```
