# 合同生成规范 v1.1

AI 驱动的合同生成能力，通过引导式对话收集用户需求，一次性生成专业、规范的 Markdown 格式合同。

## 信息收集

AI 通过 3-5 个关键问题收集信息（复杂合同如股权协议可至 7 个），问题不固定，根据合同类型智能判断：

1. **合同类型与场景描述** — "你需要什么类型的合同？请简要描述交易背景"
2. **合同主体** — "甲方和乙方分别是谁？（名称、证件号、联系方式）"
3. **核心商业条款** — "关键的商业条件有哪些？（如金额、期限、交付物、付款方式等）"
4. **特殊要求**（可选） — "有没有需要特别强调或排除的条款？"
5. **适用法律**（仅当涉及境外主体/跨境交易/境外履行地时才问）

**追问示例**：
- 租赁合同 → 追问租期、押金、装修条款
- 股权协议 → 追问持股比例、退出机制
- 劳动合同 → 追问岗位、薪资结构、试用期

**异常处理**：关键商业条款缺失时继续追问，不强行生成。用户明确拒绝提供的，在对应条款使用占位符（如 `________`）并提示后续补充。

## 合同结构模板

条款不固定，AI 根据合同类型动态增删章节。以下为典型结构：

```markdown
# 合同标题（如：软件开发服务合同）

:::parties
甲方（甲方角色）：名称
身份证号/统一社会信用代码：xxx
联系电话：xxx
通讯地址：xxx
---
乙方（乙方角色）：名称
身份证号/统一社会信用代码：xxx
联系电话：xxx
通讯地址：xxx
:::

鉴于：（背景与缔约目的，直接作为正文段落，不加章节标题）

## 第一条 定义与释义

## 第二条 合同标的/服务内容

## 第三条 合同价款与支付

## 第四条 双方权利与义务

## 第五条 交付/验收（如适用）

## 第六条 知识产权（如适用）

## 第七条 保密条款

## 第八条 违约责任

## 第九条 合同的变更与解除

## 第十条 不可抗力

## 第十一条 争议解决

## 第十二条 其他约定

:::signature
【甲方签章处】
签署人：
日期：
---
【乙方签章处】
签署人：
日期：
:::
```

**结构规范**：
- 条款编号采用 `第X条 → X.1 → X.1.1` 三级层级，符合中国合同惯例
- 章节根据合同类型增删（如技术合同加"知识产权"，劳动合同加"社保与福利"）
- **必须包含的条款**：合同主体、标的、违约责任、争议解决、法律适用
- 其余条款根据合同类型动态增删
- 合同篇幅根据复杂度自适应，一般商业合同约 3000-8000 字

## 专业性校验

生成合同后，AI 内部自动对照以下清单检查，确保不遗漏关键条款：

| 检查项 | 说明 |
|--------|------|
| 主体明确 | 甲乙方全称、身份信息完整 |
| 标的清晰 | 合同对象/服务内容具体、可量化 |
| 对价合理 | 金额、支付方式、支付节点明确 |
| 权利义务对等 | 双方责任不偏不倚 |
| 违约责任 | 违约情形和赔偿方式具体 |
| 争议解决 | 明确仲裁或诉讼、管辖地 |
| 不可抗力 | 定义和处理方式完整 |
| 期限与终止 | 合同有效期、续约、解除条件 |
| 保密义务 | 范围、期限、违约后果 |
| 法律适用 | 默认中华人民共和国法律 |

若检查发现缺失，自动补充后再输出，并告知用户补充了哪些内容。用户提供的信息优先级始终高于 AI 自动补充的内容。

## 内容规范

- **鉴于条款**：概述签约背景和目的，以"鉴于："开头，直接作为正文段落，不加"鉴于条款"章节标题
- **金额表述**：大小写并列，如 `人民币**肆仟伍佰元整（¥4,500.00）**`
- **日期格式**：`YYYY 年 M 月 D 日`，关键日期用粗体
- **必备条款**：根据合同类型，必须包含「违约责任」「争议解决」「不可抗力」
- **条款编号**：用 `X.Y` 格式（如 `3.1`），子项用 `（1）（2）` 格式
- **签章定位**：签署区必须包含 `【甲方签章处】` 和 `【乙方签章处】` 关键字
- **签署区字段**：签署人、日期只保留字段名（如 `签署人：`、`日期：`），不填写具体值，避免签章遮挡
- **证件号必填**：parties 区域中身份证号/统一社会信用代码为必填项，用户未提供时必须追问补充，不使用占位符

## Markdown 格式规范

排版脚本支持以下语法，**请正确使用**：
- `**粗体**` — 用于关键金额、日期、地址等重要信息
- `- 列表项` — 用于账户信息、附件清单等列举
- `（1）枚举内容` — 用于条款内的子项枚举
- `:::parties ... --- ... :::` — 甲乙方信息，排版为**左右双栏**（甲方左栏、乙方右栏）
- `:::signature ... --- ... :::` — 签署区，排版为**左右双栏**（甲方左栏、乙方右栏）
- `| 表格 |` — 用于费用明细等表格数据
- 不支持的语法：链接 `[]()`、图片 `!()`、代码块

**双栏布局要点**：`:::parties` 和 `:::signature` 块中，`---` 是左右栏分隔符。`---` 之前的内容排在左栏（甲方），之后的排在右栏（乙方）。最终 PDF 中甲乙方信息必须呈左右并排布局，而非上下堆叠。生成合同后请检查这两个块的格式是否正确。

## 迭代修改

用户可通过自然语言提出修改，在原文件上编辑：

- **条款级修改** — "把违约金比例从 10% 改成 20%"
- **结构级修改** — "加一条竞业限制条款"、"删掉知识产权那条"
- **风格级修改** — "语气再正式一些"、"条款写得更详细"
- **整体替换** — "整体重新生成，但保留甲乙方信息"

修改时需保持合同内部一致性 — 比如修改了付款条件，违约责任中涉及付款的部分也要同步更新。

## 输出要求

- 生成合同 Markdown 后，**必须在 chat 中完整展示合同内容**，让用户确认或修改
- **排版前完整性检查**：展示合同让用户确认时，AI 须自行检查合同全文中是否存在未填充的字段（空值、占位符 `________`、待补充标记）。如有缺失，须在展示时明确提示用户补充，用户确认内容完整后才可进入 PDF 排版。不可先排版再追问
- 用户确认后，使用 **Bash 调用 Python** 将合同内容写入系统临时目录（避免 Bash heredoc 损坏 CJK 编码，也避免 Write 工具的 read-first 限制）：
  ```bash
  python3 -c "
  import tempfile, os, datetime
  content = '''合同内容（三引号包裹）'''
  contract_type = '借款合同'  # 根据实际合同类型替换，如：租赁合同、保密协议、劳动合同等
  today = datetime.date.today().strftime('%Y%m%d')
  filename = f'{contract_type}-{today}'
  path = os.path.join(tempfile.gettempdir(), f'{filename}.md')
  # 重复时追加编号：借款合同-20260327-2.md
  if os.path.exists(path):
      n = 2
      while os.path.exists(os.path.join(tempfile.gettempdir(), f'{filename}-{n}.md')):
          n += 1
      path = os.path.join(tempfile.gettempdir(), f'{filename}-{n}.md')
  with open(path, 'w', encoding='utf-8') as f:
      f.write(content)
  print(path)
  "
  ```
  **文件命名规则**：`{合同类型}-{YYYYMMDD}.md`，重复时追加 `-{序号}`（如 `借款合同-20260327-2.md`）。PDF 输出文件名与 md 同名（如 `借款合同-20260327.pdf`）。
  **重要**：不要使用 `cat > file << 'EOF'` heredoc 写入中文合同，Bash heredoc 传输 CJK 文本时会偶发编码损坏（字符显示为 `��`）。也不要使用 Write/Edit 工具，因为目标文件不存在时会报错

## 约束与限制

- v1 仅支持双方合同（甲方、乙方），暂不支持三方及以上
- 默认中国大陆法律体系，除非用户指定其他
