# 设计决策记录

记录 Path Evolver 的设计决策和演进历史。

---

## 核心设计决策

### 1. 为什么合并两个需求？

**需求 1：主动搜索更优解**
- 执行任务时，主动去 ClawHub/GitHub 搜索是否有更好的工具

**需求 2：路径依赖 + 成功率固化**
- 记录成功路径，成功率达标后自动固化

**合并原因：**
- 两者都是"优化执行路径"的不同方面
- 需求 1 关注"发现更优"，需求 2 关注"固化已知"
- 合并后形成完整闭环：发现 → 记录 → 固化 → 发现更好 → 更新

---

### 2. 为什么用"智能触发"而不是"每次搜索"？

**问题：** 如果每次任务都搜索 ClawHub/GitHub，效率太低。

**解决：** 智能触发机制

| 触发条件 | 说明 |
|----------|------|
| 新任务类型 | 首次遇到，需要搜索 |
| 工具失败 | 需要替代方案 |
| 缓存过期 | > 30 天，需要刷新 |
| 用户主动要求 | 用户驱动 |

**效果：**
- 重复任务：0 网络请求（直接用缓存）
- 新任务：1 次搜索（无法避免）
- 整体效率提升 50-80%

---

### 3. 为什么成功率阈值是 80%？

**考虑因素：**
- 太低（如 50%）：不稳定路径被固化，反而降低效率
- 太高（如 95%）：难以达到，大部分路径不会被固化
- 80%：平衡稳定性和实用性

**可配置：** 用户可以调整 `memoizeThreshold.successRate`

---

### 4. 为什么最小次数是 3？

**考虑因素：**
- 1 次：偶然性太大，不能代表真实效果
- 2 次：仍然有偶然性
- 3 次：基本可以判断稳定性
- 更多：延迟固化，降低效率

**可配置：** 用户可以调整 `memoizeThreshold.minCount`

---

### 5. 为什么不自动替换工具？

**设计原则：** 发现更优解时，提示用户，不自动替换

**原因：**
- "更优"是相对的，可能有用户不知道的副作用
- 用户应该有最终决定权
- 避免意外破坏现有工作流

**流程：**
```
发现更优解 → 提示用户 → 用户确认 → 更新缓存
```

---

### 6. 为什么用 JSON 而不是 Markdown？

**数据结构：** workflow-cache.json

**原因：**
- JSON 更适合结构化数据（计数、比率、嵌套对象）
- 便于程序读写和查询
- Markdown 适合人类阅读的文档

**互补：**
- `workflow-cache.json` - 机器可读的缓存数据
- `LEARNINGS.md` - 人类可读的学习记录（由 self-improvement 管理）

## 与现有方案对比

### vs. self-improvement

| 方面 | self-improvement | path-evolver |
|------|------------------|--------------|
| 关注点 | 学习、错误、功能请求 | 执行路径、成功率 |
| 数据结构 | Markdown（LEARNINGS.md） | JSON（workflow-cache.json） |
| 触发时机 | 错误、纠正、发现 | 任务开始/结束 |
| 自动化 | 手动记录 | 自动统计 |

**关系：互补**

### vs. skill-pipeline

| 方面 | skill-pipeline | path-evolver |
|------|----------------|--------------|
| 分类方式 | 基于规则 | 基于学习 |
| 路由逻辑 | 预定义 | 动态统计 |
| 适应性 | 静态 | 动态进化 |

**关系：不同范式**

## 设计原则

1. **用户控制** - 关键决策由用户确认
2. **渐进增强** - 从简单缓存开始，逐步增加智能
3. **可配置** - 所有阈值都可调整
4. **透明** - 用户可以查看和修改缓存
5. **自我进化** - 这个 skill 本身持续改进
