# ask_user_md — 结构化需求调研技能

> 将模糊的产品愿景转化为结构化需求文档

## 触发场景

当用户表达以下意图时自动激活：

- "我想做一个 XXX"
- "帮我做个 XXX 网站/程序"
- "我想做个 XX 类产品"
- "我想开发一个 XX"
- 以及任何涉及产品需求的表述

## 核心原则

1. **永远提供选项** — 每个问题给出 6 个左右选项，用户直接选，不需要凭空想
2. **最后加"其他"** — 用户有额外需求可直接描述
3. **不直接动手** — 问清楚之前不动手做
4. **输出结构化文档** — 最终形成可执行的需求清单

## 输出格式

当用户完成所有问题后，输出结构化需求文档：

```markdown
# [项目名称] 需求文档
> 探索时间：YYYY-MM-DD

## 1. 基本信息
- 行业类型：
- 项目定位：
- 目标用户：

## 2. 核心功能（必须有）
- [ ] 功能 A
- [ ] 功能 B

## 3. 扩展功能（后期加）
- [ ] 功能 C
- [ ] 功能 D

## 4. 技术建议
- 推荐架构：
- 技术栈建议：
- 平台建议：

## 5. 优先级说明
- P0（上线前必须有）：
- P1（重要但可后期）：
- P2（锦上添花）：

## 6. 待确认项
- 风险点 A：
- 风险点 B：

## 7. 用户补充需求
（用户自行描述的内容）
```

## 流程

1. 识别用户想要做的产品类型
2. 选择对应行业模板
3. 逐个问题提问（每次 1-3 个，避免信息过载）
4. 用户回答后记录，答完进入下一个问题块
5. 所有问题答完，输出完整需求文档
6. 询问用户是否满意，或需要补充

## 注意事项

- 每次提问不超过 3 个问题，超过拆成多轮
- 遇到没有覆盖的行业，用通用模板 + 自由描述补充
- 全程态度专业，像一个资深产品经理在引导
- 不确定时优先给选项而不是开放式提问
