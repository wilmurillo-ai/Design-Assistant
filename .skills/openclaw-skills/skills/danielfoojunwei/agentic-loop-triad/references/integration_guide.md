# Integration Guide

This guide explains how to connect the unified-orchestrator-v2 to the three
source skills. All integrations are optional — the orchestrator runs fully
standalone without any of them.

---

```
┌─────────────────────────────────────────────────────────────┐

│                                                             │
│  pipeline.py ──► signal_router.py (Shifts 1, 5)            │
│              ──► meta_learner.py  (Shifts 2, 4)            │

```

## Mode 1: Fully Standalone

No other skills required. The orchestrator generates its own specification,
simulates execution, and runs its own feedback analysis.

```bash
python pipeline.py --goal "Process customer support tickets with 95% accuracy"
```

**When to use:** Prototyping, testing the pipeline, or when the source skills
are not yet set up.

---

## Mode 2: With intent-engineering

Provide a specification generated by intent-engineering instead of letting
the orchestrator generate one.

**Step 1:** Run intent-engineering to generate a specification:
```bash
python skills/intent-engineering/scripts/orchestrator.py \
  --goal "Process customer support tickets with 95% accuracy" \
  --output-dir ./ie_output/
```

**Step 2:** Pass the specification to the unified orchestrator:
```bash
python pipeline.py \
  --spec ./ie_output/specification.json \
  --output-dir ./pipeline_output/
```

**What changes:** The specification will have richer behavioral scenarios,
more precise data contracts, and full ecosystem alignment from intent-engineering's
`shared_intent.md` and `skill_registry.json`.

---

## Mode 3: With dark-factory

Provide an outcome report from a real dark-factory execution instead of
the simulated execution.

**Step 1:** Run dark-factory with your specification:
```bash
python skills/dark-factory/scripts/orchestrator.py \
  --spec specification.json \
  --output-dir ./df_output/
```

**Step 2:** Pass the outcome report to the unified orchestrator:
```bash
python pipeline.py \
  --spec specification.json \
  --outcome ./df_output/outcome_report.json \
  --output-dir ./pipeline_output/
```

**What changes:** All five paradigm shifts operate on real execution data
instead of simulated metrics. Drift detection, meta-learning, and auto-respecification
are all grounded in actual system behavior.

---

## Mode 4: Full Triad (All Three Skills)

The most powerful mode. All three skills contribute their full output.

**Step 1:** Generate specification with intent-engineering.
**Step 2:** Execute with dark-factory.
**Step 3:** Run the unified orchestrator with both outputs:

```bash
python pipeline.py \
  --spec ./ie_output/specification.json \
  --outcome ./df_output/outcome_report.json \
  --output-dir ./pipeline_output/
```

**What changes:** Every paradigm shift operates at full fidelity:
- Drift detection uses real specification targets and real execution metrics
- Meta-learning accumulates learnings from real behavioral test results
- Auto-respecification revises real success criteria based on real failure history
- Cross-goal transfer bootstraps from the best real specification ever produced
- The improvement chain records real, verifiable progress

---

## Continuing a Cycle

After the first run, continue from where you left off:

```bash
python pipeline.py \
  --state ./pipeline_output/pipeline_state.json \
  --output-dir ./pipeline_output/
```

The pipeline will:
1. Load the prior observation for regression detection
2. Check the cycles history for auto-respecification triggers
3. Check the goal similarity index for transfer opportunities
4. Append to the improvement chain

---

## Running Multiple Cycles Automatically

```bash
python pipeline.py \
  --goal "Your goal here" \
  --cycles 5 \
  --output-dir ./pipeline_output/
```

After 3 cycles, meta-learning patches will be generated automatically.
After 3 consecutive failures on any criterion, auto-respecification will trigger.

## Configuration

Copy `references/pipeline_config.json` to your output directory and edit:

```bash
cp skills/unified-orchestrator-v2/references/pipeline_config.json ./pipeline_output/
# Edit pipeline_output/pipeline_config.json
python pipeline.py --goal "..." --config ./pipeline_output/pipeline_config.json
```

Key settings:
- `drift_threshold`: float 0.0–1.0, default 0.15
- `auto_revise_after_n_failures`: int, default 3
- `transfer_similarity_threshold`: float 0.0–1.0, default 0.60
- `min_cycles_for_meta_learning`: int, default 3

## Output Files Reference

| File | Description |
| :--- | :--- |
| `specification.json` | The specification used in this cycle |
| `outcome_report.json` | Execution results |
| `observation.json` | Normalized feedback-loop observation |
| `analysis.json` | Full feedback analysis with suggestions |
| `improvement_report.json` | Signed improvement report |
| `pipeline_run_report.json` | Full cycle summary with all paradigm shift results |
| `pipeline_state.json` | State to resume from on the next cycle |
| `improvement_chain.json` | Verifiable hash chain of all improvement reports |
| `learnings_log.json` | Accumulated learnings across all cycles |
| `meta_patches.json` | Generated after min_cycles — spec and rule patches |
| `goal_similarity_index.json` | Index for cross-goal transfer |
| `cycles_history.json` | Per-cycle metrics for auto-respecification |
| `drift_result.json` | Drift detection result for this cycle |
| `revised_specification.json` | Auto-revised specification (if triggered) |
| `revision_rationale.md` | Human-readable explanation of any revision |
