# POST_SEARCH_DECISION_RULES

## 目的
定義「搜尋到 OB 筆記之後，下一步該做什麼」。避免搜到就停，或直接亂改。

## 1. 搜尋後先判斷四件事
- 這篇是要 **讀**、**引用**、**修改**、還是 **搬移/改名**？
- 這篇位於哪個母區？是 `Skills`、`OpenClaw`、`AIout`、`Library`、`Server`、`Projects`、`Prompts`、`OldNotes`、還是根目錄？
- 這篇屬不屬於亞瑟自己管理的目錄？
- 這次操作會不會影響 links、結構、或既有治理規則？

## 2. 搜尋結果的典型後續動作
### A. 只讀 / 摘要 / 回答
適用：
- 使用者只是問內容
- 只是要引用片段
- 只是要確認是否存在

做法：
- 讀該 note
- 摘要重點
- 不主動修改

### B. 更新現有 note
適用：
- 使用者明確要補內容
- 只是文字更新
- 不涉及換位置或改檔名

做法：
- 直接編輯 `.md` 可接受
- 但先確認這篇不是亞瑟自己管理且不希望被動到的內容

### C. 搬移 / 改名
適用：
- 落點錯了
- 命名錯了
- 需要歸位到正確母區

做法：
- 優先 `obsidian-cli move`
- 先判斷 link refactor 風險
- 先確認不是在亂動亞瑟自己管理的區

### D. 建立新 note
適用：
- 搜完發現根本不存在
- 現有 note 不適合硬塞新內容
- 這是一個新主題、新結案、新 spec

做法：
- 先決定母區與檔名
- 再建立新 note
- 依 `WRITING_RULES.md` 套用命名與開頭格式

## 3. 落點優先判斷
先想：
- `Skills / Prompts / OldNotes / Server / OpenClaw / Projects / Library / AIout / 根目錄`

再問：
- 這篇是 skill 專屬文件嗎？
- 是 OpenClaw 治理文件嗎？
- 是 AI 輸出 / workflow result 嗎？
- 是歷史參考 / memory / summary 嗎？
- 是亞瑟的隨手記嗎？

## 4. 不要做的事
- 不要搜尋到就直接改
- 不要因為找到了最像的 note，就把不相干內容硬塞進去
- 不要在沒有確認主控權時，改亞瑟自己管理的內容
- 不要用 shell 大批改名，跳過 link refactor 風險判斷

## 5. 最後一問
每次搜尋到 note 後，最後都問自己：
> 我現在是要讀它，更新它，搬它，還是應該新開一篇？
