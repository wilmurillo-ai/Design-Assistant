# Token 效率分析：中英文混合策略

## 一、当前状态

**SKILLS_INDEX.yaml:**
- name/description: 中文
- 技术术语: 英文 (pdb2gmx, RMSD, BAR)
- 参数名: 英文 (lambda_windows, sc-alpha)

**脚本注释:**
- 功能说明: 中文
- 技术术语: 英文
- 命令/代码: 英文

**错误信息:**
- 错误描述: 中英混合
- 修复命令: 英文

## 二、Token 效率对比

### 测试样本

**纯英文:**
```
System Preparation
From PDB to simulation system
ACPYPE missing → auto-install or try antechamber
Ligand topology missing → multi-method parameterization
```

**纯中文:**
```
系统准备
从 PDB 到可模拟系统
ACPYPE 缺失 → 自动安装或尝试 antechamber
配体拓扑缺失 → 多方案参数化
```

**中英混合 (推荐):**
```
系统准备
从 PDB 到可模拟系统
acpype 缺失 → 自动安装或尝试 antechamber
配体拓扑缺失 → 多方案参数化
```

### Token 估算

**Claude 的 tokenizer 特点:**
- 英文: ~1 token/word (平均 4-5 字符)
- 中文: ~1 token/字 (1-2 字符)
- 混合: 取决于分词

**实际测试 (粗略估算):**

| 版本 | 字符数 | 估算 token | 相对效率 |
|------|--------|-----------|---------|
| 纯英文 | ~180 | ~40 | 100% (基准) |
| 纯中文 | ~120 | ~60 | 67% |
| 中英混合 | ~130 | ~45 | 89% |

**结论:**
- 纯英文最省 token
- 中英混合次之 (比纯英文多 ~10-15%)
- 纯中文最费 token (比纯英文多 ~50%)

## 三、推荐策略

### 策略 A: 激进省 token (纯英文)

**优点:**
- Token 最少
- AI 理解最快

**缺点:**
- 人类可读性差 (中国用户)
- 失去语义清晰度

**适用场景:**
- 纯 AI-to-AI 交互
- 不需要人类阅读

### 策略 B: 平衡策略 (中英混合) ⭐ 推荐

**优点:**
- Token 适中 (比纯英文多 10-15%)
- 人类可读性好
- 语义清晰

**缺点:**
- 不是最省 token

**适用场景:**
- AI-to-AI 为主，人类偶尔查看
- 需要平衡效率和可读性

**具体规则:**
```yaml
# 使用中文的地方
- 功能名称: "系统准备" (而非 "System Preparation")
- 动作描述: "从 PDB 到可模拟系统" (而非 "From PDB to...")
- 错误描述: "配体拓扑缺失" (而非 "Ligand topology missing")

# 使用英文的地方
- 技术术语: RMSD, RMSF, BAR, PMF (不翻译)
- 命令名称: pdb2gmx, editconf, gmx (不翻译)
- 参数名称: lambda_windows, sc-alpha (不翻译)
- 文件名称: ligand.gro, topol.top (不翻译)
- 工具名称: acpype, antechamber (小写，不翻译)

# 使用符号的地方
- 箭头: → (而非 "to" 或 "到")
- 范围: 0.3-0.7 (而非 "0.3 to 0.7")
- 列表: - (而非 "1. 2. 3.")
```

### 策略 C: 保守策略 (纯中文)

**优点:**
- 人类可读性最好
- 语义最清晰

**缺点:**
- Token 最多 (比纯英文多 50%)

**适用场景:**
- 人类为主要用户
- 不在乎 token 消耗

## 四、实施建议

### 立即优化 (阶段 2.5)

**目标:** 在当前中英混合基础上，进一步优化

**优化点:**

1. **统一技术术语为小写英文**
```yaml
# 当前
"ACPYPE 缺失"

# 优化后
"acpype 缺失"
```

2. **精简动词**
```yaml
# 当前
"自动安装或尝试 antechamber"

# 优化后
"自动安装 / 尝试 antechamber"
```

3. **使用符号替代文字**
```yaml
# 当前
"lambda 窗口数量: 13 (非均匀分布)"

# 优化后
"lambda_windows: 13 (非均匀)"
```

4. **精简注释**
```bash
# 当前
# 这个参数控制软核势的强度，范围是 0.3 到 0.7

# 优化后
# sc-alpha: 0.5 (0.3-0.7) 软核强度
```

### 预期效果

**优化前 (当前):**
```yaml
name: "自由能计算"
description: "配体结合自由能 (TI/FEP/BAR) - 完整流程"
auto_fix:
  - "ACPYPE 缺失 → 自动安装或尝试 antechamber"
  - "配体拓扑缺失 → 多方案参数化"
key_params:
  - "lambda 窗口数量: 13 (非均匀分布)"
  - "软核参数 alpha: 0.5 (范围 0.3-0.7)"
```
**估算:** ~50 tokens

**优化后:**
```yaml
name: "自由能计算"
description: "配体结合自由能 (TI/FEP/BAR)"
auto_fix:
  - "acpype 缺失 → 自动安装 / 尝试 antechamber"
  - "配体拓扑缺失 → 多方案参数化"
key_params:
  - "lambda_windows: 13 (非均匀)"
  - "sc-alpha: 0.5 (0.3-0.7)"
```
**估算:** ~40 tokens

**节省:** ~20%

## 五、总结

### 推荐方案

**采用策略 B (中英混合) + 微优化**

**原则:**
1. 功能/动作用中文 (清晰)
2. 术语/参数用英文 (省 token)
3. 符号优于文字 (→ / - :)
4. 精简冗余词 ("的"、"是"、"进行")

**预期效果:**
- 比纯英文多 10% token
- 比当前版本省 15-20% token
- 保持良好可读性

### 是否需要立即优化？

**建议:** 在阶段 2 完成后，作为阶段 2.5 进行微优化

**原因:**
- 不影响核心功能
- 可以批量处理
- 效果明显 (省 15-20% token)
