# Claude Code 工作流参考

## 常用开发工作流

### 1. 新功能开发

```markdown
**触发条件**: 用户需要实现新功能

**步骤**:
1. 需求澄清 - 确认功能细节和边界条件
2. 技术方案 - 设计实现方案和架构
3. 代码实现 - 编写代码和单元测试
4. 自检验证 - 运行测试确保功能正常
5. 文档更新 - 更新相关文档

**示例提示词**:
- "帮我实现一个用户认证系统，支持邮箱登录和 JWT"
- "创建一个数据导入功能，支持 CSV 和 Excel 格式"
```

### 2. Bug 修复

```markdown
**触发条件**: 发现 bug 或异常行为

**步骤**:
1. 问题复现 - 确认 bug 的触发条件
2. 根因分析 - 定位问题源头
3. 修复方案 - 设计修复策略
4. 代码修复 - 实施修复并添加测试
5. 回归验证 - 确保修复不引入新问题

**示例提示词**:
- "这个函数在处理空数组时抛出异常，帮我修复"
- "用户报告登录后页面空白，帮我排查原因"
```

### 3. 代码重构

```markdown
**触发条件**: 代码需要改进可维护性或性能

**步骤**:
1. 代码分析 - 识别需要重构的部分
2. 重构计划 - 制定安全的重构步骤
3. 渐进重构 - 小步重构，每步验证
4. 测试覆盖 - 确保重构后测试通过
5. 性能对比 - 验证性能改进

**示例提示词**:
- "这个模块太复杂了，帮我重构提高可读性"
- "优化这个函数的性能，它现在是瓶颈"
```

### 4. 代码审查

```markdown
**触发条件**: 需要审查代码质量

**审查清单**:
- [ ] 代码风格一致性
- [ ] 命名清晰准确
- [ ] 函数职责单一
- [ ] 错误处理完善
- [ ] 边界条件覆盖
- [ ] 注释和文档充分
- [ ] 测试覆盖率足够
- [ ] 无明显安全漏洞
- [ ] 性能考虑合理

**示例提示词**:
- "审查这个 PR，重点关注安全性和性能"
- "检查这个模块是否有潜在的 bug"
```

### 5. 技术调研

```markdown
**触发条件**: 需要评估技术方案或库

**步骤**:
1. 需求定义 - 明确调研目标和约束
2. 方案收集 - 列出可行的技术方案
3. 对比分析 - 评估各方案优劣
4. 原型验证 - 快速验证关键假设
5. 建议输出 - 给出推荐方案和理由

**示例提示词**:
- "比较一下 Express.js 和 Fastify 的性能和生态"
- "调研一下实现实时通信的最佳方案"
```

## 模型选择指南

| 任务类型 | 推荐模型 | 理由 |
|----------|----------|------|
| 简单修复 | claude-haiku | 快速、经济 |
| 常规开发 | claude-sonnet | 平衡质量和速度 |
| 复杂架构 | claude-opus | 深度思考和规划 |
| 代码审查 | claude-sonnet | 细节把控好 |
| 技术调研 | claude-opus | 分析能力强 |
