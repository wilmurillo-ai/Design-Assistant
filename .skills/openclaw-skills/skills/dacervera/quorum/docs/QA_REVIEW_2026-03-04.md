## Summary

The README and RUBRIC_BUILDING_GUIDE are well-written and tonally consistent, but contain significant factual inaccuracies around critic counts and a **critical JSON schema mismatch** between the guide's examples and the actual rubric format used by the system. These will mislead users trying to build rubrics or understand what the system does today vs. what's planned.

---

## CRITICAL

### C-001 — JSON Schema Mismatch: RUBRIC_BUILDING_GUIDE examples use wrong field names

**Location:** `docs/RUBRIC_BUILDING_GUIDE.md` — Sections 5, 7, 8

The guide's JSON examples use a field schema that does **not match** the actual built-in rubric schema used by the reference implementation.

| Field in Guide | Actual field in built-in rubrics |
|---|---|
| `assertion` | `criterion` |
| `assessment_guidance` | *(not present)* |
| `evidence_requirements` (array) | `evidence_required` (string) |
| `title` | *(not present as separate field)* |
| *(not present)* | `why` |
| *(not present)* | `category` |

**Evidence:**
- Built-in rubric schema (`reference-implementation/quorum/rubrics/builtin/agent-config.json`): uses `id`, `criterion`, `severity`, `evidence_required`, `why`, `category`
- Guide's criteria examples: use `id`, `title`, `assertion`, `severity`, `assessment_guidance`, `evidence_requirements`

**Impact:** Any rubric a user builds following this guide's examples will fail to load or parse correctly, since the field names don't match what the loader expects.

**Fix:** Either update the guide examples to match the actual schema (preferred, keeps source of truth in the code), or document that the guide describes an aspirational extended schema and provide a separate "current supported schema" callout box.

### C-002 — Depth Table in README: Standard and Thorough critic counts are wrong

**Location:** `README.md` — "You Decide How Deep I Go" section

The README's depth table claims:

| Depth | Critics |
|---|---|
| Quick | 2 |
| Standard | 4 + tester |
| Thorough | 6-9 + fix loops |

**Actual configuration** (`reference-implementation/quorum/configs/`):
- `quick.yaml`: `correctness`, `completeness` — **2 critics** ✓
- `standard.yaml`: `correctness`, `completeness` — **2 critics**, `max_fix_loops: 0`
- `thorough.yaml`: `correctness`, `completeness` — **2 critics**, `max_fix_loops: 0`

All three profiles currently use exactly 2 critics with no fix loops. The "4 + tester" and "6-9 + fix loops" rows describe planned/future capabilities, not current reality.

**Impact:** Users will expect more than they get. Discovering this mismatch after installing is a trust-breaking experience.

**Fix:** Update the depth table to accurately reflect current capability. Add a clear "coming soon" row or note for planned expansion. The "What I Can Do Today" section below already correctly says "2 critics (Correctness, Completeness)" — the depth table contradicts it.

### C-003 — README architecture diagram shows unimplemented critics as active

**Location:** `README.md` — "How I Work Under the Hood" diagram

The ASCII diagram shows Security, Architecture, Delegation, and Tester critics in active boxes within the critic pool. Only Correctness and Completeness are implemented (`reference-implementation/quorum/critics/` contains only `correctness.py` and `completeness.py`).

**Impact:** Users reading the diagram will assume all depicted critics are operational.

**Fix:** Add a legend or note to the diagram: "Currently active: Correctness, Completeness. Others: planned." Alternatively, dim or annotate future critics differently.

### C-004 — RUBRIC_BUILDING_GUIDE references non-existent `security` critic

**Location:** `docs/RUBRIC_BUILDING_GUIDE.md` — Section 7 (Packaging), "Which critics?" guidance

The guide recommends `"critics": ["correctness", "completeness", "security"]` for security/compliance rubrics. The `security` critic does not exist in the current implementation.

**Evidence:** `reference-implementation/quorum/critics/` — only `base.py`, `completeness.py`, `correctness.py`.

**Impact:** Rubrics built with `"critics": ["security"]` will fail or silently skip that critic.

**Fix:** Note that `security` and `architecture` critics are planned but not yet available. For now, only `correctness` and `completeness` are functional.

## HIGH

### H-001 — Normative strength table missing LOW severity

**Location:** `docs/RUBRIC_BUILDING_GUIDE.md` — Section 4, "What You're Looking For" table

The table maps:
- SHALL → CRITICAL
- SHOULD → HIGH
- MAY → MEDIUM

But the actual built-in rubrics include **LOW severity** (e.g., `AC-009` in `agent-config.json`). The guide never mentions LOW as a valid severity level, leaving users no guidance on when to use it.

**Fix:** Add a row: `Informative / non-normative context → LOW`. Clarify that LOW is available for style guidance or best-practice-but-not-required items.

### H-002 — README "built-in rubrics" don't match `examples/rubrics/`

**Location:** `README.md` — "Where I Am Right Now" (Shipped section) and "Rubric Roadmap"

README says built-in rubrics are: `research-synthesis` and `agent-config`.

`examples/rubrics/` contains: `research-synthesis-rubric.json` and **`swarm-config-rubric.json`** (not `agent-config`).

The actual built-ins in `reference-implementation/quorum/rubrics/builtin/` are `research-synthesis.json` and `agent-config.json` — these match the README claim. But `examples/rubrics/swarm-config-rubric.json` is undocumented and its relationship to the built-in `agent-config` rubric is unclear.

**Fix:** Clarify the distinction between built-in rubrics (shipped with the package) and example rubrics (in `examples/rubrics/`). Either document `swarm-config-rubric.json` or remove it if it's stale/superseded by `agent-config`.

### H-003 — `thorough.yaml` has malformed model reference

**Location:** `reference-implementation/quorum/configs/thorough.yaml`

```yaml
model_tier2: claude-opus-4   # Use tier 1 for all critics in thorough mode
```

Missing the `anthropic/` provider prefix (other profiles use `anthropic/claude-opus-4-0`). This is a config file bug, not a doc bug, but it directly undermines the README claim that thorough mode works. Flagged here because the README describes thorough mode as production-ready.

**Fix:** Change to `anthropic/claude-opus-4-0` (consistent with other profiles and LiteLLM naming conventions).
