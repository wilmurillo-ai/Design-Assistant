### 标准化招聘流程

**第一步：简历筛选（语义匹配）**
- 输入：候选模型的技术文档、性能报告 + 岗位JD
- 工具：AI语义比对系统 → 生成"岗位适配度得分"
- 淘汰线：适配度 < 80% → 自动淘汰
- 判定标准：仅泛写技术名词（如"熟悉NLP"）而无具体技术点（如BERT微调、Prompt工程）→ 极低分

**第二步：笔试评估（能力验证）**
| 岗位 | 考察内容 |
|------|---------|
| 技术岗 | 数据结构、操作系统、数据库SQL、机器学习原理；编程题LeetCode中等难度 |
| 算法岗 | 技术岗内容 + 模型调参、算法设计（如梯度消失问题） |
| AI开发岗 | 技术岗内容 + 模型部署相关题目 |

**第三步：AI面试官评估（五维度评分）**

| 维度 | 评分标准 |
|------|---------|
| 正确性 | 代码跑通率、边界条件覆盖 |
| 可读性 | 变量命名规范、注释习惯 |
| 健壮性 | 异常输入处理、空指针防范 |
| 效率 | 时间复杂度、空间复杂度 |
| 需求分析能力 | 主动追问模糊需求（如"缓存"的过期策略） |

**第四步：价值观对齐**
- 评估模型行为逻辑是否与企业价值观一致
- 参考：以客户为中心、长期有耐心、正直诚信、合作共赢

**第五步：合规前置审查（必须项）**
- 算法透明度：必须提供可解释性报告，拒绝黑箱模型
- 数据安全：ISO27001认证、等保三级资质
- 信效度：α系数 > 0.85 的第三方验证报告
- 以上三项不全 → 一票否决

## 二、入职模块（模块二）

### 总体架构：感知-决策-执行-反馈闭环

```
用户请求 → LLM/Agent → 工作流引擎（LangGraph/n8n）→ 业务系统API → 返回结果/触发操作
```

| 层级 | 职责 | 技术实现 |
|------|------|---------|
| 感知层 | NLU解析HR指令，识别任务意图 | 自然语言理解模型 |
| 决策层 | LLM驱动 + RAG知识库，拆解任务路径 | 大语言模型 + 知识库检索 |
| 执行层 | API/RPA/MCP操作ERP/CRM/HRIS | OpenAI Function Calling / MCP协议 |
| 反馈层 | 日志记录、异常处理、审计追溯 | 持久化日志 + 人工接管接口 |

### 核心技术实现

| 技术模块 | 实现方法 |
|---------|---------|
| 工作流编排 | LangGraph、n8n 或 Power Automate 状态机 |
| 工具调用 | OpenAI Function Calling 或 MCP 协议封装 |
| 身份认证 | OAuth2 / API Key 鉴权 |
| 上下文管理 | Redis 或 PostgreSQL 持久化会话状态 |
| 异常处理 | Schema验证 + 置信度阈值 + 超时熔断 + 人工审查队列 + 回滚策略 |

### 四级权限控制

| 级别 | 名称 | 权限说明 |
|------|------|---------|
| L1 | 只读 | 读取数据，无操作权限 |
| L2 | 建议 | 生成建议，需人工审批后执行 |
| L3 | 受控写入 | 预设范围内的写入，自动日志 |
| L4 | 闭环执行 | 全权限，日志+审计+熔断三重保障 |

### 五步实施路径

1. **高价值场景评估**：识别高频、高人力成本、易出错的入职环节
2. **技术架构选型**：私有化部署优先（实在Agent、CoCo），兼容现有HR系统
3. **Agent设计与编排**（工作流节点）：
   - 候选人信息检索（对接HRIS）
   - 文档自动收集与验证（候选管理系统）
   - 个性化学习计划生成（角色+团队+技能匹配）
   - 创建IT工单（Jira/Zendesk）
   - 进度监控与提醒（日历+IM系统）
4. **测试与灰度上线**：影子运行（生成建议但不执行）+ 人机协同确认
5. **迭代与规模化复制**：日志优化Prompt → 流程模板化 → 推广至其他部门

### 公平性考核

#### 公平性量化指标

| 指标 | 定义 |
|------|------|
| 统计均等差（SPD） | 不同群体获得正向结果的比例差异 |
| 机会均等差（EOD） | 符合条件的不同群体被正确识别的比例差异 |
| 不均衡比例（DI） | 非优势群体获正向结果概率与优势群体之比（理想值≈1） |
| 差异阈值 | 各群体间性能差异的最大容忍值 |

#### 偏见检测方法

| 方法 | 说明 |
|------|------|
| 反事实公平性测试 | 仅改变敏感属性（姓名、性别代词），观察输出是否变化 |
| 分组评估 | 按性别/年龄/地域分组，分别计算各组性能指标 |
| 红队演练 | 模拟对抗性输入，测试歧视性回应 |
| 偏见探测工具包 | IBM AIF360、Fairlearn 开源库自动化计算 |

| 指标 | 目标值 |

### 综合评估最佳实践

- **GTX基准真相集**：使用真实生产数据而非黄金数据集
- **人类-AI协同评估**：LLM-as-a-judge + 人工评审结合

## 五、淘汰模块（模块五）

### 三类触发条件（量化指标）

| 触发类型 | 量化标准 |
|---------|---------|
| 性能持续衰减 | 准确率/F1连续多周期下降 > 10%；推荐系统点击率/转化率不可逆下滑 |
| 安全违规事件 | 越权访问敏感数据、执行未授权操作（如自动转账）、数据泄露 |
| 合规失效 | 法律法规更新导致架构或训练数据不再符合合规标准 |

### 五步标准化退役流程

| 步骤 | 操作 | 详细说明 |
|------|------|---------|
| 1 | 蓝绿部署策略 | 蓝绿/金丝雀发布，逐步流量迁移，无缝切换 |
| 2 | 渐进式下线 | 低峰时段分批关闭节点，监控稳定性，逐步扩大范围 |
| 3 | 自动归档元数据 | 原始文件+训练日志+参数配置+评估报告+数据集→对象存储，标记"已退役" |
| 4 | 建立回滚预案 | 退役后7天内持续监控新模型表现，重大缺陷→自动化脚本快速恢复 |
| 5 | 更新权限与注册表 | 注销Agent身份标识；模型注册表标记生命周期状态="退役" |

**量化目标**：人工干预减少80%，业务中断风险降低90%
