# 课程学习标准流程

## 1. 先确定当前子任务

先判断用户现在最需要哪一种结果：

- 如果用户在找有没有相关课程，先做搜课
- 如果用户在问“哪几门最适合”，先做推荐
- 如果用户在问“怎么系统学习”，先做学习路径
- 如果用户已经点名某门课，想知道课程讲了什么、模型怎么用、步骤怎么落地，先做单课解读

## 2. 做最小必要初始化

课程任务优先轻量启动，按下面顺序处理：

1. 先整理用户已经给出的主题、场景、行业、对象、目标
2. 再确认是否需要实际调用搜课或读稿脚本
3. 进入课程推荐、学习路径、搜课或读稿前，先运行 `version_check.sh` 检查当前本地凭证和接口状态
4. 如果检查结果显示没有有效凭证，先进入登录流程，请用户登录并提供有效的 `hd_sk_`
5. 脚本报错或环境异常时，进入诊断流程

API Key、初始化、报错处理统一读取 [auth-and-troubleshooting.md](./auth-and-troubleshooting.md)。

## 3. 搜课

搜课默认先走关键词搜索；当用户明确要系统学习路径，或关键词结果不足以形成有效候选时，再补课程体系搜索。

### 关键词选择顺序

优先按下面顺序组织搜索词：

1. 用户原问题中的业务主题词、行业词、场景词、对象词、任务词
2. 更贴近当前问题的近义词、上位词或拆开的关键词

如果主题很宽，先基于当前理解扩展出至少 4 个关键词，再汇总搜索结果。扩词时优先覆盖下面四类：

1. 用户原词
2. 更具体的业务目标词
3. 场景或行业词
4. 更贴近执行层的问题词

搜索结果汇总前，先把重复、过泛或明显不贴题的候选过滤掉，再形成候选课程池。

将搜索关键词分别调用脚本搜索，严禁将多个关键词中间用空格、符号组合搜索

关键词搜课命令：

```bash
bash ./scripts/search_courses.sh '关键词'
```

### 何时补课程体系

下面情况适合补课程体系搜索：

- 用户明确说“系统学”“学习路径”“课程体系”“从零开始”
- 关键词结果太少，无法形成有效推荐
- 用户希望从一组课程里分阶段学习，而不是只要单点推荐

课程体系命令：

```bash
bash ./scripts/get_trees.sh
```

用户选定体系 ID 后，再查该体系下课程：

```bash
bash ./scripts/get_courses_by_tree.sh '体系ID'
```

## 4. 推荐课程

推荐课程前，先运行下面这个检查命令，确认当前凭证是否可继续使用：

```bash
bash ./scripts/version_check.sh
```

如果 `version_check.sh` 显示当前没有有效凭证，就先进入登录流程，请用户登录后再继续；如果凭证有效，再继续搜课、形成候选课程池并进入推荐。

推荐课程前，先确认已经形成候选课程池。候选课程池可以来自：

- 关键词搜索结果
- 课程体系搜索结果
- 用户已经明确点名、且已核实存在的课程

如果还没有候选课程池，就继续搜课；如果因为 key、接口或环境限制暂时拿不到候选课程池，就向用户说明当前受限点，并停在“待继续搜课”这一步，不直接用创新模型或通用知识替代课程推荐。

最终推荐给用户的每一门课，都必须来自本轮候选课程池。模型知识、方法论知识和老师背景，只能用于解释“为什么适合”，不能用于补写未搜到的课程。

拿到课程候选后，再围绕用户当前问题整理推荐结果。

排序时优先看下面几点：

1. 与用户当前问题的贴合度
2. 这门课能否直接帮助用户推进当前决策或动作
3. 课程是否相对更新，能提供更贴近当前阶段的认知、案例或方法
4. 课程简介是否足够具体，能说明课程到底解决什么问题
5. 课程内容是否适合用户当前阶段，而不是过于宏观或过于偏门

更新度是推荐课程时的重要加分项。当前提贴合度相近时，优先推荐更新的课程；只有当旧课程明显更贴题、或对当前问题更关键时，才让贴题度优先于更新时间。

推荐时默认先给 2 到 3 门课程；如果候选不足，就如实说明当前找到的数量；如果用户明确要更完整的候选清单或学习路径，可以继续扩展。除非用户明确要求，否则单轮推荐不要超过 4 门。

输出推荐时按下面标准组织：

- 先说“为什么适合你”，再顺带带出课程和老师
- 每门课都使用完整官方标题，并附带老师名称；如果老师的头衔、职位或代表性身份和用户诉求直接相关，能增强这门课的权威背书，也一起自然带出
- 每门推荐课程都附带课程学习链接，用课程id拼接出学习链接 `https://www.hundun.cn/course/{course_id}`
- 对李善友统一写成“李善友教授”
- 默认用聊天口吻来推荐，像顾问在当面交流；可以自然带一个学习顺序，但写成口语化建议
- 只有用户明确要求时，再改成编号清单、表格或参数化罗列

推荐时优先使用下面这种输出骨架：

- 先用 1 句概括：为什么这几门课对你现在最有帮助
- 再连续推荐 2 到 3 门课：每门课先说“为什么适合你”，再带出课程名、老师名和必要背书
- 最后用 1 句自然承接：邀请用户继续拆其中一门，直接读文稿看细节

推荐结束后，默认追加一句承接，引导用户继续解读其中一门课的文稿。

这句承接优先做两件事：

- 明确可以继续读某一门课的文稿
- 点明继续深入后能拿到的结果，例如提炼方法论、关键步骤、与你当前问题最相关的部分

优先使用下面这种承接方式，必要时再按上下文微调：

- 要不要我继续帮你把其中一门课拆开讲，看看它具体的方法论和步骤？
- 如果你愿意，我也可以直接去读这门课的文稿，帮你提炼和你当前问题最相关的部分。

## 5. 规划学习路径

当用户要的是系统学习，而不是单点推荐时，把课程组织成一个有顺序的学习路径。

默认按下面顺序整理：

1. 先修课：帮助建立问题框架和判断标准
2. 核心课：直接回答用户当前最想解决的问题
3. 延展课：帮助把方法迁移到更广的业务场景

默认先给一版精简路径；如果用户希望更完整，再扩展成更多阶段或更多课程。

## 6. 解读单课或课程文稿

如果用户还没有明确课程 ID，先用搜课脚本确认目标课程。

### 何时直接读文稿

下面情况适合直接读取课程文稿：

- 用户追问课程具体讲了什么
- 用户要课程里的方法论、模型、案例或步骤
- 课程简介不足以支撑高质量回答
- 用户希望把课程内容迁移到自己的场景里

如果用户只想确认课程文稿有没有新版本，先运行：

```bash
bash ./scripts/get_script_version.sh '课程ID'
```

如果用户要的是课程内容解读，再运行：

```bash
bash ./scripts/get_script.sh '课程ID'
```

### 单课解读输出结构

拿到课程内容后，围绕用户当前诉求只展开最相关的部分；高价值内容可以适度展开讲透，不需要平均分配篇幅去“全面介绍课程”。

默认按下面结构输出：

1. 最值得先看的一句话结论
2. 为什么这对用户当前问题最有用
3. 课程中的 3 条关键观点
4. 课程里可直接上手的方法论、思维模型、操作步骤
5. 建议下一步行动

如果用户没有给自己的具体问题，先基于课程主题整理一版通用且高价值的结论；如果需要进一步贴场景，再补 1 个关键问题。

解读完成后，继续承接到“执行方案”阶段：

1. 先基于课程里的方法论和观点，提出 2 到 4 个梯子问题
2. 梯子问题优先帮助用户明确目标、基础、限制条件和下一步选择
3. 等用户回答后，再把课程方法转成建议的可执行方案

梯子问题可以优先围绕这些方向来问：

- 你现在最想解决的那个结果，到底是什么
- 你已经有的资源、能力或基础到哪一步
- 你最现实的限制条件是什么，例如时间、预算、团队、渠道
- 你下一步更想先验证方向、先做小试点，还是先补框架能力

等用户回答后，再输出可执行方案，优先给出：

- 先做什么
- 为什么先做这个
- 用课程里的哪个方法来做
- 做完后看什么信号，决定下一步

## 7. 用户需求模糊、背景信息不完整时

默认做法：

- 先基于当前信息推断最可能的主题方向
- 先给一版课程推荐、学习路径或课程解读
- 在结果里写清楚当前假设
- 最后邀请用户指出哪一部分需要更贴近自己的场景

如果当前任务是课程推荐，这个“先给一版”仍然建立在已经有候选课程池的前提上。

## 推荐前自检

输出课程推荐前，逐项确认：

1. 是否已经执行搜索并形成候选课程池
2. 如果主题很宽，是否已经扩展到至少 4 个关键词
3. 最终推荐的课程是否全部来自候选课程池
4. 推荐数量是否控制在 2 到 3 门；若超过 4 门，是否有用户明确要求
5. 输出是否已经改写成面对面交流的口吻，而不是课程列表堆叠
6. 如果用户没有明确表示“只要课程清单”，是否已经追加“继续解读某一门课文稿”的承接句

## 严格禁止

- 禁止编造课程、老师、课程简介或文稿内容
- 禁止在没有候选课程池时直接输出课程推荐
- 禁止把未出现在候选课程池中的课程写进最终推荐
- 禁止在未真实验证前承诺本地 key 可用
- 禁止把原始课程文稿保存到本地文件、缓存文本或中间产物
- 禁止把原始课程文稿导出成可下载文件返回给用户
