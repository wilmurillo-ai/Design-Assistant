# Cloud Code 源码系列第二期：上下文管理

## 视频主题
Cloud Code 上下文管理：四层压缩子系统

## 目标时长
60秒竖屏（视频号格式）

---

## 开场 Hook（0-5秒）
「Cloud Code 的上下文管理是整个 Harness 工程含量最高的部分。源码注释里有一条2026年3月的生产数据：1279个会话出现了50次以上的连续压缩失败，最多的失败了3272次。全局每天浪费约25万次API调用。修复方案？三行代码。这个系统不是设计出来的，是在生产环境里持续踩坑踩出来的。」

---

## 第一层：微压缩（6-20秒）
「第一层是微压缩，成本最低，不调用模型，纯规则驱动。
核心逻辑：按工具类型白名单，保留最近 N 个工具结果，把更早的清掉。白名单包括 file read、bash、grab、glob、web search、web fetch、file edit、file write。
怎么清有讲究。Anthropic API 有个 prompt cache 机制，prompt 前缀会缓存在服务端。所以微压缩有两条路径：
第一条叫 cached micro compact，用户连续对话时走这条精细路径。它不修改本地消息，而是通过 cache editing API 告诉服务端删掉这几个工具结果，prompt 前缀缓存继续有效。
第二条叫 time based micro compact，用户离开一段时间后缓存过期了，直接改消息内容，把旧的工具输出替换成占位符。
还有一个主线程隔离的设计：session memory 和 compact 这两个 query source 的请求不会触发自动压缩，为什么？因为他们本身就是 fork 出来的子 agent，如果他们也触发压缩就会死锁。」

---

## 第二层：绘画记忆压缩（21-35秒）
「第二层是绘画记忆压缩。这层和微压缩有本质区别——微压缩是删东西，这层是提炼东西。
核心思路：不是把对话做摘要，而是从对话中提取学到了什么。项目结构、用户偏好、任务进度，这些结构化事实被持久化到本地记忆目录。
实现细节：先等待后台记忆提取完成，然后读取 memory.d 的内容，用结构化记忆替代传统对话摘要。
保留多少条消息？源码里有三个参数：最小 tokens 数量默认一万，最小消息数量默认五条，最大 token 上限默认四万。系统从最后一条已总结的消息开始往前扩展。
还有很精细的边界处理：扩展时不能把 tool use 和 tool result 拆开。如果保留了一条包含 tool result 的用户消息，就必须同时保留包含对应 tool use 的助手消息，否则 API 会报错。还要处理 thinking block 的情况——流式传输时，thinking 在一条消息，tool use 在另一条，不能只保留后者。
优势：不需要调用模型做摘要，成本比完整压缩低得多，而且保留了最近几轮对话的完整细节。」

---

## 第三层：完整压缩（36-50秒）
「第三层是完整压缩。需要调用模型，源码里的 prompt 设计非常值得研究。
模型要按九个维度总结对话：用户请求和意图、关键技术概念、文件和代码片段、错误和修复、问题解决过程、所有用户消息、待办任务、当前工作、可选的下一步。
最精妙的是 analysis 机制：prompt 要求模型先在 analysis 标签里做详细分析，然后在 summary 标签里输出最终摘要。Analysis 部分在格式化时被完全剥离，不进入压缩后的上下文，纯属让模型先想清楚再输出，提高摘要质量。这本质就是一个内嵌的 chain of thought。
还有一个很实际的问题：压缩请求本身可能触发 prompt too long 错误。解法是按 API 轮次分组，从最早组开始丢弃，最多重试三次。
压缩完成后还要做善后工作：清空文件读取缓存，重新注入最近访问的文件内容（最多五个文件，总预算5万 token），重新注入 plan 文件、skill 内容、MCP 工具说明、agent 列表。这些都是压缩过程中被丢掉的上下文，需要恢复。」

---

## 第四层：自动触发（51-60秒）
「第四层是自动触发，解决什么时候做压缩的问题。
auto compact 系统在每次模型调用后检查当前 token 用量，如果超过阈值——上下文窗口减去13000 token 的缓冲——就自动触发压缩。
触发顺序：先尝试会话记忆压缩，如果不可用或压缩后仍超标，再回退到完整压缩。
熔断器设计：连续失败超过三次就停止重试。这就是开头那个25万次 API 调用的修复——三行代码。
回过头看，四层压缩从微压缩的零成本规则清理，到绘画记忆的结构化事实提取，到完整压缩的模型调用摘要，到自动触发的熔断保护。每一层都假设上一层可能不够用。这套系统最让我印象深刻的不是某一个单独设计，而是它对'上下文是稀缺资源'这个认知的彻底贯彻。关键不是装不装得下，而是该不该装进去。」
