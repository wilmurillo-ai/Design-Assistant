# Cloud Code 上下文管理 - 精简版

## 目标：55秒

## 精简脚本

Cloud Code 的上下文管理是整个 Harness 工程含量最高的部分。源码注释里有一条2026年3月的生产数据：1279个会话出现50次以上的连续压缩失败，最多的失败3272次。全局每天浪费约25万次API调用。修复方案三行代码。这个系统不是设计出来的，是在生产环境里持续踩坑踩出来的。

第一层微压缩，成本最低，纯规则驱动。按工具类型白名单保留最近结果。怎么清有讲究——Anthropic API 有 prompt cache 机制，连续对话走 cached micro compact 精细路径，不动缓存只删工具结果；缓存过期后走 time based micro compact，直接替换成占位符。

第二层绘画记忆压缩，不删东西而是提炼东西——从对话中提取项目结构、用户偏好、任务进度，持久化到本地记忆目录。不需要调用模型，成本比完整压缩低得多。

第三层完整压缩，需要调用模型。要求模型按九个维度总结对话。最精妙的是 analysis 机制——模型先在 analysis 标签里分析，再在 summary 标签里输出摘要，本质就是内嵌的 chain of thought。

第四层自动触发，每次模型调用后检查 token 用量，超过阈值就触发压缩。熔断器设计：连续失败超过三次就停止重试。

关键不是装不装得下，而是该不该装进去。上下文是稀缺资源，需要主动管理。管理的好，agent 可以数小时会话保持连贯。管理的不好，25万次 API 调用就这么烧掉了。
