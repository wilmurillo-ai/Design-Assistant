# Cloud Code 上下文管理 - 精炼版

## 目标：60秒竖屏视频

## 精炼脚本（约900字，适配TTS约160字/分钟）

Cloud Code 的上下文管理是整个 Harness 工程含量最高的部分。源码注释里有一条2026年3月的生产数据：1279个会话出现了50次以上的连续压缩失败，最多的失败了3272次。全局每天浪费约25万次API调用。修复方案？三行代码。这个系统不是设计出来的，是在生产环境里持续踩坑踩出来的。

第一层是微压缩。成本最低，不调用模型，纯规则驱动。按工具类型白名单保留最近 N 个工具结果。怎么清有讲究——Anthropic API 有个 prompt cache 机制，prompt 前缀会缓存在服务端。所以微压缩有两条路径。第一条 cached micro compact，连续对话时走精细路径，通过 cache editing API 告诉服务端删工具结果，缓存继续有效。第二条 time based micro compact，缓存过期后直接改消息内容，把旧工具输出替换成占位符。

第二层是绘画记忆压缩。和微压缩的本质区别是——不是删东西，是提炼东西。不是把对话做摘要，而是从对话中提取学到了什么：项目结构、用户偏好、任务进度。这些结构化事实被持久化到本地记忆目录。它不需要调用模型，成本比完整压缩低得多。

第三层是完整压缩。需要调用模型，prompt 要求模型按九个维度总结对话：用户请求、关键技术概念、文件代码片段、错误修复、解决过程、待办任务。最精妙的是 analysis 机制——要求模型先在 analysis 标签里做详细分析，然后在 summary 标签里输出摘要。Analysis 部分在格式化时被完全剥离，纯属让模型先想清楚再输出，本质就是内嵌的 chain of thought。

第四层是自动触发。系统在每次模型调用后检查 token 用量，超过阈值就自动触发压缩。触发顺序：先尝试会话记忆压缩，不可行再回退完整压缩。还有熔断器设计——连续失败超过三次就停止重试。这就是那25万次 API 调用的修复。

关键不是装不装得下，而是该不该装进去。每一层都假设上一层可能不够用。管理的好，agent 可以在数小时会话中保持连贯。管理的不好，25万次 API 调用就这么烧掉了。
