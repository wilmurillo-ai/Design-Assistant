# Bug 报告标准模板

## 标题格式

```
[模块名] 简短描述现象（动词 + 对象 + 结果）
```

**示例：**
- `[登录] 输入正确密码后提示"账号或密码错误"`
- `[订单] 下单成功后库存未扣减`
- `[用户中心] 头像上传超过 2MB 时页面白屏`

---

## Bug 报告完整模板

```markdown
## Bug 标题
[模块] 现象描述

## 基本信息
| 字段 | 内容 |
|------|------|
| **严重程度** | P0 Blocker / P1 Critical / P2 Major / P3 Minor |
| **优先级** | 高 / 中 / 低 |
| **发现版本** | v1.2.3 |
| **测试环境** | 测试环境 / 预发环境 / 生产环境 |
| **操作系统** | Windows 11 / macOS 14 / iOS 17 / Android 14 |
| **浏览器/App版本** | Chrome 123 / Safari 17 / App v2.1.0 |
| **发现人** | 张三 |
| **发现时间** | 2026-03-30 |
| **指派给** | 开发负责人姓名 |

## 前置条件
> 复现此 Bug 前必须满足的环境或数据状态

- 用户已注册，账号状态正常
- 已登录系统
- ...

## 复现步骤
1. 打开 xxx 页面（URL: https://...）
2. 执行操作 A
3. 执行操作 B
4. 观察结果

## 实际结果
> 描述实际看到的现象，要具体可观测

页面显示"账号或密码错误"弹窗，停留在登录页，未跳转。

## 预期结果
> 描述正确行为应该是什么

登录成功，跳转至首页，顶部显示用户昵称。

## 复现频率
- [x] 必现（100%）
- [ ] 偶现（约 50%）
- [ ] 难以复现（< 10%）

## 附件
- [ ] 截图（标注问题区域）
- [ ] 视频录屏
- [ ] 接口请求/响应截图（Charles / DevTools）
- [ ] 日志文件

## 根因假设（可选）
> 测试人员对问题原因的初步判断，供开发参考

可能是登录接口密码加密方式与数据库存储格式不一致。

## 备注
其他需要说明的信息。
```

## 严重程度（Severity）定义

| 级别 | 名称 | 定义 | 示例 |
|------|------|------|------|
| **P0** | Blocker | 核心功能完全不可用，阻塞测试，或导致数据丢失/安全漏洞 | 无法登录、支付接口报错、用户数据泄露 |
| **P1** | Critical | 主要功能严重损坏，无可接受的绕过方案 | 下单后订单状态错误、搜索功能返回空结果 |
| **P2** | Major | 功能部分损坏，存在可接受的临时绕过方案 | 筛选条件无效但可手动翻页找到数据 |
| **P3** | Minor | 界面问题、文案错误、低影响 UX 缺陷 | 按钮颜色偏差、提示语错别字 |

## Bug 生命周期流转

```
新建(New) → 已分配(Assigned) → 修复中(In Progress) → 待验证(Resolved)
  → 验证通过(Closed)
  → 验证不通过(Reopened) → 修复中...
  → 无法复现(Cannot Reproduce)
  → 不予修复(Won't Fix) → 关闭前需说明理由
```

---

## 好 Bug 报告的标准

1. **可复现** — 步骤精确，任何人按步骤操作都能看到同样结果
2. **一个 Bug 一个单** — 不要把多个问题混在同一条 Bug 里
3. **有对比** — 实际结果 vs 预期结果，差异清晰
4. **有证据** — 截图/录屏/日志，让开发不用猜
5. **标题明确** — 看标题就知道问题在哪、是什么现象
