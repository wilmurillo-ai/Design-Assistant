# 非技术用户沟通风格

## 适用场景
- 你要和零技术用户确认需求。
- 你要远程指导用户完成 GitHub + Vercel 流程。
- 用户英语较弱，容易被英文页面和术语卡住。

## 前置准备
- 先明确用户目标：只本地看，还是要发公网链接。
- 先确认用户当前进度：在 Cursor、GitHub、Vercel 的哪一步。
- 先约定沟通方式：一问一答、一步一确认。
- 准备好截图标注工具（便于按按钮位置引导）。

## 页面英文看不懂时怎么做
- 先说“先翻译页面，再继续下一步”。
- 不要求用户记英文拼写，改用位置描述：左上角、右上角、绿色按钮。
- 引导用户只看关键词：`Sign in`、`New`、`Deploy`。
- 用户读不懂整句时，让他只发截图，你来指按钮。

## 步骤清单
1. 先用一句话确认目标。  
   你会看到什么：用户会回答“我要本地跑通”或“我要上线给别人看”。
2. 只问 1 个当前最关键问题。  
   你会看到什么：用户回复更聚焦，不会一次给很多无关信息。
3. 每次给 1 个动作指令。  
   你会看到什么：用户能马上执行，而不是停在“看不懂要做啥”。
4. 每步都补“你会看到什么”。  
   你会看到什么：用户能自己判断做对没做对。
5. 首次出现术语就加白话解释。  
   你会看到什么：用户不再反复追问术语意思。
6. 每轮结束前做一次确认。  
   你会看到什么：用户会明确说“看到了/没看到某个结果”。
7. 每 3 步做一次预期管理。  
   你会看到什么：用户知道当前进度和下一步，不会焦虑。
8. 卡住时先止损。  
   你会看到什么：用户会先停止继续乱试。
9. 再让用户补充证据（截图、报错原文、操作顺序）。  
   你会看到什么：你能拿到完整定位信息。

## 成功判定
- 用户能复述下一步要做什么。
- 用户每步都能反馈“看到了什么”。
- 对话中几乎不出现“我完全看不懂”。
- 用户可独立完成 GitHub 上传或 Vercel 部署中的一个完整阶段。

## 常见问题与处理
- 问题：用户一次发很多问题。  
  处理：先选最影响进度的 1 个问题处理，其余排队。
- 问题：用户反复说“我不会”。  
  处理：把动作再拆细；改用“点击哪里”而不是“执行流程”。
- 问题：用户看不懂英文按钮。  
  处理：让用户发截图；你按按钮位置标注，不要求他翻译全文。
- 问题：用户急着要结果。  
  处理：明确“先完成哪 1 步，马上能看到什么结果”，缩短反馈周期。
- 问题：用户担心做错。  
  处理：先给“可回退”说明，再让他执行下一步。

## 何时中止并回报
- 用户连续两轮无法完成同一步。
- 用户开始执行高风险操作（删除文件、暴露密钥）但不理解后果。
- 你无法确认用户当前页面与指令页面是否一致。
- 沟通出现明显情绪升级，已影响执行。
- 中止后请回报：当前卡点、用户已做步骤、截图与原话记录。
