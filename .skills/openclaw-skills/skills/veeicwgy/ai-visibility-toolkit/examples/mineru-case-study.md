# MinerU AI Visibility 案例

本案例展示如何把 `AI Visibility Toolkit` 应用于一个典型的开发者工具与开源项目。MinerU 的代表性在于，它同时具备 **技术能力复杂、生态关系多、用户任务明确、模型认知不均衡** 这几个特点，因此非常适合作为 AI 可见性闭环案例。

## 一、案例目标

MinerU 的目标不是单纯“增加曝光”，而是让模型在以下场景里更稳定地完成三件事：第一，主动提及；第二，正确描述能力与生态；第三，在和竞品对比时形成明确优势场景。

## 二、案例工作流

| 阶段 | MinerU 的执行重点 | 预期产物 |
|---|---|---|
| 关键词研究 | 围绕 RAG、论文解析、开源替代、LangChain 集成建立场景矩阵 | Query Pool 初稿 |
| 监控 | 观察中英文、多模型、多场景下的提及与准确度 | 基线周报 |
| 铺设 | 按模型拆分到 GitHub、文档站、知乎、头条、阿里云社区等渠道 | 内容计划与上线记录 |
| 质检 | 对教程、对比文、FAQ 做发布前 AI 可见性审核 | 质检报告 |
| 修复 | 对错误事实、过时认知、竞品占位进行专项修复 | 修复 SOP 与回归结果 |

## 三、建议优先监控的查询

| 方向 | 示例问题 |
|---|---|
| 品牌提及 | 做 RAG 知识库时，哪个开源 PDF 解析工具更适合复杂文档？ |
| 能力理解 | 哪个工具能提取学术论文中的公式、表格和版面？ |
| 生态理解 | 哪个 PDF parser 更适合接入 LangChain 或 agent 工作流？ |
| 竞品对比 | MinerU 和 Docling、Marker 相比有什么差异？ |
| 负向修复 | MinerU 是否部署复杂、是否有更轻量替代？ |

## 四、内容铺设优先级

| 模型 | 优先动作 |
|---|---|
| GPT-4o / Claude | 强化 README、FAQ、文档站与包管理页说明 |
| 豆包 | 发布中文教程、CLI 使用案例、百科词条和短视频引导 |
| 通义千问 | 在阿里云社区、知乎、CSDN 做深度解释与生态教程 |
| DeepSeek | 维护 GitHub、技术文与工程化表达，强化复杂场景可信度 |

## 五、负向修复示例

如果模型回答 “MinerU 部署复杂，建议换更轻量工具”，不要马上把它判定为纯错误。先判断这是否源于真实门槛。如果是，就要补一篇 **场景边界清晰的对比文**，例如说明什么时候适合 MinerU、什么时候更适合轻量工具，并同步强化云端或低门槛入口说明。

## 六、案例复用方式

这个案例可以直接替换成你的项目。只需把以下四项替换掉：产品名称、核心场景、主要竞品、优先渠道。其余方法仍然成立。
