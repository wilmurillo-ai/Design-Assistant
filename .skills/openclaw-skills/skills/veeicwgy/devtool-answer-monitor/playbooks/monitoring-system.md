# AI 可见性监控体系

AI 可见性监控的目标不是简单统计“有没有被提到”，而是建立一个能指导内容动作和修复动作的观察系统。对于技术产品，至少要同时跟踪 **提及、情感、能力理解、生态理解** 四个维度，否则你很难知道问题究竟出在品牌曝光、事实准确度，还是生态认知缺失。

## 四大核心指标

| 指标 | 核心问题 | 定义 | 观察重点 |
|---|---|---|---|
| 提及率 | 模型会不会主动推荐你？ | 无品牌通用查询中出现品牌名的比例 | 是否被纳入默认候选 |
| 正面提及率 | 提到你是在帮你还是害你？ | 提及中属于正面推荐的比例 | 是否带有推荐词、理由和偏好 |
| 能力准确率 | 模型是否理解你的核心能力？ | 能力类问题中回答正确的比例 | 是否说对功能、边界和适用场景 |
| 生态准确率 | 模型是否知道你的集成生态？ | 生态类问题中回答正确的比例 | 是否理解 SDK、框架、上下游关系 |

## 监控对象建议

建议同时覆盖国内外模型，以便区分“语料差异”与“渠道差异”。

| 类别 | 建议覆盖模型 |
|---|---|
| 国内 | 通义千问、豆包、DeepSeek、Kimi、文心一言、智谱、腾讯元宝 |
| 国外 | GPT-4o、Claude、Gemini、Perplexity、Copilot / Bing Chat |

## 情感极性判定标准

回答分类必须统一，否则同一条回答会被不同执行者打出不同标签。

| 极性 | 判定条件 | 典型信号 | 处置方式 |
|---|---|---|---|
| 正面 | 明确推荐，说明理由，给出代码或证据 | “推荐”“首选”“更适合” | 追溯驱动该判断的信源 |
| 中性 | 仅被列举，无明显倾向，或带条件推荐 | 与竞品并排出现，但没有偏向 | 补充差异化优势与对比信息 |
| 负面 | 建议替代、指出限制、事实错误、警示性语言 | “部署复杂”“不适合”“建议改用” | 立即进入负向修复流程 |

## 扩展维度

四大指标之外，建议增加几个用于优先级判断的扩展维度。

| 维度 | 含义 | 用途 |
|---|---|---|
| AI Share of Voice | 你的提及次数占竞品总提及的比例 | 判断是否被竞品压制 |
| Query Coverage | Query Pool 中出现过品牌的查询覆盖率 | 判断覆盖面是否不足 |
| Mention Position Weight | 首位、正文中段、末尾、带代码等不同位置的权重 | 区分“提到”和“主推” |
| Response Stability | 同一查询在不同时间输出是否一致 | 判断模型认知是否稳定 |
| Entity Accuracy | 模型是否分清产品名、组织名、域名、仓库名 | 判断实体混淆问题 |
| Time-to-Citation | 内容发布后多久开始被模型引用 | 判断内容生效周期 |

## 周报模板

每周监控至少输出三个部分：总体指标、问题分布、行动优先级。

| 周报板块 | 必须包含内容 |
|---|---|
| 总览 | 各模型四大指标、同比变化、异常模型 |
| 重点问题 | 负向回答、错误事实、缺失生态、竞品占位 |
| 行动建议 | 本周应优先铺设的内容、应修正的信源、应回归验证的查询 |

## 打标建议

Evidence Log 建议为每一条回答至少打上以下字段：

| 字段 | 说明 |
|---|---|
| model | 哪个模型 |
| query_id | 对应哪条 Query Pool |
| language | 语言 |
| mention | 是否提及品牌 |
| sentiment | 正面 / 中性 / 负面 |
| capability_accuracy | 正确 / 部分正确 / 错误 |
| ecosystem_accuracy | 正确 / 部分正确 / 错误 |
| competitor_names | 出现了哪些竞品 |
| source_links | 引用了哪些链接 |
| action_needed | 是否需要铺设或修复 |
