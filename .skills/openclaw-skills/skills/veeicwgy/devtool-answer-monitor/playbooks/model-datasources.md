# 模型数据来源指南

模型并不会从同一个地方学习内容。对 AI 可见性来说，**先判断模型从哪里学习，再决定去哪里铺内容**，比盲目扩写文章更重要。尤其是开发者工具与开源项目，很多有效信源并不在传统营销站，而是在代码、文档、知识库和技术社区。

## 使用方式

本文件不把下表视为绝对事实表，而是作为一张 **运营决策地图**。每个模型的数据来源会随着版本更新、联网能力和产品策略发生变化，因此应按月回顾，而不是长期固化。

## 技术产品的高权重信源清单

| 信源 | 作用 |
|---|---|
| GitHub README | 决定模型能否快速理解项目定位、License、Quick Start、生态集成 |
| 官方文档站 | 决定能力边界、版本信息、FAQ、结构化引用质量 |
| 包管理页 | 如 PyPI、npm、pkg.go.dev，用于建立语言生态可见度 |
| 技术博客 | 用于承接场景词、对比词、教程词和版本更新词 |
| 问答社区 | 用于积累问题表达与可检索讨论，如知乎、Stack Overflow、CSDN |
| 百科与知识页 | 用于统一品牌定义、组织关系和权威概念说明 |

## 三个判断原则

### 原则一：优先修正权威源而不是复制内容

如果模型错误来自 README、百科、包管理页或高权重教程，优先修正源头，而不是先多发十篇外围内容。

### 原则二：优先建设“被抽取友好”的页面

模型更容易学习和复述结构化内容。FAQ、对比表、参数表、版本更新说明、明确的实体关系图，通常比纯情绪化长文更容易被引用。

### 原则三：按模型分配预算

不同模型需要不同内容预算。对于 GPT / Claude / DeepSeek，一条高质量 README 更新可能比十篇碎片化内容更有效；对于豆包、通义、元宝，则中文平台投放的重要性会明显上升。
