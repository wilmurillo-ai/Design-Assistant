# AI 可见性全链路工作流架构

本 playbook 用于把 AI 可见性工作从零散的内容动作升级为 **可监控、可干预、可回归验证** 的系统。对于开发者工具与开源项目，这个系统尤其重要，因为模型是否推荐一个技术产品，常常取决于其是否在技术信源、代码生态、文档站和社区讨论中被持续、准确地表达。

## 闭环总览

```text
产品定位与实体真相
        ↓
关键词研究与场景矩阵
        ↓
Query Pool 设计
        ↓
多模型监控与打分
        ↓
问题归因（缺失 / 中性 / 错误 / 负向 / 竞品占位）
        ↓
内容铺设与权威覆盖
        ↓
发布前 AI 可见性质检
        ↓
负向修复与信源纠偏
        ↓
回归监控与周报复盘
```

## 五个执行阶段

| 阶段 | 核心动作 | 判断标准 | 典型输出 |
|---|---|---|---|
| Foundation | 明确产品定位、用户任务、核心实体 | 是否知道用户会问什么、模型该如何正确描述你 | 产品真相表、场景列表 |
| Research | 建立关键词矩阵与 Query Pool | 监控语句是否覆盖品牌、能力、生态、竞品、负向场景 | 关键词矩阵、Query Pool |
| Observe | 跑多模型监控并统一评分 | 是否同时看到提及、情感、能力理解、生态理解 | 周报、异常清单 |
| Intervene | 按模型数据来源铺设内容并做质检 | 内容是否进入高权重信源、是否足够可引用 | 渠道计划、质检记录 |
| Repair | 对负向内容做类型化修复并回归验证 | 修复动作是否落到信源、是否带来指标改善 | 修复 SOP、回归结果 |

## 统一对象模型

AI 可见性工作流建议围绕四个对象协同运行。

| 对象 | 定义 | 谁来维护 |
|---|---|---|
| Entity Truth | 产品的标准事实集合，如名称、版本、License、能力边界、生态关系 | 产品团队 |
| Query Pool | 用来反复监控 LLM 回答的查询集合 | 增长 / 内容 / 产品 |
| Evidence Log | 模型回答、引用链接、截图、时间戳、标签 | 监控执行人 |
| Action Backlog | 待执行的铺设动作、修复动作、验证动作 | 项目 owner |

## 核心判断逻辑

当模型回答不理想时，不要马上继续发文章。先判断问题属于哪一类。

| 问题类型 | 典型表现 | 正确动作 |
|---|---|---|
| 未提及 | 模型完全不说你 | 扩 Query Pool、做高权重信源铺设 |
| 中性提及 | 提到了但没推荐 | 强化对比、数据、场景边界与代码示例 |
| 信息错误 | License、部署方式、能力说错 | 找错误信源并覆盖权威事实 |
| 负面评价 | 推荐竞品并贬低你 | 承认限制，重塑适用场景，建设替代路线 |
| 竞品占位 | 相关查询只出现竞品 | 提升竞品对比词与替代词覆盖 |

## 运营节奏建议

建议按周运行监控闭环，按月回顾信源策略，按季度刷新关键词矩阵。对于刚启动 AI 可见性项目的团队，前四周应优先完成基线建立，而不是追求立刻扩量。

| 周期 | 必做事项 |
|---|---|
| 每周 | 抽样运行 Query Pool，生成四维指标周报，更新异常清单 |
| 每两周 | 执行重点内容铺设与负向修复回归验证 |
| 每月 | 更新模型来源判断、渠道优先级、实体真相表 |
| 每季度 | 重做关键词研究与 Query Pool 扩容 |
