## 3分钟选型表（流程与BPR模块）

| 场景 | 优先方法 | 何时升级 |
|---|---|---|
| 已有明确目标，需分解动作与预防风险 | 方法20 系统展开图 | 出现跨部门断点时升级到方法22 |
| 原因链复杂，需分层找突破口 | 方法21 树图+强度分析 | 涉及流程重构时升级到方法22 |
| 流程重复/错误/遗漏多，需快速优化 | 方法22 流程再造体系（主卡） | 若涉及组织重构，调用方法23/24子卡 |
| 需公司级流程与组织一体重构 | 方法23 BPR（子卡A） | 若一线执行脱节，叠加方法24 |
| 需先从一线作业单元改起 | 方法24 Bottom-up BPR（子卡B） | 成熟后回归方法23做全局定案 |
| 已有改造方案，需安排关键路径与工期 | 方法29 CPM（子卡C） | 若执行偏差频发，叠加PDCA周复盘 |
| 需要统一“计划-执行-检查-评审”闭环 | 方法30 PDCA（执行主闭环） | 若计划编排复杂，叠加方法29 |

### 方法20：系统展开图（目标反推与预防性设计）

来源要点（PPT）：
- 确定最终目标，预先分析可能存在不良。

#### 1) 方法原理与适用边界

- 先设计后执行：`目标 -> 路径 -> 风险 -> 验证`。
- 适用于工艺优化、质量提升、复杂改进方案。

#### 2) 专家级框架

1. 目标层：量化目标。
2. 策略层：2-5条主路径。
3. 动作层：每条路径拆动作。
4. 验证层：每个动作设置放行标准。

#### 3) 最小模板

```markdown
【目标】<指标/时间/口径>
【路径】<策略1..n>
【动作】<动作+责任人>
【验证】<测试方式+通过标准>
```

### 方法21：树图 + 阻碍强度分析（根因分层与优先突破）

来源要点（PPT）：
- 将问题层层分解，找出最根本原因，并做强度排序。

#### 1) 方法原理与适用边界

- 树图负责“分层”，强度分析负责“优先”。
- 适用于多因叠加、跨部门问题。

#### 2) 专家级框架

1. 问题层
2. 大原因层
3. 小原因层
4. 策略层
5. 强度评估（影响/紧迫/可控）

#### 3) 最小模板

```markdown
【问题】<问题+影响>
【分层】<大原因/小原因>
【评分】<影响/紧迫/可控>
【Top10突破】<原因+动作+负责人+里程碑>
```

### 方法22：流程再造体系（主卡：牛皮纸法 + BPR）

来源要点（PPT）：
- 牛皮纸法：现状流程可视化与问题标注。
- BPR：工作机能整合与流程重排，形成新组织承接。
- Bottom-up：从作业单元上卷，完成标准化落地。

#### 1) 方法原理与适用边界

- 这是流程改进的主干方法，覆盖三层：
  - 流程优化（局部）
  - 流程重构（跨部门）
  - 组织承接（公司级）
- 适用：流程冗余、接口扯皮、变更落地难。

#### 2) 统一实施路径（8步）

1. 绘制现状流程（含时间、责任、输入输出）。
2. 标注问题点（重复、错误、遗漏、等待、返工）。
3. 量化影响（成本、时效、质量、客户）。
4. 执行四类动作：合并、删除、重组、简化。
5. 形成目标流程并设置风险补偿控制点。
6. 评估是否需要组织重构：若需要调用方法23。
7. 评估是否需一线先行：若需要调用方法24。
8. 纳入`CPM + PDCA`实施与复盘。

#### 3) 常见误用与纠偏

- 只改图不改职责：同步更新RACI。
- 只删步骤不控风险：补充控制点与回滚方案。
- 流程改了系统不改：制度/系统/培训同步上线。

#### 4) 主卡交付模板

```markdown
【现状流程】<边界/角色/关键节点>
【问题清单】<重复/错误/遗漏/等待>
【改造动作】<合并/删除/重组/简化>
【目标流程】<端到端说明+控制点>
【实施计划】<里程碑/责任人/指标>
```

### 方法23：BPR子卡A（组织-流程一体重构）

#### 1) 何时调用

- 需要跨部门重排职责、调整组织架构、重设协同接口时。

#### 2) 关键动作

1. 现状组织-流程全景图。
2. 机能整合与RACI重定义。
3. 分阶段切换（试点->扩面->全量）。

#### 3) 子卡模板

```markdown
【现状诊断】<纵向/横向/组织/流程问题>
【重构方案】<机能整合+组织调整>

```

### 方法24：BPR子卡B（Bottom-up由下而上改造）

#### 1) 何时调用

- 纸面流程与一线执行脱节，需要从作业单元起步时。

#### 2) 关键动作

1. 作业单元盘点（Xij/Yij）。
2. 作业级两轮“比对-调整”。
3. 上卷至机能层与组织层定案。
4. 输出SOP/DP并持续迭代。

#### 3) 子卡模板


### 方法29：最佳路线分析法（CPM）子卡C（关键路径排程）

关系判定（新增协议）：
- 类型：`细化方法`（用于既有方案的实施排程，不单独构成新主链路）。
- 放置：E模块（流程与BPR），作为流程改造执行阶段子卡。

来源要点（PPT）：
- 通过工作项目、时间轴、责任人与确认机制，识别关键路径并安排推进顺序。
- 典型任务如：确定岗位职责、完善作业标准、系统人员培训、建立目标管理体系。

#### 1) 何时调用

- 改造动作已明确，但执行顺序、并行关系、工期控制尚不清晰时。

#### 2) 关键动作（6步）

1. 列出全部工作包（WBS最小可交付粒度）。
2. 明确前后依赖关系与可并行任务。
3. 估算每项工期并确定责任人。
4. 计算关键路径与总工期。
5. 标注里程碑与确认点（Gate）。
6. 对关键路径任务设置预警与纠偏机制。

#### 3) 常见误用与纠偏

- 误用：只排时间，不排依赖。
  - 纠偏：无依赖关系的甘特图不等于CPM。
- 误用：关键路径上堆叠高风险任务。
  - 纠偏：关键任务需优先配资源和缓冲。
- 误用：责任人不唯一导致确认失效。
  - 纠偏：每个任务必须唯一Owner并有确认标准。

#### 4) 子卡模板

```markdown
【CPM任务表】
| 任务ID | 工作项目 | 前置任务 | 工期(天) | 责任人 | 确认标准 |
|---|---|---|---:|---|---|
| A | 确定岗位职责 | - | 3 | <人> | <标准> |
| B | 完善作业标准 | A | 5 | <人> | <标准> |
| C | 系统人员培训 | A | 8 | <人> | <标准> |
| D | 建立目标管理体系 | B,C | 4 | <人> | <标准> |

【关键路径】
- 路径：<A -> C -> D>
- 总工期：<xx天>
- 关键里程碑：<M1/M2/M3>
```

### 方法30：管理循环法（PDCA）执行主闭环

关系判定（新增协议）：
- 类型：`独立方法`（定义完整执行闭环与阶段工具映射）。
- 放置：E模块（流程与BPR）主执行方法，并作为跨模块默认闭环。

来源要点（PPT）：
- P（计划）：
  1. 明确问题点（点检表、管制图）
  2. 分析原因（鱼骨图、5WHY、脑图）
  3. 找出主要原因（系统图、柏拉图、5WHY）
  4. 确定改善计划（PERT、CPM、推移图）
- D（执行）：实施行动方案（并行工程、瓶颈分析）
- C（检查）：确认改善效果（质保技术）
- A（评审）：
  7. 标准化制订（IE工具、流程改善、文档管理）
  8. 识别遗留问题（鱼骨图、系统对照分析）

#### 1) 何时调用

- 任一改进项目从“分析”进入“落地”阶段时。
- 需要建立周/月复盘节奏，避免一次性整改后反弹时。

#### 2) 关键动作（8步）

1. P1：定义问题与口径。
2. P2：完成根因分析并证据化。
3. P3：筛主因并设目标值。
4. P4：制定计划（任务、依赖、时间、资源）。
5. D：执行方案并记录偏差。
6. C：对照目标验证效果。
7. A1：将有效做法标准化。
8. A2：识别遗留问题并进入下一轮PDCA。

#### 3) 常见误用与纠偏

- 误用：把PDCA当“周报流程”，缺少量化目标。
  - 纠偏：每轮必须有目标值与验收口径。
- 误用：只做D，不做C/A。
  - 纠偏：C/A是防反弹核心，必须占固定节奏。
- 误用：A阶段只做总结，不做标准化。
  - 纠偏：A阶段必须形成制度/流程/文档更新。

#### 4) 子卡模板

```markdown
【P 计划】
- 问题点：<定义+口径>
- 主因：<主因清单>
- 目标：<目标值+时间>
- 计划：<任务+责任人+依赖+里程碑>

【D 执行】
- 执行动作：<动作记录>
- 偏差：<偏差项+原因>

【C 检查】
- 效果对比：<基线/当前/目标>
- 结论：<达成/未达成>

【A 评审】
- 标准化：<制度/流程/文档更新>
- 遗留问题：<问题+下一轮动作>
```
