## 3分钟选型表（问题定义与结构化模块）

| 场景 | 优先方法 |
|---|---|
| 议题模糊、团队意见分散 | 方法1 是什么-为什么-怎么做 |
| 需要从“工具集合”快速设计诊断链路 | 方法2 六段作战链 |
| 需校准服务方案与客户真实需求匹配 | 方法3 结构图分析法 |

### 方法1：是什么-为什么-怎么做（问题分析总线）

来源要点（PPT）：
- `是什么`：明确要达到的目标。
- `为什么`：明确目标达成的必要性。
- `怎么做`：整合资源，制订切实可行解决方案。

#### 1) 方法原理与适用边界

- 这不是“提问话术”，而是咨询分析的最小闭环：`目标定义 -> 价值证明 -> 实施路径`。
- 适用：
  - 议题模糊、跨部门分歧大、方案争论先于问题定义的场景。
  - 方案评审前的快速对齐场景。
- 不适用：
  - 已有清晰目标和约束，且仅需排期执行时（直接进入CPM/PDCA更高效）。

#### 2) 专家级操作步骤（7步）

1. 用一句话写出`是什么`：目标对象、期望变化、时间边界。
2. 用指标定义目标：至少包含基线、目标值、时间点、口径。
3. 写出`为什么`的三层价值：业务价值、客户价值、组织价值。
4. 把“必要性”量化：不做的损失、晚做的机会成本、错误做法的风险。
5. 写`怎么做`的方案骨架：工作流、资源包、里程碑、责任人。
6. 做可行性校验：资源可得性、组织阻力、依赖项、关键假设。
7. 生成执行版一页纸：问题定义、必要性证据、行动路径、90天里程碑。

#### 3) 常见误用与纠偏

- 误用：把`是什么`写成动作（如“开展培训”）。
  - 纠偏：`是什么`必须是结果状态，不是活动本身。
- 误用：`为什么`只写口号（如“为了增长”）。
  - 纠偏：至少给出1个财务或客户维度证据。
- 误用：`怎么做`只有方向没有约束。
  - 纠偏：必须补齐资源、时间、责任、风险四要素。

#### 4) 诊断问题清单（访谈可直接使用）

- `是什么`：
  - 目标对象是谁？
  - 成功被谁判定？判定标准是什么？
  - 时间窗口是什么？
- `为什么`：
  - 当前痛点若不处理，3个月后会怎样？
  - 该议题与年度目标的连接点是什么？
  - 是否存在监管/客户/竞争倒逼？
- `怎么做`：
  - 关键路径的第一个可验证里程碑是什么？
  - 哪个部门是瓶颈？
  - 最小可行试点（MVP）怎么定义？

#### 5) 交付模板（专家版）

```markdown
【是什么】
- 目标：在<时间>将<对象指标>从<基线>提升至<目标值>，口径为<统计口径>。

【为什么】
- 业务必要性：<收入/成本/效率/风险>影响为<量化值>。
- 客户必要性：<满意度/NPS/留存>影响为<量化值>。
- 组织必要性：与<战略主题/KPI>直接关联。

【怎么做】
- 路径：<阶段1> -> <阶段2> -> <阶段3>
- 资源：负责人<人名/角色>，预算<金额>，系统/数据<依赖项>
- 里程碑：D30、D60、D90
- 风险与预案：<风险1>/<预案1>，<风险2>/<预案2>
```

#### 6) 与其他方法组合策略

- 前置：当议题来源杂乱时，先用`脑图`收敛议题，再进入本方法。
- 后置：
  - 需要根因穿透时，接`鱼骨图 + 5WHY`。
  - 需要优先级排序时，接`ABC分析法`。
  - 需要落地闭环时，接`PDCA + CPM/PERT`。

### 方法2：知识与分析工具总框架（6段作战链）

来源要点（PPT）：
1. 分析思路：`是什么 -> 为什么 -> 怎么做`
2. 分析工具：`脑图、鱼骨图、系统展开图、树图`
3. 分析方法：`5W2H + 28问 + 5WHY`
4. 重点因素界定：`ABC分析法（层别图）`
5. 改善计划制订：`最佳路线分析法（CPM）`
6. 改善行动实施：`管理循环法（PDCA）`

#### 1) 方法原理与适用边界

- 这是“从问题到落地”的完整链路，而不是并列工具清单。
- 本质是六段递进：
  - 定义问题
  - 结构化拆解
  - 深度追因
  - 识别关键少数
  - 制订执行路径
  - 形成持续改善闭环
- 适用：
  - 经营改善、运营提效、质量治理、组织协同类项目。
- 不适用：
  - 纯探索性创新且短期无需落地验证的议题。

#### 2) 专家级操作步骤（12步）

1. 用`是什么-为什么-怎么做`定义问题边界和目标。
2. 用`脑图`穷举影响因子，形成问题全景。
3. 用`鱼骨图`按维度归因，建立因果假设池。
4. 用`树图/系统展开图`分解目标到可执行子任务。
5. 用`5W2H+28问`对每个关键任务做可行性质询。
6. 对高风险环节执行`5WHY`，穿透到机制层根因。
7. 汇总全部问题项，用`ABC`做80/20优先级排序。
8. 将A类问题转成项目包，定义责任人与收益目标。
9. 用`CPM`排定关键路径、里程碑和并行任务。
10. 进入`PDCA-P`阶段，固化计划与验收标准。
11. 执行`PDCA-D/C`，按周复盘偏差和效果。
12. 完成`PDCA-A`，标准化、复制与下一轮优化。

#### 3) 六段作战链交付物（必须产出）

1. 问题定义卡（是什么/为什么/怎么做）
2. 因素全景图（脑图+鱼骨图）
3. 根因说明书（5WHY证据链）
4. 优先级清单（ABC分层表）
5. 项目推进图（CPM关键路径图）
6. 闭环复盘表（PDCA周/月复盘）

#### 4) 常见误用与纠偏

- 误用：直接上PDCA，未完成根因验证。
  - 纠偏：必须先完成`5WHY + ABC`，否则是“忙碌闭环”。
- 误用：把所有问题都列为A类。
  - 纠偏：A类占比应受控，建议不超过问题总数20%。
- 误用：CPM只排时间，不排责任和依赖。
  - 纠偏：每个节点必须含责任角色、前置依赖、交付标准。

#### 5) 诊断问题清单（项目经理视角）

- 当前项目处于六段链哪一段？是否越级跳步？
- 当前结论来自数据证据，还是经验判断？
- A类问题对应的资源是否已锁定？
- 关键路径上是否存在单点依赖？
- PDCA中的`C`与`A`是否有固定节奏和模板？

#### 6) 交付模板（专家版）

```markdown
【阶段1 问题定义】
- 目标：<指标提升目标>
- 必要性：<不做的损失/延迟成本>

【阶段2-3 拆解与追因】
- 脑图结论：<Top5影响因子>
- 鱼骨与5WHY根因：<Top3机制层根因>

【阶段4 优先级】
- A类问题：<问题1/问题2/...>
- 选择依据：<影响度x紧迫度x可控度>

【阶段5 执行计划】
- CPM关键路径：<里程碑1 -> 里程碑2 -> 里程碑3>
- 责任与资源：<Owner/预算/系统支持>

【阶段6 闭环治理】
- PDCA节奏：周检视、月复盘、季评审
- 标准化输出：流程、制度、看板、培训
```

#### 7) 与其他方法组合策略

- 前置增强：
  - 战略议题先加`PESTEL/五力/7S`，再进入六段作战链。
- 后置增强：
  - 方案稳定后接`BSC`，将行动映射为长期指标治理。

### 方法3：结构图分析法（服务设计与客户事实对照）

来源要点（PPT）：
- 左侧（我方）：我们服务的切入点、提供什么服务、打击点、服务项目是否合理、服务保障是什么。
- 右侧（客户）：客户现状、客户需求、客户最感兴趣点、客户需求绩效、客户为何信任我们。
- 原则：`事实为基础、客户为导向`，反对`假设为基础、自我为导向`。

#### 1) 方法原理与适用边界

- 本方法本质是“供需镜像校准”：
  - 左边是我方设计意图。
  - 右边是客户真实约束与价值判断。
- 只有左右一一映射且证据闭合，方案才具备成交与落地可能。
- 适用：
  - 咨询售前诊断、方案论证、项目中期纠偏、复购评估。
- 不适用：
  - 纯内部流程优化且无外部客户界面的场景（可简化为内部价值链分析）。

#### 2) 专家级操作步骤（10步）

1. 先写右侧客户事实（现状、需求、兴趣点、绩效、信任依据）。
2. 每条客户事实都必须绑定证据源（访谈、数据、现场观察、第三方）。
3. 再写左侧我方设计（切入点、服务包、打击点、项目合理性、保障机制）。
4. 对每一行做双向映射：`我方动作 -> 客户价值` 与 `客户需求 -> 我方响应`。
5. 标记“断链项”：左侧有动作但右侧无价值证据，或右侧有需求但左侧无响应。
6. 给每个断链项定义修复策略：删除、替换、补证据、补能力。
7. 对“客户最感兴趣点”做优先级排序（影响度x可感知度x可交付性）。
8. 将“客户需求绩效”转成可验收指标（口径、频率、阈值、责任人）。
9. 建立“信任机制包”：案例、方法论、专家履历、质控机制、里程碑承诺。
10. 输出“结构图评审版一页纸”，作为提案与启动会共同底稿。

#### 3) 五组核心对照关系（必须逐项检查）

1. `我方切入点` <-> `客户现状`
2. `我方服务内容` <-> `客户需求`
3. `我方打击点` <-> `客户最感兴趣点`
4. `我方项目合理性` <-> `客户需求绩效`
5. `我方服务保障` <-> `客户信任来源`

#### 4) 常见误用与纠偏

- 误用：先写“我们能做什么”，再去找客户理由。
  - 纠偏：必须先写客户事实，再反推我方设计。
- 误用：把“客户感兴趣”误判为“客户会付费”。
  - 纠偏：兴趣点必须映射到可量化绩效或风险降低。
- 误用：把“服务保障”写成口号（如“专业团队”）。
  - 纠偏：保障必须可验证（机制、频率、标准、责任）。

#### 5) 诊断问题清单（售前/交付都可用）

- 客户现状是否由至少两类证据交叉验证？
- 客户需求与兴趣点是否存在冲突（短期需求 vs 长期目标）？
- 我方打击点是否指向客户最关心的业务指标？
- 服务项目的边界是否清晰（做什么、不做什么）？
- 客户信任建立依赖个人关系还是制度化能力？

#### 6) 交付模板（专家版）

```markdown
【结构图对照表】
1. 切入点 <-> 客户现状
- 我方切入：<问题入口>
- 客户现状：<事实证据>
- 对照结论：<匹配/不匹配>

2. 服务内容 <-> 客户需求
- 我方服务：<模块A/B/C>
- 客户需求：<需求1/2/3>
- 对照结论：<覆盖率与缺口>

3. 打击点 <-> 兴趣点
- 我方打击点：<关键抓手>
- 客户兴趣点：<最在意结果>
- 对照结论：<是否击中>

4. 项目合理性 <-> 需求绩效
- 我方项目设计：<阶段与动作>
- 客户绩效指标：<KPI口径与阈值>
- 对照结论：<可验收性>

5. 服务保障 <-> 信任来源
- 我方保障：<治理机制/质控/专家/里程碑>
- 客户信任依据：<案例/试点/承诺机制>
- 对照结论：<信任强度>
```

#### 7) 与其他方法组合策略

- 前置：
  - 在复杂议题先用`脑图`收集需求要素，再进入结构图对照。
- 后置：
  - 对照发现的“断链项”进入`5WHY`做根因分析。
  - 修复动作进入`CPM + PDCA`形成交付闭环。
