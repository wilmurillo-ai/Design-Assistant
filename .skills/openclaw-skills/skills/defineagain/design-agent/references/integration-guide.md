# Integration Guide — How Other Skills Call design-agent

## When to Call This Skill

Any skill that generates visual output should check whether design-agent is needed:

| Skill | Call design-agent when... |
|---|---|
| `article-writing` | Generating HTML newsletters or formatted articles |
| `pdf-generator` | Creating styled PDFs with headers, typography |
| `novel-editor` | Generating HTML scene excerpts or HTML chapter previews |
| `writer` | Producing formatted long-form output with visual hierarchy |
| Any skill generating HTML/React | First output of a UI component or page |

## How to Call

### From another skill's SKILL.md

Add to the calling skill's description field:
```
Also invoke design-agent when the user asks to improve, style, or beautify
any visual output, or when the skill generates HTML/React/PDF for the first time.
```

### From an agent (human request)

```
User: Make this newsletter look better
→ Agent calls design-agent
→ design-agent provides design tokens + workflow
→ agent applies them to newsletter HTML
```

### From a script

```bash
# Extract design from a reference URL manually
# Then read design-tokens into context
cat references/design-tokens.md
```

## Calling Pattern

When design-agent is called, it expects:

1. **The output medium** — HTML, React, PDF, Newsletter, etc.
2. **The reference style** — a URL, a style name (minimal/editorial/brutalist/luxury), or "use defaults"
3. **The content to be styled** — the raw output before design is applied

design-agent then returns:
- Design tokens (merged base + reference)
- A `DESIGN.md` file written to the project directory
- A design review checklist response

## What to Tell design-agent

When invoking, include:
- What you're building (newsletter, UI panel, PDF page, etc.)
- Any existing brand tokens or reference URLs
- The target aesthetic (corporate, editorial, playful, luxury, etc.)
- Whether this is a one-off or an ongoing project (affects token depth)

## Design Review Before Shipping

Any visual output generated by another skill should pass through the design-agent checklist:

```
- Colors from tokens (no ad-hoc hex)
- Typography from scale (no arbitrary font sizes)
- Spacing from grid (no magic numbers)
- Component states defined
- Animation durations from token set
```

If the calling skill can't apply all tokens (e.g., PDF output), it should note which tokens are unresolvable and document why.

## Creating a Project DESIGN.md

Once design-agent has provided tokens, the calling skill should write a `DESIGN.md` into the project directory:

```markdown
# DESIGN.md — Project Name

## Colors
<!-- from design-agent tokens -->

## Typography
<!-- from design-agent tokens -->

## Spacing
<!-- from design-agent tokens -->

## Components
<!-- project-specific components -->
```

This becomes the single source of truth for design in that project, replacing ad-hoc instructions.
