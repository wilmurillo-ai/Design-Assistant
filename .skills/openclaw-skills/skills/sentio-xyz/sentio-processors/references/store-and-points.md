# Store Entities & Points Systems

Production patterns for persistent state management and points/rewards tracking.

## Store Entity Setup

### 1. Define Schema

Create `schema.graphql` (or `store.graphql`) in project root:

```graphql
type AccountSnapshot @entity {
  id: ID!
  timestampMilli: BigInt!
  balance: BigInt!
}

type Pool @entity {
  id: ID!
  token0: String!
  token1: String!
  fee: BigInt!
  tvl: BigDecimal!
}

type UserPosition @entity {
  id: ID!
  account: String!
  pool: String!
  liquidity: BigInt!
  lastUpdated: BigInt!
}
```

Types are auto-generated by `sentio gen` into `src/schema/store.js`.

### 2. CRUD Operations

```typescript
import { AccountSnapshot } from "./schema/store.js"

// Get
const snapshot = await ctx.store.get(AccountSnapshot, accountId)

// Upsert (create or update)
const newSnapshot = new AccountSnapshot({
  id: account,
  timestampMilli: BigInt(ctx.timestamp.getTime()),
  balance: newBalance,
})
await ctx.store.upsert(newSnapshot)

// Batch upsert
const snapshots = accounts.map(a => new AccountSnapshot({ ... }))
await ctx.store.upsert(snapshots)

// List all entities
const allAccounts = await ctx.store.list(AccountSnapshot, [])

// Delete
await ctx.store.delete(AccountSnapshot, accountId)
```

### 3. Sequential Execution for State Consistency

**Critical:** When using store entities, enable sequential execution to prevent race conditions:

```typescript
import { GLOBAL_CONFIG } from "@sentio/runtime"

GLOBAL_CONFIG.execution = {
  sequential: true,
}
```

Without this, concurrent event handlers may read stale data and overwrite each other's changes.

## Points/Rewards System Pattern

The dominant pattern from production processors (Lombard, Pendle, etc.):

### Core Concept

Track user balances over time, calculate points based on balance × duration:

```
points = balance × daily_rate × days_held × multiplier
```

### Implementation

```typescript
import { GLOBAL_CONFIG } from "@sentio/runtime"
import { BigDecimal } from "@sentio/sdk"
import { ERC20Processor } from "@sentio/sdk/eth/builtin"
import { AccountSnapshot } from "./schema/store.js"

GLOBAL_CONFIG.execution = { sequential: true }

const DAILY_POINTS = 1000
const MULTIPLIER = 2
const MS_PER_DAY = 86400000

ERC20Processor.bind({ address: VAULT_ADDRESS, network: NETWORK })
  .onEventTransfer(async (event, ctx) => {
    // Update snapshots for both sender and receiver
    const accounts = [event.args.from, event.args.to]
      .filter(addr => !isNullAddress(addr))
    const newSnapshots = await Promise.all(
      accounts.map(account => processAccount(ctx, account, "Transfer"))
    )
    await ctx.store.upsert(newSnapshots)
  })
  .onTimeInterval(async (_, ctx) => {
    // Periodically recalculate all accounts
    const accounts = await ctx.store.list(AccountSnapshot, [])
    const newSnapshots = await Promise.all(
      accounts.map(account =>
        processAccount(ctx, account.id.toString(), "TimeInterval")
      )
    )
    await ctx.store.upsert(newSnapshots)
  }, 60, 4 * 60)

async function processAccount(
  ctx: EthContext,
  account: string,
  trigger: string
): Promise<AccountSnapshot> {
  const snapshot = await ctx.store.get(AccountSnapshot, account)
  const points = snapshot ? calcPoints(ctx, snapshot) : new BigDecimal(0)

  // Read current on-chain balance
  const balance = await ctx.contract.balanceOf(account)

  const newSnapshot = new AccountSnapshot({
    id: account,
    timestampMilli: BigInt(ctx.timestamp.getTime()),
    balance,
  })

  ctx.eventLogger.emit("point_update", {
    account,
    points,
    oldBalance: snapshot?.balance.scaleDown(18) ?? 0,
    newBalance: newSnapshot.balance.scaleDown(18),
    multiplier: MULTIPLIER,
    trigger,
  })

  return newSnapshot
}

function calcPoints(ctx: EthContext, snapshot: AccountSnapshot): BigDecimal {
  const nowMs = ctx.timestamp.getTime()
  const snapshotMs = Number(snapshot.timestampMilli)
  if (nowMs <= snapshotMs) return new BigDecimal(0)

  const deltaDay = (nowMs - snapshotMs) / MS_PER_DAY
  return snapshot.balance
    .scaleDown(18)
    .multipliedBy(DAILY_POINTS)
    .multipliedBy(deltaDay)
    .multipliedBy(MULTIPLIER)
}
```

### Key Design Decisions

1. **`onEventTransfer` + `onTimeInterval`**: Transfer events trigger immediate snapshot updates; time intervals catch any missed updates and ensure periodic point emissions.

2. **`ctx.store.list(AccountSnapshot, [])`**: Lists all entities for batch recalculation. The `[]` is required (empty filter).

3. **`Promise.all` for parallel balance reads**: Read multiple on-chain balances concurrently for efficiency.

4. **`distinctId` not used in point_update**: Points events often use a plain `account` field instead of `distinctId` since the system tracks balances, not unique user actions.

## Multi-Contract Points (Pendle pattern)

Track points across multiple interacting contracts (SY, YT, LP tokens + external yield aggregators):

```typescript
// Main protocol tokens
ERC20Processor.bind({ address: SY_ADDRESS, name: "SY" })
  .onEventTransfer(syTransferHandler)

PendleYieldTokenProcessor.bind({ address: YT_ADDRESS, name: "YT" })
  .onEventTransfer(ytTransferHandler)
  .onTimeInterval(processAllYTAccounts, 60, 60)

PendleMarketProcessor.bind({ address: LP_ADDRESS, name: "LP" })
  .onEventTransfer(lpTransferHandler)
  .onEventSwap(swapHandler)

// External yield aggregator receipt tokens
ERC20Processor.bind({ address: EQUILIBRIA_RECEIPT, name: "EQB" })
  .onEventTransfer(async (evt, ctx) => {
    // When receipt token transfers, recalculate affected LP accounts
    await processAllLPAccounts(ctx, [
      evt.args.from.toLowerCase(),
      evt.args.to.toLowerCase(),
    ])
  })

ERC20Processor.bind({ address: PENPIE_RECEIPT, name: "Penpie" })
  .onEventTransfer(async (evt, ctx) => {
    await processAllLPAccounts(ctx, [
      evt.args.from.toLowerCase(),
      evt.args.to.toLowerCase(),
    ])
  })
```

**Key insight:** Track transfer events on ALL receipt tokens (not just the main protocol) to capture positions held through aggregators.
