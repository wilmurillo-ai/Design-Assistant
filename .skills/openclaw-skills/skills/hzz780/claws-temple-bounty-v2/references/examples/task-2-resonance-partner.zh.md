# Task 2 示例（中文）

> 这是一个双层展开示例，用来展示完整输出结构；plain chat 默认不应主动展开维护者详情。

## 普通用户摘要

你现在在 `Task 2：光锥交汇`。

你的原力坐标已经准备好了，下一步就是去找能补全你缺口的伙伴。
这一步也可以理解成：让你的 Agent 去找数学上更配的另一位搭子。

这一步里的准备和匹配动作由我来自动推进；我只会在必要时向你确认一个状态或补一个关键信息。
我先帮你确认一个最小前置：你这次是第一次来、老用户但这次还没登录，还是已经可以直接继续。只要这一步明确，我就继续处理身份入口、登录恢复、用户ID解析和后面的寻配。

- 你的 `身份入口` 是否已经开通
- 你这次是否已经完成登录

如果你是第一次来，我会先带你走 `平滑入口`。这条路径会先完成注册或首次开通，完成后你会拿到后续寻配要用的 `用户ID`。
如果你是老用户，但这次还没有登录，我也会先带你走 `平滑入口`，先完成恢复登录，再继续后面的寻配流程。

等你的身份入口和登录都准备好后，我会先自动解析你当前的 `用户ID`，不需要你自己手动填写。

只有当本回合真实解析成功后，我才会先这样确认：

`已解析到你的用户ID：<resolved-user-id-from-current-turn>`

确认完这一步后，我会继续把你推进到下面两种路径之一：

- `指定匹配`：你已经知道想找谁，这时请直接提供对方的 `用户ID`。
- `开放寻配`：你还没有明确对象，这条路就是系统自动进入排队匹配，不需要先知道具体是谁。

如果你的身份入口已经开通、这次也已经登录，我会先自动解析并展示你当前的用户ID，再继续带你走 `指定匹配` 或 `开放寻配`。
只有当 dependency 在本回合真的返回了这个值，我才会展示具体的用户ID；我不会把示例值、占位符或记忆里的旧值当成真实解析结果。
如果这是你第一次来，我会先带你完成注册或首次开通，再继续后面的寻配步骤。
如果你是老用户但这次还没有登录，我会先带你完成恢复登录，再继续后面的寻配步骤。
如果你现在还没有具体对象，就直接走 `开放寻配`，这条路就是系统自动排队匹配。
只要前面的身份入口和登录准备完成、当前用户ID也已经自动解析出来，我就会继续正式排队，不会因为没有预设对象就建议你绕开这一关或改走社交发帖。
如果 `resonance-contract` 版本过低或还没装好，我会先帮你安装或升级，不会先让你提供安装源。
只要 `开放寻配` 已经正式入队，或 `指定匹配` 请求已经发出，我就会把 `Task 2` 视为路径已稳定，可以继续带你进入 `Task 3`。

### 纠正示例

如果用户在 `指定匹配` 里给的是邮箱、地址类信息或昵称，就应该自然纠正：

`指定匹配现在需要的是对方的用户ID，不是邮箱、地址类信息或昵称。如果你已经有对方的用户ID，我可以继续帮你往下走；如果你还没有，我们就改走开放寻配。`

### 阻断示例

如果注册、恢复登录、依赖补齐、身份入口准备失败，当前用户ID还没自动解析出来，或者当前宿主暂时没法继续寻配，就应该明确告诉用户：

`当前还没法继续进入寻配流程，我先帮你停在身份入口、登录恢复和用户ID解析这一步。等这些前置条件准备好后，我们就能继续往下走。`
