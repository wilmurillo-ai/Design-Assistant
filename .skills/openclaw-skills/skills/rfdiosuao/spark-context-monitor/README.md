# 📊 Context Usage Checker - 上下文使用率查询

> **版本：** 1.0.0  
> **作者：** Spark  
> **最后更新：** 2026-04-06

实时查询当前会话的 Token 使用情况，支持进度条可视化和智能预警。

---

## 🎯 功能介绍

### 核心功能
- 📊 **实时查询** - 随时查看当前会话的 Token 消耗
- 📈 **进度条可视化** - 直观的进度条展示使用率
- ⚠️ **智能预警** - 超过阈值时自动提醒
- 🎯 **多模型支持** - 自动识别不同模型的上下文窗口

## ⚙️ 配置说明

### 预警阈值

| 状态 | 阈值 | 图标 | 提示 |
|------|------|------|------|
| 正常 | < 70% | 📊 | 无额外提示 |
| 警告 | 70% - 89% | ⚠️ | 建议清理上下文 |
| 严重 | ≥ 90% | 🚨 | 立即清理 |

### 支持的模型

| 模型 | 上下文窗口 |
|------|-----------|
| qwen3.5-plus | 256,000 tokens |
| qwen3.5 | 256,000 tokens |
| qwen-max | 32,000 tokens |
| Claude 系列 | 200,000 tokens |
| GPT-4 系列 | 128,000 tokens |
| 其他模型 | 128,000 tokens (默认) |

## 🔧 技术原理

### Token 估算算法
采用字符数估算法计算 Token 数量：
```
中文字符：约 1.5 字符/token
英文字符：约 4 字符/token
```

**精度说明：** 估算误差通常在 ±10% 以内，实际 Token 数以平台为准。

### 进度条生成
- 总长度：20 个字符
- 已使用部分：`▓`
- 剩余部分：`░`
- 计算公式：`已使用格数 = (使用率 / 100) × 20`

## 📝 使用场景

### 场景 1：长对话管理
**问题：** 多轮对话后，AI 开始遗忘早期内容  
**解决：** 定期查询上下文使用率，在达到 70% 时主动使用 `/new` 开启新会话

### 场景 2：Token 成本控制
**问题：** 不清楚每次对话消耗多少 Token  
**解决：** 实时显示 Token 使用量，帮助评估对话成本

### 场景 3：调试与优化
**问题：** 需要优化 Prompt 长度  
**解决：** 监控不同 Prompt 的 Token 消耗，找到最优方案

## 🐛 常见问题

### Q1: 为什么显示的 Token 数和平台统计不一致？
**A:** 本 Skill 使用字符数估算算法，实际 Token 数以平台为准。误差通常在±10% 内，不影响使用。

### Q2: 可以修改预警阈值吗？
**A:** 可以，修改源码中的常量：
```typescript
const WARNING_THRESHOLD = 70;      // 警告阈值（%）
const CRITICAL_THRESHOLD = 90;     // 严重警告阈值（%）
```

### Q3: 支持哪些平台？
**A:** 支持所有 OpenClaw 集成的平台：
- ✅ 飞书
- ✅ Telegram
- ✅ WhatsApp
- ✅ Discord
- ✅ Slack
- ✅ 其他 OpenClaw 支持的渠道

### Q4: 会影响回复速度吗？
**A:** 几乎无影响。查询耗时约 5-10ms，用户无感知。

### Q5: 如何清空上下文？
**A:** 使用 OpenClaw 内置命令：
- `/new` - 开启全新会话
- `/compact` - 压缩上下文（保留核心信息）
