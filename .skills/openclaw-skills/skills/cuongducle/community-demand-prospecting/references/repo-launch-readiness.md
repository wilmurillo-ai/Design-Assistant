# Repo Launch Readiness

Use this reference when the user provides a repository, README, release page, or rough product idea instead of a finished marketing brief.

## Goal

Convert a dev-centric artifact into a marketable product brief before doing outreach.

Do not jump straight to promotion if the repo is not ready to convert attention into installs, stars, trials, or feedback.

## Audit Areas

Review these areas in order:

### 1. Positioning

Check whether the repository clearly answers:

- what it is
- who it is for
- what pain it solves
- why it is simpler or better
- what action the visitor should take next

Common problems:

- headline describes implementation, not user value
- audience is too narrow
- use case is buried
- differentiator is unclear

### 2. Conversion Assets

Check for:

- screenshot or GIF
- release asset or installation path
- short demo
- concise feature list
- social proof or credibility signals

Common problems:

- no visual proof
- no clear download/install path
- too much engineering detail before user value

### 3. Trust Friction

Check for:

- unsigned binaries
- scary install steps
- unclear permissions
- sparse docs
- missing release notes

If trust friction is high, tell the user that promotion should wait until the blocker is reduced or at least explained clearly.

### 4. Channel Fit

Choose channels by answering these questions first:

- Is the primary user a builder, operator, or broad end user?
- Is discovery usually recommendation-driven, search-driven, or community-driven?
- Is install friction low or high?
- Does the product require trust before trial?
- Is the buying cycle immediate, evaluative, or long?

Then map the product to channel types such as:

- builder communities
- practitioner communities
- recommendation-driven communities
- founder or operator communities
- directories or marketplaces
- comparison or search-oriented content

Prefer the smallest set of channels that matches the product's true discovery pattern.

### 5. Launch Mode

Choose one of these modes:

- `Feedback launch`: early, honest, ask for opinions and bug reports
- `Utility launch`: simple product, direct install value, lightweight promo
- `Story launch`: strong founder story or frustration narrative
- `Comparison launch`: clear alternative to an existing tool

## Recommendation Pattern

When auditing a repo, return:

- one-sentence positioning
- top three blockers
- channels to prioritize
- channels to avoid for now
- first post angle
- first CTA

## Typical Improvements

For developer-built products, suggest practical fixes such as:

- rewrite README headline and first paragraph
- add one screenshot
- add one GIF or short demo
- add install command or release instructions
- broaden top-level positioning while keeping niche use cases below
- add GitHub topics and repo description
- clarify whether the project is stable, beta, or experimental
