# MIT有序创业24步法 - 阶段五：团队与产品验证

第五阶段将创业从"纸上谈兵"推进到"实战验证"。需要确定可以投入市场的最小产品，准备向真实客户展示的方案，并组建有能力执行落地的核心团队。本阶段包含步骤19-21。

## 前置条件

建议在前四个阶段的基础上进行，确保有清晰的产品定义和市场进入策略。

## 工作流程

### 步骤 19: 定义最小可行商业产品（MVBP）

确定可以验证商业模式的最小产品——注意MVBP与MVP的区别。

**MVBP vs MVP：**
- MVP（最小可行产品）：最小化功能范围，验证产品是否有用
- MVBP（最小可行商业产品）：不仅验证产品有用，还要验证客户愿意付费

**MVBP的判定标准：**
1. 能解决步骤3中客户画像的核心痛点
2. 能支撑步骤11中定价策略的价格点
3. 能通过步骤14中的客户购买决策标准
4. 能在步骤18的假设验证中证明商业可行性
5. 具备完整的端到端客户体验（从认知到购买到使用）

**MVBP规划清单：**
- 核心功能列表（不超过5个）
- 非功能性要求（性能、安全、可用性）
- 技术架构选择
- 开发时间线（建议4-8周内完成）
- 资源需求（人力、资金、工具）
- 上线计划和初始客户获取方案

**输出要求：** MVBP产品规格文档，含功能范围、时间线、资源需求。

### 步骤 20: 展示解决方案

向客户展示产品解决方案，收集真实反馈。

**展示方式选择：**

| 方式 | 适用阶段 | 优势 | 局限 |
|------|----------|------|------|
| 产品Demo | 产品已有可运行版本 | 最直观 | 需要产品已完成 |
| 原型演示 | 产品开发中 | 快速验证交互 | 可能与最终产品有差距 |
| 概念视频 | 早期验证 | 制造成本低 | 缺乏交互体验 |
| 案例研究 | 有早期客户案例 | 说服力强 | 需要真实数据 |
| 客户见证 | 有满意客户 | 信任度高 | 需要真实客户配合 |
| 数据报告 | 有使用数据 | 客观可信 | 需要足够数据积累 |

**有效展示的5个要素：**
1. 开场30秒：快速建立共鸣，说明客户痛点
2. 产品展示：聚焦核心功能，控制在10分钟内
3. 价值证明：用数据和案例说话
4. 互动环节：邀请客户操作和提问
5. 下一步：明确的行动号召

**反馈收集框架：**
- 客户最喜欢什么？
- 客户觉得缺少什么？
- 客户的购买意愿有多强？（1-10分）
- 客户愿意付多少钱？
- 客户有什么顾虑或疑虑？

**输出要求：** 客户展示方案（Demo脚本/演示大纲）+ 反馈收集表。

### 步骤 21: 建立核心团队

组建有能力将创业愿景落地的核心团队。

**四种核心角色：**

| 角色 | 职责 | 关键特质 | 典型背景 |
|------|------|----------|----------|
| 愿景型（Visionary） | 设定战略方向，激励团队，对外讲故事 | 远见卓识、感染力、战略思维 | 创始人、连续创业者 |
| 执行型（Executor） | 落实计划，解决问题，推动执行 | 结果导向、抗压能力强、组织能力 | 运营、项目经理 |
| 技术型（Technician） | 开发产品，技术创新，技术决策 | 技术深度、创造力、学习能力 | CTO、技术负责人 |
| 商业型（Business） | 市场开拓，商务合作，商业运营 | 商业敏感度、谈判能力、人脉资源 | 市场负责人、BD |

**团队组建原则：**
- 早期团队3-5人为佳，角色互补
- 技术团队和商业团队的比例要平衡
- 合伙人之间要有共同的价值观和愿景
- 明确股权分配和决策机制（避免日后纠纷）
- 建立开放透明的沟通文化

**常见团队问题与应对：**
- 技术团队强但商业能力弱 → 引入商业合伙人或顾问
- 合伙人之间分工不清 → 明确角色边界和决策权
- 早期招人难 → 用愿景和股权吸引，提供学习成长机会
- 团队扩张过快 → 先验证商业模式再扩大团队

## 输出格式

完成3个步骤后，生成《团队与产品验证报告》，包含：
1. MVBP产品规格文档
2. 客户展示方案（Demo脚本 + 反馈收集表）
3. 团队组建方案（角色分工 + 招聘计划 + 股权建议）

## 实用建议

- MVBP的完成时间不应超过8周，否则说明范围太大
- 客户展示的最佳时机是产品可用但不完美的时候——太早没说服力，太晚错过反馈窗口
- 早期团队的股权分配是日后矛盾的常见根源，建议提前明确并签署协议
- 如果用户是科研人员创业，特别需要补充商业型角色，避免"技术自嗨"
