# 七阶段工作流详细说明（Phase 0–6）

---

## Phase 0：需求澄清

**目标**：将用户的自然语言需求转化为可执行的明确需求。

**第一步：确认用户角色**
- 📋 产品/业务同学 → 输出 spec.md + plan.md
- 🔧 数仓开发同学 → 输出 spec.md + plan.md + task.md

角色不明确时直接询问，不默认。

**必须确认的信息**：

1. **需求范围**：新建表 / 修改已有表 / 新增指标 / 数据迁移 / 其他？涉及哪些数据域 / 业务主题？
2. **数据分层定位**：ODS / DWD / DWS / ADS / DIM？
3. **数据来源**：源系统是什么？MySQL / Oracle / Kafka / 埋点？
4. **更新频率**：全量 / 增量？多久调度一次？
5. **关键指标或字段**：业务方最关心哪些产出？
6. **上下游依赖**：是否依赖其他已有表？是否有下游使用方？

**澄清策略**：
- 需求明确的点不重复询问
- 将多个问题合并提问，一次最多问 5 个问题
- 对用户提供的示例数据 / 截图 / 文档，先理解后确认

**退出条件**：所有必填信息已获取，进入 Phase 1。

---

## Phase 1：元数据收集

**目标**：获取需求相关的表结构、字段、分区、存储等元数据信息。

**五种采集方式**（可组合使用）的完整配置说明见 `resources/conventions/metadata-config.md`，该文件是元数据获取方式的唯一定义来源。

**方式选择建议**：

| source_type | 适用场景 |
|-------------|----------|
| `manual` | 少量表（< 10）、快速验证、无系统访问权限 |
| `jdbc` | 有数据库直连权限，需采集大量表或完整库 |
| `openapi` | 平台提供标准 REST API，支持 AK/SK 鉴权 |
| `web` | 仅有前端数据门户入口，无法直连数据库或调用 API |
| `hdfs` | 元数据以文件形式存储在 HDFS，常见于离线数仓元数据导出场景 |

**信息不足处理**：
- 标注 `[待补充]` 的字段，汇总后一次性询问用户
- 关键字段（主键、分区键、业务时间字段）缺失时阻断流程，必须获取

**退出条件**：核心元数据齐备，进入 Phase 2。

---

## Phase 2：生成 Spec

**目标**：输出标准化的 spec.md 需求规格文档（业务友好，聚焦做什么）。

**输入**：Phase 0 需求 + Phase 1 元数据

**输出**：完整的 `spec.md`，结构见 `resources/spec-template.md`

**生成要求**：
- 使用业务语言，避免 SQL 语法和技术术语（技术实现细节在 plan.md 中定义）
- 所有字段必须有业务含义说明，不允许仅有技术描述
- 数据范围和业务规则用自然语言描述，附具体口径举例
- 风险与约束（§6）完整填写，验收标准（AC）可由业务同学验证
- 不确定的内容标注 `[待确认]`，不伪造信息

**退出条件**：spec.md 生成完毕，进入 Phase 3 等待确认。

---

## Phase 3：用户确认 Spec ⏸

⛔ **强制确认点，不得跳过，不可自行推断用户已同意。**

展示 spec.md 全文后等待用户明确回复（确认通过 / 指出修改点）才能继续。

收到修改意见时：更新对应内容，标注 `[已更新]`，重新展示后再次等待确认。

**退出条件**：用户明确表示通过，进入 Phase 4。

---

## Phase 4：生成 Plan

**目标**：基于确认的 spec.md，生成技术方案层执行计划（plan.md）。

**输出**：`plan.md`，按 `resources/plan-template.md` 生成。重点产出：
- **ETL 技术方案**（§1）：将 spec §5 的业务规则转化为 JOIN 逻辑、SQL 过滤条件、分步转换伪代码、技术选型决策
- **任务清单**（§3）：每行包含实现要点和 conventions 应用决策，作为 Phase 6 生成 task 的直接输入

**退出条件**：plan.md 生成完毕，进入 Phase 5 等待确认。

---

## Phase 5：用户确认 Plan ⏸

⛔ **强制确认点，不得跳过，不可自行推断用户已同意。**

展示 plan.md 全文后等待用户明确回复（确认通过 / 调整意见 / 跳过某任务）才能继续。

**退出条件**：用户明确表示通过，进入 Phase 6。

---

## Phase 6：生成 Task

**目标**：基于已确认的 spec.md 和 plan.md，生成所有可直接执行的代码和配置，组织为 task.md。

**输入**：spec.md（做什么）+ plan.md（怎么做，含 ETL 技术方案和 conventions 应用决策）

**输出**：`task.md`，按 `resources/task-template.md` 生成，使用 plan.md 任务清单中的实现要点和 conventions 应用决策内联所有规范参数。

### 数仓开发同学输出

| 产物 | 说明 |
|------|------|
| DDL 建表语句 | Hive 建表，含分区、存储格式、表属性 |
| ETL SQL 脚本 | Spark SQL / HiveQL |
| Azkaban 调度配置 | Job 文件 + flow 文件 |
| 数据质量检查 | 基于 Spec 中质量规则生成 |

若 task 数量超过 10 个，按模块（数据分层或业务域）拆分为多个文件（如 `task-ods.md`、`task-dwd.md`），并在主 `task.md` 中列出索引。

### 产品/业务同学
不生成代码，仅输出 spec.md 和 plan.md。
