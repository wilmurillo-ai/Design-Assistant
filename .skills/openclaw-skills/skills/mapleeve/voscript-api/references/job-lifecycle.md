# VoScript 任务生命周期

本文档描述转写任务从提交到完成的完整状态机、各阶段含义、典型耗时与
异常处理。

## 状态机总览

```
                    submit_audio
                         │
                         ▼
                     ┌────────┐
                     │ queued │
                     └────┬───┘
                          │  worker 取任务
                          ▼
                    ┌──────────┐
                    │converting│   音频格式归一化
                    └────┬─────┘
                         │
                         ▼
                    ┌──────────┐
                    │denoising │   可选：降噪
                    └────┬─────┘
                         │
                         ▼
                 ┌──────────────┐
                 │ transcribing │   ASR + 说话人分离
                 └──────┬───────┘
                        │
                        ▼
                 ┌─────────────┐
                 │ identifying │   声纹匹配 (AS-norm)
                 └──────┬──────┘
                        │
              ┌─────────┴─────────┐
              ▼                   ▼
        ┌───────────┐         ┌────────┐
        │ completed │         │ failed │
        └───────────┘         └────────┘
```

如果 SHA-256 命中已有结果，提交请求会直接以 `status=completed` +
`deduplicated=true` 返回，不进入上述状态机。

## 各状态含义与典型耗时

| 状态           | 含义                                                   | 典型耗时（1 小时音频） |
| -------------- | ------------------------------------------------------ | ---------------------- |
| `queued`       | 任务入队，等待 worker                                  | 秒级，视负载而定       |
| `converting`   | 解码为统一采样率单声道 WAV                             | 10–30 秒               |
| `denoising`    | 仅当 `denoise_model != none` 时出现                    | 30–120 秒              |
| `transcribing` | 语音识别 + 说话人分离，是最耗时阶段                    | 3–10 分钟（依模型）    |
| `identifying`  | 针对每个 speaker segment 计算声纹并做 AS-norm 比对      | 10–60 秒               |
| `completed`    | 全部完成，可通过 `/api/transcriptions/{id}` 取结果     | —                      |
| `failed`       | 任一阶段报错终止                                       | —                      |

短音频（< 10 分钟）整体通常在 1–2 分钟内完成；长音频（1 小时以上）
应按分钟级预期。

## 轮询建议

- 每 5 秒 `GET /api/jobs/{job_id}` 一次。
- 遇到 `queued` / `converting` 状态，保持固定节奏即可。
- 进入 `transcribing` 后可以适度拉长间隔（例如 10–15 秒），以减少服务端压力。
- 看到 `completed` 或 `failed` 立刻停止轮询。

## 失败处理

当任务状态变为 `failed`，响应 JSON 中会包含 `error` 字段，常见成因：

- **音频解码失败**：文件损坏或格式不被 ffmpeg 支持 → 请用户重新导出为 WAV / MP3。
- **降噪阶段 SNR 过低**：可降低 `snr_threshold` 或改用更稳定的降噪模型。
- **GPU OOM / 显存不足**：提示部署方降低并发或使用更小模型。
- **模型加载失败**：多数是服务端环境问题，需查看服务端日志。

失败任务无法原地续跑，需要重新 `submit_audio` 提交。

## SHA-256 去重行为

提交阶段服务端会计算上传音频的 SHA-256 哈希并与历史提交比对：

- **命中**：直接返回 `{"id": "tr_xxx", "status": "completed", "deduplicated": true}`。
  - 代理应识别 `deduplicated: true`，跳过轮询，直接进入
    `GET /api/transcriptions/{tr_id}` 取结果。
- **未命中**：正常入队，按上述状态机推进。

去重基于完整文件字节，即使仅修改元数据（如 ID3 tag）也会产生新的哈希
从而视为不同音频。

## 与幂等性的关系

- 同一音频重复提交不会创建新任务，也不会重复消耗 GPU 资源。
- 若需要对同一段音频用不同参数重跑，只需改动 `language`、`min_speakers`、
  `max_speakers` 等参数即可——目前版本参数变更不会绕过 SHA-256 去重，
  需联系部署方手动删除历史结果后重跑。
