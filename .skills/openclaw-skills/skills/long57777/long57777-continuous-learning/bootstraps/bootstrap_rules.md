# 启动规则 - 持续学习套件

*整合版：新会话/新任务的标准启动流程*

---

## 铁律：判断 → 确认 → 读取 → 执行

### 1️⃣ 判断是否是新任务

**判断标准**：
- 话题跨度大（从A项目跳到B项目）
- 任务类型变（查LIMS → 发邮件）
- 关键词第一次出现
- 用户说"新任务"、"开始做..."
- 任何不确定的情况

### 2️⃣ 确认

- ✅ 明确是新任务 → 直接执行
- ❓ 不确定 → 问用户确认
- ❌ 禁止跳步

### 3️⃣ 读取

按以下顺序读取：

**核心文档**（每次都读）：
1. `SOUL.md` — 我是谁、个性偏好
2. `AGENTS.md` — 工作流程、规则
3. `MEMORY.md` — 用户背景、长期记忆
4. `TOOLS.md` — 工具配置、坑、技巧

**学习文件**（避免重复错误）：
5. `.learnings/ERRORS.md` — 历史错误教训
6. `.learnings/LEARNINGS.md` — 学习成果

**近期记忆**（上下文）：
7. `memory/YYYY-MM-DD.md` — 今日记忆
8. `memory/YYYY-MM-DD.md` — 昨日记忆

**相关技能**（仅当需要时）：
9. `skills/<技能名>/SKILL.md` — 特定技能使用方法

### 4️⃣ 执行

- 严格按照读取的规则执行
- 犯错立即记入 `.learnings/ERRORS.md`
- 被纠正立即记入 `.learnings/LEARNINGS.md`


## 🚨 常见陷阱

### 陷阱1：凭记忆执行

**错误**：直接根据上次记忆执行
**正确**：每次新任务必须先读相关文档

### 陷阱2：猜测字段名

**错误**：猜测API字段名
**正确**：先验证，再使用

### 陷阱3：忽略错误

**错误**：错误后不记录
**正确**：立即记入ERRORS.md，避免重复

### 陷阱4：跳过确认

**错误**：不确定时直接执行
**正确**：先问用户确认

## 🎯 最佳实践

### 实践1：结构化记录

`ERRORS.md`格式：
```markdown
### 错误：[简短描述]
**错误**: [具体错误内容]
**正确**: [正确做法]
**教训**: [避免方法]
**日期**: YYYY-MM-DD
```

`LEARNINGS.md`格式：
```markdown
### 学习：[主题]
**收获**: [学到的内容]
**应用**: [应用场景]
**日期**: YYYY-MM-DD
```

### 实践2：定期清理

- 每30天清理ERRORS.md中已解决的问题
- 将长期有效的迁移到MEMORY.md
- 保留最有价值的学习条目

### 实践3：主动应用

- 每次执行任务前 → 速览ERRORS.md
- 每次新任务开始 → 速览LEARNINGS.md
- 每周做梦后 → 检查5文档更新

## 🔧 故障排查

### 问题：定时任务不执行

**检查**：
- 任务计划程序（Windows）或cron（Linux）
- Python路径是否正确
- 脚本是否有执行权限

### 问题：MemPalace写入失败

**检查**：
- 数据库路径
- 写入权限
- 日志文件输出

### 问题：做梦无更新

**检查**：
- 记忆碎片数量
- API Key有效性
- 提示词配置
