# FlowClaw

**Automate your agent workflows. Stay in control of what happens next.**

FlowClaw orchestrates your AI agent teams through automated multi-step workflows — with approval gates that pause and wait for your go-ahead before anything critical runs.

## Features

- ✅ **Approval Gates** — Workflows pause and wait for your go-ahead before anything critical runs. You stay in control, always.
- 🔄 **YAML Workflow Definitions** — Write your workflows in simple, readable YAML. Version-control them, share them, tweak them without touching code.
- 🤖 **Multi-Agent Orchestration** — Route tasks to the right OpenClaw agent for each step. FlowClaw handles the sequencing so you don't have to.
- 📊 **Notion Integration** — Pick up tasks from Notion and write status back automatically.
- 🚀 **n8n Integration** — Trigger workflows from n8n via HTTP. Drop-in compatible.
- 🔔 **Discord Notifications** — Know what's happening at every step, in real time.
- 🛡️ **Runs Reliably** — Idempotent execution, SQLite-backed state, structured logging, Gunicorn multi-worker, health + metrics endpoints.

## Platform Support

| Platform | Status | Notes |
|----------|--------|-------|
| macOS    | ✅ Supported | Primary development platform. LaunchAgent + Gunicorn startup. |
| Linux    | ✅ Supported | systemd service template included. |
| Windows  | ❌ Not supported | Gunicorn does not run on Windows. Startup scripts are bash-only. |

### Why Windows is not supported

FlowClaw relies on components that are not available on Windows:

- **Gunicorn** — the WSGI server used for production deployments does not run on Windows
- **Bash startup script** — `src/scripts/start-workflow-executor.sh` requires bash
- **LaunchAgent** (`config/launchagent.plist`) — macOS only
- **systemd service** (`config/workflow-executor.service`) — Linux only

The Python application itself (`src/workflow-executor.py`) uses `pathlib.Path` and cross-platform APIs throughout, but the production deployment stack is Unix/macOS only.

If you need Windows support, you would need to:
1. Replace Gunicorn with a Windows-compatible WSGI server (e.g. Waitress)
2. Add a PowerShell startup script
3. Use a Windows Service or Task Scheduler instead of LaunchAgent/systemd

## Security

FlowClaw is designed with security at every boundary. Here is what it guarantees:

### Authentication
- **All API endpoints require authentication.** Provide `Authorization: Bearer <key>` or `X-API-Key: <key>`.
- HMAC constant-time comparison prevents timing attacks on key comparison.
- If `WORKFLOW_EXECUTOR_API_KEY` is not set, all requests are rejected with HTTP 503.

### Input Validation
Every externally-supplied value is validated before use:

| Input | Validation rule |
|-------|----------------|
| `workflowName` | Alphanumeric, hyphens, underscores — 1–64 chars. Regex: `^[a-zA-Z0-9_-]{1,64}$` |
| `taskId` | String, max 64 chars |
| `run_id` | Must match timestamp format generated by this service: `^[0-9]{8}-[0-9]{6}-[0-9a-f]{6}$` |
| `pageId` (Notion) | Valid UUID (Notion page ID format): `^[0-9a-f]{8}-?...-?[0-9a-f]{12}$` |
| `status` (Notion) | Allowlisted values only (see `_VALID_NOTION_STATUSES`) |
| `approved_by` | String, max 128 chars |
| `notes` | String, silently truncated to 1000 chars |
| `dueDate` | ISO 8601 date string: `YYYY-MM-DD` |
| `assignedTo` | Allowlisted agent IDs only |
| `agentId` | Allowlisted agent IDs only (`frontend`, `backend`, `creative`, `quality`, `devops`, `main`) |
| `channel` | Allowlisted channel keys from env-configured `DISCORD_CHANNEL_*` vars |
| `channelId` | Must be in allowlist of configured channel IDs |
| Task text | Max 50,000 chars |
| Name fields | Max 512 chars |
| All request bodies | Content-Type must be `application/json`, body must be a JSON object |

### Path Safety
- All file operations use `_safe_path()` which resolves paths and verifies they stay under the designated base directory (`FLOWCLAW_DIR` or `WORKSPACE`).
- Path traversal attempts (`../../../etc/passwd`) are rejected with `ValueError`.
- QA scripts must have a `.py` extension — shell scripts and binaries cannot be executed via the QA step.
- Staging URLs passed to QA scripts must start with `http://` or `https://`.

### Command Execution
- **No `shell=True` anywhere** — all subprocess calls use list-form arguments, preventing shell injection.
- **No `os.system()`, `eval()`, or `exec()`** — only `subprocess.Popen` and `subprocess.run` are used.
- Agent IDs are allowlisted before being passed to the CLI.
- Project names are validated before being used as working directories.
- Timeout values are safely coerced to `int`, defaulting on coercion failure — no crash if YAML supplies a float or string.

### Credential Isolation
- **Default:** FlowClaw reads credentials **only** from environment variables or `.env` files.
- It does **not** automatically inherit credentials from `~/.openclaw/openclaw.json`.
- To opt in to OpenClaw config file loading, explicitly set `FLOWCLAW_LOAD_OPENCLAW_CONFIG=true`. This gives FlowClaw access to all credentials in your global OpenClaw config — only use it if you understand the implications.

### Discord Channel Security
- Discord channel IDs are configured via environment variables (`DISCORD_CHANNEL_PROJECT_UPDATES`, etc.) — never hardcoded.
- The `/notify/discord` endpoint enforces an allowlist: only configured, non-empty channel IDs are accepted.
- Requests to unrecognised or unconfigured channel IDs return HTTP 403.

### Logging
- Structured JSON logging to stderr — no plaintext format strings.
- API keys and tokens are never logged.
- Error messages omit internal paths and stack traces from HTTP responses.

### Database
- SQLite database uses WAL mode and parameterised queries throughout — no string interpolation into SQL.
- Workflow state expires after a configurable timeout (default 1 hour) and is deleted on resume.

### Service Binding
- Default bind address is `127.0.0.1` (loopback only).
- To expose on the network, change `HOST` in `.env` and ensure authentication is configured. Never expose without `WORKFLOW_EXECUTOR_API_KEY`.

## Quick Start

**Prerequisites:** Python 3.8+, OpenClaw with configured agents. Optional: n8n, Notion API key, Discord bot token.

1. Install dependencies:
   ```bash
   pip3 install -r src/requirements.txt
   ```

2. Copy and configure environment variables:
   ```bash
   cp config/example.env .env
   # Edit .env — at minimum, set WORKFLOW_EXECUTOR_API_KEY
   ```

3. Start the executor:
   ```bash
   bash src/scripts/start-workflow-executor.sh
   # or in dev mode:
   python3 src/workflow-executor.py
   ```

4. Verify it's running:
   ```bash
   curl -H "Authorization: Bearer your-key" http://127.0.0.1:8765/health
   ```

5. Import `src/n8n-workflow.json` into your n8n instance and update the placeholder values.

6. Configure n8n to call `http://127.0.0.1:8765/workflow/execute`

→ See [INTEGRATION-STEPS.md](docs/INTEGRATION-STEPS.md) for full setup details.

## Built to Keep Running

FlowClaw runs as a persistent service and handles the edge cases you'd rather not think about:

- **Idempotent execution** — trigger the same workflow twice, it only runs once
- **Gunicorn multi-worker** — concurrent requests handled, no bottlenecks
- **Structured logging** — readable in your terminal, parseable by log tools
- **Health + metrics endpoints** — `/health` and `/metrics` work out of the box
- **Service templates** — systemd (Linux) and LaunchAgent (macOS) configs included

## API Endpoints

All endpoints require `Authorization: Bearer <WORKFLOW_EXECUTOR_API_KEY>` or `X-API-Key: <WORKFLOW_EXECUTOR_API_KEY>`.

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/health` | GET | Health check (no auth required) |
| `/workflow/execute` | POST | Execute a workflow |
| `/workflow/approve` | POST | Approve a pending workflow |
| `/workflow/resume` | POST | Resume a paused workflow |
| `/workflow/history` | GET | Query run history for a task |
| `/workflow/pending-approvals` | GET | List pending approvals |
| `/workflows/list` | GET | List available workflow YAML files |
| `/notify/discord` | POST | Send a Discord notification |
| `/notion/update` | POST | Update a Notion page status |
| `/notion/create-subtask` | POST | Create a Notion subtask |
| `/metrics` | GET | Prometheus metrics |
| `/dashboard` | GET | Web dashboard |

## Required Environment Variables

| Variable | Required | Description |
|----------|----------|-------------|
| `WORKFLOW_EXECUTOR_API_KEY` | **Required** | API key for all requests |
| `NOTION_API_KEY` | Optional | Notion integration token |
| `NOTION_TASKS_DATABASE_ID` | Optional* | Notion database UUID for `/notion/create-subtask` |
| `DISCORD_BOT_TOKEN` | Optional | Discord bot token |
| `DISCORD_CHANNEL_PROJECT_UPDATES` | Optional | Discord channel ID |
| `DISCORD_CHANNEL_PROJECT_CALLS` | Optional | Discord channel ID |
| `DISCORD_CHANNEL_CHATTERIA` | Optional | Discord channel ID |
| `PORT` | Optional | Server port (default: 8765) |
| `HOST` | Optional | Bind address (default: 127.0.0.1) |
| `MAX_WORKERS` | Optional | Gunicorn workers (default: 4) |
| `OPENCLAW_GATEWAY_URL` | Optional | OpenClaw gateway WebSocket URL (default: `ws://127.0.0.1:18789`) |
| `OPENCLAW_GATEWAY_TOKEN` | Optional* | Gateway auth token. Required when gateway auth is enabled and `FLOWCLAW_LOAD_OPENCLAW_CONFIG=false`. Auto-loaded from `openclaw.json` when `FLOWCLAW_LOAD_OPENCLAW_CONFIG=true`. |
| `FLOWCLAW_LOAD_OPENCLAW_CONFIG` | Optional | Set `true` to load from `openclaw.json` (see Credential Isolation) |

*Required if using the `/notion/create-subtask` endpoint.

```bash

```
