# قالب وكيل SDR للشركات B2B

> حوّل أي شركة تصدير B2B إلى آلة مبيعات مدعومة بالذكاء الاصطناعي في 5 دقائق.

قالب مفتوح المصدر وجاهز للإنتاج لبناء ممثلي تطوير المبيعات AI (SDRs) الذين يتعاملون مع **خط المبيعات الكامل** — من التقاط العملاء المحتملين إلى إغلاق الصفقات — عبر WhatsApp وTelegram والبريد الإلكتروني.

مبني على [OpenClaw](https://openclaw.dev)، تم اختباره في الواقع مع شركات التصدير B2B الحقيقية.

## البدء السريع

### الخيار A: مستخدمو OpenClaw (أمر واحد)

إذا كان لديك بالفعل [OpenClaw](https://openclaw.dev) قيد التشغيل:

```bash
clawhub install b2b-sdr-agent
```

تم. يقوم المهارة بتثبيت نظام السياق الكامل من 7 طبقات، وdelivery-queue، وsdr-humanizer في مساحة عملك. ثم قم بالتخصيص:

```bash
# قم بتحرير الملفات الرئيسية لعملك
vim ~/.openclaw/workspace/skills/b2b-sdr-agent/references/IDENTITY.md
vim ~/.openclaw/workspace/skills/b2b-sdr-agent/references/USER.md

# أو انسخها إلى مساحة العمل الرئيسية
cp ~/.openclaw/workspace/skills/b2b-sdr-agent/references/*.md ~/.openclaw/workspace/
```

استبدل جميع `{{placeholders}}` بمعلومات شركتك الفعلية، وسيكون وكيل AI SDR الخاص بك مباشرًا.

### الخيار B: النشر الكامل (5 دقائق)

#### 1. الاستنساخ والتكوين

```bash
git clone https://github.com/iPythoning/b2b-sdr-agent-template.git
cd b2b-sdr-agent-template

# قم بتحرير ملفات مساحة العمل السبعة لعملك
vim workspace/IDENTITY.md   # معلومات الشركة، الدور، خط الأنابيب
vim workspace/USER.md       # منتجاتك، ICP، المنافسون
vim workspace/SOUL.md       # شخصية الذكاء الاصطناعي والقواعد
```

#### 2. إعداد تكوين النشر

```bash
cd deploy
cp config.sh.example config.sh
vim config.sh               # املأ: IP الخادم، مفتاح API، رقم WhatsApp
```

#### 3. النشر

```bash
./deploy.sh my-company

# الإخراج:
# ✅ اكتمل النشر: my-company
# Gateway:  ws://your-server:18789
# WhatsApp: مفعّل
# Skills:   b2b_trade (28 مهارة)
```

هذا كل شيء. وكيل AI SDR الخاص بك مباشر على WhatsApp وجاهز للبيع.

## ما يفعله

### أتمتة المبيعات الكاملة لخط الأنابيب (10 مراحل)

| المرحلة | ما يفعله الذكاء الاصطناعي |
|-------|-----------------|
| **1. التقاط العملاء المحتملين** | اكتشاف تلقائي للرسائل الواردة (WhatsApp/Telegram/إعلانات CTWA)، إنشاء سجلات CRM |
| **2. تأهيل BANT** | محادثة طبيعية لتقييم الميزانية، السلطة، الحاجة، الجدول الزمني |
| **3. إدخال CRM** | التقاط البيانات المهيكلة — الاسم، الشركة، البلد، درجة ICP، اهتمام المنتج |
| **4. البحث والإثراء** | بحث Jina AI على الويب + تحليل موقع الشركة، خط أنابيب إثراء من 3 طبقات |
| **5. عرض الأسعار** | توليد عروض أسعار PDF تلقائيًا، متعدد اللغات، إرسالها للمالك للموافقة |
| **6. التفاوض** | تتبع العروض المضادة، التوصية بالاستراتيجية، التصعيد عند الحاجة |
| **7. التقارير** | تقارير خط الأنابيب اليومية 09:00، تنبيهات التوقف 15:00، ملخصات أسبوعية |
| **8. الرعاية** | متابعات تلقائية، أخبار الصناعة، رعاية ما بعد البيع، مراجعات ربع سنوية |
| **9. التواصل عبر البريد الإلكتروني** | سلاسل بريد إلكتروني بارد مخصصة (اليوم 1/3/7/14)، متابعة تلقائية |
| **10. التنسيق متعدد القنوات** | تنسيق عبر القنوات (WhatsApp + البريد الإلكتروني + Telegram)، تبديل تلقائي |

### الجدولة التلقائية (13 مهام Cron)

| الجدول | المهمة |
|--------|--------|
| كل 30 دقيقة | مسح صندوق بريد Gmail لردود العملاء |
| يوميًا 09:00 | تقرير خط الأنابيب للمالك عبر WhatsApp |
| يوميًا 10:00 | اكتشاف العملاء المحتملين بالذكاء الاصطناعي (تناوب الأسواق: أفريقيا/الشرق الأوسط/جنوب شرق آسيا/أمريكا اللاتينية) |
| يوميًا 11:00 | فحص سلسلة متابعة البريد الإلكتروني |
| يوميًا 15:00 | اكتشاف العملاء المحتملين المتوقفين |
| أسبوعيًا الأربعاء | حملة الرعاية |
| أسبوعيًا الجمعة | جمع معلومات المنافسين |
| أسبوعيًا الاثنين | تقرير الملخص الأسبوعي |

### محادثات شبيهة بالبشر
- يستجيب بلغة العميل (يتم اكتشافها تلقائيًا)
- وتيرة رسائل طبيعية مع مهارة delivery-queue (تأخيرات 3-90 ثانية، تقسيم الرسائل)
- إرسال واعٍ بالمنطقة الزمنية (تخطيط 7 مناطق زمنية للأسواق)
- التكيف الثقافي (الشرق الأوسط، أفريقيا، جنوب شرق آسيا، أمريكا اللاتينية، أوروبا)
- لا يكشف أبدًا عن هويته كذكاء اصطناعي — يقدم نفسه كمستشار مبيعاتك

### تسجيل ICP الديناميكي
- درجة أولية بناءً على 5 أبعاد مرجحة (حجم الشراء، مطابقة المنتج، المنطقة، القدرة على الدفع، السلطة)
- **يتعدّل تلقائيًا** بناءً على التفاعل: رد سريع +1، طلب عرض أسعار +2، ذكر منافس +2، عدم رد لمدة 7 أيام -1
- العملاء المحتملون الساخنون (ICP>=7) يتم تمييزهم تلقائيًا، ويُبلّغ المالك فورًا

### ذاكرة ذكية (3 محركات)
- **Supermemory**: ملاحظات البحث، معلومات المنافسين، رؤى السوق — يتم الاستعلام قبل التواصل
- **MemoryLake**: سياق الجلسة، ملخصات المحادثة — يتم استدعاؤها تلقائيًا لكل محادثة
- **MemOS Cloud**: أنماط السلوك عبر الجلسات — يتم التقاطها تلقائيًا

### نظام مكافحة فقدان الذاكرة من 4 طبقات

تفقد وكلاء الذكاء الاصطناعي السياق في المحادثات الطويلة وعبر الجلسات. **بنية مكافحة فقدان الذاكرة من 4 طبقات** تضمن أن وكيل AI SDR الخاص بك لا ينسى أبدًا:

```
الرسالة الواردة ──→ L1 استدعاء MemOS التلقائي (حقن الذاكرة المهيكلة)
    │
    ├──→ L3 تخزين ChromaDB لكل رسالة (عزل العملاء، وسم تلقائي)
    │
    ├──→ L2 ملخص استباقي عند 65% من الرموز (ضغط haiku، صفر فقدان معلومات)
    │
    └──→ L4 لقطة CRM يومية 12:00 (احتياطي للتعافي من الكوارث)
```

| الطبقة | المحرك | ماذا تفعل |
|--------|--------|-----------|
| **L1: MemOS** | ذاكرة مهيكلة | استخراج تلقائي لـ BANT والالتزامات والاعتراضات كل رسالة. حقنها في System Prompt عند بدء المحادثة. |
| **L2: ملخص استباقي** | مراقبة الرموز | ضغط عند 65% من استخدام السياق عبر نموذج فئة haiku. جميع الأرقام وعروض الأسعار والالتزامات محفوظة حرفيًا. |
| **L3: ChromaDB** | مخزن متجهات لكل رسالة | كل رسالة محادثة مخزنة بعزل `customer_id`. وسم تلقائي لعروض الأسعار والالتزامات والاعتراضات. استرجاع دلالي عبر الجلسات. |
| **L4: لقطة CRM** | نسخ احتياطي يومي | تخزين حالة خط الأنابيب الكاملة يوميًا في ChromaDB كتعافي من الكوارث. إذا فشلت أي طبقة، L4 لديها البيانات. |

**النتيجة**: وكيل AI SDR الخاص بك يتذكر كل عميل وكل عرض أسعار وكل التزام — حتى بعد 100+ رسالة أو أسابيع من الصمت أو إعادة تشغيل النظام.

## المهارات

قدرات مبنية مسبقًا توسع وكيل AI SDR الخاص بك:

| المهارة | الوصف |
|-------|-------------|
| **delivery-queue** | جدولة الرسائل مع تأخيرات شبيهة بالبشر. حملات التنقيط، متابعات موقوتة. |
| **supermemory** | محرك الذاكرة الدلالية. التقاط تلقائي لرؤى العملاء، البحث عبر جميع المحادثات. |
| **sdr-humanizer** | قواعد المحادثة الطبيعية — الوتيرة، التكيف الثقافي، الأنماط المضادة. |
| **lead-discovery** | اكتشاف العملاء المحتملين بالذكاء الاصطناعي. البحث على الويب عن مشترين محتملين، تقييم ICP، إدخال تلقائي في CRM. |
| **chroma-memory** | تخزين المحادثات لكل رسالة مع عزل العملاء، ووسم تلقائي، ولقطات CRM. |
| **telegram-toolkit** | أوامر البوت، لوحات المفاتيح المضمّنة، معالجة الملفات الكبيرة، واستراتيجيات الأسواق التي تعتمد Telegram أولاً. |
| **quotation-generator** | توليد فواتير أولية PDF تلقائيًا مع ترويسة الشركة، دعم متعدد اللغات. |

## لوحة التحكم

بعد النشر، يأتي وكيل AI SDR مع لوحة تحكم ويب مدمجة:

```
http://YOUR_SERVER_IP:18789/?token=YOUR_GATEWAY_TOKEN
```

تعرض لوحة التحكم:
- حالة البوت في الوقت الفعلي واتصال WhatsApp
- سجل الرسائل وسلاسل المحادثات
- حالة تنفيذ مهام Cron
- مراقبة صحة القنوات

يتم إنشاء الرمز تلقائيًا أثناء النشر ويُطبع في الإخراج. احتفظ به بشكل خاص — أي شخص لديه URL+الرمز لديه وصول كامل.

> **ملاحظة أمنية**: اضبط `GATEWAY_BIND="loopback"` في config.sh لتعطيل الوصول عن بُعد للوحة التحكم. الافتراضي هو `"lan"` (يمكن الوصول إليه من الشبكة).

## النشر

### المتطلبات الأساسية
- خادم Linux (يُوصى بـ Ubuntu 20.04+)
- Node.js 18+
- مفتاح API لنموذج الذكاء الاصطناعي (OpenAI، Anthropic، Google، Kimi، إلخ.)
- حساب WhatsApp Business (اختياري ولكن موصى به)

### التكوين

يوجد جميع التكوين في `deploy/config.sh`. الأقسام الرئيسية:

```bash
# الخادم
SERVER_HOST="your-server-ip"

# نموذج الذكاء الاصطناعي
PRIMARY_API_KEY="sk-..."

# القنوات
WHATSAPP_ENABLED=true
TELEGRAM_BOT_TOKEN="..."

# CRM
SHEETS_SPREADSHEET_ID="your-google-sheets-id"

# المسؤول (من يمكنه إدارة الذكاء الاصطناعي)
ADMIN_PHONES="+1234567890"
```

### تكوين WhatsApp

بشكل افتراضي، يقبل وكيل AI SDR الرسائل من **جميع جهات اتصال WhatsApp** (`dmPolicy: "open"`). هذا هو الإعداد الموصى به لوكلاء المبيعات — تريد أن يتمكن كل عميل محتمل من الوصول إليك.

| الإعداد | القيمة | المعنى |
|---------|--------|--------|
| `WHATSAPP_DM_POLICY` | `"open"` (افتراضي) | قبول الرسائل من أي شخص |
| | `"allowlist"` | قبول فقط من `ADMIN_PHONES` |
| | `"pairing"` | يتطلب رمز الاقتران أولاً |
| `WHATSAPP_GROUP_POLICY` | `"allowlist"` (افتراضي) | الرد فقط في المجموعات المدرجة في القائمة البيضاء |

للتغيير بعد النشر، عدّل `~/.openclaw/openclaw.json` على الخادم:

```json
{
  "channels": {
    "whatsapp": {
      "dmPolicy": "open",
      "allowFrom": ["*"]
    }
  }
}
```

ثم أعد التشغيل: `systemctl --user restart openclaw-gateway`

### عزل IP لـ WhatsApp (متعدد المستأجرين)

عند تشغيل وكلاء متعددين على نفس الخادم، يجب أن يكون لكل منهم IP خروج فريد حتى يرى WhatsApp أجهزة مستقلة. هذا يمنع الحظر المتقاطع بين الحسابات.

```bash
# بعد نشر عميل، اعزل IP WhatsApp الخاص به:
./deploy/ip-isolate.sh acme-corp

# أو مع منفذ SOCKS5 محدد:
./deploy/ip-isolate.sh acme-corp 40010
```

**كيف يعمل:**

```
                  ┌─ wireproxy :40001 → WARP Account A → CF IP-A
                  │    ↑
tenant-a ─────────┘    ALL_PROXY=socks5://host:40001

tenant-b ─────────┐    ALL_PROXY=socks5://host:40002
                  │    ↓
                  └─ wireproxy :40002 → WARP Account B → CF IP-B
```

كل مستأجر يحصل على:
- حساب [Cloudflare WARP](https://1.1.1.1/) مجاني مخصص
- مثيل [wireproxy](https://github.com/pufferffish/wireproxy) معزول (~4MB ذاكرة)
- IP خروج Cloudflare فريد لجميع حركة المرور الصادرة (بما في ذلك WhatsApp)

للتفعيل التلقائي أثناء النشر، اضبط `IP_ISOLATE=true` في `config.sh`.
