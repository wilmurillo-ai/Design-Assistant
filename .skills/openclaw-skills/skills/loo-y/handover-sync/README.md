# handover-sync

English version: [README.en.md](./README.en.md)

`handover-sync` 是一个面向开发会话收尾阶段的文档同步 skill。它的目标不是生成一段泛泛的“今日总结”，而是基于仓库中的真实证据，产出可交接、可追溯、可继续推进的 handoff 文档。

这个 skill 适合在以下场景使用：

- 结束当天开发前，补齐交接文档
- 需要把今天的改动、问题、验证和 TODO 系统化记录下来
- 想同步更新 `handover.md`、`README.md` 和相关运行说明
- 希望下一个人或下一个 agent 能直接接手，而不是重新读完整段对话

## 它解决什么问题

很多“会话总结”只有一句话，例如“今天修了几个 bug，后续继续优化”。这种内容几乎不能交接，也不能帮助后续继续工作。

`handover-sync` 的作用是把开发收尾动作标准化，确保文档能回答三个核心问题：

1. 今天到底改了什么
2. 现在还有什么风险、限制或未完成事项
3. 下一个人接手时应该先做什么

## 核心原则

- 以仓库证据为准，不凭记忆写总结
- 写事实，不写空泛表述
- 明确区分“已经验证”和“尚未验证”
- 明确标记临时方案、MVP 方案和长期目标
- 默认同时同步多个文档，而不是只改一个文件

## 默认工作流程

`handover-sync` 的执行流程大致如下：

1. 确认仓库根目录和总结日期
2. 读取现有交接文档和相关说明文件
3. 根据用户语言选择中文或英文模板
4. 通过仓库检查收集证据，例如：
   - `git status --short`
   - `git diff --name-only`
   - `git diff --stat`
   - `git log --since "<date> 00:00:00" --oneline`
   - 读取当天修改过的关键文件
5. 从证据中整理出可交接叙事
6. 默认同步更新：
   - `handover.md`
   - `README.md` 中已过时的相关部分
   - 必要的 TODO / runbook / 使用说明
7. 回读修改后的文档，确认信息足够支持下一位接手

## 输出内容要求

这个 skill 要求 handoff 文档至少覆盖以下内容：

- 本次会话目标 / 当前阶段目标
- 当前仓库状态
- 实际遇到的问题
- 原因判断与结论
- 已落地的修复
- 已执行的验证
- 踩过的坑、被否定的方案、关键约束
- 接手后如何继续
- 当前仍存在的问题和边界
- 最终产品目标（如果讨论过）
- 可执行的后续 TODO

重点不是“写得像总结”，而是“写完后别人真的能接着做”。

## 语言策略

- 用户本轮使用中文时，输出中文
- 用户本轮使用英文时，输出英文
- 用户明确要求双语时，输出双语

如果现有文档长期使用某种语言，则默认保持该文档原有语言，除非用户明确要求切换。

## 默认更新范围

除非用户明确缩小范围，否则这个 skill 默认会一起更新：

- `handover.md`
- `README.md` 中与本次结论直接相关的部分
- 附近相关的 TODO、运行说明或注意事项

如果项目当前理解已经变化，就不应该只改其中一个文件。

## 仓库结构

当前仓库的关键文件如下：

- `SKILL.md`
  这个 skill 的正式定义，包括用途、工作流、写作规则和默认更新策略。
- `references/session_summary_template.md`
  中文交接模板。
- `references/session_summary_template.en.md`
  英文交接模板。

## 适合怎样的请求

下面这类请求都适合触发这个 skill：

- “总结一下今天做了什么”
- “更新交接文档”
- “把 README 和 handover 一起同步一下”
- “把今天的问题、修复、验证和 TODO 记下来”
- “整理一下当前状态，方便下个人接手”

## 快速使用示例

下面是几种直接可用的触发方式：

```text
总结一下今天做了什么，并同步更新 handover.md 和 README.md
```

```text
把今天的问题、修复、验证结果和后续 TODO 按固定模板写进交接文档
```

```text
在结束前整理一下当前状态，方便下一个 agent 接手
```

如果你希望范围更明确，也可以直接限定时间窗口或目标，例如：

```text
按这个 skill 总结今天 2026-03-22 的改动，只更新 handover.md
```

```text
从上次 handover 之后到现在，补一份英文交接并同步 README
```

## 不做什么

这个 skill 不是为了：

- 生成聊天式简短 recap
- 替代真实代码检查
- 用模糊措辞掩盖未验证内容
- 只写结论而不写证据和上下文

如果没有实际检查仓库、变更文件和验证动作，就不符合这个 skill 的设计目标。

## 使用建议

如果你想让交接文档稳定可用，推荐在每次阶段性收尾时都使用这个 skill，而不是等到上下文已经混乱后再补文档。

这样可以让仓库里的 `handover.md` 和 `README.md` 始终反映真实状态，减少信息断层，也能显著降低换人接手成本。
