# Session Summary Template

这个文件用于约定本项目每次开发结束后的总结和交接方式。

以后如果你说：

- “按 `SESSION_SUMMARY_TEMPLATE.md` 更新总结”
- “按固定交接模板更新 README / handover”
- “把今天的问题、修复、验证、坑点和 TODO 一起补齐”

就按本文件的结构执行，而不是只写一句笼统 TODO。

## 目标

这份 handover 不是“今天做了什么”的流水账，而是要做到：

1. 当前对话结束后，隔一天再看也能迅速恢复上下文。
2. 即使后续换人接手，也能看懂今天到底改了什么、为什么这样改、踩过哪些坑、现在还缺什么。
3. 接手方只看 `handover.md` 和相关 `README.md`，就能知道下一步先做什么，以及怎么开始验证。

## 默认更新范围

每次需要做“阶段总结”时，至少同步更新以下内容：

1. `handover.md`
2. `README.md` 中与当前结论直接相关的部分
3. 如有必要，补充运行说明、注意事项、命令示例或 TODO

不要只补其中一个文件，除非用户明确要求。

## 固定输出结构

每次 handover 至少覆盖以下几部分。

### 1. 本次会话目标 / 当前阶段目标

先交代这次会话原本想解决什么：

- 目标功能或问题范围
- 涉及的模块、页面、脚本或构建链路
- 这次改动是最终方案、阶段方案，还是 MVP 临时方案

这一段是为了让接手方先建立方向感，而不是直接掉进细节里。

### 2. 当前仓库状态

必须写清楚当前代码处于什么状态，例如：

- 这次主要改了哪些文件
- 改动是否已经提交
- 是否有仍未整理的工作区改动
- 是否依赖某些配置、环境变量、外部工具、账号权限

如果仓库状态会影响接手方式，必须明确写出来。

### 3. 今天实际遇到的问题

必须写清楚：

- 具体报错或异常现象
- 触发条件
- 为什么它是问题
- 影响到什么核心能力

不要写成泛泛而谈的“有兼容性问题”。
要写成可定位的描述，例如：

- Windows 下 `tkinter` 可导入，但 `tk.Tk()` 因 `Tcl/Tk` 初始化失败直接报错
- `os.startfile()` 播放音频是非阻塞的，因此无法保证上一条语音播放完再进入下一轮

### 4. 原因判断与结论

必须明确写出这次排查后的判断结果，包括：

- 问题是在代码逻辑、运行环境、第三方依赖、系统权限、分发方式，还是产品假设本身
- 哪些猜测已经被排除
- 最终确认的根因或当前最可信结论

如果不是 100% 确认，也要写“当前判断”而不是装作确定。

### 5. 这次已经落地的修复

必须写清楚：

- 改了哪些文件
- 每个改动解决什么问题
- 当前方案是长期方案还是 MVP 临时方案

推荐格式：

- 文件路径
- 改动内容
- 作用

### 6. 已验证结果

必须写清楚这次实际验证过什么，而不是只写“已修复”。

例如：

- 是否成功启动
- 是否成功进入主循环
- 是否真实触发过区域选择
- 是否真实播放过两段音频并确认阻塞顺序
- 是否跑过 `compileall`、测试、构建或其他基本验证

不要写“已验证”四个字就结束，必须说清楚验证动作和结果。

### 7. 踩过的坑 / 已否定方案 / 关键约束

这一段必须有。

要写清楚：

- 哪些方案已经尝试过但被否定
- 为什么这些方案不成立
- 当前方案依赖哪些前提条件
- 后续接手时最容易重复踩的坑是什么

这一段的作用是防止后续接手的人把今天已经踩过的坑再踩一遍。

### 8. 接手后如何继续

必须明确写出接手方的启动路径：

- 应该先看哪个文件
- 应该先跑哪个命令
- 应该先验证哪个行为
- 什么结果代表“当前状态正常”
- 如果结果不对，优先排查哪里

不要让接手方自己猜“下一步大概做什么”。

### 9. 当前仍存在的问题 / 边界

必须单列写出还没解决的问题，避免后续接手的人误以为已经彻底完成。

例如：

- 只是开发态可用，还不适合分发
- 仍依赖某个系统工具
- 仍未做窗口级捕获
- 仍无自动化测试
- 只验证过单平台，未验证多平台

### 10. 最终想实现的产品目标

如果这次讨论已经涉及“长期方案”或“分发目标”，必须写明最终目标，而不只是写眼前补丁。

例如：

- 最终交付物应是 Windows 安装包或桌面应用
- 用户应开箱即用，而不是面对 `.venv`、`pip install`、`ffmpeg`
- 当前阶段只是为了先打通主链路，不代表最终实现方式

### 11. 后续 TODO

TODO 不能只写一句话，必须拆成可执行事项。

每条 TODO 要尽量回答：

- 要做什么
- 为什么要做
- 做完后解决什么问题

如果有先后顺序，必须按顺序排。

例如：

1. 自动检测 `ffplay`
2. 缺失时自动下载并缓存
3. 播放逻辑优先使用项目内缓存
4. 后续做成安装包

## 写法要求

- 少写空话，不写“优化体验”“提升稳定性”这种没有上下文的句子。
- 多写具体事实、具体约束、具体文件、具体命令、具体行为。
- 如果这次只是 MVP 修复，要明确标注“这是 MVP 方案，不是最终分发方案”。
- 如果用户已经表达了产品目标，例如“最终要做成给普通用户安装的分发包”，必须把这个目标写进总结。
- 如果这次没验证，就直接写“未验证”以及原因，不要假装已经验证过。
- 如果今天试过错误方向，也要写出来，避免后人重复投入。

## 默认执行规则

以后只要用户提出以下任一意图，就默认按本模板执行：

- 要求“总结一下今天做了什么”
- 要求“更新交接文档”
- 要求“补一下 TODO”
- 要求“把问题和后续方案写下来”
- 要求“方便以后换人接手”

## 推荐口令

以后你可以直接说：

- “按 `SESSION_SUMMARY_TEMPLATE.md` 更新今天的交接”
- “按固定总结模板补 handover 和 README”
- “把今天的问题、修复、验证、坑点、TODO 按模板写进去”

看到这些说法时，就按本文件执行。
