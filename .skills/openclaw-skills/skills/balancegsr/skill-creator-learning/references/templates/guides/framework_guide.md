# 认知框架生成指南

> 本文件在结项模式触发时加载（仅完整模式）。指导如何从模块总结中提炼项目级认知框架。

## 什么是认知框架

认知框架不是学习笔记的汇总，而是对整个知识领域的**重新建构**——用学完之后的视角重新组织知识结构，呈现各部分之间的关系和判断标准。

好的认知框架应该能让未来的自己（或他人）快速重建对这个领域的理解，而不需要重新阅读所有笔记。

## 生成步骤

### 第一步：加载素材

读取所有模块总结（`summary_module_N.md`），提取：
- 每个模块的核心框架
- 每个模块的关键决策/判断标准
- 每个模块的"仍需深化"

同时参考 learning_plan.md 中的模块结构，理解原始规划意图。

### 第二步：识别知识类型

根据学习内容的整体特征，判断最适合的框架形式：

| 知识类型特征 | 推荐框架形式 | 适用场景 |
|---|---|---|
| 系统有明确的层级或组件关系 | 分层图 / 架构图 | 技术架构、组织结构、系统设计 |
| 核心是做选择和判断 | 决策树 / 判断流程 | 方案选型、诊断排查、策略选择 |
| 核心是按步骤执行 | 步骤清单 / 检查表 | 操作流程、方法论、工作流 |
| 核心是理解概念关系 | 概念关系图 / 对比矩阵 | 理论体系、概念辨析、分类学 |

如果领域跨多种类型，可以组合使用。与用户确认框架形式后再开始生成。

### 第三步：构建核心结构

这是框架最重要的部分。目标是用用户自己的语言重建知识领域的架构：

- 不是按模块顺序堆叠（那是学习顺序，不是知识结构）
- 而是按知识本身的逻辑关系组织
- 突出模块之间的连接——哪些概念跨越了多个模块？哪些模块是理解另一个模块的前提？

与用户讨论核心结构，确认是否反映了他们的真实理解。

### 第四步：提炼判断标准

从各模块总结的"关键决策/判断标准"中提炼：

- 学完后能做出哪些之前不能做的决策？
- 面对某类问题时，判断的依据和优先级是什么？
- 有哪些常见的误区或陷阱？

这部分的价值在于：知识会遗忘，但判断标准留存下来就能指导实践。

### 第五步：标注关联和边界

- 这个知识领域与用户已有的其他知识/技能如何连接？
- 这个领域的边界在哪？学完之后还有哪些方向可以继续深入？
- 汇总所有模块的"仍需深化"，作为后续学习路线图

### 第六步：保存

按照笔记格式规范中的"项目认知框架"模板，保存为 `framework_[项目主题].md`。

保存后通知用户项目完成，建议花几分钟回顾框架文档，确认是否完整反映了学习成果。
