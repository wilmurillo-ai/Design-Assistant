💡 Redis 为什么快？4个核心原因
  • ① 纯内存操作 — 内存读写 ns 级，磁盘 ms 级，差1000倍
  • ② 单线程模型 — 无锁、无切换、无竞态，O(1)操作直接定位
  • ③ I/O多路复用 — epoll 一个线程处理上万并发连接
  • ④ 高效数据结构 — String/Hash/List/Set/Sorted Set 都是 O(1) 或 O(log N)
内存数据库 单线程 epoll
📖 为什么早期坚持单线程？
  • ✅ 够用 — 早期 CPU 单核，单线程跑满足够快，官方跑分10-30万 QPS
  • ✅ 简单 — 无锁无竞态，代码只有几千行，好维护
  • ✅ 正确选择 — 内存场景下，单线程是最优解，不是缺陷
💡 设计哲学：简单才是力量
"如果你需要多线程，说明场景已经不适合 Redis 了" — Antirez
🔧 6.0 引入 I/O 多线程：解决了什么？
  • ❌ 单线程 recv() 串行 — 1万个连接同时就绪要排队，总耗时巨大
  • ✅ 多线程并发读 socket — 不用排队，并行处理多个连接
  • ⚠️ 命令执行保持单线程 — 多线程写内存要加锁，开销反而更大
架构公式：
I/O线程池（并发读/写）→ 主线程（命令执行）→ I/O线程池（并发写回）
per-client buffer 不需要锁 6.0新特性
⏱️ 12步请求链路时间分解
① 生成命令 ~1-10µs （可忽略）
② TCP RTT 200-500µs（同机房） 🔴 最严重瓶颈
⑤ recv() 系统调用 ~200ns-1µs （系统开销）
⑧ 执行命令 O(1) ~1-5µs / O(N²) ~200+µs （看命令复杂度）
⑩ send() + 网络回传 ~200-500µs （网络延迟）
🔴 网络RTT最贵 🟡 序列化开销 🟢 buffer读写极快
🚀 实战优化：让单机 QPS 达到最高
  • 1️⃣ Pipeline 批量命令 — 1万条命令从1万次RTT降到100次，省99%时间
  • 2️⃣ 连接池 — 消除建连开销（2-3个RTT），复用连接
  • 3️⃣ 避免慢命令 — KEYS*/SORT/HGETALL 大集合 换成 SCAN/ZSCAN
  • 4️⃣ Lua 脚本 — 原子化批量操作，减少 RTT
  • 5️⃣ 控制 value 大小 — 控制在 10KB 以内，避免大 key
🔌 连接池大小怎么定？
  • 📐 公式： 连接池 ≈ 预期QPS / 5000 + 10~20余量
  • 💡 单连接能力： 无Pipeline ~3000QPS，用Pipeline放大几十倍
  • ⚠️ 不是越大越好 — 太大浪费资源，可能触发 maxclients 上限
推荐值：
1万QPS → 15-20连接 | 5万QPS → 30-50 | 10万+ → 50-100+分片
⚙️ I/O 线程配置多少合适？
  • 📐 公式： min(CPU核心数 - 2, 8)
  • 💡 主线程也要CPU — 别让I/O线程把主线程的CPU抢了
  • ⚠️ 超过8个无意义 — 收益递减，开销增大
配置示例（redis.conf）：
io-threads-do-reads yes
io-threads 6
8核机器推荐6线程 4核不建议开 压测验证

🎯 总结：两条核心原则

① 减少网络往返 — Pipeline、Lua脚本、连接池，核心都是减少 RTT

② 简化命令复杂度 — O(1)优先，避免大key，不用O(N²)命令

单机组性能目标：简单GET/SET 同机房 30-50万QPS
