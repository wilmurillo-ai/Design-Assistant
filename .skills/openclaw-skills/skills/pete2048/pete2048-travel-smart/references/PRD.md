# TravelSmart PRD - 智能出行决策助手

> 本文档基于真实用户场景编写，用于指导 V1 开发方向。

---

## 一、真实用户痛点

### 痛点 A：高速上突然想下高速吃饭

> **场景**：五一/国庆自驾出游，高速上突然发现前面大堵车。心想反正堵着也是堵着，不如先下高速找个地方吃饭休息。
>
> **问题**：随便选了个出口下去，结果离出口 10km 才有镇子，绕了一大圈还找不到好吃的。浪费时间又折腾。
>
> **本质**：我不知道「哪个出口下去之后附近有好吃的、而且离得不远」。导航只给我路线，不给我决策建议。

### 痛点 B：半路临时决定找地方住

> **场景**：下午 5 点从景点出来，发现到下个目的地还要 4 小时，精力不够了，决定中途住下。但不知道哪里合适。
>
> **顾虑**：要有停车场（我开车）、价格合适、入住人数对、第二天去景点顺路、最好回程也顺路。
>
> **问题**：美团搜酒店只给我「附近有什么」，不给我「综合我所有条件哪个最优」。我要自己在十几个选项里权衡。

### 痛点 C：景点太堵，想停车打车进去

> **场景**：开车去某个热门景点，入口堵死了，车根本开不进去。想找个地方停车，然后打车进去。
>
> **顾虑**：哪里好停车？哪里打车方便？哪个点离景点最近、不走冤枉路？
>
> **问题**：滴滴/高德只告诉我怎么叫车，不告诉我「应该在哪个具体位置停车等车最省事」。

## 二、市场竞品分析

### 现有产品的盲区

| 产品 | 解决了什么 | 没解决什么 |
|------|----------|----------|
| 高德/百度地图 | 导航路线、实时路况 | 不告诉你「出口附近有什么好」 |
| 美团/大众点评 | 餐饮/酒店评分 | 不知道你在高速上，不考虑绕行代价 |
| 携程旅行 | 酒店+门票 | 不做「多因子决策」，只是搜索工具 |
| 滴滴/嘀嗒 | 打车 | 不告诉你「最佳停车候车点」 |

### 结论

**没有任何产品专门解决「自驾途中做决策」的问题**。市场存在空白。

---

## 三、产品定位

### 一句话定位

> TravelSmart 是「**自带决策能力的地图**」——不只告诉你哪里近，而是告诉你「**综合最优的决策点在哪里、为什么**」。

### 核心差异化

- **知道你在「路上」**：不只是搜索附近，而是理解你「刚从高速下来/正在前往景点/中途需要停留」的上下文
- **考虑绕行代价**：把「距离」和「好不好」放在一起权衡，不只是排序
- **多因子综合决策**：一个推荐结果，背后同时考虑了距离、评分、价格、顺路程度

### 目标用户

- **核心**：自驾游爱好者（60%汽油+%25高速费车主）
- **次要**：节假日长途开车出行人群
- **扩展**：商业用途（物流/车队调度）

## 四、V1 功能定义

**三个独立场景，按优先级排序：**

### 场景 1：高速出口推荐（V1 核心，优先级 P0）

**一句话**：告诉我在哪个高速出口下，附近有好吃的+好玩的，绕行不远。

**用户输入**：
- 高速名称（如 G4、京港澳）
- 当前位置（经纬度，或直接输入服务区/地名）
- 目的地（可选，用于判断回程顺路）

**系统输出**：
- 推荐 3 个最优出口，每个包含：
  - 距当前距离（km）
  - 绕行代价（km）
  - 餐饮/景点评分（高德 POI 聚合）
  - 是否有连锁品牌/农家乐
  - 综合打分及理由

**技术方案**：
1. 高德「高速出口 POI」搜索
2. 高德「周边搜索」找出口 5km 内餐饮/景点
3. 高德「驾车路线规划」计算绕行距离
4. 加权评分算法排序
5. LLM 生成推荐理由（可选）

---

### 场景 2：途中住宿推荐（V1 扩展，优先级 P1）

**一句话**：告诉我附近哪个酒店「有停车场+价格合适+第二天行程顺路」。

**用户输入**：
- 当前位置（经纬度）
- 住宿预算（价格区间）
- 入住人数
- 次日目的地（可选，判断顺路程度）
- 常用出发地（可选，判断回程是否顺路）

**系统输出**：
- 推荐 3 个最优酒店，每个包含：
  - 距当前位置距离
  - 价格 + 星级/评分
  - 是否有停车场
  - 距次日目的地距离
  - 综合打分及理由

**技术方案**：
1. 高德「周边搜索」半径 10km 内酒店
2. 高德「地理编码」次日目的地
3. 高德「驾车路线规划」计算到次日目的地距离
4. 价格/评分/距离/顺路程度多因子评分
5. LLM 生成推荐理由（可选）

---

### 场景 3：景点打车点推荐（V1 探索，优先级 P2）

**一句话**：告诉我应该在哪个位置停车，然后打车去景点最省事。

**用户输入**：
- 出发地（当前位置）
- 目的地（景点名称）

**系统输出**：
- 推荐 3 个最优打车点，每个包含：
  - 距出发地距离
  - 距目的地距离
  - 预估打车费用
  - 步行距离
  - 停车难度（POI 密度估算）
  - 综合打分及理由

**技术方案**：
1. 高德「驾车路线规划」找最近可行停车区
2. 高德「周边搜索」停车 POI
3. 高德「距离测量」各点到目的地
4. 多因子评分（距离近+好停车+好打车）

## 五、评分算法设计

### 核心公式

```
Score = 距离得分 × w1 + 评分得分 × w2 + 价格得分 × w3 + 顺路得分 × w4
```

### 各场景权重

| 场景 | w1(距离) | w2(评分) | w3(价格) | w4(顺路) |
|------|---------|---------|---------|---------|
| 高速出口 | 0.35 | 0.35 | 0.0 | 0.30 |
| 途中住宿 | 0.20 | 0.20 | 0.30 | 0.30 |
| 打车点 | 0.40 | 0.20 | 0.10 | 0.30 |

> 注：高速出口不关心价格（你总得吃饭）；住宿最关心价格；打车点最关心距离

### 分数归一化

- 距离：取反归一化（最近 = 100）
- 评分：直接用高德评分（0-5 → 0-100）
- 价格：相对预算偏离度（越接近预算中值越高）
- 顺路：绕行/总行程比例（越小越高）

## 六、技术架构

```
用户输入（地址/位置）
       ↓
┌──────────────────┐
│  Geocoding       │ ← 高德地理编码，地址 → 经纬度
└──────────────────┘
       ↓
┌─────────────────────────────────────────┐
│  Scene Router                            │
│  场景路由（高速/住宿/打车）                  │
└─────────────────────────────────────────┘
       ↓
┌─────────────────────────────────────────┐
│  Data Layer（并行请求）                    │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │ 高速出口POI │  │ 周边餐饮  │  │ 路线规划 │ │
│  │ 高德 API │  │ 高德 API │  │ 高德 API │ │
│  └──────────┘  └──────────┘  └────────┘ │
└─────────────────────────────────────────┘
       ↓
┌─────────────────────────────────────────┐
│  Scoring Engine                         │
│  多因子加权评分 + 排序                    │
└─────────────────────────────────────────┘
       ↓
推荐结果 → 用户
```

### V1 数据源（全部高德，免费 5000次/日）

| 数据 | API | 用途 |
|------|-----|------|
| 地理编码 | `/geocode/geo` | 地址→经纬度 |
| 高速出口 | `/place/text` + 关键字过滤 | 找出口 |
| 周边 POI | `/place/around` | 餐饮/酒店/停车场 |
| 驾车路线 | `/direction/driving` | 绕行距离计算 |
| 距离测量 | `/distance` | 点到点距离 |

### LLM 使用（可选，V1 暂缓）

- V1 不用 LLM，直接用规则评分，降低依赖和成本
- V2 加入 LLM 生成「推荐理由」，提升体验
- 模型：`MiniMax-M2.7-highspeed`（Max-极速版可用）
