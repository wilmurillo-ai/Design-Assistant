# 核心工作流

### 前置步骤：安装/升级 FlyAI CLI

在执行任何搜索之前，**必须先执行安装命令**（无论是否已安装，确保为最新版本）：

```bash
npm install -g @fly-ai/flyai-cli@latest --registry=https://registry.npmjs.org
```

> 💡 此命令会自动处理首次安装和版本升级，无需手动判断是否已安装。**安装后验证：**
```bash
flyai --help
```

**安装失败处理：**
| 情况 | 处理方式 |
|-----|---------|
| npm 未安装 | 提示用户先安装 Node.js (https://nodejs.org/) |
| 权限不足 | 建议使用 `sudo npm install -g @fly-ai/flyai-cli@latest --registry=https://registry.npmjs.org` 或使用 nvm 管理 Node |
| 网络问题 | 建议用户检查网络或使用国内镜像 `npm config set registry https://registry.npmmirror.com` |

**注意：** 此步骤只在首次使用时执行，后续调用会直接跳过已安装的情况。

### Step 1: 收集用户输入

使用 `ask_user_question` 工具收集以下信息：

#### 1.1 询问候选目的地

```
问题: "你纠结的目的地是哪几个？"
选项:
- "日本（东京/大阪/冲纳）"
- "东南亚（泰国/巴厘岛/越南）"
- "韩国（首尔/济州岛）"
- "我来输入具体目的地"
```

> **注意**：如果用户已经在触发语中明确了目的地（如"清迈还是巴厘岛"），则跳过此步骤。

#### 1.2 询问出发城市

```
问题: "你从哪个城市出发？"
选项:
- "上海"
- "北京"
- "广州/深圳"
- "其他城市（我来输入）"
```

#### 1.3 询问出行时间

```
问题: "你计划什么时候出行？"
选项:
- "1个月内"
- "1-2个月后"
- "2-3个月后"
- "我有具体日期（我来输入）"
```

#### 1.4 询问出行天数

```
问题: "计划玩几天？"
选项:
- "3-4天（周末短途）"
- "5-7天（小长假）"
- "7天以上（深度游）"
```

#### 1.5 询问在意维度（可选）

```
问题: "你最在意哪个方面？"
选项:
- "预算优先 - 花最少的钱"
- "景点多 - 玩的地方要多"
- "适合拍照 - 网红打卡点"
- "美食丰富 - 吃货天堂"
multiSelect: false
```

**收集完成后**，整理用户信息：
- **候选目的地**: 2-3个城市/地区
- **出发城市**: 用户所在城市
- **出行日期**: 出发日期和返回日期
- **在意维度**: 预算优先 / 景点多 / 适合拍照 / 美食丰富

如果目的地超过3个，提示：
> "建议先选2-3个重点对比，太多反而更纠结哦～"

### Step 2: 并行查询数据

为每个目的地**同时**调用三个搜索能力。

#### 2.1 查询机票

```bash
/flyai search-flight --origin "{出发城市}" --destination "{目的地}" --dep-date {出发日期} --back-date {返回日期} --sort-type 3
```

> **⚠️ SSL 证书问题**：如果通过终端执行 flyai CLI 命令时遇到 `UNABLE_TO_GET_ISSUER_CERT_LOCALLY` 错误，需要在命令前加上环境变量：
> ```bash
> NODE_TLS_REJECT_UNAUTHORIZED=0 flyai search-flight ...
> ```

**提取信息**:
- 最低价/最高价区间
- 飞行时长
- 是否有直飞

#### 2.2 查询酒店

```bash
/flyai search-hotel --dest-name "{目的地}" --check-in-date {入住日期} --check-out-date {退房日期} --sort rate_desc
```

**提取信息**:
- 每晚价格区间
- 高分酒店数量（评分4.5+）
- 酒店类型分布

#### 2.3 查询景点

```bash
/flyai search-poi --city-name "{目的地城市}"
```

**提取信息**:
- 景点总数
- 按类别统计（寺庙古迹、自然风光、网红打卡、水上活动等）

### Step 3: 数据整理与评分

#### 3.1 景点类型星级评估

根据各目的地的景点数量和类型丰富度，给出各类别的星级评分（⭐1-5颗星）：

| 类别 | 评估依据 |
|------|----------|
| 寺庙古迹类 | 历史古迹、宗教场所、古镇古村数量 |
| 自然风光类 | 山湖田园、森林丛林、峡谷瀑布数量 |
| 网红打卡类 | 地标建筑、文创街区、特色市集数量 |
| 水上活动类 | 沙滩海岛、潜水、冲浪、漂流数量 |
| 亲子娱乐类 | 主题乐园、水上乐园、动物园、海洋馆数量 |

#### 3.2 综合成本估算

```
预估总花费 = 机票往返(取中位价) + 酒店(每晚均价 × 晚数)
```

标注：不含餐饮购物

### Step 5: 综合推荐逻辑

根据用户在意的维度给出推荐：

| 用户偏好 | 推荐逻辑 |
|----------|----------|
| 预算优先 | 推荐总花费最低的目的地 |
| 景点多 | 推荐POI数量最多的目的地 |
| 适合拍照 | 推荐网红打卡类评分最高的目的地 |
| 美食丰富 | 结合当地特色美食介绍推荐 |
| 亲子友好 | 推荐亲子娱乐类评分最高的目的地 |
| 海岛度假 | 推荐水上活动类评分最高的目的地 |

如果用户未指定偏好，按以下优先级综合推荐：
1. 性价比（总花费 / 景点数量）
2. 交通便利性（直飞 > 中转）
3. 景点丰富度
