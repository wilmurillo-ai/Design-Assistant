# 常见任务编排模式

针对高频出现的复杂任务类型，提供具体的编排方案。

---

## 模式 1：批量企业研究（20+ 家公司）

**典型触发**："研究这 200 家公司的基本信息"

**推荐策略**：策略 D（混合编排）

```
阶段 1 - 数据收集（策略 A 串行）：
  - 已知信息直接填（上市公司公开数据）
  - 不确定的搜索验证
  - 每 10 家写入一次 JSON 文件
  - 上下文膨胀时只保留当前批次在工作记忆中

阶段 2 - 数据整理（策略 B 并行，可选）：
  - 前提：数据已在文件中，sub-agent 只做格式化（不搜索）
  - 每个 sub-agent 处理一个分类的公司
  - 超时 3 分钟（纯文件处理）

阶段 3 - 输出生成（策略 A 串行）：
  - 生成 Excel 文件
  - 或分批写入飞书文档（每批 5-10 行）

阶段 4 - 验证：
  - 抽检 5-10 家公司数据准确性
  - 检查 Excel 格式/飞书表格完整性
```

**关键教训**（来自 2026-04-03 ISV 任务）：
- sub-agent 不要做搜索密集任务，会超时
- 飞书表格单次写入有行数限制，大表用 Excel
- 已知公司（上市公司）的公开数据不需要搜索

## 模式 2：大文档迁移/批量写入

**典型触发**："把这些内容搬到飞书知识库" / "批量创建 20 个文档"

**推荐策略**：策略 C（分段检查点）

```
每段 5-10 个文档：
  1. 创建文档（feishu_doc create / feishu_wiki create）
  2. 写入内容（feishu_doc write/append）
  3. 记录检查点：文档 token + 状态
  4. 验证：读回 1-2 个确认内容完整
  
失败处理：
  - 记录失败文档的 ID 和错误原因
  - 这段完成后统一重试失败的
  - 重试 1 次仍失败的标记为"需人工处理"
```

**飞书 API 注意**：
- 429 限流：每 2-3 个操作间隔 1 秒
- 文档 body 不要重复标题
- 用 append 不用 write（防覆盖）
- resource_deleted 立即放弃不重试

## 模式 3：多步骤配置/部署

**典型触发**："升级 OpenClaw" / "配置一个新环境"

**推荐策略**：策略 A（串行）+ 每步验证

```
每一步：
  1. 备份当前状态
  2. 执行变更（一次只改一个东西）
  3. 验证变更效果
  4. 记录到 daily notes
  5. 确认无误后下一步

回滚点：
  - 每步都有备份 → 可以逐步回滚
  - 回滚顺序：反向执行
```

**绝对不用 sub-agent**：配置操作需要完整上下文，一步出错要立即停。

---

## 模式 4：数据处理管道

**典型触发**："分析这个 Excel 的数据" / "处理这批 JSON"

**推荐策略**：根据数据量选择

- < 100 条：策略 A（直接处理）
- 100-500 条：策略 C（分段 50 条一批）
- 500+ 条：策略 D（Python 脚本批处理 + 主 session 验证）

```
大数据量建议用 Python 脚本：
  1. 主 session 写处理脚本
  2. exec 运行脚本（一次性处理全量数据）
  3. 主 session 验证输出
  4. 生成报告

优势：不消耗 LLM token，处理速度快，不怕上下文膨胀
```

---

## 模式 5：多角色审查流程

**典型触发**："投资决策" / "需要风控审查"

**推荐策略**：策略 A（串行 + 单个 sub-agent 审查）

```
主 session 完成研究报告 →
  启动 1 个 sub-agent 做独立审查（不是并行多个）→
  主 session 综合决策

Sub-agent 注意：
  - 只起 1 个，不要多个
  - 给完整自包含的 prompt + 待审查内容
  - 超时 5-8 分钟
  - 结果通过文件传递
```

## 反模式清单（不要这么干）

| 反模式 | 为什么不行 | 正确做法 |
|--------|-----------|---------|
| 给 sub-agent 重搜索任务 | 上下文膨胀 → 超时 → fallback 乱跑 | 主 session 做搜索 |
| 同时起 5+ sub-agent | EC2 资源撑不住 | ≤ 3 个并行 |
| sub-agent 往主 session 回消息 | announce 搞乱上下文 | 结果写文件 |
| 全量写入飞书文档 | write 覆盖可能丢数据 | 用 append/insert |
| 跑完不验收就交付 | "做了"≠"做对了" | 抽检 + 结构校验 |
| 中间结果只在上下文里 | compaction 来了全丢 | WAL：先写文件 |
| 不设检查点跑大批量 | 断了从头来 | 每 10-20 个存一次 |
