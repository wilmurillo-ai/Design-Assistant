## 🔴 核心检查项 (CORE)

### CORE-1: 业务规则完整性

**检查点**:
- [ ] 所有业务场景是否都有对应的处理规则
- [ ] 规则之间是否存在冲突或矛盾
- [ ] 规则是否有明确的优先级定义
- [ ] 边界条件是否有明确定义

**检查问题**:
1. 是否存在未定义处理规则的业务场景？
2. 不同规则之间是否存在逻辑冲突？
3. 当多个规则同时适用时，优先级如何确定？
4. 极端情况（最大值、最小值、空值）是否有处理规则？

**验收标准**:
- 所有业务场景覆盖率 100%
- 规则冲突数 = 0
- 优先级定义清晰度 ≥ 90%

**✅ 通过示例**:
```markdown
业务规则：用户申请贷款
- 规则 1: 贷款金额 ≤ 用户信用额度
- 规则 2: 贷款期限 ≤ 36 个月
- 优先级：规则 1 > 规则 2
- 边界：贷款金额最小 100 元，最大 100 万元
```

**❌ 失败示例**:
```markdown
业务规则：用户申请贷款
- 需要审核（未说明审核规则）
- 有额度限制（未说明具体额度）
```

### CORE-2: 输入输出定义

**检查点**:
- [ ] 所有功能的输入参数是否有明确定义
- [ ] 所有功能的输出结果是否有明确定义
- [ ] 输入数据的格式、类型、范围是否有约束
- [ ] 输出数据的格式、类型、范围是否有约束

**检查问题**:
1. 每个功能模块的输入参数是否完整列出？
2. 每个功能模块的输出结果是否完整列出？
3. 输入数据的合法性校验规则是否明确？
4. 输出数据的格式规范是否明确？

**验收标准**:
- 输入参数定义完整率 100%
- 输出结果定义完整率 100%
- 数据约束明确率 ≥ 95%

**✅ 通过示例**:
```markdown
功能：用户登录
输入：
- username: string, 必填，4-20 个字符
- password: string, 必填，8-32 个字符，包含大小写字母和数字
输出：
- token: string, JWT 令牌
- userInfo: object, 用户信息
  - userId: string
  - nickname: string
```

**❌ 失败示例**:
```markdown
功能：用户登录
输入：用户名和密码
输出：登录成功或失败
```

### CORE-3: 异常处理

**检查点**:
- [ ] 所有可能的异常情况是否有识别
- [ ] 每种异常情况是否有对应的处理策略
- [ ] 异常处理是否有用户友好的提示
- [ ] 异常恢复机制是否有定义

**检查问题**:
1. 是否识别了所有可能的异常情况（网络、数据、权限、系统等）？
2. 每种异常情况是否有明确的处理流程？
3. 用户看到的错误提示是否清晰、可理解？
4. 系统是否支持异常后的恢复操作？

**验收标准**:
- 异常情况识别覆盖率 ≥ 90%
- 异常处理策略完整率 100%
- 用户提示友好度 ≥ 85%

**✅ 通过示例**:
```markdown
异常情况：
1. 网络异常：显示"网络连接失败，请检查网络设置"，支持重试
2. 数据异常：显示"数据加载失败，请稍后重试"，记录日志
3. 权限异常：显示"您没有权限执行此操作"，引导申请权限
```

**❌ 失败示例**:
```markdown
异常情况：
- 出错了（未说明什么错误）
- 请联系管理员（未说明联系方式）
```

## 🟡 完善检查项 (COMPLETE)

### COMPLETE-1: 用户角色定义

**检查点**:
- [ ] 所有用户角色是否有明确定义
- [ ] 每个角色的权限范围是否清晰
- [ ] 角色之间的关系是否有说明
- [ ] 角色切换机制是否有定义

**检查问题**:
1. 系统涉及哪些用户角色？
2. 每个角色可以执行哪些操作？
3. 不同角色之间的权限边界在哪里？
4. 用户是否可以切换角色？如何切换？

**验收标准**:
- 角色定义完整率 100%
- 权限描述清晰度 ≥ 90%

**✅ 通过示例**:
```markdown
用户角色：
1. 普通用户：查看产品、购买产品、查看订单
2. 管理员：管理产品、管理订单、管理用户
3. 角色切换：不支持切换，一个账号一个角色
```

**❌ 失败示例**:
```markdown
用户角色：
- 有用户和管理员（未说明权限区别）
```

### COMPLETE-2: 业务流程覆盖

**检查点**:
- [ ] 主流程是否有完整描述
- [ ] 分支流程是否有完整描述
- [ ] 流程之间的衔接是否清晰
- [ ] 流程状态是否有明确定义

**检查问题**:
1. 核心业务流程是否有完整的步骤描述？
2. 条件分支（if/else）是否有明确说明？
3. 不同流程之间如何切换和衔接？
4. 每个流程节点的状态是否明确？

**验收标准**:
- 主流程覆盖率 100%
- 分支流程覆盖率 ≥ 85%
- 状态定义清晰度 ≥ 90%

**✅ 通过示例**:
```markdown
流程：订单支付
主流程：选择支付方式 → 确认支付 → 支付成功 → 更新订单状态
分支流程：
- 支付失败：显示失败原因 → 支持重试
- 支付超时：取消订单 → 通知用户
状态：待支付、支付中、已支付、支付失败、已取消
```

**❌ 失败示例**:
```markdown
流程：订单支付
- 用户支付，然后完成（未说明分支和状态）
```

### COMPLETE-3: 数据一致性

**检查点**:
- [ ] 数据来源是否有明确说明
- [ ] 数据存储方式是否有定义
- [ ] 数据同步机制是否有说明
- [ ] 数据冲突解决策略是否有定义

**检查问题**:
1. 数据从哪里来？（数据库、API、用户输入）
2. 数据如何存储？（表结构、字段定义）
3. 多系统/多端数据如何保持同步？
4. 数据冲突时以哪个为准？

**验收标准**:
- 数据来源明确率 100%
- 存储定义完整率 ≥ 90%
- 同步策略清晰度 ≥ 85%

**✅ 通过示例**:
```markdown
数据来源：
- 用户信息：用户注册时输入，存储于 MySQL users 表
- 订单信息：用户下单生成，存储于 MySQL orders 表
数据同步：
- APP 和 Web 端通过 API 实时同步
- 数据冲突：以服务器时间戳为准
```

**❌ 失败示例**:
```markdown
数据：
- 有用户数据和订单数据（未说明来源和存储）
```

### COMPLETE-4: 接口定义完整性

**检查点**:
- [ ] 所有外部接口是否有定义
- [ ] 接口请求参数是否有说明
- [ ] 接口响应格式是否有说明
- [ ] 接口错误码是否有定义

**检查问题**:
1. 系统需要调用哪些外部接口？
2. 每个接口的请求参数（名称、类型、必填）是否明确？
3. 每个接口的响应数据结构是否明确？
4. 接口错误码和错误信息是否有定义？

**验收标准**:
- 接口定义完整率 100%
- 参数说明清晰度 ≥ 95%

**✅ 通过示例**:
```markdown
接口：POST /api/v1/login
请求参数：
- username: string, 必填
- password: string, 必填
响应：
- code: number, 0 表示成功
- data: { token: string, userInfo: object }
错误码：
- 401: 用户名或密码错误

```

**❌ 失败示例**:
```markdown
接口：登录接口
- 传入用户名密码，返回 token
```

### COMPLETE-5: 验收标准可测性

**检查点**:
- [ ] 每个功能是否有明确的验收标准
- [ ] 验收标准是否可量化
- [ ] 验收标准是否可执行
- [ ] 验收标准是否有优先级

**检查问题**:
1. 每个功能点是否有对应的验收标准？
2. 验收标准是否可以用数字或明确的是/否来判断？
3. 测试人员是否可以根据验收标准执行测试？
4. 验收标准的优先级是否明确（P0/P1/P2）？

**验收标准**:
- 功能验收标准覆盖率 100%
- 可量化率 ≥ 90%
- 可执行率 100%

**✅ 通过示例**:
```markdown
功能：用户登录
验收标准：
- P0: 输入正确的用户名密码，登录成功（是/否）
- P0: 输入错误的密码，显示错误提示（是/否）
- P1: 连续 5 次错误，账号锁定 30 分钟（是/否）
```

**❌ 失败示例**:
```markdown
功能：用户登录
验收标准：
- 登录功能正常（无法量化）
- 用户体验良好（主观判断）
```

### COMPLETE-6: 验收标准格式 (GWT)

**检查点**:
- [ ] 验收标准是否使用 GWT 格式（Given-When-Then）
- [ ] 每条验收标准是否包含三要素（前置条件/操作/预期结果）
- [ ] GWT 格式是否正确（Given 描述前提、When 描述操作、Then 描述结果）
- [ ] 验收标准是否使用中文描述

**检查问题**:
1. 验收标准是否采用"Given...When...Then..."格式？
2. 每条验收标准是否包含前置条件（Given）？
3. 每条验收标准是否包含操作/事件（When）？
4. 每条验收标准是否包含预期结果（Then）？
5. 验收标准是否使用中文而非英文？

**验收标准**:
- GWT 格式覆盖率 ≥ 80%
- 三要素完整率 100%
- 中文描述率 100%

**✅ 通过示例**:
```markdown
验收标准：
1. Given 用户已注册，When 输入正确密码登录，Then 登录成功并跳转到首页
2. Given 用户密码错误，When 点击登录，Then 显示"密码错误"提示
```

**❌ 失败示例**:
```markdown
验收标准：
- 用户可以登录（不是 GWT 格式）
- Login should work（英文描述）
```

## 🟢 优化检查项 (OPTIMIZE)

### OPTIMIZE-1: 性能指标定义

**检查点**:
- [ ] 关键操作是否有响应时间要求
- [ ] 系统并发能力是否有定义
- [ ] 数据处理量是否有预估
- [ ] 性能优化策略是否有说明

**检查问题**:
1. 关键操作的响应时间要求是多少？（如：页面加载 < 2s）
2. 系统需要支持多少并发用户？
3. 预计每天/每月处理多少数据量？
4. 有哪些性能优化策略？（缓存、异步、分页等）

**验收标准**:
- 关键性能指标定义率 ≥ 80%
- 优化策略可行性 ≥ 85%

### OPTIMIZE-2: 安全合规检查

**检查点**:
- [ ] 敏感数据是否有加密说明
- [ ] 权限验证是否有定义
- [ ] 合规要求是否有说明
- [ ] 审计日志是否有定义

**检查问题**:
1. 敏感数据（密码、身份证号等）如何加密存储？
2. 权限验证在哪些环节执行？
3. 是否符合行业合规要求？（如金融、医疗）
4. 关键操作是否有日志记录？

**验收标准**:
- 安全措施覆盖率 ≥ 85%
- 合规要求明确率 ≥ 90%

- [ ] 架构是否支持水平扩展

**检查问题**:
1. 新增功能模块是否会影响现有功能？

## 📊 检查项使用说明

### 在 PRD 生成流程中的集成点

| 阶段 | 使用的检查项 | 集成方式 |
|------|-------------|---------|
| **需求分解** | CORE-1, CORE-2, CORE-3 | 生成检查清单，指导 AI 生成 |
| **PRD 生成** | COMPLETE-1 ~ COMPLETE-6 | 作为生成提示词的一部分 |
| **质量检查** | 全部 14 项 | 作为质量门禁的评分标准 |
| **优化建议** | OPTIMIZE-1 ~ OPTIMIZE-5 | 生成优化建议报告 |

### 检查项评分规则

| 等级 | 得分 | 说明 |
|------|------|------|
| **完全符合** | 100% | 所有检查点均满足 |
| **基本符合** | 70-99% | 大部分检查点满足 |
| **部分符合** | 40-69% | 部分检查点满足 |
| **不符合** | <40% | 大部分检查点不满足 |

**通过标准**:
- 核心检查项：必须 100% 通过
- 完善检查项：≥80% 通过
- 优化检查项：≥60% 通过
