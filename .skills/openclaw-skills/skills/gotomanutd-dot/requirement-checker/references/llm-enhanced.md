**版本**: 1.0  
**更新日期**: 2026-04-03

## 🤖 为什么需要 LLM 增强？

### 基础规则检查的局限性

| 问题 | 规则检查 | LLM 增强 |
|------|---------|---------|
| **语义理解** | ❌ 基于关键词匹配 | ✅ 理解上下文语义 |
| **误判率** | ❌ 较高（可能误报） | ✅ 较低（智能判断） |
| **建议质量** | ❌ 模板化建议 | ✅ 针对性建议 |
| **证据引用** | ❌ 无法定位原文 | ✅ 可引用具体段落 |
| **复杂逻辑** | ❌ 难以处理 | ✅ 可分析复杂关系 |

## 🔧 工作原理

### 混合检查架构

```
┌─────────────────────────────────────────┐
│         需求文档内容                      │
└───────────────┬─────────────────────────┘
                │
        ┌───────┴───────┐
        │               │
        ▼               ▼
┌──────────────┐  ┌──────────────┐
│  规则检查器   │  │  LLM 检查器   │
│  (关键词)    │  │  (语义分析)   │
└──────┬───────┘  └──────┬───────┘
       │                 │
       └────────┬────────┘
                │
                ▼
      ┌─────────────────┐
      │   结果合并器     │
      │  (去重 + 互补)   │
      └────────┬────────┘
               │
               ▼
      ┌─────────────────┐
      │   最终检查报告   │
      └─────────────────┘
```

### 检查流程

1. **规则检查**：快速扫描，基于关键词和正则表达式
2. **LLM 检查**：深度分析，基于语义理解
3. **结果合并**：
   - 去重：同一问题不重复报告
   - 互补：规则检查发现的问题补充给 LLM 结果
   - 优先级：LLM 结果优先（更准确）

## 📊 检查能力对比

### 示例 1: 流程描述检查

**需求内容**:
```
用户提交订单后，系统会进行一系列处理，最终完成支付。
```

**规则检查**:
```
✅ 通过（检测到"后"、"进行"等关键词）
```

**LLM 增强**:
```
⚠️ 警告
问题：检测到多步处理描述，但未明确具体步骤
建议：应详细说明"一系列处理"包含哪些步骤，如：
      1. 订单校验
      2. 库存扣减
      3. 支付发起
      4. 结果通知
原文："系统会进行一系列处理"
```

---

### 示例 2: 改造内容标注

**需求内容**:
```
修改资产查询接口，增加基金类型筛选功能。
```

**规则检查**:
```
⚠️ 警告：检测到改造内容涉及接口，但未使用括号【】标注
建议：请用【接口名称】形式明确标注
```

**LLM 增强**:
```
⚠️ 警告
问题：提到修改接口但未明确接口名称
建议：应明确写出接口名称，如【assetQueryApi】
      同时说明增加的具体字段和取值规则
原文："修改资产查询接口"
```

---

### 示例 3: 异常处理

**需求内容**:
```
系统异常时显示错误提示。
```

**规则检查**:
```
✅ 通过（检测到"异常"、"错误"等关键词）
```

**LLM 增强**:
```
⚠️ 警告
问题：异常处理描述过于笼统，未说明具体场景
建议：应分别说明以下异常的处理方式：
      - 网络异常：重试机制 + 用户提示
      - 数据校验失败：高亮错误字段
      - 权限不足：隐藏功能或提示
      - 系统超时：超时时间 + 重试策略
原文："系统异常时显示错误提示"
```

## ⚙️ 配置说明

### 环境变量

```bash
# 阿里云 DashScope（推荐）
export DASHSCOPE_API_KEY="sk-xxx"
export OPENAI_BASE_URL="https://dashscope.aliyuncs.com/compatible-mode/v1"

# 或使用 OpenAI
export OPENAI_API_KEY="sk-xxx"
export OPENAI_BASE_URL="https://api.openai.com/v1"
```

### 模型选择

| 模型 | 适用场景 | 成本 | 速度 |
|------|---------|------|------|
| **qwen-plus** | 日常检查（推荐） | ¥ | ⚡⚡⚡ |
| **qwen-max** | 重要文档评审 | ¥¥ | ⚡⚡ |
| **qwen-max-longcontext** | 超长文档（>10 万字） | ¥¥¥ | ⚡ |

### 命令行参数

```bash
# 指定模型
python check_requirement_llm.py --model qwen-max requirement.md

# 指定 API Key（覆盖环境变量）
python check_requirement_llm.py --api-key "sk-xxx" requirement.md

# 禁用 LLM（仅规则检查）
python check_requirement_llm.py --no-llm requirement.md
```

## 🎯 最佳实践

### 1. 日常开发：规则检查

```bash
# 快速自查，秒级反馈
python check_requirement.py requirement.md
```

### 2. 正式评审：LLM 增强

```bash
# 提交评审前深度检查
python check_requirement_llm.py --model qwen-max requirement.md
```

### 3. 批量检查：混合模式

```bash
# 先用规则检查快速筛选
for file in PRDs/*.md; do
  python check_requirement.py "$file" > /dev/null 2>&1
  if [ $? -ne 0 ]; then
    echo "需要改进：$file"
    # 对问题文档进行 LLM 深度检查
    python check_requirement_llm.py "$file"
  fi
done
```

### 4. CI/CD 集成

```yaml
# GitHub Actions 示例
- name: Check PRD Quality
  run: |
    python check_requirement_llm.py --no-llm docs/PRD-${{ github.event.number }}.md
    # 合规率低于 80% 则失败
```

## ⚠️ 注意事项

### 1. API 成本

- **qwen-plus**: 约 ¥0.02/次（万字文档）
- **qwen-max**: 约 ¥0.08/次（万字文档）
- 建议：日常用规则检查，重要文档用 LLM

### 2. 隐私保护

- 敏感文档建议使用本地规则检查
- 或使用私有化部署的模型
- 不要上传含有机密信息的文档

### 3. 网络依赖

- LLM 检查需要网络连接
- 无网络时自动降级为规则检查
- 可设置 `--no-llm` 强制本地检查

### 4. 结果解读

- LLM 建议仅供参考
- 重要决策仍需人工评审
- 结合领域知识判断建议合理性
