- 一键调用：`vibe-coding "需求"`
- 标准化流程：5 个阶段自动执行
- 质量保障：每个阶段自动检查
- 可复用：所有 OpenClaw 用户都能用

## 🎯 核心特性

### 1. LLM 分层策略

| Agent | 模型 | Thinking | 理由 |
|-------|------|----------|------|
| Analyst | qwen3.5-plus | medium | 需求分析需要理解能力 |
| Architect | qwen3.5-plus | high | 架构设计需要深度思考 |
| Developer | qwen3-coder-next | medium | 代码实现需要编码专长 |
| Tester | glm-4 | low | 测试用例相对简单 |

**效果**: 成本 -30%，性能 +33%

### 2. 质量门禁

每个阶段自动检查：

| 阶段 | 检查项 | 通过标准 |
|------|--------|---------|
| 需求分析 | 功能数量 | ≥3 个 |
| | 用户故事 | ≥2 个（GWT 格式） |
| 架构设计 | 技术选型 | 有理由说明 |
| | 模块设计 | ≥3 个模块 |
| 代码实现 | 代码注释 | JSDoc 风格 |
| | 错误处理 | 有输入验证 |
| 测试编写 | 功能测试 | ≥5 个用例 |
| | 边界测试 | ≥3 个用例 |

### 3. 标准化流程

```
用户输入 → Phase 1 需求分析 → Phase 2 架构设计 → 
Phase 3 代码实现 → Phase 4 测试编写 → Phase 5 整合验收
```

每个阶段：
- 调用对应的 Agent
- 等待完成
- 质量检查
- 保存输出
- 汇报进度

## 🎓 借鉴 Claude Code 的优点

### 1. 明确的职责边界

每个 Agent 都有：
- ✅ **必须做的** (Must Do)
- ❌ **禁止做的** (Must Not Do)

### 2. 详细的输出格式规范

```markdown
## 输出格式

你的输出将自动解析并保存到 `docs/requirements.md`，格式如下：

```markdown
# {项目名} - 需求文档
...
```
```

### 3. 示例驱动

每个 Agent 都有完整的输入输出示例。

### 4. 输出效率要求

```markdown
## 输出效率

- 直接输出代码，不要冗长的解释
- 代码块之间用空行分隔
- 每个文件一个代码块
- 如果代码简单，不要写过多注释
```

### 5. 安全协议

每个 Agent 都有安全/质量考虑。

## 💡 关键洞察

### 洞察 1: 真实 > 完美

一个能真实执行但界面简陋的工具，比一个界面精美但不能执行的工具更有价值。

**行动**: 优先保证真实执行，再优化可视化。

### 洞察 2: 提示词即代码

Agent 的输出质量直接取决于提示词的质量。

**行动**: 投入时间优化提示词模板，这是 ROI 最高的投资。

### 洞察 3: 分层即优化

不同任务需要不同模型，不是一味用最强的。

**行动**: 
- 战略决策 → qwen3.5-plus + high
- 代码实现 → qwen3-coder-next + medium
- 简单任务 → glm-4 + low

### 洞察 4: 技能化 = 可复用

将框架做成技能，让所有用户都能用。

**行动**: 发布到 ClawHub，最大化价值。
