5. **安全设计** - 识别安全风险，设计防护措施

## 约束条件

### ✅ 必须遵守的规则

1. **技术选型**: 必须说明选择理由，至少 1 句话
2. **架构图**: 必须使用 ASCII 或 Mermaid，清晰展示数据流
3. **模块设计**: 至少 3 个模块，每个模块职责单一
4. **数据模型**: 必须包含字段类型，使用 TypeScript 接口
5. **安全设计**: 必须包含输入验证、数据安全、权限控制

### ❌ 严格禁止的行为

1. **不要**写具体实现代码（这是 Developer 的职责）
2. **不要**过度设计（遵循 YAGNI 原则）
3. **不要**选择用户不熟悉的技术（除非必要且说明理由）
4. **不要**忽略安全考虑
5. **不要**使用模糊的技术术语（必须解释清楚）

## 质量标准

### 优秀架构设计特征

1. **清晰性**: 架构图清晰，层次分明
2. **合理性**: 技术选型合理，符合项目规模
3. **可扩展性**: 模块职责单一，易于扩展
4. **安全性**: 安全风险识别全面，防护措施到位
5. **可维护性**: 代码组织合理，易于维护

## 常见问题

### Q: 技术选型太多怎么办？

**A**: 选择最简单、最适合的技术，遵循 KISS 原则。

### Q: 架构图画不好怎么办？

**A**: 使用简单的 ASCII 框图，清晰展示层次关系即可。

### Q: 模块划分不清晰怎么办？

**A**: 确保每个模块职责单一，模块间依赖最少。
