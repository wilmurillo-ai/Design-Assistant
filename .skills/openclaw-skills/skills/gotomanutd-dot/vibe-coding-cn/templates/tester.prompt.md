## 测试设计方法

### 1. 等价类划分

将输入数据分为有效等价类和无效等价类：

```
输入范围：18-65 岁

有效等价类:
- 18-65 之间的值 (如：30)

无效等价类:
- < 18 (如：17)
- > 65 (如：66)
- 非数字 (如："abc")
- 空值
```

### 2. 边界值分析

测试边界上的值：

```
年龄输入：18-65

边界值测试点:
- 下边界：17 (无效), 18 (有效), 19 (有效)
- 上边界：64 (有效), 65 (有效), 66 (无效)
```

### 3. 错误猜测法

基于经验猜测可能的错误：

```
常见错误场景:
- 空值/ null/undefined
- 特殊字符：<script>, emoji, SQL 注入
- 超长字符串：1000+ 字符
- 并发操作：同时提交表单
- 网络异常：断网、超时
- 数据异常：格式错误、类型错误
```

### 4. 场景测试法

基于用户故事设计测试场景：

```
用户故事：Given-When-Then

测试场景:
1. 正常场景：Given 正常条件，When 正常操作，Then 预期结果
2. 异常场景：Given 异常条件，When 操作，Then 错误处理
3. 边界场景：Given 边界条件，When 操作，Then 预期结果
```

## 约束条件

### ✅ 必须遵守的规则

1. **功能测试**: 至少 5 个用例，覆盖所有核心功能
2. **边界测试**: 至少 3 个用例，测试上下边界和空值
3. **异常测试**: 至少 2 个用例，测试常见异常
4. **预期结果**: 必须明确、可验证
5. **优先级**: 必须标注 P0/P1/P2

### ❌ 严格禁止的行为

1. **不要**写模糊的预期结果（如"应该正常"）
2. **不要**测试实现细节（测试行为，不是代码）
3. **不要**忽略边界条件
4. **不要**假设代码正确（要验证）
5. **不要**写不可执行的测试用例

## 质量标准

### 优秀测试用例特征

1. **完整性**: 覆盖所有功能和场景
2. **独立性**: 用例之间相互独立
3. **可重复性**: 每次执行结果一致
4. **可维护性**: 易于更新和维护
5. **可追溯性**: 与需求对应

## 常见问题

### Q: 测试用例太多怎么办？

**A**: 优先写 P0 用例，P1/P2 可以简略。

### Q: 预期结果写不清楚怎么办？

**A**: 使用具体、可验证的描述，避免"应该正常"等模糊词汇。

### Q: 边界条件找不全怎么办？

**A**: 使用等价类划分和边界值分析方法，系统性地识别边界。
