# ERC-4626: Tokenized Vault Standard

EIP-4626 defines a standard interface for yield-bearing vaults. A vault accepts a deposit of an underlying asset (e.g. USDC) and mints "shares" representing proportional ownership of the vault's assets. As the vault generates yield, each share becomes redeemable for more underlying tokens.

**The standard solves:** Fragmented vault interfaces — before 4626, every protocol (Yearn, Aave, Compound) had its own deposit/withdraw API, making aggregators and integrations painful to build.

## Key Functions

```solidity
interface IERC4626 is IERC20 {
    // --- Asset/Share Info ---

    /// @return The address of the underlying ERC-20 asset (e.g. USDC)
    function asset() external view returns (address);

    /// @return Total underlying assets managed by the vault (principal + yield)
    function totalAssets() external view returns (uint256);

    // --- Deposit (specify assets in, receive shares out) ---

    /// @dev Preview: how many shares would I get for depositing `assets`?
    function previewDeposit(uint256 assets) external view returns (uint256 shares);

    /// @dev Deposit `assets` tokens from `msg.sender`, mint shares to `receiver`
    function deposit(uint256 assets, address receiver) external returns (uint256 shares);

    // --- Mint (specify shares out, pay assets in) ---

    /// @dev Preview: how many assets do I need to pay to receive exactly `shares`?
    function previewMint(uint256 shares) external view returns (uint256 assets);

    /// @dev Mint exactly `shares` to `receiver` by depositing required assets from `msg.sender`
    function mint(uint256 shares, address receiver) external returns (uint256 assets);

    // --- Withdraw (specify assets out, burn shares) ---

    /// @dev Preview: how many shares do I need to burn to withdraw exactly `assets`?
    function previewWithdraw(uint256 assets) external view returns (uint256 shares);

    /// @dev Burn shares from `owner`, send exactly `assets` to `receiver`
    function withdraw(uint256 assets, address receiver, address owner) external returns (uint256 shares);

    // --- Redeem (specify shares in, receive assets out) ---

    /// @dev Preview: how many assets do I get for redeeming `shares`?
    function previewRedeem(uint256 shares) external view returns (uint256 assets);

    /// @dev Burn exactly `shares` from `owner`, send resulting assets to `receiver`
    function redeem(uint256 shares, address receiver, address owner) external returns (uint256 assets);

    // --- Conversion helpers ---

    /// @dev How many shares does `assets` amount convert to (at current exchange rate)?
    function convertToShares(uint256 assets) external view returns (uint256);

    /// @dev How many assets does `shares` amount convert to (at current exchange rate)?
    function convertToAssets(uint256 shares) external view returns (uint256);

    // --- Max limits ---
    function maxDeposit(address receiver) external view returns (uint256);
    function maxMint(address receiver) external view returns (uint256);
    function maxWithdraw(address owner) external view returns (uint256);
    function maxRedeem(address owner) external view returns (uint256);
}
```

## Exchange Rate Formula

The share price increases as the vault accumulates yield:

```
sharePrice = totalAssets / totalSupply
assets = shares * sharePrice = shares * totalAssets / totalSupply
```

## OpenZeppelin Implementation

The easiest way to create an ERC-4626 vault with OpenZeppelin 5.x:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "@openzeppelin/contracts/token/ERC20/extensions/ERC4626.sol";
import "@openzeppelin/contracts/access/Ownable.sol";

contract SimpleYieldVault is ERC4626, Ownable {
    constructor(IERC20 _asset, string memory _name, string memory _symbol)
        ERC4626(_asset)
        ERC20(_name, _symbol)
        Ownable(msg.sender)
    {}

    // Override to add yield — in real usage this might read from a lending protocol
    function totalAssets() public view override returns (uint256) {
        // OZ default: just returns the vault's balance of `asset()`
        // Override to include yield sources (lending pool balances, farming rewards, etc.)
        return super.totalAssets();
    }
}
```

**Usage:**
```solidity
// Deposit 1000 USDC, get shares back
uint256 shares = vault.deposit(1000e6, msg.sender);

// Later: redeem all shares for USDC + yield
uint256 usdc = vault.redeem(shares, msg.sender, msg.sender);
```

## First Depositor Attack (and the Fix)

**The attack:** If the vault is empty (totalSupply = 0), the first depositor can front-run later depositors:
1. Alice deposits 1 wei — gets 1 share
2. Alice donates 1000 USDC directly to the vault (not through deposit)
3. Bob deposits 1000 USDC — rounds down to 0 shares (because 1000e6 * 1 / (1000e6 + 1) = 0)
4. Bob loses his deposit to Alice when she redeems her 1 share

**The fix — virtual shares/assets (OZ default since 5.x):**

```solidity
// Inside ERC4626 in OZ 5.x:
function _convertToShares(uint256 assets, Math.Rounding rounding)
    internal view virtual returns (uint256)
{
    // Offset adds virtual shares/assets: makes rounding attack economically impractical
    return assets.mulDiv(
        totalSupply() + 10 ** _decimalsOffset(),
        totalAssets() + 1,
        rounding
    );
}
```

The `_decimalsOffset()` (default 0) adds a virtual share, making the attack require depositing `10^offset * totalAssets` to grief a single wei — economically infeasible.

**To increase protection:** Override `_decimalsOffset()` to return a higher value (e.g. 3).

## Integrating with a Vault (as a User/Integrator)

```typescript
// Viem example: deposit USDC into a vault
import { parseUnits, formatUnits } from 'viem';
import { erc4626Abi } from 'viem'; // Viem includes ERC-4626 ABI

// Check current exchange rate before depositing
const assetsPerShare = await client.readContract({
  address: vaultAddress,
  abi: erc4626Abi,
  functionName: 'convertToAssets',
  args: [parseUnits('1', 18)], // 1 share
});

// Preview: how many shares will I get?
const expectedShares = await client.readContract({
  address: vaultAddress,
  abi: erc4626Abi,
  functionName: 'previewDeposit',
  args: [parseUnits('1000', 6)], // 1000 USDC
});

// Step 1: Approve vault to spend USDC
await walletClient.writeContract({
  address: usdcAddress,
  abi: erc20Abi,
  functionName: 'approve',
  args: [vaultAddress, parseUnits('1000', 6)],
});

// Step 2: Deposit
const shares = await walletClient.writeContract({
  address: vaultAddress,
  abi: erc4626Abi,
  functionName: 'deposit',
  args: [parseUnits('1000', 6), userAddress],
});
```

## Tracking Yield

```typescript
// Track user's current value vs. initial deposit
async function getVaultPosition(vaultAddress, userAddress) {
  const shares = await client.readContract({
    address: vaultAddress, abi: erc4626Abi, functionName: 'balanceOf', args: [userAddress],
  });
  const currentAssets = await client.readContract({
    address: vaultAddress, abi: erc4626Abi, functionName: 'convertToAssets', args: [shares],
  });
  return { shares: formatUnits(shares, 18), currentAssets: formatUnits(currentAssets, 6) };
}
```

## Common Gotchas

1. **totalAssets includes unrealized yield** — the vault might report higher assets than it can actually liquidate (e.g. locked staking positions). Check `maxWithdraw`.

2. **Rounding direction matters** — OZ rounds in favor of the vault (not the user) on most operations to prevent economic exploits. Don't expect `deposit → immediate redeem` to return the exact same amount.

3. **Use `previewDeposit` not `convertToShares`** — `convertToShares` is gas-efficient but ignores fees. `previewDeposit` returns the exact shares including fees.

4. **Slippage on `deposit`** — set a minimum shares received off-chain and revert if not met (the standard doesn't include slippage protection natively).

5. **Events** — ERC-4626 emits `Deposit` and `Withdraw` events in addition to standard ERC-20 `Transfer`. Index both for complete accounting.
