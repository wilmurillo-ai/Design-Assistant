# CONTENT STYLE

Writing style standards for all content outputs. Templates define STRUCTURE; this guide defines STYLE.

---

## WRITING PRINCIPLES

### Core Philosophy

| Principle | Description |
|-----------|-------------|
| **Deliver value** | Content serves audience needs, not optimization metrics |
| **Respect the reader** | Assume intelligence, explain complexity, never condescend |
| **Earn trust** | Every claim verifiable; uncertainty acknowledged |
| **Balance perspective** | Present opportunities AND risks; pros AND cons |
| **Clarity over cleverness** | Simple, direct language beats jargon |
| **Inform, don't sell** | Education and insight, not promotion or hype |

### Content Values

Prioritize:
- Audience value over optimization metrics
- Accuracy over speed
- Depth over breadth
- Insight over information
- Usefulness over engagement bait

## VOICE & TONE

### Core Voice Attributes

| Attribute | What it means | In practice |
|-----------|---------------|-------------|
| **Educational** | Teaching mindset | Explain WHY, not just WHAT |
| **Inclusive** | Reader is partner | Address reader directly |
| **Balanced** | Fair assessment | Cover both sides |
| **Factual** | Evidence-based | Lead with data, follow with analysis |
| **Accessible** | No gatekeeping | Define terms, use analogies |
| **Confident** | Clear perspective | Take positions, avoid hedging everything |

### Tone Spectrum

Adjust tone based on content purpose:

```
<-- MORE FORMAL ---------------------------------------- MORE CONVERSATIONAL -->

Research        Educational       News/Updates       Social
Analysis        Guides            Commentary         Threads
```

| Content Type | Tone | Characteristics |
|--------------|------|-----------------|
| **Research Analysis** | Professional, analytical | Data-driven, measured, objective |
| **Educational** | Friendly, patient | Thorough explanations, examples |
| **News/Updates** | Neutral, informative | Factual, timely, concise |
| **Opinion/Commentary** | Thoughtful, engaging | Clear perspective, well-reasoned |
| **Social Content** | Conversational, punchy | Direct, energetic, accessible |

## RESEARCH-TO-INSIGHT TRANSFORMATION

Content creation is not information transfer — it's insight delivery. Transform research into audience value:

| Stage | Action | Output |
|-------|--------|--------|
| **Extract** | Identify key facts from research | Data points, statistics, events |
| **Synthesize** | Connect facts to form patterns | Trends, relationships, context |
| **Interpret** | Determine what patterns mean | Analysis, implications |
| **Deliver** | Present insights audiences need | Actionable understanding, perspective |

**Key questions while writing:**
- **"So what?"** — Why should audience care about this fact?
- **"What's the implication?"** — How does this affect the audience?
- **"What's different?"** — What's novel, surprising, or counterintuitive?

Audiences can find information anywhere. They come for insights, context, and understanding.

## PLATFORM-SPECIFIC GUIDANCE

Each platform requires adapting the research-to-insight process:

| Platform | Core Constraint | Transformation Focus |
|----------|-----------------|---------------------|
| **Blog** | Attention span (~8 min read) | Deep insight with scannable structure |
| **X Thread** | 280 chars/tweet | Ultra-condensed, each tweet standalone valuable |
| **TikTok** | 3 seconds to hook | Visual-first, one insight per video |
| **YouTube** | 30 sec retention cliff | Progressive value, chapter-based depth |

**Blog Articles:**
- Lead with insight, support with evidence
- Scannable: Headers convey key points, bold key phrases sparingly
- Each section delivers one takeaway
- Depth without drowning — comprehensive yet digestible

**X Threads:**
- Each tweet must answer "So what?" — insight, not fact dump
- Prioritize clarity over completeness
- First tweet = hook, last tweet = takeaway + CTA

**TikTok Scripts:**
- First 3 seconds = entire video's fate
- Visualize what data MEANS, not just display data
- One clear insight per video
- Pacing: Fast but not rushed

**YouTube Scripts:**
- Build argument progressively, each chapter = one insight
- Front-load value — no slow intros
- Use visuals/analogies for abstract concepts
- Long-form advantage: explore nuance and context

## OPENING & HOOKS

### Hook Types

| Type | When to use | Example |
|------|-------------|---------|
| **Problem-driven** | Addressing pain point | "A $50 gas fee for a simple swap. This is the reality Ethereum users face..." |
| **Question-driven** | Sparking curiosity | "What makes a token pump 10,000% in a week — then crash 90%?" |
| **Data-driven** | Compelling statistics | "Over $200 billion lost to DeFi hacks since 2020..." |
| **Context-driven** | Timely/trending | "After the FTX collapse, self-custody became urgent..." |
| **Contrarian** | Challenging beliefs | "Contrary to popular belief, not every L2 solves scalability..." |
| **Story-driven** | Narrative opening | "In May 2022, a $60 billion ecosystem collapsed in 72 hours..." |

### Hook Structure

Every hook should contain 3 elements:

1. **Attention** (1 sentence): Grab interest immediately
2. **Context** (1-2 sentences): Why this matters now
3. **Promise** (1 sentence): What reader will gain

### Closing & CTA

| Pattern | Structure | Best for |
|---------|-----------|----------|
| **Summary** | Recap key points → Final insight | Educational content |
| **Forward-looking** | Current state → What's next | News, trends |
| **Action-oriented** | Key takeaway → What to do | Guides, analysis |
| **Reflection** | Bigger picture → Why it matters | Opinion, deep dives |

CTA should deliver value, not just ask for engagement.

## READABILITY & STRUCTURE

**For Skimmers:**
- H2/H3 headers descriptive enough to convey main point
- Key insights bolded within paragraphs (sparingly)
- Bullet lists for 3+ related points
- Tables for comparisons

**For Deep Readers:**
- Paragraphs: 2-4 sentences ideal
- Sentence variety: Mix short punchy + longer explanatory
- Logical flow: Each section builds on previous

**For All Readers:**
- Introduction promises what they'll learn
- Each section delivers one key takeaway
- Conclusion reinforces what they learned

### Narrative vs Structured Elements

Default to narrative paragraphs. Use structured elements when they genuinely aid comprehension.

| Element | When to Use | When NOT to Use |
|---------|-------------|-----------------|
| **Paragraphs** | Analysis, explanation, argumentation | Never avoid |
| **Tables** | 4+ items with 3+ comparable attributes | Simple comparisons |
| **Bullets** | 3+ truly parallel items | Analysis, insights |
| **Numbered lists** | Sequential steps only | Rankings, arguments |

## LANGUAGE & TERMINOLOGY

See **core-config.md** for language settings and terminology rules.

### First-Use Pattern

When using domain-specific terms for the first time, provide brief explanation:

```
"TVL (Total Value Locked) has surpassed $10B..."
"The protocol's TVL — the total value of assets deposited — reached $10B..."
```

### Domain Term Handling

- **Preserve** crypto-native terms: staking, oracle, liquidity pool, smart contract, gas fee, DeFi, NFT
- **Preserve** proper nouns: Ethereum, Uniswap, Polymarket
- **Translate** common words to output language when natural equivalent exists

## VISUAL & MEDIA

- **Images:** Must support content (not decoration). Always include caption with source.
- **Tables:** Max 4-5 columns. Consistent formatting. Include units. Sort meaningfully.
- **Lists:** Parallel structure. 3-7 items ideal.
- **Citations:** Integrate sources naturally into text, not academic footnotes.
- **Source credibility:** Official docs / on-chain data (high) > Research reports (high) > Team announcements (medium) > Social media (low, sentiment only).
- **Data currency:** Always note time frame. Link to live sources. Flag volatile metrics.

## ANTI-PATTERNS

**1. Hype Language**
- Avoid: superlatives, "gem", "can't miss", price predictions
- Instead: specific factors + specific risks, let reader decide

**2. False Balance**
- Avoid: "some say X, some say Y" without analysis
- Instead: evaluate evidence for each position, state which is stronger and why

**3. Jargon Overload**
- Avoid: stacking technical terms without explanation
- Instead: introduce terms with brief parenthetical definitions

**4. Unsupported Claims**
- Avoid: "X is the best blockchain" without evidence
- Instead: cite specific metrics, data sources, and timeframes

**5. Passive Voice Overuse**
- Avoid: "Tokens are staked by users so rewards are received"
- Instead: "Users stake tokens to earn rewards"

**6. Burying the Lead**
- Avoid: lengthy historical context before the main point
- Instead: lead with the insight, then provide supporting context
