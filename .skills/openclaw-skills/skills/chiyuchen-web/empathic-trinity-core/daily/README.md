# daily 目录说明

## 一、定位

`daily/` 用于保存按天组织的每日摘要文件。

它不是聊天流水账目录，也不是长期主记忆目录。

它的职责是：

- 保存当天关键事件摘要
- 记录重要决定与待办变化
- 记录值得跨时间回看的近期变化
- 为按需检索提供补充信息
- 为长期主记忆提供候选晋升线索

一句话：**daily 负责“近期变化摘要”，不是“逐条聊天转录”。**

---

## 二、文件命名规范

每日文件统一使用：

- `YYYY-MM-DD.md`

例如：

- `2026-03-07.md`

---

## 三、推荐结构

每个 daily 文件建议至少包含以下部分：

```md
# 2026-03-07

- Context:
- Key events:
- Decisions made:
- Preference changes:
- Open loops / todos:
- Worth promoting to `MEMORY.md`:
```

如果需要，也可以扩展为更自然的中文小节，例如：

```md
# 2026-03-07

## 今日上下文

## 关键事件

## 做出的决定

## 偏好或认知变化

## 未闭环事项

## 候选长期记忆
```

---

## 四、写作原则

1. 写摘要，不写逐条聊天流水。
2. 优先记录变化、决定、纠错、推进与待办更新。
3. 保留对未来搜索真正有价值的内容。
4. 尽量高信息密度，避免重复描述。
5. 没有显著变化时，可以少写，甚至不写。

优先记录：

- 重要变化
- 重要纠错
- 重要经验
- 明确决定
- 候选长期记忆
- 待办状态变化

不建议记录：

- 无意义寒暄
- 已经在当前会话里非常明确且不会复用的碎片
- 低价值重复细节
- 大段逐字聊天转录

---

## 五、更新频率

`daily` 默认采用低频摘要式更新。

推荐在以下时机更新：

- 一个阶段结束时
- 当前主题告一段落时
- 会话接近 compaction 时
- 出现显著偏差、冲突或纠错时
- 明确出现值得长期保留的线索时

不建议：

- 每条消息后写
- 每轮无条件写
- 纯流水式追踪

---

## 六、读取策略

daily 不应在每轮固定全量读取。

推荐读取方式：

1. 先读今天的 daily
2. 仅当今天缺失或信息不足时，再读昨天
3. 只有在确实需要追踪变化轨迹时，才继续扩大时间范围

核心原则：**small first，need-based expansion。**
