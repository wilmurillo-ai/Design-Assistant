# Memory V8

Memory V8 是一套建立在 OpenClaw 之上的低上下文占用记忆运行架构。

它不是模型，也不是新的 provider，而是对 OpenClaw 原生会话、工作区文件记忆、长期主记忆与检索机制的一次重新组织。

## 一句话定义

**以会话连续性为短期工作内存，以摘要式 daily 作为近期变化层，以 `MEMORY.md` 作为长期主记忆，以按需触发检索作为补充召回层的一套低占用记忆运行协议。**

## 三个文件

本目录继续保持简洁结构，仅保留三个核心文件：

1. `README.md`
   - 对外总览
   - 说明这套系统是什么、解决什么问题、三个文件分别负责什么

2. `ARCHITECTURE.md`
   - 完整运行架构
   - 说明记忆分层、按需加载、冲突处理、检索触发、晋升机制与上下文占用控制
   - 供接入者与其它 agent 理解运行方式

3. `daily/README.md`
   - 说明 daily 的定位、结构、写法、更新频率
   - 明确 daily 是摘要式日志，而不是流水账

## 当前架构结论

- 当前 session 只承担短期工作内存，不承担永久长期记忆职责
- `MEMORY.md` 只保存长期稳定且高价值的信息，不保存流水式历史
- `daily` 只保存高信息密度摘要，用于记录关键事件、决定、变化与待办更新
- daily 默认先读今天；只有今天缺失或信息不足时再读昨天
- 检索采用按需触发，不再每轮固定预载全部记忆
- 冲突处理优先尊重当前现实、时间锚点与任务相关性

## 与 OpenClaw 的关系

Memory V8 复用 OpenClaw 原生能力：

- 当前会话上下文
- workspace 文件记忆
- `memory_search` / `memory_get`
- memory 索引与持久化
- bootstrap 文件注入

当前接入状态：

- 长期主记忆采用 `MEMORY.md`
- recent memory 采用 `memory/YYYY-MM-DD.md` 摘要式 daily
- memory 索引存放在 `.openclaw/memory/{agentId}.sqlite`
- 记忆加载采用按需触发策略

## 为什么这样设计

目标不是“尽量多记”，而是“在尽量少的上下文占用下，保留真正有用的记忆能力”。

因此，Memory V8 重点解决四件事：

- 什么应该常驻
- 什么应该只在需要时加载
- 什么应该沉淀成长期认知
- 什么应该停留在近期摘要层

## 给其它 agent 的最短使用方法

先理解三层：

1. 当前会话 = 短期工作内存
2. `daily` = 近期变化摘要层
3. `MEMORY.md` = 长期主记忆层

然后按以下原则工作：

- 优先利用当前会话上下文
- 只在涉及 prior work、decisions、dates、people、preferences、todos 时加载或检索记忆
- 默认先看今天的 daily；不足时再看昨天
- 不要把所有历史记录一股脑塞进上下文
- 不要把短期变化直接写进主记忆
- 冲突时根据现实、时间锚点与任务相关性判断

## 一句话总结

**Memory V8 = 会话优先 + 摘要式 daily + 精简 `MEMORY.md` + 按需触发检索 + 现实优先的冲突判断。**
