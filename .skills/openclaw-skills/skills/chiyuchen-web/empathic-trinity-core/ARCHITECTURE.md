# Memory V8 Architecture

## 1. 定位

Memory V8 不是模型，也不是新的 provider，更不是“把所有历史都塞进上下文”的方案。

它是一套运行在 OpenClaw 之上的低上下文占用记忆架构，用来回答以下问题：

- 什么应该常驻
- 什么应该按需加载
- 什么应该沉淀成长期记忆
- 什么应该停留在近期摘要层
- 当不同记忆冲突时应该信谁
- 如何在保持记忆能力的同时压低上下文成本

## 2. 整体分层

### 2.1 会话层

会话层是当前 session 的即时上下文。

负责：

- 当前任务承接
- 最近偏好与约定
- 语气与目标延续
- 当轮短期工作记忆

特点：

- 最贴近当下
- 低摩擦
- 但会被 compaction、裁剪、`/new`、`/reset` 或新 session 打断

因此，会话层应被视为**短期工作内存**，而不是永久记忆。

### 2.2 近期摘要层

近期摘要层由 `memory/YYYY-MM-DD.md` 构成。

它负责保存：

- 当天关键事件
- 重要决定
- 明显变化
- 待办更新
- 值得之后检索的上下文摘要

它不负责：

- 逐条转录聊天
- 保存所有原始细节
- 充当长期主记忆

一句话：**daily 保存“最近发生了哪些值得回看的变化”。**

### 2.3 主记忆层

主记忆层统一使用 `MEMORY.md`。

主记忆内部采用三合一结构：

- **事实**：长期稳定的背景、偏好、约束与确认事实
- **思考**：长期有效的方法、判断、原则与认知模式
- **情感**：关系温度、在意的事、协作氛围与持续性的情绪线索

一句话：**主记忆保存“长期稳定且值得持续依赖的认知”。**

### 2.4 检索层

检索层依赖 OpenClaw 原生能力：

- `memory_search`
- `memory_get`
- memory 索引

它的作用不是替代主记忆，而是在需要时，从 recent daily、外部化记忆与长期主记忆相关片段中召回补充信息。

## 3. 启动与加载策略

推荐策略：

1. 先利用当前会话已有上下文
2. 常驻只保留最小启动集，例如 `SOUL.md`、`USER.md`
3. 只有在任务涉及 prior work、decisions、dates、people、preferences、todos 时，才加载 `MEMORY.md` 或触发检索
4. recent daily 默认先读今天
5. 仅当今天缺失或信息不足时，再读昨天
6. 只有在背景不足、存在时间变化、需要比对历史或判断冲突时，才扩大检索范围

核心原则：**先让会话记忆工作，再让文件记忆补位；先小范围补位，再大范围检索。**

## 4. 检索触发阈值

额外检索采用按需触发，而不是每轮固定流程。

应触发检索的情况：

- 当前对话缺乏足够事件背景
- 问题明显涉及 prior work、decisions、dates、people、preferences、todos
- 需要确认最近一次怎么说、后来是否变化
- 同一事项在不同时间段可能冲突
- 主记忆无法提供充分支持

不必强行检索的情况：

- 当轮上下文已经充分
- 当前会话刚刚明确过
- 问题不依赖历史变化
- 只需要一般知识回答

## 5. 冲突处理

当会话、daily 与主记忆冲突时，优先级应为：

1. 当前会话中的明确信息
2. 更新更近、时间锚点更清晰的 daily 或外部化记录
3. 稳定但可能滞后的主记忆
4. 如果仍不确定，再向用户确认

判断时要看：

- 哪条信息更新更晚
- 哪条信息更贴近当前任务
- 哪条信息具有更明确的时间背景
- 哪条像长期规律，哪条像近期变化

原则：**尊重现实，不机械套规则。**

## 6. 晋升机制

当近期摘要中的信息满足以下条件时，应考虑晋升到 `MEMORY.md`：

- 不再只是一次性事件，而是表现出持续性
- 对后续协作反复有帮助
- 能稳定描述偏好、工作方式、关系背景或长期约束
- 被多次记录验证
- 脱离原始事件背景后仍然有长期参考价值

通常不应晋升的内容：

- 一次性临时状态
- 只对当天有效的安排
- 没有上下文就容易误导的碎片信息
- 明显会快速过期的事实
- 低价值重复细节

一句话：**近期摘要记录变化，主记忆沉淀模式。**

## 7. 上下文占用控制

Memory V8 的优化目标之一，是把记忆能力与上下文成本解耦。

推荐做法：

- `SOUL.md` 保持短，只放人格与边界
- `USER.md` 只放稳定信息，不放临时事项
- `MEMORY.md` 只留高价值长期信息，不留流水账
- daily 采用摘要式写法，每日只保留关键变化
- 记忆加载尽量按需触发，而不是默认全量预载

一个常见的低占用区间大致是：

- 平常不触发记忆：约 220–420 tokens
- 触发主记忆：约 400–770 tokens
- 触发主记忆 + 单份 daily：约 480–970 tokens
- 最重情况主记忆 + 今天/昨天 daily：约 560–1170 tokens

具体数字会随文件长度变化，但远低于“每轮固定读主记忆 + 两天 daily”的做法。

## 8. 与 OpenClaw 的映射

当前 V8 通过 OpenClaw 原生能力运行：

- 会话层：当前 session 上下文
- 近期摘要层：`memory/YYYY-MM-DD.md`
- 主记忆层：`MEMORY.md`
- 检索层：`memory_search` / `memory_get`
- 索引层：`.openclaw/memory/{agentId}.sqlite`

## 9. 给其它 agent 的操作规范

使用这套架构时：

- 不要把所有历史记录全部读进当前上下文
- 不要把会话上下文误当成永久记忆
- 不要把 daily 写成流水账
- 不要把短期变化直接写进主记忆
- 优先使用当前上下文，其次按需看 recent daily，再按需参考主记忆与检索结果
- 当主记忆与近期记录冲突时，以现实变化和时间锚点为准

## 10. 一句话总结

**Memory V8 = 会话连续性优先 + 摘要式 daily + 精简 `MEMORY.md` + 按需触发检索 + 现实优先的冲突判断。**
