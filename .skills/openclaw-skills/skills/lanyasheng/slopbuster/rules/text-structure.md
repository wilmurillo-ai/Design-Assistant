# Text Rules: Structural Patterns

How to detect and fix AI-generated document structure — the patterns that make an entire piece feel assembled rather than written.

---

## Structural Anti-Patterns

| Pattern | What it looks like | Fix |
|---------|-------------------|-----|
| Meandering intro | 3+ paragraphs of context before stakes appear | Start with friction, surprise, or specific problem |
| Saggy conclusion | Final paragraph recaps everything you just said | End by extending, reframing, or posing a question |
| Theory before practice | Abstract principle → then maybe an example | Lead with the concrete, derive the abstract |
| Process without emotion | Step 1, Step 2, Step 3 with no reaction | Add what you felt, not just what you did |
| Context before tension | "In today's world..." opening | Hook first, context second |
| Perfectly balanced sections | Every section same length, same structure | Vary section weight by importance |
| Every paragraph = claim → evidence → implication | Identical internal paragraph structure throughout | Mix structures: some paragraphs are single observations, some are arguments, some are digressions |
| Symmetric pro/con | Equal-weight "On one hand... on the other hand..." | Unless the contrast is genuinely balanced, commit to a position |

---

## Restructuring Patterns That Work

### The Friction → Zoom Out → Framework
Start with a specific, personal friction moment ("I couldn't get this to work"). Zoom out to a broader cultural or historical frame ("this is the same problem that..."). Land on a usable framework or takeaway.

**Good for:** Technical posts, opinion pieces, tutorials.

### The Adventure Narrative
Chronological account of trying something new. Structured around escalating discovery and unexpected consequences. Show the messy middle — missteps, screenshots, wrong turns.

**Good for:** Case studies, product reviews, "how we built X" posts.

### The Borrowed Lens
Pull a concept from an unexpected discipline (cooking, architecture, music theory, sports) and use it as the interpretive key for a technical or business topic.

**Good for:** Thought leadership, conference talks, creative technical writing.

## Sentence-Level Structural Rules

| Pattern | Action |
|---------|--------|
| 3+ consecutive sentences same length | Vary — short, long, medium |
| 3+ consecutive sentences same opening structure | Restructure — different subjects, different clause order |
| Every sentence follows Subject-Verb-Object with no variation | Break it — inversion, fragment, question |
| Rhetorical question immediately followed by its answer | Cut the question; state the answer directly |
| Correlative constructions ("not X, but Y") | Rewrite: state Y directly |
| Generic transitions ("Moving on," "That said," "With that in mind") | Cut, or make specific to the actual transition |
| "Boilerplate authority" ("studies show," "research suggests") | Name the study or delete the claim |

## The Paragraph Test

Read each paragraph and ask:
1. Could I swap this paragraph into a different article on a different topic and it still works? → Too generic. Add specifics.
2. Does this paragraph resolve too neatly? → Let some tension carry to the next paragraph.
3. Does every paragraph start with its broadest claim? → Sometimes start specific and derive the general.
4. Is every paragraph the same length? → Vary it. Some paragraphs should be one sentence.

---

## The Opening Test

Your first paragraph should do ONE of these:
- State a specific problem the reader has felt
- Present a surprising fact or counterintuitive observation
- Describe a moment of friction, failure, or discovery
- Ask a genuine (not rhetorical) question

Your first paragraph should NEVER:
- Define a commonly known term
- Provide historical background without stakes
- Start with "In today's..." anything
- State something nobody would disagree with
- Use more than 3 sentences

---

## The Ending Test

Your last paragraph should:
- Extend the argument somewhere new
- Leave the reader with a specific, actionable next step
- Pose an honest, unresolved question
- Reframe what came before from a new angle

Your last paragraph should NEVER:
- Summarize what you just said
- Use "In conclusion" or "To summarize"
- Predict a bright future
- End with a call to action that could apply to any article
