# 质量检查清单

README 审计的完整 22 项检查与评分细则。

## 评分总览

总分 100 分，6 个类别。每项检查为 pass/fail，pass 得满分，fail 得 0。

---

## Hook（钩子）— 25 分

让读者在 5 秒内决定是否继续。

| # | 检查项 | 分值 | 判定标准 |
|---|--------|------|----------|
| H1 | 项目名清晰 | 5 | 标题与 repo 名一致，不含冗余前缀（如 `awesome-`、`my-`） |
| H2 | 一行描述 | 8 | < 120 字符，描述「做什么 + 给谁用」，和 package.json/Cargo.toml 描述一致 |
| H3 | Hero Visual | 7 | 存在 GIF/截图/架构图/终端录屏之一，且在前 50 行内 |
| H4 | 徽章合理 | 5 | 有 CI + License 徽章，总数 ≤ 8 个，无 failing badge |

---

## Onboarding（上手）— 25 分

让读者在 2 分钟内从零跑通。

| # | 检查项 | 分值 | 判定标准 |
|---|--------|------|----------|
| O1 | Quick Start ≤ 3 步 | 8 | 从安装到看到输出不超过 3 步，每步有可复制的代码块 |
| O2 | 安装命令完整 | 7 | 包含所有前置依赖、包管理器命令、平台差异说明 |
| O3 | Usage 有代码块 | 5 | 至少一个语法高亮的代码示例，展示核心功能 |
| O4 | 代码块有语法高亮 | 5 | 所有代码块标注了语言（```js, ```python 等） |

---

## Content（内容）— 20 分

让读者判断是否满足需求。

| # | 检查项 | 分值 | 判定标准 |
|---|--------|------|----------|
| C1 | Features 结构化 | 5 | 功能用列表或表格展示，不是散文段落 |
| C2 | Configuration 文档化 | 5 | 所有配置项（env vars / CLI flags / 配置文件）有说明 |
| C3 | API 概览或链接 | 5 | 公共 API 有概览表或链接到完整文档 |
| C4 | 信息时效性 | 5 | 无描述已删除功能、无过时截图、版本号与实际一致 |

---

## Trust（信任）— 15 分

让读者放心使用和贡献。

| # | 检查项 | 分值 | 判定标准 |
|---|--------|------|----------|
| T1 | License 存在 | 5 | LICENSE 文件存在 + README 底部有 SPDX 声明 |
| T2 | Contributing 指南 | 4 | 有 Contributing 节或链到 CONTRIBUTING.md |
| T3 | CI 徽章状态 | 3 | CI badge 存在且为 passing（允许无 CI 的小项目跳过） |
| T4 | 维护者信息 | 3 | 至少有一个联系方式或维护者列表 |

---

## Structure（结构）— 10 分

符合认知漏斗，信息不冗余。

| # | 检查项 | 分值 | 判定标准 |
|---|--------|------|----------|
| S1 | TOC | 3 | 超过 100 行有 Table of Contents，且链接有效 |
| S2 | 认知漏斗顺序 | 4 | 节顺序：Hook → Onboarding → Content → Trust → Meta（不倒序） |
| S3 | 无信息重复 | 3 | 同一信息不在多个节中重复出现 |

---

## Polish（打磨）— 5 分

细节决定专业度。

| # | 检查项 | 分值 | 判定标准 |
|---|--------|------|----------|
| P1 | 无死链 | 2 | 所有 `[text](url)` 和 `![](url)` 指向有效目标 |
| P2 | Markdown 格式正确 | 2 | 无格式错误（未关闭的标签、错误的缩进、缺少空行） |
| P3 | 移动端可读 | 1 | 表格不超过 5 列，代码块不超过 80 字符宽 |

---

## 审计执行流程

```
1. 读取 README.md 全文
2. 读取项目元数据（package.json / Cargo.toml / pyproject.toml）
3. 逐项检查 H1-P3
4. 每项记录 pass/fail + 扣分原因
5. 计算各类别小计和总分
6. 映射到等级（S/A/B/C/D）
7. 识别匹配的反模式（参见 SKILL.md 15 个反模式表）
8. 按影响力排序 Top 3 改进建议（优先修高权重项）
9. 输出审计报告
```

## 改进建议优先级

当多项检查 fail 时，按此顺序建议修复：

1. **H2 一行描述**（8 分，最高单项）— 投入最小收益最大
2. **O1 Quick Start**（8 分）— 直接影响新用户留存
3. **H3 Hero Visual**（7 分）— 视觉冲击力
4. **O2 安装命令**（7 分）— 上手障碍
5. **其余项**按分值降序
