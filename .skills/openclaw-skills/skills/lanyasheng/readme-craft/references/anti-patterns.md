# 15 个 README 反模式

| # | 反模式 | 问题 | 修复 |
|---|--------|------|------|
| 1 | **空仓库** | 没有 README | 写一个，哪怕只有 5 行 |
| 2 | **纯文字墙** | 没有标题、代码块、视觉断点 | 用 Markdown 结构化 |
| 3 | **垃圾场** | 用户文档+贡献者文档+API+CHANGELOG 全塞一个文件 | 拆分到 docs/、CONTRIBUTING.md、CHANGELOG.md |
| 4 | **死链** | 链接指向不存在的页面 | 用相对链接，定期检查 |
| 5 | **不能跑的代码** | 示例代码过时或有语法错误 | 像项目代码一样 lint 示例 |
| 6 | **缺许可证** | 法律风险让人不敢用 | 加 LICENSE 文件 + README 底部声明 |
| 7 | **模糊描述** | 「A utility for managing things」 | 说清楚解决什么问题、给谁用 |
| 8 | **省略安装** | 认为 `npm install` 人人知道 | 写完整命令，包括前置依赖 |
| 9 | **只说不做** | 没有代码示例，全是文字描述 | 每个功能至少一个代码块 |
| 10 | **时光胶囊** | README 描述的功能半年前就删了 | 改代码时同步改 README |
| 11 | **无声消失** | 项目停止维护但没有通知 | 顶部加 archived/unmaintained 徽章 |
| 12 | **专家假设** | 跳过「显而易见」的步骤 | 假设读者聪明但零上下文 |
| 13 | **营销文案** | README 在卖东西而不是说明东西 | 客观描述，让人自己判断 |
| 14 | **README 当 API 文档** | 完整 API 参考放在 README 里 | README 放概览，详细文档链出去 |
| 15 | **图片传话** | 关键信息只在截图里，无法搜索/无障碍 | 图片配文字说明 |

## 检测方法

审计时逐项匹配：

- **#7 模糊描述**：一行描述是否包含「utility」「tool」「things」等泛化词？是否说明了具体解决什么问题？
- **#8 省略安装**：安装节是否只有一行 `npm install`？有无前置依赖说明？
- **#9 只说不做**：Features 节是否全是散文？有无代码块？
- **#3 垃圾场**：README 是否超过 500 行且没有链接到外部文档？
- **#10 时光胶囊**：README 中提到的功能在源码中是否还存在？

每检测到一个反模式，在审计报告的 Anti-patterns 节中列出编号和具体位置。
