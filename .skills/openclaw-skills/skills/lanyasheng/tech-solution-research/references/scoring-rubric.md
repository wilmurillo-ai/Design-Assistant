# Scoring Rubric — 7 维评分标准

## 使用说明

本评分标准用于技术方案调研中的量化评估。每个维度 1-5 分，可根据具体场景调整权重。

**评分原则**：
- 5 分 = 行业领先，无明显短板
- 4 分 = 优秀，有少量可接受缺点
- 3 分 = 合格，满足基本需求
- 2 分 = 有缺陷，需谨慎使用
- 1 分 = 不满足需求，不推荐

---

## 1. 功能（Functionality）

评估方案是否满足业务需求。

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 核心功能 100% 覆盖，扩展性强，插件/生态丰富，超出预期 |
| 4 | 核心功能 90%+ 覆盖，有扩展机制，生态较好 |
| 3 | 核心功能 70-90% 覆盖，基本够用，扩展性一般 |
| 2 | 核心功能 50-70% 覆盖，关键功能缺失或需自行实现 |
| 1 | 核心功能 <50% 覆盖，无法满足基本需求 |

### 评估要点

- [ ] **核心功能覆盖度**：必须有的功能是否都支持？
- [ ] **扩展性**：是否支持插件/自定义/二次开发？
- [ ] **生态丰富度**：第三方库、模板、工具是否丰富？
- [ ] **未来路线图**：是否有清晰的发展规划？

### 证据来源

- official-docs：功能清单、API 文档
- github：issues 中功能请求的处理情况
- community：用户分享的使用场景
- runtime-test：实际功能验证

---

## 2. 性能（Performance）

评估方案的速度、资源消耗、并发能力。

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 行业基准领先，资源占用极低，支持高并发 |
| 4 | 性能优秀，略低于顶尖但差距可接受 |
| 3 | 性能中等，满足一般场景，高负载下可能瓶颈 |
| 2 | 性能较差，一般场景就有延迟/资源问题 |
| 1 | 性能不可接受，严重影响用户体验 |

### 评估要点

- [ ] **响应时间**：典型操作的延迟
- [ ] **吞吐量**：单位时间处理能力
- [ ] **资源占用**：CPU、内存、磁盘、网络
- [ ] **并发能力**：同时处理请求数
- [ ] **可扩展性**：水平/垂直扩展能力

### 证据来源

- official-docs：官方基准测试（需验证）
- github：性能相关的 issues/benchmarks
- community：第三方基准测试、用户反馈
- runtime-test：**必须有自己的实测数据**

### 实测建议

```bash
# 示例：API 响应时间测试（将 URL 替换为你的目标服务）
wrk -t12 -c400 -d30s https://example-service.local/api

# 示例：内存占用测试
node --inspect benchmark.js
# 配合 Chrome DevTools Memory 面板
```

## 3. 开发体验（Developer Experience）

评估开发者使用方案的愉悦程度和效率。

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | API 设计优雅，文档完善，工具链强大，学习曲线平缓 |
| 4 | 开发体验良好，文档较全，有常用工具 |
| 3 | 开发体验一般，文档基本够用，需要一定摸索 |
| 2 | 开发体验较差，文档缺失/过时，工具简陋 |
| 1 | 开发体验糟糕，反人类设计，难以使用 |

### 评估要点

- [ ] **API 设计**：是否直观、一致、符合直觉？
- [ ] **文档质量**：是否完整、清晰、有示例？
- [ ] **错误信息**：是否友好、有帮助？
- [ ] **调试工具**：是否有调试/ profiling 工具？
- [ ] **学习曲线**：新手上手难度
- [ ] **IDE 支持**：自动补全、类型提示、linting

### 证据来源

- official-docs：文档完整性、示例质量
- community：教程数量和质量、用户评价
- runtime-test：实际开发过程中的体验记录

### 评估方法

**新手测试**：让未接触过该方案的开发者完成一个简单任务，记录：
- 完成时间
- 查阅文档次数
- 遇到的困惑点
- 主观满意度

## 4. 成本（Cost）

评估方案的总拥有成本（TCO）。

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 完全免费或成本极低，维护成本可忽略 |
| 4 | 成本较低，在预算范围内，ROI 清晰 |
| 3 | 成本中等，需要权衡 ROI |
| 2 | 成本较高，需要充分论证 ROI |
| 1 | 成本过高，超出预算或 ROI 不清晰 |

### 成本构成

| 成本类型 | 说明 | 评估要点 |
|----------|------|----------|
| **License 费用** | 软件授权费 | 开源/商业？按什么计费？ |
| **云资源成本** | 服务器/存储/带宽 | 预估月/年费用 |
| **学习成本** | 团队学习时间 | 培训时间 × 人力成本 |
| **开发成本** | 集成/定制开发 | 开发时间 × 人力成本 |
| **维护成本** | 日常运维/升级 | 运维时间 × 人力成本 |
| **机会成本** | 选择 A 放弃 B 的损失 | 对比备选方案 |

### 计算公式

```
总拥有成本 (TCO) = License + 云资源 + (学习 + 开发 + 维护) × 人力成本 + 机会成本

年度 TCO = 初期投入 / 预计使用年限 + 年度运营成本

ROI = (收益 - 成本) / 成本 × 100%
```

### 证据来源

- official-docs：定价页面、TCO 计算器
- community：用户分享的实际成本
- internal-assets：历史项目成本记录

## 5. 集成难度（Integration Complexity）

评估与现有系统集成的难易程度。

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 开箱即用，与现有技术栈完美兼容，零配置 |
| 4 | 集成简单，少量配置即可，有官方集成指南 |
| 3 | 集成难度中等，需要一定定制，社区有参考案例 |
| 2 | 集成较复杂，需要较多定制，参考案例少 |
| 1 | 集成困难，与现有系统冲突，需要大量改造 |

### 评估要点

- [ ] **API 兼容性**：是否与现有 API 风格一致？
- [ ] **依赖复杂度**：依赖数量、大小、冲突风险
- [ ] **配置复杂度**：需要多少配置项？
- [ ] **数据迁移**：是否需要数据迁移？复杂度？
- [ ] **向后兼容**：升级是否破坏现有功能？
- [ ] **集成文档**：是否有集成指南/最佳实践？

### 证据来源

- official-docs：集成指南、迁移文档
- github：issues 中的集成问题
- community：集成教程、踩坑记录
- internal-assets：历史集成经验

## 6. 安全性（Security）

评估方案的安全记录和实践能力。

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 安全记录优秀，有专业安全团队，定期审计 |
| 4 | 安全记录良好，漏洞响应及时，有安全实践 |
| 3 | 安全记录一般，有漏洞但及时修复 |
| 2 | 安全记录较差，有严重漏洞或响应慢 |
| 1 | 安全风险高，有未修复的严重漏洞 |

### 评估要点

- [ ] **漏洞历史**：过去 12 个月的 CVE 数量和严重程度
- [ ] **响应速度**：漏洞披露到修复的时间
- [ ] **安全特性**：是否有内置安全机制（认证、授权、加密等）
- [ ] **合规性**：是否符合相关合规要求（GDPR、SOC2 等）
- [ ] **安全实践**：是否有安全开发流程、代码审计
- [ ] **依赖安全**：依赖是否有已知漏洞

### 证据来源

- github：Security Advisory、CVE 记录
- official-docs：安全文档、合规认证
- community：安全讨论、漏洞披露

### 安全检查清单

- [ ] 检查 GitHub Security Advisory
- [ ] 检查依赖的 CVE（npm audit / pip audit / etc.）
- [ ] 查看过去 12 个月的漏洞修复记录
- [ ] 确认是否有安全更新通知机制
- [ ] 评估默认配置的安全性

## 7. 长期维护（Long-term Maintainability）

评估方案的可持续性和长期支持能力。

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 背后有稳定组织支持，社区活跃，版本更新规律，长期支持承诺 |
| 4 | 维护良好，社区较活跃，更新频率稳定 |
| 3 | 维护正常，社区一般，更新频率可接受 |
| 2 | 维护不稳定，社区冷清，更新缓慢 |
| 1 | 已停止维护或接近 EOL，社区消失 |

### 评估要点

- [ ] **背后组织**：个人项目/公司支持/基金会？
- [ ] **社区活跃度**：Star 增长、issue 响应、PR 合并
- [ ] **版本更新频率**：多久发布一个新版本？
- [ ] **向后兼容性**：升级是否破坏现有代码？
- [ ] **EOL 政策**：是否有明确的版本支持周期？
- [ ] **替代方案**：如项目停止，是否有迁移路径？

### 关键指标

| 指标 | 5 分 | 3 分 | 1 分 |
|------|------|------|------|
| Star 增长（月） | >500 | 50-500 | <50 |
| Issue 响应时间 | <24h | 1-7 天 | >7 天 |
| 版本更新频率 | 月更 | 季更 | 半年+ |
| 核心贡献者 | >10 人 | 3-10 人 | <3 人 |
| 最新 commit | <1 周 | 1-4 周 | >1 月 |

### 证据来源

- github：贡献者数量、commit 频率、issue 响应
- official-docs：Roadmap、EOL 政策
- community：用户对未来稳定性的担忧/信心

## 评分计算示例

```
方案 A 评分：
- 功能：4 分 × 20% = 0.8
- 性能：5 分 × 10% = 0.5
- 开发体验：4 分 × 25% = 1.0
- 成本：3 分 × 15% = 0.45
- 集成难度：4 分 × 15% = 0.6
- 安全性：4 分 × 5% = 0.2
- 长期维护：4 分 × 10% = 0.4
- **加权总分：3.95 分**

方案 B 评分：
- 功能：5 分 × 20% = 1.0
- 性能：3 分 × 10% = 0.3
- 开发体验：3 分 × 25% = 0.75
- 成本：4 分 × 15% = 0.6
- 集成难度：3 分 × 15% = 0.45
- 安全性：5 分 × 5% = 0.25
- 长期维护：3 分 × 10% = 0.3
- **加权总分：3.65 分**

结论：方案 A（3.95 分）略优于方案 B（3.65 分）
```
