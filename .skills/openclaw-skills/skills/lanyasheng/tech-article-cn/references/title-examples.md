# 好标题 vs 坏标题对比库

> 蒸馏自 Cole Schaefer 方法论 + Ogilvy 标题框架 + 中文技术社区高赞文章分析

## 核心原则

标题是 80% 的钱。如果你花 5 小时写文章，至少 1 小时应该花在标题上。

写 10-15 个标题候选，放一放，回来选最好的那个。

## 四种标题策略

### 1. 悬念钩子 — 给一个味道但不揭底

```
✅ "我们的 API 网关每天处理 2 亿请求。上个月它挂了 47 分钟。"
✅ "这个 bug 藏了 6 个月，根因是一行看起来完全正确的代码"
✅ "为什么我们放弃了 GraphQL（在用了 18 个月之后）"

❌ "一次关于 API 网关的实践分享"
❌ "GraphQL 的优缺点分析"
```

### 2. 直给结论 — 告诉读者他能得到什么

```
✅ "50 个服务从 REST 迁到 gRPC：我们踩的 7 个坑"
✅ "把编译时间从 47 分钟砍到 3 分钟的 5 个改动"
✅ "React 18 并发模式完全指南：从零到生产"

❌ "浅谈微服务迁移的一些实践"
❌ "编译优化的一些思考"
```

### 3. 反直觉/争议 — 挑战读者的已有认知

```
✅ "Serverless 不会成为主流——来自三年实践者的反思"
✅ "为什么 100% 测试覆盖率让我们的代码更烂了"
✅ "我们把微服务合回了 monolith，性能提升了 10 倍"

❌ "关于 Serverless 的一些个人看法"
❌ "测试覆盖率的利与弊"
```

### 4. 具体数字 — 数据自带说服力

```
✅ "从 800ms 到 12ms：一次数据库查询优化的完整过程"
✅ "3 个月，120 个文件，0 个 bug：我们的 TypeScript 迁移记录"
✅ "日均 1 亿消息：我们的消息队列选型之路"

❌ "数据库查询优化实践"
❌ "TypeScript 迁移分享"
```

## 坏标题模式（禁用）

| 模式 | 示例 | 问题 |
|------|------|------|
| "浅谈"开头 | 浅谈 XX 的最佳实践 | 自我贬低，读者凭什么花时间看"浅"的东西 |
| "深入理解" | 深入理解 XX 的原理与实战 | 空洞承诺，没说清楚深入了什么 |
| "关于...的一些思考" | 关于微服务的一些思考 | 太模糊，可以是任何内容 |
| "论..." | 论分布式系统的一致性 | 学术论文语气，读者预期枯燥 |
| 全英文术语堆砌 | Kubernetes + Istio + Envoy 实践 | 关键词罗列不是标题 |
| "最佳实践" | XX 最佳实践大全 | 通用到放在任何文章都成立 |

## 标题长度

- 中文技术文章：**15-25 个字**效果最好
- 太短（<10 字）缺信息量
- 太长（>30 字）无法一眼扫完
- 含数字的标题点击率高 30%+（来自掘金社区数据）

## 标题公式

### 公式 1: [数字] + [对象] + [动作] + [结果]

```
"50 个服务从 REST 迁到 gRPC：延迟降了 40%"
"3 行配置修复了困扰我们 2 个月的内存泄漏"
```

### 公式 2: [意外结果] + [背景]

```
"编译时间反而变长了——升级到 Gradle 8 的教训"
"性能提升 10 倍的改动只有 5 行代码"
```

### 公式 3: [时间] + [挑战] + [暗示故事]

```
"凌晨 2 点的 P0 故障：一次 DNS 缓存引发的血案"
"上线前 3 小时发现的致命 bug"
```

### 公式 4: 问题式

```
"为什么你的 Docker 镜像有 2GB？"
"gRPC 真的比 REST 快吗？我们测了 6 个月"
```

## 标题自检

写完标题后问自己：

```
□ 读标题就知道文章讲什么？
□ 有具体数字或具体技术名称？
□ 有冲突/悬念/反直觉的元素？
□ 去掉"浅谈""深入""一些"了吗？
□ 15-25 个字（中文）？
□ 换成任何公司/项目的标题都成立吗？（如果是，太通用了）
```

## 句子级打磨要点

> 蒸馏自 Cole Schaefer / Ogilvy 方法论

### 短第一句

每段、每节的第一句话：10 字以内好，8 字以内更好，5 字以内最佳。

```
✅ "编译挂了。" → 读者立刻想知道为什么
✅ "这不对。" → 制造紧张感

❌ "在经过多次尝试和反复调试之后，我们终于发现了编译失败的根本原因。"
```

### 节奏变化

连续 3 句同样长度 = 读者走神。短句和长句交替，像呼吸。

```
✅ "查询慢了。不是一点点慢，是从 50ms 变成了 3 秒那种慢。我们看了监控，
    看了日志，甚至怀疑是不是机房网络出了问题。结果根因在索引上。一个月前
    有人改了字段类型，索引失效了，没人注意到。"

→ 短(3字)→长(18字)→长(24字)→短(5字)→长(16字)
```

### 简单词优先

用一个字能说清的不用两个字。"用" 不用 "使用"。"改" 不用 "修改"。

技术术语保持原样（Kubernetes、gRPC），但连接词/描述词用最简单的。

### 杀副词

扫描所有 "地" "了" "着" 结尾的修饰语。如果删掉不影响意思就删。

```
❌ "显著地提升了性能"
✅ "性能提升了 10 倍"

❌ "成功地完成了迁移"
✅ "迁移完成"
```
