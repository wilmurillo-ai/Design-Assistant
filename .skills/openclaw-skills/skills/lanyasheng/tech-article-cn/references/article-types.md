# 文章类型详细指南

> 以下模板的字数估算是每个段落的**最低内容量**，适用于掘金等短文平台（1500-3000 字）。
> 内部技术分享长文（10000-20000 字）需要在每个段落基础上扩展 3-5 倍：加更多细节、子案例、数据分析、代码走读。

## 1. 案例复盘型

最受欢迎的技术文章类型。读者想知道"别人怎么做的，我能学到什么"。

### 结构模板

```
HOOK: 结果数据 + 冲突/意外点
  "我们用 3 个月把 50 个服务从 REST 迁到 gRPC。P95 降了 40%，但中间差点把生产搞挂。"

背景 (~200字): 最少量上下文
  - 我们是什么团队，做什么业务
  - 遇到了什么问题促使我们做这件事
  - 不需要介绍技术本身（读者知道 gRPC 是什么）

决策过程 (~300字): 为什么这么选
  - 考虑了哪些方案
  - 选择标准是什么
  - 最终选了什么，为什么

实施路径 (~500字): 怎么做的
  - 分几个阶段
  - 关键的技术决策（有代码/架构图）
  - 时间线

踩坑实录 (~500字): 3-5 个具体故事
  - 每个坑: 症状→排查→根因→修复
  - 用代码/日志/截图说明
  - 说清楚为什么会踩——这才是价值

结果 (~200字): 数据对比
  - before/after 数据表
  - 超出预期的地方
  - 没达到预期的地方（诚实）

CLOSE (~100字): 不是"总结"
  - 下一步计划
  - 还没解决的问题
  - 给读者的建议
```

### 好案例 vs 坏案例

```
❌ 坏案例:
"我们采用了微服务架构，实现了服务解耦，提升了开发效率。
通过引入容器化技术，实现了快速部署。
经过团队的共同努力，项目顺利上线。"

→ 问题: 没有数据、没有细节、没有故事、放到任何项目都成立

✅ 好案例:
"第 23 个服务迁完的时候，我们发现 gRPC 的连接池在高并发下会
泄漏。Grafana 上看到连接数从正常的 50 涨到 1200+，然后 OOM。
排查了 2 天，最终定位到 grpc-go 的 keepalive 参数和我们的
负载均衡策略冲突。修复就一行配置，但定位过程消耗了整个 Sprint。"

→ 有时间线、有具体数字、有排查过程、有情绪
```

## 2. 问题深潜型

解决了一个有趣的技术问题，把排查过程写成侦探故事。

### 结构模板

```
HOOK: 症状 + 影响范围
  "线上告警: API 响应时间从 50ms 飙到 3s。影响 2000+ 用户。凌晨 2 点。"

症状描述 (~200字):
  - 什么时候开始的
  - 表现是什么（监控截图/日志片段）
  - 影响范围

排查过程 (~800字): 这是文章的核心
  - 第一个假设 → 验证 → 排除（为什么排除）
  - 第二个假设 → 验证 → 发现线索
  - 顺着线索深挖 → 定位根因
  - 用代码/日志/调用链说明每一步

根因分析 (~300字):
  - 根因是什么（一句话）
  - 为什么会出现这个问题（架构/设计/代码层面）
  - 代码级别的分析（关键代码段 + 注释）

修复方案 (~200字):
  - 怎么修的（diff 或关键代码）
  - 为什么选这个方案
  - 修复后的验证数据

CLOSE: 防护措施 + 反思
  - 加了什么监控/告警
  - 团队流程有什么调整
  - 这个问题揭示了什么更深层的问题
```

## 3. 技术选型型

技术选型文章的核心价值：不是告诉读者"X 好"，而是展示评估过程，让读者能用你的方法做自己的选型。

### 结构模板

```
HOOK: 选型的背景约束
  "我们需要一个消息队列。要求: 日均 1 亿消息，P99 < 10ms，团队没人用过 Kafka。"

需求分析 (~300字):
  - 硬约束（必须满足）
  - 软约束（最好满足）
  - 非约束（不关心的）

候选方案 (~400字):
  - 列出 3-5 个候选
  - 每个 1-2 句话说清楚它是什么

评估对比 (~500字):
  - 评估维度表（性能/运维成本/团队学习曲线/社区生态...）
  - 每个维度的具体测试数据（不是网上抄的，是自己跑的）
  - POC 代码和测试方法

最终选择 (~200字):
  - 选了什么
  - 最关键的决策因素是什么
  - 放弃其他方案的具体原因

CLOSE: 使用 N 个月后的回顾
  - 当初的判断对了吗
  - 有什么意外
  - 如果重来会怎么选
```

## 4. 原理解析型

帮读者建立心智模型。不是翻译官方文档，而是"我搞懂之后用人话跟你讲"。

### 结构模板

```
HOOK: 用一个具体问题引入
  "为什么 React 18 的 useEffect 在 StrictMode 下会执行两次？"

它是什么 (~200字):
  - 一句话定义（用你自己的话）
  - 它解决什么问题
  - 它不解决什么问题（边界很重要）

它怎么工作 (~800字):
  - 从简单到复杂，分 2-3 层讲
  - 每层一个最小可运行示例
  - 配架构图/流程图

为什么这样设计 (~300字):
  - 设计者面临的 trade-off
  - 他们选了什么，放弃了什么
  - 引用源码/RFC/设计文档

实际表现 (~200字):
  - 在你的场景下的真实数据
  - 和预期的差异
  - 使用注意事项

CLOSE: 延伸
  - 相关概念的指引
  - 进一步学习的资源
```

## 5. 教程实战型

教程的目标：读者跟着做，10 分钟内跑通。

### 结构模板

```
HOOK: 最终效果展示
  截图/GIF/demo 链接——让读者先看到终点

前提条件 (~100字):
  - 需要什么环境
  - 需要什么前置知识
  - 预估完成时间

步骤 (~1500字):
  - 每步: 做什么 → 代码 → 预期结果
  - 代码块必须可直接 copy-paste 运行
  - 每步结束有验证方法（"运行后应该看到..."）

常见问题 (~300字):
  - 3-5 个常见坑
  - 每个: 错误信息 → 原因 → 修复

CLOSE: 下一步
  - 可以扩展什么
  - 进阶资源
  - 完整代码 repo 链接
```

## 6. 趋势观点型

最难写的类型。需要有真实经验支撑的判断，不是转述他人观点。

### 结构模板

```
HOOK: 反直觉的判断
  "我认为 Serverless 在未来 3 年内不会成为主流。这是一个不受欢迎的观点。"

现状 (~200字):
  - 当前的事实（有数据支撑）
  - 主流叙事是什么

我的判断 (~300字):
  - 我认为会怎样
  - 清晰表达立场（不要 hedging）

依据 (~500字):
  - 来自个人实践的证据（最有说服力）
  - 来自行业数据的支撑
  - 来自技术原理的推导

反驳预期反对意见 (~200字):
  - "有人会说 X"→ 我的回应
  - 承认自己判断可能错的地方

CLOSE: 行动建议
  - 如果我是对的，读者应该怎么做
  - 如果我是错的，最坏情况是什么
```
