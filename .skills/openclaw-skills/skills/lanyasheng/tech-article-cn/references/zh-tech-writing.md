# 中文技术写作常见问题和修复

## 1. 开头综合症

中文技术文章最常见的开头有三种，全是低分开头：

### "随着...的发展" 开头

```
❌ "随着云原生技术的快速发展，微服务架构已经成为..."
❌ "随着 AI 技术的不断进步，大模型在..."
❌ "在当今数字化转型的大背景下..."

→ 问题: 这是万能开头，放到任何文章都成立。读者第一秒就判定"AI 写的"或"没话找话"。

✅ "我们的 monolith 有 120 万行代码。上周五它编译了 47 分钟。"
✅ "这个 bug 藏了 6 个月，影响了 3 万用户，根因是一行看起来完全正确的代码。"
✅ "团队 8 个人，同时改一个文件。Git 冲突率 40%。我们不得不拆服务。"
```

### "本文将介绍..." 开头

```
❌ "本文将从以下几个方面介绍..."
❌ "本文主要分享我们团队在 X 方面的实践经验"

→ 问题: 这不是开头，这是目录。读者想看结论和故事，不想看你的提纲。

✅ 直接从最有趣的部分开始。如果你的文章最有趣的部分在第三段，那第三段就是你的开头。
```

### "众所周知" 开头

```
❌ "众所周知，性能优化是..."
❌ "大家都知道，微服务架构的核心思想是..."

→ 问题: 如果众所周知，为什么还要写？

✅ 从"众所周知"的反面开始: "大多数人以为 X，但实际上 Y。"
```

## 2. 四字成语堆砌

```
❌ "高效协同、智能赋能、敏捷迭代、持续交付"
❌ "全面覆盖、深度整合、无缝对接、灵活扩展"

→ 问题: 四字词自带权威感和抽象感，读起来像会议纪要。

✅ 把每个四字词展开成具体的事:
  "高效协同" → "同一个需求，iOS 和 Android 的代码改动量从 2 天降到了半天"
  "智能赋能" → "自动生成的测试用例覆盖了 80% 的边界情况"
  "敏捷迭代" → "从周发布改成了日发布，每次改动不超过 200 行"
```

## 3. 无意义的"总分总"

```
❌ "下面我将从三个方面展开讨论:
    第一，xxx；第二，yyy；第三，zzz。
    综上所述，xxx、yyy、zzz 共同构成了..."

→ 问题: 开头说了要讲什么，中间讲了，结尾重复一遍讲了什么。读者看了三遍同样的信息。

✅ 直接说最重要的。如果有三个点，从最重要的开始，其他自然带出。
   结尾不要重复开头——给一个新的视角、一个行动项、或一个开放问题。
```

## 4. 过度谦虚

```
❌ "一些微小的尝试"
❌ "抛砖引玉，希望能给大家一些启发"
❌ "不足之处还请大家多多指正"
❌ "个人浅见，仅供参考"

→ 问题: 你花了几个月做的事，写了几千字的文章，然后说这是"微小的尝试"？
   过度谦虚在中文语境中确实有礼貌功能，但在技术文章中它会削弱你的信息的可信度。

✅ 实事求是。做了什么就说什么。
   如果确实有局限，写清楚具体的局限: "这个方案在并发超过 10k 时没有验证过。"
   这比"一些微小的尝试"有信息量得多。
```

## 5. 被动语态过多

```
❌ "该方案已被成功实施" / "性能已被优化" / "问题已被解决"
❌ "通过引入 X 技术，Y 问题得到了有效缓解"

→ 问题: 谁实施的？怎么优化的？解决了多少？被动语态隐藏了关键信息。

✅ "我们把 API 响应时间从 800ms 降到 50ms"
✅ "团队选了方案 B，因为 A 需要改 30 个服务的接口"
✅ "我花了两天排查，最后发现根因是连接池泄漏"
```

## 6. 没有具体数据

```
❌ "性能显著提升"
❌ "大幅降低了错误率"
❌ "开发效率得到了明显改善"

→ 问题: "显著""大幅""明显"是空话。什么叫显著？从多少到多少？

✅ "查询时间从 847ms 降到 12ms（70x）"
✅ "错误率从 2.3% 降到 0.04%"
✅ "新功能的平均交付时间从 2 周缩短到 3 天"

没有精确数据就给范围: "大约从 5s 降到了 500ms 左右"
连范围都没有就别说这个点。
```

## 7. 结尾综合症

```
❌ "综上所述，本文从 A、B、C 三个方面..."（重复开头）
❌ "展望未来，我们将继续探索..."（空洞承诺）
❌ "以上就是我们的实践经验，希望对大家有帮助。"（无新信息）

✅ 好结尾的三种写法:
  1. 下一步: "接下来我们计划把这个方案扩展到离线场景，但还没解决 X 问题。"
  2. 开放问题: "一个还没想清楚的问题: 当 replica 超过 5 个时，一致性怎么保证？"
  3. 给读者行动项: "想试的话: clone 这个 repo，改 config.yaml 里的 X 字段，跑一下 benchmark。"
```

## 8. 中英文混排问题

```
技术文章经常中英文混排，注意:

✅ 专有名词用英文: Kubernetes, gRPC, React（不要翻译成"反应"）
✅ 中英文之间加空格: "我们使用 Kubernetes 部署"（不是"我们使用Kubernetes部署"）
✅ 代码/命令用 backtick: "运行 `kubectl get pods`"
❌ 不要中英文随意切换: "这个 feature 很 nice，可以 significantly improve 我们的 performance"
```

## 9. 段落太长

```
❌ 一段超过 150 字（手机屏幕上看就是一面墙）

✅ 每段一个观点，超过 3 句就考虑拆分
✅ 段落间用空行隔开
✅ 关键信息放段首——很多读者只看每段第一句
```

## 10. 自检清单

发布前逐条过:

```
□ 开头前 3 句有没有具体数据或故事？
□ 有没有"随着...的发展"、"众所周知"、"本文将"？
□ 每个"显著/大幅/明显"都替换成了具体数字？
□ 四字成语都展开成了具体描述？
□ 结尾是不是在重复开头？
□ 最长的段落不超过 150 字？
□ 中英文之间有空格？
□ 被动句都改成了主动句？
□ 删掉"抛砖引玉"和"一些微小的尝试"了吗？
```
