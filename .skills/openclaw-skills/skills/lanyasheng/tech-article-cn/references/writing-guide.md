# 写作指南（Phase 2-4 详细规则）

从 SKILL.md 拆出的详细写法指导。SKILL.md 保留流程骨架和规则引用，具体怎么写看这里。

## TL;DR 写法

长文（>20000字）SHOULD 加 TL;DR。放在 HOOK 之前或之后，用 3-5 条判断句概括全文核心观点。

```
❌ "本文介绍了 Agent 架构的各个方面"（这是目录不是 TL;DR）
❌ "Agent 很重要，Harness 也很重要"（废话）

✅ "更贵的模型带来的提升，很多时候没有想象中那么大，反而 Harness 和验证测试质量对成功率的影响更大"
✅ "调试 Agent 行为时，应优先检查工具定义，因为多数工具选择错误都出在描述不准确"
✅ "评测系统本身的问题，很多时候比 Agent 出问题更难发现"
```

TL;DR 的核心：**判断句，不是描述句**。每条都应该是一个可以被反驳的观点，而不是一段内容介绍。

## 小节标题：有信息量，偶尔用问句

好标题让读者一眼知道这段讲什么。问句可以用，但不能堆砌——全是问句读起来像 FAQ，反而成了新的 AI 味。

```
❌ "Harness 的重要性"（太虚）
❌ "上下文分层结构"（没有观点）
❌ "为什么要分层？"（如果全文都是这种问句，很累）

✅ "Harness 比模型更关键"（有判断）
✅ "上下文为什么要分层"（关键转折处用问句）
✅ "4 个安全阀"（简洁有信息量）
✅ "Checkpoint + Rollback：破坏性命令前拍快照"（告诉读者这段讲什么）
```

小节标题 SHOULD 有信息量（读标题就知道这段的结论或主题）。问句适合用在关键转折处（30-50% 为宜），不要全用问句。

## HOOK 写法（最重要的 3 行）

```
❌ "本文将介绍我们团队在微服务迁移方面的实践经验"
❌ "随着云原生技术的发展，微服务架构越来越受到关注"
❌ "在当今快速发展的技术环境中..."

✅ "3 个月，50 个服务，从 REST 到 gRPC。延迟降了 40%，但中间差点把生产搞挂。"
✅ "我们的 API 网关每天处理 2 亿请求。上个月它挂了 47 分钟。这是我们排查出的根因。"
✅ "大多数 gRPC 迁移文章告诉你怎么写 proto 文件。没人告诉你第 30 个服务迁完后会遇到什么。"
```

HOOK 的核心：**具体数据 + 悬念/冲突/意外**。让读者觉得"这个人真做过"。

## Phase 3: 写草稿 — 先完成后完美

### 写作节奏

- 每段先写**论点句**（这段的核心观点，一句话），再补充证据和细节
- 段落间不需要过渡句。如果必须过渡，内容本身就是最好的过渡
- 技术细节用代码块、数据表格、架构图呈现。不要用文字描述代码
- 长段落（>150字）必须拆分。读者在手机上读你的文章

### 证据优先级

| 证据类型 | 说服力 | 示例 |
|---------|-------|------|
| 好做法 vs 差做法对比表 | ★★★★★ | 两栏对比，读者一眼看出区别 |
| 可运行代码 | ★★★★★ | 读者 copy-paste 就能跑 |
| 性能数据对比 | ★★★★★ | before/after，带环境信息 |
| 真实故障/踩坑 | ★★★★ | 具体时间线、症状、修复过程 |
| 架构图/流程图 | ★★★★ | 自制 SVG/Mermaid，不要用网上找的通用图 |
| 团队讨论/决策过程 | ★★★ | "我们考虑了 A 和 B，选了 A 因为..." |
| 外部引用 | ★★ | 有出处的论文/官方文档 |
| 个人感受/观点 | ★★ | 真诚但不能替代证据 |

### 对比表格：信息密度最高的武器

对比表是技术文章最有效的证据形式之一。两栏对比让读者不需要动脑就能理解差异。好的技术文章每篇通常有 5+ 个对比表格。

```
常用对比维度:
- 好做法 vs 差做法（最常用）
- 方案 A vs 方案 B（技术选型必备）
- 设计目标 vs 反模式
- 旧方案 vs 新方案（迁移类文章）
```

每个核心概念 SHOULD 配备 **图/表/代码 三件套中的至少两种**。只有文字描述的技术概念 = 读者记不住。

### HTML 图表

架构图、流程图、对比图 MUST 用 HTML+CSS 色块渲染，不要用 ASCII art 代码块。详细风格规范见 `html-diagram-style.md`，核心要点：

- 统一容器：`background:#fafafa;border:1px solid #e5e7eb;border-radius:12px;padding:24px`
- 标签用药丸样式：`border-radius:6px;padding:4px 10px;font-size:12px`
- 6 种语义颜色（绿/蓝/黄/红/紫/灰）各有 浅底+边框+文字 三色
- 层级标题：`font-size:11px;font-weight:700;letter-spacing:1px` 大写风格
- 不用 emoji、不用渐变、不用 `position:absolute`

### "为什么" 追问模式

不要只描述 what（"系统用了 AsyncGenerator"），要追问 why（"为什么用 AsyncGenerator？三个原因..."）。每个决策背后都有 Tradeoff，写出来比隐藏更有价值。

```
❌ "Claude Code 的消息扣留机制会拦截三类消息"
✅ "Claude Code 的消息扣留机制会拦截三类消息。为什么不直接传给调用方？因为 SDK 消费者看到 error 就会终止会话，但内部可能还有恢复路径"

❌ "我们选择了分批迁移策略"
✅ "我们选择了分批迁移策略。为什么不一次性切换？因为第 15 个服务切完后我们发现了一个序列化兼容性问题，如果全量切换就是全站故障"
```

每个技术决策/设计选择 SHOULD 追加一个 why 段。读者要的不是信息搬运，是洞察。

### 句子级打磨

- **短第一句**: 每段开头句 10 字以内。"编译挂了。"比"在经过反复调试后我们发现..."好 10 倍
- **节奏变化**: 短句→长句→短句。连续 3 句同样长度 = 读者走神。朗读一遍就知道
- **简单词优先**: "用"不用"使用"、"改"不用"修改"、"is"不用"serves as"
- **杀副词**: 扫描所有"显著地""成功地""有效地"。删掉不影响意思就删
- **具体替换模糊**: "很多公司" → 哪些？"性能提升" → 从多少到多少？"studies show" → 哪个 study？

### 人味注入

去掉 AI 模式只是一半。干净但无菌的文字一样一眼能看出是 AI 改过的。

- **有观点**: 纯中立陈述 → 加一句反应。"这个数据让我们放弃了方案 A"
- **有不确定**: "我可能错了，但..."比"综合分析表明..."更让人信任
- **有个人经历**: "我们花了两天排查"比"问题被定位"有温度
- **有跑题**: 完美结构感觉算法生成的。偶尔的旁白和跑题是人的标志
- **有具体感受**: "这个 bug 让人抓狂"比"this is concerning"有力

### 每段自检

写完一段后问自己：
1. 这段的论点句是什么？如果没有，加一个
2. 证据具体吗？"性能提升了"→"P95 从 847ms 降到 12ms"
3. 这段放到别人的文章里也成立吗？如果是，你的独特视角在哪

## CLOSE：结尾怎么写才不像 AI

"总结"是最弱的结尾。读者刚读完全文，不需要你帮他总结。好的结尾给出新的东西。

**三种结尾模式（按场景选一种）：**

| 模式 | 适用场景 | 做法 |
|------|---------|------|
| **反模式表** | 工程实践类 | 5-8 个常见错误 + 问题 + 怎么修，单独成章 |
| **三段反思** | 源码分析/技术选型类 | "值得学习" + "值得商榷" + "如果重新设计" |
| **编号要点** | 综述类/长文 | 10 条压缩后的核心判断，每条一个可被反驳的观点 |

```
❌ "综上所述，本文介绍了 Agent 的架构设计..."（重复目录）
❌ "总结一下，我们从以下几个方面..."（浪费读者时间）

✅ 反模式表：列出 5-8 个常见错误 + 症状 + 修法，读者可以直接当 checklist 用
✅ 三段反思：哪些设计值得学、哪些可以商榷、如果你来做会怎么改
✅ 编号要点：10 条压缩后的核心判断，每条都是独立的可执行洞察
```

## 标题优化

```
❌ 浅谈微服务架构的最佳实践
❌ 深入理解 gRPC 原理与实战
❌ 关于 API 网关的一些思考

✅ 50 个服务从 REST 迁到 gRPC：我们踩的 7 个坑
✅ API 网关挂了 47 分钟：一次 P0 故障的完整复盘
✅ 为什么我们放弃了 GraphQL（在用了 18 个月之后）
```

好标题 = **具体 + 有信息量 + 引发好奇**。数字、时间线、反直觉结论都是好信号。
