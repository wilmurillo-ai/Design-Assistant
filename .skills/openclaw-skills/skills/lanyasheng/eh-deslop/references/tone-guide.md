# 场景语气指南与快速修复

> 不同场景下的语气校准和常见问题的快速修复清单。
> 来源：human-writing 项目实践 + slopbuster voice-and-soul guide。

---

## 场景语气矩阵

### 技术分享（给同行）

**定位**: 懂行的同行，不是老师

**假设**: 读者有技术背景，想要具体细节，不需要基础概念解释

**证据优先级**: 代码示例 > 性能数据 > 真实案例 > 架构图

**示例**:
```markdown
上下文压缩会丢失"为什么这么做"的决策记忆。
我们在 COMPACT_STATE.md 中写入语义快照——
做到哪一步、关键决策是什么、有什么阻塞。

新会话启动时优先读这个文件。
覆盖写入，不追加——试过追加式，很快变成日志，没人会读。
```

**要点**: 直接说做法和原因。不需要"首先让我解释一下什么是上下文压缩"。

### 愿景/方案分享（给决策者）

**定位**: 有实践支撑的建设者，不是推销员

**假设**: 读者持怀疑态度，需要用实例说服

**证据优先级**: 前后对比 > 真实案例 > 具体数字 > 回应质疑

**示例**:
```markdown
我们试过让 AI 自行判断任务类型——
同一个 Crash 有时走 problem-fix，有时走 tdd。

确定性路由消除了这种随机性。
11 种工作流，每种有对应的入口 Skill 和委派策略。
```

**要点**: 先说问题（试过，不行），再说解法，用具体例子支撑。

### 教程/指南（给新手）

**定位**: 犯过错的过来人，不是居高临下的教导者

**假设**: 读者想跟着做，能复制粘贴，想学到东西

**证据优先级**: 可运行示例 > 预期输出 > 常见坑 > 排查步骤

**示例**:
```markdown
1. Clone 模板（2 分钟）
   cp -r template/.agents template/.cursor template/AGENTS.md ./

2. 填 PROJECT_KNOWLEDGE.md（5-10 分钟）
   技术栈、构建命令、目录结构、绝对禁止项。

3. 开始用
   AI 会自动读基础文件、匹配工作流、更新记忆。
   不需要"教"AI。在它犯错时纠正就行。
```

**要点**: 每步有时间预期，有可执行命令，最后一句解除焦虑。

## 快速修复清单

### 太正式？

- "利用" → "用"
- "为了" → "来" 或直接删
- "在此基础上" → 删掉，直接说下一件事
- "进行了优化" → "优化了"
- 删掉"很"、"非常"、"十分"、"相当"

### 太通用？

- 加一个具体数字
- 提到一个真实项目/公司
- 加一段代码示例
- 提一个具体的边界情况
- 引用用户反馈

### 太推销？

- 把最高级替换成对比（"最好的" → "比 X 好在..."）
- 把"革命性的" → "不同在于"
- 删感叹号
- 砍掉第一段（通常是营销废话）
- 加一段"做不到什么"

## 结构修复

### 开头

好的开头应该做 **其中一件**：
- 说一个读者感受过的具体问题
- 给一个意外的事实或反直觉的观察
- 描述一个摩擦、失败或发现的瞬间
- 问一个真正的（非修辞性的）问题

坏的开头：
- 定义一个大家都知道的概念
- 没有利害关系的历史背景
- "在当今..."
- 没有人会不同意的陈述
- 超过 3 句

### 结尾

好的结尾应该：
- 把论点延伸到新的方向
- 给读者一个具体的、可执行的下一步
- 留一个真正的、未解决的问题
- 从新角度重新框定前文

坏的结尾：
- 总结你刚说的
- "总结来看" / "综上所述"
- 预测光明的未来
- 适用于任何文章的行动号召

---

## 修订流程

1. **初稿**: 把想法写下来，不管质量
2. **二稿**: 砍 30% 的内容
3. **三稿**: 给每个声明加具体数据
4. **四稿**: 读出声——如果不会在 GitHub Issue 评论里这么说，就重写
