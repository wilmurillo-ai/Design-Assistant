# 工作流 Skill 模板

本文件是 auto-create-skill 生成工作流 Skill 时的参考模板。
根据实际需求调整，不需要照搬所有部分。

---

## 模板结构

```markdown
---
name: <skill-name>
description: >
  <一句话说明 Skill 的用途>。

  当用户提到以下情况时使用本 Skill：
  - <触发短语 1>
  - <触发短语 2>
  - <触发短语 3>
  （尽可能列举所有可能的触发方式，宁多勿少）

  本 Skill 由 auto-create-skill 创建，可通过 auto-create-skill 进行修改。
---

# <Skill 标题>

## 概述

<简要描述这个工作流做什么、适用场景、不适用场景>

| 参数名 | 类型 | 必填 | 默认值 | 说明 | 示例 |
|--------|------|------|--------|------|------|
| param1 | string | 是 | - | 参数说明 | PROJ-1234 |

在工作流开始前，收集所有必填参数。如果用户没有提供必填参数，主动询问。
可以一次性询问所有缺失的必填参数，避免多次来回。

## 完整示例：简单 BUG 修复 Skill

下面是一个完整的实际示例，展示如何把用户描述的 BUG 修复流程转化为 Skill：

```markdown
---
name: simple-bugfix
description: >
  简单 BUG 的自动修复流程：通过 Jira ID 查询 BUG 信息，自动修复代码，
  用户验证后提交到 bugfix 分支并推送远端。

  当用户提到以下情况时使用本 Skill：
  - "修复简单 BUG" / "简单 BUG 修复"
  - "用简单流程修这个 bug"
  - "修一个简单的 bug，ID 是 XXX"
  - 提到 Jira ID 并要求修复，且标记为简单级别
  - "使用 simple-bugfix 流程"

  本 Skill 由 auto-create-skill 创建，可通过 auto-create-skill 进行修改。
---

# 简单 BUG 修复流程

## 概述

适用于排查难度为「简单」的 BUG。流程全自动化，仅在修复完成后需要用户验证。
不适用于中等和复杂难度的 BUG（这些需要先与用户确认排查思路）。

## 输入参数

| 参数名 | 类型 | 必填 | 默认值 | 说明 | 示例 |
|--------|------|------|--------|------|------|
| jira_id | string | 是 | - | Jira Issue ID | PROJ-1234 |

### 获取参数

开始前确认用户已提供 jira_id。如果没有，直接询问：
「请提供需要修复的 BUG 的 Jira ID。」

## 工作流步骤

### 步骤 1：查询 BUG 信息 [AUTO]

**目的**：获取 BUG 的完整信息，包括描述、重现步骤、相关代码模块等。

**操作**：
- 使用 Jira MCP 查询 issue 详情
- 提取关键信息：标题、描述、重现步骤、关联组件、优先级

**工具调用**：
- MCP Server: jira
- 方法: get_issue
- 参数: issue_key = {jira_id}

**失败处理**：
- 如果 Jira ID 不存在，通知用户检查 ID 是否正确
- 如果 Jira MCP 连接失败，提示用户检查 MCP 配置

---

### 步骤 2：分析并修复代码 [AUTO]

**目的**：根据 BUG 信息定位问题代码并进行修复。

**操作**：
- 根据 BUG 描述和关联组件，在代码库中定位相关文件
- 分析问题原因
- 实施修复
- 确保修复不引入新问题（如果项目有测试，运行相关测试）

**失败处理**：
- 如果无法定位问题，向用户说明情况并建议升级到中等流程
- 如果测试失败，展示失败信息并询问用户意见

---

### 步骤 3：通知用户验证 [CONFIRM]

**目的**：让用户确认修复结果是否正确。

**操作**：
- 向用户展示：
  - 修改了哪些文件
  - 每个文件的具体改动（diff 格式）
  - 修复逻辑的简要说明
- 等待用户确认

**用户响应处理**：
- 如果用户确认通过 → 继续步骤 4
- 如果用户发现问题 → 返回步骤 2，根据反馈重新修复

---

### 步骤 4：提交并推送 [AUTO]

**目的**：将修复提交到对应的 bugfix 分支并推送到远端。

**操作**：
```bash
# 创建并切换到 bugfix 分支
git checkout -b bugfix/{jira_id}

# 添加修改的文件
git add <modified files>

# 提交，commit message 包含 Jira ID
git commit -m "fix({jira_id}): <BUG 标题的简要描述>"

# 推送到远端
git push origin bugfix/{jira_id}
```

**失败处理**：
- 如果分支已存在，询问用户是否要覆盖
- 如果推送失败，展示错误信息并建议手动处理

## 完成

工作流完成后通知用户：
- BUG 修复已提交到 `bugfix/{jira_id}` 分支
- 远端推送成功
- 后续可以创建 PR 进行 code review
```

## 带条件分支的示例：BUG 分级修复

```markdown
---
name: bugfix-workflow
description: >
  根据 BUG 难度自动选择修复流程。支持简单、中等、复杂三个级别，
  每个级别有不同的处理流程。

  触发条件：
  - "修复 BUG" / "修 bug"
  - 提到 Jira ID 并要求修复
  - "使用 bugfix 流程"
---

# BUG 分级修复流程

## 输入参数

| 参数名 | 类型 | 必填 | 默认值 | 说明 | 示例 |
|--------|------|------|--------|------|------|
| jira_id | string | 是 | - | Jira Issue ID | PROJ-1234 |
| level | enum(simple/medium/complex) | 是 | - | BUG 难度级别 | simple |

## 流程分支

根据 level 参数选择流程：

### 如果 level == "simple"：

  步骤 1 [AUTO] → 查询 Jira
  步骤 2 [AUTO] → 直接修复代码
  步骤 3 [CONFIRM] → 用户验证
  步骤 4 [AUTO] → 提交推送到 bugfix/{jira_id}

### 如果 level == "medium"：

  步骤 1 [AUTO] → 查询 Jira
  步骤 2 [AUTO] → 分析问题，定位代码位置
  步骤 3 [CONFIRM] → 向用户展示分析结果和修复思路，等待确认
  步骤 4 [AUTO] → 用户确认后，按确认的思路修复
  步骤 5 [CONFIRM] → 用户验证修复结果
  步骤 6 [AUTO] → 提交推送到 bugfix/{jira_id}

### 如果 level == "complex"：

  步骤 1 [AUTO] → 查询 Jira
  步骤 2 [AUTO] → 深入分析问题，列出所有可能的原因
  步骤 3 [CONFIRM] → 向用户展示分析，讨论问题根因
  步骤 4 [INPUT] → 用户确认根因后，讨论修复方案
  步骤 5 [CONFIRM] → 用户确认修复方案
  步骤 6 [AUTO] → 按方案修复
  步骤 7 [CONFIRM] → 用户验证修复结果
  步骤 8 [AUTO] → 提交推送到 bugfix/{jira_id}

（每个分支的步骤需要在下方展开详细定义，格式同上面的单流程示例）
```
