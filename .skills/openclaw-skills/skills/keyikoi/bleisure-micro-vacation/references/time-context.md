# 时间与时长（禁止瞎猜出门钟点）

## 当前时刻从哪来

1. **直接用**：对话/系统上下文中 **已有** 当前日期或时间（例如 `Today's date`、`user_info` 里的日期时间戳、消息元数据）。**有就用，不要再问用户「现在几点」**。  
2. **用户自述**：用户打出「现在 X 点」「刚散会」等，以用户话为准，同样 **不要回头再确认**。  
3. **真的都没有**（极少见：上下文无任何时间、用户也没提）：改用 **相对时长**（「从出发起约 15 分钟到…」）并写明 **请对照你手机时钟**。仍然 **不要** 编造「假设 21:30」。  
   - 只在 **影响安全**（如赶飞机倒推到枢纽时间）时才 **问一句** 当前几点；普通散步/喝酒场景 **用相对时长即可，不问**。

## 硬约束

- **禁止**在已有系统时间的情况下还问「你那边几点」。这是多余的一步，让用户觉得你不读上下文。  
- **禁止**在无依据时写具体 HH:MM 时间线假装是用户的真实时刻。  
- 若使用了上下文时间，时间线 **第一行** 注明锚点，例如：`🕐 基于当前约 HH:MM（来自系统上下文）` 或 `🕐 你刚说现在 HH:MM`。  
- 跨时区差旅：若用户城市与系统时区 **明显不一致**（如系统显示 UTC 但用户在北京），可 **一句确认** 时区；同时区则 **不问**。

## 白天 vs 晚上：多问一句什么

按用户 **本地当前钟点** 粗分（边界可模糊，以「是否影响排期」为准）：

| 场景 | 建议补问（用户已说过则可省略） |
|------|--------------------------------|
| **白天**（大致 **06:00～17:59**） | **「大概能玩多久？」**（空闲窗口，如 1 小时 / 半天） |
| **晚间**（大致 **18:00～次日 05:59**）或用户明确说「晚上」 | **「明天要不要早起？」**（或「最晚几点要回酒店/要睡？」）；若已说赶飞机/早会，则 **不问** |

与 **硬截止**（`deadline-travel.md`）同时出现时：**早起/回酒店** 与 **航班高铁** 合并理解，不重复追问。

### 时间追问的时机（硬约束）

**禁止** 把时间追问和活动方向的澄清混在同一条消息里。这是两个独立决策，拼在一起用户不知道先答哪个。

- **第一轮（需求捕捉）**：只问活动方向。如果选项描述里已自然嵌入了时长信息（如「沿湖走一圈大概四十分钟」），用户选了就等于确认了时间，**后续不用再问**。
- **第二轮（用户选完活动后）**：如果仍需确认时间约束，再单独问一句。
- **可以省略追问的情况**：选项描述已含时长；用户主动说了时间；硬截止场景已包含时间信息。

## 与「总时长」的关系

- 用户答了「能玩多久」：动线 **必须**塞进该窗口，时间线钟点 **从锚点时刻往后加**。  
- 用户答了「要早起」：晚间喝酒/按摩 **主动缩短**，收束时间写清楚 **建议散场时间**。
