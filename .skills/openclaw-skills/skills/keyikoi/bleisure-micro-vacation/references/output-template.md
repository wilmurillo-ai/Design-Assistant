# 输出模板（差旅晚间微度假）

输出时 **严格保持以下顺序**。全文控制在 **短消息体量**（约等于一条长微信 + 一张头图），不要扩展成长文攻略。

## 全局语气（贯穿所有区块）

严格遵守 `SKILL.md` 中「情绪价值与人设」硬约束。简记：
- **像朋友发微信**，不像客服汇报。用「你」不用「您」。
- 推荐讲 **感受**（「走过去刚好够你从工作状态里切出来」），弱化 **数据**（评分、距离数字可留但不当主角）。
- 时间线 **故意松**，用「觉得累了就」「什么时候想回就回」放权。
- 结尾 **推出门**，不总结信息。

---

## 0. 硬截止时优先（可选块）

若本次为 **航班/高铁/末班** 等硬截止场景，**先**输出此块，**再**接第 1 节氛围图：

```markdown
### ⏱ 时间底线（先看完再逛）

- **建议到枢纽：**不晚于 …（假设：…）
- **路上预留：**约 …（交通方式：…）
- **建议最晚从游玩点动身：**约 …
- **说明：**以票面、航司/车站当日规则与实时导航为准。
```

## 1. 氛围头图（最高优先级）

在标题前或紧跟标题后，插入 **至少一张** Markdown 图片：

```markdown
![一句话氛围描述，如：亮马河夜景](https://可嵌入的HTTPS图片URL)
```

- 无合规 URL 时，改用：

```markdown
**氛围参考**：在小红书搜索 `关键词1 关键词2`，选近期笔记封面作感觉参考（链接以 App 内为准）。
```

## 2. 共情开场 + 标题（短，带温度，时段驱动）

开场第一句 **必须** 按 `SKILL.md` 中「时间段情绪地图」写：取当前时段的情绪内核 + 城市/地标特征，**禁止照抄模板示例**。如果命中「叠加层」（连续出差/周末/节假日等），把叠加句自然嵌进开场里，不要变两段。

然后紧跟一个 **有画面感的标题**。

```markdown
[共情开场：1 句，按时间段情绪地图 + 城市特征写]

### 🌙 [有画面的标题，不是汇报腔]

📍 **推荐去：**[地点昵称]  
离你约 **X 分钟**（[步行/打车，说明起点]）
```

**好的开场 + 标题示例**（不同时段）：

**黄金档 17:30–19:30**：
> 最好的时间到了——白天是出差，从现在起是旅行。
> ### 🌉 悦目桥上被星星包住

**还没睡 22:00–00:00**：
> 还没困？出差的夜特别长。与其躺着刷手机，不如出去走一圈。
> ### 🌙 玄武湖边灯光打在水面上

**偷来的早晨 06:00–09:00**：
> 这个城市刚醒，你也是。趁还没进会议室，先出去走走。
> ### ☀️ 晨光里的断桥只有环卫和你

**一个人的午饭 11:30–14:00**：
> 又是一个人吃午饭？别将就了。
> ### 🍜 这家店值得专门走一趟

**不要写的开场**：
- ❌ 没有共情直接上标题：「### 推荐方案：前滩太古里」
- ❌ 用客服腔开场：「根据您的位置，为您推荐以下方案」
- ❌ 开场和时段无关：大半夜写「美好的一天从这里开始」

## 3. 安全（一行即可，硬约束）

```markdown
**安全提示：**夜间尽量走照明好、人多的公开路段；避开无灯小巷与偏僻区域，以现场为准。
```

（若动线全程商场/平台室内，可改写成半句，但不可整段省略。）

---

## 4. 为什么适合现在（最多 3 条，每条一行，讲感受不讲参数）

每一条的写法：**把信息翻译成「这跟你当下的感受有什么关系」**。

```markdown
**为什么适合现在：**

- …
- …
- …
```

**好的写法**：
- 这里晚上灯光很好人不多，适合一个人不带目的地走
- 走一圈刚好用掉你的这段时间
- 这个点去刚好，再晚就关了，像是专门给你留的

**不要写的**：
- ❌ 该景点 TripAdvisor 评分 4.8
- ❌ 距离您 800 米，步行约 12 分钟
- ❌ 该区域夜间治安良好（这是安全提示该做的事，不是推荐理由）

---

## 5. 时间线节奏（第二优先级，故事感 > 行程表）

**锚点规则**：第一行必须交代时间从哪来，例如 `🕐 从当前约 HH:MM 起（来源：对话上下文 / 你刚说的现在 X 点）`。**禁止**无依据写「假设 21:30 出门」。无可靠时刻时改用 **相对时长**（出发后约 15 分钟…）并提示用户对照本地时钟。详见 `references/time-context.md`。

**语气要求**：时间线是「你的小冒险」，不是 schedule。出差人日程已经够紧了，微度假的时间线 **要故意松**。用「觉得累了就」「什么时候想回就回」「不用导航，迷路也没关系」这种放权语气。

```markdown
**怎么玩：**

🕐 从当前约 HH:MM 起（来源：…）
🚶 HH:MM 先往 X 方向走，[一句描述，不要只写地名]
🌉 HH:MM [到了之后做什么，用松弛的语气]
☕ 走到觉得累了就找个店坐下
🔙 什么时候想回去了就回，不用卡点
```

**好的时间线写法**：
- 🚶 先往东走，穿过胡同，不用导航，跟着灯光走就行
- ☕ 走累了就近找个店坐一会儿，这条街上选择多
- 🔙 什么时候想回去了原路折返，不赶

**不要写的时间线**：
- ❌ 19:30 出发 → 19:42 到达景点 A → 19:55 拍照 → 20:08 步行至景点 B（精确到分钟的行程表）

```markdown
（可加半句：**不赶**，或「可按体力砍掉 ☕ 段」）
```

## 7. 高德路线（可点击，强制）

在「总时长」与「小红书」之间插入，**至少一条**官方 URI（步行优先）；**硬截止**时另加 **赴枢纽** 的 `car` 或 `bus` 链。

```markdown
**🗺 高德路线（点开即规划，以 App 内为准）：**

- [步行：起点简称 → 终点简称](https://uri.amap.com/navigation?from=经度%2C纬度%2C起点名编码&to=经度%2C纬度%2C终点名编码&mode=walk&src=travel-evening-micro-vacation&callnative=1)

**坐标来源：**联网检索 POI「…」与「…」的近似 GCJ-02；若与你位置不符，请在 App 内拖动起点改为酒店/当前位置。
```

- 多段动线（步行无法单链表达）：最多 **2～3 条**，标注「第一段 / 第二段」。
- 附一句：**以高德实际路线为准**；微信内点不开 App 时试「浏览器打开」。

拼参、GCJ-02、`mode` 取值见：`references/amap-navigation-uri.md`。

## 8. 结尾收尾（推出门，不总结）

在小红书区块 **之后**，用 **一句话** 收尾。这句话的作用是 **推他出门**，不是总结信息。

```markdown
[一句收尾的话]
```

**好的收尾**：
- 去吧，明天的事明天再说。
- 45 分钟也够了，出去一趟心情就不一样。
- 一个人走走也挺好的，不用跟谁交代。
- 拍张照回来，下次你同事出差可以推荐给他。
- 别在酒店待着了，出去看看这个城市的晚上。

**不要写的收尾**：
- ❌ 祝您旅途愉快！
- ❌ 希望以上推荐对您有帮助。
- ❌ 如有其他需求请随时告诉我。

## 9. 小红书（真实链接优先，关键词兜底）

本区块有两种模式，取决于 CDP 是否可用（详见 `references/xiaohongshu-cdp.md`）。

### 模式 A：CDP 可用 → 真实高赞链接

按 `xiaohongshu-cdp.md` 流程抓取后，展示点赞最高的 1～3 条笔记：

```markdown
**小红书推荐笔记：**

- [笔记标题](https://www.xiaohongshu.com/...) · ❤️ 1.2万
- [笔记标题](https://www.xiaohongshu.com/...) · ❤️ 3860

链接可能失效，也可以在 App 里搜 `关键词` 看更多。
```

要点：
- 按 **点赞数降序**，取 Top 1～3
- 点赞数保留原始格式（如 `1.2万`），不自己换算
- 仍附一个搜索关键词作为兜底

### 模式 B：CDP 不可用 → 关键词模式（静默回退）

不提 CDP 失败，自然地只给关键词：

```markdown
**小红书怎么搜：**

- `关键词 A`
- `关键词 B`
```

## 自检清单

### 情绪与语气
- [ ] 全文用「你」不用「您」，无客服话术（祝您愉快、希望有帮助等）
- [ ] 开场第一句 **按「时间段情绪地图」写**：已对照当前时段的情绪内核，且结合了城市/地标特征，不是照抄模板
- [ ] 若命中叠加层（连续出差/周末/节假日），叠加句已 **自然嵌入** 开场，未单独成段
- [ ] 推荐理由讲的是 **感受**，不是堆参数
- [ ] 时间线语气 **松弛放权**（觉得累了就、什么时候想回就回），不是精确到分钟的行程表
- [ ] 结尾是 **推出门的一句话**，不是总结或客套

### 需求捕捉
- [ ] 若用户 **仅给位置、未说想干嘛**，已按 `references/intent-triage.md` 用 **4 选项问法** 澄清，未直接套全文模板
- [ ] **出选项前已先查起点**（高德 POI / WebSearch），确认 A/B/C 推荐的地点 **确实在起点附近**（步行 ≤15 分钟或打车 ≤10 分钟），未凭城市印象编选项
- [ ] A/B/C **都通过「换城市名」测试**——没有「出去走走」「找个地方坐一会儿」之类万能废话，都具体到 **街名/店名/地标**
- [ ] **D = 自由输入**（引导用户说想干嘛），不是「不出去/算了/睡觉」
- [ ] 选项维度覆盖合理（动/坐/吃），并根据时段做了调整
- [ ] **每轮只问一件事**——活动方向和时间约束 **没有** 拼在同一条消息里
- [ ] 老用户有 log 偏好时，开场 **一句带过**，选项排序已微调

### 功能与格式
- [ ] 若为硬截止，**时间底线块**已置于动线前（见 `references/deadline-travel.md`）
- [ ] 头图或替代指引已给出
- [ ] **安全提示**已给出（至少一行）
- [ ] 时间线锚点来自 **上下文当前时间** 或 **用户自述**，非虚构；白天已问 **能玩多久** / 晚上已问 **早起或回酒店**（除非用户已声明或硬截止已覆盖）
- [ ] 时间线钟点闭合、总时长与空档匹配
- [ ] **高德 `uri.amap.com/navigation` 链接已给出**（步行为主，GCJ-02，**已写坐标来源**）；有 API Key 时起点终点坐标 **来自高德 POI 搜索**（非 WebSearch 猜测）
- [ ] 无大段说明文、无过多 bullet
- [ ] **小红书链接已自动抓取**：CDP 可用时 **必须** 附真实高赞链接（标题 + 点赞数），**不可跳过**；仅 CDP 确认不可用时静默回退为关键词
- [ ] **`standups.md` 已追加**本轮 standup，或已勾选 **跳过：原因**（用户拒记 / 未生成动线）
