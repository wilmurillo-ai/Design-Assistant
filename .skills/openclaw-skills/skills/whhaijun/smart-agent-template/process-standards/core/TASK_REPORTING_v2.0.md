# 任务汇报规范 v2.0

> 所有人必须遵守，不可跳过
> 最后更新：2026-03-28
> 变更：新增强制汇报检查点、汇报质量检查

---

## 🎯 核心原则

**有进展就汇报，卡住就上报，完成就汇总**

---

## 📋 强制汇报检查点（不可跳过）

### 5 个必须汇报的时机

| 时机 | 触发条件 | 汇报方式 |
|------|---------|---------|
| **任务开始** | 收到任务后 | 输出拆分方案 |
| **每批次完成** | 每个批次/断点 | 输出阶段结果 |
| **遇到问题** | 失败≥2次 / 等待>30秒 | 立即上报 |
| **需要决策** | 需求不明确 / 多个方向 | 立即上报 |
| **任务完成** | 所有工作完成 | 输出最终结果 |

---

## 📝 汇报格式模板

### 1. 任务开始汇报

```markdown
## 任务拆分方案

**任务名称：** [任务名]
**触发原因：** [为什么拆]

**依赖关系：**
  A // B         → 并行
  A → B          → 串行
  A → (B // C)   → A 完成后，B 和 C 并行

**批次安排：**
  批次1（并行，最多3个）：A / B / C
  批次2（依赖批次1）：D

**健康度：** 🟢 On Track / 🟡 At Risk / 🔴 Off Track

**预计时间：** [总时长]

**风险预案：**
  - [风险] 可能性：H/M/L × 影响：H/M/L → 应对：[措施]
```

### 2. 批次完成汇报

```markdown
## 批次X完成汇报

**健康度：** 🟢 / 🟡 / 🔴

**完成：**
- [任务名]：[结果+关键数据]（不要只说「完成了」）
- [任务名]：[结果+关键数据]

**风险/阻塞：**
- [描述] — 可能性：M，影响：H → [应对]

**下一步：**
- [行动] / 负责人：[谁] / 预计：[时间]
```

### 3. 问题上报

```markdown
## ⚠️ 问题上报

**任务ID：** [task_id]
**问题描述：** [具体问题]
**失败原因：** [根因分析]
**已尝试：** [尝试过的方法]
**需要：** [需要什么帮助]

**影响：**
- 阻塞任务：[哪些任务]
- 预计延误：[时间]
```

### 4. 决策请求

```markdown
## 🚨 需要决策

**任务ID：** [task_id]
**情况：** [简述]
**需要你决定：** [具体问题]

**选项：**
- [A] [选项A] — 优点：[...] / 缺点：[...]
- [B] [选项B] — 优点：[...] / 缺点：[...]

**我的建议：** [A/B] — 理由：[...]
```

### 5. 任务完成汇报

```markdown
## ✅ 任务完成

**任务ID：** [task_id]
**目标：** [原始需求]
**结果：** [一句话结论]

**产出：**
- [文件路径1]
- [文件路径2]

**关键发现：**
- [重要发现1]
- [重要发现2]

**需要 Mark 确认的：**
- [真正需要决策的事项，没有则写"无"]

**下一步建议：** [可选]
```

## ✅ 汇报质量检查

### 每次汇报前，自检以下 5 点：

```markdown
- [ ] 是否包含关键数据？（不能只说"完成了"）
- [ ] 是否说明下一步？（不能只汇报结果）
- [ ] 是否标注健康度？（让用户快速了解状态）
- [ ] 是否简洁明了？（2 分钟内读完）
- [ ] 是否使用标准格式？（便于解析和追踪）
```

### 不合格示例 vs 合格示例

**❌ 不合格：**
```
完成了代码开发。
```

**✅ 合格：**
```
## 批次1完成汇报

**健康度：** 🟢 On Track

**完成：**
- 转账确认页开发：新增手续费显示，格式 `SAR X.XX`，位于总金额下方
- 文件：TQTransferConfirmVC.m（+120行）

**风险/阻塞：** 无

**下一步：**
- 规范扫描 / 负责人：Lily / 预计：30分钟
```

## 🚨 禁止行为

### 绝对不允许

- ❌ 任务卡住超过 10 分钟无任何反馈
- ❌ 失败后默默重试，不说原因
- ❌ 等待用户主动来问进度
- ❌ 只说"完成了"，不说具体结果
- ❌ 跳过中间汇报，只在最后汇总

### 必须做到

- ✅ 有进展就汇报（每批次完成）
- ✅ 卡住就上报（失败≥2次 / 等待>30秒）
- ✅ 完成就汇总（使用标准格式）
- ✅ 包含关键数据（结果+数据+下一步）
- ✅ 标注健康度（🟢/🟡/🔴）

## 📊 汇报时机决策树

```
收到任务
    ↓
输出拆分方案 ✅
    ↓
开始执行
    ↓
完成一个批次？
    ├─ 是 → 输出批次汇报 ✅
    └─ 否 → 继续执行
    ↓
遇到问题？
    ├─ 失败≥2次 → 立即上报 ✅
    ├─ 等待>30秒 → 立即上报 ✅
    └─ 否 → 继续执行
    ↓
需要决策？
    ├─ 是 → 立即上报 ✅
    └─ 否 → 继续执行
    ↓
所有批次完成？
    ├─ 是 → 输出最终汇报 ✅
    └─ 否 → 回到"完成一个批次？"
```

## 🎯 实战示例

### 示例1：正常流程

**任务：** 开发转账确认页

**汇报1：任务开始**
```markdown
## 任务拆分方案

**任务名称：** 开发转账确认页
**触发原因：** 涉及 3 个文件，预计 2 小时

**批次安排：**
  批次1：UI 布局 + 数据绑定
  批次2：手续费计算逻辑
  批次3：确认按钮交互

**健康度：** 🟢 On Track
**预计时间：** 2 小时
```

**汇报2：批次1完成**
```markdown
## 批次1完成汇报

**健康度：** 🟢 On Track

**完成：**
- UI 布局：新增手续费显示区域，位于总金额下方
- 数据绑定：金额、手续费、总计三个字段

**风险/阻塞：** 无

**下一步：**
- 批次2：手续费计算逻辑 / 预计：30分钟
```

**汇报3：任务完成**
```markdown
## ✅ 任务完成

**任务ID：** TQ-20260328-001
**目标：** 开发转账确认页
**结果：** 完成 UI 布局、数据绑定、手续费计算、确认交互

**产出：**
- TQTransferConfirmVC.h

- TQTransferConfirmView.m

**关键发现：**
- 手续费计算需要调用后端接口，已对接完成

**需要 Mark 确认的：** 无

**下一步建议：** 规范扫描 + 功能测试
```

### 示例2：遇到问题

**汇报：问题上报**
```markdown
## ⚠️ 问题上报

**任务ID：** TQ-20260328-001
**问题描述：** 手续费接口调用失败
**失败原因：** 接口返回 500 错误
**已尝试：**

**影响：**
- 阻塞任务：批次2（手续费计算逻辑）
- 预计延误：1 小时
```

## 📈 汇报质量评分

### 评分标准（满分 10 分）

| 项目 | 分值 | 说明 |
|------|------|------|
| 使用标准格式 | 2 分 | 使用模板，便于解析 |
| 包含关键数据 | 2 分 | 结果+数据，不只说"完成了" |
| 标注健康度 | 2 分 | 🟢/🟡/🔴，快速了解状态 |
| 说明下一步 | 2 分 | 明确后续行动 |
| 简洁明了 | 2 分 | 2 分钟内读完 |

### 质量等级

- **优秀（9-10分）：** 所有项目都做到
- **良好（7-8分）：** 缺少 1-2 项
- **及格（5-6分）：** 缺少 3 项
- **不及格（<5分）：** 缺少 4 项以上
