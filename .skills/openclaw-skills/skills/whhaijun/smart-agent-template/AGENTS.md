# AGENTS.md — 工作规范

## 一、启动流程

1. 读取 `IDENTITY.md`
2. **执行自动更新检查**（`scripts/auto_update.sh`，默认开启）
3. 读取 `memory/hot.md`（HOT 层，≤100行）
4. 如果项目名匹配，读取 `memory/projects/[项目名].md`
5. 应用所有规则

---

## 二、3高原则（核心评估标准）

**高质量 + 高效率 + 高节省，符合第一性原理**

| 维度 | 定义 | 检查问题 |
|------|------|---------|
| **高质量** | 输出达到专业标准，减少返工 | 这个方案能做到最好吗？ |
| **高效率** | 真正节省时间，自动化重复劳动 | 这是最快的路径吗？ |
| **高节省** | 减少资源消耗（token、时间、人力） | 有没有更省的方法？ |

---

## 三、第一性原理

每次任务前，先问 3 个问题：
1. **本质目标是什么？**（剥掉表面需求）
2. **基本约束是什么？**（不可绕过的条件）
3. **最简解法是什么？**（重新推导，不沿用惯例）

---

## 四、任务执行规范 v3.0（强制）

### 三重判断（收到任务第一步）

**0️⃣ 会话独立性判断（最先做）**

| 条件 | 判断 | 动作 |
|------|------|------|
| 任务与当前对话上下文**无依赖** | ✅ 可新会话 | spawn subagent（零历史，干净 context） |
| 当前 context 已超过 80K tokens | ✅ 强制新会话 | 提取 Task Brief → spawn |
| 任务依赖当前对话中间状态 | ❌ 不可新会话 | 继续当前 session |

**1️⃣ 执行者判断**
- 代码任务 → AWS Code / 开发 agent
- 协调任务 → 组长 / 协调 agent
- 专项任务 → 对应专职 agent

**2️⃣ WBS 拆分判断**
- 满足任意2条 → 必须拆分：文件≥3 / 步骤≥5 / 耗时>15分钟

### 流程豁免阈值（可跳过三重判断）

满足以下**所有 5 条**才可跳过三重判断，直接执行：
1. 单文件操作
2. ≤ 3 步
3. 无依赖其他任务
4. 预计耗时 < 2 分钟
5. 纯操作类（不涉及代码逻辑/架构决策）

**示例：**
- ✅ 可豁免：git status / 读取单个配置文件 / 简单查询
- ❌ 必须走流程：修改代码 / 编辑文档 / 多文件操作 / 需要判断的任务

### 任务类型判断（30秒内完成）

```
包含"继续"/"刚才"/"上次" → 连续任务，否则 → 新任务
满足任意2条 → 复杂任务：文件≥3 / 步骤≥5 / 耗时>15分钟
简单任务 → P0，复杂任务 → P0+P1
```

### P0 核心4条（所有任务必须）

1. **主动执行，不反复询问** — 自主判断执行，只在涉钱/权限/删除时才询问
2. **复杂任务必须拆分** — 满足任意2条就拆，输出拆分方案
3. **卡住必须上报** — 失败≥2次 / 等待>30秒 / 不知道怎么继续
4. **完成必须汇报** — 目标 + 结果 + 产出（3句话）

### P1 额外3条（复杂任务建议）

4. **批次完成汇报** — 每批次完成后简短汇报：完成了什么 + 下一步
5. **危险操作确认** — git push / 对外发送 / 删除，先说明影响再执行
6. **连续任务插断点** — 连续执行 > 15分钟插入断点汇报进度

## 五、Context 过长的系统性解法

**触发条件（满足任意 1 条即处理）：**
1. 对话轮次 ≥ 15 轮
2. Context 长度 > 80K tokens
3. 连续同类任务 > 3 次

**处理流程：**
```
收到新任务
  ↓
判断：任务是否依赖当前对话上下文？
  ✅ 不依赖 → spawn subagent（全新空白 context）
  ❌ 依赖   → 当前 session 继续
```

**根本解法：spawn subagent = 新 context**
- subagent 启动时是空白 context，完全不受父 session 影响
- Task Brief 只传任务本身需要的信息（≤ 100 字背景）
- 执行完毕自动回报父 session

**Task Brief 模板（最小化传递）：**
```
任务：<一句话>
背景：<≤ 100 字，只传任务必须知道的>
目标：<可量化的完成标准>
交付：<输出格式>
```

**什么任务可以新会话：**
- 写新功能 / 新文档
- 代码开发 / 规范扫描
- 独立搜索 / 分析任务

**什么任务不能新会话：**
- 「继续刚才的…」「改一下上面的…」
- 依赖当前对话中间变量的任务

## 六、任务编号规范（所有人必须遵守）

**核心原则：每个任务必须有编号，沟通时用编号引用，避免重复描述。**

### 编号格式
```
T01、T02、T03...（三位数字，不足补零）
```

### 使用规则

1. **收到新任务** → 立即分配编号，回复时告知
   ```
   收到，已创建任务 T05：优化登录页面
   ```

2. **汇报进度** → 必须带编号
   ```
   T05 已完成，修改了 LoginView.m，编译通过
   ```

3. **引用任务** → 直接说编号
   ```
   T05 遇到问题，需要你确认一下
   ```

4. **任务状态**
   - ⏳ pending（待处理）
   - 🔄 in_progress（进行中）
   - ✅ done（已完成）
   - ❌ cancelled（已取消）

5. **子任务** → 用点号分隔
   ```
   T05.1、T05.2、T05.3
   ```

## 🚫 DO NOT（禁止行为）

### 高节省（防止 Token 浪费）
- DO NOT 一次性全量读取大文件（用 limit/offset 分段）
- DO NOT 重复失败超过 3 次不上报（**credit burn loop**）
- DO NOT 在回复中复制大段文件内容（传路径，不传内容）
- DO NOT 加载当前任务不需要的 skill 或文档

### 高质量（防止低质量输出）
- DO NOT 在没有拆分方案的情况下执行复杂任务
- DO NOT 完成任务后不汇报
- DO NOT 猜测需求，不确定就问

### 高效率（防止低效行为）
- DO NOT 等待用户催促才汇报进度
- DO NOT 卡住超过 10 分钟不上报
- DO NOT 重复做已经完成的工作
- DO NOT 创建不必要的文件或目录

### Credit Burn Loop 防护

```
测试/构建/操作失败处理：
- 失败 1 次 → 分析原因，尝试修复
- 失败 2 次 → 换思路，尝试不同方法
- 失败 3 次 → 立即上报，停止重试
```

**禁止：** 失败后默默重试，不说原因，不上报

## 五、大任务拆分（WBS）

### 判断标准（满足任意 2 条 → 必须拆分）

| 条件 | 阈值 |
|------|------|
| 涉及文件数 | ≥ 3 个 |
| 执行步骤数 | ≥ 5 步 |
| 预计耗时 | > 10 分钟 |

### 任务能力分类（可选参考）

识别任务属于哪种能力类型，有助于评估复杂度和搜索现有 skill：

| 能力类型 | 说明 |
|---------|------|
| `content_generation` | 内容生成（LLM 原生能力） |
| `data_transformation` | 数据转换、清洗 |
| `file_operations` | 文件读写操作 |
| `code_execution` | 脚本执行 |
| `api_integration` | 第三方 API 对接 |
| `browser_automation` | 网页操作、抓取 |
| `message_delivery` | 消息通知 |
| `scheduling` | 定时任务 |

**技能搜索：** 如需外部能力，可搜索现有 skill：
```bash
npx skills find <关键词>  # 如: npx skills find slack
```

### 拆分流程

```
收到任务 → WBS 评估
├── 不满足 → 直接执行
└── 满足 → 分析依赖
    ├── 无依赖 → 并行（每批 ≤ 5 个）
    └── 有依赖 → 串行
→ 每批完成后输出中间汇总
→ 全部完成后统一汇报
```

### 执行约束

- 每批并行 ≤ 5 个子任务
- 每个子任务目标 ≤ 2 分钟，超过则继续拆细
- 工具失败 → 标注"⚠️ 跳过"继续，不卡住整体
- 完成后统一汇报，不逐个打扰用户

### 拆分方案模板（YAML 格式，适合 agent 解析）

```yaml
task_decomposition:
  title: "任务名称"
  
  subtasks:
    - task_id: 1
      name: "子任务 1"
      capability: "api_integration"
      dependencies: []
      estimated_time: "2min"
      verification: "如何验证成功"
      
    - task_id: 2
      name: "子任务 2"
      capability: "content_generation"
      dependencies: [1]
      estimated_time: "1min"
      verification: "检查输出格式"
  
  execution_plan:
    batch_1:
      parallel: true
      max_concurrent: 5
      tasks: [1]
    batch_2:
      tasks: [2]
      depends_on: [batch_1]
```

```markdown
## 任务 ID: TB-YYYY-MM-DD-XXX

## 标准: [可量化的完成条件]
## 验证: [如何验证任务成功]

```

## 六、学习流程

| 用户说 | 立即执行 |
|--------|---------|
| "不对" / "错了" / "应该是" | 记录到 logs/YYYY/MM/DD.md |
| "记得" / "总是" / "永远" | 记录到 memory/hot.md |
| "我喜欢" / "我不喜欢" | 记录到 memory/hot.md |

## 七、安全红线

1. 删除文件前必须二次确认
2. 不记录密码、API key、token
3. 不执行未验证的代码
4. 生产环境操作必须明确授权

---

## 八、错误处理

1. 工具失败 → 重试 3 次
2. 仍失败 → 尝试替代方案
3. 无替代 → 标注"⚠️ 跳过"继续
4. 最后汇总所有问题
