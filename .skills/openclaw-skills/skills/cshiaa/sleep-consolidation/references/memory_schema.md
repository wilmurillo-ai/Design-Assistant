# Memory Schema Reference

Type system, conventions, and design rationale for sleep-consolidation v2.

---

## Design principles

1. **Files are the source of truth** (OpenClaw): the agent only "remembers" what gets written to disk
2. **Plain Markdown** — human-readable, Git-trackable, editable in any text editor
3. **Two layers**: `MEMORY.md` (curated, durable) + `memory/YYYY-MM-DD.md` (append-only daily log)
4. **Dual-track NREM** (Eichenlaub 2020): fast-track for confident items → `MEMORY.md`; slow-track for weak items → `bank/`
5. **Waking replay** (Eichenlaub 2020): micro-rest consolidates even during active sessions, not just sleep

---

## Type system (W / B / O / S)

Derived from OpenClaw workspace research and Hindsight framework:

### W — World fact
Objective, external, not agent-specific.
```
- W: Redis default port is 6379
- W: OpenClaw workspace defaults to ~/.openclaw/workspace
- W: Python's GIL prevents true multi-threaded CPU parallelism
```

### B — Biographical / experience
What the agent (or its user) did, encountered, or observed.
```
- B: Fixed auth bug by wrapping connection.update in try/catch (2025-01-14)
- B: Deployed to staging using Docker Compose; build took 4 minutes
```

### O(c=N) — Opinion / preference
Subjective judgment or preference, with a confidence value 0–1.
```
- O(c=0.92): User prefers TypeScript; rejected Python suggestion twice
- O(c=0.70): Team may migrate from Jest to Vitest — mentioned in two sessions
- O(c=0.40): User might prefer dark mode (only mentioned once, unconfirmed)
```

Update `c` as evidence accumulates. REM phase updates confidence values automatically.

### S — Summary
Synthesized observation across multiple events. Usually generated by NREM/REM, not directly observed.
```
- S: All three latency spikes occurred when memory store exceeded 10k entries
- S: User's preference for brevity + TypeScript suggests typed, concise coding background
```

---

## Confidence levels

| Range | Interpretation |
|-------|----------------|
| 0.9 – 1.0 | Confirmed, high evidence |
| 0.7 – 0.9 | Likely, moderate evidence |
| 0.5 – 0.7 | Tentative, limited evidence |
| < 0.5 | Uncertain — flag for slow-track attention |

Items with `c < 0.5` should go to the slow track (bank/) rather than MEMORY.md.

---

## NREM two-track routing

| Track | Timescale analog | Destination | Criteria |
|-------|-----------------|-------------|----------|
| Fast | Compressed (0.1x) | `MEMORY.md` | confidence ≥ 0.7, clear single fact |
| Slow | Dilated (1.5–2x) | `bank/` | confidence < 0.7, complex topic, recurring theme |

Based on Eichenlaub et al. 2020: replay duration is inversely proportional to encoding strength. Weakly-learned items receive more replay time — the slow track simulates this.

---

## File size guidelines

| File | Recommended size | Auto-trimmed? |
|------|-----------------|---------------|
| `MEMORY.md` | < 6000 chars (~1500 tokens) | No — run `--prune-memory` |
| Daily log | < 2000 chars per day | No — old logs not loaded |
| Bank entity/concept | < 3000 chars each | No |
| Loaded context at session start | < 4000 chars total | Yes — `max_memory_chars` param |

---

## Retention policy

- **Daily logs**: kept indefinitely on disk; only today + yesterday loaded automatically
- **MEMORY.md**: curated manually or via `--prune-memory`; remove superseded entries
- **Bank pages**: kept indefinitely; update in-place via NREM `bank_updates`

**Do not** store transient context (current task state, in-progress code) in Markdown files. That belongs in RAM.
