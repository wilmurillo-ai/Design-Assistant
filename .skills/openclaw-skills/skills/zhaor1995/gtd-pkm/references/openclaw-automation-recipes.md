# OpenClaw 自动化配方

这个配方描述的是一套适合 GTD 知识系统的 OpenClaw 工作协议。

## 1. 工作区文件职责

推荐在 `~/.openclaw/workspace` 明确这些文件：

| 文件 | 用途 |
| --- | --- |
| `AGENTS.md` | 代理每次会话必须遵守的工作协议 |
| `USER.md` | 对用户的长期认知 |
| `MEMORY.md` | 长期记忆与关键决策 |
| `HEARTBEAT.md` | 心跳轮询时要执行的清单 |
| `TOOLS.md` | 本地工具、账号、设备、特殊约束 |
| `TODO/YYYY-MM-DD.md` | 短周期行动项 |

## 2. AGENTS.md 应该约束什么

建议让 `AGENTS.md` 至少覆盖：

1. 每次会话先读哪些文件
2. 什么信息写入 `memory/YYYY-MM-DD.md`
3. 什么信息沉淀进 `MEMORY.md`
4. 什么情况下主动写 TODO
5. Git 操作规范
6. 知识库目录规则
7. 什么时候使用 heartbeat，什么时候使用 cron

## 3. HEARTBEAT.md 应该做什么

心跳适合批量轻量检查，不适合重型任务。

推荐清单：

1. 回看今天的最近对话和上下文
2. 检查今天的 TODO 是否有逾期事项
3. 看 `00-收集箱` 是否需要清空
4. 看 `99-资源库/待阅读队列.md` 是否需要提醒
5. 看失败的 cron 任务是否要重试
6. 判断是否需要主动提醒用户
7. 周期性地整理 `MEMORY.md`

## 4. 推荐 cron 设计

### 每日固定任务

| 时间 | 任务 |
| --- | --- |
| `06:00` | 生成前一天日志 |
| `09:00` | 资讯或待阅读队列提醒 |
| `12:00` 或 `18:00` | 工作节奏/待办回顾 |
| `23:00` | 当日收尾提醒 |
| `00:00` | 自我反思与系统进化 |
| `00:30` | 配置备份或知识库健康检查 |

### 每周任务

| 时间 | 任务 |
| --- | --- |
| 周日晚上 | 周回顾、项目盘点、归档建议 |

## 5. 任务设计建议

### 日志生成

输入来源优先级：

1. 主会话真实聊天记录
2. `memory/YYYY-MM-DD.md`
3. 当天 TODO 和项目笔记

日志应该以用户视角记录，不要把 AI 的猜测写成用户想法。

### 配置备份

优先只保留一种机制：

- 方案 A：OpenClaw cron 负责备份
- 方案 B：系统 crontab 负责纯 shell 备份

不要两套同时做同一件事。

### 知识库健康检查

适合做这些：

- 目录完整性
- 命名规范
- 标签统计
- 错位文件识别
- 长期未更新的项目提示

## 6. 推荐的 GTD 自动化闭环

### Capture

- 用户发链接、碎片想法、临时任务时，先落到 `00-收集箱` 或 `99-资源库`

### Clarify

- 每天至少一次，把收集箱内容分类

### Organize

- 把可执行事项写入 `TODO`
- 把项目材料移入 `10-工作` 或 `20-项目`
- 把学习结果沉淀到 `40-学习`

### Reflect

- 每日写日志
- 每周回顾项目与行动
- 每月收敛标签与归档

### Engage

- 从 `40-学习` 和 `20-项目` 中抽取可输出内容到 `50-知识输出`

## 7. 避坑清单

- 不要同时用系统 crontab 和 OpenClaw cron 备份同一批文件
- 不要让知识库检查任务因为投递目标格式错误而长期失败
- 不要把运行脚本和知识正文混在一个 Git 仓库，除非用户明确接受
- 不要在 heartbeat 里塞太多重任务，避免 token 和噪音
- 不要让自动化只会“提醒”，要让它具备最小闭环能力，比如生成报告、更新 TODO、记录失败原因
