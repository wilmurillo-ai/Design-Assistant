# OpenClaw 可迁移模式

这份文档描述的是适合多数 OpenClaw 知识系统的通用模式，不依赖某一台机器、某一个人格设定或某一个知识库结构。

把它当成“设计原则库”，而不是“现成环境的复刻说明”。

## 1. 工作区的最小协议

无论具体人设如何，长期运行的 OpenClaw 最好都把工作区拆成几层：

- `AGENTS.md`：会话启动规则和工作原则
- `USER.md`：对用户的长期、稳定认知
- `MEMORY.md`：需要跨天保留的重要结论
- `memory/YYYY-MM-DD.md`：日常上下文和当天记录
- `HEARTBEAT.md`：定时轮询时的轻量检查清单
- `TODO/`：短周期行动项

### 为什么这样拆

- `AGENTS.md` 负责“怎么工作”
- `USER.md` 负责“为谁工作”
- `MEMORY.md` 负责“什么值得长期记住”
- `memory/` 负责“今天发生了什么”
- `TODO/` 负责“接下来要做什么”

如果这些职责混在一个文件里，代理会越来越难维护。

## 2. 记忆的分层模式

推荐至少区分两层：

### 日常层

- 当天聊天要点
- 临时约定
- 还没完全验证的观察

适合写进 `memory/YYYY-MM-DD.md`。

### 长期层

- 用户稳定偏好
- 多次验证后的工作原则
- 项目中的关键决策
- 以后还会继续影响判断的经验

适合写进 `MEMORY.md`。

### 常见错误

- 把流水账全塞进长期记忆
- 把长期偏好只写在当天日志里
- 以为“记住了”就不用写文件

## 3. Heartbeat 与 cron 的分工

这是 OpenClaw 自动化里最重要的边界之一。

### Heartbeat 适合做什么

- 批量查看待办、收集箱、失败任务
- 回看最近对话上下文
- 判断是否需要主动提醒
- 轻量维护 `USER.md` 或 `MEMORY.md`

### cron 适合做什么

- 固定时间运行的总结类任务
- 固定时间的提醒或回顾
- 周期性生成报告
- 独立执行、不依赖主会话上下文的工作

### 不建议的做法

- 把重型整理任务塞进 heartbeat
- 把同一件事同时放进 heartbeat 和 cron
- 用 cron 替代所有主动判断

## 4. 自动化任务的四个类别

一个健康的 OpenClaw 知识系统，自动化通常分成这四类：

### 节奏提醒

例子：

- 每日回顾
- 每周回顾
- 待阅读提醒
- 项目推进提醒

### 知识维护

例子：

- 收集箱清空提醒
- 知识库结构检查
- 标签收敛建议
- 命名规范检查

### 记录沉淀

例子：

- 生成当日日志
- 汇总周报
- 从对话中提取长期记忆候选项

### 系统维护

例子：

- 备份配置
- 导出任务清单
- 检查失败任务并重试

## 5. 知识库与运行配置要分开

如果用户想把 OpenClaw 接入个人知识库，推荐把两部分职责拆开：

- `~/.openclaw`：运行配置、代理状态、插件、工作区协议
- 知识库仓库：Markdown 知识内容、模板、规范、归档

### 这样做的好处

- 知识仓库更干净
- Git 历史更聚焦
- 自动化脚本与知识内容不会互相污染
- 更容易把知识库迁移到新设备或新 OpenClaw

## 6. GTD 化时最值得保留的模式

下面这些模式，通常值得保留：

### 先收集，再澄清

- 碎片输入先落 `00-收集箱`
- 原始资料先落 `99-资源库`
- 不要求用户当场就想清楚去向

### 让目录表达阶段

- 目录回答“它处于哪一步”
- 标签回答“它是什么主题”

### 让自动化形成闭环

自动化不该只会“提醒你一下”，最好至少还能：

- 生成一份报告
- 更新一个 TODO
- 标记一条失败原因
- 给出明确下一步动作

### 给代理留自我修正机制

推荐定期让代理思考：

- 最近哪些自动化有效
- 哪些提醒在制造噪音
- 哪些用户偏好已经稳定
- 哪些文档该精简、该更新

## 7. 可迁移的反模式

下面这些错误，在不同 OpenClaw 实例里都很常见：

### 重复自动化

- 同一份备份跑两套机制
- 同一份提醒同时从多个入口发出

### 路径写死

- 把知识库路径、投递渠道、特定用户名直接写死进技能
- 在 skill 里默认某个机器才存在的目录

### 文档职责混乱

- `USER.md` 写成流水账
- `MEMORY.md` 塞满短期信息
- `AGENTS.md` 同时承担太多人设细节和业务规则

### 把本地经验伪装成通用真理

- 某套目录刚好适合一个人，就要求所有人照搬
- 某个 cron 时间点适合一个作息，就默认别人也一样

## 8. 在新 OpenClaw 中落地时的检查项

在任何新环境里，先确认这几件事：

1. 工作区文件职责是否清晰
2. 知识库与运行配置是否分离
3. 是否已有重复自动化
4. GTD 的收集层和归档层是否存在
5. 标签是否承担了过多目录职责
6. 是否有定期复盘与自我修正入口

只有这些边界清楚了，skill 才能真正迁移，而不是把旧环境的问题一起复制过去。
