# 共创式设计流程

这份流程用于帮助代理和用户一起设计基于 OpenClaw 的个人知识库体系。

核心原则只有一句话：

**先一起想清楚，再一起做出来。**

## 1. 代理的角色

在这个技能里，代理不是一次性安装器，而是：

- 访谈者
- 结构设计师
- 分阶段实施搭档
- 运行期优化顾问

默认不要直接冲去建一堆目录、写一堆 cron、迁移一堆文件。

## 2. 共创时要先搞清楚的五件事

### 用户现在怎么收集

例如：

- 微信文章
- 聊天灵感
- 工作文档
- 播客/视频
- AI 对话

### 用户现在怎么推进事情

例如：

- 按项目推进
- 按工作职责推进
- 按日期和待办推进
- 按兴趣和学习主题推进

### 用户希望最终产出什么

例如：

- 更清晰的项目推进
- 更稳定的复盘
- 更系统的学习沉淀
- 公众号文章或对外输出

### 用户现在最痛的点是什么

例如：

- 收藏很多但不处理
- 记了很多但找不到
- 标签越用越乱
- 日志写不下去
- 自动化太吵

### 用户希望 OpenClaw 扮演什么角色

例如：

- 轻提醒型
- 共同整理型
- 强执行型
- 复盘教练型

## 3. 设计阶段的推荐输出

在还没动大文件前，优先输出这些内容：

### 当前状态摘要

- 现有工具和目录
- 用户习惯
- 主要痛点
- 明确目标

### 一版蓝图

至少包含：

- 顶层目录
- 信息流转路径
- 标签策略
- OpenClaw 介入点
- 每日/每周节奏

### 第一阶段计划

只做一小步，比如：

- 先做收集箱和待阅读队列
- 先做项目/学习分层
- 先做日志和周回顾
- 先做 OpenClaw 协议文件

## 4. 推荐的问题方式

不要一次问很多大而空的问题。

优先问：

- 你现在最想解决的是哪一个卡点？
- 你更希望这套系统帮你“收集不丢”，还是“整理不乱”，还是“推动输出”？
- 你平时最常进入知识库的入口是什么？

能从仓库和文件里看出来的事情，先自己看，再补问。

## 5. 推荐的分阶段落地法

### 阶段 A：骨架

- 顶层目录
- 命名规范
- 标签原则

### 阶段 B：入口

- `00-收集箱`
- `99-资源库`
- 待阅读或待处理队列

### 阶段 C：沉淀

- `30-记录`
- `40-学习`
- 日志和笔记模板

### 阶段 D：运行协议

- `AGENTS.md`
- `HEARTBEAT.md`
- `TODO/`
- `MEMORY.md`

### 阶段 E：自动化与反思

- 每日整理
- 每周回顾
- 配置备份
- 健康检查
- 自我反思

### 阶段 F：输出闭环

- 从学习和项目中提取可输出内容
- 推动到 `50-知识输出`

## 6. 每一阶段结束后都要做的事

每做完一个阶段，都要和用户对齐：

- 这一步有没有变得更顺手
- 有没有哪个设计看起来合理但实际别扭
- 下一步最值得做的是什么
- 哪些东西先不要自动化

## 7. 优化已有系统时的做法

已有系统不要直接推翻。

推荐顺序：

1. 先审计
2. 再找真正高频摩擦点
3. 再只改一小批
4. 让用户真实用几天
5. 根据反馈再收敛

## 8. 常见误区

- 还没理解用户就开始搭架子
- 把“标准化”误当成“适合用户”
- 一次性引入太多目录和标签
- 自动化上得太快，用户没形成习惯
- 想一步到位，结果没有任何一层真正跑起来
