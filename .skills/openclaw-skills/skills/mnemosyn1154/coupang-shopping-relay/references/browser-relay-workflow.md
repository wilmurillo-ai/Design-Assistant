# Browser Relay Workflow

이 문서는 Chrome browser relay 기반 쿠팡 쇼핑 스킬의 세부 운영 지침입니다.

## 핵심 운영 모델

- relay는 사용자의 로그인된 일반 Chrome 세션을 사용합니다.
- relay로 수행하는 동작은 탐색, 클릭, 입력, 텍스트 읽기 중심이어야 합니다.
- relay가 페이지를 조작하더라도, 쿠팡 쪽에는 일반 사용자 브라우저 상호작용처럼 보여야 합니다.
- **단건 모드** 기본 루프는 `읽기 -> 사용자 확인 -> 단일 액션 -> 다시 읽기`입니다.
- **단건 모드**에서는 한 번에 여러 품목을 연속으로 처리하기보다, 한 품목 액션 후 장바구니 재검수를 넣는 편이 안정적입니다.
- **배치 모드**에서는 재료별로 검색 → 자동 선택 → 담기를 연속 처리하고, 장바구니 재검수는 전체 배치 완료 후 한 번만 수행합니다.

## 실제 테스트에서 확인된 점

### 비교적 안정적이었던 영역

- 브라우저 실행과 연결
- 쿠팡 접속
- 로그인 상태 확인
- 검색창 사용
- 검색 결과 읽기
- 장바구니 기존 품목 읽기
- 장바구니 일부 품목 삭제
- 스크린샷 캡처

정리하면, 읽기와 현재 상태 파악은 비교적 안정적이었습니다.

### 흔들렸던 영역

- 검색 결과 클릭 후 상세 페이지 이동 안정성
- 상세 페이지에서 장바구니 담기 후 반영 확인
- 장바구니 추가 품목 즉시 확인
- 동적 ref 변화
- 여러 탭이 생기면서 포커스가 꼬이는 문제
- 장바구니 페이지 스냅샷이 불완전하게 수집되는 문제

정리하면, 쓰기 액션과 연속 이동은 읽기보다 불안정했습니다.

## 권장 운영 흐름

### 단건 모드 (기본)

가장 안정적인 기본 흐름:

1. 로그인 상태 확보
2. 검색 결과에서 후보 탐색
3. 후보를 1~3개로 좁힘
4. 사용자에게 후보, 가격, 규격을 제시
5. 사용자 승인 후 한 번에 한 품목만 장바구니 담기
6. 장바구니 화면으로 이동해 캡처와 재검수 수행
7. 필요하면 다음 품목으로 반복
8. 결제 직전에는 대상 품목, 배송 시점, 총액만 따로 정리

### 배치 모드

사용자가 여러 재료/상품을 한 번에 요청한 경우:

1. 로그인 상태 확보
2. 재료 목록 파싱
3. 각 재료마다:
   a. 불필요한 탭 정리 (상세 페이지 탭 모두 닫기), 쿠팡 검색 페이지에서 새로 시작
   b. 검색 실행 → wait → snapshot
   c. 자동 상품 선택 기준에 따라 최적 상품 선택 (광고 건너뛰기, 로켓배송 우선, 옵션 없는 상품 우선)
   d. 상세 페이지 이동 → wait → snapshot → 장바구니 담기
   e. 1~2초 대기 → 성공 판정 (장바구니 가기 버튼/레이어, 아이콘 숫자 증가, 토스트 중 하나)
   f. 실패 시 기록하고 다음 재료로 진행
4. 전체 배치 완료 후 장바구니 이동 → wait → snapshot → 실제 담긴 품목 일괄 검증
5. 최종 결과 보고 (성공 품목, 실패 품목, 합계)

배치 모드에서 중간 상태 보고 금지. 사용자에게 중간에 묻지 않음.

### 피해야 할 흐름

- **단건 모드**: 검색 → 상세 → 담기 → 뒤로 → 다른 상품 담기 → 선택 삭제 → 결제를 한 번에 몰아서 처리하는 것
- **배치 모드**: 배치 모드에서는 매 재료마다 검색부터 새로 시작하는 것을 권장. 이전 검색 결과 탭을 재활용하거나 뒤로 가기로 돌아가려 하지 않음

## 탭과 포커스 관리

동적 페이지에서 탭이 여러 개 생기면 오류가 늘어납니다.

### 단건 모드

원칙:
1. 새 탭이 열리면 현재 활성 탭이 어디인지 먼저 확인
2. 원하는 상품 상세 탭인지 제목이나 URL로 다시 확인
3. 쓰기 액션 전에는 현재 탭이 맞는지 다시 확인
4. 작업이 끝난 보조 탭은 가능하면 정리하고, 장바구니 탭 하나를 기준 탭으로 유지

새 탭이나 포커스가 꼬이면:
1. 현재 열린 탭 목록을 확인
2. 제목과 URL로 장바구니/상세/검색 탭을 재구분
3. 장바구니 탭을 다시 기준으로 삼고 상태를 재확인

### 배치 모드

원칙:
1. 매 재료 시작 전에 현재 열린 쿠팡 상세 페이지 탭을 모두 닫음
2. 쿠팡 검색 페이지(`https://www.coupang.com/np/search?q=`)에서 새로 시작
3. 이전 상품 상세 탭을 재사용하려 하지 않음 — 깔끔하게 매번 초기화
4. 탭이 꼬이면 멈추고 재정렬을 시도하기보다, 모든 탭을 닫고 검색부터 다시 시작

## Snapshot 전 Wait 의무화

쿠팡은 비동기 로딩이 심하므로 wait 없이 찍은 snapshot은 신뢰하지 않음. 페이지 이동 후, 장바구니 담기 후, 검색 결과 로딩 후에는 반드시 `openclaw browser wait` 명령으로 핵심 요소가 로딩될 때까지 기다린 후 snapshot을 찍어야 함.

### 페이지별 wait 대상

| 페이지 | wait 대상 | 비고 |
| --- | --- | --- |
| 검색 결과 | 상품 카드 1개 이상 로딩 | 광고 카드만 먼저 뜰 수 있으므로 실제 상품 카드 확인 |
| 상품 상세 | 장바구니 담기 버튼 나타남 | 옵션 선택 UI도 로딩 확인 필요 |
| 장바구니 | 장바구니 품목 목록 로딩 | 추천 영역이 먼저 뜨는 경우가 많으므로 품목 목록 기준 |
| 담기 클릭 후 | 1~2초 대기 후 성공 UI 확인 | 토스트, 레이어, 장바구니 가기 버튼 |

### 배치 모드에서의 wait 적용

배치 모드 매 재료 루프에서 wait가 필요한 시점:
1. 검색 실행 후 → wait → snapshot (검색 결과 파싱)
2. 상세 페이지 이동 후 → wait → snapshot (담기 버튼 확인)
3. 장바구니 담기 클릭 후 → 1~2초 대기 → snapshot (성공 UI 확인)
4. 전체 배치 완료 후 장바구니 이동 → wait → snapshot (일괄 검증)

## 추천 응답 형식

### 검색 결과를 보여줄 때

다음처럼 번호를 고정해 보여주세요:

1. 상품명
가격: 12,900원
배지: 로켓배송
평점: 4.8 / 리뷰 152개

2. 상품명
가격: 13,500원
배지: 일반배송

번호는 현재 대화 맥락에서만 유효합니다. 다음 화면으로 이동한 뒤에는 같은 번호가 다른 항목을 의미할 수 있으니, 상태 변경 전에는 최신 화면 기준으로 대상을 다시 확인하세요.

### 장바구니를 보여줄 때

다음 정보를 우선해서 읽으세요:
- 위에서부터 순서
- 상품명
- 옵션/규격
- 수량
- 가격

가능하면 다음처럼 보여주세요:

1. 생수 2L 6개입
수량: 2
가격: 7,900원

2. 쿠팡 물티슈 100매
수량: 1
가격: 12,900원

장바구니에는 서로 무관한 품목이 섞일 수 있습니다. 따라서 가능하면 다음 두 구획으로 나눠 보여주는 것이 좋습니다:
- 이번 주문 대상
- 비대상 또는 보류 품목

## 상태 변경 전 확인 규칙

### 장바구니 추가

**단건 모드:**

실행 전 확인할 것:
- 상품명이 사용자가 고른 항목과 일치하는가
- 가격이 비정상적으로 바뀌지 않았는가
- 옵션 선택이 필요한가

사용자에게 한 줄로 다시 확인:
- "`OOO`, `12,900원` 상품을 장바구니에 담을게요."

**배치 모드:**

실행 전 내부적으로만 확인:
- 상품명이 요청한 재료와 관련 있는가
- 자동 선택 기준(로켓배송 우선, 광고 건너뛰기, 옵션 없는 상품 우선)에 부합하는가
- 사용자에게 묻지 않고 바로 진행

**공통 — 클릭 후 판독 순서:**
1. 클릭 직후 바로 다른 페이지로 이동하지 말고 1~2초 정도 현재 페이지에서 성공 UI를 확인
2. 혜택 레이어, 추천 팝업, 장바구니 가기 레이어 같은 중간 UI가 떴는지 먼저 인지
3. `장바구니 가기` 버튼이나 명시적 성공 문구가 보이면 그 경로를 우선 신뢰
4. **단건 모드**: 그 다음 장바구니로 이동해 신규 품목 또는 수량 증가를 재검수
5. **배치 모드**: 장바구니 이동 없이 성공으로 간주하고 다음 재료로 진행. 위 1~3번 중 하나라도 확인되면 성공. 모두 안 보이면 실패로 기록하고 다음으로 이동.

즉, 실제 클릭 실패와 클릭 후 상태 판독 실패를 구분해야 합니다. 쿠팡에서는 클릭 자체보다 클릭 이후 상태 해석이 더 어려운 경우가 많습니다.

### 장바구니 삭제

실행 전 확인할 것:
- 어떤 항목을 지우는지 현재 장바구니 기준으로 특정했는가
- 같은 상품이 여러 개면 옵션이나 가격까지 맞는가

사용자에게 한 줄로 다시 확인:
- "`장바구니 2번`, `쿠팡 물티슈 100매`를 삭제할게요."

### 수량 변경

실행 전 확인할 것:
- 대상 항목이 맞는가
- 목표 수량이 명확한가

사용자에게 한 줄로 다시 확인:
- "`장바구니 1번`, `생수 2L 6개입` 수량을 `2개`로 변경할게요."

### 주문 직전 검수

최종 주문 제출은 기본 모드에서 하지 않지만, 주문 직전 검수는 다음 순서로 수행할 수 있습니다:

1. 배송지 라벨과 도착 예정 문구 확인
2. 이번 주문 대상 품목만 따로 정리
3. 각 상품 수량과 금액 확인
4. 할인, 적립, 배송비, 총 결제 예정 금액 확인
5. 결제수단 라벨만 읽고, 변경은 하지 않음
6. 긴 화면이면 스크린샷 캡처로 다시 확인

사용자에게 짧게 다시 확인:
- "`이번 주문 대상 3개`, `총 결제 예정 15,880원`, `내일 도착 예정`으로 보입니다. 최종 제출은 하지 않고 여기까지 검수할게요."

## 성공 판정 기준

### 장바구니 추가 성공

아래 중 하나 이상이면 성공으로 볼 수 있습니다:
- 장바구니 추가 토스트 또는 확인 문구 노출
- 장바구니 아이콘 카운트 증가
- 장바구니 페이지 재조회 시 새 항목 추가 또는 수량 증가 확인

위 조건이 없으면 "버튼은 눌렀지만 실제 반영은 확인되지 않았다"고 설명하세요.

### 장바구니 삭제 성공

아래 조건이 필요합니다:
- 삭제 후 장바구니를 다시 읽었을 때 대상 항목이 없어짐

### 수량 변경 성공

아래 조건이 필요합니다:
- 장바구니를 다시 읽었을 때 대상 항목 수량이 요청한 값과 일치함

### 주문 직전 검수 성공

아래 조건이 모두 읽히면 성공으로 볼 수 있습니다:
- 주문 대상 품목 목록
- 총 결제 예정 금액
- 도착 예정 또는 배송 관련 핵심 문구
- 결제수단 라벨

## 실패 상황별 대응

### 로그인 화면이 뜸

- 사용자가 Chrome에서 직접 로그인하도록 안내
- 로그인 완료 후 같은 페이지에서 이어서 진행

### 추가 인증 또는 CAPTCHA

- 사용자가 직접 브라우저에서 처리하도록 안내
- 자동으로 우회 시도하지 않음

### 애매한 대상

예:
- 같은 물티슈가 2개 있음
- 옵션만 다른 상품이 여러 개 있음

대응:
- 가격, 옵션, 수량을 함께 보여주고 다시 물음

### 버튼을 눌렀는데 확인이 안 됨

대응:
- 현재 화면 상태 설명
- 팝업, 추천 영역, 혜택 레이어, 장바구니 가기 버튼 존재 여부를 먼저 확인
- 장바구니 재조회
- 그래도 불확실하면 성공으로 말하지 않음

### 장바구니 스냅샷이 잘리거나 불완전함

대응:
1. 스냅샷 재수집
2. 장바구니를 다시 맨 위에서부터 읽음
3. 필요하면 스크린샷 첨부
4. 그래도 불완전하면 현재 확인 가능한 항목만 요약하고 한계를 밝힘

### 탭 포커스가 꼬임

대응:
1. 열린 탭을 다시 식별
2. 장바구니 또는 현재 대상 상세 탭으로 포커스를 재정렬
3. URL과 제목을 확인한 뒤 다음 액션 진행

## 추천 자동화 단위

좋은 단위:
- 검색
- 후보 가격 비교
- 용량 필터링
- 냉장/냉동/완제품 구분
- 장바구니 현재 상태 읽기
- 합계 계산
- 장바구니 스크린샷 제공
- **배치 모드**: 재료별 검색 → 자동 선택 → 담기 연속 처리 (매 재료마다 검색부터 새로 시작)

단건 모드에서 위험한 단위:
- 여러 상세 페이지를 연속 순회하며 한 번에 다수 상품을 담는 작업
- 상세 이동, 담기, 삭제, 결제를 한 번에 몰아서 수행하는 작업

배치 모드에서 위험한 단위:
- 이전 검색 결과 탭을 재활용하여 뒤로 가기로 돌아가는 작업
- 매 품목마다 장바구니 페이지로 이동해서 재검수하는 작업 (전체 완료 후 한 번만)

## 사용자 응답 UX

자동화 중에는 매번 길게 설명하기보다 다음만 짧게 알려주는 편이 좋습니다:
- 현재 상태
- 담은/삭제한 품목
- 예상 금액
- 다음 액션

예:
- `하림 1kg 삭제됨`
- `500g 후보는 아직 반영 안 됨`
- `현재 식단 관련 합계 9,690원`
- `500g 추가 시 예상 15,880원`

## 외부 스킬 참고

검색 결과에서 상품명, 가격, 배지, 리뷰 수를 구조화된 형태로 파싱하는 로직을 강화하기 위해 아래 ClawHub 스킬을 참고:
- `clawringhouse` — AI shopping concierge: 검색 결과 구조화, 비교 패턴
- `openclaw-ecommerce` — 가격 추적, 알림, 상품 데이터 구조화: 상품 데이터 정규화 로직

특히 배치 모드에서 자동 상품 선택 시 검색 결과 파싱 안정성이 중요하므로, 상품 카드에서 다음을 구조화하여 추출:
- 상품명 (광고 태그 여부 포함)
- 가격 (할인 전/후)
- 배지 (로켓배송/로켓와우/로켓프레시)
- 리뷰 수/평점
- 옵션 필요 여부
