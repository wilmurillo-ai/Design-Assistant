# Coupang Browser Automation Improvement Log

이 문서는 실제 쿠팡 브라우저 자동화 테스트에서 관찰된 안정 구간, 실패 패턴, 개선 아이디어를 누적하기 위한 로그입니다.

## 테스트 목적

- 로그인된 브라우저 세션에서 쿠팡 장보기 보조
- 목표 작업:
  - 상품 검색
  - 장바구니 읽기
  - 장바구니 정리
  - 후보 상품 추가
  - 결제 직전까지 이동

## 현재까지 정상 동작

### 1. 브라우저 연결

- OpenClaw browser 자체는 정상 실행
- 전용 브라우저 프로필 실행 가능
- 쿠팡 페이지 열기 가능

### 2. 로그인 세션 유지

- 쿠팡 로그인 상태 확인 가능
- 로그아웃, 마이쿠팡, 사용자명 일부 노출로 로그인 상태 판별 가능

### 3. 조회 계열 작업

- 검색창 입력 가능
- 검색 결과 읽기 가능
- 장바구니 현재 품목 읽기 가능
- 장바구니 품목명/가격 파악 가능
- 스크린샷 캡처 가능

정리:
- 읽기, 조회, 검색, 현재 상태 파악은 비교적 안정적입니다.

## 불안정 또는 실패한 동작

### A. 장바구니 조작 불안정

증상:
- 삭제 버튼 클릭 후 기대한 장바구니 상태 변경 대신 다른 상품 상세 페이지로 튀는 현상 발생
- 장바구니에 머물러야 하는데 엉뚱한 상품 PDP(Product Detail Page)로 이동함

재현 상황:
- 장바구니에서 더여백 관련 품목 삭제 시도 중 발생
- 삭제 후 ref 또는 페이지 상태가 즉시 바뀌며 포커스가 꼬임

추정 원인:
- 쿠팡 장바구니가 삭제 후 DOM을 크게 다시 그림
- 삭제 텍스트나 버튼 주변의 클릭 타이밍이 어긋나서 다른 클릭 이벤트가 잡힐 수 있음
- OpenClaw snapshot ref가 재렌더링 후 빠르게 무효화됨

개선 아이디어:
- 삭제 후 반드시 현재 URL 확인
- 장바구니 개수 확인
- 장바구니 탭 유지 여부 확인
- 삭제 액션 후 즉시 추가 액션하지 말고 새 snapshot 재수집
- 장바구니 작업은 한 번에 1개만 처리

### B. 검색 결과에서 상세 페이지 이동이 불안정

증상:
- 검색 결과 클릭은 되지만 새 탭으로 열리기도 하고, 기존 탭에서 열리기도 하고, 대기 명령은 timeout 나기도 함
- 상세 페이지 이동 성공 여부가 일관되지 않음

재현 상황:
- 닭가슴살 500g 검색 결과 클릭 시 자주 발생

추정 원인:
- 쿠팡 검색 결과 페이지의 링크 처리 방식
- 새 탭/기존 탭 혼재
- 이동 후 페이지 로딩 대기 조건이 불안정

개선 아이디어:
- 클릭 후 항상 tabs로 새 탭 생성 여부 확인
- 가장 최근 쿠팡 상품 탭으로 포커스
- URL 매칭은 더 넓게 `vp/products`를 사용
- timeout 실패해도 실제 탭 생성 여부를 먼저 재확인

### C. 장바구니 담기 후 반영 확인이 불안정

증상:
- 상품 상세에서 장바구니 담기 클릭까지는 되는 것처럼 보이나 실제 장바구니에 상품이 반영되지 않거나, 반영 여부를 안정적으로 확인하지 못함

재현 상황:
- 다향 닭가슴살 500g 후보 추가 시도 중 반복

추정 원인:
- 장바구니 담기 후 쿠팡 레이어나 팝업 처리
- add-to-cart 성공 UI가 스냅샷에 안정적으로 잡히지 않음
- 실제 장바구니 반영 전에 사용자가 다른 탭이나 상태로 이동했을 가능성
- ref 재사용 타이밍 문제

개선 아이디어:
- 장바구니 담기 후 즉시 장바구니 탭으로 가기보다 현재 페이지에서 성공 UI를 먼저 확인
- 성공 UI가 확인되면 그 버튼으로 이동
- 성공 UI가 없으면 장바구니 직접 진입 후 장바구니 개수 변화 또는 상품명 존재 여부 확인

추가 해석:
- 일부 사례는 실제 클릭 실패가 아니라 클릭 이후 상태 판독 실패였을 가능성이 높음
- 쿠팡 혜택, 추천, 장바구니 가기 레이어가 중간에 뜨면 실패처럼 오인할 수 있음
- 장바구니 반영 완료 전에 너무 빨리 재확인하면서 false negative가 났을 수 있음

### D. 장바구니 페이지 snapshot 품질 불안정

증상:
- 장바구니 페이지가 일부만 읽히거나 추천 상품 위주로 보이고 실제 담긴 품목 구조가 불완전하게 잡힘
- 어떤 때는 잘 읽히고, 어떤 때는 거의 못 읽음

재현 상황:
- 장바구니 페이지 진입 직후
- 삭제나 이동 직후 자주 발생

추정 원인:
- 로딩이 덜 끝난 상태에서 snapshot 수행
- 장바구니 DOM이 비동기로 늦게 붙음
- 추천 영역이나 광고 영역이 먼저 잡힘

개선 아이디어:
- 장바구니 페이지 진입 후 바로 snapshot 하지 말고 잠깐 대기하거나 장바구니 텍스트 요소 확인 후 snapshot
- 텍스트 snapshot이 이상하면 스크린샷 캡처를 fallback으로 사용
- 장바구니는 텍스트 읽기와 이미지 확인을 병행

### E. ref 재사용 불가 문제

증상:
- 방금 쓴 ref가 몇 초 후 바로 무효화됨
- Unknown ref 오류 발생

재현 상황:
- 삭제 직후
- 페이지 이동 직후
- 추천이나 동적 목록이 섞인 쿠팡 페이지 전반

추정 원인:
- DOM 재렌더링 빈도 높음
- 페이지 구조가 동적으로 바뀜

개선 아이디어:
- 쿠팡에서는 ref를 장시간 유지한다고 가정하지 않음
- 액션마다 새 snapshot을 기준으로 ref 재획득
- 특히 삭제 후, 담기 후, 탭 전환 후, 페이지 복귀 후에는 무조건 ref 갱신

## 실무상 얻은 운영 규칙

### 규칙 1

읽기와 쓰기를 분리한다.

- 읽기:
  - 검색
  - 가격 비교
  - 장바구니 확인
- 쓰기:
  - 삭제
  - 담기
  - 결제 단계 이동

### 규칙 2

한 번에 한 액션만 한다.

- 검색 후 바로 여러 품목 연속 담기 금지
- 삭제도 1개 삭제 후 재확인 후 다음 삭제

### 규칙 3

장바구니 작업 후에는 무조건 장바구니를 재확인한다.

- URL 확인
- 장바구니 개수 확인
- 상품명 확인
- 필요시 스크린샷 캡처

### 규칙 4

결제 전에는 텍스트 요약과 화면 캡처를 둘 다 제공한다.

- 사용자가 최종 검수 가능하도록 함

목표 조건:

- 생 닭가슴살
- 냉장 우선
- 400~600g 우선
- 수비드, 훈제, 실온보관 제외

실제 검색에서 유효했던 후보:

- 다향 닭가슴살 (냉장), 500g, 1개

- 하림 또는 자연실록 400~600g 일부

자동화 전략:

1. 먼저 후보 2~3개 가격과 용량 비교
2. 사용자 승인 후 1개만 장바구니 담기

## v1.3.0 변경 기록 — wait 의무화, 탭 초기화 구체화, 외부 스킬 참고 (2026-04-05)

### 변경 이유
배치 모드 도입 후 snapshot 안정성과 탭 관리 정밀도를 높여야 실전 성공률이 올라감.

### 주요 변경 사항
1. **snapshot 전 wait 의무화**: 검색 결과(상품 카드 로딩), 상세 페이지(담기 버튼 로딩), 장바구니(품목 목록 로딩), 담기 후(성공 UI 확인) 각 시점에서 wait 후 snapshot. wait 없이 찍은 snapshot은 신뢰하지 않음.
2. **매 재료 사이 탭 초기화 구체화**: 쿠팡 상세 페이지 탭을 모두 닫고 검색 페이지에서 새로 시작. 이전 상세 탭 재사용 금지.
3. **외부 스킬 참고 추가**: ClawHub의 `clawringhouse`(AI shopping concierge), `openclaw-ecommerce`(가격 추적/상품 데이터 구조화) 참고하여 검색 결과 파싱 안정성 강화.
4. **배치 모드 실행 흐름 요약 추가**: 전체 배치 루프를 의사코드로 명시 (탭 정리 → 검색 → wait → 선택 → 상세 → wait → 담기 → 대기 → 판정 → 다음).

- browser-relay-workflow.md: wait 의무화 섹션 추가, 배치 모드 흐름에 wait 단계 반영, 탭 초기화 구체화, 외부 스킬 참고 섹션 추가
- operating-memory.md: 배치 모드 루프에 wait 단계 반영

## v1.2.0 변경 기록 — 배치 모드 도입 (2026-04-05)

### 변경 이유
다수 재료 요청 시 매번 확인으로 진도가 나가지 않음. 배치 모드 도입으로 UX 개선.

### 주요 변경 사항
1. **배치 모드 추가**: 여러 재료를 한 번에 요청하면 자동으로 배치 모드 전환. 검색 → 자동 선택 → 담기를 재료별 연속 처리, 최종 결과만 보고.
2. **자동 상품 선택 기준**: 로켓배송 우선, 리뷰 많은 순, 광고 건너뛰기, 옵션 없는 상품 우선. 탈락 후보 2~3개 내부 보관.
3. **실패 처리 변경**: 하나 실패해도 다음 재료로 계속 진행. 전체 완료 후 실패 항목만 따로 표시.
4. **중간 보고 제거**: 배치 모드에서 진행 상황 메시지 금지.
5. **방어 규칙 완화**: 배치 모드에서 사용자 확인 없이 자동 선택/담기 진행. 단건 모드는 기존 규칙 유지.
6. **성공 판정 기준 완화**: 장바구니 가기 버튼/레이어, 아이콘 숫자 증가, 토스트 중 하나라도 있으면 성공.
7. **중간 검증 간소화**: 배치 모드에서 개별 품목마다 장바구니 재검수 안 함. 전체 완료 후 한 번만.
8. **탭 관리 단순화**: 배치 모드에서 매 재료 시작 전 탭 초기화. 이전 탭 이어받지 않음.
9. **피해야 할 흐름 수정**: 배치 모드에서 연속 담기를 정상 흐름으로 인정. 매 재료마다 검색부터 새로 시작 권장.

- browser-relay-workflow.md: 권장 운영 흐름, 탭 관리, 상태 변경 전 확인, 추천 자동화 단위 섹션 수정

## 한 줄 결론

쿠팡 자동화는 조회와 요약은 안정적이고, 삭제, 담기, 연속 이동은 재검수형 반자동으로 운영해야 합니다.

## 이번 교정에서 추가된 최종 결론

장바구니 담기 실패처럼 보였던 사례 중 일부는 실제 클릭 실패보다, 담기 직후 노출되는 쿠팡 혜택, 추천, 장바구니 레이어와 빠른 상태 재판독 때문에 생긴 오판 가능성이 높습니다.

이후 판독 순서:
1. 장바구니 담기 클릭
2. 1~2초 정도 현재 페이지에서 성공 UI 확인
3. 팝업이나 혜택 레이어 존재 여부 확인
4. `장바구니 가기` 버튼이 보이면 그 경로를 우선 사용
5. 장바구니에서 최종 재검수
