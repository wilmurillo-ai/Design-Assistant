## 📊 1. Token 消耗分析

| L2 | 活跃项目 | 5,000 | 5,000 | ✅ 正常 |

### 问题发现

**🚨 索引文件过大（66KB）**
- 当前索引包含每个章节的完整摘要（200字符）
- 23个文件 × 平均3个章节 × 200字符 = 约14KB摘要
- 加上元数据，实际66KB

**对比分析：**
- 手动回忆（全量）: 58,796 token
- 分层回忆（当前）: 49,973 token
- 节省仅 15%（远低于预期）

## ⚠️ 4. 潜在问题

### 问题1: 索引文件过大 🔴 严重

**原因**: 每个章节都保存200字符摘要

**影响**: 
- L3预算1k，实际34k
- 导致总token超支149%

**解决方案**:
```python
# 优化方案：索引只保留定位信息，不保存摘要
{
  "topics": {
    "游戏开发": [
      {"file": "2026-03-23.md", "lines": "100-200"},  # 无摘要
      {"file": "2026-03-24.md", "lines": "50-150"}
    ]
  }
}
```

**预期效果**: 索引从66KB降到约5KB（~2.5k token）

### 问题2: 索引更新频率 ⚠️ 中等

**当前**: 需手动运行 `build-index.py`

**建议**: 
- 心跳检查时自动更新索引（每日一次）
- 或检测到新日志文件时自动更新

### 问题3: 与self-improving-enhancement协作 ⚠️ 轻微

**潜在冲突**:
- 两个技能都管理记忆文件
- self-improving-enhancement 有自己的 `~/self-improving/` 目录

**建议**: 明确分工
- tiered-recall: 负责加载策略
- self-improving-enhancement: 负责记忆内容管理

## 💡 5. 改进建议

### 紧急修复（v1.0.1）

1. **精简索引文件**
   - 移除章节摘要
   - 只保留文件名+行号
   - 目标: 从66KB降到5KB

2. **动态token预算**
   - L1日志按实际大小计算
   - 如果L1小，可以把预算给L2

### 优化改进（v1.1.0）

3. **智能日志截取**
   - 只加载相关章节（非全文）
   - 根据活跃项目过滤日志

4. **索引自动更新**
   - 心跳时检查是否需要更新
   - 检测到新日志自动重建

5. **可配置参数**
   ```json
   {
     "recentDays": 2,  // 可改为3
     "maxTokensPerLayer": {...},
     "autoUpdateIndex": true
   }
   ```

### 功能增强（v2.0.0）

6. **项目关联记忆**
   - 记录每个项目的相关日期
   - 切换项目时自动加载相关记忆

7. **记忆热度统计**
   - 统计哪些记忆被频繁访问
   - 热点记忆提升加载优先级
