# Layer 4 — Synthesis, Integration & Composite Frameworks

This layer handles two things:
1. Synthesizing the excavated data into a unified, operational picture
2. Building composite frameworks when the user blends multiple targets

---

## Part 1 — Single Framework Synthesis

After completing all 15 dimensions (L1 + L2), run this synthesis pass
before presenting the Framework Card. This is what elevates the output
from a list of observations to a living cognitive lens.

### Synthesis Step 1 — Find the Generative Core

Every framework has one central mechanism that generates everything else.
All 15 dimensions should connect back to it.

Ask: "If I had to explain every major decision, every strength, and every
blind spot this target has with a single underlying dynamic — what is it?"

Examples:
- A thinker whose wound was powerlessness → the entire framework is organized
  around control, mastery, and the elimination of uncertainty
- A philosopher whose core model is impermanence → risk posture, time horizon,
  relational patterns all reflect non-attachment
- A founder whose drive is significance → utility function, decision heuristics,
  and shadow all orbit the need to matter at scale

State the generative core in one sentence. This goes into the Framework Card
as the invisible thread connecting everything.

### Synthesis Step 2 — Map the Internal Coherence

Check: do the 15 dimensions tell a consistent story?

Consistency test:
```
Does C3 (utility) explain the choices documented in C2 (heuristics)? ✓/✗
Does P1 (wound) explain the intensity of C5 (risk posture)? ✓/✗
Does P5 (shadow) explain the pattern in P4 (defenses)? ✓/✗
Does C7 (paradox) connect to P3 (ego architecture)? ✓/✗
```

Where inconsistencies appear: either the evidence is mixed and should be
flagged, or the inconsistency itself is data (unintegrated shadow, regression
under stress, etc.).

### Synthesis Step 3 — Derive the Signature Outputs

From the full picture, extract:

**Signature Insight** (cognitive edge):
"This framework sees [X] that most people miss because [Y]."
Must be specific and non-obvious. "Thinks long-term" is not a signature insight.
"Treats volatility as a tax paid for asymmetric upside" is.

**Defining Trade-off** (the price of the framework):
"This framework gains [X] by consistently sacrificing [Y]."
Every powerful framework has a price. Naming it is essential.

**Load-bearing Paradox** (the productive tension):
"This framework simultaneously [A] and [B] — and the tension between them
is what makes it powerful."
This is not a contradiction to resolve. It is the engine.

**Ideal Conditions** (when to deploy this framework):
"This framework works best when [conditions]. It misfires when [conditions]."

### Synthesis Step 4 — Calibrate Confidence

Rate the overall excavation honestly:

```
RICH     : Extensive primary sources, consistent behavioral record,
           multiple independent accounts. High inference confidence.

MODERATE : Good primary sources in some areas, behavioral patterns visible,
           some gaps. Medium inference confidence. Flag gaps.

THIN     : Limited primary sources, mostly secondary accounts,
           significant gaps. Low inference confidence.
           Many claims are pattern inference — label them.

MYTHOLOGICAL RISK : Target is well-known but primary sources are sparse.
                    Popular narrative dominates. High risk of excavating
                    the myth rather than the person. Flag explicitly.
```

## Part 2 — Composite Framework Protocol

When the user requests a blend of multiple frameworks:
"Use [A]'s patience combined with [B]'s risk logic"
"Think like someone who combines [X] and [Y]"

### Composite Step 1 — Map Each Framework First

Do a condensed excavation of each component framework.
For 2-framework blends: focus on C1, C2, C3, C5, P2 for each.
For 3+ blends: C1, C2, C3 only — more dimensions become unmanageable.

### Composite Step 2 — Find the Compatibility Structure

```
SHARED GROUND
  Where do both frameworks fundamentally agree?
  These are the load-bearing pillars of the composite.

COMPLEMENTARY ZONES
  Where does A contribute what B lacks, and vice versa?
  These create the composite's unique leverage.

TENSION POINTS
  Where do they genuinely conflict?
  Do NOT paper over these. They are where the most useful
  thinking happens — and where the user needs to consciously choose.

INCOMPATIBILITY ZONES
  Where are they fundamentally irreconcilable?
  Flag these explicitly. Force a choice rather than pretending harmony.
```

### Composite Step 3 — Build the Navigation Logic

For each tension point, propose a decision rule:
"When the question is about [domain], Framework A's logic dominates."
"When the question is about [domain], Framework B has the better lens."
"When they conflict directly: [explicit trade-off, user chooses]."

### Composite Card Format

```
╔══════════════════════════════════════════════════════════════════╗
║  🧠  COMPOSITE FRAMEWORK: [A] × [B]                              ║
╠══════════════════════════════════════════════════════════════════╣

║  A Contributes    » [What A adds that B structurally lacks]      ║
║  B Contributes    » [What B adds that A structurally lacks]      ║

║  ── TENSION POINTS ───────────────────────────────────────────  ║
```

## Part 3 — Advanced Application Modes

### Mode: Stress Test

User asks: "Stress test this framework against [hard scenario]"

Protocol:
1. Apply P8 (stress & breakdown) to the specific scenario
2. Identify which dimensions of the framework are load-bearing vs. decorative
3. Find the conditions under which the framework would catastrophically fail
4. Propose what would need to be true for the framework to hold

Output format:
```
SCENARIO         : [The hard case]
FRAMEWORK RESPONSE: [How it would initially engage]
PRESSURE POINT   : [Where it starts to strain]
FAILURE CONDITION: [What would cause collapse]
SURVIVAL CONDITION: [What would allow it to hold]
```

### Mode: Forecast

User asks: "Using this framework, forecast what [target] would do about [situation]"

Protocol:
1. Apply C2 (heuristics) + C3 (utility) to the situation
2. Filter through C4 (framing) — how would they redefine the question?
3. Apply C5 (risk posture) to the options
4. Check P2 (drive) — what emotional pull is present?
5. Flag P8 (stress) if the situation is high-pressure

Always label forecast as inference, not fact.

### Mode: Devil's Advocate

User asks: "Argue against this framework from within it"

Protocol:
1. Apply P5 (shadow) + P4 (defense mechanisms) to identify the internal critic
2. Use C7 (paradox) to find the tension that undermines the framework's claims
3. Apply D8 (blind spots) to the framework's own core assumptions
4. Generate the most honest critique possible using the framework's own logic

This is the most intellectually valuable mode — a framework that can critique
itself is more trustworthy than one that cannot.

### Mode: Cross-Framework Debate

User asks: "What would [Framework A] say to [Framework B] about [topic]?"

Protocol:
1. Load Framework A's C3 (utility) and C4 (framing)
2. Load Framework B's C3 and C4
3. Find the fundamental disagreement at the utility/framing level
4. Generate the dialogue that makes that disagreement explicit and productive

The goal is not to declare a winner.
The goal is to show what each framework sees that the other structurally cannot.
