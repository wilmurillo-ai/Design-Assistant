# MVP 完备性审查与文档对齐设计文档

- 日期：2026-04-10
- 项目：`bing-keyword-image-downloader`
- 设计主题：按 MVP 标准审查当前多来源图片下载器，并完成 README / SKILL / 开发文档对齐
- 执行状态：本设计对应工作已完成；本文档作为设计依据保留，不承担逐步进度勾选职责。实际执行结果见 `docs/superpowers/plans/2026-04-10-mvp-completion-alignment.md`。

## 1. 目标

本轮工作的目标不是继续扩展功能，而是按已经确认的 MVP 标准收口当前仓库。

本轮的 MVP 完备定义为：

1. 当前多来源下载流程可运行。
2. 历史去重逻辑可用。
3. 文件编号续写逻辑可用。
4. 自动化测试 fresh 通过。
5. `README.md`、`SKILL.md`、开发计划文档与当前真实实现一致。

本轮不新增来源、不继续架构重构、不补做更高阶段能力，只做验证、审查和文档对齐。

## 2. 范围

### 2.1 纳入范围

- 审查当前代码是否已经满足 MVP 标准
- fresh 运行完整自动化测试
- fresh 运行一轮 CLI 冒烟验证
- 更新 `README.md`
- 更新 `SKILL.md`
- 更新 `docs/superpowers/plans/2026-04-10-multi-source-image-downloader.md`
- 提交上述修改，并使用中文 git 提交信息

### 2.2 不纳入范围

- 不新增图片来源
- 不引入新的去重算法
- 不补做独立 `pipeline.py`
- 不修改核心业务能力边界
- 不重写 `evals/evals.json`
- 不做大规模压力测试或性能优化

## 3. 现状判断

当前仓库已经具备多来源 MVP 的主体实现：

- `scripts/bing_image_downloader.py` 已承担 CLI 调度入口
- `image_downloader/sources/bing.py` 与 `image_downloader/sources/demo.py` 已提供来源实现
- `image_downloader/storage.py` 已承担索引、元数据与重复跳过逻辑
- `image_downloader/reporting.py` 已承担运行摘要生成
- 测试已覆盖模型、来源、存储、CLI 与多来源集成行为

当前主要问题不在功能缺失，而在文档与计划仍有少量历史残留，容易让后续阅读者误判当前状态。

## 4. 方案选择

### 方案 A：只更新 README 和 SKILL

优点：改动最小。

缺点：开发计划文档仍会保留过时描述，无法完整体现当前 MVP 已完成。

### 方案 B：MVP 收口 + 文档去过时（推荐）

内容包括：

- 完整测试与冒烟验证
- README 对齐
- SKILL 对齐
- 计划文档中明显过时项去除或改写
- 最终中文提交

优点：范围适中，能完整表达“当前多来源 MVP 已完成”的事实。

缺点：需要比单纯改 README 多做一轮文档梳理。

### 方案 C：文档全面重写

优点：材料最整齐。

缺点：明显超出本轮“按 MVP 收口”的必要范围。

最终选择方案 B。

## 5. 执行设计

### 5.1 功能验证

分为两层：

1. 自动化验证
   - 运行当前完整 unittest 集合
   - 以 fresh 输出作为“测试通过”的证据

2. CLI 冒烟验证
   - 使用现有脚本执行一轮小规模下载
   - 验证脚本可运行、可输出摘要、可落盘到 `downloads/<keyword>/`

这里的真实验证目标是证明 MVP 主流程仍然正常，而不是继续做更大规模的实网压力测试。

### 5.2 文档整理策略

#### README.md

对齐以下内容：

- 当前真实目录结构
- 当前真实测试命令
- MVP 能力边界
- 多来源能力与去重能力说明

如果目录结构中遗漏了已有文件或测试模块，应补齐。

#### SKILL.md

对齐以下内容：

- 何时触发该 skill
- 当前真实依赖文件
- 当前执行命令模板
- 当前能力边界
- 对失败原因和保存目录的说明

重点是让 skill 描述与 README 和实际脚本行为一致。

#### 开发计划文档

本轮不是重写整份计划，而是修正其中最明显的历史残留，使其不再误导后续阅读者。

重点处理：

- 当前仓库并不存在但计划中仍写出的文件，如 `tests/test_pipeline.py`
- 已被当前脚本和模块吸收的职责，不再继续表述为“待实现”
- 用明确文字表明：当前阶段已按 MVP 标准完成多来源下载、历史去重、编号续写与测试验证

## 6. 验收标准

本轮完成必须同时满足：

1. 完整测试 fresh 通过
2. CLI 冒烟验证 fresh 成功
3. `README.md`、`SKILL.md`、开发计划文档与当前真实实现一致

## 7. 风险与处理

### 风险 1：文档描述超过当前真实能力

处理方式：所有文字以当前实际代码和测试结果为准，不写未实现能力。

### 风险 2：把计划文档改成“像最终总结”，失去计划上下文

处理方式：只清理过时项，并补充当前阶段完成状态，不抹掉原始设计意图。

### 风险 3：真实下载受第三方源站影响，导致下载数不稳定

处理方式：冒烟验证只证明流程可运行；若未下载满目标数量，需明确说明这是源站可访问性问题，不直接等同于脚本失败。
