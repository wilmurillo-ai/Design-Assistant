# 多来源图片下载器设计文档

- 日期：2026-04-10
- 项目：`bing-keyword-image-downloader`
- 设计主题：将当前 Bing 专用关键词图片下载脚本升级为“多来源图片下载器”

## 1. 背景

当前项目最初是一个面向 Bing 图片搜索结果的关键词批量下载脚本，核心能力包括：

- 按关键词请求 Bing 图片搜索页
- 从 HTML 中提取候选原图链接
- 分页收集更多候选结果
- 顺序下载图片到本地目录
- 当部分第三方源站返回 403、超时、SSL 错误或 404 时，跳过失败项并继续补位下载

该项目后续又被整理成了一个可复用的 Claude skill，用于帮助其他 agent 完成“按关键词从 Bing 公开图片中批量下载图片”的任务。

随着实际使用推进，项目暴露出两个核心问题：

1. **重复图片问题明显**
   - 当前逻辑主要是 URL 级去重。
   - 现实中常见“同一张图片对应不同 URL”“同图不同 CDN 地址”“同图不同尺寸或参数”“多次运行重复下载”等情况。
   - 因此会出现虽然下载数量增加了，但有效唯一图片数量并不理想的问题。

2. **单来源限制明显**
   - 当前实现强依赖 Bing 搜索结果页结构。
   - 只支持一个来源，扩展新来源时容易把抓取逻辑、去重逻辑、下载逻辑进一步耦合在一起。
   - 如果继续沿用单脚本堆叠功能的方式，后续维护和测试复杂度会快速上升。

基于这些问题，本次设计的目标不是继续给当前脚本局部打补丁，而是将项目整体升级为一个支持统一接口、可扩展多来源、具备系统化去重能力的图片下载工具。

---

## 2. 设计目标

本次设计的目标是将项目升级为一个：

> **支持多来源、统一候选模型、统一去重流水线、可保存历史索引与元数据的关键词图片下载器。**

升级后的系统应具备以下核心能力：

1. 支持按关键词从多个来源收集图片候选。
2. 将不同来源返回的数据统一标准化，避免上层逻辑依赖具体网站结构。
3. 在统一流水线中处理去重、筛选、排序与质量控制。
4. 将最终选定的图片稳定下载到本地，并保存必要的元数据。
5. 通过历史索引减少跨次运行的重复下载。
6. 为后续继续添加来源、过滤规则、报告能力和更强去重算法提供清晰的架构基础。

---

## 3. 范围边界

为了避免项目范围失控，第一阶段只关注“多来源统一架构 + 去重治理 + 结果管理”，明确**不**将以下能力纳入当前实施范围：

- 不做需要登录态的网站抓取
- 不做复杂反爬绕过与代理池系统
- 不做 GUI 图形界面
- 不做服务端 API 化
- 不做数据库服务化存储
- 不做分布式或大规模并发下载系统
- 不做图像识别分类
- 不默认引入高误判风险的感知相似去重作为核心能力

第一阶段重点是：

- 支持多来源统一接口
- 将 Bing 逻辑从脚本式实现重构为来源模块
- 引入标准去重体系
- 建立历史索引
- 保存元数据
- 提供更可解释的任务报告
- 改善测试结构，保证后续可维护性

## 4. 架构方案选择

### 4.1 被比较的三个方向

在本次设计中，主要考虑了三种方向：

#### 方案 A：单脚本扩展型
在现有主脚本中继续增加多个来源函数，依靠命令行参数选择不同来源。

**优点：**
- 改动最小
- 最快可做出可运行版本

**缺点：**
- 来源逻辑、去重逻辑、下载逻辑会进一步耦合
- 代码会越来越难理解和测试
- 不适合后续持续增加来源与功能

#### 方案 B：模块化来源型
将来源采集、统一建模、去重流水线、下载落盘、历史索引、结果报告拆分为独立模块。

**优点：**
- 结构清晰
- 更适合长期维护
- 更利于逐步扩展来源和治理能力
- 更容易分层测试
- 很适合当前项目“要扩来源，也要解决重复问题”的真实需求

**缺点：**
- 相较单脚本方案需要更明确的重构
- 早期文件和模块会变多

#### 方案 C：插件注册表型
进一步抽象为插件系统，来源、去重器、过滤器都通过注册表发现和注入。

**优点：**
- 扩展性最强
- 未来长期演化空间最大

**缺点：**
- 对当前项目规模来说抽象偏重
- 初期复杂度过高
- 容易为架构而架构

### 4.2 最终选择

本设计选择：

> **方案 B：模块化来源型架构**

选择理由如下：

1. 当前项目已经不只是一个临时脚本，而是在演化成可复用工具和 skill。
2. 用户的核心诉求是“多来源 + 重复治理 + 更多功能”，这些能力天然适合分层实现。
3. 模块化方案比单脚本更稳，比插件化更克制，适合作为当前阶段的工程平衡点。
4. 该方案可以为后续继续升级到更强的插件体系保留空间，但不会在第一阶段引入过多抽象负担。

## 5. 模块划分

本设计建议将项目拆分为以下核心模块。

### 5.1 CLI 层
负责：

- 接收命令行参数
- 解析任务配置
- 组织一次完整执行流程
- 输出最终摘要

CLI 的职责是调度，不直接承担来源解析、去重判定或下载细节。

### 5.2 Sources 层
负责：

- 根据关键词从具体来源收集图片候选
- 对每个来源的抓取逻辑进行隔离
- 将来源结果转换为统一候选结构

来源层的核心原则是：

> **来源负责发现候选，不负责决定最终保留哪些图片。**

### 5.3 Models 层
负责统一定义数据结构，例如：

- 图片候选对象
- 任务配置对象
- 下载结果对象
- 报告统计对象

统一建模可以避免不同来源之间使用不兼容字段，降低上层逻辑对来源实现细节的耦合。

### 5.4 Pipeline 层
负责系统治理，包括：

- 合并多个来源候选
- URL 去重
- URL 规范化
- 内容去重
- 尺寸/格式/大小过滤
- 排序与截断

该层的核心职责是：

> **统一决定哪些候选值得下载。**

### 5.5 Downloader 层
负责：

- 发起真实下载
- 推断扩展名
- 保存文件
- 重试与错误处理
- 返回落盘结果

下载器只负责“把已选中的候选稳定落盘”，不负责来源判断和策略决策。

### 5.6 Storage / Index 层
负责：

- 记录已下载历史
- 保存文件 hash
- 保存原始 URL 与规范化 URL
- 记录关键词、来源、文件路径和时间
- 支持跨次运行去重与断点续跑

该层是解决“重复下载问题”的关键基础设施。

### 5.7 Reporting 层
负责汇总与输出：

- 候选总数
- 各来源贡献数
- 被去重的数量
- 被过滤的数量
- 下载成功数与失败数
- 最终保存目录

报告层的目标是让任务结果可解释，而不是只给出一个“下载完成”的模糊结论。

## 6. 统一数据流

一次完整任务建议遵循如下数据流：

1. **CLI 接收参数并构造任务配置**
   - 例如关键词、目标数量、来源列表、抓取深度、输出目录、去重级别、是否保存元数据等。

2. **Sources 按统一接口收集候选**
   - 每个来源接收相同风格的输入，例如关键词、页数、数量。
   - 每个来源返回统一格式的候选列表。

3. **Pipeline 合并并治理候选池**
   - 合并多来源候选
   - 执行 URL 去重、规范化 URL 去重、过滤和排序
   - 控制最终进入下载环节的候选池规模

4. **Downloader 下载并落盘**
   - 对通过筛选的候选执行真实下载
   - 推断扩展名
   - 命名并写入文件
   - 处理错误和重试

5. **Storage / Index 记录历史**
   - 为成功结果写入索引
   - 记录 hash、URL、来源、路径等信息
   - 为后续跨次运行去重提供依据

6. **Reporting 输出结果摘要**
   - 输出各来源贡献、去重数量、下载成功数量、失败原因统计和保存路径

这一数据流的核心原则可以总结为：

> **来源负责发现，流水线负责决策，下载器负责落盘，索引负责记忆，报告负责解释。**

## 7. 去重策略设计

### 7.1 重复问题的层次

项目中的重复图片问题不应被视作单一问题，而应分为以下四层：

1. **原始 URL 重复**
   - 完全相同的图片地址。

2. **规范化 URL 重复**
   - 参数不同但本质资源相同的地址。

3. **内容完全相同重复**
   - 不同 URL 指向完全相同的文件内容。

4. **感知上高度相似重复**
   - 尺寸、压缩、裁剪不同，但肉眼几乎相同的图片。

### 7.2 第一阶段默认落地的去重能力

第一阶段推荐实现以下“标准去重”：

1. **原始 URL 去重**
2. **规范化 URL 去重**
3. **文件内容 hash 去重**
4. **历史索引去重（跨次运行）**

这套组合在复杂度和收益之间较为平衡，足以解决大部分现实重复问题。

### 7.3 感知相似去重的定位

感知相似去重（例如感知哈希）应作为后续增强，而不是第一阶段的默认能力。原因是：

- 实现复杂度更高
- 容易带来误判
- 测试和调参成本更高

在第一阶段，应优先把 URL 级、内容级和历史级去重做扎实。

### 7.4 单次运行去重与跨次运行去重

#### 单次运行中应做到：
- 来源内 URL 去重
- 跨来源候选合并后的 URL 去重
- 下载成功后的内容 hash 去重

#### 跨次运行中应做到：
- 根据规范化 URL 和已知 hash 检查历史重复
- 避免每次执行都重新下载同样内容的图片

### 7.5 去重级别建议

为兼顾灵活性和默认体验，后续可暴露不同去重级别概念：

- **基础去重**：原始 URL 去重
- **标准去重**：原始 URL + 规范化 URL + 内容 hash + 历史索引
- **严格去重**：在标准去重基础上进一步加入感知相似去重（后续增强）

默认推荐使用“标准去重”。

## 8. 第一阶段功能范围

第一阶段建议优先实现以下内容。

### 8.1 必做项

1. **统一 source 接口**
   - 所有来源模块返回统一候选结构。

2. **BingSource 重构**
   - 将现有 Bing 抓取逻辑从脚本中抽离为标准来源模块。

3. **DemoSource 或本地测试来源**
   - 提供一个不依赖网络的来源，用于测试统一接口、流水线与集成流程。
   - 该来源同时有助于向其他 agent 展示如何新增来源。

4. **多来源调度能力**
   - 支持在一次任务中选择一个或多个来源。

5. **标准去重能力**
   - 实现原始 URL、规范化 URL、内容 hash 与历史索引去重。

6. **历史索引**
   - 记录成功结果，用于跨次运行防重复。

7. **元数据保存**
   - 为每张成功保存的图片记录来源、关键词、URL、hash、路径、时间等信息。

8. **任务报告**
   - 输出来源贡献数、候选池大小、去重数量、下载成功数、失败数和保存目录。

### 8.2 可选项

第一阶段可作为增强但不是硬性要求的内容：

- 最小文件大小过滤
- 指定图片格式过滤
- 简单失败重试
- dry-run（只收集与去重，不下载）

### 8.3 暂缓项

以下内容建议在第一阶段之后再评估：

- 感知相似去重
- 大规模真实新来源接入
- 并发下载
- 图形界面
- 服务化接口
- 登录态网站支持
- 复杂反爬对抗

## 9. CLI 设计原则

### 9.1 保持单入口

第一阶段建议仍保留一个主入口，而不是拆成大量子命令。原因是：

- 便于学习和使用
- 便于 skill 复用
- 不会过早把简单场景复杂化

### 9.2 参数设计应围绕清晰能力

CLI 应围绕以下概念组织：

- `keyword`
- `--limit`
- `--sources`
- `--pages`
- `--output`
- `--dedupe-level`
- `--min-bytes`
- `--save-metadata`
- `--resume`
- `--dry-run`

这些参数不要求第一阶段全部实现，但结构设计应为此保留空间。

### 9.3 推荐默认行为

建议未来默认配置为：

- 默认来源：`bing`
- 默认去重：标准去重
- 默认保存元数据：开启
- 默认启用历史索引：开启
- 默认输出目录：按关键词组织

这能使工具行为更稳定，也更适合 skill 使用场景。

## 11. 测试策略

随着项目从单脚本升级为分层系统，测试必须从“脚本能跑”升级为“各层行为可验证”。

### 11.1 来源测试
验证每个来源是否：

- 能返回统一格式候选
- 能正确处理空结果
- 能正确处理分页
- 能遵守来源内的去重规则

这类测试应尽量依赖 mock，而不依赖真实网络。

### 11.2 流水线测试
重点验证：

- 原始 URL 去重
- 规范化 URL 去重
- 内容 hash 去重
- 历史索引去重
- 过滤逻辑
- 排序逻辑

这是解决重复图片问题的核心测试层，应作为重点投入。

### 11.3 存储测试
验证：

- 文件命名
- 元数据写入
- 索引更新
- 跨次运行重复判定
- 断点续跑基础能力

### 11.4 集成测试
验证一条完整链路：

- source 返回候选
- pipeline 去重和筛选
- downloader 落盘
- storage 更新索引
- reporting 汇总结果

### 11.5 冒烟测试
保留少量真实网络测试，用于确认：

- 主入口可执行
- 现有 Bing 抓取逻辑没有完全失效

真实网络测试不应作为单元测试主力，以免引入不稳定性。

- skill 触发条件

## 14. 风险与注意事项

### 14.1 来源失效风险
搜索引擎或素材站页面结构与访问策略可能变化，因此来源模块必须保持隔离，避免一个来源失效影响整体系统结构。

### 14.2 误去重风险
去重能力越强，误杀不同图片的风险越高。第一阶段应优先采用低误判的标准去重，避免过早启用强感知相似去重。

### 14.3 过度扩张风险
如果第一阶段同时追求多来源、并发下载、复杂 UI 和高级相似度算法，项目会变得过重，难以稳定落地。应坚持分阶段演进。

### 14.4 名称与定位滞后风险
当前项目名和 skill 名仍偏向 Bing 专用。若后续完成多来源升级，应择机统一项目命名与对外描述，以免能力和名称不一致。

---

## 15. 最终结论

本设计建议将当前项目从一个 Bing 专用下载脚本，升级为一个：

> **采用模块化来源型架构的多来源关键词图片下载器。**

其核心不是简单“多接几个站点”，而是围绕以下四项能力建立清晰基础：

1. 多来源统一接口
2. 统一候选模型
3. 去重与过滤流水线
4. 历史索引与元数据管理

第一阶段的成功标准不应只是“能下载更多图片”，而应是：

- 能在统一架构下继续扩来源
- 能显著减少重复图片
- 能解释下载结果与去重结果
- 能通过测试保障后续演进

这将为项目从“可运行脚本”升级为“可复用、可维护、可扩展的图片下载 skill”奠定基础。
