# Feishu Setup and Write Rules

本文档覆盖飞书文档写入时的规则和注意事项。安装和认证步骤见 `lark-cli-install-and-auth.md`。

## 一、写飞书前先遵守共享规则

优先读取并遵守：
- `lark-shared`
- `lark-doc`

核心原因：
- `--as user` 和 `--as bot` 行为差异很大
- 用户资源通常需要 `--as user`
- 写入操作前必须确认用户意图

## 二、默认身份

| 场景 | 推荐身份 |
|------|----------|
| 访问用户已有飞书文档 | `--as user` |
| 追加到用户自己的飞书文档 | `--as user` |
| 上传到用户云空间 | `--as user` |
| 企业级自动化 / bot 资源 | `--as bot`（需用户明确要求）|

不要默认使用 `--as bot` 去写用户自己的文档。

## 三、飞书写入前的最小确认项

在新环境中，写飞书前至少确认：

1. 是否已经有目标文档 ID / URL
2. 如果没有，是新建文档还是先让用户指定文档
3. 当前身份是否有权限访问该文档
4. 当前任务是 append、replace 还是 create
5. 写入对象是 doc/docx 还是 wiki 节点映射出的真实对象

**上下文复用：** 如果当前对话中已出现过目标文档的 URL 或 ID，直接复用，不必重复询问。

## 四、历史去重依赖

如果任务要求"继续追加""不要重复写已经记录过的平台"，必须先：
- 读取已有飞书文档内容
- 抽取历史候选名单（名称 + 规范化域名）
- 以历史文档为第一去重基线

如果拿不到已有文档内容：
- 不要声称已完成历史去重
- 最多只能完成"本轮内部去重"

## 五、写入格式纪律

### 1. 先汇总，再写入
不要边搜索边直接写飞书。标准顺序应为：
- 搜索
- 抓取与验证
- 去重
- 风险分层
- 生成完整 Markdown
- 再由主代理串行写入

### 2. 主代理串行写入
如果使用 subagent：
- subagent 只负责搜索、抓取、初步核验
- 最终飞书写入只能由主代理执行

### 3. 保持真实换行
写入 Markdown 时：
- 使用真实换行
- 不要依赖脆弱的 `\n` 拼接作为主要手段

### 4. 避免 shell 引号与反引号坑
在命令行拼接长 Markdown 时：
- 避免反引号包裹 URL
- 避免多层引号嵌套导致 shell 解释错误
- 优先构造稳定的单次写入内容

### 5. 写入命令参考
```bash
# 追加到文档末尾
lark-cli docs +update --doc "<doc_url>" --mode append --body "<markdown>" --as user

# 替换整个文档
lark-cli docs +update --doc "<doc_url>" --mode replace --body "<markdown>" --as user
```

## 六、无飞书能力时的降级方式

如果新环境没有飞书写入能力或认证未完成：
- 先输出完整结构化 Markdown
- 明确告诉用户"尚未写入飞书"
- 将结果保留为后续可直接粘贴或再调用飞书工具写入的版本
