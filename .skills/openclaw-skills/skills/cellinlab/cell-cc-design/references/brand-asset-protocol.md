# Brand Asset Protocol

> **Load when:** 任务涉及具体品牌、产品、公司、App、硬件或已命名服务
> **Skip when:** 只是抽象概念页、内部练习稿、完全不依赖真实品牌识别的探索
> **Why it matters:** 品牌任务的识别度首先来自真实资产，不来自“猜一个像它的配色”
> **Typical failure it prevents:** 产品根本没核实就开做；只抽象提色值却没有 logo / 产品图 / UI；做出来“像科技产品”但不像用户提到的那个品牌

## Rule 0: Verify Facts Before Styling

只要任务里出现了明确品牌或产品名，先核实事实，再设计。

至少确认：

- 这个产品 / 服务是否真实存在
- 是否已经发布
- 当前版本或主要形态是什么

如果这些事实不清楚，就先查，不要凭印象写。

## Asset Priority

识别度优先级通常是：

1. `Logo`
2. `产品图 / 官方渲染图`
3. `UI 截图 / 界面素材`
4. `颜色`
5. `字体`

因此，不要只做色板抽取就以为完成了品牌研究。

## Minimum Asset Checklist

遇到品牌任务时，优先向用户或项目索取：

- Logo：SVG 或高清 PNG
- 产品图：实体产品、包装、设备或官方渲染图
- UI 截图：如果是数字产品，这一项几乎是必需的
- 品牌色 / 字体 / design system
- 官方网站、品牌指南或 press kit

## Search Order

如果用户没有提供，按这个顺序补：

1. 官方网站或品牌页
2. 官方 press / media kit
3. 产品详情页
4. App Store / 产品截图
5. 用户给的竞品或参考

## Fallback Rules

如果真实资产不足：

- 明确告诉用户缺了什么
- 优先用诚实 placeholder，而不是编造“看起来像”的品牌物料
- 可以继续做结构和方向，但不要假装已经完成品牌化

## Deliverable Honesty

结果里至少说明：

- 用到了哪些真实资产
- 哪些资产是缺失的
- 当前更像“品牌方向稿”还是“品牌成稿”
