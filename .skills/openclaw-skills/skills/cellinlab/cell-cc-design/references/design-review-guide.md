# Design Review Guide

> **Load when:** 用户要求 review 一个设计、比较多个方向、或在交付前想做一次专家视角审查
> **Skip when:** 还在最初搭骨架，尚未形成可审查的设计
> **Why it matters:** 没有统一评审维度时，设计反馈很容易沦为“好像差点意思”
> **Typical failure it prevents:** 反馈空泛；找不到真正的问题；修复建议无法执行

## Review Dimensions

默认从 5 个维度看：

1. `Direction Fit`
   - 这个设计和目标气质、品牌或场景是否匹配
2. `Visual Hierarchy`
   - 主次是否清晰，用户知道先看哪里
3. `Craft Quality`
   - 对齐、间距、颜色、字重是否足够精细
4. `Functional Clarity`
   - 设计是否服务任务，而不是被装饰绑架
5. `Distinctiveness`
   - 是否有辨识度，还是落入常见模板

## Severity Levels

输出问题时，尽量按严重度排序：

- `Critical`
  - 不修会直接影响理解、使用或交付
- `Important`
  - 方向没错，但会明显拉低质量
- `Polish`
  - 细节优化项

## Output Structure

评审时优先给：

- 1 句总体判断
- 3-5 条按严重度排序的问题
- 最后给 `Quick Wins`

## Quick Wins Rule

如果用户时间有限，最后附上 3 条最高杠杆修复项：

- 改完就会明显提升整体质量
- 不要求大改结构
- 用户或下一个 agent 可以直接执行
