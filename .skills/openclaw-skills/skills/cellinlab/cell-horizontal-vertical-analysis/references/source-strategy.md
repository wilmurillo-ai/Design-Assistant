## 1. Source Priority

尽量按这个顺序找资料：

1. 官方页面、官方博客、原始文档、官方账号
2. 创始人 / 核心人物的一手表达
3. 原创采访、权威媒体、权威数据库
4. 社区讨论、论坛、Reddit、知乎、GitHub issues
5. 二手总结与转载，只作线索

规则：

- 关键事实不要只靠第 4 或第 5 类来源
- 同一种错误被多个二手来源重复，并不等于被验证

## 2. Search Patterns

不要只搜对象名，要把问题带进去。

### 2.1 纵向查询

- `对象名 + founded / launched / first release / 起源 / 创始人`
- `对象名 + year`
- `对象名 + funding / acquisition / crisis / pivot / roadmap`
- `对象名 + interview / podcast / keynote`

### 2.2 横向查询

- `对象名 + competitor / alternative / vs`
- `对象名 + review`
- `对象名 + reddit / github / 知乎 / x`
- `对象名 + pricing / market share / benchmark`

### 2.3 学术或技术对象

如果研究对象涉及 AI、算法、模型、技术范式、学术概念：

- 必查论文、技术报告或标准文档
- `arXiv` 是优先入口之一
- 还可以查官方论文页、会议官网、维护者文档

## 3. Sampling

不要只看最近热帖。

最少抽样：

- 最近 30 天的公开动态
- 稳定期样本
- 早期或转向期样本
- 跨平台或跨来源样本

需要用户口碑时，优先收：

- 高赞问题
- 重复出现的夸点和骂点
- 典型使用场景

## 4. Verification

这些信息尤其要谨慎验证：

- 成立 / 发布日期
- 融资、并购、营收、用户量
- 创始人背景
- 市场份额
- 技术路线归属
- 关键路线转向

推荐验证动作：

1. 先找一手表达
2. 再找另一种类型的来源交叉核实
3. 记录日期
4. 如果冲突，直接写冲突，不要擅自抹平

## 5. Sufficiency Check

写报告前自检：

- 纵轴能讲出完整故事吗
- 有没有明显断层年份或关键节点空白
- 横轴的主要对手是不是漏了
- 核心判断有没有至少两类来源支撑
- 有没有哪些结论其实只是“感觉像”

如果答案不稳，就回到检索阶段。

## 6. Research by Object Type

### 产品 / 技术

优先找：

- release notes
- docs
- GitHub
- benchmark
- 开发者口碑

### 公司

优先找：

- 官方公告
- 管理层访谈
- 招股 / 融资 / 并购信息
- 组织和战略转向信号

### 概念 / 范式

优先找：

- 首次提出者
- 关键论文或文章
- 概念的演化争论
- 与相邻概念的边界

### 人物

优先找：

- 长访谈
- 原始演讲
- 公开写作
- 关键决策记录
