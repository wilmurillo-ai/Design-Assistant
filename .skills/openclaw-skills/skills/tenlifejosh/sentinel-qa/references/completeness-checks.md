## 1. COMPLETENESS PHILOSOPHY

### Why Completeness Matters
An asset is only as strong as its least complete part. The best headline in the world
doesn't save a Gumroad listing with a broken download. The most beautiful cover art
doesn't save a book with missing chapters.

Completeness checks answer one question: **Is everything that should be here, here?**

Not: Is it good?
Not: Is it correct?
Just: Is it all there?

### Completeness vs Quality
- **Completeness**: All required parts exist
- **Quality**: Those parts are done well

Run completeness first (fast, binary). Run quality second (takes judgment).
A complete-but-poor asset can be improved. An incomplete asset cannot ship.

## 2. DIGITAL PRODUCT COMPLETENESS

### Gumroad Product: Required Elements
```
LISTING PAGE:
- [ ] Product title (not empty, not placeholder)
- [ ] Product description (200+ words)
- [ ] Cover image (1280×720px minimum)
- [ ] Price (non-zero)
- [ ] Category (selected)
- [ ] Tags (minimum 3)
- [ ] Thank-you page URL (set)

PRODUCT FILE:
- [ ] At least one downloadable file attached
- [ ] File is correct version (verify filename/date)
- [ ] File opens successfully
- [ ] All advertised sections/pages exist in file

OPTIONAL BUT EXPECTED IF PROMISED:
- [ ] Bonus files attached (if promised in description)
- [ ] Preview/sample PDF (if offered)
- [ ] Accompanying resources (worksheets, templates, if described)
```

### PDF Product: Required Pages/Sections
```
MINIMUM PAGE SET FOR PAID PDF:
- [ ] Cover page (title, author/brand)
- [ ] Copyright/legal page
- [ ] Table of contents (if 10+ pages)
- [ ] Introduction or "how to use this" section
- [ ] All content sections listed in description
- [ ] Appropriate closing/conclusion

FOR WORKBOOKS & JOURNALS:
- [ ] All worksheet/exercise pages exist
- [ ] Instructions for filling out are present
- [ ] Enough writing/response space for expected content
- [ ] Any referenced resources mentioned in exercises exist

FOR KDP BOOKS:
- [ ] Front matter (title page, copyright, TOC)
- [ ] All chapters in correct order
- [ ] Back matter (about author, other products, if applicable)
- [ ] Page count is even
- [ ] Last page is a right-hand page OR intentionally left blank
```

## 3. EMAIL CAMPAIGN COMPLETENESS

```
SINGLE EMAIL:
- [ ] Subject line (not empty)
- [ ] Preview text (or explicitly set to none)
- [ ] From name (correct sender name)
- [ ] From email address (authenticated domain)
- [ ] Email body (all sections present)
- [ ] Primary CTA (button or link)
- [ ] CTA destination URL (not placeholder)
- [ ] Unsubscribe link (required by law)
- [ ] Physical mailing address (required by CAN-SPAM)
- [ ] Brand logo or name in header
- [ ] Correct reply-to address

EMAIL SEQUENCE:
- [ ] All emails in sequence are present (correct count)
- [ ] Timing/delays set for each email
- [ ] Each email has unique subject line
- [ ] Sequence trigger condition set
- [ ] Sequence exit condition set (bought, unsubscribed, etc.)
- [ ] Welcome email is first (if new subscriber sequence)
- [ ] No email is missing from middle of sequence
```

## 4. SOCIAL MEDIA POST COMPLETENESS

```
TEXT POST:
- [ ] Caption written (not empty)
- [ ] CTA present or intentionally absent
- [ ] Hashtags added (if part of strategy)

IMAGE POST:
- [ ] Caption written
- [ ] Image attached (correct file)
- [ ] Image has alt text (for accessibility)
- [ ] Image is correct aspect ratio for platform

THREAD:
- [ ] All tweets/posts written (count matches plan)
- [ ] Thread is in correct order
- [ ] Last tweet has clear conclusion/CTA
- [ ] No placeholder "[next tweet]" items remaining
```

## 6. SOP COMPLETENESS

```
STANDARD OPERATING PROCEDURE:
- [ ] Purpose section (why does this exist?)
- [ ] Scope section (what does this cover?)
- [ ] Owner section (who runs this?)
- [ ] Prerequisites section (what's needed before starting?)
- [ ] Step-by-step procedure
- [ ] Verification section (how do you know it worked?)
- [ ] Common failures + recovery section
- [ ] Version number and date
- [ ] Review/update schedule

MINIMUM STEP REQUIREMENTS:
- [ ] Every step starts with an action verb
- [ ] Steps are numbered sequentially
- [ ] Decision points are explicit (if X then Y, else Z)
- [ ] Expected outcome stated for ambiguous steps
```

## 7. KDP BOOK COMPLETENESS

```
PAPERBACK INTERIOR:
Front Matter:
- [ ] Title page
- [ ] Copyright page
- [ ] Table of contents (if applicable)
- [ ] Dedication (if applicable)
- [ ] Preface/introduction (if applicable)

Body:
- [ ] All chapters/sections present
- [ ] Chapter titles match TOC
- [ ] No chapters missing from middle of book
- [ ] No obviously incomplete chapters ("more to come later")

Back Matter:
- [ ] About the author page
- [ ] Other products/books page (recommended)
- [ ] References/bibliography (if academic/nonfiction)
- [ ] Blank page if needed to make page count even

COVER:
- [ ] Front cover design complete
- [ ] Spine has title, author name, and publisher logo/name
- [ ] Back cover has description, author photo/bio, barcode area
- [ ] ISBN/barcode area reserved (lower right of back cover)

KDPMETADATA:
- [ ] Title exact match to cover
- [ ] Subtitle set (if applicable)
- [ ] Full author name
- [ ] Description (500+ words recommended)
- [ ] 7 keywords set
- [ ] 2 categories set
- [ ] Age/grade range set
- [ ] Pricing set for all markets
```

## 8. COMPLETENESS VERIFICATION PROTOCOL

### The Systematic Verification Process

**Step 1: Identify the asset type**
What kind of asset is this? Match to the appropriate checklist above.

**Step 2: Check for required elements first**
Run through the required elements checklist. Mark each:
- ✅ Present and appears complete
- ⚠️ Present but may be incomplete (flag for content QA)
- ❌ Missing entirely

**Step 3: Check for placeholder indicators**
Scan the entire asset for:
- Text in [brackets] or {{curly braces}}
- "Lorem ipsum" or "placeholder" text
- Empty fields
- "TBD", "Coming soon", "to be added"
- Items that clearly reference something not yet created

**Step 4: Cross-reference with description/brief**
If a product description says it includes "5 worksheets and 3 bonus checklists":
- Count the worksheets (are there 5?)
- Count the bonus checklists (are there 3?)

**Step 5: Issue completeness verdict**
- COMPLETE: All required elements present, no placeholders
- INCOMPLETE: Missing elements or placeholders found (list them specifically)
- CANNOT DETERMINE: Need additional information to verify

## 9. THE MISSING ELEMENT LOG

### How to Log Completeness Failures

When an asset fails completeness review:

```
## COMPLETENESS FAILURE REPORT

Asset: [Name]
Asset Type: [Type]
Date Reviewed: [Date]
Reviewer: Sentinel Agent

MISSING ELEMENTS:
1. [Element name]: [What's missing, where it should be, what to add]
2. [Element name]: [What's missing, where it should be, what to add]

PLACEHOLDER ELEMENTS:
1. [Location]: "[Exact placeholder text found]" → needs to be replaced with [description of what's needed]

ASSIGNMENT:
Assigned to: [Agent who created the asset]
Due: [Urgency level]
Re-review required: YES

NOTES FOR CREATOR:
[Any helpful context about why these elements are required or how to approach filling them in]
```

### Tracking Completeness Failures Over Time
If the same type of element is missing repeatedly → update the checklist.
Completeness checks should get better over time, not stay the same.

For example: if covers are missing from 3 consecutive Gumroad products → that's a process gap, not individual error.
Fix: Add cover creation to the product build SOP, before Sentinel review.
