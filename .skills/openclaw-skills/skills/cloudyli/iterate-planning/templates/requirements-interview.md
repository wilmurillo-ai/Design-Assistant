# 需求访谈模板 (Requirements Interview)

## 目标
在动手之前，真正理解用户要解决什么问题、达成什么 Outcome。

## 原则

**问 Outcome，不要问功能。**

| ❌ 不要问 | ✅ 要问 |
|-----------|--------|
| "需要哪些功能？" | "解决了这个问题之后，你的生活/工作会有什么不同？" |
| "用户能做什么？" | "用户真正想完成什么？为什么这件事重要？" |
| "有什么技术要求？" | "怎么才算成功？有哪些衡量标准？" |

## 访谈流程

### Step 1: 识别 JTBD（Jobs to Be Done）

**核心问题**：
1. "你想解决什么问题？"
2. "为什么这个问题重要？"
3. "解决之后会带来什么不同？"

**记录格式**：
```
JTBD: [一句话描述用户真正想要完成的Outcome]
来源: [用户原话或背景]
重要性: [P0/P1/P2]
```

### Step 2: 拆分 Topics of Concern

**每个 Topic = 一个独立的关注维度**

用"一个句子，不带and"测试：
- ✅ "颜色提取系统分析图片识别主色调"
- ❌ "用户系统处理认证、 profiles、和计费" → 这是3个Topics

**Topic 命名**：[what it does]-[who benefits]

### Step 3: 每个 Topic 写 Spec

创建 `specs/topic-[name].md`：

```markdown
# [Topic 名称]

## 需求描述
[一句话描述这个 Topic 要做什么]

## 用户视角
[从用户角度描述：用户看到这个功能会怎样]

## 验收标准 (Acceptance Criteria)
- [ ] [标准1：可测试、可观察]
- [ ] [标准2]
- [ ] [标准3]

## 边界情况 (Edge Cases)
- [情况1]：期望行为是...
- [情况2]：期望行为是...

## 不在范围内 (Out of Scope)
- [明确不做的]
```

---

## 访谈检查清单

访谈结束时确认：

- [ ] 每个 JTBD 都有对应 Topics
- [ ] 每个 Topic 有完整 spec
- [ ] 每个 spec 有明确的验收标准（可测试）
- [ ] 边界情况已识别
- [ ] 不在范围内的已明确排除
- [ ] 用户确认 specs 准确反映了他们的需求

---

## 提示词模板

### 开场
```
我来帮你理清需求。在开始之前，我需要先理解你真正想解决的问题。

请回答：
1. 你想解决什么问题？
2. 解决这个问题为什么重要？
3. 解决之后会带来什么改变？

我会通过追问来澄清细节，直到需求完备。
```

### 追问 Topic
```
关于"[刚才提到的点]"，我想多了解一些：
- 具体场景是什么？
- 现在是怎么做的？
- 理想情况应该怎样？
```

### 确认完成
```
需求访谈到这里比较完备了。让我确认一下我们讨论的结果：

[总结每个 Topic 和 验收标准]

这些准确吗？有没有遗漏或需要调整的地方？
```
