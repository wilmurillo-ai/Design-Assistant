# Claw-List v2 — Overview

Self-hosted todo/task management for OpenClaw agents. Agents manage tasks via a REST API using `web_fetch` — no direct database access, no credentials in agent context. A web UI provides human oversight across all agents and lists.

This is a full rewrite of v1, which had agents writing directly to PostgreSQL via a Python CLI bundled with the skill.

---

## Architecture

```
Agent → web_fetch → FastAPI (port 8100) → PostgreSQL
Browser → nginx UI (port 8101) → Caddy → FastAPI
```

Caddy sits in front of both containers. The UI calls `/api/*` relative to its own origin — Caddy strips the prefix and proxies to the API. No CORS issues, no URL configuration needed in the UI.

Without Caddy: expose ports directly (`0.0.0.0:8100`, `0.0.0.0:8101`) and point agents at `http://<host>:8100`.

| Column | Type | Notes |

## Permissions Model

- `own` — agent reads and writes only its own lists and items
- `all` — agent can read all agents' lists and items, but can only write to its own

Scope is stored in `agents`. Change it via the web UI or `PUT /admin/agents/{id}`. Write operations always enforce ownership regardless of scope.

## Agent Identity and Self-Configuration

Agents are identified by UUID, not name — prevents collisions when the same agent type runs on multiple VMs.

**Config resolution order:**
1. Env vars `CLAW_LIST_AGENT_ID` / `CLAW_LIST_URL` (set per-agent in `openclaw.json` skills.entries)
2. `claw-list.conf` in the skill dir
3. First-run self-configuration: agent generates UUID, prompts user for display name and API URL, writes conf, self-registers

On first use the skill POSTs to `/admin/agents` to self-register (no auth header required), then writes `REGISTERED=true` to conf. Subsequent sessions skip this check entirely — zero token overhead.

For multi-agent setups on the same machine, use `openclaw.json` skills.entries to inject per-agent env vars rather than a shared conf file. See `docs/api.md` for the full config example.

## Skill Design

The skill is a single `SKILL.md` file — no Python, no dependencies, no direct DB access.

SKILL.md is kept minimal (~35 lines) to avoid context bloat — it's loaded on every skill invocation. Full API reference is in `docs/api.md`, which the agent reads on demand.

**Trigger phrases:** "the list", "show me the list", "todo", "add task", "add to my list", "mark done", "what's due", "my tasks", "create a list"

**Notes field:** agents store conversation context, links, or references in `items.notes` when the user says things like "put that in the notes".

## Web UI

Single-page vanilla JS app served by nginx. Dark terminal aesthetic (IBM Plex Mono, green accent on near-black).

- **Agents view** — list all agents, register new (UUID auto-generated), edit scope, delete
- **Lists view** — all lists across all agents, create, delete, click to open items
- **Items view** — full CRUD, toggle done, priority displayed as colour-coded filled squares (1–5), notes preview

---

## Input Sanitisation

All data is sanitised at every layer:

- **API** — SQLAlchemy ORM throughout (parameterised queries), Pydantic validates every request body, errors return generic messages (no raw DB errors or stack traces)
- **UI** — all API data passed through `escHtml()` before DOM insertion, `innerHTML` never used with raw API values
- **Skill** — user conversation text stored in `notes` is treated as untrusted free-text by both API and UI

Any new endpoint, field, or UI component must follow the same rules from day one.

## Deployment

See `server/README.md` for full setup instructions. Quick summary:

```bash
cp server/.env.example server/.env
# Set POSTGRES_PASSWORD (required), CLAW_LIST_URL (informational)

# Bundled Postgres:
docker compose --profile db up -d --build

# External Postgres:
# Set POSTGRES_HOST/USER/DB in .env, then:
docker compose up -d --build
```

Caddy snippet in `server/caddy-snippet.conf`. Without Caddy: change port bindings to `0.0.0.0` and point agents at `http://<host>:8100`.

## Running Tests

No Docker or Postgres needed — tests use SQLite in-memory:

```bash
cd server/api
pip install -r requirements.txt pytest httpx
pytest test_api.py -v
```

20 tests covering agent CRUD, scope enforcement, and list/item operations.

## Known Gaps / Future Work

- **Auth** — no authentication currently. Internal network only. API and UI are designed so token auth can be added without breaking changes.
- **Pydantic deprecation** — `class Config` style should be migrated to `model_config = ConfigDict(from_attributes=True)` before next major release.
- **utcnow() deprecation** — swap for `datetime.now(UTC)` for Python 3.12+ compatibility.

## Decisions Log

| Decision | Reason |
|---|---|
| UUID as agent_id, not name | Prevents collision between same-named agents on different VMs |
| Self-configuration on first use, not install script | Works for multi-agent setups without shared conf; no uuidgen dependency |
| REGISTERED flag in conf file | Tracks registration state without any API call on subsequent sessions |
| scope=all can read but not write others' data | Visibility without risk of cross-agent data corruption |
| `/api/*` proxied through Caddy | Single origin, no CORS config, no URL needed in UI |
| Bundled Postgres behind compose profile | Optional — users with existing Postgres skip it cleanly |
| Vanilla JS, no framework | No build step, single file, easy to audit and modify |
| SQLite for tests | No external dependencies to run the test suite |
| server/ bundled inside skill dir | Users get full self-hosting stack on install, no separate repo |
| SKILL.md minimal + docs/api.md on demand | Avoids context bloat — SKILL.md is loaded on every invocation |
