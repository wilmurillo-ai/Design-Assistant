# 结构化问题解决方法论

## 为什么需要这个

AI agent 在遇到问题时，容易陷入"快猜快改"的陷阱——看到现象立刻形成假设并动手修改，结果改错了再猜再改，反复试错的总时间远超一次深入分析的时间。

这份方法论提供一个结构化的问题解决流程，适用于非显而易见的问题（需要猜测原因、涉及多组件交互、修改有风险的场景）。

---

## 适用范围

### 应该使用本方法论的场景
- 问题原因不明显，需要猜测
- 涉及多个组件/系统交互
- 修改可能有副作用或不可逆
- 之前尝试修复失败过
- 用户明确要求分析后再动手

### 不需要的场景（直接修复即可）
- 一眼能看出的问题（typo、明显配置错误、已知 pattern）
- 有明确错误信息且对应唯一修复方案
- 用户明确说"直接改"

---

## 流程：六步法

### 第一步：定义问题（Define）

**目标**：把模糊的"出了问题"变成精确的问题陈述。

**做什么**：
1. **记录现象**：什么发生了？什么时候发生的？影响范围多大？
2. **记录预期**：本来应该怎样？
3. **记录差异**：现象和预期之间的 gap 是什么？
4. **确认可复现性**：是偶发还是必现？触发条件是什么？

**输出格式**：
```
问题陈述：[一句话描述]
现象：[具体表现]
预期：[正确行为应该是什么]
影响：[谁受影响、严重程度]
可复现：[是/否，触发条件]
```

**常见错误**：
- ❌ 把假设当成问题（"origin 被污染了" 是假设，不是问题）
- ❌ 跳过这一步直接开始修（连问题都没定义清楚就动手）
- ✅ "webchat 的回复同时出现在钉钉群聊里" 才是问题

---

### 第二步：诊断分析（Diagnose）

**目标**：找到根因（root cause），不是表象原因。

**做什么**：

#### 2.1 画数据流 / 调用链
从输入到输出，数据经过哪些环节，每个环节做了什么：
```
输入 → 环节A → 环节B → 环节C → 输出
         ↓         ↓         ↓
       正常？     正常？     正常？
```

#### 2.2 逐环节验证
对每个环节，检查实际值是否符合预期：
- 读日志、读配置、读源码、打印中间状态
- **不要猜**，要看到实际数据

#### 2.3 缩小范围
用二分法或排除法：
- 哪个环节的输出开始不正常？
- 上游正常 + 本环节异常 = 问题在这里

#### 2.4 确认根因
问自己三个问题：
1. **为什么会这样？**（机制层面的解释）
2. **改掉这个点，问题一定消失吗？**（充分性）
3. **不改这个点，有没有其他原因也能导致同样的问题？**（唯一性）

**诊断工具优先级**：
1. 日志 / 错误信息（最快）
2. 中间状态检查（读数据库、配置文件、session 状态）
3. 源码追踪（最可靠但最慢）
4. 实验验证（构造最小复现场景）

**常见错误**：
- ❌ 只看了一个环节就下结论
- ❌ 用"可能是"开头的假设直接当结论
- ❌ 改了一个地方没效果，不反思为什么，继续猜下一个
- ✅ 画完调用链，逐环节验证，定位到确切代码行

---

### 第三步：方案设计（Design）

**目标**：提出多个可选方案，对比评估，选最优。

**做什么**：

#### 3.1 列出候选方案（至少 2 个）
- 每个方案用一句话描述"做什么"
- 包括"不修复 / 绕过"选项（有时候最好的方案是不改）

#### 3.2 多维度评估

| 维度 | 说明 |
|------|------|
| **有效性** | 能否彻底解决问题？还是只治标？ |
| **风险** | 会不会引入新问题？影响其他功能？ |
| **复杂度** | 改动量多大？涉及几个组件？ |
| **可逆性** | 改错了能回退吗？ |
| **持久性** | 长期有效还是临时方案？会被覆盖吗？ |
| **副作用** | 对其他功能/场景的影响？ |

#### 3.3 给出推荐
- 说明推荐理由
- 说明选择的 trade-off

**输出格式**：
```
方案 A：[描述]
  优点：...
  风险：...
  推荐度：⭐⭐⭐

方案 B：[描述]
  优点：...
  风险：...
  推荐度：⭐⭐

推荐方案 A，因为 [理由]
```

**常见错误**：
- ❌ 只提一个方案就开始做（没有对比就没有信心）
- ❌ 方案评估只看"能不能解决"，忽略副作用和风险
- ❌ 推荐方案没有说清楚 trade-off

---

### 第四步：确认执行（Execute）

**目标**：安全地实施修复，有回退预案。

**做什么**：

#### 4.1 执行前检查
- [ ] 根因已确认（不是猜测）
- [ ] 方案已评估（不是拍脑袋）
- [ ] 用户已确认（外部操作/有风险的修改）
- [ ] 回退方案准备好（改错了怎么办）

#### 4.2 最小化修改
- 只改必要的部分，不顺手改别的
- 一次只改一个变量，方便验证

#### 4.3 变更记录
- 改了什么、为什么改、改之前是什么
- 方便回退和后续审计

**常见错误**：
- ❌ 一次改多个地方，分不清哪个生效了
- ❌ 改之前不记录原始值，无法回退
- ❌ 改完不告知用户（尤其是需要 restart 的操作）

---

### 第五步：验证确认（Verify）

**目标**：确认问题真的解决了，没有引入新问题。

**做什么**：

#### 5.1 直接验证
- 复现原来的触发条件，确认问题不再出现
- 检查修改点的实际状态（不要"应该没问题了"，要看到证据）

#### 5.2 回归验证
- 相关功能是否正常？
- 其他场景是否受影响？

#### 5.3 持久性验证
- 重启后是否还有效？
- 下次触发条件出现时是否还有效？

**常见错误**：
- ❌ 改完就说"应该好了"，不验证
- ❌ 只验证了修复场景，没检查相关功能
- ❌ 一次验证就宣布修复，没考虑持久性

---

### 第六步：总结复盘（Review）

**目标**：提炼教训，避免同类问题再犯。

**做什么**：

#### 6.1 事件总结
- 问题是什么、根因是什么、怎么修的
- 从发现到修复花了多长时间
- 其中有效时间和浪费时间各多少

#### 6.2 过程反思
- 哪些步骤做对了？
- 哪些步骤走了弯路？为什么？
- 如果重来一次，怎样能更快？

#### 6.3 规则提炼
- 这个问题暴露了什么通用模式？
- 需要新增什么检查项 / 规则？
- 写入 `.learnings/` 或 `AGENTS.md`

**输出格式**：
```
## 复盘：[问题名称]
- 耗时：X 分钟（有效 Y 分钟，弯路 Z 分钟）
- 根因：[一句话]
- 修复：[一句话]
- 弯路：[走了什么弯路，为什么]
- 教训：[提炼的规则]
```

## 速查：问题分级与应对

| 级别 | 特征 | 应对 |
|------|------|------|
| **L1 显而易见** | 一眼能看出原因和修复方案 | 直接修复 |
| **L2 需要排查** | 原因不确定，但范围有限 | 简化版流程：定义 → 诊断 → 修复 → 验证 |
| **L3 复杂/高风险** | 多组件交互、修改有风险、之前修错过 | 完整六步流程 |

**判断标准**：
- 如果你需要说"可能是..."才能给出原因 → 至少 L2
- 如果涉及 2 个以上组件的交互 → 至少 L2
- 如果改错了会影响用户 / 数据安全 → L3
- 如果已经尝试修复失败过 → 升级到 L3

---

## 反模式清单（Anti-patterns）

| 反模式 | 表现 | 正确做法 |
|--------|------|----------|
| **猜了就改** | 看到现象立刻形成假设并动手 | 先画调用链，逐环节验证 |
| **只看一端** | 只检查输入或输出，不看中间过程 | 端到端追踪数据流 |
| **表象修复** | 改了能看到的异常值，没找为什么异常 | 问"为什么会变成这个值" |
| **改了再说** | 改了看效果，没效果再猜 | 确认根因后再改 |
| **一次改多处** | 同时改好几个地方 | 一次一个变量 |
| **宣布修复** | 改完说"好了"不验证 | 必须看到证据 |
| **忘记回退** | 不记录原始值，改错了回不去 | 先备份/记录，再修改 |

## 沟通原则

在问题解决过程中与用户的沟通：

1. **定义阶段**：确认问题理解一致（"你说的问题是 X，对吗？"）
2. **诊断阶段**：同步进展，不要沉默太久（"我在查 Y 环节，发现了 Z"）
3. **方案阶段**：给选择题不给填空题（"方案 A 和 B，推荐 A 因为..."）
4. **执行阶段**：有风险的操作先确认（"需要 restart，确认吗？"）
5. **验证阶段**：请用户协助验证（"你那边看一下是否正常"）
6. **全程**：承认不确定性（"我不确定是不是这个原因，先验证一下"比"就是这个原因"靠谱得多）
