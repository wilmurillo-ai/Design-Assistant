## 设计目标

4A 架构图的核心不是“把信息连起来”，而是“把信息放在正确层级，再表达层间映射关系”。

输出时优先追求：

- 层次清晰
- 边界明确
- 跨层可追踪
- 同层元素类型一致

不要用简单流程图替代分层架构图。

## 主图规范

主图是 4A 总览分层架构图，必须显式包含 4 个容器：

- 业务架构层
- 应用架构层
- 数据架构层
- 技术架构层

推荐布局：

- 上下四层，业务在上、技术在下
- 或左右四栏，但必须顺序固定且有清晰层标题

每层至少包含：

- 业务层：角色、场景、能力域、业务规则中的关键项
- 应用层：渠道、核心应用、共享服务、外部系统中的关键项
- 数据层：数据域、主实体、数据流/所有权中的关键项
- 技术层：运行平台、集成方式、中间件、安全/观测中的关键项

主图里的连线应该表达：

- 业务能力由哪些应用承接
- 应用读写哪些数据域
- 应用由哪些技术能力支撑

不要让主图变成一条从左到右的步骤链。

## 业务架构图规范

业务架构图关注业务对象，不关注技术调用顺序。

优先包含：

- 关键角色
- 关键场景
- 业务能力域
- 业务规则 / KPI / 风险约束

推荐布局：

- 角色区
- 场景区
- 能力区
- 规则/KPI 区

## 应用架构图规范

应用架构图关注系统边界、应用分工和接口协作。

优先包含：

- 渠道入口
- 核心应用/Skill
- 共享服务
- 第三方系统 / 外部系统
- 系统间接口关系

推荐布局：

- 渠道层
- 业务应用层
- 共享平台层
- 外部系统层

## 数据架构图规范

数据架构图关注数据对象与治理边界，不只关注单次处理流程。

优先包含：

- 数据域
- 主实体
- 数据来源
- 数据消费者
- 数据所有权 / 主数据口径
- 治理与合规约束

推荐布局：

- 来源区
- 核心数据区
- 治理区
- 消费区

## 技术架构图规范

技术架构图关注技术支撑层次，不关注业务步骤。

优先包含：

- 运行环境
- 接入层 / 网关 / 集成层
- 中间件 / 执行引擎
- 基础设施
- 安全、审计、可观测性

推荐布局：

- 接入层
- 服务与编排层
- 执行与集成层
- 基础设施与共性能力层

## 跨层映射规则

至少要满足以下三条：

- 每个核心业务能力都能映射到一个或多个应用承接点
- 每个核心应用都能映射到一个或多个关键数据对象
- 每个关键应用或共性服务都能映射到技术支撑

在主图中，跨层连线只表达映射，不表达完整时序。

## 反模式

- 把 4A 主图画成阶段路线图
- 把业务架构图画成用户点击步骤
- 把应用架构图画成 API 调用时序图
- 把数据架构图画成单次 ETL 流程
- 把技术架构图画成部署命令或操作手册
