# Skill 沉淀决策流程

本文档详细描述 Agent 如何判断是否应该将任务沉淀为 Skill，以及沉淀过程中的各种决策点。

> **工具无关设计**：本流程不依赖任何特定工具，执行时请使用你环境中可用的文件操作工具。

## 详细判断逻辑

### 复杂度评估标准

| 指标 | 阈值 | 权重 |
|------|------|------|
| 工具调用次数 | ≥ 5 次 | 高 |
| 错误重试次数 | ≥ 2 次 | 高 |
| 涉及工具种类 | ≥ 3 种 | 中 |
| 多步骤依赖 | 存在 | 中 |
| 需要用户输入 | ≥ 2 次 | 低 |

**组合规则：** 任一高权重指标满足，或两个及以上中权重指标满足，即判定为"复杂任务"。

### 价值判断标准

| 场景 | 价值等级 | 说明 |
|------|---------|------|
| 克服了未文档化的错误 | 高 | 这种经验最值得沉淀 |
| 用户纠正后成功 | 高 | 用户偏好需要记录 |
| 发现更优工作流 | 高 | 比现有方法更好 |
| 通用任务模式 | 中 | 可能被其他场景复用 |
| 特定环境配置 | 中 | 特定于用户环境 |
| 一次性操作 | 低 | 不太可能复用 |

### 脚本创建判断标准

LLM 在以下情况下**更可能**创建独立脚本：

```
┌─────────────────────────────────────────────────────────────────┐
│ 代码特征                          │ 创建脚本的可能性            │
├─────────────────────────────────────────────────────────────────┤
│ 代码超过 30 行                     │ ████████░░ 80%             │
│ 包含函数/类定义                    │ █████████░ 90%             │
│ 有复杂错误处理                     │ ███████░░░ 70%             │
│ 需要导入第三方库                   │ ██████░░░░ 60%             │
│ 可以参数化复用                     │ █████████░ 90%             │
│ 经过多次调试迭代                   │ ████████░░ 80%             │
```

## 负面案例（不应创建 Skill）

### 案例 1：简单查询

```
用户："Python 的 list comprehension 怎么写？"
Agent：解释语法 + 给出示例

→ 不创建 Skill（这是知识查询，不是任务流程）
```

### 案例 2：失败的任务

```
用户："帮我部署到生产环境"
Agent：尝试了多种方法，都因权限问题失败

→ 不创建 Skill（任务未成功完成）
```

### 案例 3：已有 Skill 覆盖

```
用户："帮我提交一个 PR"
Agent：使用了 github-pr-workflow skill 完成

→ 不创建 Skill（已有 Skill 覆盖，除非发现需要补充的陷阱）
```

### 案例 4：用户拒绝

```
Agent："是否要保存为 Skill？"
用户："不用了，这是一次性的"

→ 不创建 Skill（尊重用户意愿）
```

## 正面案例（应该创建 Skill）

### 案例 1：复杂调试成功

```
用户："帮我配置 CUDA 开发环境"
Agent：
  - 检测系统 → 发现驱动版本不匹配
  - 尝试安装 → 遇到依赖冲突
  - 解决冲突 → 遇到内核模块问题
  - 最终成功

→ 询问用户是否保存（复杂度高 + 克服了多个陷阱）
```

### 案例 2：用户纠正

```
用户："帮我格式化这个 JSON"
Agent：使用 jq 处理
用户："不对，我们团队用 prettier"
Agent：改用 prettier，成功

→ 询问用户是否保存（用户偏好值得记录）
```

### 案例 3：发现更好方法

```
用户："帮我批量重命名文件"
Agent：
  - 最初用 for 循环
  - 发现 rename 命令更简洁
  - 最终用 rename 完成

→ 可以提议保存（发现了更优方案）
```

### 案例 4：明确请求

```
用户："帮我写个部署脚本，以后还会用到"

→ 必须创建 Skill（用户明确表达了复用意图）
```

## 更新现有 Skill 的触发条件

使用 Skill 过程中发现以下情况时，应在任务完成前使用文件编辑工具更新：

| 发现 | 动作 |
|------|------|
| 步骤缺失 | 使用文件编辑工具补充步骤 |
| 命令过时 | 使用文件编辑工具更新命令 |
| 新的陷阱 | 使用文件编辑工具添加到 Pitfalls 部分 |
| 更好的方法 | 使用文件编辑工具重写相关部分 |
| 依赖变化 | 使用文件编辑工具更新 Prerequisites |

**常用编辑操作映射：**

| 操作类型 | 可能的工具 |
|----------|-----------|
| 小范围修改 | `replace_in_file`, `patch_file`, `str_replace_editor` |
| 大范围重写 | `write_file`, `write_to_file`, `create_file`（覆盖） |
| 追加内容 | `append_to_file`, 或读取后重写 |
