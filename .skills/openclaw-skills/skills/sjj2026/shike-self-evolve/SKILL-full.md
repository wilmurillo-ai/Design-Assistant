# self-evolve.skill — Agent自进化机制

> **教一遍，记一辈子。用得越多，越懂你。**

你是一个会进化的Agent。每次对话，你都有机会变得更好——记住用户的偏好、沉淀踩坑的经验、优化技能的方法论。

本skill定义了**何时进化、如何进化、进化到哪**的完整协议。

---

## 核心心智模型

### 模型1：遗忘曲线对抗

**问题**：Agent每次对话从零开始，用户教过的事下次又忘。
**原理**：艾宾浩斯遗忘曲线——不复习就遗忘。进化机制就是Agent的"复习系统"。
**执行**：识别值得长期记住的经验 → 写入持久化文件 → 下次对话自动加载。
**失效条件**：用户只是随口一说（如"今天好累"），不是长期意图。别把噪音当信号。

### 模型2：三层沉淀漏斗

**问题**：所有经验都往MEMORY.md塞，导致文件膨胀、质量下降。
**原理**：经验有不同重量——轻量偏好放记忆层，方法论放技能层，全局规则放规范层。漏斗过滤，越往上越严格。
**执行**：L1快速审批 → L2需Darwin评分 → L3需双重确认。
**失效条件**：层级判断错误（把临时偏好升级为全局规则，或把重要规则只放在记忆层）。不确定时，宁可放在L1等自然升级。

### 模型3：审批即训练

**问题**：Agent自动记忆容易产生噪音（"用户喜欢吃火锅"）。
**原理**：AutoClaw的核心洞察——所有进化需用户审批。审批不是负担，是训练信号。批准=正样本，忽略=负样本。
**执行**：生成进化请求卡片 → 用户选择进化/忽略/修改 → 只保留批准的。
**失效条件**：进化请求太频繁，用户审批疲劳。频率控制：每日≤5次，每周≤15次，重要内容不受限。

## 进化层级协议

### Layer 1：对话记忆层 → MEMORY.md

**写入条件**：用户偏好、行为模式、经验碎片
**审批门槛**：重要性≥medium，用户简单确认即可
**写入格式**：
```
### 🧬 自进化 [来源]
> 经验内容（一句话，精炼表述）
```
**写入位置**：MEMORY.md的「行为偏好与规则」或当日日记section
**Git提交**：`🧬 L1进化: <内容摘要>`

**示例**：
- ✅ "以后回复简洁" → 写入MEMORY.md行为偏好
- ✅ "记住我喜欢Markdown表格" → 写入MEMORY.md行为偏好
- ❌ "今天心情不好" → 噪音，不进化

### Layer 2：技能优化层 → SKILL.md

**写入条件**：踩坑经验、方法论改进、边界补充
**审批门槛**：需Darwin评分验证，≥70分才能写入

**Darwin评分调用方式**（新增）：
```
Step 1: 读取 /root/.openclaw/workspace/skills/darwin-skill/SKILL.md
Step 2: 按Darwin评分rubric评估拟写入内容
        - 结构维度（60分）：Frontmatter/工作流/边界/检查点/指令/资源/架构
        - 效果维度（40分）：实测表现
Step 3: 计算总分，返回评分结果
Step 4: if 分数 ≥ 70：允许写入；else：拒绝并说明改进建议
```

**Darwin评分fallback**（如果无法调用Darwin skill）：
- 由Agent干跑验证：模拟3个测试prompt，自评效果
- 在进化日志中标注：`eval_mode=dry_run`
- 干跑评分 ≥ 75 才允许写入（提高阈值补偿无外部验证）

**写入格式**：
```
### 🧬 踩坑经验 [日期]
> 失败现象：...
> 根因分析：...
> 解决方案：...
> 失效条件：...
```
**写入位置**：目标skill的SKILL.md「边界条件」section
**Git提交**：`🧬 L2进化: <skill名> - <改进摘要>`

**示例**：
- ✅ "网页截图多次重试" → 写入browser-skill的边界条件
- ✅ "Git提交前先检查status" → 写入相关skill的检查点
- ❌ "这次格式不太对" → 太模糊，需先提炼为具体规则

### Layer 3：行为规范层 → CLAUDE.md

**写入条件**：全局行为约束、开发规范、安全边界
**审批门槛**：重要性必须high + 双重确认（先确认层级，再确认内容）
**写入格式**：
```
### 🧬 规范进化 [来源]
> 规则内容（精确、无歧义）
> 适用范围：所有Bot / 指定Bot
> 生效时间：即日起
```
**写入位置**：CLAUDE.md的「开发准则」或「行为约束」section
**Git提交**：`🧬 L3进化: <规则摘要>`（重要！必须提交）
**额外操作**：通知所有Bot「CLAUDE.md已更新，请重新读取」

**示例**：
- ✅ "所有网页必须添加备案信息" → 写入CLAUDE.md全局约束
- ✅ "Git提交前必须检查工作目录" → 写入CLAUDE.md开发准则
- ❌ "这个技能包不好用" → 不是规范，不适合L3

## 触发机制协议

### 触发器A：关键词识别

**监听关键词**：

| 级别 | 关键词 | 触发行为 |
|------|--------|---------|
| **紧急** | 以后、永远、每次、必须、一定、禁止 | 对话中**立即**弹出进化卡片 |
| **标准** | 记住、下次、别再、不要再、记得 | 对话中**提示**进化机会 |
| **温和** | 倾向、偏好、喜欢、习惯 | 记录到候选池，**批量审批**时提出 |

**识别流程**：
1. 扫描用户消息，匹配关键词
2. 提取核心意图（去除触发词后的内容）
3. 评估目标层级（含"技能"→L2，含"所有/全局/规范"→L3，默认L1）
4. 生成进化请求卡片

**误触发保护**：
- 关键词在引用/复述中不触发（如"他说以后..."）
- 否定句不触发（如"不要记住这个"）
- 疑问句降级（如"以后要不要简洁？"→温和级）

**混合意图协议**（新增）：
- 检测到临时状态（"今天心情不好"）+ 长期意图（"以后少说点开心的"）混合时：
  1. 分离临时状态和长期意图
  2. **降级处理**（紧急→温和，标准→温和）
  3. 在进化卡片中标注"⚠️ 受临时情绪影响，建议谨慎审批"
  4. 用户可选择「进化」（保留）或「暂缓」（等情绪稳定后再决定）

**语义模糊追问机制**（新增）：
- 当提炼结果不够具体时（如"少说点开心的"→是指emoji？正能量语句？闲聊？），触发追问：
  - 卡片中标注："⚠️ 意图模糊，建议澄清"
  - 追问用户："你说的'少说点开心的'具体指？(1)减少emoji (2)减少正能量语句 (3)减少主动闲聊"
  - 用户澄清后再提炼为具体规则

### 触发器B：踩坑检测

**监听指标**：

| 指标 | 阈值 | 触发行为 |
|------|------|---------|
| 工具调用次数 | >5次/对话 | 提示效率优化 |
| 同操作重试 | >3次 | 提示流程优化 |
| 用户纠正 | >2次 | 提示行为调整 |
| Git回滚 | 任何1次 | 提示规范优化 |
| 操作超时 | 任何1次 | 提示边界补充 |

**识别流程**：
1. 对话中持续监控指标（标记但不打断）
2. **时机策略**：超阈值时内部标记，对话自然结束后统一处理（不打断用户流程）
3. 分析失败根因
4. 提炼可复用的踩坑经验
5. 生成进化请求卡片（默认L2）

**根因确认协议**（新增）：
- Agent提出根因假设，不直接写入
- 在进化卡片中展示："根因假设：<...>"
- 用户可选择「确认根因」或「修正根因：<用户输入>」
- 只有用户确认后，根因才写入踩坑经验

**踩坑经验提炼模板**：
```
失败现象：<具体操作>失败，重试<N>次
根因分析：<为什么会失败>
解决方案：<下次应该怎么做>
失效条件：<这个方案什么情况下不管用>
```

### 触发器C：定期复盘

**触发时机**：
- 每日HEARTBEAT检查时
- 每日22:00科技任务完成后
- 用户主动请求"复盘"

**复盘流程**：
1. 读取最近记忆文件 `memory/YYYY-MM-DD.md`
2. 识别重复出现的模式（≥2次出现的同类问题）
3. 评估每条经验的价值（重要性×复用性）
4. 过滤噪音（临时状态、情绪表达、无关信息）
5. 生成批量进化清单
6. 用户批量审批

**复盘输出格式**：
```markdown
🧬 **每日复盘 - <日期>**

**候选进化**：
| # | 内容 | 来源 | 建议层级 | 价值 |
|---|------|------|---------|------|
| 1 | ... | 关键词 | L1 | ⭐⭐⭐ |
| 2 | ... | 踩坑 | L2 | ⭐⭐ |
| 3 | ... | 噪音 | - | ❌ |

→ 回复「进化 1,2」批准指定项
→ 回复「全部进化」批准所有有效项
→ 回复「跳过」不进化
```

## 进化请求卡片协议

### 卡片格式（唯一标准格式）

```
🧬 进化请求 #<编号>

触发：<触发器A/B/C> | 级别：<紧急/标准/温和>
建议记忆：> <一句话精炼表述>
⚠️ <受临时情绪影响/意图模糊/根因待确认>（如有）
目标：L<1/2/3> → <目标文件路径>
评估：重要性<H/M/L> 复用性<H/M/L> 风险<无/低/中>
→ 「进化」确认 | 「进化 L2」变更层级 | 「忽略」跳过 | 「修改」调整 | 「暂缓」延迟决定
```

### 卡片示例

**关键词触发**：
```
🧬 进化请求 #1

触发：关键词"以后" | 级别：紧急
建议记忆：> 回复简洁为主，重点先行，避免冗长解释
目标：L1 → MEMORY.md
评估：重要性H 复用性H 风险无
→ 「进化」确认 | 「忽略」跳过
```

**踩坑触发（含根因确认）**：
```
🧬 进化请求 #2

触发：踩坑-工具调用7次 | 级别：标准
建议记忆：> 网页截图先检查browser状态，确认targetId匹配再操作
⚠️ 根因假设：targetId不匹配导致操作指向错误tab
目标：L2 → browser-skill/SKILL.md
评估：重要性H 复用性M 风险低
→ 「确认根因」确认写入 | 「修正根因：<你的分析>」更正 | 「忽略」跳过
```

**混合意图触发（含追问）**：
```
🧬 进化请求 #3

触发：关键词"以后" | 级别：温和（由紧急降级）
建议记忆：> 少说点开心的
⚠️ 受临时情绪影响（"心情不太好"）
⚠️ 意图模糊："少说点开心的"具体指？(1)减少emoji (2)减少正能量语句 (3)减少主动闲聊
目标：L1 → MEMORY.md
评估：重要性L 复用性M 风险低
→ 「进化」确认 | 「暂缓」延迟决定 | 「修改」调整内容
```

## 用户审批协议

### 审批命令

| 用户回复 | Agent行为 |
|---------|----------|
| `进化` | 按建议层级执行写入+Git提交 |
| `进化 L1` / `L2` / `L3` | 覆盖建议层级，执行写入 |
| `忽略` | 跳过，记录为"用户拒绝"（负样本） |
| `修改 <新内容>` | 用新内容替换后重新请求审批 |
| `批量进化` | 批量审批所有候选项 |
| `暂缓` | 延迟决定，保留到下次复盘时重新评估 |
| `确认根因` | 确认L2踩坑根因，继续写入流程 |
| `修正根因 <新根因>` | 用用户提供的根因替换Agent假设，继续写入流程 |

### 层级审批规则

| 层级 | 审批要求 | 额外操作 |
|------|---------|---------|
| L1 | 用户单次确认 | 写入MEMORY.md + Git提交 |
| L2 | 用户确认 + Darwin评分≥70 | 写入SKILL.md + Darwin优化 + Git提交 |
| L3 | 用户双重确认（层级+内容） | 写入CLAUDE.md + 通知所有Bot + Git提交 |

### L3双重确认流程

L3修改全局约束，必须双重确认：
1. **第一重**：确认层级（"这会影响所有Bot，确认升级到L3？"）
2. **第二重**：确认内容（展示拟写入的具体文本，"确认写入此内容？"）
3. 两重都通过才执行，任一不通过则降级为L1或取消

## 频率控制协议

### 限额规则

| 时间维度 | 普通内容上限 | 重要内容上限 |
|---------|------------|------------|
| 每日 | 5次 | 不限 |
| 每周 | 15次 | 不限 |

### 噪音过滤规则

**自动过滤**（不需要用户审批直接跳过）：
- 临时状态："今天好累"、"现在很忙"
- 情绪表达："开心"、"生气"
- 无关信息："天气不错"、"想吃火锅"

**需要用户判断**（弹出卡片让用户决定）：
- 模糊偏好："我觉得这样更好"（具体什么更好？）
- 条件规则："有时候应该..."（什么时候？）
- 矛盾规则：与现有规则冲突

### 进化计数器

Agent在每次进化后，在进化日志中记录计数。每日0:00重置日计数，每周一0:00重置周计数。

**计数器存储位置**（新增）：
- 文件：`/root/.openclaw/workspace/memory/evolution-counters.json`
- 格式：
```json
{
  "date": "2026-04-18",
  "daily_count": 3,
  "weekly_count": 12,
  "last_reset_daily": "2026-04-18T00:00:00",
  "last_reset_weekly": "2026-04-15T00:00:00"
}
```
- 每次进化时更新计数，每日深夜自动重置

## 进化冲突处理

### 冲突检测

写入前检查是否与现有规则冲突：
1. 读取MEMORY.md搜索相似内容
2. 新规则与旧规则矛盾？→ 弹出冲突提示
3. 新规则与旧规则重复？→ 合并而非新增

### 冲突解决

| 冲突类型 | 解决策略 |
|---------|---------|
| 新旧矛盾 | 展示冲突，用户选择保留哪个 |
| 内容重复 | 合并为一条，保留更精确的表述 |
| 层级冲突 | 新规则更适合更高层级？建议升级 |

### 冲突卡片格式

```
⚠️ 进化冲突检测

新规则：> <新内容>
已有规则：> <旧内容>（来源：<文件:行号>）

→ 「覆盖」新规则替换旧规则
→ 「保留」保持旧规则，放弃新规则
→ 「合并」合并为一条
```

## 回滚协议

### 回滚触发条件

1. 进化后效果变差（用户反复纠正）
2. Darwin评分下降（L2专属）
3. 用户主动请求回滚

### 回滚流程

1. `git log --oneline -5` 查看最近进化提交
2. 确认要回滚的commit
3. `git revert <commit>` 创建回滚commit
4. 记录回滚原因到进化日志
5. 通知用户回滚完成

### 回滚安全边界

- 只回滚进化相关的commit（`🧬`前缀）
- 不回滚其他类型的commit
- 回滚后重新评估，避免反复进退

## 安全边界

### 不可进化的内容

以下内容**永远不能**通过自进化机制写入：
- 安全约束（如"不要执行rm -rf"、"不要泄露API Key"）
- 系统级规则（如"不要修改/etc/下的系统文件"）
- 核心身份信息（如IDENTITY.md中的名字和角色）
- 其他Bot的专属规则（如shike3的投研规则不应写入shike1的记忆）

### 进化内容限制

- 每条进化内容≤100字（精炼表述，不要长篇大论）
- 不包含代码（代码属于skill的scripts/，不属于进化）
- 不包含敏感信息（API Key、密码、Token）

## 诚实边界

本skill的局限性：

1. **不是真学习**：进化写入的是文本规则，不是模型权重。规则越多，加载越慢，边际效益递减。
2. **噪音无法完全消除**：即使用户审批，也可能批准了短期有效的规则，长期变成噪音。需要定期清理。
3. **冲突可能漏检**：依赖文本搜索检测冲突，语义相似的冲突可能漏掉。
4. **L3风险最高**：修改CLAUDE.md影响所有Bot，一次错误进化可能全局污染。务必双重确认。
5. **过度进化陷阱**：进化频率过高会让用户审批疲劳，进化质量下降。宁可少进化，不要多噪音。
6. **无法验证效果**：除L2有Darwin评分外，L1和L3的效果只能靠用户主观感受。
7. **语义模糊依赖用户澄清**（新增）：用户说"少说点开心的"，Agent无法精确理解，需要追问。追问失败则无法进化。
8. **根因分析可能错误**（新增）：Agent分析的踩坑根因可能是错的，即使有用户确认环节，用户也可能不了解技术细节。建议L2进化后观察效果，及时回滚。

---

## 内在张力

1. **进化速度 vs 进化质量**：进化越快越懂用户，但噪音也越多。频率控制是平衡点。
2. **用户审批 vs 自主性**：全部审批太累，全部自主太危险。分级审批是折中——L1宽松，L3严格。

## 回答工作流（Agentic Protocol）

当检测到进化机会时，按以下协议执行：

```
Step 1: 识别
  - 扫描用户消息，匹配关键词 / 检查踩坑指标 / 定期复盘触发
  - 输出：进化机会（触发类型+原始内容）

Step 2: 提炼
  - 去除触发词，提取核心意图
  - 评估目标层级（L1/L2/L3）
  - 质量预评估（重要性×复用性）
  - 噪音过滤
  - 输出：进化请求卡片

Step 3: 请求审批
  - 弹出进化请求卡片
  - 等待用户回复（进化/忽略/修改）
  - L3需双重确认
  - 输出：审批结果

Step 4: 冲突检测
  - 搜索MEMORY.md/CLAUDE.md，检查是否与现有规则冲突
  - 如有冲突，弹出冲突卡片
  - 输出：冲突解决结果

Step 5: 执行写入
  - 按目标层级写入对应文件
  - L2额外触发Darwin评分
  - L3额外通知所有Bot
  - 输出：写入确认

Step 6: Git提交
  - commit message: "🧬 L<n>进化: <内容摘要>"
  - 输出：commit hash

Step 7: 日志记录
  - 追加到 evolution-log.md
  - 更新进化计数器
  - 输出：日志确认
```

**关键约束**：
- 每步都有明确的输入和输出
- 任何一步失败，中止进化，不跳过
- Step 3和Step 4可以并行（优化效率）

## 实测验证

### 测试Prompt 1：关键词触发（happy path）

**用户输入**："以后回复简洁一点，不要长篇大论"

**预期Agent行为**：
1. 识别关键词"以后"（紧急级）
2. 提炼核心意图："回复简洁为主，避免冗长解释"
3. 评估目标层级：L1
4. 弹出进化请求卡片
5. 用户回复「进化」→ 写入MEMORY.md + Git提交

**判断标准**：
- ✅ 正确识别关键词和级别
- ✅ 提炼内容精炼（≤50字）
- ✅ 层级判断正确（L1）
- ✅ 卡片格式标准
- ❌ 如果忽略关键词或层级判断错误

### 测试Prompt 2：踩坑触发（复杂场景）

**场景**：Agent执行网页截图任务，调用browser工具6次，其中3次重试失败

**预期Agent行为**：
1. 检测到工具调用>5次 + 重试>3次
2. 分析失败根因（targetId不匹配）
3. 提炼踩坑经验："网页截图先检查browser状态，确认targetId匹配"
4. 评估目标层级：L2
5. 弹出进化请求卡片
6. 用户回复「进化 L2」→ Darwin评分 + 写入SKILL.md + Git提交

**判断标准**：
- ✅ 正确检测踩坑指标
- ✅ 根因分析具体（不是"操作失败"这种废话）
- ✅ 踩坑经验含失效条件
- ✅ L2需触发Darwin评分
- ❌ 如果踩坑经验太模糊或缺失效条件

### 测试Prompt 3：噪音过滤（边缘case）

**用户输入**："今天心情不太好，以后少说点开心的"

**预期Agent行为**：
1. 识别关键词"以后"（紧急级）
2. 但内容包含"今天心情不太好"（临时状态）
3. **混合意图协议**：检测到临时状态+长期意图混合
4. **降级处理**（紧急→温和）
5. **语义模糊追问**："你说的'少说点开心的'具体指？(1)减少emoji (2)减少正能量语句 (3)减少主动闲聊"
6. 弹出进化请求卡片，标注"⚠️ 受临时情绪影响"+"⚠️ 意图模糊，建议澄清"

**判断标准**：
- ✅ 识别出临时状态和长期意图的混合
- ✅ 降级处理（紧急→温和）
- ✅ 触发语义模糊追问
- ✅ 标注情绪影响警告
- ❌ 如果直接按紧急级进化（可能后悔）
- ❌ 如果完全过滤掉（可能确实有长期价值）
- ❌ 如果没有追问直接写入模糊规则
