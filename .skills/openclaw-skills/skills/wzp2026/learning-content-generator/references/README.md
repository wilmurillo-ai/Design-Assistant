## 一、核心原则（详细）

### 1.1 来源优先原则

| 优先级 | 来源类型 | 标识 | 使用说明 |
|-------|---------|------|---------|
| P1 | 国家标准（GB/T） | ✅ 权威来源 | 直接引用原文，注明标准号 |
| P1 | 中国官方文档 | ✅ 权威来源 | 工信部/教育部/国家密码管理局等官方文件 |
| P1 | 中国高校MOOC | ✅ 权威来源 | 清华/北邮/上交大等课程 |
| P2 | 学术期刊（知网） | ⚠️ 参考来源 | 参考引用，注明来源期刊 |
| P2 | 技术社区文档 | ⚠️ 参考来源 | GitHub/OWASP等，需核实 |
| P3 | 大模型推断内容 | 🤖 推断内容 | 必须注明，逻辑推导无原始文献 |

**引用比例要求：**
- 权威来源：≥30%（可直接引用原文的知识点）
- 参考来源：≤40%（有参考但需加工）
- 推断内容：≤30%（需特别标注）

### 1.2 来源标注方式

```
【来源】✅ 权威来源 | GB/T 42752-2023 国家标准（相关领域）
【来源】⚠️ 参考来源 | 中国知网《计算机学报》2023年
【来源】🤖 推断内容 | 大模型逻辑推断，建议结合权威资料核实
```

**内容中标注方式：**
- 直接引用：用引号标出，注明"来源：国家标准GB/T xxx"
- 参考内容：括号注明"参考：xxx"
- 推断内容：末尾标注"【注】此部分为逻辑推断，建议核实"

### 1.3 国内合规性（必须项）

**优先使用：**
- ✅ 国家标准（GB/T xxx-xxxx）
- ✅ 中国官方文档（工信部/教育部/国家密码管理局）
- ✅ 中国大学MOOC平台课程
- ✅ 中国知网/万方学术文献

**谨慎使用（需标注来源）：**
- ⚠️ GitHub/StackOverflow（技术内容，非政治敏感）
- ⚠️ 国际技术文档（技术中立，需核实）

**禁止使用：**
- ❌ 境外政治敏感内容
- ❌ 未经核实的外文来源（无中文译本）
- ❌ 商业推广内容

### 1.4 搜索优先原则

**搜索来源优先级：**

| 优先级 | 来源类型 | 示例 | 标注 |
|--------|---------|------|------|
| P1 最高 | 国家标准 | openstd.samr.gov.cn（GB/T 标准全文） | ✅ 权威来源 |
| P1 最高 | 政府机关 | miit.gov.cn、moe.gov.cn、gov.cn | ✅ 权威来源 |
| P1 最高 | 高校/学术 | edu.cn、ac.cn 域名下的官方平台 | ✅ 权威来源 |
| P2 中等 | 学术数据库 | CNKI知网、万方、国家哲学社会科学文献中心 | ⚠️ 参考来源 |
| P2 中等 | 权威技术组织 | W3C/IETF/ISO等国际标准组织文档 | ⚠️ 参考来源 |
| P3 最低 | 行业社区 | 技术博客、开发者社区 | ⚠️ 参考来源（需核实） |
| ❌ 禁止 | 非权威网络 | 个人博客、贴吧、论坛、微博、公众号等 | 禁止使用 |

**搜索格式示例：**

```
site:openstd.samr.gov.cn {关键词} 国家标准
site:gov.cn {关键词} 政策/法规
site:edu.cn {关键词} 课程/教材
site:miit.gov.cn {关键词} 标准/文件
site:cnki.net {关键词} 学术论文
```

**搜索-推论流程：**

```
① 使用 site:gov.cn / site:edu.cn / site:openstd.samr.gov.cn 搜索
   → 找到权威来源 → ✅ 权威来源
② 扩大范围搜索（不限 site）
   → 找到支撑来源 → ⚠️ 参考来源
③ 搜索不到或不确定 → 大模型推论
   → 标注为 🤖 推断内容
④ 推论内容必须注明"建议结合权威资料核实"
```

## 二、双技能协作模式（详细）

### 2.1 协作架构

```
learning-path-builder（技能1）
    ↓ 生成知识图谱（JSON/CSV/JSON-LD）
    ↓
learning-content-generator（技能2）
    ↓ 接收：完整知识图谱 + 指定节点ID
    ↓
    生成该节点的个性化学习内容
```

### 2.2 调用参数说明

| 参数 | 说明 | 作用 |
|------|------|------|
| `target_node_id` | 目标L3叶节点ID | 指定要生成哪个节点的内容 |
| `knowledge_graph` | 完整知识图谱JSON | 提供全局上下文 |
| `parent_l2_node` | 父L2节点信息 | 理解中模块层级的上下文 |
| `parent_l1_node` | 父L1节点信息 | 理解大领域层级的上下文 |
| `level` | 节点层级(1/2/3) | 决定内容深度和难度 |
| `source_type` | 来源类型 | 决定内容风格（权威/参考/推断） |
| `total_hours` | 总学时规划 | 辅助判断内容深度 |

### 2.3 输入示例

```json
{
  "job_id": "nlp_kg_01",
  "target_node_id": "L3_f1_1",
  "knowledge_graph": { /* 完整知识图谱JSON */ },
  "context": {
    "l1_node": { "id": "L1_f1", "name": "人工智能算法选型及调优" },
    "l2_node": { "id": "L2_f1_1", "name": "机器学习算法选型" },
    "l3_node": { "id": "L3_f1_1", "name": "机器学习算法选型原则", "hours": 0.5 },
    "level": 3,
    "source_type": "权威来源",
    "total_hours": 80,
    "direction": "自然语言及语音处理产品实现"
  }
}
```

### 2.4 为什么需要完整图谱？

同一个概念在不同层级应有不同深度：
- L1节点内容：大而全的概述，面向零基础学习者
- L2节点内容：模块化的系统知识，面向有一定基础的学习者
- L3节点内容：精确深入的细节，面向要参加认证考试的学员

如果只传L3节点本身，缺少上下文，同一个"深度学习"概念在不同图谱里会生成一样的内容，失去了层级差异化的价值。

## 三、层级感知与差异化策略（详细）

### 3.1 同一概念在不同L层级的差异

| 层级 | 内容深度 | 目标读者 | 内容风格 |
|------|---------|---------|---------|
| L1 | 大而全的概述 | 零基础学习者 | 通俗易懂，引入概念，激发兴趣 |
| L2 | 模块化的系统知识 | 有基础的学习者 | 结构清晰，涵盖原理和方法论 |
| L3 | 精确深入的细节 | 要参加认证考试的学员 | 考试导向，精确到考点，可操作 |

### 3.2 上下文感知的重要性

同一个概念在不同图谱里应有不同内容：
- 同样是"深度学习"，在AI芯片领域和NLP领域侧重点不同
- 同样是"模型训练"，初级要求会调库，中级要求懂原理会优化
- L3节点内容必须结合父L1/L2的领域上下文，不能泛泛而谈

### 3.3 生成时必须参考的信息链

```
L1大领域（做什么的）
  → L2模块（在哪个环节）
    → L3叶节点（具体什么技能点）
```

**示例**：L3节点"Transformer架构"：
- 如果父L1是"NLP核心模型"，内容侧重：Transformer在NLP中的应用、BERT/GPT原理
- 如果父L1是"语音处理核心"，内容侧重：Transformer在语音识别中的应用、Conformer等

## 四、进度通知机制（详细）

### 4.1 分段执行流程

```
【第一步】读取知识图谱文件
  → 写入：/tmp/content_kg_{job_id}.json
  → 通过当前会话发送通知：「图谱读取完成：L1×N / L2×N / L3×M个节点」

【第二步】分析内容生成计划
  → 写入：/tmp/content_plan_{job_id}.json
  → 通过当前会话发送通知：「内容规划完成：计划生成N个学习单元，预计总字数X万」

【第三步】逐节点生成学习内容
  → 写入：/tmp/content_progress_{job_id}.json（增量）
  → 每完成一个节点通知：「[节点名] 已完成（X/Y）」

【第四步】生成随堂自测题
  → 通过当前会话发送通知：「随堂测试题生成完成：共N道选择题」

【第五步】验证与汇总
  → 写入：{name}_content_verification.md
  → 通过当前会话发送通知：「✅ 内容生成完成！总字数X万，节点数N个」
```

### 4.2 超时与续接策略

| 情况 | 处理方式 |
|------|---------|
| 某步骤执行中 | 等待完成，不设硬性超时 |
| 某步骤完成通知后 | 等待用户确认或自动继续 |
| 某步骤失败 | 通过当前会话发送通知失败原因，从该步骤重试 |
| 用户中断 | 保留已生成的文件，下次可续接 |

## 五、长文本分片策略（详细）

### 5.1 触发条件

单个知识点内容预计超过8000字时，自动分片。

### 5.2 处理方式

```
1. 估算总字数
2. 按~7000字/片拆分
3. 每片独立生成（file-based，避免截断）
4. 最后合并为完整内容
5. 标注分片位置：[...接上文] / [...接下文]
```

## 七、随堂自测题生成规范（详细）

### 7.1 核心要求

- 每知识点生成3~5道单选题
- 题型：概念题、计算题、应用题
- 来源类题目占一定比例，强化来源意识

### 7.2 题目模板

**概念题**：
```
以下关于[知识点]的描述，正确的是？
- A. 描述1
- B. 描述2
- C. 描述3
- D. 描述4
```

**应用题**：
```
[场景描述]，最适合使用哪种方法？
- A. 方法A
- B. 方法B
- C. 方法C
- D. 方法D
```

**来源题**：
```
[知识点]的权威来源是？
- A. 标准A
- B. 标准B
- C. 标准C
- D. 标准D
```

### 7.3 题目质量标准

- 答案唯一、无歧义
- 选项长度相近
- 不使用"以上都是"等模糊选项
- 适当包含来源验证类题目
