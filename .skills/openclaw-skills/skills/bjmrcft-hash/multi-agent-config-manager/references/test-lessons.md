# 多代理编排引擎 — 端到端测试复盘

## 第一轮测试（2026-03-31，红相股份虚假陈述案）
- **结果**：3个代理全部超时或产出不全
- **问题**：超时120s太短、子代理不写文件、无降级机制
- **修复**：超时预设表、强制write工具、降级协议

## 第二轮测试（2026-04-01，航天长峰）
- **结果**：2/3成功，1个缺失
- **问题**：exec写文件卡审批、消息风暴
- **修复**：强制write+read验证、禁止无关输出

## 第三轮测试（2026-04-03，新能源汽车出海）
- **结果**：3/3并行成功 + Critic审核通过 + 聚合完成
- **问题1**：Critic超时0 tokens → 重试×1.5解决
- **问题2**：数据矛盾未发现 → Critic发现4处严重矛盾
- **问题3**：主代理跳过聚合 → 协议规定不可跳过
- **修复**：强制聚合阶段、数据统一口径

## 第四轮测试（2026-04-03，装修对比）
- **结果**：3/3并行 + Critic审核 + 聚合完成，全流程通过
- **新功能验证**：分层目录、看板系统、自动触发
- **发现的问题**：
  1. 主代理静默（需催促才继续）→ 改为 completion event 自动触发
  2. 缺少任务看板 → 新增计划看板 + 执行看板
  3. 输出文件混乱 → 三层目录隔离
  4. progress/ 目录无用 → 删除

## 关键教训
1. **文件存在 = 唯一成功判据**：子代理称"已完成"不算，必须文件存在
2. **降级比失败好**：Critic超时→重试→成功，比放弃好
3. **流程不可跳过**：聚合阶段即使100%成功也必须执行
4. **AI的"核实"≠真实核实**：Critic发现的矛盾只能在已有信息内推理，无法外部验证
