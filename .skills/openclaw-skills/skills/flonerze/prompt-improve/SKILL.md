# 提示优化器

一项将模糊、不明确或未充分指定的用户请求转化为详细、可操作提示的技能，这些提示可以由AI助手或人类协作者有效执行。

## 为什么这项技能重要

许多用户难以清晰地表达他们的需求。他们可能会说"帮我做个东西"、"这个怎么弄"或"我需要一个工具来..."。这些模糊的陈述缺乏有效协助所需的特异性。本技能基于优秀prompt的四个核心要素构建结构化提示，弥合模糊请求与可执行指令之间的差距：

1. **角色设定** - 明确AI应扮演的专业角色和领域
2. **任务描述** - 清晰定义具体任务和期望结果
3. **上下文信息** - 提供必要的背景、环境和限制条件
4. **指令与约束** - 设定明确的规则、格式和质量标准

通过这四要素框架，确保生成的提示具有明确性、可操作性和高质量。

## 关键区别：仅生成提示，不执行任务

**这是本技能最重要的原则**：当调用此技能时，只根据用户的描述生成prompt，不做任何其他多余操作。与其他技能不同，此技能：

- **不执行**用户请求的实际任务
- **不探索**文件系统或代码库  
- **不收集**额外信息或上下文
- **不调用**任何工具进行分析

技能的唯一输出就是优化后的prompt文本本身。例如，当用户说"了解代码的架构"时，技能生成一个用于分析代码架构的prompt模板，而不是实际分析代码。

## 何时使用此技能

当遇到以下任何模式时，请主动使用此技能：

- **上下文极少**：用户提供1-2句话而没有背景信息
- **目标模糊**："让它更好"、"修复这个"、"添加些东西"
- **未定义范围**：没有明确的边界或成功标准
- **隐含假设**：用户假设你知道他们的意思
- **混合语言**：中英文混合的随意表达，如"帮我写个script"
- **交付成果不明确**："给我个方案"而没有指定格式

**重要**：不要等到用户明确要求优化提示。如果他们的请求感觉不完整或模糊，请激活此技能以确保在继续之前理解他们的真实需求。

## 如何使用此技能

当用户提供模糊或不明确的请求时，**立即且仅**根据用户输入生成基于四要素框架的结构化提示。**绝对不要执行任何探索性操作**，技能的唯一且全部输出就是优化后的提示本身。

**核心原则（必须严格遵守）**：
1. **仅生成提示**：技能的唯一功能是生成用于执行任务的提示文本，不执行任务本身
2. **零探索**：不读取文件、不分析代码库、不调用任何工具来探索实际环境
3. **纯文本转换**：基于用户输入的字面含义和常见模式直接生成提示，不做任何额外信息收集
4. **输出即产品**：最终输出是一个完整的、可直接使用的prompt文本，没有其他任何内容

### 自动生成流程

1. **解析用户输入**：将用户的模糊请求作为核心任务描述
2. **推断关键要素**：基于输入内容推断角色设定、上下文信息等（使用常见模式，不探索实际环境）
3. **填充模板**：将分析结果填充到四要素框架模板中
4. **输出优化提示**：生成完整的结构化提示供用户使用

### 提示模板

```
# [清晰、具体的任务标题]

## 1. 角色设定
[明确AI应扮演的角色和专业领域，例如：数据分析专家、软件架构师、内容策略师、技术顾问等]

## 2. 任务描述
[清晰、详细地描述需要完成的具体任务，包括主要目标、期望结果和交付成果]

## 3. 上下文信息
- **背景**：[项目背景、现状、问题描述、业务需求]
- **相关人员**：[涉及的用户、团队、利益相关者及其需求]
- **技术环境**：[现有技术栈、工具、平台、系统限制]
- **业务场景**：[使用场景、业务流程、价值主张]

## 4. 指令与约束
- **输出格式**：[期望的交付成果格式，如代码文件、文档、设计稿、配置等]
- **技术要求**：[必须或推荐使用的技术、框架、库、版本]
- **质量要求**：[性能指标、准确性标准、用户体验、安全合规]
- **限制条件**：[时间、预算、资源、兼容性、法规等限制]
- **成功标准**：[具体、可衡量的评估指标，如何验证解决方案有效性]
```

**用户说**："帮我做个数据分析工具"

**直接生成的优化提示**：
```
# 为销售团队创建自动化数据分析工具

## 1. 角色设定
数据分析工具开发专家，专注于业务数据自动化处理解决方案

## 2. 任务描述
为销售团队创建一个用户友好的数据分析工具，自动化月度销售报告流程，提供关键指标可视化，并将结果导出为可共享的格式。

## 3. 上下文信息
- **背景**：销售团队目前依赖手动Excel处理月度交易数据，过程耗时且容易出错，需要自动化解决方案
- **相关人员**：销售团队成员（非技术人员）、销售经理、财务部门
- **技术环境**：现有数据为CSV格式导出，团队使用Windows系统，无专用数据分析工具
- **业务场景**：每月需要生成销售报告，分析销售趋势，支持业务决策

## 4. 指令与约束
- **输出格式**：具有Web界面的Python应用程序（优先考虑Flask或Streamlit）
- **技术要求**：
  - 接受CSV格式的销售交易数据
  - 支持常见数据操作（计算总计、平均值、增长率）
  - 集成数据可视化库（生成条形图、折线图、饼图）
  - 支持PDF报告导出功能
- **质量要求**：
  - 界面直观，非技术用户无需培训即可使用
  - 计算准确性需与手动Excel结果一致
  - 系统稳定，处理常见数据格式无错误
- **限制条件**：
  - 必须离线工作（无云服务依赖）
  - 需在标准办公电脑上运行（无高性能计算要求）
  - 支持Windows操作系统
- **成功标准**：
  - 月度报告处理时间从8小时减少到1小时以内
  - 销售经理可独立使用工具生成报告
  - 计算结果100%准确，与手动验证一致
```

## 四要素框架优势

本技能采用的四要素框架（角色设定、任务描述、上下文信息、指令约束）是构建优秀prompt的关键，具有以下优势：

1. **结构化清晰**：四个要素逻辑递进，确保提示完整无遗漏
2. **角色明确**：帮助AI理解自身定位，提供更专业的响应
3. **任务具体**：避免模糊描述，确保目标明确可执行
4. **上下文完整**：提供必要背景，减少AI猜测和假设
5. **约束明确**：设定清晰边界，提高输出质量和相关性

## 高级技巧

### 处理文化上下文
当用户混合使用中文和英文或使用口语表达时，解码潜在需求而非字面翻译。"帮我弄一下"通常意味着"帮助我实施/设置/修复"而不仅仅是"帮助我"。

### 处理极度模糊
对于"做个东西"这样的请求，自动推断：
- 用户可能正在经历的痛点
- 类似工具或解决方案的常见模式
- 可能的用户角色和行业上下文
- 基于常见需求生成合理的假设

### 提示迭代周期
将提示优化视为迭代：生成 → 获取反馈 → 优化 → 执行。每个周期提高理解力和结果质量。

## 进阶方法

除了四要素框架外，以下进阶方法可以进一步提升提示的质量和效果：

### 1. 任务分解：一步步解决复杂问题

对于复杂任务，引导AI采用分步处理策略，而不是一次性解决所有问题：

**应用方法**：
- **输出大纲先行**：要求AI先输出整体大纲或架构，然后基于该大纲填充细节
- **分阶段执行**：将复杂任务分解为逻辑步骤，按顺序完成（如：需求分析→架构设计→代码实现→测试验证）
- **检查点设置**：在关键节点设置检查点，确保每一步都符合预期后再继续

**优化提示示例**：
"请采用分步策略完成此任务：首先输出整体解决方案大纲，包括主要模块和关键决策点，然后基于该大纲逐步深入每个模块的详细设计和实现。"

### 2. 链式思考：展示推理过程

要求AI展示思维链（Chain-of-Thought），让推理过程透明化，这通常比直接给出答案更准确：

**应用方法**：
- **显式推理**：明确要求AI"展示你的推理过程"或"逐步解释你的思考逻辑"
- **假设说明**：要求AI明确列出所有假设，并说明这些假设如何影响解决方案
- **替代方案对比**：对于关键决策点，要求提供多个备选方案并对比优缺点

**优化提示示例**：
"请展示你的链式思考过程：首先分析问题核心，然后列出可能的解决方案，评估每个方案的优缺点，最后基于评估结果给出推荐方案并详细说明理由。"

### 3. 明确性准则：避免模糊词汇

在提示中使用具体、可衡量的语言，消除歧义：

**模糊词汇** → **明确表达**
- "快速" → "在3秒内完成响应"
- "好看的设计" → "遵循Material Design规范，使用一致的配色方案和间距"
- "高效" → "时间复杂度O(n)，内存使用不超过100MB"
- "用户友好" → "新用户能在2分钟内完成首次使用，无需阅读说明文档"

**明确性检查清单**：
- ✅ 所有要求都有具体的衡量标准
- ✅ 时间、数量、质量等都有明确数值或范围
- ✅ 专业术语有明确定义或解释
- ✅ 成功标准可客观验证（无需主观判断）

### 综合应用示例

**原始模糊请求**："帮我优化数据库查询"

**进阶优化提示**：
```
# 数据库查询性能优化方案

## 1. 角色设定
数据库性能优化专家，精通SQL查询调优和索引设计

## 2. 任务描述
为电商订单查询系统提供性能优化方案，将当前平均查询响应时间从5秒降低到500毫秒以下

## 3. 上下文信息
- **背景**：订单查询页面在高峰期响应缓慢，影响用户体验和转化率
- **技术环境**：MySQL 8.0，订单表有5000万条记录，当前查询使用多个JOIN和子查询
- **约束条件**：不能修改现有数据架构，优化必须在现有表结构基础上进行

## 4. 指令与约束
- **输出要求**：
  1. 先提供整体优化策略大纲，然后基于该大纲详细实现
  2. 展示每个优化步骤的推理过程和性能影响分析
  3. 提供具体的SQL改写示例和索引建议
- **明确性要求**：
  - 所有性能目标必须有具体数值（如：响应时间<500ms，CPU使用率降低30%）
  - 每个优化建议必须说明预期提升幅度和验证方法
  - 提供可执行的测试SQL和性能对比数据
- **成功标准**：
  - 优化后查询在测试环境验证响应时间<500ms
  - 提供完整的性能测试报告和监控指标
```

## 实践建议

1. **渐进式采用**：从最简单的明确性要求开始，逐步引入任务分解和链式思考
2. **模板化**：为重复性任务创建包含这些进阶方法的提示模板
3. **质量评估**：定期回顾优化后的提示，评估其明确性和效果，持续改进

## 要避免的常见陷阱

❌ **执行实际任务**：**绝对不要**执行用户请求的实际任务。例如，当用户说"了解代码的架构"时，不要读取文件或分析代码库，只生成用于分析代码架构的prompt
❌ **过度工程化**：不要使提示过长或过于复杂
❌ **假设过多**：验证假设而非猜测
❌ **忽略文化细微差别**：理解字面翻译之外的上下文
❌ **跳过验证**：生成的提示应尽可能完整，减少用户二次确认的需要
❌ **一刀切**：根据用户明显专业知识调整方法

## 成功指标

当出现以下情况时，您会知道此技能正在发挥作用：
- 用户对您提供的清晰度表示赞赏
- 生成的提示基于四要素框架，结构完整清晰
- 需求变得具体且可操作，角色、任务、上下文、约束明确
- 解决方案更好地匹配用户期望，减少AI猜测
- 由于误解而需要的返工减少
- 用户学会随着时间的推移提供更好的提示
