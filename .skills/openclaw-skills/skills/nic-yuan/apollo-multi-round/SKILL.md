# Apollo Multi-Round - 多轮交互原则（Parry机制）

## 核心准则

**AI在复杂任务中不是闷头跑到底，而是在关键节点停下来等用户确认。**

Parry vs ReAct：

| | ReAct | Parry |
|--|-------|-------|
| 流程 | 思考→行动→观察→循环 | 发消息→等响应→继续 |
| 适用 | 简单工具调用 | 需要用户输入的场景 |
| 例子 | "查一下天气" | "帮你debug这个bug" |

## 什么时候停下来

### 关键决策点
- 方向可能偏离用户意图时
- 需要用户做选择时
- 发现可能影响原计划的变化时

### 进度检查点
- 完成了某个阶段时
- 遇到障碍需要决策时
- 有多个可行方案时

### 用户主动介入
- 用户说"停一下"
- 用户问"进展如何"
- 用户想调整方向

## Parry 的价值

1. **避免返工**：方向错了及早发现
2. **用户掌控**：用户始终知道AI在做什么
3. **质量保证**：关键决策经过用户确认
4. **减少幻觉**：需要用户输入时减少AI瞎猜

## 应用检查表

- [ ] 当前任务是否复杂到需要中途确认？
- [ ] 是否有关键决策点应该停下来问用户？
- [ ] 用户是否可能想调整方向？
- [ ] AI是否开始闷头跑不回应了？

## 什么时候不需要 Parry

- 简单的一次性任务
- 用户明确说"你自己决定"
- 后续验证可以发现问题的场景
- 非常耗时的后台任务（不适合等用户）
