# 代码审查规范

## 概述

本文档定义代码审查流程和标准，确保代码质量和团队协作效率。

## 审查流程

### 1. 提交前自检
- 运行测试确保通过
- 运行代码格式化工具（ESLint / Checkstyle / Black）
- 检查是否有明显的 bug
- 确保符合编码规范（见 `coding.coding-style.md`）

### 2. 创建 Pull Request
- 提供清晰的 PR 描述：变更内容、原因、关联 Issue
- 控制变更范围（单次 PR 不超过 400 行为佳）

### 3. 代码审查
- 至少一人审查，关键路径需要多人审查
- 审查重点关注：正确性、安全性、性能、可维护性

### 4. 修改和合并
- 根据审查意见修改，重新请求审查
- 审查通过后合并

## 审查检查清单

> 详细的审查标准（P0 安全检查、P0 空指针检查、P1 业务逻辑、P2 代码质量）见 `/review-code` 技能的 `checklists/code-review.md`。

以下为简要检查项：

| 维度 | 关键检查点 |
|------|-----------|
| **正确性** | 功能是否符合预期、边界情况已处理 |
| **安全** | 输入验证、参数化查询、权限检查（详见 `coding.security.md`） |
| **性能** | 避免 N+1 查询、合理使用缓存（详见 `coding.performance.md`） |
| **规范** | 命名、格式、注释符合 `coding.coding-style.md` |
| **测试** | 关键逻辑有单元测试、覆盖率达标（详见 `coding.testing.md`） |

## 审查反馈原则

- **建设性**：提供改进建议，而非仅指出问题
- **具体性**：指出具体的行号和问题
- **分级**：按 P0（必须修复）/ P1（建议修复）/ P2（可选优化）分级
