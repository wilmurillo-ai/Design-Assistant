# /gen-prd - PRD 生成技能

**版本**: 1.0.0  
**阶段**: Phase 1

---

## 概述

`/gen-prd` 技能用于通过交互式需求分析，生成完整的产品需求文档（PRD）。

---

## 使用方式

### 基本用法

```bash
# 直接使用
/gen-prd 我要做一个用户积分系统

# 从 start 技能引导后使用
/start 我要做一个积分系统
# ... 需求澄清后 ...
/gen-prd

# 指定输出路径
/gen-prd --output=docs/prd/custom-prd.md
```

### 参数说明

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `requirement` | string | 否 | - | 需求描述 |
| `output` | string | 否 | `docs/prd/YYYY-MM-DD-{project-name}.md` | 输出文件路径 |

---

## 输入

- 需求描述（直接输入或从 start 技能传入）
- 需求澄清信息（通过交互问答收集）

---

## 输出

- `docs/prd/YYYY-MM-DD-{project-name}.md` - PRD 文档

---

## PRD 文档结构

| 章节 | 说明 | 必填 |
|------|------|------|
| 项目背景 | 项目概述、目标、范围 | 是 |
| 目标用户 | 用户画像、用户规模 | 是 |
| 功能需求 | 功能列表、功能详情 | 是 |
| 用户故事 | 用户故事及验收标准 | 是 |
| 核心数据流与闭环 | 数据流路径、闭环保证 | **是（必填）** |
| 非功能性需求 | 性能、安全、其他需求 | 是 |
| 技术栈确认 | 前后端技术栈 | 是 |
| 验收标准 | 功能验收、质量验收 | 是 |

---

## 核心数据流与闭环

PRD 中**必须**包含「核心数据流与闭环」节，描述：

1. **核心数据流**：数据从哪里来、到哪里去
2. **闭环保证**：如何保证数据一致性和完整性

**示例**：

| 数据流编号 | 数据流名称 | 触发条件 | 数据来源 | 数据去向 | 闭环说明 |
|------------|------------|----------|----------|----------|----------|
| DF-001 | 积分获取 | 用户消费 | 订单系统 | 积分账户 | 订单完成后触发积分计算，事务保证一致性 |
| DF-002 | 积分兑换 | 用户兑换 | 积分账户 | 商品系统 | 积分扣减与商品发放原子操作 |

---

## 需求澄清问题

本技能会通过交互问答收集以下信息：

1. **目标用户**：企业内部员工 / C 端消费者 / B 端商家
2. **核心功能**：根据需求描述列出选项
3. **用户规模**：<1 万 / 1-10 万 / >10 万
4. **集成需求**：用户系统 / 订单系统 / 支付系统
5. **技术栈偏好**：推荐方案 / 指定技术栈

## 与其他技能的关系

### 上游

- `/start` → 需求澄清后引导至 `/gen-prd`

### 下游

- `/gen-prd` → `/gen-design`（系统设计）
- `/gen-prd` → `/validate --mode=dataflow`（验证数据流）

---

## 常见问题

### Q1: PRD 生成后可以修改吗？

可以。PRD 是活文档，建议在需求变更时及时更新。

### Q2: 如何确保 PRD 的完整性？

本技能会在生成 PRD 时自动检查必填章节，特别是「核心数据流与闭环」节。

### Q3: PRD 与设计文档的关系？

PRD 侧重业务需求，设计文档侧重技术实现。设计文档应与 PRD 中的数据流对齐。
