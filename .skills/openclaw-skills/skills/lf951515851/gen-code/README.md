# /gen-code - 代码生成技能

**版本**: 2.0.0  
**合并来源**: gen-code (v1.5.0) + gen-with-context (v1.0.0)

---

## 概述

`/gen-code` 技能用于按任务和规范生成高质量代码，支持：

- **新项目生成** - 使用标准技术栈和最佳实践
- **遗留项目上下文感知生成** - 基于设计主线和代码库分析，生成与存量风格一致的代码

技能会自动检测项目类型（通过检查 `docs/design/design-line.md` 或 `docs/analysis/codebase-analysis.md`），并自动选择合适的生成模式。

---

## 使用方式

### 基本用法

```bash
# 生成单个任务的代码
/gen-code Task-001

# 生成任务范围（分批生成）
/gen-code Task-001..010

# 基于设计文档生成
/gen-code --design=docs/design/order-design.md

# 指定输出目录
/gen-code Task-001 --output-dir=src/points/
```

### 参数说明

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `task` | string | 是* | - | 任务编号或任务描述 |
| `design` | string | 否 | - | 设计文档路径 |
| `output-dir` | string | 否 | 自动推断 | 输出目录路径 |
| `language` | enum | 否 | 自动识别 | 生成语言：`java` / `vue` / `react` / `typescript` |

*task 和 design 至少提供一个

## 项目类型自动检测

技能会自动检测项目类型：

### 新项目

**检测条件**：项目中不存在 `docs/design/design-line.md` 或 `docs/analysis/codebase-analysis.md`

**生成模式**：
- 使用标准技术栈（Vue3 + Ant Design + Spring Boot 3 + MyBatis-Plus）
- 遵循默认编码规范
- 使用最佳实践

### 遗留项目

**检测条件**：项目中存在 `docs/design/design-line.md` 或 `docs/analysis/codebase-analysis.md`

**生成模式**：
- 读取设计主线和代码库分析
- 学习现有代码风格
- 生成与存量风格一致的代码
- 遵循设计主线定义的架构约束

**提示示例**：
```
检测到当前项目为遗留项目（已存在设计主线或代码库分析报告）。
将自动启用上下文感知模式，基于现有架构和代码风格生成代码。

设计主线: docs/design/design-line.md
代码分析: docs/analysis/codebase-analysis.md

是否继续？
```

## 上下文感知生成

在遗留项目模式下，生成的代码将遵循：

### 1. 风格一致性

| 方面 | 说明 |
|------|------|
| 命名风格 | 驼峰/下划线，类名/方法名/变量名规范 |
| 代码格式 | 缩进、换行、空行、括号风格 |
| 注释风格 | Javadoc/行内注释、注释模板 |
| 日志规范 | 日志框架、日志级别、日志格式 |

### 2. 架构一致性

| 方面 | 说明 |
|------|------|
| 分层结构 | Controller/Service/Repository 分层 |
| 设计模式 | 现有项目使用的模式（策略、工厂等） |
| 异常处理 | 异常类定义、处理方式 |
| 事务处理 | 事务边界、传播机制 |

### 3. 技术栈一致性

| 方面 | 说明 |
|------|------|
| 依赖库 | 使用项目已有的依赖库 |
| 配置 | 遵循项目配置（Spring、数据库等） |
| 工具类 | 使用项目的工具类 |
| 安全规范 | 认证、授权、加密等 |

### 4. 边界约束

- **服务边界**：微服务架构下，不跨服务直接访问其他服务的数据库或内部实现；仅通过已定义的 API、事件或网关集成。
- **微前端边界**：微前端架构下，不破坏子应用边界（不随意跨子应用暴露状态、不混用不同子应用的组件/路由）；新功能归属明确子应用或基座，并遵循现有接入方式。

## 质量门禁

生成代码后自动执行质量门禁检查：

### P0 强制检查项

- **复杂度检查**：方法长度 ≤ 80 行、嵌套深度 ≤ 3 层、参数数量 ≤ 5 个、圈复杂度 ≤ 10
- **安全检查**：SQL 注入防护、XSS 防护、敏感信息保护、权限控制
- **空指针防护**：外部输入 null 检查、集合判空、Optional 正确使用
- **事务正确性**：多表操作在事务中、事务范围合理、异常回滚配置正确
- **测试覆盖**：核心业务逻辑有测试、异常分支有测试

### P1 设计一致性检查

- **接口一致性**：API 路径与设计文档一致、请求/响应字段与设计一致
- **数据一致性**：数据库字段与设计一致、数据类型与设计一致
- **数据流一致性**：输入来源与设计一致、输出目标与设计一致、集成点实现正确
- **上下文一致性**（遗留项目）：代码风格与现有项目一致、架构分层与现有项目一致、命名规范与现有项目一致

### 代码健康度评分

| 等级 | 分数范围 | 说明 | 处理建议 |
|------|----------|------|----------|
| A+ | 95-100 | 优秀 | 可直接使用 |
| A | 90-94 | 良好 | 可直接使用 |
| B+ | 85-89 | 较好 | 建议优化后使用 |
| B | 80-84 | 合格 | 需小幅优化 |
| C | 70-79 | 需改进 | 必须优化后使用 |
| D | 60-69 | 较差 | 需要重构 |
| E | <60 | 不合格 | 需要重新生成 |

**强制要求**：代码健康度评分必须 ≥ 80 分（B 级及以上）才能输出，否则必须重新生成。

## 典型工作流

### 工作流 1：新项目开发

```bash
# 1. 生成 PRD
/gen-prd "新功能需求"

# 2. 生成详细设计
/gen-design

# 3. 拆解任务
/gen-tasks

# 4. 生成代码（自动识别为新项目）
/gen-code Task-001

# 5. 生成测试
/gen-test src/main/java/.../PointsService.java

# 6. 代码审查
/review-code src/main/java/.../
```

### 工作流 2：遗留项目迭代

```bash
# 1. 分析项目（如尚未分析）
/analyze --phase=deep

# 2. 生成 PRD
/gen-prd "新功能需求"

# 3. 生成详细设计
/gen-design

# 4. 验证设计是否符合设计主线
/validate --mode=domain

# 5. 拆解任务
/gen-tasks

# 6. 生成代码（自动识别为遗留项目，启用上下文感知）
/gen-code Task-001

# 7. 生成测试
/gen-test src/main/java/.../PointsService.java

# 8. 代码审查
/review-code src/main/java/.../
```
