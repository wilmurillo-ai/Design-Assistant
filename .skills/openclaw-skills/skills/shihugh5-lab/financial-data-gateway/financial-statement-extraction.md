# Skill: 财务报表数据提取 (financial-statement-extraction)

> 从中国A股上市公司年报PDF中精确提取合并财务报表的结构化数据。
> 这是整个财务分析流水线的根节点——所有下游分析（重构、估值、对比、报告）都依赖本Skill的输出质量。

## 处理流程

### 步骤1：文档预处理与基础解析

**目标**：读取PDF，识别基本属性，判断是否可处理。

**操作**：
1. 用 `pdfplumber` 打开PDF，获取总页数
2. 检测PDF类型：
   - 提取前5页文本，如果文本长度 < 100字/页 → 扫描件（需OCR，当前版本不支持，报错退出）
   - 否则 → 可复制文本PDF（正常处理）
3. 从前10页提取元数据：
   - 公司名称：匹配"XX股份有限公司"或"XX集团"
   - 股票代码：匹配6位数字
   - 报告类型：年报/半年报/季报（匹配"年度报告"/"半年度报告"/"季度报告"）
   - 报告期：匹配"20XX年度"或"20XX年X月X日"
   - 语言：中文/英文（检测中文字符占比）
4. 检测单位：扫描报表页，匹配"单位：元/万元/千元/亿元"

**输出**：`meta` 字典

**异常处理**：
| 错误类型 | 触发条件 | 返回值 |
|:---|:---|:---|
| scanned_pdf | 扫描件PDF | `{"error": "scanned_pdf", "message": "扫描件PDF暂不支持，请提供可复制文本版本"}` |
| encrypted_pdf | 加密PDF | `{"error": "encrypted_pdf"}` |
| warning | 页数 < 20 | 警告"可能不是完整年报" |

---

### 步骤2：目录与章节结构化

**目标**：定位三张合并财务报表的精确页码范围。

**策略（双层定位）**：

**第一层：AI智能定位（主策略）**
1. 扫描PDF中间30%-70%的页面（报表通常在这个范围）
2. 提取每页前200字摘要
3. 发送给LLM，让AI判断每张报表的起止页码
4. AI Prompt 关键约束：
   - 报表固定顺序：合并资产负债表 → 合并利润表 → 合并现金流量表 → 合并所有者权益变动表 → 母公司报表
   - 典型页数：资产负债表2-3页，利润表2页，现金流量表2-3页
   - "母公司"三个字是绝对分界线
   - 宁可少包含一页，也不要多包含母公司的页

**第二层：关键词匹配（降级策略）**
- 当AI定位失败时自动降级
- 扫描全文，匹配标题关键词：
  - "合并资产负债表" → bs_start
  - "母公司资产负债表" → bs_end
  - "合并利润表"/"合并损益表" → is_start
  - "母公司利润表" → is_end
  - "合并现金流量表" → cf_start
  - "母公司现金流量表"/"合并所有者权益变动表" → cf_end
- 用"下一个标题"作为当前报表的结束边界

**母公司数据过滤（三层防线）**：
1. **页级过滤**：检测每页前500字，纯母公司页面直接跳过
2. **混合页处理**：如果母公司标题不在页面开头（前80字），保留该页交给行级过滤
3. **行级截断**：提取表格时，遇到"母公司XX表"标题行立即停止

**向前扩展**：如果AI遗漏了合并报表的首页（标题在前一页末尾），自动检测并补上。

**实战经验（踩坑记录）**：
| 公司 | 问题描述 | 解决方案 |
|:---|:---|:---|
| 中炬高新2024 | 第74页末尾是"合并利润表"标题，第75页才是数据。AI只给了75-77页，漏掉了74页 | 向前扩展机制 |
| 中炬高新2024 | 第76页上半部分是合并利润表的综合收益，下半部分是母公司利润表的开头 | 混合页+行级截断 |
| 海天味业2024 | 第87页是合并现金流量表末尾+母公司现金流量表开头 | 同上 |

---

### 步骤3：表格识别与数据提取

**目标**：从定位到的页面中提取科目名称和数值。

**操作**：
1. 用 `pdfplumber.extract_tables()` 提取每页的所有表格
2. 遍历每行：
   - 第1列 → 科目名称（原始文本）
   - 后续列 → 跳过附注引用列（"七、XX"格式或罗马数字），取第一个有效数字
3. 科目标准化（`_normalize`）：
   - 去掉序号前缀（"一、"、"二、"）
   - 去掉"加："、"减："前缀
   - 去掉括号内说明文字
   - 通过映射表统一科目名称（如"股本"→"实收资本"，"营业总收入"→"营业收入"）
4. 数字解析（`_parse_num`）：
   - 处理千分位逗号
   - 处理括号负数（中英文括号）
   - 处理注释符号（①②③等）
   - 跳过零值（避免覆盖有效数据）
5. 去重策略：`if std not in target` — 先提取到的数据优先（合并报表在前，母公司在后）

**科目覆盖**：
| 报表 | 科目数量 | 主要内容 |
|:---|:---:|:---|
| 资产负债表 | ~50 | 流动资产、非流动资产、流动负债、非流动负债、所有者权益 |
| 利润表 | ~25 | 收入、成本、费用、利润、每股收益 |
| 现金流量表 | ~15 | 经营/投资/筹资三大活动 |

**BS_ONLY过滤**：当提取利润表/现金流量表时，自动过滤掉资产负债表专属科目（防止跨页溢出）。

---

### 步骤4：附注与补充信息提取（v2.1规划）

> 当前版本暂不实现，作为v2.1的扩展目标。

**规划内容**：
- 从附注中提取会计政策摘要（折旧方法、收入确认、存货计价）
- 提取应收账款账龄分析表
- 提取存货分类明细
- 提取固定资产、无形资产明细
- 提取有息负债明细（短期借款+长期借款+应付债券）
- 提取关联交易清单
- 提取或有负债、承诺事项
- 提取分部信息（业务分部、地区分部）

**技术挑战**：附注格式极不统一，需要更强的AI理解能力。考虑用LLM直接从附注文本中提取结构化数据。

---

### 步骤5：数据清洗与标准化

**目标**：统一单位、统一科目名称、清理异常值。

**操作**：
1. **单位统一**：
   - 检测原始单位（元/千元/万元/亿元）
   - 统一转换为亿元（乘以对应系数）
   - 特殊处理：每股收益（基本/稀释）不做单位转换，保留原始元/股
2. **科目标准化**：
   - 通过 `_MAP` 映射表（~120个映射规则）统一科目名称
   - 处理同义词：如"股本"="实收资本"，"所有者权益合计"="股东权益合计"
   - 处理格式差异：如"负债和所有者权益（或股东权益）总计"→"负债和股东权益总计"
3. **异常清理**：
   - 去掉乱码字符
   - 去掉空格、换行
   - 去掉注释符号（①②③、*、注）

---

### 步骤6：内部核验与勾稽关系校验

**目标**：验证提取数据的正确性和完整性。

**已实现的校验**：

| 校验项 | 规则 | 通过标准 | 失败处理 |
|:---|:---|:---|:---|
| 资产负债表等式 | 资产总计 = 负债合计 + 股东权益合计 | 差异 < 0.01% | 差异 > 1% 触发降级重提取 |
| 净利率合理性 | 净利率 > 100% 可能混入母公司数据 | < 100% | 自动触发降级重提取 |
| 毛利率合理性 | 毛利率 > 95% 可能异常 | < 95% | 警告 |
| 完整性-资产负债表 | 科目数量 | ≥ 20项 | 警告"提取不完整" |
| 完整性-利润表 | 科目数量 | ≥ 10项 | 警告 |
| 完整性-现金流量表 | 科目数量 | ≥ 8项 | 警告 |
| 关键科目检查 | 必须包含核心科目 | 全部存在 | 错误 |

**关键科目**：资产总计、营业收入、净利润、经营活动现金流净额

**v2.1规划的校验**：
- 利润表净利润 + 调整项 ≈ 经营活动现金流（间接法勾稽）
- 本期期末余额 = 上期期末余额 + 本期变动（需要上期数据）
- 利润表净利润与所有者权益变动表匹配
- 同一科目在主表与附注之间一致性校验

## 已知限制

| 限制 | 说明 | 解决方案 |
|:---|:---|:---|
| 扫描件PDF | 当前不支持，需要OCR预处理 | v3.0规划支持OCR |
| 英文财报 | 当前科目映射表只覆盖中文A股年报，不支持SEC 10-K/10-Q | v2.2规划英文支持 |
| 附注提取 | 当前版本不提取附注明细 | v2.1规划附注提取 |
| 上期数据 | 当前只提取本期/期末数，不提取上期/年初数 | v2.1规划 |
| 跨页表格 | 依赖pdfplumber的表格识别，极端跨页情况可能丢失数据 | - |
| 特殊行业 | 银行、保险、证券的报表结构与一般企业不同，科目映射可能不完整 | v2.3规划行业模板 |
