## 问题与修复

### 问题 1: FTS 索引不一致 ✅ 已修复

**现象**: 
- chunks 表：1567 条
- chunks_fts 表：1395 条
- 差异：172 条（测试数据导致）

**原因分析**:
- SQLite FTS5 索引是异步更新的
- 测试中快速连续写入 170+ 条测试数据
- 原有代码未主动触发 FTS 同步

**已实施修复**:
在 `insert_chunk` 方法中添加 FTS 同步逻辑：
```python
# 同步 FTS 索引（确保一致性）
try:
    # 使用 rowid 直接插入，避免 FTS5 rebuild 的延迟
    self.cursor.execute('''
        INSERT OR IGNORE INTO chunks_fts(rowid, text, id, path, source, model, start_line, end_line)
        VALUES (last_insert_rowid(), ?, ?, ?, ?, ?, ?, ?)
    ''', (text, chunk_id, path, source, model, start_line, end_line))
    self.conn.commit()
except Exception as fts_error:
    logger.warning(f"⚠️  FTS 同步失败：{fts_error}，将在下次监控中修复")
```

**验证结果**:
- ✅ 修复后 chunks=1567, fts=1567，差异=0
- ✅ FTS 索引完全同步

### 问题 3: 数据库过大 ⚠️ 待优化

**现象**:
- 数据库大小：273.85 MB
- 超过警戒线：100 MB

**原因分析**:
- embedding 向量占用大量空间（768 维 float 数组）
- 每个记忆块约 200KB（含 embedding）
- 1567 条 × 200KB ≈ 313MB（与实际相符）

**影响**:
- 磁盘占用较大
- 备份时间较长
- 不影响功能

**建议优化**:
1. 使用更小的 embedding 维度（如 384 维）
2. 使用量化 embedding（int8 代替 float32）
3. 定期归档旧记忆到冷存储
4. 使用向量数据库（如 Chroma、Qdrant）替代 SQLite 存储 embedding
