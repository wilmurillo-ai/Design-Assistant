# 全流程日志规范

## 目的
记录一次完整流程的可追溯日志，覆盖从输入读取 → 任务拆解 → 执行引擎 → 开发与自检 → 结果写回 → 评分回写全过程。

## 日志产物

### A. 主日志（推荐）
- 文件名：`运行日志_{date}_{batch_seq}.md`
- 位置：与任务 JSON 同目录（或用户指定目录）
- 原则：**每次执行新建文件，不覆盖**

#### 模板
```
# 运行日志_{date}_{batch_seq}

## Step1 读取输入
- 输入JSON路径：
- goal_prompt路径：
- 摘要：

## Step2 任务拆解
- 拆解方法摘要：
- 生成任务数：
- 输出JSON路径：

## Step3 任务执行引擎
- 当前task_id：
- query：
- script_registry命中：否（为空）
- OB召回结果：无/有
- 置信度评分：
- 是否进入开发：是/否

## Step3.1 开发流程
- 技术方案文件：
- 开发要点：
- 自检结论：

## Step4 结果写回
- task_result摘要：
- task_state更新：

## Step5 结果评分
- script_score：
- score_reason：

## Step6 回写说明
- 脚本注册表：禁止新增，记录未回写说明
```

### B. 任务 JSON 内嵌日志（可选）
在每个任务对象中新增：
- `task_log`: 数组，按时间顺序记录步骤摘要

示例：
```json
"task_log": [
  {"ts":"2026-03-16 08:30", "step":"execute", "note":"进入OB召回，无结果"},
  {"ts":"2026-03-16 08:32", "step":"dev", "note":"完成技术方案v1"}
]
```

## 归档规则
- 日志与输出 JSON 同目录
- 多批次建议按 `date/batch_seq/` 分层目录
- 任何异常都需追加记录到主日志
