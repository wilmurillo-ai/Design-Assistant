# 长篇小说架构师

完整的中文长篇小说策划与写作技能，用于把零散设定整理成真正可写、可扩展、可长期维护的小说工程。

## 适用场景

- 整理零散灵感，建立可持续推进的长篇框架
- 修复世界观、时间线、人物逻辑之间的冲突
- 设计系列总纲、分卷大纲、章节卡、线索表
- 强化人物弧线、关系拉扯、团队冲突与互相拯救
- 将重设定题材改造成更适合连载平台的开篇结构
- 搭建或整理 Markdown 小说项目目录
- 生成简介、宣传语、封面文案等发布素材

## 核心目标

- 让故事先成立，再让设定发光
- 让人物推动情节，而不是被设定拖着走
- 让谜题逐层揭示，而不是一次性解释完
- 让每份文档都能直接服务写作，而不是变成资料堆

## 默认工作方式

除非用户明确只要某个局部产物，否则按下面的顺序推进。

### 1. 先找故事脊柱

先回答四个问题：

- 到底发生了什么
- 谁被迫卷入
- 他为什么不能离开
- 故事最终逼向什么选择

如果这四件事还不清晰，不要急着堆设定。

### 2. 建立压力系统

优先明确这些内容：

- 外部危险
- 内在伤口
- 团队摩擦
- 思想对手
- 拖延的代价

故事一旦没有压力，设定再宏大也会变空。

### 3. 把抽象设定翻成戏剧问题

避免只写“有一个古老文明”“有一个失落遗迹”“主角想知道真相”。

优先改写成：

- 这个遗迹正在影响现实中的天气、地貌、秩序或生命
- 主角如果误判，会有人死，会失去亲人，或会把更大的东西放出来
- 每个真相都伴随代价，而不是只带来解释

### 4. 分层拆解项目

把内容拆到不同层级：

- 系列层：主题、终局、三卷功能
- 分卷层：每卷任务、反派、谜题推进、高潮与代价
- 章节层：事件、转折、钩子、人物关系变化
- 场景层：动作、证据、误判、冲突和情绪

### 5. 产出真正可用的写作文件

优先生成最小够用的一组资产：

- 世界观核心设定
- 时间线
- 术语表
- 人物总表与单人档案
- 人物关系表
- 系列总纲 / 分卷总纲
- 分章卡
- 线索表 / 伏笔回收表
- 正文章节初稿

### 6. 需要连载时再做平台化改造

如果目标是连载平台，必须额外检查：

- 开篇是否足够快地给出异常与危险
- 主角是否有清晰的私人理由留下
- 章节结尾是否保留追读钩子
- 解释是否挂在行动、冲突和代价上

需要这部分时，读取 `references/platform-serialization.md`。

## 写作决策原则

### 故事优先，设定其次

- 不把文档写成设定百科
- 不让人物沦为说明设定的嘴
- 不让关键章节只负责解释，不负责推进

### 谜题分层释放

推荐顺序：

1. 异常事件
2. 物证出现
3. 解释冲突
4. 历史回声
5. 结构真相
6. 最终选择

### 每次揭示都要改变局势

每个重大真相至少带来一项后果：

- 危险升级
- 关系破裂
- 判断被推翻
- 证据损坏或流失
- 必须做出更艰难的选择

### 严格保护连续性

当世界观、时间线或人物动机变化时，同步检查：

- 时间线
- 术语表
- 人物档案
- 分章卡
- 线索回收表

不要在项目里无限生成平行版本，优先更新主文件。

## 常用交付物

根据用户卡点，优先输出：

- 书名与一句话 premise
- 完整人物表
- 三卷总纲
- 单卷卷大纲
- 分章卡
- 人物关系表
- 设定冲突修复方案
- 连载平台版简介 / 宣传语 / 封面文案
- 正文章节初稿

## 质量检查清单

交付前至少自检一次：

- 主角是否有真实驱动力，而不是单纯“想知道真相”
- 设定是否真正压到人物命运上
- 每卷是否有明确任务和情绪走势
- 谜题释放顺序是否合理
- 时间线是否自洽
- 前 10 到 12 章是否具备连载抓力
- 相关文档是否已经同步更新
