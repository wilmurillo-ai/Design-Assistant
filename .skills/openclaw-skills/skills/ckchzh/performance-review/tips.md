# 绩效评估使用技巧与最佳实践

## 📋 绩效自评的核心原则

### 1. 包装≠造假
自评的本质是**用专业的方式呈现你的工作价值**。
- ❌ 造假：把没做的事说成做了
- ✅ 包装：把做了的事说得更有影响力

### 2. 数据说话
上级最关注的是你的成果对团队/公司的贡献。
- ❌ "优化了系统性能"
- ✅ "通过缓存策略优化，将接口响应时间从800ms降至200ms，
     用户体验满意度提升15%"

### 3. 自评的黄金比例
- 成果展示：60%（你做了什么，产出了什么）
- 能力成长：20%（你提升了什么）
- 不足与计划：20%（你的改进方向）

## 🎯 使用场景速查

### 场景A：季度/年终自评
```bash
# 第一步：生成自评框架
review.sh self "你的岗位" "成果1,成果2,成果3"

# 第二步：把模糊目标转化为下季度OKR
review.sh goal "下季度目标1,目标2,目标3"
```

### 场景B：给同事写360反馈
```bash
# 给同事写正面+建设性反馈
review.sh feedback "同事" "项目配合积极,但沟通可以更主动"

# 给下属写评语
review.sh feedback "下属" "执行力强但缺乏独立思考"

# 给上级的反馈
review.sh feedback "上级" "给团队的支持充分,期望更多方向性指导"
```

### 场景C：准备晋升
```bash
# 晋升述职报告
review.sh promotion "高级工程师" "3" "系统重构,带团队,技术创新"

# 配合KPI数据
review.sh kpi "系统可用性:99.9%,项目交付:100%,团队培养:带新人3人"
```

|-----------|-----------|-----------|

## 📊 SMART目标功能详解

`goal` 命令帮你把模糊目标转化为可衡量的OKR：

**SMART原则：**
- **S**pecific（具体）：不是"做好"，而是"做到什么程度"
- **M**easurable（可衡量）：有数字、有基线、有目标值
- **A**chievable（可达成）：够得着但有挑战
- **R**elevant（相关）：与团队目标对齐
- **T**ime-bound（有时限）：明确deadline

**转化示例：**
- ❌ "提升用户体验" → ✅ "Q2将NPS评分从40提升至55"
- ❌ "加强团队建设" → ✅ "Q2完成2场技术分享会，组织1次团建"
- ❌ "学习新技术" → ✅ "Q2完成Kubernetes认证，并在团队内推广"

## 🤝 360反馈功能详解

`feedback` 命令使用 **SBI反馈模型**：

- **S**ituation（情境）：在什么项目/场景中
- **B**ehavior（行为）：TA具体做了什么
- **I**mpact（影响）：带来了什么影响

**四种反馈对象的语气区别：**
| 对象 | 语气 | 关注点 |
|------|------|--------|
| 同事 | 平等、协作 | 合作贡献 |
| 下属 | 鼓励+指导 | 成长和表现 |
| 上级 | 尊重、建设性 | 管理支持 |
| 跨部门 | 客观、专业 | 协作效率 |

## 💡 绩效评估实战技巧

### 日常积累（最重要！）
- 每周花5分钟记录本周成果
- 保存重要的数据截图和邮件
- 记录同事/上级的正面反馈
- 绩效评估时就不用回忆了

### 自评写作要点
1. **开头**：用一句话概括整体表现
2. **成果**：按重要性排列，每个成果用STAR法则
3. **成长**：展示你的学习曲线
4. **不足**：选1-2个真实但不致命的点
5. **计划**：用SMART格式写下季度目标

### 和上级沟通
- 自评提交前，先和上级口头确认一下重点
- 如果有分歧，用数据说话
- 绩效面谈时带着你的成果清单

## ⚖️ 绩效评级参考

| 评级 | 含义 | 自评要点 |
|------|------|---------|
| S (卓越) | 远超预期 | 需要有突破性贡献的证据 |
| A (优秀) | 超出预期 | 展示超额完成的数据 |
| B (良好) | 符合预期 | 稳定交付+一些亮点 |
| C (合格) | 基本达标 | 如实陈述，重点写改进计划 |
| D (待改进) | 未达预期 | 诚实面对+详细改进方案 |
