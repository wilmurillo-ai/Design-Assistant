# JD-Writer 使用技巧与最佳实践

## 📋 写JD的核心原则

### 1. 结构清晰
好的JD应该包含以下模块，缺一不可：
- 职位基本信息（职位名称、部门、汇报关系）
- 岗位职责（具体做什么，5-8条）
- 任职要求（硬性+软性，分层列出）
- 薪资福利（透明化薪资范围）
- 公司/团队介绍（卖点）

### 2. 避免常见错误
- ❌ 职责写成"完成领导交办的其他任务"——万能废话
- ❌ 要求写成"精通XX"——精通是什么标准？
- ❌ 薪资写"面议"——投递率直降30%
- ❌ 要求过多——超过10条会吓跑候选人
- ✅ 用"负责/主导/参与"区分职责权重
- ✅ 用"熟悉/掌握/了解"区分技能层次
- ✅ 注明薪资范围，哪怕是"15-25K"这样的区间

### 3. 职位名称怎么写
- "高级Java开发工程师" > "Java工程师"（加级别）
- "用户增长产品经理" > "产品经理"（加方向）
- 避免内部黑话："P7级别"外面人看不懂

## 🎯 使用场景速查

### 场景A：从零开始写JD
```bash
# 第一步：生成完整JD模板
jd.sh write "产品经理" "字节跳动" --level 中级

# 第二步：根据实际情况修改模板内容
# 第三步：做包容性检查
jd.sh inclusive "你修改后的JD文本"
```

### 场景B：优化现有JD（投递少）
```bash
# 第一步：分析现有JD的问题
jd.sh optimize "你的现有JD文本"

# 第二步：对比市场薪资，看是否有竞争力
jd.sh benchmark "前端开发" "深圳"

# 第三步：检查是否有无意识偏见
jd.sh inclusive "你的现有JD文本"
```

### 场景C：跨城市招聘
```bash
# 了解不同城市薪资差异
jd.sh benchmark "后端开发" "杭州"
jd.sh benchmark "后端开发" "成都"
jd.sh benchmark "后端开发" "武汉"
```

## 💰 薪资参考功能详解

`benchmark` 命令提供：
- 按级别划分的薪资范围（初级→中级→高级→专家/总监）
- 城市薪资系数（一线=100%，新一线≈80%，二线≈65%）
- 福利待遇三档参考（基础/中等/优质）
- JD中薪资表述的推荐写法

**支持城市**：北京、上海、深圳、杭州、广州、成都、武汉、南京、西安、长沙、重庆、苏州

**支持职位**：前端开发、后端开发、产品经理、数据分析、UI设计、测试工程师、运营、人力资源、销售、市场营销

> 💡 未匹配的职位会展示所有已有职位供参考

## 🤝 包容性检查功能详解

`inclusive` 命令检查六大维度：
1. **性别偏见**：男性优先、限男、小姐姐、形象好气质佳...
2. **年龄歧视**：35岁以下、90后、年轻...
3. **学历歧视**：985/211、第一学历、全日制...
4. **地域歧视**：本地户口、限本地...
5. **健康歧视**：身体健康、无传染病...
6. **隐性排斥**：能加班、996、服从安排、不计较得失...

检查结果包含：
- 包容性评分（0-100分）
- 触发词定位
- 每个问题的具体修改建议
- 相关法律参考

## 📊 提升JD投递率的数据

根据招聘平台数据：
- 标明薪资范围的JD，投递率提高 **30-50%**
- 结构清晰的JD，阅读完成率提高 **40%**
- 包含团队介绍的JD，投递意愿提高 **25%**
- 去除歧视性用语的JD，候选人多样性提高 **60%**
- JD长度控制在500-800字最佳

## 🔧 高级用法

### 批量生成
如果需要招多个岗位，可以：
```bash
for role in "前端开发" "后端开发" "产品经理"; do
  jd.sh write "$role" "你的公司" --level 中级
  echo "========================"
done
```

### 配合其他工具
- 写完JD后用 `job-hunter` 的简历匹配功能测试JD质量
- 用 `meeting-notes` 记录招聘会议的决策
- 用 `email-writer` 撰写候选人沟通邮件

## ⚖️ 法律红线提醒

1. 《就业促进法》禁止基于性别、民族、种族、宗教信仰的就业歧视
2. 《妇女权益保障法》禁止在招聘中限定性别或提高女性录用标准
3. 《劳动法》规定不得以传染病病原携带为由拒绝录用
4. 各地可能有更严格的地方性反歧视法规

**写JD前，先用 `inclusive` 检查一遍，规避法律风险！**
