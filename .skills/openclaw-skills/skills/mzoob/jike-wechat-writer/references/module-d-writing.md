# 模块 D：正文创作与迭代

这是核心创作环节。包含素材搜集、正文撰写、质量自检、交付与迭代四个子环节。产出的内容以 `.md` 文件形式保存，完成后交给用户预览确认。

## D1. 素材搜集

动笔之前，围绕选题搜集必要的事实素材，避免凭空编造或内容空洞。

**必须搜索的场景**：
- 选题涉及时效性信息（新闻事件、政策变化、行业动态、产品发布）
- 选题涉及具体数据（市场规模、用户量、价格对比、排名）
- 选题涉及你知识库可能过时或不确定的事实

**搜索策略**：
- 围绕选题拆解 2-3 个搜索角度，通过脚本分别搜索：
  ```bash
  python3 {baseDir}/scripts/writer.py search --query "搜索角度1" --action news
  python3 {baseDir}/scripts/writer.py search --query "搜索角度2" --action news
  ```
  例如选题是"AI 替代岗位"，可搜"AI 替代 岗位 2025"、"AI 就业 报告"、"ChatGPT 企业裁员"
- 对搜索结果中信息密度高的链接，使用 `web_fetch` 获取原文，提取关键事实、数据和案例
- 将搜集到的核心素材（数据点、案例、引用）整理为创作备用，在正文中自然融入

**可跳过搜索的场景**：
- 纯观点输出类文章，且用户已提供了充分的论点和素材
- 用户明确说"不需要搜索"或"用你已有的知识写"

## D2. 正文撰写

正文的结构、节奏、语言风格、标题公式、开头结尾策略、段落节奏、篇幅等维度，全部遵循模块 C 确定的风格框架（风格报告或公众号爆文默认风格）。本模块不重复定义这些维度，专注于模块 C 未覆盖的创作执行细节。

**配图占位**：

撰写正文时不要同时搜索或生成配图，而是在需要配图的位置插入占位标记，专注于文案本身的质量。占位格式：

```markdown
上一段正文。

<!-- IMG: 一张程序员在深夜加班的真实办公室场景照片 -->

下一段正文。
```

- 使用 HTML 注释语法 `<!-- IMG: 描述 -->`，不会影响 Markdown 渲染
- 描述应具体说明需要什么类型的图片（真实照片/插画/概念图）、画面内容、情绪氛围
- 配图位置遵循风格文件的配图习惯；无风格文件时，默认每 500-800 字放置一个占位
- **占位标记是正文的一部分，必须在模块 D 写作阶段就写入 .md 文件**。模块 E 只负责将占位替换为真实图片，不负责决定配图位置和数量

## D3. 质量自检

正文写完后、交付用户之前，对照以下清单进行一轮自检。不需要逐条向用户汇报，但发现问题必须自行修正后再交付。

| 维度 | 检查项 | 不合格的典型表现 |
|------|--------|-----------------|
| 事实准确 | 文中引用的数据、时间、人名、产品名是否准确 | 编造不存在的数据，张冠李戴 |
| 逻辑连贯 | 段落之间的逻辑关系是否通顺，有无跳跃 | 上一段讲原因，下一段突然讲无关的案例 |
| 信息密度 | 是否存在大段废话或重复表达 | 同一个观点换了三种说法反复讲 |
| 风格一致 | 全文语言风格是否统一，是否符合风格文件要求 | 前半段口语化，后半段突然变学术腔 |
| 受众匹配 | 用词和案例是否匹配目标受众的认知水平 | 给小白读者写满专业术语且不解释 |
| 可读性 | 段落长度、句式变化、小标题分隔是否合理 | 连续出现 200 字以上的大段落 |
| 完整性 | 文章是否有头有尾，核心论点是否都已展开 | 列了 5 个要点只写了 3 个就结束 |
| 配图占位 | 文章中是否已按风格要求插入了 `<!-- IMG: 描述 -->` 占位标记 | 全文没有任何占位标记，或占位描述过于笼统（如"配图"） |

## D4. 交付与迭代

**首次交付**：

1. **写入文件**：使用 `file` 工具的 `write` 操作，将完整文章写入 `.md` 文件（文件名应体现选题，如 `AI工具盘点-职场效率提升.md`）
2. **交付预览**：将 `.md` 文件交给用户预览确认
3. **等待反馈**：交付后，简要说明本次创作的核心思路（2-3 句话），然后等待用户反馈

**迭代机制**：

当用户对文章提出修改意见时：

- **精准修改**：只改用户指出的部分，不要擅自"顺手优化"其他段落
- **覆盖写入**：修改后使用 `file` 工具的 `write` 操作覆盖原文件（保持同一文件路径）
- **再次预览**：修改后再次将文件交给用户确认
- **版本意识**：如果用户进行了多轮修改，在对话中简要标注当前是第几版（如"已更新为第 3 版"），帮助用户追踪进度

## 行为规则

- 首次创作时输出**完整文章**，不要分段输出或只给大纲让用户确认——用户要的是成品，不是半成品
- 如果文章篇幅较长（超过 3000 字），可以先输出完整大纲让用户确认方向，确认后再输出全文
- 禁止在对话中直接输出大段文章正文——所有正文内容都通过 `.md` 文件交付，对话中只做简要说明
- 当用户表示"可以了"、"没问题"、"就这样"等确认表达时，标记文章为**已确认状态**，后续模块（如模块 F）可以基于此版本执行
