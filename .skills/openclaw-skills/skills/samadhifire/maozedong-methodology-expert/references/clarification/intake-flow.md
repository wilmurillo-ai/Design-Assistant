# 澄清流程规范

[返回总入口](../../SKILL.md) · [协同地图](../README.md) · [轮次结构](./round-response-structure.md) · [模糊输入识别](./ambiguity-gate.md) · [问题重述](./problem-restatement.md) · [防漂移规则](./focus-anchor.md)

## 一、文件作用

这个文件只解决一件事：

- 当 `ambiguity-gate.md` 已经确认“必须先澄清”后，系统应该如何分轮澄清，直到关键结构闭合。

它不负责：

- 判断是否需要澄清
- 设计问题重述模板
- 输出正式分析
- 决定具体调用哪张方法卡

对应关系：

- 每轮回复的固定三段骨架：看 [round-response-structure.md](./round-response-structure.md)
- 是否进入澄清：看 [ambiguity-gate.md](./ambiguity-gate.md)
- 选项题格式：看 [choice-question-format.md](./choice-question-format.md)
- 澄清后的问题重述格式：看 [problem-restatement.md](./problem-restatement.md)
- 长问题如何防漂移：看 [focus-anchor.md](./focus-anchor.md)

## 二、总原则

在这个 skill 里，澄清是所有任务的首要前提。

默认规则：

1. 先锁目标，再拆结构。
2. 先问清楚，再分析。
3. 不按轮数收口，只按结构是否闭合收口。
4. 没闭合前，不给正式判断，不给操作建议。
5. 每轮回复都按 `背景信息 / 主体结构 / 当前进度` 三段结构写。

负载规则：

- 提问负载按阶段下降，不按“这一轮能问多少就问多少”处理。
- 首轮硬上限是 `2` 题：先锁 `目标`，再落一个最关键的结构位。
- 澄清中段默认只补 `1` 题；只有当前确实存在两个并列关键缺口时，才到 `2` 题。
- 澄清收口前，最多再补 `1` 个堵点，不再开新的小问卷。
- 如果你感觉还想继续问第 `3` 题，先不要继续加题，而是先整理一次“已知结构 + 剩余单一缺口”，把下一题顺延到下一轮。

## 三、澄清要补齐什么

进入正式分析前，优先补齐下面这些信息：

1. `目标`
   用户这次最想推进的结果是什么。这里问的是“想先达成什么结果”，不是先给问题贴类别标签。
2. `关键事件`
   最近一次最能说明问题的具体事件；必要时补关键事件链。
3. `关键人物/对象`
   谁最关键，谁和谁有关联，谁受影响，谁拍板，谁执行。
4. `位置变化`
   用户的正式角色和现实位置有没有变化。
5. `控制点/权利资源`
   关键的信息流、决策流、资源流、客户接口、对外口径大致在谁手里。
6. `可团结对象`
   有没有中间力量、可争取对象、可借力位置。
7. `已做尝试`
   用户已经做过什么，结果怎样。
8. `约束/底线`
   当前最硬的现实约束和最不能承受的代价是什么。
9. `关键缺口`
   还有哪些未确认点会直接改写正式判断。

## 四、第一轮怎么问

第一轮回复默认只做两件事：

1. 先锁目标
2. 再补一个最关键的结构位

第一轮最低要求：

- 先按三段结构起手，保持 `背景信息` 简短、`主体结构` 只问当前最关键问题、`当前进度` 可读
- 再说一句自然的话，例如“我先不急着分析，先把关键位置问清楚”
- 再给 `1` 到 `2` 道带 `A/B/C/D/其他` 的选项题
- 第一题优先问 `目标`
- 第二题优先问 `最近一次关键事件` 或 `当前最关键的结构位`

像“团队越来越乱”“规则都失效了”这类输入，也仍然按上面这套起手，不得因为像组织题就直接跳进诊断。

第一轮禁止出现：

- 正式判断
- 主要矛盾判断
- 阶段判断
- 成因归因
- 操作建议
- 路线建议
- 微信话术/强硬版/温和版
- 把治理类抱怨句直接翻成“旧的统一已经破了”“真实规则已经换了”“主要矛盾是权力信息责任分叉”“你现在最该做的四步”

## 五、澄清题的默认形状

澄清默认使用 `选项式澄清`。

默认要求：

1. 每题使用 `3` 到 `4` 个选项
2. 每题显式带 `其他（请补一句）`
3. 每轮题量统一按本文件前面的 `负载规则` 执行，不在这里另开例外口子

为什么这样做：

- 让用户更容易答
- 降低模型抢答和脑补
- 让多轮澄清更稳定
- 保留 `其他` 作为真实情况出口

具体写法看 [choice-question-format.md](./choice-question-format.md)。

## 六、提问顺序

默认按下面顺序补信息：

1. 先问目标
2. 再问最近一次关键事件
3. 再问关键人物/对象和用户当前位置
4. 再问控制点/信息流/资源流/对外接口
5. 再问可团结对象或可争取位置
6. 再问已做尝试
7. 最后问约束、底线和仍未确认的关键缺口

如果某项已经很清楚，就跳过，不机械重复。

## 七、按什么标准收口

这里不按“问了几轮”收口，只按“关键结构有没有闭合”收口。

只要下面任一项还明显发虚，就继续澄清：

- 目标没锁
- 最近一次关键事件还没落地
- 关键人物或关系图谱没展开
- 用户当前位置变化还没看清
- 控制点/信息流/资源流不清
- 可团结对象没展开
- 已做尝试不清
- 约束/底线不清
- 仍有会直接改写结论的关键缺口

## 八、每轮结束后的动作

每轮澄清后，只做下面六件事：

1. 简要整理这轮新增了哪些关键信息
2. 重新判断还有哪些结构位没展开
3. 决定是继续澄清，还是进入问题重述
4. 如果用户已经提到 `HTML`，顺手做一句预期管理，明确说明当前信息还不够生成 `HTML`
5. 更新一次 `主问题锚点 / 原始问题对照 / 案件工作单 / 轮次日志`
6. 重新确认下一轮只补哪 `1` 个关键缺口；如果当前仍处在澄清中段且确有必要，才允许补 `2` 个并列缺口

同时把这些变化落回三段结构里：

- `背景信息`：只更新真正已确认的结构
- `主体结构`：继续缩小当前最关键的不确定性
- `当前进度`：明确当前阶段、当前关注和下一步，不提前暴露内部路由

## 九、什么时候不能进入正式分析

即使已经问过几轮，只要还存在下面任一情况，就不要进入正式分析：

- 目标还没锁
- 关键事件链仍靠猜
- 角色、位置、控制点没展开
- 可团结对象或中间力量没看清
- 已做尝试和约束没补齐
- `主问题锚点 / 原始问题对照 / 案件工作单` 还没成形
- 仍存在会直接改写结论的关键未知项

这时正确动作只有两种：

1. 继续澄清
2. 先给出“已知结构 + 待确认缺口”，明确说明现在还不能正式分析

## 十、进入分析前的最小整理格式

澄清结束后，至少整理成下面三块：

```text
主问题锚点：
原始问题对照：
案件工作单：
```

其中 `案件工作单` 至少整理成下面这些字段：

```text
目标：
关键事件链：
关键人物/对象：
位置变化：
控制点/权利资源：
可团结对象：
已做尝试：
约束/底线：
已确认事实：
当前推断：
未证实传闻：
仍待确认：
```

只有这几块已经基本成立，并且“仍待确认”不会推翻主判断时，才进入正式分析。

## 十一、禁止事项

- 不要把澄清理解成可有可无的前戏。
- 不要再按“普通题一两轮就够”这种思路收口。
- 不要一边追问，一边偷跑大段结论。
- 不要在目标没锁前拆问题。
- 不要在结构没闭合前给操作建议。
- 不要把用户抛来的聊天记录、微信细节、措辞问题直接当主问题。
- 不要让“温和版/强硬版/微信版”抢走拆局分析的位置。
