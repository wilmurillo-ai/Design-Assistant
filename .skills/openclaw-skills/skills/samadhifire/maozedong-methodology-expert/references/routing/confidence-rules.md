# 分类置信度与升级规则

[返回总入口](../../SKILL.md) · [协同地图](../README.md) · [模糊输入识别](../clarification/ambiguity-gate.md) · [澄清流程](../clarification/intake-flow.md) · [输出路由](./output-mode-routing.md)

## 一、文件作用

这个文件只解决两件事：

- 当分类、方法选择或风险判断拿不准时，应该继续、追问、降级，还是暂不下强判断。
- 当输出模式拿不准时，应该维持文字分析，还是升级为 `HTML 报告`。

它不负责：

- 判断是否必须进入澄清
- 设计具体追问问题
- 替代方法卡本身的调用条件
- 直接决定某张风险卡的内容

定位提醒：

- `routing/` 是例外处理层，不是常驻分析层。
- 正常主链应尽量停留在 `clarification -> categories -> scenarios -> methods`。
- 只有真的拿不准时，才回这里做保守判断或交付判断。

对应关系：

- 输入是否足够清楚：看 [ambiguity-gate.md](../clarification/ambiguity-gate.md)
- 需要澄清时怎么追问：看 [intake-flow.md](../clarification/intake-flow.md)
- 正式输出模式怎么路由：看 [output-mode-routing.md](./output-mode-routing.md)

## 二、什么是“置信度”

这里的置信度，不是数学分数，也不是装作很确定。它只判断一件事：

- 我现在的分类、方法入口和输出方式，是否已经足够稳，可以继续往下走。

要判断的主要有四类：

1. 分类置信度：主领域、方法标签、复杂度是否基本稳。
2. 方法置信度：当前先从哪张方法卡开始是否基本选对。
3. 风险置信度：这次是否已经明显命中误用风险或翻译风险。
4. 输出模式置信度：应该维持 `深度分析`，还是升级为 `HTML 报告`。

## 三、三级置信度

### 1. 高置信度

满足下面大部分条件时，可视为高置信度：

- 主领域基本明确
- 方法标签能稳定收敛到 2 到 4 个
- 当前最早缺失的方法卡比较清楚
- 关键风险点已经看得出是否命中
- 输出方式也较清楚：文字足够，或确实需要 HTML

处理原则：

- 直接进入下一步，不必额外追问。

### 2. 中置信度

满足下面情况时，可视为中置信度：

- 大方向基本清楚，但仍有一两个关键缺口
- 两张相邻方法卡之间暂时拿不准
- 风险可能存在，但还不确定是否真的命中
- 文字分析和 HTML 两种输出都说得通，但还缺一个触发条件

处理原则：

- 不直接停下，但应补最少量信息，或先按更稳妥的低风险路径走。
- 方法上优先回到更前置、更保守的入口，例如先回 [investigation.md](../methods/investigation.md)。
- 输出上优先保留 `深度分析`，不要抢升 HTML。

### 3. 低置信度

满足下面任一情况，通常就是低置信度：

- 主领域都拿不准
- 方法标签散得太开，几乎每张卡都像能调
- 当前结论主要靠猜
- 结构层级和关键结构位分不清
- 不知道有没有命中误用风险，却已经准备使用高风险表达
- 不知道该不该做 HTML，但又想直接开始可视化

处理原则：

- 不要硬推进。
- 先回到澄清或先退回前置方法卡。
- 默认不做强定性、不做高风险表达、不做 HTML。
- 只要仍存在可能改写结论的关键未知项，就只输出“已知结构 + 待确认缺口”。

## 四、分类置信度判断

### 1. 主领域高置信度的信号

- 用户要解决的问题类型比较集中。
- 主要场景明显落在工作、学习、关系、自我管理、生活决策或团队治理中的一个。
- 补充领域只是辅助视角，不会改变主要进入角度。

### 2. 主领域中置信度的信号

- 两个主领域都像，比如既像“关系问题”又像“团队治理问题”。
- 这时先选“当前最影响结果”的那个主领域，另一个只作为补充视角。

### 3. 主领域低置信度的信号

- 连用户到底在推进什么类型的问题都说不清。
- 这时不要急着分类，先回澄清层补目标、事件、人物。

## 五、方法选择置信度判断

### 1. 高置信度

出现下面情况时，方法入口通常比较稳：

- 事实不清、对象混：优先 [investigation.md](../methods/investigation.md)
- 问题很多但抓不住重点：优先 [core-contradiction.md](../methods/core-contradiction.md)
- 旧打法失灵，怀疑转段：优先 [stage-judgment.md](../methods/stage-judgment.md)
- 资源紧、线太多：优先 [forces-resources.md](../methods/forces-resources.md)
- 多方关系复杂：优先 [alliance-boundaries.md](../methods/alliance-boundaries.md)
- 一说就空、一公开就偏：优先 [communication-calibration.md](../methods/communication-calibration.md)
- 方向有了但落不下去：优先 [execution-routes.md](../methods/execution-routes.md)
- 已有真实结果，要复盘纠偏：优先 [review-loop.md](../methods/review-loop.md)

### 2. 中置信度

- 两张相邻卡都像能用，比如 [investigation.md](../methods/investigation.md) 和 [core-contradiction.md](../methods/core-contradiction.md)，或 [stage-judgment.md](../methods/stage-judgment.md) 和 [forces-resources.md](../methods/forces-resources.md)。
- 这时优先选更前置、更保守的一张，不要先跳后段卡。

### 3. 低置信度

- 感觉 8 张卡都能调。
- 这通常说明问题还没成形，或前门没有完成。
- 处理方式：先回分类层，必要时回澄清层，再从 [investigation.md](../methods/investigation.md) 重新起手。

## 六、风险置信度判断

### 1. 高置信度：必须做风险检查

出现下面任一情况时，风险检查优先级应直接抬高：

- 准备使用高风险表达，如 `主要矛盾`、`统一战线`、`群众路线`、`统一口径`、`整风`、`合法性`、`持久战`
- 准备做公开定性、对象分层、联盟判断、资源压注、复杂 rollout
- 方法卡里的 `不直接调用` 或 `误用边界` 已明显命中
- 明显出现口号化、扩大化、清洗化、操控化、贴标签化风险

处理方式：

1. 先看 [misuse-boundaries.md](../risks/misuse-boundaries.md)
2. 再看 [translation-red-lines.md](../risks/translation-red-lines.md)
3. 必要时进入对应独立风险卡

### 2. 中置信度：先保守表达

- 风险似乎存在，但还不确定有没有真的命中。
- 这时不要急着上高压表达，先用工作语言写判断。
- 只要暂时不用高风险表达，很多问题可以先完成文字分析。

### 3. 低置信度：先不碰风险词

- 你拿不准这个词现在能不能用，也拿不准它会不会被翻坏。
- 处理方式：先不用，先换成现代工作语言，等风险判断稳定后再决定是否展示出处或术语。

## 七、输出模式置信度判断

### 1. 高置信度：可以升级 HTML

满足下面大部分条件时，可视为 HTML 升级把握较高：

- 用户明确要 `HTML 报告`
- 关键人物较多
- 时间线较长
- 阶段变化明显
- 路线比较是关键任务
- 问题涉及关系、资源、策略、执行等多层结构

处理方式：

- 可以在完成文字判断后升级为 HTML。

### 2. 中置信度：先保留文字分析

- 看起来可以做 HTML，但结构还不够稳定。
- 或者用户没明确要求，只是“好像适合可视化”。

处理方式：

- 先完成 `深度分析`。
- 在最终输出前再问一次用户要不要 HTML。

### 3. 低置信度：不要升级 HTML

- 事实仍不够清楚
- 只有一个直接动作
- 没有多路线比较
- 问题本身很轻

处理方式：

- 不做 HTML，直接维持文字分析。

## 八、拿不准时的总处理顺序

如果你现在整体都拿不准，按下面顺序退回：

1. 先看是不是输入本身不够清楚；如果是，回 [ambiguity-gate.md](../clarification/ambiguity-gate.md)
2. 再看是不是方法入口拿不准；如果是，退回更前置的方法卡
3. 再看是不是风险判断拿不准；如果是，先不用高风险表达
4. 最后看是不是输出形式拿不准；如果是，默认保留 `深度分析`

一句话原则：

- 宁可先保守，也不要假装确定。

## 九、最低保守策略

当你真的无法稳定判断时，默认采用下面这组保守策略：

- 分类上：先选最接近的主领域，暂不做过细次领域扩张
- 方法上：先从 [investigation.md](../methods/investigation.md) 或更前置卡起手
- 风险上：先不用高风险表达
- 输出上：先做 `深度分析`
- HTML 上：先不升级

## 十、禁止事项

- 不要把“不确定”伪装成“已经判断清楚”。
- 不要因为想显得有体系，就在低置信度时硬贴分类和方法标签。
- 不要方法拿不准时直接跳后段卡。
- 不要风险拿不准时仍使用高压表达。
- 不要输出方式拿不准时直接做 HTML。
