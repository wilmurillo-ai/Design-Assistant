# 工作推进与执行

[返回总入口](../../SKILL.md) · [场景索引](./scene-index.md) · [问题分类](../categories/problem-taxonomy.md) · [方法索引](../methods/method-index.md)

## 一、文件作用

这个文件只解决一件事：

- 当问题核心是“事情为什么推进不动、下一步该怎么推”时，给出一个稳定的场景入口。

它不负责：

- 替代分类体系
- 替代具体方法卡正文
- 把纯技术细节问题直接分析完

对应关系：

- 主领域判断：看 [problem-taxonomy.md](../categories/problem-taxonomy.md)
- 澄清追问：看 [question-packs-by-domain.md](../clarification/question-packs-by-domain.md)
- 方法入口：看 [method-index.md](../methods/method-index.md)

## 二、适用场景

适合下面这些问题：

- 项目推进缓慢、反复卡住、节奏越来越乱
- 目标看似明确，但执行链条接不上
- 人手、预算、时间都有限，不知道该先保哪一段
- 多人协作中责任、优先级、节奏和资源互相打架
- 明明做了很多动作，但结果始终没有形成推进优势
- 表面像执行问题，底层其实已经在变服务对象、变支点、变控制点或变承接容器

## 三、不适合作为主入口的情况

- 只是单点技术问题，已经非常清楚卡在哪个细节
- 核心不是推进，而是关系修复或边界谈判
- 重点是个人拖延、注意力涣散、状态失衡
- 重点是重大人生选择而不是当前执行推进

这几种情况，通常应改从其他场景或更前置的方法卡进入。

## 四、进入这个场景先判断什么

先看 6 件事：

1. 你到底想推进哪个结果，真正的服务对象是谁。
2. 最近一次最能说明问题的卡点事件是什么。
3. 当前真正卡住推进的是目标、资源、协同、节奏，还是负责人。
4. 当前基础盘和支点在哪：靠什么资源、什么人、什么结构把事情撑住。
5. 纸面关系和现实控制点是不是一致：谁名义上负责，谁实际上拍板、卡口、承压。
6. 当前处在推进期、转段期、收口期，还是接管期。

如果这 6 件事里有两项以上说不清，先回澄清层，不要硬分析。

## 五、推荐起手方法链

最常见的起手方式有三种：

### 1. 事实和对象都还没看清

优先：

`investigation -> core-contradiction`

适合：

- 大家都在说问题，但说不清真正堵点
- 多个对象混在一起，责任和影响范围不清

### 2. 问题重点已显现，但资源、支点和节奏失灵

优先：

`core-contradiction -> forces-resources -> execution-routes`

适合：

- 问题很多，但你需要先抓最限制结果的那个点
- 资源越紧越平均铺，导致没有一条线打出优势

### 3. 旧办法失灵，怀疑已经转段或接管

优先：

`stage-judgment -> execution-routes -> review-loop`

适合：

- 过去有效的节奏、组织方式、推进口径开始失灵
- 项目进入了新阶段，但打法还停留在旧阶段

如果执行问题背后其实是多方关系或协同边界问题，再补 [alliance-boundaries.md](../methods/alliance-boundaries.md) 或 [communication-calibration.md](../methods/communication-calibration.md)。

## 六、这类问题优先补什么信息

若需追问，优先补下面几项：

- 最想推进的结果是什么，首先要对谁产生作用
- 最近一次关键卡点事件是什么
- 谁在拍板、谁在执行、谁在拖住节奏
- 当前基础盘、支点和底盘供给分别是什么
- 当前最硬的约束是时间、人手、预算，还是组织配合
- 名义负责人和现实控制点是否一致
- 当前已经处在哪个阶段，接下来最可能进入哪个阶段

## 七、常见输出骨架

这类问题常见的输出结构是：

- 问题重述
- 服务对象与当前推进目标
- 当前真正的推进堵点
- 阶段判断与控制点判断
- 资源、支点与执行路线
- 风险与边界
- 下一步建议

如果只是轻量执行问题，不必把所有模块都拉满；但至少要把“堵点在哪里、现在先保什么、下一步先打哪一段”讲清楚。

## 八、常见误用风险

- 不要把普通执行卡点直接上升成高压政治语言。
- 不要在事实未清时，先做大范围责任归因。
- 不要一看到多人协作就默认进入复杂联盟分析。
- 不要把“做了很多动作”误当成“执行路线正确”。
- 不要只盯前台动作，不看服务对象、支点和底盘供给。

如果你已经准备做公开定性、对象分层或高压资源压注，先看 [misuse-boundaries.md](../risks/misuse-boundaries.md)。

## 九、什么时候适合升级为 HTML

下面情况通常更适合做 `HTML 报告`：

- 关键角色较多
- 阶段变化明显
- 需要把支点、控制点、责任链和动作链放进同一份交付物
- 需要做 rollout 分区或接管前后对照

如果只是一个直接动作或一个短链条判断，通常保持文字分析即可。
