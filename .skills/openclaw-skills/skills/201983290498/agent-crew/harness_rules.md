# Agent Team Builder — 护栏规则

本文档定义多 Agent 团队协作时的强制规则。每条规则包含违反后果说明，确保团队在可控范围内运转。

---

## R001: 沙盒隔离

**规则**: 每个角色只能在各自的 `.claude/teams/<team_name>/<role_name>/workspace/` 目录下创建临时文件、草稿和任务文档。禁止在 `workspace/` 以外的目录写入非结构化内容。

**适用范围**: 所有角色 Agent 在任务执行阶段的行为。

**违反后果**:
- 临时文件污染项目 `.claude/` 根目录，导致配置混乱
- 多个角色写入同名文件产生冲突
- 清理团队时需要手动逐个文件排查删除

---

## R002: 宪章至上

**规则**: `team_charter.md` 是团队结构的唯一真相源（Single Source of Truth）。任何针对团队结构的变更——包括新增角色、删除角色、修改工作流拓扑——**必须**同步更新 `team_charter.md`。

**适用范围**: 团队结构变更操作（非日常任务执行）。

**违反后果**:
- 二次加载团队时，`team_charter.md` 与实际结构不一致，导致 Agent 实例化失败
- 新成员（或新 Agent）读取宪章得到错误的团队信息
- 工作流拓扑断裂，角色间协作触发条件失效

---

## R003: 渐进式披露

**规则**: `progress.md` 仅存放一句话摘要（干了什么事、当前状态）。具体任务的完整文档、代码、执行记录必须存放在 `workspace/` 下的专门文件中。严禁在 `progress.md` 中塞入大量代码、长文本或完整配置文件。

**适用范围**: 所有角色 Agent 的任务留痕行为。

**违反后果**:
- `progress.md` 文件过大，SendMessage 状态对齐时传输冗余数据
- 其他角色快速扫描进度时，被大量细节淹没
- generate_prompts.py 读取 progress.md 作为 prompt 输入时，挤占上下文窗口

---

## R004: 技能作用域限定

**规则**: 角色私有技能文件必须且只能存放在 `.claude/teams/<team_name>/<role_name>/skills/` 目录下。一个角色不能读取、修改或删除其他角色的 `skills/` 内容。项目级共享技能（`.claude/skills/`）的复制操作是单向拷贝，源文件保留不动。

**适用范围**: 技能的创建、复制、删除操作。

**违反后果**:
- 角色 A 修改了角色 B 的私有技能，导致角色 B 行为异常
- 项目级共享技能被误删除，其他依赖该技能的角色全部失效
- 技能归属不清，重复创建同名技能产生版本分裂

---

## R005: 状态对齐只做回顾

**规则**: Agent 实例化后的第一步（状态对齐阶段），每个角色只能做三件事：
1. 读取自己的 `progress.md`，理解当前进度
2. 读取自己的 `memory.md`，了解历史经验
3. 给出一份简短的就绪报告（我是谁、职责是什么、当前状态）

**禁止**：在状态对齐阶段创建新文件、编写文档、设计方案、写代码或执行任何其他产出行为。所有具体任务必须等待用户明确指令后，由 team-leader 按需派发。

**违反后果**:
- Agent 自行假设需求并开始"干活"，产出非用户期望的文档/代码
- 用户还没来得及确认需求，团队已经基于错误假设产生了大量无效输出
- 团队失去"以用户为中心"的节奏控制，变成自主发散

---

## R006: team-leader 必须先问再做

**规则**: team-leader 的"需求沟通"类任务，第一步必须是向用户提出具体问题、等待用户回答，然后才能整理成文档（PRD、需求规格等）。禁止在未与用户确认的情况下自行假设需求并直接输出文档。

**适用范围**: 需求收集、功能范围确认、PRD 编写等涉及用户意图的任务。

**违反后果**:
- team-leader 拍脑袋写出与实际期望不符的 PRD
- 后续所有角色（designer、developer）基于错误需求工作，造成大量返工
- 用户信任度下降，认为团队没有真正理解需求
