# 系统架构说明

## 设计理念

Memory Manager的设计遵循以下核心原则：

### 1. 简化优先
- 避免复杂的目录层级
- 使用直观的文件命名
- 减少自动化，增加可控性

### 2. 代理中心化
- 主代理作为记忆中心
- 子代理通过主代理协调
- 清晰的权限边界

### 3. 按需管理
- 无定期自动任务
- 所有操作按需执行
- 用户完全控制

### 4. 记录完整
- 所有操作都有记录
- 记录可追溯可审计
- 便于问题排查

### 目录结构详解

#### 1. 主记忆 (main/)
```
main/
├── long-term/      # 长期记忆
│   ├── preferences/    # 用户偏好
│   ├── knowledge/      # 知识库
│   └── decisions/      # 重要决策
├── daily/          # 每日日志
│   ├── 2026-04-17.md
│   ├── 2026-04-16.md
│   └── ...
└── shared/         # 共享文件（活跃）
    ├── character_profiles.md
    ├── plot_outlines.md
    └── ...
```

**设计考虑**：
- `long-term/`: 低频访问，高价值记忆
- `daily/`: 高频更新，短期记忆
- `shared/`: 当前活跃，需要共享的文件

#### 2. 共享池 (shared_pool/)
```
shared_pool/
├── writing/        # 写作相关
├── analysis/       # 分析相关
├── research/       # 研究相关
├── projects/       # 项目相关
└── templates/      # 模板文件
```

**设计考虑**：
- 按类型分类，便于查找
- 跨代理共享的集中位置
- 项目隔离，避免混乱

#### 3. 归档系统 (archive/)
```
archive/
├── project_progress_archived_20260417.md
├── test_archive_archived_20260417.md
└── ...
```

**设计考虑**：
- 扁平结构，无子目录
- 文件名包含归档日期
- 便于搜索和清理

#### 4. 记录系统 (logs/)
```
logs/
├── operations/     # 操作记录
│   ├── op_20260417_135217.json
│   ├── op_20260417_135537.json
│   └── ...
└── exchanges/      # 交换记录
    ├── ex_20260416_142612.json
    ├── ex_20260416_145436.json
    └── ...
```

**设计考虑**：
- 按类型分开记录
- 时间戳文件名，便于排序
- JSON格式，便于解析

#### 5. 临时文件 (temp/)
```
temp/
└── to_delete/      # 待删除文件
    ├── file1_delete_20260417.md
    └── file2_delete_20260417.md
```

**设计考虑**：
- 删除前暂存，防止误删
- 保留期可配置（默认30天）
- 定期清理机制

#### 6. 代理访问 (agent_access/)
```
agent_access/
└── agent_mapping.json  # 代理映射配置
```

**设计考虑**：
- 配置文件代替符号链接（兼容性）
- 集中管理代理路径
- 便于扩展新代理

## 数据流设计

### 1. 文件生命周期
```
创建 → 使用 → 评估 → 决策 → 处理
   ↓       ↓       ↓       ↓       ↓
共享目录 → 活跃使用 → 定期检查 → 归档/保留/删除 → 记录
```

### 2. 代理间记忆交换
```
主代理共享池 → 配置映射 → 子代理访问
       ↑                           ↓
   记录交换 ←─────────────── 子代理使用
```

### 3. 操作记录流
```
用户指令 → 工具执行 → 结果验证 → 记录生成 → 文件保存
   ↓           ↓           ↓           ↓           ↓
明确指令    脚本工具    状态检查    JSON格式    日志目录
```

## 工具架构

### 1. 工具设计原则
- **极简化**：避免复杂语法结构
- **单一职责**：每个工具只做一件事
- **错误容忍**：失败时提供明确信息
- **记录完整**：所有操作都有记录

### 2. 工具分类

#### 检查类工具
- `memory_check.ps1`: 系统状态检查
- `agent_check.ps1`: 代理状态检查

#### 操作类工具
- `archive_tool.ps1`: 文件归档
- `simple_logger.ps1`: 操作记录

#### 配置类
- `config/default.json`: 系统配置

### 3. 工具调用链
```
用户指令
    ↓
memory_check.ps1 (状态检查)
    ↓
agent_check.ps1 (代理检查)
    ↓
archive_tool.ps1 (文件操作)
    ↓
simple_logger.ps1 (自动记录)
    ↓
日志文件 (JSON格式)
```

## 配置管理

### 1. 配置文件结构
```json
{
  "system": {},      // 系统信息
  "paths": {},       // 路径配置
  "agents": {},      // 代理配置
  "retention": {},   // 保留策略
  "naming": {},      // 命名规则
  "operations": {},  // 操作配置
  "monitoring": {},  // 监控配置
  "categories": {}   // 分类配置
}
```

### 2. 配置加载逻辑
```
启动 → 检查默认配置 → 检查自定义配置 → 合并配置 → 应用配置
   ↓           ↓               ↓             ↓           ↓
工具执行    config/         config/      优先级:      环境变量
          default.json    custom.json    custom > default
```

### 3. 配置验证
- 路径存在性检查
- 权限验证
- 格式验证
- 依赖检查

## 安全设计

### 1. 权限控制
- 主代理完全控制
- 子代理只读访问（默认）
- 操作需要明确授权
- 记录所有访问

### 2. 数据保护
- 敏感信息不共享
- 归档前内容检查
- 删除前暂存期
- 定期备份机制

### 3. 审计追踪
- 所有操作有记录
- 记录不可修改
- 时间戳精确
- 操作者标识
