# Duty Table — Dynamic Model Assignment

## duty-table.json Structure

```json
{
  "assessedAt": "ISO timestamp",
  "nextAssessment": "ISO timestamp",
  "availableAgents": {
    "claude": { "cli": "claude", "status": "active", "models": {...} },
    "codex":  { "cli": "codex",  "status": "active|benched", "models": {...} },
    "gemini": { "cli": "gemini", "status": "active", "models": {...} }
  },
  "dutyTable": {
    "architect":  { "agent": "claude", "model": "claude-opus-4-6", "nonInteractiveCmd": "..." },
    "builder":    { "agent": "codex",  "model": "gpt-5.3-codex",  "nonInteractiveCmd": "..." },
    "reviewer":   { "agent": "gemini", "model": "gemini-2.5-flash","nonInteractiveCmd": "..." },
    "integrator": { "agent": "claude", "model": "claude-opus-4-6", "nonInteractiveCmd": "..." }
  },
  "history": [...]
}
```

## Role Resolution Flow

When `spawn-agent.sh` receives a role (e.g. `builder`):

1. Read `duty-table.json` → get `agent` + `model` for that role
2. If `fallback-swap.sh` exists, test primary model first
3. If primary fails, swap to fallback and update duty table
4. If no duty table exists, use hardcoded defaults:
   - architect → claude/opus
   - builder → claude/sonnet
   - reviewer → claude/sonnet
   - integrator → claude/opus

## assess-models.sh

Tests all configured models with a simple probe ("Reply with HELLO"):
- 45-second timeout per model
- Checks output for error patterns (401, 429, quota, rate limit)
- Assigns optimal 3-vendor spread based on results:
  - architect: always Claude Opus (best reasoning)
  - builder: Codex preferred, fallback to Sonnet
  - reviewer: Gemini Pro/Flash preferred, fallback to Sonnet
  - integrator: always Claude Opus

### Fallback Mode
If both Codex AND Gemini are unavailable → all-Claude table:
- architect: Claude Opus
- builder/reviewer/integrator: Claude Sonnet

### Scheduling
Run daily via cron or OpenClaw cron job:
```bash
# Manual
bash assess-models.sh
bash assess-models.sh --dry-run    # Test without updating
bash assess-models.sh --fallback   # Force Claude-heavy table
```

## model-fallback.sh — Mid-Run Failover

Called by the runner when a token/rate limit is detected.

**Per-role fallback chains:**

| Role | Chain (priority order) |
|------|----------------------|
| architect | claude-opus → claude-sonnet → gemini-flash |
| builder | codex → claude-sonnet → gemini-flash → claude-opus |
| reviewer | gemini-flash → gemini-pro → claude-sonnet → claude-opus |
| integrator | claude-opus → claude-sonnet → gemini-flash |

For each candidate:
1. Skip the model that just failed
2. Run `try-model.sh` (15s health check)
3. Return first working model

Output format: `agent|model|nonInteractiveCmd`

## Runner Auto-Retry

The runner script (generated by `spawn-agent.sh`) wraps agent execution:

```
Agent runs → exits with error
  ↓
Is it a token/rate/quota error?
  ├─ No → exit with original error code
  └─ Yes → call model-fallback.sh
       ├─ Found fallback → switch agent, retry (max 2 retries)
       │   └─ Update duty-table.json for future spawns
       └─ No fallback → exit with error, notify human
```

### Error Detection Patterns
```
rate.limit | 429 | quota | token.limit | exceeded.*limit |
capacity.*exceeded | too.many.requests | billing | budget |
overloaded | model.*limit
```

## Manual Override

Set a manual override in duty-table.json:
```json
{
  "manualOverride": {
    "enabled": true,
    "setBy": "WB",
    "note": "Codex benched today. All roles use Claude."
  }
}
```

Remove `manualOverride` key to return to automatic assessment.
