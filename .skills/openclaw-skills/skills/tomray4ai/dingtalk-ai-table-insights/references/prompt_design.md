# AI 表格洞察 - Prompt 设计指南

## 核心原则

### 1. 角色定义清晰
```
你是一位资深的数据分析专家和业务顾问，擅长从多维度数据中发现洞察、识别风险、给出可执行建议。
```

**为什么重要：** 明确角色让大模型进入专业状态，输出更有深度。

---

### 2. 任务背景完整
```
用户希望你分析最近 {days} 天内访问的 {len(tables_data)} 个 AI 表格，生成一份有深度的洞察报告。
```

**包含要素：**
- 时间范围
- 数据规模
- 输出目标

---

### 3. 分析框架结构化

使用**四维分析法**：

| 维度 | 目的 | 关键问题 |
|------|------|----------|
| 🔍 数据一致性检查 | 发现矛盾 | 不同表格的相同指标是否一致？ |
| 💡 趋势洞察 | 发现模式 | 数据变化意味着什么？表格间有何关联？ |
| 🚨 风险预警 | 识别问题 | 什么可能导致损失？优先级如何？ |
| 📋 行动建议 | 指导行动 | 具体做什么？谁来做？何时完成？ |

**为什么有效：**
- 覆盖完整分析流程（发现问题 → 分析原因 → 评估风险 → 给出建议）
- 每个维度有明确的分析要点，避免大模型泛泛而谈

---

### 4. 输出要求具体化

### ❌ 不好的要求
```
请分析数据，给出洞察。
```

### ✅ 好的要求
```
【输出要求】
- 每个洞察要有数据支撑（引用具体数值）
- 说明洞察的业务含义（这意味着什么）
- 按【高/中/低】优先级标注风险
- 建议要包含：做什么 + 谁来做 + 何时完成
```

**关键要素：**
- 数据支撑（避免空泛）
- 业务含义（体现洞察价值）
- 优先级排序（帮助决策）
- 可执行性（责任 + 时间 + 目标）

---

### 5. 提供输出示例

给出完整的示例输出，让大模型理解：
- 格式风格（Markdown、emoji、加粗）
- 内容深度（具体到什么程度）
- 长度范围（800-1500 字）

**示例的作用：**
- 统一输出风格
- 设定质量基准
- 减少迭代次数

---

### 6. 特殊情况处理

告诉大模型遇到以下情况如何处理：

| 情况 | 处理方式 |
|------|----------|
| 数据量不足（<5 条） | 说明"样本量有限，结论仅供参考" |
| 部分表格无法访问 | 说明"共 X 个表格，成功分析 Y 个" |
| 无明显问题 | 如实说明"数据一致性良好，未发现明显异常" |
| 发现严重问题 | 用 🚨🚨🚨 标记，放在报告开头 |

**为什么重要：** 避免大模型"硬编"结论，保持专业性。

## Prompt 迭代记录

### v1.0 (初始版本)
- 基础四维分析框架
- 简单输出要求

### v2.0 (优化版本)
- 增加角色定义
- 细化每个维度的分析要点
- 添加完整输出示例
- 增加特殊情况处理

### 未来优化方向
1. **行业适配** - 针对销售/项目/人力等不同场景定制分析维度
2. **对比分析** - 增加"与上周/上月对比"的维度
3. **图表建议** - 让大模型建议适合的数据可视化方式
4. **追问机制** - 发现异常后，主动建议需要补充的数据

## Prompt 使用技巧

### 1. 温度设置
- **分析类任务**: temperature=0.3-0.5（偏保守，减少幻觉）
- **建议类任务**: temperature=0.5-0.7（适度创意）

### 2. 上下文管理
- 表格数据放在 prompt 中间（前后都有指令，避免被忽略）
- 重要指令放在最后（大模型对末尾内容更敏感）

### 3. 调试方法
- 先用小数据集测试（1-2 个表格）
- 检查输出是否符合预期
- 逐步增加数据量

## 常见问题分析

### Q1: 输出太泛泛
**原因：** 分析要点不够具体
**解决：** 在每个维度下增加"分析要点"子项，给出具体问题

### Q2: 建议不可执行
**原因：** 没有明确要求包含责任人、时间
**解决：** 在行动建议部分明确要求"做什么 + 谁来做 + 何时完成"

### Q3: 数据引用混乱
**原因：** 没有要求注明来源
**解决：** 在输出要求中明确"提到数据时注明来源表格"

### Q4: 报告过长
**原因：** 没有长度限制
**解决：** 明确"总长度 800-1500 字，适合钉钉阅读"
