# Documentation Conventions Discovery Guide

How to discover and follow any project's documentation conventions. Instead of prescribing a fixed style, this guide teaches you to infer conventions from the project's existing documentation.

## Step 1: Sample Existing Documentation

Read 3-5 existing documentation files of the same type you're about to create or edit. Look for:

- **Frontmatter schema**: What fields are used? Which are required?
- **Heading hierarchy**: What level do pages start at? How deep do they go?
- **Code block style**: Are language tags used? Filename annotations? Line highlighting?
- **Callout/admonition format**: How are notes, warnings, and tips formatted?
- **Cross-references**: How do docs link to each other?
- **Structural patterns**: What sections appear consistently (intro, usage, API, examples)?

## Step 2: Check for a Style Guide

Many projects document their own conventions. Search for:

- `CONTRIBUTING.md` — often has a documentation section
- `docs/CONTRIBUTING.md` or `docs/STYLE_GUIDE.md`
- `.markdownlint.json` / `.markdownlint.yml` — enforced Markdown rules
- `.prettierrc` with Markdown settings
- `docs/README.md` — sometimes describes how to write docs
- `.editorconfig` — indentation and formatting rules

## Common Convention Patterns

### Frontmatter

Most documentation systems use YAML frontmatter. Common schemas:

**Minimal (most projects):**
```yaml
---
title: Page Title
description: Brief description of the page content.
---
```

**Extended (larger doc sites):**
```yaml
---
title: Page Title
description: Brief description.
sidebar_label: Short Title    # Docusaurus
sidebar_position: 3           # Docusaurus
weight: 30                     # Hugo
nav_title: Short Title         # Custom
tags: [api, reference]         # Various
---
```

**Sphinx (RST):**
```rst
Page Title
==========

:description: Brief description.
:tags: api, reference
```

Always match the exact frontmatter fields used by existing pages in the same directory.

### Code Blocks

**Language tags** — always include them:
````markdown
```python
def example():
    pass
```
````

**Filename annotations** — use if the project does:
````markdown
```javascript title="src/config.js"
module.exports = { ... }
```
````

Different doc systems use different annotation syntax:

| System     | Filename Syntax                          |
| ---------- | ---------------------------------------- |
| Docusaurus | ` ```js title="file.js" `               |
| VitePress  | ` ```js [file.js] `                      |
| Sphinx     | ` .. code-block:: python` + `:caption:`  |
| mdBook     | ` ```rust,edition2021 `                  |
| Generic MD | ` ```js filename="file.js" `             |

**Multi-language examples** — if the project provides examples in multiple languages, follow the same pattern (tabs, switcher components, or sequential blocks).

### Callouts and Admonitions

Projects use different callout styles. Match what exists:

**Blockquote style:**
```markdown
> **Note**: Important information here.

> **Warning**: Be careful with this.
```

**Docusaurus-style:**
```markdown
:::note
Important information here.
:::

:::warning
Be careful with this.
:::
```

**GitHub-style:**
```markdown
> [!NOTE]
> Important information here.

> [!WARNING]
> Be careful with this.
```

**Sphinx-style (RST):**
```rst
.. note::
   Important information here.

.. warning::
   Be careful with this.
```

### Headings

- Check what level pages start at (`#` vs `##`) — some systems use `#` only in frontmatter title
- Check maximum depth used in the project
- Check for consistent ordering (e.g., Overview → Usage → API → Examples → Related)

### Links and Cross-References

Different systems use different link formats:

| System     | Format                                          |
| ---------- | ----------------------------------------------- |
| Standard MD | `[text](../relative/path.md)`                  |
| Docusaurus | `[text](/docs/category/page)` (URL-based)       |
| VitePress  | `[text](./relative/path.md)` (file-based)       |
| Sphinx     | `` :ref:`label` `` or `` :doc:`path` ``         |
| Hugo       | `{{< ref "path/to/page.md" >}}`                |

Check how existing docs link to each other and follow the same pattern.

### Tables

Check whether the project uses:

- Standard Markdown tables
- HTML tables (for complex layouts or responsive scroll wrappers)
- RST grid or simple tables
- Custom table components

## Step 3: Check for Auto-Generation

Some documentation is generated from source code. Signs include:

- Comments like `<!-- Generated by ... -->`, `<!-- DO NOT EDIT -->`, `<!-- auto-generated -->`
- Identical structure across many files (suggesting a template)
- A generation script in `package.json`, `Makefile`, or `scripts/`
- Config files for TypeDoc, JSDoc, Sphinx autodoc, rustdoc, godoc

If docs are auto-generated, update the **source** (code comments, templates, config) rather than the generated output.

## Step 4: Validate Your Changes

### Linting

Look for documentation linting in the project:

- `markdownlint` — Markdown style enforcement
- `vale` — prose linting (style, grammar)
- `rstcheck` — RST validation
- `sphinx-build -W` — Sphinx with warnings as errors
- Custom lint scripts in `package.json` or `Makefile`

### Build verification

If the project has a documentation build step, run it to catch:

- Broken internal links
- Invalid frontmatter
- Missing images or assets
- Syntax errors in code blocks

### Preview

Many doc systems have a local preview:

```bash
# Check package.json, Makefile, or docs README for the exact command
npm run docs:dev          # Docusaurus, VitePress, etc.
mkdocs serve              # MkDocs
sphinx-autobuild docs/ _build/  # Sphinx
mdbook serve              # mdBook
hugo server               # Hugo
```

## Writing Style Guidelines

When no explicit style guide exists, follow these universal principles:

- **Match the existing voice**: Read nearby docs and match their tone (formal vs conversational, "you" vs passive)
- **Be specific**: Use exact names for functions, parameters, and config options
- **Show, then explain**: Lead with a code example, then explain what it does
- **Keep it scannable**: Use headings, lists, and tables for quick reference
- **Avoid assumptions**: Don't assume the reader knows project internals unless the doc is explicitly for contributors
