# 复杂场景：一句话约会的 OpenClaw 技能分析与设计方案

## 1. 场景重构与分析

**用户输入示例：** “我需要你帮我约张三、李四在本周内，参加一个会议，内容有关新产品发布，时间大概 1 小时。”

这个场景远不是一个简单的“调用 API”就能完成的。它涉及到一个完整的、具有状态记忆的 Agent Workflow（工作流）。
根据你的描述，这个过程可以拆解为以下几个核心能力块：

### 1.1 意图识别与缺失信息追问 (Skill 1 或 Agent 前置)
*   **输入：** 用户的自然语言。
*   **处理：** 提取关键信息（参会人：张三、李四；时间范围：本周；主题：新产品发布；时长：1小时）。
*   **边界情况处理：** 如果用户只说“帮我约张三和李四开个会谈新产品”，Agent 必须能识别出缺失了“时间范围”和“预计时长”，主动反问：“好的，请问期望的时间范围（如本周/下周）和会议大概需要准备多长的时间呢？”

### 1.2 日程忙闲交集计算 (Skill 2 - 即我们之前写的核心逻辑)
*   **输入：** 提取好的结构化数据（参会人 user_ids, start_time, end_time, duration）。
*   **处理：** 查所有人的 busy 状态，计算 free 交集。
*   **复杂化考量：**
    *   **时段筛选规则：** 不能仅仅算出交集，还要过滤掉**非工作时间**（比如晚上 8 点之后的交集就不能推给用户）。
    *   **时长匹配：** 找出的空闲块必须 `>= 1小时`。
    *   **没有完全交集的情况如何降级？** 如果张三和李四本周连 1 小时的共同空闲时间都没有怎么办？是建议时间减半？还是只找各自有一半空闲的时间？（通常的处理是返回给 Agent：**“张三和李四本周没有持续1小时的共同空位。张三X日Y时有空，李四Z日W时有空，是否需要扩大范围到下周，或者缩短会议时间？”**）。

### 1.3 拟人化的话术生成与卡片/消息分发 (Skill 3)
*   **需求理解：** “以我的身份向参会人分别询问是否有空，用语礼貌得体。”
*   **处理机制差异：** 如果你作为应用机器人发消息，你顶多是“xx机器人替[你的名字]询问”，无法完全假扮成你在私人聊天里的对话。
*   **动作：** 虽然不能假扮本人，但语气必须是由 LLM 润色过的。在包含“候选时间按钮”的卡片前，要加上礼貌的导语（这是 OpenClaw Agent 发送时的 prompt 设定的功劳）。

### 1.4 状态轮询与事件流驱动 (Skill 4 - 核心挑战)
*   **用户要求：** “建立一个30秒的轮询，半个小时以后，变成2分钟的轮询... 看看各位参会人是否有回复，回复的内容是什么，根据回复的内容决定下一步”。
*   **架构冲突：** 
    *   **飞书官方实践 ❌ 不推荐轮询机制**：去不断轮询几百上千个人的聊天记录是极其浪费资源且飞书 API 有限频（Rate Limit）的。
    *   **正确实践 ✅ 基于 Callback/Webhook 的事件驱动机制**。不要去“看他们回没回”，而是让飞书在他们“点了卡片按钮”或者“在这个卡片下回复了文字”时，**主动推给你**。

## 2. 深入探讨：轮询 vs 事件驱动 (Webhook)

这是最核心的认知升级。如果你要做长达几个小时的状态追踪，轮询是会把你的代码写成“屎山”且容易崩溃的。

### 为什么不要用循环去“看记录”？
- **Node.js 进程死掉怎么办？** `setTimeout` 或者 `setInterval` 在服务器重启时全都丢了，这表示昨天发起的排期今天直接变成了孤儿进程。
- **飞书 API 频率限制：** 如果你有 100 个正在等回音的会议，每 30 秒查 10 个人的聊天，你很快就会被飞书封印（Token 限流）。

### 正确的 OpenClaw Agent 架构设计：
**我们把它变成一个“状态机 (State Machine)”。**

1.  **分发任务：** Agent 给他们发了卡片（包含时间1、时间2、都没空），同时把这个会议的“排期事务 ID”存进数据库（带有一个状态 `status: 'WAITING'`）。这就结束了当前回合（不阻塞 Node.js 进程）。
2.  **被动等待触发：** 飞书的人点击了卡片，或者回复了消息。飞书的 Webhook 发送一个 POST 请求给你。
3.  **状态收集：** 你的服务收到 Webhook，发现是“排期事务 ID 001”的李四回复的。他在数据库里更新李四的状态为 `READY(选了时间2)`。
4.  **裁决与跟进 (Agent 决策)：** 你的服务检查数据库：
    *   **情况 A：所有人都选了时间2。** 触发终点动作：去建日程，并通知所有人选定了。
    *   **情况 B：张三选了时间1，李四选了都没空。** 你的服务将这一僵局交给 LLM（提示词可以写：当前收集情况冲突，请决定下一步策略）。LLM 会判断，并借由 Skill 再发一条消息给李四：“了解，那我再看看下个礼拜的时间，下周三你有空吗？”
    *   **情况 C：“定时器”机制。** 我们需要一个 Cron Job 或者消息队列自带的 Delayed Task（比如 30 分钟后检查）。30 分钟到了，任务苏醒，检查数据库，发现张三还没回。触发 Agent 的“催办动作”发一句话：“张总不好意思打扰，麻烦看下上面的会您哪个时间方便~”。

## 3. 把这些落地成具体的 OpenClaw 技能结构

为了满足你这个复杂的宏大场景，你需要封装的不是一个简单的一件到底脚本，而是一组给 Agent 挂载的 **原子化原子技能 (Atomic Skills)**，以及配合一个状态管理中心。

### 给你规划的 Skill 体系长这样：

#### 技能一：日历排期计算器 (`calc_free_time`)
*   **输入：** `userIds` (谁), `timeRange` (那几天), `duration` (多长).
*   **逻辑：** 拉取所有人日历 $\rightarrow$ 合并 $\rightarrow$ 取补集 $\rightarrow$ 过滤非工作时间 $\rightarrow$ 过滤小于 `duration` 的时段 $\rightarrow$ 选出最优的 N 个推荐选项。
*   **输出：** 给 LLM 返回 `["10-24 10:00", "10-25 14:00"]` 或者是 `None` (并附注冲突原因，比如刘总这两周全满了)。

#### 技能二：状态发包与意向收集单 (`dispatch_schedule_cards`)
*   **输入：** `options` (供选时间), `userIds`, `meetingTopic`, `messageText` (LLM润色过的话术).
*   **逻辑：** 为这批人生成一个唯一的 `SessionID`，写入数据库状态为待定。调用飞书把卡片和 LLM 的话术发出去。
*   **输出：** `SessionID` (交给 Agent 挂起记录)。

#### 技能三：排期综合决策器 (`check_schedule_status` 或者被 Webhook 触发)
*   **输入：** 用户的点击行为，或文字回复。
*   **逻辑：** 当飞书 Webhook 拍你的肩告诉你有动静了，你从库里捞出这个 `SessionID` 的所有参与者状态。如果大家意见一致，就调用**技能四**；如果不一致，把状态喂给 LLM：“当前进展是大家时间对不上，请生成斡旋话术”。

#### 技能四：最终敲定执行器 (`commit_calendar_event`)
*   **输入：** 最终决定的确切 `time`, `userIds`, `topic`.
*   **逻辑：** 调用飞书 `/calendar/v4/events` 真的创建日程，并在飞书上发消息给所有人：“已经帮你们锁定并约好会议啦！”

---

## 💡 总结你的下一步怎么做

1.  如果在 OpenClaw 里，**你不需要写前面那个 30 秒、2 分钟的死循环来检查他们有没有回复**。依靠回调。
2.  对于“未写明时间要求会反问”，这本身就是 LLM 的基础能力，你只要在 System Prompt 里写好：“你是一个排期小助手，如果是约会议查时间，必须强制向用户确认好时间范围和时长，如果没有，必须反问收集”。
3.  这几个动作应该拆成 **不同的子工具 (Tools/Skills)** 提供给 Agent，而不是写在一个几千行的面条代码函数里。当你的回调接口接收到别人的拒绝时，它是带着之前的记忆去找 LLM 问接下来该说啥去安抚和另找时间的。
