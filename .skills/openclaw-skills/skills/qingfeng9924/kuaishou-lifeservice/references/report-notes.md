# 商家诊断报告生成器 - 完整文档

本文档整合了商家诊断报告 Skill 的完整使用说明和技术细节。

---

## 一、Skill 概述

本 skill 用于查询快手本地生活商家诊断数据，并自动生成 markdown 格式诊断报告，包含总分、交易、流量、视频、直播、用户六大模块的详细数据。

### 核心触发场景

- **经营表现**：用户问"经营表现如何"、"最近经营情况"、"经营状况"等
- **GMV表现**：用户问"GMV表现"、"交易额多少"、"支付金额"、"核销金额"等
- **经营数据**：用户问"经营数据"、"商家数据"、"店铺数据"、"诊断数据"等

### 触发关键词

以下关键词会触发此skill：
- 经营表现、经营数据、商家数据、店铺数据
- GMV、交易额、支付金额、核销金额
- 流量数据、播放量、曝光量
- 用户数据、支付用户、核销用户
- 直播数据、直播GMV、直播转化
- 视频数据、视频GMV、视频转化
- 诊断报告、商家诊断、诊断得分
- 自建渠道、美团渠道

---

## 二、默认参数说明

**重要**：以下参数在用户未明确指定时使用默认值：

1. **渠道类型**：默认 `1 2`（全渠道：自建+美团）
2. **日期范围**：用户指定周期范围排除今日，未指定周期询问用户周期
   - 计算方式：`endDate = 昨天`，`beginDate = endDate - 6天`
3. **日期限制**：最多只能查询31天（1个月）的数据
   - 超过31天会提示错误并拒绝查询

---

## 三、输入参数

在使用前需要确认以下信息：

### 1. 日期范围（可选，用户指定周期范围排除今日，未指定周期询问用户周期）

- `beginDate`：开始日期，格式 `YYYYMMDD`
- `endDate`：结束日期，格式 `YYYYMMDD`

### 2. 渠道类型（可选，默认全渠道）

- 默认查询全渠道（1和2），不允许查询单个渠道

### 3. 认证信息

- `ACCESS_TOKEN`：必须提供（环境变量）

## 五、执行步骤

### 步骤1：确认环境变量

确保已设置必要的环境变量：

```bash
export ACCESS_TOKEN='your-access-token'
```

### 步骤2：运行查询脚本

使用以下命令查询数据并生成报告（默认全渠道，近7日不包含今天）：

```bash
python3 "<skill_directory>/scripts/query_report.py" \
  --begin-date 20260315 \
  --end-date 20260321 \
  --channel 1 2 \
  --format markdown
```

### 步骤3：输出报告

脚本会自动生成markdown格式的诊断报告，包含：

1. **诊断总分**：总分及各子模块得分
2. **交易诊断**：支付金额、核销金额等
3. **流量诊断**：视频播放量、直播播放量等
4. **视频诊断**：视频支付金额、视频核销金额等
5. **直播诊断**：直播支付金额、直播核销金额等
6. **用户诊断**：支付用户数、核销用户数等

## 六、输出格式要求

**⚠️ 核心要求**：
1. 最终报告**必须包含原始数据**，不可省略！
2. 查询数据**必须完整无丢失**，所有API返回的数据都要完整展示！
3. **只要涉及到这个skill的功能就查询所有数据，且不允许丢失任何字段！**
4. 时间范围**不能包含今天**，endDate 必须是昨天或之前的日期！
5. **不要添加接口返回以外的信息**，所有展示的数据必须来自API返回结果！
6. **禁止编造任何信息**，不要添加推断、建议、评价等非原始数据内容！
7. **不支持查询单个模块**，即使提问只涉及某个模块，仍需返回全部六大模块的数据！

### 1. 报告结构

每个模块**必须**包含：
- **📋 数据表格**：**必须展示**，使用 markdown 表格展示所有原始数据，不可省略

### 2. 数据完整性要求

- **禁止数据丢失**：所有API返回的数据字段必须完整展示，不可遗漏任何字段
- **时间范围准确**：查询时间范围不能包含今天，endDate 必须是昨天或之前
- **渠道完整**：默认查询全渠道（自建+美团），确保数据覆盖完整
- **必须展示所有字段**：只要涉及到这个skill的功能就查询所有数据，且不允许丢失任何字段

### 3. 全量数据要求

- **禁止查询单个模块**：不支持 `--metric` 参数，始终查询全部六大模块
- **全量返回**：即使提问只涉及某个模块（如"交易数据"），仍需返回全部六大模块的完整数据

### 4. 数据真实性要求

- **禁止添加接口返回以外的信息**：所有展示的数据必须来自API返回结果
- **禁止编造数据**：不要添加计算值、推断值或估算值
- **禁止添加不存在字段**：只展示API实际返回的字段

### 5. 表格展示规则

- 所有原始数据必须以 markdown 表格形式展示
- 表格应包含**所有返回的数据字段**，不可遗漏任何字段
- 数值应格式化（大数字转为万、亿）
- 必须包含"超越同类商家"和"大致所在位置"列

## 八、指标详细说明

### 1. 诊断得分 (merchantDiagnosisScore)

综合评估商家在多个维度的表现，包括：

- **总分 (totalScore)**：综合诊断得分
- **交易 (trade)**：交易维度得分
- **流量 (traffic)**：流量维度得分
- **视频 (photo)**：视频维度得分
- **直播 (live)**：直播维度得分
- **用户 (user)**：用户维度得分

**分数范围**：0-100分

**百分位说明**：
- `percent` 字段表示超越同类型商家的百分比
- 例如 `90.00%` 表示超越了90.00%的同类型商家
- 大致所在位置 = 100% - percent

**API响应示例**（脱敏）：
```json

    "name": "总分",

    "value": 85.50,

    {"key": "traffic", "name": "流量", "type": "subScore", "value": 82.00, "percent": 87.00},
```

### 2. 交易诊断 (merchantDiagnosisTradeCardOverview)

核心交易指标：

| 指标 | 字段名 | 说明 |
|------|--------|------|

## 九、数据特点

### 1. 百分位信息

每个指标都包含 `percent` 字段，表示超越同类型商家的百分比。

**解读示例**：
```
支付金额: 50.00万，percent: 90%
→ 支付金额为50.00万，超越了90%的同类型商家
→ 大致所在位置：前10%
```

### 2. error_code=807000

**现象**：接口返回数据时，`extra.error_code` 可能为 `807000`，`description` 为"业务处理失败"。

**说明**：这是正常现象，表示"有数据但带业务告警"的返回形态。从结果看，数据是有效的，可以正常使用。

### 3. 缺失环比数据

**现象**：当前API返回中不包含上一周期数据。

**影响**：无法自动计算环比变化率。

**解决方案**：如需环比，需要显式查询上一个时间周期的数据。

### 4. 数据字段完整列表

每个模块返回的数据字段：

| 模块 | 字段名 | 类型 | 说明 |
|------|--------|------|------|
| 诊断得分 | `key` | string | 指标标识 |
| 诊断得分 | `name` | string | 指标名称 |
| 诊断得分 | `type` | string | 类型：score/subScore |
| 诊断得分 | `value` | number | 分数值 |
| 诊断得分 | `percent` | number | 超越同类商家百分比 |
| 其他模块 | `metricKey` | string | 指标标识 |
| 其他模块 | `title` | string | 指标名称 |
| 其他模块 | `value` | number | 数值 |
| 其他模块 | `percent` | number | 超越同类商家百分比 |

## 十三、环比计算说明

**当前API响应只提供百分位信息，不提供上一周期数值**。要支持环比计算，需要：

1. 显式查询上一个对比周期
2. 计算两个周期相同指标的环比变化率

**公式**：
```
环比增长率 = (本期值 - 上期值) / 上期值 × 100%
```

**示例**：
```
本期支付金额: 50.00万
上期支付金额: 40.00万

环比增长率 = (50.00 - 40.00) / 40.00 × 100% = 25.00%
```

**实现方式**：
```bash
# 查询本期
python3 query_report.py --begin-date 20260301 --end-date 20260307 --channel 1 --format json > current.json

# 查询上期
python3 query_report.py --begin-date 20260222 --end-date 20260228 --channel 1 --format json > previous.json

# 手动计算环比或编写脚本处理
```

## 十六、常见问题

### Q1: 为什么不同模块的分数差异很大？

**原因**：不同模块反映不同维度的能力，可能存在"偏科"现象。

**建议**：重点关注短板模块，制定针对性优化策略。

### Q2: 如何判断某个指标是好是坏？

**参考标准**：
- 超越90%以上：优秀
- 超越70%-90%：良好
- 超越50%-70%：中等
- 超越50%以下：需提升

### Q3: 支付金额和核销金额为什么不同？

**原因**：
- 支付金额：用户下单并支付的金额
- 核销金额：用户实际消费/使用的金额
- 差额：未消费/退款的部分

**关注点**：核销率反映用户的实际履约情况。

### Q4: 如何提升视频诊断得分？

**建议方向**：
1. 提升视频内容质量
2. 增加视频发布频率
3. 优化视频挂载商品
4. 提升视频转化效率

### Q5: 超过31天会怎样？

**限制**：查询时间范围超过31天时，脚本会拒绝查询并提示错误。

**错误信息**：
```
❌ 错误：查询时间范围不能超过31天（当前范围 50 天）
   开始日期：20260201
   结束日期：20260322
   请缩小查询范围后重试。
```

**解决方案**：缩小查询范围，确保时间跨度不超过31天。

### Q6: 出现 error_code=807000 怎么办？

**说明**：这是正常现象，表示"有数据但带业务告警"的返回形态。数据仍然有效，可以正常使用。

### Q7: ACCESS_TOKEN 过期怎么办？

**错误信息**：`401 unauthorized`

**解决方案**：
1. 重新获取 ACCESS_TOKEN
2. 更新环境变量：`export ACCESS_TOKEN='new-token'`

## 十八、注意事项

1. **数据延迟**：部分指标可能存在1-2天的延迟
2. **数据口径**：不同时间查询的数据可能略有差异
3. **渠道归因**：跨渠道的数据可能存在归因差异
4. **异常数据**：如发现明显异常，建议重新查询确认
5. **日期范围限制**：最多只能查询31天（1个月）的数据
6. **时间范围**：endDate 不能包含今天，必须是昨天或之前的日期
7. **认证有效**：确保 `ACCESS_TOKEN` 有效且未过期
8. **依赖安装**：确保 `requests` 库已安装：`pip install requests`
9. **日期格式**：日期格式必须为 `YYYYMMDD`
10. **渠道类型**：渠道类型必须为 `1` 或 `2`
11. **数据完整性**：返回数据按照示例进行展示，不要进行总结和丢失
