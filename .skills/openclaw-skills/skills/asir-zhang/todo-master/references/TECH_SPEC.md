# OpenClaw Todo Skill Technical Design (V1)

## 1. Scope

This document defines the implementation-ready technical design for a local-file Todo skill.
It covers:

1. Command interface (CLI contract for `scripts/todo.py`)
2. Data model and file schemas
3. Monthly partition and cross-month query behavior
4. Validation, status transitions, and error handling
5. Acceptance criteria and test checklist

Out of scope for V1:

1. Cloud sync
2. Multi-device conflict resolution
3. GUI
4. Natural-language date parser (free text parser)

## 2. Delivery Package

Required package layout:

```text
openclaw-todo-skill/
  SKILL.md
  config.json
  scripts/
    todo.py
  data/
    index.json                  # auto-generated
    todos-YYYY-MM.json          # auto-generated monthly partitions
  examples/                     # optional
  tests/                        # optional
```

Notes:

1. `SKILL.md` is the skill entry document.
2. `config.json` stores initialization state and storage path selection.
3. Runtime data files under `data/` are generated by script if missing.
4. Script must run on Python 3.8+ and standard library only.

## 2.1 Initialization Gate

1. First-time usage requires explicit `init` command.
2. During initialization, user must choose:
   1. default path (`<skill_root>/data`)
   2. custom path (`--data-dir`) and it must be absolute
3. If not initialized, all commands except `init` must fail with `ERR_NOT_INITIALIZED`.

## 3. Data Model

## 3.1 Todo Object

```json
{
  "id": "t_19c4f994c3a_131a2a",
  "title": "Prepare Q2 proposal",
  "type": "long",
  "status": "open",
  "created_at": "2026-02-12T18:00:00+08:00",
  "updated_at": "2026-02-12T18:00:00+08:00",
  "plan_date": "2026-02-14",
  "due_date": "2026-03-20",
  "tags": ["work", "planning"],
  "note": "collect metrics first",
  "archive_month": "2026-02"
}
```

Field constraints:

1. `id`: required, unique, immutable
2. `title`: required, non-empty, trimmed
3. `type`: required, enum `long|short`
4. `status`: required, enum `open|done|canceled|deleted`
5. `created_at`: required, ISO 8601 with timezone, default Asia/Shanghai (`+08:00`)
6. `updated_at`: required, ISO 8601 with timezone, default Asia/Shanghai (`+08:00`)
7. `plan_date`: optional date (`YYYY-MM-DD`)
8. `due_date`: optional date (`YYYY-MM-DD`)
9. `tags`: optional string array, deduplicated
10. `note`: optional string
11. `archive_month`: required `YYYY-MM`, derived by write rule

Type-specific rule:

1. `type=short`: `plan_date` required
2. `type=long`: `plan_date` optional, `due_date` optional

## 3.2 Monthly File Schema

Path pattern: `data/todos-YYYY-MM.json`

```json
{
  "version": 1,
  "month": "2026-02",
  "todos": []
}
```

Constraints:

1. `month` must match filename month
2. `todos[*].archive_month` must equal `month`
3. `id` uniqueness required globally (validated through index)

## 3.3 Index File Schema

Path: `data/index.json`

```json
{
  "version": 1,
  "months": ["2026-01", "2026-02"],
  "id_map": {
    "t_abc123": "2026-02"
  },
  "stats": {
    "open": 10,
    "done": 2,
    "canceled": 1,
    "updated_at": "2026-02-12T10:00:00Z"
  }
}
```

Use:

1. Fast location by `id`
2. Discover available month partitions
3. Cache summary counts for quick views

## 4. Partitioning and Archive Rules

## 4.1 Archive Month Assignment

On create:

1. If `type=long` and `due_date` exists, use `due_date[:7]`
2. Else if `plan_date` exists, use `plan_date[:7]`
3. Else use current month of `created_at`

On update:

1. If `due_date` or `plan_date` changed and archive month changed, move record to target month file
2. Keep `id` stable and update `id_map`

On done/canceled/deleted:

1. Keep in current partition (no forced re-bucket in V1)

## 4.2 Cross-Month Query

Query engine determines candidate months first:

1. If filter has date range (`from/to`) and `by=plan|due|created`, calculate month span
2. Load only involved `todos-YYYY-MM.json`
3. Merge results, then sort in-memory

Sort default:

1. `status=open` first
2. then `due_date` ascending (null last)
3. then `created_at` descending

## 5. Command Contract

Command executable:

```bash
python3 scripts/todo.py <subcommand> [options]
```

## 5.1 Create

```bash
python3 scripts/todo.py add --type long --title "Prepare Q2 proposal" --due 2026-03-20 --tag work --tag planning
python3 scripts/todo.py add --type short --title "Send reimbursement docs" --plan 2026-02-14
```

Rules:

1. `--type` and `--title` required
2. `--plan` required when `--type short`
3. Date format strict `YYYY-MM-DD`

## 5.0 Initialize

```bash
python3 scripts/todo.py init --default
python3 scripts/todo.py init --data-dir /absolute/path
```

Rules:

1. `--default` and `--data-dir` are mutually exclusive.
2. `--data-dir` must be absolute path.
3. Initialization writes `config.json` and creates storage/index files.

## 5.2 Read

```bash
python3 scripts/todo.py list
python3 scripts/todo.py list --status all --by due --from 2026-02-01 --to 2026-02-28
python3 scripts/todo.py list --status open --due-state overdue
python3 scripts/todo.py list --status open --due-state not-overdue
python3 scripts/todo.py show --id t_xxx
python3 scripts/todo.py overdue
```

Defaults:

1. `list` defaults to `--status open`
2. If `--by` omitted with range, default `--by created`
3. Output human-readable by default, JSON via `--json`
4. `--due-state` supports `overdue|not-overdue|all`, default `all`
5. Overdue rule uses China local date: `due_date < today(Asia/Shanghai)` and `status=open`

## 5.3 Manage

```bash
python3 scripts/todo.py done --id t_xxx
python3 scripts/todo.py reopen --id t_xxx
python3 scripts/todo.py cancel --id t_xxx
python3 scripts/todo.py rm --id t_xxx
python3 scripts/todo.py update --id t_xxx --title "New title" --due 2026-03-25
```

Behavior:

1. `done/reopen/cancel` mutate `status` and `updated_at`
2. `rm` is soft delete in V1 (`status=deleted`)
3. `update` supports partial patch

## 6. Validation Rules

Input validation:

1. Empty title rejected
2. Invalid enum rejected
3. Invalid date format rejected
4. `from > to` rejected
5. `type=short` without `plan_date` rejected
6. Duplicate tags normalized

Data integrity validation:

1. Missing monthly file auto-created
2. Corrupt JSON causes explicit error and non-zero exit
3. Atomic write required (`.tmp` + replace)

## 7. Status Flow

Allowed transitions:

1. `open -> done`
2. `open -> canceled`
3. `done -> open`
4. `canceled -> open`
5. `open -> deleted`
6. `done -> deleted`
7. `canceled -> deleted`
8. `deleted -> open`

Disallowed:

1. `done -> canceled` directly
2. `canceled -> done` directly
3. `deleted -> done` directly
4. `deleted -> canceled` directly

Disallowed transitions return validation error.

## 8. Error Handling

Exit codes:

1. `0`: success
2. `2`: invalid args / validation error
3. `3`: todo not found
4. `4`: storage read/write failure
5. `5`: data corruption detected

Error output contract:

1. Always print one-line machine-friendly code prefix, example `ERR_VALIDATION: ...`
2. Always include storage path context when storage-related
3. Uninitialized gate returns `ERR_NOT_INITIALIZED` with guided init commands

## 9. SKILL.md Requirements

`SKILL.md` must include:

1. Capability summary: write/read/manage
2. Field semantics: long vs short
3. Time-range query examples with `--by`
4. Monthly archive explanation
5. Default rules and caveats
6. Troubleshooting section for common errors

## 10. Acceptance Checklist (V1)

Functional:

1. Create both long and short todo types
2. Query by time range with `by=created|plan|due`
3. Query supports both `--due-state overdue` and `--due-state not-overdue`
4. Persist data into monthly files
5. Read across month partitions correctly
6. Perform status transitions under allowed rules
7. Return correct non-zero code on invalid operation

Reliability:

1. Script initializes empty storage automatically
2. Atomic writes prevent partial file corruption
3. Index remains consistent after add/update/delete

Cross-platform:

1. Works on macOS, Linux, Windows with Python 3.8+
2. No external dependency required

## 11. Implementation Sequence

1. Build storage layer (`index.json`, month file IO, atomic writer)
2. Implement `add/show/list`
3. Implement `done/reopen/cancel/rm/update`
4. Implement range query month planner
5. Add validation and exit code mapping
6. Finalize `SKILL.md` and usage examples

## 12. Deferred for V2

1. Weekly/monthly review reports
2. Auto-rollover for unfinished short todos
3. Import/export adapters (CSV, markdown)
4. File locking for multi-process writes

## 13. Locked Product Decisions

1. Skill entry filename uses uppercase `SKILL.md`
2. For `type=long`, `archive_month` prioritizes `due_date`
3. `rm` uses soft delete (`status=deleted`)
4. Default timezone is China time (`Asia/Shanghai`)
5. Query supports both overdue and not-overdue filtering
