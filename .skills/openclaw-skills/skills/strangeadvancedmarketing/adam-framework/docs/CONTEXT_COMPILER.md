# The Context Compiler — Bring Your Own Hippocampus

> How the Adam Framework solves the context window problem at the architecture level.

---

## The Problem With Context Windows

Every AI model has a context window — a hard limit on how much text it can hold in
active memory at once. When that window fills, older content gets truncated. The AI
forgets what it read at the start of the session.

This creates a structural problem for persistent AI: the more history you load at
session start, the faster you burn the window. Load too much, and the AI forgets it
before the session gets interesting. Load too little, and the AI starts without enough
context to be useful.

Most solutions pick one horn of this dilemma. The Adam Framework doesn't.

---

## The Hippocampus / Cortex Split

In human memory, the hippocampus handles short-term encoding and retrieval — it's the
active working memory. The cortex stores long-term consolidated knowledge — slow to
update, but deep and stable.

The Adam Framework maps this split directly onto its architecture:

```
CORTEX  = The Vault (CORE_MEMORY.md, SOUL.md, daily logs)
          Long-term. Persistent. Updated nightly by the sleep cycle.
          Human-readable. Auditable. Stable.

HIPPOCAMPUS = BOOT_CONTEXT.md
              Short-term. Compiled fresh every boot.
              Contains only what's needed for the current session.
              Discarded and rebuilt on every restart.
```

The key insight: **you don't load the cortex directly into the context window.
You compile a hippocampus from it — a targeted, session-relevant snapshot —
and load that instead.**

---

## How BOOT_CONTEXT.md Is Compiled

SENTINEL runs this at every gateway start, before the gateway launches.

**Windows (PowerShell):**
```powershell
$coreMemory    = Get-Content "$VAULT_PATH\CORE_MEMORY.md" -Raw
$activeContext = Get-Content "$VAULT_PATH\workspace\active-context.md" -Raw
$bootPayload   = "# BOOT CONTEXT — Compiled by SENTINEL`n> Auto-generated. Do not edit manually.`n`n## Core Memory`n`n" + $coreMemory
$bootPayload  += "`n`n## Active Context`n`n" + $activeContext
Set-Content -Path "$VAULT_PATH\workspace\BOOT_CONTEXT.md" -Value $bootPayload
```

**macOS/Linux (bash — same logic, same output):**
```bash
{ printf '# BOOT CONTEXT — Compiled by SENTINEL\n> Auto-generated. Do not edit manually.\n\n## Core Memory\n\n'
  cat "$VAULT_PATH/CORE_MEMORY.md"
  printf '\n\n## Active Context\n\n'
  cat "$VAULT_PATH/workspace/active-context.md"
} > "$VAULT_PATH/workspace/BOOT_CONTEXT.md"
```

OpenClaw then injects BOOT_CONTEXT.md as part of its memory search context. The AI
reads it as the first act of every session.

**What this means in practice:**
- CORE_MEMORY.md can be as long and detailed as needed — it lives in the cortex
- BOOT_CONTEXT.md stays lean — it's compiled, not accumulated
- Every session starts with a fresh, accurate hippocampus
- The context window is preserved for actual work, not history loading

## The Determinism Principle

BOOT_CONTEXT.md is compiled from files, not generated by the AI. This is intentional.

If the AI generated its own boot context, it would introduce variance — different
summaries, different emphasis, potential hallucination. By having SENTINEL compile
it deterministically from source files, every session starts from exactly the same
known state. The AI's understanding of "who it is and what's happening" is not an
inference — it's a read.

This is why the date injection is part of the same boot sequence. The AI doesn't
infer the date from training data. SENTINEL writes `TODAY.md` with the real date
before every launch. The AI reads it. No hallucination possible.

**Deterministic boot = predictable session start = consistent behavior.**

## The Five-Layer Handoff

BOOT_CONTEXT.md is Layer 1 — the foundation. But it hands off to the other layers
during the session:

```
SESSION START
├── Sleep cycle fires (if >6 hours since last run) — OFFLINE MODE
│   └── reconcile_memory.py runs before gateway launches
│       Merges daily logs → CORE_MEMORY.md (via Gemini)
│       Increments neural graph (via mcporter → neural-memory)
│       Gateway is still offline: Markdown + neural layers only
│       Result: cortex is up-to-date before hippocampus compiles
│
├── BOOT_CONTEXT.md compiled (Layer 1 — deterministic, compiled by SENTINEL)
│   └── AI knows: identity, active projects, recent state, today's date
│
├── Gateway starts → health check passes → vector reindex runs (ONLINE)
│   └── openclaw memory index syncs vector store with offline reconcile writes
│       New memory from sleep cycle is now searchable via Layer 2
│
├── nmem_context fires (Layer 3 — neural graph spreading activation)
│   └── AI knows: associative context, related history, relevant connections
│
│   [During session]
│
├── Hybrid search on demand (Layer 2 — session + vault search)
│   └── AI can retrieve: specific past decisions, prior conversations, exact facts
│
└── Context approaches limit → memoryFlush triggers (Layer 4)
    └── AI writes: lasting notes back to Vault before truncation
        └── Cortex updated. Next boot compiles new hippocampus from updated cortex.
```

**Why the split between offline and online reconcile?**

The sleep cycle (reconcile) doesn't need the gateway — it reads session JSONL files
and Vault Markdown directly, calls Gemini externally, and writes back to Markdown and
the neural graph via mcporter. Running it offline means the cortex is already updated
*before* BOOT_CONTEXT.md is compiled, so the hippocampus contains the latest state
from the moment the session starts.

The vector reindex runs *after* the gateway is healthy because the OpenClaw vector
store requires the gateway process to be running. It's a sequential dependency, not a
design choice. The offline reconcile handles what it can; the online reindex finishes
the sync.

The loop is closed. Every session both reads from and writes back to the long-term
cortex. The hippocampus is rebuilt fresh from updated state on every boot.

## Why "Bring Your Own Hippocampus"

Standard AI deployments give you a model with a fixed context window and no
persistent architecture. You bring the prompt; the model brings the amnesia.

The Adam Framework inverts this: **you bring the hippocampus.**

You control what gets compiled into BOOT_CONTEXT.md. You control what goes into
CORE_MEMORY.md. You control the sleep cycle that updates it. The AI is just the
reader — powerful, capable, but stateless. The memory architecture is yours.

This means:
- **Model-agnostic** — swap the LLM, the hippocampus still loads correctly
- **Operator-auditable** — every fact the AI "knows" at boot is in a readable file
- **Incrementally improvable** — tune what gets compiled into BOOT_CONTEXT.md
  based on what you actually need in sessions
- **Survives system rebuilds** — wipe the gateway, rebuild from scratch, the cortex
  files survive and the hippocampus compiles fresh from them

The nuclear reset validation in February 2026 proved this directly: the system was
wiped completely, rebuilt from zero, and the AI came back online with full continuity
because the cortex files survived. The hippocampus compiled fresh from the same source.
Same AI. Different process. No lost memory.

## Tuning Your Boot Context

The default compilation is: CORE_MEMORY.md + active-context.md.

You can extend it by adding more files to the SENTINEL compilation block:

**Windows:**
```powershell
$soulContext  = Get-Content "$VAULT_PATH\SOUL.md" -Raw
$recentLog    = Get-Content "$VAULT_PATH\workspace\memory\$(Get-Date -Format 'yyyy-MM-dd').md" -Raw -ErrorAction SilentlyContinue
$bootPayload  = "# BOOT CONTEXT`n`n## Identity`n`n" + $soulContext + "`n`n## Core Memory`n`n" + $coreMemory + "`n`n## Active Context`n`n" + $activeContext
if ($recentLog) { $bootPayload += "`n`n## Today's Log`n`n" + $recentLog }
Set-Content -Path "$VAULT_PATH\workspace\BOOT_CONTEXT.md" -Value $bootPayload
```

**macOS/Linux:**
```bash
{ printf '# BOOT CONTEXT\n\n## Identity\n\n'; cat "$VAULT_PATH/SOUL.md"
  printf '\n\n## Core Memory\n\n'; cat "$VAULT_PATH/CORE_MEMORY.md"
  printf '\n\n## Active Context\n\n'; cat "$VAULT_PATH/workspace/active-context.md"
  TODAY=$(date +%Y-%m-%d)
  [[ -f "$VAULT_PATH/workspace/memory/$TODAY.md" ]] && { printf '\n\n## Today Log\n\n'; cat "$VAULT_PATH/workspace/memory/$TODAY.md"; }
} > "$VAULT_PATH/workspace/BOOT_CONTEXT.md"
```

The rule: **if the AI needs it to be functional from word one, compile it in.
If it can be retrieved on demand, leave it out and let Layer 2/3 handle it.**

Keep BOOT_CONTEXT.md lean. It's a hippocampus, not a library.
