## 概述

OnKos 是一个基于智能体的长篇小说创作伙伴系统。通过6级分层摘要、SQLite事实相关性引擎、本地语义检索和多角色协作，实现2000章（600万字）规模的连贯性保证。

### 核心理念

- **智能体主动创作**: 智能体根据大纲和上下文主动推进故事
- **连贯性优先**: 千万字创作最核心的挑战是一致性，所有技术手段服务于连贯性
- **6级分层记忆**: 全书→阶段→弧线→卷→章节→场景，确保任意时刻的上下文窗口有效
- **事实驱动**: 所有角色、世界、物品状态由事实表管理，创作时按相关性自动筛选
- **统一存储**: 知识图谱、伏笔、弧线均存储在 SQLite，解决百万字规模下的性能瓶颈

## 创作流程

### 1. 项目初始化

```bash
python scripts/project_initializer.py --project-path ./my-novel --action init --title "书名" --genre "玄幻"
```

创建标准项目结构:
- `data/novel_memory.db` - SQLite统一数据库（记忆库、知识图谱、伏笔、弧线）
- `data/characters/` - 角色心理模型
- `outline/` - 大纲目录
- `drafts/` - 草稿目录

### 2. 世界观搭建

以 Architect 角色进行，将设定录入知识图谱（SQLite）:
```bash
python scripts/knowledge_graph.py --db-path ./my-novel/data/novel_memory.db --action add-node --name "青云宗" --type faction
```

### 3. 角色建模

以 CharacterMind 角色进行:
```bash
python scripts/character_simulator.py --characters-dir ./my-novel/data/characters --action create --name "林风" --role protagonist --big-five '{"openness":0.8}'
```

### 4. 弧线与阶段规划

2000章必须拆分为阶段和弧线:
```bash
python scripts/arc_manager.py --db-path ./my-novel/data/novel_memory.db --action create-phase --arc-id "rise" --arc-title "崛起篇" --start-chapter 1 --end-chapter 200
python scripts/arc_manager.py --db-path ./my-novel/data/novel_memory.db --action create-arc --arc-id "trial" --arc-title "宗门试炼" --start-chapter 1 --end-chapter 40 --phase-id "rise"
```

### 5. 章节创作

1. 获取6级上下文
2. 阅读章节大纲
3. 撰写正文
4. 存储章节（支持 --content-file）
5. 提取实体和事实（含重要性层级）
6. 检测事实变更（对比章节内容与现有事实，识别新/更新/冲突）
7. 检查伏笔状态

### 6. 事实变更检测

在存储章节并提取实体后，调用 `detect-fact-changes` 自动对比章节内容与现有事实:

```bash
python scripts/command_executor.py --project-path <项目> --action detect-fact-changes --content "章节内容..." --chapter 15 --genre fantasy
```

输出内容:
- **提取的实体**: 章节中出现的角色/地点/物品名
- **当前事实（按实体分组）**: 每个实体的已有属性和值
- **分析提示**: 引导智能体识别三类变更

智能体分析流程:
1. **新事实**: 章节中出现了新实体或新属性，事实库中无记录 -> 调用 `set-fact` 录入
2. **更新事实**: 章节中某属性值发生了变化（如境界提升、位置转移） -> 调用 `set-fact` 更新（自动supersede旧值）
3. **冲突事实**: 章节内容与已有事实矛盾 -> 智能体判断是情节需要还是写作错误，决定更新事实或修改内容

importance选择规则:
- 角色核心属性（修为/身份/阵营） -> `permanent`
- 阶段性状态（临时位置/短期关系） -> `arc-scoped`
- 一次性事件（战斗结果/对话内容） -> `chapter-scoped`

### 7. 质量审计

以 Reviewer 角色执行质量审计（health_score）和连续性检查（双层OOC）

## 2000章架构

### 6级分层摘要体系

| 层级 | 粒度 | 预算字符 | 用途 |
|------|------|----------|------|
| L0: 全书 | 整体 | 600 | 宏观把控 |
| L0: 当前状态 | 滚动更新 | 800 | 最新进展 |
| L1: 阶段(Phase) | 100-300章 | 400 | 阶段内连贯 |
| L2: 弧线(Arc) | 30-60章 | 600 | 弧线内连贯 |
| L3: 卷(Volume) | 20章 | 800 | 卷内衔接 |
| L4: 章节(Chapter) | 单章 | 1200+800 | 章间衔接 |
| L5: 场景(Scene) | 段落 | 2500 | 细节检索 |

总预算约12000字符，上下文覆盖率可达80%。

### 事实相关性引擎

事实表新增三个维度:
- **importance**: `permanent` / `arc-scoped` / `chapter-scoped`
- **valid_from**: 事实生效的起始章节
- **valid_until**: 事实失效的章节

检索优先级:
1. 所有 permanent 事实（角色核心属性、世界观基石）
2. 大纲中提及实体的活跃事实
3. 当前弧线范围内的 arc-scoped 事实
4. 近10章的 chapter-scoped 事实

超过80条事实时按优先级截断，避免上下文溢出。

### SQLite统一存储

知识图谱（kg_nodes/kg_edges）和伏笔（hooks）统一存储在SQLite:
- 支持百万字规模下的高性能查询
- 支持关系查询（路径搜索、邻居发现）
- 支持事务和并发安全

## 实体提取与语义检索

### 实体提取

实体提取器基于多策略融合，每个提取结果附带置信度评分（high/medium/low）:
- **jieba posseg 词性标注**: 主力方法，识别 nr/ns 等词性
- **称谓模式匹配**: 基于分词边界验证（"X师兄"、"X道长"）
- **复姓优先匹配**: 长名优先避免截断（"欧阳明月"完整识别）
- **对话归属识别**: 从"X说道"等模式提取说话者
- **跨类型去重**: 角色名与地名/物品名互斥
- **题材自适应**: 根据题材自动切换称谓后缀、地点关键词、物品关键词和事件模式

#### 题材自适应

实体提取器支持4种题材配置，通过 `--genre` 参数指定:

| 题材 | 称谓示例 | 物品示例 | 事件示例 |
|------|----------|----------|----------|
| fantasy(玄幻) | 师兄/掌门/真人 | 功法/丹药/法器 | 突破/拜师 |
| urban(都市) | 总裁/局长/警官 | 文件/装备 | 升职/收购 |
| wuxia(武侠) | 掌门/堂主/镖头 | 武功/暗器/兵器 | 修炼/结拜 |
| scifi(科幻) | 舰长/博士/司令 | 装备/能源 | 进化/入侵 |

不指定时从项目 `data/project_config.json` 的 `genre` 字段自动读取，默认为 fantasy。

### 语义检索

记忆引擎支持两层检索，自动降级:
- **FTS5 关键词搜索**: jieba预分词 + 空格连接，解决中文分词问题
- **ONNX 语义检索**: text2vec-base-chinese（768维，INT8量化版），章节范围预过滤 + numpy向量化

章节范围预过滤: 语义搜索限定在当前弧线章节范围内，减少无关场景干扰。numpy向量化: 批量计算余弦相似度，避免逐条循环。

## 长篇连贯性保障

### 事实表引擎

事实表自动管理所有可变状态:
- 角色修为变化（importance=permanent）
- 物品归属转移（importance=arc-scoped）
- 地点状态变更（importance=chapter-scoped）
- 关系演变（importance=arc-scoped）

每个事实带有版本链、有效期和重要性标签，支持历史查询和矛盾检测。

### 伏笔生命周期

1. **种植**: 在大纲中标注，写作时埋入（可指定strength强度0-1）
2. **潜伏**: 定期检查未闭合伏笔
3. **暗示提及**: 更新last_hint_chapter，维持读者记忆
4. **部分回收**: 递增partial_count，逐步揭示伏笔
5. **预警**: 超过预期回收章节时预警（urgency紧迫度自动衰减）
6. **回收**: 确认回收方式与种植呼应
7. **遗忘预警**: 超过100章未提及的伏笔标记为"可能遗忘"

### OOC防护（双层检测）

- **第一层: 字符串匹配**: 禁止行为红线检测（绝对不可违反）
- **第二层: 智能体裁判决**: Big Five人格冲突分析（需情节压力解释的可疑行为）

### 位置连续性

检测同一角色在短距离内出现在矛盾地点，自动关联知识图谱中的角色信息。

## 常见问题与对策

### 问题1: 写着写着角色性格变了
**对策**: 每次创作前加载角色心理模型，创作后进行双层OOC检测

### 问题2: 前面埋的伏笔忘了回收
**对策**: 使用伏笔追踪器（SQLite），每章检查未闭合伏笔和遗忘预警

### 问题3: 2000章后事实太多上下文放不下
**对策**: 事实相关性引擎按重要性+时效性过滤，permanent事实永远保留

### 问题4: 支线太多收不回来
**对策**: 情节分支器监控活跃支线数量，超过5条预警

### 问题5: 上下文窗口不够覆盖
**对策**: 6级分层摘要压缩，只加载当前需要的上下文，预算12000字符

### 问题6: 修改已写章节后怕连锁错误
**对策**: 使用修订影响分析，对比修订前后内容，自动识别受影响的后续章节、事实和伏笔

#### 修订影响分析

当需要修改已写章节时，务必先执行影响分析:
```bash
python scripts/continuity_checker.py --project-path <项目> --action analyze-revision --chapter 50 --old-content-file 旧版.txt --new-content-file 新版.txt
```

输出内容:
- **风险等级**: high(实体删除)/medium(存在受影响内容)/low(无明显连锁影响)
- **实体变化**: 新增/删除的角色、地点、物品
- **受影响章节**: 后续引用了变化实体的章节列表
- **受影响事实**: 与变化实体相关的事实条目
- **受影响伏笔**: 描述或关联角色涉及变化实体的伏笔
- **修订建议**: 具体的检查和修复指引

## 核心概念详解

### 事实重要性层级

录入事实时必须选择正确的 importance:

| 层级 | 含义 | 典型场景 | 上下文优先级 |
|------|------|---------|------------|
| `permanent` | 永久设定，全书有效 | 角色核心属性、世界观基石 | 始终包含 |
| `arc-scoped` | 弧线内有效 | 阶段性关系、临时状态 | 弧线内包含，需配合 valid-from/valid-until |
| `chapter-scoped` | 近期有效 | 临时位置、短时状态 | 近10章内包含 |

同一实体的同一属性再次录入时，旧事实自动被替代(superseded)。替代事实的 valid-from 在未来时，旧事实仍然可见。

### 摘要层级（6级）

| 层级 | 粒度 | 更新频率 |
|------|------|---------|
| `book` | 全书 | 结构性变更时 |
| `phase` | 阶段（100-300章） | 阶段完成时 |
| `arc` | 弧线（30-60章） | 弧线完成时 |
| `volume` | 卷（10-30章） | 卷完成时 |
| `chapter` | 章节 | 每章完成时 |
| `scene` | 场景 | 自动分割 |

### 角色定位

`protagonist`(主角) / `antagonist`(反派) / `mentor`(导师) / `sidekick`(伙伴) / `npc`

创建角色时需定义: Big Five人格、核心特质、禁忌行为、说话风格、典型行为模式。详见 [references/agent_roles.md](agent_roles.md)。

### 知识图谱节点类型

`person`(人物) / `faction`(势力) / `location`(地点) / `item`(物品) / `event`(事件)
