# Blueprint Cross-Skill Handoff Design

## Status

Proposed

## Goal

Make `kai-business-blueprint` the upstream semantic source for downstream presentation skills without forcing `kai-report-creator` and `kai-slide-creator` to understand the full blueprint schema.

## Decision Summary

Use `solution.blueprint.json` as the only durable machine truth.

Do not generate human-readable handoff documents. The only human-facing artifact remains the existing blueprint viewer and diagram exports.

Introduce a thin shared adapter contract that projects blueprint data into a normalized, presentation-ready summary object. Downstream skills consume that projection and then generate their own native IR:

- `kai-report-creator` -> `.report.md`
- `kai-slide-creator` -> `PLANNING.md` or equivalent internal deck plan

The adapter is not a second source of truth. It is a derived compatibility surface generated by `kai-business-blueprint`, never hand-edited by downstream skills.

## Problem

Today `kai-business-blueprint` already produces a strong semantic artifact:

- business capabilities
- actors
- flow steps
- systems
- relations
- viewer / export outputs

But `kai-report-creator` and `kai-slide-creator` are optimized for narrative and layout generation, not for parsing a domain graph directly.

If both downstream skills parse raw blueprint JSON on their own:

- blueprint schema details leak into multiple downstream skills
- schema evolution creates duplicate maintenance
- narrative mapping logic diverges between report and slide
- downstream prompts become bloated with low-level graph handling concerns

## Design Principles

1. `solution.blueprint.json` is the only durable semantic source.
2. Viewer and diagram exports are presentation artifacts, not handoff artifacts.
3. Downstream skills should consume a stable narrative projection, not the raw full graph.
4. Report and slide keep their own native planning/generation workflows.
5. No downstream skill writes back into `solution.blueprint.json`.
6. Schema evolution should be absorbed in one place.
7. Adapter execution ownership must be unambiguous.

## Proposed Architecture

### Artifact Flow

```text
raw notes / presales materials / workshop input
  -> business blueprint extraction
  -> solution.blueprint.json
       -> viewer / svg / drawio / excalidraw / mermaid
       -> blueprint adapter projection
            -> report IR (.report.md)
            -> slide IR (PLANNING.md or internal plan)
```

### Responsibility Split

`kai-business-blueprint`
- owns semantic extraction
- owns blueprint schema
- owns viewer / diagram exports
- owns canonical adapter generation
- owns adapter compatibility guarantees

`kai-report-creator`
- owns report narrative structure
- owns section-level component selection
- owns `.report.md` generation and HTML rendering
- consumes canonical projection output, not raw blueprint parsing logic

`kai-slide-creator`
- owns deck narrative arc
- owns page-level storytelling and visual rhythm
- owns deck plan generation and HTML slides rendering
- consumes canonical projection output, not raw blueprint parsing logic

## Why This Contract Is Thin Enough

This projection is intentionally not a second blueprint schema.

It does not duplicate:

- low-level relation graph details
- every flow step property
- every editor field
- every artifact export

It only preserves what downstream narrative generators need repeatedly:

- what problem is being solved
- which capabilities matter
- which actors matter
- which flows define the story
- which systems anchor the architecture story
- which framing angles are likely useful

## Downstream Mapping

### Report Import Path

Add a prompt-native entry path:

```text
"Use this business blueprint JSON or projection JSON to generate a report IR."
```

Internal flow:

```text
blueprint.json
  -> canonical BlueprintNarrativeProjection
  -> report-specific section planner
  -> .report.md
  -> HTML report
```

Recommended default report section mapping:

1. Executive summary
2. Business goals and scope
3. Capability map and ownership
4. End-to-end operational flows
5. Application / system landscape
6. Key assumptions, risks, and constraints
7. Recommended rollout or next actions

Rules:

- prefer `callout`, `timeline`, `diagram`, and selective `table`
- only emit KPI or chart components if the blueprint source includes real quantitative data
- use `capabilityGroups`, `coreFlows`, and `systems` as the main visual anchors

### Slide Import Path

Add a prompt-native entry path:

```text
"Use this business blueprint JSON or projection JSON to generate a deck plan."
```

Internal flow:

```text
blueprint.json
  -> canonical BlueprintNarrativeProjection
  -> deck storyline planner
  -> PLANNING.md or internal deck plan
  -> HTML slides
```

Recommended default deck arc:

1. Title and transformation thesis
2. Current-state pain or complexity
3. Business goals and scope
4. Capability map
5. Core business flow
6. System landscape
7. Design choices / target operating model
8. Rollout or next-step summary

Rules:

- default to 8-slide business architecture narrative
- expand to 10-12 slides only when blueprint scope is broad enough
- prefer one dominant idea per page
- use capability map and system landscape as visual anchors, not dense graph dumps

## Prompting Contract

### Skill-Level Entry Points

`kai-business-blueprint`
- remains the producer of `solution.blueprint.json`
- should expose the canonical projection surface used by downstream skills
- in stage 1, runs the adapter itself as a helper module or CLI command
- downstream skills consume projection output from this skill rather than reproducing projection logic in prompt space

`kai-report-creator`
- must recognize business blueprint JSON or projection JSON passed in the prompt as machine input
- must treat blueprint import as machine input, not ask for a separate human narrative brief unless required data is missing
- must not re-implement raw blueprint traversal logic independently of the canonical adapter

`kai-slide-creator`
- must recognize business blueprint JSON or projection JSON passed in the prompt as machine input
- must generate deck planning from blueprint-derived narrative, not from raw graph dumping
- must not re-implement raw blueprint traversal logic independently of the canonical adapter

### Missing Information Behavior

When blueprint lacks information needed for narrative generation:

- use `context.clarifyRequests` and `context.clarifications` first
- if still missing, emit placeholders or explicit assumptions in the downstream IR
- surface adapter diagnostics to the user or downstream logs
- do not invent quantitative metrics
- if diagnostics contain only warnings and no substantive narrative data, downstream skills generate a skeleton IR with explicit placeholder sections instead of fabricating detail

## Versioning and Evolution

### Stability Strategy

Treat the adapter contract as a stable compatibility surface.

Blueprint schema may evolve more freely behind it as long as the adapter still produces:

- stable top-level sections
- stable field semantics
- stable narrative intent

### Change Policy

Safe blueprint changes:

- adding new blueprint fields
- adding richer relation metadata
- adding editor-only metadata

Potentially breaking changes:

- renaming or restructuring `library.capabilities`, `library.flowSteps`, `library.systems`
- changing actor/system identity semantics
- changing how scope, assumptions, or clarify requests are stored

For potentially breaking changes, only the adapter layer should absorb the difference. Downstream skills should continue targeting the adapter contract.

## Alternatives Considered

### Option A: Downstream Skills Parse Raw Blueprint Directly

Rejected.

Pros:

- no extra abstraction layer
- lower short-term implementation cost

Cons:

- duplicates mapping logic
- higher coupling
- brittle under schema evolution

### Option B: Generate Human-Readable Handoff Markdown

Rejected.

Pros:

- easy to inspect manually

Cons:

- user explicitly does not need handoff docs for humans
- introduces redundant narrative artifacts
- encourages manual drift

### Option C: Generate Separate Consumer-Specific Machine Briefs

Partially rejected.

Examples:

- `solution.report-brief.json`
- `solution.slide-brief.json`

Reason:

- acceptable as a cache later
- not acceptable as durable truth
- should only be generated from blueprint, never edited directly

## Recommendation

Adopt the adapter-based design now and implement it in two stages:

### Stage 1

- document the adapter contract
- make `kai-business-blueprint` the canonical projection owner from day one
- implement the adapter execution surface inside `kai-business-blueprint`
- add prompt-native blueprint/projection import handling to report and slide skills
- keep downstream logic focused on their native IR generation only

### Stage 2

- harden the canonical adapter surface as a reusable helper module or CLI if the first implementation starts as prompt-level guidance
- optionally add generated cache artifacts for debugging or reproducibility

## Open Questions

1. Should `BlueprintNarrativeProjection` be persisted for debugging, or remain purely in-memory by default?
2. Should `kai-business-blueprint` expose the canonical adapter as a helper module, a CLI command, or both?
3. Should slide/report import default to different storylines by industry, or stay generic in v1?

## Recommendation for Review

Challenge this design on:

- whether the adapter is actually thin enough
- whether the contract hides too much blueprint richness
- whether stage 1 can stay lightweight without creating hidden duplication
- whether report and slide entry points are stable enough for future skills
