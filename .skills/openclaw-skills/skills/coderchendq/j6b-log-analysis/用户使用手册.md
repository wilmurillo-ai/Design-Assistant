## 1. 技能概述

### 1.1 什么是J6B泊车日志分析Skill？

J6B泊车日志分析Skill是一个专为J6B泊车项目开发的AI助手技能，帮助开发人员快速定位问题、分析日志、诊断系统性能。

### 1.2 适用人群

| 角色 | 适用场景 |
|------|----------|
| **开发工程师** | 代码调试、问题定位、性能优化 |
| **测试工程师** | 实车测试、故障分析、日志收集 |
| **系统工程师** | 系统维护、性能监控、问题排查 |

### 1.3 核心价值

| 价值点 | 说明 |
|--------|------|
| **快速定位** | 一句话找到模块日志路径，无需记忆 |
| **智能分析** | 识别常见错误模式，提供解决建议 |
| **性能诊断** | 一键监控系统资源，标准化报告输出 |
| **知识复用** | 积累故障案例，经验团队共享 |

### 1.4 支持的功能

| 功能编号 | 功能名称 | 优先级 |
|----------|----------|--------|
| F1 | 日志文件定位 | P0 |
| F2 | 基础日志读取 | P0 |
| F2+ | 常见错误识别 | P0 |
| F3 | 日志下载 | P0 |
| F4 | 进程状态查看 | P1 |
| F5 | 系统资源性能分析 | P0 |
| F6 | 泊车业务进程状态检测 | P0 |

### 3.6 系统资源性能分析（F5）

**功能说明**：分析板端的CPU、内存、线程状态，按标准模板输出报告

#### 方式一：一键监控（推荐新手）

**特点**：
- 🎨 彩色输出：高CPU红色、警告黄色
- 📊 TOP 15排名：CPU占用、内存占用
- 📝 自动日志：保存到 `/userdata/log/cpu_monitor/`
- 🔄 自动清理：7天后自动删除旧日志

**使用方法**：

```bash
# 前台运行（实时查看）
/userdata/monitor_cpu.sh

# 后台运行（适合长期监控）
nohup /userdata/monitor_cpu.sh > /dev/null 2>&1 &

# 查看监控日志
tail -f /userdata/log/cpu_monitor/cpu_monitor_*.log

# 停止后台进程
kill $(pidof monitor_cpu.sh)
```

#### 方式二：手动三步法（推荐熟练用户）

```
第一步：hogs -i -s 1 → 找出占用CPU的进程
第二步：top -p <PID> → 查看该进程详情
第三步：分析 → 判断是算法太重还是逻辑锁死
```

#### 诊断结论速查表

| PIDS | SYS | STATE | 可能原因 | 解决方法 |
|------|-----|-------|----------|----------|
| >50% | <30% | 多Run | 算法过重 | 优化算法、降低频率 |
| >50% | >70% | 多Run | 死循环/系统调用频繁 | 检查代码逻辑 |
| 正常 | 正常 | 多Mtx/Sem | 锁竞争 | 减小锁粒度、缩短持锁时间 |
| 正常 | 正常 | 多NSlp | 正常 | 无需处理 |
| 高 | 高 | 多Mtx/Sem | 可能死锁 | 检查锁逻辑 |

**功能说明**：检测13个泊车核心业务进程的运行状态和在线时长

**监控的业务进程**（共13个）：

| 序号 | 进程名 | 功能 |
|------|--------|------|
| 1 | sensorcenter | 传感器中心 |
| 2 | image_preprocess | 图像预处理 |
| 3 | perception_rd | 车位检测 |
| 4 | perception_od | 目标检测 |
| 5 | perception_fusion | 感知融合 |
| 6 | dr | 航迹推算 |
| 7 | loc | 定位建图 |
| 8 | pad | 泊车辅助决策 |
| 9 | gridmap | 栅格地图 |
| 10 | planning | 规划控制 |
| 11 | ui_control | UI控制 |
| 12 | adaptercom | 适配通信 |

**进程状态判定**：

| 状态 | 说明 | 判定 |
|------|------|------|

## 4. 系统资源分析报告

### 4.1 概述

当用户要求分析系统资源时，AI会自动按照标准模板输出7章报告，确保格式统一、信息完整。

### 4.2 报告结构

| 章节 | 标题 | 内容说明 |
|------|------|----------|
| 第1章 | 系统健康度 | 一眼看清系统整体状态：CPU、内存、进程数、线程数，含CPU统计(Min/Max/Avg/Idle) |
| 第2章 | CPU占用 TOP 15 | 按CPU使用率排序的前15个进程，含SYS占比和判定 |
| 第3章 | 内存占用 TOP 15 | 按内存使用量排序的前15个进程，含判定 |
| 第4章 | 线程状态异常 | 检测Mtx/Sem异常线程，提示锁竞争或死锁风险 |
| 第5章 | 泊车业务进程状态 | 13个核心业务进程的PID、运行状态、在线时长 |
| 第6章 | TOP完整输出 | top命令的完整原始输出，供深入分析 |
| 第7章 | 重点关注项 | 汇总异常项，给出可能原因和排查建议 |

### 4.3 判定阈值

| 指标 | 🟢 正常 | 🟡 注意 | 🔴 告警 |
|------|---------|---------|---------|
| 单进程 CPU | < 10% | 10% ~ 50% | > 50% |
| 系统总 CPU | < 50% | 50% ~ 80% | > 80% |
| 单进程内存 | < 200M | 200M ~ 500M | > 500M |
| 可用内存 | > 2000M | 500M ~ 2000M | < 500M |
| 线程 Mtx/Sem | 0 个 | 1 ~ 3 个 | > 3 个 |

### 4.4 健康度判定规则

| 等级 | 条件 |
|------|------|
| 🟢 良好 | 所有指标均在正常范围 |
| 🟡 注意 | 存在 🟡 指标，无 🔴 |
| 🔴 告警 | 存在任何 🔴 指标 |

## 5. 使用场景

### 场景1：泊车失败，需要查看日志

**操作步骤**：

1. **定位日志路径**
   ```
   你: 泊车失败了，我该看哪个日志？

   AI: 先看planning和psd模块日志：
   /app/apa/log/planning/
   /app/apa/log/psd/
   ```

2. **查看最新错误**
   ```bash
   grep -i "error\|fail" /app/apa/log/planning/*.log | tail -50
   ```

3. **实时跟踪**
   ```bash
   tail -f /app/apa/log/planning/*.log
   ```

4. **如需下载分析**
   ```bash
   scp root@192.168.1.10:/app/apa/log/planning/*.log ./
   ```

**操作步骤**：

1. **对AI说**："帮我分析一下系统资源状态"

2. **AI自动输出标准7章报告**，包含：
   - 系统健康度总览
   - CPU/内存 TOP 15
   - 线程异常检测
   - 泊车业务进程状态
   - 重点问题及排查建议

## 6. 常见问题

### Q1: 如何连接到板端？

**A**:
```bash
# 千兆网口（推荐）
ssh root@192.168.1.10

# 百兆网口
ssh root@192.168.2.10

# 测试连接
ping 192.168.1.10
```

### Q2: 日志被覆盖了怎么办？

**A**:
- 日志采用轮转循环写入，旧日志会被覆盖
- 重要日志请及时下载到本地
- 可以使用 `find /app/apa/log/ -name "*.log" -mmin -10` 查找最近修改的日志

### Q3: QNX命令和Linux有什么不同？

**A**:
| 功能 | QNX | Linux |
|------|-----|-------|
| 进程查看 | `pidin` | `ps` |
| 结束进程 | `slay` | `kill` |
| 系统日志 | `slog2info` | `journalctl` |
| 资源监控 | `hogs`, `top` | `top`, `htop` |

### Q4: 如何长期监控系统性能？

**A**:
```bash
# 后台运行监控脚本
nohup /userdata/monitor_cpu.sh > /dev/null 2>&1 &

# 查看监控日志
tail -f /userdata/log/cpu_monitor/cpu_monitor_*.log
```

### Q5: 找不到问题原因怎么办？

**A**:
1. 收集相关日志
2. 查看 [troubleshooting.md](references/troubleshooting.md) 故障案例库
3. 记录以下信息上报：
   - 故障现象
   - 复现步骤
   - 相关日志
   - 系统信息

### Q6: 报告中的判定标准可以调整吗？

**A**: 可以。判定阈值定义在 `references/report-template-system-resource.md` 中，修改后所有报告将按新标准输出。当前默认阈值：

| 指标 | 🟡 注意 | 🔴 告警 |
|------|---------|---------|
| 单进程 CPU | > 10% | > 50% |
| 系统总 CPU | > 50% | > 80% |
| 单进程内存 | > 200M | > 500M |
| 可用内存 | < 2000M | < 500M |
