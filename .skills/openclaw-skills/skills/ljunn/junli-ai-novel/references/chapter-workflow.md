# 单章创作流程细则

用于需要完整写前分析、场景拆分、写后检查和定向返修时。

默认服务平台向长篇网文连载，目标优先级是：追读牵引 > 卖点兑现 > 连载稳定 > 文本局部花活。

## 1. 写前分析

每次开写新章节前：

1. 读取 `task_log.md`
2. 读取 `docs/大纲.md`
3. 读取上一章正文
4. 读取相关角色与设定
5. 检查活跃伏笔和时间线
6. 将目标章节在 `docs/大纲.md` 中标记为“进行中”

无论是否群像，都先确认：

7. 本章主 POV 是谁
8. 当前 POV 在本章开头能看到、听到、知道、误判到什么
9. 本章有哪些关键信息需要暂时对读者隐藏，不能提前泄露

如果是群像模式，再额外确认：

10. 本章推进哪条主线，哪条线只做背景推进
11. 读者需要被提醒的上一条关键线索是什么

无论是否群像，再额外确认：

12. 本章冲突主要来自资源、规则还是历史
13. 冲突代价是什么，谁会因此受损
14. 本章涉及的世界规则是底层铁律还是表层规则
15. 本章准备让读者追问的核心问题是什么
16. 这个问题属于“发生了什么 / 会怎样 / 为什么”中的哪一类
17. 本章是在延迟、局部解答，还是生成新问题
18. 本章赌注是否足够清晰，结果是否真的未定
19. 旧问题这章是否需要回扣，避免被读者遗忘或衰减
20. 本章主角承受的压力或委屈，会在何处转化为反制、收益或关系回应
21. 本章深度内容准备藏进哪个选择、代价或后果里，而不是直接讲道理

写前方案至少包含：

- 本章主 POV
- 本章信息边界与保密信息
- 本章目标
- 承接上章的方式
- 本章核心冲突
- 本章核心爽点或情绪爆点
- 本章情绪补偿路径
- 结尾钩子方向
- 本章准备让哪些场景做到“一笔多用”，避免每段只承担单一功能

## 2. 场景拆分

正文前先拆成 3-5 个场景。每个场景至少写清：

- 场景类型：铺垫 / 冲突 / 高潮 / 转折 / 收尾
- 情绪走向：起点情绪 -> 终点情绪
- 叙事目的：本场景要推进什么
- 暗线：该场景中隐藏推进的伏笔、利益或误导
- 副功能：这一场除了推进情节，还顺手承载哪 1-2 个功能（如人设、关系、伏笔、世界、反差、资源信息）

场景内部可以再拆成 2-4 个微推进点，但它们只是后台写作骨架，不是正文里要显形的编号节点。

使用微推进点时，至少满足：

- 每个推进点都由上一个推进点自然引出，而不是横向跳项
- 相邻推进点之间至少有一种桥接：动作余波、情绪惯性、视线或注意力转移、空间移动、信息触发、因果反应
- 读者读正文时不该明显感觉到“这里切到下一个 Beat 了”

推荐写法：

```text
场景一：[名称]
- 地点：
- 人物：
- 核心事件：
- 场景类型：
- 情绪走向：
- 目的：
- 暗线：
- 微推进链：
  - 起手动作：
  - 局部受阻 / 新信息出现：
  - 反应 / 选择 / 失衡：
  - 推向下一场景的余波：
```

不要把“微推进链”当成正文结构照抄。它只是提醒你控制内部节奏、因果和转场手感。

## 3. 正文撰写

使用 `references/chapter-template.md` 作为正文约束，但不要把模板说明文字写进最终章节文件。

最终写入 `manuscript/` 时默认输出纯正文：

- 从第一句故事正文直接开始
- 不写 `# 第X章` 一类 Markdown 标题
- 不写 `## 本章概要`、`## 正文`、`## 章节备注`
- 不写 `---` 分隔线
- 章节摘要、伏笔备注、下章预告写回 `task_log.md`、`docs/大纲.md` 或 `plot/伏笔记录.md`

正文硬性要求：

- 默认目标：3000-5000 字
- 平台向默认开头：前 20% 必须有即时冲突、异常信息、强认知钩子或高吸引力动作
- 如果用户明确要求文学感较强的网文开篇，可不立刻爆动作，但前几段必须同时建立认知钩子和情感钩子，不能只剩气氛
- 每章至少推进一条主线或重要支线
- 每章至少回应一个旧悬念
- 每章至少设置一个新钩子或升级现有钩子
- 每章至少有一个核心爽点、情绪爆点、关系爆点或信息收益
- 如果本章安排主角受挫，正文内或紧邻章节必须给出明确补偿基础

正文提醒：

- 日常不是填充物，要么服务人物，要么服务关系，要么服务世界，要么服务伏笔或信息
- 冲突最好能追溯到既有世界观，而不是临时起意
- 局部解答之后，尽量立刻引出下一层问题，避免悬念链断掉
- 新设定优先通过角色经历、冲突后果和环境细节带出
- 人物成长优先写成“被困境逼出来的渐进变化”
- 文笔优先按“四层模型”运转：旁白负责必要交代，画面负责场面成立，思想负责 POV 内部推进，情绪尽量藏进动作、环境和细节，不直说大词
- 场景不要写成悬浮皮影戏；尽量让人物和环境发生实际接触，例如抓、碰、躲、借、踩、闻、撞、顺手利用身边物件
- 优先追求“一笔多用”：一个动作、物件或细节，最好同时服务情绪、人设、伏笔、关系或后续功能
- 群像模式下，单章默认只设置 1 个主 POV，必要时最多 2 个
- 直接对话不是越多越好；高冲突场景可以密，但大段对白前后要有动作、环境、心理或转述承接
- 连续三句以上直接对话时，把它当成复查信号，而不是格式目标；检查是否有信息能改为叙述承载
- 不要把每个推进点都写成独立小段落或独立任务块，正文应当像一股连续水流，而不是清单推进
- 相邻推进点之间优先补“过桥层”：上一动作留下的余波、人物反应、环境变化、视线落点、身体动作延续或因果触发
- 如果一个转折来得太突然，先补半拍反应或观察，再推进下一步
- 正文默认贴紧本章主 POV，优先让读者和角色一起看到、猜到、误会到
- 不要在同一段里同时钻进两个人的脑内；切 POV 必须有明确收益，且最好通过场景切换或明显段落转换完成
- 非当前 POV 的态度和判断，不要直接宣布；让当前 POV 从动作、语气、停顿、视线和行为里读出来
- 能留成悬念的信息不要急着解释，尤其不要提前暴露对方真实想法来破坏代入和拉扯

按需读取以下资料：

- 开头与章节结构：`references/chapter-guide.md`
- 文学感较强的网文开篇：`references/literary-opening.md`
- 对话：`references/dialogue-writing.md`
- 扩写：`references/content-expansion.md`
- 日常叙事：`references/daily-narrative.md`
- 钩子：`references/hook-techniques.md`
- 连贯性：`references/consistency.md`
- 非线性结构：`references/nonlinear-narrative.md`

## 4. 写后检查

默认检查入口：

- `references/quality-checklist.md`
- `references/consistency.md`

最低检查项：

1. 人设一致性
2. 世界观与法则一致性
3. 伏笔推进与回收状态
4. 时间线连续性
5. 节奏和情绪曲线
6. 字数是否达标
7. 是否有 AI 痕迹、抽象心理、空泛感悟式结尾
8. 本章是否真的推进剧情、改变关系或升级风险
9. 悬念是否仍保持前台未决状态
10. 主角受挫后是否获得补偿基础、信息收益、关系回应或价值确认

群像模式额外检查：

11. 本章是否真的推进了某条角色线，而非原地切视角
12. 是否出现“伪群像”角色，只围绕某个主角打转
13. 多条线之间是否存在交汇、映照或冲突，而不是彼此无关

写后必须额外做一次“AI 痕迹净化复检”，至少检查：

- 高风险套语是否成串复用；若可以改写，优先改成具体动作、环境、代价或后果
- 是否出现解释腔和说明文化连接词堆积，而不是让因果落在场面里
- 对话是否过密；能用动作、心理、环境或第三人称转述传达的信息，不必全写成台词
- 章节结尾是否在总结情绪、总结主题或替作者发言；若有，改停在动作、发现、选择或危险上

## 5. 自我优化闭环

不要只检查不修。优先做定向修复，不要全文推倒重来。

处理规则：

- 字数不足：扩展场景后果、人物反应、感官细节、信息交锋，不要注水
- 情绪跳变：补桥接段、调整推进链顺序与桥接方式、增强因果触发
- 补偿缺失：补足反制基础、阶段性回报、关系回应或信息收益
- 爽点不足：优先检查是否缺少情绪补偿节点，再强化代价、反转、决断、兑现
- 说理过重：把主题改写进选择、代价、后果和人物反应里
- 毒点偏多：删除 AI 词、抽象情绪、空泛感叹和过度解释
- AI 套语偏多：不要只删词，改写成更具体的动作、物体、声音、关系压力或后果
- 对话过密：删掉重复表达，把部分台词改成动作、环境、心理或转述
- 结尾升华味重：删除总结句，改用未决动作、发现、选择或危险收尾
- Beat 感太重：删掉编号式推进痕迹，合并碎段，补上余波、反应和因果桥接
- 转折生硬：不要急着跳下一点，先补动作延续、情绪过桥、视线转移或环境反馈
- 连贯性冲突：优先遵守 L3 宪法记忆，其次 L2 项目记忆
