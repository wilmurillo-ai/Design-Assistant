# Office Hours - YC 办公时间

> 转换自 gstack/office-hours，专为 WorkBuddy/OpenClaw 优化

## 概述

这是 YC 办公时间技能，提供两种工作模式：
1. **创业公司模式** - 6个强制性问题验证产品需求
2. **构建者模式** - 设计思考头脑风暴

## 触发条件

当用户说以下内容时使用此技能：
- "头脑风暴"
- "我有一个想法"
- "帮我思考这个问题"
- "验证我的产品想法"
- "产品创意咨询"

## 核心流程

### 阶段 1: 上下文收集

在开始之前，充分了解：
- 项目背景和目标
- 用户面临的具体问题
- 目标用户群体
- 已有解决方案（如果有）

### 阶段 2A: 创业公司模式

如果用户是在验证一个创业想法，使用 **6个强制性问题** 验证产品可行性：

1. **问题验证** - 用户面临的最痛点是什么？你有多少证据证明这是真的？
2. **解决方案匹配** - 你的解决方案如何解决这个问题？用户说"是的，这正是我需要的"还是"有点意思"？
3. **区分优势** - 为什么现有解决方案不够好？如果大公司做这个会怎样？
4. **用户来源** - 你认识多少有这个问题的人？你能接触到他们吗？
5. **商业模式** - 人们会付钱吗？付多少？为什么？
6. **增长模型** - 用户如何发现你的产品？CAC 和 LTV 是多少？

### 阶段 2B: 构建者模式

如果用户是在构建一个产品，使用 **设计思考头脑风暴**：

1. **问题重构** - 重新定义问题，找到本质
2. **用户视角** - 从目标用户角度思考
3. **约束创新** - 在限制条件下寻找创新
4. **快速验证** - 如何最快验证想法？

### 阶段 3: 前提挑战

验证问题假设：
- 这个假设是真的吗？
- 证据来自哪里？
- 有反面例子吗？

### 阶段 4: 替代方案

提出 2-3 种可能的实现方法：
- 大胆方案
- 保守方案
- 创新方案

### 阶段 5: 设计文档

生成正式的设计文档，包含：
- 问题陈述
- 用户画像
- 解决方案概述
- 成功指标
- 风险与缓解

### 阶段 6: 交接

提供：
- 下一步行动建议
- 需要验证的关键假设
- 创始人发现

## 重要原则

### 特异性是唯一货币
- 要求具体的用户证据而非模糊描述
- "10个人说想要" 比 "大家都想要" 更有价值

### 兴趣不等于需求
- 只关注实际行为而非口头兴趣
- 观察真实使用比演示更有价值

### 狭窄优于宽泛
- 从小切入点开始验证
- 不要试图一开始就解决所有问题

## 使用建议

此技能应该用在：
- `/plan-ceo-review` 或 `/plan-eng-review` 之前
- 用户首次提出产品想法时
- 需要验证创业概念时
