# Ralph Wiggum (Claude Code 插件) - 七步法讲明白

> 本文用「深入浅出讲明白」七步法结构写成，帮你快速搞懂核心。

---

## 1. 开篇：从一个痛点说起

你用 Claude Code 写项目，肯定遇到过这种情况：

你说"帮我修复这个登录页的几个 bug"，Claude 改完两个文件，说"好了，我修完了"，你一跑，构建失败，还有两个 bug 没修好。你得再把报错粘回去，让它继续改，改完再跑，又出新问题，再粘，再改……

这个循环全是你来推动，一会儿都不能离开。有没有办法让 Claude 自己循环下去，修好所有问题再叫你？

有，这个办法就是 Ralph Wiggum 插件。今天咱们把它讲透。

---

## 2. 澄清：它不是这些，别搞错了

❌ **它不是新的大模型** —— 不是说训练了一个叫 Ralph 的新 AI，它就是个 Claude Code 的插件，套在外面帮你控制节奏

❌ **它不会替你写代码** —— 写代码的还是 Claude，它只做一件事： Claude 写完想停的时候，拦住它，让它继续改，直到满足要求

❌ **它不是银弹** —— 不能把烂需求变成好产品，如果你的需求本身模糊不清，它只会循环很多轮还是做不对，它解决的是迭代节奏问题，不是需求问题

---

## 3. 本质：一句话讲透它到底是什么

**Ralph Wiggum 是给 Claude Code 加上的一个「自动迭代控制回路」，让 AI 按照你设定的规则，持续自我纠错直到达标，全程不需要人工干预。**

说人话就是：你把任务和验收标准写清楚，告诉它最多改多少轮，然后你就可以去喝杯咖啡，等它自己把所有 bug 修好、所有测试跑绿，最后给你一个能运行的结果。

---

## 4. 拆解：三个维度读懂核心

### 核心特性：两个机制保证可控

1. **Stop Hook 拦截**：利用 Claude Code 原生的停止钩子，在每次 Claude 准备结束会话时介入，不满足条件就不允许退出
2. **双保险控制**：`max-iterations` 限制最多循环次数（防止烧钱），`completion-promise` 明确验收标准（只有输出指定文本才停止）

### 核心价值：把 AI 拉回真实工程节奏

真实开发从来不是一次性写完就完事，都是「写→跑→错→改→再跑」循环出来的。

原来的 AI 编程是「一次性输出」，写完就下班，剩下的烂摊子你收拾。Ralph 让 AI 自己收拾烂摊子，它把反馈（报错、测试结果）自动带到下一轮，让 AI 基于反馈持续修正，直到真的搞定。

### 典型应用场景

最适合这四类需要多轮纠错的"脏活累活"：

| 场景 | 为什么适合 |
|------|------------|
| **Bug 修复** | 尤其适合修了又冒出来的顽固 bug，自动复现→定位→修复→验证 |
| **补测试用例** | 天然需要循环：补一个→失败→改代码→再补下一个，全绿就是清晰的完成信号 |
| **依赖迁移/大版本升级** | 最怕半吊子（能编译跑不起来），循环跑构建测试，把所有隐藏问题都扫出来 |
| **重构代码** | 改一块→验证一块→再改一块，回合制推进稳得一批 |

## 5. 对比：一张表格看懂区别

|Ralph Wiggum|普通 Claude Code 使用|
|---|---|
|AI 自动持续迭代，人可以走开|人必须全程推动，每轮都要手动粘报错|
|验收标准提前明确，达标才停|写完就停，不管有没有真的搞定|
|适合收尾，解决烂尾问题|适合从零开始写第一版代码|
|对需求清晰度要求高|模糊需求也能写，但留下一堆尾巴|

---

## 6. 收尾：回到开头，聊聊它对我们的意义

回到开头那个痛点：为什么我们用 AI 开发，总觉得"能用，但差点意思"，总得自己收拾尾巴？

因为原来的模式就是"AI 写一段，人擦一次屁股"，AI 永远只做第一轮，后面的脏活都留给人。

Ralph 改变了这个规则：**它让 AI 把活干完再下班**。你只需要提需求、定规则，然后就可以等结果，不用一轮一轮陪跑。

这不是什么技术革命，就是一个很小但很实用的改进——把工程里最枯燥、最磨人的迭代推进工作，自动化了。对咱们独立开发者来说，就是省了时间，能把精力花在更重要的设计决策上，而不是粘报错粘到手酸。
