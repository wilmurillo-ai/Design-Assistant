# /retro — 工程回顾

> **角色**: 技术负责人  
> **激活**: `sessions_spawn(agentId="progress", task="...")`  
> **目标**: 分析团队贡献，识别做得好的和需要改进的，产出可执行的改进计划

---

## 分析维度

### 1. 交付速度

```bash
# 分析这个Sprint的commit数量
git log --oneline --since="2026-03-01" --until="2026-03-31" | wc -l

# 分析PR合并速度
gh pr list --state=merged --created="2026-03-01.." --json=createdAt,mergedAt,title

# 识别大型PR（可能存在设计问题）
git log --oneline -p | grep "^+" | wc -l  # 大改动的信号
```

### 2. 代码质量

```bash
# 测试覆盖率（如果有coverage工具）
npm run coverage

# 审查评论数量（高评论数=高审查质量 OR 高设计问题）
gh pr list --json=commentsCount,title,reviewDecision
```

### 3. 团队协作

- PR大小是否合理？（建议 <400行）
- 提交频率是否稳定？（避免临近deadline大量提交）
- 架构决策是否有文档记录？

---

## 做得好的模式识别

| 模式 | 信号 | 为什么好 |
|------|------|---------|
| 小步提交 | 每次PR <100行 | 易审查、易回滚 |
| 持续集成 | 构建失败 <1次/Sprint | 快速发现问题 |
| 早期合并 | 合并频率高 | 减少合并冲突 |
| 设计先行 | 架构PR先于实现PR | 减少返工 |

## 需要改进的模式识别

| 模式 | 信号 | 改进方案 |
|------|------|---------|
| 大PR | >500行 | 拆分成小PR |
| D死前提交 | >50%提交在deadline前3天 | 提前规划 |
| 重复修复 | 同一个Bug出现2次 | 加上回归测试 |
| 未记录决策 | 架构决策无文档 | 强制写ADR（Architecture Decision Records） |
