# 并行子任务执行器

> AI干大事会超时？那就拆开干，并行干，超时继续干。

## 核心理念

**痛点**：AI处理大任务（100+文件、批量分析、多步骤流程）容易超时或质量下降。

**解法**：
1. **拆分**：大任务 → N个子任务（每个≤5分钟）
2. **并行**：同时spawn 3-5个子agent
3. **监控**：实时检查进度，超时自动继续
4. **汇总**：结果聚合 + 完成度报告

**实战验证**：
- 130个PDF → 40个框架（今天上午实战）
- 7+1批次并行，效率提升3-6倍
- 640个知识库文件健康检查，20秒完成

## 适用场景

- 批量文件处理（100+文件）
- 多维度调研（6个并行Agent）
- 大规模数据分析
- 多步骤Pipeline执行
- 任何会超时的"大任务"

## 执行流程

### Phase 0: 任务分析（入口）

**先判断任务类型**：

| 任务特征 | 拆分策略 | 并行数 |
|---------|---------|--------|
| 文件批量处理 | 每批5-10个文件 | 5 |
| 多维度调研 | 每维度1个agent | 6 |
| Pipeline多阶段 | 每阶段1个agent | 3 |
| 大数据分析 | 每批100条数据 | 5 |
| 复杂推理 | 每个推理分支1个agent | 3 |

**输入格式**：
```
任务描述：[具体任务]
总数量：[文件数/数据量/步骤数]
单任务耗时：[预估每个子任务耗时]
```

**输出确认**：
```
拆分策略：每批5个文件，共27批
并行数：同时spawn 5个子agent
预估总时间：27批 × 5分钟 ÷ 5并行 = 27分钟
确认执行？(y/n)
```

---

### Phase 1: 任务拆分

**自动拆分逻辑**：
```
总数量 = N
单任务耗时 = T分钟
超时阈值 = 5分钟（硬性限制）

每批数量 = floor(5分钟 / T分钟)
总批次数 = ceil(N / 每批数量)
并行数 = min(5, 总批次数)
```

**生成任务队列**：
```json
{
  "batches": [
    {"id": 1, "items": ["file1", "file2", "file3", "file4", "file5"]},
    {"id": 2, "items": ["file6", "file7", "file8", "file9", "file10"]},
    ...
  ],
  "parallel_count": 5,
  "timeout_per_batch": 300000,
  "progress_file": "parallel-progress.json"
}
```

**保存进度文件**：`parallel-progress.json`
```json
{
  "total_batches": 27,
  "completed_batches": [],
  "failed_batches": [],
  "current_batch": 0,
  "start_time": "2026-04-18T13:00:00Z"
}
```

---

### Phase 2: 并行执行

**执行策略**：
```
while (未完成的批次 > 0):
  可并行数 = min(parallel_count, 未完成批次数)
  
  spawn 可并行数 个子agent:
    - 每个子agent负责一个批次
    - 超时设置：5分钟
    - 输出保存到：results/batch-{id}.json
  
  等待所有子agent完成或超时
  
  检查进度：
    - 成功 → 记录到completed_batches
    - 超时/失败 → 记录到failed_batches（稍后重试）
  
  更新progress文件
  
  展示进度条：
    [████████████░░░░░░░░] 15/27 (55.6%) | 预计剩余12分钟
```

**spawn子agent的参数**：
```
{
  "task": "处理batch-{id}的文件",
  "items": [...],
  "timeout_seconds": 300,
  "output_path": "results/batch-{id}.json"
}
```

---

### Phase 3: 进度监控（实时）

**监控指标**：
- 完成批次数 / 总批次数
- 成功率 = completed / (completed + failed)
- 预估剩余时间
- 当前活跃agent数

**进度展示**：
```
并行执行中...
┌──────────────────────────────┐
│ [████████████░░░░░░░░] 15/27 │
│ 完成率：55.6%                │
│ 成功：15  失败：0  待处理：12 │
│ 预估剩余：12分钟              │
│ 活跃agent：5                 │
└──────────────────────────────┘
```

**超时处理**：
```
子agent超时 → 记录到failed_batches
策略：
1. 首次超时 → 稍后重试（降级策略）
2. 二次超时 → 标记为"需人工介入"
3. 更新progress文件（断点续传）
```

---

### Phase 4: 断点续传（恢复）

**恢复机制**：
```
检查parallel-progress.json是否存在？
- 存在 → 读取进度，从未完成的批次继续
- 不存在 → 从Phase 0重新开始

恢复逻辑：
1. 读取completed_batches和failed_batches
2. 未完成的批次 = 总批次 - completed - failed
3. 从未完成批次继续执行Phase 2
```

**恢复场景**：
- 用户中断（Ctrl+C）
- 网络断开
- 主agent超时
- 系统重启

**恢复确认**：
```
发现上次执行进度：
- 已完成：15批
- 失败：2批（将重试）
- 待处理：10批
从上次进度继续？(y/n)
```

---

### Phase 5: 结果汇总

**汇总策略**：
```
读取所有results/batch-{id}.json

合并逻辑：
1. 成功结果 → 合并到final-results.json
2. 失败结果 → 记录到failed-items.json
3. 统计分析 → 生成汇总报告

输出：
- total_processed: 总处理数
- success_count: 成功数
- failed_count: 失败数
- success_rate: 成功率
- execution_time: 总耗时
- parallel_efficiency: 并行效率提升倍数
```

**汇总报告格式**：
```
并行执行完成！
┌──────────────────────────────┐
│ 总批次：27                   │
│ 成功：25  失败：2            │
│ 成功率：92.6%                │
│ 总耗时：27分钟               │
│ 并行效率：5倍（vs串行135分钟）│
│ 失败项已保存到failed-items.json │
└──────────────────────────────┘
```

## 参数配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `parallel_count` | 5 | 同时spawn的子agent数量 |
| `timeout_per_batch` | 300秒 | 每批次超时阈值 |
| `max_retries` | 2 | 失败批次最大重试次数 |
| `progress_file` | parallel-progress.json | 进度保存文件 |
| `results_dir` | results/ | 结果保存目录 |

**动态调整**：
- 复杂任务 → 降低parallel_count（避免资源争抢）
- 简单任务 → 提高parallel_count（提升效率）
- 网络不稳定 → 降低timeout_per_batch（快速失败重试）

---

## 错误处理

### 子agent失败
- 首次失败 → 重试（降级策略）
- 二次失败 → 标记为"需人工介入"
- 保存失败原因到failed-items.json

### 主agent中断
- Ctrl+C → 保存进度，下次恢复
- 网络断开 → 自动恢复
- 超时 → 保存进度，提示用户继续

### 结果合并失败
- JSON解析失败 → 跳过该批次，记录错误
- 文件不存在 → 检查是否子agent未完成
