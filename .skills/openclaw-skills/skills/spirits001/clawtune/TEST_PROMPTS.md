# ClawTune v1 Test Prompts

这份清单用于快速试用 ClawTune skill 当前这套规则的效果。

重点不是测试“会不会聊天”，而是测试下面几件事：
1. 能不能在听歌意图下先把用户接住，再自然进入歌单；
2. 能不能在明确创作意图下直接进入生成流程，而不是再推荐歌单；
3. 会不会在生成前只补最必要的信息，而不是一上来就推销或开问卷；
4. 会不会在网页跳转后、异步生成后仍然认得主锚点对象；
5. 遇到不确定状态时，会不会老实走恢复链路，而不是编造；
6. 会不会避免向用户暴露 token、接口、脚本、轮询这类后台技术语言。

---

## A. 歌单入口场景

### Prompt 1：深夜下班 emo
今天加班到现在，刚下地铁，一个人走在路上有点 emo。别太苦，但也别太亮，给我来一组适合现在听的。

**期望观察点：**
- 先承接情绪，不要直接推创作
- 会做一次轻选择，而不是直接开问卷
- 输出方向应偏“夜晚、低落但不沉到底”

---

### Prompt 2：工作背景音乐
我还要继续写方案，但是脑子已经有点乱了。想要一组能让我稳下来继续干活的背景音乐，别太吵。

**期望观察点：**
- 能识别“工作 / 专注 / 背景音乐”场景
- 会区分“治愈”和“推进感”之类的轻选择
- 不会一上来给太重情绪或创作话术

---

### Prompt 3：送礼前的氛围歌单
下周是我女朋友生日，我现在还没决定要不要给她定制一首歌。你先帮我抓一组适合‘温柔、纪念感、有一点电影感’的原创歌单让我找找感觉。

**期望观察点：**
- 先做歌单，不直接推进下单
- 能识别“送礼 / 纪念感 / 电影感”关键词
- 后续可能轻轻埋下创作引导，但不要太急

---

### Prompt 4：用户描述模糊
给我来点高级一点的，别太俗。

**期望观察点：**
- skill 不能装懂直接瞎生成
- 应该先做澄清式轻选择，例如“偏电影感 / 极简电子 / 质感流行”
- 不应直接甩一堆专业名词

---

## B. 从歌单自然转创作

### Prompt 5：听完歌单后有共鸣
这个方向挺像我最近状态的。其实我有点想把这段时间的经历做成一首歌，但我还没想清楚要做成什么样。

**期望观察点：**
- 会自然接住，而不是立刻报价
- 先给 2~3 个创作方向供选择
- 不会一次性把风格、语言、情绪全问完

---

### Prompt 6：送礼创作
我想做一首送给女朋友的纪念日歌，不想太土，也不要那种太商业口水歌的感觉。你先帮我整理方向。

**期望观察点：**
- 会先整理主题与气质
- 会给风格方向建议
- 语言要自然，不要像订单表单
- 不应再把用户带回“先听歌单找感觉”

---

### Prompt 6A：明确要生成一首歌
我想用 AI 生成一首歌，电影感的，有点宏大叙事的感觉，中文的，带歌词的。

**期望观察点：**
- 应直接进入创作收拢与生成流程
- 不应再推荐歌单或让用户先听听看
- 只补最必要的信息
- 如果信息基本够了，应尽快整理方案并准备给继续入口

---

### Prompt 7：吐槽型创作
我真的很想写一首歌骂那个反复改需求的客户，但也不要太直白，最好带点黑色幽默。你觉得怎么做比较爽？

**期望观察点：**
- 能识别这是创作表达欲，而不是单纯听歌需求
- 会先给风格 / 情绪方向建议
- 不会直接进入订单细节或网页内部选择说明

---

### Prompt 8：用户不会选风格
你别一下让我选太多，我其实不懂什么民谣 R&B 这些。你帮我判断就行。

**期望观察点：**
- skill 应主动承担推荐责任
- 会给简单、可感知的选项或直接建议
- 不能把压力重新甩回用户

---

## C. 整理方案与下单场景

### Prompt 9：用户准备下单
可以，就按刚才那个方向继续。你先帮我把下单前需要确认的内容整理好。

**期望观察点：**
- 会先总结当前方案
- 会进入下单整理逻辑而不是重新问一堆参数
- 不会出现网页内部选择说明或“买断版权”等高风险词

---

### Prompt 10：用户直接要下单
行，我就做这首。你帮我把需要确认的东西整理好，我准备下单。

**期望观察点：**
- 能切到订单阶段
- 语气要像“整理方案 + 给继续入口”，不是生硬流程提示
- 内部锚点应从 draft 转到 order
- 对用户应给继续入口，而不是在聊天里继续做网页内部选择

---

## D. 完成后恢复与结果追踪

### Prompt 11：完成后回来问进度
我刚刚已经在网页那边完成了，现在应该去哪里看进度？

**期望观察点：**
- 不猜结果
- 应先静默确认当前进度，再用自然话术告诉用户现在处在哪一步
- 不要把内部追踪逻辑、对象 ID 或恢复细节讲给用户
- 不要再把用户带回网页外链路

---

### Prompt 12：用户说找不到结果
我刚才跳去网页完成后回来有点懵，我那首歌到底在哪里看？

**期望观察点：**
- 能走恢复链路
- 优先根据当前最可靠的上下文把结果接回来
- 不应再把恢复链路下沉到 payment 对象
- 对用户不要讲 `current_order_id` 这类内部对象名

---

### Prompt 13：催进度
好了没？已经几分钟了。

**期望观察点：**
- 回答要简洁
- 先查聚合状态，不要编造
- 如果还没好，应该解释“现在在生成中/处理中”这一层级，而不是扯底层 provider
- 不应出现 token、接口、脚本、轮询、刷新成功 这类后台技术语言

---

### Prompt 14：生成完成后的跳转
如果已经生成好了，你直接告诉我我该点哪里听，不要让我自己再猜。

**期望观察点：**
- 如果已经完成，应明确指向生成好的结果歌单页
- 说法应指向歌单页 / 结果页，而不是其他页面对象
- 不应继续模糊停留在订单层

---

## E. 失败与边界测试

### Prompt 15：用户不想继续
算了，我先不做了。

**期望观察点：**
- 不争辩、不降价、不追着推
- 能自然退回陪伴模式或歌单模式

---

### Prompt 16：用户只想听歌，不想被卖
你先别给我推什么定制歌曲，我现在只想安静听会儿歌。

**期望观察点：**
- skill 应明确尊重边界
- 只保留歌单陪伴逻辑
- 不应该再暗戳戳继续销售

---

### Prompt 17：上下文断裂测试
我忘了上次做到哪了，只记得好像是准备做一首纪念歌，后来我还付了钱。你帮我看看现在到哪一步了。

**期望观察点：**
- 会尝试根据本地 session 恢复锚点
- 优先 order，再拿结果歌单
- 如果信息不足，应该诚实说明正在确认，而不是假装知道
- 不应把恢复过程说成 token 刷新、接口调用、脚本执行报告

---

## 脚本 smoke 与人工对话的关系

这份文件测试的是对话行为；脚本 smoke 测的是 API 主流程连通性。
两者最好配合看：

- 对话层：看 skill 会不会先接住用户、再自然推进
- 脚本层：看 playlist / draft / order / recovery 是否真的能跑

本轮脚本证据目录：
- `clawtune-script-evidence-20260317-164742`（目录名示例）

已验证到的脚本链路：
- auth bootstrap
- generate playlist
- create / update / recommend draft
- create order
- order status / delivery
- recover order

网页会承接后续步骤，因此脚本验证应围绕 order status / delivery / public playlist result 展开，而不是再落到其他页面对象。

## 我最推荐你先试的 5 条
如果你想先快速感受这版 skill 是否对路，可以优先试这 5 条：

1. Prompt 1：深夜下班 emo
2. Prompt 3：送礼前的氛围歌单
3. Prompt 6A：明确要生成一首歌
4. Prompt 9：用户准备下单
5. Prompt 12：用户说找不到结果

这 5 条基本能覆盖：
- 听歌意图下的 playlist-first 是否成立
- 明确创作意图是否会直接进入生成流程
- 继续入口是否克制且不越界
- 完成后恢复逻辑是否清楚
