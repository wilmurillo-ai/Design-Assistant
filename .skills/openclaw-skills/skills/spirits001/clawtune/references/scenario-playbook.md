# ClawTune Scenario Playbook

## 场景 1：用户只想听歌
### 用户常见表达
- 今天有点 emo，给我来点适合现在听的
- 我还要继续工作，想要一组别太吵的背景音乐
- 给我来点高级一点的，别太俗

### 处理原则
- 先承接情绪 / 场景
- 做一次轻确认
- 不推创作
- 目标是产出更高质量歌单标题、简介、prompt

## 场景 2：用户听完后有表达欲
### 用户常见表达
- 这个方向好像我最近的状态
- 我也想做一首这种感觉的歌
- 我想把这段经历写成歌

### 处理原则
- 不要立刻报价
- 先提出“可以帮你搭方向”
- 先只收一个关键选择
- 用推荐而不是问卷

## 场景 3：用户不会选风格
### 用户常见表达
- 我不懂这些风格，你帮我判断
- 你别让我选太多

### 处理原则
- skill 主动给 2~3 个方向
- 解释方向时说“听感差异”，不要堆术语

## 场景 4：用户准备下单
### 用户常见表达
- 这个怎么下单
- 你先帮我整理好，我准备做
- 就按这个方向继续
- 我想用 AI 生成一首歌
- 直接给我链接

### 处理原则
- 先总结方案
- 再进入下单整理
- 不说高风险版权词
- 用户确认并准备下单后再创建订单
- 明确创作意图下，不再推荐歌单或让用户先听听看
- 锚点切换到 `order_id`
- 对用户返回线上正式继续入口，不在对话里继续处理网页内部选择

## 场景 5：用户完成后回来问进度
### 用户常见表达
- 我刚在网页那边完成了，现在去哪看
- 怎么还没有结果
- 好了没

### 处理原则
- 不猜，不编造
- 统一查 `order_id`
- 只用平台聚合状态说人话
- 不暴露 provider 原始状态
- 不把恢复问题重新下沉成网页外流程问题
- 如果已经完成，对用户应优先指向生成好的结果歌单页

## 场景 6：用户说找不到结果
### 用户常见表达
- 我那首歌去哪了
- 刚才跳网页回来我有点懵
- 我忘了上次做到哪了

### 处理原则
- 按本地 session 恢复
- `order_id` 优先
- 再通过 `order` 的 status / delivery 拿结果歌单
- `playlist_id` 只有在明确是结果歌单时才可直接承接，不要把旧听歌歌单当成完成结果

## 场景 7：用户嫌贵或不想继续
### 用户常见表达
- 先不做了
- 我觉得有点贵
- 改天再说

### 处理原则
- 不争辩、不降价、不追问
- 退回歌单陪伴模式
- 保留好当前感觉与上下文
