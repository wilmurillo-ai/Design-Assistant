# 记忆系统配置模板 (memoryFlush + memorySearch)

**版本**: 1.0.0  
**用途**: 配置 OpenClaw 记忆系统（防止失忆 + 语义检索）  
**位置**: `~/.openclaw/config/core.json` 或单独配置文件

---

## 📋 配置说明

OpenClaw 记忆系统包含两个核心组件：

1. **memoryFlush** - 在上下文压缩前自动保存关键信息（防止失忆）
2. **memorySearch** - 语义记忆检索（秒回历史信息）

---

## 一、memoryFlush 配置

### 问题背景

**为什么 OpenClaw 聊着聊着 AI 会"失忆"？**

原因是 OpenClaw 的**上下文压缩（compaction）**触发了。

每个模型都有上下文窗口限制（Claude 是 200K token）。当对话接近这个限制时，OpenClaw 会自动把旧对话压缩成摘要腾出空间——压缩过程中可能丢失细节。

### 解决方案

开启 memoryFlush，在压缩触发之前，先让 AI 把重要信息写入文件。

### 配置示例

```json
{
  "agents": {
    "defaults": {
      "compaction": {
        "reserveTokensFloor": 20000,
        "memoryFlush": {
          "enabled": true,
          "softThresholdTokens": 4000
        }
      }
    }
  }
}
```

### 参数说明

| 参数 | 含义 | 推荐值 | 说明 |
|------|------|--------|------|
| `reserveTokensFloor` | 压缩后至少保留多少 token 的最近对话 | 20000 | 值越大，保留的最近对话越多 |
| `memoryFlush.enabled` | 是否开启压缩前自动保存 | true | 强烈建议开启 |
| `memoryFlush.softThresholdTokens` | 距离压缩多少 token 时触发保存 | 4000 | 提前 4000 token 触发，给 AI 留出保存时间 |

### 工作原理

```
对话进行 → 接近上下文限制
    ↓
距离压缩还有 4000 token (softThresholdTokens)
    ↓
触发 memoryFlush
    ↓
AI 自动将关键信息写入 memory/YYYY-MM-DD.md
    ↓
继续对话 → 触发压缩
    ↓
压缩后保留 20000 token 最近对话 (reserveTokensFloor)
```

### 验证方式

开启 verbose 模式（`/verbose`）可以看到提示：
```
🧹 Auto-compaction complete: Flushed 3 key memories to file
```

## 二、memorySearch 配置

### 问题背景

**为什么需要 memorySearch？**

- 手动翻记忆文件效率低
- 关键词搜索精度差（同义词、表述差异）
- 语义检索可以理解"部署问题" = "nginx 配置错误"

### 推荐方案：SiliconFlow bge-m3（免费）

国内用户最快上手的方案——SiliconFlow 提供的 bge-m3 向量模型**完全免费**。

### 配置示例

```json
{
  "memorySearch": {
    "enabled": true,
    "provider": "openai",
    "remote": {
      "baseUrl": "https://api.siliconflow.cn/v1",
      "apiKey": "{{SiliconFlow API Key}}"
    },
    "model": "BAAI/bge-m3"
  }
}
```

### 参数说明

| 参数 | 含义 | 推荐值 | 说明 |
|------|------|--------|------|
| `enabled` | 是否启用 memorySearch | true | 开启语义检索 |
| `provider` | 向量模型提供商 | openai | 使用 OpenAI 兼容 API |
| `remote.baseUrl` | API 基础 URL | https://api.siliconflow.cn/v1 | SiliconFlow 地址 |
| `remote.apiKey` | API Key | (需申请) | 在 SiliconFlow 控制台获取 |
| `model` | 向量模型 | BAAI/bge-m3 | 免费、中英文双语、1024 维度 |

### 为什么推荐 bge-m3？

| 特性 | 说明 |
|------|------|
| **完全免费** | SiliconFlow 提供免费额度，个人使用足够 |
| **中英文双语** | 对中文支持好，检索精度高 |
| **1024 维度** | 精度和性能的平衡点 |
| **OpenAI 兼容** | 配置简单，无需特殊适配 |

### 获取 API Key

1. 访问 https://siliconflow.cn/?ref=tbbbk.com
2. 注册账号
3. 进入控制台 → API Keys
4. 创建新 Key
5. 复制到配置中

## 三、结构化日志规范

memorySearch 的底层是向量语义检索，**结构化日志的命中率比非结构化日志高得多**。

### 搜索示例

**搜索词**: "nginx 部署问题"

❌ **非结构化日志**（命中率低）:
```markdown
今天折腾了一天，先搞了数据库迁移，然后部署了新版本，
nginx 配置改了一下，报错了，查了半天...
```

✅ **结构化日志**（命中率高）:
```markdown
### [PROJECT:MyApp] nginx 反代配置
- **结论**: 用 nginx 反代部署成功
- **教训**: upstream 要用 127.0.0.1 不要用 localhost（IPv6 问题）
- **标签**: #nginx #deploy #myapp
```

### 日志模板

```markdown
### [PROJECT:项目名称] 简短标题
- **结论**: 一句话总结
- **文件变更**: 涉及的文件路径
- **教训**: 踩坑点（如有）
- **标签**: #tag1 #tag2
```

### 标签规范

推荐使用以下标签类别：

| 类别 | 标签示例 | 说明 |
|------|---------|------|
| 项目 | `#myapp` `#website` `#bot` | 项目名称 |
| 操作 | `#deploy` `#config` `#debug` | 操作类型 |
| 技术 | `#nginx` `#docker` `#python` | 技术栈 |
| 问题 | `#port-conflict` `#permission` `#network` | 问题类型 |

## 四、自动记忆维护

防止过期日志干扰 memorySearch 结果，建议每周自动维护一次。

### HEARTBEAT.md 配置

在 `HEARTBEAT.md` 中添加：

```markdown
## 记忆维护（每周一次）

读取 `memory/heartbeat-state.json`，检查 `lastMemoryMaintenance` 字段。
如果距今 >= 7 天：

1. 读最近 7 天的 `memory/YYYY-MM-DD.md` 日志
2. 提炼有价值的信息到对应文件（projects.md / lessons.md）
3. 压缩已完成一次性任务的日志为一行结论
4. 删除过期信息
5. 更新 `heartbeat-state.json` 的 `lastMemoryMaintenance` 为今天
```

### heartbeat-state.json

创建 `memory/heartbeat-state.json`:

```json
{
  "lastMemoryMaintenance": "2026-03-08",
  "lastChecks": {
    "email": 1703275200,
    "calendar": 1703260800,
    "weather": null
  }
}
```

## 六、验证与调试

### 验证 memoryFlush

```bash
# 1. 开启 verbose 模式
/verbose

# 2. 进行长对话（接近上下文限制）

# 3. 检查日志
tail -f ~/.openclaw/logs/gateway.log | grep "memoryFlush"

# 4. 检查记忆文件
ls -la ~/agents/writer/memory/
```

### 验证 memorySearch

```bash
# 1. 重建索引
openclaw memory index

# 2. 测试搜索
openclaw memory search "nginx 部署问题"

# 3. 检查搜索结果精度
# 应该返回包含 #nginx #deploy 标签的日志
```

### 常见问题

**Q1: memoryFlush 未触发？**
- 检查 `memoryFlush.enabled` 是否为 `true`
- 对话长度可能还未接近限制
- 开启 verbose 模式查看日志

**Q2: memorySearch 精度低？**
- 检查日志是否为结构化格式
- 检查标签是否规范
- 重新建立索引：`openclaw memory index`

**Q3: SiliconFlow API 报错？**
- 检查 API Key 是否正确
- 检查账户余额（免费额度可能用完）
- 检查网络连接
