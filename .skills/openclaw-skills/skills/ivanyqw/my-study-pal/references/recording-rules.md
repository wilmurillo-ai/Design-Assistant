# 概念解释记录与用户画像写入规则

本文件定义 `my-study-pal` 在使用过程中如何写入 `mystudy/`。

与其他文件的区别：
- `setup.md` 负责首次启用、补齐骨架和可运行状态检查
- `references/blueprint.md` 负责定义最小可运行骨架
- `references/study-storage.md` 负责定义目录、文件职责、模板内容
- `recording-rules.md` 负责定义每次概念解释对话发生时怎么记、记多少、记到哪里

## 总原则

- 默认原则：只要是形成了明确概念主题的解释型回答，就要落盘，不再以“可记可不记”处理
- 只有纯寒暄、纯确认、或没有带来有效理解推进的消息，才可以不记录
- 记录要服务后续继续解释和追问，不是机械存档
- 优先记“以后再遇到这个概念还用得上”的信息
- 用户画像要服务内容组织和风格适配，不是为了堆资料
- `study-summary.md` 中的时间统一精确到时分秒
- `study-detail/` 中默认直接保留完整对话过程，不再额外总结“结果”“建议”等纪要项
- 每次写入前先读取已有文件，避免重复、冲突和覆盖
- 如果内容涉及隐私或敏感信息，只保留必要摘要
- 如果系统尚未达到可运行状态，先完成骨架补齐，再进入长期记录流程

## 一、何时创建或更新 `study-summary.md`

在以下场景更新：

1. 用户开始一个新概念主题时，必须更新
2. 某概念理解完成一个阶段时，必须更新
3. 用户明确说“记录一下”“帮我保存进度”“今天先到这”时，必须更新
4. 已经形成明显的阶段性结果，适合沉淀概览时，必须更新
5. 同一天连续学习多个概念时，每出现一个新的明确概念主题，都要补入 summary，不得遗漏

### 写入内容

每条主题记录至少包含：
- 概念主题
- 最近解释时间（精确到时分秒）
- 是否完成

## 二、何时创建或更新 `study-detail/` 文件

在以下场景创建或追加详情：

1. 发生了一次有实质内容的概念解释型对话时，必须创建或追加
2. 用户围绕同一概念进行了理解、辨析、比较、语境判断或追问时，必须追加
3. 后续大概率还会继续追问，需要留下脉络时，必须追加
4. 只要本轮回答已经包含定义、例子、区分点、纠偏点中的任一项，就视为“有实质内容”，不得跳过 detail 记录

### 最少保留字段

- 概念主题
- 日期
- 完整对话过程

## 三、何时创建或更新 `user-profile.md`

在以下场景更新：

1. 用户明确透露自己的工作 / 专业 / 业务领域
2. 用户明确透露自己的爱好领域
3. 用户近期学习了某些概念，且这些信息对后续举例有价值
4. 用户明确提出长期教学偏好
5. 用户明确提出长期语言风格偏好

### 写入原则

- 只记录对后续讲解有帮助的信息
- “长期偏好”才写入用户画像
- 临时偏好默认不覆盖长期设置，可在 detail 中备注
- “最近的 5 个学习知识点”优先保留最近、最可能复用的内容，超过 5 个时替换最旧或最不相关项

## 四、写入粒度规则

### 适合详细记的情况

- 出现了清晰的概念拆解
- 给出了关键区分点、语境判断或类比
- 用户显露出明确易混点
- 本次内容对后续继续解释明显有价值
- 用户画像新增了对后续取材很关键的信息

### 适合简略记的情况

- 只是一个很小的追问
- 只是确认一句话
- 只是补充一条轻量用户画像信息

### 不必记的情况

- 纯寒暄
- 与概念解释无关的闲聊
- 没有形成任何有效理解推进
- 仅有“好”“知道了”“学会了”这类确认，且没有新增概念内容

## 五、用户画像更新规则

### 1. 工作 / 专业 / 业务领域

- 优先记录稳定领域，不要把一次临时任务误记成长期领域
- 如果用户有多个身份，可并列记录

### 2. 爱好领域

- 只记录用户明确表达过、且适合拿来举例的兴趣领域
- 不要因为一次随口提到就强行写入长期画像

### 3. 最近的 5 个学习知识点

- 用于后续解释时做类比、串联和复用
- 保持最多 5 个
- 优先保留最近且可迁移的概念

### 4. 教学偏好

- 重点记录“讲解式”下的输出结构和可配置项
- “引导式”先保留结构，不强行补内容
- 用户说“以后默认这样”这类表述时，必须同轮写入

### 5. 语言风格偏好

- 记录用户长期偏好的表达皮肤

## 六、推荐写入流程

1. 判断本次是否属于概念解释型对话
2. 检查系统是否达到可运行状态；如果没有，先按 `setup.md` 和 `references/blueprint.md` 完成骨架补齐
3. 判断是否出现了可复用的用户画像信息
4. 读取 `study-summary.md`、目标 detail 文件、`user-profile.md`
5. 更新概念解释记录
6. 需要时更新用户画像
7. 再次检查是否出现重复、冲突或过度冗长

## 七、主题命名建议

尽量使用可续接的概念主题名，例如：
- 提示词工程
- 人因工程
- 机会成本
- 边际效应
- Python 列表推导式

避免：
- 今天聊的这个
- 一个问题
- 临时测试

## 八、对话记录写法

推荐写法：
- 按实际对话顺序直接记录
- 保留用户原话和助手原话
- 不额外补写“总结”“结果”“建议”这类纪要项

目标是让后续接手的人或未来的自己直接看到当时怎么问、怎么答。
