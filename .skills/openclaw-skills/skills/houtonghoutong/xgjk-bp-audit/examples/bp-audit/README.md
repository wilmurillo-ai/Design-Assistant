# BP 目标体系全面审计 — 使用说明

## 什么时候使用

- 用户需要对 BP 目标体系进行系统性审计诊断时
- 用户想检查某个组织/个人的 BP 是否合规、是否正确承接上级、下级是否正确承接
- 用户需要分析上下级 BP 之间的差异和缺口时
- 用户想了解 BP 体系的整体健康度时

## 审计模式（按意图自动路由，严禁无脑全跑）

根据用户意图自动决定执行哪些板块，**只跑需要的板块**：

| 用户意图 | 执行板块 | 需要的数据 |
|---------|---------|----------|
| "检查BP质量"、"写得对不对"、"合不合规" | 仅板块一 | 本级数据 |
| "向上对齐"、"和上级对齐了没" | 板块一 + 二 | 本级 + 上级 |
| "下级承接"、"下面接得怎么样" | 板块一 + 三 | 本级 + 下级 |
| "GAP分析"、"差距"、"拉通上下级" | 板块一 + 二 + 三 + 四 | 全部 |
| "全面审计"、"完整审计" | 板块一 + 二 + 三 + 四 | 全部 |
| 未明确指定 | **追问用户** | — |

- 板块一（基础合规性）是所有审计的基础，任何模式都包含
- 单板块审计时报告格式相应精简，不输出未执行板块的内容
- 只有用户明确要求"全面审计"或"GAP分析"时，才执行全部四个板块

## 标准审计流程

### 第一步：确定审计范围

1. 获取启用周期：
```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py get_all_periods --format md
```
筛选"启用"状态的周期。

> **所有脚本调用必须加 `--format md`**，输出紧凑 Markdown，节省 67-80% token。

2. 确定审计对象（**搜索优先，严禁默认拉全量树**）：
   - 用户给了 `task_id` / `group_id` → 直接使用
   - 用户给了分组名称 → **优先** `search_group` 模糊搜索定位（轻量高效）：
```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py search_group --period_id "<id>" --name "集团" --format md
```
   - 用户给了目标名称 → 先搜索分组，再 `search_task` 搜索目标：
```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py search_task --group_id "<id>" --name "年度财务目标" --format md
```
   - **搜索返回多个结果时** → 必须列出每个结果的区分信息（名称、所属分组/路径、编号等），让用户确认选哪一个，**禁止自行猜测或随意选择**
   - **搜索无结果时** → 提示用户检查名称或换关键字重试
   - **仅当用户完全未指定**且搜索无法进行时 → 才获取分组树让用户选择（此接口 500KB+，极耗 token）：
```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py get_group_tree --period_id "<id>"
```

### 第二步：获取审计数据

1. 获取任务树：
```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py get_task_tree --group_id "<id>" --format md
```

2. 获取目标详情（含 KR 和 KI）：
```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py get_goal_detail --task_id "<id>" --format md
```

3. 如需上级详情（板块二、四）：
   - 从目标详情的 `向上对齐` 列表获取上级任务 ID（Markdown 中呈现为 `- 向上对齐（N项）：` 下的列表，每项含名称、`[分组名]` 和 `id`）
   - 逐个获取上级任务完整详情

4. 如需下级详情（板块三、四）：
   - 从目标/KR 详情的 `向下承接` 列表获取下级任务 ID（Markdown 中呈现为 `- 向下承接（N项）：` 下的列表，格式同上）
   - 逐个获取下级任务完整详情

### 第三步 + 第四步：逐板块审计并即时输出

**流式输出**：审完一个板块立即输出该板块结果，不等全部审完。用户能看到报告一段一段地生成。

执行顺序：
1. **先输出报告标题和审计范围**（评分留到最后）
2. **板块一**：加载 `references/quality-rules.md` → 审计 → **立即输出结果**
3. **板块二**：加载 `references/upward-check-rules.md` → 获取上级详情 → 审计 → **立即输出结果**
4. **板块三**：加载 `references/downward-check-rules.md` → 获取下级详情 → 审计 → **立即输出结果**
5. **板块四**：加载 `references/gap-analysis-rules.md` → 基于前三板块数据分析 → **立即输出结果**
6. **最后输出汇总**：回填综合评分 + 关键问题清单 + 改进建议

## 审计报告示例结构

> **核心原则**：报告中每一条结论、问题、建议都必须精确到具体的 KR/KI/目标（编号+名称），严禁使用"部分"、"某些"、"个别"等模糊指代。

```markdown
# XX中心 2026年度BP 全面审计报告

## 审计概要
- **审计对象**：XX中心 2026年BP（目标ID：2009147350299463681）
- **审计范围**：全面审计（四大板块）
- **综合评分**：7.2 / 10
- **一句话总结**：结构完整、KR指标清晰，但KI G-1.1.3和KI G-1.2.1缺少承接人，且收入指标下级覆盖率仅83%存在2500万缺口。

## 板块一：基础合规性审计

### 结构完整性：合规
- 目标 G-1 下含3个KR（G-1.1、G-1.2、G-1.3），每个KR下含2-4个KI，符合3层结构要求

### 内容质量：3个问题
| 对象 | 问题 | 等级 |
|------|------|------|
| KI G-1.1.3「推进渠道拓展」 | 描述含"推进"，过于笼统，缺少具体交付物和数量指标 | 一般 |
| KI G-1.2.1「优化成本结构」 | 描述含"优化"，无明确终点和可衡量标准 | 一般 |
| KI G-1.3.2「加强投后管理」 | 描述含"加强"，属于不可追踪举措 | 一般 |

### 逻辑自洽：1个严重问题
| 对象 | 问题 | 等级 |
|------|------|------|
| KR G-1.3「投资收益达成8000万」 | 其下2个KI（G-1.3.1「完成3个项目投资」、G-1.3.2「加强投后管理」）均为投资动作，缺少"退出/变现"类举措，KI全部完成也不能保证收益达成 | 严重 |

### 衡量标准：2个问题
| 对象 | 问题 | 等级 |
|------|------|------|
| KR G-1.2「年度利润达成5000万」 | 衡量标准缺少口径说明（净利润/毛利/EBITDA？） | 一般 |
| KR G-1.3「投资收益达成8000万」 | 衡量标准缺少数据源（哪个系统取数？） | 一般 |

### 汇报周期：合规
- KR周期（month+15）< 目标周期（month+20），符合自下而上约束
- 所有KI周期（week+5）< KR周期（month+15），合规

### 语义定义：1个问题
| 对象 | 问题 | 等级 |
|------|------|------|
| 目标 G-1「推动XX中心2026年业绩增长」 | 含动词"推动"，应改为「XX中心2026年度业绩目标已达成」 | 一般 |

### 必填项：2个严重问题
| 对象 | 问题 | 等级 |
|------|------|------|
| KI G-1.1.3「推进渠道拓展」 | 承接人为空 | 严重 |
| KI G-1.2.1「优化成本结构」 | 承接人为空 | 严重 |

## 板块二：向上对齐审计

### 对齐正确性：偏弱
| 本级目标 | 对齐上级 | 判定 | 说明 |
|---------|---------|------|------|
| G-1「推动XX中心2026年业绩增长」 | 集团 KR G-1.1「年度收入达成3亿」 | 正确 | KR G-1.1的衡量标准「收入1.5亿」能直接支撑集团收入目标 |
| G-1「推动XX中心2026年业绩增长」 | 集团 KR G-1.2「年度利润达成1亿」 | 偏弱 | KR G-1.2衡量标准仅写「利润5000万」，口径未说明是否含总部分摊 |

### 对齐完整性：2个上级KI未承接
| 未承接的上级任务 | 说明 | 等级 |
|----------------|------|------|
| 集团 KR G-1.3「投资收益达成2亿」 | XX中心有投资职能但未承接此KR | 严重 |
| 集团 KR G-1.4「资产运营效率提升」 | XX中心管理资产规模大但未承接 | 严重 |

### 人员匹配：1个问题
| 本级目标 | 本级责任人 | 上级承接人 | 判定 |
|---------|----------|----------|------|
| G-1 | 张三 | 李四 | 不匹配：集团KR G-1.1承接人为李四，但XX中心目标G-1责任人为张三 |

## 板块三：向下承接审计

### 承接覆盖率：78%（7/9个KI已承接）
| 未承接的KI | 说明 | 等级 |
|-----------|------|------|
| KI G-1.1.3「推进渠道拓展」 | 无任何下级承接，且承接人为空 | 严重 |
| KI G-1.3.2「加强投后管理」 | 无任何下级承接 | 严重 |

### 承接正确性：逐项审计
| 本级KI | 下级承接 | 判定 | 说明 |
|--------|---------|------|------|
| KI G-1.1.1「完成华东区域销售」 | [销售一部]目标「华东区域销售目标已达成」(id:xxx) | 正确 | 下级KR衡量标准「收入5000万」直接支撑 |
| KI G-1.1.2「完成华南区域销售」 | [销售二部]目标「华南区域销售目标已达成」(id:xxx) | 正确 | 下级KR衡量标准「收入4000万」直接支撑 |
| KI G-1.2.1「优化成本结构」 | [财务部]目标「成本管控体系已建立」(id:xxx) | 偏弱 | 下级KR仅关注"制度建设"，未设定成本降幅数值目标 |

### 数值覆盖明细
| 指标 | 本级要求 | 下级汇总 | 覆盖率 | 缺口 | 缺口来源 |
|------|---------|---------|--------|------|---------|
| 年度收入 | 1.5亿 | 1.25亿 | 83.3% | -2500万 | KI G-1.1.3「渠道拓展」无人承接（预估贡献2000万），华南区目标偏低500万 |
| 年度利润 | 5000万 | 4200万 | 84% | -800万 | [财务部]承接的成本优化无量化目标，无法计入利润贡献 |

## 板块四：GAP差异分析

### 承接差：中度衰减
| 上级要求 | 本级承接 | 下级执行 | 衰减等级 |
|---------|---------|---------|---------|
| 集团KR G-1.1「收入3亿」 | G-1 KR G-1.1「收入1.5亿」(50%承接) | 下级合计1.25亿 | 中度衰减 |
| 集团KR G-1.3「投资收益2亿」 | 未承接 | 无 | 完全丢失 |

### 执行差
| 本级指标 | 本级目标值 | 下级汇总值 | 执行差 | 差距率 |
|---------|----------|----------|--------|--------|
| 年度收入 | 1.5亿 | 1.25亿 | -2500万 | -16.7% |
| 年度利润 | 5000万 | 4200万 | -800万 | -16% |

### 逻辑差：1处口径不一
| 概念 | 上级理解 | 本级理解 | 影响 |
|------|---------|---------|------|
| "利润" | 集团KR G-1.2指"税后净利润" | 本级KR G-1.2未说明口径 | 可能导致统计偏差，需确认 |

### 综合风险等级：中风险（72分）

## 关键问题清单（按严重程度排序）
1. [严重] 集团 KR G-1.3「投资收益达成2亿」未被XX中心承接，存在职责盲区（板块二）
2. [严重] 集团 KR G-1.4「资产运营效率提升」未被XX中心承接（板块二）
3. [严重] KI G-1.1.3「推进渠道拓展」承接人为空且无下级承接，预估2000万收入缺口（板块一+三）
4. [严重] KI G-1.3.2「加强投后管理」无下级承接（板块三）
5. [严重] KR G-1.3「投资收益达成8000万」下缺少"退出/变现"类举措，逻辑不闭环（板块一）
6. [一般] 目标 G-1「推动XX中心2026年业绩增长」含动词"推动"，不符合静态事实定格句式（板块一）
7. [一般] KR G-1.2「年度利润达成5000万」衡量标准缺少口径说明（板块一）
8. [一般] KI G-1.2.1「优化成本结构」描述含"优化"，属不可追踪举措（板块一）
9. [一般] 目标 G-1 责任人张三与集团KR G-1.1承接人李四不匹配（板块二）

## 改进建议（按优先级排序）
1. [紧急] 补充承接集团 KR G-1.3「投资收益达成2亿」和 KR G-1.4「资产运营效率提升」，新增对应目标并指定责任人
2. [紧急] 为 KI G-1.1.3「推进渠道拓展」指定承接人，并安排下级部门承接（建议：销售三部或新渠道部）
3. [紧急] 为 KR G-1.3「投资收益达成8000万」补充"退出/变现"类举措（如：KI G-1.3.3「完成XX项目退出，实现收益XX万」）
4. [重要] 调整收入指标分配：华南区[销售二部]目标从4000万上调至4500万，或为KI G-1.1.3新增承接方补齐2000万缺口
5. [重要] 明确 KR G-1.2「年度利润达成5000万」的利润口径（净利润/毛利/EBITDA），与集团KR G-1.2统一
6. [建议] 修改目标 G-1 描述为「XX中心2026年度业绩目标已达成」
7. [建议] 修改 KI G-1.2.1 描述为「成本结构优化方案已落地，年度成本降幅达X%」
8. [建议] 确认目标 G-1 责任人与集团KR G-1.1承接人的对应关系，必要时调整
```

## 写入示例

> 仅在用户明确要求新增/补充节点时使用。建议先查询父任务详情，再执行写入。

### 用户不给 ID 时的完整定位链路

```bash
# 1. 先按周期 + 节点名称定位 group_id
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py search_group \
  --period_id "<period_id>" \
  --name "华东销售一部" \
  --format md

# 2. 再按 group_id + 目标/成果名称定位父任务 ID
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py search_task \
  --group_id "<group_id>" \
  --name "年度收入目标已达成" \
  --format md

# 3. 拿到目标 ID 后新增关键成果
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py add_key_result \
  --goal_id "<goal_id>" \
  --name "新增渠道收入增长点" \
  --format md
```

> 如果第 2 步是要找成果而不是目标，就从搜索结果里选 `类型 = 关键成果` 的那条，取它的 ID 作为 `--key_result_id` 传给 `add_action`。

### 在目标下新增关键成果

```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py add_key_result \
  --goal_id "<goal_id>" \
  --name "新增 Q2 业绩增长点" \
  --measure_standard "新增签约客户数 >= 20" \
  --plan_start_date "2026-04-01" \
  --plan_end_date "2026-06-30" \
  --format md
```

### 在关键成果下新增关键举措

```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py add_action \
  --key_result_id "<kr_id>" \
  --name "拓展三线城市分销商" \
  --measure_standard "签约 10 家以上" \
  --owner_ids "1234567890123456789" \
  --format md
```

### 使用 `--body_json` 直传原始请求体

```bash
python3 .cursor/skills-v3/bp-audit/scripts/bp-audit/bp_api.py add_key_result \
  --body_json '{"goalId":"<goal_id>","name":"新增 Q2 业绩增长点","planStartDate":"2026-04-01","planEndDate":"2026-06-30"}' \
  --format md
```

## 注意事项

1. 所有 ID 参数保持字符串原样传递，严禁 parseInt 或 Number 转换
2. 每个审计结论必须附带证据引用
3. 如果审计范围很大（多层级多分组），提前告知用户可能需要较长时间
4. 如果不确定周期或分组，向用户确认后再执行
5. GAP 分析必须在前三个板块的数据基础上执行
