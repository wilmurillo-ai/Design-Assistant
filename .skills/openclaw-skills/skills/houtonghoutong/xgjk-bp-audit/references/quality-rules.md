## 一、BP 系统核心规范（基础知识）

BP（Business Plan）是一套**责任导向的管理载体**，采用 **G-R-A 三层结构**：

- **目标（Goal）**：期望达成的结果状态（What）
- **关键成果（Result）**：判断目标是否达成的依据（So what / How to verify）
- **关键举措（Action）**：实现成果的关键行动路径（How）

BP 必须同时回答四类问题：要做成什么（目标）、做成什么样算达成（成果 + 衡量标准）、靠哪些关键行动做成（举措）、谁承担什么责任（责任模型与角色字段）。

### 三层结构

| 层级 | 名称 | 说明 |
|------|------|------|
| 第一层 | 目标 (Goal) | 期望达成的结果状态，有编码，责任人负责 |
| 第二层 | 关键成果 (Key Result) | 判断目标是否达成的依据，必须配置衡量标准 |
| 第三层 | 关键举措 (Action) | 实现成果的关键行动路径，有编码，承接人执行 |

### 三层结构的严格边界（硬规则）

**目标（Goal）边界**：
- 目标只写"结果状态"，**严格禁动词**：不写"推动/建立/提升/加强/推进/完成/落地/实施"等动作性表述
- 推荐使用"静态事实定格句式"：「[业务对象]已处于[理想状态]」
- 目标不承担衡量细节，衡量细节放到成果的衡量标准里

**关键成果（Result）边界**：
- 成果只用于判断目标是否达成，是"可验收的达成状态/阶段交付状态"，不是动作清单
- 必须符合 SMART，至少满足"具体、可衡量、有时限"
- 成果之间尽量满足 MECE（相互独立、共同穷尽）
- 每条成果必须可被衡量，必须配置衡量标准

**关键举措（Action）边界**：
- 举措只写关键行动，不写结果判断
- 必须可指派、可承接、可追踪
- 举措必须挂在对应成果下，不允许写在目标层

### 数量关系与书写复杂度控制

- 每个目标下关键成果：建议 2–5 条（优先 2–3 条；超过 3 条需证明必要性）
- 每条成果下关键举措：建议 2–6 条（优先 2–3 条；超过 3 条需证明必要性）
- 超过建议上限时，应触发"合并/拆分/聚焦"提示

### 组织拆解要求

- **集团层（顶层 org 节点）**：仅 2 层（目标 → 关键成果），不设关键举措
- **中心层**：完整 3 层（目标 → 关键成果 → 关键举措），中心目标应承接集团关键成果，中心关键举措必须可供一级部门承接
- **一级部门层**：完整 3 层，目标应承接中心关键举措，关键举措必须可供核心员工承接
- **核心员工层**：完整 3 层，个人目标应承接一级部门关键举措，关键举措必须具体到可执行动作
- 组织类型字段 `type`：`org` = 组织节点，`personal` = 个人节点

### 组织 BP 与个人 BP 的审计深度标准（Refined）

- **组织 BP**：
    - **审计重心**：战略穿透与责任闭环。
    - **必检项**：每个组织目标的 KR 和 Action 必须有对应的“向下承接”记录。如果组织 Action 没被下级部门接，则判定为“严重：管理真空”。
    - **语法要求**：目标名必须是全局态的静态事实（如：`XX业务板块收入已达成预算指标`）。
- **个人 BP**：
    - **审计重心**：个人岗位价值对齐与因果自洽。
    - **必检项**：Action 承接关系必须严密。个人目标必须对齐上级组织的 Action（或上级领导的 Action）。
    - **语法要求**：目标名必须是个人承诺态的静态事实（如：`本人负责的XX产品市场份额已达成既定目标`）。

### 承接逻辑（向上对齐）

- **核心规则**：下级用「目标」承接上级的「关键成果/关键举措」
- **康哲使用约束**：只使用「下级目标对齐上级关键举措」这一种承接方式（唯一特例：中心承接集团时，由于集团无关键举措，中心目标直接承接集团关键成果）
- **对齐路径**：
  - 集团关键成果 → 中心目标（成果→目标，唯一特例）
  - 中心关键举措 → 一级部门目标（举措→目标）
  - 一级部门关键举措 → 核心员工目标（举措→目标）
- **人员对应**：
  - 承接上级关键举措时：下级目标责任人 = 上级关键举措承接人
  - 承接上级关键成果时：下级目标责任人应与该成果在下级的实际负责方一致

## 二、结构完整性审计

### 2.1 层级匹配

- 集团层（顶层 org）：只能有 2 层（目标 → 关键成果），出现关键举措则**不合规**
- 中心/部门/员工层：必须完整 3 层（目标 → 关键成果 → 关键举措），缺少任何一层都**不合规**

**审计方法**：
1. 获取分组树，确定当前分组的层级（集团/中心/部门/员工）
2. 获取任务树，检查每个目标下是否有 KR，每个 KR 下是否有 KI
3. 集团层出现 KI → 标记为"严重：集团层不应有关键举措"
4. 非集团层缺少 KR 或 KI → 标记为"严重：结构不完整"

### 2.2 结构闭环

- 每个目标下必须有至少一个关键成果
- 每个关键成果下必须有至少一个关键举措（集团层除外）
- 存在"空壳"节点（目标下无 KR，或 KR 下无举措）视为**结构不闭环**

### 2.3 举措位置检查

- 举措必须挂在对应成果下，不允许写在目标层
- 目标层只放"终态"，举措必须与某条成果形成清晰支撑关系

### 2.4 数量合理性

- 目标下 KR 超过 5 条 → 标记为"一般：KR 数量偏多，建议合并聚焦"
- KR 下举措超过 6 条 → 标记为"一般：举措数量偏多，建议合并聚焦"
- 目标下仅 1 条 KR → 标记为"轻微：KR 单一，建议检查是否充分覆盖目标"

## 三、内容质量审计

### 3.1 描述具体性

检查目标、KR、KI 的描述是否具体：
- 是否包含明确的业务对象
- 是否有可量化的指标或可验证的交付物
- 是否有明确的时间范围

**不合格信号词**：
- "相关工作"、"各项任务"、"有关事项" → 过于笼统
- "进一步"、"不断"、"持续" → 无明确终点
- "等"、"若干"、"部分" → 范围不明

### 3.2 可衡量性

- 目标：是否描述了可观察的结果状态
- KR：是否有明确的衡量标准（数值/里程碑/交付物）
- KI：是否有明确的完成标志

### 3.3 行动指向性

- KI 是否指向具体的行动（而非愿望或状态）
- KI 完成后是否能产生可观察的产出物

## 四、逻辑自洽性审计

### 4.1 KI → KR 推导

检查每条 KR 下的所有 KI，判断：
- 如果所有 KI 都完成了，KR 是否能达成？
- 是否存在"做了但不够"的情况（KI 覆盖不足）
- 是否存在"做了但没用"的情况（KI 与 KR 无关）

**审计方法**：
1. 逐条 KR 分析其下所有 KI
2. 判断 KI 集合是否构成 KR 达成的充分条件
3. 识别缺失的关键行动（KR 需要但 KI 未覆盖的）
4. 识别多余的行动（KI 存在但与 KR 无关的）

### 4.2 KR → Goal 推导

检查每个目标下的所有 KR，判断：
- 如果所有 KR 都达成了，目标是否能实现？
- KR 之间是否满足 MECE（相互独立、共同穷尽）
- 是否存在目标的某个维度未被任何 KR 覆盖

### 4.3 常见逻辑断裂模式

| 模式 | 说明 | 严重等级 |
|------|------|---------|
| KI 空转 | KI 做了很多事但对 KR 达成无直接贡献 | 严重 |
| KR 悬空 | KR 下无 KI 或 KI 不足以支撑 KR | 严重 |
| 目标虚化 | 目标过于宏大，KR 无法覆盖目标全貌 | 一般 |
| 层级错配 | 把 KR 的内容写成了 KI，或把 KI 的内容写成了 KR | 一般 |

## 五、衡量标准审计（SMART + 最小完备要求）

### 5.1 属性归属

- 衡量标准必须是关键成果的属性，**严禁赋予独立编码**
- 衡量标准不应该作为独立的任务节点存在

### 5.2 SMART 原则

| 维度 | 要求 | 不合格示例 |
|------|------|-----------|
| S（Specific） | 具体明确 | "提升"、"优化"、"改善" |
| M（Measurable） | 可量化，有明确数值或可验证交付物 | "做好XX工作" |
| A（Achievable） | 可实现 | 不切实际的目标 |
| R（Relevant） | 与目标相关 | 衡量的不是该 KR 要达成的事 |
| T（Time-bound） | 有时限 | 无截止日期或里程碑 |

### 5.3 衡量标准最小完备要求（成果必填项）

对每条成果，衡量标准至少应包含：

| 要素 | 说明 | 缺失时的审计结论 |
|------|------|----------------|
| 指标（Metric） | 衡量什么 | 严重：无法确定衡量对象 |
| 阈值（Target） | 达到什么数值/等级算达成 | 严重：无法判断是否达成 |
| 口径（Definition） | 计算方式/范围/排除项 | 一般：可能导致理解歧义 |
| 数据源（Source） | 从哪个系统/流程取数 | 一般：可能导致取数困难 |
| 频率（Frequency） | 统计与更新频率 | 轻微：不影响达成判断 |

**审计规则**：
- 若成果出现"明显/大幅/持续/优化/提升"等模糊词，必须标记为"严重：衡量标准模糊"
- 若无数据源与口径，标记为"一般：不可验收成果"
- 最终判定标准：**"验收能否发放奖金"**——如果衡量标准无法支撑奖金发放决策，说明标准不够清晰

### 5.4 验收闭环

- 判定标准是否模糊？若标准不清导致无法评价完成度，视为**严重**漏洞

## 六、汇报周期审计

### 6.1 自下而上原则

必须满足：**举措周期 ≤ 关键成果周期 ≤ 目标周期**

### 6.2 时间边界约束

- 子任务时间必须在父任务时间范围内
- 举措/成果起止时间不应超出其所属成果/目标的起止范围

### 6.3 周期字段说明

Markdown 输出中，周期呈现为 `周期：{ruleType}+{index}` 的格式（如 `month+20` 表示每月20号、`week+3` 表示每周三），位于 `状态：xx | 周期：xx | 时间：xx` 行中。

周期类型取值：`weekday`、`week`、`month`、`doubleWeek`(两周)、`doubleMonth`(两月)、`threeMonth`
- 周期大小排序：`weekday` < `week` < `doubleWeek` < `month` < `doubleMonth` < `threeMonth`

### 6.4 审计方法

1. 解析 `周期` 字段（Markdown 中格式：`{ruleType}+{index}`，如 `month+15`）
2. 比较父子任务的周期大小
3. 解析 `时间` 字段（Markdown 中格式：`yyyy-MM-dd ~ yyyy-MM-dd`，如 `2026-01-01 ~ 2026-12-31`）
4. 检查子任务时间是否在父任务范围内

### 6.5 错误示例

- 举措每月汇报，而 KR 要求每周汇报 → **一般：上级拿不到下级数据**
- KR 每月汇报，但目标每周汇报 → **一般：数据频率不匹配**
- 举措时间超出 KR 时间范围 → **一般：时间边界越界**

## 七、关键举措颗粒度审计

### 7.1 可执行性

- 举措必须是具体的动作，而非笼统描述
- **合格**："调研 100 家医院的使用情况"、"完成 3 个场景的方案包设计"
- **不合格**："优化流程"、"提升效率"、"推进项目"、"积极配合"、"持续跟进"、"加强沟通"

### 7.2 可承接性

- 颗粒度需细化到能明确责任人
- 方便下级将其转化为自己的目标（静态事实定格句式）
- 举措过于宏大、下级无法直接承接为目标，说明颗粒度不够

### 7.3 常见违规

| 违规类型 | 说明 | 严重等级 |
|---------|------|---------|
| 结果冒充举措 | 把结果写成举措（如"达成收入5亿""实现市场份额15%"） | 一般 |
| 不可追踪举措 | 不可追踪的举措（如"积极配合""持续跟进""加强沟通"） | 一般 |
| 颗粒度过大 | 举措过于宏大，无法直接执行或承接 | 一般 |
| 颗粒度过小 | 举措过于琐碎，应合并为更有意义的行动 | 轻微 |

## 八、语义与定义审计

### 8.1 结果 vs 过程

- **目标**必须描述"结果状态"，严格禁动词，使用静态事实定格句式
- **关键成果**必须描述"可验收的达成状态"
- **关键举措**必须描述"具体行动"

### 8.2 红线（不合格即标记）

| 检查项 | 不合格表现 | 严重等级 |
|--------|----------|---------|
| 目标含动词 | "推动..."、"建立..."、"提升..."、"加强..."、"推进..."、"完成..."、"落地..."、"实施..." | 一般 |
| KR 含动作 | "完成...工作"、"开展...活动" | 一般 |
| KR 含模糊词 | "客户满意度明显提升"、"业务持续增长" | 严重（不可验收） |
| 举措含愿望 | "流程变规范"、"效率提升" | 一般 |

### 8.3 合规写法参考

**目标**：
- "XX机制在全范围内按标准运行并可被审计"
- "XX交付能力达到约定SLA并稳定运行"

**关键成果**：
- "NPS达到X（问卷口径，季度统计）"
- "P0/P1问题7日内闭环率≥Y%（工单系统，月度统计）"

**关键举措**：
- "建立月度经营分析会机制并形成复盘纪要（固定模板、固定节奏）"
- "完成XX方案评审→试点→推广三阶段交付并留痕归档"

### 8.4 判断方法

- 目标/KR 应该能回答"达成后是什么样子？"
- 举措应该能回答"具体要做什么事？"

## 九、必填项审计（最小可运行集）

| 检查项 | 说明 | 严重等级 |
|--------|------|---------|
| 责任人/承接人 | 是否填写（与层级角色一致） | 严重 |
| 起止时间 | 是否填写 | 严重 |
| 汇报周期 | 是否设置 | 一般 |
| 衡量标准 | KR 是否配置衡量标准 | 严重 |

## 十、审计输出精确性规范（强制 — v3.4 标准版）

**所有审计结论必须精确到具体对象（编码+名称），必须包含修改建议（Action/Location），且必须包含建议权重（Initial Weight）。**

### 输出格式要求 (Object / Weight / Status / Why / How)

所有审计报告详情必须采用以下五列表格格式：

1. **Object ID / Name**：任务编码 (fullLevelNumber) + 任务名称。
2. **Weight**：建议权重（即便系统为空也需建议）。
3. **Status**：🔴 (严重/必改) | 🟡 (一般/建议) | 🟢 (通过)。
4. **Why**：具体审计结论（涵盖内容质量、对齐、承接、数值等）。
5. **How**：修改位置与具体建议（Action / Location）。

### 反面示例（严禁）

| 错误写法 | 正确写法 |
|---------|---------|
| "部分KR描述不够具体" | "KR A8B8-1.2「年度利润」标准缺少口径" |
| "某些举措缺少承接人" | "KI A8B8-1.1.3「渠道拓展」承接人为空" |
| "目标表述不规范" | "目标 G-1「推动XX中心业绩增长」含动词'推动'" |
| "数值覆盖不足" | "年度收入 1.5亿，下级汇总 1.25亿，缺口 2500万" |
| "权重未设置" | (必须在表格 Weight 列给出建议值，并在 Why 列标注缺失) |

## 十一、审计清单（康哲必检项）

### 结构完整性
- [ ] 是否按"目标→关键成果→关键举措"拆解，且拆解深度符合所在层级要求（集团2层，其余3层）
- [ ] 是否把举措写在目标层（禁止）
- [ ] 目标下是否有关键成果支撑；关键成果下是否有关键举措落地（集团层除外）
- [ ] 数量是否在合理范围内（KR 2-5条，KI 2-6条）

### 内容质量
- [ ] 描述是否具体、可衡量、有明确行动指向
- [ ] 是否存在笼统描述（"相关工作"、"各项任务"等）
- [ ] 逻辑是否自洽（KI 能推导 KR，KR 能验证 Goal）

### 衡量标准
- [ ] 所有关键成果是否都配置了衡量标准
- [ ] 衡量标准是否包含"指标+阈值+口径/数据源+统计频率"
- [ ] 是否把衡量标准错误编码/拆成了多个举措（禁止）
- [ ] 是否存在模糊词（"明显/大幅/持续/优化/提升"）

### 举措颗粒度
- [ ] 举措是否具体到"可执行动作"，能明确责任与跟踪节奏
- [ ] 举措是否足以让下级直接转写为"静态事实定格"的目标终态（可承接性）
- [ ] 是否存在不可追踪举措（"积极配合""持续跟进"等）

### 汇报周期与时间边界
- [ ] 汇报周期是否满足"自下而上汇报"的基本约束：举措周期 ≤ KR周期 ≤ 目标周期
- [ ] 子任务时间是否在父任务时间范围内

### 语义定义
- [ ] 目标是否使用静态事实定格句式（禁动词）
- [ ] KR 是否描述可验收状态
- [ ] KI 是否描述具体行动

### 必填项
- [ ] 责任人/承接人是否填写（与层级角色一致）
- [ ] 起止时间是否填写
- [ ] 汇报周期是否设置
