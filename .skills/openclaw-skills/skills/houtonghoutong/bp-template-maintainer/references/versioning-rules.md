# 版本规则

母版维护采用“两层版本 + 本地目录”的管理方式。

## 一、版本对象

版本分两层管理：

1. **整套母版基线版本**
   - 指四个模板作为一个体系一起升级
   - 版本形态：`v1`、`v2`、`v3`
2. **单模板补丁版本**
   - 指只改某一个周期模板，但不改变整套共同规则
   - 版本形态：`v1.1`、`v1.2`、`v2.1`

当前官方基线：

- 月报：`P001-T001-MONTH-TPL-01_月报模板_v1.md`
- 季报：`P001-T001-QUARTER-TPL-01_季报模板_v1.md`
- 半年报：`P001-T001-HALFYEAR-TPL-01_半年报模板_v1.md`
- 年报：`P001-T001-YEAR-TPL-01_年报模板_v1.md`

这意味着当前整套母版的共同基线是 `v1`。

## 二、版本存放位置

版本管理默认放在当前工作区本地，不放云端。

### 目录规则

- 当前正在使用的最新模板：放在工作区根目录
- 历史版本归档：放在 `版本管理/归档/`
- 版本索引：放在 `版本管理/版本索引.md`

也就是说：

- 根目录保存“当前生效版本”
- `版本管理/` 保存“历史版本和版本记录”

如果后续接入 Git，Git 只是额外审计能力，不替代这套本地目录。

### 技能自带基线的位置

这个 skill 自带一套安装基线，放在：

- `assets/基线母版/v1/`

它的作用是：

- 让别人安装 skill 时自动带上第一版母版
- 作为新工作区初始化母版的来源
- 作为母版体系的可分发起点

它不直接等于目标工作区的“当前生效版本”。目标工作区一旦开始运行，版本演进在目标工作区本地管理，不回写到 skill 的 `assets/` 中，除非用户明确要求同步更新 skill 自带基线。

## 三、什么时候升整套基线版本

如果改动影响整套母版共同契约，就把整套版本从 `v1` 升到 `v2`。

典型触发条件：

- 共同 8 模块骨架变了
- 共同 BP 锚点变了
- 组织 BP 映射关系变了
- 四个周期共享的固定字段或固定提示变了
- BP 系统语义变了，整套母版都要随之重构
- 团队明确要求形成新一代母版

### 操作方式

1. 在根目录为四个模板一起生成新版本文件
2. 新文件版本号统一升到新基线，例如 `v2`
3. 同步修改文件内元数据：
   - `版本`
   - `更新时间`
4. 将旧版本文件复制归档到 `版本管理/归档/<版本号>/`
5. 更新 `版本管理/版本索引.md`

### 示例

- 旧：`P001-T001-MONTH-TPL-01_月报模板_v1.md`
- 新：`P001-T001-MONTH-TPL-01_月报模板_v2.md`

其他三个模板同步升到 `v2`。

## 四、什么时候只升单模板补丁版本

如果改动只影响某一个周期模板自己的表达方式，不改变整套共同规则，就只升该模板的补丁版本。

典型触发条件：

- 月报字段提示不清，需要优化
- 年报增加一个年终复盘提示，但不影响其他周期
- 半年报某个模块文案要更适配中程校准

### 操作方式

1. 只为受影响的那个模板生成新文件
2. 版本号升为补丁版本，例如 `v1.1`
3. 同步修改文件内元数据：
   - `版本`
   - `更新时间`
4. 将旧文件复制归档到 `版本管理/归档/<模板类型>/`
5. 更新 `版本管理/版本索引.md`

### 示例

- 旧：`P001-T001-MONTH-TPL-01_月报模板_v1.md`
- 新：`P001-T001-MONTH-TPL-01_月报模板_v1.1.md`

其他三个模板不动。

## 五、判断规则

按这个顺序判断：

1. 改动是否改变四个模板共同契约
2. 如果是，升整套基线版本
3. 如果不是，判断是否只是单一周期的局部表达优化
4. 如果是，只升单模板补丁版本
5. 如果影响两到三个模板，但不影响全部四个模板：
   - 若会破坏跨周期可比性，默认升整套版本
   - 若只是局部周期差异增强，可分别走补丁版本

默认口径：

- 共同结构变化 -> 升整套版本
- 单周期表达优化 -> 升单模板补丁版本

## 六、命名规则

版本号只出现在两个地方：

- 文件名后缀
- 文件内 `版本` 字段

正确示例：

- `P001-T001-MONTH-TPL-01_月报模板_v1.md`
- `P001-T001-MONTH-TPL-01_月报模板_v1.1.md`
- `P001-T001-MONTH-TPL-01_月报模板_v2.md`

禁止使用：

- `final`
- `new`
- `最新版`
- `修订版`
- `copy`

统一使用明确版本号。

## 七、元数据同步规则

生成新版本文件时，至少同步以下字段：

- `文档标题`
- `文档编号`
- `文档状态`
- `版本`
- `更新时间`
- `标签`，如果团队有版本标签实践

最低要求：

- 文件名版本号 = 文件内 `版本`
- `更新时间` = 实际编辑日期

## 八、禁止原地覆盖

创建新版本时，不要直接覆盖旧文件。

保留旧版本的原因：

- 便于回滚
- 便于对比差异
- 便于审计版本演进
- 便于后续 skill 仍可引用旧基线

## 九、和 BPMAP 的联动规则

- 如果整套升版且改了 BP 锚点设计或模块挂位，通常要考虑同步 `BPMAP-01`
- 如果整套升版且改了质控口径，通常要考虑同步 `BPMAP-03`
- 如果只是单模板补丁，不一定需要改 BPMAP
- `BPMAP-02` 只有在原始设计立场本身变化时才需要升版

## 十、本 skill 输出要求

这个 skill 在完成一次版本维护后，输出里必须说清楚：

- 这是整套升版还是单模板补丁
- 当前最新生效文件是哪几个
- 哪些旧文件已归档
- 是否需要同步更新 BPMAP

如果这次任务还涉及“分发给别人安装”，还要额外说明：

- 是否同步更新了 `assets/基线母版/v1/`
- skill 自带基线和当前工作区运行版本是否仍一致
