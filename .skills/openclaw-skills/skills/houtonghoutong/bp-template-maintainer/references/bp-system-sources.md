# BP系统资料使用说明

当母版需要和真实 BP 系统对齐时，使用这份资料。

## 一、资料链接

- API说明：[BP系统API说明](https://github.com/xgjk/dev-guide/blob/main/02.%E4%BA%A7%E5%93%81%E4%B8%9A%E5%8A%A1AI%E6%96%87%E6%A1%A3/BP/BP%E7%B3%BB%E7%BB%9FAPI%E8%AF%B4%E6%98%8E.md)
- 业务说明：[BP系统业务说明](https://github.com/xgjk/dev-guide/blob/main/02.%E4%BA%A7%E5%93%81%E4%B8%9A%E5%8A%A1AI%E6%96%87%E6%A1%A3/BP/BP%E7%B3%BB%E7%BB%9F%E4%B8%9A%E5%8A%A1%E8%AF%B4%E6%98%8E.md)

## 二、在本 skill 中的优先级

对母版维护来说，这两类资料的使用优先级是：

1. 当前四个周期模板
2. BP系统业务说明
3. BP系统API说明
4. BPMAP 文件

原因如下：

- 周期模板是当前已落地的母版现实
- 业务说明定义了母版不能违背的 BP 语义规则
- API 说明告诉我们系统真实有哪些字段和结构
- BPMAP 是本地设计资料，但在系统语义发生变化时，不应优先于 BP 系统资料

## 三、使用边界

BP 系统资料在这个 skill 中默认只用来回答一类问题：

- 母版的语义、字段提示、结构设计，是否已经明显偏离当前 BP 体系

默认不要拿 BP 系统资料去回答下面这些问题：

- 这套母版有没有覆盖某个组织当前全部真实 BP
- 母版里的章节数量是否必须等于系统里的目标数量
- 母版中的占位编号是否必须等于系统中的任务 ID
- 某个组织当前有 4 个/18 个/22 个目标，所以母版一定错误

这些判断属于“专属模板适配”或“填写规范生成”范围，不属于通用母版维护。

## 四、业务说明主要解决什么问题

以下问题优先看业务说明：

- 目标 / 成果 / 举措边界是否变了
- 组织 BP 和个人 BP 的边界是否变了
- 各层级拆解深度是否变了
- 衡量标准的最小完备要求是什么
- 承接规则是否调整了
- 时间边界、汇报周期、责任边界有没有新约束

也就是说，凡是“母版该不该这么写”的问题，优先看业务说明。

## 五、API 说明主要解决什么问题

以下问题优先看 API 说明：

- 系统当前有哪些稳定字段
- 系统当前真实的 Goal / Key Result / Action 结构是什么
- 系统能否直接返回完整 Markdown
- 上下对齐关系能不能通过接口直接拿到
- 哪些字段已经稳定到值得固化进母版提示

对这个 skill 来说，最有价值的接口是：

- `GET /bp/period/getAllPeriod`
  - 用于确认周期名称和启用周期
- `GET /bp/group/getTree`
  - 用于看组织节点和个人节点结构
- `POST /bp/group/getPersonalGroupIds`
  - 用于从员工跳转到个人 BP 分组
- `GET /bp/task/v2/getSimpleTree`
  - 用于看真实的目标 -> 成果 -> 举措树
- `GET /bp/task/v2/getGoalAndKeyResult`
  - 用于看目标、成果、举措及对齐字段
- `GET /bp/task/v2/getKeyResult`
  - 用于确认 `measureStandard` 等字段
- `GET /bp/document/getBpMarkdown`
  - 这是最适合母版维护的读取接口，可一次拿到完整 BP Markdown、上对齐、下承接

## 六、值得和母版对照的稳定字段

当判断母版要不要改时，优先检查这些系统字段：

- `name`
- `type`
- `measureStandard`
- `statusDesc`
- `reportCycle`
- `planDateRange`
- `taskUsers`
- `taskDepts`
- `upwardTaskList`
- `downTaskList`
- `fullLevelNumber`

这些字段不是母版本身，但能帮助判断母版里的固定提示是否还贴合系统现实。

## 七、什么情况下要因为 BP 系统资料去改母版

只有满足以下任一条件时，才因为 BP 系统资料去修改母版：

1. BP 语义规则变了，原模板提示已经误导使用者
2. 系统已经稳定暴露了某些字段或关系，值得沉淀为母版固定提示
3. 业务运行中反复出现的问题，本质上是“母版结构”和“系统现实”不一致

不要因为 API 字段很多，就把字段都塞进母版。只有能提升重复使用质量、能降低结构性误填的字段，才值得进母版。

额外限制：

- 不要把系统中的真实目标总数、真实目标名称、真实任务 ID 直接当成母版错误依据
- 如果用户没有明确要求“做专属化”，不要把母版评判成“覆盖不全”
