# 论文分析报告标准 - Paper Analysis Report Standards

本文档详细定义了SubAgent生成论文分析报告的标准和要求。

SubAgent必须遵循本文档生成6部分分析报告，确保报告质量、完整性和实用性。

---

## 核心原则

1. **完整性**：必须阅读PDF的每一个章节，不能只看摘要
2. **真实性**：所有数据必须来自论文原文，严禁编造
3. **可追溯性**：所有引用必须标注原文位置 [Section X.Y]
4. **实用性**：报告内容要对用户的研究有直接帮助

---

## 报告结构（6个主要部分）

### Part 1: 研究背景 (Research Background)

**目标**：帮助用户快速理解该论文所处的研究领域和技术脉络

**可能包含的子部分**（提示，非强制）：

- **领域概述**：
  - 该领域研究什么核心问题？
  - 有哪些主要的技术路线？
  - 该领域的发展历程（早期 → 近期）

- **前人研究基础**：
  - 该论文引用了哪些关键前作？（至少列出3-5篇，带具体标题）
  - 这些前作解决了什么问题？
  - 这些前作留下了什么未解决的问题？
  - 技术演进脉络：从Method A → Method B → This Paper

- **已有实验基础**：
  - 该领域常用的数据集有哪些？
  - 标准的评估指标是什么？
  - 主流的Baseline方法是什么？
  - 该领域公认的挑战是什么？

- **研究时机**：
  - 为什么在这个时间点发表这篇论文？
  - 该领域当时面临什么瓶颈？

**引用要求**：
- 必须引用具体的Section和Paragraph
- 示例："根据[Introduction, Paragraph 3]，该领域的主要挑战是..."

---

### Part 2: 研究问题 (Research Problem)

**目标**：让用户明白作者到底在解决什么具体问题，以及为什么这个问题值得解决

**可能包含的子部分**（提示，非强制）：

- **核心问题定义**：
  - 作者明确声明要解决什么问题？（引用原话）
  - 这个问题在实际应用中的具体表现是什么？
  - 如果不解决这个问题，会有什么后果？

- **现有方法分析**：
  - 现有方法是什么？（技术路线概述）
  - 现有方法的具体局限性有哪些？（逐条列出，引用原文）
  - 这些局限性是实验发现的还是理论分析的？
  - 现有方法在哪些场景下会失败？（具体例子）

- **问题识别洞察**：
  - 作者是如何发现这个问题的？（实验观察/理论分析/用户反馈）
  - 这个问题被忽视的原因是什么？
  - 解决这个问题的关键难点是什么？

- **解决思路预览**：
  - 作者提出的核心假设是什么？
  - 作者的核心洞察是什么？（可能只有一句话，但很关键）
  - 为什么这个思路可能有效？

**引用要求**：
- 必须引用Introduction中的Motivation部分
- 必须引用Related Work中对该问题的讨论
- 示例："作者指出[Section 1.2]，现有方法在X场景下会失败，原因是..."

---

### Part 3: 核心创新工作 (Core Innovation)

**目标**：详细理解作者具体做了什么，技术细节是什么，真正的创新点在哪里

**可能包含的子部分**（提示，非强制）：

- **方法概述**：
  - 一句话概括核心方法
  - 系统整体架构（可以用文字描述流程图）
  - 输入是什么？输出是什么？
  - 主要包含哪些模块？

- **技术细节**（关键！）：
  - 网络结构：层数、每层的维度、激活函数、连接方式
  - 关键公式：损失函数、目标函数、约束条件（用LaTeX格式）
  - 算法流程：训练流程、推理流程（可以用伪代码描述）
  - 超参数设置：学习率、Batch size、优化器等
  - 特殊技术：正则化、归一化、数据增强等

- **模块详细说明**（如果有多个模块）：
  - 每个模块的功能
  - 每个模块的输入输出
  - 模块之间的交互
  - 为什么需要这个模块？

- **创新点对比**：
  | 维度 | 前人方法 | 本文方法 | 优势 |
  |------|---------|---------|------|
  | 输入表示 | ... | ... | ... |
  | 网络结构 | ... | ... | ... |
  | 损失函数 | ... | ... | ... |
  | 训练策略 | ... | ... | ... |

- **真正的创新点**（关键！）：
  - 什么是前人绝对没做过的？（可以是组合创新，但要说清楚组合的新意）
  - 这个创新点为什么重要？
  - 没有这个创新点，性能会差多少？（如果有消融实验）

**引用要求**：
- 必须引用Methods章节的具体Section
- 公式必须标注编号 [Equation X]
- 示例："网络结构如[Section 3.2, Figure 2]所示，包含..."

---

### Part 4: 关键实验设计 (Experimental Design)

**目标**：提取所有实验细节和真实数据，让用户可以复现或对比

**可能包含的子部分**（提示，非强制）：

- **实验设置**：
  - 数据集详情：
    - 名称、版本
    - 数据规模（训练/测试样本数）
    - 数据特点（真实/合成、领域、难度）
    - 获取方式（公开/自建）
  
  - 评估指标：
    - 每个指标的全称和缩写
    - 指标的计算公式
    - 指标的物理意义
    - 为什么选这些指标？

- **Baseline方法**：
  - 对比了哪些Baseline？（列出具体方法名称）
  - 这些Baseline来自哪些论文？
  - 这些Baseline的参数设置？
  - 为什么选这些Baseline？（代表性、SOTA、公平性）

- **主实验结果**（关键！必须是真实数据）：
  - 结果表格（复制原文表格）：
    | 方法 | 指标1 | 指标2 | 指标3 |
    |------|-------|-------|-------|
    | Ours | X.XX | X.XX | X.XX |
    | Baseline1 | X.XX | X.XX | X.XX |
    | Baseline2 | X.XX | X.XX | X.XX |
  
  - 性能提升幅度：相对提升X%，绝对提升Y个点
  - 统计显著性：如果有p-value，必须记录

- **消融实验**（关键！理解什么设计有效）：
  - 消融了哪些组件？
  - 每个组件的贡献：
    | 配置 | 指标1 | 指标2 |
    |------|-------|-------|
    | Full Model | X.XX | X.XX |
    | w/o Component A | X.XX | X.XX |
    | w/o Component B | X.XX | X.XX |
  
  - 关键发现：去掉哪个组件性能下降最多？

- **额外实验**：
  - 不同场景下的性能（如果论文有）
  - 计算效率对比（速度、参数量、FLOPs）
  - 可视化结果（如果有，描述关键发现）
  - 失败案例分析（如果有）

- **实验结论**：
  - 实验验证了哪些假设？
  - 实验发现了什么意外现象？
  - 实验有什么局限性？

**引用要求**：
- 所有数据必须来自原文表格 [Table X]
- 实验设置引用 [Section 4.1]
- 结果分析引用 [Section 4.2]

---

### Part 5: 核心洞察与结论 (Key Insights)

**目标**：提炼作者通过实验得出的深层结论，指导用户的实践

**可能包含的子部分**（提示，非强制）：

- **主要发现**：
  - 论文最核心的3个发现是什么？
  - 每个发现有什么实验证据支持？
  - 这些发现是否 surprising？

- **领域洞察**：
  - 什么方法/设计是有效的？为什么？
  - 什么方法/设计是无效的？为什么？
  - 有没有反直觉的发现？
  - 性能瓶颈在哪里？

- **设计原则**：
  - 基于实验，作者提出了什么设计原则？
  - 这些原则可以迁移到其他问题吗？
  - 如何权衡不同设计选择？（精度 vs 速度，复杂 vs 简单等）

- **实践建议**：
  - 如果用户要做类似研究，应该：
    - 先尝试什么Baseline？
    - 重点关注什么组件？
    - 避免什么坑？
  - 该方法的适用场景是什么？
  - 该方法的局限性场景是什么？

- **理论贡献**（如果有）：
  - 论文提出了什么理论分析？
  - 证明了什么定理？
  - 理论对实践有什么指导？

**引用要求**：
- 引用Discussion或Conclusion部分
- 引用实验中的关键发现
- 示例："作者发现[Section 5]，X因素对性能的影响比预期更大..."

---

### Part 6: 待解决问题与未来方向 (Future Work)

**目标**：帮助用户找到潜在的研究Gap和创新点

**可能包含的子部分**（提示，非强制）：

- **本文局限性**：
  - 作者自己承认的局限性有哪些？（引用原话）
  - 方法在哪些假设下才有效？
  - 实验在哪些条件下进行？（可能不泛化）

- **未解决的技术挑战**：
  - 方法还有什么明显的缺陷？
  - 还有什么 corner case 没处理好？
  - 计算成本还有什么优化空间？
  - 理论分析还有什么不完善？

- **潜在改进方向**（关键！给用户创新灵感）：
  - 方法层面：可以结合什么新技术？
  - 应用层面：可以拓展到什么新场景？
  - 理论层面：可以加强什么分析？
  - 至少列出3个具体方向，并说明：
    - 为什么这个方向有价值？
    - 初步思路是什么？
    - 可能面临什么挑战？

- **长期研究趋势**：
  - 该领域可能向什么方向发展？
  - 什么技术可能成为下一个SOTA？
  - 什么应用可能成为热点？

**引用要求**：
- 引用Conclusion或Future Work部分
- 引用Limitations部分（如果有）
- 示例："作者指出[Section 6]，未来可以探索X方向，因为..."

## 质量标准检查清单

SubAgent在完成报告前，必须逐一检查：

### 阅读完整性
- [ ] 我读了Introduction的每一个句子
- [ ] 我读了Related Work的每一个引用
- [ ] 我读了Methods的每一个公式
- [ ] 我读了Experiments的每一个表格
- [ ] 我读了Conclusion的每一个结论

### 数据真实性
- [ ] 所有表格数据都来自原文，不是我编的
- [ ] 所有数字都能在原文找到对应
- [ ] 不确定的信息我标注了"原文未明确提及"

### 引用准确性
- [ ] 每个重要陈述都有引用
- [ ] 引用格式正确：[Section X.Y] 或 [Table N] 或 [Equation M]
- [ ] 引用的位置是准确的

### 报告完整性
- [ ] 6个部分都写了
- [ ] 每个部分都有实质性内容（不是空话）
- [ ] 总字数超过3000字
- [ ] 至少包含3个数据表格
- [ ] 至少包含10个原文引用

## 使用Paper-Reader工具

### 如何读取PDF

```bash
# 使用paper-reader工具提取完整PDF内容
python3 ~/.openclaw/skills/paper-reader/read_paper.py "{pdf_path}" --full

# 输出包括：
# - 完整文本（按章节组织）
# - 所有表格内容
# - 图片和图表
# - 元数据
```

### 阅读策略

1. **先读结构**：看章节标题，了解论文组织
2. **再读摘要**：了解整体贡献
3. **细读Introduction**：理解研究动机和问题
4. **精读Methods**：提取技术细节
5. **详读Experiments**：提取所有表格数据
6. **总结Conclusion**：提取核心结论

### 注意事项

- paper-reader可能无法完美提取所有表格，如果表格不清楚，要说明"表格数据提取不完整"
- 公式可能以图片形式存在，如果无法识别，要说明"公式以图片形式存在"
- 如果有附录，也要检查附录中的重要信息

## 特别提醒

### 关于完整性

不要假设"这部分和别的论文差不多"。**每篇论文都是独特的**，必须完整阅读。

### 关于真实性

如果你不确定某个数据，**宁可不写，也不要编**。可以写"原文未明确提及此数据"。

### 关于引用

用户需要验证你的分析，所以**必须提供可追溯的引用**。没有引用的分析无法验证。

### 关于实用性

记住：**用户读这份报告是为了做研究**。报告内容要对用户有实际帮助，而不是泛泛而谈。
