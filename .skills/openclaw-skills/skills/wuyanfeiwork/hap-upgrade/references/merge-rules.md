# HAP 升级附加操作合并规则

本文件提供的是**附加操作合并原则和典型示例**，不是操作类型白名单。

使用时遵循以下原则：

- 实际有哪些附加操作，必须以本次升级路径中抓取到的**线上升级说明页原文**为准
- 本文件中未列出的操作类型，只要在线上升级说明页中出现，也必须纳入分析和文档生成
- 遇到新操作类型时，先判断它属于“升级前操作”还是“升级后操作”，再按照其执行时机、是否可合并、是否需要保留顺序来整理
- 如果某类操作在线上文档中明确要求“仅执行最新版本一次”“按版本顺序逐条执行”“需合并为一次修改”，必须优先遵循线上文档原意
- 本文件中的表格和示例用于帮助模型理解常见合并模式，不能据此否定线上文档实际出现的其他操作

跨多个含附加操作的版本时，必须按照以下原则将所有版本的附加操作合并为一次执行或整理为有序步骤。

## 1. HAP 微服务升级前操作合并原则与示例

| 示例操作类型 | 合并方式 |
|----------|----------|
| **创建 MongoDB 数据库** | 合并为一次登录，一次性创建所有需要的库 |
| **存储组件升级（仅单机）** | 直接升级到所有跨越版本中要求的**最高版本** |
| **MongoDB 预置数据更新** | 仅执行跨越路径（含目标版本）中，**包含此操作的最新版本**的命令，无需执行中间版本或不含此操作的版本 |
| **微服务 yaml 新增服务配置（集群）** | 合并所有版本需新增的服务，一并写入 yaml |

若线上文档出现其他升级前操作，按以下思路处理：

| 判断维度 | 处理原则 |
|----------|----------|
| **是否可以合并为一次环境进入 / 一次文件修改 / 一次配置变更** | 能合并则合并，减少重复进入容器、重复编辑文件或重复重启 |
| **是否要求使用最新版本对应资源** | 若线上文档明确只需执行最新版本对应资源，则不要重复执行中间版本的同类步骤 |
| **是否依赖原版本运行状态** | 若线上文档要求在原版本服务运行状态下执行，必须放在 HAP 微服务升级前 |

## 2. HAP 微服务升级后操作合并原则与示例

| 示例操作类型 | 合并方式 |
|----------|----------|
| **进入容器执行 MySQL DDL（单机）** | 一次进入容器，按版本从低到高顺序执行所有版本的 DDL |
| **进入 config Pod 执行脚本（集群）** | 一次进入 Pod，按版本从低到高顺序执行所有版本的命令 |

若线上文档出现其他升级后操作，按以下思路处理：

| 判断维度 | 处理原则 |
|----------|----------|
| **是否必须保留版本顺序** | 只要脚本或 DDL 具备明显版本递进关系，就按版本从低到高排列 |
| **是否可以在一次进入容器 / Pod 后连续执行** | 能合并执行环境进入动作时，不要拆成多次进入 |
| **是否存在多种脚本类型** | 按线上文档实际分类保留，例如 MySQL DDL、MongoDB DDL、文件初始化等，不要混写成单一类型 |

## 3. 合并示例

**场景：从 v6.4.0 跨越到 v7.2.0（跨过 v6.5.0、v7.0.0、v7.1.0、v7.2.0，均含附加操作）**

*   **升级前合并**：镜像命名替换执行一次（v7.1.0）、MongoDB 库一次性全部创建、MongoDB 预置数据更新至含有该操作的最新版本（如路径中只有 v7.0.0 有此操作，则更新至 v7.0.0）、yaml 新增服务合并写入。
*   **升级后合并（集群）**：一次进入 config Pod，依次执行 v6.5.0 → v7.0.0 → v7.2.0 的所有脚本。

> 以上仅为示例。若本次升级路径中在线上文档遇到未在本文件列出的附加操作，仍应据实提取并按实际执行逻辑合并。
