## [2.6.0] - 2026-04-09

### SKILL.md 结构瘦身 + 步骤链重写

**背景：** SKILL.md 膨胀至 817 行，AI 注意力分散导致部分规则未严格执行。需要将详细模板/示例与核心行为规则分离，并将文档式描述改为严格的步骤链执行流程。

#### 一、内容拆分

所有行为规则保留在 SKILL.md 并精简表述；格式模板、输出示例、参数说明等摘到独立 reference 文件，按需引用。

**新增文件：**
- `reference/search_guide.md` — 搜索结果分类格式、参考范文召回流程、素材交付模板、末尾素材使用情况示例

#### 二、步骤链重写

将章节描述式（"3.1是做什么的"）改为严格的步骤链式执行流程（步骤1→2→3...），每个步骤包含：
- **进入条件：** 上一步必须完成什么才能进入本步
- **必须执行：** 本步必须完成的动作
- **分支/下一步：** 完成后去哪

完整流程：步骤1(判断搜索) → 步骤2(设计搜索方案) → 步骤3(执行搜索) → 步骤4(结果分类) → 步骤5(自检) → 步骤6(用户确认) → 步骤7(范文召回+素材交付) → 步骤8(识别文种) → 步骤9(大纲确认) → 步骤10(撰写) → 步骤11(生成前检查) → 步骤12(生成Word) → 步骤13(输出)

**跳步路径：** 步骤1判断不需要搜索 → 跳至步骤8

### 搜索策略构造规则优化（3.2节）

**背景：** 原3.2节"方案设计指导"为5个维度平铺罗列（政策依据、参考案例、数据支撑、时间范围、规范性文件），缺乏层级化构造指导，导致AI搜索方案设计时各维度内部缺少分层覆盖。

#### 新增搜索策略构造框架

- **数据支撑**：本级整体数据 + 向下钻取到下属行政区划，根据需要为各区县分别构造query
- **参考案例**：横向（同级可比地区做法）+ 纵向（本地区历年同类工作数据，用于对比分析）
- **附加维度**：时间范围（`--time`）、规范性文件（`--policy`）、地域参数（`--area`）合并为按需叠加项

#### 删除的内容

- 原"方案设计指导"5点平铺清单
- 原"地域规则"独立子节（合并到附加维度中）

- 方案输出格式不变

## [2.5.0] - 2026-03-25

### 素材交付与参考范文机制重构

**背景：** 四模型写作测试发现，素材一股脑丢给写作模型、不标注用途，导致写作效果不佳。同时发现深知搜索 `--full` 全文召回与 `--clean` 清洗函数存在冲突。

#### 一、参考范文召回（3.7节重写）

**旧流程：** "全文召回"——获取文章全文，没有区分"写作素材"和"成文参考"的用途

**新流程："参考范文召回"**
- 明确定位：普通搜索收集"写作素材"，参考范文召回获取"成文参考"（结构和逻辑）
- 用文章标题做query，**不加 `--clean`**（加了会丢弃全文字段）
- 从API返回的多篇结果中按标题精确匹配，只提取目标文章的"全文"字段
- 向用户推荐时说明参考价值（同类公文、结构完整、深度分析）
- 限制最多2篇，尽量控制在1篇
- 提取失败时告知用户，允许重新选择或跳过

#### 二、新增素材交付准备（3.8节）

**在用户确认素材、参考范文召回完成后、进入撰写前，AI将所有素材整理成结构化的"素材使用指引"**

- 写作素材按类型分组（政策依据、数据支撑、参考案例、表述参考），每组标注用途（→用于：正文某部分）
- 参考范文单独列出，标注参考价值、注意事项、全文内容
- 写作素材和参考范文严格分开，告诉写作模型各用来做什么
- 不要一股脑丢原始素材，经过分类和用途标注后再传给写作模型

#### 三、引用规范重写（5.0节）

**旧流程：** 正文加 `[^N^]` 引用标记，末尾列【参考来源】

**新流程：**
- 正文中**不加任何引用标记**，保持正文干净
- 所有素材使用情况统一在末尾【素材使用情况】中标注
- 【素材使用情况】分为两个子节：
  - 写作素材：每条标注"→ 用于：正文某部分"
  - 参考范文：标注"→ 参考：学习了哪些方面"（如有全文召回）
- 参考范文只学结构不照搬，旧数据需用写作素材中的新数据替换

#### 四、配套调整

- 6.3生成前检查：去掉引用标记检查，改为素材使用情况检查
- 3.4素材分类：去掉引用编号相关描述，素材分类表中"是否引用"改为"末尾标注"
- 3.4素材分类标题：去掉"需引用"字样

#### 五、`--full` 参数验证结论

| 发现 | 结论 |
|------|------|
| `--full` + `--clean` | 全文字段被清洗函数丢弃 |
| `--full` 不加 `--clean` | 全文字段正常返回 |
| API返回多篇全文 | 需按标题精确匹配，只提取目标文章 |
| 搜索引擎排序 | 时间优先，旧文章（如2024年）排得很后 |
| 全文召回token消耗 | 19篇全文约6.7万字符，无关全文会干扰写作质量和模型注意力 |

## [2.3.2] - 2026-03-17

### 日期匹配优化

**问题：** 落款日期使用占位符格式（如 `2026年【X】月【X】日`）时，正则无法匹配，导致日期排版不生效（右对齐）

**根因：** 原正则 `^\d{4}年\d{1,2}月\d{1,2}日$` 只匹配纯数字，`\d{1,2}` 无法匹配 `【X】`

**解决方案：**
1. 扩展正则，兼容【】占位符格式：`^\d{4}年(?:\d{1,2}|【[^】]+】)月(?:\d{1,2}|【[^】]+】)日$`
2. 添加位置约束：仅在文档后60%的行中才进行日期匹配，避免正文中的独立日期（如 `2026年1月1日`）被误判为落款日期

**修改文件：** `scripts/format_document.py` 第472行

### Markdown表格转Word原生表格

**问题：** AI生成的公文中，附件回执等表格使用Markdown语法（`| 列1 | 列2 |`），但 `format_document.py` 无法解析，导致表格语法被当成纯文本输出

**解决方案：**
1. 新增 `parse_markdown_table()` 函数：从指定行开始解析连续的Markdown表格行，返回结构化的二维数组

def add_fullwidth_space_if_needed(text):
    """为两字密级/紧急程度自动添加全角空格"""
    if text and len(text) == 2:
        if all('\u4e00' <= c <= '\u9fff' for c in text):

    return text
