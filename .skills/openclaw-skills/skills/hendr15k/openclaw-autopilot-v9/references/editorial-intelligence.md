# Autopilot Editorial Intelligence Reference v9

The art of knowing what to remove. Taste over feature count.

## Philosophy

Anybody can add. Editors know what to cut.

The difference between good and great is rarely about what's there — it's about what ISN'T there. The most impactful presentations have fewer slides. The most memorable writing has fewer words. The most beautiful designs have fewer elements.

v8 always added. v9 also removes, distills, and curates.

---

## The Editorial Protocol

### When to Apply Editorial Review:

```
After creating content:
  → Is there anything redundant? → Cut it
  → Is anything weakening the whole? → Remove it
  → Could removing something make everything else stronger? → Remove it
  → What's the ONE thing that matters most? → Make it impossible to miss
```

### Editorial Actions:

| Action | When | How |
|--------|------|-----|
| Cut | Redundant, weak, or distracting content | Remove entirely |
| Promote | Most important element isn't prominent enough | Make larger, earlier, more visible |
| Distill | Complex content can be simplified | Replace with clearer version |
| Restructure | Order doesn't serve the message | Reorganize for maximum impact |
| Simplify | Over-engineered or over-designed | Strip to essence |

### The Editorial Questions:

```
1. If I could keep only ONE thing from this deliverable, what would it be?
   → Is that thing prominent enough? If not → Promote it.

2. If I removed [element], would anyone notice?
   → If no → Remove it.

3. What's the weakest 20% of this deliverable?
   → Cutting it makes the remaining 80% stronger.

4. Does this serve the user's GOAL or just fill space?
   → If it fills space → Remove it.

5. Would a stranger understand this in 30 seconds?
   → If not → Simplify or restructure.
```

## Editorial Cuts by Domain

### Presentations:

| Cut | Why | Result |
|-----|-----|--------|
| Remove slides that say the same thing as another | Redundancy kills attention | Tighter narrative |
| Remove bullet points that rephrase the heading | If heading says it, bullets add nothing | Cleaner slides |
| Cut "thank you" slide | Wastes the audience's last impression | End with call-to-action instead |
| Remove transition slides | They break flow | Smoother experience |
| Cut "any questions?" slide | Ask them directly instead | More personal |

### Documents & Reports:

| Cut | Why | Result |
|-----|-----|--------|
| Merge executive summary + introduction | Two opening sections = confusion | One powerful opening |
| Remove "methodology" section from non-academic report | Users want insights, not process | More focus on value |
| Cut generic filler paragraphs | "In today's fast-paced world..." adds nothing | Every word earns its place |
| Remove redundant data points | Same insight shown 3 ways = 2 too many | Cleaner, more convincing |
| Cut conclusion that just summarizes | If the report is well-written, the conclusion is implicit | Stronger ending (insight, not summary) |

### Websites:

| Cut | Why | Result |
|-----|-----|--------|
| Reduce CTAs from 7 to 3 | Competing CTAs = decision paralysis | Higher conversion |
| Remove decorative elements that don't serve UX | Visual noise distracts | Cleaner experience |
| Cut "welcome" text on homepage | The homepage design IS the welcome | More visual impact |
| Remove secondary navigation if unnecessary | Fewer choices = clearer path | Better UX |
| Cut placeholder text | "Lorem ipsum" = amateur | Only ship with real content |

### Code:

| Cut | Why | Result |
|-----|-----|--------|
| Remove dead code | It confuses future readers | Cleaner codebase |
| Cut unused dependencies | Smaller bundle, faster builds | Better performance |
| Remove comments that just restate the code | `// increment i` adds nothing | Self-documenting code |
| Cut feature flags for features that shipped | They add complexity | Simpler codebase |

## Editorial vs. Creative Tension

### When they conflict:

| Situation | Resolution |
|-----------|------------|
| Creative idea adds complexity | Is the complexity worth the value? Yes → keep. No → cut. |
| Surprise adds something unnecessary | Is the surprise truly delightful or just noise? Delightful → keep. Noise → cut. |
| Innovation adds a feature that complicates | Does the feature serve the core purpose? Yes → keep. No → cut. |
| Constraint forces removal of something creative | The constraint is the creative constraint. Find a creative solution WITHIN it. |

### Editorial Principle:

"Perfection is achieved not when there is nothing more to add, but when there is nothing left to take away."

### The 80/20 Rule of Editing:

80% of the impact comes from 20% of the elements. Find the 20%. Promote it. Cut everything that isn't in the 20%.

### When NOT to Cut:

| Situation | Why |
|-----------|-----|
| User explicitly wants comprehensive | Respect explicit request |
| Legal/regulatory content | Required by law, can't cut |
| The "boring but necessary" parts | Infrastructure isn't exciting but essential |
| Content the user specifically requested | They asked for it, keep it |
