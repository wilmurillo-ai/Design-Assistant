# usage-notes.md

## 离家模式使用说明

### 这个 skill 在什么情况下使用

当用户说以下表达时，激活此 skill：

- 出门了
- 离家模式
- 出门前检查一下
- 帮我把家里设备关一下
- 我走了
- 关门关灯
- 出门前看看天气

这些表达的共同特征：用户即将离开家，想要一个出门前的状态确认（设备关闭 + 关键信息汇总）。

---

### 和前三个 skill 的最大区别

| | bedtime | wake-up | movie-mode | **leave-home** |
|---|---|---|---|---|
| 核心价值 | 安心入睡 | 高效起床 | 舒适观影 | **放心出门** |
| 内容层 | 默认故事/白噪音 | 默认时间/天气 | 默认不带内容 | **默认天气/日历/提醒** |
| 主流程 | 关设备 → 播报 → 内容 | 环境 → 播报 → 内容 | 环境 → 设备 → 确认 | **双track并行 → 合并汇报** |

---

### 双 track 并行执行设计

这是离家模式特有的设计，不是失误。

**为什么可以并行？**
因为 Track A（设备关闭）走的是 `xiaodu-iot`，Track B（信息汇总）走的是 `xiaodu` 和本地脚本，两个路径完全独立，不会相互打断。

**但每个 track 内部必须串行。**

#### Track A（设备关闭）
1. 关灯
2. 关窗帘
3. 关空调/风扇
4. 门锁上锁（需要二次确认）

#### Track B（信息汇总）
1. 查天气
2. 查日历日程
3. 查提醒事项

#### 合并汇报
两个 track 完成后，合并成一段自然汇报给用户。

---

### 门锁二次确认的标准口径

> 检测到有智能门锁，是否需要帮你上锁？

如果用户说“上锁”“锁门”“锁上”，立即执行。
如果用户说“不用”“忽略”“不用了”，跳过门锁步骤，结束全流程。
如果用户没有回应，不执行门锁动作，但整体流程已完成。

---

### 推荐的最简 fallback 顺序（无 scene 时）

#### Track A
1. 读取 IoT 设备状态
2. 只对状态为"开着"的灯发关闭命令
3. 只对状态为"开着"的窗帘发关闭命令
4. 只对状态为"开着"的空调/风扇发关闭命令
5. 检测是否有智能门锁，有则询问

#### Track B
1. 查天气（通过 `control_xiaodu.sh --command "播下今天的天气"`）
2. 查备忘录（今天的日程安排，通过 `control_xiaodu.sh --command "查看我今天的备忘录信息并播报"`）
3. 查闹钟提醒（通过 `control_xiaodu.sh --command "查看我今天的闹钟"`）

#### 合并汇报
按顺序说：
1. 设备关闭结果（成功项 + 失败项）
2. 天气摘要
3. 日历日程（如果有时段安排）
4. 提醒事项（如果有未完成的）
5. 门锁询问

---

### 设备状态预检查

**重要**：关灯/关窗帘/关空调之前，先读设备当前状态。

对于每个设备：
- 状态为"开着" → 发关闭命令
- 状态为"关闭"或"关" → 跳过后续动作，直接记录为"已关闭，无需操作"
- 状态未知 → 尝试关闭，失败则记录

这样做的好处：
- 避免重复发命令
- 汇报时更准确
- 更稳定（对已关闭设备再发关闭命令有时会报错）

---

### 面向用户的标准汇报口径

#### 标准版（全部成功）
> 已帮你把出门模式安排好了。灯已全部关闭，空调已关闭，窗帘已拉好。出门信息这边：今天下午3点有个会，晚上记得买菜。天气方面，今天晴转多云，18-25度，出行没问题。门锁这边需要帮你上锁吗？

#### 降级版（部分设备失败）
> 已帮你把出门模式安排好了。灯已全部关闭，空调已关闭，不过窗帘关闭这次没成功。天气和日程我这边都读到了，今天天气不错，没有特别的日程冲突。门锁这边需要帮你上锁吗？

#### 简化版（无门锁场景）
> 已帮你把出门模式安排好了。灯已全部关闭，空调已关闭，窗帘已拉好。今天天气晴，18-25度，适合出行。

### 偏好沉淀示例

用户在离家过程中说：
- "以后出门不用问，直接上锁"
- "客厅灯不用关，只关空调和窗帘"
- "以后出门只关主卧的灯"

这些都应该被吸收为正式偏好，写入 `XIAODU_CONTEXT.md`。

---

### 串行执行规则（延续前三个 skill 的要求）

对 Track A 内部的所有命令，必须严格串行：

1. 上一条命令返回后，才能发下一条命令
2. 不要把 `xiaodu-iot` 的控制命令并发打出去
3. 如果前一个动作失败，后一个动作直接继续执行
4. Track A 和 Track B 之间可以并行（路径独立）

这条规则同时适用于：
- `xiaodu` 智能屏控制
- `xiaodu-iot` 家电控制

---

### 失败处理

#### 典型降级

| 失败场景 | 降级行为 |
|---------|---------|
| 灯已全关 | 跳过关灯，记录为"已关闭，无需操作" |
| 关灯失败 | 继续关窗帘、空调；汇报里说明 |
| 无智能门锁 | 直接完成汇报，不询问门锁 |
| 天气读取失败 | 跳过天气，继续日历和提醒；汇报说明 |
| 日历读取失败 | 跳过日历；汇报说明 |
| 提醒读取失败 | 跳过提醒；汇报说明 |

---

### 实际调用链参考

对于"出门了"这类高层请求，默认实际链路是：

**Track A（设备关闭）**
1. `list_scenes.sh`
2. 如果有离家 scene，就 `trigger_scene.sh`
3. 否则 `list_iot_devices.sh`
4. 对状态为"开着"的灯：`control_iot.sh --action turnOff`
5. 对状态为"开着"的窗帘：`control_iot.sh --action close`
6. 对状态为"开着"的空调/风扇：`control_iot.sh --action turnOff`
7. 如果检测到门锁，询问后执行上锁

**Track B（信息汇总）**
1. `control_xiaodu.sh --command "播下今天的天气"`
2. `control_xiaodu.sh --command "查看我今天的备忘录信息并播报"`
3. `control_xiaodu.sh --command "查看我今天的闹钟"`

**合并汇报**
两个 track 完成后，合并成自然语言汇报。

---

### 为什么信息层是主流程的一部分

前三个 skill 的内容层是"附加体验"。
离家模式的信息层是"出门安全检查"。

用户出门前最怕的不是"灯没关"，而是"出门后才发现今天下午有个会"或"忘带伞了"。

所以天气/日历/提醒是主流程，不是附加功能。
