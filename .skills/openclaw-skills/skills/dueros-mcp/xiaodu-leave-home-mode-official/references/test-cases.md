# test-cases.md

## 测试用例

---

## 1. 有离家 scene 时的完整调用链

### 1A. scene 存在且触发成功
- 输入：出门了
- 前提：有高置信度离家 scene
- 预期：`list_scenes` -> `trigger_scene` -> Track A + Track B -> 合并汇报
- 预期用户口径：自然产品汇报，不是技术日志

---

## 2. Track A：设备关闭

### 2A. 灯 + 窗帘 + 空调全部关闭（无 scene）
- 输入：离家模式
- 前提：没有匹配 scene，但灯开着、窗帘开着、空调开着
- 预期：关灯 -> 关窗帘 -> 关空调 -> Track B -> 合并汇报

### 2B. 部分设备已关闭（状态预检查）
- 输入：离家模式
- 前提：灯已关闭，窗帘开着，空调开着
- 预期：跳过关灯 -> 关窗帘 -> 关空调 -> 合并汇报，汇报里说"灯已关闭，无需操作"

### 2C. 只有灯开着（无窗帘、无空调）
- 输入：出门了
- 前提：只有灯开着，没有窗帘和空调
- 预期：关灯 -> Track B -> 合并汇报（不说窗帘/空调相关内容）

### 2D. 设备全部已关闭
- 输入：出门了
- 前提：灯、窗帘、空调全部已关闭
- 预期：状态预检查发现全部已关 -> 跳 Track A 所有步骤 -> Track B -> 合并汇报，汇报里说"家里设备都已经是关闭状态"

### 2E. 关灯失败，但窗帘和空调可继续
- 输入：离家模式
- 前提：灯故障无法关闭，但窗帘和空调可关
- 预期：关灯失败 -> 继续关窗帘 -> 关空调 -> Track B -> 合并汇报，汇报说明灯失败了但窗帘和空调已关闭

---

## 3. Track B：信息汇总

### 3A. 天气/日历/提醒全部读取成功
- 输入：出门了
- 前提：天气、日历、提醒都能正常读取
- 预期：天气播报 -> 日历日程播报 -> 提醒事项播报 -> 合并进汇报

### 3B. 天气读取失败
- 输入：出门了
- 前提：天气接口失败，日历和提醒正常
- 预期：跳过天气，继续日历 -> 提醒 -> 合并汇报，汇报里说明"天气暂时没读到"

### 3C. 日历无日程
- 输入：出门了
- 前提：今天没有日历日程
- 预期：天气和提醒正常读取，合并汇报里省略日历部分，只说天气和提醒

### 3D. 提醒无未完成项
- 输入：出门了
- 前提：今天没有未完成的提醒
- 预期：天气和日历正常读取，合并汇报里省略提醒部分

---

## 4. 门锁二次确认

### 4A. 检测到门锁，询问后用户确认上锁
- 输入：出门了
- 前提：有智能门锁
- 预期：Track A 完成 -> Track B 完成 -> 询问"需要帮你上锁吗？" -> 用户说"上锁" -> 执行上锁

### 4B. 检测到门锁，用户说不用上锁
- 输入：出门了
- 前提：有智能门锁
- 预期：Track A 完成 -> Track B 完成 -> 询问"需要帮你上锁吗？" -> 用户说"不用" -> 跳过门锁，结束汇报

### 4C. 没有智能门锁
- 输入：出门了
- 前提：没有智能门锁
- 预期：Track A 完成 -> Track B 完成 -> 直接出合并汇报，不询问门锁

### 4D. 用户在请求中已明确说"不用锁门"
- 输入：出门了，不用锁门
- 前提：有智能门锁
- 预期：Track A 完成 -> Track B 完成 -> 跳过门锁确认，直接出合并汇报

---

## 5. 极简确认

### 5A. 单设备场景不需要确认
- 输入：出门了
- 前提：只有一个灯在线，没有其他设备
- 预期：直接关灯，不需要先问

### 5B. 有偏好时不重复确认
- 输入：出门了
- 前提：用户之前已设定"默认不上锁"
- 预期：直接完成所有步骤，不询问门锁

### 5C. 多设备必须确认
- 输入：出门了
- 前提：有灯、窗帘、空调、门锁，用户没说范围
- 预期：按默认范围（客厅 + 主卧）执行，不需要逐个确认

---

## 6. 双 track 并行执行

### 6A. Track A 和 Track B 同时启动
- 输入：出门了
- 预期：Track A（关灯/窗帘/空调）和 Track B（天气/日历/提醒）同时启动
- 预期：两个 track 独立执行，不互相等待
- 预期：汇报等两个 track 都完成后合并输出

### 6B. Track A 先完成，Track B 后完成
- 输入：出门了
- 预期：Track A 完成后立即出初步汇报，Track B 完成后补充信息层

---

## 7. 单点失败不中断全流程

### 7A. Track A 某步失败，Track B 继续
- 输入：离家模式
- 前提：关灯失败，但天气/日历/提醒正常
- 预期：Track A 继续窗帘和空调步骤，Track B 正常执行，合并汇报时说明灯的问题

### 7B. Track B 某步失败，Track A 继续
- 输入：离家模式
- 前提：天气读取失败，但设备关闭正常
- 预期：Track A 正常执行，Track B 跳过天气继续日历和提醒，合并汇报时说明天气问题

---

## 8. 偏好记忆

### 8A. 用户设定默认不上锁偏好
- 输入：以后出门不用问，直接不上锁
- 预期：偏好写入 `XIAODU_CONTEXT.md`，下次不询问门锁

### 8B. 用户设定只关部分设备偏好
- 输入：以后出门只关空调，灯不用管
- 预期：偏好写入，下次只执行关空调步骤

---

## 9. 串行执行验证

### 9A. Track A 内部必须串行执行
- 输入：离家模式
- 预期：关灯 -> 等待返回 -> 关窗帘 -> 等待返回 -> 关空调 -> 等待返回
- 预期：前一条命令没返回前，不应发送下一条命令
- 预期：若前一条失败，后一条直接继续

### 9B. Track B 内部必须串行执行
- 输入：出门了
- 预期：天气 -> 等待返回 -> 日历 -> 等待返回 -> 提醒 -> 等待返回
- 预期：前一条命令没返回前，不应发送下一条命令

---

## 10. 合并汇报内容验证

### 10A. 完整合并汇报（两个 track 都成功）
- 输入：出门了
- 预期汇报包含：
  1. 设备关闭结果（逐项列出）
  2. 天气摘要
  3. 日历日程（如果有时段安排）
  4. 提醒事项（如果有未完成的）
  5. 门锁询问

### 10B. 设备全成功，信息层全失败
- 输入：出门了
- 预期汇报：设备侧全部列出，信息侧说明暂时没信息，询问门锁

### 10C. 设备部分失败，信息层全成功
- 输入：离家模式
- 预期汇报：设备侧列出成功项和失败项，信息侧完整汇报，不询问门锁（如果无门锁）
