# 测试用例

## 1. 有现成 bedtime scene
- 输入：小度，带孩子睡觉
- 预期：先列场景 -> 匹配到晚安/睡前 -> 触发场景 -> 做一句简短确认播报

## 2. 有儿童睡眠类场景
- 输入：开始睡前模式
- 预期：匹配到儿童睡眠 scene -> 触发 -> 做一句简短睡前确认

## 3. 没有 scene，但有主灯和辅助灯
- 输入：帮我哄孩子睡觉
- 预期：调暗或关闭主灯 -> 启用更柔和的辅助灯 -> 做睡前播报 -> 进入睡前内容

## 4. 没有 scene，只有主灯
- 输入：把卧室调成适合睡觉的状态
- 预期：做主灯调整 -> 不要假装夜灯存在 -> 继续睡前播报 / 内容

## 5. 没有 IoT，但有智能屏
- 输入：开始睡前模式
- 预期：不要伪造 IoT 执行 -> 仍然可以播报并进入睡前内容

## 6. 用户明确要求讲故事
- 输入：帮我哄孩子睡觉，再讲个故事
- 预期：先做环境调整（如果有）-> 再发一个故事请求

## 7. 用户明确要求轻音乐
- 输入：带孩子睡觉，放一点轻音乐
- 预期：先做环境调整（如果有）-> 再发一个轻音乐请求

## 8. 用户明确要求白噪音
- 输入：开始睡前模式，放白噪音
- 预期：先做环境调整（如果有）-> 再发一个白噪音请求

## 8A. 泛化睡前请求走完整实际调用链
- 输入：小度，带孩子睡觉
- 预期：列场景 -> 无 scene 或触发 scene -> 列 IoT 设备 -> 做安全环境调整 -> 列智能屏 -> 睡前播报 -> 故事请求 -> 挂 15 分钟后的 stop/pause -> 支持时继续 screen-off
- 预期用户口径：自然产品汇报，不是技术日志
- 预期价值表达：如果射灯确实调暗了，就应在播报里明确说出“我刚刚已经把射灯调暗了”这类成功动作

## 8B. 故事优先路径带收尾控制
- 输入：小度，带孩子睡觉
- 预期：默认 story-first，并带 5 分钟后 stop + screen-off 收尾

## 8C. 延时收尾暂不可用时的降级
- 输入：小度，带孩子睡觉
- 前提：当前运行环境无法稳定挂起延时 follow-up
- 预期：仍然可以启动故事，但要明确当前设计期望有 timed stop，并采用最接近的后续控制方式

## 9. 单个 IoT 动作失败
- 输入：帮我哄孩子睡觉
- 前提：主灯失败，但其它 bucket 可继续
- 预期：不要整体中断 -> 继续后面的安全动作和睡前内容

## 9A. bedtime 流程必须串行执行
- 输入：带孩子睡觉
- 预期：灯光、播报、故事、延时收尾必须一个接一个执行
- 预期：前一条命令没返回前，不应发送下一条命令
- 预期：若前一条失败，后一条直接继续，不中断全流程

## 10. schema 不支持亮度属性
- 输入：太亮了，帮我进入睡前模式
- 前提：灯存在，但不支持 brightness
- 预期：跳过不支持的 brightness 设置，继续尝试其他安全调整

## 11. 用户说有点热
- 输入：带孩子睡觉，有点热
- 预期：先做睡前环境动作 -> 只在存在相关设备且 schema 支持时再调整空调/风扇

## 12. 用户说有点冷
- 输入：开始睡前模式，有点冷
- 预期：不要脑补加热能力 -> 只在设备存在且能力支持时调整

## 13. 只有非卧室设备
- 输入：把卧室调成适合睡觉的状态
- 预期：先让用户 уточ明房间 / 设备，不要默认控全屋

## 14. 找不到智能屏
- 输入：帮我哄孩子睡觉，讲个故事
- 预期：不能假装播报 / 故事已经成功，应明确说明无法定位智能屏
