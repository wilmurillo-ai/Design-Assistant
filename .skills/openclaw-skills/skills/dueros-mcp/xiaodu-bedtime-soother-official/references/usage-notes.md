# 使用说明

## 依赖模型

这个 orchestrator 必须把 `xiaodu-control-official` 视为事实来源。
如果某个 bucket、设备族或动作没有在依赖 skill 中明确出现，就不要纳入规划。

本 skill 的统一规划宇宙只允许使用：
- scene
- lights
- curtains
- thermal comfort
- smart-screen speech
- smart-screen assistant/media

本 skill 依赖 `xiaodu-control-official` 作为控制平面。
不要在这里重做底层 XiaoDu MCP 操作。
依赖 skill 已经提供了稳定脚本封装：

- 智能屏发现与播报
- 智能屏助手类命令
- IoT 设备列表与控制
- 场景列表与触发

复用这些脚本可以减少 prompt 噪音，避免底层参数错误，让编排逻辑聚焦在睡前体验本身。

## 什么时候优先使用现成 IoT 场景

当以下条件同时满足时，优先使用已有场景：

1. scene 名称与睡前意图高度匹配
2. 这个 scene 明显比临时拼装的全屋控制更安全
3. 用户说的是高层意图，不是精确到单设备的细调

高置信度 bedtime scene 例如：

- 晚安
- 睡眠
- 睡前
- 儿童睡眠
- 夜间

为什么优先场景：

- 更快
- 更稳
- 更符合家庭真实设备分组
- 更少参数错配风险

如果场景成功触发，再补一句简短的智能屏确认即可。

## 什么时候做分步控制

在这些情况下改走分步控制：

- 没有匹配的睡前场景
- 场景存在但含义模糊
- 用户加了额外约束，比如“太亮了”“有点热”
- 用户要的是具体调整，不是泛化模式

推荐的睡前 fallback 顺序：

1. 调暗或关闭主灯
2. 如果存在夜灯 / 床头灯，就启用柔和灯光
3. 如果 schema 支持，就把灯光调得更暖更柔和
4. 如果有窗帘，就关闭
5. 如果用户明确说热 / 冷 / 闷，再考虑空调 / 风扇
6. 做一句简短睡前播报
7. 继续进入睡前内容主流程

默认睡前内容优先级：
1. 故事
2. 白噪音
3. 轻音乐
4. 只播报

如果走默认 story-first，还应附带默认收尾步骤：
1. 开始播放睡前故事
2. 大约 15 分钟后停止播放
3. 然后尝试关闭屏幕

默认落地方式：
- `bash scripts/bedtime_story_tail.sh --device-name "..." --delay-minutes 15`
- 或者改用 `--cuid + --client-id`

这一步是为了避免故事落到长播放列表后一直不停。

## 实际调用链参考

对于高层睡前请求，默认实际链路是：

1. `list_scenes.sh`
2. 如果有 bedtime scene，就 `trigger_scene.sh`
3. 否则 `list_iot_devices.sh`
4. 然后做一个或多个安全的 `control_iot.sh`
5. `list_devices.sh`
6. `speak.sh`
7. `control_xiaodu.sh` 启动睡前内容
8. 如果选中故事，再用 `scripts/bedtime_story_tail.sh` 做延时 stop + screen-off

这能保证整条编排链是分层且显式的。
除非用户明确说要 content-only，否则不要跳过环境与播报阶段，直接开始故事。

## 智能屏与 IoT 路由边界

以下动作优先走 `xiaodu-iot`：

- 灯
- 窗帘
- 空调
- 风扇
- 卧室环境调整
- 场景触发

以下动作优先走 `xiaodu`：

- 简短确认播报
- 睡前故事
- 白噪音 / 轻音乐
- 故事后的延时 stop / pause
- 屏幕关闭（如果助手指令支持）
- 其他智能屏助手能力

不要在明明有 IoT 可控路径时，反而用 `control_xiaodu` 去控制灯、窗帘、空调。

## 面向用户的汇报风格

面向用户时，优先给**产品化、自然汇报**，不要给技术动作日志。
一段好的自然汇报通常覆盖：

1. 已经帮用户安排好了什么
2. 现在正在发生什么
3. 接下来自动会发生什么

而且要把**真实已经成功的关键动作**讲出来，不要把价值藏在后台。
如果灯光确实调暗了、窗帘确实关了、故事确实开始了，就应该体现在用户可听见的汇报里。

示例：

- 已经帮你把睡前模式安排好了。
- 我刚刚已经把射灯调暗了，现在房间已经更适合休息。
- 我也已经在小度上开始讲睡前故事了。
- 为了不一直播放下去，我还安排了一个收尾动作：5 分钟后会自动停掉故事，并尝试把屏幕关掉。

可以把这理解成“动作完成回执的产品化表达”，而不是技术日志。

避免这样对用户说：
- 我先执行了 list_scenes
- 我调用了 control_xiaodu
- 我刚刚起了一个 nohup 任务
- 我执行了某个脚本

## 串行执行规则

对同一条 bedtime 流程里的控制动作，必须严格串行：

1. 上一个命令返回后，才能发下一个命令
2. 不要把多条 `xiaodu` 助手命令并发打给同一台设备
3. 不要把同一条 bedtime 流程里的 IoT 控制并发打给设备链路
4. 如果前一个动作失败，后一个动作直接继续执行，不要让整个流程因为单点失败而中断

这条规则同时适用于：
- `xiaodu` 智能屏控制
- `xiaodu-iot` 家电控制

原因：在当前小度链路里，并发很容易让前一条指令被打断，或者让设备解析 / 控制阶段变得不稳定。

## 失败处理

这个 skill 对局部失败应当有容错。
不要因为一个动作失败就让整个睡前流程中断。

好的降级示例：

- 没有 scene -> 自动改走 step fallback
- 主灯控制失败 -> 继续尝试其他安全 bucket 和睡前内容
- 没有智能屏 -> 仍可做安全 IoT 动作
- 没有 IoT -> 仍可完成播报和睡前内容
- 亮度属性不支持 -> 跳过亮度设置，继续后续动作
- 延时 stop 无法挂起 -> 仍可启动故事，但要明确当前环境需要用最接近的后续机制收尾

## 为什么不能重做底层

除非必要，不要绕过 `xiaodu-control-official`，因为它已经：

- 分清了 `xiaodu` 与 `xiaodu-iot`
- 带有更安全的脚本约束
- 定义了能力边界与 schema 预期
- 让排障更容易聚焦在同一依赖层

### 当前环境下的实际路由建议

- 当前没有 bedtime scene，所以 scene-first 会很快 fall through。
- 当前没有确认到夜灯 / 床头灯 / 窗帘。
- `三菱电机空调` 在客厅，默认不要拿它当孩子卧室睡前设备。
- 当前默认优先屏顺序：
  1. `小度智能屏3`
  2. `小度智能屏2`
  3. `小度添添闺蜜机Pro 4K Max`
- 默认不要主动碰的 IoT：
  - `三菱电机空调`
  - `夏普电视`
  - `体脂秤`
- 当前最稳的默认做法是：
  1. 先选一个偏好的智能屏
  2. 如果有真实可达且相关的灯，就做安全环境调整
  3. 做一句简短安抚播报
  4. 进入睡前内容
  5. 如果是故事，就挂 15 分钟后收尾

### 哪些 query 默认应保持 screen-first

在当前环境里，以下 query 默认更适合 screen-first：

- 带孩子睡觉
- 帮我哄孩子睡觉
- 给孩子讲个睡前故事
- 放白噪音
- 放轻音乐
- 说一句晚安

这些 query 可以包含安全灯光调整，但不应扩展成不必要的全屋 IoT 控制。

### 哪些 query 只有明确目标后才碰 IoT

- 把卧室调成适合睡觉的状态
- 灯太亮了，准备睡觉
- 有点热，开睡前模式

只有在以下条件之一成立时才应该碰 IoT：
- 用户明确指定了设备或房间
- 刷新后的设备列表里确实出现了相关且可达的卧室设备

## 常见失败场景与降级

### 没有 bedtime scene

退到 per-device bedroom adjustment + bedtime content。

### 没有夜灯

不要假装夜灯已经打开。
可以尝试把其他可达灯光调得更柔和。

### 用户明确要故事或白噪音

尽量先完成环境调整，再发一个智能屏内容请求。

### 用户只给了一个泛化睡前请求

除非环境或助手行为强迫降级，否则不要停在只播报。
泛化睡前请求通常应该继续进入一项睡前内容。
优先顺序：
1. 故事
2. 白噪音
3. 轻音乐
4. 只播报

如果最终选择故事，默认也应带上 5 分钟后收尾。

### 只有非卧室设备

不要默认控全屋。
先请用户指定房间或设备。

### 没有 IoT，但有智能屏

仍然可以完成一段温和播报，并继续睡前内容。
