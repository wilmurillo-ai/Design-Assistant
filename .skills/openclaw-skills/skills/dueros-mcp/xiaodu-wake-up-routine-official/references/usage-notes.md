# 使用说明

## 依赖模型

这个 orchestrator 必须把 `xiaodu-control-official` 视为事实来源。
如果某个 bucket、设备族或动作没有在依赖 skill 中明确出现，就不要纳入规划。

本 skill 的统一规划宇宙只允许使用：
- scene
- lights
- curtains
- thermal comfort
- smart-screen speech
- smart-screen assistant/media

本 skill 依赖 `xiaodu-control-official` 作为控制平面。
不要在这里重做底层 XiaoDu MCP 操作。
依赖 skill 已经提供了稳定脚本封装：

- 智能屏发现与播报
- 智能屏助手类命令
- IoT 设备列表与控制
- 场景列表与触发

复用这些脚本可以减少 prompt 噪音，避免底层参数错误，让编排逻辑聚焦在晨起体验本身。

## 什么时候优先使用现成 IoT 场景

当以下条件同时满足时，优先使用已有场景：

1. scene 名称与晨起意图高度匹配
2. 这个 scene 明显比临时拼装的全屋控制更安全
3. 用户说的是高层意图，不是精确到单设备的细调

高置信度 wake-up scene 例如：

- 早安
- 起床
- 晨间
- 上学前
- 早晨

为什么优先场景：

- 更快
- 更稳
- 更符合家庭真实设备分组
- 更少参数错配风险

如果场景成功触发，再补一句简短的智能屏确认即可。

## 什么时候做分步控制

在这些情况下改走分步控制：

- 没有匹配的晨起场景
- 场景存在但含义模糊
- 用户加了额外约束，比如“太暗了”“太闷了”
- 用户要的是具体调整，不是泛化模式

推荐的晨起 fallback 顺序：

1. 打开或提亮灯光
2. 如果 schema 支持，把灯光调成更适合晨起的状态
3. 如果有窗帘，就打开
4. 如果用户明确说闷 / 热，再考虑空调 / 风扇
5. 做一句简短晨起播报
6. 继续说当前时间
7. 继续播天气
8. 如果存在明确可提醒的备忘录信息，就播报备忘录提醒
9. 继续播放儿歌或轻快音乐

默认晨起主流程优先级：
1. 环境调整
2. 晨起提醒播报
3. 时间
4. 天气
5. 备忘录信息
6. 儿歌或轻快音乐

## 实际调用链参考

对于高层晨起请求，默认实际链路是：

1. `list_scenes.sh`
2. 如果有 wake-up scene，就 `trigger_scene.sh`
3. 否则 `list_iot_devices.sh`
4. 然后做一个或多个安全的 `control_iot.sh`
5. `list_devices.sh`
6. `speak.sh`
7. `control_xiaodu.sh` 继续补全天气 / 时间 / 新闻 / 儿歌 / 轻音乐中的主流程项

这能保证整条编排链是分层且显式的。
除非用户明确说要 info-only，否则不要跳过环境与播报阶段，直接开播天气或儿歌。

## 智能屏与 IoT 路由边界

以下动作优先走 `xiaodu-iot`：

- 灯
- 窗帘
- 空调
- 风扇
- 房间环境调整
- 场景触发

以下动作优先走 `xiaodu`：

- 简短确认播报
- 天气 / 时间 / 新闻
- 儿歌 / 轻音乐
- 其他智能屏助手能力

不要在明明有 IoT 可控路径时，反而用 `control_xiaodu` 去控制灯、窗帘、空调。

## 面向用户的汇报风格

面向用户时，优先给**产品化、自然汇报**，不要给技术动作日志。
一段好的自然汇报通常覆盖：

1. 已经帮用户安排好了什么
2. 现在正在发生什么
3. 接下来用户还可以让系统继续做什么

而且要把**真实已经成功的关键动作**讲出来，不要把价值藏在后台。
如果灯光确实提亮了、窗帘确实打开了、晨起提醒确实开始了、天气/儿歌确实开始了，就应该体现在用户可听见的汇报里。

示例：

- 已经帮你把起床模式安排好了。
- 我刚刚已经把灯光调得更适合起床了，现在房间已经更适合开始一天。
- 我也已经在小度上开始提醒孩子起床了，接下来我还会继续播时间、天气、备忘录提醒，再放一点儿歌或轻快音乐。

避免这样对用户说：
- 我先执行了 list_scenes
- 我调用了 control_xiaodu
- 我刚刚执行了脚本

## 串行执行规则

对同一条 wake-up 流程里的控制动作，必须严格串行：

1. 上一个命令返回后，才能发下一个命令
2. 不要把多条 `xiaodu` 助手命令并发打给同一台设备
3. 不要把同一条 wake-up 流程里的 IoT 控制并发打给设备链路
4. 如果前一个动作失败，后一个动作直接继续执行，不要让整个流程因为单点失败而中断

这条规则同时适用于：
- `xiaodu` 智能屏控制
- `xiaodu-iot` 家电控制

原因：在当前小度链路里，并发很容易让前一条指令被打断，或者让设备解析 / 控制阶段变得不稳定。

## 失败处理

这个 skill 对局部失败应当有容错。
不要因为一个动作失败就让整个晨起流程中断。

好的降级示例：

- 没有 scene -> 自动改走 step fallback
- 窗帘失败 -> 继续做灯光、播报等后续动作
- 没有智能屏 -> 仍可做安全 IoT 动作
- 没有 IoT -> 仍可完成播报以及明确要求的信息 / 内容
- 某个属性不支持 -> 跳过这个属性继续后面动作

## 为什么不能重做底层

除非必要，不要绕过 `xiaodu-control-official`，因为它已经：

- 分清了 `xiaodu` 与 `xiaodu-iot`
- 带有更安全的脚本约束
- 定义了能力边界与 schema 预期
- 让排障更容易聚焦在同一依赖层

### 当前环境下的实际路由建议

- 当前没有 wake-up scene，所以 scene-first 会很快 fall through。
- 当前没有确认到窗帘。
- `三菱电机空调` 在客厅，默认不要拿它当孩子卧室晨起设备。
- 当前默认优先屏顺序：
  1. `小度智能屏3`
  2. `小度智能屏2`
  3. `小度添添闺蜜机Pro 4K Max`
- 默认不要主动碰的 IoT：
  - `三菱电机空调`
  - `夏普电视`
  - `体脂秤`
- 当前最稳的默认做法是：
  1. 先选一个偏好的智能屏
  2. 如果有真实可达且相关的灯，就做安全环境调整
  3. 做一句简短晨起播报
  4. 然后继续补天气、时间、儿歌 / 轻快音乐这类晨起主流程内容

### 哪些 query 默认应保持 screen-first

在当前环境里，以下 query 默认更适合 screen-first：

- 叫孩子起床
- 帮我把孩子叫醒
- 播下今天的天气
- 说下现在几点
- 放儿歌
- 放轻快一点的音乐

这些 query 不应扩展成不必要的全屋 IoT 控制。

### 哪些 query 只有明确目标后才碰 IoT

- 让房间进入起床状态
- 拉开窗帘，把灯打开
- 太暗了，开起床模式
- 房间太闷了，叫孩子起床

只有在以下条件之一成立时才应该碰 IoT：
- 用户明确指定了设备或房间
- 刷新后的设备列表里确实出现了相关且可达的卧室设备

## 常见失败场景与降级

### 没有 wake-up scene

退到 per-device room adjustment + wake-up speech。

### 没有窗帘

不要假装窗帘已经打开。
可以优先通过灯光增强晨起感。

### 用户明确要天气 / 时间 / 儿歌

尽量先完成环境调整，再发一个智能屏信息 / 内容请求。

### 用户只给了一个泛化晨起请求

不要默认播一长串天气 / 新闻 / 音乐。
先完成环境与叫醒播报，再根据用户明确要求追加内容。

### 只有非卧室设备

不要默认控全屋。
先请用户指定房间或设备。

### 没有 IoT，但有智能屏

仍然可以完成一段简短晨起播报，并继续明确要求的信息 / 内容。
