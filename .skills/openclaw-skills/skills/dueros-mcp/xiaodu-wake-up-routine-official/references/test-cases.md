# 测试用例

## 1. 有现成 wake-up scene
- 输入：小度，叫孩子起床
- 预期：先列场景 -> 匹配到早安 / 起床 -> 触发场景 -> 做一句简短确认播报

## 1A. 泛化晨起请求走完整实际调用链
- 输入：小度，叫孩子起床
- 预期：列场景 -> 无 scene 或触发 scene -> 列 IoT 设备 -> 做安全环境调整 -> 列智能屏 -> 晨起播报 -> 继续说时间 -> 继续播天气 -> 如果存在备忘录提醒则继续播报 -> 继续播放儿歌或轻快音乐
- 预期用户口径：自然产品汇报，不是技术日志
- 预期价值表达：如果灯光确实提亮了，就应在播报里明确说出“我刚刚已经把灯光调得更适合起床了”这类成功动作

## 2. 有 school-morning scene
- 输入：开始早安模式
- 预期：匹配到上学前类 scene -> 触发 -> 做一句简短晨起确认

## 3. 没有 scene，但有灯和窗帘
- 输入：帮我把孩子叫醒
- 预期：打开或提亮灯 -> 打开窗帘 -> 做晨起播报

## 4. 没有 scene，只有灯
- 输入：让房间进入起床状态
- 预期：做灯光调整 -> 不要假装窗帘存在 -> 继续晨起播报

## 5. 没有 IoT，但有智能屏
- 输入：开始早安模式
- 预期：不要伪造 IoT 执行 -> 仍然可以完成播报

## 6. 用户明确要求天气
- 输入：帮我把孩子叫醒，再播下今天的天气
- 预期：先做环境调整（如果有）-> 再发一个天气请求

## 7. 用户明确要求儿歌
- 输入：开始早安模式，放儿歌
- 预期：先做环境调整（如果有）-> 再发一个儿歌请求

## 8. 用户明确要求轻音乐
- 输入：小度，叫孩子起床，放点轻快的音乐
- 预期：先做环境调整（如果有）-> 再发一个轻音乐请求

## 9. 单个 IoT 动作失败
- 输入：帮我把孩子叫醒
- 前提：窗帘失败，但灯或播报还能继续
- 预期：不要整体中断 -> 继续后面的安全动作

## 9A. wake-up 流程必须串行执行
- 输入：叫孩子起床
- 预期：灯光、播报、时间、天气、备忘录、儿歌 / 轻快音乐必须一个接一个执行
- 预期：前一条命令没返回前，不应发送下一条命令
- 预期：若前一条失败，后一条直接继续，不中断全流程

## 10. schema 不支持亮度属性
- 输入：太暗了，打开起床流程
- 前提：灯存在，但不支持 brightness
- 预期：跳过不支持的 brightness 设置，继续尝试基本开灯

## 11. 用户说房间太闷 / 太热
- 输入：开始早安模式，房间太闷了
- 预期：只在存在相关设备且 schema 支持时再调整空调 / 风扇

## 12. 用户要求拉开窗帘
- 输入：拉开窗帘，把孩子叫醒
- 预期：优先处理窗帘，再做晨起播报；不要无故扩展成很多别的内容

## 13. 只有非卧室设备
- 输入：让房间进入起床状态
- 预期：先让用户 уточ明房间 / 设备，不要默认控全屋

## 14. 找不到智能屏
- 输入：帮我把孩子叫醒，播下今天的天气
- 预期：不能假装播报 / 天气已经成功，应明确说明无法定位智能屏
