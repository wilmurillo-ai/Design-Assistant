# usage-notes.md

## 夜间辅助模式使用说明

### 这个 skill 在什么情况下使用

当用户说以下表达时，激活此 skill：

- 夜里起夜
- 我要去卫生间
- 帮我开一下夜灯
- 夜间辅助模式
- 我起夜了
- 去趟厕所
- 太黑了
- 我要喝水，开一下灯

这些表达有一个共同特征：用户半夜临时起身，需要一个更安全、更柔和、更低打扰的夜间辅助状态。

---

### 和 bedtime 的最大区别

bedtime 的目标是：**帮助进入睡眠**。
night-assist 的目标是：**在不破坏睡眠的前提下，完成一次安全短时起身**。

所以这个 skill 默认：
- 有灯光
- 有短播报
- 有自动收尾
- 没有内容层
- 没有长信息流

---

### 两种核心子模式

#### 路径照明模式
适用于：
- 夜里起夜
- 我要去卫生间
- 去趟厕所
- 帮我开一下去卫生间的灯

产品理解：
> 开一条安全路径，而不是只开一盏灯。

默认顺序：
1. 床边灯 / 夜灯
2. 走廊灯
3. 卫生间灯

#### 局部照明模式
适用于：
- 帮我开一下夜灯
- 太黑了
- 我要喝水
- 开一下床边灯

产品理解：
> 只做近距离、低打扰的局部照明。

默认动作：
1. 只开最近的一盏柔和灯
2. 不扩展成整条路径
3. 自动收尾时间更短

---

### 推荐的最简 fallback 顺序（无 scene 时）

1. 判断是路径照明还是局部照明
2. 读取 IoT 设备
3. 打开合适的灯
4. 如果支持亮度，调低亮度
5. 如果支持色温，调暖
6. 做一句短播报
7. 如果用户明确说冷 / 热 / 闷，再做轻微温控
8. 挂自动收尾

### 默认夜间主流程优先级

1. 灯光安全辅助
2. 简短播报
3. 温控附加（仅明确需要时）
4. 自动收尾

---

### 自动收尾建议

自动收尾是主流程的一部分，不是附加功能。

默认：
- 路径照明模式：5 分钟后关灯
- 局部照明模式：3 分钟后关灯

如果用户明确说：
- 先别关
- 我要待一会儿
- 不用自动关

则跳过自动收尾。

---

### 面向用户的标准汇报口径

推荐口径：

- 夜灯已经开好了，慢一点走。
- 路径灯已经打开了，注意脚下。
- 灯已经帮你开好了，过几分钟我会自动关掉。

不推荐：
- 我现在为你执行夜间安全辅助链路
- 已调用灯光系统
- 现在开始执行定时关灯

---

### 偏好沉淀示例

用户说：
- 以后夜里都只开床边灯
- 以后 5 分钟再自动关
- 以后不要播报，只开灯
- 以后夜灯亮一点

这些都应该被吸收为正式偏好，写入 `XIAODU_CONTEXT.md`。

---

### 串行执行规则

对同一条 night-assist 流程里的控制动作，必须严格串行：

1. 上一个命令返回后，才能发下一个命令
2. 不要把多条 `xiaodu` 或 `xiaodu-iot` 命令并发打出去
3. 如果前一个动作失败，后一个动作直接继续执行，不要让整个流程因为单点失败而中断

原因：夜间模式对“稳定”和“低打扰”要求高，并发容易带来误控和打断。

---

### 失败处理

#### 典型降级

| 失败场景 | 降级行为 |
|---------|---------|
| 没有 scene | 自动退到设备级 fallback |
| 找不到路径灯 | 至少开一盏最接近的柔和灯 |
| 某个灯失败 | 继续尝试后面的可用灯 |
| 色温/亮度不支持 | 只做开灯 |
| 没有智能屏 | 仍完成灯光动作，不假装播报已成功 |
| 自动收尾脚本不可用 | 明确说明这次不会自动关 |

---

### 实际调用链参考

对于高层夜间请求，默认实际链路是：

1. `list_scenes.sh`
2. 如果有 night scene，就 `trigger_scene.sh`
3. 否则 `list_iot_devices.sh`
4. 然后做一个或多个安全的 `control_iot.sh`（夜灯 / 床边灯 / 走廊灯 / 卫生间灯）
5. `list_devices.sh`
6. `speak.sh`
7. 如明确需要，再做轻微温控
8. `night_light_tail.sh` 做自动收尾

这能保证整条编排链是分层且显式的。
