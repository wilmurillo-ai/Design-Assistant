# usage-notes.md

## 观影模式使用说明

### 这个 skill 在什么情况下使用

当用户说以下表达时，激活此 skill：

- 开始观影模式
- 我要看电影
- 把房间调成适合看电影的状态
- 准备看电影了
- 进入影院模式
- 打开家庭影院
- 帮我安排观影

这些表达有一个共同特征：用户想要一个“可以开始看电影”的整体状态，而不是指定某个具体影片或平台。

---

### 观影模式和 bedtime / wake-up 的最大区别

bedtime 和 wake-up 都会默认带内容层（睡前故事 / 晨起儿歌）。

**观影模式默认不带内容层。**

因为：
- 看电影是高度个性化的行为
- 选错平台或内容很容易惹恼用户
- 用户说“看电影”时，通常意图是“准备环境”而不是“马上开始播”

所以观影模式的默认理解是：
> **先把房间和设备准备好，用户说要看了再继续内容层。**

---

### 推荐的最简 fallback 顺序（无 scene 时）

1. 把灯光调暗或关闭
2. 如果有窗帘就关闭
3. 如果用户明确说闷 / 热 / 冷，再考虑空调 / 风扇
4. 打开或准备电视 / 投影
5. 调整音量
6. 切到合适的信号源（如有必要）
7. 做一句简短确认播报
8. 如果用户明确指定了内容，再继续进入内容播放层

### 默认观影主流程优先级

1. 环境调整（灯光 / 窗帘）
2. 显示设备准备（电视 / 投影）
3. 音量 / 信号源 / 播放模式
4. 简短确认播报
5. 内容层（仅在用户明确指定时继续）

---

### 面向用户的标准汇报口径

推荐口径（默认准备完成）：

> 已经帮你把观影模式安排好了。
> 我刚刚把灯光收暗了，房间也调整得更适合看电影，电视这边也已经准备好。
> 现在已经可以开始看了，如果你想指定电影或者平台，我接着帮你切过去。

备选口径（有部分降级）：

> 我已经先帮你把观影环境准备好了。灯光已经调整好，不过当前没找到合适的投影目标；现在电视这边已经可以开始看了。

---

### 极简确认示例

#### 需要确认的情况
用户说“我要看电影”，但同时有电视和投影在线。

可以问：
> 用电视还是投影？

#### 不需要确认的情况
只有一个合理目标设备，或用户已经明确说了“开电视”“开投影”。

---

### 偏好沉淀示例

用户在观影过程中说：
- “以后看电影都用电视，音量中等”
- “窗帘不用关，留一点光”
- “以后不用问，直接切 HDMI”

这些都应该被吸收为正式偏好，写入 `XIAODU_CONTEXT.md`。

---

### 串行执行规则

对同一条观影流程里的控制动作，必须严格串行：

1. 上一个命令返回后，才能发下一个命令
2. 不要把多条 `xiaodu` 助手命令并发打给同一台设备
3. 不要把同一条观影流程里的 IoT 控制并发打给设备链路
4. 如果前一个动作失败，后一个动作直接继续执行，不要让整个流程因为单点失败而中断

这条规则同时适用于：
- `xiaodu` 智能屏控制
- `xiaodu-iot` 家电控制

原因：在当前小度链路里，并发很容易让前一条指令被打断，或者让设备解析 / 控制阶段变得不稳定。

---

### 失败处理

#### 典型降级

| 失败场景 | 降级行为 |
|---------|---------|
| 没有观影 scene | 自动退到设备级 fallback |
| 没有灯 / 窗帘 | 跳过该项，继续设备层 |
| 没有电视 / 投影 | 至少完成环境准备 + 智能屏确认，然后明确告知播放设备未确认 |
| 切信号源失败 | 保持已准备设备的状态，继续其他可做动作 |
| 内容层失败 | 保持在“环境和设备已准备好”的状态，不假装电影已开始播放 |

---

### 实际调用链参考

对于高层观影请求，默认实际链路是：

1. `list_scenes.sh`
2. 如果有观影 scene，就 `trigger_scene.sh`
3. 否则 `list_iot_devices.sh`
4. 然后做一个或多个安全的 `control_iot.sh`（灯光、窗帘、电视/投影、音量、信号源）
5. `list_devices.sh`
6. `speak.sh`
7. 如果用户明确指定内容，再继续 `control_xiaodu.sh` 或进一步 IoT 控制

这能保证整条编排链是分层且显式的。
除非用户明确说要自动播放内容，否则不要跳过环境和设备准备阶段，直接进入内容播放。
