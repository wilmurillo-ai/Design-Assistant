# VERIFICATION_PROTOCOL.md — skill-collision-engine

## 验证目标

确保碰撞引擎真正产生认知涌现而非机械组合，且工厂能自进化。

## 五步自验证流程

### Step 1 — 认知涌现验证（非机械组合）
- [ ] 碰撞输出是否包含"涌现认知"（两个输入Skill都不单独拥有的认知）？
- [ ] 是否明确定义了 FA∩VB 和 FB∩VA 交叉区域的发现过程？
- [ ] 涌现认知是否标注了来源（来自哪个碰撞方向）？
- **通过标准**: 涌现认知占比 ≥30%（非简单复制）；交叉区域发现过程完整

### Step 2 — SOUL 领域适配验证
- [ ] 是否提供 ≥3 个职业的适配示例？
- [ ] 适配后的 Skill 是否保留了 SOUL 五律核心框架？
- [ ] 领域已知/未知是否与职业实际匹配？
- **通过标准**: ≥3示例 + 五律保留 + 领域准确性

### Step 3 — 碰撞矩阵验证
- [ ] 是否列出所有已发布 Skill 的碰撞状态矩阵？
- [ ] 矩阵中已碰撞项是否与实际发布历史一致？
- [ ] 是否标注了待碰撞方向和潜在新 Skill？
- **通过标准**: 矩阵完整 + 历史一致 + 未来方向标注

### Step 4 — 工厂元进化验证
- [ ] 是否定义了进化循环（读取反馈→提取模式→更新参数）？
- [ ] 碰撞参数是否可动态调整（方向权重、领域亲和度、阈值）？
- [ ] 是否通过验证日志驱动优化（非随机调整）？
- **通过标准**: 进化循环完整 + 参数可调 + 日志驱动

### Step 5 — 外部知识源验证
- [ ] 是否支持 ≥3 种外部知识源？
- [ ] 每种知识源是否有明确的碰撞输入格式？
- [ ] 外部碰撞是否也执行四向碰撞推理？
- **通过标准**: ≥3源 + 格式明确 + 四向碰撞
