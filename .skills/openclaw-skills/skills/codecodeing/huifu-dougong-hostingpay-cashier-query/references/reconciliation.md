# 交易结算对账单查询

> 本文依据 2025-11-21 官方文档整理，适用于 `v2/trade/check/filequery`。

## 接口概述

| 属性 | 值 |
|------|-----|
| 请求方式 | `POST` |
| 汇付 API 端点 | `https://api.huifu.com/v2/trade/check/filequery` |
| SDK Request 类 | `V2TradeCheckFilequeryRequest` |
| Content-Type | `application/json` |

说明：

- 支持 JSON 报文。
- 请求和响应整体都需要签名，验签规则见 `huifu-dougong-hostingpay-base/references/tech-spec.md`。
- 官方特别说明：同步返回中的 `file_details` 是 `jsonArray`，对嵌套 JSON 做验签时无需排序。
- 本接口只有同步返回；如果文件还未生成或正在跑批，可通过 `task_details` 查看任务状态。

## 应用场景

- 支持渠道商或商户查询交易、分账、出金、结算、用户结算等对账文件。
- 适用于财务对账、账单下载归档、补拉历史账单和排查跑批状态。
- 调用前需要由运营为渠道商号开通对账单功能。

## SDK 映射

`V2TradeCheckFilequeryRequest` 的独立 setter 字段：

| 字段 | setter | 类型 | 必填 | 说明 |
|------|--------|------|------|------|
| reqDate | `setReqDate()` | String(8) | Y | 本次查询请求日期，`yyyyMMdd` |
| reqSeqId | `setReqSeqId()` | String(128) | Y | 本次查询请求流水号 |
| huifuId | `setHuifuId()` | String(18) | Y | 汇付客户 ID |
| fileDate | `setFileDate()` | String(8) | Y | 文件生成日期，`yyyyMMdd` |

说明：

- 在本 skill 库的 Java SDK 接入实践中，`bill_type` 通过 `setExtendInfo(Map<String, Object>)` 传入更稳妥。
- `file_date` 表示文件生成日期，不应直接等同于“昨天”；如果是调接口补生成，要填写交易日期 `+1` 天，也就是原对账单生成日期。
- 请求字段名固定为 `file_date`；“文件生成日期”只是中文释义，不要在 DTO、Controller 或前端参数里改成 `bill_date`、`generate_date` 等别名。

### 字段命名约束

- 请求字段名固定为 `file_date`。
- “文件生成日期”只是中文含义，不是建议把字段改成 `bill_date`、`generate_date`。
- 生成 Controller、DTO、前端请求参数或 OpenAPI 描述时，优先直接暴露 `file_date`。
- 请求字段名固定为 `bill_type`。
- “文件类型”是该字段的中文名，不是建议改成 `file_type` 的意思。
- 生成 Controller、DTO、前端请求参数或 OpenAPI 描述时，优先直接暴露 `bill_type`。
- 除非用户已有存量 API 契约必须兼容，否则不要再做 `file_type -> bill_type` 的二次映射。

## 对账文件生成与下载说明

### 跑批时间

| 文件类别 | 跑批时间 | 建议取文件时间 |
|----------|----------|----------------|
| 交易数据、分账对账文件 | 每日 `03:00` | 产品介绍最新口径建议 `12:00` 后 |
| 出金对账单 | 每日 `10:30` | 建议 `1` 小时后 |
| 结算对账单 | 每日 `17:00` | 建议 `1` 小时后 |

说明：

- 每日跑批生成上一个工作日或自然日的对账文件，按 `T+1 / D+1` 规则生成。
- 接口支持 `1` 年内账单下载，控台下载暂未见时间限制说明。
- 如果 `file_details` 为空，优先查看 `task_details`；若 `task_stat=S` 且没有文件，代表前一日无记录。
- 官方产品介绍 / FAQ 口径说明：常规账单下载链接通常对应压缩文件，压缩包内多为 csv；当单文件记录数超过 `400000` 时，会拆分多个 csv。
- `SETTLE_FUND_BILL` 的官方模板文件是 `.xlsx`，不要把所有账单都按 csv 解析。

### 常见文件名规则

| 文件类型 | 常见文件名 |
|----------|------------|
| `TRADE_BILL` / `TRADE_BILL_MONTH` | `年月日_商户号_order_批次号.csv` |
| `SETTLE_BILL` / `SETTLE_BILL_MONTH` | `年月日_商户号_settle_批次号.csv` |
| `WITHDRAWAL_BILL` | `年月日_商户号_outcash_批次号.csv` |
| `SPLIT_BILL` | `年月日_商户号_split_批次号.csv` |
| `SETTLE_USER_BILL` | `年月日_商户号_settleuser_批次号.csv` |
| `SETTLE_FUND_BILL` | `年月日_商户号_settlefund.xlsx` |

## 业务返回码

| 返回码 | 返回描述 | 处理 |
|--------|----------|------|
| 00000000 | 查询成功 | 继续解析 `file_details` 和 `task_details` |
| 其他 | 参见官方业务返回码页面 | 结合 `resp_desc`、`task_stat` 和请求入参排查 |

## 文档勘误与实现备注

- `huifu_id` 在请求参数表长度写成 `18`，`file_details` 里写成 `32`，`task_details` 里又写成 `16`；官方同页不一致，实现时不要把该字段长度硬编码得过短。
- `file_details` 参数表没有列出 `file_id`，但官方成功示例里出现了 `file_id`；当前 reference 仍以参数表为准，不把它视作稳定字段。
- 官方成功示例里 `file_details.file_Name` 使用了大写 `N`，而参数表字段名写的是 `file_name`；消费返回报文时要兼容大小写不一致的历史数据。
- `task_details.bill_type` 的官方枚举列表没有列出 `SETTLE_FUND_BILL`，但请求参数和 `file_details` 已列出该类型；当前按“请求与文件结果支持、任务列表描述遗漏”处理。
- 官方补充说明里还给出了“全域资金入账明细单”“电子账户资金变动明细文件”等模板，但当前请求参数 `bill_type` 没给出对应枚举值；在没有新增正式枚举前，不要自行扩展请求值。
- 历史版本文档曾把 `file_date` 简化成“查昨天”，这会误导补生成和月账单场景；本文已按官方修正为“文件生成日期”，并补充了补生成时填写“交易日期 `+1` 天”的规则。
- 根据产品介绍补充了更保守的取数建议：交易/分账文件建议 `12:00` 后再下载；如果业务对时效敏感，应优先按 `task_details.task_stat` 判定文件是否就绪。
