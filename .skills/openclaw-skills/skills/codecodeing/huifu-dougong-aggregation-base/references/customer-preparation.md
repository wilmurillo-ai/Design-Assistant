# 聚合支付客户前置准备清单

> 这份文档用于约束聚合支付 skill 在编码前先确认“参数从哪里来”。这里的前置准备不只是收集字段值，还包括官方产品介绍和开发指引里明确要求的业务开通、应用配置、授权绑定和终端采集动作。如果来源不明确，模型不应自行推断或伪造参数值。

## 参数来源分类

| 来源类型 | 典型字段 | 说明 |
|---------|---------|------|
| 汇付平台固定配置 | `sys_id`、`product_id`、`huifu_id`、RSA 密钥 | 由汇付开放平台 / 控台提供 |
| 客户业务配置 | `notify_url`、`fee_flag`、`acct_id`、`channel_no` | 由客户业务侧或控台确认 |
| 前端 / 用户授权结果 | `sub_openid`、`buyer_id`、`buyer_logon_id` | 运行时值，模型不能猜 |
| 终端 / 设备采集 | `auth_code`、`device_ip`、`devs_id`、`customer_ip` | 反扫、终端报备、银联场景常见 |
| 上游订单沉淀 | `req_date`、`req_seq_id`、`hf_seq_id`、`party_order_id` | 查询 / 关单 / 退款必须复用 |

## 全局必备配置

| 配置项 | 用途 | 没有会怎样 |
|-------|------|-----------|
| `sys_id` | 公共请求参数 | 请求无法落地 |
| `product_id` | 公共请求参数 | 汇付会直接报产品号错误 |
| `huifu_id` | 商户主体标识 | 业务请求无法定位商户 |
| RSA 私钥 / 公钥 | 请求签名、响应验签 | 请求或验签失败 |
| `notify_url` / `refund_notify_url` | 异步结果通知 | 只能依赖轮询，不稳定 |
| 对应渠道业务开通状态 | 微信 / 支付宝 / 银联交易前提 | 官方开发指引明确要求先完成商户进件和渠道业务开通 |

## 官方开发指引确认的通用前置动作

- `Lightning_intro.md` 和《快速开始》都明确要求：开发新业务或变更旧业务前，先按产品文档中的“开通功能和准备材料”在合作伙伴控台或商户控台完成业务新增 / 变更。
- 多个渠道开发指引都明确写了：用户前端页面收到支付完成回调，不等于后端可以直接认定交易成功；后端仍需调用查询订单 API 确认最终状态。
- 运行时授权值、扫码值、终端采集值、渠道绑定关系，都是“业务前置准备”的一部分，不是接口层补字段时临时猜出来的。

## 聚合下单前要准备什么

### 所有 trade_type 通用

| 字段 / 配置 | 来源 | 说明 |
|------------|------|------|
| `trade_type` | 业务场景确认 | 决定 `method_expand` 结构 |
| `goods_desc` | 业务订单 | 不是示例值，应该来自真实商品 / 订单语义 |
| `req_seq_id` | 平台流水号生成规则 | 必须当天唯一 |
| `time_expire` | 业务超时策略 | 如不明确可不传，不要乱填过期时间 |
| `acct_split_bunch` | 分账业务配置 | 只有要做分账时才准备 |
| `terminal_device_data` | 终端 / 设备采集链路 | 反扫、终端报备、银联场景常见 |
| `tx_metadata` | 其他扩展能力开关 | 保留为扩展入口，但不要把已确认属于顶层的补贴类字段混塞进去 |

### 微信类场景

| `trade_type` | 必须提前明确的值 | 来源 |
|-------------|------------------|------|
| `T_JSAPI` | `sub_appid`、`sub_openid` | 官方要求先准备微信公众号、开通微信业务、配置支付授权目录；`sub_openid` 必须通过当前公众号 `appid` 的网页授权流程获取 |
| `T_MINIAPP` | `sub_appid`、`sub_openid` | 官方要求先准备微信小程序、开通微信业务、完成微信配置；`sub_openid` 必须通过当前小程序 `sub_appid` 获取，且二者不能错配 |
| `T_APP` | `sub_appid` | 应用配置 |
| `T_MICROPAY` | `auth_code` | 官方要求通过扫码设备实时采集用户付款码；值本身来自用户当次付款码，不应预置到配置中 |

### 支付宝类场景

| `trade_type` | 必须提前明确的值 | 来源 |
|-------------|------------------|------|
| `A_JSAPI` | `buyer_id` 或 `buyer_logon_id` | 官方要求先开通支付宝业务；`buyer_id` 必须通过支付宝 `user_id` 获取流程拿到，不能猜 |
| `A_NATIVE` | 视业务决定是否传门店、商品、营销扩展 | 客户业务配置 |
| `A_MICROPAY` | `auth_code` | 官方要求通过扫码设备实时采集用户支付宝付款码 |

### 银联类场景

| `trade_type` | 必须提前明确的值 | 来源 |
|-------------|------------------|------|
| `U_JSAPI` | `user_id`、`customer_ip`，官方说明 `qr_code` 也要明确 | 官方要求先开通银联业务；`user_id` 需先经“网页授权获取 `auth_code` -> 调获取银联用户标识接口”获得，`customer_ip` 必须来自真实客户端 |
| `U_NATIVE` | 是否需要 `front_url`、`payee_info` | 客户业务配置 |
| `U_MICROPAY` | `auth_code`，以及是否需要 `pid_info` | 官方要求通过扫码设备实时采集用户云闪付付款码；`pid_info` 来自服务商配置 |

## 渠道级前置准备矩阵

| 场景 | 客户开发前必须完成什么 | 关键运行时值 |
|------|----------------------|-------------|
| 微信公众号支付 | 准备公众号、完成商户进件并开通微信业务、配置公众号支付授权目录 | `sub_openid` |
| 微信小程序支付 | 准备小程序、完成商户进件并开通微信业务、完成微信配置、确认 `sub_appid` 绑定关系 | `sub_openid` |
| 支付宝 JS 支付 | 完成商户进件并开通支付宝业务 | `buyer_id` / `buyer_logon_id` |
| 银联 JS 支付 | 完成商户进件并开通银联业务、准备银联网页授权回调链路 | `user_id`、`customer_ip` |
| 各类付款码反扫 | 准备扫码枪或终端采集链路 | `auth_code` |

## 扩展字段相关准备项

| 对象 | 需要客户先准备什么 | 说明 |
|------|------------------|------|
| `acct_split_bunch` | 分账接收方 `huifu_id`、账户号、比例 / 金额规则 | 未准备好不要让模型硬拼分账串 |
| `terminal_device_data` | `device_ip`、`devs_id`、定位 / 设备指纹 | 反扫和报备终端场景很关键 |
| `combinedpay_data` | 补贴方 `huifu_id`、`acct_id`、金额 | 请求顶层字段，属于补贴业务配置，不可猜 |
| `combinedpay_data_fee_info` | 手续费承担方 `huifu_id`、`acct_id` | 请求顶层字段，需要真实承担方信息 |
| `trans_fee_allowance_info` | 补贴手续费金额和活动来源 | 请求顶层字段，需要明确的活动或配置支持 |

## 查询 / 关单 / 退款前要沉淀什么

| 接口 | 开发前必须保证已保存 | 说明 |
|------|------------------|------|
| 交易查询 | `req_date`、`req_seq_id`、`hf_seq_id`、`party_order_id` 中至少一组 | 不能等到查询时再猜 |
| 交易关单 | 原交易 `org_req_date` + `org_req_seq_id` 或 `org_hf_seq_id` | 依赖原交易标识 |
| 关单查询 | 关单请求自身标识 + 原交易标识 | 两层流水都要可追溯 |
| 退款 | `org_hf_seq_id`、`org_party_order_id`、`org_req_seq_id` 三选一 | 原交易定位键来自上游沉淀 |
| 退款查询 | 原退款请求标识、原交易标识 | 便于轮询确认 |
| 对账单查询 | 对账功能开通状态、`file_date` 语义 | 功能未开通时接口也不可用 |

## 权限 / 开通项检查

| 能力 | 影响点 |
|------|--------|
| 分账权限 | `acct_split_bunch` |
| 延迟入账权限 | `delay_acct_flag` |
| 退款权限 | 退款接口 |
| 终端报备 | `terminal_device_data.devs_id` |
| 手续费 / 贴息 / 补贴配置 | `fee_flag`、`combinedpay_data`、`combinedpay_data_fee_info`、`trans_fee_allowance_info` |
| 对账单功能 | `file_date`、`bill_type` |
| 微信小程序 `sub_appid` 绑定 | `T_MINIAPP` 下单成功率 | 官方 QA 明确要求 `sub_appid` 与商户建立绑定关系 |
| 微信 `sub_appid` / `sub_openid` 一致性 | `T_MINIAPP` / `T_JSAPI` | 官方 QA 明确要求 `sub_openid` 必须从对应 `sub_appid` 获取 |

## 向客户索取材料的最小清单

### 必需

- `sys_id`、`product_id`、`huifu_id`
- RSA 私钥、公钥
- 支付 / 退款异步通知地址
- 实际要接的 `trade_type` 列表

### 按场景补充

- 微信：公众号 / 小程序应用信息、`sub_appid`、`sub_openid` 获取链路
- 支付宝：`buyer_id` / `buyer_logon_id` 的真实获取链路
- 银联 JS：网页授权回调地址、`auth_code -> user_id` 获取链路、`customer_ip`
- 反扫：`auth_code` 获取方式、终端采集能力
- 银联扩展：`front_url`、`payee_info`、`pid_info`
- 分账 / 设备 / 补贴：`acct_split_bunch`、`devs_id`、补贴账户配置

## 给模型的硬约束

- 运行时授权值、扫码值、报备值、控台配置值，都不能靠模型猜。
- 前端页面回调、支付完成页、客户端 success 回调，都不能直接当作交易成功终态；按官方开发指引，后端必须再查单确认。
- 如果客户没有提供场景必需值，应该先暴露缺口，而不是直接给出“看起来完整”的代码。
- 查询、关单、退款代码必须复用上游订单沉淀的标识，不要在下游重新假设。
