# 分步确认执行规范

本文档是分步确认机制的唯一权威规范。工作流文件中的确认步骤均以本文档为准。

---

## 核心要点（5 条）

1. **MUST 展示实际内容** — 将占位符替换为真实分析结果，不能保留 `[此处展示...]`
2. **MUST 等待明确回复** — 暂停工作流，真正等待用户回复，不能假设确认或自动继续
3. **MUST 修改后再确认** — 用户提出修改后，调整内容，再次发送确认请求，才能进入下一步
4. **MUST 包含下一步预告** — 明确告知用户确认后将进入哪个步骤
5. **优先级高于「避免阻塞」** — 分步确认是本 skill 的核心机制，不受「避免阻塞任务」建议影响

---

## 确认消息必须包含的要素

```
✅ 步骤X完成：[步骤名称]

[实际产出内容摘要或完整内容]

请确认以上内容是否准确完整，或提出修改意见。确认后我将继续下一步：[下一步名称]。
```

---

## 常见错误与正确做法

### ❌ 错误 1：只告知「已完成」，不展示实际内容

```markdown
> 步骤1已完成，请确认是否继续？
```

**问题**：用户无法知道具体完成了什么，无法有效确认。

**✅ 正确**：展示实际分析结果（见 `guides/confirmation-templates.md` 示例）。

---

### ❌ 错误 2：保留占位符，不填充实际内容

```markdown
> [此处展示上述产出的核心内容]
```

**问题**：占位符没有被替换，用户看不到实际产出。

**✅ 正确**：将占位符替换为本步骤实际产出的摘要或完整内容。

---

### ❌ 错误 3：确认前就开始下一步骤

```
1. 完成步骤 1
2. 发送确认请求
3. 立即开始执行步骤 2 的任务  ← 错误！
4. 等待用户回复
```

**✅ 正确**：发送确认请求 → 等待用户回复 → 收到确认后才开始步骤 2。

---

### ❌ 错误 4：修改后不再次确认

```
1. 步骤 1 确认请求
2. 用户提出修改
3. 调整内容
4. 直接进入步骤 2  ← 错误！
```

**✅ 正确**：调整内容 → 再次发送步骤 1 确认请求 → 用户确认后 → 才进入步骤 2。

---

## 处理用户反馈

### 情况 1：用户确认无误 ✅
回复示例：「确认」「没问题，继续」「看起来不错」
→ 记录确认，进入下一步骤

### 情况 2：用户提出修改 📝
→ 理解修改要求 → 调整产出 → **再次发送确认请求** → 重复直到满意

### 情况 3：用户要求澄清 ❓
→ 提供解释或补充 → **再次请求确认** → 不能在澄清后直接进入下一步

### 情况 4：用户要求跳过确认 ⏭️
询问：「好的，您希望跳过后续所有步骤的确认，在全部完成后一次性交付吗？还是只跳过本次确认？」
- 全部跳过 → 切换「连续执行模式」，完成所有步骤后一次性交付
- 只跳过当前 → 继续下一步，后续步骤仍需确认

## 执行前自检

发送确认请求前，确认以下事项：
- [ ] 本步骤所有核心任务已完成
- [ ] 已读取 `guides/confirmation-templates.md` 中的对应模板
- [ ] 已将占位符替换为实际产出内容
- [ ] 确认话术清晰专业，包含关键产出摘要
- [ ] 已将模板中 `[ ]` 改为 `[x]`

## 连续执行模式规范

当用户要求跳过所有确认时，按以下规范执行：

### 进入连续模式

进入前必须明确询问：
> "好的，我将完成所有步骤后一次性交付。中间过程中您将无法干预方向。是否确认切换？"

等待用户明确确认后才开始连续执行。

### 连续模式中的执行规则

1. **仍然执行质量检查**：每步完成后对照 `quality-checklist.md` 自检，自动修正发现的问题
2. **记录每步产出摘要**：在执行过程中记录每步核心产出，用于最终一次性展示
3. **遇到阻塞性问题时暂停**：若遇到需要用户决策的情况（如需求本身有矛盾、关键信息缺失），必须中断并告知用户，不能自行假设后继续

### 连续模式的最终交付

完成所有步骤后，一次性展示所有产出摘要，格式如下：

```
✅ 全流程完成，一次性交付

步骤1 产出摘要：[核心内容]
步骤2 产出摘要：[核心内容]
步骤3 产出摘要：[核心内容]
（标准模式继续列步骤4、5）

生成文档：
- [文件名1]
- [文件名2]

请整体确认以上内容是否符合预期，或告诉我需要调整的部分。
```

### 退出连续模式

连续模式仅针对本次会话的所有步骤。最终交付确认后恢复到正常确认模式。
