# 📝 工作即时记录机制

**创建日期：** 2026-03-21  
**教训来源：** 2026-03-21 记忆遗漏事件（Day-1 到 Day-6 创作未记录）

---

## ⚠️ 问题根因

**场景：**
- 白天写了 6 篇文章（Day-1 到 Day-6，约 6 万字）
- 晚上才写 daily log，只记录了公众号发布
- 第二天新会话读取 daily log，不知道白天做了什么
- 误以为 Day-1 到 Day-6 没写，计划重复创作

**根因：**
1. 依赖"事后回忆"记录 — 容易遗漏
2. 没有即时记录机制 — 靠自觉
3. 新会话信息源单一 — 只读 daily log，不查 git

---

## ✅ 解决方案：三层防护

### 第一层：工作后即时记录（最重要）

**使用脚本：** `scripts/log_work.sh`

**用法：**
```bash
# 基本用法
./scripts/log_work.sh "任务描述" "文件路径 1" "文件路径 2" ...

# 示例 1：内容创作后
./scripts/log_work.sh "Day-1 内容创作" "docs/articles/Day-01-为什么选 OpenClaw.md"

# 示例 2：配置完成后
./scripts/log_work.sh "配置 SSH 免密" ".ssh/authorized_keys"

# 示例 3：多个文件
./scripts/log_work.sh "Day-2 内容创作" "docs/articles/Day-02-从零安装 OpenClaw.md" "docs/安装指南.md"
```

**效果：**
- ✅ 自动追加到 `memory/YYYY-MM-DD.md`
- ✅ 自动 git add + commit
- ✅ 输出记录内容预览
- ✅ 全程 10 秒钟

**习惯养成：**
```
完成工作 → 执行脚本 → 记录完成
```

---

### 第二层：新会话双重检查

**AGENTS.md 强制规则：**

```markdown
新会话必读：
1. SOUL.md
2. USER.md
3. memory/YYYY-MM-DD.md（今天 + 昨天）
4. MEMORY.md（主会话）
5. ⚠️ 执行 `git log --oneline --since="today 00:00"` 检查实际工作
```

**检查逻辑：**
```
daily log 记录 + git log 实际 commit = 完整工作记录

如果有 commit 但 daily log 没记录 → 立即补记
```

---

### 第三层：Heartbeat 自动补漏

**HEARTBEAT.md 自动检查：**

```markdown
### 4. 自检：未记录的工作（优先级：高）
- 检查频率：每次 heartbeat（每 2 小时）
- 操作：
  1. 执行 `git log --oneline --since="today 00:00"`
  2. 对比 `memory/YYYY-MM-DD.md`
  3. 如果有 commit 但没记录 → 立即补记
```

## 🔧 技术实现

### **脚本核心逻辑**

```bash
#!/bin/bash
# 1. 解析参数
TASK="$1"
FILES="$@"

# 2. 追加到 daily log
cat >> memory/$(date +%Y-%m-%d).md << EOF
## ${TASK} ($(date +%H:%M))
**文件：** ${FILES}
**状态：** ✅ 已完成
EOF

# 3. 自动 git commit
git add memory/$(date +%Y-%m-%d).md
git commit -m "memory: 记录 ${TASK}"
```

## 💡 经验教训

### **核心原则**

**📝 Write-Through 原则：**
> 学到的瞬间就写入，不等待。

**应用到工作记录：**
> 完成工作后立即记录，不等待。

### **为什么重要？**

1. **记忆不可靠** — 人脑会忘，AI 会话会重启
2. **文件可靠** — 写进文件就永久保存
3. **习惯胜过自觉** — 脚本 + 流程 > 靠记忆
