# 测试用例生成流程详解

## 完整生成流程

### 步骤1：理解输入材料

**必须仔细阅读：**
- 需求文档（如果有）
- 设计文档（如果有）
- 业务规则说明
- 接口文档
- 数据库设计

**识别关键信息：**
- [ ] 所有功能点列表
- [ ] 所有业务规则
- [ ] 所有数据字段及约束
- [ ] 所有状态转换
- [ ] 所有接口定义
- [ ] 所有错误场景

---

### 步骤2：规划测试用例结构

**确定用例类型和数量：**

#### 单点测试用例（TC-XXX）
按以下顺序规划：

1. **查询功能测试**（5-10个）
   - 列表查询（分页、排序、过滤）
   - 详情查询
   - 搜索功能
   - 导出功能

2. **输入校验测试**（10-15个）
   - 必填字段校验
   - 格式校验（邮箱、手机号、日期等）
   - 长度校验（最小、最大）
   - 唯一性校验
   - 特殊字符校验

3. **边界值测试**（5-10个）
   - 数值边界（最大值、最小值、0、负数）
   - 字符串边界（空字符串、最大长度、超长）
   - 日期边界（过去、未来、当前）
   - 集合边界（空列表、单个元素、大量元素）

4. **异常场景测试**（5-10个）
   - 权限不足
   - 数据不存在
   - 并发冲突
   - 网络异常
   - 依赖服务异常

5. **用户体验测试**（3-5个）
   - 加载状态提示
   - 错误提示友好性
   - 操作反馈及时性
   - 界面响应速度

**预计单点测试用例：20-30个**

#### 流程测试用例（LC-XXX）
按业务流程规划：

1. **核心业务流程**（5-10个）
   - 主流程（正常路径）
   - 分支流程（条件分支）
   - 异常流程（错误处理）

2. **跨模块流程**（3-5个）
   - 数据流转
   - 状态同步
   - 事件触发

3. **端到端流程**（2-5个）
   - 完整业务场景
   - 多角色协作
   - 复杂数据计算

**预计流程测试用例：10-20个**

**总计：30-50个用例**

### 步骤3：生成单点测试用例（TC-XXX）

#### 3.1 查询功能测试

**模板：**
```
用例ID: TC-001
测试模块: [模块名称]
测试点: 列表查询-默认查询
测试步骤: 打开[模块]列表页面，不输入任何查询条件，点击查询按钮
预期结果: 展示所有数据，默认按创建时间倒序排序，每页显示10条，支持分页
优先级: P0
```

**覆盖场景：**
- 默认查询（无条件）
- 单条件查询
- 多条件组合查询
- 分页查询（首页、末页、中间页）
- 排序查询（升序、降序）
- 空结果查询

#### 3.2 输入校验测试

**模板：**
```
用例ID: TC-010
测试模块: [模块名称]
测试点: 必填字段校验-[字段名]为空
测试步骤: 打开新增页面，[字段名]留空，其他必填字段填写有效值，点击保存按钮
预期结果: 系统提示"[字段名]不能为空"，数据不保存
优先级: P0
```

**覆盖场景：**
- 必填字段为空
- 格式不正确（邮箱、手机号、身份证等）
- 长度超限（最小长度、最大长度）
- 唯一性冲突
- 特殊字符（SQL注入、XSS）
- 数值范围（负数、超大值）

#### 3.3 边界值测试

**模板：**
```
用例ID: TC-020
测试模块: [模块名称]
测试点: 边界值测试-[字段名]最大值
测试步骤: 打开新增页面，[字段名]输入最大允许值[具体值]，其他字段填写有效值，点击保存按钮
预期结果: 保存成功，数据正确存储
优先级: P1
```

**覆盖场景：**
- 最小值（0、1、最小允许值）
- 最大值（最大允许值）
- 最大值+1（超出范围）
- 空值（null、空字符串）
- 特殊值（负数、小数、科学计数法）

#### 3.4 异常场景测试

**模板：**
```
用例ID: TC-030
测试模块: [模块名称]
测试点: 异常场景-权限不足
测试步骤: 使用无权限的账号登录，尝试访问[功能]页面
预期结果: 系统提示"您没有权限访问此功能"，跳转到首页或提示页
优先级: P0
```

**覆盖场景：**
- 权限不足
- 数据不存在（404）
- 数据已删除
- 并发修改冲突
- 网络超时
- 服务异常（500）

#### 3.5 用户体验测试

**模板：**
```
用例ID: TC-040
测试模块: [模块名称]
测试点: 用户体验-加载状态提示
测试步骤: 打开列表页面，观察数据加载过程
预期结果: 数据加载时显示loading动画，加载完成后显示数据或"暂无数据"提示
优先级: P2
```

**覆盖场景：**
- 加载状态提示
- 错误提示清晰友好
- 操作成功反馈
- 空数据提示
- 操作确认提示

### 步骤4：生成流程测试用例（LC-XXX）

#### 4.1 流程测试的特殊要求

**连续执行模式：**
- 用例之间有数据依赖
- 后续用例基于前面用例的数据
- 测试点标注依赖关系

**示例：**
```
LC-001: 创建团队A（驻点组）
LC-002: 向团队A添加成员张三（累积LC-001）
LC-003: 尝试将张三添加到团队B（驻点组），验证同类型团队约束（累积LC-001、LC-002）
```

#### 4.2 流程测试模板

**主流程：**
```
用例ID: LC-001
测试模块: [模块名称]
测试点: [业务流程]-主流程
测试步骤: 
1. 创建测试数据A（具体数据）
2. 执行操作B（具体操作）
3. 验证结果C（具体验证点）
预期结果: 
1. 数据A创建成功，状态为[状态]
2. 操作B执行成功，触发[事件]
3. 结果C符合预期，数据为[具体值]
优先级: P0
```

**分支流程：**
```
用例ID: LC-005
测试模块: [模块名称]
测试点: [业务流程]-分支流程（累积LC-001）
测试步骤: 
基于LC-001的数据，执行条件分支操作D
预期结果: 
分支逻辑正确执行，数据状态变更为[状态]
优先级: P0
```

#### 4.3 数据计算说明

**预期结果需要说明计算过程：**
```
错误示例：
预期结果: 金额为100元

正确示例：
预期结果: 金额为100元（50元基础费用 + 50元服务费）
```

#### 4.4 场景隔离

**通过维度隔离不同场景：**
- 时间维度：不同日期、不同时间段
- 组织维度：不同机构、不同科室、不同团队
- 角色维度：不同用户、不同权限
- 数据维度：不同数据状态、不同数据类型

### 步骤5：生成CSV文件

#### 5.1 使用Write工具

**必须使用Write工具生成真实的CSV文件：**

```python
# 错误做法：markdown代码块
```csv
用例ID,测试模块...
```

# 正确做法：使用Write工具
Write(
    file_path="测试用例.csv",
    content="﻿用例ID||测试模块||..."  # 注意UTF-8 BOM标记
)
```

#### 5.2 UTF-8 BOM编码

**文件开头必须有BOM标记：**
```
﻿用例ID||测试模块||测试点||测试步骤||预期结果||优先级
```

**BOM标记是 `\ufeff` 字符，在文件开头显示为 `﻿`**

#### 5.3 双竖线分隔符

**使用 `||` 不是逗号：**
```
正确：用例ID||测试模块||测试点
错误：用例ID,测试模块,测试点
```

**原因：**
- 测试步骤和预期结果中可能包含逗号
- 双竖线不会与内容冲突

#### 5.4 文件命名

**建议命名格式：**
```
[模块名称]测试用例.csv
[模块名称]功能测试用例.csv
[日期]-[模块名称]测试用例.csv
```

## 常见问题

### Q1: 用户说"先生成几个示例"怎么办？

**A:** 使用标准话术说明完整生成的必要性：
```
"我理解时间紧迫，但完整的测试用例能避免测试遗漏。
让我生成30-50个用例，覆盖所有场景，这样能确保测试质量。"
```

### Q2: 如何判断用例数量是否足够？

**A:** 按照以下标准：
- 每个功能点至少2-3个用例（正常+异常）
- 每个业务规则至少1个用例
- 每个输入字段至少2个用例（有效+无效）
- 每个边界条件至少1个用例
- 总计至少30-50个用例

### Q3: 流程测试用例如何设计依赖关系？

**A:** 
- 使用"累积LC-XXX"标注依赖
- 确保依赖的用例先执行
- 说明基于哪些数据进行测试
- 预期结果说明数据变化

### Q4: 如何确保UTF-8 BOM编码？

**A:** 
- 在文件内容开头添加 `\ufeff` 字符
- 在Write工具中，内容以 `﻿` 开头
- 生成后可以用文本编辑器验证
