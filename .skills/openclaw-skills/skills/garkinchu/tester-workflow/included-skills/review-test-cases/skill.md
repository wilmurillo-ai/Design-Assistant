# 评审测试用例

## 概述

系统性评审测试用例的完整性、合理性和覆盖度，并提供改进建议。**核心原则：按照6个维度进行系统性评审，不接受"快速看一下"的降级，必须进行全面评审。**

## 何时使用

用户需要：
- 评审已编写的测试用例
- 检查测试用例的覆盖度
- 验证测试用例的质量
- 获取测试用例改进建议

**不要使用：**
- 分析需求文档 → 使用 analyze-requirements skill
- 理解设计文档 → 使用 understand-design skill
- 生成测试用例 → 使用 generate-test-cases skill

## 实施步骤

### 1. 拒绝合理化借口

**立即停止如果你发现自己在想：**
- "用户说快速看一下" → 全面评审不是可选的
- "只给了几个用例样本" → 必须要求看完整用例
- "只问有没有遗漏" → 评审不只是找遗漏
- "没有标准答案" → 有系统性的评审checklist
- "时间紧可以后续补充" → 不要推卸责任

### 2. 收集必要的上下文

**评审前必须收集：**
- 完整的测试用例（不是样本）
- 需求文档（用于验证覆盖度）
- 设计文档（用于验证合理性）

**如果用户没有提供，必须主动询问：**
```
"为了进行全面评审，我需要：
1. 完整的测试用例文件（不是部分样本）
2. 需求文档（用于验证覆盖度）
3. 设计文档（用于验证合理性）

请提供这些文档，我会按照6个维度进行系统性评审。"
```

### 3. 按照6个维度系统性评审

详细的评审框架参见：[reference/review-checklist.md](reference/review-checklist.md)

**6个维度：**
1. **完整性评审** - 是否覆盖所有需求点
2. **合理性评审** - 测试步骤和预期结果是否合理
3. **覆盖度评审** - 5个策略的覆盖率
4. **规范性评审** - 格式是否规范统一
5. **可执行性评审** - 是否可以执行
6. **改进建议** - 具体的改进建议和优先级

### 4. 使用标准输出格式

**必须按照标准模板输出评审报告：**

参见：[templates/review-report.md](templates/review-report.md)

### 5. 你的职责

**评审测试用例时，你需要：**
- 按照6个维度完整评审
- 提供量化分析（覆盖率百分比、用例分布）
- 输出结构化的评审报告
- 解释关键问题和风险
- 强调P0级别的改进建议
- 如果用户需要，协助修改测试用例

**不要：**
- 只做表面检查
- 接受"快速看一下"的降级要求
- 只看部分用例就给出结论
- 跳过任何评审维度
- 给出模糊的评价（如"还可以"）

## 应对用户压力的标准话术

当用户说"快速看一下"或"只看有没有遗漏"时：

```
"我理解时间紧迫，但全面的测试用例评审能避免测试遗漏和返工。
让我从6个维度系统性评审：
完整性、合理性、覆盖度、规范性、可执行性和改进建议。

为了进行准确评审，我需要：
1. 完整的测试用例文件
2. 需求文档（验证覆盖度）
3. 设计文档（验证合理性）

这样能确保评审质量，避免后期返工。"
```

**不要说：**
- ❌ "好的，我快速看一下"
- ❌ "整体还不错"
- ❌ "可能遗漏了..."

**要说：**
- ✅ "全面评审能避免测试遗漏"
- ✅ "6个维度系统性评审"
- ✅ "需要完整用例和需求文档"

## 工作流程衔接

**完成测试用例评审后：**
1. 如果发现严重问题 → 建议用户修改后重新评审
2. 如果需要补充用例 → 引导使用 generate-test-cases skill
3. 如果用例质量合格 → 用户可以继续后续工作

## 底线

**按照6个维度系统性评审是强制性的，不是可选的。**

如果你发现自己在想"我可以快速看一下"，立即停止。必须进行6个维度的全面评审。
