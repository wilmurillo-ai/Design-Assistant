# 测试用例评审6维度检查清单

## 维度1：完整性评审

**目标：** 验证测试用例是否覆盖所有需求点和设计要点

### 检查项

- [ ] 是否覆盖所有需求点？
- [ ] 是否覆盖所有设计要点？
- [ ] 是否有遗漏的测试场景？
- [ ] 功能点是否都有对应的测试用例？
- [ ] 业务规则是否都有验证？

### 检查方法

1. **列出所有需求点**（从需求文档中提取）
2. **列出所有设计要点**（从设计文档中提取）
3. **逐一对照测试用例**，标记已覆盖和未覆盖
4. **计算覆盖率** = 已覆盖数 / 总数 × 100%

### 输出格式

```markdown
**覆盖率：X%**

已覆盖的需求点：
- 需求点1 ✓ (TC-001, TC-002)
- 需求点2 ✓ (TC-003)

未覆盖的需求点：
- 需求点3 ✗ [P0] - 缺少XXX场景的测试用例
- 需求点4 ✗ [P1] - 缺少XXX边界值测试
```

---

## 维度2：合理性评审

**目标：** 验证测试用例的测试步骤和预期结果是否合理

### 检查项

- [ ] 测试步骤是否合理？
- [ ] 预期结果是否明确？
- [ ] 优先级划分是否合理？
- [ ] 测试数据是否合适？
- [ ] 前置条件是否必要？

### 常见不合理问题

**测试步骤不合理：**
- 步骤顺序错误
- 缺少关键步骤
- 步骤过于简略
- 步骤过于冗余

**预期结果不合理：**
- 预期结果模糊（如"操作成功"）
- 预期结果与测试步骤不匹配
- 预期结果缺少验证点
- 预期结果不可验证

**优先级不合理：**
- 核心功能标记为P2
- 辅助功能标记为P0
- 优先级标注不一致

### 输出格式

```markdown
**不合理用例：X个**

| 用例ID | 问题描述 | 优先级 | 修改建议 |
|--------|---------|--------|---------|
| TC-001 | 测试步骤不清晰，缺少具体数据 | P1 | 补充具体的测试数据 |
| TC-005 | 预期结果模糊"操作成功" | P0 | 明确具体的预期值和验证点 |
```

---

## 维度3：覆盖度评审

**目标：** 验证5个覆盖策略的覆盖率是否达标

### 检查项

- [ ] 正常流程覆盖率是多少？（目标100%）
- [ ] 异常流程覆盖率是多少？（目标≥80%）
- [ ] 边界值覆盖率是多少？（目标≥90%）
- [ ] 输入校验覆盖率是多少？（目标100%）
- [ ] 用户体验覆盖率是多少？（目标≥50%）

### 检查方法

1. **分类统计用例**
   - 正常流程用例数
   - 异常流程用例数
   - 边界值用例数
   - 输入校验用例数
   - 用户体验用例数

2. **计算覆盖率**
   - 覆盖率 = 已覆盖场景数 / 总场景数 × 100%

3. **识别缺失场景**
   - 列出未覆盖的场景
   - 标注优先级

### 输出格式

```markdown
| 类型 | 用例数 | 覆盖率 | 目标 | 评价 |
|------|--------|--------|------|------|
| 正常流程 | 15个 | 100% | 100% | ✓ 达标 |
| 异常流程 | 8个 | 75% | ≥80% | ✗ 不达标 |
| 边界值 | 10个 | 90% | ≥90% | ✓ 达标 |
| 输入校验 | 12个 | 100% | 100% | ✓ 达标 |
| 用户体验 | 3个 | 40% | ≥50% | ✗ 不达标 |

**缺失场景：**
- 异常流程：缺少并发冲突场景、数据权限过滤场景
- 用户体验：缺少加载状态提示、空数据提示
```

## 维度4：规范性评审

**目标：** 验证测试用例的格式是否规范统一

### 检查项

- [ ] 用例ID格式是否规范？
- [ ] 测试步骤描述是否清晰？
- [ ] 预期结果是否具体？
- [ ] 优先级标注是否准确？
- [ ] 格式是否统一？

### 常见规范性问题

**用例ID不规范：**
- TC-1（缺少前导零）
- TC001（缺少连字符）
- tc-001（小写）

**测试步骤不规范：**
- 使用序号（1、2、3）
- 没有融入前置条件和测试数据
- 语言不专业（口语化）

**预期结果不规范：**
- 过于简略（"成功"）
- 没有验证点
- 格式不统一

**CSV格式不规范：**
- 使用逗号分隔符（应该用||）
- 编码不是UTF-8 BOM
- 列顺序错误

### 输出格式

```markdown
**不规范用例：X个**

常见问题：
- 用例ID格式不统一：5个（TC-1, TC-10, TC001）
- 测试步骤使用序号：8个
- 预期结果过于简略：12个
- CSV分隔符错误：全部（使用逗号而非||）
```

---

## 维度5：可执行性评审

**目标：** 验证测试用例是否可以执行

### 检查项

- [ ] 测试步骤是否可执行？
- [ ] 测试数据是否明确？
- [ ] 前置条件是否清晰？
- [ ] 依赖关系是否合理？
- [ ] 是否有环境要求说明？

### 常见可执行性问题

**测试步骤不可执行：**
- 步骤过于抽象（"输入数据"）
- 缺少具体操作（"验证结果"）
- 步骤顺序错误

**测试数据不明确：**
- 没有具体数据（"输入有效值"）
- 数据来源不清晰
- 数据依赖关系不明确

**前置条件不清晰：**
- 没有说明前置条件
- 前置条件过于复杂
- 前置条件无法满足

**依赖关系不合理：**
- 流程测试没有标注依赖
- 依赖的用例不存在
- 循环依赖

### 输出格式

```markdown
**不可执行用例：X个**

| 用例ID | 问题描述 | 优先级 | 修改建议 |
|--------|---------|--------|---------|
| TC-010 | 缺少测试数据，"输入有效值"不明确 | P0 | 补充具体测试数据 |
| LC-002 | 前置条件不清晰，依赖LC-001但未说明 | P1 | 明确依赖关系和数据来源 |
```

---

## 维度6：改进建议

**目标：** 基于前5个维度的评审结果，提供具体的改进建议

### 输出格式

```markdown
**P0（必须修改）：X个**
1. 补充缺失的核心功能测试用例（需求点3、需求点4）
2. 修正CSV格式：使用||分隔符，UTF-8 BOM编码
3. 明确所有预期结果，补充验证点

**P1（建议修改）：X个**
1. 补充异常流程用例，提升覆盖率到80%以上
2. 规范用例ID格式，统一为TC-001格式
3. 测试步骤融入前置条件和测试数据

**P2（可选修改）：X个**
1. 补充用户体验用例，提升覆盖率到50%以上
2. 优化测试步骤描述，使用更专业的语言
```

## 评审流程

### 步骤1：准备工作
1. 收集完整的测试用例文件
2. 收集需求文档
3. 收集设计文档

### 步骤2：逐维度评审
1. 完整性评审（30分钟）
2. 合理性评审（20分钟）
3. 覆盖度评审（30分钟）
4. 规范性评审（15分钟）
5. 可执行性评审（20分钟）
6. 改进建议汇总（15分钟）

### 步骤3：输出评审报告
1. 使用标准模板
2. 提供量化分析
3. 标注优先级
4. 评估风险

---

## 质量标准

一份合格的评审报告应该：
- ✅ 覆盖所有6个维度
- ✅ 提供量化分析（覆盖率百分比）
- ✅ 标注问题优先级（P0/P1/P2）
- ✅ 提供具体改进建议
- ✅ 评估风险
- ✅ 给出总体评价
