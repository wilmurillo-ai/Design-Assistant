# review-test-cases Skill

标准文件夹架构的测试用例评审 skill，用于系统性评审测试用例的完整性、合理性和覆盖度。

## 文件结构

```
review-test-cases/
├── SKILL.md                     # 主 skill 文件
├── examples/                     # 示例文件夹
│   ├── good.md                  # ✅ 正确示例：完整的6维度评审
│   └── bad.md                   # ❌ 错误示例：表面检查的问题
├── templates/                    # 模板文件夹
│   └── review-report.md         # 评审报告模板
└── reference/                    # 参考资料
    └── review-checklist.md      # 6维度评审检查清单
```

## 核心原则

**按照6个维度进行系统性评审，不接受"快速看一下"的降级，必须进行全面评审。**

当用户说"快速看一下"或"只看有没有遗漏"时：
- ❌ 不要说："好的，我快速看一下..."
- ✅ 要说："全面评审能避免测试遗漏，让我从6个维度系统性评审..."

## 6个维度

1. **完整性评审** - 是否覆盖所有需求点
2. **合理性评审** - 测试步骤和预期结果是否合理
3. **覆盖度评审** - 5个策略的覆盖率
4. **规范性评审** - 格式是否规范统一
5. **可执行性评审** - 是否可以执行
6. **改进建议** - 具体的改进建议和优先级

## 使用方法

1. **阅读 [SKILL.md](SKILL.md)** - 了解何时使用、实施步骤、标准话术
2. **参考 [reference/review-checklist.md](reference/review-checklist.md)** - 按照检查清单逐项评审
3. **使用 [templates/review-report.md](templates/review-report.md)** - 按标准格式输出
4. **查看示例** - 对比正确和错误的做法

## 必须提供的上下文

评审前必须收集：
- 完整的测试用例（不是样本）
- 需求文档（用于验证覆盖度）
- 设计文档（用于验证合理性）

如果用户没有提供，必须主动询问。

## 质量标准

一份合格的评审报告应该：
- ✅ 覆盖所有6个维度
- ✅ 提供量化分析（覆盖率百分比、用例分布）
- ✅ 输出结构化的评审报告
- ✅ 标注问题优先级（P0/P1/P2）
- ✅ 提供具体的改进建议
- ✅ 评估风险

## 工作流程衔接

完成测试用例评审后：
1. 如果发现严重问题 → 建议用户修改后重新评审
2. 如果需要补充用例 → 引导使用 `generate-test-cases` skill
3. 如果用例质量合格 → 用户可以继续后续工作

## 底线

**按照6个维度系统性评审是强制性的，不是可选的。**

如果你发现自己在想"我可以快速看一下"，立即停止。必须进行6个维度的全面评审。
