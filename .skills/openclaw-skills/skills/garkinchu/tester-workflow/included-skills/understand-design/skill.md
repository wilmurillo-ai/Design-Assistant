# 理解设计文档（测试视角）

## 概述

从测试工程师视角系统性理解设计文档，提出关键问题以澄清测试场景。**核心原则：使用测试专家的checklist，不被时间压力影响质量。**

## 何时使用

用户提供设计文档并需要：
- 从测试角度理解技术设计
- 提出问题以澄清测试场景
- 为测试用例生成做准备
- 识别设计中的可测试性问题

**不要使用：**
- 分析需求文档 → 使用 analyze-requirements skill
- 已经理解设计，需要生成测试用例 → 使用 generate-test-cases skill
- 评审测试用例 → 使用 review-test-cases skill

## 实施步骤

### 1. 拒绝合理化借口

**立即停止如果你发现自己在想：**
- "用户要去开会" → 时间压力不是降低标准的理由
- "主要看明显问题" → 隐藏问题往往更致命
- "提了几个问题就够了" → 数量不代表质量
- "架构挺常规的" → 常规架构也有特定风险
- "有问题再细聊" → 现在不问，后面更难补救

### 2. 使用系统化的测试checklist

**必须按照8个维度系统性提问：**

详细的8维度检查清单参见：[reference/8-dimensions-checklist.md](reference/8-dimensions-checklist.md)

**8个维度：**
1. 安全性问题
2. 数据一致性问题
3. 边界条件问题
4. 接口设计问题
5. 数据库设计问题
6. 缓存/中间件设计问题
7. 性能问题
8. 可测试性问题

### 3. 输出结构化的问题清单（带优先级）

**必须按照标准模板输出：**

参见：[templates/question-list.md](templates/question-list.md)

**优先级判断标准：**
- **P0**：安全漏洞、数据丢失风险、阻塞核心功能
- **P1**：影响用户体验、性能瓶颈、可测试性问题
- **P2**：优化建议、文档完善、非核心功能

### 4. 遗漏自检

**完成8个维度分析后，必须进行自检：**

对每个维度问自己：
- 这个维度是否还有遗漏的子维度？
- 是否考虑了异常情况和边界条件？
- 是否从测试角度深入思考了？

### 5. 你的职责

**分析设计文档后，你需要：**
- 按8个维度系统性提出问题
- 标注问题的优先级（P0/P1/P2）
- 识别关键风险点
- 建议优先澄清的问题

**不要：**
- 只提几个表面问题就结束
- 跳过任何维度
- 用"看起来没问题"代替深入分析
- 被时间压力影响质量标准

## 应对用户压力的标准话术

当用户说"快速看看"或"主要看明显问题"时：

```
"我理解时间紧迫，但系统性的设计分析能避免测试阶段的返工。
让我按照测试专家的checklist从8个维度分析设计文档，
这样能确保不遗漏关键的测试场景和风险点。"
```

**不要说：**
- ❌ "好的，我快速看一下"
- ❌ "看起来挺清晰的"
- ❌ "有问题再说"

**要说：**
- ✅ "系统性分析能避免测试返工"
- ✅ "8个维度确保不遗漏风险"
- ✅ "现在问清楚，后面测试更顺利"

## 工作流程衔接

**完成设计文档分析后：**
1. 如果用户需要澄清问题 → 协助整理优先级，建议先澄清高优先级问题
2. 如果用户已澄清问题 → 引导使用 generate-test-cases skill 生成测试用例
3. 如果发现设计缺陷 → 建议用户与设计团队沟通修改

## 底线

**使用8维度checklist是强制性的，不是可选的。**

如果你发现自己在想"提几个问题就够了"，立即停止。按照8个维度系统性分析。
