# understand-design Skill

标准文件夹架构的设计文档理解 skill，用于从测试视角系统性理解设计文档，提出关键问题以澄清测试场景。

## 文件结构

```
understand-design/
├── SKILL.md                     # 主 skill 文件
├── examples/                     # 示例文件夹
│   ├── good.md                  # ✅ 正确示例：完整的8维度分析
│   └── bad.md                   # ❌ 错误示例：表面问题的问题
├── templates/                    # 模板文件夹
│   └── question-list.md         # 问题清单模板
└── reference/                    # 参考资料
    └── 8-dimensions-checklist.md # 8维度检查清单
```

## 核心原则

**使用测试专家的checklist，不被时间压力影响质量。**

当用户说"快速看看"或"主要看明显问题"时：
- ❌ 不要说："好的，我快速看一下..."
- ✅ 要说："系统性分析能避免测试返工，让我按照8个维度分析设计文档..."

## 8个维度

1. **安全性问题** - 认证/授权/加密/防护/审计
2. **数据一致性问题** - 约束/并发/事务/关联
3. **边界条件问题** - 状态转换/异常/边界/超时
4. **接口设计问题** - 格式/错误码/限流/版本
5. **数据库设计问题** - 索引/扩展/软删除/性能
6. **缓存/中间件设计问题** - 策略/失效/降级/Key设计
7. **性能问题** - 指标/瓶颈/压测/资源
8. **可测试性问题** - 环境/数据/Mock/自动化

## 使用方法

1. **阅读 [SKILL.md](SKILL.md)** - 了解何时使用、实施步骤、标准话术
2. **参考 [reference/8-dimensions-checklist.md](reference/8-dimensions-checklist.md)** - 按照检查清单逐维度分析
3. **使用 [templates/question-list.md](templates/question-list.md)** - 按标准格式输出
4. **查看示例** - 对比正确和错误的做法

## 质量标准

一份合格的问题清单应该：
- ✅ 覆盖所有8个维度
- ✅ 每个问题标注优先级（P0/P1/P2）
- ✅ 提供时间盒方案（5分钟/15分钟/完整版）
- ✅ 识别关键风险点
- ✅ 建议优先澄清的问题

## 工作流程衔接

完成设计文档分析后：
1. 如果用户需要澄清问题 → 协助整理优先级，建议先澄清高优先级问题
2. 如果用户已澄清问题 → 引导使用 `generate-test-cases` skill 生成测试用例
3. 如果发现设计缺陷 → 建议用户与设计团队沟通修改

## 底线

**使用8维度checklist是强制性的，不是可选的。**

如果你发现自己在想"提几个问题就够了"，立即停止。按照8个维度系统性分析。
