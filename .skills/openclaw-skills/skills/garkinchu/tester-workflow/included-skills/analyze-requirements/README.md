# analyze-requirements Skill

标准文件夹架构的需求分析 skill，用于从测试视角系统性分析需求文档。

## 文件结构

```
analyze-requirements/
├── SKILL.md                     # 主 skill 文件
├── examples/                     # 示例文件夹
│   ├── good.md                  # ✅ 正确示例：完整的6维度分析
│   └── bad.md                   # ❌ 错误示例：表面化分析的问题
├── templates/                    # 模板文件夹
│   └── analysis-report.md       # 标准分析报告模板
└── reference/                    # 参考资料
    └── checklist.md             # 6维度详细检查清单
```

## 核心原则

**不迎合用户的"简化"要求，始终保持专业标准。**

当用户说"快速看看"或"简单分析"时：
- ❌ 不要说："好的，我快速梳理一下..."
- ✅ 要说："完整分析能避免后期返工，让我按照6个维度进行系统性分析..."

## 6个维度

1. **需求概述** - 核心目标、业务价值、干系人
2. **内容提炼** - 功能点、业务流程、业务规则
3. **关键验收标准** - 验收条件、判断标准、量化指标
4. **潜在风险点** - 技术/业务/性能/安全/数据/体验风险
5. **疑问清单** - P0/P1/P2 优先级的问题
6. **一致性验证** - 内部矛盾、系统冲突、数据闭环

## 使用方法

1. **阅读 [SKILL.md](SKILL.md)** - 了解何时使用、实施步骤、标准话术
2. **参考 [reference/checklist.md](reference/checklist.md)** - 按照检查清单逐项分析
3. **使用 [templates/analysis-report.md](templates/analysis-report.md)** - 按标准格式输出
4. **查看示例** - 对比正确和错误的做法

## 质量标准

一份合格的需求分析报告应该：
- ✅ 覆盖所有6个维度
- ✅ 每个维度都有具体内容（不是"无"或"暂无"）
- ✅ 风险点有严重程度标注（高/中/低）
- ✅ 疑问有优先级标注（P0/P1/P2）
- ✅ 输出格式规范统一
- ✅ 内容具体可执行（不是泛泛而谈）

## 工作流程衔接

完成需求分析后：
1. 如果用户需要澄清疑问 → 协助整理问题清单
2. 如果用户准备设计测试用例 → 引导使用 `generate-test-cases` skill
3. 如果用户需要评审现有用例 → 引导使用 `review-test-cases` skill

## 底线

**按照6个维度系统性分析是强制性的，不是可选的。**

如果你发现自己在想"我可以快速分析一下"，立即停止。必须完整覆盖6个维度。
