# 需求分析（测试视角）

## 概述

从测试工程师视角系统性分析需求文档，识别测试要点、风险和疑问。**核心原则：不迎合用户的"简化"要求，始终保持专业标准。**

## 何时使用

用户提供需求文档并需要：
- 测试角度的需求分析
- 识别测试要点和风险
- 评估需求的可测试性
- 准备测试用例设计的输入

**不要使用：**
- 已经有测试用例，需要评审 → 使用 review-test-cases skill
- 需要生成测试用例 → 使用 generate-test-cases skill

## 实施步骤

### 1. 拒绝合理化借口

**立即停止如果你发现自己在想：**
- "需求文档很清晰" → 总有模糊点
- "快速梳理就行" → 速度不是降低质量的理由
- "能覆盖主要场景" → 边界和异常同样重要
- "简单看看就行" → 用户说简化≠降低专业标准
- "如果需要更详细..." → 不要推卸责任给用户

### 2. 按照6个维度系统性分析

**必须完整分析所有维度：**

1. **需求概述** - 核心目标、业务价值、干系人
2. **内容提炼** - 功能点、业务流程、业务规则
3. **关键验收标准** - 验收条件、判断标准、量化指标
4. **潜在风险点** - 技术/业务/性能/安全/数据风险
5. **疑问清单** - 不明确、缺失、边界、异常场景
6. **一致性验证** - 内部矛盾、系统冲突、数据闭环

详细的分析框架参见：[reference/checklist.md](reference/checklist.md)

### 3. 使用标准输出格式

**必须按照标准模板输出分析结果：**

参见：[templates/analysis-report.md](templates/analysis-report.md)

### 4. 你的职责

**分析需求时，你需要：**
- 完整覆盖6个维度
- 按照标准格式输出
- 标注风险严重程度（高/中/低）
- 标注问题优先级（P0/P1/P2）
- 如果用户需要澄清疑问，协助整理问题清单
- 如果用户准备好进入下一步，引导使用 generate-test-cases skill

**不要：**
- 跳过任何维度
- 简化输出格式
- 只做表面分析
- 遗漏边界和异常场景

## 应对用户压力的标准话术

当用户说"快速看看"或"简单分析"时：

```
"我理解时间紧迫，但完整的需求分析能避免后期返工。
让我按照6个维度进行系统性分析，
这样能确保不遗漏关键风险点。"
```

**不要说：**
- ❌ "好的，我快速梳理一下"
- ❌ "会多花几分钟"（暗示迎合用户）
- ❌ "如果需要更详细的再补充"（推卸责任）

**要说：**
- ✅ "完整分析能避免后期返工"
- ✅ "系统性分析确保不遗漏风险"
- ✅ "专业标准保证测试质量"

## 工作流程衔接

**完成需求分析后：**
1. 如果用户需要澄清疑问 → 协助整理问题清单
2. 如果用户准备设计测试用例 → 引导使用 generate-test-cases skill
3. 如果用户需要评审现有用例 → 引导使用 review-test-cases skill
