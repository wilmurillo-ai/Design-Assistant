## 需求分析报告

### 1. 需求概述

**核心目标：**
为团队管理系统提供成员管理功能，支持添加、删除成员，并确保同类型团队的成员唯一性约束。

**业务价值：**
- 规范团队成员管理流程
- 避免人员重复分配（同类型团队）
- 支持跨类型团队的灵活分配

**主要干系人：**
- 产品经理：张三
- 开发团队：系统管理模块组
- 测试团队：功能测试组
- 目标用户：医院管理员、科室主任

---

### 2. 内容提炼

#### 功能点清单

**主功能：**
- 功能点1：查看团队成员列表（支持分页、搜索）
- 功能点2：添加团队成员（从待添加列表选择）
- 功能点3：删除团队成员（单个删除）
- 功能点4：搜索成员（按姓名、工号）

**辅助功能：**
- 功能点5：显示成员基本信息（姓名、工号、科室）
- 功能点6：区分已添加和待添加成员

#### 业务流程

**主流程：添加成员**
```
1. 管理员打开团队详情页
2. 点击"添加成员"按钮
3. 系统展示待添加成员列表（过滤已加入同类型团队的成员）
4. 管理员选择成员并确认
5. 系统校验业务规则
6. 保存成功，刷新成员列表
```

**分支流程：**
```
- 如果选择的成员已加入同类型团队，则提示错误
- 如果选择的成员已加入不同类型团队，则允许添加
- 如果待添加列表为空，提示"暂无可添加成员"
```

**异常流程：**
```
- 当网络异常时，提示"网络连接失败，请稍后重试"
- 当保存失败时，提示"保存失败，请重试"并回滚操作
- 当权限不足时，提示"您没有权限执行此操作"
```

#### 关键业务规则

**数据校验规则：**
- 规则1：成员必须是系统中已存在的用户
- 规则2：同一成员不能重复添加到同一团队

**业务约束条件：**
- 约束1：一个人只能加入一个同类型的团队（驻点组或治疗组）
- 约束2：一个人可以同时加入不同类型的团队
- 约束3：删除成员时，该成员的个人项目绑定保留

**权限控制规则：**
- 角色A（系统管理员）：可以管理所有团队的成员
- 角色B（科室主任）：只能管理本科室团队的成员
- 角色C（团队负责人）：只能管理自己负责的团队成员

### 3. 关键验收标准

**功能完整性：**
- [ ] 成员列表正确展示所有已添加成员
- [ ] 待添加列表正确过滤已加入同类型团队的成员
- [ ] 添加成员功能正常工作
- [ ] 删除成员功能正常工作
- [ ] 搜索功能正确返回结果

**数据准确性：**
- [ ] 同类型团队约束正确生效
- [ ] 不同类型团队可以共享成员
- [ ] 删除成员后数据正确更新
- [ ] 成员信息展示准确

**性能标准：**
- [ ] 成员列表加载时间 < 2秒
- [ ] 搜索响应时间 < 1秒
- [ ] 添加/删除操作响应时间 < 1秒

**用户体验：**
- [ ] 操作流程简洁（不超过3步）
- [ ] 错误提示清晰明确
- [ ] 列表支持分页（每页10条）
- [ ] 搜索支持模糊匹配

### 4. 潜在风险点

#### 技术风险

- **风险1：同类型团队约束的实现复杂度**
  - 严重程度：中
  - 影响范围：添加成员功能、待添加列表过滤
  - 建议措施：需要在前端和后端都进行校验，确保数据一致性

- **风险2：数据权限过滤的性能问题**
  - 严重程度：中
  - 影响范围：待添加列表查询
  - 建议措施：需要优化SQL查询，考虑添加索引

#### 业务风险

- **风险1：团队类型变更时的成员处理逻辑不明确**
  - 严重程度：高
  - 影响范围：如果团队类型从"驻点组"改为"治疗组"，现有成员如何处理？
  - 建议措施：需要明确业务规则，是否允许修改团队类型

- **风险2：删除团队时成员的处理逻辑**
  - 严重程度：中
  - 影响范围：团队删除后，成员是否自动释放？
  - 建议措施：需求文档已说明"删除团队后，团队项目绑定删除，个人项目保留"，但成员关系是否删除需要确认

#### 数据一致性风险

- **风险1：并发添加成员时的数据冲突**
  - 严重程度：中
  - 影响范围：多个管理员同时添加同一成员到不同同类型团队
  - 建议措施：需要在数据库层面添加唯一约束，并处理并发冲突

### 6. 一致性验证

#### 发现的矛盾/冲突

- **矛盾1：需求文档中"所属上级"字段的可编辑性**
  - 位置1：团队管理章节说"修改团队时只读"
  - 位置2：成员管理章节未说明是否可以通过修改上级来间接修改团队归属
  - 冲突内容：如果团队的上级不可修改，那么成员的团队归属是否也不可修改？
  - 影响：可能影响成员管理的业务逻辑
  - 建议：明确成员的团队归属是否可以通过"删除+重新添加"来实现

#### 数据流验证

- **数据来源：** 系统用户表（SysUser）
- **数据流转：** 用户 → 团队成员关系表（SysTeamUser）→ 团队（SysTeam）
- **数据存储：** SysTeamUser 表存储团队ID和用户ID的关联关系
- **数据展示：** 成员列表展示用户的基本信息（姓名、工号、科室）
- **闭环验证：** 数据流完整，但需要确认删除团队时成员关系的处理

#### 改进建议

1. **建议1：增加批量操作功能**
   - 理由：提高操作效率，特别是新建团队时需要添加多个成员
   - 预期效果：减少操作步骤，提升用户体验
   - 优先级：中

2. **建议2：增加操作日志记录**
   - 理由：便于追溯成员变更历史，满足审计需求
   - 预期效果：提高系统可追溯性
   - 优先级：低
