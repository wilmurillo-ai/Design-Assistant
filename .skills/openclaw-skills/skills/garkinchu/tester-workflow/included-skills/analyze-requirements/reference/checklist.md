# 需求分析6维度检查清单

## 维度1：需求概述

**必须回答的问题：**
- [ ] 这个需求的核心目标是什么？（一句话总结）
- [ ] 这个需求解决什么业务问题？
- [ ] 业务价值是什么？
- [ ] 主要干系人有哪些？（产品、开发、测试、运营、用户）

**输出要求：**
- 用一句话清晰总结需求核心
- 明确业务价值和目标用户

---

## 维度2：内容提炼

### 功能点清单
**必须做到：**
- [ ] 列出所有功能点（不遗漏任何一个）
- [ ] 区分主功能和辅助功能
- [ ] 标注功能优先级（如果需求文档有）

### 业务流程
**必须梳理：**
- [ ] 主流程（正常路径）
- [ ] 分支流程（条件分支）
- [ ] 异常流程（错误处理）
- [ ] 流程中的角色和权限

### 关键业务规则
**必须识别：**
- [ ] 数据校验规则
- [ ] 业务约束条件
- [ ] 计算公式
- [ ] 状态转换规则
- [ ] 权限控制规则

---

## 维度3：关键验收标准

**必须明确：**
- [ ] 功能完整性标准（所有功能都实现）
- [ ] 数据准确性标准（计算、存储、展示正确）
- [ ] 性能标准（响应时间、并发量）
- [ ] 用户体验标准（易用性、流畅度）
- [ ] 兼容性标准（浏览器、设备、系统）

**输出格式：**
```
- [ ] 验收标准1：具体可验证的条件
- [ ] 验收标准2：具体可验证的条件
```

---

## 维度4：潜在风险点

### 技术实现风险
**检查项：**
- [ ] 技术方案是否可行？
- [ ] 是否依赖第三方服务？
- [ ] 是否有技术债务？
- [ ] 是否需要数据迁移？
- [ ] 是否有性能瓶颈？

### 业务逻辑风险
**检查项：**
- [ ] 业务规则是否完整？
- [ ] 是否有逻辑漏洞？
- [ ] 边界条件是否考虑？
- [ ] 异常场景是否覆盖？
- [ ] 是否有业务冲突？

### 性能风险
**检查项：**
- [ ] 大数据量场景是否考虑？
- [ ] 高并发场景是否考虑？
- [ ] 是否有性能指标要求？
- [ ] 是否需要性能优化？

### 安全风险
**检查项：**
- [ ] 是否有权限控制？
- [ ] 是否有数据加密？
- [ ] 是否有SQL注入风险？
- [ ] 是否有XSS风险？
- [ ] 是否有敏感数据泄露风险？

### 数据一致性风险
**检查项：**
- [ ] 数据流是否闭环？
- [ ] 是否有数据冗余？
- [ ] 是否有数据同步问题？
- [ ] 是否有事务处理？
- [ ] 是否有数据回滚机制？

### 用户体验风险
**检查项：**
- [ ] 操作流程是否繁琐？
- [ ] 错误提示是否友好？
- [ ] 是否有加载等待提示？
- [ ] 是否有操作反馈？

**输出格式：**
```
**技术风险：**
- 风险描述 [严重程度：高/中/低] [影响范围] [建议措施]

**业务风险：**
- 风险描述 [严重程度：高/中/低] [影响范围] [建议措施]
```

---

## 维度5：疑问清单

### P0（必须澄清）
**识别标准：**
- 核心业务规则不明确
- 关键流程缺失
- 数据定义模糊
- 验收标准不清晰

**示例问题：**
- [ ] 当XX条件时，系统应该如何处理？
- [ ] XX字段的取值范围是什么？
- [ ] XX操作的权限控制规则是什么？

### P1（建议澄清）
**识别标准：**
- 边界条件未定义
- 异常场景未说明
- 提示信息不明确
- 交互细节缺失

**示例问题：**
- [ ] 当XX字段为空时，如何展示？
- [ ] XX操作失败时，错误提示是什么？
- [ ] XX列表为空时，如何提示用户？

### P2（可选澄清）
**识别标准：**
- 优化建议
- 用户体验改进
- 非核心功能细节

**示例问题：**
- [ ] 是否需要添加XX快捷操作？
- [ ] 是否需要优化XX交互流程？

**输出格式：**
```
**P0（必须澄清）：**
- [ ] 问题描述？[涉及模块] [影响范围]

**P1（建议澄清）：**
- [ ] 问题描述？[涉及模块] [影响范围]

**P2（可选澄清）：**
- [ ] 问题描述？[涉及模块] [影响范围]
```

---

## 维度6：一致性验证

### 需求内部一致性
**检查项：**
- [ ] 不同章节描述是否一致？
- [ ] 业务规则是否有矛盾？
- [ ] 数据定义是否统一？
- [ ] 流程描述是否一致？

### 与现有系统一致性
**检查项：**
- [ ] 是否与现有功能冲突？
- [ ] 是否影响现有数据？
- [ ] 是否需要修改现有接口？
- [ ] 是否需要数据迁移？

### 数据流一致性
**检查项：**
- [ ] 数据来源是否明确？
- [ ] 数据流转是否完整？
- [ ] 数据存储是否合理？
- [ ] 数据展示是否一致？

**输出格式：**
```
**发现的矛盾/冲突：**
- 矛盾描述：[位置1] vs [位置2] [影响] [建议]

**建议：**
- 建议1：具体的改进建议
- 建议2：具体的改进建议
```

---

## 使用说明

1. **按顺序检查**：从维度1到维度6依次检查
2. **不跳过任何维度**：即使某个维度看起来"没问题"
3. **记录所有发现**：不要因为"太多"而省略
4. **标注优先级**：帮助用户聚焦关键问题
5. **提供建议**：不只是发现问题，还要给出解决方向

## 质量标准

一份合格的需求分析报告应该：
- ✅ 覆盖所有6个维度
- ✅ 每个维度都有具体内容（不是"无"或"暂无"）
- ✅ 风险点有严重程度标注
- ✅ 疑问有优先级标注
- ✅ 输出格式规范统一
- ✅ 内容具体可执行（不是泛泛而谈）
