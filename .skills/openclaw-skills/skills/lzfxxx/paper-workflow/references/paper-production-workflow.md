# Workflow

## Scope

Use this workflow when the user wants to turn existing project assets into a submission-ready paper.

## SOP

1. Confirm the venue constraints.
   - CFP or submission page
   - paper type
   - page limit
   - template
   - key dates
2. Inventory available assets.
   - code and repository structure
   - technical documentation
   - experiment logs or metrics
   - pilot data, interviews, or case notes
   - screenshots and existing figures
   - prior abstracts, slides, or internal summaries
3. Normalize the assets.
   - create module notes when source material is fragmented
   - separate core modules from extensions
   - separate real evidence from background material
4. Freeze the paper boundary.
   - choose one main contribution line
   - define what is in scope and out of scope
   - define evidence level per claim
5. Produce 2-3 abstract options.
   - let the user or decision maker choose one
   - do not expand the full paper before the direction is chosen
6. Expand to outline.
   - section purpose
   - claim per section
   - evidence per section
   - figure per section
7. Draft the manuscript.
   - use `academic-research-writer` for abstract, intro, related work, method, discussion, and conclusion
   - maintain a concise review-language version when stakeholders need it
8. Plan and build figures.
   - architecture
   - workflow or sequence
   - state machine
   - result tables or evidence screenshots
9. Run quality gates.
   - use `paper-validator`
   - fix claim-evidence mismatch before polishing prose
10. Compress and package.
   - reduce non-core content first
   - keep figures aligned with claims
   - fix metadata and declarations
11. Submit and archive.
   - update status docs
   - freeze submission version
   - archive receipt and final files

## Output Expectations

Typical outputs include:

- abstract options
- outline
- manuscript draft
- figure list
- review issue list
- submission checklist
- status update
