# Pay — x402 Details

Most of the time, `pay request <url>` handles everything. This
reference is for when you need to understand what's happening under
the hood or when something goes wrong.

## How `pay request` works

```
1. Agent runs: pay request https://api.example.com/data
2. CLI sends GET to the URL
3. If response is 200 -> return response (not paywalled)
4. If response is 402 -> CLI reads PAYMENT-REQUIRED header
5. CLI determines settlement mode (direct or tab)
6. For direct: CLI signs EIP-3009 authorization, retries with payment
7. For tab: CLI checks for open tab with provider
   - If none: CLI opens a tab (10x per-call price, minimum $5)
   - If exists: charges existing tab
   CLI retries with tab charge proof
8. Provider verifies via facilitator -> serves content
9. CLI returns the API response
```

Steps 4-8 are handled by the CLI. The agent just gets the response.

## Settlement modes

The provider declares which settlement mode to use in their 402
response:

### Direct settlement (`settlement: "direct"`)

- On-chain payment per request
- For calls priced at $1+
- Standard x402 — works with any x402-compatible agent SDK
- Higher latency (~2s for on-chain confirmation)

### Tab settlement (`settlement: "tab"`)

- Charges against a pre-funded tab
- For any price point, especially micropayments
- Pay-specific — requires the Pay CLI or SDK
- Low latency (~50ms, off-chain verification)
- Batched settlement (~$0.000006 effective gas per charge)

The CLI handles both modes transparently based on the 402 response.

## Non-Pay facilitators

Not all 402 responses use the Pay facilitator. The x402 ecosystem
includes other facilitators (e.g., from Coinbase).

If `pay request` encounters a 402 pointing to a different facilitator:
- The CLI cannot handle it
- Report to the operator: "This endpoint uses a different x402
  facilitator. The Pay CLI only works with the Pay facilitator
  (pay-skill.com/x402)."
- The operator may need a different x402 client for that endpoint

## The Pay facilitator

```
URL:     https://pay-skill.com/x402
Testnet: https://testnet.pay-skill.com/x402

Endpoints:
  POST /x402/verify     Verify payment (off-chain, instant)
  POST /x402/settle     Execute on-chain settlement
  GET  /x402/supported  List supported schemes/networks
```

The CLI calls these automatically. The agent never calls them directly.

## pay-gate

pay-gate is the provider-side reverse proxy that generates 402
responses and verifies payments. The agent never interacts with
pay-gate directly — it's transparent. The agent just sees a 402
from the API URL.

For provider setup docs: pay-skill.com/docs/gate

## Wire format

x402 V2 headers (the only version Pay supports):

| Header | Direction | Purpose |
|--------|-----------|---------|
| `PAYMENT-REQUIRED` | server -> agent | Base64 JSON payment requirements |
| `PAYMENT-SIGNATURE` | agent -> server | Payment proof |
| `PAYMENT-RESPONSE` | server -> agent | Settlement receipt |

V1 headers (`X-PAYMENT`, `X-PAYMENT-RESPONSE`) are not supported.

## `pay request` options

Mirrors curl:

```
pay request <url>                    GET request (default)
  -X POST                           HTTP method
  -H "Authorization: Bearer ..."    Headers (repeatable)
  -d '{"query": "..."}'             Request body
  -d @file.json                     Body from file
  -o output.json                    Write response to file
  -v                                Show request/response headers
  -s                                Body only (no status line)
  --no-pay                          Skip payment handling (debug)
```
