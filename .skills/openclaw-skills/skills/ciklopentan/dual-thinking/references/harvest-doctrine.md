# Harvest Doctrine
#tags: skills review

This file is non-runtime guidance for local capability harvest in `dual-thinking`.
It does not override the executable runtime law in `SKILL.md`.

## Purpose
Keep capability harvest real, evidenced, and subordinate to the runtime contract.
Use this file for examples, preservation rationale, and corpus-scan heuristics.
Do not treat it as a second runtime source of truth.

## What counts as a harvested quality
A harvested quality is valid only if all are true:
- it was actually observed in the local skill corpus
- it is reusable beyond a one-off project quirk
- it can strengthen the current artifact against a specific failure mode, validation gap, recovery seam, operator-risk seam, or publish-risk seam

## What does not count
- decorative wording
- branding voice
- project-specific noise
- vague admiration
- qualities that were imagined rather than evidenced

## Preservation policy
Capability-harvest learnings are additive by default at the review-memory level, but they are not all mandatory runtime law.
If a previously harvested pattern proves weak, redundant, or wrongly generalized, narrow or replace it explicitly rather than silently pretending it never existed.

## Good harvest examples
- importing a one-fix-one-verify rule from `node-connect` to reduce speculative recovery loops
- importing installed-vs-packaged split awareness from `openclaw-memory-canonical` to avoid false live-behavior claims
- importing exact symptom-to-command recovery routing from the orchestrator skills to reduce fuzzy repair prose

## Anti-patterns
- scanning the local corpus and then proposing nothing specific
- naming broad virtues without tying them to a concrete seam
- turning every observed good idea into mandatory runtime law
- keeping stale doctrine in `SKILL.md` after its runtime value has been compressed into a smaller binding rule
