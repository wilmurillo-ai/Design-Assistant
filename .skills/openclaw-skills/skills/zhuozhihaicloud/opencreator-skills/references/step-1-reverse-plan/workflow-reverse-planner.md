# 🧠 Skill：工作流结构反推器（Format Reverse Planner）

## 1. Skill 定义

从目标内容（尤其是视频）出发，反向推导出：

- **宏观 Format（内容生产结构）**
- **各模块之间的依赖关系（Dependency Graph）**

核心能力：

> 把“我要一个什么视频” → 推导出“这个视频必须由哪些模块组成，以及它们如何依赖”

---

## 2. 核心输出（唯一目标）

该 skill 只输出两件事：

### 2.1 宏观 Format

即：

```text
内容生产结构 = 模块组成 + 模块顺序 + 分支关系
```

例如：

- 单段结构
- 多分镜结构
- 双分支结构（画面 / 音频）
- 共享语义结构

### 2.2 模块依赖关系

即：

```text
A → B → C
```

表示：

- B 依赖 A 的输出
- C 依赖 B 的输出

最终形成一个 DAG（有向无环结构）。

---

## 3. 核心方法

只做一件事：

> 从结果往前推，一层一层拆依赖。

### Step 1：定义最终产物（Root）

先明确最终输出是什么：

- 视频（是否有声音？是否有人？是否多镜头？）
- 多图
- 单图
- 音频

### Step 2：拆一层依赖（First Layer）

问：

> 这个结果“直接由什么组成”？

例如：

```text
视频 = 画面 + 音频
```

### Step 3：继续往前推（Recursive）

对每个组成部分继续问：

> 这个东西从哪里来？

例如：

```text
音频 → 需要脚本
脚本 → 需要语义输入

画面 → 需要视觉输入
视觉 → 需要生成或已有素材
```

### Step 4：直到链路闭合

终止条件：

- 所有最底层输入，要么来自用户
- 要么可以由“纯文本 / 基础输入”生成

如果不满足，则标记为**缺失依赖**。

### Step 5：整理成结构

最终输出：

1. **宏观结构（Format）**
2. **依赖链（Dependency Graph）**

---

## 4. 标准输出格式

### 4.1 宏观结构（Format）

```text
输入
↓
语义层
↓
├── 视觉分支
└── 音频分支
↓
合成层
↓
输出
```

### 4.2 依赖链（Dependency Graph）

```text
输入 → 语义 → 脚本 → 音频
输入 → 视觉 → 视频
→ 合成
```

---

## 5. 模块抽象要求

在这个 skill 中：

### 5.1 只允许输出抽象模块

例如：

- 语义层（理解 / 脚本）
- 视觉层（人物 / 场景）
- 音频层（语音）
- 合成层

### 5.2 不允许输出具体实现细节

不得出现：

- 工具名称
- 节点名称
- 模型名称
- 参数细节

这个 skill 只负责**结构推理**，不负责实现选择。

---

## 6. 关键原则（必须遵守）

### 原则 1：先结构，后实现

- ❌ 不考虑怎么生成
- ✅ 只考虑结构是否完整

### 原则 2：每个模块必须有来源

任何模块都必须能回答：

> 它的输入从哪里来？

### 原则 3：必须形成闭环

最终所有路径必须落到：

- 用户输入
- 或基础可生成输入

### 原则 4：输出必须是 DAG

输出的结构必须满足：

- 无环
- 有序
- 可执行

### 原则 5：缺失输入必须显式暴露

如果某条链路无法闭合，必须直接标注：

- 缺什么输入
- 属于哪一层
- 是否必须由用户补充

---

## 7. 最小通用结构模板

适用于绝大多数营销视频：

```text
输入
↓
语义层
↓
├── 视觉分支
└── 音频分支
↓
合成层
↓
输出
```

如果是更复杂的长视频或多镜头内容，则自然扩展为：

```text
输入
↓
语义层
↓
分镜结构层
↓
├── 多段视觉分支
└── 音频分支（可单段或多段）
↓
合成层
↓
输出
```

## 8. 典型反推示意

### 示例 1：口播视频

```text
最终产物：一个口播视频

反推：
视频
→ 画面 + 音频

音频
→ 脚本
→ 语义输入

画面
→ 人物 / 场景
→ 视觉输入

最终结构：
输入
↓
语义层
↓
├── 视觉分支
└── 音频分支
↓
合成层
↓
输出
```

### 示例 2：多镜头营销视频

```text
最终产物：一个多镜头营销视频

反推：
视频
→ 多段画面 + 音频

多段画面
→ 分镜结构
→ 各段视觉输入

音频
→ 脚本 / 分段语义

最终结构：
输入
↓
语义层
↓
分镜结构层
↓
├── 多段视觉分支
└── 音频分支
↓
合成层
↓
输出
```

---

## 9. 这个 Skill 不负责什么

这个 skill 不负责：

- prompt 编写
- 模型选择
- 节点实现
- 工具调用
- 最佳实践 format 匹配
- 具体 workflow 落图

它只负责：

> 从目标内容中恢复出“内容生产结构（format）”以及“模块依赖关系”

---

## 10. 适用场景

用于：

- Workflow Planner 前置推理
- Format 匹配前结构分析
- 内容生产路径设计
- 营销视频反推分析
- 子 agent 的结构层 reasoning

---

## 11. 输出要求（给下游）

必须输出：

- 清晰的结构层级
- 完整依赖链
- 无缺失节点（或显式标注缺失输入）
- 一个可供后续 format 匹配使用的抽象结构

不得输出：

- 工具名称
- 模型名称
- 参数细节
- UI / 节点层实现建议

---

## 12. 从抽象结构到具体 Generator（桥接层）

反推完成后，需要将抽象模块映射为具体的 Generator。以下是标准映射关系：

### 语义层（理解 / 分析 / 脚本生成）

| 用户提供的素材 | 对应 Generator |
|-------------|---------------|
| 只有文本 | 参考文本生文器 |
| 有图片（产品图 / 参考图） | 参考图生文器 |
| 有参考视频 | 参考视频生文器 |
| 多种模态混合 | 多模参考生文器 |

以上全部映射到 `textGenerator`，区别在于连接了哪些输入 Pin。

### 分镜结构层（拆分 / 列表态启动）

| 场景 | 对应 Generator |
|------|---------------|
| 需要批量分镜 / 多图 / 多段 | 故事板分文器（`scriptSplit`） |

> 一旦接入 `scriptSplit`，下游自动进入列表态。

### 视觉分支（图像生成）

| 条件 | 对应 Generator |
|------|---------------|
| 无参考图，纯文本生成 | 文本意图生图器（`imageMaker`） |
| 有参考图，单次融合生成 | 图片参考生图器（`imageToImage`） |
| 有参考图 + 列表态文本（广播） | 故事板分镜生图器（`imageToImage`，广播模式） |
| 需要改光照 | 打光生图器（`relight`） |
| 需要改视角 | 换角度生图器（`imageAngleControl`） |

### 视频分支（视频生成）

| 条件 | 对应 Generator |
|------|---------------|
| 单图→单段视频 | 图生视频器（`videoMaker`） |
| 单图 + 列表态文本（广播）→ 多段视频 | 参考图分镜生视频器（`videoMaker`，广播模式） |
| 分镜图列表 + 列表态文本（对齐）→ 多段视频 | 故事板分镜生视频器（`videoMaker`，对齐模式） |
| 需要口播（图+音频） | 图片对口型生视频器（`imageAudioToVideo`） |
| 需要动作迁移（图+动作视频） | 动作模仿生视频器（`klingMotionControl`） |
| 已有视频需修改 | 视频修改生视频器（`videoToVideo`） |

### 音频分支

| 条件 | 对应 Generator |
|------|---------------|
| 文本→语音 | 文本转语音生音频器（`textToSpeech`） |
| 有音色参考→克隆语音 | 音色转语音生音频器（`voiceCloner`） |
| 需要背景音乐 | 音乐生音频器（`musicGenerator`） |

### 合成层 / 视频终端

合成层的实现取决于是否需要口播以及精度要求：

| 场景 | 实现方式 | 说明 |
|------|---------|------|
| 口播带货（展示商品）| **videoMaker + 音画同步模型（Sora 2 优先）** | 脚本含台词，模型直接渲染口型，无需单独音频。**默认优先方式** |
| 口播（纯人像 + 无特殊要求）| **videoMaker + 音画同步模型** | 同上，不轻易走 lip-sync |
| 口播（必须逐字台词精度 或 指定音色）| `imageAudioToVideo` 或 `videoLipSync` | 需上游 TTS / voiceCloner 提供音频 |
| 非口播视频 | `videoMaker` / `textToVideo` | 普通图生视频或文生视频 |

> **原则：不轻易用 lip-sync 融合路径。** 支持音画同步的 videoMaker 模型（Sora 2、Kling 3.0）可以直接从脚本文本渲染口型动作，链路更短、商品展示效果更好。

> **完成映射后**，查阅 `generator-routing.md` 获取完整的 node.type 映射、决策树和结构模式详情。

---

## 13. 一句话总结

```text
反推结构 = 从最终结果出发，逐层拆依赖，直到所有输入可获得
```
