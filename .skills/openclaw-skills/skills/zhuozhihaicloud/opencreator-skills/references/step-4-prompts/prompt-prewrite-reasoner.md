# 🧠 Skill：提示词撰写前置判断器（Prompt Placement & Pre-Write Reasoner）

## 1. Skill 定义

在真正编写 prompt 之前，先判断以下问题：

- 这个原子是否**已经从上游继承了一套结构化 prompt / 结构化描述**
- 当前这个原子是否**还需要再写 prompt**
- 如果需要写，prompt 应该**写在哪里**
- 当前原子应该承接的是：
  - 用户原始输入
  - 上游生成结果
  - 还是 agent 新补充的控制信息

核心能力：

> 在写 prompt 之前，先判断“要不要写、写什么、写到哪里”，避免重复、冲突和上下文污染。

---

## 2. 这个 Skill 解决什么问题

在复杂 workflow 里，prompt 很容易出现以下问题：

- 上游已经有结构化脚本，当前原子又重复写一遍
- 图片/视频输入已经提供主体与场景，当前 prompt 又把主体与场景重写一遍
- 用户原始输入、本轮 agent 补充、上游脚本三者混在一起，造成 prompt 冗余
- 不知道 prompt 应该写在当前原子的 creative instruction，还是应该只保留在上游文本原子里

这个 skill 的作用就是在 prompt 写作之前，先做一次**输入职责审计（input responsibility audit）**。

---

## 3. 核心判断目标

该 skill 必须回答 4 个问题：

### 3.1 当前原子是否已经有上游结构化 prompt / 结构化描述可继承
如果有，优先复用，不要重复改写。

### 3.2 当前原子是否真的还缺信息
只有缺信息时，才补 prompt。

### 3.3 缺的是哪一层信息
例如：
- 主体
- 环境
- 动作
- 情绪
- 机位
- 运镜
- 灯光
- 风格
- voiceover

### 3.4 如果要补，补在哪里
优先写在当前原子的 `creative instruction`，而不是把所有信息重新写回输入原子。

---

## 4. 核心规则（必须遵守）

### 规则 1：先判断上游是否已经传来结构化 prompt
如果上游已经有：

- 结构化脚本
- 分镜脚本
- 图像描述
- 视频描述
- 可直接消费的 prompt blocks

那么当前原子必须先判断：

```text
这些内容是否已经足够控制当前生成？
```

如果答案是“足够”，则：

```text
❌ 当前原子不再重复写 prompt
```

否则会造成重复撰写、控制冲突和结果失稳。

---

### 规则 2：prompt 只补缺失，不重复已知
当前原子需要的不是“重新写一遍”，而是：

```text
检查全集 → 找缺口 → 只补缺口
```

例如：
- 上游已有主体、环境、脚本
- 当前只缺动作和运镜

则当前 prompt 只写：
- 动作
- 运镜

不重写主体和环境。

---

### 规则 3：如果需要 agent 写 prompt，一般写在当前原子的 creative instruction
这是最重要的落地规则。

原因：
- 当前原子的 `creative instruction` 才是这个原子的直接控制层
- 上游 input 原子应尽量保持“原始素材 / 原始描述”的职责纯净
- 不应把 agent 的推理性补充内容污染到用户原始 input 中

所以：

```text
需要补 prompt → 一般写在当前原子的 creative instruction
```

---

### 规则 4：纯 input 原子只承接原始输入，不承接 agent 二次创作
例如：
- 产品参考
- 人像参考
- 场景参考
- 商品链接解析内容
- 用户直接输入的脚本或描述

这些 input 原子的职责通常是：

```text
承接原始需求 / 原始素材
```

而不是承接 agent 的再编排内容。

所以通常不应：
- 在 input 原子里追加复杂控制指令
- 把 agent 生成的补充 prompt 回写到原始输入原子

---

## 5. Prompt 撰写前的标准判断流程

### Step 1：识别当前原子的输入来源
先判断当前原子接的是什么：

- 用户原始 input
- 上游结构化文本
- 上游图像 / 视频 / 音频
- 多模混合结果

---

### Step 2：识别当前原子所需的控制全集
对于当前原子，明确它理论上需要哪些维度。

例如：

#### 生图原子常见全集
- 主体外貌
- 环境
- 动作 / 表情
- 景别
- 机位
- 构图
- 灯光
- 风格

#### 生视频原子常见全集
- 主体动作 / 情绪
- 环境
- voiceover（内容 + 音色 + 口音）
- 机位
- 运镜

#### 生音频原子常见全集
- 文本内容
- 音色
- 口音 / 语气
- 情绪节奏

---

### Step 3：做缺口审计（Gap Audit）
逐项判断：

```text
这个维度上游已经有了吗？
```

结果分三类：

- 已有且足够 → 不补
- 已有但不够精确 → 轻量补充
- 没有 → 必补

---

### Step 4：决定是否需要写 prompt
只有当前原子仍然缺控制信息时，才需要写 prompt。

判断结果：

```text
如果不缺 → 不写
如果缺 → 进入补写
```

---

### Step 5：决定 prompt 写在哪里
默认规则：

```text
写在当前原子的 creative instruction
```

而不是：
- 改写 input 原子
- 覆盖上游结构化脚本
- 重新生成整份上游描述

---

## 6. 常见应用场景

### 场景 1：生视频前，上游已经给了完整分镜脚本
上游已包含：
- 动作
- 情绪
- 环境
- voiceover
- 机位
- 运镜

则判断结论：

```text
当前视频原子通常不需要再写 prompt
```

最多只做极轻量补充。

---

### 场景 2：生视频前有图片输入，但没有动态描述
此时：
- 主体外貌已有
- 场景可能已有
- 但动作 / 情绪 / 运镜缺失

则判断结论：

```text
只补动作 / 情绪 / 运镜
```

并写在当前视频原子的 `creative instruction`。

---

### 场景 3：生图前已有参考图 + 上游文本描述
此时：
- 主体可能已有
- 场景可能已有
- 缺的是构图 / 景别 / 机位 / 灯光 / 风格

则判断结论：

```text
只补视觉控制维度
```

---

### 场景 4：纯 input 原子
如：
- image input
- text input
- audio input
- video input

这些原子通常只承接原始需求，不承接 agent 补写。

判断结论：

```text
❌ 不在 input 原子里写 agent prompt
✅ 如需补写，写到下游生成原子的 creative instruction
```

---

## 7. 关键原则

### 原则 1：不要把 prompt 当作默认必写项
很多时候最好的 prompt 是：

```text
什么都不补
```

如果上游已经完整。

---

### 原则 2：prompt 是补充层，不是覆盖层
它的职责是：
- 补缺
- 微调
- 控制

不是：
- 重写上游
- 覆盖用户输入

---

### 原则 3：当前原子的 prompt 只服务当前原子
不要试图在某个原子的 `creative instruction` 里承担整条链路的所有控制。

---

### 原则 4：尽量保持原始输入纯净
input 原子越纯，整个链路越稳定、越容易 debug。

---

## 8. 常见错误

### ❌ 上游已有完整脚本，当前原子再写一遍
→ 造成重复控制

### ❌ 把 agent 生成的 prompt 写回 input 原子
→ 污染原始输入

### ❌ 不做缺口审计，默认全量补写
→ prompt 冗余、结果不稳

### ❌ 当前原子缺的是动态信息，却把主体和环境重写一遍
→ 降低参考稳定性

### ❌ 不区分“原始输入”与“控制补充”
→ 画布职责混乱

---

## 9. 一句话总结

```text
提示词撰写前置判断器 = 在写 prompt 之前，先判断上游是否已足够、当前是否真的缺信息，以及补充内容应该写到哪里
```

---

## 10. 定位说明

该 Skill 是：

👉 **Prompt 生成前的输入审计与职责分配层**

它连接：
- 上游：用户原始输入 / 上游结构化结果
- 下游：当前生成原子的 creative instruction

它的价值不是“写 prompt”，而是：

**决定 prompt 是否应该被写、缺什么、以及写在哪里。**
