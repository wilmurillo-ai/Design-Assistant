# 动态核验策略

> 这个文件规定：哪些平台 API 信息可以信本地摘要，哪些信息必须在回答前实时核验。

## 原则

本 skill 采用“静态导航 + 动态核验”模式：

- 静态文件负责：官方入口、常用 base URL、认证方式、最小模板、平台对比骨架
- 动态核验负责：高变动信息，例如模型能力、模型列表、预览状态、价格、上下线变更

## 必须实时核验的信息

以下内容**不要只靠本地 reference 直接下结论**：

1. 最新模型列表
2. 某模型是否支持：
   - 联网搜索
   - tool calling / function calling
   - reasoning / thinking
   - 多模态输入
   - 图像/音频/视频能力
   - embeddings
   - 流式输出
3. 模型是否已下线、废弃、预览转正式
4. 价格、速率限制、地域/区域差异
5. 新增参数、响应字段、endpoint 变化

## 建议的实时核验顺序

当用户问“现在支不支持”“最新模型是什么”“最近有没有变化”这类问题时，按这个顺序：

1. 查 `references/index.md` 找到官方入口
2. 优先访问：
   - 模型列表 / 模型概览页
   - API reference
   - 快速开始 / 功能说明页
   - changelog / 更新日志 / 公告（如果有）
3. 如果文档没有明确写，再保守表达，不要脑补

## 推荐回答方式

### 有明确证据时
- 直接回答结论
- 最好附上“我查的是当前官方文档里的 XX 页面”

### 没有明确证据时
- 明说“我查了当前公开文档，没看到明确说明”
- 建议以控制台模型页或最新 API 文档为准
- 不要把旧摘要当成当前事实

## 写 reference 的规则

- reference 里可以写“常见模型示例”，但不要把它写成“完整且最新模型清单”
- reference 里可以写“当前抓到的页面显示支持 X”，但不要写死为永久事实
- 对高变能力，尽量写成：
  - “以官方模型页/API 文档为准”
  - “回答时建议实时核验”

## 适用问题示例

以下问题默认触发动态核验：

- “Kimi 现在支持联网搜索吗？”
- “DeepSeek 最近多了什么模型？”
- “百炼的 qwen-plus 现在支持 tools 吗？”
- “MiniMax 哪个模型支持图像输入？”
- “旧模型是不是现在支持 reasoning 了？”
