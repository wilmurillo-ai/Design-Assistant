# AI-База знаний PRO — Путь внедрения Пошаговый план от установки до рабочего источника правды для команды.
Не спешить. Каждый этап — фундамент для следующего. --- ## День 1: Установка и первый knowledge-аудит **Время: 2-3 часа** ### Шаг 1: Заполнить config.yaml (30-45 минут) Это самый важный шаг. Чем точнее заполнен config — тем точнее работает система. Обязательные поля:
- Профиль компании (команда, отрасль, стадия)
- Метрики базы знаний (docs_count, search_queries, repeated_questions, onboarding_days)
- Структура категорий и уровни доступа
- Список ролей для онбординга
- Owner'ы по ключевым разделам **Правило:** если не знаете точные цифры — поставьте приблизительные. Лучше примерно, чем пусто. ### Шаг 2: Зафиксировать 5 самых дорогих gap-зон (30 минут) Ответить честно:
1. Какие вопросы команда задаёт каждый день и которые жгут ваше время?
2. Где онбординг занимает больше всего времени и где ошибаются новички?
3. Где конфликтуют регламенты или существуют «зомби»-инструкции?
4. Что невозможно найти без звонка конкретному человеку?
5. Что случится с процессом, если ключевой сотрудник заболеет? Отправить: `аудит базы` — система покажет приоритеты для первого наполнения. ### Шаг 3: Первые записи (60-90 минут) Загрузить 10 самых частых вопросов:
```
добавь в базу FAQ: В: [вопрос] О: [ответ]
``` Создать первый SOP для главного процесса:
```
SOP: [название процесса]
``` **Результат дня 1:** config заполнен, gap-зоны зафиксированы, первые 10 FAQ и 1-2 SOP в базе. --- ## Неделя 1: Наполнение структуры **Ежедневно (30-60 минут):** | День | Что делать | Что получаете |
|---|---|---|
| Пн | Таксономия: категории, теги, уровни доступа | Структура для всех будущих записей |
| Вт | FAQ по команде: топ-20 повторных вопросов | Основа для снятия нагрузки L1 |
| Ср | SOP по 3-5 ключевым процессам | Регламенты в одном месте |
| Чт | Онбординг для 2-3 ролей: день 1 / неделя 1 / месяц 1 | Первый маршрут по роли |
| Пт | Первый аудит базы: `аудит базы` | Что есть, чего не хватает, приоритеты | **Что должно произойти к концу недели 1:**
- 25-30 записей в базе (FAQ + SOP + онбординг)
- Базовая таксономия настроена
- Owner'ы назначены по ключевым разделам
- Первый аудит показал план работы **Чего НЕ нужно делать на неделе 1:**
- Не пытаться загрузить всё подряд — начать с критичного
- Не делать записи формально — лучше 5 живых, чем 50 пустых заглушек
- Не забыть назначить owner'а хотя бы для каждого раздела --- ## Месяц 1: База начинает работать ### Недели 2-4: углубление и чистка **Неделя 2:**
- Собрать FAQ по клиентским обращениям (из поддержки, продаж)
- Настроить поиск: проверить, что важные записи находятся по смыслу
- Провести первые онбординги по новому маршруту — собрать обратную связь **Неделя 3:**
- Запустить `статистика базы` — посмотреть первые no-answer queries
- Добавить записи по top-3 gap-зонам из аналитики
- Проверить устаревшие записи — первый stale-review **Неделя 4:**
- Провести полный `аудит базы` — дубли, пробелы, записи без owner'а
- Оценить: онбординг нового сотрудника по маршруту vs старый хаос
- Зафиксировать первые метрики: сколько повторных вопросов снизилось ### Чек-лист конца месяца 1: - [ ] 50+ активных записей в базе
- [ ] Онбординг по ролям протестирован минимум на 1 новом сотруднике
- [ ] Первые no-answer queries закрыты новыми записями
- [ ] Аудит проведён — дубли и stale-записи убраны
- [ ] Команда использует поиск (не только чат)
- [ ] Owner'ы назначены на каждую категорию
- [ ] Повторные вопросы к сильным сотрудникам заметно снизились --- ## Квартал 1: Источник правды для всей команды ### Месяц 2: зрелость - База стала первой точкой обращения для команды, а не последней
- Новые онбординги идут по маршруту, без ручного сопровождения наставника
- Аналитика показывает реальные gap-зоны — план наполнения системный
- Revision cycle работает: owner'ы обновляют свои разделы по расписанию
- Первый экспорт базы для бота или FAQ на сайте ### Месяц 3: результат - Замер: время онбординга снизилось с 30 до 10 дней
- Замер: количество повторных вопросов в чатах упало на 30-50%
- Знания уже не привязаны к конкретным людям — процессы передаются через базу
- Стратегический аудит: что добавить в следующем квартале, что архивировать **Это ключевой момент.** Первый квартал с настроенной базой знаний — компания видит, где ещё остаются knowledge-gap зоны, но уже с данными, а не с догадками. ### Чек-лист конца квартала 1: - [ ] 100+ активных записей в базе
- [ ] Время онбординга сократилось минимум на 30-50%
- [ ] No-answer queries снизились — команда находит ответы
- [ ] Revision cycle работает — записи обновляются owner'ами
- [ ] 3+ ролей с полными онбординг-маршрутами
- [ ] Экспорт для бота или публичного FAQ настроен
- [ ] CEO / руководитель видит метрику снижения ручной нагрузки --- ## Типичные ошибки первого месяца | Ошибка | Почему плохо | Что делать |
|---|---|---|
| Загрузить всё подряд без структуры | База превращается в свалку, хуже чем было | Начать с категорий и 5 gap-зон |
| Не назначить owner'ов | Записи устаревают, никто не обновляет | Owner назначается при создании каждой записи |
| Делать записи-заглушки | Команда перестаёт доверять базе | Лучше 20 живых записей, чем 200 пустых |
| Не измерять no-answer queries | Не видно, где настоящие дыры | Включить аналитику с первого дня |
| Использовать только поиск, забыть про аудит | Устаревшие записи подрывают доверие | Аудит каждые 4 недели |
| Не обучить команду пользоваться базой | Продолжают спрашивать в чатах | Показать поиск на примере первого же вопроса | --- ## Рекомендуемый ритм (после первого месяца) ```
ЕЖЕДНЕВНО: — Поиск по базе вместо вопроса в чат (команда) — Добавление новых записей по мере появления знаний ЕЖЕНЕДЕЛЬНО (30-60 мин): — Owner'ы проверяют свои разделы — Статистика: no-answer queries и stale-warning ЕЖЕМЕСЯЧНО (1-2 часа): — Полный аудит базы — Обновление устаревших записей — Архивирование мёртвых документов ЕЖЕКВАРТАЛЬНО (2-3 часа): — Стратегический аудит gap-зон — Ревью структуры и таксономии — Метрики: онбординг, повторные вопросы, no-answer
``` --- *AI-База знаний PRO | RAAI | Путь внедрения*
