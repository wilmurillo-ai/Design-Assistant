# AI-Поддержка PRO — Путь внедрения Пошаговый план от установки до управляемого SLA-контура поддержки.
Не спешить. Каждый этап — фундамент для следующего. --- ## День 1: Установка и первый контур SLA **Время: 2-3 часа** ### Шаг 1: Заполнить config.yaml (30-45 минут) Это самый важный шаг. Чем точнее заполнен config — тем точнее работает система. Обязательные поля:
- Каналы поддержки и приоритеты
- SLA по уровням (critical / high / medium / low)
- Матрица эскалации (L2, L3, emergency)
- Автоэскалация (angry client, VIP, legal threat)
- Команда: L1/L2/L3 owner'ы **Правило:** если не знаете точные параметры SLA — поставьте текущие фактические значения. Даже «мы отвечаем за 2 часа» лучше, чем пусто. ### Шаг 2: Зафиксировать топ-5 типов обращений (30 минут) Ответить честно:
1. Какие обращения повторяются чаще всего?
2. Где возникают angry clients (тип проблемы)?
3. Какие тикеты регулярно эскалируются на L2?
4. Где операторы теряют больше всего времени?
5. Что вы хотите уметь автоматизировать первым? Это основа для первых шаблонов и категоризации. ### Шаг 3: Первый тест системы (15 минут) Отправить тестовое обращение: `обработай обращение: [реальный текст типичного тикета]` Увидеть:
- корректность категоризации;
- правильность приоритета;
- маршрут L1/L2/L3;
- качество ответа клиенту. **Результат дня 1:** config заполнен, топ-5 типов зафиксированы, первый тест пройден. --- ## Неделя 1: Настройка контура и тестирование сценариев **Ежедневно (30-60 минут):** | День | Что делать | Что получаете |
|---|---|---|
| Пн | SLA-контур: `SLA отчёт` — проверить текущий статус | Видите базовую картину по очереди |
| Вт | Angry client: тест `клиент злится` + `эскалация клиента` | Правильный тон и маршрут для конфликта |
| Ср | Возвраты: `возврат денег [сумма]` — тест L1/L2 разделения | Знаете лимиты и правила возвратов |
| Чт | Онбординг оператора: `онбординг оператора [имя]` | Стандарт для нового члена команды |
| Пт | Daily report: `отчёт по обращениям` — утвердить формат | Ежедневная картина поддержки | **Что должно произойти к концу недели 1:**
- Все 5 ключевых сценариев протестированы
- SLA-контур работает и предупреждает о рисках
- Angry client и VIP маршруты настроены
- Daily report настроен и понятен руководителю **Чего НЕ нужно делать на неделе 1:**
- Не переписывать все шаблоны сразу
- Не добавлять слишком много исключений и кастомных правил
- Не ожидать идеальных данных — начать с приблизительных --- ## Месяц 1: Система начинает работать ### Недели 2-4: углубление **Неделя 2:**
- Собрать первую статистику: топ-3 категории тикетов, SLA compliance, доля эскалаций
- Пересобрать шаблоны ответов, которые дали плохую оценку
- Проверить, правильно ли работает эскалация по VIP и legal threats **Неделя 3:**
- NPS/CSAT: `NPS отчёт` — первый взгляд на качество сервиса
- Проверить нагрузку L2: кто и что эскалируется туда, можно ли часть вернуть на L1
- Онбординг реального нового оператора — проверить эффективность чек-листа **Неделя 4:**
- Полный ревью месяца: SLA compliance, angry clients, escala rate, NPS динамика
- Выявить главные точки просадки качества
- Сформировать приоритеты на следующий месяц ### Чек-лист конца месяца 1: - [ ] SLA compliance первого ответа ≥ 90%
- [ ] Angry clients эскалируются автоматически, не теряются
- [ ] Daily report читается и используется руководителем
- [ ] NPS/CSAT мониторится — есть baseline
- [ ] Один новый оператор прошёл онбординг по стандарту
- [ ] L2 нагрузка под контролем — есть картина кто и почему эскалирует --- ## Квартал 1: Управляемая поддержка ### Месяц 2: зрелость - Все 10 режимов используются операторами и руководителем
- SLA держится в ритме: первый ответ ≥ 94%, решение ≥ 90%
- Angry clients снизились в 2 раза относительно старта
- NPS/CSAT улучшились — есть управленческие гипотезы что влияет
- Онбординг нового оператора занимает дни, а не недели ### Месяц 3: результат - Поддержка работает как система, а не как набор героических людей
- Руководитель видит узкие места до того, как они бьют по клиентам
- NPS стал инструментом: понятно что снижает качество и что делать
- Стратегический вопрос: где следующая автоматизация **Ключевой момент.** Через квартал руководитель поддержки управляет системой по метрикам, а не тушит пожары вручную. ### Чек-лист конца квартала 1: - [ ] SLA первого ответа ≥ 94% (было ≤ 82%)
- [ ] Публично опасных кейсов снизилось минимум на 50%
- [ ] Доля эскалаций L1→L2 снизилась
- [ ] NPS вырос или стабилизировался
- [ ] Онбординг нового оператора ≤ 5 дней
- [ ] Ежедневный и еженедельный отчёт — часть рабочего ритма --- ## Типичные ошибки первого месяца | Ошибка | Почему плохо | Что делать |
|---|---|---|
| Не настроить L2/L3 owner'ов | Эскалации уходят в никуда | Назначить в первый же день |
| SLA только на бумаге | Нет мотивации держать стандарт | Включить daily SLA report с первого дня |
| Не тестировать angry client сценарий | Опасные кейсы попадают в общую очередь | Обязательно тест на неделе 1 |
| Игнорировать NPS | Не видно, где качество падает | Замерять с первой недели |
| Слишком много кастомных правил | Сложность → ошибки → недоверие системе | Начать с базовых, добавлять постепенно |
| Операторы обходят систему | ROI = 0 | Объяснить выгоду команде, показать отчёты | --- ## Рекомендуемый ритм (после первого месяца) ```
ЕЖЕДНЕВНО (5-10 мин): 20:00 — Daily report: SLA, angry clients, перегруз L2 ПО СОБЫТИЮ: — Входящий тикет → категоризация → маршрут → ответ — Эскалация → уведомление L2/L3 → статус в отчёт ЕЖЕНЕДЕЛЬНО (30-45 мин): Понедельник — Weekly summary: SLA compliance, топ-3 причины эскалаций Пятница — NPS/CSAT ревью: что тянет качество вниз ЕЖЕМЕСЯЧНО (1-2 часа): — Полный ревью: SLA тренды, NPS динамика, доля эскалаций — Ревью шаблонов ответов — что дало плохую оценку — Онбординг ревью: как новые операторы входили ЕЖЕКВАРТАЛЬНО (2-3 часа): — Стратегический ревью качества поддержки — Что автоматизировать следующим — Обновление матрицы эскалации
