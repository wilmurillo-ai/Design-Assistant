# AI-Монитор PRO — Путь внедрения Пошаговый план от установки до полного операционного контура инфраструктуры.
Не спешить. Каждый этап — фундамент для следующего. --- ## День 1: Установка и картирование инфраструктуры **Время: 2-3 часа** ### Шаг 1: Заполнить config.yaml (45-60 минут) Это самый важный шаг. Чем точнее заполнен config — тем точнее работает система. Обязательные поля:
- Список серверов с ролями и критичностью
- Список сервисов с привязкой к бизнес-влиянию
- Стоимость 1 часа простоя (`cost_of_1h_major_downtime_rub`)
- SLA-цель (`sla.default_target`)
- Эскалация по severity **Правило:** если не знаете точную стоимость простоя — поставьте минимальную оценку. Лучше примерно, чем 0. ### Шаг 2: Определить карту критичности (30 минут) Ответить на вопросы:
1. Какой сервис, упав, останавливает выручку немедленно?
2. Какой сервис влияет на всех пользователей vs часть?
3. Кто incident commander при critical incident?
4. Кто CEO получает алерт при SLA breach? Назначить ownership:
```yaml
ownership: incident_commander_role: "CTO" postmortem_owner: "CTO" ceo_visibility_required: true
``` ### Шаг 3: Первый дашборд (15 минут) Отправить: `дашборд` — увидеть формат отчёта и какие данные нужны. **Результат дня 1:** config заполнен, карта критичности готова, дашборд настроен. --- ## Неделя 1: Настройка алертинга и playbook **Ежедневно (30-60 минут):** | День | Что делать | Что получаете |
|---|---|---|
| Пн | Настройка порогов — cpu/ram/disk/response_time | Первые умные алерты |
| Вт | Протестировать playbook: `инцидент: сервис упал` | Видите формат и качество реакции |
| Ср | SLA-контур: `SLA` — проверить буфер и расчёт | Понимаете где риск breach |
| Чт | CEO-отчёт: `отчёт для CEO` — утвердить формат | CEO видит риски без техшума |
| Пт | Утренний брифинг: `утренний отчёт` — первый daily | Техкоманда видит состояние системы | **Что должно произойти к концу недели 1:**
- Критичные алерты настроены и протестированы
- Playbook проверен на 3 типовых сценариях
- SLA-контур показывает буфер
- CEO-отчёт утверждён
- Daily briefing запущен **Чего НЕ нужно делать на неделе 1:**
- Не пытаться настроить все 10 режимов сразу
- Не добавлять слишком много алертов — начать с критичных
- Не ждать идеального config — лучше рабочий, чем идеальный в теории --- ## Месяц 1: Система начинает работать ### Недели 2-4: углубление **Неделя 2:**
- Собрать первую статистику по алертам: что сработало, что шум
- Провести `аудит алертов` — пересобрать шумные пороги
- Первый real incident: пройти по playbook в боевых условиях **Неделя 3:**
- Проверить capacity: `capacity` — где ресурсы под давлением
- Post-mortem на любой incident (даже minor) — выработать привычку
- CEO-отчёт недели — проверить понимание со стороны CEO **Неделя 4:**
- Подвести итоги месяца: MTTR, количество инцидентов, alert noise ratio
- Пересмотреть и обновить пороги по накопленным данным
- Проверить, соблюдается ли SLA в текущем темпе ### Чек-лист конца месяца 1: - [ ] Все критичные сервисы в мониторинге
- [ ] Playbook протестирован на 5+ сценариях
- [ ] Alert noise ratio снизился — меньше бесполезных сигналов
- [ ] Post-mortem проведён минимум 1 раз
- [ ] CEO получает еженедельный нетехнический отчёт
- [ ] Capacity headroom известен
- [ ] MTTR замерен — есть baseline для улучшения --- ## Квартал 1: Полный операционный контур ### Месяц 2: зрелость - Все 10 режимов доступны и используются
- Инженеры работают по playbook, а не импровизируют
- Post-mortem-дисциплина: каждый critical incident → разбор за 48 часов → action items
- SLA управляется проактивно: команда видит буфер, а не узнаёт о breach постфактум
- Capacity planning связан с бюджетом ### Месяц 3: результат - MTTR снизился с 60 до 15 минут (или прогресс по этому направлению)
- Alert noise значительно снизился — команда доверяет алертам
- CEO видит инфраструктурный риск на языке денег
- Стратегический вопрос: где следующие точки хрупкости при росте нагрузки **Ключевой момент.** Через квартал компания управляет инфраструктурой по рискам, а не реагирует на сбои. Это качественно другой уровень операционной зрелости. ### Чек-лист конца квартала 1: - [ ] MTTR снизился минимум на 30%
- [ ] Alert noise ratio < 30%
- [ ] 3+ post-mortem с выполненными action items
- [ ] SLA compliance ≥ цели
- [ ] Capacity plan готов на следующий квартал
- [ ] CEO понимает топ-3 инфраструктурных риска --- ## Типичные ошибки первого месяца | Ошибка | Почему плохо | Что делать |
|---|---|---|
| Не определить стоимость простоя | CEO-отчёт теряет смысл — нет языка денег | Поставить хотя бы минимальную оценку |
| Слишком много алертов с нуля | Alert fatigue появляется быстрее, чем контроль | Начать с 5-10 критичных метрик |
| Post-mortem без action items | Те же инциденты повторяются | Каждый post-mortem = минимум 2-3 action items с owner'ом |
| Скрыть мониторинг от CEO | CEO слеп к бизнес-рискам инфраструктуры | Утвердить формат CEO-отчёта в первую неделю |
| Игнорировать capacity | Неожиданный рост нагрузки → паника | Capacity review — раз в 2 недели | --- ## Рекомендуемый ритм (после первого месяца) ```
ЕЖЕДНЕВНО (5-10 мин): 08:00 — Утренний брифинг для техкоманды ПО СОБЫТИЮ: — Алерт → playbook → эскалация по severity — SLA breach → CEO alert → action plan ЕЖЕНЕДЕЛЬНО (30-45 мин): Пятница — CEO-отчёт по рискам Пятница — Ревью алертов недели + настройка шумных ЕЖЕМЕСЯЧНО (1-2 часа): — Ревью capacity headroom — Post-mortem summary и ревью action items — Обновление карты критичности при изменениях ЕЖЕКВАРТАЛЬНО (2-3 часа): — Capacity plan на следующий квартал — Стратегический ревью инфраструктурных рисков
