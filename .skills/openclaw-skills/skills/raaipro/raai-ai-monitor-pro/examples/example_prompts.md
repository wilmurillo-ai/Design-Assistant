# AI-Монитор PRO — примеры запросов 15 готовых сценариев использования. --- ## 1. Дашборд инфраструктуры Запрос:
```
дашборд
``` Когда использовать: утренняя проверка, после деплоя, при подозрении на проблемы.
Что получите: таблица всех серверов и сервисов с CPU/RAM/Disk/uptime/response time, активные алерты, SLA статус, бизнес-влияние прямо сейчас. --- ## 2. Алерт по превышению порога Запрос:
```
алерт: db-primary RAM 91%, растёт последние 2 часа
``` Когда использовать: мониторинг прислал уведомление, нужен контекст и план действий.
Что получите: оформленный алерт с severity, рекомендациями по диагностике и ссылкой на playbook. --- ## 3. Инцидент — сервис упал Запрос:
```
инцидент: payment-service не отвечает с 14:32, HTTP 503
``` Когда использовать: сервис упал, нужен пошаговый план восстановления.
Что получите: Сценарий 3.1 — что проверить, как диагностировать, как восстановить, кого уведомить. --- ## 4. Инцидент — база данных медленная Запрос:
```
инцидент: PostgreSQL медленная, запросы висят по 8-15 секунд
``` Когда использовать: тайм-ауты в приложении, пользователи жалуются на тормоза.
Что получите: Сценарий 3.2 — SQL запросы для диагностики блокировок и долгих запросов, план исправления. --- ## 5. SSL сертификат истекает Запрос:
```
инцидент: SSL сертификат api.example.com истекает через 12 дней
``` Когда использовать: получили алерт об истечении сертификата.
Что получите: Сценарий 3.3 — команды для проверки, обновления и настройки автопродления. --- ## 6. SLA отчёт за месяц Запрос:
```
SLA отчёт за апрель
``` Когда использовать: конец месяца, подготовка отчёта для клиентов или руководства.
Что получите: таблица uptime по всем сервисам, breach log с причинами, финансовые потери, прогноз на следующий месяц. --- ## 7. Бизнес-метрики с акцентом на потери Запрос:
```
бизнес-метрики: покажи потери выручки от инцидентов за эту неделю
``` Когда использовать: оценка влияния технических проблем на бизнес, обоснование инвестиций в инфраструктуру.
Что получите: потери в рублях по каждому инциденту, конверсионные метрики, тренды. --- ## 8. Добавить инцидент в журнал Запрос:
```
добавь инцидент: payment-service был недоступен 2026-04-08 с 03:22 до 05:09.
Причина: OOM, memory leak в воркере. Потери: 43200 руб. Исправление: рестарт + патч v2.1.3.
``` Когда использовать: инцидент закрыт, нужно зафиксировать в журнал.
Что получите: заполненный шаблон инцидента с severity, TTR, root cause, financial impact. --- ## 9. Утренний брифинг Запрос:
```
утренний отчёт
``` Когда использовать: начало рабочего дня, быстрый осмотр что произошло за ночь.
Что получите: инциденты ночью, текущие алерты, прогноз нагрузки, плановые работы на сегодня. --- ## 10. Еженедельный отчёт для CEO Запрос:
```
отчёт для CEO за неделю 14-20 апреля
``` Когда использовать: еженедельный отчёт руководству, который не нужно переводить с технического.
Что получите: "система работала 99.94% времени, 3 инцидента, потери 7700 руб, рекомендации по инвестициям". --- ## 11. Capacity planning — когда закончится место Запрос:
```
capacity: когда закончится disk на prod-02, текущий рост 0.23% в день, сейчас 74%
``` Когда использовать: планирование инфраструктурных расходов, предотвращение disk full.
Что получите: прогноз в днях, рекомендации по расширению, бюджет. --- ## 12. Post-mortem разбор Запрос:
```
post-mortem для инцидента: payment-service был недоступен 1ч47мин 8 апреля.
Root cause: утечка памяти в обработчике очереди. 43200 руб потерь.
``` Когда использовать: закрытие инцидента, обязателен для severity HIGH и CRITICAL.
Что получите: заполненный шаблон post-mortem с 5 Whys, timeline, action items с дедлайнами. --- ## 13. Настройка кастомных порогов Запрос:
```
установи пороги для payment-service:
- CPU: warning 70%, critical 85%
- Response time: warning 500ms, critical 1000ms
- Error rate: warning 0.5%, critical 2%
``` Когда использовать: первичная настройка, после изменения архитектуры, тюнинг под реальную нагрузку.
Что получите: подтверждение обновлённых порогов, рекомендации если пороги слишком жёсткие/мягкие. --- ## 14. Playbook при DDoS Запрос:
```
инцидент: DDoS, трафик вырос в 40 раз, легитимные пользователи не попадают на сайт
``` Когда использовать: аномальный трафик, сервис перегружен.
Что получите: Сценарий 3.7 — команды для идентификации атакующих IP, настройка rate limiting, действия по Cloudflare. --- ## 15. Полный обзор инцидентов за период Запрос:
```
покажи журнал инцидентов за апрель: статистику по severity, среднее TTR, повторные инциденты
``` Когда использовать: ежемесячный review, поиск системных проблем, подготовка к capacity review.
Что получите: таблица инцидентов, MTBF, MTTR, список повторных инцидентов требующих системного решения. --- ## Комбинированные сценарии ### После инцидента (полный цикл):
```
1. алерт: payment-service 503
2. инцидент: payment-service не отвечает
3. [после восстановления] добавь инцидент: ...
4. post-mortem для инцидента: ...
5. вечерний отчёт
``` ### Утренняя рутина CTO:
```
1. дашборд
2. утренний отчёт
3. SLA статус на сегодня
``` ### Подготовка к встрече с CEO:
```
1. отчёт для CEO за [период]
2. бизнес-метрики за [период]
3. capacity: риски на следующий квартал
```
