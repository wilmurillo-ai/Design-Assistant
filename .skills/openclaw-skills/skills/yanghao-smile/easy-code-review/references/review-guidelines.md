# 代码审核指南

本文档为AI代码审核助手提供详细的审查标准和最佳实践。

## 审查维度

### 1. 需求符合性审查

#### 1.1 功能完整性检查

**审查要点：**
- [ ] 所有需求点是否都有对应实现
- [ ] 实现是否完整，是否存在半成品代码
- [ ] 边界情况是否考虑周全

**常见问题：**
- 功能实现了，但缺少错误处理
- 只处理了正常流程，忽略了异常情况
- 实现了UI，但缺少后端逻辑

#### 1.2 技术约束检查

**审查要点：**
- [ ] 是否使用了指定的框架/库
- [ ] 是否遵循了技术架构要求
- [ ] API设计是否符合规范

**示例：**
```
需求：使用React实现表单
问题：代码中使用了原生DOM操作，不符合React范式
```

#### 1.3 业务规则验证

**审查要点：**
- [ ] 业务逻辑是否正确实现
- [ ] 数据验证规则是否符合要求
- [ ] 权限控制是否到位

### 2. 非必要修改检测

#### 2.1 文件修改必要性评估

**评估标准：**

| 文件类型 | 必要性判断 |
|---------|-----------|
| 业务逻辑文件 | 仅与需求直接相关的文件为必要 |
| 配置文件 | 仅添加新依赖时必要，修改现有配置需确认 |
| 测试文件 | 添加新功能测试必要，修改无关测试需确认 |
| 文档文件 | 更新API文档必要，大规模重写需确认 |
| 构建脚本 | 添加新构建步骤必要，其他需确认 |

#### 2.2 高风险修改模式

**应当特别关注的修改：**

1. **依赖版本变更**
   ```json
   // package.json
   "dependencies": {
     "react": "17.0.0" -> "18.0.0"  // ⚠️ 大版本升级
   }
   ```

2. **配置文件修改**
   ```javascript
   // webpack.config.js 或 tsconfig.json
   // 需要确认是否必要
   ```

3. **不相关模块修改**
   ```
   需求：添加登录功能
   修改：修改了用户列表页面逻辑  // ⚠️ 可能不必要
   ```

4. **大规模重构**
   ```
   需求：修复一个bug
   修改：重写了整个模块  // ⚠️ 过度修改
   ```

#### 2.3 修改范围评估公式

```
修改必要度 = 相关性分数 × 影响系数 × 复杂度系数

相关性分数：
- 直接相关：1.0
- 间接相关：0.6
- 可能相关：0.3
- 无关：0.0

影响系数：
- 新增文件：0.8
- 修改现有文件：1.0
- 删除文件：1.5

复杂度系数：
- 简单修改（<10行）：0.5
- 中等修改（10-50行）：1.0
- 复杂修改（>50行）：1.5
```

### 3. 代码质量检查

#### 3.1 代码风格

**检查项：**
- 缩进一致性
- 命名规范
- 注释质量
- 代码组织结构

#### 3.2 最佳实践

**前端代码：**
- [ ] 组件是否合理拆分
- [ ] 状态管理是否合理
- [ ] 是否存在性能问题（如不必要的重渲染）
- [ ] 是否考虑了响应式设计

**后端代码：**
- [ ] 是否遵循RESTful规范
- [ ] 是否有适当的错误处理
- [ ] 是否有SQL注入等安全风险
- [ ] 是否有适当的日志记录

**通用检查：**
- [ ] 是否有硬编码值
- [ ] 是否有魔法数字
- [ ] 是否有重复代码
- [ ] 是否有复杂度过高的函数

#### 3.3 安全检查

**常见安全问题：**
```javascript
// ❌ SQL注入风险
const query = `SELECT * FROM users WHERE id = ${userId}`;

// ✅ 参数化查询
const query = 'SELECT * FROM users WHERE id = ?';

// ❌ XSS风险
element.innerHTML = userInput;

// ✅ 安全渲染
element.textContent = userInput;

// ❌ 敏感信息暴露
console.log('User password:', password);

// ✅ 避免记录敏感信息
console.log('User login attempt');
```

### 4. 变更影响分析

#### 4.1 影响范围评估

**分析步骤：**
1. 识别修改的函数/类/模块
2. 分析调用关系链
3. 评估对其他模块的影响
4. 识别可能的副作用

#### 4.2 兼容性检查

**检查项：**
- [ ] 是否会破坏现有API
- [ ] 是否需要数据库迁移
- [ ] 是否需要配置文件更新
- [ ] 是否影响其他服务

## 特殊场景处理

### 场景1：需求描述模糊

**处理策略：**
1. 尝试从代码变更反推意图
2. 列出可能的解读供用户确认
3. 基于最小修改原则评估

### 场景2：大规模重构

**评估标准：**
- 是否有明确的重构需求说明
- 重构范围是否合理
- 是否有配套的测试更新

### 场景3：紧急修复

**审核原则：**
- 优先保证修复的有效性
- 允许适度的技术债
- 标记需要后续优化的部分

## 工具集成

```bash
# pre-commit hook
#!/bin/sh
# 自动触发代码审核
claw skill run code-review-assistant --diff
```

```yaml
# .github/workflows/review.yml
name: Code Review
on: [pull_request]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run Code Review
        run: claw skill run code-review-assistant
```
