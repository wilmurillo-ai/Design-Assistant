# Design Digest - 维护策略

## 概述

本文档说明 Design Digest skill 的长期维护策略，确保持续稳定运行和数据质量。

---

## 数据源管理

### 添加新数据源

**触发条件：**
- 用户提交 issue 建议新源
- 自己发现高质量设计/AI 资讯站点
- 原有源失效需要替换

**操作步骤：**
1. 验证站点是否支持 web_fetch（HTML 内容可提取）
2. 编辑 `SKILL.md` 的"数据源"部分
3. 更新 README/发布说明中的源数量
4. 提交 commit

**示例：**
```markdown
**🎨 工具更新**
- New Site: https://example.com/design/blog
```

### 数据源有效性检查

**频率：** 每月一次

**方式：**
- 手动运行一轮 web_fetch 测试
- 或编写自动化脚本批量验证
- 记录失效站点，考虑移除或降级

**失效处理：**
- 临时 404/502：等待下次验证
- 连续 3 次失败：考虑移除
- 完全停运：立即移除

**更新方式：**
- 修改 `SKILL.md` 源列表
- Git commit: `maintenance: remove失效源 [源名称]`
- 推送到主分支

## 摘要质量调优

### 调整触发

**按需：**
- 用户反馈摘要质量不佳
- 发现新需求或痛点
- 数据源内容风格变化

### 可调优项

#### 1. 摘要长度

编辑 `SKILL.md` 中的"摘要规则"：
```markdown
- 每条最终摘要 ≤ 3 句（英文）+ 3 句（中文）
- 总输出 ≤ 2000 字（中文计）
```

#### 2. 输出语言

编辑"输出模板"：
- 保留双语：保持现有格式
- 改单语：只保留一种语言

#### 3. 分类组织

调整分类数量和名称：
```markdown
- 🎨 工具更新
- 🤖 AI 动态
- 🏗️ 设计系统
- 📖 设计深度
- 💡 灵感发现
```

#### 4. 筛选策略

调整三层筛选权重：
- 新鲜度：7 天内（可调整为 3/5/14 天）
- 相关性：关键词匹配（可更新关键词列表）
- 价值度：头部产品优先、信息密度高（微调优先级规则）

### 未来：style.md 支持

计划引入独立配置文件 `style.md`，支持：
- 用户自定义摘要模板
- 无需修改 SKILL.md
- 更灵活的配置管理

## Skill 兼容性维护

### OpenClaw 版本跟进

**频率：** 每季度

**检查项：**
- OpenClaw 发布新版本时的 Breaking Changes
- Skill API 变更（如工具调用格式、session 管理等）
- 配置文件结构变化

**操作：**
- 阅读更新日志
- 更新 `SKILL.md` 中过时的 API 调用
- 测试新版本兼容性
- 发布补丁版本

### 测试计划

每次重大更新后：
1. 在不同 OpenClaw 版本测试
2. 验证所有命令正常工作
3. 检查去重和状态保存
4. 测试定时任务集成

## 性能优化

### Token 控制

**监控点：**
- 每次运行的 token 消耗
- web_fetch 的字符数限制
- 输出长度是否超限

**优化策略：**
- 超长输出：增加摘要截断规则
- 抓取效率：并行 fetch（如果 API 支持）
- 缓存优化：避免重复抓取同一 URL

### 去重策略

**当前方式：**
- URL 去重（`seen[url] = date`）
- 7 天后自动清理

**优化空间：**
- 添加标题/内容哈希去重（避免同一内容不同 URL）
- 支持手动清理命令 `/design clear`
- 调整保留天数（可配置）

## 用户支持

### Issue 响应

**优先级：**
- P0 - 源失效、无法安装：24 小时内
- P1 - 功能 bug、异常：3 天内
- P2 - 新需求、改进建议：1 周内

### 文档更新

**触发条件：**
- 常见问题重复出现
- 安装/使用流程不清晰
- 新功能上线

**方式：**
- 更新 README.md 的 FAQ 部分
- 添加"常见问题"章节到 PUBLISH.md
- 补充代码示例

## 版本规划

### 版本号规则

- `1.x.x` - 大版本（架构变更、新平台支持）
- `x.1.x` - 小版本（新功能、数据源更新）
- `x.x.1` - 补丁（Bug 修复）

### 发布节奏

- **功能发布**：按需，充分测试后
- **Bug 修复**：紧急时立即发布
- **数据源更新**：随 issue/PR 同步，可累积后批量发布

### 变更日志

每次发布需更新 `CHANGELOG.md`（未来创建）：
```
## [1.1.0] - 2026-03-31
### 新增
- 支持中文文档 (README.zh-CN.md)
- 添加 5 个新数据源
### 修复
- 修复去重逻辑问题
### 改进
- 优化 token 使用
```

## 与 follow-builders 的关键区别

| 方面 | follow-builders | design-daily |
|-------|----------------|---------------|
| 数据源存储 | `default-sources.json` | 直接写在 `SKILL.md` |
| 更新方式 | GitHub Actions 自动更新 | 手动编辑 + git push |
| 透明度 | 依赖自动化流程 | 完全透明，所见即所得 |
| 复杂度 | 较高（需配置 Actions） | 低（改一个文件即可） |
| 适用场景 | 频繁更新的大型源列表 | 中小规模精选源 |

**优势：**
- 更简单透明
- 无需配置 CI/CD
- 用户可直接提交 PR 参与

## 应急预案

### 源站大面积失效

**症状：**
- 多个源 404/503
- 搜索源无结果

**应急措施：**
- 立即发布 issue 通知用户
- 提供备用源列表
- 考虑切换到更稳定的源

### OpenClaw API 变更

**症状：**
- Skill 报错
- 工具调用失败

**应急措施：**
- 查看 OpenClaw 更新日志
- 快速适配新版 API
- 发布兼容性补丁

---

## 总结

维护目标：
1. **数据质量**：确保信息源有效、内容相关
2. **稳定可靠**：及时修复 bug、兼容版本更新
3. **用户友好**：快速响应问题、清晰文档
4. **持续改进**：根据反馈优化功能、支持更多场景

遵循本策略，Design Digest 可以长期稳定运行，为用户提供价值。
