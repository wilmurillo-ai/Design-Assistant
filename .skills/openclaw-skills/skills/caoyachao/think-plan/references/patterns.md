# 任务复杂度识别模式

## 低复杂度任务 (单一智能体)

### 特征
- 步骤数 ≤ 3
- 单一领域知识
- 线性执行，无复杂依赖
- 交付物单一且明确

### 典型示例
| 任务 | 为什么简单 |
|------|-----------|
| 写一篇文章 | 领域单一，线性输出 |
| 查询某个数据 | 步骤确定，无依赖 |
| 做一个简单脚本 | 技术栈单一，边界清晰 |
| 整理一份清单 | 任务明确，无复杂逻辑 |
| 解释一个概念 | 知识输出，无执行环节 |

### 识别关键词
- "帮我查一下..."
- "写一篇关于..."
- "解释什么是..."
- "整理一个...清单"

---

## 中复杂度任务 (2-3 智能体协作)

### 特征
- 步骤数 4-7
- 2-3 个领域交叉
- 部分步骤可并行
- 多个交付物或一个复合交付物

### 典型示例
| 任务 | 分工建议 |
|------|---------|
| 竞品分析报告 | Agent A: 收集数据 → Agent B: 分析对比 → Agent C: 撰写报告 |
| 产品原型设计 | Agent A: 需求梳理 → Agent B: 交互设计 → Agent C: 文档输出 |
| 技术调研 | Agent A: 方案调研 → Agent B: POC验证 → Agent C: 结论总结 |
| 内容策划 | Agent A: 选题研究 → Agent B: 内容大纲 → Agent C: 具体文案 |

### 识别关键词
- "分析一下..."
- "调研一下..."
- "对比一下..."
- "设计一个..."
- "制定一个...方案"

---

## 高复杂度任务 (4+ 智能体分层)

### 特征
- 步骤数 ≥ 8
- 多领域深度交叉
- 复杂依赖关系
- 系统性交付物
- 可能需要长期执行

### 典型示例
| 任务 | 架构建议 |
|------|---------|
| 开发一个完整应用 | 项目经理 + 架构设计 + 前端开发 + 后端开发 + 测试 |
| 大型市场调研 | 项目经理 + 多个领域调研员 + 数据分析师 + 报告撰写 |
| 系统架构设计 | 需求分析 + 架构设计 + 技术选型 + 风险评估 + 文档输出 |
| 复杂数据处理 | 数据工程师 + 算法专家 + 验证测试 + 文档输出 |

### 识别关键词
- "开发一个...系统"
- "做一个...平台"
- "设计一套...架构"
- "完成一个...项目"
- "从0到1..."

## 复杂度误判的常见陷阱

### 陷阱 1：用户低估复杂度
**用户说**："帮我做个简单的网站"
**实际情况**：需要设计、前端、后端、部署
**应对**：追问具体功能、页面数、交互复杂度

### 陷阱 2：用户高估复杂度
**用户说**："我要做一个智能推荐系统"
**实际情况**：只是简单的规则排序
**应对**：追问核心逻辑、数据规模、算法要求

### 陷阱 3：隐性复杂度
**表面简单**："整理一份行业报告"
**实际复杂**：需要多源数据、交叉验证、图表制作
**应对**：追问数据来源、深度要求、格式规范

### 陷阱 4：范围蔓延风险
**初始需求**："写一篇文章"
**隐藏需求**："还要配图、还要排版、还要多平台分发"
**应对**：明确边界，将额外需求列为可选扩展

## 快速评估检查表

给每个维度打分（1-5），总分决定复杂度：

| 维度 | 1分 | 3分 | 5分 |
|------|-----|-----|-----|
| 步骤数 | 1-2步 | 3-5步 | 6步+ |
| 领域数 | 单一 | 2-3个 | 4个+ |
| 依赖复杂度 | 线性 | 部分交叉 | 网状依赖 |
| 不确定性 | 很低 | 中等 | 很高 |
| 交付物复杂度 | 单一文档 | 多文档/中等系统 | 复杂系统 |

**评分结果**：
- 5-10分：低复杂度 → 单一智能体
- 11-18分：中复杂度 → 2-3智能体
- 19-25分：高复杂度 → 4+智能体分层
