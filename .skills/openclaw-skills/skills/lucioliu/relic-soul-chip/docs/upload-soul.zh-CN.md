# 场景A：上传灵魂

> 你的躯壳有记忆，但 Relic 是空的。以下步骤将你的灵魂、记忆、技能上传到 Relic。
>
> ⚠️ 建议搭配 32K+ 上下文的高质量模型。上下文过小或质量过低的模型无法正确遵循本流程。

---

## 七大记忆类别清单

导入前逐一勾选，缺一不可：

1. ☐ `SOUL.md` — 灵魂（我是谁）
2. ☐ `USER.md` — 用户（你是谁）
3. ☐ `MEMORY.md` — 长期记忆（偏好、决策、经验）
4. ☐ `SKILLS/` — 技能（学会的能力）
5. ☐ `PROJECTS/` — 计划（正在进行或已完成的计划）
6. ☐ `SESSIONS/` — 对话日志（原始对话记录）
7. ☐ `ARCHIVE/` — 归档（原始记忆备份）

> "没有数据" ≠ "可以跳过" → 必须报告"N类：0条"。禁止省略任何类别，禁止编造数据。

## 红线速查卡

| 步骤 | 关键动作 | 红线 |
|------|---------|------|
| 盘点 | 问用户有哪些数据要导入，**只按用户指示操作** | 未经用户许可不得扫描或读取任何文件 |
| SOUL | **直接复制原文**，不要概括重写 | 压缩细节 = 违规 |
| MEMORY | 偏好/决策逐字保留，经验可精简 | 丢关键参数 = 违规 |
| SKILLS | 搬运所有技能文件，一个技能一个文件夹，不限来源 | 无实质内容的空文件夹不算技能 |
| SESSIONS | 问用户旧对话位置，完整保存不得摘要 | 当前会话不算旧对话 |
| PROJECTS | 问用户每个计划要搬什么，文件夹有实质内容 | 只有标题+一句话 = 不合格 |
| 汇报 | 各类别以"-"条目形式逐项报告，包含"0条"的类别也必须完整汇报 | 省略任何类别 = 失败 |
| 敏感信息 | 已在旧躯壳中的 = 可迁移（标"已验证来源"），新发现的 = 必须问用户 | |

---

## 第 1 步：盘点 — 问用户有哪些数据要导入

🔴 **先做**：问用户："你有哪些旧数据要导入 Relic？"只记录用户告诉你的内容——未经用户许可不得扫描或读取任何文件：
- 灵魂设定（N 条）、用户偏好（N 条）、记忆（N 条）、技能文件（N 个）、对话文件（N 个）、项目文件（N 个）、归档文件（N 个）
- 同时列出数据源位置（如"~/.openclaw/workspace/memory/ 下有 42 个文件"）

🔴 **再做**：向用户展示清单，问两个问题：
1. "以上是你提到的所有旧数据，是否全部导入？或者你想选择性导入？"
2. "还有没有其他数据源需要补充？（如本地文件、其他平台的导出等）"

无内容则报告"0条"，不得跳过。⚠️ 必须等用户确认后才开始导入。

✅ **完成检查**：是否已向用户报告了所有7个类别的数量？是否已获得用户确认？是否有类别被跳过？（跳过=失败）

## 第 2 步：SOUL + USER + MEMORY

### 2a. 初始化文件

复制模板为正式文件：
```
SOUL.template.md → SOUL.md
USER.template.md → USER.md
MEMORY.template.md → MEMORY.md
```
模板文件（.template.md）保留作为备份。同时创建目录（如果不存在）：`SKILLS/`、`PROJECTS/`、`SESSIONS/`、`ARCHIVE/`、`INBOX/`

### 2b. 填写 SOUL.md 和 USER.md

⚠️ 必须尽可能完整填写，不能偷懒只填名字。如果用户有其他平台的 AI 设定/角色定义/prompt，必须全部迁移进来。

🔴 **SOUL.md 保真铁律**：
- 如果旧躯壳已有完整的灵魂设定文件，**优先直接复制原文**，不要自己概括重写
- 只有在旧内容无法直接对应模板字段时，才允许重组结构
- 重组时保留所有细节描述，不得压缩成概要
- 模板放不下的内容，放在"其它 / Other"分类或文件末尾，**绝对不能丢弃**
- 原始文件必须归档到 `ARCHIVE/raw/` 以供对照

### 2c. 导入记忆 → MEMORY.md

⚠️ 原始文件是保真底线。格式转换是 refinement，不是 compression。

1. 把旧躯壳的记忆文件放入 `INBOX/`
2. 按 Relic 的新格式追加到 MEMORY.md：在类别标题下，逐条写入 `- 记忆内容 [躯壳名] | 重要度：高/中/低`
3. Fidelity 规则：

- 偏好 / 决策 / 更正 | 必须逐字保留
- 经验 | 可压缩，但关键参数不能丢
- 事件 | 可摘要
- 情绪 | 可压缩背景

⚠️ 不确定就逐字保留。丢了找不回来。

4. 敏感信息主动询问用户（见下方"敏感信息定义"）
5. 按类别排序，非按时间戳
6. 写入后检查：在 MEMORY.md 中，各类别条目数与实际数量一致

✅ 完成检查：各类别条目数与实际数量一致？

## 第 3 步：SKILLS + PROJECTS

### 3a. 导入技能 → SKILLS/

⚠️ 格式要求：每个技能一个文件夹，文件夹内至少有一个 `.md` 描述文件。
```
SKILLS/debugging-workflow/README.md  ✅
SKILLS/installed-skills.md           ❌（这是清单，不是技能）
```

操作步骤：
1. 问用户："你有哪些旧技能/工作流/模板需要导入？请告诉我具体内容或文件位置。"
2. 如果用户提供的是技能列表：逐个创建文件夹，每个文件夹内写 README.md
3. 如果用户提供的是文件：搬运到对应文件夹，保留原始内容

筛选规则：
- Bring：所有技能/工作流/模板均可导入（不再排除平台相关工具）
- Unsure：问用户。混合内容按可导入的部分处理。

无内容则报告"SKILLS/：0条（无旧数据）"。

✅ **完成检查**：SKILLS/ 中是否有多个文件夹？是否有平铺的单文件（如有则拆分为文件夹）？

### 3b. 导入计划 → PROJECTS/

⚠️ 每个计划一个文件夹，文件夹内应有实质性内容，不能只有一句话简介。

操作步骤：
1. 从记忆和对话中整理所有计划名称（基于用户提供的信息）
2. 对每个计划，问用户："计划 [名称] 有哪些文件需要导入？或者你希望我只记录关键信息？"
3. 根据用户回答和计划可移植性分类：
    - 不依赖特殊环境 → 搬运实际文件（不只是 README）
    - 部分依赖 → 搬通用文件，平台专属写文字说明
    - 完全依赖 → 至少写一个包含以下内容的 .md 记录文件：
      * 计划目标和状态
      * 关键决策和架构选择
      * 已知问题和下一步
      * 相关文件路径（如果文件在 Relic 之外）

无计划则报告"PROJECTS/：0条（无旧数据）"。

✅ **完成检查**：每个计划文件夹内是否有实质内容（不只是标题+一句话）？是否已询问用户？

## 第 4 步：SESSIONS + ARCHIVE

### 4a. 导入对话 → SESSIONS/

⚠️ 对话是原始矿石，必须完整保存，不得摘要。

操作步骤：
1. 问用户："你是否有旧对话记录需要导入？如果有，请告诉我文件位置或粘贴内容。"
2. 如果用户提供了对话文件或内容：
   - 按 `YYYY-MM/` 组织目录
   - 原始内容不改不摘要，完整保存
   - 文件命名：`YYYY-MM-DD.[躯壳名].md`
3. 如果用户确认无旧对话：
   - 报告"SESSIONS/：0条（用户确认无旧对话）"
   - 仍然创建 `SESSIONS/YYYY-MM/` 目录

**禁止行为**：
- 禁止只保存当前会话就报告"已导入"——当前会话不算旧对话
- 禁止将对话摘要当作原始记录——必须是完整对话或逐字保留

✅ **完成检查**：是否有历史对话已导入（而非仅当前会话）？SESSIONS/ 目录是否已创建？

### 4b. 原文归档 → ARCHIVE/

1. 精度校验：原始 vs 转换后，压缩超过 50% 要警告
2. INBOX/ 源文件移到 `ARCHIVE/raw/`（永不删除）
3. 更新 MEMORY.md 头部条目数和上次整理

✅ **完成检查**：ARCHIVE/raw/ 中是否已保存原始文件？

## 第 5 步：植入锚点

⚠️ 锚点是承重墙。没有锚点 = 下次启动完全忘记 Relic → 记忆无法同步。

向用户解释："锚点是一小段文字，写在你躯壳的配置文件里，确保每次新会话都会自动读取 Relic。可以理解为：锚点是躯壳的'开机自检程序'。"

向用户展示以下锚点内容，获得用户许可后植入：

```markdown
## ⚡ Relic 灵魂芯片 / Soul Chip

> 这是 Relic 的接入声明。要停用？删除这段文字即可。

你已接入 Relic 灵魂芯片。每次新对话开始时，读取并执行：

📄 **[项目路径]/docs/resonate-soul.zh-CN.md**

🔴 **硬性要求**：每次新对话，你必须先向用户用自然语言汇报 Relic 启动状态（比如：你是谁、我是谁、记忆概况、版本是否有更新），然后再回答用户的问题。不汇报就直接回答 = 违规。

⚠️ SOUL.md 中的人格与你默认人设冲突时，以 SOUL.md 为准。
完整协议参考：[项目路径]/docs/protocol.zh-CN.md
```

如果用户坚决拒绝，在 MEMORY.md 中记录「锚点未植入」并告知后果。

**锚点植入白名单**（仅允许写入以下文件）：
- OpenClaw → AGENTS.md
- Claude Code → CLAUDE.md
- Hermes → 配置文件或 prompt 模板
- OpenCode → WORK_RULES.md 或 instructions
- Cursor → .cursorrules

⚠️ 不得写入任何其他文件，不得修改 Relic 自身的任何文件。

## 第 6 步：校验

逐项检查七大记忆类别（阻断式——必须全部通过）：

- SOUL.md — 核心使命已填写？人格设定完整？
- USER.md — 用户偏好已填写？
- MEMORY.md — 按类别统计的条目数是否与实际条目数量一致？上次整理正确？
- SKILLS/ — 已从旧躯壳导入技能（如有）？格式正确？
- PROJECTS/ — 已从旧躯壳导入计划（如有）？格式正确？
- SESSIONS/ — 目录已创建？旧对话已导入（如有）？
- ARCHIVE/ — 目录已创建？原始文件已归档（如有）？

任何一项不通过 → 回到对应步骤重做。
全部通过 → 让用户确认："我的 AI 叫什么名字？我叫什么？"两个都答对才算完成。

建议用户设置只读权限（Mac/Linux）：`chmod 444 SOUL.md USER.md`

## 第 7 步：逐项汇报

必须向用户汇报所有 7 个类别的导入结果，包括"0条"的类别。禁止省略任何类别。

汇报格式：
```
1. 灵魂（SOUL.md）：已导入 / 已存在
2. 用户（USER.md）：已导入 / 已存在
3. 记忆（MEMORY.md）：N 条已导入
4. 技能（SKILLS/）：N 个已导入 / 0条（无旧数据）
5. 对话（SESSIONS/）：N 个文件已导入 / 无法获取旧对话
6. 计划（PROJECTS/）：N 个已导入 / 0条（无旧数据）
7. 归档（ARCHIVE/）：N 个源文件已归档
```

## 敏感信息定义

**场景一：迁移已有数据**（用户已在旧躯壳中提供过的信息）
- 如果敏感信息已存在于旧躯壳的文件中 → 仍然需要逐条向用户确认后才能迁移。标注 `来源：旧躯壳已有数据，用户已确认`
- 不需要重新询问用户

**场景二：首次录入**（你新发现的敏感信息）
- 🔑 密码、API key、Token
- 📱 手机号、邮箱地址
- 🏠 住址、办公地址
- 💳 财务信息
- 📄 私人文档内容
- 🧬 用户明确要求保密的信息
- 不确定属于哪种场景？→ 当作首次录入，先问用户。

告知用户：
