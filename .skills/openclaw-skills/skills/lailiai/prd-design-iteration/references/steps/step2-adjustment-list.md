# Step 2: AI分析并输出调整清单

## 目标
基于Step 1收集的信息，AI推断出功能模块列表和本次迭代的调整清单。

## 2.1 功能模块推断

### 推断逻辑

**输入**：Step 1的主流程 + 迭代需求

**推断方法**：
1. 从主流程中提取动词短语（配置、创建、执行、审核、生成）
2. 将相似动作归并为一个功能模块
3. 从迭代需求中提取新提到的功能词

**示例**：

主流程：
```
1. 业务部门配置回访模板
2. 业务部门创建回访任务并分配给客服专员
3. 客服专员执行回访并记录结果
4. 业务部门审核回访质量
5. 系统生成回访数据报表
```

推断出的模块：
```
M1 回访模板管理 - 对应步骤1"配置回访模板"
M2 回访任务管理 - 对应步骤2"创建任务、分配"
M3 回访执行 - 对应步骤3"执行回访、记录结果"
M4 质量审核 - 对应步骤4"审核回访质量"
M5 数据报表 - 对应步骤5"生成数据报表"
```

如果迭代需求提到：
```
需求：希望系统能自动根据规则生成回访任务
```

则新增推断：
```
M6 规则引擎 - 新增模块，用于配置自动生成规则
```

### 输出格式

创建 `project/01-adjustment-list.md`：

```markdown
# 调整清单

## 现有功能模块（推断）

基于主流程推断出以下功能模块：

| 模块编号 | 模块名称 | 对应流程 |
|---------|---------|---------|
| M1 | [模块名] | 步骤X：[流程描述] |
| M2 | [模块名] | 步骤Y：[流程描述] |
...

## 本次调整清单

根据迭代需求，本次需要调整以下模块：

| 模块编号 | 模块名称 | 调整类型 | 调整说明 | 对应需求 |
|---------|---------|---------|---------|---------|
| M2 | 回访任务管理 | 调整 | 增加自动生成任务功能 | 需求1 |
| M6 | 规则引擎 | 新增 | 新增规则配置模块 | 需求1 |
| M3 | 回访执行 | 调整 | 增加客户历史记录展示 | 需求2 |

### 调整类型说明

- **新增**：全新的功能模块，之前不存在
- **调整**：在现有模块基础上增加/修改功能
- **删除**：移除现有模块或功能
```

## 2.2 调整类型判断规则

### 新增（ADD）
触发条件：
- 迭代需求中出现"新增"、"增加XX模块"、"需要一个新的XX"
- 推断的模块在现有主流程中找不到对应步骤

### 调整（MODIFY）
触发条件：
- 迭代需求中出现"在XX模块增加"、"XX功能优化"、"改进XX"
- 现有模块功能不变，但增加新能力或修改交互

### 删除（DELETE）
触发条件：
- 迭代需求中出现"删除"、"移除"、"废弃"、"不再需要XX"

## 2.3 用户确认

用 `present_files` 呈现 `01-adjustment-list.md`，询问用户：

> "我理解您需要调整以上模块，对吗？
> 
> - 如果模块推断有误，请告诉我正确的模块名称
> - 如果有遗漏的模块，请补充
> - 如果调整类型判断错误（新增/调整/删除），请指正"

### 常见调整情况

**情况1：用户补充遗漏模块**
```
用户："还需要调整M5数据报表，增加新的统计维度"
AI：更新调整清单，增加M5的调整项
```

**情况2：用户修正调整类型**
```
用户："M6不是新增，是调整现有的任务配置模块"
AI：将M6从"新增"改为"调整"，并更新模块名称
```

**情况3：用户合并模块**
```
用户："M2和M6其实是一个模块，都是任务管理"
AI：合并为一个模块M2，调整说明合并
```

## 确认完成标志

用户明确回复：
- "确认"
- "没问题"
- "就是这样"
- "对的"

或用户修正后再次确认。

**确认完成后进入Step 3。**
