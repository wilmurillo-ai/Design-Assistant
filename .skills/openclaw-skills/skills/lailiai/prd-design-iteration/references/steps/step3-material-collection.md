# Step 3: 分类收集调整内容

## 目标
按调整类型（新增/调整/删除）分别收集必要的材料，确保有足够信息进行精准设计。

## 处理顺序
建议按以下顺序处理（先易后难）：
1. 删除类（DELETE） - 只需分析影响
2. 新增类（ADD） - 需求描述即可
3. 调整类（MODIFY） - 需要截图/原型，最复杂

---

## 3.1 新增界面（Type A）

### 适用场景
调整清单中标记为"新增"的模块。

### 收集流程

对于每个新增模块，依次询问3个问题：

#### 问题1：需求描述
**提问**：
> "请描述【{模块名}】这个新增界面/模块的需求和功能。"

**引导格式**：
```
功能：做什么
用户：谁用
场景：什么时候用
目标：解决什么问题
```

**示例**：
```
功能：配置自动生成回访任务的规则
用户：业务部门使用
场景：在需要批量定期回访时，提前配置好规则，系统按规则自动生成任务
目标：减少手动创建任务的工作量
```

#### 问题2：界面字段
**提问**：
> "这个界面需要哪些字段？请列出字段名称、类型和说明。"

**引导格式**：
```
字段名 | 类型 | 说明 | 是否必填
```

**示例**：
```
规则名称 | 文本 | 规则的标识名称 | 是
触发条件 | 下拉多选 | 选择触发规则的条件（如：新客户注册、订单完成） | 是
生成频率 | 单选 | 每日/每周/每月 | 是
目标客户 | 复杂筛选组件 | 按业务类型、地区等筛选客户 | 是
回访模板 | 下拉单选 | 选择使用哪个回访模板 | 是
分配策略 | 单选 | 平均分配/按地区分配/按客服负载分配 | 是
状态 | 系统字段 | 启用/禁用 | 否
```

**注意**：
- 如果用户只说字段名，追问类型和说明
- 如果字段超过15个，提醒"是否可以分页或分组"

#### 问题3：前序/后续界面
**提问**：
> "这个界面的前序界面是什么（从哪里进入）？后续界面是什么（点击后去哪里）？
> 
> 如果有**现有系统的截图**，请上传前序和后续界面的截图，这样我能更精准地设计跳转。如果没有截图，请描述跳转关系。"

**引导格式**：
```
前序界面：从XX页面点击XX按钮进入
后续界面：
  - 点击"保存"后回到XX页面
  - 点击"取消"后回到XX页面
```

**示例**：
```
前序界面：从M2任务管理页面，点击顶部"规则配置"按钮进入
后续界面：
  - 点击"保存规则"后回到任务管理页面，顶部显示成功提示
  - 点击"取消"后回到任务管理页面
  - 点击"预览生成结果"后弹窗显示按规则会生成哪些任务
```

**如果用户上传截图**：
- 保存截图到 `project/02-add/screenshots/`
- 在文档中引用截图路径
- 分析截图中的导航结构、按钮位置、页面布局

### 输出

为每个新增模块创建 `project/02-add/{模块名}.md`：

```markdown
# 新增模块：{模块名}

## 需求描述

**功能**：[做什么]
**用户**：[谁用]
**场景**：[什么时候用]
**目标**：[解决什么问题]

## 界面字段

| 字段名 | 类型 | 说明 | 必填 |
|-------|------|------|------|
| [字段1] | [类型] | [说明] | 是/否 |
| [字段2] | [类型] | [说明] | 是/否 |
...

## 前后跳转

**前序界面**：[从哪里进入]
（如有截图：见 `screenshots/[文件名]`）

**后续界面**：
- [操作1]：[跳转目标]
- [操作2]：[跳转目标]
（如有截图：见 `screenshots/[文件名]`）

## 备注

[其他需要说明的信息]
```

## 3.2 调整当前界面（Type B）

### 适用场景
调整清单中标记为"调整"的模块。

### 收集流程

对于每个调整模块，依次询问2个问题：

#### 问题1：调整说明
**提问**：
> "请说明要对【{模块名}】做什么调整？具体要增加/修改/移除什么功能或元素？"

**引导格式**：
```
调整内容：[具体要改什么]
调整原因：[为什么要改]
期望效果：[改完后是什么样]
```

**示例**：
```
调整内容：在任务管理页面顶部增加"自动生成"按钮，点击后进入规则配置页面
调整原因：目前只能手动创建任务，希望增加自动生成入口
期望效果：业务部门可以快速进入规则配置，减少操作步骤
```

#### 问题2：当前界面截图/原型（关键！）
**提问**：
> "🎯 **关键步骤**：请上传【{模块名}】**当前界面的截图**或HTML原型。
> 
> 这样我能精准知道：
> - 在哪个位置加按钮/字段
> - 现有布局是怎样的
> - 应该如何调整才不破坏原有设计
> 
> 如果暂时没有截图，请尽可能详细描述当前界面的布局和主要元素。"

**如果用户提供截图**：
- 保存截图到 `project/03-modify/screenshots/`
- 仔细分析截图：
  - 页面标题在哪里
  - 主要操作按钮在哪里（顶部/右侧/底部）
  - 数据展示区域是什么形式（表格/卡片/列表）
  - 筛选/搜索组件在哪里
- 在调整说明中标注变更位置（如："在顶部操作栏右侧增加按钮"）

**如果用户无法提供截图**：
追问以下细节：
> "请描述当前界面：
> 1. 顶部有什么？（标题、面包屑、操作按钮）
> 2. 中间主体是什么？（表格、表单、卡片）
> 3. 如果是表格，有哪些列？
> 4. 右侧或底部有什么操作？"

### 输出

为每个调整模块创建 `project/03-modify/{模块名}.md`：

```markdown
# 调整模块：{模块名}

## 调整说明

**调整内容**：[具体要改什么]
**调整原因**：[为什么要改]
**期望效果**：[改完后是什么样]

## 当前界面

### 截图
（如有截图：嵌入截图或引用路径 `screenshots/[文件名]`）

### 界面结构描述
（如无截图，根据用户描述整理）

**顶部区域**：
- [元素1]
- [元素2]

**主体区域**：
- [表格/表单/卡片等]
- [列/字段清单]

**操作区域**：
- [按钮/链接]

## 变更位置标注

**要增加的元素**：
- 位置：[在哪里加]
- 内容：[加什么]
- 样式：[按钮/链接/下拉等]

**要修改的元素**：
- 原内容：[现在是什么]
- 新内容：[改成什么]
- 位置：[在哪里]

**要删除的元素**：
- 位置：[在哪里]
- 内容：[删什么]
- 原因：[为什么删]

## 调整后效果描述

[用文字描述调整后的界面应该是什么样]
```

## 3.3 删除界面（Type C）

### 适用场景
调整清单中标记为"删除"的模块。

### AI自动分析（无需用户输入）

对于每个删除模块，AI自动执行以下分析：

#### 分析1：流程影响
检查该模块在主流程中的位置，分析删除后流程如何衔接。

**示例**：
```
当前流程：
1. 配置模板
2. 创建任务
3. 执行回访
4. 审核质量 ← 要删除
5. 生成报表

删除后流程：
1. 配置模板
2. 创建任务
3. 执行回访
4. 生成报表 ← 直接跳到这里

影响：客服提交回访后，直接进入报表生成，中间不再有审核环节
```

#### 分析2：数据影响
检查该模块涉及的数据字段，分析哪些字段会废弃。

**从Step 1中提取的数据字段**：
```
回访记录：记录ID、任务ID、客户ID、客服ID、回访时间、
         回访结果、问题答案JSON、审核状态、审核意见
```

**分析**：
```
废弃字段：
- 审核状态（删除审核模块后不再需要）
- 审核意见（删除审核模块后不再需要）

保留字段：
- 其他字段继续使用

历史数据处理建议：
- 已有的审核状态和审核意见建议归档保留，不删除
- 新创建的回访记录不再包含这两个字段
```

#### 分析3：依赖检查
检查其他模块是否依赖被删除的模块。

**检查方法**：
1. 在其他模块的描述中搜索被删除模块的关键词
2. 检查数据报表是否使用被删除模块的数据

**示例**：
```
依赖检查结果：
- M5 数据报表中有"审核通过率"指标，依赖审核状态字段
- M3 回访执行中有"返工"状态，需要审核模块触发

影响处理：
- M5 数据报表需要删除"审核通过率"相关图表
- M3 回访执行需要移除"返工"状态及相关逻辑
```

### 输出

为每个删除模块创建 `project/04-delete/{模块名}-impact.md`：

```markdown
# 删除影响分析：{模块名}

## 删除概述

**删除模块**：[模块名]
**删除原因**：[引用用户在迭代需求中的说明]

## 流程影响

### 当前流程
[列出当前完整流程，标注被删除步骤]

### 删除后流程
[列出删除后的流程，标注如何衔接]

### 流程完整性
- ✅ 流程仍然完整 / ⚠️ 需要补充衔接逻辑

## 数据影响

### 废弃字段
| 数据对象 | 废弃字段 | 原用途 | 处理建议 |
|---------|---------|--------|---------|
| [对象名] | [字段名] | [用途] | [建议] |

### 保留字段
[列出不受影响的字段]

### 历史数据处理
[说明已有数据如何处理]

## 依赖检查

### 被依赖情况
[列出哪些模块依赖被删除的模块]

### 影响处理
[说明如何处理这些依赖]

## 删除风险评估

- **流程风险**：[高/中/低] - [说明]
- **数据风险**：[高/中/低] - [说明]
- **依赖风险**：[高/中/低] - [说明]

## 建议

[AI给出的删除建议或替代方案]
```

### 用户确认

用 `present_files` 呈现影响分析，询问用户：

> "删除【{模块名}】会有以上影响，确认要删除吗？
> 
> - 如果确认删除，我将在后续设计中处理这些影响
> - 如果风险太高，建议改为"隐藏"或"禁用"而非完全删除"
