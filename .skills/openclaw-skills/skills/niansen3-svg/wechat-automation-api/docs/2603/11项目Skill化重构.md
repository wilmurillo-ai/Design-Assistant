# 微信自动发信 API 升级：兼容 OpenClaw Skill 规范重构记录

**日期**：2026-03-11

## 1. 背景与目标
原项目 `wechat-automation-api` 专注于作为独立的 HTTP 后台服务运行，通过队列系统进行异步的消息发送。
为了将其接入类似于 **OpenClaw** 的大模型智能体系统作为**执行原子能力的 Skill** 进行即时调用，我们需要满足以下原则：
1. **单一职责和快速熔断**：智能体通过执行本地脚本发送消息，需要直接、纯净和即时的反馈（成功/失败），不能依赖复杂的异步常驻队列。
2. **部署兼容性**：保留原有的 HTTP 后台发送功能，以便其他常规后台任务仍能排队发信。
3. **极简架构（第一性原理）**：避免为了“兼容”而让单个文件变得过度复杂。从调用入口切分是最简单优雅的做法。

## 2. 核心架构调整
### 2.1. 目录平移与解耦
- 创建 `scripts/` 收敛所有核心逻辑脚本 (`app.py`, `wechat_controller.py`, `message_queue.py`)。
- 在 `scripts/app.py` 中重构路径寻址逻辑，采用基于 `__file__` 的相对绝对路径定位，确保程序能在任意目录正常地拉起针对项目根目录的 `config.json` 与 `wechat_automation.log`。

### 2.2. 分立调用入口
不再将命令行执行与服务启动混合在 `app.py` 中。
- **作为 Skill 使用**：新增 `scripts/skill_cli.py`。这是给智能体系统挂载的官方唯一执行入口。程序直接实例化 `WeChatController` 操控微信窗体进行消息或图片的发送，并返回简单的标准退出码或控制台输出以供 Agent 判断。
- **作为后台 HTTP 服务使用**：新增同名快捷启动文件 `run.bat` 至项目根目录。内部隐藏实现为触发 `python scripts/app.py` 开启原有 HTTP 网关功能和队列调度功能。 

### 2.3. 规范化声明
- 新增 `SKILL.md`：详细约束了该 Skill 的运行前置条件、调用示例及其作为纯粹独立服务启动的方法，为智能体识别本外挂能力提供准确的知识图谱。

## 3. 测试与验证
1. 通过 `python scripts/skill_cli.py --to [name] --content [msg]` 直接调用完成测试，进程快速退出，返回0，日志显示发送成功。
2. 双击 `run.bat`，端口依然正常拉起，旧有的 `POST /` API 可无损兼容运行。

**结论**：重构非常顺利，达成预期设计的模块解耦目标并彻底保留了业务向后兼容性。
