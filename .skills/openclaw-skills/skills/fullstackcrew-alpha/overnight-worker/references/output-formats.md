# 输出格式规范

定义 Overnight Worker 各类产出物的标准格式，确保输出一致性和可读性。

---

## Markdown 报告 (.md)

适用：调研报告、分析报告、技术文档

### 结构规范

```markdown
# {报告标题}

> **生成时间**: {ISO 8601}
> **任务来源**: Overnight Worker
> **数据时效**: 截至 {日期}

## 摘要

{3-5 句话概括核心发现，信息密度要高}

## 目录

- [章节1](#章节1)
- [章节2](#章节2)
...

## 章节1: {标题}

### 1.1 {子标题}

{正文内容}

> 💡 **关键发现**: {一句话高亮}

### 1.2 {子标题}

{正文内容}

| 维度 | 对象A | 对象B | 对象C |
|------|-------|-------|-------|
| {维度1} | {值} | {值} | {值} |

## 章节2: {标题}

...

## 结论与建议

1. **{建议1}**: {说明}
2. **{建议2}**: {说明}

## 附录

### 数据来源
- [{来源名}]({URL}) — 访问日期: {日期}

### 局限性说明
- {说明哪些信息未能获取及原因}

---
*由 Overnight Worker v1.0.0 自动生成 | {日期}*
```

### 格式要点
- 标题层级不超过 3 级 (H1-H3)
- 表格用于对比数据，不用于叙述
- 每个章节不超过 500 字
- 关键发现用引用块 (blockquote) 高亮
- URL 引用放在附录，正文中用脚注

## CSV 数据表 (.csv)

适用：结构化数据、对比数据、列表数据

### 格式规范

```
# 文件编码: UTF-8 with BOM
# 分隔符: 逗号
# 引号: 包含逗号或换行的字段用双引号包裹
# 首行为表头
```

### 示例

```csv
名称,类别,价格,评分,来源URL,备注
"产品A","SaaS",29.99,4.5,"https://example.com","市场领导者"
"产品B","SaaS",19.99,4.2,"https://example.com","性价比最高"
```

### 附带说明文件

每个 CSV 文件都应附带 `{同名}-meta.md` 说明文件：

```markdown
# {数据表名} 数据说明

## 字段定义
| 字段名 | 类型 | 说明 |
|--------|------|------|
| 名称 | string | 产品/对象名 |
| 类别 | string | 分类标签 |
...

## 数据统计
- 总记录数: {N}
- 数据时间范围: {范围}
- 缺失值: {哪些字段有缺失，比例}

## 数据来源
- {来源说明}
```

## JSON 结构化数据 (.json)

适用：API 兼容数据、程序化消费的数据

### 格式规范

```json
{
  "_meta": {
    "generated_by": "overnight-worker",
    "version": "1.0.0",
    "timestamp": "2024-01-01T00:00:00Z",
    "task": "原始任务描述",
    "record_count": 0
  },
  "data": [
    {
      "id": "唯一标识",
      "field1": "value1",
      "field2": "value2"
    }
  ],
  "summary": {
    "total": 0,
    "key_findings": [
      "发现1",
      "发现2"
    ]
  }
}
```

### 格式要点
- 顶层必须有 `_meta` 字段
- 数据放在 `data` 数组中
- 摘要放在 `summary` 对象中
- 使用 2 空格缩进
- 日期时间统一用 ISO 8601 格式
- 金额统一用最小单位（如：分）并标注货币

## 代码审查报告 (.md)

适用：代码审查、安全审计

### 结构规范

```markdown
# 代码审查报告

> **项目**: {项目名}
> **审查范围**: {文件/模块列表}
> **审查时间**: {ISO 8601}

## 审查摘要

| 统计项 | 数量 |
|--------|------|
| 审查文件数 | {N} |
| 🔴 严重问题 | {N} |
| 🟡 一般问题 | {N} |
| 🟢 改进建议 | {N} |

## 🔴 严重问题

### [S-001] {问题标题}

- **文件**: `{file_path}:{line}`
- **类别**: {安全/数据丢失/...}
- **描述**: {问题描述}
- **影响**: {潜在影响}
- **建议修复**:

```{lang}
// 修复前
{问题代码}

// 修复后
{建议代码}
```

---

## 🟡 一般问题

### [W-001] {问题标题}

- **文件**: `{file_path}:{line}`
- **类别**: {代码异味/可维护性/...}
- **描述**: {问题描述}
- **建议**: {改进建议}

---

## 🟢 改进建议

### [I-001] {建议标题}

- **文件**: `{file_path}:{line}`
- **描述**: {建议说明}

---

## 总评

{1-2 段对代码整体质量的评价，包括做得好的方面}
```

## 晨间摘要 (morning-summary.md)

格式已在 SKILL.md 阶段 5 中定义，此处不再重复。

关键原则：
- **信息密度最高的文件**——用户可能只看这一个文件
- 关键发现每条不超过 2 行
- 需要人工跟进的项必须说清楚为什么 Agent 无法完成
- 文件列表要有一句话说明每个文件是什么
