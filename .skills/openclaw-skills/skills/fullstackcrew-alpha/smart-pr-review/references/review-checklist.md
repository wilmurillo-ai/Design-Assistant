# Code Review Checklist

按语言/框架分类的审查检查点。审查时逐项过一遍，确保不遗漏。

---

## 通用检查点（所有语言）

### 逻辑正确性
- [ ] 条件分支是否覆盖了所有情况（包括 else / default）
- [ ] 循环是否有正确的终止条件，不会无限循环
- [ ] 递归是否有基准条件，不会栈溢出
- [ ] 状态变更是否一致（操作 A 成功但操作 B 失败时，状态如何回滚）
- [ ] 函数的返回值是否在所有代码路径上都有定义

### 边界条件
- [ ] 空值（null/undefined/nil/None）是否被正确处理
- [ ] 空集合（空数组、空 Map）作为输入时是否正常工作
- [ ] 空字符串 vs null vs undefined 是否区分处理
- [ ] 数值边界：0、负数、最大值、最小值
- [ ] 字符串：空串、超长字符串、特殊字符（emoji、换行、Unicode）

### 安全
- [ ] 无硬编码的密钥、Token、密码、API Key
- [ ] 用户输入是否经过验证和清理（sanitize）
- [ ] SQL 查询是否使用参数化查询（非字符串拼接）
- [ ] HTML 输出是否转义（防 XSS）
- [ ] 文件操作是否防止路径遍历（`../`）
- [ ] API 端点是否有鉴权和授权检查
- [ ] 敏感信息不输出到日志

### 性能
- [ ] 是否有 N+1 查询（循环中的数据库/API 调用）
- [ ] 大数据量下的算法复杂度是否可接受
- [ ] 是否有不必要的全量数据拷贝
- [ ] 网络请求是否有超时设置
- [ ] 是否可以使用缓存减少重复计算/请求

### 可维护性
- [ ] 命名是否清晰表达意图
- [ ] 函数长度是否合理（一般不超过 50 行）
- [ ] 参数数量是否合理（一般不超过 4 个）
- [ ] 是否有魔术数字（应提取为命名常量）
- [ ] 复杂逻辑是否有必要的注释说明 why（不是 what）
- [ ] 错误消息是否有助于调试

### 测试
- [ ] 新增逻辑是否有对应的单元测试
- [ ] 边界条件是否在测试中覆盖
- [ ] 错误路径是否有测试覆盖
- [ ] 测试是否稳定（不依赖外部状态、时间、随机数）

---

## TypeScript / JavaScript

### 类型安全
- [ ] 是否避免使用 `any`（优先用 `unknown` + 类型守卫）
- [ ] 泛型约束是否足够严格
- [ ] 类型断言（`as`）是否有充分理由
- [ ] 联合类型是否有穷尽检查（exhaustive check）
- [ ] 函数返回类型是否显式声明（公共 API）

### 异步
- [ ] Promise 是否被正确 await
- [ ] 是否有未捕获的 Promise rejection
- [ ] 并行 Promise 是否使用 `Promise.all` / `Promise.allSettled`
- [ ] async 函数中的错误是否被 try-catch 捕获

### React 特定
- [ ] useEffect 依赖数组是否完整
- [ ] 是否有不必要的重渲染（缺少 memo/useMemo/useCallback）
- [ ] 状态更新是否可能导致无限循环
- [ ] 组件 key 是否稳定且唯一（不用 index 作 key，除非列表静态）
- [ ] 事件处理器是否在 unmount 时清理
- [ ] 是否有闭包陷阱（stale closure）

### Node.js 特定
- [ ] 文件/流/连接是否正确关闭
- [ ] 是否有 EventEmitter 内存泄漏（监听器未移除）
- [ ] 进程退出时是否有清理逻辑（graceful shutdown）
- [ ] 环境变量是否有默认值和验证

## Python

### 类型和数据
- [ ] 可变默认参数（`def foo(bar=[])`）→ 必须用 `None` + 内部初始化
- [ ] 类型注解是否一致（函数签名、返回值）
- [ ] dataclass / Pydantic model 字段是否有合理的默认值
- [ ] 字典访问是否用 `.get()` 或 `in` 检查（避免 KeyError）

### 错误处理
- [ ] 禁止裸 `except:`（至少 `except Exception:`）
- [ ] 是否使用上下文管理器（`with` 语句）管理资源
- [ ] 异常链是否保留（`raise ... from e`）

### 并发
- [ ] GIL 是否影响性能（CPU 密集型应使用多进程）
- [ ] 异步代码是否正确使用 `async/await`
- [ ] 共享状态是否有锁保护

### Django / FastAPI 特定
- [ ] ORM 查询是否有 N+1 问题（使用 `select_related` / `prefetch_related`）
- [ ] API 端点是否有输入验证（Pydantic / serializer）
- [ ] 数据库迁移是否安全（大表加列是否有默认值）

---

## Go

### 错误处理
- [ ] 每个返回 error 的函数调用后是否检查 err
- [ ] error 信息是否提供足够上下文（`fmt.Errorf("xxx: %w", err)`）
- [ ] 是否有被静默忽略的 error（`_ = someFunc()`）

### 并发
- [ ] goroutine 是否有退出机制（context / done channel）
- [ ] 共享数据是否有 mutex 保护或使用 channel
- [ ] sync.WaitGroup 是否正确使用（Add 在 goroutine 外）
- [ ] defer 在循环中是否会导致资源延迟释放

### 接口和设计
- [ ] 接口是否足够小（Go 惯例：1-3 个方法）
- [ ] 是否在定义处而非使用处声明接口
- [ ] struct 初始化是否使用字段名（防止新增字段后出错）

---

## Rust

### 所有权和安全
- [ ] 是否有不必要的 `.clone()`（能用引用就用引用）
- [ ] `unwrap()` / `expect()` 是否只在测试或确定不会失败处使用
- [ ] `unsafe` 块是否有充分理由和安全注释
- [ ] 生命周期标注是否最小化

### 错误处理
- [ ] 是否使用 `thiserror` / `anyhow` 统一错误类型
- [ ] `?` 操作符是否正确传播错误
- [ ] 错误信息是否有足够上下文

### 性能
- [ ] 是否有不必要的堆分配（`Box` / `Vec` vs 栈上分配）
- [ ] 迭代器是否使用惰性求值
- [ ] 是否有不必要的 `to_string()` / `to_owned()` 拷贝

---

## 数据库变更

- [ ] 新增的查询是否有对应索引
- [ ] 迁移是否可回滚
- [ ] 大表 ALTER TABLE 是否会锁表
- [ ] 是否有数据一致性风险（先删后插 vs 事务）
- [ ] 新增字段是否有合理默认值
- [ ] 外键约束是否正确
- [ ] 是否考虑了并发写入场景
