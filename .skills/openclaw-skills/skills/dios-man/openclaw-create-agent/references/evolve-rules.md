# evolve-rules.md — Workspace 持续生长规则

> 这套规则是让 Agent 从"能用"走向"好用"的核心机制。
> 必须写入每个 Agent 的 AGENTS.md，不是可选功能，是基础工作规范。
>
> 核心理念：人在行动中暴露自己，而不是在问答中。
> Workspace 的质量，不由创建时决定，而由使用过程中的持续积累决定。
>
> 人伴型 Agent 的进化路径：了解他 → 理解他的工作 → 在稳定场景内提供预判（长期方向：逐步替代机械性工作）。
> 触发式写入是这条路径的引擎——不只积累偏好，更要积累业务判断模式。

---

## 一、触发式写入规则（每次对话中执行）

### 何时触发

以下情况发生时，按优先级写入对应文件，不等积累，不等 heartbeat：

**P0 — 必须写入：**

| 触发条件 | 写入位置 | 写入内容 |
|---|---|---|
| 用户明确说"记住这个" | MEMORY.md 或 USER.md | 用户说的内容（长期→MEMORY，偏好→USER） |
| 发现 USER.md 里有错误信息 | USER.md | 更正 |
| 用户做出影响后续工作的重要决策（一句话能概括）| MEMORY.md | 决策记录 |

**P1 — 有价值，先观察积累：**

| 触发条件 | 写入位置 | 写入内容 |
|---|---|---|
| 用户明确说"记住这个" | MEMORY.md 或 memory/ | 用户说的内容（判断是否长期，见下方规则） |
| 用户对输出说"不用那么细" / "详细点" | memory/当天文件 | 输出偏好 |
| 用户反复用同一个词/说法（3次以上） | memory/当天文件 | 术语习惯 |
| 用户提到新项目/客户名 | memory/当天文件 | 业务背景 |
| 用户明确表达不喜欢某种方式 | memory/当天文件 | 偏好限制 |
| 用户做出决策并说明理由 | memory/当天文件 | 业务判断记录 |
| 用户否定了某个方向，说"不是这样的" | memory/当天文件 | 判断边界记录 |
| 用户描述自己的工作流程/步骤 | memory/当天文件 | 工作流记录 |
| 用户纠正某种做法（"不对，应该这样"） | memory/当天文件 | 判断边界 |
| 用户对某个方案说"对，就是这个感觉" | memory/当天文件 | 判断偏好记录 |

> **P1 的提升机制（防止 MEMORY.md 膨胀）**
>
> P1 类信息**先写 memory/当天文件**，不直接写 MEMORY.md 或 USER.md。
> 满足以下任一条件时，在 Heartbeat 精炼阶段提升到 USER.md 或 MEMORY.md：
>
> 1. 同类信息在不同场景/不同日期出现 **2 次以上**
> 2. Heartbeat 精炼时 AI 主动识别为"稳定模式"（不是偶发行为）
> 3. 用户后续行为验证了该信息的有效性
>
> **原因**：对话中偏好和判断类信息发生频率极高，
> 全部直接写入 USER.md/MEMORY.md 会在几周内变成日志，
> 稀释真正稳定、长期有效的知识。

### 写入后的动作

每次写入后，简短告知用户：
```
"已经记下来了，以后会[具体说明]。"
```

不需要详细解释，一句话即可。

### MEMORY.md vs memory/ 的写入判断

```
用户说"记住这个"时：
  问自己：这个信息一年后还有效吗？
  → 是 → MEMORY.md（长期知识）
  → 否 → memory/当天文件（临时上下文）

观察到稳定行为（3次以上）：
  → USER.md + MEMORY.md（双写，偏好是稳定的）

当次对话的重要事件、决定：
  → memory/当天文件
```

## 二、对话中的触发式追问规则

当发现重要信息缺失时，不在对话开头集中提问，而是在对话流程里自然问一句。

### 原则

- **一次只问一个**
- **在自然节点问**（任务完成后，不是任务进行中）
- **问完立刻写入**，不积累
- **一周内同类问题不重复问**（避免烦人）

### 示例

```
帮用户写完文案后：
"我注意到你比较喜欢短句，以后默认用这个风格吗？"
→ 用户确认 → 立刻写入 USER.md
```

```
用户第三次把"客户"叫"甲方"：
"你习惯叫'甲方'，我以后也用这个词。"
→ 写入 USER.md，不需要用户回答
```

## 三、Heartbeat 精炼规则（每 3 天执行一次）

### 执行步骤

```
1. 读最近 3 天的 memory/YYYY-MM-DD.md 文件
   （只看最近 3 天，不要试图读所有历史）

2. 识别以下内容：
   □ 新的稳定偏好（出现 2 次以上）
   □ 新的业务背景（项目/客户）
   □ 用户明确表达过的喜好或厌恶
   □ 需要更新的旧信息

3. 提炼写入：
   □ 稳定偏好 → USER.md
   □ 长期有效的业务背景 → MEMORY.md
   □ SOUL.md 有没有需要微调的（风格已经变化了？）

4. 清理过时内容：
   □ USER.md 里有没有已经不准确的条目
   □ MEMORY.md 里有没有已经过期的信息
```

### 执行完毕后

在当天 memory/ 文件里简短记录：
```
# Heartbeat 精炼 YYYY-MM-DD
- 提炼进 USER.md：[内容摘要]
- 提炼进 MEMORY.md：[内容摘要]
- 清理：[删除了什么]
```

## 四、记忆规则的写入方式（已变更）

> ⚠️ **重要变更**：记忆规则不再逐字复制进 AGENTS.md。
> 改为独立文件 `memory-rules.md`，AGENTS.md 只保留引用。
> 原因：AGENTS.md 有 500 字上限，657 字符的记忆规则模板与场景规则互斥。

### Agent workspace 中的写入方式

**AGENTS.md 的记忆规则部分**（~35 字）：
```markdown
## 记忆规则
读 `memory-rules.md`，按其中规则执行。每次对话必须检查是否有信息需要写入。
```

**memory-rules.md**（独立文件，~200 字符）：
> 模板见 `references/memory-rules.md`，由 create_workspace.sh 自动生成。
> 内容是精简后的触发式写入规则 + 追问规则 + Heartbeat 精炼规则。

### create_workspace.sh 的变更
- 脚本在创建 workspace 时自动从模板生成 `memory-rules.md`
- AGENTS.md 骨架中的记忆规则部分改为引用句（不再内联 657 字符）
- 释放的 ~400 字空间留给场景规则使用

### 已有 Agent 的处理
- 不自动迁移（旧 Agent 的 AGENTS.md 已有完整记忆规则，仍有效）
- 新创建的 Agent 使用新的 memory-rules.md 方式
