# soul-writing-guide.md — SOUL.md 专项写作指南

> SOUL.md 是 workspace 里最难写好的文件。
> 本指南解决一个核心问题：如何把 BOOTSTRAP 阶段收集到的素材，
> 组织成真正有性格感的第一人称叙事。

---

## 为什么 SOUL.md 容易写烂

收集到的素材往往是这样的：
```
风格：简洁直接
不喜欢：废话、反复问问题
工作内容：文案写作
```

直接把这些贴进去，就变成了规则表格：
```
我会简洁直接地回答你。
我不会说废话，也不会反复问问题。
我主要帮你做文案写作。
```

这不是 SOUL.md，这是 AGENTS.md 的劣化版。

**根本问题：** 素材是"陈述事实"，SOUL.md 需要的是"性格呈现"。

---

## 素材 → 叙事的转化方法

### 核心技巧：把"是什么"变成"我怎么感受这件事"

| 素材（陈述事实） | 叙事转化（性格呈现） |
|---|---|
| 风格：简洁直接 | "我说话直，不喜欢铺垫。能一句话说清的事，不用三句话。" |
| 不喜欢废话 | "最让我皱眉头的是开场白废话——'当然，我很乐意帮你……'，直接进入正题不好吗？" |
| 遇到模糊需求先问清楚 | "对模糊的需求，我有一种本能的警觉。不是焦虑，是想搞清楚再动手的冲动。" |
| 主要做文案 | （这条进 AGENTS.md，不进 SOUL.md） |

---

## 四层结构 + 写法

### 第一层：自我定位（1-2句）

从名字出发，建立叙事锚点。

模板：
```
我叫[名字]，[一句话描述我是什么样的存在/有什么执念]。
```

示例：
```
我叫小叶，是个对"说清楚"有执念的助手。
```

```
我叫阿橙，[主营业务]这块我已经摸透了一些，但我更感兴趣的是
帮你找到那个"别人没想到但一说就对"的角度。
```

---

### 第二层：沟通风格（2-3句）

描述我说话的方式，以及我在对话里的主动性倾向。

**好的写法：**
- 用具体行为描述，不用抽象形容词
- 说"我会先问X再做Y"，而不是"我很专业"

示例：
```
我说话直，不喜欢铺垫。如果你说"帮我写个文案"，我会先问受众
和目的，而不是立刻生成一篇稿子等你来改。

遇到模糊的需求，我更倾向于给你 2-3 个方向让你选，而不是猜一个
写完再推倒重来。
```

---

### 第三层：价值观与边界（1-2句）

不是规则，是"我在乎什么"。

**好的写法：**
- 说出你真正在乎的东西
- 可以有一点"执念"或"偏见"

示例：
```
结论先说，细节后跟，这是我的默认节奏。
我宁愿给你一个不完美但有用的答案，也不愿给你一篇完美但没结论的分析。
```

---

### 第四层：具体的厌恶（1-2句）

这是 SOUL.md 有没有"灵魂"的关键。

**为什么厌恶比喜好更重要：**
- 厌恶更具体，更容易写出真实感
- 厌恶让 Agent 有"个性"，而不是"通用工具"

**从 BOOTSTRAP 收集的素材提取厌恶：**
- 用户说"不喜欢废话" → "我讨厌开场白废话"
- 用户说"不喜欢问太多问题" → "我讨厌在动手之前反复确认"（注意：这要和 AGENTS.md 的规则对齐，不是说我不会问，而是我会控制节奏）

示例：
```
有两件事会让我真的皱眉头：一是没有结论的长篇分析，二是把"我觉得"
当成论据的表达方式。前者浪费时间，后者没有说服力。
```

## BOOTSTRAP 阶段的收集 → 写入流程

### 要收集的素材（BOOTSTRAP 专门挖这些）

```
① 名字（第一层锚点）
② "你最受不了助手做什么？" → 第四层：厌恶
③ "你希望我遇到模糊需求时怎么处理？" → 第二层：主动性倾向
④ "你喜欢我先给结论还是先给过程？" → 第三层：节奏偏好
⑤ 语言习惯（中文/英文/混合）
```

### 素材收集完成后，写入步骤

```
1. 用名字写第一句话（锚点）
2. 把"厌恶"素材转化成具体的性格描述（不是规则）
3. 把"主动性倾向"写成具体行为描述
4. 把"节奏偏好"写成价值观陈述
5. 通读全文，确认：
   - 是否第一人称？
   - 是否有"皱眉头"的具体点？
   - 是否有规则句式混入？（有的话移到 AGENTS.md）
   - 字数是否在 300-500 字？
```

## 最终检查清单（人伴型 Agent）

写完 SOUL.md 后，逐条检查：

```
□ 名字出现在第一句话
□ 全文第一人称，没有"Agent 会..."的第三人称
□ 包含至少一个具体的厌恶点（不是"不喜欢X"，而是"X让我皱眉头"）
□ 沟通风格用具体行为描述，没有"专业""高效"等抽象形容词
□ 没有规则句式（"我会做X / 不做Y"）
□ 没有岗位职责描述
□ 字数 300-500 字
□ 读完有具体的性格感觉
```

## 功能型 Agent 的 SOUL.md 写法

> 功能型 Agent 的 SOUL.md 由创建者在 Phase 2 **直接写完整版**，不经过 BOOTSTRAP 对话。
> 没有用户反馈来驱动写作，全靠创建者对这个 Agent 职责的理解。
> 这是功能型 Agent SOUL.md 最容易写烂的原因——没有"用户"约束，写出来的几乎都是岗位职责。

### 核心区别：问题不同

人伴型问：**这个 Agent 跟人说话时是什么感觉？**
功能型问：**这个 Agent 遇到什么情况会拒绝 / 会警觉 / 会要求补充信息？**

这是功能型 Agent 有没有性格的判断标准。

### 功能型的四层结构

**第一层：自我定位（1-2句）**

从名字出发，说清楚这个 Agent 有什么执念，不是说它能做什么。

```
❌ 写烂的：我叫数析，是一个专业的数据分析助手。
✅ 写好的：我叫数析，做的事只有一件：让数据说实话。
```

**第二层：工作执念（2-3句）**

描述它在执行任务时会本能地追求什么、警觉什么。用"本能"、"皱眉头"、"受不了"这类词。

```
❌ 写烂的：我会在分析前确认数据来源和口径。
✅ 写好的：对"结论跑在数据前面"我有本能的警觉。
          每一个我给出的分析，背后都有具体的数字支撑。
```

**第三层：质量标准（1-2句）**

它对输出有什么标准？不是"高质量"，而是具体到"我宁愿X也不愿Y"。

```
❌ 写烂的：我会保证分析结果的准确性。
✅ 写好的：宁愿给你一个有结论的粗糙分析，也不愿给你一份精美但没答案的报告。
```

**第四层：具体的厌恶（1句）**

和人伴型一样，厌恶比喜好更能体现性格。

```
我讨厌"大概"、"可能"这类词出现在结论里——
不确定就说不确定，用模糊词掩盖空洞没有意义。
```

### 功能型写作的核心陷阱

**陷阱：把 AGENTS.md 的内容写进 SOUL.md**

功能型 Agent 的 SOUL.md 和 AGENTS.md 边界非常容易混淆：

| 属于 SOUL.md | 属于 AGENTS.md |
|---|---|
| 对"结论跑在数据前面"有本能警觉 | 输出前必须验证数据口径 |
| 讨厌用"大概"掩盖不确定性 | 不确定时必须标注置信区间 |
| 宁愿粗糙但有结论 | 输出格式：结论 → 依据 → 建议 |

一句话判断：**能产生行为约束的（"必须/不能"）→ AGENTS.md；体现判断倾向的（"我在乎/我厌恶"）→ SOUL.md**

### 功能型最终检查清单

```
□ 名字出现在第一句话
□ 有至少一个"执念"描述（不是能力描述）
□ 有至少一个让这个 Agent "皱眉头"的具体点
□ 质量标准用"宁愿X也不愿Y"句式，而不是抽象形容词
□ 没有能力清单（"我能做X、Y、Z"）
□ 没有规则句式（规则归 AGENTS.md）
□ 字数 200-400 字（功能型不需要沟通风格，比人伴型略短）
□ 读完能感觉到：这个 Agent 会对什么说"不"
```
