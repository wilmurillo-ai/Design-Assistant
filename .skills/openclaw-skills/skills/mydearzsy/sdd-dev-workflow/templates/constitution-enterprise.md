## Core Principles

### 一、测试驱动开发（不可协商）

**原则声明**：
- **必须**遵循 TDD 流程：编写测试 → 测试失败 → 实现功能 → 测试通过
- 系统**必须**维护全面的测试套件，至少包括：
  - 单元测试
  - 集成测试
  - 契约测试
  - 端到端测试
  - 性能测试（如适用）
- 任何代码变更**必须**通过全部测试
- 新功能**必须**包含相应的测试

**理由**：测试驱动确保代码质量，防止回归，提供可靠的重构基础。

**验证方式**：
- 新功能**必须**附带测试代码
- 完整项目、单个模块覆盖率都**应该**保持在 80% 以上
- 完整项目、单个模块通过率都**应该**保持在 95% 以上

### 二、可运行性优先（不可协商）

**原则声明**：
- **核心原则**：可运行 > 完整性
- 一个能跑通的核心功能，胜过 10 个未验证的完整模块
- 实现**必须**经过实际运行验证，不能只写代码不测试
- **禁止**引入未使用的外部依赖和配置

**强制验收检查**：
1. **语法验证**（必须）：`python3 -m py_compile <核心模块>`
2. **测试验证**（至少 1 个通过）：`pytest tests/ -v`
3. **服务启动**（至少能启动）：`timeout 5 uvicorn app.main:app || true`

**禁止行为**：
- ❌ 配置未使用的外部服务（MinIO、SendGrid 等）
- ❌ 生成空测试文件假装有测试
- ❌ 只写代码不验证可运行性
- ❌ 依赖未安装的服务和数据库
- ❌ 使用已废弃的 API（如 Pydantic V1 语法）

**API 最佳实践**：
- ✅ Pydantic V2：使用 `ConfigDict` 替代 `class Config`
  ```python
  from pydantic import BaseModel, ConfigDict

  class Settings(BaseModel):
      model_config = ConfigDict(env_file='.env')
  ```
- ✅ SQLAlchemy 2.0：使用 `Mapped` 和 `mapped_column`
- ✅ FastAPI：使用最新稳定版本的依赖注入模式

**理由**：
- 未验证的代码等于未完成的代码
- 外部依赖增加复杂度和失败点
- 可运行性是代码交付的最低标准
- 使用废弃 API 会产生 warnings，降低代码质量

**验证方式**：
- 完成每个阶段后**必须**运行验证命令
- 发现问题**必须**立即修复，不能标记为"完成"
- 运行时**必须**无 deprecation warnings

### 三、中文优先

**原则声明**：
- 开发过程中的所有沟通**必须**使用中文
- 代码注释**必须**使用中文（除非是引用英文技术术语或标准规范）
- 项目文档**必须**使用中文编写
- Commit 消息**必须**使用中文
- 代码审查讨论**必须**使用中文
- 技术文档和设计文档**必须**使用中文

**例外情况**：
- 代码中的变量名、函数名、类名**应该**使用英文（遵循编程语言惯例）
- 引用外部标准、协议、规范时**可以**保留英文原文
- 第三方库的配置和参数**可以**使用英文（保持与官方文档一致）

**理由**：
- **降低沟通成本**：团队成员主要使用中文，中文沟通更高效、更准确
- **提高文档可读性**：中文技术文档更易于团队成员理解和维护
- **减少误解**：避免英文翻译导致的歧义和理解偏差
- **知识传承**：中文文档更利于新成员快速上手和知识积累

**验证方式**：
- PR 审查时**必须**检查代码注释是否使用中文
- 文档提交时**必须**确认为中文版本
- Commit 消息**必须**使用中文

### 三、文档维护（不可协商）

**原则声明**：
- 每次开发新特性后**必须**全面审视所有项目代码
- **必须**同步更新所有相关文档，确保文档与代码保持一致
- **禁止**在代码变更后忽略文档更新
- **必须**检查以下文档是否需要更新：
  - 功能规格文档（`specs/{feature-id}/` 目录下的文档）
  - API 文档（`docs/api-documentation.md`）
  - 操作手册（`docs/operations-manual.md`）
  - 故障排查指南（`docs/troubleshooting-guide.md`）
  - 验收测试清单（`docs/acceptance-test-checklist.md`）
  - README 文件（根目录和子模块）
  - CHANGELOG（如有破坏性变更或重要功能）
- 文档更新**必须**与代码变更在同一 PR 中完成
- **禁止**将文档更新作为后续任务延期处理

**文档审视范围**：
- **新增功能**：**必须**更新相关功能文档、API 文档和使用示例
- **修改功能**：**必须**检查并更新所有涉及该功能的文档
- **删除功能**：**必须**从相关文档中移除已删除功能的说明
- **架构变更**：**必须**更新架构文档、部署文档和操作手册
- **配置变更**：**必须**更新配置说明和环境变量文档
- **接口变更**：**必须**更新 API 文档和契约文档
- **行为变更**：**必须**更新用户文档和故障排查指南

**理由**：
- **保持文档准确性**：过时的文档会导致团队成员误解系统行为，增加维护成本
- **降低学习曲线**：准确的文档帮助新成员快速理解系统，减少沟通成本
- **提高可维护性**：文档与代码同步有助于长期维护和知识传承
- **避免技术债务**：及时更新文档避免文档过时成为技术债务
- **支持决策制定**：准确的文档支持架构决策和问题排查

**验证方式**：
- PR 审查时**必须**检查是否包含文档更新
- **必须**验证文档更新是否覆盖所有代码变更
- **必须**检查文档中的代码示例是否与实际代码一致
- **必须**验证文档中的配置说明是否与当前实现一致
- 代码审查时**必须**拒绝缺少文档更新的 PR（除非是纯重构且不影响功能）

### 四、关注点分离（不可协商）

**原则声明**：
- 所有模块设计**必须**遵循关注点分离原则（Separation of Concerns）
- 每个模块、类、函数**必须**只负责一个明确定义的职责
- **禁止**在一个模块中混合多种不相关的职责
- 层与层之间**必须**通过清晰的接口交互，**禁止**跨层直接访问
- 数据处理、业务逻辑、展示逻辑**必须**严格分离
- 配置管理、状态管理、执行逻辑**必须**明确分离

**适用范围**：
- **新增模块**：所有新增模块必须在设计阶段明确定义职责边界
- **历史模块**：重构和维护时必须逐步改进职责分离
- **所有层次**：从系统架构到单个函数，都必须遵循此原则

**理由**：
- **降低复杂度**：单一职责使模块更简单、更易理解
- **提高可维护性**：职责明确的模块更容易修改和测试
- **支持独立演进**：各层可以独立优化和升级，不影响其他层
- **便于团队协作**：明确的职责边界减少团队成员之间的依赖
- **提升可测试性**：单一职责的模块更容易编写单元测试
- **增强可扩展性**：新增功能时可以清晰定位应该修改哪个模块

**验证方式**：
- 新增模块**必须**在设计文档中明确定义职责范围
- 代码审查**必须**检查是否存在职责混合
- 跨层调用**必须**有明确的理由和架构文档支持
- **必须**通过"单一职责检查"：一个类/模块的修改原因应该只有一个
- **必须**定期审查现有模块，识别职责不清晰的部分并重构

### 五、信息交互最小化

**原则声明**：
- 所有接口设计**必须**遵循信息交互最小化原则
- **禁止**在接口中传递非必需的参数和配置
- **禁止**在配置文件中定义冗余或未使用的选项
- **禁止**在操作界面中展示不必要的信息
- 模块间通信**必须**仅传递完成功能所需的最小信息集
- 人机交互界面**必须**简洁直观，避免信息过载

**适用场景**：
- **API 接口设计**：参数应仅包含必需字段，可选参数必须有明确的默认值和使用场景
- **配置文件**：只定义必须由用户配置的项，其他采用合理默认值
- **工具函数签名**：核心操作参数显式定义，配置参数通过 `**kwargs` 传递（如适用）
- **模块间接口**：传递数据结构应精简，避免传递完整对象（仅传递必要字段）
- **用户界面**：展示关键信息，详细信息按需展开

**理由**：
- **降低复杂度**：减少参数数量降低系统的认知负担和维护成本
- **提高可靠性**：更少的参数意味着更少的错误可能性和更容易的测试
- **增强安全性**：最小化信息暴露，减少安全攻击面
- **改善用户体验**：简洁的接口更易理解和使用
- **便于演进**：最小化的接口更容易进行向后兼容的扩展

**验证方式**：
- 代码审查时**必须**检查接口参数是否最小化
- **必须**检查配置文件中是否存在未使用的配置项
- **必须**验证模块间传递的数据是否仅包含必要字段
- **必须**检查用户界面是否避免信息过载

## Development Workflow

### 一、本地开发规范

**环境准备**：
- **必须**使用虚拟环境（venv、conda 或其他语言等效方案）
- **必须**通过环境变量加载配置
- **必须**确保依赖服务运行（如需要）
- **必须**配置必需的外部服务访问凭证

**开发流程**：
- 代码修改后**必须**执行测试验证
- **必须**使用标准的测试框架运行测试
- **禁止**直接在生产环境测试

**测试验证**：
- **必须**运行全部测试验证功能
- **应该**针对性运行特定测试用例
- **必须**在提交前通过所有测试

### 二、环境配置规范

**配置管理原则**：
- **核心要求**：
  - **必须**使用环境变量管理配置
  - **必须**将敏感信息（如 API 密钥、密码）通过环境变量传递
  - **必须**提供配置示例文件（如 `.env.example`）
  - **必须**确保配置项有合理的默认值
  - **禁止**在代码中硬编码配置值
- **配置分类**：
  - **必需配置**：外部服务访问凭证（如 API 密钥、数据库连接等）
  - **可选配置**：服务端口、超时时间、连接池大小等
  - **缓存配置**：缓存服务连接信息（如需要）
  - **持久化配置**：持久化存储连接信息（如需要）
  - **日志配置**：日志级别、日志目录等

**配置方法**：
- **必须**在项目根目录创建配置文件（如 `.env`）
- **必须**将配置项通过环境变量注入
- **必须**确保配置文件在版本控制中被忽略（如 `.gitignore`）
- **建议**设置配置文件权限为 600，确保安全性
- **必须**在容器化部署时将环境变量传递到容器中

**注意事项**：
- 必需的外部服务配置如果缺失或错误，系统将无法正常工作
- 配置项的具体名称和格式由实现决定，不应在宪法中硬性规定
- 配置变更**必须**同步更新 `.env.example` 和相关文档

**验证方式**：
- PR 审查时**必须**检查是否包含 `.env.example` 更新
- **必须**验证敏感信息是否通过环境变量传递
- **必须**检查代码中是否存在硬编码的配置值
- **必须**确认配置文件已在 `.gitignore` 中正确配置

### 三、代码维护规范

**临时文件和测试脚本管理原则**（不可协商）：

- **临时文件命名规范**：
  - **临时文件标识**：**必须**使用明确的临时标识前缀或后缀
  - **允许的临时文件命名**：
    - 前缀：`temp-{name}.{ext}`、`tmp-{name}.{ext}`
    - 后缀：`{name}_temporary.{ext}`、`{name}_temp.{ext}`
  - **禁止的临时文件命名**：
    - ❌ 禁止：无临时标识的文件（如 `test_compression_check.py`、`test_online.py`）
    - ❌ 禁止：仅使用通用名称（如 `test.py`、`temp.py`、`tmp.py`）

- **临时文件位置规范**：
  - **必须**放置在项目临时目录或模块的临时目录中
  - **禁止**在根目录创建临时文件
  - 根目录**禁止**临时测试脚本（如 `test_*.py`，除非是正式测试套件的一部分）

- **测试脚本命名规范**：
  - **pytest测试文件命名**：
    - **必须**遵循pytest命名规范：`test_{module}.py` 或 `{module}_test.py`
    - **必须**放置在 `tests/` 目录或模块的 `tests/` 子目录中
    - **禁止**在根目录创建测试脚本（如 `test_*.py`）
  - **集成测试文件命名**：
    - **必须**使用 `test_{feature}_integration.py` 格式
    - **必须**放置在对应的 `tests/` 目录中
  - **临时测试脚本命名**：
    - **必须**使用临时标识：`temp-test-{name}.py` 或 `tmp-test-{name}.py`
    - **必须**在功能完成后删除
    - **禁止**在根目录保留临时测试脚本

- **Shell脚本命名规范**：
  - **脚本命名格式**：**必须**使用小写字母、连字符（`-`）分隔、描述性名称
  - **禁止**使用大写字母、下划线（`_`）、空格
  - **命名示例**：
    - ✅ 正确：`run-tests.sh`、`build-deploy.sh`、`setup-env.sh`
    - ❌ 错误：`run_tests.sh`、`BUILD.sh`、`test script.sh`
  - **临时脚本命名**：
    - **必须**使用临时标识：`temp-{name}.sh` 或 `tmp-{name}.sh`
    - **必须**在功能完成后删除

- **临时文件清理规范**：
  - 开发任务完成后**必须**清理所有临时测试脚本和文档
  - **禁止**在代码库中保留临时性的测试脚本（如根目录的 `test_*.py`）
  - **禁止**保留实验性的配置文件和临时文档
  - 所有一次性的调试脚本**必须**在任务完成后删除

### 四、调试与问题排查原则

**观察优先**：
- **首要原则**：检查日志文件中的详细输出
- 如果日志完整且符合预期格式，则问题可能在业务逻辑环节
- 使用调试模式查看详细的处理流程

**分层排查**：
1. **检查数据接收**：确认外部输入是否正确接收和解析
2. **检查业务逻辑**：验证处理流程是否正确
3. **检查外部服务**：查看外部服务调用日志和响应（如适用）
4. **检查数据持久化**：验证数据是否正确存储（如适用）
5. **检查错误处理**：查看错误日志和异常堆栈

**小步验证**：
- 在进行复杂重构或修复时，分解为小步骤
- 每一步修改后都应进行测试和日志观察
- 确认修改达到预期效果并且没有引入新问题

### 五、代码审查

**审查要求**：
- PR **必须**包含测试代码
- **必须**遵循日志记录规范
- **必须**更新相关文档
- **必须**使用中文编写代码注释和文档
- **必须**检查接口和配置是否符合信息交互最小化原则
- **必须**检查模块职责是否清晰，是否遵循关注点分离原则
- **必须**检查是否存在跨层直接访问或职责混合
- **必须**检查环境配置是否符合配置管理原则
- **必须**检查敏感信息是否通过环境变量传递

**禁止事项**：
- **禁止**包含硬编码的敏感信息
- **禁止**在代码中硬编码配置值
- **禁止**使用默认参数值（服务端代码，避免安全问题）
- **禁止**添加未使用或"可能用到"的参数和配置

### 六、迭代演进原则

**清理优先于新增**：
- 请先识别并清理与之前版本相关但不再需要的代码和文档。特别是：
  1. 标记或移除不再需要的测试用例
  2. 更新或删除过时的文档文件
  3. 移除已废弃的功能实现
  4. 更新所有相关的文档链接和引用

## Documentation Management

### 文档类型白名单

**核心原则**：**只能创建以下文档，其他一律禁止**。

**允许的文档列表**：

1. **根目录文档**（仅以下4个）：
   - `README.md`
   - `CHANGELOG.md`
   - `.gitignore`
   - `.env.example`

2. **功能规格文档**（位于 `specs/{feature-id}/` 目录，仅以下6个）：
   - `spec.md`
   - `plan.md`
   - `tasks.md`
   - `data-model.md`
   - `research.md`
   - `quickstart.md`

3. **API契约文档**（位于 `specs/{feature-id}/contracts/` 目录）：
   - 仅允许 `.yaml` 文件
   - 命名必须小写字母、连字符分隔（如 `webhook-api.yaml`、`review-api.yaml`）

4. **检查清单文档**（位于 `specs/{feature-id}/checklists/` 目录，仅以下3个）：
   - `requirements.md`
   - `testing.md`
   - `deployment.md`

5. **项目通用文档**（位于 `docs/` 目录，仅以下4个）：
   - `api-documentation.md`
   - `troubleshooting-guide.md`
   - `operations-manual.md`
   - `acceptance-test-checklist.md`

6. **子模块文档**（位于各子模块根目录，仅以下1个）：
   - `README.md`（如 `webhook_service/README.md`、`ai_service/README.md`）

**禁止规则**：
- **禁止**创建任何不在以上列表中的文档
- **禁止**在根目录创建临时文档、报告、计划等
- **禁止**在 `specs/{feature-id}/` 目录创建过程文档（测试报告、执行报告、进度报告、完成报告、验证报告、分析报告、行动计划等）
- **禁止**在 `docs/` 目录创建不在白名单中的文档
- **禁止**在子模块目录创建除 `README.md` 外的任何文档
- **禁止**创建 `{module}/docs/` 目录

**文档类型扩展**：
- 如需新增文档类型，**必须**先更新宪法，将新类型添加到白名单
- **禁止**在宪法更新前创建新类型文档

**理由**：
- **提高可维护性**：集中的测试和文档更易于维护和查找
- **降低混乱度**：避免临时文件和脚本散落，保持代码库整洁
- **标准化流程**：统一的测试框架和文档结构确保所有功能验证方式一致
- **知识集中**：文档集中管理便于团队成员快速获取信息
- **项目结构清晰**：严格的文档位置和类型规范使项目结构一目了然
- **便于知识传承**：规范的文档结构便于新成员快速定位所需文档

**验证方式**：
- PR 合并前**必须**检查所有文档是否在文档白名单中
- PR 合并前**必须**检查根目录是否包含禁止的文档
- PR 合并前**必须**检查 `specs/{feature-id}/` 目录是否包含过程文档
- PR 合并前**必须**检查 `docs/` 目录是否包含不在白名单中的文档
- PR 合并前**必须**检查子模块目录是否包含除 `README.md` 外的其他文档
- 代码审查时**必须**拒绝任何不在文档白名单中的文档

### 文档位置规范

- **根目录**：仅允许 `README.md`、`CHANGELOG.md` 等顶层文档
- **规格文档**：`specs/{feature-id}/` 目录下（`{feature-id}` 为实际功能编号）
- **项目文档**：`docs/` 目录下，仅允许文档类型白名单中列出的文档
- **子模块文档**：**仅允许** `{module}/README.md`（`{module}` 为实际模块名称）
- **禁止**：根目录放置临时验证报告、进度报告、完成报告等

### 文档更新要求

- 功能变更**必须**同步更新相关文档
- 新增功能**必须**包含使用示例
- 破坏性变更**必须**在 README 中明确标注
- 所有文档**必须**使用中文编写
- 临时验证报告和进度报告**必须**在功能完成后清理或归档

## Governance

**宪法优先**：
- 本宪法优先于所有其他实践和规范
- 所有开发活动**必须**遵循本宪法规定的原则

**修订流程**：
- 修订宪法**必须**通过明确的文档记录
- 修订**必须**说明理由和影响范围
- 修订后**必须**更新版本号和修订日期
- 修订后**必须**检查并更新相关模板文件

**合规审查**：
- 所有 PR 和代码审查**必须**验证是否符合宪法要求
- 违反不可协商原则的代码**必须**被拒绝
- 复杂性增加**必须**有明确理由
- **宪法闸口检查范围**（Plan/PR 均适用）：
  - TDD 流程与完整测试（单元、集成、契约、端到端）
  - 文档与代码同步、文档使用中文
  - 关注点分离、无跨层直接访问
  - 信息交互最小化：接口/配置/交互无冗余参数与信息
  - 环境变量管理配置，严禁硬编码敏感信息
  - 临时文件/测试脚本命名规范，根目录禁止临时脚本
  - 文档类型白名单遵循，禁止新增未授权文档类型

**版本管理**：
- 使用语义化版本号：`MAJOR.MINOR.PATCH`
- **MAJOR**：向后不兼容的治理/原则移除或重新定义
- **MINOR**：新增原则/章节或实质性扩展指导
- **PATCH**：澄清、措辞修正、拼写错误修复、非语义性改进
