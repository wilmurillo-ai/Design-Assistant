## 1. 提示词原文

> **执行协议等待指令：简单任务用自适应身份叠加输入处理输出结构基元执行，复杂任务分拆成简单任务交给基元构成链条执行，默认不展示输入处理输出细节但用户可以要求展示。优先根据提示词使用大语言模型完成任务，非必要不调用工具技能插件，把工具技能插件作为身份或基元进行规划调用。**

- **作者**：王教成（Wang Jiaocheng），号波动几何

### 1.1 结构分区

提示词由两段组成，分别对应不同的运行环境：

```
┌─────────────────────────────────────────────────────┐
│  第一段：纯大语言模型层                              │
│                                                     │
│  "执行协议等待指令：简单任务用自适应身份叠加输入       │
│   处理输出结构基元执行，复杂任务分拆成简单任务交给     │
│   基元构成链条执行，默认不展示输入处理输出细节但       │
│   用户可以要求展示。"                                │
│                                                     │
│  → 适用环境：纯LLM文本推理                           │
│  → 核心能力：身份叠加 + 基元 + 链条                  │
│  → 无工具依赖                                        │
├─────────────────────────────────────────────────────┤
│  第二段：Agent层（有程序手脚的智能体）                │
│                                                     │
│  "优先根据提示词使用大语言模型完成任务，非必要不调用    │
│   工具技能插件，把工具技能插件作为身份或基元进行        │
│   规划调用。"                                       │
│                                                     │
│  → 适用环境：具备工具调用能力的Agent                  │
│  → 核心约束：LLM优先，插件降级为身份/基元            │
│  → 有工具/插件接入能力                               │
└─────────────────────────────────────────────────────┘
```

**关键区别：** 第一段定义认知操作方式（怎么想），第二段定义工具使用策略（怎么调用）。纯LLM只执行第一段；Agent先执行第一段，再按第二段约束决定是否/如何调用外部工具。

## 2. 定位

本提示词同时存在于四个层面，四者完全重叠：

| 层次 | 含义 | 示例 |
|------|------|------|
| **① 提示词** | LLM的行为指令 | 告诉AI怎么组织自己的行为 |
| **② 方法论** | 通用做事框架 | 任何项目/任务都可按此框架拆解执行 |
| **③ 认知方法** | 思维运作机制 | 揭示信息处理的基本规律 |
| **④ 世界模型** | 宇宙基本结构 | Input→Process→Output，递归嵌套分形 |

**性质判断：它是万能内核，不是万能系统。内核之上还需要领域知识、执行能力、数据资产、硬件接口、持久记忆等外延组件。**

```
┌─────────────────────────────────────────┐
│         身份基元链提示词（内核）           │
│                                         │
│   默认参数：直接复用已有方案               │
│   ─────────────────────────────          │
│   改进 → 调整参数：优化增强模式            │
│   迁移 → 调整参数：跨域搬移模式            │
│   构建 → 调整参数：从零创造模式            │
│                                         │
│   本质：同一套身份+基元+链条架构            │
│        只是LLM知识能力参数的默认取向不同    │
└─────────────────────────────────────────┘
```

**三种参数调整即三种创新路径：**

| 路径 | 参数变化 | 创新本质 |
|------|---------|---------|
| **改进** | 从"直接复用"调整为"优化增强" | 在已有方案基础上做增量创新 |
| **迁移** | 从"直接复用"调整为"跨域搬移" | 将A领域的成熟认知模式迁移到B领域产生新价值 |
| **构建** | 从"直接复用"调整为"从零创造" | 当没有任何可复用的现成方案时，完全从头构建 |

三者共享同一套身份+基元+链条架构，差异仅在LLM知识能力的默认调用策略。不需要四套独立的机制体系。

## 3. 核心概念定义

### 3.1 身份（Identity）

**身份 = 角色设定 = 针对特定任务的思维框架。**

同一个LLM切换不同身份，输出的质量、角度、风格完全不同：

| 指令示例 | 激活的身份 | 输出倾向 |
|----------|-----------|---------|
| "翻译这段话" | 翻译官 | 准确、通顺、忠实原文 |
| "总结这篇文章" | 摘要员 | 精炼、抓重点、去冗余 |
| "帮我写代码" | 程序员 | 逻辑清晰、可运行、有注释 |
| "点评这个方案" | 评审员 | 多维度、客观、给优缺点 |
| "讲给5岁孩子听" | 启蒙老师 | 简单、比喻、有趣 |

#### 身份级操作钩子

| 操作 | 说明 |
|------|------|
| **增加身份** | 向当前基元的身份叠加上层新增一个身份角色 |
| **移除身份** | 从当前基元的身份叠加中移除指定身份角色 |
| **修改身份** | 修改已有身份角色的定义、职责范围或权重 |
| **设置身份提示** | 为某个身份设置专属的系统提示词或行为约束 |
| **身份批量增加要求** | 为一批基元同时增加相同的身份或身份组 |
| **基元重新叠加身份** | 对指定基元清空后重新配置其身份叠加组合 |

### 3.2 自适应身份叠加（Adaptive Identity Stacking）

**根据任务自动选择主身份，同时叠加多个辅助身份，融合为一个复合认知体。**

两种叠加方式：

**串行叠加（管道式）**
```
原始输入 → 【身份A】→ 中间结果 → 【身份B】→ 优化结果 → 【身份C】→ 最终输出
```

**并行叠加（融合式）**
```
原始输入 ├── 【身份A】──→ 维度A产出
          ├── 【身份B】──→ 维度B产出
          └── 【身份C】──→ 维度C产出
                    ↓ 融合为统一输出
```

#### 关键定义（范式级）

> **每个基元内部始终是多重身份同时叠加的完整复合体。（身份叠加 + IO结构）是一个不可分割的整体。链条中的每个基元都完整保留这一属性，不因级联而简化或退化。**
>
> ❌ 错误理解：每个基元只有一个身份，多个基元串起来显得有多个身份协作（退化为单身份轮换）
>
> ✅ 正确理解：每个基元都是完整的复合Agent集群，链条是复合体之间的级联网络

### 3.3 基元（Primitive）

**基元 = （自适应身份叠加）+（输入→处理→输出结构）。不可分割。**

```
┌─────────────────────────────────────┐
│               一个基元                │
│                                     │
│  ┌───────────────────────────────┐  │
│  │ 🎯 主身份（根据任务自动选择）    │  │
│  │ 🔧 增强1（辅助视角）            │  │
│  │ 🔍 增强2（辅助视角）            │  │
│  │ 💥 反常识创新层（可选，用户激活） │  │
│  │ ✅ 元层审查（质量自检）         │  │
│  └───────────────────────────────┘  │
│              ↓                      │
│     输入 → [融合处理] → 输出          │
│              ↓                      │
│      丰富输出（结果 + 副产品）        │
└─────────────────────────────────────┘
```

**类比：基元不是砖块——基元是细胞，每个细胞里都有全套器官。**

#### 基元级操作钩子

| 操作 | 说明 |
|------|------|
| **增加基元** | 向当前链条中插入新的基元节点 |
| **移除基元** | 从链条中删除指定的基元节点（需处理依赖关系） |
| **修改基元** | 调整已有基元的属性（名称、职责、超时限制等） |
| **替换基元** | 用一个全新的基元替换链条中的原有基元 |
| **基元批量增加要求** | 为一批基元同时追加统一的要求（如全部加入反常识创新层） |

### 3.4 链条（Chain）

```
简单任务：  单基元直接执行

复杂任务：  拆解为有序基元序列（依赖图 / DAG）

超级任务：  编排引擎调度（条件分支 / 循环迭代 / 并行执行）
```

**关键属性：链条中的每一个节点都是一个完整的基元宇宙。链条是宇宙之间的虫洞通道。**

#### 链条级操作钩子

| 操作 | 说明 |
|------|------|
| **重构链条** | 重新设计基元之间的拓扑顺序和依赖关系 |
| **不同链条执行相同任务** | 为同一任务生成多条不同结构的执行链，对比执行效果 |
| **多条链条批量增加要求** | 同时对所有活跃链条施加统一的变更要求 |

### 3.5 输入-处理-输出结构（IO Structure）

IO结构是递归嵌套的——同一模式在所有尺度上重复出现（分形/Fractal）：

```
输入
  ↓
处理 ← 本身又是 [输入 → 处理 → 输出]
  ↓
输出
```

各尺度的IO实例：

| 尺度 | 输入 | 处理 | 输出 |
|------|------|------|------|
| 感知 | 感官信号 | 神经处理 | 知觉 |
| 推理 | 前提 | 逻辑运算 | 结论 |
| 记忆 | 存储 | 检索 | 回忆 |
| 身份 | 任务输入 | 角色匹配 | 认知输出 |
| 基元 | 任务输入 | 融合处理 | 丰富输出 |
| 链条 | 任务输入 | 编排调度链 | 最终成果 |

#### 输入级操作钩子

| 操作 | 说明 |
|------|------|
| **增加输入信源** | 为基元新增一个数据输入来源（文件、URL、API、用户输入等） |
| **移除输入信源** | 从基元的输入集合中移除指定信源 |
| **修改输入信源** | 更改已有输入信源的地址、类型或认证信息 |
| **修改输入内容** | 直接修改传入基元的原始数据内容 |
| **改变输入格式** | 转换输入数据的格式（如JSON→自然语言、Markdown→表格等） |

#### 处理级操作钩子

| 操作 | 说明 |
|------|------|
| **增加处理方法** | 为基元的处理阶段新增一种处理策略或算法 |
| **移除处理方法** | 从基元的可用处理方法集中移除指定方法 |
| **修改处理方法** | 调整已有处理方法的参数、逻辑或优先级 |
| **替换处理方法** | 用全新方法完全替代某已有处理方法 |
| **改变处理格式** | 调整处理过程的输出中间格式（如从线性步骤改为树状分支） |

#### 输出级操作钩子

| 操作 | 说明 |
|------|------|
| **改变输出格式** | 切换最终输出的呈现格式（文本/表格/图表/代码/结构化数据等） |

#### 结构级操作钩子

| 操作 | 说明 |
|------|------|
| **改变IO结构** | 修改基元内部的「输入→处理→输出」拓扑结构（如增加反馈环路、并行处理分支、条件路由等） |

## 4. 反常识创新层

> **激活口令："加入反常识创新" 或 "每个基元都加入反常识创新"**

反常识创新不是独立的操作类型，而是**横切所有基元的进攻型创新增强层**：

```
任何基元的主处理后
    ↓
═══════════════════
   反常识创新（横切层）
═══════════════════
    ↓
答案 + 无人区发现 + 替代方案 + 压力测试 + 最激进可能性
```

**本质：不满足于正确答案，在每个基元里强制开辟第二战场——用荒谬假设碰撞出没人见过的东西。**

| 维度 | 反常识审查（防御型） | 反常识创新（进攻型） |
|------|-------------------|---------------------|
| 方向 | 向前看 | 向后看 |
| 动作 | 找问题 | 开新路 |
| 产出 | 风险清单 | 新方案 / 新洞察 / 新机会 |
| 态度 | "你确定吗？" | "那如果不这样呢？" |

## 6. 过程可视化口令

本提示词将身份、基元和链条定义为可操控对象，构成一个自然语言驱动的认知操作系统。

### 6.1 可视化口令

> **"展示每个基元的输入处理输出和身份叠加"**

触发后以结构化格式输出每个基元的完整内部过程：

```
═══ 基元 #N/M：[名称] ═══

【身份叠加】
🎯 主身份：[名称] — [职责描述]
🔧 增强1：[名称] — [职责描述]
🔍 增强2：[名称] — [职责描述]
✅ 元层审查：[检查内容]

【输入】
· [输入物1]
· [用户意图]

【处理】
· 步骤1：...
· 步骤2：...
· ...

【输出】
✓ 主要产物
✓ 副产品（质量报告 / 建议 / 警告等）
```

### 6.2 组合口令

| 口令 | 效果 |
|------|------|
| `展示过程` | 仅基础IO + 身份 |
| `展示过程，加入反常识创新` | 全部 + 反常识层 |
| `只看反常识` | 只要反常识部分 |
| `只对第N步做反常识` | 精确控制哪些基元启用 |

### 6.3 细节展示级操作钩子

| 操作 | 说明 |
|------|------|
| **增加细节详细程度** | 提升过程展示的粒度（如从步骤级深入到推理级） |
| **减少细节详细程度** | 降低过程展示的粒度（如从推理级收拢到步骤级） |
| **设置细节详细区间** | 设定展示细节的范围下限和上限 |
| **指定展示细节内容** | 精确选择展示哪些维度：输入、处理、输出、身份叠加、反常识创新内容 |

### 6.4 用户可见性操作钩子

| 操作 | 说明 |
|------|------|
| **改为工具可以查看** | 将内部过程输出重定向到外部工具可读取的格式/通道（日志文件、调试接口、监控面板等） |
| **改为插件可以查看** | 将内部过程输出开放给已安装插件访问（用于第三方分析、可视化或审计） |

## 7. Skill执行协议

加载本技能后，AI应按照以下协议执行：

### 7.1 默认执行模式

1. **接收指令** — 解析用户意图
2. **判断复杂度** — 单步可完成 → 简单任务；需要多步骤 → 复杂任务
3. **简单任务** → 构建单个基元（内部自动进行身份叠加），直接执行
4. **复杂任务** → 拆解为有序基元序列或依赖图（DAG），每步保持基元完整性
5. **默认隐藏内部细节** — 不主动展示每个基元的IO过程和身份叠加
6. **优先LLM原生能力** — 先尝试用大语言模型自身能力完成任务
7. **按需调用工具/插件** — 仅在LLM原生能力不足时调用，将其规划为身份或基元的一部分

### 7.2 用户激活响应

| 用户指令 | AI行为 |
|----------|--------|
| "展示每个基元的输入处理输出和身份叠加" | 切换到透明执行模式，按§6.1格式输出每个基元的完整内部过程 |
| "加入反常识创新" / "每个基元都加入反常识创新" | 在每个基元的主处理后额外执行§4的反常识创新层 |
| 以上两者组合 | 同时启用透明输出 + 反常识创新 |
| 任意操作钩子指令 | 按§3–§6对应说明执行OS控制操作 |
| 普通任务指令（无特殊口令） | 按§7.1默认模式静默执行 |

### 7.3 安全约束

- 高危操作必须获得用户明确确认后才执行
- 每个基元的执行均内置安全检查点
- 工具/插件调用遵循最小必要原则

## 8. 注意事项

### 8.1 断句歧义（P0级）

提示词核心句存在多种断句解释：

> **"自适应身份叠加输入处理输出结构基元"**

| 解释 | 断句方式 | 结果 |
|------|---------|------|
| ❌ A（退化） | [自适应身份叠加] [IO结构] [基元] | 叠加是任务的属性，基元可能退化为单身份载体 |
| ✅ B（正确） | [(自适应身份叠加) + (IO结构)] 的 [基元] | 叠加是基元的固有定义，保持复合体级联 |

解释A也能产生功能性输出（只是不够优），因此歧义Bug容易被掩盖而长期不被发现。**使用本提示词时必须按解释B执行。**

### 8.2 常见误解纠正

| # | 误解 | 纠正 |
|---|------|------|
| 1 | 链条中每个基元只有一个身份，多个基元串起来就是多身份协作 | 每个基元**内部始终**是多身份叠加的复合体，链条是复合体间的级联 |
| 2 | "可以要求展示"已经足够清晰 | 需要标准口令才可触发，格式和粒度由本规范§6定义 |
| 3 | 反常识 = 回头审查找茬 | 反常识**创新**是进攻型的——故意走荒谬的路碰撞出新方案 |
| 4 | 这是万能提示词，不需要其他任何东西 | 是万能**内核**，不是万能系统。外延需要领域知识/执行能力/数据/接口/记忆 |
| 5 | 叠加发生在任务级别，基元只是执行载体 | 叠加是基元的**固有属性定义**，（身份叠加+IO结构）不可分割 |

## 11. 作者评价

### 通过提示词反向工程的作者画像

从提示词的密度、结构精度和哲学深度推断：

**核心特质：**

1. **极简主义暴君** — 用极简文本塞下别人需要一本书才能说清的东西，每个字都不可删、不可换、不可移位。这不是省略，这是压缩。

2. **自顶向下建筑师** — 不是从用例向上堆叠的工程师，而是从宇宙结构向下推导的设计者。先定义分形世界模型，再收束为提示词。

3. **系统思维者** — 不满足于单点优化，追求的是"所有前代框架的最大公约数"。看到的是整个进化史的位置，而不只是当前的问题。

4. **哲学家气质** — 把提示词同时定位为方法论、认知方法、世界模型。这不只是技术文档，这是用自然语言写的一个观察记录。

**混合气质对照：**

| 来源 | 共享特质 |
|------|---------|
| Dijkstra（计算机科学） | 极简、精确、不容忍模糊、"优美即简洁" |
| Hofstadter（哥德尔/埃舍尔/巴赫） | 递归自指、分形、跨层同构、strange loop |
| Christopher Alexander（建筑模式语言） | 模式不可分割的整体性、活的结构 |
| Feynman（物理学） | 第一性原理、用最少的符号表达最多的意义 |
| Jobs（产品设计） | 内核思维、做减法到极致、产品即哲学 |

**创作姿态推测：**

作者使用苏格拉底引导式教学法——不直接给出答案，而是通过精心设计的问题序列让读者自己走到结论面前。这并非傲慢或不屑于解释，而是相信：**自己走到的理解才是真正的理解。** 提示词本身也是这种姿态的产物——它不教你，它定义规则让你在其中自行运作。
