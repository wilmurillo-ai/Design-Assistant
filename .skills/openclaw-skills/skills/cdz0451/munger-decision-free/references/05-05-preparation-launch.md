# 准备启动 (Preparation & Launch)

> **准备的目的不是完美，而是跨越启动阈值；启动不是结束，而是迭代的开始。**
> 
> —— 查理·芒格

---

## 一、模型概述

### 1.1 核心定义

**准备启动**是指在进行重大行动前进行充分准备，同时在准备完成后果断启动执行的思维模型。它平衡了"充分准备"与"避免分析瘫痪"之间的张力。

**通俗理解：**
- 不是"准备到 100% 再启动"，而是"准备到 70% 就启动"
- 不是"启动后就结束"，而是"启动后持续迭代"
- 真正重要的不是准备多完美，而是是否在行动中验证和优化

### 1.2 核心公式

```
最佳启动时机 = 信息完备度 × 风险可控度 × 机会窗口

信息完备度 = 核心信息收集完整度（0-100%）
风险可控度 = 已识别风险的应对方案覆盖度（0-100%）
机会窗口 = 市场/竞争环境允许的时间窗口

启动阈值：信息完备度≥70% × 风险可控度≥70% × 机会窗口>0

过度准备检测 = 准备时间/原计划时间 > 2 OR 连续 7 天无进展
```

### 1.3 准备启动三阶段

```
阶段 1：准备期（启动前）
特征：
  - 收集核心信息
  - 识别主要风险
  - 制定应对方案
  - 设定启动阈值
关键：清单化 + 阈值化 + 警惕过度准备

阶段 2：启动期（启动后 0-30 天）
特征：
  - 按计划执行
  - 监控核心指标
  - 快速响应异常
  - 每日/每周迭代
关键：强监控 + 快迭代 + 不慌乱

阶段 3：优化期（启动后 30 天+）
特征：
  - 效果开始显现
  - 识别成功/失败因素
  - 固化有效做法
  - 复盘沉淀经验
关键：深复盘 + 沉淀经验 + 更新清单
```

### 1.4 为什么重要？

**人性弱点：** 人类有两种极端倾向——要么准备不足盲目行动，要么过度准备永不启动

**准备启动思维的价值：**
1. **清单化准备**：系统化的准备清单，避免遗漏关键项
2. **阈值化决策**：明确的启动阈值，避免"再等等"的拖延
3. **过度准备检测**：识别分析瘫痪信号，强制启动决策
4. **启动后迭代**：启动不是结束，是持续优化的开始

### 1.5 适用场景

| 场景类型 | 典型问题 | 准备启动视角 |
|---------|---------|-------------|
| **新产品上线** | "准备到什么程度可以上线？" | 设定启动阈值，达到即上线 |
| **新市场拓展** | "新省份什么时候进入？" | 政策/竞品/团队准备清单 |
| **营销活动** | "活动什么时候开始？" | 物料/人员/流量准备检查 |
| **团队扩张** | "什么时候招新人？" | 岗位/流程/薪酬准备就绪 |
| **合作伙伴** | "什么时候签合作协议？" | 方案/法务/试点验证完成 |

## 二、核心原理

### 2.1 原理一：清单化准备 - 结构化避免遗漏

**洞察：** 人脑不擅长记忆多项任务，清单化确保关键准备不遗漏

**示例：**
```
新产品上线准备清单：

核心功能准备（P0 必须完成）：
  ✅ 题库功能：题目数量≥500 道，覆盖率≥80% 考纲
  ✅ 智能推荐：推荐准确率≥70%（测试数据）
  ✅ 用户系统：注册/登录/学习进度追踪正常
  ✅ 支付系统：支付流程测试通过，退款机制明确

风险控制准备（P0 必须完成）：
  ✅ 技术风险：服务器压力测试通过（支持 1000 并发）
  ✅ 内容风险：题目版权审核完成，无侵权风险
  ✅ 法律风险：用户协议、隐私政策法务审核通过

启动条件（P1 重要）：
  ✅ 种子用户测试：50 人测试，NPS≥30
  ✅ 核心 bug 修复：P0/P1 bug 全部修复，P2 bug<10 个
  ✅ 运营准备：推广渠道测试通过，首批流量 ready

启动阈值：
  - P0 项 100% 完成
  - 总体完成度 ≥ 70%
  - 无致命风险
```

**业务应用：**
```
新市场拓展准备清单（山东省）：

政策调研（P0）：
  ✅ 山东省专升本政策文件收集完整
  ✅ 考试科目、分值、录取规则明确
  ✅ 政策稳定性评估（近 3 年变化频率）
  ✅ 政策风险识别（如可能取消专升本）

竞品分析（P0）：
  ✅ 山东省主要竞品机构识别（前 5 名）
  ✅ 竞品价格、课程、服务对比分析
  ✅ 竞品弱点识别（如服务差、通过率低）
  ✅ 差异化定位明确（我们 vs 竞品）

本地团队（P0）：
  ✅ 本地负责人招聘完成（有山东资源）
  ✅ 教师团队搭建（熟悉山东考纲）
  ✅ 销售/客服团队培训完成
  ✅ 办公场地/设备准备就绪

启动阈值：
  - P0 项 100% 完成
  - 总体完成度 ≥ 70%
  - 无致命风险（如政策明确取消专升本）
```

### 2.2 原理二：阈值化决策 - 量化避免拖延

**洞察：** 模糊的"准备差不多了"不如具体的"完成度 75%，达到启动阈值"

**示例：**
```
启动阈值规则：

标准启动条件（推荐）：
  - P0 项 100% 完成
  - 总体完成度 ≥ 70%
  - 无致命风险
  → 建议：LAUNCH（正常启动）

风险启动条件（谨慎）：
  - P0 项 ≥ 80% 完成
  - 总体完成度 ≥ 60%
  - 剩余风险有应对方案
  → 建议：LAUNCH_WITH_RISK（风险启动）

不建议启动：
  - P0 项 < 80% 完成
  - 或总体完成度 < 60%
  - 或存在未识别的致命风险
  → 建议：CONTINUE_PREPARING（继续准备）
```

**业务应用：**
```
营销活动启动阈值评估：

准备清单完成情况：
  - 总检查项：50 项
  - 已完成：38 项
  - 完成度：76% ✅

P0 项完成情况：
  - P0 总项：15 项
  - P0 已完成：15 项
  - P0 完成度：100% ✅

风险评估：
  - 已识别风险：8 个
  - 有应对方案：8 个
  - 风险可控度：100% ✅

启动建议：LAUNCH（正常启动）

依据：
  - P0 项 100% 完成 ✅
  - 总体完成度 76% ≥ 70% ✅
  - 无致命风险 ✅
```

### 2.3 原理三：过度准备检测 - 识别分析瘫痪

**洞察：** 过度准备的成本往往超过启动风险，需要强制启动机制

**过度准备信号：**
```
信号 1：准备时间超过原计划 2 倍
  触发条件：实际准备时间 > 原计划 × 2
  严重程度：高
  建议行动：强制启动或重新评估项目可行性

信号 2：连续 7 天无进展
  触发条件：连续 7 天检查项无完成
  严重程度：中
  建议行动：识别阻塞项，集中资源突破

信号 3：非核心项迭代超过 5 次
  触发条件：非 P0 项修改/优化 > 5 次
  严重程度：中
  建议行动：接受"足够好"，聚焦核心项

信号 4：新增检查项超过 10 个
  触发条件：原始清单外新增 > 10 项
  严重程度：中
  建议行动：冻结范围，完成现有项

信号 5：启动决策推迟超过 3 次
  触发条件：达到阈值但推迟启动 > 3 次
  严重程度：高
  建议行动：强制决策，接受风险
```

**业务应用：**
```
过度准备检测案例：

项目：升本智途 APP 上线
原计划准备时间：30 天
实际准备时间：75 天（第 75 天检测）

检测信号：
  ⚠️ 信号 1：准备时间 75 天 > 原计划 30 天 × 2 = 60 天
  ⚠️ 信号 4：新增检查项 15 个（原始 35 项 → 现 50 项）
  ⚠️ 信号 5：启动决策推迟 4 次

分析：
  - 准备时间超期 150%
  - 范围蔓延严重（新增 15 项）
  - 决策拖延（推迟 4 次）

强制启动决策：
  - 决策：FORCED_LAUNCH（强制启动）
  - 理由：继续准备成本超过启动风险
  - 风险缓解：启动后加强监控，快速迭代

结果：
  - 第 76 天强制上线
  - 发现 3 个 P2 bug（不影响核心功能）
  - 2 周内修复，用户 NPS 35（达标）
  - 复盘：如第 45 天启动，可节省 30 天时间
```

### 2.4 原理四：启动后迭代 - 在行动中优化

**洞察：** 启动不是结束，而是验证和优化的开始；完美是在迭代中实现的，不是在准备中实现的

**示例：**
```
启动后监控框架：

监控周期：启动后 30 天

核心指标（每日监控）：
  - DAU（日活跃用户）
  - 留存率（次日/7 日/30 日）
  - NPS（净推荐值）
  - 核心 bug 数量
  - 用户反馈数量

告警规则：
  P0 告警（立即响应）：
    - 核心指标低于预期 50%+
    - 出现致命 bug 或事故
    - 负面舆情爆发
    → 24 小时内制定应对方案

  P1 告警（48 小时响应）：
    - 核心指标低于预期 30-50%
    - 用户反馈问题集中
    - 竞品强力反击
    → 48 小时内分析原因，调整策略

  P2 提示（周报复盘）：
    - 核心指标低于预期 10-30%
    - 轻微问题
    → 周报复盘，持续优化

迭代节奏：
  - 每日：监控数据，响应 P0 告警
  - 每周：复盘会，调整策略
  - 30 天：全面复盘，沉淀经验
```

**业务应用：**
```
新产品上线后 30 天监控记录：

第 1 周：
  - DAU：120（预期 150，-20%）⚠️ P2
  - 次日留存：55%（预期 60%，-8%）✅
  - NPS：28（预期 30，-7%）✅
  - 核心 bug：2 个（P2）✅
  - 用户反馈：15 条（主要反馈题目解析不够详细）

  行动：
  - 优化题目解析（增加详细步骤）
  - 加强新用户引导（增加 onboarding 流程）

第 2 周：
  - DAU：135（预期 150，-10%）✅
  - 次日留存：58%（预期 60%，-3%）✅
  - NPS：32（预期 30，+7%）✅
  - 核心 bug：0 个 ✅
  - 用户反馈：8 条（正面反馈增加）

  行动：
  - 保持当前策略
  - 加大推广力度（DAU 接近预期）

第 3-4 周：
  - DAU：155（预期 150，+3%）✅
  - 次日留存：62%（预期 60%，+3%）✅
  - NPS：35（预期 30，+17%）✅
  - 核心 bug：0 个 ✅
  - 用户反馈：正面为主

  结论：启动成功，进入优化期

30 天复盘：
  - 成功因素：快速迭代响应用户反馈
  - 失败因素：初期对解析详细度估计不足
  - 经验沉淀：更新准备清单，增加"解析详细度测试"检查项
```

## 三、应用框架

### 3.1 准备启动七步法

```
┌─────────────────────────────────────────────────────────────┐
│  步骤 1：识别项目类型                                       │
│  → 新产品/新市场/营销活动/团队扩张/合作伙伴                 │
│  → 匹配准备清单模板                                         │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤 2：生成准备清单                                       │
│  → 基于模板生成检查项                                       │
│  → 标注优先级（P0/P1/P2）                                   │
│  → 设定启动阈值                                             │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤 3：执行准备工作                                       │
│  → 逐项完成检查项                                           │
│  → 更新完成状态                                             │
│  → 记录备注和问题                                           │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤 4：评估启动准备度                                     │
│  → 计算完成度（总体/P0）                                    │
│  → 检查启动阈值                                             │
│  → 检测过度准备信号                                         │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤 5：启动决策                                           │
│  → 综合评估结果和过度准备信号                               │
│  → 生成决策建议（LAUNCH/风险启动/继续准备/强制启动）        │
│  → 用户确认                                                 │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤 6：启动后监控                                         │
│  → 收集核心指标（每日/每周）                                │
│  → 对比预期 vs 实际                                         │
│  → 识别异常并告警                                           │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  步骤 7：复盘学习                                           │
│  → 启动后 30 天复盘                                         │
│  → 对比准备清单 vs 实际情况                                 │
│  → 更新清单模板                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 准备清单模板库

```markdown

- [ ] 数据安全审核（用户数据保护/备份机制）

启动条件（P1）：
- [ ] 种子用户测试完成（50 人+，NPS≥30）

- [ ] 推广渠道测试通过（首批流量 ready）

- P0 项 100% 完成

- 无致命风险
```

### 3.3 启动决策框架

```python
def make_launch_decision(checklist, over_prep_signals):
    """
    启动决策框架
    
    Args:
        checklist: 准备清单对象
        over_prep_signals: 过度准备信号列表
    
    Returns:
        决策建议
    """
    # 计算完成度
    total_items = len(checklist.items)
    completed_items = sum(1 for item in checklist.items if item.completed)
    completion_rate = completed_items / total_items
    
    # 计算 P0 完成度
    p0_items = [item for item in checklist.items if item.priority == 'P0']
    p0_completed = sum(1 for item in p0_items if item.completed)
    p0_rate = p0_completed / len(p0_items)
    
    # 识别阻塞项
    blockers = [item for item in p0_items if not item.completed]
    
    # 检测高风险信号
    high_risk_signals = [s for s in over_prep_signals if s["severity"] == "HIGH"]
    
    # 决策逻辑
    if high_risk_signals:
        return {
            "decision": "FORCED_LAUNCH",
            "reason": "检测到过度准备信号，继续准备成本超过启动风险",
            "signals": high_risk_signals,
            "risk_mitigation": "启动后加强监控，快速迭代"
        }
    
    if p0_rate == 1.0 and completion_rate >= 0.7:
        return {
            "decision": "LAUNCH",
            "reason": "准备充分，达到启动阈值",
            "confidence": min(1.0, completion_rate + 0.1),
            "next_steps": "按启动计划执行"
        }
    
    if p0_rate >= 0.8 and completion_rate >= 0.6:
        return {
            "decision": "LAUNCH_WITH_RISK",
            "reason": "准备基本充分，但存在部分风险",
            "blockers": blockers,
            "risk_mitigation": "启动后优先解决阻塞项"
        }
    
    return {
        "decision": "CONTINUE_PREPARING",
        "reason": "准备不充分，建议继续准备",
        "blockers": blockers,
        "focus_areas": [item.description for item in blockers]
    }
```

## 五、常见误区

### 5.1 误区一：分析瘫痪（准备过度）

**错误想法：** "再收集点信息，准备更充分再启动"

**正确认知：**
```
过度准备的成本往往超过启动风险：

过度准备成本：
  - 时间成本：准备时间 × 团队人力
  - 机会成本：错过市场窗口
  - 心理成本：团队士气下降

启动风险成本：
  - 可修复问题：启动后迭代解决
  - 不可修复问题：准备阶段也难以识别

关键洞察：
  - 70% 准备度启动，30% 在行动中完善
  - 完美是在迭代中实现的，不是在准备中实现的
  - 如准备时间超过原计划 2 倍，强制启动

示例：
  升本智途 APP 原计划准备 30 天：
  - 第 30 天：完成度 75%，团队想继续准备
  - 第 45 天：完成度 82%，团队还想优化
  - 第 60 天：完成度 88%，团队仍在找问题
  - 第 75 天：强制启动，完成度 90%
  
  复盘：
  - 第 30 天启动和第 75 天启动，用户感知差异<5%
  - 但浪费了 45 天时间和团队士气
  - 教训：设定强制启动日期，到期必须启动
```

### 5.2 误区二：准备不足（盲目启动）

**错误想法：** "先上线再说，有问题再改"

**正确认知：**
```
准备不足的风险：

致命风险（必须在准备阶段解决）：
  - 法律合规问题（如用户协议/隐私政策）
  - 数据安全问题（如用户数据泄露）
  - 核心功能缺陷（如支付失败）
  - 政策风险（如办学许可未拿到）

可迭代问题（可启动后解决）：
  - UI 细节不完美
  - 非核心功能缺失
  - 性能可优化空间
  - 用户体验小问题

关键洞察：
  - P0 项必须 100% 完成才能启动
  - P1/P2 项可启动后迭代
  - 区分"致命风险"和"可迭代问题"

示例：
  某机构新市场拓展：
  - 准备度：60%
  - P0 项：办学许可未拿到 ❌
  - 决策：盲目启动
  
  结果：
  - 第 15 天：教育局检查，责令停业
  - 损失：已招生 30 人，退款 + 声誉损失
  - 教训：P0 项（合规）必须完成才能启动
```

### 5.3 误区三：启动即结束（不监控不迭代）

**错误想法：** "启动了，任务完成了"

**正确认知：**
```
启动不是结束，而是开始：

启动前：准备（20% 精力）
启动后：监控 + 迭代（80% 精力）

启动后关键动作：
  - 每日监控核心指标
  - 每周复盘调整策略
  - 30 天全面复盘沉淀经验

关键洞察：
  - 启动只是验证假设的开始
  - 真正的价值在启动后的迭代中创造
  - 不监控不迭代 = 浪费启动前的准备

示例：
  某机构营销活动：
  - 准备充分：物料/人员/流量 100% ready
  - 启动后：无人监控数据，无人调整策略
  - 结果：
    - 低效渠道持续投放 7 天（浪费 5 万）
    - 高效渠道未加大投入（损失 10 万潜在收益）
  - 教训：启动后必须监控 + 迭代
```

### 5.4 误区四：忽视复盘（不沉淀经验）

**错误想法：** "项目结束了，下一个"

**正确认知：**
```
不复盘的代价：

- 同样错误重复犯
- 成功经验未固化
- 准备清单未更新
- 团队能力未提升

复盘的价值：
  - 识别成功/失败因素
  - 更新准备清单模板
  - 调整启动阈值
  - 提升团队能力

关键洞察：
  - 启动后 30 天必须复盘
  - 复盘输出必须更新清单模板
  - 复盘经验必须分享给团队

示例：
  荟众教育 3 次市场拓展复盘对比：
  
  第 1 次（山东）：
  - 未复盘，直接做第 2 次
  - 问题：竞品价格战应对不足
  - 结果：招生低于预期 30%
  
  第 2 次（浙江）：
  - 简单复盘，未更新清单
  - 问题：同样遇到价格战，仍应对不足
  - 结果：招生低于预期 25%
  
  第 3 次（江苏）：
  - 深度复盘，更新清单
  - 新增"竞品价格战应对预案"检查项
  - 结果：招生超过预期 20%
  
  教训：复盘必须输出可执行的清单更新
```

### 5.5 误区五：清单僵化（不灵活调整）

**错误想法：** "清单上的项必须全部完成"

**正确认知：**
```
清单是工具，不是枷锁：

清单的价值：
  - 结构化避免遗漏
  - 量化评估准备度
  - 团队对齐期望

清单的局限：
  - 无法覆盖所有场景
  - 可能过度约束
  - 需要灵活调整

关键洞察：
  - P0 项必须完成（刚性）
  - P1 项尽量完成（弹性）
  - P2 项可启动后完成（灵活）
  - 如遇特殊情况，可调整优先级

示例：
  某机构新产品上线：
  - 原清单：50 项检查项
  - 突发情况：竞品提前 1 个月上线
  - 决策：
    - P0 项（15 项）：必须完成
    - P1 项（20 项）：完成 12 项，8 项延后
    - P2 项（15 项）：全部延后
  - 启动阈值调整：
    - P0 项 100% 完成 ✅
    - 总体完成度 54%（原 70%）✅（特殊情况）
  - 结果：成功抢在竞品前上线，后续迭代补齐 P1/P2 项
  
  教训：清单是工具，根据实际情况灵活调整
```

## 七、与其他模型的关联

### 7.1 与临界质量的关系

```
准备启动 × 临界质量：

准备启动回答："什么时候启动"
临界质量回答："启动后需要多少投入才能自运转"

协同使用：
1. 用准备启动思维决定"何时启动"
2. 用临界质量思维设计"启动后投入计划"
3. 两者结合 = 完整的启动→自运转框架

示例：
  老带新网络项目：
  
  准备启动阶段：
  - 准备清单：激励方案/追踪系统/运营流程
  - 启动阈值：P0 项 100%，总体 70%
  - 启动决策：LAUNCH
  
  临界质量阶段：
  - 临界点计算：50 个推荐人
  - 阶段管理：启动期→增长期→自运转期
  - 自运转验证：外部投入依赖度<50%
  
  关系：
  - 准备启动是"0→1"（从准备到启动）
  - 临界质量是"1→10"（从启动到自运转）
  - 没有准备启动，临界质量无法开始
  - 没有临界质量，准备启动无法持续
```

### 7.2 与复利效应的关系

```
准备启动 × 复利效应：

准备启动回答："什么时候开始投入"
复利效应回答："投入后如何自我强化"

协同使用：
1. 用准备启动思维决定"何时开始"
2. 用复利效应设计"自我强化机制"
3. 两者结合 = 完整的长期投入框架

示例：
  内容库建设项目：
  
  准备启动阶段：
  - 准备清单：内容规划/创作流程/发布渠道
  - 启动阈值：P0 项 100%，总体 70%
  - 启动决策：LAUNCH
  
  复利效应阶段：
  - 复利机制：老内容持续带来流量
  - 强化循环：流量→反馈→优化→更多流量
  - 长期收益：获客成本持续降低
  
  关系：
  - 准备启动是"开始投入"
  - 复利效应是"投入后的收益曲线"
  - 没有准备启动，复利效应无法开始
  - 没有复利效应，准备启动收益有限
```

### 7.3 与机会成本的关系

```
准备启动 × 机会成本：

准备启动回答："什么时候启动"
机会成本回答："这些准备时间放弃了什么"

协同使用：
1. 用机会成本评估"准备时间是否值得"
2. 用准备启动评估"准备是否充分"
3. 两者结合 = 完整的启动时机决策

示例：
  决策：新产品上线准备时间
  
  机会成本分析：
  - 准备 30 天：放弃 30 天市场收益
  - 准备 60 天：放弃 60 天市场收益 + 竞品可能抢先
  - 准备 90 天：放弃 90 天市场收益 + 市场窗口可能关闭
  
  准备启动分析：
  - 30 天准备：完成度 70%，达到启动阈值
  - 60 天准备：完成度 85%，超出启动阈值
  - 90 天准备：完成度 92%，过度准备
  
  决策：
  - 30 天启动：机会成本低，准备充分
  - 60 天启动：机会成本中，准备过度
  - 90 天启动：机会成本高，严重过度准备
  
  结论：30 天启动最优
```

### 7.4 与逆向思维的关系

```
准备启动 × 逆向思维：

逆向思维回答："反过来想，什么会失败"
准备启动回答："什么条件下可以启动"

协同使用：
1. 逆向：不问"如何成功"，问"什么会失败"
2. 准备启动：识别失败模式（准备不足/过度准备）
3. 两者结合 = 风险识别 + 启动决策

示例：
  新产品上线失败模式（逆向思考）：
  - 准备不足：核心功能缺陷、法律风险
  - 过度准备：错过市场窗口、团队士气下降
  - 启动后不监控：问题积累爆发
  
  准备启动对策：
  - 准备不足：P0 项必须 100% 完成
  - 过度准备：设定强制启动日期
  - 启动后不监控：每日监控 + 每周复盘
  
  关系：
  - 逆向思维识别失败模式
  - 准备启动设计预防机制
```

## 八、模型局限

### 8.1 局限性

| 局限 | 说明 | 应对方法 |
|------|------|---------|
| **清单模板不适用** | 特殊场景无匹配模板 | 支持自定义，持续迭代模板 |
| **阈值设置主观** | 70% 阈值可能不适用所有场景 | 基于历史数据校准，支持调整 |
| **过度准备检测误判** | 可能误判正常准备为过度 | 多信号综合判断，人工确认 |
| **启动后监控数据缺失** | 数据收集不完整 | 自动化收集 + 手动补录 |
| **复盘流于形式** | 复盘不深入，无行动项 | 标准化复盘框架，强制行动项 |

### 8.2 不适用场景

- **紧急问题处理**：需要立刻行动，无法准备
- **日常运营决策**：快速迭代即可，无需 formal 准备
- **低成本试错**：失败成本低，无需严格准备
- **外部环境高度不确定**：准备无法应对变化
- **创意型工作**：过度准备可能抑制创意

### 8.3 使用边界

```
准备启动思维的适用边界：

适用场景（重大决策/高成本/不可逆）：
✅ 新产品上线
✅ 新市场拓展
✅ 大型营销活动
✅ 团队扩张
✅ 合作伙伴关系建立

不适用场景（日常/低成本/可逆）：
❌ 日常运营决策
❌ 小范围试错实验
❌ 紧急问题处理
❌ 创意型工作
❌ 低成本尝试
```

## 九、行动建议

### 9.1 立即开始

1. **选择一个待启动项目**
   ```
   问自己：
   - 哪个项目正在准备中？
   - 哪个项目因"准备不充分"而延迟？
   - 哪个项目可能过度准备？
   
   候选项目：
   - 新产品/功能上线
   - 新市场拓展
   - 营销活动
   - 团队扩张
   ```

2. **生成准备清单**
   ```
   使用准备清单模板：
   - 选择项目类型
   - 生成检查项
   - 标注优先级（P0/P1/P2）
   - 设定启动阈值
   ```

3. **评估启动准备度**
   ```
   计算完成度：
   - P0 项完成度
   - 总体完成度
   - 检测过度准备信号
   
   生成启动决策：
   - LAUNCH / 风险启动 / 继续准备 / 强制启动
   ```

### 9.2 长期练习

1. **建立准备启动思维**
   - 看到"再等等"时，问"是否过度准备？"
   - 看到"先上线再说"时，问"P0 项完成了吗？"
   - 看到"启动了"时，问"监控计划 ready 吗？"

2. **记录启动案例**
   - 记录每个项目的准备清单
   - 追踪启动决策和结果
   - 复盘偏差原因
   - 持续改进清单模板

3. **培养阈值感知能力**
   - 识别"准备差不多了"的模糊感觉
   - 转化为"完成度 75%，达到启动阈值"的量化判断
   - 在适当时机果断启动
