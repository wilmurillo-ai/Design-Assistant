# Failure Analysis Guide - 失敗分析指南

## 分析框架

### 第一步：分類失敗類型

```
交易失敗 → 
├── 執行失敗（未上鏈）
├── 上鏈失敗（Reverted）
├── 成功但錯誤（買錯/賣錯）
└── 成功但虧損（時機/價格）
```

### 第二步：收集數據

每次失敗記錄：

- 時間戳
- Token 對
- 交易類型（Buy/Sell）
- 預期參數
- 實際結果
- 錯誤信息
- 市場狀況

### 第三步：根因分析

使用 5 Whys 方法：

```
問題：交易失敗
1. 為什麼？→ Gas 不足
2. 為什麼？→ Gas 價格設置過低
3. 為什麼？→ 沒考慮網絡擁堵
4. 為什麼？→ 沒實時獲取 Gas 建議
5. 為什麼？→ 缺少 Gas API 集成

根因：缺少動態 Gas 調整機制
```

## 常見問題與解決方案

### 1. Gas 相關失敗

**症狀：** Transaction reverted / Out of gas

**分析：**
```python
# 檢查 Gas 使用情況
estimated_gas = await contract.estimateGas()
actual_gas = tx_receipt.gasUsed
gas_ratio = actual_gas / estimated_gas

IF gas_ratio > 0.9:
    # 接近限制，需要增加
```

**解決：**
- 增加 Gas Limit 20-30%
- 使用動態 Gas 價格
- 避開網絡高峰時段

### 2. 滑點相關失敗

**症狀：** Amount out minimum / Price impact too high

**分析：**
```
檢查：
- 設置滑點 vs 實際滑點
- 流動池深度
- 交易規模占比
```

**解決：**
- 增加滑點容忍度
- 減小單筆交易規模
- 使用多筆拆分執行

### 3. 價格波動失敗

**症狀：** 執行價格與預期偏差大

**分析：**
```
檢查時間差：
訊號產生 → 決策 → 執行 → 上鏈
     ↓
  價格已變
```

**解決：**
- 縮短決策延遲
- 使用更高 Gas 優先上鏈
- 設置更寬的價格容忍度

### 4. RPC 相關失敗

**症狀：** Timeout / Connection refused

**分析：**
```
檢查：
- RPC 節點健康狀態
- 請求速率限制
- 網絡連接穩定性
```

**解決：**
- 使用多個 RPC 節點
- 實施重試機制
- 添加降級策略

### 5. 邏輯錯誤失敗

**症狀：** 買錯/賣錯/數量錯

**分析：**
```
檢查：
- 參數傳遞鏈路
- 數據來源準確性
- 決策邏輯正確性
```

**解決：**
- 添加參數驗證
- 實施交易前確認
- 添加白名單機制

## 修復驗證流程

### 修復後測試

1. **模擬測試**
   - 使用歷史數據回測
   - 驗證修復邏輯

2. **小額測試**
   - 使用最小金額測試
   - 確認實際執行正常

3. **監控觀察**
   - 連續監控 24-48 小時
   - 確認問題不再出現

### 回歸測試

每次修復後確認：
- [ ] 原問題已解決
- [ ] 未引入新問題
- [ ] 其他功能正常

## 學習與優化

### 知識庫更新

每次失敗分析後：
1. 記錄到失敗案例庫
2. 更新檢查清單
3. 優化監控規則
4. 改進腳本邏輯

### 趨勢分析

每週回顧：
- 失敗類型分佈
- 高頻問題識別
- 系統性問題發現
- 優化優先級排序
