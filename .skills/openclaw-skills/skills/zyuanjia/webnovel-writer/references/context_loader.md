# 上下文智能摘要机制

> 解决核心问题：50万字+长篇小说的设定信息无法全部塞进上下文窗口，写作时需要智能加载最相关的内容。

## 一、分层加载策略

### L0 - 始终加载（每章写作必带，约2-3KB）

这些信息量小但极重要，全部加载不会浪费上下文：

```
1. 写作硬性规则（来自设定词典/设定集的"写作规则"部分）
2. 设定词典全文（专有名词+时间线+禁用词，通常5KB以内）
3. 近5章摘要表（来自追踪表）
4. 上一章最后500字原文
5. 当前写作卡（本章大纲要点）
```

**操作方式**：写作前直接 `read` 设定词典 + 写作追踪表 + 上一章文件尾部。

### L1 - 按需加载：角色设定（约1-3KB/角色）

只加载**本章出场角色**的设定。根据写作卡判断谁出场。

**操作方式**：
1. 从写作卡提取角色名列表
2. 用 `exec` + `grep` 从设定集提取对应角色段落：
   ```bash
   grep -n "###\|####" 设定集.md
   ```
3. 用 `read` 的 offset/limit 只读目标角色的部分

**示例**：写作卡写"陆远和沈清在咖啡馆见面"→ 只加载陆远和沈清的角色档案，不加载老钱、赵恒、方哲。

### L2 - 按需加载：大纲与剧情线（按卷/章范围）

502章的大纲文件可达67KB+，绝不能全读。

**操作方式**：
1. 根据当前章节号，用 `grep` 定位当前卷的范围：
   ```bash
   grep -n "第.*卷\|Vol\|卷[0-9]" 大纲文件.md | head -30
   ```
2. 用 `read` offset/limit 只读当前卷大纲（通常2-5KB）
3. 额外读取前后各1卷的摘要标题，了解上下文

### L3 - 按需加载：场景/道具/关系（极小概率）

仅在写作卡涉及特殊设定时加载（如系统的具体运作规则、特定场景描写）。

**操作方式**：维护一个"设定索引"（见下方模板），写作时查索引决定是否需要深入读取。

## 二、章节相关性匹配

### 匹配流程

```
写作卡输入 → 提取关键词 → 匹配设定索引 → 决定加载内容
```

### 关键词提取规则

从写作卡中提取以下维度：

| 维度 | 示例 | 需要加载 |
|------|------|----------|
| 角色名 | "陆远""沈清""赵恒" | L1：对应角色设定 |
| 场地名 | "地下金库""投行" | L3：场景描写设定 |
| 道具/物品 | "余额""迈凯轮" | L3：相关设定条目 |
| 事件关键词 | "交易""假死""毕业" | L2：相关卷的大纲回顾 |
| 情感/关系 | "决裂""告白""信任" | L1：相关角色档案的情感线部分 |

### 匹配执行模板

写作前的标准化操作序列：

```markdown
## 写作上下文包生成步骤

1. 读取当前写作卡
2. 从写作卡提取：出场角色、涉及场景、关键事件
3. 加载 L0（始终加载项）
4. 根据角色列表加载 L1（grep 定位 + read 摘取）
5. 根据章节号加载 L2（当前卷大纲）
6. 检查 L3 索引，判断是否需要额外加载
7. 组装上下文包，开始写作
```

## 三、摘要压缩方法

### 3.1 角色设定压缩

完整角色档案可能1000字+，写作时只需核心要素。压缩模板：

```markdown
## [角色名] 写作摘要
- 身份：[一句话]
- 当前状态：[情感/处境，从追踪表取]
- 与本章关系：[为什么出场]
- 关键特征：[外貌/口头禅/习惯，最多3条]
- 称呼规则：[怎么称呼别人/被谁怎么称呼]
```

**压缩示例**（原文800字 → 压缩后120字）：

```markdown
## 老钱 写作摘要
- 身份：钱国明，55岁，交易部总监，陆远上司
- 当前状态：刚发现前妻被系统交易真相，情绪不稳定
- 与本章关系：向陆远透露关键线索
- 关键特征：爱喝浓茶、说话慢吞吞、住老小区三居室
- 称呼规则：陆远叫他"钱哥"，他叫陆远"小陆"
```

### 3.2 大纲压缩

502章大纲按卷压缩为"卷摘要卡"：

```markdown
## 第3卷 摘要卡（第61-90章）
- 核心事件：陆远发现孙磊余额异常 → 深入调查 → 揭示交易系统冰山一角
- 关键转折：第75章孙磊余额归零
- 情感线：陆远与沈清从合作到产生好感
- 出场主要角色：陆远、沈清、孙磊、赵恒
- 承上：第2卷沈清首次出场
- 启下：第4卷方哲登场
```

### 3.3 设定词典不压缩

设定词典本身已经是精简的条目表，全文保留。5KB在上下文中完全可接受。

## 四、增量更新策略

### 4.1 章节写作后的增量维护

每章写完后，agent 自动执行：

```
1. 更新写作追踪表的"近5章摘要"
2. 如果本章引入新专有名词 → 追加到设定词典
3. 如果角色状态发生变化 → 更新追踪表的角色状态列
4. 如果发现设定冲突 → 标记到待修复列表
```

### 4.2 角色状态快照（增量，不传全量）

在追踪表中维护角色状态表，每章只更新变化的角色：

```markdown
| 角色 | 最近变化章节 | 当前状态 | 关键备注 |
|------|-------------|----------|----------|
| 陆远 | Ch35 | 决定深入调查 | 从被动转主动 |
| 沈清 | Ch33 | 不知情 | 未出场2章 |
| 老钱 | Ch34 | 情绪低落 | 被王建国警告后 |
| 赵恒 | Ch33 | 远程协助 | 未到场 |
```

写作时只需读取这张表，不需要重新加载全部角色档案。

### 4.3 设定变更日志

设定文件如果被修改，维护变更记录：

```markdown
## 设定变更日志
| 日期 | 文件 | 变更内容 | 影响范围 |
|------|------|----------|----------|
| 2026-04-10 | 设定词典 | 新增"刘阳"条目 | 第8-9卷 |
| 2026-04-10 | 设定集 | 修改老钱背景：前妻张敏细节 | 第3卷起 |
```

写作前检查变更日志，只加载近期变更的相关部分。

## 五、写作上下文包模板

### 标准模板（写作时组装）

```markdown
# 第[NN]章 写作上下文包

## 📋 写作卡
[从大纲中提取的本章要点，200-500字]

## ⚙️ 硬性规则提醒
[3-5条最易违反的规则，从设定词典中挑选]

## 👥 本章角色摘要
### [角色A] 写作摘要
- 身份/状态/关键特征/称呼规则

### [角色B] 写作摘要
- 身份/状态/关键特征/称呼规则

## 📖 近5章摘要
| 章节 | 关键事件 | 角色变化 |
|------|----------|----------|
| ... | ... | ... |

## 🔗 上一章尾部
[上一章最后500字原文]

## 📍 当前卷方向
[当前卷的1-2句话方向概括]

## ⚠️ 特别注意
[写作卡中涉及的特殊设定/易错点]
```

### 实际示例

以《我的死亡日期消失了》第35章为例：

```markdown
# 第35章 写作上下文包

## 📋 写作卡
陆远决定不再被动等待，主动深入调查永盛财务数据。深夜独自加班，
发现第二层加密数据指向一个海外账户。赵恒打来电话提醒他小心。
陆远在回家路上经过牛肉面馆，想起老钱说的话，决定第二天找老钱
单独谈谈。章末：陆远打开家门，发现门没锁。

## ⚙️ 硬性规则提醒
- 全程第三人称"陆远"视角，禁止"我"
- 公司名"永盛集团"，禁止其他写法
- 不提前剧透余额归零等重大异常
- 每章2500字左右

## 👥 本章角色摘要
### 陆远
- 身份：32岁投行合伙人，能看到寿命余额
- 当前状态：从被动转主动，决定深入调查
- 关键特征：西装三件套、黑眼圈、口头禅"正常营业"
- 称呼：老钱叫他"小陆"

### 赵恒
- 身份：陆远搭档，实际是系统人格化（600岁）
- 当前状态：远程协助，不在江城
- 关键特征：外表35岁、永远笑眯眯、说话带哲学味
- 称呼：叫陆远"陆老弟"

## 📖 近5章摘要
| 章节 | 关键事件 | 角色变化 |
|------|----------|----------|
| Ch30 | 陆远完成永盛尽调初稿 | 无 |
| Ch31 | 发现财务系统后门 | 开始怀疑 |
| Ch32 | 找孙磊确认，孙磊否认 | 孙磊异常 |
| Ch33 | 赵恒远程分析数据 | 发现海外关联 |
| Ch34 | 王建国警告"别多管闲事" | 处境危险 |

## 🔗 上一章尾部
[第34章最后500字...]

## 📍 当前卷方向
第2卷核心：陆远在永盛项目中越陷越深，逐步发现交易系统在现实中的痕迹。

## ⚠️ 特别注意
- 赵恒的电话要用他特有的说话方式（带哲学味但不啰嗦）
- 牛肉面馆是公司楼下，8块一碗——不要写错位置
- 门没锁是悬念钩子，本章不揭晓原因
```

## 六、Agent 操作指令

写作前 agent 执行的标准化流程（伪代码）：

```
function build_context_packet(章节号, 写作卡):
    # L0：始终加载
    规则 = read("设定词典.md") 的"写作硬性规则"部分
    词典 = read("设定词典.md") 全文
    追踪表 = read("写作追踪表.md")
    近5章摘要 = 追踪表.近5章摘要
    上一章尾部 = read("正文/第{章节号-1}章.md", offset=尾部500字)
    
    # L1：角色
    出场角色 = extract_characters(写作卡)
    for 角色 in 出场角色:
        位置 = grep("设定集.md", "### " + 角色)
        角色摘要 = read("设定集.md", offset=位置, limit=50行)
        压缩为写作摘要格式
    
    # L2：大纲
    当前卷 = determine_volume(章节号)
    卷范围 = grep("大纲.md", "第{当前卷}卷")
    卷大纲 = read("大纲.md", offset=卷范围.start, limit=卷范围.length)
    
    # 组装上下文包
    return 组装为模板格式
```

### 具体可执行的 shell 命令

```bash
# 查找角色在设定集中的位置
grep -n "^### \|^## " 设定集.md

# 查找当前卷在大纲中的位置
grep -n "第.*卷\|卷[0-9]" 大纲.md

# 读取上一章尾部
tail -20 "正文/第${PREV_CHAPTER}章.md"

# 从设定词典提取特定条目
grep -A2 "永盛\|投行\|沈清" 设定词典.md
```

## 七、容量估算与限制

| 内容层 | 典型大小 | 加载频率 |
|--------|----------|----------|
| L0：硬性规则 + 设定词典 | ~5KB | 每章 |
| L0：近5章摘要 + 上一章尾部 | ~2KB | 每章 |
| L1：角色摘要（2-3人） | ~1KB | 每章 |
| L2：当前卷大纲 | ~3KB | 每章 |
| L3：特殊设定 | ~1KB | 按需 |
| **合计** | **~12KB** | - |

12KB 约等于 4000-5000 tokens，在大多数模型的上下文窗口中完全可控。对比全量加载（设定集17KB + 大纲67KB + 词典5KB = 89KB），压缩率约 86%。

## 八、何时触发全量加载

以下场景需要牺牲效率换取完整性：

1. **新卷开始**：加载前卷完整摘要 + 新卷完整大纲
2. **重大剧情转折**：加载所有相关角色的完整档案
3. **一致性检查时**：加载设定词典全文 + 角色状态表全文
4. **回顾性章节**（闪回、回忆）：加载相关历史时间线设定
