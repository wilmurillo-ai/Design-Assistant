# 测试覆盖策略详解

## 概述

本文档为 5 大测试类型提供详细的用例设计方法、检查清单和示例。
在 SKILL.md 第三步生成用例时按需参考对应章节。

---

## ① 正向功能测试

### 设计方法
- 按需求文档逐条对应，每个功能点至少一条正向用例
- 覆盖完整的主流程（从入口到最终结果）
- 包含多种合法输入的组合

### 检查清单
- [ ] 每个需求功能点都有对应正向用例
- [ ] 主流程端到端走通（创建→查看→编辑→删除）
- [ ] 数据持久化验证（操作后刷新数据是否还在）
- [ ] 多种合法输入格式都能正常处理

### 示例（以"用户注册"为例）
```
TC-REG-001 | 使用有效信息完成注册
步骤: 填写合法邮箱+密码(8位以上)+手机号 → 点击注册
预期: 注册成功，跳转欢迎页，数据库生成新用户记录
```

## ② 逆向 / 异常测试

### 设计方法：等价类划分
将输入分为有效等价类和无效等价类，每类至少一条用例。

常见无效等价类：
| 类型 | 示例 |
|------|------|
| 空值 | 空字符串、null、undefined |
| 格式错误 | 邮箱无@、手机号含字母 |
| 类型错误 | 数字字段填文本 |
| 长度超限 | 超过最大字符数 |
| 非法字符 | SQL注入、XSS脚本、特殊符号 |
| 逻辑非法 | 结束时间早于开始时间 |
| 重复操作 | 重复注册已存在账号 |

### 检查清单
- [ ] 所有必填字段都有"为空"的用例
- [ ] 格式有要求的字段有"格式错误"的用例
- [ ] 有长度限制的字段有"超长"的用例
- [ ] 重复/冲突操作有对应用例
- [ ] 操作失败后的错误提示是否友好、准确
- [ ] 失败后系统状态是否正确回滚

### 特殊输入值备忘
```
文本类：
  - 空字符串 ""
  - 纯空格 "   "
  - 超长字符串（超出 maxLength）
  - SQL注入：' OR 1=1; --
  - XSS：<script>alert(1)</script>
  - 特殊符号：!@#$%^&*()
  - 中文、日文、阿拉伯文（国际化）
  - emoji：😀🎉

数字类：
  - 0
  - 负数 -1
  - 小数 0.01
  - 超大数 99999999999
  - 非数字字符

日期类：
  - 今天
  - 过去日期（如果不允许）
  - 未来日期（如果不允许）
  - 非法日期 2024-13-01
  - 跨时区
```

## ③ 边界值测试

### 设计方法：边界值分析
对每个有范围限制的字段，测试以下 5 个点：

```
最小值-1  |  最小值  |  中间值  |  最大值  |  最大值+1
  ❌      |    ✅    |    ✅    |    ✅   |    ❌
```

### 常见边界场景

| 场景 | 边界点 |
|------|--------|
| 字符长度（如密码8-20位）| 7位、8位、14位、20位、21位 |
| 数量/金额（如购买1-99件）| 0件、1件、50件、99件、100件 |
| 日期范围 | 范围首日、范围末日、范围外±1天 |
| 分页（每页20条）| 0条、1条、20条、21条 |
| 文件大小（限制10MB）| 9.9MB、10MB、10.1MB |
| 评分（1-5分）| 0分、1分、3分、5分、6分 |

### 检查清单
- [ ] 识别所有有范围/长度/数量限制的字段
- [ ] 每个限制都有恰好在边界处的用例（合法边界+非法边界）
- [ ] 提示信息是否准确反映边界规则

## ④ 业务规则测试

### 设计方法：状态机 + 规则矩阵

**状态机流转测试：**
画出状态流转图，测试每条允许路径和每条禁止路径。

```
订单状态示例：
待支付 →（支付）→ 待发货 →（发货）→ 待收货 →（确认）→ 已完成
                ↓（取消）      ↓（取消）
               已取消        已取消

需测试：
✅ 合法流转：每条箭头对应一条正向用例
❌ 非法流转：已完成→待支付（应报错）、已取消→待发货（应报错）
```

**规则矩阵测试：**
多个条件组合影响结果时，用判定表列出组合。

```
优惠券规则示例：
条件: 订单金额>=100 | 会员等级>=2 | 节假日
结果: 折扣比例

组合1: ✅ | ✅ | ✅ → 9折
组合2: ✅ | ✅ | ❌ → 95折
组合3: ✅ | ❌ | ✅ → 98折
...
每种组合各一条用例
```

### 检查清单
- [ ] 识别所有业务状态及流转规则
- [ ] 合法流转路径全部覆盖
- [ ] 非法流转路径（越级、逆向）覆盖
- [ ] 计算逻辑用多组具体数值验证
- [ ] 关联字段联动（A变化时B是否正确更新）
- [ ] 唯一性约束（重复数据被拒绝）

## ⑤  用户体验测试

### 操作反馈
- [ ] 点击按钮有即时视觉反馈（按下状态）
- [ ] 异步操作有 loading 状态
- [ ] 操作成功有明确成功提示（Toast/弹窗）
- [ ] 操作失败有明确失败原因

### 错误提示质量
```
❌ 差的提示：「操作失败」「系统错误」
✅ 好的提示：「手机号格式不正确，请填写11位数字」
              「该邮箱已被注册，请直接登录或找回密码」
```

### 流程连贯性
- [ ] 完成操作后自动跳转到合理位置
- [ ] 长流程（多步骤）有进度指示
- [ ] 取消操作后返回正确的上一页
- [ ] 浏览器前进/后退不破坏状态

## 用例覆盖度自查清单

生成完测试用例后，逐项确认：

**功能覆盖**
- [ ] 需求文档中每个功能点都有对应用例
- [ ] 正向/逆向比例合理（建议 3:7 或 4:6）

**数据覆盖**
- [ ] 有范围限制的字段都做了边界值测试
- [ ] 必填字段都做了空值测试
- [ ] 格式有要求的字段做了格式错误测试

**业务覆盖**
- [ ] 业务规则（计算、状态流转）有对应用例
- [ ] 唯一性/约束条件有验证

**质量覆盖**
- [ ] 有并发/防重复提交测试（如适用）
- [ ] 有错误提示质量验证

**优先级分布**
- [ ] P0: 10-15%  P1: 30-40%  P2: 35-45%  P3: 5-10%
