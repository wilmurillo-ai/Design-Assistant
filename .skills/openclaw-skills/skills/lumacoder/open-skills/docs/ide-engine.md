# IDE Engine 设计原理

> 面向多 IDE/AI 编辑器的内容分发与同步引擎设计。本文档提取核心设计思想，供跨项目借鉴。

---

## 1. 统一适配器抽象（Adapter Abstraction）

将所有目标 IDE/AI 工具抽象为统一的接口，不关心具体 IDE 的差异，只关心两个核心问题：
- **输出到哪里**（文件路径或目录路径）
- **生成什么内容**（字符串或目录树）

引擎通过统一的适配器接口调度所有 IDE，新增 IDE 支持时不需要改动核心生成逻辑。

---

## 2. 配置驱动预设（Configuration-Driven Presets）

避免为每个 IDE 创建独立的类文件。通过一张**预设配置表**定义所有支持的 IDE：
- `id`：适配器唯一标识
- `name`：显示名称
- `filePath`：目标输出路径（相对于工作区根目录）
- `type`：单文件（`file`）或目录（`directory`）
- `defaultEnabled`：是否默认启用
- `isRuleType`：输出的是规则（rules）还是技能（skills）

运行时根据用户配置，从这张表中动态实例化对应的适配器对象。新增一个 IDE 支持，只需要在配置表中加一行。

---

## 3. 模板方法流水线（Template Method Pipeline）

基类定义一条固定的生成流水线：

```
加载用户规则 → 合并去重 → 排序 → 生成输出
```

子类不修改这条流水线，而是通过**重写钩子方法**来定制行为：
- `getOutputType()`：返回单文件还是目录
- `generateHeaderContent()`：定制文件头部（元数据、标题、目录索引）
- `shouldOrganizeBySource()`：目录模式下是否按规则源分组
- `shouldGenerateIndex()`：目录模式下是否生成索引文件
- `shouldCopySkillDirectory()`：遇到 `SKILL.md` 时是否复制整个父目录
- `transformSkillContent()`：对内容进行编辑器适配转换

这种设计让核心逻辑稳定，差异点通过钩子局部化。

---

## 4. 双模态输出（Dual-Mode Output）

引擎同时支持两种输出形态：

### 4.1 单文件模式
适用于 `.cursorrules`、`.clinerules`、`.windsurfrules` 等单文件配置。所有规则合并到一个 Markdown 文件中。

### 4.2 目录模式
适用于 `.claude/skills/`、`.github/skills/` 等需要保持目录结构的场景。支持两种组织策略：
- **保持目录结构**：按规则源内部的相对路径完整输出
- **平铺结构**：所有文件直接放在输出目录下，不保留子目录层级

目录模式还会自动生成 `index.md` 索引文件，可选择按源分组或全局平铺。

---

## 5. 规则源标记系统（Source Marker System）

单文件模式的核心创新：用** HTML 注释标记**把文件变成可解析、可部分更新的结构化文档。

标记分为三层：

### 5.1 全局包裹标记（Block Markers）
包裹整个文件内容，用于识别"这个文件是否被引擎管理"。如果用户自己创建了同名文件且没有该标记，引擎会拒绝覆盖，防止数据丢失。

### 5.2 规则源标记（Source Markers）
按规则源（如远程仓库、用户规则）分组，每个源一个区块：
```markdown
<!-- OPEN-AI-RULES:BEGIN-SOURCE source="remote-repo" count="3" -->
...规则内容...
<!-- OPEN-AI-RULES:END-SOURCE source="remote-repo" -->
```

### 5.3 单条规则标记（Rule Markers）
标记单条规则的边界和元数据（source、id、priority）：
```markdown
<!-- OPEN-AI-RULES:BEGIN-RULE source="remote-repo" id="naming-convention" priority="high" -->
...规则正文...
<!-- OPEN-AI-RULES:END-RULE -->
```

这套标记系统让单文件配置文件具备了"数据库表"一样的可寻址、可替换能力。

---

## 6. 部分更新机制（Partial Update）

基于规则源标记系统，引擎支持**增量更新**而不必全量重写文件。

更新流程：
1. **解析**现有文件，按标记拆分为规则源区块、用户内容区块
2. **识别**需要更新的目标规则源区块
3. **替换**目标区块为新内容
4. **保留**其他规则源区块和用户手动添加的内容
5. **重新组装**文件

这解决了多规则源场景下的关键问题：当某个远程规则源更新时，不需要覆盖其他源的内容，也不会丢失用户在文件末尾手动添加的注释或补充规则。

---

## 7. 用户规则保护（User Rules Protection）

引擎自动识别并保护工作区中用户自定义的内容：
- **规则适配器**：自动加载 `ai-rules/` 目录下的文件，作为虚拟规则源 `user-rules`
- **技能适配器**：自动加载 `ai-skills/` 目录下的文件，作为虚拟规则源 `user-skills`

合并策略：
- 用户规则与远程规则按 `id` 去重合并
- 通过排序配置决定冲突时的优先级（例如高优先级规则覆盖低优先级）
- 单文件模式下，用户规则区块用专用标记包裹，在部分更新时始终保留
- 目录模式下，用户规则的文件名参与去重，同名文件以用户本地版本为准

---

## 8. 内容转换引擎（Skill Transformer）

同一份 `SKILL.md` 源文件，不同 IDE 可能有不同的格式要求。引擎在复制/生成时，根据目标适配器 ID 对内容进行**语义转换**。

转换维度包括：
- **Frontmatter 重映射**：字段名转换（如 `allowed-tools` → `tools`）
- **Frontmatter 删除**：移除目标 IDE 不支持的字段
- **工具名映射**：将通用工具名转换为目标编辑器的专有工具名
- **Bash Preamble 剥离**：移除某些编辑器不兼容的 bash 脚本前言
- **Header 注入**：为目标 IDE 添加特定的文件头注释
- **Gstack 专有内容清理**：移除只在特定生态生效的扩展标记

转换失败时**静默降级**，返回原始内容，确保生成流程不中断。

---

## 9. 冲突解决与排序策略

规则合并时的去重与排序策略：

### 9.1 排序维度
- `priority`：按优先级排序（high > medium > low）
- `id`：按规则标识符字母序排序
- `none`：保持原始顺序

### 9.2 去重逻辑
1. 先将所有规则（远程 + 用户）合并
2. 应用排序
3. 按 `id` 去重，排序后的顺序决定了冲突时谁保留（排在前面的优先）

### 9.3 优先级相同时的兜底
当两条规则优先级相同时，用 `id` 作为次要排序条件；没有 `id` 的规则被视为最小值，排在最前面。

---

## 10. 生成安全与回退（Generation Safety & Fallback）

### 10.1 单文件安全写入
如果目标文件已存在但没有引擎的全局标记，说明这是用户自己创建的文件。引擎会：
- 拒绝覆盖
- 提示用户备份或重命名
- 在 UI 中提供"打开文件位置"的快捷操作

### 10.2 目录模式清理策略
目录模式下，每次生成后会**递归清理旧文件**：
- 扫描输出目录下的所有文件
- 对比"期望文件集合"（由当前规则列表推导出的应存在的文件）
- 删除不在期望集合中的文件
- 删除空目录
- 对于 `SKILL` 目录，采用**源目录镜像同步**：与源目录对比，删除源目录中不存在的文件

### 10.3 转换降级
内容转换过程中发生任何异常，都返回原始内容，不中断整体生成流程。

---

## 11. 设计借鉴清单

| 设计点 | 可借鉴场景 |
|--------|-----------|
| 配置驱动预设 | 任何需要支持多平台/多目标输出的工具 |
| 模板方法 + 钩子 | 需要稳定核心流程但允许大量局部定制的框架 |
| 单文件标记系统 | 需要把多个数据源合并到一个可部分更新的文本文件中 |
| 部分更新 | 需要增量同步、保护用户手动修改内容的代码生成器 |
| 用户规则保护 | 官方内容与用户自定义内容需要共存并合并的系统 |
| Skill Transformer | 同一份源内容需要根据不同目标做语义转换的场景 |
| 目录模式清理 | 动态生成目录树，需要自动回收旧文件/旧目录的构建工具 |
