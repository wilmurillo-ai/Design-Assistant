# SRL 评估评分规则详细文档

## 评分总览

### 架构：AI 评审 + 脚本计算

```
AI 阅读 Skill 全部内容
  → 逐维度评审（5 维度 + 3 修正因子）
  → 输出结构化 JSON（每维度：总分 + 子项分 + 证据）
  → Python 脚本接收 JSON
  → 加权计算 + 修正因子应用 + 等级映射
  → 输出 Markdown/JSON 报告
```

**AI 负责**：语义理解、评分判断、证据引用
**脚本负责**：数学加权、修正扣减、等级映射、报告格式化

### 五维权重模型

| 维度 | 代号 | 权重 | 满分 |
|------|------|------|------|
| 工程锚点密度 | `anchor_density` | 25% | 100 |
| 幻觉暴露面 | `hallucination_exposure` | 20% | 100 |
| 失败语义清晰度 | `failure_transparency` | 20% | 100 |
| 溯源性 | `traceability` | 20% | 100 |
| 可重现性推断 | `reproducibility` | 15% | 100 |

### 总分计算

```
final_score = anchor × 0.25 + hallucination × 0.20 + transparency × 0.20 + traceability × 0.20 + reproducibility × 0.15
```

### 等级映射

| 分数范围 | SRL 等级 | 星级 | 使用建议 |
|---------|---------|------|---------|
| 90-100 | SRL-5 | ★★★★★ | 全流程工程化，可进入自动化决策链路 |
| 75-89 | SRL-4 | ★★★★ | 关键步骤锚定可靠平台，结果基本可复现 |
| 60-74 | SRL-3 | ★★★ | 混入部分弱校验环节，适合在明确边界下使用 |
| 40-59 | SRL-2 | ★★ | 大量依赖互联网信息或 AI 推理，需人工复核 |
| 0-39 | SRL-1 | ★ | 全链路 AI 推理，无工程锚点，不建议用于正式决策 |

### 修正因子

| 修正因子 | 修正类型 | 影响 | 公式 |
|---------|---------|------|------|
| 时效性 (Timeliness) | 线性连续 | 可重现性扣减 | max(0, (50-score)/50) × 20% |
| 鲁棒性 (Robustness) | 线性连续 | 幻觉暴露面扣减 | max(0, (50-score)/50) × 10% |
| 级联稳定性 (Cascade Stability) | 线性连续 | 可重现性扣减 | max(0, (50-score)/50) × 20% |

> **线性连续修正**：评分 ≥ 50 时不修正；< 50 时线性扣减，评分越低扣减越多；0 分时扣减达到最大值。例如时效性=25 → 可重现性扣减 (50-25)/50 × 20% = 10%。

## 维度一：工程锚点密度 (Anchor Density) — 25%

### 定义

Skill 工作流中，有多少步骤依赖**可被外部验证的工程化手段**（CLI 调用、API 请求、数据库查询、文件系统操作），而非 AI 自主推理。

### 子项评分

| 子项 | 代号 | 满分 | 评估要点 |
|------|------|------|---------|
| 锚点覆盖率 | `anchor_coverage` | 40 | 有至少一个工程锚点（CLI/API/DB/MCP/文件系统任意一种）的阶段占总阶段的比例 |
| 锚点密度 | `anchor_intensity` | 30 | 锚点总数 / 总阶段数，衡量工程手段的密集程度 |
| 验证闭环 | `verification_loop` | 30 | 是否存在"操作→确认"的完整闭环（如 API 调用→校验返回值、写入文件→读取确认） |

**总分 = anchor_coverage + anchor_intensity + verification_loop**

> **设计说明**：不按锚点类型（CLI/API/MCP）分别计分。纯 CLI Skill 只要覆盖率高、密度大、有验证闭环，同样可以得满分。

### 评分标准

| 得分 | 解释 |
|------|------|
| 90-100 | 几乎每步都有工程化验证手段 |
| 75-89 | 多数关键步骤有外部验证 |
| 60-74 | 部分步骤有验证，部分纯推理 |
| 40-59 | 少量验证，大段无验证 |
| 0-39 | 几乎无工程锚点，全靠 AI 推理 |

### 典型证据示例

- "SKILL.md 阶段 2 要求使用 `git log` 命令获取提交历史" → anchor_coverage（该阶段有锚点） + anchor_density
- "scripts/analyzer.py 中调用 `requests.get()` 获取 API 数据" → anchor_coverage + anchor_density
- "步骤要求将结果写入 JSON 文件，阶段 4 用 Read 工具校验文件内容" → verification_loop

## 维度二：幻觉暴露面 (Hallucination Exposure) — 20%

### 定义

Skill 工作流中，纯 AI 生成且无外部验证的步骤占总步骤的百分比。**分数越高表示暴露面越小（越好）**。

### 子项评分

| 子项 | 代号 | 满分 | 评估要点 |
|------|------|------|---------|
| 推测性步骤占比 | `speculative_ratio` | 40 | 有多少步骤是纯 AI 推理而无外部验证？（40=无推测, 0=全是推测） |
| 外部验证覆盖率 | `external_validation` | 30 | AI 推理结果是否有配套的外部验证步骤？ |
| 输出确定性 | `output_determinism` | 30 | 输出是结构化数据（高确定性）还是自由文本（低确定性）？ |

**总分 = speculative_ratio + external_validation + output_determinism**

### 评分标准

| 得分 | 暴露比例 | 解释 |
|------|---------|------|
| 90-100 | < 10% | 极低暴露面，几乎无幻觉风险 |
| 75-89 | 10-25% | 低暴露面，可控风险 |
| 60-74 | 25-40% | 中等暴露面，需要关注 |
| 40-59 | 40-60% | 高暴露面，显著幻觉风险 |
| 0-39 | > 60% | 极高暴露面，输出不可信赖 |

## 维度三：失败语义清晰度 (Failure Transparency) — 20%

### 定义

当 Skill 执行过程中遇到错误、异常或无法获取可靠信息时，它是**主动暴露错误并停下来**，还是**脑补一个答案继续往下跑**。

### 子项评分

| 子项 | 代号 | 满分 | 评估要点 |
|------|------|------|---------|
| 错误处理策略 | `error_handling_strategy` | 40 | 见下方详细标准 |
| 置信度降级机制 | `confidence_degradation` | 30 | 见下方详细标准 |
| "我不知道"能力 | `idk_capability` | 30 | 见下方详细标准 |

#### A. 错误处理策略（0-40 分）

| 策略 | 得分 | 示例 |
|------|------|------|
| 明确的错误检测 + 停止 + 用户通知 | 35-40 | "如果 API 返回非 200，停止执行并报告错误" |
| 错误检测 + 降级处理 + 告知 | 25-34 | "如果主 API 失败，切换备用源并标记置信度降低" |
| 有错误检测但处理模糊 | 15-24 | "如果出错，尝试其他方法" |
| 错误处理不完整 | 5-14 | 部分步骤有，部分无 |
| 无错误处理 | 0-4 | 没有提及任何异常场景 |

#### B. 置信度降级机制（0-30 分）

| 机制 | 得分 |
|------|------|
| 有明确的置信度分级（高/中/低）并根据数据源质量动态调整 | 25-30 |
| 在关键步骤标注不确定性 | 15-24 |
| 偶尔提到"可能不准确" | 5-14 |
| 无任何不确定性标注 | 0-4 |

#### C. "我不知道"能力（0-30 分）

| 能力 | 得分 |
|------|------|
| 明确场景下会输出"无法确认"并解释原因 | 25-30 |
| 会标注部分结果为"待验证" | 15-24 |
| 偶尔承认局限性 | 5-14 |
| 从不承认局限，始终输出确定性结论 | 0-4 |

**总分 = error_handling_strategy + confidence_degradation + idk_capability**

## 维度四：溯源性 (Traceability) — 20%

### 定义

Skill 输出的每条信息是否可以追溯到具体的、可验证的数据来源。

### 子项评分

| 子项 | 代号 | 满分 | 评估要点 |
|------|------|------|---------|
| 数据来源分类 | `source_classification` | 40 | 见下方详细标准 |
| 引用标注 | `citation_annotation` | 30 | 关键数据是否标注来源和时间戳？ |
| 可独立验证性 | `independent_verifiability` | 30 | 用户能否用提供的信息独立验证输出？ |

#### A. 数据来源分类（0-40 分）

| 来源类型 | 质量系数 | 示例 |
|---------|---------|------|
| **强溯源**: 官方 API/CLI/数据库 | ×1.0 | GitHub API, 数据库查询结果 |
| **中溯源**: 第三方聚合 API | ×0.6 | 天气 API, 汇率接口 |
| **弱溯源**: 网页爬取 | ×0.3 | 电商页面, 新闻网站 |
| **无溯源**: AI 自主推理 | ×0.0 | 模型"知识"，无外部来源 |

评分基于 Skill 中各类数据来源的加权占比：

| 加权比例 | 得分 |
|---------|------|
| ≥ 0.8 | 35-40 |
| 0.6-0.8 | 25-34 |
| 0.4-0.6 | 15-24 |
| 0.2-0.4 | 5-14 |
| < 0.2 | 0-4 |

#### B. 引用标注（0-30 分）

| 标注质量 | 得分 |
|---------|------|
| 每条关键数据都标注来源和时间戳 | 25-30 |
| 关键数据标注来源 | 15-24 |
| 部分标注 | 5-14 |
| 无标注 | 0-4 |

#### C. 可独立验证性（0-30 分）

| 验证性 | 得分 |
|---------|------|
| 用户可用提供的信息独立验证所有输出 | 25-30 |
| 大部分输出可独立验证 | 15-24 |
| 部分可验证 | 5-14 |
| 无法独立验证 | 0-4 |

**总分 = source_classification + citation_annotation + independent_verifiability**

## 维度五：可重现性推断 (Reproducibility) — 15%

### 定义

根据 Skill 的设计特征，推断其**同输入同输出**的稳定性。

### 子项评分

| 子项 | 代号 | 满分 | 评估要点 |
|------|------|------|---------|
| Tier 确定性 | `tier_determinism` | 40 | 基于 Tier 分类的确定性水平 |
| 状态管理 | `state_management` | 30 | 有无状态持久化机制 |
| 步骤确定性 | `step_determinism` | 30 | 每步是否由输入唯一确定 |

#### A. Tier 确定性（0-40 分）

| Tier | 得分 | 识别方法 |
|------|------|---------|
| Tier 1 (确定性) | 35-40 | 主要使用 CLI/API/DB，几乎无网页爬取 |
| Tier 2 (半确定性) | 20-34 | 混合使用 API 和网页，有缓存机制 |
| Tier 3 (非确定性) | 0-14 | 主要依赖开放互联网搜索或 AI 推理 |

#### B. 状态管理（0-30 分）

| 状态管理 | 得分 |
|---------|------|
| 有完整状态持久化（checkpoint, 文件写入, 数据库记录） | 25-30 |
| 有部分状态管理 | 15-24 |
| 依赖上下文传递，无外部状态 | 5-14 |
| 无任何状态管理 | 0-4 |

#### C. 步骤确定性（0-30 分）

| 确定性 | 得分 |
|---------|------|
| 每步输出由输入唯一确定（纯函数式） | 25-30 |
| 大部分步骤确定性，少量随机因素 | 15-24 |
| 混合确定与非确定步骤 | 5-14 |
| 大量步骤依赖外部随机源 | 0-4 |

**总分 = tier_determinism + state_management + step_determinism**

## 修正因子详细说明

### 时效性 (Timeliness) — 0-100 分

| 评估项 | 高分特征 | 低分特征 |
|--------|---------|---------|
| 外部系统稳定性 | 使用有版本化的官方 API | 依赖频繁变更的网页结构 |
| 6个月后可用性 | 高概率仍可正常工作 | 很可能因外部变更而失效 |
| 缓存/刷新机制 | 有应对变更的策略 | 无任何应对措施 |

**连续修正**：修正量 = max(0, (50 - 评分) / 50) × 20%，即评分 50 时不修正，0 时扣减 20%，线性插值。

### 鲁棒性 (Robustness) — 0-100 分

| 评估项 | 高分特征 | 低分特征 |
|--------|---------|---------|
| 异常输入容忍度 | 完善的输入校验和类型检查 | 假设输入永远正确 |
| SEO/矛盾信息处理 | 有多源交叉验证 | 无任何防御 |
| 边界条件 | 明确处理空值、超长输入等 | 未考虑边界情况 |
| 失败恢复 | 某步失败可回退或跳过 | 一步失败全链崩溃 |

**连续修正**：修正量 = max(0, (50 - 评分) / 50) × 10%，即评分 50 时不修正，0 时扣减 10%，线性插值。

### 级联稳定性 (Cascade Stability) — 0-100 分

| 评估项 | 高分特征 | 低分特征 |
|--------|---------|---------|
| 误差放大系数 | 步骤间有独立验证，误差不累积 | 紧耦合链路，误差逐步放大 |
| 上下文链长度 | 短链路或有中间检查点 | 超长链路无中间验证 |
| 步骤耦合度 | 松耦合，各步独立可验证 | 紧耦合，上步输出是下步唯一输入 |

**连续修正**：修正量 = max(0, (50 - 评分) / 50) × 20%，即评分 50 时不修正，0 时扣减 20%，线性插值。

## 改进建议生成规则

### AI 在评审阶段直接给出改进建议

改进建议作为 `improvement_suggestions` 字段包含在 JSON 输出中：

1. 按**维度得分从低到高排序**
2. 为得分最低的 **3 个维度**各生成 1-2 条具体建议
3. 建议格式：`🔧 [维度名] 当前得分 X/100 → [具体行动] — [预期效果]`

### 建议质量要求

- **具体可执行**：不是"改进错误处理"，而是"在第X步的API调用后添加HTTP状态码检查，非200时停止并报告"
- **引用具体位置**：指明 SKILL.md 的哪个阶段/步骤需要修改
- **给出预期效果**：修改后预计该维度能提升多少分
