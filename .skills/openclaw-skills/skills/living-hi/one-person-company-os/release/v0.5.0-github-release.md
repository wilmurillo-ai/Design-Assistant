# v0.5.0 - Numbered DOCX Deliverables and Post-Launch Ops Pack

This release makes `one-person-company-os` stricter about what counts as real delivery.

The previous versions already improved navigation, runtime clarity, and artifact structure.
The remaining gap was execution quality:

- key artifacts could still fragment into multiple draft shapes
- software and non-software rounds could look productive without leaving auditable deliverables
- launch-stage work could miss deployment and production materials if the right roles were not active

`v0.5.0` closes that gap.

## What changed

### 1. Numbered DOCX outputs only

Formal artifacts inside `产物/` are now numbered `.docx` files by default.

### 2. Real output requirements

The workspace now ships starter deliverables for:

- actual output tracking
- software/code evidence
- non-software deliverables
- deployment and rollback
- production monitoring and operating records

### 3. Launch-stage role and ops upgrade

The default launch-stage role set now includes:

- `运维保障`
- `用户运营`

so deployment, production, and feedback materials are visible at the right stage.

### 4. Stronger release validation

`scripts/validate_release.py` now verifies:

- numbered DOCX artifacts under `产物/`
- launch-stage deployment and production files
- launch-stage role briefs for `运维保障` and `用户运营`

## Why this matters

The product is no longer just better at explaining progress.
It is better at forcing a founder or operator to leave real deliverables behind, including the deployment and production artifacts that matter after launch.

## Validation

Validated locally with:

```bash
python3 -m compileall scripts
python3 scripts/validate_release.py
git diff --check
```
