# 微信公众号写作参考：结构模板与标题（大卫风格）

## 文章结构模板

根据选定的文章类型选择对应结构。默认以“大卫实践者视角”为底色，但结构与节奏允许随内容切换，不要一篇文章只靠一种模板走到底。

**字数建议**：
- 默认写 1000-2000 字：主体保持 3 个小节（或 3-5 个步骤），每节短小精悍
- 仅在内容确实较多/深度分析/用户明确长文时写 2000-3000 字：增加小节数量，但每节仍要有明确新信息
- 旗舰长文（可选）写 8000-12000 字：只在用户明确要求/主题确实需要时启用；用“主结构 + 模块混搭 + 图解”推进，避免把目录写成模板

## 结构库：媒体范式（可切换）

下面这些结构不是“必须照抄”，而是用来决定：开头怎么抓人、主体怎么推进、结尾怎么落地。每次写作至少选择其中一个作为主结构，并在大纲里明确写出来。

### M1. a16z 风格：Playbook Essay（论断 + 框架 + 战术清单）

**适用场景**：趋势判断、创业/产品/增长、AI 时代的新打法，读者希望拿到“可执行的策略”。

```
# 标题（强论断/对比/趋势判断）

[开头：一句硬观点或反常识 + 1 个具体细节/小故事，80-120字]
[Why now：为什么这件事此刻值得写，1段]

## 我想说的结论
[一句话总论：清楚、带取舍]

## 这背后的机制（框架）
[给出1个框架：变量、约束、因果链；尽量画图/列表]

## 你可以怎么做（战术清单）
1. [战术1：一句话 + 一个例子/踩坑]
2. [战术2]
3. [战术3]

## 反例与边界
[什么情况下不成立 / 代价是什么 / 风险在哪里]

## 写在最后
[读者下一步动作：周一就能做的两件小事]
```

### M2. Stratechery 风格：Analytical Memo（事件 + 背景 + 概念镜头 + 推演）

**适用场景**：行业格局、平台/生态、监管与策略推演，读者希望“想清楚”而不是“被煽动”。

```
# 标题（事件名 + 判断/推演）

[开头：一个悖论/张力 + 立场自证，80-120字]

## 背景与事实（你需要知道的）
[时间线/关键事实列表，控制在3-6条]

## 核心概念（用一个镜头看清）
[定义/拆分变量/约束；必要时给一个类比]

## 推演：会发生什么
[一阶影响：对谁直接变好/变坏]
[二阶影响：激励如何变化、谁会调整策略]

## 我会盯的3个信号
- [信号1]
- [信号2]
- [信号3]

## 我的判断
[条件 + 结论 + 代价]
```

### M3. Axios 风格：Smart Brevity（先说“是什么/为什么重要”，再补细节）

**适用场景**：快评、更新解读、读者时间很碎但想快速掌握“关键点”。

```
# 标题（信息密度高，避免修辞）

## What’s new
[一句话讲清更新/事件]

## Why it matters
[一句话讲清影响：对谁、在哪个场景、带来什么变化]

## The big picture
[把它放进更大的趋势里，2-3句]

## Yes, but
[限制、代价、反例，2-3句]

## The bottom line
[一句话结论 + 读者行动]
```

### M4. 深度解读：Explainer（发生了什么 + 为什么会这样 + 意味着什么）

**适用场景**：复杂议题解释、技术/商业结合、读者需要“搞懂全貌”。

```
# 标题（解释型：XX到底意味着什么）

[开头：一个具体场景/数据点，把读者拉进来]
[Nut graf：这篇文章到底要解释什么、为什么重要，1段]

## 发生了什么（What）
[事实/定义/范围，尽量短]

## 为什么会这样（Why）
[机制与因果链：3个点即可]

## 意味着什么（So what）
[对公司/行业/个体分别说一句]

## 你可以怎么用（Now what）
[动作建议：2-4条]
```

### M5. 叙事特稿：Narrative Feature（场景引子 + Nut graf + 三幕推进）

**适用场景**：人物、组织、重大项目、亲历故事但又需要观点与洞察支撑。

```
# 标题（故事 + 隐含主题）

[开头：直接入场景，尽量具体，120-200字]
[Nut graf：把故事上升到更大的议题，并预告你要回答的问题，1段]

## 第一幕：问题出现
[冲突/限制/代价]

## 第二幕：尝试与转折
[做法、反复、关键决定]

## 第三幕：结果与代价
[结果、意外、长期影响]

## 写在最后
[回扣开头细节 + 一句取舍式结论]
```

### M6. 对比评测：Decision Review（结论先行 + 维度对齐 + 场景推荐）

**适用场景**：模型/工具/方案对比，“怎么选”“怎么组合”。

```
# 标题（XX vs YY：怎么选）

[TL;DR：一句话推荐 + 适用人群]

## 先说结论
- [谁适合谁]
- [最大差异是什么]

## 维度对齐（同一把尺子）
| 维度 | XX | YY |
|------|----|----|

## 典型场景
1. [场景1：我怎么用]
2. [场景2]

## 结论与边界
[推荐 + 风险/限制]
```

### M10. 深度研究：Deep Research Brief（结论先行 + 多源证据 + 争议与反例 + 行动建议）

**适用场景**：用户明确要“深度研究/调研/带来源报告/多方对比”，或主题本身争议较大、口径不一，需要多源交叉验证。

```

[TL;DR：3-6 句，直接写结论、分歧点、适用边界]

## 研究问题（我想搞清楚什么）
- [问题1]
- [问题2]
- [问题3]

[反证/分歧：如果有，写出来并给来源]

### 发现 2：...

## 关键对比（同一把尺子）
| 维度 | 观点/结论 | 证据与来源 |
|------|-----------|------------|

- [缺口是什么 + 为什么缺]

## 我会怎么做（行动建议）
1. [建议1：可执行 + 适用条件]
2. [建议2]

## 参考来源
1. [标题](url)
2. ...

## 方法说明
检索范围、关键词、筛选标准、时间区间。
```

### M9. 公众号发布体：PART 分段（痛点开场 + 分段快扫读 + 行动建议）

**适用场景**：你已经有一篇结构完整的稿子，但需要更适合微信阅读的“发布版排版”，或用户明确说“分 Part/不要太学术/别太啰嗦/可扫读”。

```
# 标题（一句话讲清主题与判断）

[开头：直接写痛点与不知所措，80-150字；用一句“评测来了/数据来了”转折]
[可选：放一个最关键的链接，单独成行]

PART 01｜先看结果
[3-5 句短句，给结论与推荐]

PART 02｜评测怎么做
[用 3-5 个短段讲清：任务、维度、指标、判分]

PART 03｜重点对象怎么打
[按读者关心点展开，例如“国产模型表现”】【成本/速度取舍】等]

PART 04｜我会怎么选
[1-3 条动作建议，尽量可执行]
```

### M11. 工具/Skill 发布公告：Announcement（先说清楚是什么 + 最短路径 + 可复用闭环）

**适用场景**：发布一个新工具/新 Skill/新流程，读者关心“它到底是什么”“怎么用”“怎么改成我的”“我下一步该做什么”。

```

PART 03｜如何安装和使用
[先给安全性检查清单：摘要/说明书/文件清单；如平台有安全扫描，提一句“高置信度”]
[给两条可复制的安装命令（OpenClaw / clawhub 等）]

## 结构组合规则（让每篇都不死板）

每次写作在大纲里显式选定：
- 1 个主结构（M1-M8 或 A-I）
- 1 个开头模块（从下方选择）
- 1 个结尾模块（从下方选择）
并允许额外选择 2-4 个“中段模块”，把文章写成“骨架 + 插件”，而不是一套模板从头走到尾。

### 中段模块库（可混搭，按需选 2-4 个）

- **P1 一屏结论**：用 3-5 条 bullet 把结论先交付
- **P2 结构迁移图**：用 1-2 张图讲清“旧 → 新”
- **P3 术语极简卡**：每个新术语只给“一句定义 + 一句例子”
- **P4 模块拆解**：按组件拆 3-6 个小节，每节 3 段（结论句→解释→证据/例子）
- **P5 成本与瓶颈**：谁在付什么成本，瓶颈从哪迁移到哪
- **P6 反对意见与边界**：Yes, but；把反例写出来
- **P7 路线图三分法**：基准/乐观/风险，给可验证信号
- **P8 落地清单**：周一就能做的 3-7 个动作

### 图解规范（公众号可复刻）

- 一张图只表达一件事：结构、流程、边界、责任归属，别混在一张里
- 每张图都要有标题与图例：标题回答“这张图解释什么”，图例解释缩写与符号
- 线条只用两种含义：箭头=流向/依赖；无箭头线=关联/边界
- 推荐 5 类图：分层图、数据/控制流图、组件图、时序图、对比表
- 正文写法固定为：一句结论句 → `[图片：这张图想说明什么]` → 2-4 句解释 → 一句边界/反例

### 自动选择器（用户未指定结构时默认启用）

按内容信号自动选：主结构 + 开头模块 + 结尾模块。若同时命中多条，优先级从上到下。

- **工具教程/配置/代码可复现**：主结构 D；开头 S4；结尾 E4
- **旗舰长文/万字/深度拆解/图解说明/产业判断**：主结构 M8；开头 S7；结尾 E2
- **深度研究/调研/带来源报告/多方对比**：主结构 M10；开头 S3；结尾 E2
- **大模型/架构深度拆解（需要框架图）**：主结构 E；开头 S3；结尾 E2
- **同日更新/对比/怎么选**：主结构 M6 或 I；开头 S4；结尾 E1
- **公众号发布排版/分Part/可扫读短段**：主结构 M9；开头 S4；结尾 E1
- **热点快评/读者时间碎/先结论**：主结构 M3 或 H；开头 S4；结尾 E1
- **趋势判断/创业打法/要战术清单**：主结构 M1；开头 S2；结尾 E1
- **行业推演/平台生态/监管与策略**：主结构 M2；开头 S2；结尾 E2
- **人物/项目/亲历故事但要有洞察**：主结构 M5 或 F；开头 S1；结尾 E3
- **复杂概念解释（帮读者“搞懂全貌”）**：主结构 M4；开头 S3；结尾 E2
- **科普/浅显讲清楚/平台迁移类大趋势**：主结构 M7；开头 S6；结尾 E2

### 开头模块（选1）

- **S1 场景入场**：时间/地点/人物/动作，带一点紧张感
- **S2 反常识开门**：先抛一个“不舒服的判断”，再解释原因
- **S3 数据/事实冲击**：一个数字或一句事实，把问题钉在桌面上
- **S4 新闻钩子**：一条更新/发布/争议，说明“为什么现在要聊”
- **S5 我踩过的坑**：一句“我当时也以为……后来发现……”
- **S6 历史类比**：用一个历史锚点/旧产品/旧平台，拉出“平台迁移”的直觉
- **S7 叙事长文开头**：用一个具体场景/现场细节开篇，再用 Nut graf 抬升到议题（硅谷式是其中一种写法）

### 结尾模块（选1）

- **E1 周一就能做的两步**：给读者两个小动作
- **E2 取舍式结论**：结论 + 条件 + 代价
- **E3 回扣开头细节**：用开头的一个细节收束
- **E4 风险边界**：提醒限制、误区、适用范围
- **E5 一句金句**：少抒情，强调清醒与行动

### D. 技术实操型（工具介绍/动手教程）

**适用场景**：介绍某个技术工具、软件配置、实操步骤，类似"在笔记本上实现本地RAG""轻松使用Gemma模型"这类文章。

**风格特征**：
- 结构以**数字编号步骤**为主干（1. 下载 → 2. 配置 → 3. 使用）
- 每节**短小精悍**，100-200字，直接说做什么，不铺垫
- **工具名、路径、代码**具体准确（存储路径、Docker命令等）
- **代码块规范**：命令直接给出，不加废话解释
- **第一人称实操感**："我创建了一个工作区""我就直接选用nomic-embed-text"
- **轻评论**：偶尔一句个人判断，不展开（"这一点对企业级应用特别关键""值得称许的是……"）
- **开篇承接上文**（如有系列）：简短说明这篇文章解决什么问题，1-2句话即可

```
# 标题（"在XX上实现YY"/"轻松使用XX"/"XX入门"）

[背景/问题，1-3句话，说清楚这篇文章解决什么]

[图片：产品/工具截图]

## 1. [第一步操作名称]
[做什么，核心说明，50-100字]
[如有代码/命令：直接给出代码块]
[个人补充（可选）："我就直接选用……""以我的情况为例……"]

## 2. [第二步操作名称]
...（同上）

## 3. [第三步操作名称]
...（同上）

[结尾（可选）：1-2句话点评效果或展望，如"未来还可以在XX方面进一步调优"]
```

**注意**：
- 技术类文章**不需要**长故事开篇，直接进入主题
- 每步之间逻辑连贯（先装工具→再配置→再使用）
- 截图/配图需求较多，在需要图的地方标注 `[图片：描述]`
- 个人判断点到即止，不深入展开分析

### E. 深度技术分析型（大模型/AI技术深度解析）

**适用场景**：对某个技术领域进行系统性分析，如"ChatGLM金融大模型解析""大模型应用架构设计"，既有技术深度又有方法论提炼。

**风格特征**：
- 开篇：背景说明 + 引出核心问题（2-3段，200字内）
- 结构：**编号段落**（ 1  2  3 而非1. 2. 3.），每段一个核心主题
- **框架图先行**："按照惯例，我研究分析……画了一张图"——先给总体框架图
- 逐点拆解：每个编号段落深入讲一个技术点，有代码/表格/示例
- 结尾："X点启示"或"后续开发计划"，总结提炼

```
# 标题（"XX的三点启示"/"XX解析"/"一图读懂XX"）

[背景说明，1-2段，引出要解决的核心问题]

[图片：总体框架图]

按照惯例，我研究分析……画了一张图。这个框架包含X大块内容……

 1 
核心流程/架构

[详细讲解，200-400字]
[图片/表格/代码示例]

 2 
关键技术点1

[详细讲解，200-400字]

 3 
关键技术点2

[详细讲解，200-400字]

...

 N 
X点启示 / 后续计划

[总结提炼，3-5条]
```

**注意**：
- 编号用空格包围的数字（ 1  2  3 ），这是大卫深度技术文的标志性格式
- 必须有框架图（"按照惯例，我画了一张图"）
- 结尾必须提炼方法论（"X点启示""三个关键能力"）

### F. 故事记叙型（亲历事件完整记录）

**适用场景**：记录某个有意义的亲身经历，如"七二一大暴雨""某次项目经历"，强调故事性和细节还原。

**风格特征**：
- 开篇直接进入时间地点（"那天是2012年7月21日……"）
- 细节丰富：对话、心理活动、环境描写
- 第一人称叙事，有"我"的视角和感受
- 时间线清晰（"晚上八点半""后来才知道"）
- 结尾升华：从个人经历到普遍思考

```
# 标题（事件主题，可含时间/地点）

[开篇：直接进入场景，时间+地点+事件]
那天是XXXX年X月X日……

[故事发展：按时间线推进]
## [可选小标题，或不分节]
[细节描写，有对话、心理、环境]
"……"（直接引语对话）
我心想/我注意到/我才发现……（心理活动）

[故事高潮]
[关键场景细节]

[结尾：回到现在 + 反思升华]
现在看来……
然而……这些代价难道是真的不可避免的吗？
```

**注意**：
- 大量细节：具体的时间、对话、动作、环境
- 有温度：个人情感和判断自然流露
- 结尾必须升华：从"我的故事"到"对所有人的启发"

### G. 年度总结型（个人年度回顾）

**适用场景**：年度总结、阶段性回顾，如"我的2023年总结"。

**风格特征**：
- 开篇：简短概括这一年的关键词（1-2段）
- 结构：数字编号分板块（1/ 工作 2/ 读书 3/ 写作 4/ 徒步）
- 每板块：数据 + 具体事件 + 个人感受
- 真实感：有具体地点、人名、书名、数字
- 结尾：展望下一年，简短祝福

```

## 小标题2

[数据：今年读了X本书]
[图片：书籍封面]
[书单列表]

**注意**：
- 必须有真实数据（多少本书、多少篇文章、去过哪些地方）
