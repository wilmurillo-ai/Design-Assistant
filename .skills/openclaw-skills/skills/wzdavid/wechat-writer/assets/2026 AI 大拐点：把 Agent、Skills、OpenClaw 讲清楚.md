# 2026 AI 大拐点：把 Agent、Skills、OpenClaw 讲清楚


1979 年，第一代电子表格软件 VisiCalc 出现。很多人买 Apple II，只为了跑它。从此，个人电脑（PC）从极客玩具，变成了普通人的生产工具。

2007 年 iPhone 发布，软件从桌面走进日常生活。

我今天想说的变化，跟这两次很像。

过去两年，我们看到 AI 可以写文章、写代码……甚至上万字有深度的研究报告。进入 2026 年，一个转向在发生。AI 不只生成内容，它开始执行任务。

而且，就在你的电脑上。它会自己拆任务、调用工具、跑流程、交付结果。

这背后有三个关键词：

- Agent：让 AI 去完成任务
- Skills：让 AI 以模块化能力去调用工具
- OpenClaw：把这套思路做成本地运行的系统

我在上篇文章介绍了 AI 智能体的技术趋势，但是写得有点晦涩，我不太满意。

这篇文章，我只想用尽量小白的语言，把概念与趋势讲清楚。

PART 01｜软件形态发生巨变

过去几十年，软件这样演变。

1）PC 时代的软件

用户  
↓  
桌面软件  
↓  
PC 本地操作系统

2）互联网时代的 Web 服务

用户  
↓  
浏览器  
↓  
Web 服务

这里，还经历了 Web 1.0 到 Web 2.0 的跃迁。

3）移动互联网时代的 App

用户  
↓  
App  
↓  
云服务

4）AI 时代的智能体（正在发生）

用户  
↓  
AI 智能体  
↓  
工具 & 数据

其中：

工具系统（CLI / API / 浏览器 / 插件）  
数据与状态（文件 / 数据库 / 网页 / 消息 / 记忆）

这里有两个容易被忽略的点。

第一，Agent 后面接的不是某个编写好的软件系统。它接的是一堆可以灵活调用的工具，甚至工具都可以按需生成。

同一个任务可能要同时用 API、命令行、浏览器、企业内部系统的界面操作。工具越多，越需要一个统一的调用方式。

第二，Agent 的难点在于能不能管好长期记忆，管好上下文窗口，管好状态。

比如，文件有没有下载完、网页是不是跳转了、某一步失败后该不该重试、结果应该写回哪张表。

PART 02｜一句话讲清楚 Agent

Agent 是一个能自己决定下一步、并且会调用工具去完成任务的 AI 系统。你可以把它理解为数字员工。

人类在办工桌电脑上完成的所有工作，理论上都可以由 Agent 自动化完成。

一个典型 Agent 会反复做一个循环：

目标  
↓  
拆解与规划  
↓  
调用工具  
↓  
执行  
↓  
检查结果  
↓  
继续下一步（或修正计划）

Anthropic 在 Building effective agents 这篇文章里，做了一个很实用的区分：

https://www.anthropic.com/research/building-effective-agents

有些系统基于工作流，路径提前写死，缺乏可扩展性。过去采用 Dify、Coze、n8n 构建的 Agent 大多属于这一类。

而像 OpenClaw 这样的系统才是通用的 Agent，它在运行时根据情况决定怎么用工具、怎么推进任务。

两者都属于 Agentic AI System，但工程难度和风险并不一样。

我自己做产品时的体感也类似：

当你把任务从回答一个问题，升级到跑完一个流程，很多麻烦会突然冒出来。

比如：

- 工具调用失败和重试策略
- 多轮对话的上下文管理
- 中间结果的保存、回放、审计
- 权限边界和安全策略

所以，Agent 的关键不只是模型。更关键的是，工具系统怎么接、技能怎么封装、状态怎么记。

PART 03｜工具系统是 Agent 的手脚

如果把大模型当成大脑，那工具就是手脚。

很多人对 Agent 的误解是：模型变强了，它自然就能干活。

实际做过的人会知道，任务执行真正卡住的地方，经常出现在工具层。

一个比较实用的分类是四类工具：

1）API  
最干净，最稳定。能用 API 的场景，尽量用 API。

2）CLI（命令行）  
工程师的老朋友。人可以跑，AI 也可以跑。最重要的是可组合，一条命令的输出可以直接喂给下一条命令。

3）浏览器与 Web UI  
很多系统没有 API，或者 API 权限不好给。浏览器就成了默认接口。但它也是最脆弱的接口，页面一改，脚本就可能失效。

4）文件与数据系统  
Excel、CSV、数据库、文档库、对象存储、聊天记录。Agent 要能读，也要能写。更要能知道自己写到了哪里。

OpenAI 在 Agents SDK 里把这些统称为 tools，并且把工具调用作为 Agent 的基础能力来设计。

已经广泛被采用的 ReAct Agent 框架也表达了一个核心想法：把推理（Reasoning）与行动（Acting）交错在一起，让模型在做事过程中不断获取外部信息、修正计划。

简而言之：你想让 AI 真的干活，必须给它一套靠谱的工具系统。

这也是现在 Harness Engineering 概念出现的原因，为马儿跑得更快更稳，提供缰绳、马鞍等工具与环境。

PART 04｜Skills 是什么

工具有了，还不够。下一个问题是：怎么让 Agent 以可控的方式调用它们。

Skills 是一种很直接、但很有效的做法。

它是一份可复用的工作说明书。它把三个东西写清楚：

输入是什么  
输出是什么  
要跑哪些工具（命令/接口/脚本）

你可以把 Skill 看成指导 AI 调用工具的说明书。只是，这个说明书，使用自然语言采用 Markdown 格式写成的，命名为 SKILL.md。

说明书是一段 Prompt + 命令行 + 规则的组合。

实际上，标准作业程序（SOP）已经被广泛使用。Skills 是写给 AI 看的 SOP。

采用 Skills 有三个好处。

1）复用  
你在工作里一定有工作步骤和方法。每次让 AI 做同一类事，你都要重复打一段话。Skill 把这些高频 Prompt 和流程固化下来，想用的时候直接调用。

2）可控工具调用  
一旦自由发挥，就容易越界，或者把参数传错。Skill 把允许做什么、怎么做写清楚，调用就更可控。

3）可组合  
一个 Skills 可以按需调用其他 Skills，从而完成复杂的任务。

Skills 渐进加载的特性，使得我们管理上下文的大小也更容易。

在实践中，一个任务往往不是靠一个技能解决的。而且，我们也不希望一个技能变得过大过于臃肿。

因此，我们会分解任务，用一组 Skills 各司其职，来完成复杂任务。

怎么写 Skills 呢？本质上，我们是把完成某个特定任务所需的高频、公共的 Prompt 抽出来做复用，并且用渐进式披露的方式控制上下文，不把所有技能一股脑塞进提示词里，匹配到了再加载细节。

这也是 Skills 显著区别于其他方式的特性。

PART 05｜为什么 Skills 会在 2026 年爆起来

自从去年底发布以来，Agent Skills 规范已经成为一个新的接口层，几乎所有的 AI 编程工具与通用智能体都支持 Skills。

原因在于：

1）它更省上下文  
工具越多，上下文越贵。Skill 先只暴露元信息，需要时再展开细节，能把桌面收拾干净。

2）它更好调试  
人可以直接在终端里跑命令，看输出对不对。不对就改。对了再交给 Agent 调用。比起在一大段 Prompt 里猜它为什么没做对，舒服太多。

3）它更像资产  
你做出一个好用的 Skill，本质上是在沉淀团队经验。它可以被复制、被迭代、被组合。

这也是很多团队说 AI 时代经验终于能沉淀的原因之一。

我认为最关键的点在于，它极大地降低了门槛，用普通人都容易理解和操作的方式，来指挥 AI 智能体完成任务。

PART 06｜OpenClaw：把 Agent + Skills 做成一个可用系统

讲完 Agent 和 Skills，就可以回到爆火的 OpenClaw 了。

我把 OpenClaw 当成一个落地样板。它最打动人的点不在于给出新的概念，而在于落地。

本地优先，把记忆、工具、技能、入口都放在你自己的设备和环境里。

OpenClaw 是一个你在自己设备上运行的个人 AI 助手，它可以接入多个消息渠道，并通过 Skills 来执行任务。

我用一张简化图，把它的思路画出来：

消息入口（飞书/钉钉/企业微信…）  
  │  
  ▼  
Agent Core（理解 + 规划 + 决策）  
  │  
  ▼  
Skills（一个个可调用模块）  
  │  
  ├── 工具：CLI / API / 浏览器 / 文件  
  └── 数据：本地文件 / 记忆 / 业务系统

这里的逻辑很清晰。

消息入口层解决在哪跟智能体对话  
Agent Core 解决怎么拆任务、怎么推进  
Skills 解决怎么调用工具、怎么复用经验  
工具和数据层解决真正把事情做完

细思极恐，OpenClaw 不就是下一代软件的一个实验品吗？

它传递的价值很容易理解：

未来的软件未必都要把 UI 做得很全面很漂亮。

很多软件可能会先把自己变成更好被 Agent 调用的工具。

PART 07｜然而，可靠性与安全，别当成小事

Agent 能干活之后，风险也会变真实。

我在之前的文章中，提过 OpenClaw 的 5 类风险操作。我这里不再展开讲安全风险，只给两个提醒。

1）状态要能追溯  
你最好能回答三个问题：

它做了哪些步骤  
每一步读到了什么、写到了什么  
如果失败了，能不能回放到某一步继续

没有这些，Agent 一旦出错就只能重来一遍。这会让它很难进生产。

2）工具要分级授权  
浏览器、邮件、支付、企业后台，这些工具一旦被错误调用，后果很直接。

我的建议是把权限拆成三层：

只读  
可写但需审批  
可写且自动执行（极少数场景）

工具越强，越要小心。

PART 08｜普通人怎么跟上这波变化

如果你看完这篇文章，希望有所收获，我认为是这三件事。

1）Agent 从对话走进执行  
它是一套任务执行系统。模型是大脑，工具是手脚，状态是生命线。

2）Skills 很可能是新的接口层  
它把经验变成可复用模块，让工具调用更可控、更好调试、更能组合。

3）OpenClaw 代表新的模式  
传统的软件模式已经被打破，消息渠道入口对接 Agent 将成为满足用户需求的主要方式。

强烈建议你开始熟悉如何写 Skills，以及在类似 OpenClaw、Trae 这样的智能体中使用 Skills。

我建议两个动作。

1）先给自己写一个有用的 Skills  
例如，把一段会议录音转文字并整理要点，把一个网页抓取成 Markdown 并做摘要，把一张表自动清洗再画图，等等。你可以直接与 OpenClaw 对话，创建这样的 Skills。

2）再把 Skills 装进 OpenClaw 或 Trae  
然后，观察智能体如何触发 Skills，并正确执行 Skills 中定义的操作，从而完成你的任务需求。

在这个过程中，你会很快发现，刚写的 Skills 需要持续迭代，无论是 SKILL.md 中的提示词，还是调用和控制工具、数据和状态。

当 AI 开始干活，我们就要掌握驾驭 AI 干活的能力。

加油，人类！
