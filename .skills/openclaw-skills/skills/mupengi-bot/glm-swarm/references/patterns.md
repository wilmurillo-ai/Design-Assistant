# 사전 정의 패턴 라이브러리

## 패턴 매칭 우선순위

요청이 들어오면 아래 패턴을 순서대로 대조. 매칭되면 해당 템플릿으로 즉시 분해.
매칭 안 되면 동적 분해 (Planner가 직접 DAG 생성).

---

## 패턴 A: 조사 → 판단 → 실행

### 트리거 키워드
"알아보고 ~ 해줘", "확인하고 ~ 해", "조사해서 ~ 실행"

### 구조
```
[조사 1] ─┐
[조사 2] ─┤ 병렬
[판단]    ← t1+t2 의존
[실행]    ← 판단 의존
```

### 실전 예시
- 포토부스 문의 처리: 문의 내용 확인 + 요금표 조회 → 견적 판단 → 견적서 생성
- 경쟁사 대응: 경쟁사 가격 조사 + 우리 현황 확인 → 비교 분석 → 대응 방안 실행
- 에이전트 장애 대응: 채널 로그 확인 + 에러 로그 확인 → 원인 판단 → 수정 실행

### Context Packet 템플릿
```
조사 worker: [목적] ~를 파악하기 위한 조사 [너의 역할] ~에서 ~를 확인 [출력] 팩트 리스트 (불릿)
판단 worker: [목적] 조사 결과 기반 판단 [너의 역할] 조사 결과를 보고 ~ 판단 [출력] 판단 결과 + 근거
실행 worker: [목적] 판단 결과를 실행 [너의 역할] ~를 실행 [출력] 실행 결과 확인
```

## 패턴 B: 다중 소스 수집 → 합성

### 트리거 키워드
"여기저기 확인", "다 모아서 정리", "리포트 만들어", "브리핑"

### 구조
```
[소스 A] ─┐
[소스 B] ─┤
[소스 C] ─┤ 전부 병렬
[소스 D] ─┘
[합성]    ← 전부 의존
```

### 실전 예시
- 고객 활성도 리포트: 4개 고객사 채널 각각 확인 → 합산 리포트
- 모닝 브리핑: 일정 + 문의 현황 + 에이전트 상태 + 매출 → 종합 브리핑
- ERP 싱크 체크: DB inquiries + memory inquiries + events → 불일치 리포트

### Context Packet 템플릿
```
수집 worker: [목적] ~를 위한 데이터 수집 [너의 역할] ~에서 ~ 데이터 확인 [출력] 구조화된 데이터 (키:값)
합성 worker: [목적] 수집 결과 통합 [너의 역할] N개 소스 결과를 하나의 리포트로 합성 [출력] 리포트 포맷
```

## 패턴 C: 반복 작업 병렬화

### 트리거 키워드
"N건 처리", "다 해줘", "전부 ~", "하나씩 ~"

### 구조
```
[항목 1] ─┐
[항목 2] ─┤
  ...     ─┤ 전부 병렬 (최대 6개씩 배치)
[항목 N] ─┘
[결과 합산] ← concat (Aggregator 생략 가능)
```

### 실전 예시
- 포토부스 문의 10건 ERP 등록
- 스킬 30개 ClawHub 배포 (6개씩 배치)
- 에이전트 6명 상태 체크

### 배치 규칙
- 항목 6개 이하: 전부 한번에 병렬
- 항목 7개 이상: 6개씩 배치로 나눠서 순차 배치 실행
- 각 배치 내에서는 병렬

### Context Packet 템플릿
```
반복 worker: [목적] ~을 N건 처리 중 1건 [너의 역할] ~번째 항목 처리 [입력] 해당 항목 데이터 [출력] 처리 결과 1줄
```

## 패턴 D: 분석 → 제안

### 트리거 키워드
"분석해줘", "의견 줘", "비교해줘", "어떻게 생각해"

### 구조
```
[데이터 수집]    ─┐ 병렬
[비교 대상 수집] ─┘
[분석]           ← 수집 의존
[제안]           ← 분석 의존
```

### 실전 예시
- 모델 비용 비교: GLM 현재 비용 + 대안 모델 가격 → 비교 분석 → 최적화 제안
- 사업 전략: 현재 매출 데이터 + 시장 동향 → 분석 → 전략 제안
- 기술 선택: 옵션A 조사 + 옵션B 조사 → 비교 → 추천

### Context Packet 템플릿
```
수집 worker: [목적] 분석을 위한 데이터 수집 [너의 역할] ~에 대한 데이터 수집 [출력] 정량 데이터 위주
분석 worker: [목적] 데이터 기반 분석 [너의 역할] 수집 데이터를 비교 분석 [출력] 분석 결과 + 수치
제안 worker: [목적] 분석 기반 제안 [너의 역할] 분석 결과를 보고 실행 가능한 제안 생성 [출력] 제안 리스트 (우선순위 포함)
```

## 패턴 E: 검증 → 수정 → 배포 (코드 작업용)

### 트리거 키워드
"고쳐줘", "배포해", "업데이트해", "수정하고 푸시"

### 구조
```
[현재 상태 확인] → [수정 사항 파악] → [수정 실행] → [검증] → [배포]
```

### 주의사항
- 코드 작업은 파일 충돌 위험 → **병렬 최소화**
- 서로 다른 파일 수정이면 병렬 가능
- 같은 파일이면 반드시 순차

---

## 동적 분해 가이드라인

패턴 매칭 실패 시 Planner가 직접 분해할 때의 규칙:

1. **task당 도구 호출 최대 2회** — 넘으면 더 쪼개기
2. **task 설명은 1문장** — 모호하면 worker가 해석 달리함
3. **출력 포맷 필수 명시** — "적당히 정리"는 금지
4. **의존성은 최소화** — 병렬 가능한 것을 순차로 만들지 말 것
5. **공유 상태 명시** — task 간 공유해야 할 데이터가 있으면 Shared Scratchpad에 뭘 쓸지 지정
