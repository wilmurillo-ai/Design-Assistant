# 工作流分析（精简指南）

## 核心分析问题
创建目录结构前先思考：

### 1. 任务理解
- **最终交付物是什么？**（报告、代码、设计、分析等）
- **目标用户是谁？**（技术团队、业务方、公众等）
- **完成标准是什么？**（具体输出、质量要求、成功指标）

### 2. 工作流识别
- **自然阶段有哪些？** 大多数任务可分为3-5个主要阶段
- **依赖关系如何？** 哪些步骤必须先完成
- **哪些可以并行？** 独立的子任务
- **决策点在哪里？** 可能改变方法的关键节点

### 3. 资源分析
- **需要什么输入？**（数据文件、文档、API、用户输入）
- **使用什么工具？**（Python、专业软件、自定义代码）
- **产生什么中间产物？**（处理后的数据、分析结果、草稿）
- **最终输出是什么？**（报告、代码、可视化等）

## 常见工作流模式

### 分析研究类
```
收集 → 清洗 → 分析 → 可视化 → 报告
01_数据收集 → 02_数据清洗 → 03_分析建模 → 04_可视化 → 05_报告生成
```

### 开发工程类
```
规划 → 实现 → 测试 → 部署 → 文档
01_需求分析 → 02_编码实现 → 03_测试验证 → 04_部署上线 → 05_文档编写
```

### 创意内容类
```
研究 → 规划 → 创作 → 优化 → 发布
01_素材研究 → 02_内容规划 → 03_创作生成 → 04_优化调整 → 05_发布分发
```

## 步骤创建指南

### 好的步骤名称
- **动作导向**："提取数据"而非"数据提取阶段"
- **具体明确**："清洗客户记录"而非"处理数据"
- **风格一致**：避免混用不同命名风格
- **粒度适中**：每个步骤对应一个主要活动

### 步骤数量建议
- **3-7个步骤** 通常最合适
- **太少（1-2）**：可能遗漏重要阶段
- **太多（8+）**：可能过度细分，考虑合并

### 处理不确定性
1. **从宽泛开始**："准备"、"执行"、"交付"
2. **记录假设**：注明预期工作流程
3. **随时调整**：根据实际工作重命名、拆分或合并
4. **使用占位名**："Step1_初步分析"可后续改为"01_数据探索"

## 决策框架
- **新建步骤**：有明显阶段切换、使用不同工具/技能、产生中间产物
- **合并步骤**：活动紧密耦合、无中间产物、同一人/工具完成
- **重新排序**：依赖关系要求、需要迭代工作（如草稿→反馈→修订）

## 实践示例

### 客户细分分析
```
01_数据获取理解 → 02_数据清洗准备 → 03_特征探索分析 → 
04_聚类算法应用 → 05_细分市场解释 → 06_行动建议生成
```

### API集成项目
```
01_API文档研究 → 02_架构设计规划 → 03_核心功能实现 → 
04_异常处理优化 → 05_全面测试验证 → 06_使用文档编写
```

## 避免的常见错误
- **过度设计**：为简单任务创建10+步骤
- **结构不足**：所有内容放在一个"处理"文件夹
- **过早优化**：未充分理解任务就设计复杂结构
- **过于僵化**：结构明显不适用时拒绝调整
- **类别思维**：生搬硬套预设步骤，不从第一性原理思考

## 自适应思维
最佳结构：
1. **从任务中涌现**，而非强加于任务
2. **随理解进化**，而非初始固定
3. **服务于工作**，而非指挥工作
4. **记录过程**，而非仅组织文件
