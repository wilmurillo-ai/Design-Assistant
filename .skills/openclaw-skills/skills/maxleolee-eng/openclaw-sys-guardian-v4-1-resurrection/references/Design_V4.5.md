# 龙虾守护 (Lobster Guardian) V4.5 - Decoupled Resurrection Edition

## 1. 核心设计哲学：躯体与灵魂的异步修复
「龙虾守护」V4.5 采用了**解耦自愈架构**。它将「系统配置（物理环境）」与「记忆体系（知识体系）」视为两个独立运行但相互依存的层级，确保在极端崩溃下，系统能以最快速度恢复客观运行能力，并利用主观记忆指导环境进化。

## 2. 三层阶梯自愈逻辑
1. **L1: 物理重启/清创 (The Pulse)**：
   - 发现故障后，自动清理 OpenClaw 及 Provider 级缓存。
   - 按照 [1, 3, 5, 10] 分钟进行指数退避式重启。
2. **L2: 物理回滚 (The Rollback)**：
   - 若 L1 失败，且故障持续，系统将环境变量与 Skill 源代码回滚至物理隔离的备份库（Stable 状态）。
   - **注意**：此时不触动记忆目录，保持认知连续性。
3. **L3: 交互式涅槃 (The Resurrection)**：
   - 如果自动化路径全线崩溃，守护进程转为“引路人模式”，在控制台持续打印引导信息，由人类执行 `lobster-resurrect.sh`。

## 3. 记忆对齐机制 (The Alignment)
当物理层回滚成功后，Agent 自动进入「认知审计」流程：
- **强制锚定**：龙虾守护复活后，第一动作是强制将 T1 (AGENTS.md) 规则注入 Agent 当前 Context 顶端。
- **扫描**：读取当前的 `MEMORY.md`（或其他记忆架构）中的活跃任务。
- **比对**：检查当前物理环境（回滚后的旧版）是否支持任务所需的 Skill。
- **决策**：
  - **一致**：任务续接。
  - **冲突**：向用户发起飞书卡片，申请授权热重装缺失的 Skill 零件。

## 4. 冲突熔断策略
- **单次寿命测试 (Life-Test)**：重装技能后，如果在 5 分钟内再次发生同一故障，该技能自动进入「运行黑名单」。
- **风险规避**：防止由于记忆要求的配置导致系统陷入“安装-崩溃-重启”的死循环。

## 5. 通用性适配
- **解耦优势**：不强制绑定特定的记忆代码。
- **兼容性**：支持任何遵循「环境对齐检查」逻辑的 AI 助手运行环境。
