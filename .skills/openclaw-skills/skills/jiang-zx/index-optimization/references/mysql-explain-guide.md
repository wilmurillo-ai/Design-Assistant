# MySQL EXPLAIN 核心解读

`EXPLAIN` 是 MySQL 性能优化的核心工具。它可以模拟优化器执行 SQL 语句，展示数据库如何处理查询、是否使用索引，以及各表如何连接。

看懂 `EXPLAIN` 结果时，建议抓大放小，重点关注决定查询性能的关键字段。

## 核心字段全览

执行 `EXPLAIN` 后，常见核心列如下：

| 字段名 | 含义 | 关注重点 |
| --- | --- | --- |
| `id` | 查询的执行顺序 | `id` 值越大越先执行；`id` 相同则从上到下执行。 |
| `select_type` | 查询的类型 | 是简单查询（`SIMPLE`）、子查询（`SUBQUERY`）还是联合查询（`UNION`）。 |
| `table` | 正在访问的表名 | 可以是实际表，也可以是派生表或临时表。 |
| `type` | 访问/连接类型 | 极其重要，决定是全表扫描还是走索引。 |
| `possible_keys` | 可能用到的索引 | MySQL 认为可能会使用的索引。 |
| `key` | 实际使用的索引 | 极其重要；若为 `NULL`，通常表示没走到可用索引。 |
| `key_len` | 实际使用的索引长度 | 越短通常越好，可判断复合索引利用列数。 |
| `rows` | 预估扫描行数 | 极其重要；估算读取行数，越少越好。 |
| `filtered` | 返回结果占读取行的百分比 | 越大越好，说明有效行占比高。 |
| `Extra` | 额外附加信息 | 极其重要，包含执行细节与风险信号。 |

## 必看两个核心指标

如果时间有限，优先关注 `type` 和 `Extra`。

### 1) `type`（访问类型）

性能通常从好到坏为：

- `system` / `const`：最快。表仅一行，或通过主键/唯一索引精确等值匹配。
- `eq_ref`：极快。常见于联表，且关联条件命中主键或唯一索引。
- `ref`：很快。使用普通非唯一索引进行等值匹配。
- `range`：较好。使用索引范围扫描（如 `BETWEEN`、`<`、`>`、`IN`）。
- `index`：一般。遍历整个索引树，通常仍慢于精准索引命中。
- `ALL`：最差。全表扫描，数据量大时必须优先优化。

优化目标：通常至少达到 `range`，更理想是 `ref` 及以上。

### 2) `Extra`（额外信息）

- `Using index`：好现象，覆盖索引，通常可避免回表。
- `Using where`：有额外过滤，需结合 `rows` 与 `filtered` 判断代价。
- `Using filesort`：坏现象，发生额外排序，常因 `ORDER BY` 缺少匹配索引。
- `Using temporary`：高风险，出现临时表，常见于复杂 `GROUP BY`/`UNION`，资源开销大。
