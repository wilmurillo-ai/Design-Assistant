# 项目分类系统

> **扫描前先判断项目类型，不同类型问不同的问题。**

---

## 类型定义

### 1. Skill（OpenClaw 扩展）

**定义**：OpenClaw 的扩展 skill，通过 `clawhub install` 安装，扩展 AI 的能力。

**识别特征**：
- 目录有 `SKILL.md`
- 有 `_meta.json` 或 `skill.json`
- 目录在 `skills/` 下
- 有 `scripts/` 目录放 Python 代码

**四视角权重**：

| 视角 | 权重 | 重点问题 |
|------|------|---------|
| 用户 | 30% | SKILL.md 激活关键词是否直觉、CLI 参数设计 |
| 产品 | 30% | README/SKILL.md 承诺的功能实际做到了吗 |
| 项目 | 20% | learnings 有没有闭环、巡检有没有节奏 |
| 技术 | 20% | 代码是否可复用、依赖是否干净 |

**特有检查项**：
- SKILL.md 的 description 是否准确
- 激活关键词是否和功能匹配
- `--help` 是否清晰
- 是否支持 `--dry-run`（如果适用）

---

### 2. CLI 工具

**定义**：通过命令行界面交互的工具，用户通过终端执行。

**识别特征**：
- 有 `main.py` 或 `cli.py` 或 `__main__.py`
- `setup.py` / `pyproject.toml` 中有 `console_scripts`
- 有 `run.sh` 或 `Makefile`
- README 有安装和基本用法

**四视角权重**：

| 视角 | 权重 | 重点问题 |
|------|------|---------|
| 用户 | 35% | --help 清晰吗、错误提示说人话吗 |
| 产品 | 25% | 核心功能是否完整、参数设计是否合理 |
| 项目 | 15% | 配置管理是否规范 |
| 技术 | 25% | 参数解析是否健壮、是否有 dry-run 模式 |

**特有检查项**：
- `--help` 包含用法示例
- 子命令结构清晰
- 支持 `--dry-run` / `--verbose` / `--quiet`
- 配置文件格式是否合理（JSON/YAML/TOML）

---

### 3. Python 库 / 包

**定义**：被其他 Python 代码通过 `import` 使用的库。

**识别特征**：
- 有 `setup.py` / `pyproject.toml` / `setup.cfg`
- `pyproject.toml` 中有 `build-system` 和 `project`
- 目录结构是 `src/` 或包名直接作为目录名

**四视角权重**：

| 视角 | 权重 | 重点问题 |
|------|------|---------|
| 用户 | 25% | API 设计是否符合直觉、文档清晰吗 |
| 产品 | 25% | README 承诺的接口是否都实现了 |
| 项目 | 15% | 版本管理是否规范 |
| 技术 | 35% | 类型提示是否完整、有无循环依赖、测试覆盖 |

**特有检查项**：
- `__init__.py` 导出是否合理（不过度导出）
- API 是否符合 Python idiom（len vs count，filter vs find）
- 类型提示是否完整
- `__version__` 是否暴露

---

### 4. Web 应用

**定义**：有前端界面的 Web 服务，用户通过浏览器交互。

**识别特征**：
- 有 `templates/` 或 `static/` 或 `frontend/` 目录
- `requirements.txt` / `package.json` 中有 Web 框架
- 有 `app.py` / `server.py` / `main.py` 启动入口

**四视角权重**：

| 视角 | 权重 | 重点问题 |
|------|------|---------|
| 用户 | 35% | 核心路径流畅吗、错误提示友好吗 |
| 产品 | 25% | 主要功能是否完整、用户旅程是否顺畅 |
| 项目 | 15% | 部署配置是否规范 |
| 技术 | 25% | 前后端分离是否合理、API 设计是否规范 |

**特有检查项**：
- 核心用户路径（注册 → 登录 → 核心功能）是否有阻断
- 表单验证是否友好
- 错误页面是否定制化
- 移动端适配情况

---

### 5. 通用项目

**不符合以上任意类型的项目。**

**四视角权重**：均衡分布，用户/产品/项目/技术各 25%。

---

## 自动判断流程

```
扫描项目时，自动判断类型：

1. 有 SKILL.md + _meta.json？ → Skill
2. 有 console_scripts 或 __main__.py？ → CLI 工具
3. 有 build-system + import 结构？ → Python 库
4. 有 templates/static/前端框架？ → Web 应用
5. 其他 → 通用项目
```

---

## 类型优先级映射

不同类型项目的关注重点：

| 类型 | 最重要 | 其次 | 再次 |
|------|--------|------|------|
| Skill | 产品（承诺兑现） | 用户（激活体验） | 技术（代码可复用） |
| CLI | 用户（用起来顺） | 技术（健壮性） | 产品（功能完整） |
| Python 库 | 技术（API 设计） | 用户（文档清晰） | 产品（声明一致） |
| Web 应用 | 用户（体验流畅） | 产品（功能完整） | 技术（安全性能） |
| 通用 | 用户 > 产品 > 项目 > 技术 |
