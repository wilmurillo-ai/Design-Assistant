# 项目巡检标准

> **四视角巡检框架**：用户 → 产品 → 项目 → 技术。

---

## 四视角概览

| 视角 | 核心问题 | 详细标准 |
|------|---------|---------|
| 👤 用户 | 用起来顺手吗？ | [user/user-perspective.md](user/user-perspective.md) |
| 📦 产品 | 说到做到了吗？ | [product-requirements.md](product-requirements.md) |
| 🏗 项目 | 运作方式健康吗？ | 本文档 |
| ⚙️ 技术 | 代码本身健康吗？ | [code-standards.md](code-standards.md) |

---

## 一、用户视角巡检

详见：[user/user-perspective.md](user/user-perspective.md)

**核心问题**：这个工具用起来顺手吗？

### 快速检查清单

- [ ] CLI 参数设计直观，--help 清晰
- [ ] 新人 3 步以内能跑起来
- [ ] 错误提示说人话，有修复建议
- [ ] 失败有回滚，不留半成品状态
- [ ] 核心操作 3 步以内完成

---

## 二、产品视角巡检

详见：[product-requirements.md](product-requirements.md)

**核心问题**：README/SKILL.md 承诺的功能，实际做到了吗？

### 快速检查清单

- [ ] README 承诺的功能 ≥ 80% 已实现
- [ ] 文档里 ❌ 标记的痛点 ≤ 2 个未解决
- [ ] SKILL.md 的激活关键词和实际功能匹配
- [ ] 示例代码能正常运行

---

## 三、项目视角巡检

**核心问题**：项目的运作方式健康吗？

### 3.1 learnings 闭环

| 检查项 | 标准 | 当前状态 |
|--------|------|----------|
| learnings 历史有记录 | ≥ 5 条记录 | |
| learnings 影响下次巡检 | 上次拒绝的本次有标记 | |
| 没有反复被拒绝的同类问题 | 同一原因拒绝 ≤ 2 次 | |

### 3.2 巡检节奏

| 检查项 | 标准 | 当前状态 |
|--------|------|----------|
| 巡检有固定周期 | 周巡检或双周巡检 | |
| 巡检结果有跟踪 | 上次发现本次有跟进 | |
| 迭代有记录 | changelog 或 commit 历史 | |

### 3.3 配置合理性

| 检查项 | 标准 | 当前状态 |
|--------|------|----------|
| 配置项有文档 | --help 或 README 说明 | |
| 默认值合理 | 不需要调参就能跑起来 | |
| 敏感配置不硬编码 | API key 等在环境变量或配置文件 | |

### 3.4 依赖管理

| 检查项 | 标准 | 当前状态 |
|--------|------|----------|
| 依赖版本固定 | requirements.txt 或 package.json 锁定版本 | |
| 无高危漏洞 | 依赖无已知 CVE | |
| 依赖不过度 | 核心功能依赖 ≤ 5 个主要包 | |

---

## 四、技术视角巡检

详见：[code-standards.md](code-standards.md)

**核心问题**：代码本身健康吗？

### 快速检查清单

- [ ] 无重复代码（>3 处相同逻辑）
- [ ] 函数长度合理（单个函数 ≤ 100 行）
- [ ] 有测试（核心逻辑有测试覆盖）
- [ ] 无硬编码的秘密（API key 等在环境变量）
- [ ] 依赖无已知安全漏洞
