# 架构设计标准

## 架构评审触发条件

- 新项目立项
- 重大架构变更
- 技术选型重大调整
- 用户反馈严重性能/可用性问题

## 评审检查清单

### 1. 功能性
- [ ] 核心业务流程全覆盖
- [ ] 功能边界清晰
- [ ] 扩展点预留充分

### 2. 性能
- [ ] QPS 评估完整
- [ ] 响应时间目标明确
- [ ] 数据存储容量规划

### 3. 可用性
- [ ] 无单点故障
- [ ] 故障隔离机制
- [ ] 降级熔断策略

### 4. 扩展性
- [ ] 水平扩展能力
- [ ] 模块解耦
- [ ] 插件机制

### 5. 安全性
- [ ] 认证授权机制
- [ ] 数据加密
- [ ] 审计日志

### 6. 可维护性
- [ ] 代码规范
- [ ] 文档完整
- [ ] 监控告警

### 7. 成本
- [ ] 建设成本
- [ ] 运维成本
- [ ] 扩展成本

## 架构设计原则

1. **高可用**：无单点、故障自恢复、多级降级
2. **可扩展**：服务无状态、水平扩展优先、模块解耦
3. **简单性**：KISS、避免过度设计、渐进式演进
4. **安全性**：零信任、纵深防御、安全默认值
5. **可观测性**：日志、指标、链路追踪统一

## 常见架构模式选择

| 场景 | 推荐架构 |
|------|----------|
| 传统 Web 应用 | 分层架构 |
| 大型复杂系统 | 微服务 |
| 异步处理 | 事件驱动 |
| 读写分离 | CQRS |
| 复杂业务领域 | DDD |
| 业务与技术解耦 | 六边形架构 |
