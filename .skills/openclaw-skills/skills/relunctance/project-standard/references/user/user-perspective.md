# 用户视角巡检标准

> **核心问题：这个工具/产品用起来顺手吗？**

用户视角关注的是"实际使用体验"，不是"功能是否完整"。一个功能完整的工具，用起来可能非常别扭。

---

## 检查维度

### 1. CLI / 交互设计

**好用的 CLI 长什么样？**

- [ ] 参数名符合直觉（`--dry-run` 而不是 `--simulate-mode`）
- [ ] 有合理的默认值（不需要用户指定所有参数才能跑）
- [ ] `--help` 或 `-h` 清晰易懂，不只是列出参数还解释用法
- [ ] 子命令结构清晰（`git clone` 而不是 `git --clone`）
- [ ] 没有不必要的交互式输入（能用参数解决的不要 prompt）

**典型问题：**
- 参数名和实际行为不符：`--dry-run` 在某些子命令下不起作用
- 需要记住奇怪的参数顺序
- 没有 `dry-run` 模式，用户担心会破坏数据不敢用

### 2. 学习门槛

**新人上手要多久？**

- [ ] README 有"快速开始"，3 步以内能跑起来
- [ ] 有示例输入输出的截图或截图
- [ ] 没有"前提条件"的黑洞（需要先装 X 才能装 Y，需要先配 Z 才能配 W）
- [ ] 错误信息不只是说"Error"，还会说"你可能需要做什么"

**典型问题：**
- README 只有功能列表，没有实际使用示例
- 需要手动 clone 多个仓库、配置多个环境变量才能跑起来
- 缺少"3 步上手"指南

### 3. 错误提示

**报错说人话还是说机器话？**

- [ ] 错误信息说清楚"什么错了"，不只是"Error occurred"
- [ ] 有修复建议（"你可能需要：..."）
- [ ] 区分错误类型：配置错误 vs 运行时错误 vs 数据错误
- [ ] 日志层级清晰（ERROR 是 ERROR，WARNING 是 WARNING，INFO 是 INFO）

**典型问题：**
- `"Error: something went wrong"` — 完全不知道哪里出了问题
- 网络错误显示成"配置错误"，误导用户去改配置
- 权限错误不说明需要什么权限

### 4. 容错性

**某一步失败了会怎样？**

- [ ] 操作有原子性，失败不会留半成品状态
- [ ] 有备份/回滚机制
- [ ] 失败时有清晰的错误信息和恢复指南
- [ ] 幂等设计：重复执行不会产生副作用

**典型问题：**
- 写入文件失败但进度已经更新，状态不一致
- 没有回滚，失败后只能手动清理
- 网络中断后不知道进度在哪

### 5. 操作流程度

**完成一个操作要几步？**

- [ ] 核心操作 3 步以内能完成
- [ ] 能用配置文件解决的不要每次敲参数
- [ ] 支持 pipeline/chain 操作，方便自动化
- [ ] 有批处理模式，不需要循环调用

**典型问题：**
- 一个操作需要 10 个参数，大部分是合理默认值
- 没有配置文件，每次都要带一长串参数
- 不能串成 pipeline，自动化困难

## 输出模板

```markdown
## 👤 用户视角巡检结果

### 整体评分
[ 优秀 / 良好 / 及格 / 不及格 ]

### 良好项
- ...

### 需改进项
| 问题 | 严重程度 | 改进建议 |
|------|----------|----------|
| ... | 高/中/低 | ... |

### 改进优先级
1. [最紧急的]
2. [次紧急的]
...
```

---

## 评判标准

| 等级 | 分值 | 说明 |
|------|------|------|
| 优秀 | 90-100 | 新人 10 分钟内能上手，所有维度无明显问题 |
| 良好 | 75-89 | 有少量改进项，不影响核心使用 |
| 及格 | 60-74 | 有中等问题，需要改进才能有好的使用体验 |
| 不及格 | <60 | 有严重问题，普通用户无法顺利使用 |
