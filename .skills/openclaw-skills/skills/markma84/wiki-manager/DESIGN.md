# Wiki 关键词系统设计笔记

> 创建：2026-04-09
> 背景：2026-04-08 搬家后搭建，彧哥+空空+小蜂 三人共识

---

## 解决什么问题

- memory_search 全量扫描冷区，太慢
- OpenClaw 原生 wiki 太重，不适合轻量化关键词场景
- 小蜂需要"按需加载、快速检索"的知识索引

---

## 核心理念

**外轻内重：表面极简，背后全是结构**

| 分工 | 职责 |
|---|---|
| Wiki（索引文件） | 轻量关键词 + 一句话定义，不存大段正文 |
| 向量库 | 完整语义记忆，关键词的详细上下文 |
| 原生 Memory | 只管开机唤醒，不参与日常检索 |

---

## 三层记忆分工

```
原生 Memory  → 开机唤醒，熟人感
Wiki         → 轻量关键词索引，快速识题
向量库       → 完整真实记忆，精准上下文
```

---

## 221 三池结构

| 池 | 上限 | 说明 |
|---|---|---|
| core | 200 | 最高频，核心专属术语 |
| buffer | 200 | 次高频，缓冲梯队 |
| new | 100 | 新增词，观察期 |
| **合计** | **500** | **永不膨胀** |

---

## 循环规则（频次制，不看时间）

```
new 高频用（score≥3）→ 升 buffer
buffer 高频用（score≥3）→ 升 core
core 用得少 → 降 buffer
buffer 用得少 → 降 new → 淘汰
```

**核心逻辑：重要内容会反复出现，频次是价值的真实信号**

---

## score 计算规则

- **触发 +1**：我主动查 wiki 并获取到某个关键词
- **只计"真正用上"**：查到但没用不加，查到并用于思考/回复才加
- **触发者**：我（关键词系统是给我用的）
- **频次制**：不看时间，只看实际调用次数

---

## 新增关键词触发规则

| 来源 | 触发者 | 说明 |
|---|---|---|
| 历史对话 | 我（从记忆中恢复时） | 昨天搬家丢的内容，由我主动捞 |
| 实时对话 | 我（判断重要新概念时） | 彧哥问到我不认识的词，我主动加 |

**为什么不怕加错**：赛马机制 + 频次制，不合适的词自然被刷出去

---

## 混合型 Skill 结构

```
SKILL.md     → 操作逻辑（查/增/升降级/淘汰/备份）
wiki.json    → 持久数据（500词条 + 三池 + 定义）
DESIGN.md   → 设计笔记（本文档）
```

- **SKILL.md**：管"怎么做"
- **wiki.json**：管"存什么"
- **DESIGN.md**：管"为什么这么设计"

---

## compact 恢复

compact 后只 reload wiki.json 即可恢复全部认知：
```bash
cat wiki/wiki.json
```
