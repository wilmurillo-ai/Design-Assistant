# PRD 编写工作流程

## 阶段一：需求收集

### 1.1 背景调研
- 产品定位是什么？
- 目标用户是谁？
- 解决什么痛点？
- 核心功能有哪些？

### 1.2 输出物
```
## 1. 产品概述

### 1.1 产品背景
[2-3段产品背景描述]

### 1.2 产品目标
- 目标1：具体可量化
- 目标2：具体可量化
- 目标3：具体可量化

### 1.3 系统范围

| 组件 | 说明 |
|------|------|
| [组件名] | [组件说明] |
```

## 阶段二：用例建模

### 2.1 识别参与者
问自己：
- 谁会使用这个系统？
- 哪些外部系统会与本系统交互？
- 是否有自动化参与者？

### 2.2 识别用例
问自己：
- 参与者能看到/修改什么？
- 需要哪些数据？
- 需要通知参与者什么？

### 2.3 建立关系
- 包含关系（Include）：公共子流程
- 扩展关系（Extend）：可选扩展
- 泛化关系（Generalization）：继承

### 2.4 用例优先级
- P0：核心流程，MVP必须
- P1：重要功能，第一版
- P2：优化体验，后续迭代
- P3：探索性功能，暂不考虑

## 阶段三：用例规格编写

### 3.1 基本事件流
每个步骤格式：
```
1. [参与者]执行[动作]
2. [系统]响应[结果]
3. ...
```

### 3.2 备选事件流
格式：
```
[条件编号]. [条件描述]
   → 如果[条件发生]
   → 系统[响应处理]
```

### 3.3 业务规则
- 数据格式要求
- 业务逻辑约束
- 状态转换规则

### 3.4 数据字典
每个数据表包含：
- 主键/外键定义
- 字段类型和长度
- 取值范围
- 必填/可选项
- 业务含义

## 阶段四：交互设计

### 4.1 页面流程
```
用户 ──操作──► 页面A ──触发──► 页面B ──返回──► 结果
```

### 4.2 时序图要素
- 对象/参与者
- 消息（同步/异步）
- 激活条
- 返回箭头

### 4.3 状态机设计
状态定义：
- 初始状态
- 中间状态
- 终态
- 状态转换条件

## 阶段五：UI规范

### 5.1 布局规范
- 栅格系统
- 留白规范
- 响应式断点

### 5.2 组件规范
- 组件状态（正常/悬停/禁用/加载）
- 交互反馈
- 动画时长

### 5.3 视觉规范
- 色彩系统（主色/辅助色/功能色）
- 字体层级
- 图标风格

## 阶段六：非功能需求

### 6.1 性能指标
- 响应时间
- 并发能力
- 资源消耗

### 6.2 安全要求
- 认证授权
- 数据加密
- 审计日志

### 6.3 兼容性
- 浏览器支持
- 设备适配
- 网络要求

## 常见问题

### Q: 用例太抽象怎么办？
A: 检查是否包含具体的操作步骤，用例应该是可执行的。

### Q: 用例太细碎怎么办？
A: 将连续的操作合并，用例应该描述一个完整的业务目标。

### Q: 如何确定前置条件？
A: 问自己：如果没有这个条件，后续步骤能执行吗？

### Q: 备选事件流写不全怎么办？
A: 从错误处理、边界条件、异常情况三个维度思考。
