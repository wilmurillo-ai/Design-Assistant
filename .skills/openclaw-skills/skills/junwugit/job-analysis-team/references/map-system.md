## MAP 的核心逻辑

MAP = **multiphase analysis of performance**。它主要为团队训练开发而设计。

核心路径：

> team mission / goal -> team functions -> individual position tasks -> KSAOs -> ratings -> training content

MAP 的关键不是一开始就列所有任务，而是从团队整体使命开始，逐步放大细节，直到足以决定训练什么。

## 三个组织因素

Levine 和 Baker 用三个因素把团队训练需求分成 8 个 cell：

1. **Individual training vs team training**：训练单个成员，还是训练团队整体。
2. **Interpersonal vs production**：训练人际/团队关系，还是训练技术生产任务。
3. **Mature vs immature team**：团队是否已经有任务经验。这里的 immature 指没有相关任务经验，不是性格幼稚。

每个 cell 会对应一组可行的 descriptors、sources、methods、units of analysis。选择 cell 的目的，是避免在不可行来源上浪费时间。例如新设备训练还没有 incumbent，就不能主要依赖现任团队成员提供资料。

## MAP 实施流程

1. **确定训练目的和 cell**：individual/team、interpersonal/production、mature/immature。
2. **定义 team mission**：团队要达成的最终目标。
3. **识别 team functions**：为达成 mission，团队整体必须完成哪些功能。
4. **生成 position tasks**：把 function 分解成各成员角色下的任务。课本建议任务生成会议目标是 12-15 个 team functions 和至少 50 条 individual position tasks。
5. **描述工具和设备关系**：不仅记录有哪些设备，还要说明哪个成员使用、谁共享、信息如何传递。
6. **画 flowchart / time chart**：表示顺序、并行、判断点、循环、刚性/弹性时序。
7. **选择 sources 和 methods**：可用 supervisors、high-ranking officials、experts、trainers、written documents；方法可用 interviews、technical conferences、records review。
8. **生成 KSAOs**：SME 根据任务提出完成任务需要的 KSAO。
9. **任务评分**：每个 task 评 criticality 和 difficulty to learn。这里用 difficulty to learn，是因为目的服务 training。
10. **KSAO 评分**：评估 KSAO 是否新成员必须具备，以及是否能区分优秀成员与普通成员。
11. **汇总与训练决策**：高 criticality、高 difficulty to learn、能区分绩效的 KSAO 更可能进入训练；还要考虑时间和预算。
12. **报告和复核**：写明过程和输出，请专家审查准确性与完整性，再修订。

## Flowcharts 和 time charts

**Flowchart** 用来表达任务流：

- 矩形表示行动或任务执行。
- 菱形表示判断点。
- 箭头表示活动顺序。
- 可以表示循环，例如没击中目标后重新尝试。

**Time chart** 用来表达多个成员活动在时间上的关系：

- 横轴表示时间。
- 不同水平行表示不同成员或角色。
- 箭头表示各自活动。
- 可以看出哪些活动必须先后发生，哪些可以并行或调整。

讲解差异时说：flowchart 更像"流程逻辑图"，time chart 更像"多角色时间谱"。

## 两人空战模拟例子

Levine 和 Baker 用一个两人微电脑空战模拟说明 MAP。一个人操作 joystick 控制方向，另一个人操作 keyboard 控制速度和武器。任何一人都不能单独完成任务；共同 mission 是击落敌机。

在这个例子中：

- 团队没有任务经验，所以偏向 immature team cell。
- 演示重点是 production training，因此关注 individual production 和 team production。
- mission 确定后，把功能拆到成员任务。例如 joystick 角色在 aiming function 下可分为 search、approach、maintain lock。
- 设备分析不仅列出 computer、joystick、keyboard、maps、headphones，还要说明哪个成员使用什么、哪些设备共享。

## 数据来源、评分和训练决策

模拟空战例子中，实际使用的来源是 experts 和 trainers；方法包括访谈和 technical conference 来开发 task inventory。

评分逻辑：

- **Tasks**：criticality + difficulty to learn。
- **KSAOs**：essential for new team members + distinguishes superior from average。

训练优先级通常来自这些评分的组合，但不应机械决定；训练预算、时间、可训练性和组织目标都要进入判断。

## 信息存储与检索

MAP 最终会产出报告、任务清单、评分、KSAO 和流程/时间图。课本强调，工作分析结果如果只放在文件柜里，很难复用。更好的做法是把资料组织成可检索数据库或网络资源，让人能按 individual position 或 team as a whole 检索。

## Team cognitive task analysis

Team CTA 是把第 3 章 cognitive task analysis 扩展到团队。Klein 等人的方法来自 critical incidents，但更强调决策过程：

- 当时注意到哪些 cues？
- 还考虑过哪些 alternatives？
- 为什么做出这个判断？
- 团队成员之间如何共享或整合信息？

Team CTA 能捕捉行为观察不容易看到的"团队心智"或团队决策过程，常见应用包括警察/军事与平民互动、消防指挥决策、专家团队协调。

## 对话使用提示

用户说"我要给团队做培训分析"时，用 MAP 四问开局：

1. 训练对象是个人还是团队？
2. 训练内容偏人际协作还是技术生产？
3. 团队对任务是 mature 还是 immature？
4. 团队的 mission 是什么？

用户提到"流程混乱、互相等待、时机错位"时，优先讲 flowchart 和 time chart，而不是先讲 KSA。
