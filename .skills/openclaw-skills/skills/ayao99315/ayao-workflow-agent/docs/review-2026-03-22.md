# Code Review — 2026-03-22

## Critical（必须修复）
- [scripts/update-task-status.sh:38] 将 `TASK_ID`、`NEW_STATUS`、`COMMIT_HASH`、`TMUX_SESSION` 直接拼进 `python3 -c` 源码；同类写法也出现在 [scripts/on-complete.sh:36]。`task_id` 只要包含单引号就能打断字符串并执行任意 Python 语句，属于直接代码注入。建议全部改成 `python3 - "$arg1" "$arg2"` + `sys.argv`/环境变量传参，禁止拼接解释器源码。
- [scripts/dispatch.sh:197] 生成 tmux runner 时把 `TASK_ID`、`SESSION`、`LOG_FILE` 等未转义地写进 shell 脚本；`task_id` 含 `$()`/反引号会在 agent 侧再次求值，含 `/` 时又会让 [scripts/dispatch.sh:144] 和 [scripts/dispatch.sh:148] 的日志/临时脚本路径直接失效。建议运行参数只通过环境变量或严格转义后的数组传入，并对文件名使用白名单 slug。
- [scripts/generate-image.sh:127] `--backend` 被直接拼成 `$BACKENDS_DIR/${backend_name}.sh` 并 `source`；通过 `../../...` 可路径穿越到 `backends/` 之外并执行任意 `.sh`。建议把 backend 限制为固定 allowlist，且改成子进程执行而不是 `source`。
- [scripts/spawn-agent.sh:85] `agent-pool.json` 被多个脚本无锁 read-modify-write：同类问题还出现在 [scripts/update-agent-status.sh:16]、[scripts/health-check.sh:66]、[scripts/cleanup-agents.sh:61]，而 [scripts/on-complete.sh:73] 又会并发触发这些写入。结果是丢更新、互相覆盖、甚至写出损坏 JSON。建议统一封装单一写入口，使用 `flock` + 临时文件 + `os.replace`。
- [scripts/dispatch.sh:153] 任务先被标记成 `running`，之后才真正 `tmux send-keys`（[scripts/dispatch.sh:325]）。如果 session 不存在或 tmux 故障，脚本会直接退出，但任务会永久卡在 `running`。建议先验证 tmux 目标存在，或在 dispatch 失败 trap 中回滚状态/改成 `failed`。
- [scripts/swarm-new-batch.sh:119] 仍有 `running` 任务时也会 archive 并重建全局 `active-tasks.json`；而 [scripts/update-task-status.sh:16] 和 [scripts/on-complete.sh:17] 始终只操作这个单例文件。旧 agent 晚到 completion 会打到新批次，随后被“task not found”静默吞掉。建议 `new-batch` 在存在活跃任务时拒绝执行，或把 batch/task-file 显式传进 dispatch → on-complete → update-task-status 全链路。

## High（强烈建议修复）
- [scripts/update-task-status.sh:98] 找不到任务时只打印 warning 然后 `exit 0`，把批次切换、重复 completion、task_id 不匹配这类真实错误全部伪装成成功。建议返回专用非零码，并让上游通知/告警。
- [scripts/update-task-status.sh:118] 即使拿了 `flock`，仍然是对 `active-tasks.json` 原地覆盖写；进程被 kill、磁盘写满或 Python 异常时，核心状态文件可能被截断。建议在锁内写临时文件、`fsync`、再 `os.replace`。
- [scripts/on-complete.sh:73] `update-agent-status.sh` 与 `agent-manager.sh` 都被后台 fire-and-forget 启动，且 stderr 被吃掉；一旦失败，agent-pool 和自动扩缩容会静默漂移。建议关键状态更新同步执行，或至少把后台失败写入固定日志并告警。
- [scripts/agent-manager.sh:68] manager 自己没有互斥锁，而 [scripts/on-complete.sh:76] 会在每个任务完成后并发触发它；再叠加 [scripts/spawn-agent.sh:46] 的“先找空槽、后创建 session”流程，多个 manager 很容易同时决定扩容并发生碰撞/过度扩容。建议为 manager 和 spawn 都加全局锁。
- [scripts/health-check.sh:62] 检测到 tmux 已死或 shell prompt 可见时，只更新 pool/发通知，不把任务从 `running` 切走。这样 scheduler 看到的仍是活跃任务，系统会长期停留在错误状态。建议通过 `update-task-status.sh` 把任务标成 `failed`/`stuck`/`retrying`。
- [scripts/review-dashboard.sh:74] `full` review 通过 `task_id in review_task.name` 做子串匹配，`T1` 会被 `T10-REVIEW` 误判为已 review，可能直接放行发版。建议改成显式 `review_of` 字段、`depends_on` 反查或严格的 ID 精确匹配。
- [scripts/review-dashboard.sh:121] 即使缺少必须的 review，脚本也只打印红字但仍然 `exit 0`，上游自动化无法把它当真正的 release gate。建议在存在任何未满足的 `full` review 时 `sys.exit(1)`。
- [scripts/generate-image.sh:157] 参数校验失败时 `parse_args "$@" || return 0` 会直接成功退出；[scripts/generate-image.sh:166] backend/stub 失败也被吞掉，最后 [scripts/generate-image.sh:167] 仍打印“✅ Image generated”。这会让调用方在没有产物时误判成功。建议用户输入错误返回非零，且只有确认输出文件存在后才打印成功。
- [scripts/swarm-config.sh:76] 读取配置失败时会悄悄把 `data` 重置成 `{}`，随后下一次 `set` 就会覆盖掉原有配置；当前写路径也没有锁。建议损坏配置时 fail fast，不要自动清空，并为 `set` 增加互斥与原子替换。

## Medium（建议改进）
- [scripts/dispatch.sh:231] Claude `--output-format json` 模式把 stderr 全部重定向到 `/dev/null`，认证失败、CLI 异常和运行时报错会丢失诊断信息，只留下一个 exit code。建议把 stderr 单独 tee 到日志或 sidecar。
- [scripts/update-task-status.sh:19] 代码默认依赖 `flock`（macOS 还依赖 Homebrew util-linux），但没有显式依赖检查或降级路径；缺少该命令时只会报一个模糊的 shell 错误。建议在脚本入口做依赖探测并给出可操作提示。
- [scripts/dispatch.sh:47]、[scripts/swarm-config.sh:21]、[scripts/spawn-agent.sh:23]、[scripts/health-check.sh:62] 普遍假设 `python3`、`tmux`、`openclaw` 一定存在；缺失时不是静默吞掉，就是在半路以不清晰的方式失败。建议统一做 preflight check，并区分“不可用但可降级”和“必须中止”的依赖。
- [scripts/agent-manager.sh:31] 通过 `eval "$(python3 ...)"` 把 Python 输出再喂回 shell。虽然当前输出主要是数字/布尔值，但这种模式脆弱、可维护性差，也让未来字段变更更容易引入注入问题。建议改成 JSON 输出后由 shell 显式解析，或直接把逻辑留在 Python。
- [scripts/on-complete.sh:199] token milestone 去重文件固定写到全局 `/tmp/agent-swarm-token-warned.json`，没有按 batch/project 隔离。新 swarm 继承旧阈值状态后，后续里程碑提醒会丢失。建议按 `batch_id` 或任务文件指纹命名。

## Low / Nitpick（可选）
- [scripts/monitor.sh:1] 这个脚本没有 `set -euo pipefail`，session 列表里还保留了过时的 `cc-frontend` 名称，看起来已经与当前命名体系脱节；继续保留会产生误导性的监控信号。
- [scripts/health-check.sh:50] `ISSUES_FOUND` 在 `python3 | while read ...` 管道体内赋值，但这个 `while` 运行在 subshell 里，外层变量实际上不会被更新；虽然当前没有消费这个变量，但会误导后续维护者。

## Summary
整体评估：核心链路已经具备可用雏形，但共享状态层、输入边界处理和失败语义还停留在“单操作者脚本”级别，离稳定的多 agent 编排器还有明显距离。现在最危险的不是功能缺失，而是“看起来成功、实际上状态已经漂移”。

最需要优先处理的前 3 个问题：
1. 彻底移除 `python3 -c` / here-doc 中对 `task_id`、`session`、`backend` 等用户输入的源码级拼接，先堵住注入面。
2. 给 `active-tasks.json`、`agent-pool.json` 和批次切换建立统一的锁与原子写入机制，避免并发覆盖和晚到 completion 丢失。
3. 修复 `dispatch.sh` / `health-check.sh` / `review-dashboard.sh` / `generate-image.sh` 的假成功路径，让错误能被真实传播和阻断。
