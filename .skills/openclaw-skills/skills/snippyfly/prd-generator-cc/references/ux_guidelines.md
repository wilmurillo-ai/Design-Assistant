## 1. 核心设计原则

在 PRD 中描述交互设计时，遵循以下原则：

**清晰优于聪明**：界面行为应可预期，避免过于"聪明"的隐式逻辑。  
**即时反馈**：用户每一个操作都应获得视觉/文字反馈，不超过 100ms。  
**容错设计**：所有破坏性操作（删除、取消、提交）应提供确认步骤或撤销机制。  
**渐进披露**：复杂功能采用分步展示，避免一次性呈现过多信息。

## 2. 反馈与状态规范

### 2.1 加载状态

| 场景 | 加载时间 | 推荐方案 |
|------|----------|----------|
| 页面级加载 | 任意 | 骨架屏（Skeleton Screen）|
| 按钮操作（提交、保存）| < 3s | 按钮显示 loading 动画，禁用点击 |
| 局部内容刷新 | < 3s | 局部 Spinner，不阻塞其余操作 |
| 超时（> 3s）| — | 显示错误提示 + "重试"按钮 |

**PRD 描述示例：**  
> 用户点击"提交"后，按钮切换为 Loading 状态并禁用，系统在 3s 内返回结果；若超时，显示 Toast 错误提示"提交失败，请检查网络后重试"并恢复按钮可点击状态。

### 2.2 Toast 通知

| 类型 | 触发场景 | 显示时长 |
|------|----------|----------|
| 成功（绿色）| 操作成功 | 3s 后自动消失 |
| 警告（橙色）| 需用户注意但不阻断操作 | 5s 或手动关闭 |
| 错误（红色）| 操作失败 | 需手动关闭 |
| 信息（蓝色）| 中性提示 | 3s 后自动消失 |

### 2.3 操作确认弹窗

**适用场景**：删除、批量操作、不可逆变更。

弹窗结构：
- 标题：明确说明动作（"删除订单"，而非"确认"）
- 正文：简述后果（"删除后无法恢复"）
- 按钮：主操作按钮用危险色（红色），取消按钮在左侧

## 3. 表单与输入规范

### 3.1 输入框

- 占位符文字（Placeholder）不应替代标签（Label）
- 字段验证应在失焦（onBlur）时触发，而非实时触发（避免频繁报错打扰用户）
- 错误信息显示在字段下方，红色文字，不超过 1 行

### 3.2 必填与选填

- 必填字段用 `*` 标注，并在表单顶部说明"* 为必填项"
- 选填字段可在标签旁注明"（选填）"

### 3.3 提交按钮

- 主操作按钮置于表单底部右侧
- 取消/返回按钮置于左侧，样式为次要按钮（不填充色）
- 表单未修改时，"保存"按钮应置灰（disabled）

### 3.4 文件上传

- 支持拖拽和点击两种方式
- 明确说明支持的文件类型和大小上限（如："仅支持 PDF、JPG，单文件不超过 10MB"）
- 上传进度条实时展示

## 4. 导航与信息架构

### 4.1 面包屑（Breadcrumb）

适用于层级超过 2 级的页面，帮助用户定位当前位置。  
格式：`首页 > 分类 > 当前页`，最后一级不可点击。

### 4.2 标签页（Tabs）

- 标签数量建议 2–6 个；超过 6 个考虑使用下拉或左侧菜单
- 当前激活标签用底部蓝线或加粗字体区分
- 切换标签时，如有未保存内容，需弹窗提醒

### 4.3 侧边栏/抽屉（Drawer）

- 适用于需要保持主内容可见的辅助操作（筛选、详情预览）
- 从右侧滑出，宽度建议 360–480px（移动端全屏）
- 提供明显的关闭按钮（右上角 ✕）

## 5. 数据展示规范

### 5.1 表格

| 要素 | 规范 |
|------|------|
| 列宽 | 根据内容自适应，关键列（如状态）固定宽度 |
| 排序 | 支持排序的列显示排序图标，默认状态为中性（非升非降）|
| 分页 | 默认每页 20 条；提供 20/50/100 条切换 |
| 选中行 | 选中行高亮背景色 |
| 操作列 | 操作按钮固定在最右列，hover 时显示 |

### 5.2 数字与金额

- 金额：统一使用千分位分隔符（如 ¥ 1,234.56），保留两位小数
- 百分比：保留一位小数（如 12.3%）
- 大数量：超过 10000 时使用"万"为单位（如 1.2 万）

### 5.3 时间显示

| 场景 | 格式 |
|------|------|
| 绝对时间 | YYYY-MM-DD HH:mm |
| 相对时间（7天内）| X 分钟前 / X 小时前 / X 天前 |
| 日期范围 | 2024-01-01 至 2024-03-31 |

## 6. 错误处理规范

### 6.1 错误分级

| 级别 | 场景 | 处理方式 |
|------|------|----------|
| 严重错误 | 页面崩溃、服务不可用 | 全屏错误页，提供"返回首页"按钮 |
| 操作失败 | 提交失败、接口报错 | Toast 错误提示 + 保留用户输入 |
| 表单验证 | 必填项空缺、格式错误 | 行内错误文案，聚焦至第一个错误字段 |
| 权限不足 | 访问受限页面 | 403 提示页，说明原因并提供联系途径 |

### 6.2 错误文案要求

- **说明发生了什么**：避免"出错了"这种模糊表述
- **告知用户怎么做**：提供明确的下一步行动
- **避免技术术语**：不暴露错误码或堆栈信息给普通用户

❌ 差示例：`Error 500: Internal server error`  
✅ 好示例：`数据加载失败，请刷新页面重试。如问题持续，请联系客服。`

## 7. 空状态设计

**空状态**是指列表、搜索结果为空时的界面状态。每个空状态应包含：

1. **插图**（可选但推荐）：视觉化传达"空"的概念
2. **标题**：简洁说明当前状态（"暂无订单"）
3. **描述**（可选）：解释原因或提示用户如何填充（"完成第一笔购买后，订单记录将显示在这里"）
4. **行动按钮**（如适用）：引导用户采取下一步操作（"去购物"）

**常见空状态场景：**

| 场景 | 建议文案 |
|------|----------|
| 首次使用，无数据 | "欢迎使用！开始创建你的第一个 [X]" |
| 搜索无结果 | "未找到匹配结果，试试调整搜索词或筛选条件" |
| 权限限制 | "你暂无权限查看此内容，请联系管理员" |
| 数据被清空 | "[X] 已清空，你可以随时重新添加" |

## 8. 移动端注意事项

在 PRD 中描述移动端交互时，注意以下差异：

- **触控目标**：最小点击区域 44×44pt（约 7mm），避免元素过于密集
- **底部操作栏**：主要操作按钮建议放置于底部固定区域（Safe Area 之上）
- **手势操作**：左滑删除、下拉刷新等手势需在 PRD 中明确说明
- **键盘遮挡**：输入框聚焦时，确保输入区域不被软键盘遮挡（需开发注意）
- **网络异常**：移动端网络不稳定概率更高，需重点描述弱网/断网状态的降级方案

## 9. 无障碍要求

**最低要求（WCAG 2.1 AA 标准）：**

- **色彩对比度**：正文文字对比度 ≥ 4.5:1；大号文字（≥18pt）≥ 3:1
- **不以颜色传达唯一信息**：状态标签除颜色外还需有文字说明
- **键盘可访问**：所有交互元素可通过键盘 Tab 键访问
- **图片 Alt 文本**：所有有意义的图片需提供描述性 alt 属性
- **焦点样式**：键盘焦点需有明显的视觉样式（不可移除 outline）

## 10. 文案规范

### 10.1 按钮文案

- 使用动词开头，明确动作：**保存**、**提交**、**删除**、**导出**
- 避免模糊表述：❌ "确认" → ✅ "确认删除"
- 长度控制在 4 个字以内（移动端 2–3 字为佳）

### 10.2 提示文案

- 以用户视角书写，使用"你"而非"您"（根据产品调性决定）
- 正向表述优先：❌"不能超过 100 字" → ✅"最多输入 100 字"
- 技术限制用口语化表述：❌"File size exceeds limit" → ✅"文件太大，请压缩后重新上传（最大 10MB）"

### 10.3 标题层级

| 层级 | 使用场景 | 字号参考 |
|------|----------|----------|
| H1 | 页面标题（每页唯一）| 24–28pt |
| H2 | 主要区块标题 | 18–20pt |
| H3 | 次级区块或卡片标题 | 14–16pt |
| 正文 | 说明文字、列表 | 12–14pt |
| 辅助文字 | 注释、时间戳、标签 | 10–12pt |
