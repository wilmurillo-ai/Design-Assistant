## 各步骤详情

### Step 1: 上下文收集

**输入**：用户触发评审时的上下文（文件路径、git diff、PR 信息）

**任务**：
1. 确定评审范围（单个文件 / 目录 / 全项目）
2. 判断评审模式：
   - **增量（Delta）**：仅评审新增/修改的代码
   - **全量（Full）**：评审整个代码库
   - **PR**：评审 Pull Request 的完整 diff
3. 识别 C++ 版本和编译标准（`__cplusplus`、`.cpp` 文件特征）

**输出**：评审上下文对象

---

### Step 2: C++ 专项预检

**输入**：评审上下文 + C++ 源码

**任务**：
1. 运行 `cpp` skill 的 C++ 专项规则（内存、UB、所有权）
2. 运行 `code-review-sr` 的 regex 本地静态分析预检
3. 合并初步问题列表

**本地静态分析规则（C++）**：
```javascript
const localRules = [
  // 内存
  { pattern: /new\s+\w+\s*\[/, type: 'memory', severity: 'high' },
  { pattern: /delete\s+\w+\s*\[/, type: 'memory', severity: 'high' },
  { pattern: /\bdelete\b(?!\[)/, type: 'memory', severity: 'medium' },
  
  // 安全
  { pattern: /sprintf\s*\(/, type: 'security', severity: 'high' },
  { pattern: /strcpy\s*\(/, type: 'security', severity: 'high' },
  { pattern: /scanf\s*\([^,]+,\s*%s/, type: 'security', severity: 'high' },
  
  // 弃用特性
  { pattern: /std::auto_ptr/, type: 'deprecated', severity: 'medium' },
  { pattern: /#define\s+\w+\s+\d+/, type: 'modern-cpp', severity: 'low' },
  
  // UB 风险
  { pattern: /=\s*\{[^}]*\}\s*;/, type: 'ub', severity: 'medium' }, // 未初始化
  { pattern: /\*\s*\w+\s*=\s*\d+/, type: 'pointer', severity: 'high' }, // 指针运算
];
```

**输出**：预检问题列表

---

### Step 3: 三路并行 AI 评审

**输入**：源码 + 预检问题列表

**并行启动 3 个 subagent**（每个使用不同模型和角色）：

#### Reviewer-1: 功能正确性
```
角色：资深 C++ 软件工程师，专注于逻辑正确性和测试覆盖
关注点：
- 业务逻辑是否正确实现了需求
- 边界条件和异常是否处理
- 是否有测试覆盖关键路径
- 是否有遗漏的 corner case
- 断言（assert）使用是否恰当
```

#### Reviewer-2: 性能与内存
```
角色：C++ 性能工程师，专注于内存管理和性能优化
关注点：
- 内存分配策略是否合理（堆/栈/池）
- 是否有不必要的拷贝（缺少 move 语义）
- 循环和算法复杂度是否最优
- 虚函数调用开销是否可接受
- 缓存友好性（data locality）
- RAII 是否正确使用
```

#### Reviewer-3: 安全与最佳实践
```
角色：C++ 安全专家，专注于安全漏洞和 Modern C++
关注点：
- 缓冲区溢出风险（strcpy, sprintf,scanf）
- 格式化字符串漏洞
- 指针运算安全性
- 输入验证完整性
- Modern C++ 合规（C++17/20/23）
- 数据竞争和线程安全
- 智能指针使用是否正确
```

**输出**：3 份独立的评审报告

---

### Step 4: 问题汇总与去重

**任务**：
1. 合并 3 个 reviewer 的问题列表
2. 识别重复问题（相同文件 + 相同行号 + 相同类型）
3. 升级/降级严重性（基于共识）
4. 按严重性排序：P0 > P1 > P2 > P3

**输出**：去重后的统一问题列表

---

### Step 5: 生成评审报告

**任务**：
1. 计算各维度得分（内存安全、逻辑正确性、性能、安全、可读性）
2. 计算总分（加权平均）
3. 性价比分析（修复成本 vs 收益）
4. 生成修复优先级列表

**输出**：结构化评审报告

---

### Step 6: 用户确认

**展示内容**：
- 问题汇总（按严重性展示）
- 代码评分
- 修复建议及优先级

**用户选择**：
- 继续修复（进入 Step 7）
- 结束评审（输出最终报告）
- 调整评审参数（重新评审）

---

### Step 7: 验证修复

**任务**：
1. 执行 `code-review-fix` 自动修复
2. 重新运行预检确认问题已解决
3. 生成最终评审报告

**输出**：修复确认 + 最终评分

---

## 退出条件

- 连续 **2 轮** 无 P0/P1 问题
- 达到 **10 轮** 评审上限
- 用户主动结束评审
