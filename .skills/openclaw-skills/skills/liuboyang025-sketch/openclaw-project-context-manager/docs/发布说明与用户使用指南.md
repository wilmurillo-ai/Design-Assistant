# 发布说明与用户使用指南

## 一、这是什么

`project-context-manager` 是一个用于长期项目上下文管理的通用 skill。

它主要解决的问题不是“单轮问答”，而是：

- 一个共享主会话里同时推进多个长期项目
- 会话中断、归档或上下文丢失后，需要恢复项目现场
- 需要把项目工作与普通事务分开
- 需要在关键阶段把当前状态固化下来，便于后续继续

一句话：

> 它用文件化规则来管理项目的进入、退出、恢复与固化。

---

## 二、它不是什么

这个 skill 不是：

- OpenClaw 原生多会话 UI 的替代品
- 自动猜中当前项目的黑盒系统
- 完全无配置的全自动项目管理器

它提供的是：

- 一套明确的项目边界规则
- 一套可恢复的文件结构
- 一套项目发现、进入和恢复的工作流

---

## 三、适用场景

适合：

- 需要长期推进的项目
- 需要多轮协作、记录断点、保留决策的任务
- 希望在普通事务之外单独管理项目上下文的情况
- 会话中断后还需要稳定恢复的工作

不太适合：

- 一次性问答
- 单次资料查询
- 临时报错排查
- 很短的脚本任务
- 不需要恢复和阶段固化的轻任务

---

## 四、最小使用方式

### 方式 A：接管已有项目目录

当你已经有一个项目文件夹时，可以对 assistant 说：

- 这是我的项目目录，帮我接管
- 把这个目录纳入项目管理
- 给这个目录补齐项目恢复结构

assistant 应该：

1. 确认项目路径
2. 检查该目录是否已有恢复结构
3. 缺失时询问是否初始化
4. 接管后，把该项目登记进项目注册表

### 方式 B：初始化一个新项目

当你还没有项目目录，但确定这是长期项目时，可以说：

- 帮我建立一个新项目
- 这是个长期项目，给我建项目结构
- 以后要持续推进，帮我按项目机制管理

assistant 应该：

1. 识别这是长期项目
2. 询问项目路径或项目根目录
3. 初始化标准项目结构
4. 把项目登记进项目注册表

---

## 五、推荐项目结构

每个项目建议至少包含：

- `00_恢复入口.md`
- `00_文档总索引与当前进度.md`
- `01_项目会话与恢复机制说明.md`
- `99_关键决策记录.md`
- `checkpoints/`
- `session-backups/`

这些文件分别负责：

- 定义恢复协议
- 记录当前断点与文档总览
- 说明项目会话机制
- 记录关键决策
- 保存阶段性快照与会话备份

---

## 六、项目发现如何工作

当用户说：

- 恢复项目
- 进入项目
- 回到项目
- 按恢复入口恢复项目

推荐流程是：

1. 读取项目注册表
2. 扫描所有已登记项目路径
3. 检查路径与关键恢复文件是否有效
4. 列出当前可识别项目
5. 让用户选择进入哪个项目
6. 用户确认后，再进入该项目的恢复入口

注意：

- 单个项目里的 `00_恢复入口.md` 解决的是“已知项目后的恢复”
- 项目注册表解决的是“先发现有哪些项目”

---

## 七、为什么有时会要求你确认项目目录

为了让项目能够在未来稳定恢复，assistant 在以下情况可能会提示你确认目录：

- 首次把项目纳入 skill 管理时
- 恢复项目但当前没有目录记录时
- 项目路径失效时
- 无法形成可靠候选项目列表时

但它不应该高频打扰：

- 若路径已经登记且有效，不应反复询问
- 对同一目录问题，默认只提醒一次
- 只有在用户再次要求恢复或路径状态变化时，才适合再次提醒

---

## 八、注册表是必需的吗

不是绝对必需，但在多项目场景下非常推荐。

当前推荐的最小实现是：

- 注册表文件：`<workspace>/projects-registry.json`

它记录：

- 项目名称
- 项目路径
- 当前状态
- 关键恢复文件信息

如果没有注册表，assistant 就更难在“恢复项目”时先列出候选项目。

## 十、常见问题

### 1. 为什么 assistant 不直接猜我要进哪个项目？

因为在多项目场景下，猜测很容易出错。

这个 skill 的原则是：
- 先发现候选项目
- 再让用户确认
- 确认后再恢复

### 2. 为什么我明明有恢复入口，assistant 还是要问项目目录？

因为恢复入口只能在“已知项目路径”后发挥作用。
如果 assistant 还不知道项目在哪里，就没法先发现并进入该项目。

### 3. 为什么不把所有桌面文件夹都当项目？

因为这样误识别太多，也会污染项目列表。
当前推荐只把“已确认纳入管理的项目”写进注册表。

## 十一、发布建议

如果你准备把这个 skill 对外发布，建议：

1. 去掉个人真实路径与真实项目名
2. 把本地样例改成通用示例
3. 明确写清边界：它不是原生多会话替代品
4. 提供最小可运行示例
5. 提供常见错误与恢复说明

## 十二、当前版本定位

当前版本已经具备：

- 项目进入 / 退出 / 恢复 / 固化规则
- 已有项目接管机制
- 新项目初始化机制
- 项目目录确认机制
- 已登记项目扫描机制
- 项目注册表维护规则

后续仍可继续增强：

- 注册表自动维护
- 项目归档规则
