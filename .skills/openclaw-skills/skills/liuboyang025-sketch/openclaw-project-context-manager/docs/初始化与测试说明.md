# 初始化与测试说明

## 一、目的

本文件用于说明 `project-context-manager` 当前版本的初始化方式、已确认的最小运行机制，以及推荐测试流程。

当前版本重点验证三件事：

1. 能否把一个长期项目纳入 skill 管理
2. 能否在会话中断后按恢复入口恢复单个项目
3. 能否在多个已登记项目之间先发现候选项目，再让用户选择进入哪个

---

## 二、当前最小运行机制

当前版本建议由以下几部分组成：

1. `SKILL.md`
   - 定义项目进入 / 退出 / 恢复 / 固化的规则

2. `PROJECT_SYSTEM.md`（可选但推荐）
   - 定义 workspace 级项目系统规则
   - 适合采用统一项目根目录时使用

3. `projects-registry.json`
   - 用于登记已纳入 skill 管理的项目
   - 记录项目名、项目路径、状态、恢复入口文件等信息

4. 每个项目自己的恢复文件
   - `00_恢复入口.md`
   - `00_文档总索引与当前进度.md`
   - `99_关键决策记录.md`
   - `checkpoints/`
   - `session-backups/`

---

## 三、当前建议的恢复逻辑

当用户说：
- 恢复项目
- 进入项目
- 回到项目
- 按恢复入口恢复项目

建议执行顺序为：

1. 读取 `projects-registry.json`
2. 扫描所有已登记项目路径
3. 检查路径与关键恢复文件是否有效
4. 列出当前可识别项目
5. 让用户选择进入哪个项目
6. 选择后，再读取对应项目的 `00_恢复入口.md`
7. 按该项目的恢复协议继续恢复

如果当前只有一个有效项目，也建议先向用户确认，而不是静默默认进入。

---

## 四、首次纳入项目时的推荐流程

### 情况 A：用户已有项目目录

1. 用户提供项目路径
2. 检查该目录是否已有恢复结构
3. 如无结构，则提示是否初始化
4. 如已有结构，则直接接管
5. 将项目登记进 `projects-registry.json`

### 情况 B：用户还没有项目目录

1. 识别这是长期项目，而不是一次性任务
2. 询问是否建立标准项目目录
3. 用户确认后初始化结构
4. 将项目登记进 `projects-registry.json`

## 六、建议测试用例

### 测试 1：单项目恢复

场景：
- 当前只有一个已登记且有效项目
- 用户说“恢复项目”

期望：
- assistant 先扫描已登记项目
- 告知当前仅识别到一个项目
- 仍建议用户确认是否进入

### 测试 2：多项目恢复

场景：
- 当前有多个已登记项目
- 用户说“进入项目”

期望：
- assistant 先列出所有可识别项目
- 让用户选择
- 选择后再进入对应恢复入口

### 测试 3：路径失效

场景：
- 某个项目已登记，但目录被移动或删除

期望：
- assistant 检测到路径失效
- 给出低打扰提示
- 请求用户更新路径，而不是假装恢复成功

### 测试 4：缺少目录记录

场景：
- 用户让 assistant 恢复项目
- 但当前没有该项目的目录登记

期望：
- assistant 提示补充目录
- 不应直接猜测项目

## 七、当前限制

1. 当前版本已补规则，但注册表还属于轻量实现
2. 目前主要解决“已登记项目发现与恢复”问题
3. 尚未建立更完整的自动注册、更新、清理机制
4. 后续可以继续补：
   - 注册表维护规则
   - 项目删除 / 归档规则
   - 路径迁移后的更新流程
   - 更明确的用户使用说明

---
