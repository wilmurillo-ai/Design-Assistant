# 项目会话与恢复机制说明

## 一、目的

本说明用于规范长期项目在主会话中的工作方式，避免项目内容与普通事务混淆，并保证会话中断后仍可恢复。

---

## 二、当前工作方式

- 聊天表面上仍可能发生在主会话中
- 但项目上下文不依赖聊天连续性
- 项目恢复依赖项目目录中的恢复入口、索引、决策、checkpoint 与相关记录

---

## 三、进入项目的方式

用户表达以下意图时，视为进入项目：

- 进入项目
- 恢复项目
- 回到项目
- 继续当前项目

在多项目场景下，应先列出项目并让用户确认目标项目。

---

## 四、退出项目的方式

用户表达以下意图时，视为退出项目：

- 退出项目
- 先切到普通事务
- 当前问题和项目无关

退出项目后：
- 后续普通事务不写入项目记录
- 如后续内容疑似项目相关，应先提醒用户确认是否重新进入项目

---

## 五、会话中断后的恢复方式

恢复顺序：

1. 读取 `00_恢复入口.md`
2. 扫描当前项目目录
3. 读取索引与决策文件
4. 读取当前断点文件
5. 必要时补读最近 checkpoint / memory / 摘要

---

## 六、固化机制

### 用户明确触发时
用户明确下达以下指令时，应直接执行固化：

- 固化当前项目
- 更新恢复文件并保存当前断点
- 保存当前阶段进展

### 关键节点提醒
在以下节点，应主动提醒用户是否固化：

- 完成关键文档后
- 当前断点变化后
- 新决策形成后
- 新增强相关文件后
- 进入新阶段时
- 会话可能暂停或中断前

---

## 七、边界规则

- 项目记录保存的是有效可恢复上下文，不是全部聊天原文
- 项目外普通事务不应纳入项目记录
- 恢复时不能只依赖固定文件，必须先扫描目录并识别新增文件

---

## 八、重要边界说明

本机制管理的是“项目上下文”，不是 OpenClaw 原生 UI 级别的多会话切换。

也就是说，它不等同于在 WebChat 中真正创建了一个独立、持久、可切换的原生项目会话，而是通过文件化项目结构、恢复入口和上下文切换协议来实现长期项目管理。
