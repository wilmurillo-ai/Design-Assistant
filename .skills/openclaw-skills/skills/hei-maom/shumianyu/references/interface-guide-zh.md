# 接口与产品接入说明（中文）

## 这份 Skill 现在做什么

这份 Skill 适合这样的链路：

1. 上游持续送入分段 ASR 文本流
2. 系统只做缓存和累计，不对用户实时展示临时字幕
3. 当整段结束、录音结束或调用方明确提交结束时
4. 再把整段 ASR 结果统一整理成流畅书面语
5. 只有在显式传入 `target_language` 时，才输出翻译结果

它不是一个绑定某家云服务的技能说明，而是一个**可替换接口的行为规范**。

## 为什么改成“结束后统一整理”

这种模式适合：
- 不追求实时字幕
- 更看重最终文本质量
- 希望减少中间态文本闪动或反复改写
- 想在整段上下文完整后再做更自然的书面化处理

这样做的好处：
- 不需要维护临时字幕和回写窗口
- 可以基于整段上下文统一断句和润色
- 更适合会议纪要、采访整理、语音输入后处理

## 推荐的单次最终输出模式

### 1）分段接收
上游持续送入 ASR chunk。

建议：
- 先做轻量归一化
- 处理明显重复片段
- 把 chunk 合并进当前缓冲区

### 2）等待结束信号
在结束前：
- 不输出最终面向用户的文本
- 不生成临时字幕
- 只维护内部缓冲区

可接受的结束信号包括：
- `end_of_stream`
- `is_session_final`
- `commit=true`
- 用户点击“结束”或“提交”
- 上游 ASR 会话关闭

### 3）统一整理
收到结束信号后：
- 对整段文本统一断句
- 去掉口头禅、重复、自我修正
- 改写成流畅书面语
- 若存在 `target_language`，再翻译最终结果

### 4）输出最终结果
默认只输出：
- 最终书面语结果

如果设置了 `target_language`，默认输出：
- 目标语言的最终结果

## 推荐的抽象接口

为了方便你后面替换 API，不要把 Skill 写成某个固定厂商的 SDK 调用说明。
应当只约定“职责”，不约定“厂商实现”。

推荐保留这些抽象层：

### `ingest_asr_chunk(...)`
职责：接收最新的 ASR 分段，并做最基础的归一化。

输入可包含：
- 当前 chunk 文本
- 时间戳
- 置信度
- 说话人信息（如果有）

输出建议：
- 标准化后的 chunk
- 可并入缓冲区的数据结构

### `append_to_buffer(...)`
职责：把每个 chunk 合并到当前整段缓冲区。

建议：
- 保留原始顺序
- 尽量避免重复拼接
- 可记录元数据，但不要把具体存储方式写死

### `detect_stream_end(...)`
职责：判断现在是否已经到达整段结束时刻。

可参考信号：
- 用户手动提交
- 识别流关闭
- 上游 final 会话标记
- 显式 commit 参数

输出建议：
- `ended: true/false`
- `reason: manual_commit | upstream_final | stream_closed | explicit_end`

### `rewrite_to_written(...)`
职责：把累计的整段口语内容统一改写成流畅书面语。

建议：
- 删除口语杂质
- 统一断句
- 保留意思和事实
- 不新增内容
- 优先整段可读性

### `translate_text(...)`
职责：当 `target_language` 存在时，对最终结果做翻译。

建议：
- 基于最终书面语再翻译
- 保留术语、数字、人名、产品名
- 使用领域词库稳定译法

### `commit_final_output(...)`
职责：提交最终结果并清空或归档当前缓冲区。

建议：
- 输出完成后重置会话状态
- 为下一段录音或下一次说话做准备

## 推荐的数据流

可以按下面这个顺序接：

1. 上游 ASR 连续送 chunk
2. `ingest_asr_chunk(...)`
3. `append_to_buffer(...)`
4. `detect_stream_end(...)`
5. 如果未结束：继续累计
6. 如果已结束：
   - 取出整段缓冲区
   - `rewrite_to_written(...)`
   - 如果有 `target_language`，再 `translate_text(...)`
   - 输出最终结果
   - `commit_final_output(...)`

## 领域词库怎么用

如果你有产品名、人名、术语表，可以作为 `domain_lexicon` 传入。

用途：
- 修正 ASR 同音误识别
- 统一专有名词写法
- 翻译时保持术语一致

注意：
- 词库应当“优先参考”，不要“强行覆盖”
- 如果上下文明显不支持某个术语，不要硬替换

## 输出建议

你现在这版要求是：
- 不要实时输出
- 默认不翻译
- 用户手动指定目标语言才翻译
- 面向用户时只输出最终结果

因此在产品集成层，建议默认只保留一个用户可见输出对象：
- `final_output`

如果内部确实需要状态，也只保留接收中、已结束、已提交这类内部状态，不需要临时字幕对象。

## 一个简单例子

实时输入：
1. `今天我们先把登录流程跑通`
2. `然后再看支付和埋点`
3. `如果有时间再补一下告警`
4. 用户点击“结束”

系统行为：
- 结束前：不输出书面稿
- 结束后最终结果：`今天先完成登录流程，再处理支付和埋点。如果有时间，再补充告警能力。`

如果这时传了 `target_language = en`，则最终输出为：
- `Today, we will complete the login flow first, then handle payments and tracking. If time allows, we will also add alerting.`

## 落地原则

记住这几个原则：

- 结束前只累计，不展示最终内容
- 结束后一次性统一整理
- 默认不翻译，按需翻译
- 接口职责抽象化，方便以后替换实现
- 优先整段书面质量，而不是分段即时反馈
