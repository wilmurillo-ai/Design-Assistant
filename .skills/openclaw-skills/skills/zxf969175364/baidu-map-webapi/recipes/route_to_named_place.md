# 地名导航：用户输入地名 → 规划驾车路线

> **适用摘要**: 用户说出目的地名称（如"故宫"、"北京大学"），需要规划从当前位置出发的驾车路线。

## 触发意图

遇到以下用户需求时使用本 recipe：
- "帮我导航去故宫"
- "规划到XX地方的驾车路线"
- "从A到B怎么开车去"
- "我想去[地名]，帮我规划路线"

## 前置条件

| 条件 | 要求 |
|---|---|
| 必要输入 | 起点坐标（用户当前位置信息，BD09LL 格式 或者当前地址信息）、目的地关键词（地名/POI名） |
| 城市信息 | 建议提供城市名，提升搜索准确率 |
| 坐标系 | 起点需为百度坐标（BD09LL），GPS 原始坐标需先转换 |
| 权限 | 标准 AK 即可 |

## 调用链

```
用户输入地名 + 起点坐标
  → Step 1: administrative_region_search（搜索地名，获取 uid 和坐标）
  → Step 2: driving_route_planning（起点坐标 + 终点坐标 + 终点 uid）
  → 返回路线方案（距离、耗时、分段详情）
```

> **为什么不能直接用地名字符串作为终点？**
> 驾车路线规划接口的终点参数只接受坐标格式，
> 必须先通过搜索建议接口将地名转换为坐标和 uid，才能正确规划路线。
> 传入 uid 还能提升路线规划准确性（精确到 POI 入口）。

## 分步说明

### Step 1: 判断用户是否提供自己当前位置, 如没有提供, 请向用户确认当前位置信息

### Step 2: 地名/地址 → 坐标 + uid

**目的**: 将用户输入的自身位置与终点的地名或地址文本，转换为算路所需的坐标和 uid

**参考**: `recipes/address_to_poi.md`（统一入口，含输入类型判断与调用方式）

> `address_to_poi.md` 内部会自动区分：
> - 结构化地址（含门牌号）→ `geocoding`
> - POI 名称/地标 → `administrative_region_search`

**从响应获取**: 目的地的坐标和 POI 唯一标识符（uid）

### Step 3: driving_route_planning（驾车路线规划）

**目的**: 根据起终点坐标规划驾车路线

**参考**: `../references/driving_route_planning.md`

**需要提供**: 起点坐标、终点坐标、终点 uid（提升精度）、坐标系类型

**从响应获取**:
- 预计行驶耗时
- 路线距离
- 过路费估算
- 分段路线详情

## 常见错误

| 错误 | 原因 | 解决方法 |
|---|---|---|
| 直接将地名字符串传给终点参数 | 终点只接受坐标格式 | 必须先调用搜索建议接口获取坐标 |
| 起点使用 GPS 原始坐标但未声明坐标系 | 坐标系不匹配导致偏移 | 声明正确坐标系类型或先做坐标转换 |
| 搜索建议返回结果为空 | 关键词过于模糊或城市指定错误 | 提供更具体的关键词，或不限制城市让全局搜索 |
| 路线规划无结果 | 起终点过于接近，或坐标在不可达区域 | 检查坐标有效性，距离过近时不必规划路线 |

## 变体

- **起点也是地名**: 先对起点也调用搜索建议，获取起点坐标和 uid 后再规划路线
- **步行路线**: 改用步行路线规划接口，参考 `../references/walking_route_planning.md`
- **骑行路线**: 改用骑行路线规划接口，参考 `../references/cycling_route_planning.md`
- **需要多条备选路线**: 开启备选路线返回，最多返回 1-3 条路线
- **有限行车牌**: 传入车牌号，系统自动规避限行路段
