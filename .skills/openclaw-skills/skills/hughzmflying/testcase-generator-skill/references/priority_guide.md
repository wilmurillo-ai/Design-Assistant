# 测试用例优先级定义指南

## 优先级等级

### P0 - Critical (关键)

**定义**: 核心功能，必须通过。短路不上线。

**适用场景**:
- 主要业务流程
- 核心功能
- 涉及资金、数据安全的操作
- 登录/支付/交易等

**示例**:
- 用户登录/登出
- 资金转账
- 订单提交
- 密码修改
- 数据删除

**SLA**: 必须100%通过

---

### P1 - High (高)

**定义**: 重要功能，短路不上线。

**适用场景**:
- 重要业务功能
- 常用功能入口
- 数据展示关键字段
- 配置类功能

**示例**:
- 商品搜索
- 用户信息查看
- 订单查询
- 基础配置修改

**SLA**: 通过率>95%

---

### P2 - Medium (中)

**定义**: 一般功能，迭代修复。

**适用场景**:
- 辅助功能
- 边界条件
- 异常处理
- 较少使用的功能

**示例**:
- 高级搜索筛选
- 导出功能
- 批量操作
- 历史记录查看

**SLA**: 通过率>85%

---

### P3 - Low (低)

**定义**: 建议优化，版本规划。

**适用场景**:
- 优化建议
- UI样式问题
- 性能优化
- 兼容性优化

**示例**:
- 页面加载优化
- 样式微调
- 提示信息优化

**SLA**: 通过率>70%

---

## 优先级判断矩阵

| 功能类型 | 核心路径 | 辅助路径 | 边界/异常 |
|---------|---------|---------|----------|
| 核心业务 | P0 | P1 | P1 |
| 重要功能 | P1 | P2 | P2 |
| 一般功能 | P2 | P3 | P3 |
| 辅助功能 | P3 | - | - |

---

## 优先级调整规则

### 升级规则

1. **安全相关** → P0
2. **影响主要流程** → P0/P1
3. **数据一致性** → P0
4. **用户投诉** → P1

### 降级规则

1. **使用频率低** → 降一级
2. **有替代方案** → 降一级
3. **影响范围小** → 降一级

## 注意事项

1. **覆盖率**: P0必须100%覆盖核心路径
2. **执行**: 优先执行P0、P1用例
3. **回归**: 每次发布前确保P0用例全部通过
4. **维护**: 定期review优先级设置的合理性
