# Handoff V2 使用文档

## 一句话理解

`handoff` 就是把当前工作现场封成一个“交接包”，然后在新会话里先恢复理解，得到你的同意后再继续执行。

## 心智模型

整个流程只有三步：

1. **冻结现场**：`handoff out`
2. **找到交接包**：`handoff list`
3. **恢复并确认**：`handoff in <id|latest>`

## 快速开始

### 1) 暂停当前会话

你说：

```text
handoff out
```

AI 会：

- 生成一个 handoff 包
- 把当前任务压缩进 `handoff.md`
- 返回一个可以直接复制的继续命令

返回示例：

```text
Handoff saved.

  ID:   20260308-1800-abc123
  File: ~/.agents/handoff_context/20260308-1800-abc123/handoff.md

  Next session → handoff in 20260308-1800-abc123
```

### 2) 新会话继续

你说：

```text
handoff in 20260308-1800-abc123
```

或：

```text
handoff in latest
```

AI 会：

- 读取 handoff 包
- 用 3-5 条 bullets 复述当前局面
- 告诉你准备执行的下一步
- 先问你是否继续执行
- 只有在你同意后，才打开关键文件并继续执行

### 3) 忘了 ID 时

你说：

```text
handoff list
```

AI 会列出最近几条 handoff：

```text
20260308-1800-abc123  实现用户认证功能
20260308-1530-def456  修复数据库查询 bug
20260308-1200-ghi789  重构配置管理模块
```

## 什么时候该用

适合：

- 上下文快满了
- 当前任务做到一半，要换会话
- 需要把当前状态交给另一个会话
- 想快速恢复最近一次工作现场

不适合：

- 做完整总结
- 做长期归档
- 记录全部对话过程

## 交接包里有什么

每个 handoff 包固定有 8 个槽位：

1. 当前任务
2. 当前状态
3. 已完成
4. 关键决策
5. 关键约束
6. 关键文件
7. 下一步
8. 待确认

它不是一份复盘，而是一份“恢复执行说明书”。

## 长度建议

- 目标长度：100 行内
- 建议上限：120 行内
- 复杂任务的例外上限：150 行内
- 超过 120 行时，应该先压缩一次
- 超过 150 行时，通常说明 handoff 写散了

## 一个好的 handoff 长什么样

```md
## 当前任务
- 实现用户登录功能

## 当前状态
- 后端 API 已完成并测试通过
- 正在对接前端表单

## 已完成
- 创建 User 模型（models/user.py）
- 实现 JWT 认证（auth/jwt.py）
- 添加登录 API（api/login.py:12）

## 关键决策
- 使用 JWT 而非 Session

## 关键约束
- 密码必须加密存储

## 关键文件
- `api/login.py:12`：登录逻辑
- `auth/jwt.py:45`：Token 生成

## 下一步
- 在前端调用 `/api/login`

## 待确认
- 无
```

它的特点是：

- 短
- 硬
- 可执行
- 不需要重新解释背景

## 常见问题

### `handoff in latest` 是什么意思？

读取最近一次创建的 handoff 包。

### 为什么恢复后不直接执行？

因为 `v2` 的规则是：先恢复理解，再征得你的同意，然后才继续执行。这样更稳，不容易恢复错方向。

### `Session ID` 为什么有时是 `N/A`？

因为当前环境不一定能拿到原始会话标识；这不影响恢复执行。

### handoff 包可以手动改吗？

可以。它就是一份 Markdown 文档，必要时可以手动补充或压缩。
