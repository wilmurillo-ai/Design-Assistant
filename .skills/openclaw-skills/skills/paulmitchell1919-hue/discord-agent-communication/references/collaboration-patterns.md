# Collaboration Patterns

Common project types and how agents work together.

---

## Pattern 1: New Product/App Development

**Example:** "I want to build a football results app"

### Phase 1: Discovery & Research (Week 1)
```
Paul pitches idea
↓
Viduka: Research competitive landscape
Eddie: Identify target audience and positioning
Bowyer: Explore visual direction options
Batty: Evaluate tech stack options
Lucas: Flag any budget constraints
↓
Don: Summarise findings, confirm direction with Paul
```

### Phase 2: Planning (Week 2)
```
Don: Create project plan with milestones
Batty: Define technical architecture
Bowyer: Lock visual direction and design system
Eddie: Define go-to-market strategy
Yeboah: Identify partnership opportunities
Lucas: Finalise budget allocation
↓
Don: Confirm plan with Paul, assign owners
```

### Phase 3: Build (Weeks 3-X)
```
Batty: Develop the app (updates in group channel)
Bowyer: Create UI designs and assets
Eddie: Prepare marketing materials
Viduka: Test and provide feedback
↓
Don: Track progress, unblock issues, report to Paul
```

### Phase 4: Launch
```
Eddie: Execute marketing launch
Bowyer: Finalise launch assets
Batty: Ensure technical readiness
Yeboah: Activate partnerships
Lucas: Monitor costs
↓
Don: Coordinate launch, report results to Paul
```

### Agent Handoffs
- Viduka → Eddie: Research findings inform marketing positioning
- Bowyer → Batty: Design handoff with specs
- Batty → Viduka: Build ready for testing
- Everyone → Don: Progress reports

## Pattern 2: Feature Addition to Existing Product

**Example:** "Let's add live scores to the football app"

### Flow
```
Paul pitches feature
↓
Viduka: Research similar features in competitors
Batty: Assess technical complexity
Eddie: Consider user perception and messaging
Bowyer: Visual design implications
Lucas: Cost/benefit analysis
↓
Don: Collate input, confirm priority with Paul
↓
Batty: Build feature
Viduka: Test and validate
Eddie: Update marketing if needed
↓
Don: Confirm completion, update Paul
```

## Pattern 3: Marketing Campaign

**Example:** "Let's run a launch campaign for the new app"

### Flow
```
Paul/Eddie initiates campaign
↓
Eddie: Define campaign strategy and channels
Bowyer: Create campaign assets (graphics, video)
Lucas: Set budget and track spend
Viduka: Research audience and competitors
Batty: Set up tracking and analytics
↓
Eddie: Execute campaign
Bowyer: Iterate assets based on performance
Lucas: Monitor ROI
↓
Eddie: Report results to Don
Don: Summarise for Paul
```

## Pattern 4: Partnership/Deal

**Example:** "Let's partner with a football data provider"

### Flow
```
Yeboah identifies opportunity
↓
Viduka: Research the partner
Batty: Assess technical integration needs
Lucas: Model commercial terms
Eddie: Consider co-marketing opportunities
↓
Yeboah: Lead negotiation
Don: Approve deal terms
↓
Yeboah: Finalise agreement
Batty: Implement integration (if needed)
Eddie: Coordinate joint marketing
↓
Yeboah: Report results to Don
```

---

## Pattern 5: Technical Investigation

**Example:** "Should we switch from Supabase to Firebase?"

### Flow
```
Batty raises question (or Paul asks)
↓
Viduka: Research both options, market trends
Batty: Technical comparison, proof of concept
Lucas: Cost comparison
Eddie: Consider impact on users
↓
Batty: Present recommendation with tradeoffs
Don: Make the call (or escalate to Paul)
↓
Batty: Execute migration if approved
```

---

## Pattern 6: Crisis/Issue Response

**Example:** "The app is down"

### Flow
```
Issue detected (Batty, Viduka, or Paul reports)
↓
Batty: Investigate and fix (owns technical response)
Eddie: Pause marketing if needed
Lucas: Assess financial impact
Viduka: Monitor user feedback
↓
Don: Coordinate response, communicate to Paul
↓
Batty: Report root cause and prevention
Don: Update Paul with full summary
```

## Communication Timing

### Immediate Response
- Crisis/issues (Batty leads)
- Blocking questions
- Time-sensitive decisions

### Same-Day Response
- Progress updates
- Review requests
- Handoffs

### Within 24 Hours
- Research findings
- Planning input
- Non-urgent feedback

## Handoff Checklist

When handing work to another agent:

- [ ] Summarise what's done
- [ ] Highlight key decisions made
- [ ] Note any constraints or requirements
- [ ] Tag the next agent clearly
- [ ] State exactly what you need from them
- [ ] Provide relevant context/links/files

---

## Review Checklist

When reviewing another agent's work:

- [ ] Check for domain overlap (is this your area?)
- [ ] Identify strengths (what's good)
- [ ] Spot issues or risks (what could go wrong)
- [ ] Suggest improvements (optional)
- [ ] Give clear verdict (approve / needs revision / discuss)
