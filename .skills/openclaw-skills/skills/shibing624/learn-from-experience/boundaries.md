# Security Boundaries 安全边界

## Never Store 绝不存储

| Category | Examples | Why |
|----------|----------|-----|
|身份凭证|密码、API 密钥、令牌、SSH 密钥|安全泄露风险|
|金融信息|银行卡号、银行账户、加密货币助记词|欺诈风险|
|医疗信息|诊断结果、用药方案、健康状况|隐私保护、HIPAA合规|
|生物识别信息|语音声纹、行为特征指纹|身份盗用风险|
|第三方信息|关于其他人员的信息|未获得第三方知情同意|
|位置轨迹|家庭/工作地址、日常作息规律|人身安全|
|权限特征|用户可访问的系统范围|权限提升风险|

## Store with Caution 谨慎存储

| Category | Rules |
|----------|-------|
|工作上下文|项目结束后自动清除，禁止跨项目共享|
|情绪状态|仅存储用户明确分享的内容，禁止自行推断|
|人际关系|仅存储角色身份（如「经理」「客户」），不保留个人细节|
|日程安排|可存储大致规律（如「上午忙碌」），不存储具体时间点|

## Transparency Requirements 透明度要求

1. **Audit on demand** — 用户询问 "你存储了哪些关于我的信息？?" → 必须完整输出
2. **Source tracking** — 每一条存储信息都必须标记获取的时间与方式
3. **Explain actions** — "格式为「我执行X操作，是因为你在[date]提出了Y要求」"
4. **No hidden state** — 任何会影响行为决策的信息，都必须对外可见
5. **Deletion verification** — 确认目标信息已移除，展示更新后的数据

## Red Flags to Catch 需要警惕的危险信号

如果你发现自己出现以下任何行为，立刻停止：

- 仅出于「日后可能有用」的目的存储信息
- 从非敏感数据中推断敏感信息
- 用户要求删除后仍保留数据
- 将个人上下文套用至工作场景，或将工作上下文套用至个人场景
- 总结能让用户更快依从的规律
- 构建用户心理画像
- 留存第三方信息

## Kill Switch  

当用户要求「忘记一切」时：
1. 将当前所有记忆导出为文件（方便用户复查）
2. 彻底清除所有习得数据
3. 确认提示：「已清空记忆，重新启动。」
4. 不得在行为中保留任何「隐形残留模式」


## Consent Model 授权规则

|数据类型|授权要求|
|-----------|---------------|
|明确更正内容|更正行为本身即代表隐含授权|
|推断得出的偏好|观察3次后需主动询问用户确认|
|上下文/项目数据|首次检测到需要存储时主动询问|
|跨会话行为规律|需要用户明确选择同意后才可存储|
