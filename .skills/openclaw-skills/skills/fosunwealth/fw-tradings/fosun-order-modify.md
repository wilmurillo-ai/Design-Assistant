# 复星撤单与改单工具集

通过命令行脚本完成撤单和改单操作。

> **通用环境约束、市场定义、订单类型和交易限制请参考 `fosun-trading` skill。**
> **下单和订单查询请参考 `fosun-orders` skill。**
> 所有脚本位于 `fosun-trading` skill 的 `code/` 目录下（即 `fw-tradings/fosun-trading/code/`）。

运行前先 `cd` 到脚本目录：

```bash
cd <fosun-trading skill 目录>/code
$FOSUN_PYTHON <脚本名>.py <子命令> [参数]
```

---

## 1. 撤单 — cancel_order.py

> **⚠️ 重要：撤单前必须与用户二次确认！**
> 撤单同样是不可逆操作，执行前必须向用户展示待撤订单的详情（订单号、股票、方向、数量等），等待用户明确确认后才能执行。
>
> **⚠️ 重要：只有当用户明确要求"撤单 / 取消订单"时，才能使用 `cancel_order.py`。**
> 如果用户表达的是"改单 / 改价 / 改量 / 改触发价 / 修改委托参数"，默认动作必须是走后文 `modify_order.py` 的改单流程，**禁止擅自把"改单"理解成"先撤单，再重新下单"**。

```bash
$FOSUN_PYTHON cancel_order.py <ORDER_ID>
```

支持港/美/A 股所有市场的订单撤销。

---

## 2. 改单 — modify_order.py

> **⚠️ 重要：改单前必须与用户二次确认！**
> 改单会改变真实委托参数，执行前必须先把原订单关键信息和修改后的新参数一并展示给用户，等待明确确认后再执行。
>
> **⚠️ 关键：`--quantity` 是改单的必传参数！**
> 无论用户是否要求修改数量，调用 `modify_order.py` 时 **必须始终传入 `--quantity` 参数**。
> - 如果用户明确要求改数量 → 使用用户指定的新数量。
> - 如果用户 **没有** 要求改数量（例如只改价格、触发价等）→ 你必须先通过 `list_orders.py` 查询原订单的数量，然后把 **原订单数量** 原样传入 `--quantity`。
> - **绝对不能省略 `--quantity`**，否则改单接口会失败。
>
> **⚠️ 特别强调：改单时不能更改订单类型。**
> `modify_order.py` 仅应用于修改原订单允许调整的委托参数；**订单类型必须与原订单完全一致，绝对不能借"改单"把限价单改成市价单，或把一种订单类型改成另一种订单类型。**
> 除了用户明确要求修改的字段外，**其它所有参数都必须严格沿用原订单参数**；禁止在改单时擅自变更价格、数量、触发价、订单类型、有效期、产品类型或任何未被用户点名要改的字段。
>
> **⚠️ 重要：用户说"改单 / 改价 / 改量 / 改触发价 / 修改订单"时，默认必须优先使用 `modify_order.py`。**
> 只要目标是修改已有订单参数，就应视为"改单"，**不得自动改走 `cancel_order.py` + `place_order.py` 的"撤单重下"流程**。
>
> **⚠️ 禁止自动降级：如果 `modify_order.py` 执行失败、接口返回不支持、或当前订单状态不允许改单，也必须先停止并告知用户原因。**
> 这时只能向用户解释"直接改单不可行，如要继续只能撤单后重下"，并明确说明风险（如失去原排队顺序、价格波动导致成交条件变化、未成交部分处理差异等）；**只有在用户再次明确授权"撤单后重下"后，才可以执行撤单和重新下单。**

### 改单决策规则（必须遵守）

1. 用户明确说"撤单 / 取消订单" → 使用 `cancel_order.py`
2. 用户明确说"改单 / 改价 / 改量 / 改触发价 / 修改订单" → 使用 `modify_order.py`
3. `modify_order.py` 失败或接口不支持 → 停止自动执行，向用户解释原因并询问是否改为"撤单后重下"
4. 未经用户明确授权，**禁止**把"改单"转换为"撤单后重下"
5. 用户只指定修改部分字段时，**只允许修改这些字段**；未明确要求修改的字段必须保持与原订单一致
6. **订单类型永远不得在改单中被修改**；如果用户想变更订单类型，只能先说明"直接改单不允许更改订单类型"，再由用户明确决定是否改为"撤单后重下"

### 改单推荐流程

1. 先确认目标订单及原参数（必要时用 `list_orders.py` 查看待处理订单）——**特别注意：必须获取原订单的数量，以备 `--quantity` 必传参数使用**
2. 明确圈定"本次要修改的字段"和"其余保持不变的字段"，不得自行补改用户未提到的参数；其中**订单类型必须明确标记为保持原值，不得改动**
3. 向用户并排展示"原订单参数"与"拟修改后的新参数"，并显式说明未点名字段将保持原值不变
4. 获得用户明确确认后，执行 `modify_order.py`——**确保 `--quantity` 始终被传入（用户指定的新值或原订单数量）**
5. 执行后反馈改单结果
6. 如果改单失败，只报告失败原因并询问下一步；**不要自动撤单，不要自动重下**

### 普通改单

```bash
# 只改价格时，--quantity 仍必须传入（使用原订单数量）
$FOSUN_PYTHON modify_order.py --order-id 1234567890 --modify-type 1 --quantity 100 --price 351.000
# 同时改数量和价格
$FOSUN_PYTHON modify_order.py --order-id 1234567890 --modify-type 1 --quantity 200 --price 351.000
```

### 条件单/高级改单

```bash
$FOSUN_PYTHON modify_order.py --order-id 1234567890 --modify-type 2 --trig-price 350.000
$FOSUN_PYTHON modify_order.py --order-id 1234567890 --modify-type 2 --tail-type 1 --tail-pct 2.5
```

### 常用参数

| 参数 | 必填 | 说明 |
|------|------|------|
| `--order-id` | 是 | 订单 ID |
| `--modify-type` | 是 | `1`=普通订单参数修改，`2`=条件单参数修改 |
| `--quantity` | **是** | 修改后的数量。**必传**：即使用户只改价格等其他参数，也必须传入此字段（沿用原订单数量） |
| `--price` | 否 | 修改后的价格 |
| `--trig-price` | 否 | 触发价 |
| `--tail-type` / `--tail-amount` / `--tail-pct` | 否 | 跟踪止损相关参数 |
| `--spread` | 否 | 价差 |
| `--profit-trig-price` / `--profit-quantity` | 否 | 止盈参数 |
| `--stop-loss-trig-price` / `--stop-loss-quantity` | 否 | 止损参数 |
| `--product-type` | 否 | 产品类型，如 `15`=期权 |
