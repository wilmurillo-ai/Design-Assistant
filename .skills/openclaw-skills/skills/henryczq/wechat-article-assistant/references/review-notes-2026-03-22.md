# WeChat Article Assistant Review Notes (2026-03-22)

## 总体结论

当前 Skill 已经具备可用的产品雏形：

- 本地 CLI 主入口明确
- SQLite 持久化结构完整
- 登录、公众号管理、同步、文章详情抓取主链路已经打通
- 已摆脱对旧 Web 服务的依赖

目前更像是 **“可用的一版”**，下一步重点不是扩功能，而是做 **接口稳定性、同步健壮性、文档对齐、工程化收口**。

---

## 一、核心评价

### 1. 优点

#### 架构层面
- 模块划分基本合理：`cli / mp_client / service / db`
- 单机 Skill 场景下，SQLite + requests 的复杂度控制得合适
- 没有过度设计，落地性强

#### 能力层面
- 支持扫码登录、导入登录态、登录校验
- 支持公众号搜索、添加、删除、同步配置
- 支持文章列表同步与单篇正文抓取
- 支持图片下载、本地 Markdown 导出、正文缓存
- 支持通过 Inbound Context 自动发二维码/通知

#### OpenClaw 适配
- Skill 文档已经从“服务型接口”转向“本地命令型接口”
- 登录场景考虑了 `channel / target / account`
- `sync-due` 适合作为外部调度入口

### 2. 当前短板

#### 稳定性
- 部分接口返回结构不一致
- 增量同步策略偏乐观
- 到点同步触发条件太硬

#### 文档一致性
- 环境变量存在表述漂移
- 个别行为文档未完全解释清楚

#### 工程化
- 多表写操作缺少事务保护
- SQL 逐渐散落
- 测试主要还是 smoke test，回归保障不够

---

## 二、问题清单（按优先级）

### P1-1 接口稳定性：`article-detail` 缓存命中与实时抓取返回结构不一致

#### 现象
`article_detail()` 在缓存命中时返回字段较少；首次抓取时返回字段更完整。

#### 影响
调用方需要分支兼容两套结构，增加接入复杂度，也容易埋 bug。

#### 建议
统一返回 payload 结构，至少保证以下字段始终存在：

- `article_id`
- `fakeid`
- `aid`
- `title`
- `author_name`
- `account_name`
- `digest`
- `link`
- `create_time`
- `create_time_formatted`
- `markdown_content`
- `html_content`
- `text_content`
- `images`
- `saved_json_path`
- `saved_md_path`
- `saved_html_path`
- `cached`

#### 优先级理由
这是典型接口契约问题，收益高、改动不大，适合优先收口。

### P1-2 文档/实现对齐：环境变量命名存在漂移

#### 现象
`SKILL.md` 中提到：
- `WECHAT_ARTICLE_ASSISTANT_HOME`
- `WECHAT_ARTICLE_OPENCLAW_HOME`

而接口文档中主要强调 `WECHAT_ARTICLE_OPENCLAW_HOME`。

#### 影响
使用者可能不知道到底哪套变量真正生效，导致排障成本上升。

#### 建议
明确并统一策略：

- 如果代码实际兼容两套，就在文档中写清优先级
- 如果实际只保留一套，就把全部文档统一到一套变量名

#### 优先级理由
这是使用门槛问题，尤其会影响自动化部署和排障。

### P1-3 数据一致性：多表写操作未使用事务

#### 现象
以下逻辑存在多次连续写库，但未显式包事务：

- `delete_account`
- `add_account`
- 登录成功后写入登录态和二维码状态
- `article_detail` 落地 detail + article
- `sync_account`

#### 影响
一旦中途失败，SQLite 可能留下半完成状态，后续排查会很痛苦。

#### 建议
为这些关键写路径增加事务保护，确保：
- 要么全部成功
- 要么整体回滚

#### 优先级理由
这是数据层面的可靠性问题，越早补越省事。

### P1-4 调度健壮性：`sync_due` 只支持“当前分钟精确命中”

#### 现象
当前 `sync_due` 的判断是：
- 小时完全相等
- 分钟完全相等
- 同一分钟内防重复

#### 影响
如果外部 cron 晚执行 1 分钟，这次同步就直接错过。

#### 建议
改成“补跑窗口”策略，例如：
- 允许过去 2~5 分钟内本应执行但未执行的任务继续补跑
- 仍保留防重复逻辑

#### 优先级理由
这是调度场景里非常常见的真实问题，改完后稳定性会明显提升。

### P1-5 同步策略：`sync_account` 的增量判断偏乐观

#### 现象
当前同步逻辑在遇到“已存在文章”后就倾向停止继续抓取。

#### 风险
如果远端列表存在：
- 置顶
- 重发
- 编辑顺序变化
- 返回页不稳定

就可能漏掉新文章。

#### 建议
优化为更稳的策略，例如：

- 默认多抓 1~2 页冗余
- 或者按“最近 N 页 + upsert 去重”处理
- 允许通过参数控制 `max_pages`

#### 优先级理由
同步的正确性是 Skill 的核心价值之一，不能太依赖理想排序。

## 三、中优先级问题

### P2-1 `list-account-articles --remote` 的 `count` 语义不够清晰

#### 现象
微信返回的是 publish 列表，展开后实际文章数可能多于 `count`，代码又会再切片。

#### 影响
调用者难以理解 `count` 是：
- 抓取页大小
- 还是最终返回条数

#### 建议
文档里明确说明当前语义，或者拆成两个参数：
- `fetch_count`
- `return_count`

### P2-2 缓存策略说明不足

#### 现象
`article-detail` 有缓存，但文档没充分说明缓存有效期和刷新规则。

#### 建议
在接口文档中补充：
- 默认永久缓存
- `--force-refresh true` 才重新抓
- 图片文件若被删除会如何处理
- 文章更新不会自动刷新

### P2-3 单账号模型需要更明确表达

#### 现象
当前数据库设计本质上是“单登录态单例”，但这件事在设计文档里可以再说得更明确。

#### 建议
在 design / interface 文档里明确声明：
- 当前不支持多公众号平台账号并行
- 所有同步与抓取默认基于当前唯一登录态

### P2-4 消息发送耦合方式偏重

#### 现象
`openclaw_messaging.py` 通过 subprocess 调 `openclaw message send`。

#### 影响
虽然能跑，但：
- 对 CLI 可执行环境有依赖
- 错误语义较粗
- 后续参数变化时容易脆

#### 建议
后续可考虑：
- 让 Skill 返回消息载荷，由上层统一发送
- 或者抽更稳定的消息适配层

#### 说明
这不是当前最急的问题，但值得记一笔。

## 四、低优先级但值得持续优化

### P3-1 `cli.py` 逐渐变胖
建议后续拆出更清晰的 command/facade 层。

### P3-2 SQL 分散在 service 中
后续可考虑引入轻量 repository 层，而不是 ORM。

### P3-3 异常分类还可更细
比如代理错误、解析错误、限流错误进一步细分。

### P3-4 Markdown 导出质量仍有打磨空间
公众号复杂版式转 Markdown 还有提升余地。

### P3-5 缺少关键单元测试
建议优先补：
- URL 归一化
- article identity 解析
- HTML 正文解析
- 账号精确匹配
- 缓存命中结构一致性

---

## 五、推荐的修复顺序

### 第一阶段：先补“接口与稳定性”
1. 统一 `article-detail` 返回结构
2. 环境变量文档与实现完全对齐
3. 给关键多表写操作加事务
4. `sync_due` 改为支持补跑窗口
5. 同步策略增加冗余页/更稳的截断逻辑

### 第二阶段：补“可维护性”
6. 明确 `count` 语义
7. 文档补缓存策略
8. 单账号模型写清楚
9. 补核心解析测试

### 第三阶段：做“工程化收尾”
10. command/facade 拆分
11. repository 收敛 SQL
12. 优化消息发送适配
13. 提升 Markdown 导出质量

---

## 六、最终判断

如果只问一句话评价：

**这不是一个玩具 Skill 了，已经是个有实用价值的本地工具链；但现在最需要的不是继续加功能，而是把边界收紧、把契约定稳。**
