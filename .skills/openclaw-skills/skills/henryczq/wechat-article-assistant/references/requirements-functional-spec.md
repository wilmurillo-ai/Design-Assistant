## 1. 文档目的

本文档用于说明 `wechat-article-assistant` Skill 的产品需求、能力边界、详细功能、兼容范围、已知限制与后续优化方向。

它面向以下读者：

- Skill 使用者：了解当前能做什么、怎么接入 OpenClaw。
- 维护开发者：了解当前实现目标、数据结构、限制条件和演进方向。
- 后续协作人员：作为继续扩展代理、导出、测试体系、模块拆分时的统一基线。

## 2. 项目背景

原有方案存在两层结构：

- 一个独立部署的 Web 服务/Nuxt/Nitro 服务，负责登录、代理转发、数据落库、文章抓取等。
- 一个外层 Skill/CLI，通过调用 `localhost:3010` 提供的 HTTP API 来完成公众号搜索、文章同步、文章详情获取等能力。

该方案存在以下问题：

- 依赖单独部署的 Web 服务，使用成本和维护成本较高。
- Skill 与实际业务逻辑分离，排障时需要同时检查 Web 服务与 Skill。
- OpenClaw 使用时不够“原生”，无法完全以一个本地 Skill 的方式独立运行。
- 登录态、代理、调试信息、测试脚本分散，不利于迁移、联调和复用。

因此，新方案目标是将这些能力统一到本地 Python Skill 中，不再依赖独立 Web 服务。

## 3. 目标与非目标

### 3.1 目标

- 将旧 Web 端对外暴露的核心能力迁移为本地 Python Skill。
- 统一以 OpenClaw/Trae Skill 的方式调用，不要求常驻 Web 服务。
- 以 SQLite 作为本地持久化存储。
- 支持公众号平台登录、登录态保存与有效性检查。
- 支持代理配置，并允许同步与文章详情分别开关。
- 支持公众号搜索、添加、删除、同步配置管理。
- 支持远端文章清单查询、本地文章清单查询、文章详情抓取与导出。
- 支持外部调度入口，由 OpenClaw 定时任务或系统计划任务调用。
- 支持调试日志与独立 smoke test 脚本，便于联调与问题定位。

### 3.2 非目标

- 不实现一个新的 Web UI 或管理后台。
- 不内置常驻 scheduler 进程。
- 当前不做多公众号平台账号并行管理。
- 当前不做复杂权限系统、加密密钥托管或远程多用户共享。
- 当前不覆盖旧 Web 后台中的所有“管理台内部接口”，尤其是评论、相册、worker 统计类接口。

## 4. 目标用户与典型场景

### 4.1 目标用户

- 在 OpenClaw 中通过 Skill 管理公众号文章的自动化使用者。
- 需要定期同步公众号文章的运营/内容处理流程维护者。
- 需要将文章详情抓取为 Markdown/JSON 进行后续处理的自动化链路。
- 需要排查登录失效、代理异常、抓取失败的维护开发者。

### 4.2 典型场景

#### 场景 A：首次接入

- 用户导入历史 cookie/token 文件，验证登录态。
- 用户搜索公众号并添加到本地库。
- 用户设置同步时间，后续交给 OpenClaw 定时任务调用 `sync-due`。

#### 场景 B：日常同步

- 定时任务调用 `sync-due`。
- Skill 根据 SQLite 中的同步配置筛选本分钟应执行的公众号。
- Skill 同步远端文章列表，写入本地 SQLite，并记录同步日志。

#### 场景 C：人工查阅某个公众号

- 用户执行远端文章清单查询，快速预览最新文章。
- 用户在本地查询已同步文章。
- 用户按 `aid` 或 `link` 抓取单篇文章详情，导出为 `article.json + article.md`。

#### 场景 D：登录或代理异常排查

- 用户执行 `doctor` 检查登录态是否可用。
- 用户检查代理是否启用、是否应用到当前操作、连通性是否正常。
- 用户通过 `--debug` 或日志文件查看详细请求链路。

## 5. 系统边界与总体方案

### 5.1 总体设计

系统由一个本地 Python CLI 组成，运行方式如下：

- 入口脚本：`python scripts/wechat_article_assistant.py`
- 数据存储：SQLite
- 目录根：优先使用 `WECHAT_ARTICLE_ASSISTANT_HOME`，兼容 `WECHAT_ARTICLE_OPENCLAW_HOME`；默认仍使用 `~/.openclaw/media/wechat-article-openclaw/`
- 日志：可选文件日志与控制台日志
- 对外调用方式：OpenClaw/Trae 调用本地 Skill CLI

### 5.2 核心模块

- `cli.py`
  负责命令参数解析、全局日志开关、命令路由、统一返回格式。
- `login_service.py`
  负责二维码登录、轮询登录状态、导入旧登录态、查询当前登录态、清理登录态。
- `mp_client.py`
  负责访问微信接口、请求重试、限速、代理处理、代理健康检查、登录校验。
- `account_service.py`
  负责公众号搜索、根据关键字添加、根据文章链接反查公众号、管理同步配置。
- `sync_service.py`
  负责单账号同步、批量同步、到点同步、同步日志。
- `article_service.py`
  负责文章清单入库、文章详情抓取、图片本地化、Markdown/JSON 导出。
- `database.py` 与 `schema.py`
  负责 SQLite 初始化与表结构。
- `log_utils.py`
  负责调试日志开关、日志文件与控制台输出。
- `scripts/dev/smoke_test.py`
  负责开发期/联调期的步骤化冒烟测试。

## 7. 详细功能说明

### 7.1 登录管理

#### 7.1.1 二维码登录

功能目标：

- 生成微信公众号平台登录二维码。
- 应先读取 Inbound Context，并将解析出的 `channel`、`target`、`account` 显式传给登录脚本。
- 脚本可基于上述参数通过 OpenClaw 消息发送二维码图片。
- 支持轮询状态或阻塞等待登录完成。
- 登录成功后保存 token 和 cookies 到 SQLite。

关键流程：

1. 上层先读取 Inbound Context，得到 `channel`、`chat_id -> target`、`account_id -> account`。
2. 调用 `login-start` 时显式传入 `--channel`、`--target`、`--account`，并生成二维码图片落地保存。
3. 保存 `login_qrcode_session`，记录 `sid`、二维码路径、临时 cookie、过期时间。
4. `login-poll` 查询扫码状态。
5. 扫码确认后执行登录动作，获取 token、昵称、头像。
6. 保存到 `login_session`，供后续所有同步与搜索操作复用。

状态要求：

- 应能区分“等待扫码”“扫码待确认”“登录成功”“二维码过期”“账号未绑定”等状态。
- 应保留最近一次二维码会话状态，便于排障。

#### 7.1.2 导入已有登录态

功能目标：

- 允许从旧系统导出的 cookie/token 文件直接导入。
- 导入后可选立即校验有效性。

适用场景：

- 本地已有可用 cookie 文件。
- 不希望每次重新扫码登录。

#### 7.1.3 登录态检查与清理

功能目标：

- 支持主动校验当前登录态是否有效。
- 支持清空登录态与二维码会话。

行为要求：

- `login-info --validate true` 应触发真实校验。
- `doctor` 应主动报告“是否有登录会话”和“登录态是否校验通过”。

### 7.2 代理配置

#### 7.2.1 代理持久化

功能目标：

- 代理配置写入 SQLite。
- 允许统一保存代理地址与开关状态。

#### 7.2.2 操作级开关

当前支持两个开关：

- `apply_article_fetch`
  控制文章详情抓取是否使用代理。
- `apply_sync`
  控制远端文章列表抓取和同步是否使用代理。

#### 7.2.3 代理健康检查

功能目标：

- 在 `doctor` 中报告代理是否启用、是否应用到当前操作、是否可连通。
- 当代理配置错误时，能给出足够明确的错误信息，而不是仅显示“请求失败”。

当前实现特点：

- 已能识别“代理已启用但未配置 URL”“当前操作未启用代理”“代理连通性检查失败”等状态。
- 已验证当代理域名不是标准 HTTP CONNECT 代理时，会在 `doctor` 中明确报出 `400 Bad Request` 类型错误。

### 7.3 公众号管理

#### 7.3.1 关键字搜索

功能目标：

- 在登录态有效时，根据关键字搜索公众号候选。
- 返回 fakeid、昵称、别名、头像、服务类型、简介等信息。

#### 7.3.2 本地账号添加

支持三种添加方式：

- 手动指定 `fakeid + nickname`
- 根据关键字自动添加唯一匹配项
- 根据文章链接反查公众号后自动添加

当前设计：

- 添加后同步配置表会自动初始化。
- 可选在添加后立即执行首次同步。

#### 7.3.3 根据文章链接反查公众号

功能目标：

- 输入一篇公众号文章链接，尝试解析出公众号名称。
- 以解析出的公众号名称再次执行微信搜索，定位公众号候选。

适用场景：

- 已知文章链接，但不知道 fakeid 或公众号昵称。

依赖条件：

- 能正常访问文章页面。
- 当微信对文章页返回“环境异常”时，该能力同样依赖可用代理。

#### 7.3.4 本地账号管理

支持：

- 列出本地已维护的公众号
- 删除公众号及其本地文章与详情缓存
- 查看与设置同步配置

### 7.4 文章同步

#### 7.4.1 单账号同步

功能目标：

- 根据公众号 fakeid，从微信后台分页抓取文章列表。
- 自动去重。
- 将新文章写入 SQLite。
- 更新公众号统计信息与最近同步结果。
- 写入同步日志。

当前实现特点：

- 首次同步会持续翻历史页，直到抓完或遇到已存在文章。
- 已修复早期“只抓固定前几页”的问题。
- 已记录 `pages_fetched`，便于评估同步规模与排障。

#### 7.4.2 批量同步与到点同步

支持：

- `sync-all`
  同步所有启用的公众号。
- `sync-due`
  只同步当前分钟命中的公众号。

调度策略：

- Skill 本身不启动常驻调度器。
- 建议由 OpenClaw 内置定时任务或系统计划任务调用 `sync-due`。

#### 7.4.3 同步日志

记录内容包括：

- fakeid
- 公众号昵称
- 状态
- 消息
- 本次新增文章数
- 开始时间与结束时间

### 7.5 文章清单查询

#### 7.5.1 远端文章清单

功能目标：

- 直接查询微信后台文章列表。
- 可选择是否把查询结果写入本地数据库。

当前行为：

- 微信接口返回的是“群发消息页”，单个 publish 记录可能展开为多篇文章。
- 当前实现会额外返回：
  - `fetched_count`
  - `returned_count`
- 已修复 `--count 3` 却直接返回大量展开文章的问题。

#### 7.5.2 本地文章清单

功能目标：

- 查询本地已同步文章。
- 支持按公众号、起始偏移、数量、关键字进行过滤。

#### 7.5.3 最近文章

功能目标：

- 按最近 N 小时查询本地新增文章。
- 用于定期筛选最新内容或做后续流水线处理。

### 7.6 文章详情抓取与导出

#### 7.6.1 详情抓取

支持两种定位方式：

- 按 `aid`
- 按 `link`

处理过程包括：

- 读取本地缓存
- 访问文章页
- 解析标题、作者、公众号名、正文、发布时间
- 将 HTML 转 Markdown
- 可选下载图片并替换为本地路径
- 写入 `article_detail`
- 同步更新 `article` 主表

#### 7.6.2 导出与落盘

默认行为：

- 保存 `article.json`
- 保存 `article.md`
- 默认下载图片
- 默认不额外保存 HTML

可选行为：

- `--include-html true` 时保存 HTML
- `--force-refresh true` 时跳过缓存重新抓取

#### 7.6.3 缓存策略

- 文章详情抓取成功后写入 SQLite。
- 后续再次请求优先读缓存。
- 除非显式强制刷新，否则不重复抓取。

## 9. 非功能需求

### 9.1 稳定性

- 微信接口调用应带重试机制。
- 应具备基本限速，避免频率过高导致被拦。
- 对常见失败类型应返回可读错误。

### 9.2 可观测性

- 提供 `doctor` 命令。
- 提供调试日志开关。
- 提供同步日志表。

### 9.3 可维护性

- 采用服务层分模块设计。
- 将测试脚本与正式 CLI 分离。
- 文档应覆盖需求、功能、接口和数据结构。

### 9.4 安全性

- 当前登录态允许明文存 SQLite。
- `doctor` 已避免直接把完整 cookie 明文回显到聊天结果。
- 不在默认文本输出中泄露不必要的敏感字段。

### 9.5 性能

- SQLite 开启 WAL。
- 大部分命令以单机串行使用为前提。
- 当前未特别针对高并发多 worker 优化。

## 11. 当前已知限制

- 文章详情抓取和基于文章页的公众号反查，仍可能被微信返回“环境异常”页面拦截。
- 当前只支持单个公众号平台登录账号。
- 当前不内置常驻调度器。
- 当前只支持标准 HTTP/HTTPS 代理模式。
- 提供的 `wechat.zzgzai.online`、`wechat2.zzgzai.online` 在当前测试中不是可直接使用的标准 CONNECT 代理入口，健康检查会返回 `400 Bad Request`。
- 当前没有单独的 `proxy-clear` 命令，只能通过 `proxy-set --enabled false` 禁用代理。

## 12. 优化建议与后续路线

### 12.1 高优先级

#### 建议一：补充网关代理模式

背景：

- 当前标准代理模式要求目标地址能作为 HTTP/HTTPS 代理使用。
- 测试显示 `wechat.zzgzai.online` 与 `wechat2.zzgzai.online` 并不符合该模式。

建议：

- 在 `proxy_config` 中新增 `proxy_mode` 字段。
- 支持两类模式：
  - `http_proxy`
  - `gateway`
- `gateway` 模式下，由 Skill 将目标请求转发到用户自有服务端，由服务端代抓文章或接口数据。

#### 建议二：补充 article-export 命令

目标：

- 对齐旧 Web `download` 接口能力。
- 支持按 `html / markdown / text / json` 导出。

收益：

- 更适合自动化下游消费。
- 减少调用方自己再转换格式的成本。

### 12.2 中优先级

#### 建议三：补充 proxy-clear 命令

目标：

- 明确区分“禁用代理”和“清空代理配置”。

#### 建议四：细化 smoke test 断言

目标：

- 让 smoke test 不只打印返回结果，也能自动校验关键字段。

#### 建议五：增加文章详情抓取的降级策略

方向：

- 当直接抓取正文失败时，优先返回本地已同步元数据。
- 如果未来接入网关代理，可在失败时自动降级到网关模式。

### 12.3 中长期

#### 建议六：模块目录分层

当前文件大小还可接受，但后续继续增长时建议拆为：

- `scripts/core/`
- `scripts/services/`
- `scripts/dev/`
- `scripts/cli/`

#### 建议七：补充自动化测试层次

推荐至少分三层：

- 单元测试
- 本地 SQLite 集成测试
- 联网 smoke test

## 13. 测试与验收建议

### 13.1 基础验收

- `quick_validate.py` 校验通过。
- `doctor` 能正确返回登录与代理状态。
- `login-import` 能导入并校验旧 cookie。
- `search-account` 能返回公众号候选。
- `add-account-by-keyword` 能添加唯一匹配公众号。
- `sync` 能实际写入文章和同步日志。
- `list-account-articles --remote true --count N` 返回数量应符合请求预期。

### 13.2 详情验收

- 在可用代理条件下，`article-detail` 能成功落地：
  - `article.json`
  - `article.md`
  - 可选 `article.html`
  - 图片目录

### 13.3 代理验收

- `doctor` 应能报告代理是否真正可用。
- 代理异常时，错误信息应明确指出是“代理连接失败”而非笼统请求失败。

### 13.4 调试验收

- `--debug` 能在控制台输出调试信息。
- `--debug` 或日志文件开关开启后，日志文件应写入 `logs/wechat_article_openclaw.log`。
