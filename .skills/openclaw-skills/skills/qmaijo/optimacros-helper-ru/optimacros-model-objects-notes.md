## 1. Короткая карта модели

Если упростить, модель в Optimacros обычно выглядит так:

- **Справочники** — описывают сущности предметной области: товары, клиенты, проекты, ЦФО, каналы, склады и т. п.
- **Измерение времени** — системная ось времени: годы, кварталы, месяцы, недели, дни, периоды.
- **Версии** — сценарность: факт, план, прогноз, прошлый год, оптимистичный прогноз и т. д.
- **Мультикуб** — контейнер данных, где значения живут на пересечении измерений.
- **Кубы внутри мультикуба** — отдельные показатели/поля: выручка, себестоимость, маржа, флаг блокировки, комментарий и т. д.
- **Выборки справочников / выборки кубов / выборки версий** — подмножества для отчётов, интерфейсов и ограничения контекста.
- **Форматы данных** — определяют, что именно хранит куб или свойство: число, текст, дата, логический, измерение, «нет данных».
- **Свойства** — дополнительные атрибуты элементов справочников, времени, версий и др.

Практически:
- **справочник отвечает на вопрос “кто/что?”**
- **время отвечает на вопрос “когда?”**
- **версия отвечает на вопрос “в каком сценарии?”**
- **мультикуб отвечает на вопрос “где лежат значения?”**
- **куб отвечает на вопрос “какой именно показатель считаем/храним?”**

## 2. Справочники

### Что это
Справочник — структура данных, где информация хранится в виде списка элементов. Справочники образуют каркас модели.

Примеры:
- продукты
- сотрудники
- статьи P&L
- проекты
- клиенты
- регионы

### Важная идея
Справочник — это не просто список для интерфейса. Это **измерение модели**, по которому потом строятся кубы, отчёты, фильтры и правила расчётов.

### Системные и пользовательские
- **Системные справочники** — встроены платформой.
- **Пользовательские справочники** — создаются моделлером под задачу.

### Иерархия
Справочники могут быть:
- иерархическими
- плоскими
- композитными
- нумерованными

#### Иерархические
У элементов есть родитель/дети. Подходят для организационных структур, товарных категорий, P&L, географии.

#### Плоские
Без иерархии. Нужны для транзакционных или простых списков, где нет уровней.

#### Нумерованные
Вариант плоского справочника: отображаемые наименования могут повторяться, но каждый элемент имеет уникальный идентификатор.

Это полезно, когда:
- названия могут дублироваться
- нужен технический ID
- важна загрузка/интеграция по ключу

**Важно:** отображаемое имя может повторяться, но **имя элемента внутри справочника должно быть уникальным**.

#### Композитные
Подходят, когда иерархия собрана из нескольких справочников/уровней и каждый уровень по сути плоский.

Из примера в документации:
- `Geo A1` → верхний уровень
- `Regions A2` → следующий
- `Territories A3` → нижний

Хороший сценарий — сбалансированная иерархия, где каждый уровень понятен и стабилен.

Ограничение из документации: у **родительского справочника композита не может быть выборка (сабсет)**.

### Базовые поля элемента справочника
По скринам и описаниям типично видны:
- имя элемента
- код
- справочник
- родитель
- папка
- комментарии
- флаги `Нумерованный`, `Плоский`
- отображаемое имя

### Отображаемое имя
Можно настроить, какое свойство показывается пользователю вместо технического имени элемента.

Это удобно, когда:
- нужен внутренний ID, а пользователю — человекочитаемое название
- один и тот же элемент должен по-разному отображаться в разных задачах

### Свойства справочников
У справочника могут быть именованные свойства — по сути дополнительные поля у каждого элемента.

Например у товара:
- бренд
- категория
- регион
- единица измерения
- активен/не активен

Ключевая мысль: свойства справочника часто используются как:
- атрибуты для фильтрации
- вход в формулы
- источник отображаемых имён
- маппинг для связей между измерениями

### Выборки элементов справочника
Выборка — это подмножество элементов справочника.

Зачем нужна:
- сократить объём отображения
- собрать отдельный рабочий срез
- ограничить интерфейс и ввод
- переиспользовать одни и те же элементы в разных представлениях

Пример:
- выборка `Фрукты`
- выборка `Активные SKU`
- выборка `Только регионы продаж`

## 3. Измерение времени

### Что это
Время в Optimacros — системное измерение с отдельной настройкой и собственными справочниками:
- годы
- полугодия
- кварталы
- месяцы
- периоды
- недели
- дни

### Что важно для модели
Время — это не просто список дат, а **стандартная ось планирования и анализа**.
От выбора календаря зависит:
- какие элементы времени доступны
- какой базовый временной справочник попадёт в мультикуб автоматически
- как будут работать YTD / YTG / кварталы / полугодия / итоги по периоду

### Свойства времени
Для времени можно создавать свойства, как и для обычных справочников.

Например:
- количество дней в месяце
- смещение периода
- пользовательский порядок
- флаги принадлежности к специальным группам

Из документации видно, что свойства времени поддерживают:
- формулу
- формат
- комментарии
- код
- способ подведения итогов (`None`, `Sum`, `Average`, `Input`)

### Custom Order
Позволяет переопределить порядок отображения элементов времени.

Полезно, когда:
- нужен нестандартный порядок периодов
- пользователь хочет поднять отдельные периоды выше/ниже системного порядка

### Тип календаря
Для моделей по месяцам обычно используется `Calendar Months`.

Настройки времени позволяют задать:
- месяц начала финансового года
- текущий год
- текущий месяц
- количество прошлых лет
- количество будущих лет
- показ YTD, YTG, квартальных, полугодовых и общих итогов

### Практический вывод
До проектирования мультикубов стоит заранее определить:
- календарь модели
- нужен ли финансовый год, отличный от календарного
- нужен ли YTD/YTG
- нужен ли помесячный, недельный или дневной уровень

Иначе потом придется переделывать ось времени в отчётах и формулах.

## 4. Версии

### Что это
Версии — системное измерение сценарности.

По умолчанию модель создаётся как минимум с:
- `Actual`
- `Forecast`

Но можно добавлять и свои версии:
- План
- Прогноз оптимистичный
- Прогноз пессимистичный
- Прошлый год

### SwitchOver (дата начала прогнозного периода)
Это одна из ключевых настроек версий.

Смысл:
- до определённой даты прогнозная версия может брать значения из факта
- после этой даты версия становится самостоятельной

То есть версия `Forecast` может частично жить на факте, а дальше — на собственных значениях.

Это удобно для rolling forecast и сценариев планирования.

### Выборки версий
При создании мультикуба вместо полного измерения `Версии` можно использовать **выборку версий**.

Это полезно, когда:
- в отчёте нужны не все версии
- хочется ограничить сценарии в конкретном мультикубе
- нужно упростить структуру пользовательского интерфейса

**Важно:** документация предупреждает, что использование/переключение выборок версий может заметно увеличивать размер модели в памяти. Это надо учитывать при проектировании.

## 5. Мультикубы

### Что это
Куб — массив данных, записанных в едином формате.
Мультикуб — контейнер для данных, состоящий из одного или более кубов с одинаковыми измерениями.

Простое понимание:
- **мультикуб = таблица/контейнер показателей на общей размерности**
- **кубы = строки/метрики/поля внутри этого контейнера**

### Что может быть измерением мультикуба
При создании мультикуба можно использовать:
- измерения времени
- версии
- пользовательские справочники
- выборки кубов
- выборки версий
- иногда специальные системные измерения

### Автоматическое добавление времени
При создании нового мультикуба базовое временное измерение часто подставляется автоматически. Какое именно — зависит от настроек календаря модели.

### Когда нужен отдельный мультикуб
Обычно новый мультикуб создают, если:
- набор измерений отличается от существующих
- другая гранулярность данных
- нужны отдельные правила доступа/блокировки/редактирования
- это самостоятельная область расчёта

Хорошая практика: не лепить в один мультикуб всё подряд. Если метрики живут на разной зернистости, лучше разделять.

### Именование и размещение
На вкладке расширенных настроек можно:
- задать имя мультикуба
- положить его в папку
- настроить порядок в списке

Имя должно быть уникальным в модели.

## 6. Кубы внутри мультикуба

### Что такое куб внутри мультикуба
Это отдельное поле/показатель, которое хранится на общей размерности мультикуба.

Примеры кубов:
- `Выручка`
- `Себестоимость`
- `Прибыль`
- `GM%`
- `Комментарий`
- `Блокировка ввода`
- `Текущий регион`

### Что у куба есть кроме имени
По документации и экранным формам у кубов есть параметры вроде:
- формула
- формат
- комментарии
- идентификатор
- автопересчёт
- последнее обновление
- формула в стабильном представлении
- формула в нормализованном представлении
- комментарий к ошибке
- стили
- условия редактирования
- выборка времени для блокирования

### Omit dimensions
Параметр позволяет исключить измерение из куба.

Смысл:
- мультикуб общий, но конкретному кубу не нужна часть измерений
- визуально и логически куб работает как более «узкий»

Это полезно, когда внутри одного мультикуба:
- часть показателей должна быть по SKU × месяц × регион
- а часть — только по SKU × месяц

Но лучше применять осознанно: если отличия по размерности слишком велики, иногда правильнее вынести показатель в отдельный мультикуб.

### Условия редактирования
Можно запретить ввод/редактирование/импорт в ячейки куба по значению логического куба.

Практический паттерн:
- есть булев куб `Can Edit?`
- если `TRUE` — редактирование разрешено
- если `FALSE` — ячейка заблокирована

Это фактически прикладной механизм DCA/управления вводом.

### Выборка времени для блокирования
Позволяет заблокировать ввод по выбранным периодам времени.

Например:
- факт за закрытые месяцы нельзя редактировать
- ввод разрешён только для будущих месяцев

### Стили кубов
Можно отдельно управлять:
- форматом данных
- форматированием ячеек
- цветом/выделением
- отображением итогов
- видом сохранения/переноса текста и т. п.

Это уже больше про представление, но для пользовательских рабочих форм бывает критично.

## 7. Форматы данных: что хранит куб или свойство

Формат данных определяет не только отображение, но и **тип значения** и допустимые действия.

### 7.1 Число
Подходит для сумм, объёмов, цен, процентов, коэффициентов.

Когда использовать:
- финансовые метрики
- количественные расчёты
- KPI

### 7.2 Текст
Для комментариев, описаний, произвольных строк.

Из документации видно, что у текста важны настройки:
- перенос строки
- выравнивание
- поведение при изменении размеров ячейки
- режим `Секрет` (маскирование содержимого звёздочками)

`Секрет` полезен для чувствительных значений, но включать его надо осознанно: после выключения функционала куб очищается, чтобы не раскрыть данные.

### 7.3 Логический
Для флагов `TRUE/FALSE`.

Типовые применения:
- признак активности
- флаг выбора
- блокировка ввода
- управляющие условия в формулах

### 7.4 Дата
Для календарных дат.

Нюанс из документации:
- ряд финансовых и календарных функций работает с точностью до **Date**, не `Date-Time`
- при необходимости надо следить, чтобы значения не были скрыто `Date-Time`

### 7.5 Измерение
Формат ссылки на элемент измерения.

Можно выбрать тип значения:
- справочник
- версия
- период времени
- выборка кубов

Это один из самых важных форматов для моделирования, потому что он позволяет хранить **ссылку на другой элемент модели**, а не просто текст.

Типовые кейсы:
- у товара хранить ссылку на регион
- у проекта хранить ссылку на менеджера
- в кубе выбора хранить выбранную версию/период/элемент справочника

Дополнительные режимы, которые видны в документации:
- иерархичный
- булеан-куб
- зависимый контекст
- зависимый куб

То есть формат измерения может не просто показывать список значений, а ограничивать допустимый выбор на основе другого контекста.

### 7.6 Нет данных
Специальный пустой формат.

Используется как:
- визуальный разделитель
- технический куб-разделитель
- пустая прослойка в рабочих формах

Важный плюс: **кубы формата “Нет данных” не занимают оперативную память**.
Обычно такие кубы называют вроде `TECH`.

## 8. Зависимый выбор и связанные сущности

Это важный практический блок, который напрямую связан с моделированием пользовательского ввода.

### Что имеется в виду
В формате `Измерение` можно организовать зависимый выбор значений:
- список доступных элементов зависит от выбора в другой ячейке
- выбор ограничивается родительским справочником
- выбор зависит от фильтра-куба
- выбор зависит от текущего периода / версии / другого измерения

По сути это механизм **контекстного списка значений**.

### Типовой пример
Есть:
- продукт
- месяц
- регион

Для конкретного продукта и месяца можно разрешить выбор только тех регионов, которые доступны по фильтру.

Это сильно помогает в интерфейсах ввода, где нужно:
- исключить ошибочный выбор
- упростить пользователю список
- жёстко направить ввод в допустимые значения

### Что это значит для проектировщика
Когда видишь в ТЗ фразы вроде:
- «выпадающий список должен зависеть от выбранного клиента»
- «регион нужно ограничить товаром и месяцем»
- «менеджер выбирается только в рамках подразделения»

— это почти всегда история про:
- формат `Измерение`
- зависимый куб / зависимый контекст
- фильтр-куб
- свойства/маппинг справочников

## 9. Позиции итогов и представление

Для измерений внутри мультикуба можно управлять положением итогов в иерархии и тем, как таблица выглядит для пользователя.

Это не меняет саму логику данных, но влияет на:
- читаемость отчёта
- поведение ввода
- удобство анализа

Полезно разделять в голове:
- **структура модели**
- **формулы и расчёт**
- **представление**

Одни и те же данные можно плохо или хорошо показать без изменения расчётов.

## 10. Как всё это связано между собой

Ниже — практическая схема мышления.

### Уровень 1. Каркас
Сначала создаётся каркас модели:
- справочники
- время
- версии

### Уровень 2. Атрибуты и связи
Потом настраиваются:
- свойства справочников
- выборки
- отображаемые имена
- связи через формат `Измерение`
- зависимые выборы

### Уровень 3. Данные и расчёты
Дальше строятся мультикубы:
- выбираются измерения
- добавляются кубы
- задаются форматы
- пишутся формулы
- настраиваются условия редактирования и блокировки

### Уровень 4. Представления и UX
После этого настраиваются:
- вид таблиц
- порядок элементов
- стили
- скрытие/показ колонок
- визуальные разделители

## 11. Практические правила моделирования

### 1. Не путай справочник и свойство
Если сущность участвует в разрезах анализа — чаще всего это **справочник**.
Если это просто атрибут элемента — чаще **свойство**.

Пример:
- `Клиент` — справочник
- `ИНН клиента` — свойство клиента
- `Канал продаж` — справочник, если нужен разрез P&L
- `Email ответственного` — свойство

### 2. Не тащи разную гранулярность в один мультикуб без нужды
Если часть метрик живёт по дням, а часть — по месяцам, если часть по SKU, а часть по категориям — сначала подумай, не лучше ли разделить мультикубы.

### 3. Формат `Измерение` почти всегда лучше текста для ссылочных значений
Если значение должно ссылаться на элемент модели, храни его как измерение, а не как строку.
Иначе теряешь:
- контроль допустимых значений
- навигацию
- зависимые выборы
- более надёжные формулы

### 4. Выборки — это не только “удобный фильтр”, но и часть архитектуры
Они помогают:
- упрощать интерфейс
- ограничивать контекст
- строить специальные отчётные формы
- уменьшать визуальный шум

### 5. Время и версии надо проектировать в начале
Потому что они влияют почти на все мультикубы и формулы.

### 6. `Нет данных` — полезный технический инструмент
Это удобный разделитель в форме, который не ест память.

### 7. Условия редактирования лучше закладывать как отдельную модельную логику
Обычно это не «ручная настройка на глаз», а отдельный булев куб с прозрачным правилом.

## 13. Быстрый словарь

- **Справочник** — список элементов предметной области
- **Элемент справочника** — конкретный товар / клиент / статья / проект
- **Свойство** — атрибут элемента
- **Выборка** — подмножество элементов
- **Измерение** — ось анализа в мультикубе
- **Время** — системное измерение периодов
- **Версия** — системное измерение сценариев
- **Мультикуб** — контейнер кубов на одной размерности
- **Куб** — отдельный показатель/поле внутри мультикуба
- **Формат данных** — тип значения в кубе или свойстве
- **Omit dimensions** — исключить часть измерений для куба
- **SwitchOver** — дата, где прогноз перестаёт брать факт
- **Условия редактирования** — логика блокировки ввода
- **Нет данных** — технический пустой формат без потребления памяти

---

## 14. Что особенно важно запомнить

Если нужен совсем короткий рабочий вывод, то он такой:

1. **Справочники, время и версии — это фундамент модели.**
2. **Мультикуб — это контейнер, куб — это конкретный показатель внутри него.**
3. **Формат данных определяет не только вид, но и смысл значения.**
4. **Формат `Измерение` — главный инструмент для ссылок, выбора и зависимого контекста.**
5. **Выборки и блокировки — часть архитектуры, а не просто UI-украшение.**
6. **Гранулярность данных надо продумывать заранее, иначе мультикубы быстро становятся неудобными.**
