# Optimacros — скрипты, доступы, роли и безопасность

Источник: страницы 889–935 из архива `Скрипты и доступы`.

Это **рабочий конспект** по четырём темам:
- как устроены макросы и скрипты;
- где в Optimacros настраиваются права;
- какие есть уровни и механизмы ограничения доступа;
- какие практические нюансы безопасности важно помнить при моделировании.

Связанные заметки:

## 1. Быстрая карта: как в Optimacros устроен контроль доступа

Если упростить, в системе есть несколько слоёв безопасности:

1. **Роли пользователей**
   - определяют общий доступ к модели и объектам;
   - настраиваются через ролевую модель.

2. **Доступ к объектам модели**
   - к мультикубам;
   - версиям;
   - справочникам;
   - формам;
   - макросам.

3. **Видимость на панели “Содержимое”**
   - объект может существовать и быть доступным, но скрываться из навигации для роли.

4. **МДП / UAM (доступ к элементам справочников)**
   - ограничивает доступ не ко всему справочнику целиком, а к его элементам.

5. **Доступ к атрибутам справочников**
   - отдельно ограничивает редактирование/просмотр атрибутов элементов.

6. **Workflow**
   - ограничивает редактирование по статусу элементов справочника.

7. **DCA-формулы**
   - задают доступ к отдельным ячейкам кубов и свойств через формулу, которая возвращает `Read`, `Write` или `None`.

8. **User Ban и дополнительные ограничения пользователя**
   - запрет входа в модель;
   - лимиты копирования/экспорта;
   - блокировка вкладки МДП у моделера.

**Практический вывод:** в Optimacros почти всегда работает не один механизм, а комбинация нескольких. Например:
- роль даёт доступ к мультикубу;
- видимость показывает его в меню;
- МДП ограничивает доступ к части элементов;
- DCA режет ввод по ячейкам;
- Workflow закрывает часть данных по статусу.

## 2. Макросы и скрипты

## 2.1. Что это такое

Optimacros разделяет:
- **макрос** — набор команд/инструкций для автоматизации задачи;
- **скрипт** — программный сценарий на **JavaScript V8**.

Скрипты доступны из меню **Макросы → Скрипты**.

## 2.2. Типы скриптов

По документации выделены три типа:
- **Обычные скрипты** — работают внутри модели.
- **Интеграционные скрипты** — работают за пределами модели, могут забирать/передавать данные через **FTP** или **DWH (OLTP)**, выполнять команды/запросы.
- **Комбинированные скрипты** — сочетают оба подхода.

**Что это значит practically:**
- обычный скрипт — про внутреннюю автоматизацию модели;
- интеграционный — уже про внешние данные и интеграционный периметр;
- комбинированный — самый рискованный с точки зрения доступа, потому что связывает внутренние объекты и внешний контур.

## 2.3. Что важно в таблице скриптов

В таблице скриптов заметны ключевые поля:
- **Планировщик** — запуск по расписанию в `Cron`-формате;
- **Папка** — группировка;
- **Режим блокировки**;
- **Дата / Время последнего запуска**;
- **Последний статус** (`success` / `fail` и т. п.);
- **Лимит памяти, МБ**;
- **Лимит времени, сек**;
- **Пользовательские смещения** — визуальное выделение;
- **Комментарий**.

Также есть:
- история запусков;
- фильтрация по типу запуска: `interface`, `scheduler`, `api`;
- источник данных: текущая или архивная таблица;
- цепочка запусков (иерархия связанных запусков).

## 2.4. Режимы блокировки скриптов

Это один из самых важных технических блоков.

### Shared
- позволяет скриптам работать параллельно;
- подходит для сценариев, где скрипт **не меняет метаданные**;
- если Shared-скрипт пытается менять метаданные, система выдаст ошибку.

### Unique
- монопольная блокировка;
- тоже позволяет параллелизм с частью других операций, но даёт возможность изменять метаданные без ошибки;
- два скрипта, меняющих метаданные, одновременно работать не смогут.

### Complete
- системный режим, недоступный для выбора;
- используется для блокировки модели в момент применения системных изменений.

### Custom
- отсутствие предзаданной блокировки;
- скрипт во время работы сам запрашивает нужный режим;
- подходит для сценариев, где модель может вообще не использоваться или не требует жёсткой блокировки.

**Практические выводы:**
- если скрипт меняет **метаданные** (например, элементы справочника, структуру и т. п.), `Shared` опасен/недостаточен;
- для изменений структуры безопаснее думать в сторону `Unique`;
- при работе с OLTP и чисто внешними операциями `Custom` может позволить не блокировать модель лишний раз;
- для ядерных/массовых сценариев блокировку нужно продумывать отдельно, иначе легко словить конфликт редактирования.

## 2.5. Ограничение ресурсов скрипта

Моделер/администратор может задавать:
- **лимит памяти** — по умолчанию 100 МБ, допустимые пределы примерно `> 1 МБ и < 2048 МБ`;
- **лимит времени** — по умолчанию 21600 сек (6 часов), допустимые пределы `> 1 сек и < 86400 сек`.

**Зачем это важно:**
- защита от зависших или прожорливых сценариев;
- контроль стабильности окружения;
- минимизация ущерба от неудачного или слишком тяжёлого скрипта.

## 2.6. Скрипты и внешний контур

В документе отдельно упомянуты интеграции с **OLTP**, включая варианты на базе:
- **MySQL**,
- **ClickHouse**,
- **PostgreSQL**.

**Безопасностный смысл:** любой интеграционный скрипт потенциально расширяет поверхность атаки и требует отдельного контроля:
- какие учётки используются;
- какие права есть у этих учёток;
- может ли скрипт только читать данные или ещё и писать/менять структуру;
- где лежат секреты и как они маскируются.

## 3. Общая модель ролей и прав

## 3.1. Базовая идея

В Login Center / системе безопасности пользователям изначально назначаются **минимальные базовые права**, которые затем расширяются по необходимости.

Это хороший принцип: не выдавать Full Access по умолчанию всем подряд.

## 3.2. Основные роли

### User
- базовая роль;
- даёт доступ к рабочему пространству и разрешённым моделям;
- если роль `User` отозвать, пользователь теряет доступ к тому, что ему было доступно раньше;
- это самая ограниченная роль в системе.

### Admin
Администратор рабочего пространства может:
- назначать других администраторов;
- назначать моделлеров;
- выдавать доступ к конкретной модели и назначать роль пользователя в модели;
- отключать доступ к конкретной модели через `No Access`.

Ограничение:
- `Admin` **не может** назначать роль **Service Admin**.

### Modeler
Моделер:
- создаёт и настраивает модель;
- работает со справочниками, мультикубами, дашбордами, контекстными таблицами;
- может менять роли пользователей **в рамках модели**, если администратор уже дал ему доступ к этой модели;
- может настраивать доступ пользователей к объектам модели;
- может отзывать доступ к модели у пользователя (`No Access`).

Но есть важное ограничение:
- если моделлер отозвал доступ пользователя к модели, **вернуть его обратно уже не может** — это под силу только администратору рабочего пространства.

### Service Admin
- привилегированная роль пользователя воркспейса;
- имеет расширенные административные возможности, например принудительное прерывание запросов.

**Важный нюанс:** права сервисного администратора чувствительны для работоспособности внутренних сервисов. После удаления/потери таких прав восстановление может быть нетривиальным.

## 4. Ролевая модель: что именно можно ограничить

Настройки по ролям доступны через **Центр безопасности → Доступ**.

Для ролей можно отдельно настроить доступ к:
- **мультикубам**;
- **версиям**;
- **справочникам**;
- **формам**;
- **макросам**.

Для большинства объектов используется одна и та же тройка уровней:
- **Write** — просмотр и редактирование;
- **Read** — только просмотр;
- **None** — нет доступа.

### 4.1. Мультикубы
- `Write` — можно смотреть и редактировать данные мультикуба;
- `Read` — только смотреть;
- `None` — нет доступа.

### 4.2. Версии
То же правило:
- `Write`;
- `Read`;
- `None`.

### 4.3. Справочники
Для роли можно включать доступ к конкретным справочникам.

### 4.4. Формы
Для формы редактирования атрибутов элемента справочника также задаётся:
- `Write` — можно смотреть и редактировать;
- `Read` — только смотреть;
- `None` — нет доступа.

### 4.5. Макросы
Можно выдать доступ только к определённым макросам, а не ко всем подряд.

**Практический вывод:** если пользователю не нужен запуск автоматизации, лучше не давать доступ ко всем макросам скопом.

---

## 5. Видимость на панели “Содержимое”

Это отдельный слой от реальных прав.

Для ролей настраивается видимость:
- мультикубов;
- представлений мультикубов;
- дашбордов;
- контекстных таблиц;
- форм.

Есть важная механика папок:
- параметр **New On / New Off** позволяет автоматически показывать или скрывать новые объекты в папке для конкретной роли.

**Практический смысл:**
- доступ и видимость — не одно и то же;
- можно скрыть объект из навигации, даже если технически доступ есть;
- это помогает упростить интерфейс и уменьшить риск случайного использования “не тех” объектов.

## 6. МДП / UAM — доступ к элементам справочников

## 6.1. Что это

МДП (`UAM`, User Access Management) — механизм, который позволяет управлять доступом **к элементам конкретного справочника** для каждого пользователя.

Доступ хранится и задаётся через системный справочник **Access Levels**.

## 6.2. Режимы МДП для справочника

Для справочника можно выбрать:
- **None** — МДП отключён;
- **Cascade** — права на родителя наследуются дочерними элементами;
- **Direct** — права задаются по элементам без наследования.

### Когда что использовать
- `Cascade` — когда иерархия стабильная и права в основном наследуются сверху вниз;
- `Direct` — когда нужно тонко управлять отдельными элементами без наследования.

## 6.3. Поведение по умолчанию

Если для справочника включить `Cascade` или `Direct`, то:
- всем пользователям автоматически выставляется **None** для всех элементов справочника;
- пользователь, который **добавил новый элемент**, автоматически получает к нему **Write**.

Это важный момент: включение МДП без последующей настройки матрицы доступа легко делает справочник “закрытым для всех”.

## 6.4. Скрипты и МДП

Скрипты могут переключать режимы МДП между:
- `None`;
- `Cascade`;
- `Direct`.

Документация отмечает нюанс:
- скрипт может менять режим с `Cascade` на `Direct`;
- но так как такой переход относится к моделированию, скрипт также может временно переводить МДП в `None`;
- если обычный пользователь (не моделлер) попытается выполнить такой сценарий или менять режим МДП, он получит ошибку.

### Скриптовая синхронизация UAM
Отдельно показан паттерн:
- создать мультикуб, где строки — элементы справочника, колонки — `Users`, а куб — в формате `Access Levels`;
- заполнять в нём `Read/Write/None`;
- запускать скрипт, который синхронизирует значения в МДП справочника.

**Практическая польза:**
- можно централизовать и автоматизировать управление доступом;
- удобно для массового обновления прав.

**Подводный камень:** запускающий пользователь должен иметь **Write** ко всем соответствующим элементам в управляющем мультикубе, иначе синхронизация может работать некорректно.

## 6.5. Ограничение МДП для моделера

Есть отдельная возможность **скрыть/заблокировать вкладку МДП** для пользователя с правами моделера.

Это настраивается через JSON в колонке `Настройки`, например через параметр `limitedUAM`:
- `true` — блокировать все МДП;
- число / список идентификаторов — скрыть МДП только для конкретных справочников.

**Важно:** доступ к колонке `Настройки` есть только у администраторов.

## 7. Доступ к атрибутам справочников

Это отдельный механизм от МДП по элементам.

Если пользователю дан доступ к элементу справочника, это ещё не всегда означает, что ему надо разрешать менять:
- имя;
- код;
- родителя;
- произвольные атрибуты элемента.

Для этого существует флаг **“Доступ к атрибутам”** у справочника и отдельная вкладка настройки прав по атрибутам.

Ключевые нюансы:
- после включения этой настройки всем пользователям автоматически ставится **None** на доступ к атрибутам;
- если включить флаг и не настроить матрицу прав, атрибуты окажутся скрыты/закрыты для всех;
- пользователь, который **добавил новый атрибут**, автоматически получает к нему **Write**.

**Практический смысл:** это полезно для конфиденциальных или чувствительных атрибутов — например, служебных кодов, статусов, идентификаторов, внутренних признаков.

## 8. Workflow — доступ по статусу

## 8.1. Что делает Workflow

Workflow управляет правом редактирования:
- атрибутов элементов справочников;
- данных в мультикубах, где этот справочник (или его выборка) используется как измерение.

## 8.2. Статусы

Для элементов справочника используются статусы:
- **Not Started** — не начат;
- **In Progress** — в работе;
- **Done** — завершён.

Цветовая логика в интерфейсе:
- красный — `Not Started`;
- жёлтый — `In Progress`;
- зелёный — `Done`.

## 8.3. Какие права даёт статус

- `Not Started` и `Done` → **Read**;
- `In Progress` → **Write**.

То есть Workflow в первую очередь открывает редактирование **только на стадии работы**, а остальные статусы делает read-only.

## 8.4. Наследование статусов

Для иерархических справочников статус родительского элемента **наследуется дочерними**.

Но если создаётся новый элемент, ему по умолчанию присваивается **In Progress**, даже если родитель уже в другом статусе.

Это важно: новый дочерний элемент сразу становится редактируемым.

## 8.5. Исключение кубов из Workflow

Если для конкретного куба в мультикубе не нужно, чтобы на него распространялись правила Workflow, в режиме редактирования у соответствующего куба снимают отметку **“Включить согласование”**.

**Практический вывод:** Workflow не обязан покрывать весь мультикуб целиком; его можно применять точечно.

## 9. DCA — доступ к ячейкам через формулы

## 9.1. Что такое DCA

DCA (`Dynamic Cell Access`) — это формульное выражение, которое для каждой ячейки возвращает значение уровня доступа в формате справочника **Access Levels**:
- `Write`;
- `Read`;
- `None`.

Если DCA-формула возвращает пустую ячейку, применяется доступ **Write**.

**Это важный подводный камень:** пустой результат не запрещает ввод, а наоборот открывает его.

## 9.2. Где используется

DCA задаётся в свойствах куба/свойства справочника как условие редактирования.

Формула должна быть корректной и возвращать значение для каждой ячейки, аналогично обычной формуле в вычисляемом кубе.

## 9.3. Типовые паттерны из документации

### Паттерн 1. Матрица доступа по времени
Отдельный мультикуб хранит по месяцам `Read/Write/None`, а целевой куб ссылается на него в DCA.

Пример идеи:
- январь — `Write`;
- февраль — `Read`;
- март — пусто → по умолчанию `Write`;
- апрель/май — `None`.

### Паттерн 2. Доступ только с текущего периода
Пример формулы из документации по смыслу:
- если месяц меньше текущего месяца → `Read`;
- иначе → `Write`.

Это хороший паттерн для закрытия прошлых периодов и открытия текущего/будущего горизонта.

### Паттерн 3. Доступ по выборке кубов
DCA можно завязать на логический куб “Разрешить ввод?”.
Если флажок включён → `Write`, иначе → `Read`.

### Паттерн 4. Доступ в разрезе пользователей (`Users`)
Матрица доступа может учитывать системный справочник **Users**:
- один пользователь видит/редактирует только свои комбинации клиент × продукт;
- другой — другой набор.

### Паттерн 5. Полуавтоматическое формирование правил
Можно хранить “ответственных” пользователей в кубах формата `Users` (например, менеджер / младший менеджер), а затем формулой собирать из этого матрицу `Access Levels`.

**Практический смысл DCA:**
- точечный доступ до ячейки;
- динамическая логика по времени, пользователю, виду показателя, признаку в матрице;
- очень мощный инструмент, но требует дисциплины при проектировании.

## 10. User Ban и персональные ограничения пользователя

## 10.1. User Ban

В таблице пользователей модели есть две связанные колонки:
- **User Ban**;
- **User Ban Message**.

В `User Ban` есть два режима:
- **Without Ban** — доступ есть;
- **Hard Ban** — вход в модель заблокирован.

При `Hard Ban` пользователь не может:
- войти в модель;
- скачать её;
- удалить;
- переименовать;
- перевести в offline;
- переместить.

В `User Ban Message` можно показать текст, который пользователь увидит при попытке входа/действия.

## 10.2. Ограничения по сервисным ролям

Для пользователя с ролью **Service Admin** и для **создателя модели** в колонке `User Ban` по умолчанию стоит `Without Ban`, и это значение нельзя изменить.

То есть полностью забанить такого пользователя через этот механизм нельзя.

## 10.3. Дополнительные настройки пользователя

В колонке `Настройка` можно задавать JSON с персональными ограничениями, в частности:
- `limitedCopy` — ограничение числа ячеек на копирование/вставку за один раз;
- `limitedExport: true` — запрет/ограничение экспорта;
- `limitedUAM` — блокировка вкладки МДП для моделера.

**Практический смысл:** можно не только дать/забрать доступ, но и сузить потенциально рискованные действия пользователя.

## 11. Частные модели (Private models)

Частная модель предназначена для ограничения доступа к конкретной модели.

Доступ к ней получает **владелец модели** (`model owner`). Без его разрешения:
- другие пользователи;
- моделлеры;
- администраторы;
- сервисные администраторы

не получают доступ автоматически.

Владелец модели может настроить временный доступ для других пользователей, но сам владелец остаётся центром контроля.

Обычные моделлеры:
- не могут менять доступ владельцу;
- не могут назначать роли другим пользователям в обход владельца.

**Практический вывод:** private model — отдельный слой изоляции поверх обычной модели прав.

## 12. Что особенно важно для безопасности

## 12.1. Скрипты лучше запускать под отдельными учётками

Документация прямо рекомендует:
- создавать **отдельные учётные записи для скриптов**.

Это важно потому, что так проще:
- ограничивать права;
- разделять ответственность;
- отслеживать историю действий;
- не смешивать пользовательскую и техническую активность.

## 12.2. Не выдавать лишний доступ “на вырост”

Из общей модели ролей следует хороший принцип:
- сначала минимум прав;
- потом точечно расширять.

Особенно это касается:
- доступа к макросам;
- прав Modeler/Admin;
- экспорта;
- копирования;
- доступа к атрибутам;
- прав на МДП.

## 12.3. Пустая DCA-формула = не запрет, а Write

Это очень важный нюанс. Если рассчитывали “ничего не вернулось — значит закрыто”, то в Optimacros это наоборот открывает доступ к редактированию.

## 12.4. Включение МДП/доступа к атрибутам может всё закрыть

После активации:
- МДП (`Cascade`/`Direct`) выставляет всем `None` по элементам;
- доступ к атрибутам тоже выставляет всем `None`.

Если после включения не заполнить матрицу — пользователи потеряют доступ.

## 12.5. Workflow и DCA решают разные задачи

- **Workflow** — стадийность и управление процессом;
- **DCA** — формульная логика по ячейкам.

Их не стоит путать. Часто нужна комбинация, а не замена одного другим.

## 12.6. Видимость — не безопасность

Скрыть объект из панели “Содержимое” полезно для UX, но это не полноценная замена настройке прав доступа.

## 12.7. Интеграционные скрипты = повышенный риск

Если скрипт работает с FTP / DWH / OLTP / внешними БД:
- проверять секреты и учётки;
- давать только необходимые права;
- ограничивать экспорт/копирование там, где это возможно;
- следить за логами запусков и цепочкой запусков.

## 12.8. Моделер не всесилен

Есть важные границы полномочий:
- моделлер может многое внутри модели;
- но не всё может вернуть назад после отзыва доступа;
- часть критичных операций остаётся за администратором рабочего пространства.

---

## 13. Практический шаблон проектирования безопасного доступа

Если делать модель “по-рабочему”, обычно разумно идти так:

1. Создать роли по функциям:
   - просмотр;
   - ввод;
   - моделирование;
   - сервисные задачи.

2. Ограничить доступ ролей к объектам:
   - мультикубы;
   - версии;
   - справочники;
   - формы;
   - макросы.

3. Настроить видимость на панели “Содержимое”, чтобы убрать лишнее из интерфейса.

4. Для чувствительных справочников включить:
   - МДП по элементам;
   - при необходимости доступ к атрибутам.

5. Для процессных зон включить Workflow.

6. Для тонкого контроля ввода настроить DCA:
   - по времени;
   - по пользователю;
   - по признакам в матрицах доступа;
   - по выборкам кубов.

7. Для скриптов:
   - отдельные техучётки;
   - минимальные права;
   - правильный режим блокировки;
   - лимиты памяти и времени.

## 14. Короткая шпаргалка

### Роли
- `User` — базовый доступ
- `Admin` — управление правами в workspace
- `Modeler` — настройка модели
- `Service Admin` — привилегированный админ-функционал

### Уровни доступа
- `Write` — просмотр и редактирование
- `Read` — только просмотр
- `None` — нет доступа

### Механизмы
- **Ролевая модель** — доступ к объектам
- **Visibility** — видимость в навигации
- **UAM / МДП** — доступ к элементам справочника
- **Доступ к атрибутам** — отдельный контроль свойств элемента
- **Workflow** — доступ по статусу
- **DCA** — доступ по формуле на уровне ячейки
- **User Ban** — полный запрет входа

### Скрипты
- язык: **JavaScript V8**
- типы: обычные / интеграционные / комбинированные
- запуск: interface / scheduler / api
- блокировки: `Shared`, `Unique`, `Custom` (`Complete` — системный)

---

## 15. Главное, что стоит помнить

1. В Optimacros безопасность — это не одна галка, а набор слоёв.
2. Роль, видимость, МДП, Workflow и DCA решают **разные** задачи.
3. Пустое значение DCA даёт `Write`, а не `None`.
4. Включение МДП или доступа к атрибутам без настройки матрицы может закрыть всё всем.
5. Скрипты, особенно интеграционные, требуют отдельного контроля прав и блокировок.
6. Для скриптов лучше использовать отдельные сервисные учётки.
7. Видимость на панели “Содержимое” полезна для UX, но сама по себе не заменяет реальные права.
8. Если нужно рабочее проектирование доступа — сначала роли и объекты, потом UAM/Workflow/DCA, потом уже косметика интерфейса.
