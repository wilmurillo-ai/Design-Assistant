## 1. Общая модель синтаксиса

### Базовая форма
- Основной стиль — функциональный: `FUNC(arg1, arg2, ...)`
- Параллельно используются:
  - инфиксные операторы: `+`, `-`, `*`, `/`, `&`, `IN`
  - логические конструкции: `IF ... THEN ... ELSE ...`
  - выборки и агрегации в SQL-подобном стиле: `SELECT`, `SUM`, `SUMIF`, `TEXTSUM`, `TEXTSUMIF`
  - работа со справочниками/иерархиями: `ITEM`, `PARENT`, `PROPERTY`, `FINDITEM`, `DISPLAY_NAME` и др.

### Нотация и шаблоны
- Аргументы разделяются запятыми.
- Для перечисления элементов часто используются квадратные скобки:
  - `X IN [Y1, Y2, Y3]`
  - `IS_ANCESTOR(X, [Y1, Y2])`
- Для группировки — круглые скобки:
  - `(X1 + X2) * X3`
- В текстовых функциях строковые константы должны быть в двойных кавычках.

### Представления формулы
На стр. 281 разобран важный технический момент:
- `Формула` — пользовательское представление, если запись синтаксически корректна.
- Если формула синтаксически некорректна или ссылки не удаётся корректно отобразить после изменений структуры, система показывает стабильное/нормализованное представление и подсвечивает проблему.
- После исправления формулы красная подсветка исчезает.

**Практический вывод:** если после удаления/переименования объектов формула «сломалась», сравнивай:
- обычное поле формулы,
- стабильное представление,
- нормализованное представление,
- комментарий к ошибке.

## 2. Операторы и базовая логика

### Арифметика
- `X1 + X2`
- `X1 - X2`
- `X1 * X2`
- `X1 / X2`
- `(X1 + X2) * X3`

### Важное поведение деления
- В Optimacros деление на ноль в ряде случаев не даёт аварийную ошибку, а возвращает `0`.
- Аналогично функция `REM(X, Y)` тоже возвращает `0`, если делитель `Y = 0`.

**Подводный камень:** это удобно для устойчивости расчёта, но может скрывать проблемы качества данных.

### Логика
- `IF x THEN y ELSE z`
- `(x) AND (y)`
- `(x) OR (y)`
- `NOT(x)`

### Проверка вхождения
- `X IN [Y1, Y2, ...]`
- `IS_IN(X, [Y1, Y2, ...])`

## 3. Выборки, связи между кубами и агрегации

Это один из главных блоков документации.

### SELECT
`SELECT` — базовый способ взять значение из другого куба/мультикуба с фиксацией измерений.

Типовые формы:
- `SELECT(X, Y:Y1)`
- `SELECT(X, Z:Y)`
- `SELECT(X, Y:FIRSTNONBLANK())`
- `SELECT(X, Y:LASTNONBLANK())`
- `SELECT(X, Y:FIRSTNONBLANK(), Z:Z1)`
- `SELECT(X, Y:LASTNONBLANK(), Z:Z1)`

Паттерн из примеров:
- сравнение версий через отдельный мультикуб:
  - `SELECT('P&L Classic_промежуточный'.'Data', 'Versions':'Versions'.'Прошедший год')`
  - `SELECT('P&L Classic_промежуточный'.'Data', 'Versions':'Versions'.'Факт')`
  - `SELECT('P&L Classic_промежуточный'.'Data', 'Versions':'Versions'.'Прогноз')`

**Когда применять:**
- тащим значение из куба-источника,
- фиксируем версию, период, элемент справочника или иной срез,
- строим «кубы сравнения» или витрины над исходной моделью.

### SELECT + TEXTSUM
На стр. 291 отдельно отмечен паттерн:
- `SELECT(TEXTSUM(...), Y1:y1, ..., Yk:yk)`

Это частый случай, когда внутри аргумента `X` у `SELECT` используется текстовая агрегация `TEXTSUM`.

### SELF
- `SELF(X1: Y1, ..., Xn: Yn)`

Используется для явного сопоставления контекста/измерений в пределах текущего расчёта.

### COLLECT
- `COLLECT()`

По примерам — вспомогательный приём для построения отчётных/сравнительных мультикубов.

### SUM и SUMIF
Типовые формы:
- `SUM(X)`
- `SUM(X, Y1, ..., Ym)`
- `SUM(X, Y1, ..., Ym, AS(Z1 ± ... ± Zk))`
- `SUMIF(X, B)`
- `SUMIF(X, Y1, ..., Ym, B)`
- `SUMIF(X, Y1, ..., Ym, AS(...), B)`

**Идея:**
- `SUM` — агрегация числовых значений,
- `SUMIF` — агрегация только по тем ячейкам, где условие истинно.

### MIN / MAX / AVG и IF-варианты
- `MIN(X1, ..., Xn)` / `MIN(Выборка кубов)`
- `MAX(X1, ..., Xn)` / `MAX(Выборка кубов)`
- `AVG(X1, ..., Xn)` / `AVG(Выборка кубов)`
- `MINIF(...)`
- `MAXIF(...)`
- `AVGIF(...)`

По стр. 361:
- `AVGIF(X1, ..., Xn, B)` вычисляет среднее только по тем аргументам, для которых соответствующая ячейка логического куба `B` равна `TRUE`.
- Принцип работы аналогичен `SUMIF`.

**Типовой кейс:** считать среднее/максимум/минимум только для разрешённых периодов, активных SKU, выбранных каналов и т.п.

### DIMENSIONSUM и TIMESUM
- `DIMENSIONSUM(X, Y)`
- `DIMENSIONSUM(X, Y, Y1, Y2)`
- `TIMESUM(X, Y1, Y2)`

**Назначение:**
- `DIMENSIONSUM` — суммирование по измерению/измерениям,
- `TIMESUM` — суммирование по временному диапазону.

## 4. Справочники, элементы, свойства и иерархии

Это второй самый важный блок после `SELECT/SUM`.

### Получение текущего элемента и навигация
- `ITEM(X)`
- `PARENT(X)`
- `FIRST(ITEM(X))`, `FIRST(Y)`
- `LAST(ITEM(X))`, `LAST(Y)`
- `LEVEL(X)`
- `D_RANK(X)`
- `MAX_D_RANK(X)`
- `LONG_ID(X)`
- `DISPLAY_NAME(X)`

### DISPLAY_NAME
По стр. 321:
- `DISPLAY_NAME(X)` возвращает отображаемое имя элемента измерения.
- Работает для системных и пользовательских справочников, версий и времени.
- Возвращает текст.

Аргумент `X` может быть:
- `ITEM(...)`,
- кубом/свойством, имеющим формат другого справочника.

Типовые примеры:
- `DISPLAY_NAME(ITEM('Контакты'))`
- `DISPLAY_NAME('Куб контакты')`
- `DISPLAY_NAME(PARENT(ITEM('Города')))`

**Практическая ценность:**
- удобно выводить человекочитаемое имя вместо кода/техссылки,
- часто комбинируется с `PARENT`, `ITEM`, иерархическими свойствами и текстовыми функциями.

### Свойства и поиск
- `PROPERTY(X:'Property', X)`
- `FINDITEM(X, Y)`
- `FINDITEM(X, Y, Z1)`
- `FINDITEM(X, Y, Z1, Z2)`
- `FINDITEM_EXACT(...)`
- `FINDBYNAME(X, Y)` / `FINDBYNAME_EXACT(...)`
- `FINDBYCODE(X, Y)` / `FINDBYCODE_EXACT(...)`

**Когда применять:**
- маппинг текста/кода на элемент справочника,
- поиск по имени или коду,
- более строгий поиск через варианты `EXACT`.

**Подводный камень:** если есть неоднозначные, неполные или «грязные» значения, обычные функции поиска могут давать неожиданные совпадения — в таких случаях лучше использовать `EXACT`-варианты.

### Иерархическая логика
- `IN_DIMENSION(X)` / `IN_DIMENSION(X, Y)`
- `IN_DIMENSION_EXACT(X)` / `IN_DIMENSION_EXACT(X, Y)`
- `IS_PARENT()`
- `IS_ANCESTOR(X, [Y1, Y2, ...])`
- `IS_DESCENDANT(X, [Y1, Y2, ...])`
- `IN_HIERARCHY(X, [Y1, Y2, ...])`

**Типовые применения:**
- проверка принадлежности текущего элемента определённой части иерархии,
- условное включение расчётов только для листовых/родительских узлов,
- фильтрация наборов данных по иерархическому признаку.

### RAND_ITEM
По стр. 341:
- `RAND_ITEM()`
- `RAND_ITEM(X)`

Возвращает случайный элемент измерения.

Особенности:
- при обновлении функции возвращается новый случайный элемент,
- если аргумент не указан, берётся измерение, используемое как формат куба,
- если передан иерархический справочник, возвращаются только **листовые** элементы (без потомков).

**Подводный камень:** результат нестабилен между пересчётами — не стоит использовать в логике, где нужен воспроизводимый результат.

## 5. Числовые функции и математика

### Базовые функции

### REM
По стр. 381:
- `REM(X, Y)` возвращает остаток от деления.
- `X` — делимое, `Y` — делитель.
- Если `Y = 0`, функция возвращает `0`.
- Знак остатка совпадает со знаком делимого.

**Практический вывод:**
- можно безопасно считать остаток без отдельной защиты от деления на ноль,
- но если нулевой делитель означает ошибку данных, её лучше проверять явно.

### Статистические и специальные функции
- `GAUSS(X, Y, M, S)`
- `INORMSDIST(X)`
- `INORMSINV(P)`
- `ERLANGC(X, Y, Z)`
- `ERLANGB(X, Y, Z)`
- `ERLANGC_SLA(X, T, Y, Z)`
- `AGENTS(X, ERLANGC_SLA(NUM_ARG, T, Y, Z))`

**Смысл блока:** специализированные функции для вероятностных и нагрузочных расчётов (например, call center / queueing-задачи).

## 7. Текстовые функции

### Базовые функции
- `NAME(X)`
- `CODE(X)`
- `SUBSTITUTE(X, Y, Z)`
- `LEFT(X, N)`
- `RIGHT(X, N)`
- `LOOKUPTEXT(X, Y, N)`
- `FINDTEXT(X, S, N)`
- `CLEAN(X)`
- `LENGTH(X)`
- `MATCH(X, Y)`
- `UPPER(X)`
- `LOWER(X)`
- `X & Y`

### Склейка текста
- Оператор конкатенации: `X & Y`

Типовой паттерн:
- `CODE(ITEM('Products')) & " - " & DISPLAY_NAME(ITEM('Products'))`

### FINDTEXT
По стр. 441:
- `FINDTEXT(X, S, N)` ищет позицию подстроки `X` в строке `S`.
- `N` — начальная позиция поиска (опционально).
- Индексация начинается с `1`.
- Текстовые/символьные константы должны быть в двойных кавычках.

Важно:
- если значение **не найдено**, функция возвращает **не null, а дату** (так указано в примечании документации).
- поэтому результат нельзя безопасно проверять сравнением с нулём; для контроля надо использовать функции работы с `NaN`/спецпроверки результата.

**Подводный камень:** это нетривиальное поведение, на нём легко поймать ложноположительную логику.

### TEXTSUM и TEXTSUMIF
Это ключевые функции текстовой агрегации.

Базовая идея:
- `TEXTSUM(...)` — объединяет текстовые значения,
- `TEXTSUMIF(...)` — делает то же, но по условию.

Из зафиксированных по страницам паттернов:
- можно использовать `LIMIT=<число>`
- можно использовать `DISTINCT`
- можно задавать `VALUE=<формула>`
- можно задавать `ORDER=(...):DESC`
- можно задавать `SEPARATOR="..."`

По стр. 461:
- `VALUE=CURRENT_VALUE` позволяет включить в итоговую строку текущее значение ячейки и дополнить его.
- В `VALUE` можно задавать и произвольную константу.
- Через `ORDER` можно явно задать обратный порядок склейки.

Примеры паттернов:
- `TEXTSUM('Источник'.'Продукт', LIMIT=16, DISTINCT)`
- `TEXTSUM('Источник'.'Продукт', LIMIT=16, DISTINCT, VALUE=CURRENT_VALUE & " oooooo")`
- `TEXTSUM('Источник'.'Продукт', LIMIT=16, DISTINCT, VALUE=" oooooo ")`
- `TEXTSUM('Источник'.'Регион', ORDER=('Нум 10'):DESC, SEPARATOR='***')`

**Когда особенно полезно:**
- собрать список товаров/регионов/контрагентов в одну ячейку,
- сформировать поясняющий текст из множества строк,
- показать только уникальные элементы,
- ограничить длину списка и отсортировать его перед выводом.

**Подводные камни:**
- без `DISTINCT` можно получить дубли,
- без `LIMIT` легко разрастить строку,
- если важен порядок — задавай `ORDER`, иначе порядок может быть не тем, который ожидаешь,
- `CURRENT_VALUE` полезен для дозаписи, но может неожиданно дублировать уже имеющийся текст.

## 8. Функция CURRENT_VALUE

По стр. 291:
- `CURRENT_VALUE` возвращает значение текущей ячейки куба.

На практике чаще всего встречается не как самостоятельная бизнес-функция, а как аргумент внутри других выражений — например, в `TEXTSUM(..., VALUE=CURRENT_VALUE & "...")`.

**Полезный паттерн:**
- дополнять текущее содержимое ячейки вместо полной перезаписи.

## 9. Матричные и продвинутые функции

### MULTIPLY
По стр. 481 описан сценарий матричного перемножения.

Суть:
- функция позволяет перемножать полные матрицы, заданные в мультикубах,
- нужно явно сопоставить измерения строк/столбцов и маппинг через свойства.

Из примера:
- строятся справочники `A`, `B`, `C`, `D`,
- в одном мультикубе строки = `A`, столбцы = `B`,
- во втором строки = `C`, столбцы = `D`,
- элементы `C` ссылаются через свойство на элементы `B`,
- результат записывается в куб со строками `A` и столбцами `D`.

Пример вызова из документации:
- `MULTIPLY('M1'.'Куб1'['*A', '*B'], 'M2'.'Куб2'['s.C', '*D'], '*C'.'p.C')`

**Практический смысл:**
- матричные преобразования,
- аллокации и трансформации с маппингом по измерениям,
- сложные пересчёты между двумя справочными осями.

### DORP и DORPIF
По стр. 501:
- `DORP(source, featureCube1, featureCube2, ..., featureCubeN)`
- `DORPIF(source, featureCube1, featureCube2, ..., featureCubeN, condition)`

Назначение:
- ищут значение в источнике данных с учётом иерархической структуры измерений, представленных в кубе-источнике и кубе назначения.

Аргументы:
- `source` — куб маппинг с данными,
- `feature` — куб формата одного из измерений куба назначения,
- `condition` — логическое выражение для `DORPIF`.

Особенности:
- `DORP/DORPIF` возвращают **набор данных**, а не одно значение,
- чтобы получить одно значение, поверх обычно нужна агрегация (`MAX`, `MIN` и т.д.),
- `DORPIF` срабатывает только если `condition = TRUE`.

**Практический вывод:** это не простая lookup-функция, а более сложный иерархический механизм сопоставления.

## 10. Повторяющиеся практические паттерны

### 10.1. Сравнение версий через отдельный мультикуб
Паттерн из страниц с `SELECT`:
1. Есть базовый куб с измерением `Versions`.
2. Создаётся отдельный отчётный мультикуб с кубами `Факт`, `Прогноз`, `Прошлый год` и расчётными отклонениями.
3. Каждый куб-заготовка вытягивает нужную версию через `SELECT(...)`.
4. Разницы считаются обычной арифметикой:
   - `'Факт' - 'Прогноз'`
   - `'Факт' - 'Прошедший год'`

### 10.2. Текстовая витрина по множеству строк
Типовой шаблон:
- `TEXTSUM(..., DISTINCT, LIMIT=..., ORDER=..., SEPARATOR=...)`

Использование:
- вывести список объектов в одной ячейке,
- дать пояснение по набору пересекающихся элементов,
- собрать «топ-N названий» в текст.

### 10.3. Условная агрегация по логическому кубу
Шаблон:
- `SUMIF(...)`
- `AVGIF(...)`
- `MAXIF(...)`
- `MINIF(...)`

Использование:
- считать только активные, разрешённые, выбранные элементы,
- не городить длинные каскады `IF` вокруг каждого аргумента.

### 10.4. Работа с иерархией через человекочитаемые имена
Шаблон:
- `DISPLAY_NAME(ITEM(...))`
- `DISPLAY_NAME(PARENT(ITEM(...)))`

Использование:
- отчёты и формы, где нужен не код, а отображаемое имя,
- подписи для регионов/городов/контрагентов,
- сборка понятного текстового вывода.

### 10.5. Мягкая обработка проблемных делителей
Шаблоны:
- `X / Y`
- `REM(X, Y)`

Использование:
- расчёт не падает на нуле,
- но в критичных местах лучше добавить явную валидацию качества входных данных.

## 11. Подводные камни и что проверять первым делом

### 11.1. Сломанные ссылки после изменений структуры
Если удалили куб/элемент/ссылку:
- формула может остаться «похожей на живую», но перейти в стабильное/нормализованное представление,
- проверяй комментарий к ошибке и поля представления формулы.

### 11.2. Деление на ноль не всегда видно
- `/` и `REM` могут вернуть `0`, а не явную ошибку.
- Это удобно, но маскирует плохие данные.

### 11.3. Поиск текста — не через сравнение с нулём
По `FINDTEXT`:
- если не найдено, поведение нестандартное;
- не полагайся на проверку `= 0`.

### 11.4. `RAND_ITEM` нестабилен
- при обновлении/пересчёте значение меняется,
- для воспроизводимых бизнес-правил не подходит.

### 11.5. `TEXTSUM` без явных параметров может вести себя не так, как ждёшь
- порядок вывода может быть неожиданным,
- возможны дубли,
- строка может стать слишком длинной,
- `CURRENT_VALUE` может дублировать текст.

### 11.6. `CURRENT_DATE()` работает от UTC
- если модель мыслит локальным временем, учитывай сдвиг.

### 11.7. `DORP/DORPIF` возвращают не скаляр
- часто поверх нужно ещё `MAX`, `MIN` или другая агрегация.

## 14. Что особенно полезно помнить на практике

Если нужен быстрый старт при моделировании в Optimacros, в первую очередь держи в голове вот что:
1. `SELECT` — основной инструмент для вытягивания данных из других кубов.
2. `SUM/SUMIF/AVGIF` — основной инструмент контролируемой агрегации.
3. `ITEM/PARENT/PROPERTY/DISPLAY_NAME` — основа работы со справочниками.
4. `TEXTSUM` — удобный способ собрать текстовую витрину по множеству строк.
5. `CURRENT_DATE`, `DATESHIFT`, `DATEPART` — основа временной логики.
6. `MULTIPLY`, `DORP/DORPIF` — уже более продвинутый слой для специальных задач.
7. При странном поведении формулы сначала проверяй: ссылки, иерархию, тип формата, UTC-смещение и скрытую обработку нулевых делителей.
