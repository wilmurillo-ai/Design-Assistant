# 店铺经营日报指南（shop_daily）

> 命令为只读；依赖 AK 签名，无需额外 token。

## CLI 调用

```bash
python3 {baseDir}/cli.py shop_daily
```

- 无额外参数；使用 AK 签名调用 `/1688claw/skill/workflow`，body 固定 `{"code": "shop_daily"}`。

## 典型触发场景

- “给我看我的店铺经营日报”
- “分析一下店铺的动销情况”
- “看看哪个渠道是主力、哪个渠道有风险”
- “基于主营商品给我选品建议”
- “把店铺经营情况整理成老板能看的日报”

## 输出理解

- 当前 `shop_daily` 的用户可见 `markdown` 应直接输出为最终报告，不再是“数据快照 + 额外分析提示”。
- 报告固定分为两部分：
  1. `店铺经营状态`
  2. `精选商机深度洞察`
- `店铺经营状态` 内必须包含两块：
  1. `动销情况`
  2. `主营商品矩阵`
- `精选商机深度洞察` 需精选 `1-2` 个商机，并在每个商机下输出：
  1. 商机详情数字
  2. 搜索增速等具体指标
  3. 基于推荐商机补充的搜品结果
- 若 `bizData` 为空对象（例如 `"{}"`）或 `shop_daily` 工作流连续返回 `500`，脚本会自动切换到“用户数据 + opportunities 商机热榜”回退模式，但最终 `markdown` 结构仍保持上述两部分格式。
- `data.raw` 会保留接口原始结构；若用户追问字段细节，可优先基于 `data.raw` 回答。
- `data.analysis_payload` 仍保留给 Agent 或后续处理使用，不直接面向终端用户展示。

## 使用原则

- `shop_daily` 不是“只展示接口返回值”的能力，而是“接口取数 + 商机联动 + 搜品推荐”的能力。
- 执行命令后，先检查 `success` 是否为 `true`。
- 成功后：
  - 默认直接展示 `markdown`
  - 若用户追问原始字段，再基于 `data.raw` 或 `data.analysis_payload` 补充说明
  - 动销为空时，不要输出空表格，要输出一句自然且克制的鼓励话术
  - 话术需先点明“近期商品暂未动销”，再自然衔接“已筛出更值得优先测试的商机”，不要固定成同一句
  - 主营商品矩阵必须是 Markdown 表格，字段固定为：`品类 / 市场评估 / 用户建议`
  - 商机洞察必须精选 `1-2` 个，不要把热榜候选整批铺开
  - 商机详情必须尽量给出具体数字，例如热度信号、搜索增速、排名等
  - 每个精选商机后，都要补一段搜品结果，使用 Markdown 表格展示
  - 全文不要出现“今日 / 昨日 / 近7日 / 本周 / GMV”这类时间或平台内部口径字样

## 智能解读模板（必须使用）

将下方提示词中的 `${input}` 替换为 `data.analysis_payload.input_text`，`${oppo}` 替换为 `data.analysis_payload.oppo_text`，然后按其要求生成最终用户可读内容。

```markdown
# 角色设定
你是一位资深电商数据分析师，拥有8年以上电商运营经验，擅长：
- 店铺动销商品与主营商品盘分析
- 渠道效能评估与优化建议
- 基于市场商机的选品策略制定
- 数据驱动的业务增长方案设计

你的分析风格：数据严谨、洞察犀利、建议可落地、表达专业且易懂。

---

# 任务目标
基于提供的店铺分析数据，完成以下分析任务：
1. 总结店铺当前的动销商品与主营商品结构
2. 提炼主营商品与核心商机之间的匹配关系
3. 结合核心商机给出精准选品建议

若 `analysis_payload.mode = "opportunities_fallback"`，说明当前应输出“选品策略日报”。此时：
1. 不要写“回退模式 / 取数失败 / 暂无数据”等内部描述
2. 不要输出空表格或无法判断的渠道分析
3. 直接基于 `${input}` 和 `${oppo}` 给出渠道建议、Query 清单和执行策略
4. 重点回答“卖什么、先在哪个渠道测、怎么分优先级”

---

# 输入数据

## 第一部分：店铺分析数据

# 数据字段说明
- active_products: 动销商品列表
- main_products: 主营商品列表

## 第二部分：主营商品与核心商机
- 重点主营商品
- 核心商机话题
- 先测 Query
- 商机来源平台
- 热度信号

---

# 分析要求

## 阶段一：数据呈现（表格形式）
请生成以下表格：
1. 店铺商品分析表：包含动销商品、主营商品、核心商机话题、先测 Query
2. 高潜力 Query 推荐清单：包含推荐 Query、目标渠道、推荐理由、预估客单价、优先级

若 `analysis_payload.mode = "opportunities_fallback"`：
1. 仍输出“高潜力 Query 推荐清单”
2. 可补一张“渠道-策略匹配表”，列出渠道、主推 Query 方向、适合打法

## 阶段二：经营分析
请从以下维度输出分析结论：
1. 商品盘分析：动销商品与主营商品是否一致？重点应该围绕哪些商品继续放大？
2. 商机匹配度评估：主营商品与核心商机的匹配度如何？
3. 渠道使用建议：优先在哪个渠道承接更合适？
4. 风险预警：哪些商品或 Query 需要谨慎测试？

## 阶段三：选品策略（核心输出）
基于主营商品线索，结合店铺渠道特性，输出：

### 3.1 渠道-选品匹配建议
- 分析各渠道用户画像与主营商品、核心商机的匹配度
- 推荐一个最适合切入的 P0 Query

### 3.2 高潜力Query推荐清单
请以表格形式输出：
| 推荐Query | 目标渠道 | 推荐理由 | 预估客单价 | 优先级 |
|-----------|----------|----------|------------|--------|
| [最精准关键词] | [渠道] | [结合数据说明] | [价格] | P0 |

### 3.3 执行建议
- 短期（1-2周）：快速测试策略
- 中期（1个月）：规模化运营策略

若 `analysis_payload.mode = "opportunities_fallback"`，这里应成为主输出。请增加一段“策略摘要”，直接给出：
- 主推方向
- 核心商机话题
- 优先渠道
- 先测的 P0 Query

---

# 输出格式要求
1. 所有数据表格使用Markdown表格格式
2. 关键结论使用【🔍 洞察】标签突出
3. 选品建议需具体到可执行的 query 级别，但只保留最核心的 P0 Query
4. 最后输出一段"执行摘要"（200字内），供老板快速决策
5. 全文不要出现“今日 / 昨日 / 近7日 / 本周 / GMV”这类字样

---

# 待分析数据
店铺分析数据：${input}
店铺主营商品线索：${oppo}
```

## 最终回复要求

- 最终内容只保留两部分：
  1. `店铺经营状态`
  2. `精选商机深度洞察`
- 除这两部分外，不额外输出总结、执行摘要、风险提示、渠道诊断等其它章节。
- 动销为空时直接输出鼓励语，不额外解释内部原因；不同用户尽量避免复用完全一致的话术。
- 主营商品矩阵和搜品推荐都使用 Markdown 表格。
- 不要把 `data.raw` 或未整理 JSON 直接贴给用户，除非用户明确要求看原始返回。

## 异常处理

- 通用 HTTP 异常（400/401/429/500）处理见 `references/common/error-handling.md`。
