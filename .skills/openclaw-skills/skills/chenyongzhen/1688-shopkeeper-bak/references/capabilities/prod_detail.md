# 商品详情指南（prod_detail）

## CLI 调用

```bash
# 方式一：抓取并保存商品详情
python3 {baseDir}/cli.py prod_detail --item-ids "692505958039,123456789012"

# 方式二：按 data_id 读取整份已保存的商品详情（仅在确需查看全量内容时使用，避免上下文过大）
python3 {baseDir}/cli.py prod_detail --data-id "20260317_143022_123"

# 方式三：按 data_id 读取指定商品详情
python3 {baseDir}/cli.py prod_detail --data-id "20260317_143022_123" --item-ids "692505958039"
```

| 参数 | 默认 | 说明 |
|------|------|------|
| `--item-ids` | 可选 | 逗号分隔的商品 ID。单独使用时表示抓取远端商品详情；与 `--data-id` 一起使用时表示按需读取指定商品 |
| `--data-id` | 可选 | 已保存商品详情的快照 ID。提供后优先走本地读取，不再请求远端接口 |

- 使用 AK 签名调用 `/1688claw/skill/workflow`
- 请求体固定为：

```json
{
  "code": "offer_detail",
  "bizParams": {
    "item_id": ["692505958039", "123456789012"]
  }
}
```

## 典型触发场景

- 用户想查看某个 1688 商品的详情信息
- 用户希望提取商品标题、类目、SKU、商家信息等结构化内容
- 用户已提供明确的商品 ID

若用户给的是 1688 商品链接，Agent 应先从链接中提取商品 ID，再调用本命令。

## 输出结构

### 抓取并保存

```json
{
  "success": true,
  "markdown": "已保存 **1** 个商品详情，保存地址：`/path/to/1688_detail_20260317_143022_123.json`。",
  "data": {
    "data_id": "20260317_143022_123",
    "detail_count": 1,
    "details": {}
  }
}
```

### 按需读取

```json
{
  "success": true,
  "markdown": "已读取 data_id=`20260317_143022_123` 中的 **1** 个商品详情。",
  "data": {
    "data_id": "20260317_143022_123",
    "detail_count": 1,
    "details": {
      "692505958039": {
        "all_info": "# 商品ID\n692505958039\n\n# 商品标题\n..."
      }
    }
  }
}
```

| data 字段 | 含义 |
|-----------|------|
| `data_id` | 本次商品详情快照 ID，可用于排查和后续引用 |
| `detail_count` | 返回的商品详情数量 |
| `details` | 原始 `bizData`，保持接口返回格式；抓取并保存时默认为空，按需读取时返回内容 |
| `details.<item_id>.all_info` | 接口返回的完整商详 Markdown 文本 |

## markdown 与 data 的分工

- `markdown`：仅返回保存/读取结果摘要，不承载商品详情正文
- `data.details`：抓取并保存时默认为空；只有在按 `data_id` 按需读取时才返回原始 `bizData`
- 脚本会同时将本次商详保存到本地快照文件，并返回对应 `data_id`
- 这样可以避免在首次抓取时把大段商详直接塞进 Agent prompt，降低 token 开销

## 这些信息有什么用

`all_info` 通常会包含：
- 商品标题
- 商品价格
- 商品类目
- SKU 属性
- 必填 CPV 属性
- 商家信息

## 典型使用场景包括：

| 场景 | 会用到的信息 | 用法 |
|------|--------------|------|
| 选品判断 | 标题、价格、类目、商家信息 | 判断商品卖点、价格带、类目是否匹配店铺定位，以及供货商是否稳定 |
| 铺货前补全 | SKU 属性、CPV 属性、类目 | 为后续铺货/上架准备规格、属性、类目映射等基础信息 |
| 内容生成 | 标题、类目、SKU、CPV 属性 | 生成下游平台标题、卖点、详情描述、规格说明 |
| 风险识别 | 标题、类目、CPV 属性、商家信息 | 识别侵权风险、审核风险、售后风险、商家可靠性风险 |
| 多商品对比 | 标题、价格、SKU、商家信息 | 对比多个候选商品，辅助决定谁更适合铺货或主推 |

其中：
- 商品标题：用于理解商品本身、提炼卖点、生成更适合下游平台的标题
- 商品价格：用于估算利润空间、判断适合做引流款还是利润款
- 商品类目：用于判断是否适合目标平台和店铺定位，也便于后续做类目映射
- SKU 属性：用于识别可售规格，决定哪些规格需要保留、突出或过滤
- 必填 CPV 属性：用于补全平台上架时的标准属性，减少铺货失败和审核风险
- 商家信息：用于判断供货商稳定性、工厂/贸易属性、资质情况

## 最佳实践

为避免将整份商详一次性带入上下文，推荐使用“先保存、再按需读取、最后总结”的渐进式流程：

1. 先抓取并保存
   - 使用 `prod_detail --item-ids ...`
   - 先只拿 `data_id` 和 `detail_count`
   - 不在首次抓取后直接展开 `all_info`

2. 明确目标后再读取
   - 用户只是想确认“有没有详情”时，不读取正文
   - 用户要看某个商品时，用 `--data-id ... --item-ids ...` 只读该商品
   - 用户要做多商品比较时，优先先读候选商品，不要默认读取整批
   - 只有确实需要全量内容时，才用 `--data-id ...` 读取整份快照

3. 读取后先做摘要，再决定是否继续展开
   - 优先提炼标题、价格、类目、SKU、商家信息等关键点
   - 先给用户摘要和结论，再根据追问补充细节
   - 若只需回答“能不能铺”“有什么风险”“怎么写标题”，通常不需要逐段转述完整 `all_info`

4. 多商品场景控制读取范围
   - 多商品分析时，先做每个商品 3-5 条关键信息摘要
   - 仅对需要深入分析的商品继续展开细节
   - 避免一次性把多个商品的完整商详全部放入 prompt

一句话原则：**先保存、后筛选、再总结，最后才按需展开全文。**

## Agent 展示规则

| 场景 | Agent 操作 |
|------|-----------|
| 首次抓取商品详情 | 先执行 `--item-ids` 抓取并保存，只向用户说明已保存和对应 `data_id` |
| 用户要看具体内容 | 再用 `--data-id` 按需读取；若只看某个商品，可同时传 `--item-ids` 过滤 |
| 查询单个商品 | 按需从 `data.details.<item_id>.all_info` 中提取重点 |
| 查询多个商品 | 按需基于 `data.details` 做横向对比 |
| 用户只想看某部分信息 | 从 `all_info` 中提取对应字段重述，如标题、价格、类目、商家 |
| 用户给的是商品链接 | 先提取 `item_id` 再调用，不要要求用户手动抄 ID |

如果用户的真实目标是“能不能铺”“该怎么卖”“该怎么写标题/卖点”，Agent 不应停留在转述详情，而应把 `all_info` 作为后续分析输入。

## 异常处理

- 通用 HTTP 异常（400/401/429/500）处理见 `references/common/error-handling.md`。
- 若返回 `detail_count: 0`，提示用户检查 `item_id` 是否正确。
- 若 `all_info` 缺失或为空，展示时注明“已查到商品，但详情内容为空”。
