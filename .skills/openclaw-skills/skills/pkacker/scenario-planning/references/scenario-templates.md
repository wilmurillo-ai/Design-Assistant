## Scenario Naming Guide

**Why naming matters:** Scenario names become shorthand for entire strategic worlds. Poor names (Best/Worst) embed bias. Good names are vivid, neutral, and memorable.

**Naming by metaphor (ILM — recommended):**

| Pattern | Examples |
|---------|----------|
| Weather | Headwind, Tailwind, Crosswind, Still Water, Perfect Storm |
| Character | The Accelerator, The Disruptor, The Steady Hand, The Pivot |
| Market condition | The Open Market, The Regulated Platform, The Fragmented Field |
| Journey | The Direct Route, The Detour, The Fork in the Road, The Dead End |
| Strategic state | Resource Constrained, Technology-Enabled, Regulatory-Bound, Partner-Led |

**Avoid:**
- Best Case / Worst Case (induces wishful/fear-based thinking)
- Optimistic / Pessimistic (same problem)
- Success / Failure (implies team caused the scenario)
- Numbers (Scenario 1, 2, 3) — too hard to internalize

**The "News Headline" technique :**
Each scenario gets a catchy, provocative news headline written as if published in the future. This immediately makes the scenario concrete and discussion-worthy. Examples:
- "Trillionaires prefer Laissez-Faire" (favorable regulation + successful monetization)
- "No-Ella! Former online sensation gone down the  tube" (adverse regulation + failed monetization)

## Narrative Writing Guide



Good scenario narratives function like news articles written from the future. They answer:
- **Who** is affected?
- **What** has changed from today?
- **Why** did it happen? (What causal chain produced this world?)
- **What does it feel like** to be managing this project right now?

**The present-tense rule:** Write scenarios in present tense, as if you are already in that world. "The project is under severe budget pressure" not "the project will face budget pressure." This makes the scenario feel real and activates genuine strategic thinking.

**Specificity rule:** Generic scenarios produce generic strategies. Every scenario should contain at least one specific, named element — a specific regulation, a specific competitor action, a specific technology, a specific team event.

**Example of a weak narrative:**
> *"In this scenario, the budget is cut and the team faces challenges delivering the project on time."*

**Example of a strong narrative (Disrupted / Headwind):**
> *"Three months into execution, the primary cloud vendor announces end-of-life for the API layer our architecture depends on, with 90 days' notice. Engineering, already at 80% capacity building core features, now faces a forced migration. Simultaneously, the organization's discretionary budget is frozen pending a Q3 board review. The original 12-month roadmap is no longer viable as written, and two senior engineers are fielding offers from a well-funded competitor."*

The strong version gives the team something to react to. It anchors discussion in specificity.

## Scenario Sets by Project Phase

### Initiation / Planning Phase (focus: scope and mandate uncertainty)

| Scenario Name | Core Driver |
|---|---|
| Greenlit & Growing | Mandate confirmed; team and budget expand faster than planned |
| Scope Held, Timeline Compressed | Approval comes with deadline brought forward |
| Descoped at Launch | Budget or strategic priority shifts before execution begins |
| Pivoted Before Start | Strategic context changes; project goal must be redefined |

### Execution Phase (focus: delivery pace and dependency risk)

| Scenario Name | Core Driver |
|---|---|
| On Track | Plan holds; milestones hit; dependencies stable |
| Accelerated Delivery | External tailwind — more resources, cleared blockers, faster vendor |
| Delayed But Recoverable | Key dependency slips; timeline extends; goal preserved |
| Descoped Delivery | Timeline holds but scope must be cut to meet deadline |

### Monitoring / Closing Phase (focus: quality and stakeholder acceptance)

| Scenario Name | Core Driver |
|---|---|
| Clean Close | All deliverables met; handover smooth |
| Conditional Close | Deliverables met with known exceptions; follow-up plan exists |
| Extended Stabilization | Post-launch issues require extended team involvement |
| Retrospective Reset | Close triggers new planning cycle due to unresolved strategic questions |

## Consistency Matrix 

For each pair of assumptions within a scenario, assess compatibility:

| Assumption A | Assumption B | Compatible? | Notes |
|---|---|---|---|
| Regulation increases significantly | Organization faces budget cuts | ✅ Yes — both can co-exist | Compliance costs add budget pressure |
| Technology disruption is severe | Vendor relationships are stable | ⚠️ Tension | Tech disruption usually destabilizes vendors |
| Team expands rapidly | Budget is constrained | ❌ No | Cannot reconcile — revise scenario |

If any pair is incompatible, the scenario must be revised. An internally inconsistent scenario is worse than no scenario — it produces false confidence.

## Probability Assignment (PMT School Only)

If using the PMT school and attaching probabilities:

**Step 1 — Assign marginal probabilities:** What is the unconditional probability of each Scenario Driver state?

**Step 2 — Assign conditional probabilities:** Given that Driver A is in State X, what is the probability that Driver B is in State Y?

**Step 3 — Calculate scenario probabilities:** Multiply the probabilities along each branch of the probability tree.

**Important warning :**
- Probability estimates in scenario planning are *subjective* — based on expert judgment, not historical data.
- Decision-makers tend to overestimate the probability of conjunctive events (A and B both occurring).
- Attaching probabilities can cause decision-makers to focus on the "most likely" scenario and mentally discount the others — defeating the purpose of scenario planning.
- **Recommendation:** For ILM-based engagements, do not attach probabilities. Present all scenarios as equally worthy of preparation.
