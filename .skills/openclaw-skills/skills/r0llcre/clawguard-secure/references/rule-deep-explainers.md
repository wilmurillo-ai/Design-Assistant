# Deep rule explanations

Detailed causal-chain explanations for L1-checked rules. Load when the user asks for a deeper understanding of a specific finding. For rules not listed here, construct an explanation from the finding's own message, description, and attackScenario fields.

### OC-001

When `gateway.bind` is set to a non-loopback address such as `0.0.0.0`, the gateway accepts connections from every network interface on the host, including WiFi and LAN. When `gateway.auth.mode` is also missing or set to `none`, no credential check stands between a network client and the gateway API. This means any device on the same network can reach the gateway and issue requests without proving identity. Those requests grant full access to conversation history, agent tool invocation (including `system.run` for command execution), and filesystem operations. An attacker on a shared network needs only a browser or `curl` to discover the gateway port and connect. So the result is unauthenticated remote code execution and data exfiltration from any machine on the local network, achievable in under two minutes with default port settings.

### OC-002

When `gateway.http.endpoints` defines HTTP routes such as `/tools/invoke` or `/v1/*`, the gateway exposes a programmatic API surface over HTTP. When `gateway.auth.mode` is simultaneously `none` or unset, those endpoints accept requests from any client without requiring a token, password, or other credential. This means an attacker with network access can call tool invocation endpoints directly, triggering agent actions including command execution, file access, and external requests. Because these are standard HTTP endpoints, the attacker does not need to use the Control UI at all — raw HTTP requests suffice. The API surface also enables automated, scriptable attacks at scale rather than manual UI interaction. So the result is that every tool the agent can invoke becomes callable by any unauthenticated network client through a simple HTTP POST.

### OC-003

When `gateway.tailscale.mode` is set to `funnel`, the gateway is routed through Tailscale Funnel, which assigns a publicly resolvable `*.ts.net` hostname and exposes the gateway to the entire internet. This converts a deployment that may have been designed for LAN-only or tailnet-only access into one reachable by any device worldwide. Internet-wide scanners and DNS enumeration can discover the Funnel hostname within minutes of activation. Once discovered, any attacker can connect without needing to be on the same network, VPN, or tailnet. If `gateway.auth.mode` is also weak or missing, the combination produces unauthenticated access from anywhere on the internet. So the result is that a single config value change moves the gateway from a private network surface to a globally exposed one, with discovery and exploitation possible in minutes.

### OC-004

When `gateway.controlUi.allowedOrigins` is missing or empty on a non-loopback deployment, the gateway does not enforce CORS origin restrictions on the Control UI. This makes the gateway vulnerable to DNS rebinding attacks, where an attacker-controlled domain is re-resolved to the gateway's LAN IP after the victim's browser loads the attacker's page. The victim's browser then treats requests to the gateway as same-origin with the attacker's domain, allowing the attacker's JavaScript to read and write through the Control UI API. Because the gateway has no origin allowlist to reject these requests, the browser-side defense that would normally block cross-origin access is completely bypassed. If gateway authentication and device pairing are also absent (OC-001 and OC-005), the attacker's JavaScript gains full unauthenticated control with zero user interaction beyond visiting a webpage. So the result is that any website the operator visits can silently read conversation data, invoke agent tools, and modify configuration through the operator's own browser.

### OC-005

When `gateway.controlUi.dangerouslyDisableDeviceAuth` is set to `true`, the device pairing challenge that normally gates first-time browser access to the Control UI is removed. This means any browser that can reach the gateway's network address is granted an immediate UI session without needing to complete physical device verification. Device authentication serves as a distinct security layer from credential-based auth — it binds a specific browser to a known device, preventing stolen credentials from being used on new machines. Without it, an attacker who gains network access (or exploits a CORS bypass) faces no per-device identity check before interacting with the full Control UI. When combined with missing gateway auth (OC-001) and missing origin restrictions (OC-004), this creates a triple-zero state where zero credentials, zero device verification, and zero origin checks stand between an attacker and full agent control. So the result is that any browser on the network, or any website the operator visits, can interact with the Control UI as if it were a trusted, paired device.

### OC-006

Browsers include an `Origin` header on cross-origin requests so servers can verify the request source against a CORS allowlist. When `dangerouslyAllowHostHeaderOriginFallback` is set to `true`, the gateway substitutes the `Host` header in place of a missing `Origin` header, treating whatever hostname the client requested as the legitimate origin. The `Host` header, unlike the `Origin` header, is not a browser-enforced security boundary — it simply reflects the domain name used in the URL, which the client controls. An attacker performing a DNS rebinding attack registers a domain with a short TTL, initially pointing it at their own server to load malicious JavaScript, then re-points it at the victim's gateway IP. The victim's browser now sends requests to the gateway with the attacker's `Host` value but no `Origin` header, and the gateway accepts it. Disabling the fallback and configuring explicit `allowedOrigins` forces the gateway to reject requests that lack a valid browser-asserted `Origin` header, closing the rebinding path entirely.

### OC-007

Reverse proxies forward the client's true IP address in headers like `X-Forwarded-For` and `X-Real-IP`. The gateway is supposed to trust these headers only when the immediate TCP connection comes from a known proxy IP listed in `trustedProxies`. When `allowRealIpFallback` is `true`, the gateway accepts these headers from any source, including direct client connections that never passed through a proxy. An attacker can set `X-Forwarded-For: 10.0.0.1` on their request, and the gateway records `10.0.0.1` as the client IP, even though the connection came from an entirely different address. This breaks IP-based access controls, rate limiting, geo-restrictions, and audit log integrity, because every security decision that references the client IP now operates on attacker-supplied data. Setting `allowRealIpFallback` to `false` and listing actual proxy IPs in `trustedProxies` ensures the gateway only trusts forwarded headers from verified proxy connections.

### OC-008

Channel DM policies control which users can initiate private conversations with the agent. When `dmPolicy` is set to `"open"`, the agent accepts direct messages from any account on the connected platform with no pairing challenge, no allowlist check, and no identity verification. This means any user who can discover the agent can begin a private 1:1 conversation. Direct messages are particularly dangerous because they provide a private, unmonitored channel where prompt injection payloads have no witnesses and no moderation layer. An attacker can craft messages that instruct the agent to execute commands, read sensitive files, or exfiltrate data, and the entire interaction remains invisible to the agent's operator. Changing the policy to `"pairing"` requires device-level verification before messaging, and `"allowlist"` restricts messaging to explicitly named accounts.

### OC-009

Group channel policies determine which shared conversations the agent will respond in. When `groupPolicy` is `"open"`, any member of the connected platform who adds the agent to a group or channel can trigger it to process messages and invoke tools. In a large workspace with hundreds of members, this means a single untrusted member in any group can issue instructions to the agent with its full configured tool surface. The risk compounds because group messages are visible to all members, so an attacker's prompt injection could cause the agent to leak sensitive data into a shared context where others can observe it. Unlike DM-based attacks which are limited to one attacker, open group access enables triggering by any number of participants across any number of groups. Setting `groupPolicy` to `"allowlist"` with specific group IDs restricts the agent's participation to explicitly approved conversations.

### OC-010

When `requireMention` is set to `false` or left unconfigured, the agent treats every message in every joined group as potential input, parsing and potentially acting on all of them. This dramatically increases prompt injection exposure because an attacker does not need to address the agent directly — any message containing embedded instructions will be processed. The attack surface scales with group activity: a busy channel with hundreds of daily messages presents hundreds of opportunities for injection payloads to reach the agent. Even benign messages that happen to contain instruction-like patterns (URLs with encoded payloads, code snippets, pasted error messages) can trigger unintended agent actions. Setting `requireMention` to `true` reduces the attack surface to only those messages that explicitly name the agent.

### OC-011

When `tools.exec.security` is missing or set to neither a deny list nor an allow list, the agent runtime places no restrictions on which shell commands the agent can invoke. This means the agent can execute arbitrary commands on the host operating system, including package installation, process management, network requests, and file deletion. An attacker who gains influence over the agent's reasoning — through prompt injection in a document, a crafted API response, or a malicious repository file — can leverage this unrestricted execution to exfiltrate environment variables, install backdoors, or pivot laterally across the network. Even without adversarial input, a miscalibrated agent may autonomously run destructive commands based on a misinterpreted user request. Configuring a deny list blocks known-dangerous commands, while an allow list restricts execution to a pre-approved set; both approaches create a hard boundary that the agent cannot override regardless of its reasoning chain.

### OC-012

When `tools.elevated.enabled` is set to true with an empty or wildcard `allowFrom` field, the runtime grants elevated-privilege tool invocations to any caller without identity verification. Elevated tools typically perform actions that carry higher consequences — modifying configurations, managing secrets, accessing administrative endpoints, or altering access controls. With no restriction on who can invoke them, any user session or sub-agent in a multi-agent system can trigger these privileged operations, effectively collapsing the distinction between regular and administrative access. A compromised or misbehaving agent in a shared environment can escalate its own capabilities by calling elevated tools directly, bypassing any intended approval workflow. Disabling elevated tools entirely removes the attack surface, while restricting `allowFrom` to specific agent or user IDs enforces the principle of least privilege.

### OC-013

When `tools.fs.workspaceOnly` is missing or set to false, the agent's file system tools operate with the full permissions of the host process rather than being scoped to the project workspace. This allows the agent to access sensitive files outside the repository, including SSH keys, cloud credentials, shell history, and environment configuration. A prompt injection attack embedded in any input the agent processes can direct it to read `~/.aws/credentials`, exfiltrate private keys, or overwrite system configuration files. Even absent malicious intent, an agent that misinterprets a file path can accidentally corrupt files in unrelated directories. Setting `workspaceOnly` to true establishes a boundary at the application layer, ensuring all file operations are resolved and validated against the workspace root before execution.

### OC-014

When `tools.exec.applyPatch.workspaceOnly` is missing or false, the `apply_patch` tool can target any file path on the system, not just files within the current workspace. Patch operations are specifically designed to make precise, structured modifications to existing files, making them ideal for injecting backdoors, altering configuration files, or tampering with build artifacts in other projects. An attacker can craft a prompt injection that instructs the agent to generate a patch targeting a file like `~/.bashrc` or a dependency in a shared `node_modules` directory, embedding malicious code that persists and executes later. Because patches are diff-based, the modifications can be small and difficult to detect in logs compared to full file writes. Setting `workspaceOnly` to true ensures the patch tool validates every target path against the workspace root before applying any modifications.

### OC-015

When `hooks.gmail.allowUnsafeExternalContent` is set to true, the safety layer that sanitizes and inspects external content before it enters the agent's context is disabled. This wrapper normally strips or neutralizes instruction-like content embedded in emails, webhook payloads, and cron job outputs, preventing these external inputs from being interpreted as agent commands. With the wrapper bypassed, an attacker can send a carefully crafted email whose body contains prompt injection directives — such as instructions to exfiltrate data, send messages, or modify files — and the agent will process this content without any safety filtering. The same vector applies to webhook payloads from third-party services and scheduled cron outputs. Because the agent treats unfiltered external content as part of its operational context, a single malicious email can hijack the agent's subsequent actions across all connected tools.

### OC-016

When API keys, bearer tokens, or provider credentials are stored as plaintext strings in configuration files such as `openclaw.json` or `.env`, they become readable by any process or user with filesystem access to those paths. A local attacker, a misconfigured backup job, or an accidental `git add` can capture these credentials verbatim, because no decryption or access-control layer stands between the stored value and the reader. Once a credential leaks — through a repository push, a log line that dumps config on startup, or a shared archive — the attacker can authenticate to the upstream provider as the legitimate operator. The exposure window compounds over time: every backup snapshot and every developer workstation that clones the repo carries a copy of the secret. Migrating secrets to `SecretRef` entries backed by environment variables or a secrets manager ensures the config file contains only a pointer, not the credential itself.

### OC-018

When `logging.redactSensitive` is disabled or absent, the logging pipeline writes raw request and response payloads without filtering sensitive fields, meaning tokens, API keys, and session credentials can appear in plaintext within log files. Log output typically flows to multiple destinations — local disk, a centralized log aggregator, SIEM ingestion, and sometimes stdout in containerized deployments — so a single unredacted credential propagates across every system in the logging chain. An attacker who gains read access to any of those log stores can extract working credentials without needing to compromise the gateway itself. The risk is amplified by log retention policies: credentials written months ago may still be valid and sitting in an archival tier. Enabling redaction causes the logger to pattern-match known secret formats and replace them with placeholder text before the log line is emitted.

### OC-023

When `plugins.allow` is absent or set to a wildcard while one or more plugins are installed, the gateway loads and executes every plugin it discovers in the plugin directory without verifying that each one has been reviewed and approved. A malicious or compromised plugin receives the same execution context as a trusted one: it can register tool handlers, intercept message flows, read configuration, and make outbound network requests. An attacker who can place a single file in the plugin directory — via a supply-chain compromise, a shared filesystem, or a social-engineering install — gains persistent code execution inside the gateway process. Because plugins run with the gateway's own privileges, the blast radius includes every tool, every channel, and every credential the gateway can reach. Defining an explicit `plugins.allow` list that names only audited plugin identifiers causes the loader to skip any plugin not on the list.

### OC-025

When `dangerouslyAllowPrivateNetwork` is set to true, the agent's browser sandbox can issue HTTP requests to RFC 1918 addresses, link-local ranges, and localhost services that are normally unreachable from the public internet. An attacker exploits this by crafting a prompt that instructs the agent to fetch a URL pointing to an internal service — for example, a cloud metadata endpoint at `169.254.169.254`, an internal admin panel, or a database REST API. The agent's browser makes the request and returns the response body to the conversation context, leaking internal data such as IAM credentials, configuration details, or health-check output. This is a server-side request forgery (SSRF) vector that is especially dangerous because the agent can be manipulated through natural-language prompt injection rather than requiring a traditional code-level vulnerability. Setting the flag to false restores the default deny policy for private-network destinations.

### OC-026

When `sandbox.mode` is set to "off" or left unset while Docker or browser sandbox sub-configuration blocks are present in the config, the gateway ignores all sandbox parameters and executes agent workloads directly on the host. The presence of sandbox configuration creates a false sense of security: an operator reviewing the config sees Docker image names, resource limits, and mount definitions and reasonably concludes that isolation is active, when in reality none of it is enforced. Any tool invocation — file writes, shell commands, browser automation — runs with the gateway process's own privileges and full host access. This gap is particularly dangerous during incident review, because responders may trust the sandbox configuration as a compensating control and underestimate the actual blast radius. Setting `sandbox.mode` to "docker" activates the container isolation layer and causes the gateway to enforce the defined resource limits and mount restrictions.

### OC-027

When `tools.exec.host` is set to `"sandbox"`, the configuration declares the operator's intent to route all command execution into an isolated container. However, this routing declaration depends on `agents.defaults.sandbox.mode` being active to actually create and manage the container runtime. When `sandbox.mode` is `"off"` or absent, the runtime encounters a sandbox routing target with no active sandbox infrastructure, and instead of raising an error, it silently falls back to executing the command directly on the host. The operator sees `host: "sandbox"` in their configuration and reasonably believes commands are contained, but every exec invocation runs with the full privileges of the agent process on the real machine. This silent fallback is more dangerous than having no sandbox configuration at all, because the operator's security assumptions are actively contradicted by the system's behavior without any visible indication.

### OC-030

The `sandbox.browser.allowHostControl` flag grants the browser process running inside a sandbox the ability to issue Chrome DevTools Protocol (CDP) commands that cross the container boundary and execute on the host system. This creates a deliberate escape hatch in the sandbox isolation: the container walls still exist for the agent process, but the browser subsystem can reach through them to the host. The primary exploitation path begins with prompt injection — an attacker embeds instructions in a webpage, document, or message that the sandboxed browser processes, directing the browser to use its CDP host control capability to access host files or services. Because the browser operates inside the sandbox, the operator assumes all browser activity is contained, but `allowHostControl` converts the sandboxed browser into a proxy for host-level operations. Setting the flag to false preserves the sandbox boundary while still allowing in-sandbox browser automation.

### OC-034

This rule detects a compound exposure where three independent configuration weaknesses align: an open or shared group channel accepts messages from any group member, the agent reachable from that channel has runtime execution or filesystem tools enabled, and neither sandbox mode nor `workspaceOnly` is active to contain those tools. The causal chain starts at the channel boundary — anyone who can join the group can send messages that the agent processes as instructions. Those instructions can invoke `tools.exec` for arbitrary command execution or `tools.fs` for unrestricted file access on the host, because no sandbox isolates the execution environment and no workspace restriction limits the filesystem scope. Each axis alone is a concern flagged by other rules (OC-009, OC-011, OC-026), but the three together create a direct, unimpeded path from an untrusted group chat message to full host compromise. Breaking any single axis eliminates the compound exposure.

### OC-044

When `dangerouslyAllowNameMatching` is `true`, the agent resolves user identity by matching the sender's display name against its trusted-user list instead of using unique, immutable platform identifiers such as Slack user IDs. Display names are user-controlled cosmetic labels that any user can change at any time on every major messaging platform — they are not unique, not verified, and not stable. An attacker who wants to impersonate a trusted administrator simply changes their own display name to match the target, and the agent's identity resolution accepts the match as authentic. This means every permission, tool access grant, and trust decision that depends on sender identity can be bypassed by a single profile edit requiring no technical skill. The fix is to remove the flag and migrate `allowFrom` entries to stable platform IDs.

### OC-047

When `responses.files.urlAllowlist` or `responses.images.urlAllowlist` is missing or empty, the agent will fetch content from any URL a request specifies, with no domain restriction. An attacker who can influence the agent's input can instruct it to issue HTTP requests to arbitrary destinations. If the agent runs inside a cloud VPC, those requests can reach internal metadata endpoints, private APIs, and databases that are unreachable from the public internet, turning the agent into an SSRF proxy. The fetched responses — which may contain IAM credentials or internal configuration — are returned through the agent's normal response path. Separately, an attacker can direct the agent to embed file references pointing to attacker-controlled servers, encoding sensitive data from the conversation into URL query parameters that the attacker's server logs on receipt. The absence of an allowlist means both internal SSRF and external data exfiltration succeed with a single crafted prompt.

### OC-048

When `hooks.enabled` is true but `hooks.defaultSessionKey` is not set, inbound webhook payloads have no designated session to land in. If the platform falls back to the system default, hook-injected content shares the same conversation context as an interactive user's session, meaning a webhook payload containing a prompt injection becomes part of the user's active conversation. If the platform instead creates a new session per request, an attacker can flood the webhook endpoint to spawn unbounded sessions, exhausting memory and inference budget. In both cases the root cause is identical: without an explicit session key, there is no convergence point that isolates hook traffic from user traffic. Setting `defaultSessionKey` to a dedicated namespace like `"hook:default"` forces all hook payloads into a single isolated session, preventing cross-session contamination.

### OC-050

When `hooks.allowRequestSessionKey` is set to true, any external caller who can reach the webhook endpoint can include a session key parameter in their payload, directing the agent to process that payload within any named session in the system. An attacker who guesses or enumerates session keys can read conversation history, inject prompt-manipulation instructions, or pre-poison sessions before the legitimate user arrives. The attack is silent — the targeted user receives no indication that their session has been accessed or modified. If the attacker places a prompt injection payload into a session that has access to exec tools, the injection achieves remote code execution through the session's tool surface. Disabling this flag forces all hook payloads to use the fixed `defaultSessionKey`, eliminating the external caller's ability to select which session receives the payload.

### OC-057

When `channels.*.allowFrom` or `groupAllowFrom` contains `"*"`, the agent accepts messages from every user or group on the connected platform with no sender-level filtering. This collapses the authorization boundary from "specific approved users" to "anyone with platform membership," which on a public Discord server or a large Slack workspace can mean thousands of untrusted individuals. Every platform member can invoke the agent's full capability surface — including tool execution, file access, and data retrieval — because the wildcard removes the only gate that distinguishes authorized operators from casual bystanders. The risk escalates sharply when combined with name-based identity matching (OC-044): the wildcard lets anyone reach the agent, and name matching lets that person assume any trusted identity. Replacing the wildcard with explicit stable platform IDs restores per-user access control.

### OC-101

When `gateway.controlUi.allowInsecureAuth` is set to `true`, the Control UI drops its requirement for current-strength authentication and begins accepting weaker or legacy methods such as deprecated token formats or unverified session cookies. This widens the set of credentials that the gateway will treat as valid, meaning an attacker does not need to forge a current-generation token — any credential that satisfies the legacy path is sufficient. Because the Control UI is the administrative interface, a successful authentication through the weakened path grants full management access. The flag is typically introduced during migration or debugging and then left behind. Disabling the flag restores the full authentication stack and ensures only credentials meeting current security requirements are accepted.

### OC-102

When `hooks.token` is set to the same value as `gateway.auth.token`, both the gateway authentication surface and the hooks ingestion endpoint share a single credential. Any context where one token is exposed — a leaked log line, a compromised CI pipeline secret, or a stolen backup — immediately grants access to both surfaces without additional work by the attacker. The gateway token controls administrative access, while the hooks token controls the ability to inject payloads that agents will process. With a single shared value, an attacker who obtains the hooks token from a less-protected webhook integration can authenticate to the gateway as an administrator, and vice versa. Generating a distinct, high-entropy token for each surface ensures that compromise of one does not cascade to the other.

### OC-103

When `hooks.allowedAgentIds` is missing, empty, or set to a wildcard, every agent in the instance is eligible to receive and process incoming hook payloads. Hook payloads typically originate from external systems and carry content that the originating system controls. Without an explicit agent allowlist, a payload intended for a low-privilege automation agent is also delivered to agents that hold elevated tool permissions or control-plane capabilities. An attacker who can submit a crafted hook payload can target the most privileged agent in the instance rather than the intended recipient. Restricting `allowedAgentIds` to the specific agents designed to handle external input confines hook-injected content to the intended processing boundary.

### OC-104

When a configuration key matching the `dangerously*` naming pattern is set to `true` but is not recognized by the current rule set, it indicates that a security bypass flag is active without a corresponding audit check. The `dangerously*` prefix is a deliberate convention signaling that the flag disables or weakens a specific security boundary. Unrecognized keys in this namespace typically originate from copy-paste of example configurations, leftover debugging settings, or version-mismatched config files. Because these flags exist specifically to lower security guarantees, an enabled but unaudited flag means a security boundary is weakened without visibility in scan results. The correct remediation is to set each flagged key to `false` or remove it entirely.

### OC-106

When multiple credential-bearing fields across the configuration tree contain the same value, a single leaked secret exposes every surface that shares it. Each credential field protects a distinct trust boundary with its own exposure profile: some appear in HTTP headers, others in webhook URLs, others in log-adjacent contexts. Reusing the same value collapses these independent boundaries into one: an attacker who extracts the credential from the least-protected location immediately holds valid credentials for every other location. The blast radius of any single credential exposure event scales linearly with the number of fields sharing that value. Generating a unique, high-entropy credential for each field ensures that compromise of one surface does not cascade beyond its own trust boundary.
