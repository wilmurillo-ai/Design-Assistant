# 验收清单详解

本文档详细解释 15 项验收清单的每一项。

---

## A. 需求一致性（3 项）

### A1. 我能用一句话说清这次修复的目标

**含义**：
- 不是"修了几个 bug"
- 而是"修复了 compress() 中 content_preserved 统计时机错误"

**为什么重要**：
- 证明你理解了问题
- 避免"修了 A，顺手把 B 也改了"的过度修改

**如何判断**：
- 如果需要超过 1 句话，说明目标不够清晰
- 如果一句话提到"顺便"、"顺便"、"顺便"，说明可能有非目标修改

---

### A2. 我知道这次"不打算修"的内容有哪些

**含义**：
- 明确边界
- 不把"看起来可以优化"的地方都改了

**示例**：
```
【不修改】
- 不修改 CLI 参数
- 不修改摘要生成逻辑
- 不新增文件
- 不重构其他函数
```

**为什么重要**：
- Bug fix 应该是收敛的
- 避免引入新 bug
- 保持代码审查简单

---

### A3. 代码改动与 issue 描述一致

**含义**：
- issue 说"修复 A bug"
- 代码就只修 A bug
- 没有夹带私货

**如何验证**：
1. 重读 issue 描述
2. 看 diff 的每一行
3. 问自己："这行是为了解决 issue 吗？"

**警告信号**：
- Diff 里出现了 issue 没提到的函数
- Diff 里出现了重构风格的改动
- Diff 里出现了"优化"字样

---

## B. 技术正确性（4 项）

### B1. 我基于正确版本开始修改

**含义**：
- 从官方最新 release 或指定版本开始
- 不是从"我本地改了一半的版本"开始

**如何验证**：
```bash
# 检查起始版本
git log --oneline | head -5
git describe --tags
```

**错误示例**：
- 从上次没提交的改动继续改
- 从别人的分支继续改
- 从旧版本继续改

---

### B2. 我没有重写整个文件

**含义**：
- 保留原有代码结构
- 只改必要的部分

**如何判断**：
```bash
# 检查改动比例
git diff --stat

# 如果一个文件改动超过 50%，要警惕
```

**正确做法**：
- 复制旧代码 → 修改 → 替换
- 不是：凭记忆重写

---

### B3. 数据结构变化已同步所有引用点

**含义**：
- 修改了数据结构（类、dict、tuple）
- 必须检查所有使用这个数据结构的地方

**示例**：
```python
# 修改前
stats = {
    'kept': len(kept),
    'removed': len(removed)
}

# 修改后（新增字段）
stats = {
    'kept': len(kept),
    'removed': len(removed),
    'content_preserved': len(kept_older) + len(kept_recent)  # 新增
}

# 必须检查
- 所有读取 stats 的地方
- 所有写入 stats 的地方
- 所有序列化/反序列化的地方
```

**如何验证**：
```bash
# 搜索所有使用点
grep -r "stats\[" .
grep -r "stats\." .
grep -r "stats\.get" .
```

---

### B4. 新逻辑不会破坏旧逻辑

**含义**：
- 新代码不会影响已有功能
- 不引入副作用

**如何验证**：
1. 运行已有的测试用例
2. 手动测试旧功能
3. 检查新代码的输入输出范围

**示例**：
```python
# 旧逻辑
def compress(text):
    return text[:100]

# 新逻辑（错误）
def compress(text):
    return text[:100] + "..."  # 增加了输出长度！

# 新逻辑（正确）
def compress(text):
    return text[:100]  # 行为不变
```

---

## C. 测试验证（4 项）

### C1. 语法检查通过

**命令**：
```bash
python3 -m py_compile scripts/xxx.py
```

**为什么重要**：
- 最基本的验证
- 能发现括号不匹配、缩进错误等

---

### C2. 导入检查通过

**命令**：
```bash
python3 -c "from scripts.xxx import ClassName; print('ok')"
```

**为什么重要**：
- 验证模块依赖正确
- 验证 import 路径正确

---

### C3. 最小样例验证通过

**含义**：
- 构造最小输入
- 验证修复点
- 验证输出正确

**示例**：
```python
# 测试修复：content_preserved 统计
def test_content_preserved():
    # 最小输入
    messages = [
        {"role": "user", "content": "Hello"},
        {"role": "assistant", "content": "World"}
    ]
    
    # 调用
    result = compress(messages)
    
    # 验证
    assert result['content_preserved'] == 2
    assert 'kept' in result
    assert 'removed' in result
```

---

### C4. 回归测试通过

**含义**：
- 旧功能仍然正常
- 没有破坏已有行为

**如何验证**：
1. 运行项目已有的测试套件
2. 手动测试核心流程
3. 检查边界情况

---

## D. 发布质量（4 项）

### D1. diff 大小与任务规模匹配

**预期**：

| 任务类型 | 预期改动量 |
|---------|-----------|
| 修 1 个小 bug | 5–30 行 |
| 修 1 组相关 bug | 20–80 行 |
| 小功能新增 | 30–150 行 |
| **超过 200 行** | **必须怀疑是否改多了** |

**如何验证**：
```bash
git diff --stat
git diff --shortstat
```

---

### D2. release note 与实际代码一致

**含义**：
- release note 说"修复了 5 个 bug"
- diff 里必须能找到 5 个对应的修复

**如何验证**：
1. 逐条对照 release note 和 diff
2. 每个"修复"都能在 diff 里找到对应代码
3. 没有暗改

---

### D3. 版本号、文档、注释已同步

**检查清单**：
- [ ] 代码中的版本号已更新
- [ ] README.md 已更新（如果有功能变化）
- [ ] CHANGELOG.md 已更新
- [ ] 代码注释已更新（如果有接口变化）
- [ ] 文档已更新（如果有使用方式变化）

---

### D4. 我可以指出这次改动的风险点

**含义**：
- 知道哪里可能出问题
- 有应对方案

**示例**：
```
【风险点】
1. 数据结构变化可能影响外部调用方
   - 应对：检查所有使用点，确保兼容
   
2. 新逻辑依赖全局状态
   - 应对：添加初始化检查
   
3. 修改了核心函数的输入输出
   - 应对：保持向后兼容，添加参数校验
```

---

## 总结

**⚠️ 这 15 项只要有 1 项答不上来，就不要发布。**

**为什么这么严格？**
- 避免返工
- 保证质量
- 建立信心

**如果某项答不上来怎么办？**
1. 停下
2. 补充验证
3. 重新检查
4. 都清楚了再发布
