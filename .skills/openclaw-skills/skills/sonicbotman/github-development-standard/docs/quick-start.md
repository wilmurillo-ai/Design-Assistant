# 快速开始

---

## 📋 5 分钟上手

### 1. 了解核心理念（1 分钟）

> **一句话记住：先定义问题，再定义改法，再写代码，再做验证，最后才发布。**

### 2. 掌握 9 步流程（2 分钟）

| 步骤 | 动作 | 时间 |
|------|------|------|
| 1 | 读 issue | 5 分钟 |
| 2 | 写"5行任务卡" | 5 分钟 |
| 3 | 确定基线版本 | 2 分钟 |
| 4 | 列改动点 | 10 分钟 |
| 5 | 编码 | 实际开发时间 |
| 6 | 本地验证 | 10-30 分钟 |
| 7 | 看 diff | 5 分钟 |
| 8 | 写发布说明 | 10 分钟 |
| 9 | 最后复盘 | 5 分钟 |

### 3. 记住 4 层验证（1 分钟）

```
1. 语法验证 (py_compile)
2. 导入验证 (import)
3. 行为验证 (最小样例)
4. 回归验证 (旧功能)
```

### 4. 使用 15 项清单（1 分钟）

**发布前必须回答：**
- A. 需求一致性（3项）
- B. 技术正确性（4项）
- C. 测试验证（4项）
- D. 发布质量（4项）

**⚠️ 只要有 1 项答不上来，就不要发布。**

## 🚀 第一个任务

### 场景：修复一个 Bug

```text
【任务类型】Bug 修复
【问题】compress() 中 summary 变量被覆盖
【目标】改用列表收集，避免覆盖
【边界】只修改 compress() 函数
【非目标】不修改 CLI、不重构其他函数
```

### 执行步骤

```bash
# Step 1: 读 issue
# 理解问题：summary 变量在循环中被覆盖

# Step 2: 写"5行任务卡"
# 已完成

# Step 3: 确定基线版本
git checkout v1.2.4
cp scripts/xxx.py scripts/xxx_fixed.py

# Step 4: 列改动点
# - 第 148 行：改用列表
# - 第 160 行：合并列表

# Step 5: 编码
# 复制旧代码 → 修改 → 替换

# Step 6: 本地验证
python3 -m py_compile scripts/xxx_fixed.py
python3 -c "from scripts.xxx_fixed import compress"
python3 test_fix.py

# Step 7: 看 diff
git diff --stat scripts/xxx_fixed.py
# 预期：5-30 行

# Step 8: 写发布说明
git commit -m "fix: 修复 summary 变量覆盖问题"

# Step 9: 复盘
# 记录经验教训
```

## ✅ 验收检查

发布前，检查以下 15 项：

### A. 需求一致性
- [ ] 我能用一句话说清这次修复的目标
- [ ] 我知道这次"不打算修"的内容有哪些
- [ ] 代码改动与 issue 描述一致

### B. 技术正确性
- [ ] 我基于正确版本开始修改
- [ ] 我没有重写整个文件
- [ ] 数据结构变化已同步所有引用点
- [ ] 新逻辑不会破坏旧逻辑

### C. 测试验证
- [ ] 语法检查通过
- [ ] 导入检查通过
- [ ] 最小样例验证通过
- [ ] 回归测试通过

### D. 发布质量
- [ ] diff 大小与任务规模匹配
- [ ] release note 与实际代码一致
- [ ] 版本号、文档、注释已同步
- [ ] 我可以指出这次改动的风险点

## 🎯 常见问题

### Q1: 我可以直接开始写代码吗？

**A: 不可以。** 必须先完成 Step 1-4，理解问题、明确目标、确定边界。

### Q2: 如果改动量超过 200 行怎么办？

**A: 需要怀疑自己。** 问自己：
- 是否改多了？
- 是否夹带了重构？
- 是否可以拆分任务？

### Q3: 15 项验收清单有 1 项答不上来怎么办？

**A: 不要发布。** 停下，补充验证，直到全部答上来。
