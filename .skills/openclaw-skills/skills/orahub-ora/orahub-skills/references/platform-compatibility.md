# Shared Platform Compatibility + OraHub Reference

This reference file defines the shared platform compatibility rules and shared OraHub CLI rules used by all OraHub image skills in this directory.

## Skill File Constraints

OpenClaw skill frontmatter rules used in this repository:

- `name` is required
- `description` is required
- If `metadata` is used, it must be written as single-line JSON
- Do not rely on multiline YAML objects for OpenClaw-specific metadata

## Inbound Media

Inbound media capabilities available on the OpenClaw platform:

- Inbound attachments may be exposed as `{{MediaPath}}` and `{{MediaUrl}}`
- `{{MediaPath}}` is the local temporary file path written by OpenClaw
- `{{MediaUrl}}` is the pseudo-URL generated by OpenClaw for inbound media
- `{{Transcript}}` is only relevant to audio transcription and does not matter for OraHub image skills

Inbound media limitations:

- By default, OpenClaw only guarantees the first matching inbound image attachment
- Do not assume multiple attachments will always be exposed unless that capability is explicitly enabled in the deployment
- If the same attachment exposes both `{{MediaPath}}` and `{{MediaUrl}}`, prefer `{{MediaPath}}`

## Outbound Media

OpenClaw only returns media when the assistant reply contains a standalone line in this exact format:

```text
MEDIA:<path-or-url>
```

Rules:

- Produce the result file first
- Then send a short text reply
- Put `MEDIA:<output_path_or_url>` on its own line
- If the reply does not contain `MEDIA:`, do not claim that the image was actually sent

If outbound `MEDIA:` delivery fails, common reasons include unreadable files or channel rejection. In that case:

- keep the text reply short
- include the local output path in text when necessary

## Memory

OpenClaw supports disk-based memory, but the skill must use the real storage model instead of inventing its own memory variables.

The real OpenClaw memory model:

- Long-term memory is stored in `MEMORY.md`
- Daily notes are stored in `memory/YYYY-MM-DD.md`
- `MEMORY.md` is loaded at the start of each DM session
- `memory_search` and `memory_get` are provided by the currently enabled memory plugin

Memory rules for OraHub image skills:

- Do not use fake variables such as `memory["x"]`
- Do not automatically persist temporary inbound attachment paths such as `{{MediaPath}}`
- Do not automatically persist OpenClaw pseudo-URLs such as `{{MediaUrl}}`
- Only persist a reusable reference when the user explicitly asks to remember it and it is a stable local path or stable `http(s)` URL
- If no stable reference was explicitly persisted, ask the user for the input again

## Shared OraHub CLI Rules

Expected when the current machine already has a working OraHub runtime:

- `orahub --version` is available
- `orahub auth verify --json` returns structured JSON
- OraHub image commands support both local file paths and `http(s)` URLs
- The current image workflows in this package use these commands:
  - `orahub photo-color-match`
  - `orahub photo-background-replace`
  - `orahub photo-passersby-removal`
  - `orahub photo-remove-background`

Runtime bootstrap policy for this package:

- The router only routes; it does not run standalone runtime or auth checks
- The selected leaf workflow should try its own workflow command with the current local runtime first
- Do not mutate the environment silently. Any install, upgrade, or auth step must be executed by the agent only after explicit user approval
- If `orahub` is missing or outdated, the agent should request approval to install or upgrade the runtime, then retry the workflow automatically
- If credentials are missing or invalid, the agent should request approval to start authentication, then retry the workflow automatically
- Only fall back to manual setup guidance when the current client cannot execute commands, cannot support the auth flow, or the user denies approval

Shared exit codes:

- `0` success
- `1` usage error
- `2` configuration error
- `3` authentication error
- `4` request error
- `5` task failure
- `6` timeout

When credentials are missing, the real `auth verify` failure shape is typically handled with device login first, with manual key setup as a fallback:

```json
{
  "ok": false,
  "exitCode": 2,
  "error": "CLI credentials not configured, run `orahub auth device-login` first, or use `orahub config set --access-key <ak> --secret-key <sk>` as a manual fallback",
  "canRetry": false
}
```

Shared handling rules:

- Do not require the router to run `orahub --version` or `orahub auth verify --json` before routing
- If a leaf workflow command fails with command-not-found, unknown-command, or similar runtime mismatch output, treat it as runtime unavailable or outdated; request user approval to run `npm install -g orahub-cli`, verify with `orahub --version`, then retry the same workflow once
- If a leaf workflow command exits with `2`, treat it as "not configured"; request user approval to run `orahub auth device-login` (primary), then retry the same workflow once. If the user prefers API keys, collect credentials and use `orahub config set --access-key <ak> --secret-key <sk>` instead
- If a leaf workflow command exits with `3`, treat it as "authentication failed"; request user approval to re-run `orahub auth device-login` or reconfigure with `orahub config set`, then retry the same workflow once
- If command execution or auth is not possible in the current client, or the user denies approval, return the equivalent manual commands and stop
- If a command exits with `4` and includes `ENOENT`, treat it as "local input path validation failed"
- If a command exits with `0` but no output file is produced, treat it as "execution failed"

## Platform-Side Error Boundary

The following should be treated as OpenClaw platform-side issues, not OraHub-side issues:

- the message exposes no attachment and no explicit path or URL
- the skill needs two images but only one image attachment is available
- `{{MediaPath}}` points to a missing temporary file
- `{{MediaUrl}}` is no longer usable
- outbound `MEDIA:` delivery fails

Recommended handling:

- ask for the missing image input
- ask the user to resend the image
- ask the user for a stable local path or `http(s)` URL
