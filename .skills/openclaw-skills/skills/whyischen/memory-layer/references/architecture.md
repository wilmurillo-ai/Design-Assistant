# 架构设计

## 设计哲学

基于 Claude Code 记忆哲学，适配 OpenClaw 基础设施。

---

## 5 核心原则

| 原则 | 说明 | OpenClaw 适配 |
|------|------|---------------|
| **带宽感知** | 上下文窗口是稀缺资源 | 飞书有消息限制，进一步优化 |
| **写纪律** | 先写内容，再更新 Index | 现有 Cron 写 Topic，再更新 MEMORY.md |
| **验证优先** | 记忆是提示，不是真相 | 投资数据从 API 获取，仅存储规则 |
| **职责分离** | Index/Topic/Transcript 职责明确 | 保持现有结构，调整内容规范 |
| **自动整理** | autoDream 夜间去重/合并 | 使用现有 Cron，无需新基础设施 |

---

## 第一层：Index

**位置**：`MEMORY.md`（工作区根目录）

**职责**：
- 存储 Topic 元数据（名称、路径、更新时间、标签）
- 全局标签索引（标签 → Topic 映射）
- LRU 缓存（最近 Top 5）
- 系统统计（上次整理时间、总数）

**约束**：
- 硬限制：≤25KB
- 行限制：≤150 字符
- 超限处理：归档最旧 20% 到 `memory/index-archive.md`

**加载策略**：始终加载到上下文

---

## 第二层：Topic

**位置**：`memory/topics/*.md`

**职责**：
- 结构化领域知识
- 用户偏好、决策规则、长期事实
- 可复用的模板和模式

**约束**：
- 建议：≤50KB/文件
- 超限处理：拆分或归档旧内容

**加载策略**：按需加载（每次会话 2-5 个）

---

## 第三层：Transcript

**位置**：`memory/transcripts/YYYY-MM/*.log`

**职责**：
- 原始对话日志
- 临时、上下文特定信息
- Topic 层的源数据

**约束**：
- 限制：500KB/文件（按日分割）
- 保留：90 天完整，之后压缩归档

**加载策略**：永不加载，仅 grep/语义搜索

---

## 数据流

```
用户请求 / Cron 触发
         ↓
写入 Transcript（完整日志）
         ↓
提取关键事实 → 更新 Topic
         ↓
压缩摘要 → 更新 Index
         ↓
夜间 autoDream 整理
```

## 重要性评分系统

每条 Topic 和 Transcript 都有重要性分数 (0.0-1.0)，用于智能排序。

### 公式

```
importance = domainBase + accessBonus + userBonus

其中：
- domainBase：领域基础权重（投资 0.8，项目 0.7，知识 0.5）
- accessBonus：访问频率奖励（最高 +0.2）
- userBonus：用户标记奖励（最高 +0.3）
```

### 领域基础权重

| 领域 | 基础权重 | 理由 |
|------|----------|------|
| 投资 | 0.8 | 高频，决策影响大 |
| 项目 | 0.7 | 持续跟踪，目标明确 |
| 资产 | 0.7 | 长期持有，变化少 |
| 知识 | 0.5 | 静态参考，按需查询 |
| 健康 | 0.6 | 个人重要，更新少 |
| 社交 | 0.4 | 临时记录，重要性低 |

## 矛盾处理

当检测到同一事实的冲突描述时：

1. **自动标记**：在 Topic 的"待解决问题"部分添加矛盾标记
2. **不强制解决**：保留两个来源，等待用户确认
3. **记录检测时间**：便于追溯

### 标记格式

```markdown
## 待解决问题

- [ ] ⚠️ **矛盾标记** (2026-04-02T23:00:00)
  - 来源 1：VIX 阈值 25 (计划 v1.0, 2026-03-25)
  - 来源 2：VIX 阈值 30 (计划 v2.0, 2026-04-02)
  - 建议：请确认哪个版本正确
```
