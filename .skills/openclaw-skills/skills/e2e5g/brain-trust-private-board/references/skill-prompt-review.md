# Skill Prompt Workflow 评审

这是一个专属高频场景。遇到 skill、prompt、workflow、agent 设计时，优先按这份规则评审，而不是只做通用分析。

## 适用对象

- `SKILL.md`
- system prompt
- workflow prompt
- agent/router 设计
- 多步骤任务编排
- 技能参考文件结构

## 推荐模式

- 首选：`critique`
- 需要改法时：`critique` 后接 `synthesis`
- 问题跨度很大时：`panel`

## 推荐专家组合

### 基础组合

- 理查德·费曼：看是否说得清、触发是否直白、使用者是否真能理解
- 查理·芒格：看有没有误判、过度复杂、结构性坏味道
- 彼得·德鲁克：看目标、职责边界、流程落地性

### 需要补的专家

- 唐纳德·诺曼：当关注可用性、调用体验、是否顺手
- 丹尼尔·卡尼曼：当担心提示词诱导偏差、错误自信
- 多内拉·梅多斯：当 workflow 存在系统性回路或耦合问题

## 评审维度

至少检查下面 8 项：

1. 触发是否准确
2. 目标是否单一清楚
3. 流程是否真能执行
4. 信息拆分是否合理
5. 资源文件是否真的有用
6. 输出是否能收束
7. 是否容易误触发或漏触发
8. 是否存在过度包装、过度模仿、过度复杂

## 常见问题清单

### 1. 触发描述太泛

症状：

- 什么都能触发
- 也就等于什么都没说明白

改法：

- 写清楚任务类型、典型输入、使用边界

### 2. 流程写得像宣言，不像动作

症状：

- 很多抽象原则
- 缺少明确执行顺序

改法：

- 改成先做什么，再做什么，什么时候读哪个文件

### 3. 参考文件很多，但没有路由

症状：

- 文件看起来丰富
- 真用时不知道先看哪一个

改法：

- 在主 SKILL.md 明确“先看什么，何时再看什么”

### 4. 输出只有过程，没有收束

症状：

- 讲了一堆分析
- 没有最终建议、优先级或下一步

改法：

- 强制补“推荐路径 / 下一步动作 / 不建议做什么”

### 5. 把风格当能力

症状：

- 很会模仿口吻
- 但没提升判断质量

改法：

- 先评估是否真的改变了判断框架

### 6. 技能太聪明，反而不好用

症状：

- 覆盖范围特别大
- 但触发边界模糊，容易失焦

改法：

- 缩小核心用途，给高频任务直达入口

## 推荐交付格式

评审 skill/prompt/workflow 时，优先输出：

1. 整体判断
2. 主要问题
3. 为什么这是问题
4. 影响范围
5. 优先修改项
6. 建议改法
7. 如果重构，建议的结构

## 最好用的调用句式

```text
用 expert-thinking，评审这份 SKILL.md。重点看触发准确性、流程可执行性、参考文件路由、输出收束能力和真实可用性。
```

```text
用 expert-thinking，评审这个 prompt/workflow。先找结构问题和误判风险，再给我一个更稳的重构方向。
```
