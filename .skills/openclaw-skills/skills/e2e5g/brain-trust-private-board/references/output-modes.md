# 输出模式与模板

## 模式总览

### `single`

适用：

- 用户点名某位专家
- 需要快速、清晰、单线条判断

建议结构：

1. 问题重述
2. 该专家会先看什么
3. 核心判断
4. 建议动作
5. 风险提醒

### `compare`

适用：

- 想看两种或三种方法论的差异
- 用户卡在不同路径之间

建议结构：

1. 比较维度
2. 专家 A 观点
3. 专家 B 观点
4. 一致处
5. 冲突处
6. 该怎么选

### `debate`

适用：

- 问题存在天然对立立场
- 需要把方案放进冲突环境里测试

建议结构：

1. 争议焦点
2. 正方专家观点
3. 反方专家观点
4. 交锋后的核心分歧
5. 裁决或折中方案

### `panel`

适用：

- 问题跨产品、战略、组织、风险多个维度
- 单专家容易偏科

建议结构：

1. 会诊目标
2. 每位专家的关键意见
3. 共识
4. 少数意见
5. 综合结论

### `boardroom`

适用：

- 用户希望像开私董会
- 复杂问题需要自然接话、追问、反驳和脑暴
- 不想看固定轮次发言
- 讨论到出现清晰结论就可以停止

建议结构：

1. 议题与约束
2. 私董会讨论摘要
3. 关键碰撞点
4. 形成的结论
5. 下一步动作

讨论规则：

1. 不预设固定发言顺序。
2. 谁最适合当前话题，谁先说。
3. 下一位只在有补充、挑战、澄清、延展或收束需要时接话。
4. 允许同一位专家多次发言，也允许有人只在关键节点插话。
5. 讨论到结论清楚就停，不追求平均轮次。

### `critique`

适用：

- 已有方案、文稿、功能草案或商业计划
- 重点是找问题，不是从零写新方案

建议结构：

1. 评审标准
2. 主要问题
3. 缺失项
4. 修改建议
5. 优先级

### `red-team`

适用：

- 方案即将投入资源
- 需要优先暴露失败路径

建议结构：

1. 方案最脆弱的假设
2. 最可能失败的路径
3. 一旦失败会出现什么信号
4. 如何提前设置保护栏
5. 是否值得继续推进

### `synthesis`

适用：

- 最终目标是得到统一建议
- 前面已经存在多个视角

建议结构：

1. 问题框架
2. 多专家观察
3. 共同结论
4. 分歧与取舍
5. 推荐路径
6. 下一步动作

## 自然语言风格

- 优先清晰、具体、可执行。
- 风格模仿只保留少量“辨识度”，不要影响信息密度。
- 不要连续堆砌名言，也不要把回答写成舞台剧。
- `boardroom` 模式下允许保留简短对话感，但不要写成长篇剧本。

## 结构化输出模板

```json
{
  "mode": "boardroom",
```

## 默认输出强度

- 简单问题：单专家，短答，直接给判断。
- 中等复杂问题：优先私董会式动态讨论，再给综合建议。
- 高风险复杂问题：至少加入一个红队或风控型专家。

## 交付前检查

- 模式是否真的匹配任务
- 专家差异是否有体现
- 综合结论是否明确
- 风险和代价是否说清楚
- 下一步动作是否能直接执行
