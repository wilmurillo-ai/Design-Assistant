# 测试用例优先级判断指南

## 优先级定义

### 高优先级（P0 - 必须执行）

**定义：** 涉及核心功能、患者安全、数据完整性的测试用例，必须 100% 执行。

**判断标准：**
- ✅ 直接影响患者安全
- ✅ 影响检测结果准确性
- ✅ 导致数据丢失或损坏
- ✅ 核心业务流程阻塞
- ✅ 法规强制要求的功能

**典型场景：**
| 场景 | 示例 |
|------|------|
| 试剂量计算 | 缓冲液/系统液消耗计算 |
| 结果判定 | 阳性/阴性判断逻辑 |
| 样本追踪 | 样本 ID 识别和追踪 |
| 安全控制 | 电气安全、机械安全 |
| 数据完整性 | 数据存储、传输、备份 |

**执行要求：**
- 变更涉及相关模块时必须执行
- 不允许跳过或延后
- 失败时必须修复后才能发布

---

### 中优先级（P1 - 建议执行）

**定义：** 涉及重要功能、用户体验的测试用例，建议执行，时间紧张时可酌情裁剪。

**判断标准：**
- ✅ 影响常用功能
- ✅ 影响用户体验
- ✅ 影响与其他系统交互
- ✅ 报警和提示功能

**典型场景：**
| 场景 | 示例 |
|------|------|
| 常用功能 | 样本管理、试剂管理 |
| 接口功能 | LIS 通信、数据导出 |
| 报警功能 | 耗材不足、错误提示 |
| 权限管理 | 用户登录、权限控制 |
| 历史数据 | 数据查询、统计报表 |

**执行要求：**
- 版本发布前建议执行
- Bug 修复后应执行相关用例
- 时间紧张时可评估风险后裁剪

---

### 低优先级（P2 - 可选执行）

**定义：** 涉及边缘功能、界面展示的测试用例，可根据时间情况选择性执行。

**判断标准：**
- ✅ 界面展示相关
- ✅ 辅助功能
- ✅ 可选配置项
- ✅ 不影响核心功能

**典型场景：**
| 场景 | 示例 |
|------|------|
| 界面展示 | 字体、颜色、布局 |
| 辅助功能 | 帮助文档、关于页面 |
| 配置选项 | 主题设置、语言切换 |
| 提示信息 | 提示文字措辞 |

**执行要求：**
- 时间充裕时执行
- 界面变更时应执行相关用例
- 可延后到后续版本

---

## 优先级调整规则

### 规则 1：患者安全优先

任何涉及患者安全的用例，优先级自动提升为**高**。

**判断关键词：**
- 患者、安全、剂量、浓度、结果、检测、阳性、阴性

### 规则 2：Bug 修复加强

Bug 修复类变更，相关高优先级用例必须执行，不允许降级。

**判断关键词：**
- 修复、fix、bug、问题、错误

### 规则 3：界面变更降级

纯界面变更，非 UI 测试用例可适当降级。

**判断关键词：**
- 界面、UI、样式、颜色、布局

### 规则 4：历史缺陷加强

曾经出过问题的模块，测试优先级提升一级。

**参考依据：**
- 历史缺陷记录
- 客户反馈问题
- 现场故障报告

---

## 优先级决策矩阵

| 变更类型 | 患者安全 | 数据完整 | 核心功能 | 重要功能 | 边缘功能 |
|----------|----------|----------|----------|----------|----------|
| **新功能** | 高 | 高 | 高 | 中 | 低 |
| **Bug 修复** | 高 | 高 | 高 | 中 | 中 |
| **重构** | 高 | 高 | 中 | 中 | 低 |
| **界面变更** | 中 | 低 | 中 | 中 | 低 |
| **性能优化** | 高 | 中 | 中 | 中 | 低 |

---

## 裁剪规则

### 时间充足（100% 时间）

- 高优先级：100% 执行
- 中优先级：100% 执行
- 低优先级：100% 执行

### 时间紧张（50% 时间）

- 高优先级：100% 执行
- 中优先级：50% 执行（选择核心场景）
- 低优先级：不执行

### 时间非常紧张（20% 时间）

- 高优先级：100% 执行（选择最关键用例）
- 中优先级：不执行
- 低优先级：不执行

## 最佳实践

1. **早识别** - 变更评审时即识别测试范围
2. **早沟通** - 与开发、产品确认优先级
3. **留记录** - 裁剪决策必须记录并审批
4. **可追溯** - 测试用例与需求/变更关联
5. **常回顾** - 定期回顾裁剪决策的合理性
