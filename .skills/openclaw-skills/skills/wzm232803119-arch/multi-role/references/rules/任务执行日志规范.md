# 任务执行日志规范

> **定位**: 全局规则，所有执行角色（研发、测试、产品经理）强制遵守
> **版本**: v1.1
> **最后更新**: 2026-03-16
> **维护人**: 大管家

---

## 一、为什么需要任务执行日志

> 记忆消耗 Token，日志不消耗 Token。出了问题先查日志，查不到再查记忆。

没有任务执行日志会导致：
- Bug 出现后只能读全量上下文找错误现场，浪费大量 Token
- 研发修复 Bug 时没有错误堆栈和复现步骤，只能靠"记忆"猜，容易走弯路
- 同一个 Bug 重复出现，没有历史记录可查
- 测试无法提供精确的错误上下文给研发，沟通成本高

---

## 二、日志分层架构

```
任务执行日志（本规范覆盖范围）
├── 任务工作日志        → sessions/{项目}/task-logs/YYYY-MM-DD_T{任务ID}_work.md
├── Bug 错误现场日志    → sessions/{项目}/task-logs/YYYY-MM-DD_T{任务ID}_bug.md
└── 修复记录日志        → sessions/{项目}/task-logs/YYYY-MM-DD_T{任务ID}_fix.md

系统运行日志（由日志管理规范覆盖）
├── 服务运行日志        → project/{项目}/logs/server.log
└── 错误日志            → project/{项目}/logs/error.log

角色错误日志（由问题分级制度覆盖）
└── 角色 P0/P1 错误    → role-extras/{角色名}-错误记录.md
```

---

## 三、日志类型与格式

### 3.1 任务工作日志（研发必写）

**触发时机**: 每次开始执行一个任务（T001、T002 等）时创建，完成时更新。

**存放位置**: `sessions/{当前项目}/task-logs/YYYY-MM-DD_T{任务ID}_work.md`

**模板**:

```markdown
# 任务工作日志 - T{任务ID}: {任务标题}

> 执行人: 研发 | 创建: YYYY-MM-DD HH:MM | 最后更新: YYYY-MM-DD HH:MM

## 任务描述
{一句话描述任务内容，来自 inbox.md 或 task-board.md}

## 执行步骤记录

### 步骤1: {步骤描述}
- 操作: {做了什么}
- 结果: ✅ 成功 / ❌ 失败
- 备注: {遇到什么问题或注意点}

### 步骤2: {步骤描述}
...

## 输出物
- 文件: {修改/新增的文件路径}
- 接口: {新增/修改的接口列表（如有）}

## 完成状态
- [ ] 进行中
- [x] 已完成
- [ ] 阻塞（原因: ）

## 遗留问题
{如有未解决的问题，列在这里，后续写入 pitfalls.md}
```

---

### 3.2 Bug 错误现场日志（测试必写，研发补充）

**触发时机**: 测试发现 Bug 时立即创建，记录完整的错误现场。

**存放位置**: `sessions/{当前项目}/task-logs/YYYY-MM-DD_T{任务ID}_bug.md`

**模板**:

```markdown
# Bug 错误现场日志 - T{任务ID}: {Bug标题}

> 发现人: 测试 | 发现时间: YYYY-MM-DD HH:MM | Bug级别: P0/P1/P2/P3

## Bug 描述
{一句话描述现象}

## 复现步骤
1. {步骤1}
2. {步骤2}
3. {步骤3}

## 期望行为
{应该是什么结果}

## 实际行为
{实际出现了什么结果}

## 错误信息（完整粘贴）
```
{错误堆栈 / 报错信息 / 异常输出}
```

## 环境信息
- 操作系统:
- 浏览器/终端:
- 相关版本:
- 数据状态: {触发时的数据条件，如"数据库为空"、"有3条记录"等}

## 截图/日志文件路径
- {如有截图或日志文件，记录绝对路径}

## 初步判断（研发填写）
- 可能原因: {一句话猜测}
- 影响范围: {哪些功能受影响}
- 修复难度: 低/中/高
```

---

### 3.3 修复记录日志（研发必写）

**触发时机**: Bug 修复完成后，补充到对应的 Bug 日志文件末尾，或单独创建。

**存放位置**: 追加到对应 `_bug.md` 末尾，或新建 `YYYY-MM-DD_T{任务ID}_fix.md`

**模板**:

```markdown
## 修复记录（追加到 Bug 日志末尾）

> 修复人: 研发 | 修复时间: YYYY-MM-DD HH:MM | 修复耗时: X小时

### 根本原因
{一句话说明根本原因，不是表象而是根因}

### 修复方案
{说明修了什么，为什么这样修}

### 修改的文件
- `{文件路径}`: {修改内容一句话描述}
- `{文件路径}`: {修改内容一句话描述}

### 验证方法
{如何验证修复有效，复现步骤是否不再触发}

### 是否需要写入 pitfalls.md
- [ ] 否（此 Bug 是一次性问题，不会再出现）
- [ ] 是 → 已写入 sessions/{项目}/pitfalls.md 第 P{序号} 条
```

## 四、日志读取规则（研发上岗时强制执行）

研发接到任务时，按以下顺序读取，**不得跳过**：

```
Step 1: 读 sessions/{项目}/context.md           → 了解项目现状
Step 2: 读 sessions/{项目}/pitfalls.md           → 检查是否有相关踩坑记录
Step 3: 读 sessions/{项目}/task-logs/            → 查看同类任务的历史执行记录
Step 4: 如果是 Bug 修复 → 读对应的 _bug.md      → 拿到完整错误现场
Step 5: 开始执行，同步创建任务工作日志
```

> ❌ 禁止不读 pitfalls.md 和 task-logs/ 就开始修复 Bug——这是最常见的重复踩坑路径

## 五、日志存放目录规范

```
sessions/{项目名}/
├── context.md
├── decisions.md
├── pitfalls.md
├── inbox.md
└── task-logs/                          ← 任务执行日志目录（新增）
    ├── YYYY-MM-DD_T001_work.md         ← 任务工作日志
    ├── YYYY-MM-DD_T002_bug.md          ← Bug 错误现场日志
    ├── YYYY-MM-DD_T002_fix.md          ← 修复记录（或追加到 _bug.md）
    └── YYYY-MM-DD_T003_work.md
```

## 六、日志保留与淘汰规则

| 日志类型 | 保留时间 | 淘汰方式 |
|---------|---------|---------|
| 任务工作日志（_work.md） | 7 天 | 迭代收尾时清理超 7 天的文件 |
| Bug 错误现场日志（_bug.md） | 30 天 | 30 天后移至 `archives/bugs/` |
| 修复记录日志（_fix.md） | 30 天 | 与 _bug.md 一起处理 |
| 写入 pitfalls.md 的坑 | 至解决 | 标记 [已解决] 后下次 review 清理 |

> **关键原则**: 日志只保留"能帮助快速定位问题"的时间窗口，过期的迁移归档而非删除。

## 七、角色职责分工

| 角色 | 职责 |
|------|------|
| 研发 | 创建任务工作日志、填写修复记录、判断是否写入 pitfalls.md |
| 测试 | 创建 Bug 错误现场日志，提供完整的错误堆栈和复现步骤 |
| 大管家 | 分配任务时确认研发已读 task-logs/，迭代收尾时清理超期日志 |
| 产品经理 | 无需写日志，但 Bug 分级时可参考 _bug.md 中的错误现场 |

## 八、与其他规范的关系

| 规范 | 关系 |
|------|------|
| 日志管理规范 | 本规范是其补充，覆盖「任务执行」这一层；日志管理规范覆盖「系统运行」层 |
| 记忆管理规范 | 任务日志是记忆的替代品：能用日志查到的，不走 memory_search |
| 问题分级制度 | Bug 日志是 P0/P1 错误记录的原始来源；修复后的教训写入 pitfalls.md |
| Bug 排查修复流程 | 本规范为该 SOP 提供日志写作的具体格式和位置 |

## 九、Phase 级实时落笔规则（v1.1 新增）

> 解决「任务中途中断导致日志为空」的根本问题。

**规则**：每个 Phase 完成后，**立即**在 `_work.md` 中打勾并写一行摘要，不等任务整体完成。

**格式**（追加到「执行步骤记录」下）：

```
- [x] Phase N 完成 — {一句话说明交付物} ({HH:MM})
```

**强制时机**：
- Phase 内任何子步骤完成后 → 打勾对应 checkbox
- 检测到响应时间 > 2 分钟 → 立即写当前进度快照，再上报大管家
- 大管家询问是否暂停 → 必须先确认日志已写，再允许暂停

**违规判定**：Phase 完成但 `_work.md` 未更新 → P1 错误
