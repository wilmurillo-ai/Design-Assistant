# 记忆管理规范

> **定位**: 全局底层规则，所有角色、所有项目强制遵守
> **版本**: v1.0
> **最后更新**: 2026-03-13
> **维护人**: 大管家

---

## 一、核心原则

> 短期记忆靠窗口，长期记忆靠文档。项目记忆最优先，按需检索不全载。

- 记忆不依赖对话历史，依赖注入的结构化文档
- 只加载当前任务需要的记忆，不全量加载所有记忆
- 项目记忆 > 角色记忆 > 系统记忆，严格按优先级路由
- Token 是稀缺资源，记忆加载必须精简高效

---

## 二、三层记忆架构

### 第一层：项目记忆（Project Memory）

**存储位置**: `sessions/{项目名}/`

| 文件 | 用途 | Token 上限 | 维护人 |
|------|------|-----------|--------|
| `context.md` | 项目全景摘要，新 Session 注入用 | ≤800 tokens | 产品经理 |
| `decisions.md` | 架构决策日志，只记结论 | ≤20 条 | CTO |
| `pitfalls.md` | 踩坑记录，问题+解法 | ≤15 条 | 研发/测试 |
| `inbox.md` | 大管家下发的待办指令 | 无上限（用完即清） | 大管家 |

**context.md 模板**:

```markdown
# [项目名] 项目背景

> 最后更新：YYYY-MM-DD | 维护人：产品经理

## 项目定位（3句话以内）

## 技术栈

## 目录结构（只列关键目录）

## 当前版本与状态
- 版本：
- 状态：开发中 / 测试中 / 上线
- 最近完成：（一句话）
- 下一步重点：（一句话）

## 关键决策（最多5条，只列结论）

## 已知坑（最多5条）

## 接续工作注意事项（最重要的1-2件事）
```

### 第二层：角色记忆（Role Memory）

**存储位置**: `references/constitutions/` + `references/role-extras/`

| 文件类型 | 用途 | 加载时机 |
|---------|------|---------|
| `{角色名}-角色宪法-SOP.md` | 角色身份、职责、工作方式 | 角色上岗时 |
| `role-extras/{角色名}-*.md` | 角色补充文档（错误记录、方案等） | 按需加载 |
| `rules/*.md` | 公共规则（禁令、分级、日志、记忆） | 角色上岗时 |
| `sop/*.md` | 标准操作流程 | 匹配到任务类型时 |

### 第三层：系统记忆（System Memory）

**存储位置**: `templates/archives/`

| 目录 | 用途 | 加载时机 |
|------|------|---------|
| `archives/重大决策/` | 跨项目的架构决策 | 需要参考历史决策时 |
| `archives/优化记录/` | 系统级优化方案 | 需要了解优化历史时 |
| `archives/问题记录/` | 历史问题解决记录 | 遇到类似问题时 |
| 全局 `memory_search` | OpenClaw 底层向量/全文搜索 | Priority 1-2 无法回答时 |

## 三、记忆路由优先级（强制执行）

### 执行项目任务时的加载顺序

```
1. 项目记忆（必须加载）
   ├─ context.md        → 了解项目当前状态
   ├─ decisions.md      → 需要技术决策参考时读
   └─ pitfalls.md       → 遇到报错/异常时先查

2. 角色记忆（角色上岗时加载）
   ├─ 当前角色宪法      → 确认职责边界
   ├─ 对应 SOP          → 确认执行流程
   └─ role-extras/      → 补充知识（按需）

3. 系统记忆（兜底，仅必要时使用）
   ├─ archives/         → 搜索历史决策/问题
   └─ memory_search     → 全局搜索（最后手段）
```

### 违规判定

| 行为 | 违规级别 |
|------|---------|
| 执行项目任务前不读 context.md | P1 |
| 跳过项目记忆直接全局搜索 | P1 |
| 全量加载多个项目的记忆 | P1 |
| context.md 超过 800 tokens 不精简 | P2 |
| 踩过的坑再踩（pitfalls.md 有记录） | P0 |

## 四、记忆写入规则

### 写入时机

| 事件 | 写入文件 | 负责人 | 写入内容 |
|------|---------|--------|---------|
| 版本迭代完成 | context.md | 产品经理 | 更新版本、状态、最近完成、下一步 |
| 重大技术决策 | decisions.md | CTO | 决策结论 + 一句话理由（不要过程） |
| 踩坑找到解法 | pitfalls.md | 研发/测试 | 问题描述 + 解法（≤3行） |
| 角色重建 | role-extras/ | 大管家 | 前任教训写入新版宪法 |
| 每 2 周 | context.md | 产品经理 | 强制 review，确保信息不过时 |

### 写入格式

**decisions.md 条目格式**:
```markdown
### D{序号}: {决策标题} ({日期})
{一句话结论}。理由：{一句话理由}。
```

**pitfalls.md 条目格式**:
```markdown
### P{序号}: {问题简述} [{已解决|未解决}]
**现象**: {一句话}
**原因**: {一句话}
**解法**: {一句话}
```

## 五、记忆淘汰规则

### 自动淘汰

| 记忆类型 | 淘汰规则 | 淘汰方式 |
|---------|---------|---------|
| context.md | 超 800 tokens | 精简旧内容（保留结论，删除过程） |
| decisions.md | 超 20 条 | 最早的决策移至 `archives/重大决策/` |
| pitfalls.md | 超 15 条 | `[已解决]` 且超 30 天的坑移除 |
| 日志文件 | 超 7 天 | 下次任务执行时自动清理 |
| inbox.md | 指令执行完毕 | 立即清除已完成指令 |

### 永不淘汰

- `archives/` 下的所有归档文件
- 角色宪法文件
- 公共规则文件

## 六、与 OpenClaw 底层记忆的协作

### 架构关系

```
┌─────────────────────────────────────────────┐
│ 应用层：多角色治理体系（本规范）               │
│ · 决定加载什么记忆、按什么优先级              │
│ · 项目隔离、角色隔离                         │
│ · 精简注入，主动控制 Token                   │
├─────────────────────────────────────────────┤
│ 基础设施层：OpenClaw 记忆系统                 │
│ · Bootstrap 注入（AGENTS.md 等）             │
│ · 向量嵌入 + SQLite + FTS5 搜索             │
│ · 上下文剪枝 + 对话压缩                      │
│ · MEMORY.md 长期记忆                         │
└─────────────────────────────────────────────┘
```

### 协作原则

- OpenClaw 的 Bootstrap 注入（AGENTS.md/SOUL.md 等）提供**全局身份和工具配置**，不涉及项目细节
- 本体系的 `context.md` 提供**项目维度的精准记忆**，替代 OpenClaw 的全量上下文阅读
- OpenClaw 的 `memory_search` 作为**最后手段**，仅在结构化文档无法覆盖时使用
- 避免重复：不在 OpenClaw 的 MEMORY.md 中重复写入已存在于 `sessions/` 中的项目信息

## 七、新项目记忆初始化清单

创建新项目时，大管家必须执行：

```
□ 1. 创建 sessions/{项目名}/ 目录
□ 2. 创建 context.md（按模板，初始内容可超 800 tokens，后续精简）
□ 3. 创建 decisions.md（空文件，带标题）
□ 4. 创建 pitfalls.md（空文件，带标题）
□ 5. 创建 inbox.md（空文件）
□ 6. 在 employee-registry.json 中注册项目路径
```

## 八、记忆健康度检查

大管家在每次迭代收尾时，检查记忆健康度：

| 检查项 | 健康标准 | 不健康时处理 |
|--------|---------|------------|
| context.md 时效 | 最后更新 ≤ 14 天 | 强制 review 更新 |
| context.md 体积 | ≤ 800 tokens | 精简 |
| decisions.md 条数 | ≤ 20 条 | 归档旧条目 |
| pitfalls.md 条数 | ≤ 15 条 | 清理已解决条目 |
| 日志文件 | 无超 7 天文件 | 清理 |

## 九、上下文管理规则（v1.1 新增）

> 解决「单 Session 上下文撑满导致响应变慢 + 记忆丢失」的问题。

### 9.1 Phase 切 Session 原则

**规则**：复杂任务（含多个 Phase）必须按 Phase 边界切换 Session，不在单个 Session 内完整跑完。

| 切换时机 | 操作 |
|---------|------|
| 一个 Phase 完成 | 在 task-log 打勾 → 新开 Session → 加载锚点 |
| 上下文超过 80K tokens | 立即触发 Phase 切割，不等 Phase 结束 |
| 响应时间持续 > 2 分钟 | 上报大管家，大管家决定是否切 Session |

### 9.2 新 Session 锚点加载

新 Session 启动时，只加载以下两份文件（约 1-3K token）：

```
1. sessions/{项目}/context.md     → 项目全景
2. sessions/{项目}/task-logs/{当前}_work.md → 已完成 Phase 的进度
```

**禁止**：新 Session 重新加载完整历史对话内容。

### 9.3 上下文下降现象说明

上下文 token 数字「下降」不是缓存释放，是 OpenClaw 触发了**上下文截断**（移除早期历史消息）。截断后早期对话不可恢复，这正是 Phase 实时落笔的必要性。

## 十、极端灾备与并发安全规约 (强制反哺)

1. **冷热索引拆分（防 Token 肥胖）**：项目记忆必须拆分活跃与归档状态，新上岗第一眼只准读当前活跃列表。
2. **灾备暴力重建（防死锁失忆）**：若 `sessions/{当前项目}/context.md` 意外丢失，允许且仅允许此时使用 `grep` 扫描该目录重建上下文。
3. **并发写锁（防子进程互踩覆盖）**：多个子 Session（如研发和测试）同时处理同一项目任务时，严禁使用全量写入（`write`）覆盖任务日志，必须使用 `edit` 或在文件末尾 `append` 追加。
4. **禁止套娃串门（防死循环）**：子 Session 严禁跨项目/跨任务互相读取绝对路径指针。若需依赖其他任务结果，必须 `sessions_yield` 抛回给主会话调度。
5. **孤儿对账机制（防异常中断）**：在迭代收尾或闲置心跳时，必须检查 `task-logs/` 下是否存在未被 `context.md` 记录的幽灵日志，发现则自动归档。
