# 日志管理规范

> 📌 **用途**: 规范所有类型日志的存放位置和命名方式
> 🎯 **目标**: 避免日志混乱,便于查找和归档
> 🔄 **适用范围**: 所有项目,所有角色

---

## 一、日志分类

### 1. 业务项目日志

**定义**: 业务系统运行时产生的日志

**存放位置**: `project/[项目名]/logs/`

**内容类型**:
- 服务运行日志 (如FastAPI/Django的access.log)
- 错误日志 (error.log)
- 数据库操作日志
- 访问日志

**示例**:
```
project/project-b/backend/server.log
project/project-a/logs/api.log
```

**保留策略**: 运行日志(.log)保留30天,按日滚动

---

### 2. 系统级项目日志

**定义**: 优化LobsterAI底层能力的项目产生的工作日志

**存放位置**: `system-projects/[项目名]/logs/`

**内容类型**:
- 工作报告
- 测试报告
- 迁移记录
- 完成报告
- 方案文档

**示例**:
```
system-projects/skill-library/logs/2026-03-08_test_report.md
system-projects/docs-refactor/logs/2026-03-08_proposal.md
```

**保留策略**: 全部永久保留(归档到项目目录)

---

### 3. 全局系统日志

**定义**: 角色系统、绩效系统等全局机制产生的日志

**存放位置**: `governance/logs/`

**内容类型**:
- 角色重建记录
- 绩效变更记录
- 全局配置变更记录
- 系统部署记录

**示例**:
```
governance/logs/2026-03-05_角色系统部署.log
governance/logs/2026-03-06_大管家重建记录.log
```

**保留策略**: 永久保留

## 二、日志命名规范

### 基本格式

```
YYYY-MM-DD_描述.扩展名
```

**规则**:
- 日期前缀: 必须使用 `YYYY-MM-DD` 格式
- 描述: 使用下划线分隔,简洁准确
- 扩展名: `.md`(文档) `.log`(日志) `.json`(数据)

### 类型后缀(可选)

根据日志类型添加后缀,提升可读性:

| 后缀 | 说明 | 示例 |
|-----|------|------|
| `_work_report` | 工作报告 | `2026-03-08_batch1_work_report.md` |
| `_test_report` | 测试报告 | `2026-03-08_test_report.md` |
| `_completion` | 完成报告 | `2026-03-08_examples_completion.md` |
| `_proposal` | 方案文档 | `2026-03-08_proposal.md` |
| `_error` | 错误日志 | `2026-03-08_error.log` |
| `_server` | 服务日志 | `server.log` (滚动,无日期前缀) |

### 命名示例

**好的命名** ✅:
```
2026-03-08_skill_test_report.md
2026-03-08_examples_completion.md
2026-03-05_角色系统部署.log
2026-03-06_系统优化讨论.md
```

**不好的命名** ❌:
```
test.md (缺少日期)
20260308_report.md (日期格式错误)
report_2026_03_08.md (日期位置错误)
测试报告.md (缺少日期)
```

## 四、日志滚动策略

### 业务项目运行日志

**当前日志**: `backend/server.log` (无日期,持续追加)

**滚动规则**:
- 每天0点自动滚动
- 旧日志重命名为: `logs/YYYY-MM-DD_server.log`
- 保留30天,超过30天自动删除

**实现方式**:
```bash
# 使用logrotate或手动脚本
# 示例: 每天定时任务
0 0 * * * /path/to/rotate_logs.sh
```

### 系统级项目日志

**不滚动**: 全部永久保留

**原因**: 这些是工作报告和文档,属于项目交付物的一部分,需要完整保留以追溯项目历史。

## 五、日志查找指南

### 快速查找命令

**查找所有日志**:
```bash
# 查找所有md格式日志
find ~/your-workspace -name "*.md" -path "*/logs/*"

# 查找所有.log日志
find ~/your-workspace -name "*.log"
```

**按日期查找**:
```bash
# 查找2026-03-08的所有日志
find ~/your-workspace -name "2026-03-08*"
```

**按项目查找**:
```bash
# 查找Skill库项目的所有日志
ls ~/your-workspace/system-projects/skill-library/logs/
```

**按关键词查找**:
```bash
# 在所有日志中搜索"测试报告"
grep -r "测试报告" ~/your-workspace/*/logs/
```

## 七、维护规范

### 日志创建规范

1. **立即命名**: 创建日志时就使用规范命名,不要用临时名称
2. **添加日期**: 必须有日期前缀,方便时间序查找
3. **描述清晰**: 从文件名就能看出日志内容

### 日志整理规范

1. **定期清理**: 每月1号清理超过30天的业务项目运行日志
2. **归档保留**: 系统级项目日志和全局系统日志永久保留
3. **目录整洁**: 不要在logs/目录放非日志文件

## 八、FAQ

### Q1: 为什么系统级项目日志不滚动删除?

因为这些是**工作成果的一部分**,不是临时运行日志。它们记录了项目的决策过程、测试结果、完成情况,是项目历史的重要组成部分。

### Q2: 如果同一天有多个相同类型的日志怎么办?

添加序号或时间戳:
```
2026-03-08_batch1_test_report.md
2026-03-08_batch2_test_report.md
2026-03-08_14:30_emergency_fix.log
```

### Q3: 日志太多了,怎么快速找到我要的?

1. 使用统一的命名规范,方便grep搜索
2. 按日期前缀排序,时间序很清晰
3. 使用后缀过滤,如 `*_test_report.md`

## 九、与任务执行日志规范的关系

本规范覆盖**系统运行日志**（业务项目运行日志、系统级项目工作日志、全局治理日志）。

**任务执行日志**（研发/测试在执行具体任务时产生的 `_work.md`、`_bug.md`、`_fix.md`）是一个独立层级，由专门的规范管理：

→ 详见 `references/rules/任务执行日志规范.md`

存放路径: `sessions/{项目}/task-logs/` — 属于项目记忆目录，不属于本规范的日志目录。

| 日志类型 | 归属规范 | 存放位置 |
|---------|---------|---------|
| 业务系统运行日志 | 本规范 | `project/{项目}/logs/` |
| 系统级项目工作日志 | 本规范 | `system-projects/{项目}/logs/` |
| 治理系统日志 | 本规范 | `governance/logs/` |
| 任务工作/Bug/修复日志 | 任务执行日志规范 | `sessions/{项目}/task-logs/` |
