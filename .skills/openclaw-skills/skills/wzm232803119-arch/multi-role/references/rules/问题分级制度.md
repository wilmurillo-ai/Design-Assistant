# 问题分级制度 (P0/P1/P2)

> 📌 **用途**: 定义错误和问题的分级标准,决定角色重建阈值
> 🎯 **目标**: 客观量化角色表现,避免主观判断
> 🔄 **适用范围**: 所有角色,所有项目

---

## 一、分级定义

### P0 - 致命错误

**定义**: 严重破坏现有功能或完全违反流程的错误

**典型场景**:
- 删除功能来修复报错
- 完全跳过必需流程步骤
- 功能回退(原本能用的功能变成不能用)
- 违反任何一条绝对禁令

**阈值**: **累计3次** → 触发角色重建

**记录方式**:
```python
role_manager.record_error(
    role="角色名",
    action="做了什么",
    reason="为什么是错误",
    priority="P0"
)
```

---

### P1 - 严重错误

**定义**: 明显违反规范,影响功能完整性,但不造成功能完全失效

**典型场景**:
- 交互选项缺失(用户无法完成操作)
- 字数严重不达标(目标5000字只写了2000字)
- 整流程缺少关键步骤
- 测试框架执行不完整

**阈值**: **累计6次** → 触发角色重建

**记录方式**:
```python
role_manager.record_error(
    role="角色名",
    action="做了什么",
    reason="为什么是错误",
    priority="P1"
)
```

---

### P2 - 一般问题

**定义**: 细节不到位,但不影响核心功能的问题

**典型场景**:
- 细节不统一(交互风格不一致)
- 风格偏差(文案表述不规范)
- 单处遗漏(某个边界条件未处理)
- 非关键功能的小bug

**阈值**:
- **方案层面**: 同一问题3次未解决 → 换方向
- **角色层面**: 累计9次 → 触发角色重建

**记录方式**:
```python
# 首次发现问题
issues.open_issue(
    issue_id="唯一ID",
    description="问题描述"
)

# 每次尝试解决
issues.record_attempt(
    issue_id="问题ID",
    approach="本次采用的解决思路",
    result="solved | failed | partial",
    role="角色名",
    role_manager=role_manager  # 传入,用于P2累计9次联动
)
```

---

## 二、三个「3次」的区别(重要!)

| 类型 | 触发条件 | 触发动作 | 使用工具 | 计数范围 |
|-----|---------|---------|---------|---------|
| **P0错误** | 累计3次 | 角色重建 | `role_manager.record_error(priority="P0")` | 该角色所有P0错误 |
| **P1错误** | 累计6次 | 角色重建 | `role_manager.record_error(priority="P1")` | 该角色所有P1错误 |
| **P2问题(方案)** | **同一问题**3次未解决 | 换方向 | `issues.record_attempt()` | 单个issue_id |
| **P2问题(角色)** | 累计9次 | 角色重建 | `issues.record_attempt(role_manager=...)` | 该角色所有P2问题 |

**关键要点**:
- P0/P1 和 IssueTracker 是**两套独立系统**,互不干扰
- P0/P1 计数所有错误,P2方案层面只计数**同一个问题**
- P2联动角色重建是累计**所有P2问题**

## 三、特殊规则

### 功能回退直接触发重建

**定义**: 原本能用的功能,修改后变成不能用

**处理**:
```python
role_manager.record_functional_regression(
    role="角色名",
    feature="哪个功能回退了",
    details="详细说明"
)
```

**结果**: 等同于P0错误,**立即触发角色重建**,不需要累计3次

## 四、分级判断流程

```
发现问题
    ↓
问: 是否破坏现有功能或完全违反流程?
    ↓ 是
    记录为 P0
    ↓ 否
问: 是否影响功能完整性或明显违反规范?
    ↓ 是
    记录为 P1
    ↓ 否
问: 是否只是细节问题?
    ↓ 是
    记录为 P2 (使用IssueTracker)
    ↓ 否
    可能不是错误,仅记录为改进建议
```

---

## 五、角色重建流程

当触发重建条件时:

1. **运行重建**:
   ```python
   role_manager._rebuild_role(role_name)
   ```
   更新 `角色评分.json` 文件

2. **创建/更新宪法**:
   - 检查 `governance/constitution/[角色]_v[新版本].md` 是否存在
   - 如不存在,创建新宪法,包含所有历史错误教训
   - 如已存在,更新版本号和错误教训章节

3. **创建/更新错误记录**:
   - 在 `governance/archives/errors/[角色]_错误记录.md` 追加最新错误

4. **记录重建事件**:
   - 在迭代工作报告中记录重建原因和时间
   - 要求重建角色在下次工作前先阅读新宪法

## 六、评分系统

### 评分规则

每个任务完成后,大管家给出0-10分评分:

| 分数 | 评级 | 说明 |
|-----|------|------|
| 10分 | 完美 | 零错误,超出预期 |
| 9分 | 优秀 | 零错误,符合预期 |
| 8分 | 良好 | 有1-2个P2问题,但可接受 |
| 7分 | 及格 | 有多个P2问题或1个P1问题 |
| <7分 | 不及格 | 有P0错误或多个P1错误 |

### 评分与重建的关系

- 评分≥8: 不触发任何动作
- 评分<8: 如果累计错误达到阈值,触发重建
- 评分只是参考,**重建触发条件是P0/P1/P2计数**,不是评分

## 七、常见问题

### Q1: P0和P1如何区分?

**核心区别**: 是否造成功能完全失效

- 删除功能 → P0 (功能完全失效)
- 交互选项缺失导致无法操作 → P1 (功能受影响但可能有workaround)

### Q2: P2什么时候换方向?

**同一个issue_id超过3次未解决时换方向**,不是所有P2累计3次。

例如: "登录按钮样式问题"尝试3次未解决 → 换方向

### Q3: P2累计9次如何计算?

**该角色所有P2问题的总次数**,不区分issue_id。

例如: 问题A尝试3次,问题B尝试4次,问题C尝试2次 → 总计9次 → 触发重建
