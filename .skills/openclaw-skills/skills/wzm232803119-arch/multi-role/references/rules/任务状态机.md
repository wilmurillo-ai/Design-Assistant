# 任务状态机

> 📌 **用途**: 定义任务生命周期中所有合法状态及转换规则
> 🎯 **适用范围**: 所有项目、所有角色、所有 Sub Agent
> 🔄 **版本**: v1.0 | 创建日期: 2026-03-19

---

## 一、状态枚举

| 状态 | 说明 |
|------|------|
| 待处理 | 任务已登记，等待执行 |
| 执行中 | Sub Agent 正在处理该任务 |
| 验收中 | Sub Agent 已完成，大管家正在验收 |
| 完成 | 大管家验收通过，任务关闭 |
| 失败 | 执行失败且已超出重试次数（3次），需人工介入 |
| 失败待处理 | 任务已失败，等待人工介入或重试决策（「失败」状态的子状态）|
| 已取消 | 用户主动取消，或大管家决定不再执行 |
| 暂停 | 因依赖未就绪、上下文超限或用户指令暂停执行 |

## 二、合法状态转换表

| 当前状态 | 可转换到 | 触发条件 | 负责方 |
|---------|---------|---------|--------|
| 待处理 | 执行中 | 大管家下发给 Sub Agent | 大管家 |
| 待处理 | 已取消 | 用户取消指令 | 用户 / 大管家 |
| 待处理 | 暂停 | 依赖未就绪 / 用户暂停指令 | 大管家 |
| 执行中 | 验收中 | Sub Agent 完成并回传摘要 | Sub Agent |
| 执行中 | 失败 | 第3次失败/超时 | 大管家 |
| 执行中 | 取消中 | 用户取消指令，Sub Agent 尚未完成当前原子操作 | 用户 / 大管家 |
| 执行中 | 已取消 | 用户取消 + Sub Agent 完成当前原子操作后 | 用户 / 大管家 |
| 执行中 | 暂停 | 上下文超限紧急暂停 / 用户暂停指令 | 大管家 |
| 验收中 | 完成 | 大管家验收通过 | 大管家 |
| 验收中 | 执行中 | 验收不通过，打回重做 | 大管家 |
| 失败 | 执行中 | 人工介入后决定重新下发 | 用户 / 大管家 |
| 失败 | 已取消 | 用户决定放弃 | 用户 |
| 暂停 | 待处理 | 暂停原因消除（依赖就绪/用户恢复指令）| 大管家 |
| 暂停 | 已取消 | 用户决定取消 | 用户 |
| 已取消 | — | 终态，不可转换 | — |
| 完成 | — | 终态，不可转换 | — |

> ⚠️ 未列出的转换均为非法转换，禁止执行。

## 三、每个状态的触发条件与负责方

### 待处理
- **触发条件**：大管家接收到用户指令，登记到任务队列
- **负责方**：大管家
- **存储**：写入 `sessions/{项目}/inbox.md` 或任务队列

### 执行中
- **触发条件**：大管家将任务下发给 Sub Agent，Sub Agent 开始执行
- **负责方**：Sub Agent（执行）/ 大管家（监控）
- **存储**：在 `sessions/{项目}/task-logs/` 创建对应日志文件，status 字段写入「执行中」

### 验收中
- **触发条件**：Sub Agent 完成执行，回传摘要 + 文件路径给大管家
- **负责方**：大管家
- **存储**：task-log 文件 status 字段更新为「验收中」

### 完成
- **触发条件**：大管家验收通过（验收清单全部通过）
- **负责方**：大管家
- **存储**：task-log 文件 status 更新为「completed」；inbox.md 中对应条目标记完成并清理

### 失败
- **触发条件**：Sub Agent 第3次失败/超时（参见系统管理流程 SOP F-2）
- **负责方**：大管家
- **存储**：task-log 文件 status 更新为「failed」；写入 inbox.md「失败待处理」；写入对应角色错题本

### 已取消
- **触发条件**：用户主动取消（参见系统管理流程 SOP G）
- **负责方**：用户（触发）/ 大管家（执行取消流程）
- **存储**：task-log 文件 status 更新为「cancelled」；写入 decisions.md 取消记录

### 暂停
- **触发条件**：① 上下文超过 180K 触发紧急暂停；② 依赖任务未就绪；③ 用户显式「暂停」指令
- **负责方**：大管家
- **存储**：task-log 文件 status 更新为「暂停」；未完成任务写入 inbox.md

## 四、异常状态强制转换规则

### 4.1 超时强制转换

| 场景 | 规则 |
|------|------|
| Sub Agent 超过 5 分钟（简单任务）未回传 | 大管家强制将该任务标记为「失败」，触发 F-2 重试流程 |
| Sub Agent 超过 15 分钟（复杂任务）未回传 | 同上 |
| 任务处于「执行中」超过阈值时间 | 无论 Sub Agent 是否响应，强制转换为「失败」，开始重试计数 |

### 4.2 崩溃/异常退出强制转换

| 场景 | 规则 |
|------|------|
| 大管家发现 task-log 状态为「执行中」但对应 session 已不存在 | 强制转换为「失败」，写入 inbox.md，等待人工确认 |
| 冷启动自检发现未关闭的「执行中」日志 | 判定为异常退出遗留，向用户汇报后由用户决定重试或取消 |
| 同时发现多个「执行中」孤儿任务 | 批量标记为「失败待处理」，统一汇报给用户 |

### 4.3 状态污染防护

- 禁止直接覆盖 task-log 文件的 status 字段，必须使用 `edit` 精确替换
- 禁止多个 Sub Agent 同时修改同一个 task-log 文件的 status 字段（并发写锁规则）
- 状态一旦进入终态（完成 / 已取消），不得被任何操作修改
