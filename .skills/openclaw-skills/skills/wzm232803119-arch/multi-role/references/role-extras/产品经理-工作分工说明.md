# 产品经理工作分工说明 v1

> 📌 **文件类型**：通用参考文档
> ✍️ **作者**：产品经理
> 📅 **适用范围**：多角色协作项目

---

## 一、项目概述

本文档描述产品经理在多角色协作项目中的工作分工与职责边界，适用于需求驱动的软件开发或内容创作类项目。

**核心流程**：需求输入 → 功能分析 → 方案设计 → 各角色协作 → 交付验收

---

## 二、角色职责总表

| 角色 | 主要职责 | 主要交付物 |
|------|---------|-----------|
| **大管家** | 统筹监督、方案审核、测试验收、迭代报告 | 审核意见、迭代报告、放行决定 |
| **产品经理** | 需求分析、流程设计、验收标准制定、内部文档 | 需求文档、流程图、分工说明、资源评估 |
| **研发** | 编码实现、Bug修复、性能优化 | 可运行代码、Bug修复记录 |
| **测试** | 多维度测试、测试报告、回归测试 | 测试报告、验证脚本 |
| **内容运营** | 对外宣传内容（Blog、社媒文案、推广素材） | 发布级内容文档 |

---

## 三、各功能模块分工说明

以下按典型项目流程的关键步骤，说明各角色在每步的具体工作内容。

---

### Step 1：用户输入与需求确认

| 维度 | 内容 |
|------|------|
| **研发负责** | 实现输入接收逻辑（参数校验、空值处理、异常抛出）；记录输入事件日志 |
| **产品经理负责** | 定义输入字段规范；定义错误提示文案；明确必填项与可选项 |
| **测试负责** | 验证非法输入时程序正确报错退出；验证正常输入后流程顺利推进 |

---

### Step 2：数据分类与分析

| 维度 | 内容 |
|------|------|
| **研发负责** | 实现分类/分析算法；基于规则或模型完成分类逻辑；输出分类结果 |
| **产品经理负责** | 定义分类体系及规则；维护关键词/规则映射表；评估分类准确率要求 |
| **测试负责** | 测试典型输入的分类结果是否符合预期；覆盖边界案例 |

---

### Step 3：数据源检查与补充

| 维度 | 内容 |
|------|------|
| **研发负责** | 实现数据存储增删查操作；实现数据不足时的自动补充机制；初始化时加载预置数据 |
| **产品经理负责** | 定义数据量阈值与触发条件；定义各类型数据的最低量要求 |
| **测试负责** | 验证数据为空时是否自动触发补充；验证补充后数据达标 |

---

### Step 4：相似度/匹配计算

| 维度 | 内容 |
|------|------|
| **研发负责** | 实现匹配/相似度算法；按阈值筛选结果；按优先级排列输出 |
| **产品经理负责** | 定义评分维度与权重；定义入选阈值 |
| **测试负责** | 验证评分范围合法；验证阈值过滤逻辑；验证结果排序正确性 |

---

### Step 5：热度/优先级判定 + 用户确认

| 维度 | 内容 |
|------|------|
| **研发负责** | 按条件判定优先级等级；展示匹配结果；实现多选项交互 |
| **产品经理负责** | 定义优先级判定规则；定义各等级的交互选项（对称性原则） |
| **测试负责** | 验证各等级判定条件；验证交互选项对称性；验证用户选择后流程正确跳转 |

---

### Step 6：内容生成 + 用户确认

| 维度 | 内容 |
|------|------|
| **研发负责** | 实现内容生成逻辑；实现多选项交互（确认/重新生成/手动修改/退出） |
| **产品经理负责** | 定义内容规格要求；定义交互选项规范；定义生成内容的参考维度 |
| **测试负责** | 验证生成内容规格达标；验证重新生成/手动修改功能正常 |

---

### Step 7：核心输出生成 + 用户确认

| 维度 | 内容 |
|------|------|
| **研发负责** | 实现核心输出生成（如按结构分段生成）；实现多选项用户确认交互 |
| **产品经理负责** | 定义输出格式和质量要求；定义结构化输出规范（各段字数/比例）；明确不可跳过的步骤 |
| **测试负责** | 验证输出内容达到质量指标；验证用户确认/重新生成/返回重来等选项均正常 |

---

### Step 8：保存 + 询问是否继续

| 维度 | 内容 |
|------|------|
| **研发负责** | 实现结果持久化（本地文件或数据库）；实现继续/结束二选项；支持连续操作时序号自增 |
| **产品经理负责** | 定义文件命名规则；定义二选项交互规范 |
| **测试负责** | 验证文件是否成功生成且内容完整；验证连续操作多轮流程正常 |

## 四、横向职责说明（跨步骤）

### 4.1 底层基础模块

| 职责 | 负责角色 | 说明 |
|------|---------|------|
| 操作日志记录 | 研发 | 记录每次操作到日志，自动清理超限旧日志 |
| 角色评分管理 | 大管家 | 记录各角色得分、错误次数、版本号；超过错误次数阈值自动触发重建 |
| 内容审核流程 | 大管家 + 内容运营 | 强制执行大纲确认→内容创作→大管家审核→放行/打回 全流程 |
| 接口变更同步 | 大管家监督、研发执行 | 任何底层接口变更（返回值/参数签名），必须在同一次修改中更新所有调用方 |

### 4.2 日常迭代流程

```
用户提出需求
    ↓
大管家拆分任务
    ↓
产品经理：读取流程图/规范文档 → 设计方案 → 提交大管家审核
    ↓
大管家审核（核心问题：方案是否会损害现有功能？是否对照了流程图？）
    ↓
研发实施（不删任何功能代码）
    ↓
测试：多维度测试（基础/功能/流程符合度/交互对称性/指标达标/用户体验）
    ↓
大管家：亲自运行验收 → 写迭代报告
    ↓
交付用户
```

### 4.3 内容运营工作流

```
大管家指派任务
    ↓
内容运营：提交大纲给用户确认
    ↓
用户确认大纲
    ↓
内容运营：创作全文
    ↓
大管家：审核打分（满分100，≥85放行；标题分不达标直接否决）
    ↓
不达标：打回给内容运营重写 → 重新提交审核
达标：输出文档给用户
```

## 五、资源消耗估算

### 5.1 当前版本（无外部 AI API）

| 资源类型 | 消耗情况 |
|---------|---------|
| AI API 调用 | 无（全部为本地规则匹配 + 模板拼接） |
| 本地计算 | 极低（脚本每次运行响应时间极短） |
| 存储空间 | 数据库：约 1-5 MB；日志文件：上限 10 MB（超限自动清理）；每次输出：约 5-10 KB |
| 网络 | 无（外部数据当前为内置预加载，未真实请求） |

### 5.2 升级为真实 AI 生成后的资源预估

| 步骤 | 预估 Token 消耗 | 备注 |
|------|---------------|------|
| 分类判断 | 约 100-200 tokens/次 | 输入：用户输入；输出：1个分类词 |
| 相似度计算 | 约 200-500 tokens/批次 | 依参考数据量而定 |
| 简介/摘要生成 | 约 200-300 tokens/次 | 约 100 字输出 |
| 核心内容生成 | 约 2000-5000 tokens/次 | 依输出规格而定 |
| **单次完整流程** | **约 2500-6000 tokens** | 包含所有步骤 |

> ⚠️ 升级 AI API 前，须先明确 API 提供商、计费模式、每日 token 预算，由产品经理纳入资源评估文档。

## 六、迭代优先级建议

| 优先级 | 待办事项 | 负责角色 | 预估工时 |
|--------|---------|---------|---------|
| P0 | 核心输出指标验证（实际输出量稳定达标） | 研发 + 测试 | 1天 |
| P0 | 完整流程回归测试（全流程不中断） | 测试 | 0.5天 |
| P1 | 用户确认步骤完善 | 研发 | 0.5天 |
| P1 | 内容生成规格稳定 | 研发 | 0.25天 |
| P2 | 接入真实 AI API | 研发 + 产品经理（方案）| 3-5天 |
| P2 | 真实数据源接入 | 研发 | 2-3天 |
| P3 | 内容运营：发布第一篇对外内容 | 内容运营 + 大管家 | 1天 |

## 七、快速参考

**各角色"绝不能做的事"**

- 产品经理：禁止删功能来修复报错；禁止跳过流程完整性自查；禁止用估算替代实际指标验证
- 研发：禁止修改接口签名后不同步更新调用方；禁止只保留 print 来规避报错
- 测试：禁止以"不报错"作为测试通过的唯一依据；禁止估算指标
- 大管家：禁止未亲自运行就允许交付；禁止接受未经流程自查的代码；禁止内容运营未审核直接输出
- 内容运营：禁止跳过用户大纲确认直接写全文；禁止大管家审核未通过就对外输出
