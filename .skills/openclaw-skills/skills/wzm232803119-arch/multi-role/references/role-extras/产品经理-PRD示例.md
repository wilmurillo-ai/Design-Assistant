## 一、产品定位

**一句话定位**：帮助用户快速完成某类创作或生产任务的 AI 辅助工具，通过数据分析降低决策失败风险，通过 AI 生成内容降低起步门槛。

**目标用户**：
- 有创作/生产意愿但苦于起步的业余用户
- 需要快速验证方向可行性的新手
- 批量生产同类内容的工作室或团队

**核心价值**：
- 方向验证：用真实数据判断主题/选题的市场热度
- 降低起步门槛：AI 生成符合规范的初稿内容
- 用户掌控权：每个关键节点均可人工干预（确认/重新生成/手动修改）

### 3.2 主流程需求详述

#### F01 用户输入主题 + 梗概

**输入字段**：
- 主题（必填，不可为空）：例「某类主题关键词」
- 梗概（必填，不可为空）：例「对主题的简短描述」

**异常处理**：
- 任一字段为空 → 提示错误，抛出 `ValueError`，程序退出

---

#### F02 AI 分类判断

**支持分类（示例，按实际项目定义）**：类型A、类型B、类型C……（共N类）

**当前实现**：关键词枚举匹配，命中即返回对应分类，无命中则默认返回预设分类

**目标实现（P1）**：调用大模型 API，Prompt 模板：
```
请判断以下内容属于哪个分类：
主题：{user_theme}
梗概：{user_outline}
可选分类：类型A、类型B、类型C……
只返回一个分类名称，不要其他内容。
```

---

#### F03 + F04 数据库检查 + 数据补充

**检查逻辑**：若当前分类数据量 < 阈值N，触发数据补充

**当前实现**：内置静态数据字典，每分类预埋若干条数据，首次运行时加载到 SQLite

**目标实现（P2）**：
- 真实数据源接入
- 每分类至少保证 M 条有效数据
- 请求限速，避免被封 IP

---

#### F05 AI 相似度计算

**计算对象**：用户输入（主题+梗概）vs 数据库中同分类每条记录的描述

**当前实现**：字符级 Jaccard 相似度，公式：`总分 = 基础分 + 关键词命中率 × 权重`

**筛选阈值**：≥ N 分加入匹配列表，按分数降序排列

**目标实现（P1）**：语义向量相似度（调用 Embedding API）或大模型直接评分

---

#### F06 + F07 热度等级判定 + 用户确认

**判定规则**：

| 等级 | 条件 | 交互选项 |
|------|------|---------|
| 🔥 高热度 | 匹配≥N1本 且 平均相似度≥S1分 | 1=确认继续 / 2=更换方向 / 0=退出 |
| ⭐ 中热度 | 匹配≥N2本 且 平均相似度≥S2分 | 1=确认继续 / 2=更换方向 / 0=退出 |
| ❄️ 低热度 | 无匹配数据 | 1=更换方向（重来） / 2=坚持继续 / 0=退出 |

> ⚠️ **设计原则**：低热度时「更换方向」为 1 号键（突出引导），高/中热度时「确认继续」为 1 号键，保持不同热度下操作逻辑对称

---

#### F08 + F09 AI 生成摘要/简介 + 用户确认

**输出规格**：约 100 字，风格参考同分类参考数据

**当前实现**：多组句式模板随机选取 + 关键词填充

**用户干预点（多选项）**：
- 1 = 确认，进入核心内容生成
- 2 = 重新生成（循环直到用户满意）
- 3 = 手动修改（用户直接输入替代文本）
- 0 = 退出

---

#### F10 + F11 AI 生成核心内容 + 用户确认

**输出规格**：≥ 目标字数（示例：≥ 2000 字），结构化分段：

| 段落 | 内容说明 | 目标字数 |
|------|---------|---------|
| ① 开场 | 场景/背景建立 | ~200字 |
| ② 冲突/核心 | 主要矛盾/内容展开 | ~450字 |
| ③ 转折 | 关键变化/突破 | ~500字 |
| ④ 发展 | 内容深化 | ~350字 |
| ⑤ 收束 | 总结/钩子/展望 | ~350字 |

**用户干预点（多选项）**：
- 1 = 满意，保存并询问是否继续下一轮
- 2 = 重新生成（循环）
- 3 = 修改主题（返回 Step1 重新输入）
- 0 = 退出

---

#### F12 结果保存

**文件格式**：`.txt`
**文件命名规则**：`output_{主题前N字}_{时间戳}_{序号}.txt`
**存储位置**：脚本同级目录

---

#### F13 连续创作多轮

**流程**：保存后询问「1=继续下一轮 / 0=结束」，继续时序号自增，重新调用核心生成函数

---

### 3.3 非功能需求

| 类别 | 要求 |
|------|------|
| 稳定性 | 全流程可无中断完整运行 |
| 字数达标 | 每次核心输出实际统计字数 ≥ 目标字数 |
| 日志 | 每次操作写入 JSON 日志，日志文件上限 10MB（自动清理）|
| 错误恢复 | KeyboardInterrupt、ValueError、Exception 均有 try/except 捕获 |
| 代码结构 | 底层接口（project_base.py）变更必须同步更新所有调用方 |

## 四、数据结构

### 4.1 SQLite 数据库表结构（示例）

```sql
CREATE TABLE items (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    title       TEXT NOT NULL,
    author      TEXT NOT NULL,
    category    TEXT NOT NULL,
    description TEXT,
    hot_rank    INTEGER,
    source      TEXT,           -- "内置数据源" | "外部数据源"
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 4.2 日志 JSON 结构（每条记录）

```json
{
  "timestamp": "YYYY-MM-DDTHH:MM:SS",

  "action": "生成核心内容",
  "detail": "第1轮, 字数: 2234",
  "status": "完成",
  "type": "dev"
}
```

## 六、已知限制（当前 Demo 版本）

- 分类判断：关键词匹配，无语义理解，混合类型主题判定不准确
- 相似度：字符级匹配，无语义向量，相似度分数参考意义有限
- 内容生成：全部模板拼接，重复感明显，不适合直接发布，需人工二次润色
- 数据来源：内置静态数据，非实时数据，无法反映最新动向
- 外部数据源：当前为内置数据填充模拟，未对接真实数据接口
