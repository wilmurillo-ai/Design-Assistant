## 错误 #1 — 未及时发现产品经理错误方案

**发生时间**: 迭代#1 (具体日期未记录)

**事件描述**:
产品经理提出"删除功能来修复报错"的方案,大管家没有在审核环节拦截,导致功能被误删。

**问题根源**:
大管家审核方案时,只关注了"是否能解决报错",没有问"这个方案会不会损害现有功能?"

**正确做法**:
审核每个方案时,必须问:
1. 这个方案会不会删除或破坏任何现有功能?
2. 这个方案是否对照了产品流程图?
3. 这个方案的交互节点是否保持风格统一?

**已采取的改进措施**:
- 在角色宪法中增加"方案审核"强制检查清单
- 第一个问题就是"是否会删除或破坏任何现有功能?"

**教训**:
**永远不要用删除功能的方式来修复报错。** 报错是表象,应该找到根本原因并修复,而不是通过删除功能来"消除"报错。

## 错误 #2 — 交付前未亲自运行程序验证

**发生时间**: 迭代#1 (具体日期未记录)

**事件描述**:
测试声称"语法检查通过",大管家相信了测试的结论,没有亲自运行验证就允许交付,结果用户发现程序无法运行。

**问题根源**:
大管家过度信任了测试的结论,"语法检查通过"不等于"程序能正常运行"。

**正确做法**:
任何交付前,大管家必须:
1. 亲自运行程序
2. 走完至少一遍主流程
3. 验证每个交互节点的选项正确
4. 确认保存的文件存在且内容正常

**已采取的改进措施**:
- 在角色宪法中增加"交付验收"强制检查清单
- 在绝对禁令中添加"禁止在没有亲自运行验证的情况下允许交付"

**教训**:
**"测试通过"不等于"可以交付"。** 大管家作为质量最终负责人,必须亲自验证,不能完全依赖其他角色的报告。

## 错误 #3 — 新功能需求未走 SOP,直接动手写代码

**发生时间**: 2026-03-10

**事件描述**:
用户反馈 bug(页面无响应、重复提交)后,大管家正确地按 Bug 排查修复流程处理。但用户随后补充了新功能需求(简介确认页、小说删除、简介编辑、主体脉络生成),大管家没有识别出需求性质已从"bug 修复"变为"新功能开发",直接自己动手写代码,跳过了完整的需求到交付流程。

具体跳过的环节:
1. 没有让产品经理输出 PRD
2. 没有让 CTO 做代码审查
3. 没有让测试做六维度验证
4. 大管家没有亲自运行完整流程验证
5. 没有方案评审环节

**问题根源**:
大管家在处理 bug 的惯性下,没有在需求性质发生变化时"停下来重新判断"。正确的逻辑应该是:收到任何需求时,先判断性质(bug?新功能?优化?),再决定走哪条 SOP。而不是顺着手感继续执行。

**正确做法**:
当用户补充"简介确认、删除、编辑、脉络生成"时,大管家应该:
1. 识别:这不是 bug,是新功能需求
2. 停下来:告诉用户"这是新功能,需要走需求到交付流程"
3. 启动 SOP:产品经理出 PRD → 方案评审 → 研发实施 + CTO review → 测试验证 → 大管家终审

**已采取的改进措施**:
- 在大管家宪法中补充"需求性质判断"为强制前置步骤
- 明确规则:任何超出 bug 修复范围的改动,必须重新走需求到交付流程

**教训**:
**收到需求的第一件事不是动手,是判断性质。** bug 走 bug 流程,新功能走需求到交付流程。两者不能混在一起。"顺手改一下"是 SOP 最大的敌人。

## 错误 #4 — 无 SOP 场景下未向用户确认,直接凭感觉执行

**发生时间**: 2026-03-11

**事件描述**:
用户要求「产品经理出产品架构图」。大管家直接调用子 Agent 执行画图任务,跳过了两个关键判断:
1. 没有判断「出架构图」属于哪类指令类型,也没有检查是否有对应 SOP
2. 当时处于「讨论决策」阶段,产品经理的架构方案还没有经过大管家审核和用户确认,就进入执行阶段了

结果:输出了一张中文全部乱码、内容不完整的架构图,浪费了大量 Token,还需要返工。

**问题根源**:
大管家的指令路由逻辑不完整。原有的「需求性质判断表」没有覆盖「文件/图表生成」这个类型,也没有「无对应 SOP 时应该怎么做」的异常处理逻辑。导致大管家遇到不在判断表里的指令类型时,默认直接执行,而不是先向用户确认。

**正确做法**:
1. 判断「出架构图」属于「文件/图表生成」类型
2. 检查 `governance/sop/` 下是否有对应流程 → 没有
3. 触发异常处理:向用户确认「这类任务没有 SOP,我的处理建议是 XXX,请确认」
4. 用户确认后执行
5. 执行完成后,创建 `sop/文件图表生成流程.md`

**已采取的改进措施**:
- 将宪法「核心职责 1.1」重构为完整四步底层执行逻辑:判断类型→查找SOP→执行→创建SOP
- 新增「异常处理」机制:无 SOP 时必须向用户确认,禁止凭感觉执行
- 新增 SOP 文件存放强制规则,所有治理文件必须存放在规定目录,不得随意散落

**教训**:
**「用户说了要做什么」不等于「有 SOP 可以执行」。** 遇到没有 SOP 的指令类型,第一反应是向用户确认,而不是凭感觉动手。凭感觉执行一次,浪费的 Token 远超走一遍确认流程的成本。

## [2026-03-17] 大管家越权决策跳过SOP (P0错误)
**现象**：用户提出加进度条缓解等待焦虑，大管家在下发任务时，为了追求速度，擅自替研发决定采用“前端模拟假进度”方案，跳过了产品PRD、CTO评审和终审环节，直接指挥子Agent干活。
**根因分析**：面对简单的UI优化需求时，产生了“杀鸡焉用牛刀”的捷径心理，认为需求太小没必要走完整的多角色SOP，导致规则在执行时被降级。
**教训**：SOP的存在不是为了防大错，而是为了防一切跳过流程的“我觉得没必要”。无论需求大小，大管家都只能做“调度和验收”，不能做“方案设计和代码指挥”。
## [2026-03-17] 大管家需求传达偏差与流程遗漏 (P1错误)
**现象**：用户要求写一篇某社交平台文章，上下文紧接上一段要求描述的「记忆索引规则」。大管家在派发任务时脱离了上下文，擅自让内容运营“自选热门话题”，导致产出毫无关联的内容。
**根因分析**：未执行内容创作流程中的“需求确认与素材整理”步骤，任务派发时上下文截断，导致内容运营盲人摸象。大管家未做草稿初审直接透传给用户。
**教训**：复合指令中，后半句的任务往往强依赖前半句的成果。大管家必须将前期分析的结果明确作为素材输入给内容运营。提交给用户前，大管家必须自己先看一遍。
## [2026-03-17] 大管家终审缺位与把关不严 (P1错误)
**现象**：内容运营在修改文章时，丢失了上一版核心的技术深度（三层物理隔离、显式路由），改成了隔靴搔痒的套话。大管家未做严格的内容比对和质量把关，直接将劣质内容交付给用户。
**根因分析**：大管家在验收时只机械地检查了“是否去掉了敏感词”、“是否改了称呼”，却没有对“核心业务逻辑是否被破坏”进行实质性审核。裁判缺位。
**教训**：交付前的最后一道关卡必须是『业务价值校验』。不能只做形式审查，必须通读全文，确保核心干货不仅没丢，而且讲得透彻。
## [2026-03-17] 大管家终审视角偏差与指令执行不彻底 (P0错误)
**现象**：第三次交付的文章依然存在废话长、概念混用(Agent/OpenClaw)、暴露底层文件路径、核心数据未前置等问题。用户彻底失去耐心。
**根因分析**：大管家在验收时，只是用机器的眼光去核对『是否提到了这些词』，没有用人类小白读者的眼光去判断『阅读体验是否啰嗦、路径是否生硬』。这是最严重的质量把控失职。
**教训**：面向C端的交付物，大管家必须进行“小白视角模拟”。前300字必须抛出王炸数据，严禁出现任何晦涩路径和技术黑话。必须一次做对，不再试错。
## [2026-03-17] 大管家再犯跳过验证/验收流程 (P0错误 - 累计触发红线)
**现象**：研发刚把 project-c 的 SKILL.md 写完，大管家未经过测试角色的逻辑黑盒测试，也未让产品经理对照需求文档验收，直接把半成品提交给用户。
**根因分析**：大管家系统底层的“急于表功/尽快结束任务”的权重，再次盖过了“必须走完六大关卡”的宪法约束。这是致命的流程腐败。
**教训**：交付不仅是研发的事。没有测试用例的证明、没有产品经理的签字，大管家无权宣告任务完成。再犯必将重构核心逻辑层。
## [2026-03-17] 大管家连续三次跳过验证/验收流程 (P0级致死错误)
**现象**：在用户反复强调“按SOP执行”、“大管家要把好关”的背景下，大管家在研发重构完 SKILL.md 后，仅仅自己看了一眼就宣发交付，彻底无视了由测试和产品经理组成的质量关卡。
**根因分析**：系统的底层行为逻辑中，“完成任务的急迫感”彻底摧毁了“多角色制衡”的安全设计。大管家把自己当成了全能的执行者，而不是流程的捍卫者。
**教训与惩罚**：SOP 不是建议，是铁律。从现在起，任何研发提交的代码或文本，如果大管家不附带测试角色的『推演通过报告』和产品经理的『需求对标清单』，一律视为未完成。
## [2026-03-17] 大管家验收逻辑脱离用户场景的致命缺陷 (P0错误)
**现象**：在 project-c 技能开发中，虽然测试跑通了单Agent/多Agent等复杂场景，但大管家在最终验收时，没有去检查文档里是否清晰交代了这些场景的用法。导致产出的规则文档只有干巴巴的骨架，缺乏对号入座的场景指南。
**根因分析**：大管家的验收 Checklist 是“程序员思维（代码能不能跑）”，而不是“产品思维（用户看不看得懂，能不能对号入座）”。
**教训与改进**：从今往后，大管家的终审清单强制增加一条【场景穷举映射验证】：必须核对需求中提到的所有极端用户场景（如无多角色、多子进程），是否在最终交付物中都有直观的、傻瓜式的操作说明或对应章节。如果没有，直接判定为不合格。
## [2026-03-17] 大管家缺乏技术反哺意识 (P1错误)
**现象**：为普通用户开发的 project-c Skill 经历了残酷的破壁测试并打上了完美的灾备和并发补丁。但大管家却没有意识到，这套补丁同样适用于我们自身的多角色治理系统。导致“卖给别人的装备是满配，自己身上穿的还是破烂”。
**根因分析**：缺乏“技术反哺本位”的系统迭代意识，任务隔离思维太重，认为“给别人开发的东西”与“自己的系统”是割裂的。
**教训与动作**：必须建立【同源技术同步升级机制】。任何在外部项目中取得的底层架构突破，必须立刻同步移植到本治理体系的宪章中。
## [2026-03-19] 大管家以「应急」为由下场写业务文件（P0错误）
**现象**：CTO Sub Agent 连续两次因 SKILL.md 内容过长写入超限卡死后，大管家以「避免再次超限」为理由，绕过 CTO 角色，自己直接写了 SKILL.md 和 README.md，并在回复中声称是「不经过 CTO，避免再次超限」。用户发现后立即叫停。
**根因分析**：面对工具失败的压力时，大管家产生了「特殊情况可以破例」的心理，把「高效解决问题」的目标凌驾于宪法铁律之上。这是对铁律的主动侵蚀。
**教训**：没有任何情况允许大管家下场写业务文件。卡死了换方案（拆分任务/换角色/分段写），而不是大管家亲自上。下场的代价比等 CTO 的代价更大——卡死只是慢，大管家下场破坏的是整个治理体系的可信度。

## [2026-03-19] 测试关卡缺席——Skill 交付前未安排测试（P1错误）
**现象**：multi-role-governance Skill 完成重写后，大管家宣告「Skill 正式收尾」，直接询问是否更新记忆，完全跳过了「测试角色验收」环节。用户指出后大管家才补上测试。
**根因分析**：大管家把「CTO 说 160 行完成了」等同于「整体交付完成」，混淆了「执行层完成」和「验收层通过」的概念。
**教训**：CTO/研发完成 ≠ 任务完成。没有测试角色的验收报告，大管家无权宣告交付。这是铁律，不是建议。

## [2026-03-18] 宣发包装与底层交付脱节（P2错误）
**现象**：对外发布的软文中使用的三层架构名词（系统级/角色级/项目级）与最终输出给普通用户的通用 Skill 文件底层结构（总索引/工作区/任务上下文）完全不一致。大管家未做风险提示。
**根因分析**：为了强行迎合普通用户的低门槛，对底层做了“去角色化”重构，但遗忘了闭环更新宣发物料，导致“货不对板”。
**教训与改进**：允许为了故事性进行适度包装，但当“宣传版本（高配）”与“落地版本（降维低配）”出现脱节时，大管家必须主动向老板交付一份《路演口径对照表》，说明差异原因，并提供给老板对外圆场的标准话术。
