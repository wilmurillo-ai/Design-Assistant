# Bug排查修复流程图(并行工作版)

## 🔥 核心改变:从串行到并行

### ❌ 旧版问题(串行模式)
```
定位(等) → 方案设计(等) → 评审(等) → 修复(等) →
代码审查(等) → 测试(等) → 验收(等) → 终审(等)
(每个环节都在等待,效率极低)
```

### ✅ 新版设计(并行模式)
```
研发定位 + CTO协助 → 并行定位
CTO+产品并行评审方案 → 同时评审
研发修复 + CTO实时review + 测试准备 → 并行工作
测试+产品并行验证 → 同时验证
```

---

## 📋 前置步骤: 研发必须先读日志

> ❌ 禁止不读 pitfalls.md 和 task-logs/ 就开始修复 Bug

研发接到 Bug 修复任务时，**必须按以下顺序读取，不得跳过**:

```
Step 1: 读 sessions/{项目}/context.md           → 了解项目现状
Step 2: 读 sessions/{项目}/pitfalls.md           → 检查是否有相关踩坑记录
Step 3: 读 sessions/{项目}/task-logs/            → 查找同类Bug的历史执行记录
Step 4: 如有历史 _bug.md → 读对应文件            → 拿到完整错误现场
Step 5: 开始执行，同步创建 _work.md 任务工作日志
```

---

## ⚡ 6个并行工作阶段

### 并行1: Bug分级评估
**并行工作**:
```
产品经理 → 评估业务影响、优先级
测试 → 评估技术严重度、影响范围
```

**同时评估,不是依次评估**

---

### 并行2: Bug定位-多角色协作
**并行工作**:
```
研发 → 定位根因、分析代码
CTO → 协助定位、提供技术支持
测试 → 补充复现信息、创建 _bug.md 日志（含完整错误堆栈+复现步骤）
```

**日志写作规范**: 测试创建 `sessions/{项目}/task-logs/YYYY-MM-DD_T{任务ID}_bug.md`，格式按 `references/rules/任务执行日志规范.md §3.2` 模板填写（共7项：Bug描述、复现步骤、期望行为、实际行为、完整错误堆栈、环境信息、截图/日志路径 + 研发填写的初步判断）。

**关键点**:
- 研发不是单打独斗
- CTO可以**实时协助**,不是等研发定位完才参与
- 测试同步提供信息支持

---

### 并行3: 方案评审-同时评审
**并行工作**:
```
CTO → 技术方案评审,检查是否涉及删代码
产品经理 → 业务方案评审,影响范围评估
```

**不是**:
```
❌ CTO评审通过 → 产品经理再评审
```

**同时评审,汇总意见**

---

### 并行4: 修复阶段 ⭐⭐ 最关键

**并行工作模式**:
```
研发修复代码 ────────┐
  ↓ 每次commit      │
CTO实时review ───────├─→ 持续交互
  ↓ 及时反馈        │
研发及时修正 ─────────┤
                    │
测试准备测试用例 ────┘
```

**关键点**:
- 研发**不是修复完全部才提交review**
- CTO**实时review每次commit**
- 测试**同步准备测试用例**,不是被动等待
- **小批次反馈,及时发现问题**

**好处**:
- 及早发现"删除其他代码"问题
- 避免修复完才发现方向错了
- CTO可以实时指导

---

### 并行5: 验证阶段-并行验证 ⭐
**并行工作**:
```
测试验证 ─────────┐
  - Bug修复验证  │
  - 回归测试     ├─→ 同时进行
                 │
产品验证 ─────────┘
  - 业务功能验证
  - 用户体验验证
```

**不是**:
```
❌ 测试通过 → 产品经理再验证
```

**同时验证,大管家汇总结果**

---

### 并行6: Bug关闭-并行归档
**并行工作**:
```
测试归档Bug记录，更新 _bug.md 状态 ──┐
研发写 _fix.md 修复记录，判断是否写 pitfalls.md ──├─→ 同时进行
产品更新文档 ──────────────────────────────────┘
```

**研发归档动作清单**:
1. 在 `_fix.md` 或追加到 `_bug.md` 末尾，写明：根本原因、修复方案、修改的文件
2. 判断此 Bug 是否需要写入 `pitfalls.md`（会再次出现的坑 → 写入；一次性问题 → 不写）
3. 如果写入 pitfalls.md，格式：`P{序号} [已解决] {坑的描述} — 解法：{一句话}`

## 🛡️ 五重防护机制(并行版)

### 防护1: 方案并行评审
- CTO + 产品经理**同时评审**方案
- CTO重点检查:"是否涉及删除/修改其他代码?"

### 防护2: 实时代码review ⭐⭐ 最关键
**工作方式**:
```
研发每次commit ─→ CTO实时review
         ↓
   发现问题及时反馈
         ↓
   研发立即修正
```

**检查重点**:
- 🚫 **删除的代码是否超出 Bug 直接定位范围（函数/方法级别）**
- 🚫 **是否有跨模块/跨功能点的代码被删除**
- ✅ commit message 是否有删除说明
- ✅ 修复逻辑是否正确
- ✅ 代码质量

**铁律**: 发现跨范围删除代码，**立即终止，重新设计方案**

### 防护3: 并行验证
- 测试 + 产品经理**同时验证**
- 回归测试检查其他功能

### 防护4: 产品验收
- 确保业务逻辑正确

### 防护5: 大管家终审
- 汇总所有反馈,最后把关

---

## 💡 实时review机制

### CTO如何实时review?

**方式1: 监控代码提交**
```bash
# CTO可以设置提醒
git log --oneline --author=研发 --since=today
```

**方式2: 定期批量review**
- 不是每个commit都立即review
- 可以每隔1-2小时批量review
- 但要及时,不要拖到最后

**方式3: 关键commit重点review**
- 涉及核心逻辑的commit
- 涉及删除代码的commit
- 涉及公共函数的commit

## 🚀 效率提升

### 串行模式耗时(假设P2 Bug)
```
定位(1天) → 方案设计(0.5天) → 评审(0.5天) →
修复(1天) → 代码审查(0.5天) → 测试(1天) →
产品验收(0.5天) → 终审(0.5天)
= 6天
```

### 并行模式耗时
```
 并行定位(0.5天,多人协作)
并行方案评审(0.5天,同时评审)
修复+实时review(1天,同时进行)
并行验证(0.5天,同时验证)
终审(0.5天)
= 3天
```

**效率提升**: 约50%

---

## ⚠️ 注意事项

### 1. 实时review不是打断工作
- 研发可以连续工作,commit后继续
- CTO可以批量review
- 但要及时,不要拖到最后

### 2. 并行验证要充分
- 测试和产品各自验证要充分
- 不是为了并行而降低标准

### 3. P0紧急Bug可加快节奏
- 各环节可以更紧密协作
- 但**代码审查不能跳过**

---

## 🚫 8项严格禁止

1. **禁止未经方案评审就修复代码**
2. **禁止删除 Bug 直接定位范围之外的代码** ⭐⭐ 最重要
   - ✅ 允许：Bug 直接定位到的函数/方法内部错误逻辑（需在 commit message 说明原因）
   - ✅ 允许：明确标注为 TODO/测试桩/临时代码（需在 commit message 说明原因）
   - ⚠️ 须评审：同文件/同模块内其他代码（需 CTO review 通过）
   - ❌ 禁止：跨文件/跨模块/其他功能点的代码（无例外）
3. **禁止注释掉其他功能的代码**
4. **禁止跳过代码审查**
5. **禁止跳过测试验证**
6. **禁止直接在生产环境修改代码**
7. **禁止同时修复多个Bug**
8. **禁止"试试看"式修复**

---

## 📋 实时review检查清单

### 研发每次commit前自检
- [ ] 我只修改了必要的代码吗?
- [ ] 我是否删除了其他功能的代码?
- [ ] commit message是否清晰?

### CTO review时检查
- [ ] 修复逻辑是否正确?
- [ ] **删除的代码是否在 Bug 直接定位范围内（函数/方法级别）？** ⭐⭐
- [ ] **commit message 是否注明了删除原因和原代码功能说明？**
- [ ] 是否有跨模块/跨功能点的代码被删除或修改？
- [ ] 代码质量是否达标?
- [ ] 是否可能引入新bug?
