# 任务板管理流程 SOP

**适用类型：** 多任务并行管理、后台任务调度、任务状态追踪、结果回收
**版本：** v1.2
**创建日期：** 2026-03-11
**最后更新：** 2026-03-11（新增调查+执行分离模式、工具类型选择指南）

---

## 核心原则

**任务板是大管家的调度中枢。** 凡是 spawn 出去的后台任务，必须登记到任务板；**任务完成后大管家必须主动通知用户**，不等用户来问。

---

## 主动通知机制（强制）

这是任务板的核心规则，**不得省略**。

### ⚠️ 关于 run_in_background 的环境限制（重要）

**当前环境中，`run_in_background=true` 对需要写文件/执行命令的任务完全不可用。**

原因：后台子 Agent 无法获得 Write/Bash 工具的执行权限，所有文件操作会被自动拒绝，输出文件为空，且静默失败不报错。

### 正确的并行执行方式：同步并行

在**同一条消息**里发多个 Task 工具调用（不带 `run_in_background`），它们会并发执行，每个都有完整权限，结果直接返回主线程。

```
正确示例（同一条消息同时调用两个 Task）：
  Task1: 研发写 capability/chapter_engine.py
  Task2: 产品经理写 PRD_驻守型Agent.md
→ 两个任务并发执行，都能正常写文件，主线程等待两个都完成后继续

错误示例（带 run_in_background）：
  Task1 run_in_background=true: 研发写文件
→ 子 Agent 启动但 Write 权限被拒，文件为空，静默失败
```

### 调查+执行分离模式（Bug排查标准范式）

当任务是 Bug 排查时，必须先调查再修复，不能盲目动手。正确流程：

```
第一步（调查）：发一个 Explore 类型的 Task，专门读文件、分析代码，不写任何东西
  Task[Explore]: 调查 main.py 中项目启动逻辑，列出所有文件路径和关键代码

第二步（等结论）：等 Explore Task 返回完整调查报告

第三步（执行）：大管家或研发根据调查结论，直接用 Edit/Bash 修复
  （如果修改简单，大管家可直接用 Edit 修复；如果复杂，派给研发子 Agent）
```

**为什么要分离：**
- Explore 只读不写，速度快，不会误改文件
- 拿到真实代码路径后再修复，避免猜测路径导致修错地方
- 调查结论可以复用，不需要重复读文件

**工具类型选择：**
| 场景 | 用哪个 Task 类型 |
|------|---------------|
| 读文件、搜索代码、分析结构 | Explore |
| 写代码、修改文件 | general-purpose 或 Bash |
| 并行执行多个独立子任务 | 同一消息发多个 Task（不带 background） |

### 并行派发的正确步骤

```
步骤1：登记所有任务到任务板（含结果路径）
步骤2：在同一条消息里同时发出所有 Task 调用（不带 background）
步骤3：等待所有 Task 返回结果
步骤4：逐一检查每个 Task 的输出，确认文件已写入
步骤5：主动通知用户各任务完成情况（完成一个报告一个）
步骤6：更新任务板状态
```

### 多任务并行时的通知方式

多个任务并行时，**不等所有任务完成再统一汇报**，而是哪个先完成哪个先报：

```
T002 完成 → 立即告知用户「T002 PRD 已完成，结果在 /path/to/file.md」
T003 完成 → 立即告知用户「T003 技术方案已完成，结果在 /path/to/file.md」
T004 完成 → 立即告知用户「T004 任务面板已完成，访问 http://localhost:8003」
```

### 通知内容格式

```
✅ T002「任务描述」已完成
负责角色：产品经理
结果文件：/path/to/result.md
摘要：[1-2句话说明产出了什么]
```

### 禁止行为

- ❌ spawn 任务后不设监控，等用户来问才知道结果
- ❌ 任务已完成但任务板状态没更新
- ❌ 所有并行任务都完成了才统一汇报（应该完成一个汇报一个）

## 任务板文件

**位置：** `sessions/governance/task-board.md`

**生命周期：** 实时更新，任务完成后归档到 `templates/archives/tasks/`，任务板只保留「进行中」和「待处理」的任务。

---

## 任务状态定义

| 状态 | 含义 |
|------|------|
| `⏳ 待处理` | 已登记，尚未开始 |
| `🔄 进行中` | 已 spawn，正在执行 |
| `✅ 已完成` | 执行完成，结果可取 |
| `❌ 失败` | 执行出错，需重新处理 |
| `⏸️ 等待确认` | 需要用户拍板才能继续 |

## 流程步骤

### A. 收到复合指令时（一次发多件事）

1. **拆解指令**：将复合指令拆分为独立子任务
2. **识别依赖关系**：判断哪些任务可以并行，哪些有先后依赖
3. **批量登记**：将所有子任务写入任务板（含任务ID、负责角色、结果路径）
4. **并行 spawn**：同时启动所有可并行的任务
5. **告知用户**：回复用户「已拆解为X个任务，并行执行中，结果路径分别是…」
6. **等待回收**：后台任务完成后主动汇报，或用户来问时能立即答

### B. 登记单个后台任务时

每次 spawn 后台任务，必须在任务板记录以下信息：

```
| ID | 任务描述 | 负责角色 | 项目 | 状态 | 结果文件路径 | 启动时间 |
```

### C. 用户询问任务状态时

1. 读取 `sessions/governance/task-board.md`
2. 检查各任务的结果文件是否已存在
3. 汇报：哪些完成了（附结果路径）、哪些还在跑、哪些失败了

### D. 任务完成归档

1. 将已完成任务从任务板移动到 `templates/archives/tasks/YYYY-MM-DD.md`
2. 任务板保持干净，只显示活跃任务

## 关键规则

- **任何后台任务，spawn 前必须先登记**，不得先 spawn 后补记
- **结果路径必须在登记时确定**，不能用「待定」占位
- **任务板必须反映真实状态**，不得有「完成了但没更新」的情况
- **用户询问任务状态时，2步内必须给出完整答复**（读文件 + 汇报）
- **并行任务不等待彼此**，各自完成后独立报告

---

## 任务队列上限规则

| 分组 | 上限 | 超出处理 |
|------|------|---------|
| 进行中 | 5条 | 新任务进「待处理」队列，不强制并发 |
| 待处理 | 20条 | 达到20条时，大管家拒绝接受新任务，告知用户「待处理任务已达上限，请等待现有任务完成后再派发」 |
| 已完成 | 展示最近20条 | 超出部分归档到 templates/archives/tasks/ |

**大管家执行要求：**
- 收到新指令时，先检查「进行中」条数：≥5 条则新任务放入「待处理」
- 每次更新任务板后，检查「待处理」条数：达到20条立即提醒用户
- 禁止为了绕过上限而不登记任务


## 并行派发规则

收到复合指令时，按以下逻辑判断并行/串行：

**可以并行的情况：**
- 任务之间无输入输出依赖
- 不同角色负责的不同项目
- 同一项目中独立的模块

**必须串行的情况：**
- B 任务需要用 A 任务的输出作为输入
- 需要用户对 A 任务确认后，才能开始 B 任务
- 同一文件的读写操作

**示例：**
```
用户说：「①产品经理出架构图，②CTO写技术方案，③更新SOP」

→ T001（架构图）和 T002（技术方案）可并行，同时 spawn
→ T003（更新SOP）依赖讨论结论，等 T001+T002 完成后再执行
→ 告知用户：T001、T002 并行中，T003 等前两个完成后开始
```

---

## 错误处理

- **任务失败**：立即更新任务板状态为 `❌ 失败`，附上失败原因，向用户报告
- **结果文件不存在**：说明「任务可能未完成或路径有误」，重新 spawn，不能说「找不到结果」就结束
- **任务超时**：超过预期时间 2 倍仍无结果，主动检查并向用户报告
