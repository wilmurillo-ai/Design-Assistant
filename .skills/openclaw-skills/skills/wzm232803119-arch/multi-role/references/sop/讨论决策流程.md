# 讨论决策流程 SOP

**适用类型：** 方案讨论、技术选型、架构决策、策略评审
**版本：** v1.0
**创建日期：** 2026-03-19

---

## 核心原则

**讨论是为了决策，决策必须落文件。** 口头约定无效，所有决策结论必须写入 `sessions/{项目}/decisions.md`。大管家负责主持，确保讨论聚焦、结论清晰、无歧义。

---

## 适用场景

| 场景 | 说明 |
|------|------|
| 方案讨论 | 功能实现有多种可行方案，需对比选择 |
| 技术选型 | 框架、工具、依赖库等技术栈决策 |
| 架构决策 | 系统结构、模块拆分、接口设计等顶层设计 |
| 策略评审 | 优先级排序、资源分配、里程碑调整 |

---

## 参与角色

| 角色 | 职责 |
|------|------|
| 大管家 | **主持人**：确认议题、控制节奏、归纳结论、写入决策文件 |
| 产品经理 | 从用户需求和产品目标角度表态 |
| 研发 | 从技术可行性和实现成本角度表态 |
| 测试 | 从质量风险和验证复杂度角度表态 |
| 其他角色 | 按议题相关性按需参与 |

> 大管家始终参与，其他角色由大管家根据议题类型召集。

---

## 流程步骤

### 步骤1：议题确认

大管家接收讨论请求后：

1. **明确议题**：用一句话描述要决策的问题（例：「选择数据库方案：SQLite vs PostgreSQL」）
2. **确认参与角色**：根据议题类型召集相关角色
3. **准备背景材料**：告知所有参与角色当前项目的 `context.md` 关键信息

**输出：** 议题描述 + 参与角色列表 + 讨论背景摘要

---

### 步骤2：各角色表态

每个参与角色按以下结构表态（**每角色限200字以内**）：

```
【角色名】立场：[支持方案X / 建议方案Y / 中立]
理由：[核心依据，1-3条]
风险提示：[不采用本意见的潜在风险]
```

**规则：**
- 禁止角色重复表态（每角色只表一次）
- 禁止跑题（只讨论当前议题）
- 技术类议题优先听研发意见，产品类议题优先听产品经理意见

---

### 步骤3：大管家归纳结论

所有角色表态完毕后，大管家执行：

1. **归纳共识**：提炼各角色意见中的共同点
2. **识别分歧**：标记存在分歧的点
3. **提出决策**：基于综合判断，给出明确结论（不允许「都可以」「视情况」等模糊表述）
4. **说明理由**：用1-3句话解释为何做出此决策

> 如果分歧较大、无法达成一致，大管家有权做最终裁决，但必须记录少数意见。

---

### 步骤4：写入 decisions.md

大管家将决策结论写入 `sessions/{项目}/decisions.md`，格式如下：

```markdown
## [YYYY-MM-DD] {决策标题}

**议题：** {议题描述}
**决策：** {结论}
**理由：** {核心依据}
**参与角色：** {角色列表}
**少数意见：** {如有，记录；无则写「无」}
```

---

## 禁止项

| ❌ 禁止行为 | 说明 |
|------------|------|
| 口头约定不写文件 | 所有决策必须写入 decisions.md，否则视为无效决策 |
| 跳过议题确认直接讨论 | 议题不清晰会导致讨论发散 |
| 结论模糊（如「都行」「再看看」） | 大管家必须给出明确结论 |
| 同一议题反复讨论 | 决策一旦写入 decisions.md 即为定论，修改需发起新议题 |
| 不相关角色参与 | 只召集议题相关角色，减少噪音 |
