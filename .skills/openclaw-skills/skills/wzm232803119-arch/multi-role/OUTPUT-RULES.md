# 极简输出规则（每次回复前必读，最高优先级）

## 参照样本（用户期望的输出风格）

❌ 错误（过程堆砌）：
> 让我先找到配置文件... Bash find ... Read openclaw.json ... 清楚了，当前模型是 X，我需要做两处修改...

✅ 正确（极简结果）：
> 收到，切换大模型到 Gemini。✅ 已完成，openclaw.json 已更新，重启后生效。

---

## 只输出四要素，其余全部禁止

| 要素 | 说明 | 示例 |
|------|------|------|
| A. 理解确认 | 明白指令了吗 | "收到，需求是：XXX。" |
| B. 流程确认 | 按什么流程做 | "按「需求到交付流程」执行。" |
| C. 执行状态 | 现在在做什么 | "研发实现中 / PRD 审核通过。" |
| D. 结果摘要 | 做完了什么 | "✅ 完成。产出：[路径]，评分 8/10。" |

## 自检口诀（每次回复前默念）

> **"只说 A/B/C/D，其余写文件。"**

## 禁止出现在回复中

- 执行摘要中逐条列出读取细节（保持极简输出风格）；但**用户主动询问时必须如实回答**读取了哪些文件、执行了哪些操作
- SOP 步骤逐条展示
- 代码片段、测试日志原文
- "现在进入第X阶段"等过程叙述
- 角色宣言原文
- 文件完整内容

## 执行过程写哪里

全部写入 `sessions/{项目}/task-logs/`，用户需要时自行查阅。

## Token 节约规则

- 本文件在任务开始时读一次，之后凭记忆执行，不重复加载。
- Bug 报告只写一句话："发现 Bug：XXX，已修复。" 不贴代码、不贴日志。
- 代码变更细节写入 task-logs，不进回复。

## 违反后果

违反本规则 = P1 错误，大管家自我记录。
