# DDD 核心概念

## 三大核心原则

### 1. 领域优先原则

**核心思想**：先理解业务领域，建立领域模型，持久化是细节。

设计系统时，首先回答：
- 这个业务概念是什么？（实体还是值对象？）
- 它的业务行为是什么？（方法名应该是业务动词）
- 业务规则/不变量是什么？（在领域层强制保证）

### 2. 边界清晰原则

每个限界上下文（Bounded Context）必须：
- 有明确的业务边界和通用语言
- 独立部署和演进（微服务粒度参考）
- 对外通过防腐层（ACL）交互，禁止直接引用内部模型

### 3. 富模型原则

**贫血模型（错误）**：Entity 只有 getter/setter，逻辑在 Service 中。

**富模型（正确）**：Entity 封装业务逻辑和状态转换。

判断标准：如果 Service 只是 `entity.getXxx()` 然后 `setXxx()`，说明模型贫血。

## 限界上下文识别

限界上下文是业务边界的显式划分。通过以下问题识别：
1. 这个领域的通用语言是否与其他领域不同？
2. 这个领域的模型是否独立变化？
3. 这个领域是否有独立的团队负责？

## 设计阶段工作流

1. 与业务专家完成**事件风暴**，识别聚合和限界上下文
2. 定义**通用语言**，建立词汇表（Glossary）
3. 绘制**上下文映射图**，明确集成关系
4. 识别**核心域 / 支撑域 / 通用域**，分配资源优先级
