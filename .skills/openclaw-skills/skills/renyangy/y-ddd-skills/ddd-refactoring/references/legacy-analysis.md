# 遗留系统分析指南

## 分析步骤

### 1. 理解现有系统架构

**技术栈盘点**：
- 框架版本（Spring MVC / MyBatis / Hibernate）
- 数据库类型和版本
- 部署模式（单体 / 微服务）
- 主要依赖库

**代码结构盘点**：
- 包结构（按层还是按模块）
- Controller / Service / DAO 层文件数量
- 代码行数分布
- 测试覆盖情况

### 2. 识别核心域 / 支撑域 / 通用域

**核心域（Core Domain）**：
- 业务差异化所在
- 团队投入最多精力的地方
- DDD 改造的首选目标

**支撑域（Supporting Domain）**：
- 必须存在但不是差异化因素
- 按需定制开发
- 优先级次之

**通用域（Generic Domain）**：
- 通用的解决方案
- 可直接采购或使用开源
- 避免自研

**识别方法**：

| 问题 | 帮助识别 |
|------|---------|
| 如果没有这个业务，公司还能运作吗？ | 核心域 vs 支撑/通用域 |
| 这个业务的复杂度高吗？ | 值得 DDD 建模 vs 直接方案 |
| 团队对这个业务的理解程度如何？ | 有领域知识 vs 需要深入挖掘 |
| 这个业务会频繁变更吗？ | 值得投资 vs 稳定不变 |

### 3. 找到承压点

**变更频率分析**：
```
通过代码版本历史，统计：
- 哪些类/模块被修改最频繁？
- 修改的原因是什么（新增功能、Bug修复、需求变更）？
```

**业务复杂度分析**：
```
识别：
- 业务规则复杂（多层 if/switch）
- 数据关联复杂（大量 JOIN）
- 跨模块调用复杂（Service 间相互依赖）
```

**风险点分析**：
```
识别：
- 多人修改的热点文件
- 缺少测试的代码区域
- 过度耦合的模块
```

### 4. 确定重构起点

**重构起点选择矩阵**：

| 维度 | 高价值 | 低价值 |
|------|--------|--------|
| 业务重要性 | 核心域 | 边缘功能 |
| 变更频率 | 高频变更 | 稳定不变 |
| 团队理解 | 领域专家熟悉 | 需要重新学习 |
| 技术风险 | 依赖简单 | 过度耦合 |

**从哪个聚合开始**：
1. 从**核心域**中选择
2. 选择**变更频繁**的区域
3. 选择**业务逻辑复杂**但可理解的部分
4. 避免从技术最复杂的部分开始（容易陷入泥沼）

### 5. 现有代码分析模板

```
## 模块名：XXX模块

### 技术现状
- 代码文件：XX个
- 代码行数：XXXX行
- 测试覆盖：XX%
- 主要依赖：XXX, YYY

### 业务现状
- 核心业务能力：XXXX
- 涉及的数据库表：X张
  - 主表：XXX（XXX条记录）
  - 关联表：XXX, YYY
- 业务规则数量：X条
- 日均调用量：XXXX次

### 改造评估
- 改造复杂度：高/中/低
- 改造收益：高/中/低
- 改造优先级：1-5
- 建议改造方式：XXX
```

## 常见遗留系统类型及策略

| 类型 | 特征 | 策略 |
|------|------|------|
| 单体大泥球 | 单一数据库，大量表交织 | 先划分限界上下文，再逐步拆解 |
| 贫血服务集 | 轻 Service + 重 DAO，无业务逻辑 | 从核心 Service 开始富化 |
| 神类（God Class） | 几千行的 Service/Manager 类 | 按职责拆分为多个聚合 |
| 跨域交织 | 多个业务域代码混在一起 | 通过防腐层隔离，逐步提取 |
