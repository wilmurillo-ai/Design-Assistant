# DDD 新项目开发流程

## 五阶段流程

```
事件风暴 → 限界上下文 → 领域建模 → 骨架生成 → 编码与审查
```

## 阶段一：事件风暴（Event Storming）

与业务专家一起，通过事件风暴识别：
- **领域事件**（Domain Event）：已发生的业务事件
- **命令**（Command）：用户或系统触发的操作
- **聚合**（Aggregate）：处理命令、维护不变量
- **策略**（Policy）：业务规则
- **读模型**（Read Model）：用户关注的信息

产出物：
- 事件风暴模型图
- 聚合列表及职责
- 限界上下文候选

## 阶段二：限界上下文划分

基于事件风暴结果，划分限界上下文：
1. 识别每个上下文的**核心业务能力**
2. 确定上下文的**边界和名称**
3. 建立上下文的**通用语言**
4. 绘制**上下文映射图**

## 阶段三：领域建模

对每个限界上下文进行详细建模：
1. 识别**聚合**及聚合根
2. 定义**实体**和**值对象**
3. 确定**领域服务**和**领域事件**
4. 设计**仓储接口**
5. 编写**通用语言词汇表**

## 阶段四：骨架生成

生成项目结构和基础代码：
1. 创建多模块 Gradle 项目
2. 按包结构规范搭建骨架
3. 实现各层的基础 stub 类
4. 配置依赖关系

## 阶段五：编码与审查

按战术设计规范编写代码：
1. 实现聚合根和实体
2. 实现值对象
3. 实现仓储接口
4. 实现应用服务
5. 实现防腐层（外部集成）
6. 编写单元测试
7. 代码审查

## 各阶段检查点

| 阶段 | 检查点 | 通过标准 |
|------|--------|---------|
| 事件风暴 | 所有核心业务流程已覆盖 | 业务专家确认 |
| 上下文划分 | 上下文边界清晰 | 无重叠的通用语言 |
| 领域建模 | 聚合职责明确 | 符合小聚合原则 |
| 骨架生成 | 依赖方向正确 | ArchUnit 验证通过 |
| 编码 | 符合战术设计规范 | 代码审查通过 |
