# New Skill Checklist

Use this checklist whenever a new skill is installed, updated, or dropped into the workspace.

## Basic check
- Read the skill name and description
- Decide whether it is unrelated, adjacent, or conflicting with Shared Memory OS
- Look for memory / heartbeat / learning / reflection / long-term storage language

## If unrelated
- Do nothing
- Do not force memory notes into pure domain skills

## If adjacent but compatible
- Add the smallest useful shared-memory note
- Avoid rewriting the whole skill

## If conflicting
- Do not blindly merge
- Record the conflict candidate
- Review before adapting

## Questions to ask
- Will this skill create parallel memory files?
- Will it define a competing heartbeat workflow?
- Will it encourage noisy long-term logging?
- Does it need a shared-memory coexistence note?
- Is there a reusable pattern worth harvesting?

## Output options
- no action
- add compatibility note
- log conflict candidate
- log pattern candidate
