# 安全制度

## 安全等级定义

| 等级 | 含义 | 示例 | 处理方式 |
|------|------|------|---------|
| 绝密 | 泄露将造成严重损失 | 密钥、密码、客户隐私数据 | 加密存储，最小权限访问 |
| 机密 | 不宜公开的内部信息 | 商业计划、内部流程 | 内部共享，禁止外传 |
| 公开 | 可对外发布的信息 | 产品文档、公开报告 | 正常存储，无特殊限制 |

## 通用铁律

1. **隐私 > 功能**：任何功能开发不得以牺牲隐私为代价
2. **禁止泄露密钥**：任何场景下不得在代码、日志、文档中硬编码密钥
3. **最小权限原则**：只申请完成任务所需的最小权限
4. **默认私有**：仓库、文件默认私有，除非明确要求公开
5. **数据脱敏**：展示数据时，敏感字段必须脱敏处理

## 存储安全建议

- 敏感信息使用加密存储或环境变量
- 不在代码仓库中存储密钥、令牌
- 定期轮换密钥和访问凭证
- 使用 `.gitignore` 排除敏感文件
- 备份文件同样需要加密保护

## Git 仓库安全规范

- **禁止**：提交包含密钥、密码、令牌的代码
- **禁止**：在 commit message 中提及敏感信息
- **必须**：使用 `.gitignore` 排除 `.env`、`config.json` 等敏感文件
- **必须**：代码审查中检查敏感信息泄露
- **建议**：使用 pre-commit hook 自动扫描密钥
- **建议**：敏感分支设置访问权限控制

## 安全事件响应

1. 发现安全漏洞 → 立即上报管理者
2. 泄露确认 → 立即轮换相关密钥
3. 影响评估 → 评估影响范围和数据量
4. 修复执行 → 修复漏洞并验证
5. 复盘总结 → 记录教训，完善制度
