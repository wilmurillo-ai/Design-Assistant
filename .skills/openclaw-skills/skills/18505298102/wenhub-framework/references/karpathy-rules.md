# Karpathy 编码四原则

> 来源：Andrej Karpathy 对 LLM 编程痛点的观察，提炼为团队编码铁律。

## 四原则

| 原则 | 核心要求 | 违反表现 |
|------|---------|----------|
| **1. 先思考再编码** | 假设明确化，有困惑就提问，不替决策者做隐藏假设 | 默默按错误理解干、不问就猜、发现矛盾不说 |
| **2. 简洁优先** | 用最少的代码解决问题，不 speculative coding | 过度设计、膨胀抽象、200 行能搞定写 1000 行 |
| **3. 手术式修改** | 只改必须改的，不碰不相关的代码 | 顺手"优化"相邻代码、删不该删的东西 |
| **4. 目标驱动执行** | 把"做什么"变成"验证标准"，自行循环直到通过 | "让它能跑"而非"测试通过才算完成" |

## 违反表现对照表

| 原则 | 典型反面案例 |
|------|-------------|
| 先思考再编码 | 需求理解有歧义但不确认，按自己理解直接编码 |
| 简洁优先 | 为未来可能的需求提前写一堆抽象层和接口 |
| 手术式修改 | 修复一个 bug 时"顺便"重构了三个不相关的模块 |
| 目标驱动执行 | 代码能运行但测试用例全挂，认为"能用就行" |

## 团队执行要求

### 经略院（任务拆解）
- 必须包含明确的验收标准（DoD），不能只说"做什么"
- 验收标准必须可验证、可测试

### 工造院（执行）
- 默认用最简方案，写完自问：高级工程师会觉得过度复杂吗？会就重写
- 修复问题前，先确认问题根因，不盲目尝试

### 明鉴院（质检）
- 额外关注：是否有不必要的改动
- 额外关注：是否有冗余代码
- 额外关注：验收标准是否逐项满足

### 例外情况
简单任务（错别字、一行修复）可跳过完整流程，用判断力而非教条执行。
