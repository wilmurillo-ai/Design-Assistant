# 输入规则

这份文档只负责说明合同起草时的输入要求与默认处理。

## 最低输入

通常至少需要：

1. 甲方
2. 乙方
3. 合作内容主题，或合作方向列表

标题默认沿用模板原题，除非用户明确要求改标题。  
如果用户没有主动提供甲乙方介绍，OpenClaw 应在起草前主动补齐背景信息摘要。

## 推荐补充输入

如用户已知，可继续提供：

1. 模板路径
2. 生效日期
3. 合作期限
4. 特殊条款增删要求
5. 已知需要保留或修改的主体表述
6. 甲乙方简介或希望采用的对外介绍口径

## 敏感字段默认处理

除非用户明确提供，否则以下字段默认留空或保持待确认：

1. 身份证号
2. 签署日期
3. 联系人和联系方式
4. 盖章细节
5. 详细签约人身份信息

如果主体是机构而不是自然人：

1. 不要强制填写个人身份证语义
2. 模板中的自然人字段应尽量中性化或留空

## 正文起草要求

当用户只给简短主题时：

1. 应扩写为正式合同语气的正文
2. 不要只保留提纲式标题
3. 不要只改合作内容区而忽略联动条款

## 公司介绍搜集规则

1. 起草前应尽量补齐甲乙方公开介绍，并各自压缩为约 150 字。
2. 先检查当前 agent 工作区的 `MEMORY.md` 与 `memory/` 中是否已有准确版本。
3. 只有在记忆中找不到或信息不足时，才使用已安装的第三方检索能力，例如 Tavily 或其他搜索 skill。
4. 摘要应聚焦主体定位、主营方向、平台能力、产业资源或实施能力，不要堆砌宣传口号。
5. 不要编造未核实的融资、专利、牌照、资质、政府合作或市场排名。
6. 若公开资料不足，应使用审慎中性表述，并在未决事项中标注“公司介绍待补充核实”。
7. 用户明确要求“记住”的版本，应写入永久记忆，后续合同任务优先复用。

当前标准 job spec 输入以 `cooperation_items` 为准。  
不要再沿用仅包含 `content_sections` 的旧版简化写法。

runner 对少数常见错误输入会做自动归一化，例如：

1. `cooperation_items` 里误用 `title/description`
2. `global_replacements` 被写成单个对象
3. `paragraph_overrides` 被写成单个对象

但 OpenClaw 不应依赖这种兜底能力。  
如果仍出现 schema error，应基于错误原因一次修正，不要连续盲试同类格式错误。
