# Troubleshooting Findings

## Issue 1: 87 TASK files
**Root cause:** Multiple invocations of `cmd_plan --force` during testing/development created duplicate TASK files. The `.ail/plans/` directory is now empty (cleaned up), but the original spike produced many duplicates of the same task.

**Evidence:**
- `seed_queue()` (line 693) calls `cmd_plan(force=True)` once → creates exactly 1 task (TASK-001)
- `cmd_plan()` has no guard against re-creation when `--force` is used; it always calls `next_task_id()` which increments the counter and writes a new file
- `cmd_plan(force=True)` was likely called multiple times during our testing (development workflow), creating TASK-001, TASK-002, ... TASK-087
- TASK-001.md and TASK-002.md (recovered from earlier reads) contain **identical content** ("为 a-current 增加完整计划文档回显能力"), confirming they were duplicates generated by repeated `--force` calls, not from `seed_queue` creating 87 at once
- `next_task_id()` (scripts/task_ids.py) simply scans for the highest number and adds 1 — no deduplication against existing task titles

**Secondary factor:** No mechanism prevents the same task from being re-generated with `--force`. Each `--force` call creates a new TASK-### file.

## Issue 2: HEARTBEAT.md redundancy
**Root cause:** HEARTBEAT.md is a legacy artifact from the old queue-based design. The new roadmap-driven flow uses only ROADMAP.md as source of truth. HEARTBEAT.md is now functionally unnecessary but is still written by `init_queue_heartbeat()` during `adopt`/`onboard` for backward compatibility with any tooling that still reads it.

**Evidence:**

1. **HEARTBEAT.md stores old-format queue rows** — the `pending_queue_rows()` / `write_pending_queue()` functions only operate on HEARTBEAT.md, never on ROADMAP.md. These are called exclusively in `cmd_adopt()` when handling existing projects with legacy state.

2. **New flow (cmd_plan / cmd_trigger / cmd_current) uses only ROADMAP.md:**
   - `cmd_plan`: writes current task to ROADMAP.md via `set_current_task()`
   - `cmd_trigger`: calls `append_done_log()` which writes to ROADMAP.md
   - `cmd_current`: reads from ROADMAP.md
   - `cmd_start` cron message explicitly tells agents: `"IMPORTANT: ROADMAP.md is the source of truth. Do not rely on the old HEARTBEAT queue flow."`

3. **HEARTBEAT.md Run Status fields** (`last_run_time`, `last_run_commit`, `last_run_result`, `last_run_task`, `cron_lock`, `mode`) — these are written by `init_queue_heartbeat()` and `update_run_status_mode()`, but the new trigger loop doesn't update them. They reflect the last time the **old** cron-based queue ran, not the new roadmap-driven execution.

4. **HEARTBEAT.md still written** because `cmd_adopt` calls `init_queue_heartbeat()` unconditionally (line 961) to set up the file structure. On a first adopt with no existing queue (`pending_queue_rows()` returns empty), it still writes the empty queue template + run status to HEARTBEAT.md.

**Could HEARTBEAT be removed?**
- Yes, from the core roadmap-driven flow
- No, if external tools/scripts still parse HEARTBEAT.md queue format
- Recommended: deprecate it, stop writing it during adopt, but keep it readable for diagnostic purposes

## Issue 3: Done Log content
**Root cause:** The 2 entries in `.ail/HEARTBEAT.md`'s Done Log are legacy entries carried over from the old system (before migration). `cmd_trigger` has **not** run during our current testing session — `.ail/ROADMAP.md` correctly shows `_暂无完成记录_`.

**Evidence:**

1. `.ail/HEARTBEAT.md` Done Log entries:
   ```
   | 2026-04-20T21:47:34Z | 6d24858 | 为关键用户流程增加集成测试 | pass |
   | 2026-04-20T22:11:00Z | aef1dca | 为关键用户流程增加集成测试 | pass |
   ```
   Timestamps are from **April 20**, before today's migration/adopt.

2. `.ail/ROADMAP.md` Done Log:
   ```
   ## Done Log
   _暂无完成记录_
   ```
   Confirms no executions have been recorded in the new system.

3. `cmd_trigger()` (line 1754) calls `append_done_log()` which writes to **ROADMAP.md**, not HEARTBEAT.md. The HEARTBEAT Done Log is a separate legacy section that predates the new flow.

4. `cmd_trigger` was never invoked during our testing (the cron never ran because we never did `a-start`, and manual trigger was not called).

**Conclusion:** No issue — the old Done Log in HEARTBEAT.md is historical. The new Done Log in ROADMAP.md is correctly empty.
