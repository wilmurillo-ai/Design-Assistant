# 图片理解模块 (Image Reader)

本模块负责理解 IM 图片并将内容转化为结构化 Markdown。

## 模块 A：图片类型识别

收到图片后，首先判断图片所属类型，以决定后续处理路径。

### 类型标签与判断特征

| 类型标签 | 判断特征 | 处理路径 |
|---------|---------|---------|
| `text-document` | 含大量文字、表格、段落排版 | → 文字提取 |
| `screenshot-ui` | 软件界面、网页截图、对话截图 | → 结构化描述 |
| `flowchart` | 含流程节点、判断框、连线、箭头 | → 文字摘要 + Mermaid 重绘 |
| `architecture-diagram` | 系统组件、服务拓扑、云架构 | → 文字摘要 + Excalidraw 重绘 |
| `mind-map` | 树状发散结构、中心主题 | → 文字摘要 + Mermaid 重绘 |
| `relationship-diagram` | ER 图、UML 类图、依赖关系 | → 文字摘要 + Mermaid 重绘 |
| `data-visualization` | 数据图表、柱状图、折线图、饼图 | → 文字摘要 + Canvas 重绘 |
| `photo-or-scene` | 实物照片、场景图 | → 简洁描述 |
| `mixed` | 图文混排、复合类型 | → 分区处理 |

### 识别要求

- 类型判断置信度低时，归为最接近的类型，不得阻塞主流程
- 同一图片可标记多个类型标签（如架构图含文字说明）
- 输出时在 YAML frontmatter 中标注 `image-type`

## 模块 B：内容提取与 Markdown 结构化输出

**核心原则**：输出结果必须是 Agent 可直接理解和引用的 Markdown。

### B1 文字内容提取

**适用场景**：`text-document`、`screenshot-ui`、`mixed`

**处理规则**：
- 保留原始文字的层级结构（标题 → 正文 → 注释）
- 表格识别后以 Markdown 表格格式输出
- 代码块、命令行识别后用 ` ``` ` 包裹
- 多栏排版按逻辑顺序（从左到右、从上到下）合并为线性 Markdown

**输出示例**：

```markdown
## 图片内容（文字文档）

### 第一部分：产品需求

用户需要一个支持...

| 功能 | 优先级 | 负责人 |
|------|--------|--------|
| 登录模块 | P0 | 张三 |

> 备注：截止日期为 Q2 末
```

### B2 UI 截图 / 界面截图

**适用场景**：`screenshot-ui`

**处理规则**：
- 描述界面结构：顶部导航、侧边栏、主内容��、弹窗等
- 标注关键交互元素：按钮文字、输入框 placeholder、错误提示
- 若含对话内容（IM 聊天截图），按时序逐条列出发言

**输出示例**：

```markdown
## 图片内容（UI 截图）

**界面类型**：Web 管理后台  
**主要区域**：
- 顶部导航栏：含「Dashboard」「用户管理」「设置」三个菜单项
- 主内容区：展示一个数据表格，共 5 列（ID / 姓名 / 邮箱 / 角色 / 操作）
- 右上角：显示「导出 CSV」按钮（蓝色，已高亮）

**关键信息**：表格第 3 行显示报错状态（红色标记），错误信息为「邮箱格式无效」
```

### B3 图表类图片的文字摘要

**适用场景**：`flowchart`、`architecture-diagram`、`mind-map`、`relationship-diagram`、`data-visualization`

**处理规则**：
- 在调用图表重绘之前，先输出简要的文字摘要
- 让 Agent 快速理解图表语义
- 提取节点、连线、层级关系等关键信息

**输出示例**：

```markdown
## 图片内容（流程图摘要）

**图表主题**：用户注册流程  
**节点数量**：约 8 个  
**主要路径**：
1. 用户填写表单 → 邮箱验证 → 创建账户
2. 验证失败 → 返回填写表单（循环）

**关键判断节点**：「邮箱是否已注册」、「验证码是否正确」
```

### B4 照片/场景图

**适用场景**：`photo-or-scene`

**处理规则**：
- 简洁描述场景主要内容
- 不触发图表重绘模块

**输出示例**：

```markdown
## 图片内容（照片）

**场景**：办公室��议室  
**主要内容**：白板上写有项目时间线，包含 Q1-Q4 四个阶段的里程碑
```

## 标准输出格式

每次处理图片，统一输出如下结构：

```markdown
---
skill: obsidian-viz
image-type: [flowchart | text-document | screenshot-ui | data-visualization | ...]
has-diagram: true | false
---

## 图片内容摘要
（2-3句话快速描述图片主要内容）

## 详细内容
（模块 B 的完整输出）

## 下一步
（如果包含图表，说明将调用哪个工具重绘；否则流程结束）
```

## 边界条件与异常处理

| 场景 | 处理方式 |
|------|---------|
| 图片模糊/低分辨率 | 输出已识别内容，注明「部分内容因图片质量无法识别」 |
| 图片为空白或纯色 | 返回 `image-type: empty`，提示用户重新发送 |
| 图表结构过于复杂（>50节点） | 输出文字摘要 + 关键路径，注明「图表过于复杂，仅生成主干结构」 |
| 非图片文件（误触发） | 立即返回，不处理，不报错 |
| 多图同时发送 | 逐张独立处理，输出合并为一个 Markdown，每张用分隔线区分 |

## 与图表重绘模块的接口

当图片类型包含图表时，向 `chart-router.md` 传递以下信息：

1. **图表类型**：flowchart / architecture-diagram / mind-map / relationship-diagram / data-visualization
2. **结构化数据**：
   - 节点列表（文字、类型、层级）
   - 连线关系（起点、终点、标签）
   - 布局建议（左右/上下/中心发散）
3. **原始图片**：作为参考（如需要）

图表重绘模块将根据这些信息选择合适的工具并生成文件。
