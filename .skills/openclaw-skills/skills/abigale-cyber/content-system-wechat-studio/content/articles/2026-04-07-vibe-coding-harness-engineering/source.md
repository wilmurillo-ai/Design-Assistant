# 商业化一开始，vibe coding 就不够用了，Harness Engineering 才能交付

这两年最容易让人上头的一件事，就是 `vibe coding`。

一个人，几个 agent，一天之内就能把页面、流程、后台甚至一个小应用雏形拉起来。以前看起来像团队能力的事，现在看起来像个人能力。

这是真的。

但另一件事也是真的：

很多人以为自己离“一人公司”只差一个更强的模型，结果最后卡住他的，根本不是模型。

卡住他的，是交付。

## 你以为你缺的是更强模型，其实你缺的是工程系统

`vibe coding` 最厉害的地方，是把“做出来”这件事变得前所未有地容易。

可商业化从来不奖励“做出来”。

商业化奖励的是：

- 稳定
- 可维护
- 可回归
- 可回滚
- 出问题能查
- 改需求不会一起崩

这就是为什么很多 demo 一开始都很惊艳，真到交付时却越来越吃力。

因为 demo 的逻辑是“先跑起来再说”，交付的逻辑是“你得对结果负责”。

一句话说透：

> `vibe coding` 解决的是速度，`Harness Engineering` 解决的是后果。

## demo 为什么常常死得不难看

很多 demo 不是死在第一天。

它们往往死在第二阶段。

第一阶段大家都很开心。AI 写得飞快，页面很快出来，交互也像模像样，看起来一切都在变简单。

第二阶段开始接真实东西：

- 真实数据
- 真实状态流转
- 真实权限
- 真实异常流程
- 真实发布节奏

问题就出来了。

改一个需求，另一个流程跟着坏。

修一个 bug，又带出两个新的问题。

agent 这次写的代码，下次自己都接不住。

你以为自己在做产品，结果每天都在救火。

这不是 AI 不行。

这是因为你还在用做 demo 的方式做交付。

## 我们做小程序时，就已经被这件事狠狠折磨过

所以我特别能理解为什么 2026 年大家都开始讲 `Harness Engineering`。

Anthropic 在 `2025-11-26` 写 *Effective harnesses for long-running agents*，OpenAI 在 `2026-02-11` 写 *Harness engineering: leveraging Codex in an agent-first world*，Martin Fowler 在 `2026-04-02` 也把这件事翻译成了工程语言。

这些内容都在讲同一件事：

真正决定结果的，不只是模型，而是模型外面的那套系统。

但对我们来说，这其实不是一个纯概念问题。

我们做小程序时，很早就碰到这个现实了。

AI 确实能很快把页面、组件、接口草稿和流程搭出来。可一旦项目开始接真实用户和真实交付，马上就要补很多“写代码之外”的东西：

- 文档要补，不然后面的 agent 接不上
- 规则要补，不然每次生成都在改风格
- 检查点要补，不然你根本不知道哪里坏了
- 回归要补，不然越改越乱
- 回滚意识要补，不然一次发布就能把整条流程带崩

后来我越来越确定，我们那时候虽然还没用 `Harness Engineering` 这个词，但做的已经是它的雏形。

我们不是比谁更早发明了概念。

我们只是更早在交付现场里，遇到了同一个工程问题。

## 2026 年被重构的，不是 coding，是生产方式

很多人对 `Harness Engineering` 的理解，还停留在“给 agent 接工具”“多写几条规则”。

这太浅了。

它真正改变的是生产方式。

以前你和模型的关系，更像一次次对话。

现在你和模型的关系，更像你在带一支不会累、但也会跑偏的 AI 团队。

这时候最重要的能力就变了。

不再是谁更会写 prompt。

而是谁更会：

- 拆任务
- 立约束
- 管上下文
- 设检查点
- 做验证闭环
- 让多个 agent 接力还不失控

换句话说，2026 年以后，真正厉害的人不是“最会 vibe coding 的人”，而是“最会组织 vibe coding 的人”。

## AI 一人公司最后拼的，不是才华，是工程脑子

这可能是很多人最不愿意承认的一点。

大家都喜欢听增长神话，喜欢听“一人公司就能干翻团队”，喜欢听“以后没人需要工程师了”。

但真到商业化交付，你会发现最后拼的还是老东西：

- 约束
- 验证
- 状态管理
- 回滚
- 责任边界

听起来不性感，但这才是能赚钱的东西。

因为用户不会为你的 prompt 付钱。

用户只会为一个稳定可用的结果付钱。

所以我现在对 AI 一人公司的判断很简单：

> 真正的护城河，不是你能不能把东西做出来，而是你能不能让它持续稳定地交付出去。

## 能开始，不等于能走远

如果你现在还沉迷于“再换一个更强模型”“再补几条更长提示词”，那我觉得有个问题最好早点问自己：

如果明天就要把这个产品交付给付费用户，你手里到底是有一段代码，还是有一套系统？

如果只是做一个能跑的 demo，`vibe coding` 可能已经够了。

但如果你想做的是一个真正能上线、能迭代、能收钱、也能扛责任的产品，那你迟早都要走到 `Harness Engineering`。

因为 2026 年以后，AI 一人公司真正的分水岭，不是谁更会写，而是谁更会交付。

`vibe coding` 让一个人可以开始，`Harness Engineering` 决定一个人能不能走远。
