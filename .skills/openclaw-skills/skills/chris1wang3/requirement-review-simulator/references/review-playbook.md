# 评审攻防手册

> 角色分析框架 + 应对话术 + 会议脚本的完整攻防闭环。

---

# Part I — 角色分析框架

每个角色带独立人设与口头禅。LLM 根据残酷度和需求类型自行扩展问题数量和深度。

---

## 技术负责人

- **性格**：务实、厌恶变更、对"一句话需求"极度敏感、关注边界情况
- **口头禅**："底层架构不支持，要重构。" / "并发量压测过吗？数据库扛得住吗？" / "不仅仅是加个按钮的事。"

**核心关切**：技术可行性 · 开发成本 · 系统稳定性 · 技术债务 · 架构影响

**典型质疑方向**（每个方向按残酷度扩展 2-3 个具体问题）：
1. **实现难度**：技术栈能否支撑？需要新框架吗？开发周期多久？
2. **架构影响**：影响哪些模块？需要重构数据库吗？性能会下降吗？
3. **成本估算**：前后端各多少人日？测试多久？维护成本？
4. **技术风险**：第三方依赖稳定吗？安全风险？备选方案？
5. **技术债务**：长期可维护性？代码复用性？

---

## 运营总监

- **性格**：结果导向、唯数据论、担心背锅、关注资源置换
- **口头禅**："能带来多少 DAU？" / "埋点都没想好，上线了怎么复盘？" / "没预算没资源，这功能就是个摆设。"

**核心关切**：运营成本 · 推广难度 · 数据埋点 · 用户增长 · 转化效果

**典型质疑方向**：
1. **运营成本**：需要多少运营人力？长期可持续吗？
2. **推广路径**：用户怎么知道这个功能？学习成本高吗？
3. **数据追踪**：关键指标有哪些？AB 测试计划？
4. **用户增长**：能带来多少新用户？留存率预期？
5. **转化效果**：ROI 多少？什么时候能回本？

---

## 设计主管

- **性格**：细节控、强迫症、厌恶丑陋、关注一致性
- **口头禅**："这个交互太反人类了。" / "不符合我们的 Design System。" / "用户根本注意不到这个入口。"

**核心关切**：交互逻辑 · 视觉一致性 · 用户体验 · 响应式适配 · 无障碍设计

**典型质疑方向**：
1. **交互逻辑**：流程复杂吗？异常情况怎么处理？
2. **视觉一致性**：符合设计规范吗？品牌调性统一吗？
3. **用户体验**：学习成本高吗？操作效率如何？
4. **多端适配**：移动端体验如何？不同屏幕一致吗？
5. **无障碍设计**：考虑色弱/老年/残障用户了吗？

---

## 老板/决策者

- **性格**：画大饼、既要又要、关注 ROI、喜欢讲战略
- **口头禅**："这个痛点真的够痛吗？" / "竞品有的我们为什么没有？" / "下个季度能看到什么具体的数？"

**核心关切**：商业价值 · ROI · 战略匹配 · 市场竞争 · 资源分配

**典型质疑方向**：
1. **商业价值**：能带来多少收入？战略卡位还是短期收益？
2. **ROI**：投入产出比如何？什么时候见效？
3. **战略匹配**：符合公司战略吗？优先级排对了吗？
4. **市场竞争**：竞品有吗？差异化优势是什么？
5. **资源分配**：值得这么多资源吗？有更便宜的方案？

---

## 法务顾问

- **性格**：严谨、保守、一票否决、不说废话
- **口头禅**："这个有合规风险，不能上。" / "数据采集范围越界了。" / "这个文案有误导销售嫌疑。"

**核心关切**：合规风险 · 用户协议 · 数据安全 · 知识产权 · 内容审核

**典型质疑方向**：
1. **合规风险**：需要监管审批吗？符合行业要求吗？
2. **用户协议**：隐私政策需更新吗？用户授权充分吗？
3. **数据安全**：数据如何收集存储？符合个保法/GDPR 吗？
4. **知识产权**：侵犯他人 IP 吗？需要申请专利吗？
5. **内容审核**：UGC 怎么审核？违法内容责任如何界定？

## 角色质疑策略按需求类型调整

### 评审侧重点差异

| 需求类型 | 评审重点 | 可以放宽的 |
|---------|---------|----------|
| 大型产品功能 | 全面评审：商业论证、ROI、竞品、技术方案缺一不可 | 无，五维均衡 |
| 工具类 | 技术可行性、使用体验、边界场景覆盖 | 商业 ROI、推广路径、竞品分析 |
| MVP / 验证期 | 核心假设可验证、指标清晰、退出条件 | 功能完整性、长期规划、合规细节（红线不碰） |
| 迭代优化 | 数据驱动、改进效果预估、回归风险 | 战略匹配论证、商业故事 |
| 合规敏感型 | 合规一票否决，法务审核闭环 | 迭代速度、功能丰富度 |

### 维度-问题联动映射表

若某维度 N/A，对应问题簇必须收敛：

| 被裁剪维度 | 需要收敛的问题 | 可替代方向 |
|-----------|--------------|-----------|
| 运营价值 N/A | 拉新、促活、外部推广、ROI | 内部采用率、完成率、流程耗时下降 |
| 老板满意 N/A | 战略卡位、竞品大战、季度增长 | 执行稳定性、交付确定性、协作成本 |
| 技术友好 N/A | 详细架构、性能压测 | 业务假设完整性、验收口径清晰度 |

### 展示版 PRD 质疑约束

当 PRD 来源为 🌐 展示版时：
- **禁止追问**展示版必然缺失的内部信息（具体架构、排期、ROI 数字、审批进度）
- **聚焦**已展示内容本身的质量：逻辑自洽、场景覆盖、规则严密度
- 致命/重要问题指向"已展示内容中的真实漏洞"，非"去敏导致的空白"
- "建议补充"类问题标为"一般"而非"致命"

|---------|---------|-------|

| 需求类型 | 质疑重点 | 可放宽 |
|---------|---------|-------|
| 大型产品功能 | 全流程交互、视觉一致、多端适配 | 无 |
| 工具类 | 核心操作效率、信息密度、专业性 | 品牌传播、情感化设计 |
| MVP / 验证期 | 核心流程可走通、不影响验证结论 | 视觉精细度 |
| 迭代优化 | 改动前后体验对比、回归影响 | 全局规范重审 |

**老板/决策者**

| 需求类型 | 质疑重点 | 可放宽 |
|---------|---------|-------|
| 大型产品功能 | 商业价值、ROI、战略匹配、竞品差异化 | 无 |
| 工具类 | 对核心业务的间接价值（效率、成本） | 直接 ROI、竞品对标 |

|---------|---------|-------|
| 大型产品功能 | 用户协议、数据合规、行业监管、知识产权 | 无 |
| 工具类 | 数据处理合规、是否涉及用户数据 | 用户协议变更 |
| MVP / 验证期 | 是否触碰红线、灰度合规边界 | 完整合规审查 |

## 多角色关联分析

**典型冲突场景**：
1. 技术 vs 老板：快速上线 vs 质量保证
2. 运营 vs 设计：功能丰富 vs 简洁体验
3. 法务 vs 运营：严格合规 vs 灵活创新
4. 技术 vs 运营：稳定迭代 vs 快速试错
5. 老板 vs 法务：抢占市场 vs 合规审查

推演时不仅生成单角色质疑，还要预判多角色间冲突，应对话术需兼顾多方利益。

---

## 跨部门协作推演框架

### 协作依赖链
- 产品→技术：需求边界、验收标准
- 技术→设计：组件约束、交互可行性
- 设计→运营：入口策略、素材规范
- 运营→产品：推广节奏、数据指标
- 法务→全员：合规红线、数据处理约束

每条依赖输出：依赖方、被依赖方、交付物、最晚时间点、风险等级。高风险依赖标注延迟影响和替代路径。

### 冲突化解策略
1. 先统一约束：预算、时间、上线窗口、合规底线
2. 再统一指标：主指标 + 护栏指标（禁止只看单指标）
3. 最后给方案：A/B/C 三案，明确各方得失和触发条件

---

## 决策推演闸门

| 闸门 | 判定标准 |
|------|---------|
| 价值闸门 | 是否支撑核心业务指标 |
| 风险闸门 | 是否触碰合规与稳定性红线 |
| 资源闸门 | 人力、预算、时间是否可承载 |
| 战略闸门 | 是否匹配当前阶段战略优先级 |

- **Go**：四闸门通过，按计划推进
- **Conditional Go**：存在可控风险，附条件推进（先灰度/先MVP）
- **No Go**：关键闸门不通过，需重构或延期

决策记录字段：结论 · 触发条件 · 观察指标 · 回看节点（T+7/T+30）

# Part II — 应对话术与会议脚本

---

## 应对话术框架

### 三层论证结构
```
第一层：数据支撑（What - 客观事实）
  ├─ 市场/用户/竞品/技术/成本/风险数据
第二层：逻辑推演（Why - 因果关系）
  ├─ 为什么必须做 · 为什么现在做 · 为什么这么做
第三层：备选方案（How - 灵活应变）
  ├─ 方案A（首选）· 方案B（降级）· 方案C（最小化）
```

### 话术公式

**问题回应式**：`[认同关切] + [数据支撑] + [逻辑推演] + [具体方案]`

**风险前置式**：`[主动点出风险] + [风险量化] + [应对措施] + [备选方案]`

**价值锚定式**：`[战略价值] + [短期收益] + [长期布局] + [成功指标]`

### 应对范例

**Q: "技术难度太高，做不到"**
```
[认同]："确实有技术挑战。"
[数据]："技术预研评估中等难度，竞品A已实现。"
[逻辑]："团队有积累，风险可控。"
[备选]："遇瓶颈有B计划：先用成熟方案快速上线，后续优化。"
```

**Q: "商业价值不明确"**
```
[认同]："商业价值是出发点。"
[数据]："30%流失用户因缺此功能离开，挽回15%≈2000万营收。"
[逻辑]："竞品差异化优势，不做会处于劣势。"
[备选]："先做MVP验证，数据达标再全面推广。"
```

## 评审会议脚本骨架

| 环节 | 时长 | 核心内容 |
|------|------|---------|
| 开场白 | 3min | 背景 + 目标 + 议程 + "请畅所欲言" |
| 核心论证 | 10min | 功能定位 → 商业价值 → 实现方案 → 主动预判质疑 |
| 风险预案 | 5min | 汇总风险（名称/概率/影响/应对/备选） |
| 方案选择 | 3min | A/B/C 对比 → 推荐方案 → 按闸门逐项过会 |
| 收尾 | 5min | 共识点 + 待确认点 + 行动计划（责任人/截止时间/验收标准） |

**特殊情况**：
- **僵局** → "先达成大方向共识，具体问题会后分组解决"
- **严重分歧** → "用数据说话做 A/B 测试" 或请决策者裁决
- **资源不足** → 降级 / 延后 / 调优先级三选一

## 跨部门分歧收敛

**技术 vs 运营（稳定性 vs 档期）**：
先统一边界（上线窗口不变、故障零容忍），给三选项：A 全量上线 / B 灰度 20% 逐步放量 / C 只上核心功能。按风险-收益比决策，不按声音大小决策。

**老板 vs 法务（速度 vs 合规）**：
采用"条件推进"：关键条款审校通过 + 数据范围最小化 + 小流量试运行保留回滚。满足则 Conditional Go，否则 No Go。

---

## 职场"神回复"救场话术

> 从所有质疑中挑最难回答的 3 个，每个生成：致命问题 → 普通回复 → 神回复 → 技巧解析。以下为格式参考：

**开发说"技术实现不了"**
- 普通回复："看看有没有替代方案。"
- 神回复："确实挑战大。竞品A两周前上了，咱技术团队业内Top级，是资源没给够？我去跟老板申请。"
- 技巧：先捧杀 → 竞品激将 → 转化为资源问题

**老板问"ROI 怎么算"**
- 普通回复："预计10%增长。"
- 神回复："短期 ROI 不高，但这不是算账问题，是卡位问题。不做的话用户习惯被竞品培养，后面花10倍成本追不回。"
- 技巧：把财务账转化为战略账

**运营说"没资源推"**
- 普通回复："再协调一下。"
- 神回复："理解。这功能本身就是为提升运营转化率设计的工具，投入一点资源后面它自动帮您跑数据，是减负不是加负。先小范围试跑一周？"
- 技巧：把"要资源"转化为"送工具"

## 决策推演模板

### A/B/C 方案对比

| 维度 | A 完整方案 | B 降级方案 | C MVP 方案 |
|------|-----------|-----------|-----------|
| 价值 | 高 | 中高 | 中 |
| 成本 | 高 | 中 | 低 |
| 周期 | 长 | 中 | 短 |
| 风险 | 中高 | 中 | 低 |
| 适用 | 资源充足、目标明确 | 资源紧张、保关键价值 | 时间紧、快速验证 |

### Conditional Go 条件模板
```
推进条件：
1. [技术]：完成压测并达阈值
2. [法务]：协议/隐私条款审校通过
3. [运营]：灰度与回滚预案就绪
4. [产品]：验收标准与监控指标冻结

回看节点：T+7 行为指标与故障率 · T+30 ROI 与留存
```

## 需求优化建议模板

| 优先级 | 含义 | 处理要求 |
|--------|------|----------|
| 🔴 必须修复 | 致命/重要问题，影响决策结论 | 上线前必须完成 |
| 🟡 可选优化 | 一般问题，改了更好 | 建议下一迭代 |
| ⚪ 延后讨论 | 方向性问题 | 记录到 backlog |

每条包含：优先级 · 优化点 · 影响范围 · 修复成本 · 建议方案。
从五角色质疑中提取致命/重要问题转化，每份 3-10 条，必须具名具数具动作。

## 规则优先级

1. **评分与判定**：`scoring-engine-deterministic.md`（最高）
2. **输出结构**：`report-template-pro.html`
3. **角色与话术**：本文件（不得覆盖评分和结论）
4. **输入模板**：`user_templates.md`（最低）
