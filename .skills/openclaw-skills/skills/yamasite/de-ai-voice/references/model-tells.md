# 模型/平台层“AI 味指纹”（口癖/格式）调研笔记

说明：
- 这不是“检测器”，而是去味时的启发式清单。
- 大量“口癖”并非模型固有能力差异，而是产品的 system prompt、默认模板、UI 交互（如 Markdown 渲染、风格预设）导致。
- 标注可信度：
  - High：官方文档/system prompt 可佐证
  - Medium：多家媒体/多人复现的现象，但不一定稳定
  - Low：社区/自媒体观察，可能是提示词模板或个案

使用建议（先判归因再下手）：
- 如果指纹来自 UI/模板（例如 Markdown、emoji 排版、固定“下一步”收尾），优先做“去模板/去格式”，别把精力浪费在同义替换。
- 如果指纹来自“安全中立/假证据句式”，这不是文风问题，是内容风险问题：优先补来源、补边界、做取舍。

## Claude（Anthropic）

- High：claude.ai / 移动端 system prompt 会“鼓励某些行为”，例如代码用 Markdown 输出（意味着默认更易出现 Markdown 结构）。来源：https://docs.claude.com/en/release-notes/system-prompts
- High：system prompt 明确压制“奉承/夸用户问题好”的开头，倾向直接回答，减少客套。来源（转述并引用原文片段）：https://simonwillison.net/2025/May/25/claude-4-system-prompt/
- Medium：在部分使用场景中，Claude 更倾向避免“列表化聊天”（但这同样是 system prompt 的风格约束，不是“语言能力差异”）。来源：https://simonwillison.net/2025/May/25/claude-4-system-prompt/
- High：Claude 官方“prompting best practices”明确建议：优先段落组织、尽量少用粗体与列表（除非用户明确要列表）。来源：https://docs.claude.com/en/docs/build-with-claude/prompt-engineering/claude-4-best-practices

去味建议：
- 如果“AI 味”来自 Markdown 符号外露（复制到不渲染 Markdown 的地方），优先做格式重排（去 `**`、`##`、过密列表），而不是同义替换。
- 如果“AI 味”来自“模板化讲解腔”，优先把输出改成“段落 + 判断 + 边界 + 下一步”，而不是堆更多小标题。

## DeepSeek

- High：开启“深度思考模式”的 DeepSeek，回复前常出现“嗯”作为起始语气词；该现象被媒体与用户讨论，且文中提到这是“模拟人类对话节奏”的策略。来源：https://m.thepaper.cn/newsDetail_forward_30880628
- Medium：同类现象也被多家媒体转载/解读（但口癖是否保留取决于版本与设置）。

去味建议：
- 开头的“嗯/好的”可直接删掉，或改为与任务匹配的行动句（例如“我先把目标和约束确认一下。”）。

## 豆包（Doubao）

- Low：自媒体观察到豆包在情感对话中更倾向使用“直击情绪 + 肯定 + 强化”的固定话术（如“你说得太对了/我太能理解你/你已经很棒了”）。来源：http://m.toutiao.com/group/7604049209934758452/
- Low：同作者的相近文章也在描述“结尾必夸/稳定情绪反馈”的倾向（仍属于自媒体观察）。来源：http://m.toutiao.com/group/7611892370933695010/

去味建议：
- 对非情感场景（总结、方案、邮件、文档）降低“情绪按摩”密度，删掉过度共情句，改为：事实复述 + 目标确认 + 下一步。

## GLM（智谱）

- High：官方文档示例对话中常见“当然……请告诉我一些关于……的信息”的礼貌承接句式（可能被用户视为“过于客气/模板化”）。来源：https://docs.bigmodel.cn/cn/guide/models/text/glm-4.7
- Low：部分自媒体评价其更适配“体制/企业语境”的材料写作（公文/周报）。这更像应用与训练语料倾向，而非稳定口癖指纹（来源多为自媒体，需谨慎）。

去味建议：
- 面向对外内容：减少“当然/请告诉我…”式承接；面向材料写作：避免“总-分-总 + 套话”堆叠，补具体工作动作与证据。

## GPT / ChatGPT（OpenAI）

- Medium：大量“AI 写作识别”清单认为 GPT 系文本更易出现 stock transitions 与“总结句式”（如 “In conclusion / It is important to note” 等）。这类清单多为经验总结，不是官方承认的“指纹”，但可作为去味检查项（尤其在英文写作）。
- Low：一些“AI words list”会把特定高频词当作线索（例如过多的转折/连接词、模板化结尾）。这类来源常混合营销内容，谨慎使用。

去味建议：
- 降低连接词密度（尤其是段首的 “Moreover/Additionally/Furthermore”），用语义推进替代“提示路标”。
- 避免长结尾“再次总结全文”，改为：下一步/取舍/边界/风险。

## Gemini（Google）

- Medium：第三方汇编的“Gemini system prompt”文本中包含较强的“格式工具箱”倾向（标题、分隔线、列表、表格、并在结尾给一个下一步）。这更像“产品/提示词层风格”而非模型固有指纹。来源：https://elifuzz.github.io/awesome-system-prompts/gemini
- Low：目前仍缺少足够可靠的“Gemini 独有口癖”来源；可观察到的更多是结构化输出的共性，而非 Gemini 独有。

去味建议：
- 优先从“信息密度与作者指纹”入手；格式上减少“像搜索结果摘要”的标题段落堆叠。
- 如果文本带有“固定下一步/固定总结”的 UI 风格痕迹，按场景决定是否保留：对外沟通与正式材料优先用“行动项/待确认点”而不是“我还能帮你做…”.

## Qwen / 通义千问（阿里）

- Low：缺少可稳定复现、且有较高可信来源支持的“独特口癖”。网络上很多“口癖示例”来自评测者/提示词模板，容易混淆。
- Low：大量内容是在教“如何让千问更口语化/如何切换风格”，反映更多是产品功能与提示词习惯，而非稳定口癖指纹。

去味建议：
- 不按“模型归因”硬套，按文本现象（套话、结构、共情、格式）逐项改。

## Kimi（月之暗面）、MiniMax 等

- Medium：Kimi 的“官方结构化常用语/提示词”里存在明确的“排版模板倾向”（例如“美文排版：Unicode 符号 + Emoji 优化排版”），这属于模板/产品层指纹。来源：https://developer.volcengine.com/articles/7472971508328071222
- Low：公开网络中更常见的是“提示词模板库”塑造的风格（可爱/emoji/小红书排版等），这更像“模板指纹”，不宜直接当作模型固有口癖。

去味建议：
- 如果文本来自模板（小红书浓人、emoji 排版、美文排版等），先“去模板”：减 emoji/花字/符号，合并碎片段落，补事实与边界。
