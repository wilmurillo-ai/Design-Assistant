# Example Outputs: 从输入到成品

用途：给 Skill 一个明确的“成品风格锚点”。当用户要写类似场景时，可先参考最接近的案例，再按对应模板输出。

## 示例 1：一页纸方案评审（决策模板）

### 输入

```text
写作场景（Scenario）：评审（要做决策）
面向读者（Audience）：技术负责人 + 产品负责人
目标（Goal）：能做取舍并决定是否推进
读者关心（Reader Concerns）：成本、风险、时间、收益
材料（Raw Notes）：
- 现状：发布频率低；故障恢复慢；跨团队协作成本高
- 方案 A：引入灰度发布 + 自动回滚（2 周）
- 方案 B：重构发布流水线（6 周）
- 约束：人手紧，不能影响本季度交付
必须包含/避免（Must include/avoid）：必须有替代方案与 trade-off；避免“更先进/更强大”
限制（Constraints）：一页纸；先给推荐结论；包含行动项与风险缓解
```

### 输出（示例）

```md
## 一句话结论（Recommendation）

在本季度交付压力与人手受限的前提下，建议优先采用“灰度发布 + 自动回滚”，用 2 周时间先解决发布风险，再决定是否投入更重的流水线重构。

## 决策结论摘要

- 推荐方案：方案 A，先引入灰度发布 + 自动回滚
- 推荐理由：投入小、见效快，能直接降低发布失败风险与恢复时间
- 不建议的方案：本季度内直接推进方案 B
- 决策时限 / 关键前提：本周内确认；前提是现有发布系统支持最小灰度与回滚钩子

## 金字塔要点（MECE）

1. 收益与影响
   - 方案 A 能优先解决“发布频率低、恢复慢”这两个当前最痛的问题
   - 2 周内可落地，收益更接近本季度目标
2. 成本与资源
   - 方案 A 所需改造范围小，当前团队可承担
   - 方案 B 需要 6 周，且会占用本季度核心交付资源
3. 风险与缓解
   - 风险：灰度规则不完善可能导致回滚判断不准
   - 缓解：先限定低风险服务试点，并补关键监控与回滚阈值
4. 替代方案与 trade-off
   - 方案 B 的长期收益更大，但本季度机会成本过高
   - 方案 A 是“先控风险，再谋优化”的折中路径

## 决策表

| 选项 | 优点 | 缺点 | 成本 | 风险 | 适用条件 |
|---|---|---|---|---|---|
| 方案 A：灰度 + 自动回滚 | 上线快、见效快、直接降低发布风险 | 长期平台化收益有限 | 2 周 | 规则不完善影响判断 | 本季度先稳发布 |
| 方案 B：重构流水线 | 长期收益更大、标准化更强 | 周期长、挤占交付资源 | 6 周 | 中途范围膨胀 | 资源充足且可接受延迟 |

## 行动项

- Owner：发布平台负责人
- Next Step：确认现有系统的灰度与回滚能力；选 1 个低风险服务试点
- ETA：本周完成评估，下周启动试点
- 验收/度量：发布失败回滚时间下降；试点服务灰度成功率稳定

## 目标达成评估

- 结论：已达成
- 理由：结论前置、取舍清楚、包含风险缓解与替代方案，足以支撑评审决策
- 下一步修改建议：
  - 增加试点服务的具体名单
  - 补一条回滚阈值的量化定义
```

## 示例 2：周报/项目进展（短格式）

### 输入

```text
写作场景（Scenario）：周报
面向读者（Audience）：直属上级 + 协作团队
目标（Goal）：快速对齐现状并推进下周工作
读者关心（Reader Concerns）：进度、风险、需要的支持
材料（Raw Notes）：
- 完成：接入新鉴权；修复 3 个 P1
- 进行中：压测、灰度、监控看板
- 风险：依赖团队接口变更未确认；压测资源不够
- 下周：灰度 10% -> 50%；补齐告警规则
必须包含/避免（Must include/avoid）：必须写清“需要谁支持什么”；避免流水账
限制（Constraints）：400 字以内；要点化
```

### 输出（示例）

```md
## 一句话结论（BLUF）

本周项目整体按计划推进，但灰度扩量前仍受接口变更确认和压测资源不足两个风险影响。

## 关键要点

1. 已完成新鉴权接入，并修复 3 个 P1 问题，核心功能稳定性提升。
2. 正在推进压测、灰度和监控看板完善，为下周扩量做准备。
3. 当前最大阻塞是依赖团队接口变更未确认，以及压测资源不足。

## 风险/阻塞

- 接口变更未确认：可能影响灰度扩量节奏
- 压测资源不足：可能导致扩量前验证不充分
- 需要支持：请依赖团队本周确认接口变更；请资源管理员补齐压测资源

## 下一步 / CTA

- 我将继续：完成 10% -> 50% 灰度方案与告警规则补齐
- 需要你确认：接口变更窗口与压测资源支持时间
- 时间点：下周三前完成扩量准备

## 目标达成评估

- 结论：已达成
- 对照 Success Criteria：状态清楚、风险突出、支持请求明确，适合快速扫读
- 下一步修改建议：
  - 若给高层看，可再压缩成 3 条 bullet
```

## 示例 3：对外技术传播（标准模板）

### 输入

```text
写作场景（Scenario）：对外文章
面向读者（Audience）：有工程背景的开发者
目标（Goal）：理解 + 知道何时用/不用 + 能跑通最小示例
读者关心（Reader Concerns）：真实收益、成本、坑点、替代方案
材料（Raw Notes）：
- 主题：解释“为什么需要 CI 缓存”，以及怎么做才不踩坑
- 读者误区：以为缓存越多越快；忽略一致性与命中率
- 需要包含：2 个真实场景 + 1 个反例 + 3 个坑点
必须包含/避免（Must include/avoid）：避免营销腔；必须写 trade-off
限制（Constraints）：1000 字以内；中文；结尾给一个可执行 checklist
```

### 输出（示例）

```md
## 一句话结论（Key Message）

CI 缓存的价值不是“缓存越多越快”，而是让稳定且可复用的步骤少做重复工作，同时控制一致性风险。

## 金字塔要点（MECE）

1. 为什么需要 CI 缓存
   - 重复下载依赖、重复构建会拉长流水线时间
   - 对高频提交项目，缓存能显著减少重复开销
2. 什么场景适合用
   - 依赖下载稳定、命中率高的项目
   - 构建步骤重复率高、产物可安全复用的项目
3. 什么场景不适合
   - 依赖变化极频繁、缓存失效率高
   - 产物与环境强绑定，缓存反而引入脏数据风险
4. 怎么避免踩坑
   - 用稳定的 cache key；不要把“经常变”的部分混进同一层
   - 定期验证命中率，不要只看缓存是否存在

## 风险与权衡

- 风险：缓存过度会增加一致性问题排查成本
- trade-off：命中率越高越快，但 key 设计越粗，脏缓存风险越高
- 反例：把所有构建产物一股脑缓存，并不会自动提升流水线效率

## 行动项

- 可执行 checklist：
  - 先找出最耗时且重复率高的步骤
  - 为依赖与构建产物分开设计 cache key
  - 观察命中率与失败率，再决定是否扩大缓存范围

## 目标达成评估

- 结论：部分达成
- 理由：结构和 trade-off 已清楚，但还缺两个更具体的真实场景例子
- 下一步修改建议：
  - 补 Node.js 与 Docker 两个场景例子
  - 增加一个“缓存命中率低反而拖慢”的反例
```
