# 考虑过但未采用的方案

## 方案A: 接入高考真题数据库 API
**描述：** 对接题库 API（万题库/猿题库），实现真题搜索
**未采用原因：** 需要付费 API key，增加部署门槛；LLM 本身对高考题目有足够覆盖
**未来考虑：** v2 可作为可选配置项

## 方案B: OCR 拍题识别
**描述：** 用户拍照上传题目，自动识别文字
**未采用原因：** 需要外部 OCR API，增加依赖；移动端截图+文字输入能覆盖大部分场景
**未来考虑：** v2 高优先级

## 方案C: 向量数据库存储知识点
**描述：** 将高考知识点向量化存储，实现语义搜索
**未采用原因：** 过度工程化；LLM 直接回答知识点问题质量已足够
**未来考虑：** 知识库规模极大时再考虑

## 方案D: 强制苏格拉底模式（禁止快速模式）
**描述：** 所有解题都必须经过引导，不允许直接给答案
**未采用原因：** 会让赶时间或需要快速验证答案的学生放弃使用
**决定：** 默认引导，允许切换

## 方案E: 多智能体（分科 Agent）
**描述：** 数学/语文/英语分别是独立 Agent
**未采用原因：** 高考生需要跨科关联和统一的学习档案；单 Agent 多模块更合适
