# 设计决策记录

## D01: 苏格拉底引导 vs 直接给答案
**决定：** 两者都支持，默认引导模式，用户可切换快速模式
**理由：** 强制引导会让赶时间的学生放弃使用；但默认引导有助于真正理解

## D02: 零外部 API 依赖
**决定：** 全部使用 OpenClaw 默认模型，不引入外部教育 API
**理由：** 降低使用门槛，开箱即用；高考知识点相对稳定，LLM 知识储备足够

## D03: 错题按知识点而非题型分类
**决定：** 知识点 × 能力层次双维归因
**理由：** 题型是表象，知识点是根因；能力层次（记忆/理解/应用/分析）决定学习策略

## D04: 情绪支持内嵌 vs 独立模块
**决定：** 自动检测关键词，内嵌触发
**理由：** 学生不会主动说「我需要情绪支持」，主动检测更自然

## D05: 院校数据静态化
**决定：** 使用静态 JSON 数据，不对接实时 API
**理由：** 高考数据年度更新，实时 API 成本高且可靠性不稳定；静态数据加免责声明更安全

## D07: 省份+年份作为必填建档字段
**决定：** 建档第2、3步强制收集高考年份和省份，所有回答基于此适配
**理由：** 各省考纲/卷型/选科规则/总分/志愿规则差异极大；通用回答会给出错误信息
**实现：** province-policy.md 覆盖全国31省市，建档后所有模块读取 profile 中的省份字段
**边界：** 政策不确定时明确说明，不猜测；建议用户核对官方最新公告

## D06: OCR 拍题放入 v2
**决定：** v1 不实现 OCR
**理由：** 需要外部 API（百度/阿里）或本地模型，增加部署复杂度；文字输入能覆盖主要场景
