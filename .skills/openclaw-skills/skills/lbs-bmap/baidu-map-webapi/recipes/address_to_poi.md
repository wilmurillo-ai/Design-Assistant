# 地址/地名转坐标与 POI UID：算路前的必要步骤

> **适用摘要**: 将用户输入的地址文本或 POI 名称，解析为算路所需的经纬度坐标（BD09LL）和 POI UID。所有涉及"输入地名/地址 → 算路/标注"的场景均须先完成此步骤。

## 何时参考本文

在以下场景中，**优先参考本文件**确定调用方式，再进行算路或地图操作：
- 用户说出地名或地标（"天安门"、"北京南站"、"故宫"、"海底捞王府井店"）
- 用户提供了详细地址（"北京市昌平区北清路甲8号"、"龙泽小区17-2-801"）
- 需要将地名/地址同时转换为**坐标 + UID** 的任何场景

---

## 判断：输入属于哪种类型？

| 输入类型 | 判断依据 | 示例 |
|---|---|---|
| **结构化地址** | 含省市区街道 + 门牌号/楼栋/单元/室号 | "北京市昌平区北清路甲8号"、"龙泽小区17-2-801" |
| **POI 名称** | 地标名/商家名/景区名，不含门牌号 | "天安门"、"故宫"、"海底捞"、"北京南站"、"首都机场T3" |

> **模糊情况**（如"北京大学"、"首都机场T3航站楼"）→ 优先选 **方案 B（POI 搜索）**，准确率更高。

---

## 方案 A：结构化地址 → geocoding API

用于含有街道门牌、小区楼栋编号等具体位置的地址。

**参考能力**: `../references/geocoding.md`

**从响应获取**:
- 坐标（主结果，始终返回）
- POI UID（地址精确匹配到 POI 时返回，**可能为空**）
- 是否精确打点
- 地址理解度评分（评分低时建议提示用户补全地址）

uid 为空的情况：地址解析到道路/区域级别（而非具体 POI），此时只能使用坐标算路。

---

## 方案 B：POI 名称/地标 → administrative_region_search API

用于地标名、商家品牌名、景区名等名称输入。

**参考能力**: `../references/administrative_region_search.md`

**从响应获取**:
- POI UID（直接可用，无需额外调用）
- 坐标
- POI 名称（建议向用户展示确认）
- 全量候选列表

> **当有多个同名候选**（连锁商家、同名景点）时，应将返回结果展示给用户选择，而非直接取第一条。

### 可选扩展：获取大型 POI 的导航引导点

机场、火车站、大型商场等的**主坐标在建筑中心**，导航时入口引导点更准确。
可在获取 uid 后，额外调用 `../references/place_detail_search.md`（开启扩展详情级别）获取导航引导点坐标。

**参考能力**: `../references/place_detail_search.md`

## 常见错误

| 错误 | 原因 | 解决方法 |
|---|---|---|
| 结构化地址坐标偏差大 | 地址过于模糊（理解度评分低） | 提示用户补充省市区 + 完整门牌号 |
| UID 为空（地址解析场景） | 地址匹配到道路/区域级别，非具体 POI | 可接受，仅用坐标算路（绑路精度略低） |
| POI 搜索返回多个同名结果 | 连锁商家 / 跨城市同名 | 展示候选列表供用户选择 |
| 搜索到外城市同名地点 | 未限制城市范围 | 补充城市名并开启城市限制 |
| 地标搜索无结果 | 关键词过长或含无关修饰词 | 简化关键词，如"首都机场T3航站楼" → "首都机场T3" |
