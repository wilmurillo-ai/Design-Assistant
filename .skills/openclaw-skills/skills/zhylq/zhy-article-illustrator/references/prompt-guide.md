# Nano Banana 配图提示词指南

本文档定义 `zhy-article-illustrator` 在 Step 3 生成图片时的提示词方法论。
目标不是“让模型随便画一张图”，而是为 Gemini Nano Banana / Gemini 原生代理
工作流产出高质量、可直接用于公众号文章的成品级专题视觉。

---

## 核心原则

### 1. 先统一，再变化

同一篇文章的多张图必须共享同一套视觉基线，再根据每个章节的内容变化布局和重点。

- 统一：主色系、图形语言、材质/光感、文字气质、完成度、信息密度
- 变化：版式方向、模块组织、图像类型、局部强调色、具体构图

不要让同一篇文章里出现“一张科技扁平、一张卡通手绘、一张海报拼贴”的混搭效果。

### 2. 先做成品，再做说明图

默认基线是“高完成度编辑视觉 / 专题配图”，不是随意插图，不是简陋 icon 图，
不是空洞概念图。

提示词必须让模型理解：

- 这是一张可以直接放进公众号正文的成品图
- 画面有明确的信息层级和版式秩序
- 既要好看，也要有清晰信息表达

### 3. 默认中文文字

图片中的可见文字默认使用简体中文。

- 主标题：中文
- 小标题 / 标签 / 注释：中文
- 英文只保留在白名单术语里，如产品名、协议名、品牌名、缩写
- 中文文字尽量短，避免一段话塞进图片导致渲染失真

### 4. 元素要具体，禁止抽象空话

不要写：

```text
创建一张关于 AI 编程的漂亮插画
```

要写：

```text
画面中央是一块深色代码编辑区，左侧是“需求输入”，右侧是“自动补全建议”，
底部是一条测试反馈时间线。右上角用一块小标签显示“效率提升”，整个画面采用
蓝青色科技编辑视觉，模块之间有清晰分区和留白。
```

### 5. 默认禁止英文 UI 残留

对 Xiaomi / Gemini 类图片模型，常见跑偏是自动生成英文界面残留，比如：

- `Documentation`
- `Dashboard`
- `Powered by`
- `User Stories`
- `PASS / Failed / Skipped`

除非这些词本身就在白名单里，否则应在提示词中显式要求全部改为简体中文。
如果画面需要展示产品名或框架名，可以保留白名单英文术语，但 UI 其余部分仍应中文化。

## 三层提示词体系

### 第一层：全局质量基线

所有文章都共享以下基线：

- 高完成度编辑视觉
- 版式感强
- 信息层级清晰
- 细节丰富但不杂乱
- 禁止低幼卡通、随意装饰、无意义图标堆砌
- 默认中文展示

### 第二层：文章级 visual bible

每篇文章先生成一份 `visual-bible.md`，定义本文统一视觉语言。

至少包括：

- `visual_theme`：这篇文章整体气质
- `color_system`：主色 / 辅色 / 强调色 / 背景
- `graphic_language`：线条、图形、材质、阴影、光感
- `layout_discipline`：模块密度、留白、信息分区方式
- `text_policy`：默认中文 + 英文白名单术语
- `negative_rules`：明确禁用的视觉方向

### 第三层：单图级结构化 prompt

每张图必须在文章级 visual bible 的基础上生成，不允许脱离文章统一风格单独发挥。

## Nano Banana 友好的提示词骨架

建议每张图都按以下顺序组织：

```text
[任务定位]
这是一张可直接用于公众号正文的高完成度专题配图，不是简单插画，不是装饰图。

[统一风格锚点]
继承本篇文章的统一视觉基调：[visual_theme]
主色系统：[color_system]
图形语言：[graphic_language]
版式纪律：[layout_discipline]

[本图目标]
本图要传达的唯一核心信息：[core_message]

[画面内容]
- 主体元素：...
- 辅助元素：...
- 背景与空间层次：...
- 必须出现的信息模块：...

[布局结构]
- 画面分区：...
- 阅读动线：...
- 标题区 / 内容区 / 强调区位置：...

[文字规则]
- 所有可见文字默认使用简体中文
- 仅以下术语允许英文：[english_terms_whitelist]
- 需要显示的标题：...
- 需要显示的标签：...
- 需要显示的注释：...
- 非白名单英文 UI 文案必须改成中文
- 不要出现英文导航、英文按钮、英文图例、英文状态词

[质量要求]
- 高信息密度但不拥挤
- 视觉层级明确
- 细节丰富
- 更像编辑专题视觉，而不是普通插图

[禁止项]
- 不要低幼卡通
- 不要随意图标拼贴
- 不要英文乱码
- 不要无意义装饰
- 不要风格漂移

[技术约束]
- 宽高比：16:9
```

## 中文文字规则

### 推荐做法

- 标题：4-12 个字
- 标签：2-8 个字
- 注释：不超过 12 个字
- 单张图的可见文字块尽量控制在 3-7 组

### 英文使用白名单

只允许以下情况保留英文：

- 产品名：`Playwright`、`Notion`
- 浏览器 / 框架名：`Chromium`、`React`
- 协议 / 标准名：`HTTP`、`REST API`
- 行业约定缩写：`ROI`、`CTR`

除此之外，默认转为中文表达。

### 不推荐做法

- 在一张图中塞进整段中文说明
- 同时出现大量中英混排
- 把所有正文内容都放进图片
- 让模型默认补出英文界面词、英文面板标题、英文图表图例

---

## 如何保持同篇统一风格

### 必须统一的维度

- 主色系
- 图形语言
- 模块边角风格（圆角 / 直角 / 卡片感）
- 明暗与材质气质
- 文字风格
- 完成度与信息密度

### 可以变化的维度

- 横向构图 / 纵向构图
- 对比图 / 流程图 / 场景图 / 架构图
- 局部强调色
- 前中后景组织方式

### 判断标准

如果把同一篇文章的 4 张图放在一起看，应当让人觉得：

- 是同一组视觉系统
- 但不是同一张图复制粘贴

## 按内容类型选择表达方式

| 内容类型 | 推荐图像形式 | 提示词重点 |
|----------|--------------|------------|
| 概念解释 | 信息图 / 类比图 | 把抽象概念转成可视化模块 |
| 对比分析 | 左右对比图 / 双栏海报 | 两侧逻辑对称，差异点明确 |
| 步骤流程 | 流程图 / 阶段路线图 | 阅读顺序、编号和箭头清晰 |
| 架构框架 | 模块图 / 层级图 | 层级、连接关系、职责分区清楚 |
| 数据统计 | 数据海报 / 图表可视化 | 少量关键数字，高对比强调 |
| 场景叙事 | 编辑场景图 / 情境插画 | 情绪氛围、人物动作、空间关系 |
| 工具产品 | UI mockup / 编辑界面图 | 界面结构准确，重点模块突出 |

## 差提示词 vs 好提示词

### 差提示词

```text
画一个关于 Playwright 的架构图，科技感一点。
```

问题：

- 太抽象
- 没有版式
- 没有具体模块
- 没有中文文字规则
- 没有说明它应该是成品级视觉

### 好提示词

```text
这是一张可直接用于公众号正文的高完成度专题配图，不是简单插画。

继承本篇文章统一的科技编辑视觉：深蓝灰背景，青蓝高亮，细线连接，圆角卡片模块，
信息密度高但留白克制。

本图要表达的核心信息是：Playwright 通过统一测试层连接多个浏览器引擎，实现跨浏览器自动化。

画面采用三层纵向架构：顶部是“测试代码层”，中部是“协议与调度层”，底部是三个并列浏览器引擎。
每个模块都有清晰中文标签，底部浏览器名称允许保留英文：Playwright、Chromium、Firefox、WebKit。

标题区放在上方左侧，中文主标题为“Playwright 三层架构”；中部连接关系清晰，底部三个模块等宽排布。

所有可见文字默认简体中文，避免英文乱码；除 Playwright、Chromium、Firefox、WebKit 外，其余 UI 文案全部改为中文，不要出现 Dashboard、Documentation、Powered by 这类英文残留。不要加入无意义装饰图标，不要做成低幼卡通，不要让画面像随手画的流程图。

宽高比 16:9。
```

## 示例模板

### 技术对比图

```text
这是一张可直接用于公众号正文的高完成度专题对比图，不是普通插画。
继承本篇文章统一的科技编辑视觉，统一蓝青色系、卡片式信息模块、细线图形语言。

本图核心信息：展示“传统测试方式”和“Playwright 自动化方式”的差异。

画面左右双栏布局：左侧是“传统方式”，右侧是“Playwright 方式”。
每栏包含 3 个关键点，采用图标 + 中文短标签 + 1 行中文注释。

所有可见文字使用简体中文，仅允许 Playwright 保留英文；不要出现 Dashboard、PASS、Documentation 等英文 UI 残留。
要求层级清晰、完成度高、像技术专栏编辑图，不要做成廉价 PPT 风。
宽高比 16:9。
```

### 流程图

```text
这是一张用于公众号正文的高完成度流程图，具有编辑专题视觉感。
继承本篇文章统一视觉基调：浅底、低饱和蓝绿色、圆角模块、轻阴影、强阅读动线。

本图核心信息：展示从“需求输入”到“自动生成”再到“测试验证”的完整流程。

画面从左到右 4 个流程节点，每个节点含中文标题和一个短说明，节点之间用清晰箭头连接。
标题、标签、注释全部用简体中文。禁止无意义装饰、禁止复杂背景干扰、禁止多余英文、禁止默认英文软件界面词。
宽高比 16:9。
```

### 场景叙事图

```text
这是一张高完成度编辑场景图，用于公众号正文，不是简单生活插画。
继承本篇文章统一的温暖叙事视觉：柔和暖色、自然光感、克制留白、杂志感构图。

本图核心信息：表现“深夜独自加班的人终于找到更高效工作方式”的情绪转折。

画面主体是一张书桌、一台笔记本、半杯咖啡、便签纸和窗外夜色，人物处于专注而放松的状态。
左上角保留一组简体中文短标题，其他文字极少，仅做辅助标签。

要求画面真实、细节丰富、情绪细腻，不要幼稚卡通，不要廉价插画感。
宽高比 16:9。
```

## 常见错误

1. **只有主题，没有信息结构**：只说“画一个关于 X 的图”
2. **没有统一风格锚点**：同篇各图各画各的
3. **中文文字过多**：想把正文塞进图里
4. **中英混乱**：该中文的不中文，该保留英文的也没明确
5. **过度装饰**：画面很好看，但没表达信息
6. **太像 PPT 截图**：缺乏编辑视觉的完成度
7. **太像随手流程图**：信息有了，但视觉层级差
8. **英文 UI 残留**：模型自动补出 Dashboard / PASS / Powered by 等无关英文

## 最终检查清单

- [ ] 这张图是否明确服务于文章某一段核心信息？
- [ ] 是否继承了同篇统一视觉基调？
- [ ] 是否明确要求默认中文展示？
- [ ] 是否只保留了必要的英文术语？
- [ ] 是否强调“高完成度编辑视觉”，而不是简单画图？
- [ ] 是否给出了清晰布局、模块和阅读动线？
- [ ] 是否写明了禁止项，避免跑偏？
