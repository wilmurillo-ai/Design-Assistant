# AhaPoint Generator - 执行流程

## 质量检查清单

执行前逐项确认：

### 准备阶段
- [ ] 确认目标领域/主题
- [ ] 确认点的类型（🔴/🟢/🟡/混合）
- [ ] 确认输出模式（单点/批量/格式化）
- [ ] 确认作者名称（用于元数据）

### 调研阶段
- [ ] 至少搜索 3 个相关关键词
- [ ] 抓取至少 5 个高质量来源
- [ ] 记录具体场景和真实抱怨/洞察
- [ ] 区分事实和个人观点

### 生成阶段
- [ ] 7 部分模板全部填充
- [ ] 一句话描述清晰简洁
- [ ] 场景故事 200-500 字
- [ ] 验证方法可执行
- [ ] 元数据完整（作者/时间/分类/状态）

### 输出阶段
- [ ] 文件名格式：`YYYYMMDD-HHMM-[类型]-[标题].md`
- [ ] 保存到 `ahapoints-protocol/points/` 目录
- [ ] 生成时间戳作为优先权证明
- [ ] 向用户确认报告质量

## 标准执行流程

### Step 1: 需求确认（2 分钟）

**问用户**：
```
1. 想探索哪个领域/主题？
2. 找什么类型的点？
   - 🔴 痛点（抱怨/困扰）
   - 🟢 妙点（创新方案）
   - 🟡 乐点（好玩有趣）
   - 混合（都来一些）
3. 要几个点？
   - 1 个深度报告
   - 3-5 个批量探索
4. 作者署名用什么？
```

### Step 2: 网络调研（5-10 分钟）

**痛点搜索示例**：
```
browser.navigate("https://www.google.com/search?q=减肥 + 抱怨 OR 吐槽 OR 太难")
browser.snapshot()
→ 提取抱怨内容、场景、人群
```

**妙点搜索示例**：
```
browser.navigate("https://www.google.com/search?q=减肥 + 创新 OR AI OR 新方案")
browser.snapshot()
→ 提取技术方案、创新产品
```

**乐点搜索示例**：
```
browser.navigate("https://www.google.com/search?q=减肥 + 好玩 OR 社区 OR 游戏")
browser.snapshot()
→ 提取有趣活动、社群
```

**记录模板**：
```markdown
## 调研笔记
- 关键词：[XXX]
- 来源 1: [标题 + 核心内容]
- 来源 2: [标题 + 核心内容]
- 来源 3: [标题 + 核心内容]
- 高频问题：[列出 3-5 个]
- 有趣洞察：[列出 2-3 个]
```

### Step 3: 识别与分类（2 分钟）

**判断标准**：

| 类型 | 识别信号 | 关键词 |
|------|----------|--------|
| 🔴 痛点 | 抱怨、困扰、不满 | "烦死了"、"太难了"、"为什么没有" |
| 🟢 妙点 | 创新、巧妙、新技术 | "新方案"、"用 AI"、"自动" |
| 🟡 乐点 | 好玩、有趣、社群 | "哈哈"、"太有意思了"、"一起" |

**输出**：
```
识别出 X 个候选点：
1. [标题] - 类型🔴/🟢/🟡 - 一句话描述
2. [标题] - 类型🔴/🟢/🟡 - 一句话描述
3. [标题] - 类型🔴/🟢/🟡 - 一句话描述
```

### Step 4: 填充模板（5 分钟/点）

**逐部分填充**：

```markdown
## 1. 点类型
[✓] 痛点 (Pain Point) 🔴

## 2. 一句话描述
> [清晰简洁，20 字以内]

## 3. 场景故事
[谁] 在 [什么情况下] 遇到了 [什么问题]
- 人物：具体画像
- 场景：时间地点情境
- 问题：具体困扰
- 字数：200-500

## 4. 为什么重要
- [✓] 影响很多人 / 频繁发生 / 现有方案不够好
[2-3 句话解释]

## 5. 潜在方案方向
[App/小程序/SaaS/硬件/内容/服务/平台]
[1-2 句话描述方向]

## 6. 验证方法
- [ ] 访谈 10 个有同样问题的人
- [ ] 做个简单的 Demo 试试
- [ ] 搜一下有没有人在做
[具体步骤 1-3 条]

## 7. 元数据
| 字段 | 内容 |
|------|------|
| **作者** | [署名] |
| **发现时间** | YYYY-MM-DD |
| **发布/更新时间** | YYYY-MM-DD HH:MM |
| **领域/分类** | [分类] |
| **相关链接** | [可选] |
| **状态** | [✓] 刚发现 [ ] 验证中 [ ] 已验证 [ ] 已放弃 |
```

### Step 5: 生成文件（1 分钟）

**文件名格式**：
```
YYYYMMDD-HHMM-[类型]-[标题].md

示例：
20260305-1430-PAIN-外卖热量估算难.md
20260305-1435-INNO-拍小票 AI 估算热量.md
20260305-1440-FUN-减肥失败者联盟.md
```

**类型代码**：
- `PAIN` = 痛点 🔴
- `INNO` = 妙点 🟢
- `FUN` = 乐点 🟡

**保存路径**：
```
/Users/olivia/.openclaw/workspace/ahapoints-protocol/points/[文件名].md
```

### Step 6: 质量复核（2 分钟）

**自检问题**：
- [ ] 一句话描述是否清晰？
- [ ] 场景故事是否具体？
- [ ] 验证方法是否可执行？
- [ ] 元数据是否完整？
- [ ] 文件格式是否正确？

**用户确认**：
```
✅ 已完成 X 个 AhaPoint 报告：

1. [文件名] - 🔴/🟢/🟡 - [一句话描述]
2. [文件名] - 🔴/🟢/🟡 - [一句话描述]
3. [文件名] - 🔴/🟢/🟡 - [一句话描述]

查看路径：ahapoints-protocol/points/

需要我：
A. 深化某个点的调研
B. 继续生成更多点
C. 修改某个报告
D. 就这样，很好
```

| 模式 | 点数 | 预计时间 |
|------|------|----------|
| 单点深度 | 1 | 15-20 分钟 |
| 批量探索 | 3 | 30-40 分钟 |
| 批量探索 | 5 | 45-60 分钟 |
| 格式化输入 | 1 | 5-10 分钟 |

## 常见问题

**Q: 用户输入很零散怎么办？**
A: 使用模式 C（格式化），帮用户整理成标准模板，补充缺失部分。

**Q: 调研不到足够信息怎么办？**
A: 换关键词，或建议用户补充自己的经历和观察。

**Q: 用户想修改已生成的报告？**
A: 直接编辑文件，更新"发布/更新时间"字段。

**Q: 如何证明优先权？**
A: 文件创建时间戳即为证明，可选添加文件哈希。
