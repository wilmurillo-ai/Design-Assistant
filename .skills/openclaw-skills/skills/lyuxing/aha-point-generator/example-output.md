## 1. 点类型
[✓] 痛点 (Pain Point) 🔴

## 2. 一句话描述
> 打工人点外卖想控制热量，但根本不知道一份宫保鸡丁到底多少卡。

## 3. 场景故事
小李在上海张江工作，996 是常态，经常加班到晚上 8 点。最近体检发现脂肪肝，
医生说要控制饮食。他想减肥，但没时间做饭，只能点外卖。

每次打开外卖 App，看到精美的图片但没有任何营养信息。想知道这份宫保鸡丁

"我不是不想控制，是根本没法控制。"

- [✓] 影响很多人（中国外卖用户超 5 亿，减肥人群超 2 亿）
- [✓] 频繁发生（外卖用户平均每周点 3-5 次）
- [✓] 现有方案不够好（商家不标营养信息，手动计算太麻烦）

**简单说明**：
外卖已成为城市打工人的刚需，但营养信息缺失让想控制热量的人无从下手。
这不是"想不想"的问题，是"能不能"的问题。

## 5. 潜在方案方向
**AI 图像识别 + 小票 OCR**

用户拍外卖小票或菜品照片，AI 自动识别：
- 菜名 → 匹配热量数据库
- 分量 → 估算克数
- 烹饪方式 → 调整油量系数

输出：这份宫保鸡丁约 650 卡，误差±15%。

**技术可行性**：
- OCR 识别小票：成熟技术
- 菜品识别：有公开数据集
- 热量数据库：可爬虫 + 人工校对

- [ ] 搜一下有没有人在做（国内外竞品分析）

**具体步骤**：
1. 周末去写字楼楼下，找 10 个拿外卖的年轻人聊天（5 分钟/人）
2. 问："你想控制热量吗？现在怎么算？愿意用什么方式？"

## 示例 2: 妙点 🟢

# 拍小票 AI 估算热量

## 1. 点类型
[✓] 妙点 (Innovation Point) 🟢

## 2. 一句话描述
> 用 AI 图像识别，拍外卖小票就能自动估算热量，误差±15%。

## 3. 场景故事
现有的热量计算 App 都要手动输入食材和重量，用户体验极差。
但外卖小票上已经有菜名、分量、价格等信息。

如果用户只需拍一张小票照片，AI 自动识别：
- "宫保鸡丁" → 匹配热量数据库（鸡丁 150g+ 花生 30g+ 油 20g）
- "大份" → 调整分量系数（1.5x）
- "微辣" → 调整油量系数（0.8x）

3 秒钟输出：这份宫保鸡丁约 650 卡，蛋白质 25g，碳水 15g，脂肪 45g。

不需要用户输入任何数据，拍照即可。

## 4. 为什么重要
- [✓] 影响很多人（减肥人群 + 健身人群 + 慢病管理人群）
- [✓] 高频需求（每天都要吃，多次决策）
- [✓] 现有方案不够好（手动输入劝退 90% 用户）
- [✓] 技术成熟（OCR+ 图像识别+ 大模型）

**简单说明**：
把"手动输入 10 分钟"变成"拍照 3 秒钟"，用户体验提升 100 倍，
可能激活之前放弃热量管理的 5000 万用户。

## 5. 潜在方案方向
**小程序 + AI API**

1. 微信小程序（无需下载，扫码即用）
2. 接入现有 OCR API（百度/腾讯/阿里）
3. 自建热量数据库（爬虫 + 人工校对）
4. 大模型微调（提高菜品识别准确率）

**MVP 功能**：
- 拍小票 → 识别菜名
- 匹配热量 → 输出结果
- 历史记录 → 每日汇总

**商业模式**：
- 免费：每日 3 次识别
- 会员：无限次 + 详细营养分析（9.9 元/月）
- B 端：外卖平台 API 接入

## 6. 验证方法
- [ ] 人工模拟：找 20 张小票，人工查热量，看用户是否觉得有用
- [ ] 竞品分析：搜"热量计算 App"，看头部产品的差评（找机会）
- [ ] 技术验证：用现有 OCR API 测试小票识别准确率

**具体步骤**：
1. 收集 20 张不同餐厅的外卖小票（自己点或找朋友要）
2. 用百度 OCR API 测试识别率（免费额度够用）
3. 人工查每个菜的热量，模拟输出结果
4. 找 5 个减肥朋友试用，问"愿意用吗？愿意付费吗？"

## 使用建议

### 学习参考
1. 先读 **痛点示例**（最常用）
2. 理解 7 部分模板的逻辑
3. 注意场景故事要具体（有人物、有时间、有地点）
4. 验证方法要可执行（明天就能做）

### 避免踩坑
- ❌ 一句话描述超过 30 字
- ❌ 场景故事太抽象（"很多人"、"经常"）
- ❌ 验证方法不可执行（"需要大量调研"）
- ❌ 元数据不完整（作者/时间必填）

### 进阶技巧
- ✅ 用对话体增加真实感（"我不是不想控制，是根本没法控制"）
- ✅ 用数据支撑重要性（"5 亿用户"、"70% 失败率"）
- ✅ 方案方向给具体技术路径（"OCR+ 大模型"）
- ✅ 验证方法量化（"10 个人"、"5 分钟/人"）
