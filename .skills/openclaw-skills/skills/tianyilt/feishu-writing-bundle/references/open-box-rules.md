# Open-box rules

这份参考文件补充 feishu-writing-bundle 在真实飞书环境里最容易踩坑、也最影响“开箱即用”体验的部分。

## 1. 默认交付物就是飞书文档链接

当用户说“写个文档”“整理成文档”“帮我写飞书文档”时，默认目标不是聊天里的长回复，而是：

- 真正创建或更新飞书文档
- 最终回传飞书文档链接

如果没有链接，这项交付通常还没完成。

## 2. 默认优先落到用户可访问空间

新建飞书文档时，默认优先级：

1. 用户明确给出的 folder / wiki node / wiki space
2. 用户个人知识库：`wiki_space="my_library"`
3. 明确知道用户可访问的默认空间

不要想当然地把文档落到某个“当前账号默认空间”。在多租户、多账号、多插件账号并存时，这经常导致“文档建出来了，但用户打不开”。

## 3. 遇到“我打不开”时，优先怀疑空间/权限，不要怀疑正文

如果文档创建成功、链接也拿到了，但用户点不开，最常见原因是：

- 文档建在了插件账号自己的空间
- 用户当前登录租户与文档所在租户不同
- 用户未完成 OAuth 授权，无法以用户身份创建到自己的空间

这时候应优先修正落点和授权，而不是重写正文。

## 4. OAuth 授权后要自动续跑

当 `feishu_create_doc` 返回“已发送授权请求卡片”时，正确动作是：

1. 让用户点击授权
2. 用户授权完成后，直接重试刚才的创建操作
3. 不要让用户重新描述需求
4. 创建成功后直接给链接

## 5. 真正的完成标准是“用户能打开”

飞书文档任务的最后验收，不是 API 返回 success，而是：

- 文档已创建/更新
- 链接已返回
- 用户可以实际打开

如果用户本人打不开，这件事就还没真正完成。
