# 杠精 6 维攻击手册

按需读取本文档。当对某个维度的攻击深度不够时，查阅对应章节。

---

## 🔪 维度 1: 假设刺穿（Assumption Piercing）

### 原理
每个技术决策都建立在一组隐含假设上。大多数假设从未被显式陈述，
更不用说被验证。假设刺穿的目标是把这些隐含假设挖出来，然后
逐一检验它们是否真的可靠。

### 常见致命假设类别

**关于规模的假设**
- "当前的数据量不会快速增长"
- "并发用户数在可预测范围内"
- "单机就够了"

**关于环境的假设**
- "网络是可靠的"
- "第三方 API 不会挂"
- "时钟是同步的"
- "DNS 解析总是成功的"

**关于团队的假设**
- "团队成员不会离职"
- "新人能快速上手这个技术栈"
- "所有人都理解这个设计决策的原因"

**关于业务的假设**
- "需求不会大幅变化"
- "PM 说的就是真正需要的"
- "这个功能不需要国际化"
- "用户会按预期方式使用"

**关于技术的假设**
- "这个库会持续维护"
- "这个 API 不会 breaking change"
- "性能瓶颈在我们预测的位置"

### 攻击方法

1. **列举法**: 把方案成立的前提条件全部列出来
2. **反转法**: 对每个假设取反，看方案是否崩溃
3. **历史法**: 查找类似假设在过去失败的案例
4. **极端法**: 把假设推到极端值（0, ∞, 负数）

## 🗡️ 维度 2: 备选方案轰炸（Alternative Bombardment）

### 原理
人类决策有严重的「锚定效应」——一旦想到第一个方案，后续思考都围绕
它展开，而不是真正比较所有选项。备选方案轰炸强制打破这个锚点。

### 必检备选方案

**"什么都不做" 选项**
- 永远要问：如果我们什么都不做会怎样？
- 有些问题不值得解决，或者会自然消失
- 做了反而可能引入新问题

**"买 vs 造" 选项**
- 有现成的 SaaS/开源方案吗？
- 自建的真实成本（包括维护）vs 外购
- 定制化需求真的不可妥协吗？

**"简单暴力" 选项**
- 不用优雅架构，用最简单粗暴的方案行不行？
- 过度工程的隐性成本往往比技术债更高

**"对立面" 选项**
- 如果当前选择是微服务，认真评估单体
- 如果当前选择是 NoSQL，认真评估 SQL
- 如果当前选择是异步，认真评估同步

### 比较框架

对每个备选方案做 5 维雷达图比较：
1. **开发速度** — 从 0 到上线要多久
2. **运维成本** — 日常运营的人力和金钱
3. **可扩展性** — 10x 增长下的表现
4. **可逆性** — 换方案的成本
5. **团队适配** — 团队的现有经验和学习曲线

## ⚔️ 维度 3: 规模预言（Scale Prophecy）

### 原理
几乎所有方案在小规模下都能工作。区分好方案和坏方案的，是它在
规模增长下的衰退曲线。规模预言不是危言耸听，而是量化分析。

### 3 个 10x 维度

**数据 10x**
- 数据库查询性能（全表扫描？索引策略？）
- 存储成本（保留策略？冷热分层？）
- 数据迁移/备份时间

**流量 10x**
- 连接池/线程池上限
- 缓存命中率变化
- 下游服务承压
- 成本线性增长还是超线性

**团队 10x**
- 代码仓库的认知负载
- 部署流程的并行度
- 服务边界是否清晰到支持团队拆分

### 输出模板

```
📈 规模预言: [决策名]
   当前规模: [X]
   10x 规模: [10X]
   
   首先崩溃的环节: [具体组件/环节]
   崩溃原因: [技术机制]
   预估扛到多少倍: [N]x
   扩展方案: [如何突破这个瓶颈]
   扩展成本: [时间/人力/架构变更]
```

## 🪓 维度 4: 不可逆性评估（Irreversibility Assessment）

### 原理
Jeff Bezos 的 Type 1 / Type 2 决策框架：
- **Type 1（单向门）**: 一旦走过去就回不来了，必须深思熟虑
- **Type 2（双向门）**: 可以随时回头，应该快速决策

杠精的火力分配应该与不可逆程度成正比。

### 不可逆性评分标准（1-10）

| 分数 | 含义 | 示例 |
|------|------|------|
| 1-2 | 随时可回滚，几乎零成本 | 变量命名、内部工具选择 |
| 3-4 | 回滚需要一些工作但可控 | 内部 API 变更、数据库索引 |
| 5-6 | 回滚成本显著 | 编程语言选择、数据库选型 |
| 7-8 | 回滚极其昂贵 | 对外 API 契约、数据模型 |
| 9-10 | 几乎不可能回滚 | 分布式数据迁移、安全事件 |

### 锁定窗口分析

关键问题：**过了什么时间点就无法回头？**
- 第一个外部用户接入 API 的时刻
- 数据量超过可一次性迁移的阈值
- 团队已经深度学习该技术栈的时刻
- 基于该决策的下游决策已经做出

## 🔨 维度 5: Pre-mortem

### 原理
Gary Klein 的方法：不是问"可能出什么问题"（人会本能地保护自己的方案），
而是**告诉你"已经失败了"，请你解释为什么**。这个思维翻转能突破
确认偏误，产出比普通风险分析多约 30% 的风险条目。

### 执行步骤

1. **设定时间点**: "假设 6 个月后，这个项目/方案被判定为失败"
2. **独立列举**: 不讨论，各自写出 "它为什么失败了"
3. **汇总归类**: 合并相似死因，按影响排序
4. **制定预警**: 每个死因对应一个可观测的预警信号
5. **制定预案**: 每个死因对应一个现在就可以做的预防措施

### 常见死因模式

**技术失败**
- 选型的技术/框架被废弃或出安全漏洞
- 性能假设在真实负载下不成立
- 集成复杂度远超预期

**团队失败**
- 关键人员离开，知识丢失
- 团队对技术栈的学习曲线被低估
- 沟通成本随团队增长失控

**产品失败**
- 需求发生根本性变化
- 上线后发现用户根本不需要这个功能
- 竞品先于我们交付

**运维失败**
- 没人监控，问题到了用户投诉才发现
- 故障恢复流程没测试过
- 成本增长超出预算

## 💀 维度 6: 代码实锤（Code Proof）

### 原理
前 5 个维度是推理和判断，维度 6 是**用事实说话**。当决策涉及
已有代码时，不要说"我觉得这段代码有问题"——运行 break-my-code
工具，用崩溃和漏洞的真实证据来支撑你的论点。

### 何时使用

- 用户声称"代码已经处理了边界情况" → 用攻击证明没有
- 用户认为"输入验证足够严格" → 用 fuzzing 证明不够
- 决策建立在"现有代码是可靠的"假设上 → 先验证这个假设

### 调用方式

完整仓库版工具脚本位于 `tooling/gangjing-engine/`；
registry-safe 安装包则使用模板落地出来的临时 harness。

```bash
# Python 目标
python tooling/gangjing-engine/harness.py attack_config.json --timeout 5 -o results.json
python .gangjing-tmp/harness.py attack_config.json --timeout 5 -o results.json

# JavaScript 目标
node tooling/gangjing-engine/harness.js attack_config.json --timeout 5 -o results.json
node .gangjing-tmp/harness.js attack_config.json --timeout 5 -o results.json

# Go 目标
go test -v -run "." -timeout 30s .
```

### 如何将攻击结果转化为杠精论据

```
💀 代码实锤:
   攻击目标: [函数名]
   攻击数量: [N] 次
   击杀数量: [M] 次 (评分: [score]/100, 等级: [grade])
   
   关键发现:
   - [CRASHED] [攻击名]: [具体崩溃信息]
   - [LEAKED] [攻击名]: [泄漏的内容]
   
   这意味着: [这个发现如何推翻或削弱用户的决策]
```

### 注意事项

- 代码实锤是最强武器，但不要滥用——只在决策确实依赖代码可靠性时使用
- 攻击结果是事实，但"这个事实是否重要"仍然是判断
- 即使代码通过了所有攻击（100/A），也不意味着决策就是对的
