# POC 验收脚本模板 v1.0

> **用途**：SKILL.md Step 8.3（POC 三段式）的具体化执行脚本。将通用的"技术测试 / 能力测试 / BPO 兜底演习"三段填充为各国特有场景。
>
> **使用原则**：
> - 技术测试和能力测试结果可以用来排名，但 BPO 兜底演习的结果是 Pass/Fail 的，不合格即淘汰
> - 所有 BPO 承诺须书面化，口头回答无效，不可进入下一步谈判
> - 验收标准须提前告知候选方，不接受"补测"（候选方在正式测试前就知道验收标准，是对其能力的真实考验）

---

## 🇻🇳 越南 T-VAN POC 验收脚本

### 段落 1：技术测试

**T-VN-01：GDT 认证码申请（核心路径）**
- 场景：提交标准 VAT Invoice XML（含越南语商品名、数字签名），验证 GDT 认证码返回
- 验收标准：
  - 正常响应时间 ≤ 30 秒（GDT 官方接口超时阈值）
  - 返回认证码（mã của cơ quan thuế）格式正确，10 位数字
  - 证书签名使用认可 CA（VNPT-CA / Viettel-CA / MISA-CA / BkavCA 之一）
- 记录：响应时间分布（P50/P95/P99）

**T-VN-02：月末高峰压测**
- 场景：模拟月末最后 2 小时的发票集中提交，并发 50 张/分钟（模拟中资 KA 客户体量），持续 30 分钟
- 验收标准：
  - 成功率 ≥ 99%（允许 GDT 超时后重试成功）
  - 超时进入本地队列，自动重试（5min 间隔，最多 12 次），重试结果须返回我方
  - 无静默失败（silent failure）：每个失败都有明确错误码和日志
- 记录：提交总量、成功量、超时量、重试成功量

**T-VN-03：多租户并发隔离**
- 场景：同时模拟 3 个不同租户（3 家中资企业客户）并发提交发票
- 验收标准：
  - 租户 A 的 PKI 证书不被用于签署租户 B 的发票（证书隔离）
  - 一个租户的 GDT 超时不影响其他租户的提交
  - 每个租户的认证码响应独立返回，不混淆

---

### 段落 2：能力测试

**C-VN-01：字段校验——越南特有必填字段**
- 场景：提交以下 4 种故意错误的发票，验证前置拦截：
  - ① 商品名仅有中文，无越南语（违反第 32/2025/TT-BTC）
  - ② 缺少付款方式字段（第 70 号法令新增必填）
  - ③ 税号（MST）格式错误（非 10 位或 13 位数字）
  - ④ 开票日期早于数字签名时间戳（签名时序违规）
- 验收标准：
  - 以上 4 种错误均在提交 GDT 前被前置拦截，返回具体错误字段和原因
  - 不允许将错误报文透传给 GDT（避免 GDT 账号被标记）

**C-VN-02：税率黑名单校验**
- 场景：提交金属制品（METAL_PRODUCTS）发票，税率填写 8%（减税税率）
- 验收标准：
  - 前置校验拦截，提示"金属制品不适用减税政策，应使用 10% 标准税率"
  - 返回具体法规依据（当期有效的减税令黑名单条款）

**C-VN-03：发票替换流程**
- 场景：已成功开具并清算的发票，因买方名称错误需发起"替换（Thay thế）"
- 验收标准：
  - 替换发票成功关联原始发票号（引用原始发票 UUID）
  - 原始发票状态标记为"已替换"，替换发票获得新认证码
  - 操作全程留审计日志（谁发起 / 何时 / 原因）

**C-VN-04：降级容灾路径**
- 场景：模拟 GDT 接口完全不可达（封锁对外 IP），验证降级行为
- 验收标准：
  - 发票进入本地缓存队列，不丢失
  - 队列状态可查（待提交 / 重试中 / 最终失败）
  - GDT 恢复后，队列自动清空，发票按顺序提交
  - 降级期间不允许系统向客户报告"发票已成功"（避免虚假确认）

### 段落 3：BPO 兜底演习（Pass/Fail）

**B-VN-01（必答）：账号锁定响应**
> 问题（须书面回答）：若我方客户（中资制造企业）因税局系统误判被锁定 GDT 账号，贵司承诺在几小时内，通过哪个具体渠道（联系人姓名/部门/电话），完成解锁？若超出工作时间，响应流程是什么？

- Pass 标准：书面承诺 ≤ 8 小时响应，有具体联系人和渠道，有非工作时间处置方案
- Fail 标准：无书面承诺、响应时限 >24 小时、无具体联系人

**B-VN-02（必答）：法规变更通知**
> 问题（须书面回答）：当越南财政部或 GDT 发布影响电子发票格式或税率的新法规时，贵司承诺在法规公告后几个工作日内以书面形式通知我方？通知内容包含哪些要素？

- Pass 标准：书面承诺 ≤ 5 工作日，通知内容含变更摘要+生效日期+影响评估
- Fail 标准：拒绝书面承诺、时限 >10 工作日

**B-VN-03（验证）：本地团队存在性**
> 验证方法：要求候选方提供其越南本地运营团队的以下信息：① 公司越南工商注册证复印件；② 本地团队组织架构图（至少包含税务合规岗位）；③ 一名本地税务合规负责人的越南电话和邮箱

- Pass 标准：有越南本地注册实体，有专职税务合规岗，联系人可在越南正常工作时间接通
- Fail 标准：无本地注册、本地团队为"远程支持"、联系人实际在境外

## 🇲🇾 马来西亚 MyInvois POC 验收脚本

### 段落 1：技术测试

**T-MY-01：Token 续期压力测试**
- 场景：Token 到期时间设为 T，在 T-4 分钟时模拟新的发票提交请求（Token Pool 提前 5 分钟续期窗口的边界条件）
- 验收标准：
  - 续期在 T-5 分钟触发（提前 5 分钟），不在 T 后才触发
  - 分布式锁保证同一时间只有一个续期操作，不出现并发续期
  - Token 续期过程中，新的发票提交请求排队等待，不丢弃
  - T-4 分钟提交的请求使用新 Token 成功提交

**T-MY-02：Submit 接口限流处理（月结日场景）**
- 场景：在 100 RPM 限制下，以 150 RPM 速率提交发票（模拟月结日集中提交）
- 验收标准：
  - 本地消息队列削峰，提交速率被自动限制在 100 RPM 以内
  - 超出速率的请求进入队列，有序消费，不触发 LHDN 429 错误
  - 队列积压情况可监控（队列深度、预计清空时间）
  - 无发票丢失

**T-MY-03：cancellable_until 时间戳边界测试**
- 场景：
  - ① 在 cancellable_until 前 5 分钟发起取消请求
  - ② 在 cancellable_until 后 5 分钟发起取消请求
- 验收标准：
  - ① 取消成功，发票状态变更为 Cancelled
  - ② 取消请求被拒绝，系统提示"72小时取消窗口已过期，请走调整发票流程"
  - cancellable_until 时间戳须持久化存储（重启后不丢失），不依赖内存

**T-MY-04：多租户 Token 隔离**
- 场景：租户 A 和租户 B 同时提交发票，租户 A 的 Token 刷新操作不影响租户 B 的提交
- 验收标准：Token Pool 按租户隔离，一个租户的 Token 失效不影响其他租户

---

### 段落 2：能力测试

**C-MY-01：Classification Code 校验**
- 场景：提交缺少 `myClassificationCode` 字段的发票（LHDN 45 个分类码必填）
- 验收标准：前置拦截，提示缺少 Classification Code，并给出当前交易类型的建议分类码

**C-MY-02：XAdES 签名验证**
- 场景：提交使用错误证书链签名的发票（Schema v1.1 要求 XAdES 签名）
- 验收标准：签名验证在提交 LHDN 前完成，错误发票被拦截

**C-MY-03：Invalid 状态诊断（Get Doc Details 专项）**
- 场景：刻意提交一张带格式错误的发票，触发 LHDN 返回 Invalid 状态
- 验收标准：
  - 系统检测到 Invalid 状态后，调用 Get Doc Details（而非 Get Submission）获取详细错误原因
  - 错误原因转义成可读的中文描述，供运营处理
  - 正常的轮询查询使用 Get Submission，不使用 Get Doc Details（避免高频调用触发限流）

**C-MY-04：离线对账（Get Recent 专项）**
- 场景：模拟系统宕机 2 小时后恢复，验证用 Get Recent 接口完成离线对账
- 验收标准：
  - 使用 Get Recent 接口（而非逐条查询）批量获取宕机期间 LHDN 的发票状态
  - 对账结果与本地队列记录比对，差异项生成待处理工单
  - 对账完成后，本地状态与 LHDN 状态一致

---

### 段落 3：BPO 兜底演习（Pass/Fail）

**B-MY-01（必答）：LHDN 线下处置能力**
> 问题（须书面回答）：若客户因身份认证问题无法在 MyInvois 平台完成操作，需要线下前往 LHDN 窗口处理，贵司在吉隆坡的团队能在几个工作日内陪同或代办？

- Pass 标准：有吉隆坡本地团队，承诺 2 个工作日内可陪同，有本地联系人
- Fail 标准：无马来西亚本地团队、无法提供线下协助

**B-MY-02（必答）：法规更新通知**
> 同越南 B-VN-02，内容替换为 LHDN / MDEC 的法规变更

**B-MY-03（必答）：LHDN 关系网络**
> 问题（须书面回答）：请列举贵司在过去 12 个月内为客户解决的 LHDN 合规异常案例（可匿名），包括异常类型、处置渠道、解决时长

- Pass 标准：有 2 个以上可描述的真实案例，有具体的处置渠道描述
- Fail 标准：无案例、案例均为境外远程处理

## 🇸🇬 新加坡 InvoiceNow（Peppol）POC 验收脚本

> 新加坡是 Post-Audit 模式，无实时 CTC，技术难度低于越南和马来西亚。POC 重点在于 Peppol 互通性和 BPO 的异常处理能力。

### 段落 1：技术测试

**T-SG-01：Peppol 发票端到端传输**
- 场景：从我方测试账号，向一个新加坡本地 Peppol ID 发送 UBL 2.1 格式发票
- 验收标准：
  - 接收方在 Peppol 网络内，发票 10 分钟内到达
  - 发票格式符合 BIS Billing 3.0 规范，无 schema 错误
  - 传输状态回调（ACK）正确返回

**T-SG-02：跨 Peppol 网络互通**
- 场景：向一个欧盟国家的 Peppol ID 发送测试发票（验证新加坡→欧盟互通）
- 验收标准：发票成功传输，接收方确认（对有欧洲业务的 MNC 客户是重要卖点）

**T-SG-03：并发稳定性**
- 场景：模拟单日 1000 张发票批量提交
- 验收标准：成功率 100%（Peppol 无限流，失败需排查格式或网络问题）

---

### 段落 2：能力测试

**C-SG-01：UBL 格式校验**
- 场景：提交缺少 `AccountingSupplierParty/Party/EndpointID` 字段的发票（Peppol ID 必填）
- 验收标准：前置拦截，提示缺少 Peppol ID，并说明获取方式

**C-SG-02：GST 税率处理**
- 场景：发票含 9% GST 标准税率和 0% 零税率（出口）的混合行项目
- 验收标准：每个行项目的税率独立计算，总税额正确，符合 IRAS 申报要求

**C-SG-03：接收方 Peppol ID 不存在**
- 场景：向一个不存在的 Peppol ID 发送发票
- 验收标准：Peppol 网络返回路由错误，系统提示"接收方 Peppol ID 不存在，请确认对方已注册 InvoiceNow"，不重试

---

### 段落 3：BPO 兜底演习（Pass/Fail）

> 新加坡 BPO 复杂度低于越南和马来西亚，重点验证数据映射和异常报文处理能力。

**B-SG-01（必答）：数据映射异常处理**
> 问题（须书面回答）：若客户 ERP 传入的发票数据字段与 Peppol UBL 2.1 字段存在映射歧义（如货币代码格式不一致），贵司的处理流程是什么？响应时限是多少？

- Pass 标准：有标准映射规则文档，异常映射 4 小时内人工确认，结果书面反馈
- Fail 标准：无映射规则文档、仅依赖客户自行调整

**B-SG-02（必答）：法规更新通知（IRAS/IMDA）**
> 同其他市场法规通知要求，内容替换为 IRAS/IMDA/Peppol 规范更新

**B-SG-03（验证）：Peppol 互通性证明**
> 要求候选方提供其在 IMDA 官方 AP 名录中的注册记录（URL + 截图），以及最近一个季度的发票传输量数据

- Pass 标准：在 IMDA 官方名录中可查，传输量数据真实可信
- Fail 标准：未在官方名录中，或传输量数据不可查证

## ⚠️ 跨市场通用 BPO 原则

1. **BPO 演习是 Pass/Fail，不是加分项**：任何一题 Fail 直接淘汰，不用加权计算
2. **书面化是硬性要求**：POC 中收到的口头 SLA 承诺无效，必须要求候选方在 POC 结束后 5 个工作日内以书面形式（邮件或文件签字）确认
3. **本地实体不可替代**：远程支持不等于 BPO 兜底，境外团队承诺"随时在线"不构成本地 BPO 能力
4. **`multiTenantProxyLegal = CONFIRMED` 是上生产的 P0 前置条件**：POC 通过后，上线前须有法务书面确认代理提交合法性，POC 结果不等于合法性确认
