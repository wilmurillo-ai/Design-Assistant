## Art Validation | 艺术形式验证标准

### 自我检测机制

本Skill包含内置的艺术形式验证器，用于判断生成的前端作品是否符合"艺术"标准。

#### 验证维度

```yaml
验证维度:
  1_原创性_originality:
    - 是否有独特的创意概念
    - 是否避免了常见UI模式的复制
    - 是否体现了创作者的个人视角

  2_意图表达_intentionality:
    - 是否有明确的艺术概念/主题
    - 技术选择是否为概念服务
    - 是否能陈述创作意图

  3_感官体验_sensory_experience:
    - 是否提供独特的视觉体验
    - 是否有动态/交互带来的惊喜
    - 是否调动了某种情绪反应

  4_形式完整_formal_completeness:
    - 是否作为完整作品存在
    - 各元素是否协调统一
    - 是否经得起凝视与探索

  5_技术诗意_technical_poetry:
    - 是否利用了浏览器媒介的独特性
    - 是否有"只有浏览器能做到"的元素
    - 技术本身是否成为表达媒介
```

#### 验证提示词

当用户请求验证时，使用以下标准：

```
艺术形式验证评估：

请从以下维度评估这个前端项目（满分100分）：

1. 原创性 (25分)
   - 概念是否独特？是否有个人视角？
   - 是否避免了模板化/工具化的复制？

2. 意图明确性 (25分)
   - 是否有清晰的艺术概念/主题？
   - 能否用一句话描述这件作品想表达什么？
   - 技术选择是否服务于概念？

3. 感官体验 (25分)
   - 视觉是否令人驻足？
   - 交互是否有意外之喜？
   - 是否唤起了某种情绪？

4. 形式完整性 (25分)
   - 作品是否自洽完整？
   - 各元素是否协调统一？
   - 是否值得反复观看？

总分 ≥ 80分：优秀的浏览器原生艺术作品
总分 60-79分：具有良好的艺术潜质，建议进一步深化
总分 < 60分：可能更偏向功能性/工具性设计
```

#### 技术诗意评估标准

除了上述四个维度外，浏览器原生艺术还应评估：

```
技术诗意 (Technical Poetry):
  - 是否利用了浏览器媒介的独特性？
    * CSS动画 vs Canvas vs WebGL的选择是否恰当
    * 是否发挥了浏览器渲染引擎的特性

  - 是否有"只有浏览器能做到"的元素？
    * 交互性：实时响应用户输入
    * 链接性：超链接形成的叙事网络
    * 响应式：不同设备的不同体验

  - 技术本身是否成为表达媒介？
    * 代码结构是否呼应艺术概念
    * 算法是否本身就是艺术创作
    * 性能限制是否转化为美学选择
```

#### 常见艺术形式判断

| 特征 | 更接近艺术 | 更接近设计 |
|------|----------|-----------|
| 目的 | 表达/探索 | 功能/可用性 |
| 评判标准 | 情感/概念深度 | 效率/转化率 |
| 完成度 | 开放性，可迭代 | 明确的目标 |
| 可复制性 | 不可复制是价值 | 可复制是优势 |
| 技术选择 | 服务于概念 | 最优解决方案 |

#### 验证流程

1. **概念层** - 询问创作者想要表达什么
2. **技术层** - 评估技术选择是否服务概念
3. **体验层** - 测试观看/交互体验
4. **反思层** - 作品是否引发了预期反应

```
自我验证问题清单：

1. 我能用一句话描述这个作品想表达什么吗？
2. 如果把这个作品的核心概念换成另一种媒介，它仍然成立吗？
3. 有什么元素是因为"技术上能做到"而加入的，而不是因为概念需要？
4. 这个作品有没有让我自己都惊讶的元素？
5. 如果观众只观看5秒钟，他们会错过什么重要的东西？
```
