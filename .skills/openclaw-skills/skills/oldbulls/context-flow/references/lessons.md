# Lessons

## Purpose

This file captures durable lessons from recovery and reflection experiences. Use it when resuming interrupted work, turning coordination mistakes into stable rules, or deciding whether a pattern should be promoted into templates or anti-patterns.

## Recovery lessons

- A raw transcript is not a reliable recovery point
- Historical memory is context, not current authorization
- `Continue where you left off` is not enough when the active task is unclear
- Resume should start from a short checkpoint: current goal, completed work, active constraints, and next step
- If the recovered thread risks reviving stale work, ask before proceeding

### Recovery checklist

- What is the current confirmed task?
- What was already completed?
- Which constraints are still valid?
- Which items cannot be resumed without reconfirmation?
- Is the next step explicit, or does it need confirmation?

### Preferred recovery pattern

- Checkpoint before interruption when possible
- Resume from checkpoint plus stable docs
- Treat injected historical notes as clues only
- Re-anchor the main line before executing more work

## Reflection lessons

- Repeated coordination mistakes should be promoted into stable rules
- Long process notes should not remain the main entry point once a shorter rule exists
- A useful reflection result is compact, actionable, and easy to reuse
- If a collaboration pattern keeps recurring, move it into templates or anti-patterns
- Governance docs should stay thinner than the process history that produced them

### Preferred distillation pattern

1. Keep the raw process record for audit
2. Extract the stable lesson
3. Promote the lesson into rules, templates, or anti-patterns
4. Stop using the long process note as the main reference

### Practical rule

When a lesson is stable enough to guide future behavior, store it in the skill references or local operating rules rather than relying on historical memory alone.
