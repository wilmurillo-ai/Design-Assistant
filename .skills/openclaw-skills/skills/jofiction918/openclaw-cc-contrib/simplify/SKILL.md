# simplify

> 三阶段代码审查：复用 → 质量 → 效率，自动发现可以改进的地方

**功能描述**：
对现有代码执行三阶段并行审查，识别出可以简化、优化和改进的地方，并给出具体修改提案。

**触发**：
- 手动触发：`/simplify` 或 `简化代码`
- 自动触发：可选，代码提交后可自动运行

## 工作原理

### 三阶段审查流程

#### 第一阶段：识别变更
- 运行 `git diff` 查看当前变更了哪些文件
- 如果没有 git 变更，就审查用户指定或最近修改过的文件

#### 第二阶段：分阶段审查

##### 审查一：代码复用检查 (Reuse)
- 寻找项目中已有的工具函数，看新写代码是否可以替换
- 标记重复实现已有功能的代码
- 标记可以用标准库/开源库替代的自定义实现

##### 审查二：代码质量检查 (Quality)
- 检查冗余状态、参数蔓延、复制粘贴代码等代码异味
- 检查泄漏抽象、字符串类型代码、不必要嵌套等问题
- 清理不必要注释，只保留解释"为什么"的注释

##### 审查三：效率检查 (Efficiency)
- 检查不必要计算、错失并发、热路径膨胀
- 检查重复无操作更新、TOCTOU反模式、内存泄漏
- 检查过于宽泛的操作

## 输出格式

```
## 🧹 Simplify 审查报告

### 阶段一：复用机会 🎣
- [ ] 发现 X 处重复代码，可以抽成公共函数
- [ ] Y 功能已有标准库 Z 可用，可以替换

### 阶段二：质量改进 ✨
- [ ] 函数 X 太长 (> 100 行)，建议拆分
- [ ] 变量命名 `a` 不够清晰，建议改为 `actualUser`
- [ ] 缺少错误处理，当 X 为 null 时会崩溃

### 阶段三：效率优化 ⚡
- [ ] 双重循环可以优化为哈希表查找，复杂度从 O(n²) → O(n)
- [ ] X 操作重复执行，可以缓存结果

---

### 📋 具体修改提案（按优先级排序）
1. **高优先级**：[具体修改方案]
2. **中优先级**：[具体修改方案]
3. **低优先级**：[具体修改方案]

---
*报告生成时间：{{date}}*
```

## 设计原则

1. **纯 Prompt 实现**：不依赖 CC 专属 Host 机制，OpenClaw 原生可运行
2. **并行思考**：三个阶段同时检查，不遗漏任何改进点
3. **提案制**：只给出改进建议，不直接修改代码，由用户确认后执行
4. **优先级排序**：区分高/中/低优先级，用户可以选择改哪些
