# 数字化解决方案设计 Skill 详细说明

本 Skill 用于系统化设计数字化解决方案，支持政府和企业数字化转型场景下的多种方案类型。

## 支持的方案类型

### 1. 规划类方案
- **适用场景**：初次接触客户，粗颗粒度规划，以打动客户为目的。
- **输出重点**：政策背景、现状问题、建设目标及思路、业务架构（简化）、关键建设内容、预期效益。

### 2. 申报类方案
- **适用场景**：客户内部汇报、立项申请、预算审批。
- **输出重点**：建设背景、需求分析、建设目标、四维度架构、主要建设内容、实施计划（简化）、效益分析、费用估算。

### 3. 可研类方案
- **适用场景**：项目立项审批、财政预算申请、专家评审。
- **输出重点**：总论、背景与必要性、需求分析（细化至业务、用户、功能、数据、性能、安全、运维）、总体建设方案、建设内容、详细技术方案与选型、实施计划、投资估算、效益分析、风险分析。

### 4. 投标类方案
- **适用场景**：公开招投标、竞争性谈判。
- **输出重点**：需求理解、总体建设方案、详细功能设计、实施与管理、安全与合规、运维服务、培训方案、报价文件。

### 5. 工作汇报类方案
- **适用场景**：项目执行过程汇报、里程碑评审、需求变更汇报。
- **输出结构**：工作背景 → 需要解决的问题 → 当前工作及完成情况 → 成果及成效 → 存在问题 → 下一步计划 → 需要领导支持。

## 各阶段详细说明

### 第一阶段：政策背景分析
1. 梳理政策环境（国家/行业/地方/国际政策）
2. 分析政策影响（指导意义、机遇、约束、趋势）
3. 识别合规要求（法律法规、行业标准、数据安全、技术标准）
4. 输出政策分析报告（政策环境、关键解读、合规清单、机遇挑战）

检查点：✅ 政策环境梳理全面、✅ 合规要求识别清晰、✅ 政策影响分析到位

### 第二阶段：需求分析
1. 行业现状分析（趋势、技术、竞争、痛点）
2. 客户现状问题分析（系统流程、痛点、瓶颈、差距）
3. 收集核心需求（业务目标、用户场景、功能范围、非功能需求、约束条件）
4. 需求优先级排序（MoSCoW 方法、MVP 范围、基础/增强/创新需求）
5. 输出需求文档（行业/客户现状分析、需求概述、功能清单、非功能需求、MVP 定义）

检查点：✅ 行业现状分析深入、✅ 客户问题诊断准确、✅ 业务价值明确、✅ 功能边界清晰、✅ MVP 范围可界定

### 第三阶段：建设思路设计
1. 确定总体建设目标（战略愿景、量化目标、价值主张、成功标准）
2. 制定建设原则（业务价值导向、用户体验中心、技术业务融合、渐进演进）
3. 设计建设路径（自建/采购/合作、传统/云原生/信创、实施模式、转型策略）
4. 规划分期建设（分期原则、各期目标范围、交付物、依赖关系）
5. 输出建设思路文档（目标愿景、建设原则、路径策略、分期规划、价值主张）

检查点：✅ 建设目标清晰可量化、✅ 建设原则指导性强、✅ 建设路径合理可行、✅ 分期规划逻辑清晰

### 第四阶段：架构设计（四维度）

#### 4.1 业务架构
- 梳理业务流程（核心/支撑/管理/跨部门协同）
- 识别业务能力（核心/支撑/管理/集成能力）
- 设计业务关系（实体/协作/流转/服务关系）
- 生成业务架构图（DOT 格式，调用脚本）
检查点：✅ 业务流程覆盖全面、✅ 业务能力边界清晰、✅ 业务关系逻辑合理、✅ 架构图清晰准确

#### 4.2 功能架构
- 划分功能模块（按业务领域/用户角色/系统层次/业务能力）
- 设计功能层次（核心/支撑/增强/集成功能层）
- 设计功能关系（依赖/调用/协作/复用关系）
- 生成功能架构图
检查点：✅ 功能模块划分合理、✅ 功能层次清晰、✅ 功能关系明确、✅ 架构图清晰准确

#### 4.3 数据架构
- 设计数据模型（实体/属性/关系/类型约束）
- 设计数据流（业务/系统/跨系统数据流，采集/存储/处理/应用）
- 设计数据标准（字典/编码/质量管理标准）
- 设计数据治理（分类分级/安全隐私/生命周期/共享开放）
- 生成数据架构图
检查点：✅ 数据模型完整、✅ 数据流清晰、✅ 数据标准统一、✅ 数据治理方案可行、✅ 架构图清晰准确

#### 4.4 技术架构
- 确定技术架构模式（参考 `references/architecture-patterns.md`）
- 设计部署架构（拓扑/分层/容器化/高可用容灾）
- 设计安全架构（网络/应用/数据/运维安全）
- 设计集成架构（API网关/服务注册/消息队列/数据交换）
- 生成技术架构图
检查点：✅ 架构模式选择合理、✅ 部署架构可行、✅ 安全架构完善、✅ 集成架构灵活、✅ 架构图清晰准确

### 第五阶段：技术选型
1. 确定技术选型维度（前端、后端、数据存储、中间件、基础设施）
2. 技术栈评估：参考 `references/technology-stack-guide.md`，评估成熟度、社区生态、团队能力、成本，进行关键技术权衡
3. 输出技术选型文档（技术栈清单、选型依据与权衡、潜在风险与备选方案）
检查点：✅ 技术选型有明确依据、✅ 考虑了团队能力匹配、✅ 关键技术有备选方案

### 第六阶段：具体建设内容
本阶段将架构设计转化为具体的建设任务和交付物。

#### 6.1 业务架构展开
- 业务流程实施设计（关键流程设计、流程优化、跨部门协同）
- 业务能力建设计划（实施路径、时间表、资源需求）
- 业务关系实现设计（协作机制、服务契约）
检查点：✅ 业务流程设计完整、✅ 能力建设计划可执行、✅ 协作机制明确

#### 6.2 功能架构展开
- 功能模块开发计划（功能点清单、开发计划、验收标准）
- 功能层次实施策略（核心/支撑/增强/集成功能的实施顺序）
- 功能关系实现设计（接口设计、数据流设计）
检查点：✅ 功能模块覆盖全面、✅ 开发计划可执行、✅ 接口设计完整

#### 6.3 数据架构展开
- 数据模型实现设计（表结构、数据字典、数据初始化）
- 数据流实现设计（ETL 方案、同步策略、缓存策略）
- 数据标准实施（数据规范、质量管控）
- 数据治理实施（数据安全、权限控制、生命周期管理）
检查点：✅ 数据模型设计完整、✅ 数据流设计可行、✅ 数据安全措施到位

#### 6.4 技术架构展开
- 部署架构实施设计（环境准备、部署方案、监控方案）
- 安全架构实施设计（安全策略、安全配置、安全测试）
- 集成架构实施设计（API 设计、中间件配置、系统集成）
检查点：✅ 部署方案可行、✅ 安全措施完善、✅ 集成方案完整

### 第七阶段：实施规划
1. 规划实施阶段：参考 `references/implementation-phases.md`，阶段划分 MVP → 功能扩展 → 优化完善
2. 任务分解与排期（WBS、任务依赖、资源需求）
3. 关键里程碑定义（MVP 发布、功能完整版、生产就绪版）
4. 输出实施计划（阶段规划、里程碑时间表、资源需求）
检查点：✅ MVP 可快速交付、✅ 阶段划分合理、✅ 里程碑可度量

### 第八阶段：风险评估
1. 识别关键风险（技术、业务、项目、运维风险）
2. 风险评估（发生概率、影响程度、风险等级）
3. 制定缓解策略（预防措施、应急预案、责任人）
4. 输出风险清单（风险分类与等级、缓解措施、监控指标）
检查点：✅ 关键风险已识别、✅ 高风险有缓解措施、✅ 风险可跟踪

### 第九阶段：效益分析与投资估算

#### 9.1 效益分析
- 经济效益：成本节约、收入增长、投资回报率（ROI）计算
- 社会效益（政府场景重点）：服务效率提升、群众满意度提升、政府治理能力提升
- 管理效益（企业场景重点）：管理效率提升、决策支持能力提升、业务协同能力提升

#### 9.2 投资估算（申报类、可研类、投标类方案必需）
1. 确定投资估算依据（参考市场价格、行业标准、类似项目）
2. 总投资估算（分项说明）：
   - 硬件设备费（服务器、存储、网络设备等）
   - 软件购置费/开发费（基础软件、定制开发）
   - 实施服务费（咨询、实施、集成）
   - 数据资源费（数据采集、清洗、迁移）
   - 安全测评费（等保测评、安全评估）
   - 培训费（培训讲师、培训材料）
   - 运维费（年度运维、技术支持）
3. 资金筹措方案（资金来源、分期资金安排、资金使用计划）
检查点：✅ 投资估算依据充分、✅ 分项明细完整、✅ 资金筹措方案可行

### 第十阶段：特殊章节（根据方案类型增加）

#### 10.1 详细功能设计方案（投标类方案必需）
- 管理后台功能设计（用户管理、角色权限、系统配置、业务功能模块详细设计）
- 运维平台功能设计（系统监控、日志管理、告警管理、性能监控、故障诊断）
- 数据服务与分析功能设计（数据查询、统计分析、报表展示、数据挖掘、智能分析）
- 功能清单输出（按照招标需求逐项响应）

#### 10.2 项目实施与管理方案（投标类方案必需）
- 项目组织架构设计（项目经理、技术负责人、开发团队、测试团队）
- 关键人员简历（资质、经验、项目案例）
- 实施方法论（敏捷开发、瀑布模型、混合模式）
- 项目进度计划（甘特图、关键路径）
- 质量管理方案（质量标准、测试方案、质量评审）
- 风险管控（风险识别、风险评估、应对措施）
- 变更管理（变更流程、变更评审、变更记录）

#### 10.3 信息安全与合规方案（投标类、可研类方案必需）
- 网络安全方案（防火墙、入侵检测、网络隔离）
- 数据安全方案（数据加密、数据脱敏、数据备份）
- 应用安全方案（身份认证、访问控制、安全审计）
- 等保测评与合规（政府场景：等保三级测评、合规评估）
- 安全管理制度与应急响应

#### 10.4 运维服务与保障方案（投标类方案必需）
- 运维服务模式（远程运维、驻场运维、混合模式）
- 运维内容（系统巡检、故障处理、性能优化）
- 服务级别协议（SLA）（响应时间、解决时间、可用性）
- 运维团队与工具（运维人员、运维平台、监控系统）
- 售后保障机制（服务热线、升级流程、投诉渠道）

#### 10.5 培训方案（投标类方案必需）
- 培训计划（培训对象、培训时间、培训周期）
- 培训内容与教材（业务操作培训、技术培训、管理培训）
- 培训方式（集中培训、在线培训、现场指导）
- 培训考核与效果保障（考核方式、效果评估、持续支持）

## 可选分支详解

- **快速原型场景**：政策背景分析 → 需求分析 → 快速架构（四维度简化）→ 技术选型 → 具体建设内容（简化）→ 原型实现
- **企业级系统**：强调安全、合规、高可用性设计
- **创新项目**：增加可行性验证阶段，采用实验性技术
- **遗留系统迁移**：增加现状评估、迁移策略设计
- **政府数字化转型场景**：增加合规评估、安全设计、政策适配、服务便民化设计，参考 `references/government-digitalization.md`

## 使用示例

### 示例 0：方案类型识别（强制流程）
- **功能**：用户输入需求后，强制询问方案类型，待用户明确后再进行方案编写。
- **执行方式**：智能体主导，第零阶段必执行。
- **关键指导**：
  - **强制要求**：用户输入需求后，**必须先询问**用户方案类型，不要通过关键词推导或猜测。
  - 询问语："请问您需要产出哪种类型的方案？可选择：规划类、申报类、可研类、投标类、工作汇报类"
  - 若用户不清楚，参考本 README 中的方案类型说明。
  - **等待用户明确反馈方案类型后**，再根据用户选择的方案类型进行解决方案内容的编写。
  - 根据确认的方案类型选择对应的执行阶段和输出结构。

### 示例 1：规划类方案 - 政务服务平台规划
- **功能**：为某政府部门设计粗颗粒度的政务服务平台规划方案。
- **执行方式**：智能体主导，执行规划类方案流程。
- **关键指导**：
  - 方案类型：规划类方案
  - 执行阶段：第零阶段 → 第一阶段（政策背景分析）→ 第二阶段（需求分析，简化）→ 第三阶段（建设思路设计）→ 4.1（业务架构，简化）→ 第六阶段（具体建设内容，简化）→ 效益分析
  - 输出重点：政策背景分析、现状问题、建设目标及思路、业务架构（简化）、关键建设内容、预期效益
  - 内容特点：粗颗粒度、强调愿景和方向，不涉及详细设计和费用

### 示例 2：申报类方案 - 企业数字化管理系统申报
- **功能**：为企业设计数字化管理系统申报方案，用于内部立项申请。
- **执行方式**：智能体主导，执行申报类方案流程。
- **关键指导**：
  - 方案类型：申报类方案
  - 执行阶段：第零阶段 → 第一阶段 → 第二阶段 → 第三阶段 → 第四阶段（四维度架构）→ 第六阶段（主要建设内容）→ 第七阶段（实施规划，简化）→ 效益分析 → 费用估算
  - 输出重点：建设背景、需求分析、建设目标、四维度架构、主要建设内容、实施计划（简化）、效益分析、费用估算
  - 内容特点：较规划类更细化，包含费用估算，架构设计涵盖四维度

### 示例 3：可研类方案 - 政府数字政府建设可研
- **功能**：为政府数字政府建设项目设计可行性研究报告。
- **执行方式**：智能体主导，执行可研类方案流程（全部阶段）。
- **关键指导**：
  - 方案类型：可研类方案
  - 执行阶段：全部阶段（第零阶段至第八阶段）+ 投资估算（第九阶段）
  - 输出重点：总论、背景与必要性、需求分析（细化）、总体建设方案、建设内容、技术方案与选型（详细）、实施计划、投资估算、效益分析、风险分析
  - 需求分析需细化：业务、用户、功能、数据、性能、安全、运维
  - 技术选型需详细：技术路线、关键技术、软硬件选型、集成方案、信创适配（政府场景）
  - 内容特点：内容最全面、最细化，包含详细的投资估算和风险分析

### 示例 4：投标类方案 - 政务云平台投标
- **功能**：为政务云平台招标项目设计投标方案。
- **执行方式**：智能体主导，执行投标类方案流程。
- **关键指导**：
  - 方案类型：投标类方案
  - 执行阶段：第零阶段 → 第二阶段（项目需求理解与分析）→ 第三阶段（总体建设方案）→ 第四阶段（四维度架构）→ 10.1（详细功能设计）→ 10.2（项目实施与管理方案）→ 10.3（信息安全与合规方案）→ 10.4（运维服务与保障方案）→ 10.5（培训方案）→ 报价文件
  - 输出重点：需求理解、总体建设方案、详细功能设计、实施与管理、安全与合规、运维服务、培训方案、报价文件
  - 详细功能设计需按照招标需求逐项响应，并附上功能清单
  - 内容特点：围绕招标需求响应，强调技术实力、实施能力、管理能力和价格竞争力

### 示例 5：工作汇报类方案 - 项目阶段性汇报
- **功能**：为项目执行过程设计阶段性工作汇报方案。
- **执行方式**：智能体主导，执行工作汇报类方案特殊流程。
- **关键指导**：

  - 输出结构：工作背景 → 需要解决的问题 → 当前正在开展的工作内容及完成情况 → 已经产生的工作成果及成效 → 当前工作开展中存在的问题 → 下一步工作计划 → 需要领导给予的支持

## 注意事项
- **方案类型识别强制要求**：用户输入需求后，**必须先询问**用户方案类型，不要通过关键词推导或猜测，等待用户明确反馈后再进行方案编写。
- **阶段迭代**：不需要线性完成所有阶段，根据方案类型选择需要的阶段执行。
- **产出导向**：每个阶段都应产出明确文档或决策，避免过度设计。
- **业务优先**：技术方案必须服务于业务目标，避免技术驱动。
- **平衡原则**：在理想方案与实际约束之间找到平衡点。
- **持续验证**：关键设计决策应通过原型、POC 或评审验证。
- **政策合规**：政府场景需严格遵守政策法规要求，确保合规安全。
- **四维度协同**：业务架构、功能架构、数据架构、技术架构需协同一致，相互支撑。
- **现状驱动**：需求分析需深入行业现状和客户现状，必要时主动向用户获取现状信息或从互联网搜索同类客户共性现状。
- **思路清晰**：建设思路设计需明确目标、原则、路径、分期，指导后续实施。
- **架构图生成**：架构设计阶段应生成可视化架构图，使用 `scripts/generate-architecture-diagram.py` 脚本。
  - 前置要求：安装 Python 依赖 `graphviz`（`pip install graphviz`）
  - 输入格式：Graphviz DOT 格式（参考 `references/architecture-dimensions.md` 中的模板）
  - 输出格式：PNG 或 SVG 格式的架构图
  - 支持类型：业务架构图、功能架构图、数据架构图、技术架构图、流程图
- **方案类型适配**：不同方案类型对应不同的输出结构和深度要求，参考 `references/solution-type-frames.md` 严格执行框架结构。
