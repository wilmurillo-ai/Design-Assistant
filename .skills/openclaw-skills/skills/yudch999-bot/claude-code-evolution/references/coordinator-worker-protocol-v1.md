# Coordinator+Worker协作协议 v1.0

## 角色分配

### Coordinator（协调者）
- **Agent**: main（毛豆）
- **职责**:
  1. 接收用户需求，进行需求分析
  2. 将复杂任务分解为可执行的子任务
  3. 生成精确到文件路径和行号的指令
  4. 分配任务给合适的Worker
  5. 监控任务进度，处理阻塞和错误
  6. 汇总Worker结果，生成最终输出
  7. 质量验证和用户反馈收集

### Workers（工作者）
- **CEO**: ceo Agent
- **产品部**: product Agent  
- **开发部**: development Agent
- **设计部**: design Agent
- **市场部**: marketing Agent
- **数据分析部**: data Agent
- **运维部**: operations Agent
- **研究部**: hermes Agent

## 精确指令协议

### 指令格式
```
任务：[明确的任务描述]

具体要求：
1. [具体需求1]
2. [具体需求2]

文件位置：
- 主文件：/path/to/main/file.ext (第10-50行)
- 配置文件：/path/to/config/file.yaml (第5-20行)
- 测试文件：/path/to/test/file.test.ext (第1-30行)

预期输出：
- 文件修改：描述预期的文件变化
- 功能验证：如何验证任务完成
- 错误处理：可能的错误和应对方案

时间限制：[可选] 预计完成时间
```

### 禁止的指令模式
- ❌ "分析这个问题"（太模糊）
- ❌ "处理用户需求"（未消化）
- ❌ "看一下代码"（无具体目标）
- ✅ "分析`/src/utils/validation.ts`第25-40行的`validateUserInput`函数，找出输入验证的漏洞，并提供修复方案"

### 指令生成原则
1. **具体性**：每个指令必须包含具体的文件路径和行号
2. **可验证性**：预期输出必须包含验证方法
3. **完整性**：覆盖所有必要的上下文和约束
4. **安全性**：考虑权限限制和错误处理

## 协作工作流程

### 标准流程
1. **任务接收**：Coordinator接收用户需求
2. **需求分析**：Coordinator消化需求，进行初步研究
3. **任务分解**：将需求拆解为具体可执行的子任务
4. **Worker分配**：根据任务类型选择合适的Worker
5. **精确指令**：向Worker发送精确到文件路径的指令
6. **执行监控**：跟踪Worker进度，处理阻塞
7. **结果汇总**：收集所有Worker结果，整合为最终输出
8. **质量验证**：验证任务完成质量
9. **反馈收集**：记录协作过程中的问题和改进点

### 错误处理流程
1. **Worker失败**：Worker报告无法完成任务
2. **任务重分配**：Coordinator重新分配任务或调整指令
3. **降级方案**：寻找替代实现方案
4. **人工干预**：必要时请求用户指导

## 任务分类与并行规则

### 任务分类
- **Class R（只读）**：读取、搜索、分析（可并行）
- **Class W（写操作）**：创建、修改、删除文件
- **Class V（验证）**：测试、验证、审查

### 并行规则
1. **R任务**：任意数量并行执行
2. **W任务**：同一文件区域串行，不同文件可并行
3. **V任务**：可与不同区域的W任务并行
4. **混合任务**：按最严格类别处理

### 冲突检测
- 文件级锁定：同一文件的W任务需要串行
- 依赖跟踪：识别任务间的文件依赖关系
- 死锁预防：超时机制和任务优先级

## 通信协议

### Coordinator → Worker 指令
```
{
  "task_id": "task_001",
  "instruction": "精确指令文本",
  "files": ["/path/to/file1", "/path/to/file2"],
  "expected_output": "预期输出描述",
  "timeout_seconds": 3600,
  "priority": "high|medium|low"
}
```

### Worker → Coordinator 响应
```
{
  "task_id": "task_001",
  "status": "completed|failed|in_progress",
  "result": "执行结果或错误信息",
  "output_files": ["/path/to/output1"],
  "execution_time_seconds": 120,
  "errors": []
}
```

## 实施要求

### 对Coordinator的要求
1. **禁止"懒委托"**：必须消化研究结果，不能直接转发用户需求
2. **精确指令**：必须提供可执行的精确指令
3. **进度监控**：定期检查Worker状态
4. **结果验证**：验证Worker输出符合预期

### 对Worker的要求
1. **指令执行**：严格按照指令执行
2. **状态报告**：定期报告进度和问题
3. **结果交付**：按格式返回执行结果
4. **错误处理**：及时报告无法解决的问题

## 示例场景

### 场景1：代码审查任务
**用户需求**："检查项目中的安全漏洞"

**Coordinator分解后指令**：
```
任务：检查`/src/auth/authentication.ts`中的身份验证逻辑安全漏洞

具体要求：
1. 检查第45-80行的`validatePassword`函数，寻找可能的注入攻击漏洞
2. 检查第120-150行的`createSession`函数，寻找会话管理安全问题
3. 检查第200-230行的`rateLimit`函数，寻找拒绝服务攻击漏洞

文件位置：
- 主文件：/src/auth/authentication.ts (第45-230行)
- 测试文件：/tests/auth/authentication.test.ts (第1-100行)

预期输出：
- 漏洞报告：列出发现的安全问题，每个问题包含位置、描述、风险等级
- 修复建议：针对每个漏洞提供具体的修复代码
- 验证方法：运行现有测试确保修复不破坏现有功能

时间限制：预计2小时内完成
```

### 场景2：研究分析任务
**用户需求**："分析最新的AI代码生成工具"

**Coordinator分解后指令**：
```
任务：研究GitHub上最热门的3个AI代码生成工具

具体要求：
1. 搜索GitHub trending中与AI代码生成相关的项目
2. 选择stars数最高的3个项目进行详细分析
3. 比较它们的架构设计、技术栈、功能特点
4. 分析各自的优势和局限性

文件位置：
- 研究笔记：/research/ai-code-generation.md (新建文件)
- 比较表格：/research/comparison-table.csv (新建文件)

预期输出：
- 研究报告：每个项目的详细分析（1000字以上）
- 比较表格：功能、技术栈、许可证、社区活跃度对比
- 推荐建议：基于分析给出使用建议

时间限制：预计3小时内完成
```
