# Token 消耗优化规范（纯现状版 v1.0）

更新时间：2026-04-05  
适用对象：云小虾 / 妮小虾（当前运行中的实际做法）

> 本规范只记录“已经实现并正在执行”的做法，不包含任何未来优化建议。

---

## 1. 目标

1. 在当前工作流下稳定降低无效 token 消耗。  
2. 保证结论质量不下降。  
3. 用可复现、可检查的方式执行。

---

## 2. 已落地做法（当前生效）

### 2.1 回复策略：优先短答
- 对确认类、状态类问题优先用短句直接回答。  
- 不重复铺垫，不重复背景，不重复总结。  
- 作用：降低 `out`。

### 2.2 日志处理：只取必要片段
- 查运行状态时只读取必要行（常见为 tail 最近 N 行或关键关键字命中行）。
- 不把整段大日志原样贴进回复。
- 作用：降低 `in` 与 `out`。

### 2.3 工具结果：先筛选再用于回答
- 工具输出先做人工/命令筛选，提取关键字段后再组织回复。
- 避免将完整 JSON/长文本直接灌入后续推理内容。
- 作用：降低上下文膨胀。

### 2.4 Token 统计输出：脚本化单行
- 每次通过统一脚本输出一行 token 统计：
  - `python3 /home/admin/.openclaw/workspace/scripts/token-show.py`
- 统计行不手工二次拼接，不额外扩写解释段。
- 作用：稳定输出长度，避免重复消耗。

### 2.5 统计聚合：增量扫描 + 去重
- `token-show.py` 使用状态文件做增量聚合，不做每次全量重扫。
- 聚合时对消息做去重，避免重复计入导致错误放大。
- 状态文件：
  - `memory/token-agg-state.json`
  - `memory/token-counter.json`
- 作用：减少统计链路带来的额外负担与误差。

---

## 3. 执行流程（当前标准）

1. 接收问题。  
2. 如需查状态/日志，先用工具读取最小必要范围。  
3. 组织简洁答案（先结论，后必要证据）。  
4. 追加 token 统计单行（调用 `token-show.py`）。  
5. 结束，不附加冗余说明。

---

## 4. 边界说明（防止误解）

以下能力不属于本规范（当前未自动化实现）：
- 按阈值自动触发摘要模式
- 自动拆分长回复为多条消息
- 自动生成高消耗周报/复盘

如需启用上述能力，应单独立项并更新规范版本。

---

## 5. 验收标准（按现状）

判定“符合本规范”需同时满足：
1. 回复无重复长铺垫，优先短答。  
2. 日志/工具结果只使用关键片段。  
3. token 统计由脚本单行输出。  
4. 不引入未定稿的“未来优化策略”。
