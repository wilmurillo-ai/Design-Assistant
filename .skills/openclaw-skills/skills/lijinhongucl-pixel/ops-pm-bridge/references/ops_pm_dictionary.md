# 运营-产品翻译词典

本文档定义运营语言与产品语言的对照关系，帮助双方快速理解对方表达。

---

## 一、运营语言 → 产品语言

### 活动运营类

| 运营说法 | 产品翻译 | 技术实现 |
|----------|----------|----------|
| 搞个大促活动 | 营销活动模块 | 活动配置后台 + 落地页 |
| 用户参与度高一点 | 提升用户参与率指标 | 任务系统 + 激励机制 |
| 病毒式传播 | 裂变增长功能 | 邀请码 + 分享奖励 |
| 限时秒杀 | 限时抢购模块 | 库存锁定 + 并发控制 |
| 优惠券多发点 | 优惠券批量发放 | 批量发放接口 + 人群筛选 |

### 用户运营类

| 运营说法 | 产品翻译 | 技术实现 |
|----------|----------|----------|

| 用户留存要高 | 留存率提升 | 会员体系 + 积分系统 |
| 老用户召回 | 用户召回系统 | 短信/推送 + 专属优惠 |
| 用户分层运营 | 用户标签体系 | 用户画像 + 分群推送 |

### 内容运营类

| 运营说法 | 产品翻译 | 技术实现 |
|----------|----------|----------|
| 内容要多 | 内容供给提升 | CMS系统 + 内容池 |
| 用户多看看 | 内容消费时长提升 | 推荐算法 + 内容分发 |
| 评论互动 | UGC互动模块 | 评论系统 + 审核机制 |

## 二、产品语言 → 运营语言

### 技术架构类

| 产品说法 | 运营翻译 | 业务价值 |
|----------|----------|----------|
| 微服务拆分 | 系统改造升级 | 提升系统稳定性，活动不挂 |

| A/B测试 | 方案对比测试 | 找到效果最好的方案 |

## 三、常见理解偏差案例

### 案例1：「用户参与度高」

| 角色 | 理解 | 问题 |
|------|------|------|
| 运营 | 活动页面浏览人数多 | 未定义具体指标 |
| 产品 | 用户完成任务数多 | 指标定义不一致 |
| **解决** | 统一为「用户完成核心任务的比例≥30%」 | ✅ |

### 案例2：「预算有限」

| 角色 | 理解 | 问题 |
|------|------|------|
| 运营 | 奖励金额有限 | 未明确具体数字 |
| 产品 | 开发资源有限 | 理解方向错误 |
| **解决** | 明确「奖励预算5万，开发资源2人月」 | ✅ |

### 案例3：「快速上线」

| 角色 | 理解 | 问题 |
|------|------|------|
| 运营 | 1周内上线 | 未考虑开发周期 |
| 产品 | 按正常排期2-3周 | 时间预期不一致 |
| **解决** | 明确「MVP版本1周，完整版本3周」 | ✅ |

## 四、翻译原则

1. **量化优先**：避免模糊表述，用数字说话
2. **场景明确**：说清楚在什么场景下做什么
3. **目标对齐**：确保双方理解的目标一致
4. **确认机制**：关键信息需要对方确认

---

## 五、快速翻译公式

### 运营需求 → 产品需求

```
原始：[运营说法]
翻译：
- 功能需求：[具体功能模块]
- 核心指标：[可量化指标]
- 目标值：[具体数字]
- 时间要求：[具体日期]
- 约束条件：[预算/资源/技术限制]
```

### 产品方案 → 运营说明

```
原始：[产品说法]
翻译：
- 业务价值：[对运营有什么帮助]
- 使用场景：[什么时候用什么功能]
- 上线时间：[什么时候能用]
- 需要配合：[运营需要做什么]
```
