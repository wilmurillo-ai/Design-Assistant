# Prompt Patterns

生成讲解动画时，优先按下面的顺序思考。

## 1. 提取输入信息

- 目标受众是谁
- 视频想说明什么
- 用户看完后应该记住哪 1 到 3 个点
- 原文里哪些句子适合做标题或结论

## 2. 先出 story beats

建议先产出一个简短分镜计划，再落成 `movie.json`。

示例：

```text
1. 开场：一句话说清“为什么做这个动画”
2. 问题：旧做法哪里低效
3. 步骤：新流程拆成三步
4. 结果：效率提升或画面更清楚
5. 结尾：一句 CTA 或总结
```

## 3. 再映射到 scene

每个 beat 要回答：

- 适合哪种 `scene.type`
- 这一段讲几秒
- 主要元素有哪些
- 哪个元素最先出现
- 观众的视线会先落在哪里
- 对应的 narration 是哪一句，语义停顿点在哪里

## 4. 生成 movie.json 时的约束

- 标题优先写成结论句，而不是分类名
- 每个 scene 都要有节奏，而不是把文字平均铺开
- 大段说明优先拆成多个短 scene
- 如果信息较少，宁可减少 scene，也不要硬凑页数
- `scene.durationMs` 和 `animation.startMs` 要以 narration 为依据，不要凭感觉定死
- narration 要尽量拆成逐句级 cue，scene 里的转场和字幕轨都应围绕 cue 工作
- 导出必须是最后一步，没有用户确认前只做 preview

## 5. 接入 TTS 时的约束

- 当用户要求配音时，先确认 provider
- 再确认 voice、language、speed、fallback
- 未确认前，不要开始语音合成
- 推荐把每个 scene 的 narration 直接作为单独音频片段来源

## 6. 引导用户选配置时的约束

- 先收窄，再确认；不要一次抛出所有配置项
- 视觉上优先让用户选 template，theme 默认跟模板走
- 如果用户没有风格偏好，给出推荐模板和 2 到 3 个备选，而不是开放式提问
- 如果用户说“你来定”，可以采用推荐模板和默认 theme，但要显式告诉用户你做了这个选择
- TTS 必须分两步：先 provider，再 voice
- 如果 provider 还没原生适配，必须明确说明现状，不能假装已经支持

## 7. scene 与字幕的生成建议

- 每个 scene 先确定 1 个中心观点，再把 narration 拆成 1 到 3 个 cue
- cue 应优先对应一句口播或一个完整语义单元
- 字幕轨默认直接从 cue 生成
- 标题、bullet reveal、结论卡和转场尽量锚定 cue，而不是继续平均分布

## 8. 自然语言编辑时的改写原则

当用户说：

- “第二段太啰嗦”
  改写为：缩短 `scene-02` 的正文元素文本，并减少时长。
- “第三页更像剪辑里的重点字幕”
  改写为：提升标题字号，改成 `pop` 或 `fade-up`，压缩次级说明。
- “流程三步逐条出来”
  改写为：给 `bullet-list` 增加 `itemStaggerMs`。
