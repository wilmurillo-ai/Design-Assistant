# O1: Standardize

**Pillar:** Operations
**North Star:** Percentage of critical processes documented

---

## What It Covers

SOPs (Standard Operating Procedures), process mapping, playbooks, checklists, knowledge management, and training documentation.

O1 is where Law 1 (Document Before You Scale) lives operationally. Before anything can be delegated, automated, or measured, it must be documented. O1 creates the foundation for O2 (Automate) and O3 (Instrument).

---

## Key Questions It Answers

- Which processes are revenue-critical and founder-dependent right now?
- What is the minimum viable SOP that allows someone else to execute this?
- How do we maintain documentation so it stays current as processes evolve?
- In what order should we document processes?
- What does "documented well enough to delegate" actually look like?

---

## Metrics

| Metric | Target |
|--------|--------|
| Percentage of critical processes documented | 80-100% |
| Time to productivity for a new hire | Under 30 days |
| Process adherence rate | 90%+ |
| Documentation freshness (updated in last 90 days) | 80%+ |

---

## Process Priority Matrix

Document in this order:

| Priority | Criteria | Examples |
|----------|----------|---------|
| P0 | Revenue-critical + founder-dependent | Client onboarding, proposal creation, first discovery call |
| P1 | High-frequency + error-prone without docs | Weekly reporting, client follow-up, billing |
| P2 | Team-dependent (new hire would not know) | Tool setup, meeting templates, QA checklist |
| P3 | Low-frequency + low-stakes | Edge case handling, vendor management |

Start with P0. Do not document P3 before P0 is complete.

---

## SOP Template

Every SOP has these sections:

```
Purpose: [why this process exists]
Owner: [who is accountable]
Trigger: [what starts this process]
Inputs: [what you need before starting]
Steps: [numbered, specific actions]
Outputs: [what you produce]
Quality Checklist: [how you verify it is done right]
Tools Required: [specific tools and access]
Time Estimate: [realistic time to complete]
Common Mistakes: [what goes wrong and how to avoid it]
Escalation: [when and how to escalate]
Version: [date last updated]
```

## Common Failures

**Documenting after things break:** Waiting for a delivery failure, a client complaint, or a missed handoff before documenting the process. Documentation works best when the process is fresh and working -- not when you are troubleshooting a broken one.

**Writing for yourself:** Writing documentation that only you understand because it assumes knowledge the reader does not have. The test: give the SOP to someone with no context and watch them execute it. Where they ask questions, the documentation is incomplete.

**Treating docs as permanent:** Writing documentation once and never updating it. A process that changes but whose documentation does not is worse than no documentation -- it actively misleads the person following it.

---

## Standardize Done Right

O1 is working when: A new hire in any function can onboard and execute their core responsibilities within 30 days using documentation alone, without daily founder involvement. P0 processes have current SOPs that have been tested by someone other than the author. Process adherence is measured and reviewed quarterly.
